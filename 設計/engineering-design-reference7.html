ENGINEERING DESIGN VOL.07

API設計

REST API DESIGN — INTERFACE TO THE WORLD

APIは一度公開すると簡単には変えられません。クライアントが既に使っていれば、URLやレスポンス形式を変える=破壊的変更。だからこそ最初から正しく設計する必要があります。本巻ではREST原則からエラー設計、バージョニングまで一気に整理します。

01

REST設計の原則

REST PRINCIPLES

RESTは「リソース指向」のAPI設計スタイル。名詞（モノ）をURLで表し、動詞（操作）をHTTPメソッドで表すのが核心。これを徹底するだけで、APIの一貫性が劇的に上がります。

rest-principles 6つの制約とリソース指向
// RESTの基本ルール
  • リソース = 名詞でURL表現
    /users 〇 / /getUsers ×
  • 操作はHTTPメソッドで
    GET/POST/PUT/PATCH/DELETE
  • ステートレス
    サーバはセッション状態を持たない
  • 統一インターフェース
    同じパターンの繰り返しで予測可能
  • キャッシュ可能
    GETはキャッシュOK
// 動詞ベース vs リソースベース [NG] 動詞ベース（RPC風） POST /getUserById POST /createUser POST /updateUser POST /deleteUser POST /getOrdersByUser // → 動詞が爆発する // → 一貫性なくAPIが肥大化 [OK] リソースベース（REST） GET /users // 一覧 GET /users/123 // 1件取得 POST /users // 作成 PUT /users/123 // 全置換 PATCH /users/123 // 部分更新 DELETE /users/123 // 削除 GET /users/123/orders // 関連リソース // パターンが統一されて // 「だいたい想像つく」 // これが REST の最大の利点
POINT — 「だいたい想像つく」が正義
良いAPIは「ドキュメントを見なくてもだいたい想像できる」もの。GET /products/42/reviews と書いてあれば、誰でも「商品42のレビュー一覧を取れる」と分かります。予測可能性こそRESTです。
02

リソース設計とURI命名

RESOURCE & URI NAMING

URI命名には定番のルールがあります。これらを守るだけで「センスのある」APIに見えます。逆に破ると、初見の利用者から即「あー…」と思われるので注意。

uri-naming 命名の鉄則
// 鉄則チェックリスト
  • ★★★ 名詞・複数形
    /users 〇 / /user ×
  • ★★★ 小文字・ハイフン
    /order-items 〇
  • ★★ 階層は2層まで
    /users/1/orders 〇
    /users/1/orders/2/items 深すぎ
  • ★★ 動詞を入れない
    /getUser ×
  • ★★ クエリは絞り込み専用
    ?status=active&page=2
// URI命名の良い例・悪い例 [BAD] /api/User /api/getUserList /api/users/getById?id=1 /api/order_items/5 /api/v1/orders/1/items/2/options/3/details [GOOD] /api/users /api/users/1 /api/users/1/orders /api/order-items/5 // フィルタ・ソート・ページングはクエリで GET /api/products? category=shoes& min_price=1000& sort=price_asc& page=2& per_page=20 // アクション系（どうしても動詞）の例外 POST /api/orders/42/cancel POST /api/orders/42/refund POST /api/users/1/password-reset // 「リソースの状態遷移を起こす操作」は // 動詞でいい。むしろそのほうが明確。 // 例: /cancel, /publish, /archive // 検索のような複雑系はPOST許容 POST /api/products/search { "keyword": "スニーカー", "filters": { ... } } // → GETでは送信不可能な複雑条件のとき
裏技 — 「複数形で統一」が一番楽
リソース名は常に複数形で統一すると判断に迷いません。「1件取得も /users/1」と決めれば、単数形にするか複数形にするかで悩む時間がゼロになります。
03

HTTPメソッドとステータスコード

HTTP METHODS & STATUS CODES

HTTPメソッドの意味的な違いを正しく使い分けるのがREST設計の基本。特に「べき等性（idempotent）」の概念は重要で、間違えるとリトライ処理で事故ります。

http-methods 5つのメソッドの性質
// メソッド別の性質
メソッド用途べき等安全
GET取得○○
POST作成××
PUT全置換○×
PATCH部分更新場合×
DELETE削除○×

べき等=何度実行しても結果が同じ。
安全=サーバ状態を変えない。

// ステータスコード早見表 [2xx 成功] 200 OK GET/PUT/PATCH 成功 201 Created POST で作成成功 202 Accepted 非同期処理を受付 204 No Content DELETE 成功・本文なし [3xx リダイレクト] 301 Moved Permanently 恒久移動 304 Not Modified キャッシュ有効 [4xx クライアントエラー] 400 Bad Request リクエスト不正 401 Unauthorized 認証されてない 403 Forbidden 権限なし 404 Not Found リソースなし 409 Conflict 状態競合 422 Unprocessable バリデーション失敗 429 Too Many Requests レート制限 [5xx サーバエラー] 500 Internal Server Error 想定外エラー 502 Bad Gateway 上流エラー 503 Service Unavailable メンテ等 504 Gateway Timeout 上流タイムアウト // よくあるミス: // - 全部 200 で返す（エラー情報を本文に入れる） // - エラーを全部 500 で返す // → どちらもクライアント側が困る // 認証 vs 認可 401: "あなたが誰か分からない" 403: "あなたが誰か分かるが権限なし"
POINT — べき等性は神
POSTはべき等でないので、ネットワーク切断時にリトライすると二重登録の事故が起きます。これを防ぐため「Idempotency-Key」ヘッダを使うのが定番（Stripeで有名）。同じキーでリトライしても1回しか実行されない仕組みです。
04

リクエストとレスポンス

REQUEST & RESPONSE

JSONの形式を揃えるのがプロのAPI。キー名のケース、日付フォーマット、ネスト構造。これらを最初に決めて、全エンドポイントで貫くのが鉄則です。

request-response JSON形式の流儀
// 形式の定石
  • キー名: snake_case 推奨
    (camelCase でも可、混在はNG)
  • 日時: ISO 8601 / RFC 3339
    2026-05-13T09:00:00+09:00
  • 金額: 整数で最小単位
    円なら円、ドルならセント
  • 真偽値: true/false
    0/1や"yes"/"no"を避ける
  • null vs 省略: 流儀を統一
  • 配列はキーで包む
    { "items": [...] }
// 良いレスポンス例 [一覧取得] GET /api/orders?page=2 HTTP/1.1 200 OK { "data": [ { "id": 42, "status": "paid", "total_yen": 5800, "created_at": "2026-05-13T09:00:00+09:00", "customer": { "id": 1, "name": "田中太郎" } } ], "meta": { "page": 2, "per_page": 20, "total": 143, "total_pages": 8 } } [作成] POST /api/orders HTTP/1.1 201 Created Location: /api/orders/43 { "data": { "id": 43, "status": "pending", ... } } // 鉄則: // - 1件でも一覧でも data キーで包む // - meta にページング情報 // - ID は POST 後に必ず返す // - Location ヘッダで新URLを示す
裏技 — 「JSON:API」または「Google Style」
迷ったら業界標準仕様に乗っかるのが楽。JSON:API（jsonapi.org）は厳密な仕様、Google API Design Guideは実用的なガイドライン。社内で流儀を1から決めるよりこれらに倣うほうが、議論時間を節約できます。
05

エラー設計

ERROR DESIGN

エラーレスポンスの設計はAPIの品質を一発で見抜ける指標。「{"error":"failed"}」しか返さないAPIはクライアントが地獄を見ます。機械が処理できる構造化エラーを返すのが現代の流儀です。

error-design 構造化されたエラーレスポンス
// 良いエラーレスポンスの要素
  • HTTPステータスコード: 4xx/5xx を正しく
  • エラーコード: 機械処理用の文字列ID
  • メッセージ: 人間向けの説明
  • フィールド情報: バリデーション失敗の対象
  • ドキュメントURL: 詳細リンク
  • リクエストID: サポート問合せ用

RFC 7807 (Problem Details)という標準仕様もあります。準拠すると外部公開APIの体裁が整います。

// 単純なエラー HTTP/1.1 404 Not Found { "error": { "code": "ORDER_NOT_FOUND", "message": "指定された注文は存在しません", "request_id": "req_abc123" } } // バリデーションエラー (複数) HTTP/1.1 422 Unprocessable Entity { "error": { "code": "VALIDATION_FAILED", "message": "入力に誤りがあります", "details": [ { "field": "email", "code": "INVALID_FORMAT", "message": "メール形式が不正です" }, { "field": "age", "code": "OUT_OF_RANGE", "message": "0以上150以下で入力してください" } ], "request_id": "req_xyz789" } } // クライアント側で: // - code を見て分岐処理 // - message をユーザーに表示 // - field で該当UIをハイライト // - request_id をサポートに伝達
POINT — エラーメッセージはユーザー向け
エラーのmessageはそのままユーザーに見せても困らない文言にするのが現代の作法。スタックトレースやSQLエラーをそのまま返すのは情報漏洩リスクです。内部詳細はログにだけ残し、ユーザーには優しい言葉で。
06

ページネーション

PAGINATION

一覧取得APIで「全件返す」のは絶対NG。10万件あったら即タイムアウトです。ページング方式にはオフセット型とカーソル型があり、用途で使い分けます。

pagination 2方式の使い分け
// オフセット型 vs カーソル型
オフセットカーソル
パラメータpage/limitcursor/limit
ジャンプ任意のページに次/前のみ
性能深くなると遅い常に高速
新規追加に強いズレるズレない
向き管理画面SNS/タイムライン
// オフセット型 GET /api/products?page=3&per_page=20 { "data": [...], "meta": { "page": 3, "per_page": 20, "total": 523, "total_pages": 27 } } // SQL内部 SELECT * FROM products ORDER BY id LIMIT 20 OFFSET 40; // page=1000 みたいになると、OFFSET 19980 // → 19980件スキップしてから20件取得 // → 致命的に遅くなる // カーソル型 (タイムライン向き) GET /api/posts?cursor=eyJpZCI6MTAwfQ&limit=20 { "data": [...], "meta": { "next_cursor": "eyJpZCI6ODB9", "has_more": true } } // SQL内部 SELECT * FROM posts WHERE id < 100 // カーソル位置から ORDER BY id DESC LIMIT 20; // どんなに深くてもインデックスで高速 // 投稿が増えてもズレない
TIP — 詰まりポイント
オフセット型は「3ページ目」のような表示と相性が良いですが、データ量が多いと深いページで激遅。Twitter/Instagramのような無限スクロールUIなら絶対にカーソル型。用途を最初に決めて選びましょう。
07

認証・認可

AUTHENTICATION & AUTHORIZATION

認証（誰か?）と認可（何ができる?）は別物。APIではBearer Token方式が主流で、トークンにはJWTか不透明（Opaque）トークンを使います。OAuth 2.0は標準仕様として覚えておきましょう。

auth 認証方式の選択
// 認証方式の比較
方式用途
Bearer (JWT)マイクロサービス間で人気。DB問い合わせ不要。失効が難しい
Bearer (Opaque)ランダム文字列。失効は楽だがDB確認が必要
API Keyサーバ間連携。シンプル
OAuth 2.0第三者にアクセス委譲
Session Cookie同一ドメインWebに最適
// Bearer Token の使い方 POST /api/auth/login { "email": "u@example.com", "password": "..." } → HTTP/1.1 200 OK { "access_token": "eyJhbGc...", "refresh_token": "abc123...", "expires_in": 3600, "token_type": "Bearer" } // 以後のリクエスト GET /api/orders Authorization: Bearer eyJhbGc... // JWT の中身 (Base64で3つに分かれる) { "sub": "user_123", "iat": 1715000000, "exp": 1715003600, "roles": ["user", "admin"] } // 認可: ロールベース vs 属性ベース [RBAC: ロール] admin → 全操作OK editor → 投稿の編集OK viewer → 閲覧のみ [ABAC: 属性] 「自分が作成した投稿のみ編集可」 「同じ組織のリソースのみアクセス可」 → より細やかな制御が可能
POINT — JWTの落とし穴
JWTは失効させにくいのが最大の弱点。一度発行すると有効期限まで強制無効化できません。短い有効期限+リフレッシュトークン方式が定番。「ログアウト即無効化」が必須ならOpaqueトークン+Redisのほうが現実的です。
08

バージョニング戦略

API VERSIONING

APIは必ず後から変更したくなります。破壊的変更を入れる時に、既存クライアントを壊さないようにする仕組みがバージョニング。URLパス方式がシンプルで主流です。

api-versioning 破壊的変更との戦い
// バージョニング方式の比較
方式例
URLパス/v1/users
シンプル。最も普及
ヘッダAPI-Version: 2
URLを汚さないが、テストしにくい
クエリ?v=2
カジュアル。あまり推奨されない
Accept ヘッダapplication/vnd.x.v2+json
厳密だが複雑

結論: 迷ったらURLパス方式（/v1/）。GitHub、Twitter、Stripeほぼ全部これ。

// バージョン管理の鉄則 [Step 1] 破壊的変更を見極める 破壊的変更: - 既存フィールドの削除/型変更 - 必須項目の追加 - エンドポイントの削除 - レスポンス構造の変更 互換変更（バージョン上げ不要）: - 新フィールドの追加（任意） - 新エンドポイントの追加 - エラーメッセージの改善 [Step 2] 古いバージョンを残す GET /api/v1/users // 既存 GET /api/v2/users // 新規 (破壊的変更含む) // 両方を並行運用。v1利用者を移行させる。 [Step 3] 廃止予告とサンセット Response Header: Deprecation: true Sunset: "Sun, 31 Dec 2026 23:59:59 GMT" Link: <https://api.example.com/v2/users>; rel="successor-version" // クライアントに「移行してね」と伝える // 半年〜1年の猶予を持って廃止 [Step 4] 旧バージョン停止 HTTP/1.1 410 Gone { "error": { "code": "VERSION_RETIRED", "message": "v1は廃止されました。v2をご利用ください" } } // 破壊的変更は最後の手段。 // なるべく互換変更で凌ぐのが本当のプロ。
