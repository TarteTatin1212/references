設計原則 — SOLID & 鉄則

DESIGN PRINCIPLES — SOLID AND BEYOND

設計原則は「やった方がいい」ではなく「やらないと後で破綻する」鉄則です。先人が数十年かけて見つけた地雷の位置を、私たちはタダで教えてもらえます。本巻ではSOLID 5原則と、DRY/KISS/YAGNI、凝集と結合を扱います。

01

設計原則の役割

WHY PRINCIPLES MATTER

設計原則は「迷った時の判断基準」です。クラスを分けるべきか / 継承すべきか / どう依存させるか。こうした分岐点で原則を知っていれば、根拠を持って選べます。感覚や好みではなく、原則で設計するのが現場の流儀です。

why-principles SOLIDの位置づけ
// SOLID = 5原則の頭文字
  • SSingle Responsibility 単一責任
  • OOpen/Closed 開放閉鎖
  • LLiskov Substitution リスコフ置換
  • IInterface Segregation インターフェース分離
  • DDependency Inversion 依存性逆転

提唱者は Robert C. Martin（通称ボブおじさん）。オブジェクト指向設計の定番中の定番です。

// 原則を知らずに書いたコード class UserService { register(user) { // DB保存も db.insert(user); // メール送信も smtp.send(user.email, "welcome"); // ログも fs.append("log.txt", ...); // 課金処理も stripe.charge(user.card); // PDF生成も pdf.create(...); } } // 問題点: // - メール仕様が変わるとUserServiceを修正 // - DBが変わるとUserServiceを修正 // - 課金が変わるとUserServiceを修正 // → 変更理由が5つもある = SRP違反 // このコードは半年後、必ず崩壊する。
POINT — 原則は道具、目的じゃない
SOLIDを守ること自体を目的にすると過剰設計になります。原則の本当の目的は「変更に強いコード」。読みやすく、テストしやすく、後から直しやすい。それを実現するための道具がSOLIDです。
02

SRP — 単一責任の原則

SINGLE RESPONSIBILITY PRINCIPLE

SRP「クラスを変更する理由は、たった1つであるべき」。一見当たり前に見えますが、現場のコードはほぼ全てこの原則を破っています。クラス名に「Manager」「Util」「Helper」が付いていたら、まず疑ってください。

srp-example 責務を1つに絞る
// SRP違反の見抜き方
  • ★★★ クラスの説明に 「と」 が入る
    例: ユーザー登録「と」メール送信「と」課金
  • ★★★ メソッドの主語が違う
    例: register / sendMail / chargeCard
  • ★★ インポートが多すぎる
    外部ライブラリを5種類以上使う = 怪しい
  • ★★ テスト時にモックが10個必要
// SRP適用前: 1クラスが全部やる class UserService { register(u) { db.insert(u); // 永続化 smtp.send(...); // 通知 stripe.charge(...); // 課金 logger.info(...); // ログ } } // SRP適用後: 責務ごとに分割 class UserRepository { save(u) { db.insert(u); } } class WelcomeMailer { send(u) { smtp.send(...); } } class PaymentProcessor { charge(u) { stripe.charge(...); } } class UserRegistration { constructor( private repo: UserRepository, private mailer: WelcomeMailer, private payment: PaymentProcessor, ) {} register(u) { this.repo.save(u); this.mailer.send(u); this.payment.charge(u); } } // 各クラスは「変更理由が1つ」 // メール仕様の変更 → Mailerだけ修正
裏技 — 「アクター」で考える
SRPの本当の定義は「クラスを変更させる『アクター（人/役割）』は1人であるべき」。経理チーム向けの集計と、マーケ向けの集計を同じクラスに入れると、両者の要望が衝突して悲劇が起きます。「誰がそれを要求するか」で分けましょう。
03

OCP — 開放閉鎖の原則

OPEN/CLOSED PRINCIPLE

OCP「拡張に対して開いており、修正に対して閉じている」。機能追加の時に既存コードを書き換えず、新しいコードを足すだけで済む状態を目指します。これが守られていると、リグレッション（壊れる）が激減します。

ocp-example if/switchの分岐をなくす
// OCPを実現する道具
  • インターフェース / 抽象クラス
    共通の型を定義
  • ポリモーフィズム
    同じメソッド名で違う動き
  • Strategyパターン
    アルゴリズムを差し替え可能に

典型サイン: if (type == "A") ... else if (type == "B") ... が出てきたらOCP違反の合図。

// OCP違反: 新方式を追加するたびに修正 class Notifier { send(type, msg) { if (type === "email") { smtp.send(msg); } else if (type === "sms") { twilio.send(msg); } else if (type === "slack") { slack.post(msg); } // LINE通知を追加するたびに // このメソッドを書き換える… } } // OCP適用: 拡張に開く interface NotificationChannel { send(msg: string): void; } class EmailChannel implements NotificationChannel { send(msg) { smtp.send(msg); } } class SmsChannel implements NotificationChannel { send(msg) { twilio.send(msg); } } // LINEを追加: 新しいクラスを作るだけ class LineChannel implements NotificationChannel { send(msg) { line.post(msg); } } // Notifier本体は1行も触らない // = 既存テスト全て通る
POINT — 「予想される拡張」だけ
全てをOCPで設計すると過剰設計になります。「ここは増えそうだ」と現実的に見える箇所にだけ適用するのが現場の知恵。1回しか使わないなら直接書いてOK。3回目に登場したらOCPを検討する、くらいのバランス感です。
04

LSP — リスコフ置換の原則

LISKOV SUBSTITUTION PRINCIPLE

LSP「派生クラスは、基底クラスと置き換えても問題なく動くべき」。継承を使うときの安全装置です。LSP違反のコードは「あれ?サブクラスを渡したら急に動かなくなった」というバグの温床になります。

lsp-example 継承の落とし穴
// 有名な「正方形と長方形」問題
Rectangle (長方形) setWidth(w) setHeight(h) area() = w * h ↑ extends Square (正方形) setWidth → heightも変える setHeight → widthも変える 問題: rect.setWidth(5); rect.setHeight(10); assert(rect.area() == 50); Square を渡すと… area() == 100 になる! (LSP違反)

数学的には正方形は長方形の特殊形。でも振る舞いとしては別物。継承は「is-a」だけでは不十分で「振る舞いとして置換可能」であることが要求されます。

// LSP違反の例 class Bird { fly() { /* 飛ぶ */ } } class Penguin extends Bird { fly() { throw new Error("ペンギンは飛べない"); } } function makeFly(bird: Bird) { bird.fly(); // Penguinが来ると爆発 } // 解決: 階層を見直す class Bird { name: string; } class FlyingBird extends Bird { fly() { ... } } class SwimmingBird extends Bird { swim() { ... } } class Sparrow extends FlyingBird {} class Penguin extends SwimmingBird {} // もうペンギンは飛ばないし、 // makeFly()も安全に動く。
裏技 — 継承より委譲
迷ったら継承（extends）より委譲（コンポジション）を選ぶのが現代の流儀。「○○は△△である」より「○○は△△を持っている」のほうが、変更に強い設計になります。継承は本当に置換可能な時だけ使いましょう。
05

ISP — インターフェース分離の原則

INTERFACE SEGREGATION PRINCIPLE

ISP「使わないメソッドへの依存を強制してはならない」。巨大なインターフェースを作って「あれもこれも実装してね」とすると、実装側は使わないメソッドを空実装で埋めるハメになります。これがISP違反。

isp-example 大きなインターフェースを分割
// ISP違反の典型
  • 巨大なインターフェース
    メソッドが10個以上
  • 空実装の存在
    { /* 使わない */ } や throw new NotSupported()
  • 用途別の実装
    同じインターフェースで全く違う動作

解決策: 「役割ごとに小さく分割」。1つのクラスが複数のインターフェースを実装してもOK。

// ISP違反: 太いインターフェース interface Worker { work(): void; eat(): void; sleep(): void; getSalary(): number; } // ロボットを実装する時… class Robot implements Worker { work() { ... } eat() { /* 何もしない */ } // 違反 sleep() { /* 何もしない */ } // 違反 getSalary() { throw new Error("無給"); // 違反 } } // ISP適用: 役割で分割 interface Workable { work(): void; } interface Feedable { eat(): void; } interface Sleepable { sleep(): void; } interface Payable { getSalary(): number; } class Human implements Workable, Feedable, Sleepable, Payable { ... } class Robot implements Workable { ... } // Robot は不要なメソッドを持たない
TIP — クライアント目線で分ける
インターフェースを設計する時は「呼び出し側（クライアント）が何を必要としているか」を起点に考えます。「実装側の都合」で1つにまとめると、必ずISP違反になります。
06

DIP — 依存性逆転の原則

DEPENDENCY INVERSION PRINCIPLE

DIP「上位モジュールは下位モジュールに依存してはならない。両者とも抽象に依存すべき」。SOLIDで最も重要かつ最も難しい原則。これを理解するとアーキテクチャ設計が一気に分かるようになります。

dip-example 依存の向きを逆転させる
// 依存の矢印を逆さに
[Before: 上が下に依存] UserService (上位) ↓ 依存 MySQLRepository (下位) → DB変えるとUserServiceも修正 [After: 両者とも抽象に依存] UserService (上位) ↓ [UserRepository] (抽象) ↑ MySQLRepository (下位) → DB変えてもUserServiceは無傷

抽象（インターフェース）を挟むことで、矢印の向きを上→下から下→中央へと逆転させる。これが「逆転」の意味です。

// DIP違反: 具体クラスに直接依存 class MySQLUserRepo { save(u) { mysql.query(...); } } class UserService { private repo = new MySQLUserRepo(); // ↑ 具体クラスに固定 register(u) { this.repo.save(u); } } // 問題: // - DB変更時にUserServiceも修正 // - テストでモック差し替え不可 // DIP適用: 抽象に依存 interface UserRepository { save(u: User): void; } class MySQLUserRepo implements UserRepository { save(u) { mysql.query(...); } } class UserService { constructor( private repo: UserRepository // ↑ 抽象（interface）に依存 ) {} register(u) { this.repo.save(u); } } // 使い方 new UserService(new MySQLUserRepo()); // 本番 new UserService(new InMemoryRepo()); // テスト
POINT — DIP = DI ではない
DIP（依存性逆転の原則）と DI（依存性注入）は別物。DIPは「考え方」、DIは「実現手段の1つ」です。DIを使えばDIPが守られるとは限らない（具体クラスを注入してたら意味なし）。抽象に依存しているかを常に確認しましょう。
07

DRY / KISS / YAGNI と凝集・結合

DRY, KISS, YAGNI & COHESION/COUPLING

SOLID以外にも、コードの品質を支える名言級の鉄則があります。DRY / KISS / YAGNIはTシャツに書きたいレベルの3大原則。そして凝集と結合はモジュール設計の最重要指標です。

dry-kiss-yagni 3大鉄則
// 3つの鉄則
  • DRYDon't Repeat Yourself
    同じコードを2度書くな。3回目に登場したら必ず共通化
  • KISSKeep It Simple, Stupid
    シンプルに保て。賢く見せようとして複雑にするな
  • YAGNIYou Aren't Gonna Need It
    必要になるまで作るな。「将来のため」の8割は使われない
// DRY違反: 同じロジックがコピペ calcDiscountForGold(p) { return p * 0.9; } calcDiscountForSilver(p) { return p * 0.95; } // DRY適用 const RATES = { gold: 0.9, silver: 0.95 }; calcDiscount(p, rank) { return p * RATES[rank]; } // YAGNI違反 class User { name: string; middleName: string; // 「将来必要かも」 preferredLang: string; // 「多言語化のため」 timezone: string; // 「グローバル展開で」 } // → どれも5年経っても使われない // YAGNI適用: 今要るものだけ class User { name: string; } // 必要になったら、その時に追加
cohesion-coupling 凝集度高く、結合度低く
// モジュール設計の2大指標
指標意味目標
凝集度
Cohesion		1モジュール内で関連性が強いか高く
結合度
Coupling		モジュール間の依存の強さ低く

High Cohesion, Low Coupling。
これがあらゆる設計の合言葉です。SOLIDも結局この2つを実現するための道具と言えます。

// 結合度の段階（低い順） [1. メッセージ結合] ★最弱 メソッド呼び出しのみ payment.charge(amount) [2. データ結合] プリミティブな引数のみ calc(price, rate) [3. スタンプ結合] オブジェクトを丸ごと渡す save(user) [4. 制御結合] フラグで動きを変える do(flag, data) // flagがcontrol [5. 共通結合] グローバル変数を共有 global currentUser [6. 内容結合] ★最悪 他クラスの内部に直接アクセス user._internal._password = 123 // なるべく上のレベルに留めること
