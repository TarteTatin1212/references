01

デザインパターンとは

WHAT ARE DESIGN PATTERNS

デザインパターンは「コードのコピペ」ではなく「設計のコピペ」です。「この問題ならこういう構造で解く」という型を共有することで、設計の議論が早くなり、コードの可読性も上がります。

gof-23-patterns GoF 23パターンの分類
// GoFパターンの3カテゴリ
  • 生成Creational（5種）
    オブジェクト生成の工夫
    Factory / Singleton / Builder / Prototype / Abstract Factory
  • 構造Structural（7種）
    クラスの組み合わせ方
    Adapter / Decorator / Facade / Proxy / Composite ほか
  • 振舞Behavioral（11種）
    オブジェクト間の通信
    Strategy / Observer / Template / Command / Iterator ほか

全23種ありますが、現場で本当に使うのは10種類くらい。まずはそれだけ覚えれば十分です。

// パターンを知ると会話が早くなる [パターン語彙なし] A: 「決済方法を後から増やせるように、 共通のインターフェースを作って、 各方式のクラスを実装で、 呼び出し側で差し替えられるように…」 B: 「えーっと、つまり?」 [パターン語彙あり] A: 「Strategyパターンで行こう」 B: 「了解、PaymentStrategy作るね」 // パターン名 = 設計の共通語 // 知ってる人同士なら一瞬で伝わる // ただし! // パターン名を振り回すのは初心者の証拠。 // 本当の達人は「これStrategyっぽいね」 // くらいの距離感で扱う。
POINT — パターンは万能薬じゃない
パターンを「適用すること」を目的にしてはいけません。「シンプルなif文で済むのにStrategyで書く」は典型的な過剰設計です。同じパターンが3回以上現れた時に、はじめてパターンとして抽出するくらいでちょうどいいです。
02

Strategy — 振る舞いを差し替える

STRATEGY PATTERN

Strategyアルゴリズム（戦略）をオブジェクトとして取り出し、差し替え可能にするパターン。GoFパターンで最も使用頻度が高いと言ってもよい定番中の定番です。OCPの実現手段でもあります。

strategy-pattern if/switchの分岐をクラスに変換
// 適用場面
  • 決済方法の選択
    クレカ / 銀行 / コンビニ / PayPay
  • 送料計算
    ヤマト / 佐川 / 自社便
  • ソートアルゴリズム
    クイックソート / マージ / バブル
  • 圧縮形式
    zip / gzip / tar

「複数のやり方がある処理」を見たら反射的にStrategy。これだけ覚えれば中級者の入口です。

// Strategy パターンの基本形 // ① 戦略のインターフェース interface PaymentStrategy { pay(amount: number): Result; } // ② 具体的な戦略を複数実装 class CreditCardPayment implements PaymentStrategy { pay(amount) { // Stripeでクレカ決済 return stripe.charge(amount); } } class PayPayPayment implements PaymentStrategy { pay(amount) { return paypay.request(amount); } } // ③ 戦略を持つコンテキスト class Checkout { constructor( private strategy: PaymentStrategy ) {} execute(amount) { return this.strategy.pay(amount); } } // ④ 利用側で戦略を選んで注入 const checkout = new Checkout( new CreditCardPayment() ); checkout.execute(5000);
TIP — 現場では「Policy」とも呼ぶ
Strategyパターンは「Policy（方針）」という別名でも呼ばれます。「割引ポリシー」「課金ポリシー」など、ビジネスルールを差し替える文脈ではPolicyのほうが自然です。同じ構造でも、命名で意図が伝わりやすくなります。
03

Factory — 生成を任せる

FACTORY PATTERN

Factoryオブジェクトの生成手順を専用クラスに任せるパターン。newを散らかさず1箇所に集めることで、生成ロジックの変更が楽になります。条件分岐で型を選ぶ場面で頻出。

factory-pattern 生成の責務を集約
// Factoryが解決する問題
  • newが各所に散らかる
    後から生成手順を変えたい時、全箇所修正…
  • 生成に複雑な手順
    引数の組み立て、依存の解決、初期化処理
  • 条件で型が変わる
    「設定によってA or Bのインスタンスを返す」

DIコンテナ（Spring、NestJSなど）は巨大なFactoryの集合体と捉えてもOK。

// Factory パターンの基本形 // ① 生成対象（共通型） interface Logger { log(msg: string): void; } class ConsoleLogger implements Logger { ... } class FileLogger implements Logger { ... } class CloudLogger implements Logger { ... } // ② Factory が選んで生成 class LoggerFactory { static create(env: string): Logger { if (env === "local") { return new ConsoleLogger(); } if (env === "prod") { return new CloudLogger({ endpoint: process.env.LOG_URL, apiKey: process.env.LOG_KEY, }); } return new FileLogger("/var/log/app.log"); } } // ③ 利用側 const logger = LoggerFactory.create( process.env.NODE_ENV ); logger.log("hello"); // 利用側は ConsoleLogger なのか // CloudLogger なのか知らなくていい
裏技 — Builderと使い分け
Factoryは「型を選んで生成」、Builderは「複雑な引数を組み立てて生成」。同じ「生成系」でも目的が違います。引数が10個ある巨大オブジェクトはFactoryよりBuilderの出番。User.builder().name(...).age(...).build() の形が代表例。
04

Singleton — 唯一の存在

SINGLETON PATTERN

Singletonシステム全体で1つしか存在しないことを保証するパターン。便利な反面、アンチパターン扱いされることも多いクセモノです。テスタビリティを下げる副作用があり、使用は慎重に。

singleton-pattern 便利だが諸刃の剣
// 適切なSingleton
  • 設定オブジェクト
    環境変数を読み込んだ結果
  • DBコネクションプール
    1接続あたりコスト高
  • ロガー
    全体で1つあれば十分
  • キャッシュ
    共有が前提

避けたい使い方: 状態を持つビジネスロジックをSingletonにする → グローバル変数と同じ罠にハマる

// Singleton の基本形 class AppConfig { private static instance: AppConfig; private settings: Record<string, any>; // ② コンストラクタを隠す private constructor() { this.settings = loadFromEnv(); } // ③ 唯一の取得口 static getInstance(): AppConfig { if (!AppConfig.instance) { AppConfig.instance = new AppConfig(); } return AppConfig.instance; } get(key: string) { return this.settings[key]; } } // 利用 const cfg = AppConfig.getInstance(); const port = cfg.get("PORT"); // ⚠ よくある罠 // テストで設定を差し替えにくい // → DI（依存性注入）のほうが現代的 // → SpringやNestJSの@Singletonは別物
TIP — Singletonの代わりにDI
現代的なコードでは「コンテナ管理のシングルトン」が主流。Springの @Component や NestJS の @Injectable は事実上のSingletonですが、テスト時に差し替え可能。クラス側でグローバル状態を持つ素のSingletonは避けるのが無難です。
05

Observer — 通知の仕組み

OBSERVER PATTERN

Observerあるオブジェクトの状態変化を、複数の関心ある側に自動通知するパターン。イベント駆動の根幹。UIフレームワーク、メッセージング、Pub/Subなど、現代の至るところで使われています。

observer-pattern 発行側と購読側を疎結合に
// 適用場面
  • UIの再描画
    Reactのstate変更で再レンダー
  • イベントバス
    ボタンクリック → 複数のリスナー
  • メール送信トリガー
    「注文確定」イベントに複数の購読者
  • Pub/Sub
    Kafka、Redis、RabbitMQ等

「○○が起きた時に△△する」という構造があれば、それはObserver。

// Observer の基本形 interface Observer<T> { update(event: T): void; } class EventEmitter<T> { private observers: Observer<T>[] = []; subscribe(o: Observer<T>) { this.observers.push(o); } unsubscribe(o: Observer<T>) { this.observers = this.observers.filter(x => x !== o); } emit(event: T) { this.observers.forEach(o => o.update(event)); } } // 注文確定イベントを発行する側 const orderEvents = new EventEmitter<Order>(); // 各購読者を登録 orderEvents.subscribe({ update: (order) => sendConfirmationMail(order) }); orderEvents.subscribe({ update: (order) => updateInventory(order) }); orderEvents.subscribe({ update: (order) => analytics.track(order) }); // 発行 orderEvents.emit(newOrder); // → 3つの処理が自動的に走る
POINT — 関心の分離の極致
Observerの本当の威力は「発行側は誰が受け取るか知らなくていい」こと。新しい購読者を追加しても発行側のコードは1行も変わりません。これがOCPの理想形です。
06

Decorator — 装飾で機能追加

DECORATOR PATTERN

Decorator既存クラスを変更せず、機能を後付けするパターン。継承の代わりに「ラップ」で機能追加するため、組み合わせが自由。HTTPミドルウェア、ストリーム処理、ログ・キャッシュの追加などで活躍します。

decorator-pattern 機能を重ね掛けする
// 直感的なイメージ
[元のオブジェクト] Coffee (300円) │ ＋ Milk装飾 (50円) │ ＋ Sugar装飾 (30円) │ ＋ Cinnamon装飾 (40円) ↓ 合計 420円

同じインターフェースを保ったまま、機能をどんどん重ねていけるのが特徴。

HTTPで言えば「認証 → ログ → キャッシュ → 本処理」のミドルウェアチェーンが完全にこれ。

// Decorator の基本形 interface DataSource { read(): string; write(data: string): void; } // ① 基本実装 class FileDataSource implements DataSource { read() { return fs.read("data.txt"); } write(d) { fs.write("data.txt", d); } } // ② 装飾の基底 abstract class Decorator implements DataSource { constructor(protected source: DataSource) {} read() { return this.source.read(); } write(d) { this.source.write(d); } } // ③ 暗号化を追加 class Encrypted extends Decorator { read() { return decrypt(super.read()); } write(d) { super.write(encrypt(d)); } } // ④ 圧縮を追加 class Compressed extends Decorator { read() { return unzip(super.read()); } write(d) { super.write(zip(d)); } } // ⑤ 重ね掛け const source = new Encrypted( new Compressed( new FileDataSource() ) ); // → 書き込み: zip → encrypt → file // → 読み込み: file → decrypt → unzip
裏技 — TypeScriptの@デコレータ
TypeScriptやPython、Javaの「@アノテーション」は、このDecoratorパターンの構文糖衣です。@Log @Cache のように複数積み重ねできるのは、まさにこのパターン由来。
07

Adapter — 形を合わせる

ADAPTER PATTERN

Adapterインターフェースの違うクラス同士をつなぐパターン。海外旅行のコンセント変換アダプタと同じ発想。既存ライブラリやレガシーAPIをラップする場面で大活躍します。

adapter-pattern 外部の形を内部の形に変換
// よくある適用場面
  • 外部APIラッパー
    Stripe / Twilio / Slack 等の差を吸収
  • レガシー連携
    古いシステムのSOAPをREST風に
  • ライブラリ差し替え
    moment → dayjs 移行時のクッション
  • テスト用ダブル
    本物の代わりにモックを噛ませる
// Adapter の基本形 // 内側で使いたい統一インターフェース interface SmsClient { send(to: string, body: string): Promise<void>; } // Twilioを使うアダプタ class TwilioAdapter implements SmsClient { constructor(private client: TwilioSDK) {} async send(to: string, body: string) { // TwilioのAPI形式に変換 await this.client.messages.create({ from: process.env.TWILIO_FROM, to: to, body: body, }); } } // AWS SNSを使うアダプタ class SnsAdapter implements SmsClient { constructor(private sns: AWS.SNS) {} async send(to: string, body: string) { await this.sns.publish({ PhoneNumber: to, Message: body, }).promise(); } } // 使う側は SmsClient だけを知っていればOK function notify(sms: SmsClient, user) { return sms.send(user.phone, "ご注文確定"); } // 後でTwilio→SNSに乗り換えても // 利用側のコードは一切変わらない
POINT — 「腐敗防止層」の正体
DDDで出てくる「腐敗防止層（Anti-Corruption Layer）」は、要するにAdapterの集合です。外部システムの汚い形がドメイン層に染み込まないよう、境界に防壁を立てる。これだけで設計の品が一段上がります。
08

Template Method — 流れを固定

TEMPLATE METHOD PATTERN

TemplateMethod処理の大枠は固定し、細部だけサブクラスに任せるパターン。「同じ流れだけど、一部だけ違う」処理を統一する時に有効。フレームワーク内部でよく使われる構造です。

template-method-pattern 骨格 + 差分実装
// 構造のイメージ
基底クラスに「流れ」を書く process() { step1(); ← 共通 step2(); ← サブクラス毎 step3(); ← 共通 step4(); ← サブクラス毎 } サブクラスはstep2/4だけ実装

「順番は決まってるけど内容が違う処理」の典型。
例: レポート生成（読込→加工→出力で形は同じ、ファイル形式だけ違う）
ETL処理（抽出→変換→保存）
テスト（setup→exec→teardown）

// Template Method の基本形 abstract class ReportGenerator { // テンプレート: 流れを固定 generate(): Buffer { const data = this.fetchData(); const formatted = this.format(data); return this.render(formatted); } // 共通処理 protected fetchData() { return db.query("SELECT ..."); } // サブクラスで実装 protected abstract format(d: any): any; protected abstract render(d: any): Buffer; } class PdfReport extends ReportGenerator { protected format(d) { return toPdfLayout(d); } protected render(d) { return pdfLib.make(d); } } class CsvReport extends ReportGenerator { protected format(d) { return toRows(d); } protected render(d) { return Buffer.from(d.join("\n")); } } // 利用側 new PdfReport().generate(); new CsvReport().generate();
TIP — Strategy との違い
Template Methodは継承で差分を埋め、Strategyは委譲で差分を埋めます。最近は「継承より委譲」の流れがあるので、迷ったらStrategyのほうが現代的。ただし「流れが固定で部品だけ差し替えたい」ケースではTemplate Methodがピタリとハマります。
