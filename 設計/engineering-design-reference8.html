01

なぜ設計を書くか

WHY DOCUMENT DESIGN

設計ドキュメントには3つの読者がいます。「他人」「未来の自分」「現在の自分」。書く行為そのものが思考を整理し、後から読む人が高速で全体像を掴め、自分の頭の中の曖昧さも炙り出されます。「コードがあれば十分」は幻想です。

why-docs 設計を書く3つの理由
// 3人の読者
  • 他人新メンバー
    「このシステムどうなってる?」を1日で理解
  • 未来6ヶ月後の自分
    「なんでこう作ったんだっけ?」の答え
  • 現在書いてる今の自分
    書くと思考が整理される

「書ける = 理解している」。書けないなら理解が不十分という証拠です。設計を書く時間は無駄ではなく投資。

// ドキュメントの種類と寿命 [長期保存] 概念モデル / アーキテクチャ - 5年単位で残る - システムの本質を記述 - ADR、C4 Lv1-2図 [中期] 機能設計書 - リリース単位で残る - ユースケース、データモデル - 機能仕様書 [短期] 議事録 / 検討メモ - 議論の経過を残す - すぐに陳腐化する - 重要な結論はADRに昇格させる // ドキュメントの理想: // 「コードで表現できることはコードで」 // 「コードで表現できないことを文書で」 // 表現できないこと: // - なぜそう作ったか (Why) // - 他の選択肢と比べた理由 // - システム全体の俯瞰図 // - 非機能要件と制約 // - 業務ドメインの背景
POINT — Whyを書け、Howは書くな
最も価値あるドキュメントは「なぜそう作ったか」を書いたもの。コードを見れば How（どう動くか）は分かりますが、Why（なぜそう決めたか）はコードに残りません。意思決定の理由を書くのがドキュメントの本義です。
02

UML主要図

UML ESSENTIALS

UMLは13種類の図がありますが、実務で使うのは4〜5種類だけ。クラス図、シーケンス図、状態遷移図、ユースケース図、アクティビティ図。これさえ押さえれば、設計コミュニケーションで困ることはありません。

uml-overview 実用5図
// 用途別の使い分け
図何を表す
クラス図静的構造。型と関係
シーケンス図動的な対話。時系列
状態遷移図状態の変化と条件
ユースケース図誰が何をするか
アクティビティ図処理フロー

使い分けの目安: 「何が(クラス)、どんな時(状態)、どう動くか(シーケンス)」。3点セットで多くのケースをカバーできます。

// ユースケース図の例 [注文システム] ┌──────────────────────┐ │ │ 顧客→ 商品検索 │ │ │ 顧客→ カートに追加 │ │ <<include>> │ │ ↓ │ 顧客→ 注文確定 ─── ログイン │ │ │ 顧客→ 注文履歴閲覧 │ │ │ 管理者→ 在庫管理 │ 管理者→ 売上集計 │ └──────────────────────┘ 記法ポイント: - アクター (人/外部システム) - ユースケース (楕円で表現) - includes 必ず実行する関連 - extends 任意で実行する関連 // 厳密性より「全体像が伝わるか」が大事 // 矢印の向きで悩むより、 // 読み手の理解優先で柔軟に
TIP — UMLは「方言」を許容
UMLの厳密な記法を完全に守る必要はありません。チーム内で意味が伝わるなら、ある程度の崩しはOK。「正しいUML」より「伝わる図」を優先しましょう。学術論文を書いてるわけではないので。
03

クラス図とER図

CLASS DIAGRAM & ER

クラス図はオブジェクト指向の世界の地図、ER図はDBの世界の地図。似ているようで目的が違います。クラス図は振る舞い込み、ER図はデータ構造に集中。

class-vs-er 2種類の構造図
// クラス図の例
┌─────────────────┐ │ <<interface>> │ │ PaymentMethod │ ├─────────────────┤ │ +pay(amt): bool │ └─────────────────┘ △ │ implements ┌─────┴─────┐ │ │ ┌──────────┐ ┌──────────┐ │CreditCard│ │ PayPay │ ├──────────┤ ├──────────┤ │-cardNo │ │-userId │ ├──────────┤ ├──────────┤ │+pay() │ │+pay() │ └──────────┘ └──────────┘ [Order] 1 ◇── 1..*[OrderItem] コンポジション（強い所有）

関連の種類: 関連(─) / 集約(◇─) / コンポジション(◆─) / 継承(△─) / 実装(△- -)

// ER図の例 (Crow's Foot 記法) ┌─────────────┐ ┌──────────────┐ │ users │ │ orders │ ├─────────────┤ ├──────────────┤ │ PK id │──┐ │ PK id │ │ email │ │ │ FK user_id │ │ name │ └─>┤< │ status │ │ created │ │ total │ └─────────────┘ │ created │ └──────┬────────┘ │ │ has many ▼ ┌──────────────────┐ │ order_items │ ├──────────────────┤ │ PK,FK order_id │ │ PK,FK product_id │ │ quantity │ │ unit_price │ └──────────────────┘ 多重度の表現: 1 1個 ──< 多数 ─<| 0または1 ──<| 1以上 ──○< 0以上 (任意) // ツール: dbdiagram.io, drawSQL, Mermaid 等 // テキストで書けるものが現代的
裏技 — 「ドメインモデル図」という選択肢
厳密なクラス図は業務メンバーには伝わりません。技術記号を取り除いた「ドメインモデル図」（箱と矢印だけ）にすると、企画/営業/PMとの議論で大活躍します。場面で図のフォーマルさを切り替えるのがプロです。
04

シーケンス図と状態遷移図

SEQUENCE & STATE DIAGRAM

時間軸を表すならシーケンス図、状態の変化を表すなら状態遷移図。どちらも動的な振る舞いを伝えるのに無くてはならない図です。バグの議論やレビューで頻出。

sequence-state 動的な側面の表現
// シーケンス図の例
User Controller UseCase Repo DB │ │ │ │ │ ├─POST────>│ │ │ │ │ ├─execute>│ │ │ │ │ ├─find─>│ │ │ │ │ ├SQL─>│ │ │ │ │<────┤ │ │ │<──────┤ │ │ │ ├─save─>│ │ │ │ │ ├SQL─>│ │ │ │<──────┤ │ │ │<────────┤ │ │ │<─201─────┤ │ │ │ │ │ │ │ │ 時間は上から下へ 横矢印は呼び出し 点線は戻り値

使いどころ: API1回の呼び出しがどのコンポーネントを通るか / 非同期処理の流れ / トランザクション境界の説明

// 状態遷移図の例: 注文の状態 [*] 初期 │ ▼ ┌───────────┐ │ PENDING │ ← 注文作成直後 └───┬───────┘ │ 支払い完了 ▼ ┌───────────┐ │ PAID │ ← 入金確認 └───┬───────┘ │ 出荷処理 ▼ ┌───────────┐ │ SHIPPED │ ← 配送中 └───┬───────┘ │ 配達完了 ▼ ┌───────────┐ │ DELIVERED │ ← 完了 (終了状態) └────────────┘ キャンセル経路: PENDING → CANCELLED PAID → REFUNDED 記述ルール: - 状態（角丸の箱） - 遷移（矢印） - イベント/条件（矢印のラベル） - 開始状態 ● - 終了状態 ⊙ // 状態遷移図を書くと「想定外の状態遷移」 // に気付くことが多い。バグ防止に超有効。 // 例: PENDING から SHIPPED へ飛んでない? // CANCELLED 後の払戻しはどう扱う?
POINT — 状態の数を意識する
1つのエンティティに状態が5個以上ある場合、必ず状態遷移図を書きましょう。書かないと必ずバグが出ます。「PAIDからSHIPPEDに飛ばすコードが無い」「CANCELLED後にCONFIRMEDに戻れちゃう」など、図にすると一発で異常に気付きます。
05

C4モデル

C4 MODEL

C4モデルはSimon Brownが提唱したアーキテクチャ図の方法論。「ズームレベル」の概念で、同じシステムを4段階の解像度で描き分けます。「全体像が見えない」「詳細すぎて全体が分からない」問題を一発で解決します。

c4-model 4段階のズーム
// 4つのレベル
  • L1System Context
    システムと外部世界の関係
    経営層・全関係者向け
  • L2Container
    主要な実行単位
    Web/API/DB/外部サービス等
  • L3Component
    コンテナ内のモジュール構成
    開発者向け
  • L4Code
    クラスレベル
    必要な時だけ。多くは省略

多くの組織はL1〜L3で十分。L4は普段省略してOK。

// L1: システムコンテキスト ┌─────────────┐ │ 顧客 │ (Person) └──────┬──────┘ │ 注文する ▼ ┌─────────────────┐ │ ECシステム │ ← 私たちのシステム └──┬───────────┬──┘ │ │ ▼ ▼ ┌──────────┐ ┌──────────┐ │ Stripe │ │ 配送API │ └──────────┘ └──────────┘ // L2: コンテナ図 (ECシステムを開く) ┌─────────────────┐ │ Webアプリ │ (Next.js) └────┬─────────────┘ │ HTTPS ▼ ┌─────────────────┐ ┌────────┐ │ API Server │───>│ Redis │ │ (Node.js) │ │(キャッシュ)│ └────┬─────────────┘ └────────┘ │ ▼ ┌─────────────────┐ │ PostgreSQL │ └─────────────────┘ // L3: API Server を開く (Component) API Server 内部: ┌────────────────────────┐ │ Controller層 │ │ OrderController │ │ UserController │ │ ───────────────────────│ │ UseCase層 │ │ PlaceOrderUseCase │ │ ListProductsUseCase │ │ ───────────────────────│ │ Domain層 │ │ Order, Product, ... │ │ ───────────────────────│ │ Infrastructure層 │ │ *Repository, *Client │ └────────────────────────┘ // ズームインに従い詳細化。 // 読み手のレベルに合わせて見せる図を選ぶ。
裏技 — C4は新人オンボーディングで神
新メンバーが入った時、「L1で全体感」→「L2でシステム構成」→「L3で担当領域」と段階的に見せると、最短で立ち上がります。いきなり詳細図を渡すと迷子になるので、解像度の階段を意識しましょう。
06

ADR — 設計決定記録

ARCHITECTURE DECISION RECORDS

ADR (Architecture Decision Record)は「なぜそれを選んだか」を1ファイル1決定で残す手法。長大な設計書より圧倒的に実用的で、近年急速に普及しています。未来の自分やチームへの贈り物です。

adr 決定理由を後世に残す
// ADR の構成要素
  • タイトル連番+短い見出し
    例: 0007-use-postgresql.md
  • Status状態
    Proposed/Accepted/Deprecated/Superseded
  • Context背景・なぜ判断が必要になったか
  • Decision決定内容
  • Consequences結果と影響（良い面/悪い面）

運用方針: 一度書いたら絶対に書き換えない。変更があれば新ADRを書き、旧ADRはSupersededにする。「履歴の保持」が肝。

// ADR の例 (Markdown) # ADR-0007: 主データベースに PostgreSQL を採用 ## Status Accepted - 2026-04-12 ## Context 本プロジェクトでは ECサイトを構築する。 - 注文/在庫の強整合性が必要 - 月間 10万件の注文を想定 - レポート集計のため SQL クエリが必要 - 既存チームの経験は MySQL/PostgreSQL - AWS環境で運用 ## Decision 主データベースに PostgreSQL 16 を採用する。 AWS RDS で運用し、リードレプリカ 1台を設置。 ## Considered Alternatives - MySQL 8: 同等の選択肢。 → JSON型と全文検索でPostgresが優位 - DynamoDB: 強整合トランザクションが弱い → 注文業務には不向き - MongoDB: スキーマレスは魅力だが → レポート集計が複雑化 ## Consequences 良い面: - ACIDで在庫整合性が保証される - 既存スキルが活かせる - AWS RDSの自動運用が利用可能 悪い面: - 水平スケールには限界がある (将来 100万件超なら分割検討) - ライセンスは無料だが運用は有料 ## References - [PostgreSQL公式] - 関連ADR: ADR-0005 (キャッシュ層に Redis)
POINT — ADRはコードリポジトリに入れる
ADRはコードと一緒にGitで管理するのが定番。docs/adr/フォルダにMarkdownで置けば、PRレビューと同じ流れで議論できます。Wikiに置くと忘れられるのがADRの宿命。コードと共に進化させましょう。
07

Mermaidで書く

DIAGRAMS AS CODE — MERMAID

Mermaidはテキストで図を書ける記法。GitHub/GitLab/Notion/VSCodeなど主要ツールが標準対応しています。図がコードと一緒にバージョン管理できるのが最大の利点。図の差分レビューも可能になります。

mermaid テキストで図を書く
// Mermaidの主な図種
  • flowchart: フロー図・概念図
  • sequenceDiagram: シーケンス図
  • classDiagram: クラス図
  • stateDiagram-v2: 状態遷移図
  • erDiagram: ER図
  • gantt: ガントチャート
  • journey: ユーザージャーニー

メリット:
・テキストなのでGitで差分管理
・コードレビューと同じ流れ
・専用ツール不要
・Notion/GitHubで即時レンダリング

// Mermaid: フロー図 ```mermaid flowchart LR A[ユーザー] --> B{ログイン済み?} B -->|Yes| C[商品ページ] B -->|No| D[ログイン画面] D --> C C --> E[カートに追加] E --> F[決済] ``` // Mermaid: シーケンス図 ```mermaid sequenceDiagram participant U as User participant API participant DB U->>API: POST /orders API->>DB: INSERT DB-->>API: order_id API-->>U: 201 Created ``` // Mermaid: 状態遷移図 ```mermaid stateDiagram-v2 [*] --> Pending Pending --> Paid: 入金確認 Pending --> Cancelled: キャンセル Paid --> Shipped: 出荷 Shipped --> Delivered: 配達完了 Paid --> Refunded: 返金 Delivered --> [*] ``` // Mermaid: ER図 ```mermaid erDiagram USER ||--o{ ORDER : places ORDER ||--|{ ORDER_ITEM : contains PRODUCT ||--o{ ORDER_ITEM : "is in" ``` // READMEに貼るだけで図がレンダリング // 「設計図 = テキスト」の時代
裏技 — PlantUMLとの使い分け
Mermaidは軽量・即時レンダリングが強み、PlantUMLはUML厳密記法・複雑図対応が強み。シンプルな図はMermaid、本格UMLはPlantUMLと使い分けるのが現代の流儀。VSCodeなら両方プラグインで対応可能です。

設計レビューの進め方

DESIGN REVIEW PROCESS

設計レビューは「実装に入る前の最後の検問所」。ここで問題を見つけられれば修正コストは小さく、見逃すと数倍の手戻りになります。「合意形成の場」として正しく運用するのが肝心。

design-review 効果的なレビューの進め方
// レビューのチェック観点
  • ★★★ 要件を満たしているか
  • ★★★ 非機能要件は?
    性能/可用性/セキュリティ
  • ★★ 設計原則を守っているか
    SOLID / DRY / 凝集・結合
  • ★★ テスト可能か
  • ★★ 運用できるか
    監視/ログ/障害対応
  • ★ 他の選択肢は検討したか
  • ★ 過剰設計になってないか
// レビュー進行の理想形 [Step 1] 事前共有 (2営業日前) - 設計書/ADR を事前に配布 - レビュアーは事前に目を通す - 当日にいきなり読ませない [Step 2] 設計者がプレゼン (15分) - 解こうとしている問題 - 採用した設計の核心 - 検討した代替案と却下理由 [Step 3] 質疑応答 (30分) - 不明点の確認 - 観点別レビュー - 「なぜ」を5回くらい掘る [Step 4] 議論と決定 (15分) - 修正必要点を確定 - 次のアクション - 再レビュー有無 // レビューでやってはいけないこと [NG] 当日初見で意見を求める [NG] 設計者を吊し上げる [NG] 個人の好みで議論 → 「私ならこう書く」は禁止 [NG] 答えを言わずに「ダメ」だけ言う → 必ず代案か観点を示す [NG] 結論を出さずに散会 → 必ず「OK/要修正/再レビュー」決定 // 良いレビュアーは「敵」じゃない // 「同じゴールを目指す仲間」 // この姿勢がチーム文化を作る
