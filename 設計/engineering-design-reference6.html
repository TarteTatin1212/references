01

ER設計の基本

ER MODELING

ER（Entity-Relationship）設計は「現実世界のモノとモノの関係を表に落とす」作業です。エンティティ（実体）、属性、関連（リレーション）、多重度の4要素を押さえれば、ほぼあらゆる業務をモデル化できます。

er-basics エンティティと関連
// ER図の基本要素
  • エンティティ名詞 = 実体
    ユーザー / 注文 / 商品 / 配送
  • 属性性質
    名前 / メール / 価格 / 数量
  • 関連動詞 = 関係
    注文する / 所有する
  • 多重度個数
    1:1 / 1:N / N:N
[User] 1 ────── N [Order] ユーザーは注文を持つ [Order] N ────── N [Product] → 中間テーブル必要 [Order] 1 ── N [OrderItem] N ── 1 [Product]
// ECサイトの基本テーブル CREATE TABLE users ( id BIGSERIAL PRIMARY KEY, email VARCHAR(255) NOT NULL UNIQUE, name VARCHAR(100) NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW() ); CREATE TABLE products ( id BIGSERIAL PRIMARY KEY, name VARCHAR(200) NOT NULL, price INTEGER NOT NULL CHECK(price >= 0) ); CREATE TABLE orders ( id BIGSERIAL PRIMARY KEY, user_id BIGINT NOT NULL REFERENCES users(id), status VARCHAR(20) NOT NULL, total INTEGER NOT NULL, created_at TIMESTAMPTZ DEFAULT NOW() ); // N:N を解決する中間テーブル CREATE TABLE order_items ( order_id BIGINT REFERENCES orders(id), product_id BIGINT REFERENCES products(id), quantity INTEGER NOT NULL, unit_price INTEGER NOT NULL, PRIMARY KEY (order_id, product_id) );
POINT — N:Nには中間テーブル
多対多（N:N）は必ず中間テーブルで解決します。「注文と商品はN:N」ならOrderItemテーブルを作る。中間テーブルに『関連の属性』（数量・単価など）が乗ることが多いのもポイント。
02

正規化

NORMALIZATION

正規化は「データの重複と矛盾をなくす」ための定石。第1〜第3正規形まで覚えれば、業務システムの9割はカバーできます。正規化されていないテーブルは、データ更新のたびに不整合の温床になります。

normalization 第1正規形〜第3正規形
// 正規化の段階
  • 1NF1セルに1つの値
    カンマ区切りなどを排除
  • 2NF主キー全体に依存
    主キーの一部にだけ依存する列を別表に
  • 3NF主キー以外に依存しない
    「主キー → 他列 → さらに他列」を排除

覚え方:
「キーに、キーの全部に、キーだけに」
(One key, the whole key, and nothing but the key)

// 正規化前: 何もかも1枚のテーブル orders | id | user_name | user_email | products | |----|-----------|-----------|----------------| | 1 | 田中 | t@x.com | りんご,バナナ | | 2 | 田中 | t@x.com | みかん | 問題: - products列にカンマ区切り → 1NF違反 - user_nameが重複 → 更新時の不整合 - メアド変更時、複数行を直す必要 // 正規化後 users | id | name | email | |----|------|---------| | 1 | 田中 | t@x.com | orders | id | user_id | |----|---------| | 1 | 1 | | 2 | 1 | order_items | order_id | product_id | qty | |----------|------------|-----| | 1 | 101 | 1 | | 1 | 102 | 1 | | 2 | 103 | 1 | products | id | name | |-----|--------| | 101 | りんご | | 102 | バナナ | | 103 | みかん | // 重複ゼロ、更新は1箇所で済む
TIP — 第3正規形まででOK
第4・第5正規形まで習った人もいるかもしれませんが、業務システムでは第3正規形までで実用上ほぼ十分です。それ以上は学術的な話。「3NFを守って、必要なら戦略的に崩す」が現場の正解です。
03

非正規化の判断

DENORMALIZATION

正規化が常に正解とは限りません。パフォーマンスのために意図的にデータを冗長化するのが非正規化。やみくもにではなく、「測定して、必要な箇所だけ」適用するのが鉄則です。

denormalization パフォーマンスのための冗長化
// 非正規化の使い所
  • 集計値のキャッシュ
    注文の合計金額をorders.totalに保持
  • スナップショット
    注文時の商品名・価格を残す（価格改定対策）
  • JOIN削減
    頻繁に表示する項目を別テーブルで保持
  • 履歴の固定化
    「過去の事実」は変わらないので冗長保持OK

注意: 非正規化は更新時に複数箇所を同期する責任を負います。設計時に「誰がどう同期するか」を必ず決めること。

// 非正規化の代表例: 注文スナップショット [NG: 正規化に固執] CREATE TABLE order_items ( order_id BIGINT, product_id BIGINT, quantity INTEGER ); // 商品の現在価格でJOINして金額計算 SELECT SUM(p.price * oi.quantity) FROM order_items oi JOIN products p ON p.id = oi.product_id; // 問題: 価格改定すると過去の注文金額が変わる! [OK: 戦略的に非正規化] CREATE TABLE order_items ( order_id BIGINT, product_id BIGINT, quantity INTEGER, -- スナップショット (注文時点の事実) product_name VARCHAR(200) NOT NULL, unit_price INTEGER NOT NULL, subtotal INTEGER NOT NULL ); // メリット: // - 商品マスタを変更しても過去注文は不変 // - JOINなしで一覧表示が高速 // - 商品削除しても注文履歴が壊れない // 「事実は時点で確定する」 // 業務系のDB設計では超重要な視点
POINT — 計測してから非正規化
「速くなりそうだから」と推測で非正規化するのは禁物。先にEXPLAIN ANALYZEで遅いクエリを特定し、インデックスやクエリ改善で解決できないと確認してから非正規化に手を出すこと。順番を間違えるとデータの一貫性だけ犠牲になります。
04

主キーと自然キー

PRIMARY KEY DESIGN

主キーの選び方には流派があります。サロゲートキー（連番・UUID）と自然キー（業務的に意味のある値）。現代の主流はサロゲートキーですが、それぞれの長所短所を理解しておく必要があります。

primary-key 3つの選択肢
// 主キーの3タイプ
タイプ長所短所
連番 BIGINT軽い、速い、インデックス効率良推測可能、移行時の衝突
UUID衝突なし、分散生成可サイズ大、インデックス効率低
自然キー
(メアド等)		意味が明確変更時に大事故、長くて遅い

結論: 多くの場合は連番BIGINTがベター。分散環境や外部公開IDが必要ならUUIDv7。

// 自然キーの落とし穴 [NG] 自然キーを主キーに CREATE TABLE users ( email VARCHAR(255) PRIMARY KEY, name VARCHAR(100) ); CREATE TABLE orders ( id BIGSERIAL, user_email VARCHAR(255) REFERENCES users(email) ); // 問題: メアド変更要望が来た // → users 1行、orders 数千行、 // logs 数万行を全部更新 // → 関連テーブル増えるほど地獄 [OK] サロゲートキー + ユニーク制約 CREATE TABLE users ( id BIGSERIAL PRIMARY KEY, email VARCHAR(255) UNIQUE NOT NULL, name VARCHAR(100) ); // メアド変更: users 1行更新で完了 // 他テーブルは user_id 参照なので無傷 // 外部公開用には UUID も併用すると安心 CREATE TABLE users ( id BIGSERIAL PRIMARY KEY, // 内部用 public_id UUID UNIQUE NOT NULL // API用 DEFAULT gen_random_uuid(), email VARCHAR(255) UNIQUE ); // URLに連番IDを晒すと推測攻撃の餌食 // → public_id で隠すのが安全
裏技 — UUIDv7という選択肢
昔のUUIDv4はランダムすぎてインデックス効率が悪かったですが、UUIDv7（時刻ベース）なら連番に近いインデックス効率を保ちつつ、衝突なしのユニーク性が手に入ります。新規プロジェクトの公開IDには有力な選択肢です。
05

インデックス設計

INDEX DESIGN

インデックスは「検索を速くする辞書のしおり」です。ただし付ければ付けるほど書き込みが遅くなるのがクセモノ。必要な列だけに、効果を測りながら張るのがプロの仕事。

index-design どこに張るか / 張らないか
// インデックスを張るべき場所
  • ★★★ 外部キー列
    JOINの高速化に必須
  • ★★★ WHERE で頻出する列
    WHERE status = ... 等
  • ★★ ORDER BY の列
    ソート処理を省略
  • ★ UNIQUE制約の列
    自動でインデックス付与

避けるべき: 性別・フラグ等のカーディナリティが低い列。値が2〜3種類しかない列にインデックスを張っても、ほぼ無意味で書き込みコストだけ増えます。

// インデックスの基本 CREATE INDEX idx_orders_user_id ON orders(user_id); CREATE INDEX idx_orders_created_at ON orders(created_at DESC); // 複合インデックス: 順序が重要! CREATE INDEX idx_orders_status_created ON orders(status, created_at); // この場合効くクエリ: SELECT * FROM orders WHERE status = 'PAID'; // OK SELECT * FROM orders WHERE status = 'PAID' AND created_at > '2026-01-01'; // OK // 効かないクエリ: SELECT * FROM orders WHERE created_at > '2026-01-01'; // NG // → 先頭列 status が指定されてない // 効果測定の必須コマンド EXPLAIN ANALYZE SELECT * FROM orders WHERE ...; // Seq Scan → Index Scan に変われば成功
POINT — インデックスの代償
インデックスは「読み速く、書き遅く」の典型的トレードオフ。1テーブルにインデックス10個張れば、INSERTのたびに10個全部を更新する必要があります。書き込み主体のテーブルでは特に慎重に。
06

トランザクション境界

TRANSACTION BOUNDARIES

トランザクションの「どこからどこまでで囲むか」を決めるのは、見過ごされがちですが超重要。広すぎるとロック競合で詰まり、狭すぎるとデータ不整合が起きます。ユースケース1つ = 1トランザクションが基本ルールです。

transaction-boundary ACIDと境界の設計
// ACID 4原則
  • AAtomicity 全部成功 or 全部巻き戻し
  • CConsistency 制約を満たした状態を維持
  • IIsolation 並行実行でも独立
  • DDurability 確定後は消えない

分離レベル: READ COMMITTED / REPEATABLE READ / SERIALIZABLE。
RDBのデフォルトは多くの場合READ COMMITTED。在庫管理などはより厳しい分離が必要。

// トランザクション境界の例 [NG] 境界が広すぎる BEGIN; // ユーザー入力を待つ… // ← この間ロックが残る UPDATE users SET ...; COMMIT; // → 他ユーザーが待たされる [NG] 境界が狭すぎる BEGIN; UPDATE stock ...; COMMIT; BEGIN; INSERT INTO orders ...; COMMIT; // → 在庫減ったが注文記録なし、の事故 [OK] ユースケース単位で囲む BEGIN; SELECT stock FROM products WHERE id = 1 FOR UPDATE; // 行ロック UPDATE products SET stock = stock - 1 WHERE id = 1; INSERT INTO orders (...) VALUES (...); INSERT INTO order_items (...) VALUES (...); COMMIT; // 全部成功 or 全部失敗 // 並行注文での在庫過剰販売も防げる // 重要: トランザクション内で // 外部API呼び出ししないこと! // → タイムアウトでDBが詰まる
裏技 — 楽観ロックと悲観ロック
悲観ロック（SELECT ... FOR UPDATE）は確実だが詰まりやすい。楽観ロック（バージョン列で衝突検出）は競合が少ない場面で高速。在庫など競合が頻発する箇所は悲観、編集画面など競合が稀な箇所は楽観、と使い分けます。
07

論理削除と監査

SOFT DELETE & AUDIT

業務システムでは「データを物理削除しない」のがほぼ鉄則。削除フラグだけ立てる「論理削除」と、変更履歴を残す「監査ログ」を組み合わせるのが定番です。「過去の事実は消さない」のが基本姿勢。

soft-delete-audit 削除と変更履歴の設計
// 論理削除のパターン
方式特徴
deleted_at列NULL=有効, 時刻=削除済み。一般的
status列ACTIVE/DELETED等。複数状態の表現に
履歴テーブル削除時に履歴表に移送。クエリは速いが手間

注意点: 論理削除を使うと全クエリにWHERE deleted_at IS NULLが必要。漏れるとバグの温床になります。ORM側でデフォルトフィルタ設定がほぼ必須。

// 論理削除の典型実装 CREATE TABLE users ( id BIGSERIAL PRIMARY KEY, name VARCHAR(100), email VARCHAR(255), created_at TIMESTAMPTZ DEFAULT NOW(), updated_at TIMESTAMPTZ DEFAULT NOW(), deleted_at TIMESTAMPTZ // NULLなら有効 ); // 削除: 物理削除しない UPDATE users SET deleted_at = NOW() WHERE id = 123; // 取得: 削除済みを除外 SELECT * FROM users WHERE deleted_at IS NULL; // 監査ログテーブル CREATE TABLE audit_logs ( id BIGSERIAL PRIMARY KEY, table_name VARCHAR(50), record_id BIGINT, action VARCHAR(10), // CREATE/UPDATE/DELETE changed_by BIGINT, // 操作したユーザー old_value JSONB, // 変更前 new_value JSONB, // 変更後 created_at TIMESTAMPTZ DEFAULT NOW() ); // アプリ層 or DBトリガで全変更を記録 // → 「いつ誰が何を変えたか」を全て追跡 // 業務系ではほぼ必須 // (金融/医療系では法的にも必須)
POINT — 個人情報の扱いだけ別
GDPR・改正個人情報保護法では「忘れられる権利」があり、削除要求があれば物理的にデータを消す必要があります。論理削除と「マスキング（個人情報部分だけ匿名化）」を組み合わせるのが現代的なやり方。「全部残す」がデフォルト、個人情報だけ例外と覚えましょう。

マイグレーション戦略

DB MIGRATION STRATEGY

DBスキーマは必ず変わります。「変えない」のではなく「安全に変える仕組み」を最初から組むのが現代的アプローチ。マイグレーションツール（Flyway、Liquibase、Prisma、TypeORM等）を使い、全変更をコードで管理しましょう。

migration-strategy 無停止デプロイの基本
// マイグレーションの鉄則
  • ★★★ 変更は必ず追加から
    列削除や名前変更は2段階で
  • ★★★ 後方互換を保つ
    旧アプリでも動く形に
  • ★★ 巨大テーブルは分割実行
    1000万行UPDATEは即障害
  • ★★ ロールバック計画
    戻せないマイグレは禁止
// 列の名前変更を安全にやる手順 [Step 1] 新列を追加 ALTER TABLE users ADD COLUMN full_name VARCHAR(200); [Step 2] アプリ更新 // 書込: 旧列・新列の両方に書く // 読込: 新列を優先、無ければ旧列 [Step 3] 既存データをコピー UPDATE users SET full_name = name WHERE full_name IS NULL; // (巨大テーブルなら分割実行) [Step 4] アプリ更新 // 書込: 新列のみ // 読込: 新列のみ [Step 5] 旧列を削除（時間を置いてから） ALTER TABLE users DROP COLUMN name; // この5段階のおかげで: // - 旧バージョンのアプリも動き続ける // - 無停止デプロイが可能 // - 問題発覚時に各段階で戻せる // 1発で名前変更 = 即障害の道 // ALTER TABLE users RENAME ... // (絶対やめましょう)
