01

アーキテクチャとは何か

WHAT IS ARCHITECTURE

アーキテクチャは「変更コストの大きい決定」の集合です。フレームワーク選定、レイヤ構成、データの流れ、デプロイ単位。これらは後から変えるのに膨大なコストがかかります。逆に言えば「最初に決めておくべき大きな線引き」です。

what-is-architecture 設計の3階層
// 抽象度と変更コスト
[マクロ] アーキテクチャ • レイヤ分割 • サービス境界 • 通信方式 • DB分割 → 変更コスト: 極大 [ミドル] モジュール設計 • クラス分割 • インターフェース • パッケージ構成 → 変更コスト: 中 [ミクロ] 実装 • 関数の中身 • 変数名 • アルゴリズム → 変更コスト: 小

上に行くほど決定が重い。だからこそ最初に時間をかけて考える価値があります。

// アーキテクチャ判断の代表例 [1] モノリス vs マイクロサービス? → 後で分割は可能だが大変 → 最初に統合は不可能に近い [2] RDB vs NoSQL? → 移行コストは数ヶ月単位 → トランザクション特性で決まる [3] 同期 vs 非同期通信? → 全体の設計思想が変わる → 後で全部書き換えになる [4] サーバーサイドレンダ vs SPA? → ユーザー体験/SEOに直結 → フロント全部作り直し [5] 認証方式? → セッション/JWT/OAuth → 後から差し替え地獄 // アーキテクトの仕事 = この大判断 // 「迷ったら専門家に相談」と // 「迷う前にトレードオフを書き出す」 // が現場の鉄則。
POINT — アーキテクチャの本質
「Architecture is about the important stuff. Whatever that is.」（Ralph Johnson）。重要なことはプロジェクトによって違う。だから万能のアーキテクチャは存在せず、「このシステムにとって重要なこと」を見極めるのがアーキテクトの仕事になります。
02

レイヤードアーキテクチャ

LAYERED ARCHITECTURE

Layered最も伝統的かつ最も広く使われるアーキテクチャ。上から「プレゼンテーション → アプリケーション → ドメイン → インフラ」と層を分け、依存は上から下にだけ流します。多くの業務アプリの土台がこれ。

layered-architecture 4層モデルの定番
// 4つの層
┌────────────────────┐ │ プレゼンテーション │ ← UI/API ├────────────────────┤ │ アプリケーション │ ← ユースケース ├────────────────────┤ │ ドメイン │ ← ビジネスルール ├────────────────────┤ │ インフラ │ ← DB/外部API └────────────────────┘ 依存は上→下のみ。 下が上を参照してはいけない。

原則: 各層は1つ下の層しか呼ばない。プレゼンが直接DBを触ると即破綻します。

// 典型的なフォルダ構成 src/ ├─ presentation/ // UI/API │ ├─ controllers/ │ │ └─ OrderController.ts │ └─ views/ │ ├─ application/ // ユースケース │ └─ usecases/ │ └─ PlaceOrderUseCase.ts │ ├─ domain/ // ビジネスルール │ ├─ entities/ │ │ └─ Order.ts │ └─ services/ │ └─ DiscountService.ts │ └─ infrastructure/ // DB/外部API ├─ repositories/ │ └─ OrderRepository.ts └─ clients/ └─ StripeClient.ts // 呼び出しの流れ OrderController // 受付 ↓ PlaceOrderUseCase // 流れ制御 ↓ Order + DiscountService // 計算 ↓ OrderRepository // 保存 ↓ DB
TIP — 詰まりポイント
レイヤードは「下層が上層を知らない」のが鉄則。Repositoryクラスの中でController用のレスポンス型を扱い始めたら黄信号。これを許すと依存が逆流して、テストが書けないコードに堕ちます。
03

MVC / MVP / MVVM

MVC / MVP / MVVM

UI ArchitectureUIまわりで定番の3つの構成。「画面・ロジック・データの分離」という目的は共通ですが、責務の分け方が違います。フレームワークによってどれを採用しているかが変わります。

mvc-mvp-mvvm UI層の3兄弟
// 3つの違い
名称特徴採用例
MVCControllerが司令塔。Viewはモデルを直接読むRails
Spring MVC
MVPPresenterが全て仲介。Viewは受動的Android
(旧)WinForms
MVVMViewModelとViewが双方向バインドVue
WPF

共通の目的: View（見た目）とビジネスロジックを混ぜないこと。これを守らないと画面ごとにコピペが量産されます。

// MVC の例（Spring風） [Model] データとビジネスルール class Order { total: number; calculateTotal() { ... } } [View] 描画担当 // order.html (Thymeleaf等) <div>合計: {{ order.total }}</div> [Controller] 司令塔 class OrderController { show(id) { const order = repo.find(id); return view("order.html", { order }); } } // MVVM の例（Vue風） [View] <input v-model="name"> <p>こんにちは {{ name }}</p> [ViewModel] data() { return { name: "" }; } // 入力 → 自動的に画面更新 // 双方向バインディングが特徴
POINT — フレームワークが決める
MVC/MVP/MVVMは自分で選ぶというより、フレームワークで決まることが多いです。Reactは強いて言えばMVVM寄り（Hooks時代はもっと自由）、Angularは構造としてMVVM、Spring MVCは名前の通りMVC。フレームワークの流儀に逆らわないのが鉄則です。
04

ヘキサゴナルアーキテクチャ

HEXAGONAL / PORTS AND ADAPTERS

Hexagonal別名「ポートとアダプタ」。アプリケーションの中心（コア）を外部世界（DB / UI / 外部API）から切り離す思想。コアは「ポート」（インターフェース）だけを知り、具体実装はアダプタとして外側に置きます。

hexagonal-architecture 中心を外界から守る
// 中心と外側
┌──────────┐ UI →│ Port IN │ │ │ │ Core │← Application │ Domain │ │ │ │ Port OUT │→ DB / API └──────────┘ Coreは外側のことを知らない。 外側は「Port」だけを通じて 中と接続する。

テストで超強力: Coreは外部依存ゼロなので、純粋な単体テストが書ける。DB/UI/APIをモックすら必要としない形が理想。

// ヘキサゴナルの実装イメージ // === Core （中心） === // ポート: 外部の窓口を抽象として定義 interface OrderRepository { // OUT save(order: Order): void; } interface Notifier { // OUT notify(msg: string): void; } // コアのビジネスロジック class PlaceOrderUseCase { // IN constructor( private repo: OrderRepository, private notifier: Notifier, ) {} execute(order: Order) { this.repo.save(order); this.notifier.notify("確定"); } } // === Adapter （外側） === // 外側で具体実装を提供 class PostgresOrderRepo implements OrderRepository { ... } class SlackNotifier implements Notifier { ... } // テスト時 new PlaceOrderUseCase( new InMemoryOrderRepo(), // 差し替え new FakeNotifier(), ); // → 本物のDB/Slack不要でテスト可能
裏技 — DIPの最大の応用例
ヘキサゴナルはSOLIDのDIP（依存性逆転）をシステム全体に適用したものと捉えると理解が早いです。中心が外側に依存しない=外側が抽象に依存する。Vol.3で学んだDIPが、ここで本領発揮します。
05

クリーンアーキテクチャ

CLEAN ARCHITECTURE

Cleanボブおじさん（Robert C. Martin）が提唱した同心円型のアーキテクチャ。ヘキサゴナルをさらに整理した形。依存性の流れは必ず外から内へという強い規律で、変更耐性とテスト容易性が極めて高くなります。

clean-architecture 同心円の4層
// 内側ほど安定・抽象
┌──────────────────────────┐ │ Frameworks & Drivers │ 外 │ (Web / DB / Devices) │ │ ┌──────────────────────┐ │ │ │ Interface Adapters │ │ │ │ (Controllers/Gateways)│ │ │ │ ┌──────────────────┐ │ │ │ │ │ Use Cases │ │ │ │ │ │ (アプリケーション) │ │ │ │ │ │ ┌──────────────┐ │ │ │ │ │ │ │ Entities │ │ │ │ │ │ │ │ (ビジネス) 内 │ │ │ │ │ │ │ └──────────────┘ │ │ │ │ │ └──────────────────┘ │ │ │ └──────────────────────┘ │ └──────────────────────────┘ 依存の方向: 外 → 内 のみ

The Dependency Rule: 依存は必ず内向き。内側は外側のことを知らない。これだけ守れば、フレームワークもDBもUIも交換可能になります。

// 各層の役割 [Entities] 企業横断のビジネスルール class Order { calculateTotal() { ... } applyDiscount() { ... } } // → 最も安定。10年変わらない。 [Use Cases] アプリケーション固有の流れ class PlaceOrderUseCase { execute(...) { ... } } // → このアプリ特有の業務手順 [Interface Adapters] 変換層 class OrderController { ... } class OrderGateway { ... } // → HTTP/DBの言葉と内部の言葉を変換 [Frameworks & Drivers] 最外側 Express / TypeORM / React // → ツール群。差し替え可能。 // 例: 「DB を PostgreSQL から // DynamoDB に変える」 // → Adaptersと最外層だけ修正 // → Entities/UseCasesは無傷 // 例: 「Web API を gRPC に変える」 // → 同じく最外層のみ
POINT — 万能ではない
クリーンアーキテクチャは規模の大きい長期保守システムで真価を発揮します。小規模CRUDアプリには過剰設計になりがち。「3画面の管理ツール」までならレイヤードで十分です。システム規模に対する仕掛けの量を見極めるのがプロの判断です。
06

モノリスとマイクロサービス

MONOLITH VS MICROSERVICES

「全部1つのアプリ」か「機能ごとにサービス分割」か。2010年代後半にマイクロサービス熱が高まりましたが、近年は「最初はモノリスから」に揺り戻しています。サイズと組織で選び分けるのが正解。

monolith-vs-microservices 2つの極とトレードオフ
// 比較表
モノリスマイクロサービス
開発速度速い初期は遅い
デプロイ1つサービス毎
スケール全体個別に可
技術選定統一個別可
運用シンプル複雑
分散トランザクション不要苦行

"Monolith First"（Martin Fowler）: まずモノリスで作り、必要になったら分割するのが安全。

// マイクロサービスが向く条件 [Yes な条件] ✓ チームが 50人 超 ✓ 機能ごとに スケール特性 が違う （検索は重い、注文は軽い、等） ✓ サービス毎に 技術スタック を分けたい ✓ 独立したデプロイ頻度が必要 ✓ 障害分離が要件 [No な条件] ✗ チームが 10人 未満 ✗ ドメインがまだ固まってない （境界の引き方を間違える） ✗ 分散トランザクションが頻発 ✗ DevOps体制が未整備 // マイクロサービスの隠れコスト - ネットワーク呼び出しの不安定さ - 分散デバッグの困難 - データ整合性（結果整合性が前提） - 監視/トレーシングの基盤構築 - サービス間の契約管理 // 結論 // 「マイクロサービスは贅沢品」 // 必要になってから移行で遅くない。
TIP — モジュラーモノリス
最近の流行は「モジュラーモノリス」。1つのアプリだが、内部は厳密にモジュール分割されている形。モノリスの運用簡単さ × マイクロサービスの境界明確さのいいとこ取り。後から分割もしやすいので、現実解として有力です。
07

イベント駆動とCQRS

EVENT-DRIVEN & CQRS

EDA「ある出来事が起きた」をきっかけに処理が連鎖していくアーキテクチャ。リアルタイム性、疎結合、スケーラビリティで威力を発揮。CQRSは「読み」と「書き」を別モデルに分けるパターンで、組み合わせて使われることが多いです。

event-driven-cqrs イベントを軸にした設計
// イベント駆動の流れ
[Service A] 発火 →「OrderPlaced」 ↓ [Event Bus] (Kafka/SQS等) ↓ ┌────┬────┬────┐ ↓ ↓ ↓ ↓ [在庫] [課金] [通知] [分析] 各サービスが並列で反応

CQRS（Command Query Responsibility Segregation）:
書き込み（Command）と読み取り（Query）を別モデルに分離。書きは正規化、読みは非正規化、と最適化を分けられる。

// イベント駆動の例 // Command側: 注文を確定 class PlaceOrderCommand { execute(order: Order) { db.save(order); eventBus.publish({ type: "OrderPlaced", orderId: order.id, items: order.items, total: order.total, }); } } // 購読側: 各サービスが反応 eventBus.subscribe("OrderPlaced", (e) => { inventory.reserve(e.items); }); eventBus.subscribe("OrderPlaced", (e) => { payment.charge(e.orderId, e.total); }); eventBus.subscribe("OrderPlaced", (e) => { mailer.sendConfirmation(e.orderId); }); // CQRS の例 [書き込みモデル] 正規化された業務DB orders, order_items, customers → 一貫性とトランザクション重視 [読み取りモデル] 非正規化された参照用 order_summary_view → JOINなしで一発取得 → ユーザー体験のため高速
裏技 — Event Sourcing
さらに進めるとEvent Sourcingに到達します。「現在の状態」をDBに保存する代わりに「イベントの履歴」だけを保存し、状態は履歴から再構成。監査・取消し・時間旅行が自由自在になる強力な手法ですが、複雑度も跳ね上がるので採用は慎重に。
08

アーキテクチャの選び方

HOW TO CHOOSE

アーキテクチャ選びの銀の弾丸はありません。プロジェクトの規模、チームの習熟度、変更頻度、寿命によって最適解は変わります。「過剰設計を避け、必要十分を選ぶ」のが現場の知恵です。

choosing-architecture 規模と特性で判断
// 規模別の目安
規模/特性推奨
個人/3画面以下MVC + シンプル分割
小〜中規模Webレイヤード
長期業務アプリレイヤード + DDD
テスト最重視ヘキサゴナル
巨大長寿命システムクリーンアーキ
組織分割大規模マイクロサービス
リアルタイム/分析イベント駆動
// 判断フローの例 Q1. 寿命は? <1年 → MVC でOK 1-3年 → レイヤード >3年 → クリーン検討 Q2. チーム規模は? 1-5人 → モノリス 5-20人 → モジュラーモノリス >20人 → マイクロサービス検討 Q3. ドメインは複雑か? Yes → DDD + クリーン/ヘキサ No → レイヤードで十分 Q4. リアルタイム要件は? Yes → イベント駆動 + CQRS No → 同期処理 で問題なし // 一番ダメなアーキテクト // 「クリーンアーキ最強だから常に使う」 // ← 過剰設計の典型 // 良いアーキテクト // 「このプロジェクトには // これが必要十分」と言える人
