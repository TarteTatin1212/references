Q.01
過剰設計になっていないか不安です。どこで線引きすればいいですか?
OVER-ENGINEERING
A. ANSWER — 「3回ルール」で判断します

設計パターンや抽象化を入れるかどうかの定番ルールが「3回ルール」です。同じパターンが3回現れた時に初めて抽象化を検討します。1回目は素直に書き、2回目は気になりつつもまだ書き、3回目で「これは型がある」と判断します。

過剰設計の典型サイン:「将来必要かも」「拡張性のために」。これらの言葉が出てきたら一旦立ち止まりましょう。YAGNI原則（You Aren't Gonna Need It）の出番です。

▲ NG — 過剰設計の例
まだ1種類しかないのに「将来増えるかも」とStrategyパターンを導入。インターフェース・抽象クラス・5つの空実装まで作って、結局1つしか使わない。

「DBを後から変えられるように」とリポジトリパターン+DIコンテナを完備。実際はSQLite以外使わない。
○ OK — 必要十分の例
まず直接書く。コードが汚くても動くものを作る。3回目の類似コードが出てきた時、リファクタリングで抽象化する。「必要になってから入れる」のが鉄則。

変更が必要になってから抽象を導入しても、テストがあれば安全に移行できる。
TIP — 過剰設計の代償
過剰設計は「動いてるけど誰も理解できないコード」を生みます。後任者は理解に時間がかかり、結局書き直されます。「シンプル>賢い」を常に意識しましょう。
Q.02
クラスやメソッドの責務分割で毎回迷います。コツはありますか?
RESPONSIBILITY SPLIT
A. ANSWER — 「と」「または」で分けます

クラスやメソッドの説明文に「〜と〜」が入ったら、その時点でSRP違反のサインです。「ユーザー登録とメール送信と課金処理を行うクラス」は3つに分けるべき。

判断基準は「変更理由が同じか」。「メール仕様の変更で触る場所」と「課金仕様の変更で触る場所」は別であるべき。同じクラスにあるなら分けるサインです。

  1. クラスを1文で説明してみる。「と」が入るなら分割候補。
  2. そのクラスを変更させる人を考える。経理チーム/マーケチーム/運用チームなど、複数の役割が出てきたら分ける。
  3. importしている外部ライブラリを数える。5つ以上なら怪しい（SMTP、Stripe、PDF、AWS S3...）。
  4. テストを書こうとしてみる。モックが10個必要なら確実に分割対象。
// 責務が混ざった典型 class UserManager { register() { ... } // 認証 sendWelcomeMail() { ... } // 通知 chargeFee() { ... } // 課金 generateReport() { ... } // 集計 } // 分割後 class UserRegistration { register() { ... } } class WelcomeMailer { send() { ... } } class PaymentProcessor { charge() { ... } } class UserReportService{ generate() { ... } }
POINT — Manager / Util / Helper は要注意
クラス名に Manager / Util / Helper / Service が付いていたら、責務が曖昧になっている可能性大。「何をするか」を具体的な名前で表現できないクラスは、責務が定まっていない証拠です。
Q.03
継承と委譲、どっちを使えばいいかいつも迷います。
INHERITANCE OR COMPOSITION
A. ANSWER — 迷ったら委譲を選びます

古典的な格言「継承より委譲（コンポジション）」が今も生きています。継承は強力ですが、親クラスの変更が子クラス全てに波及するという見えない結合を生みます。委譲なら必要な機能だけ持たせられて柔軟。

継承を使ってよいのは「明確な is-a 関係 + LSP（リスコフ置換）を満たす」場合のみ。「Penguin extends Bird」のような直感的でも振る舞いが置換不可能な関係は、継承ではなく構成で表現します。

▲ NG — 継承の濫用
UserController extends BaseController extends AbstractController

3階層も継承して、ベースクラスの仕様変更で全Controllerが壊れる。継承の連鎖は最大2階層までに。
○ OK — 委譲で機能を持つ
UserController は AuthService と Logger をコンストラクタで受け取って使う。

必要な機能だけ、必要なだけ持つ。組み合わせ自由、テスト時の差し替えも簡単。
// 継承パターン (硬い) class BaseController { protected checkAuth() { ... } protected log(msg) { ... } } class UserController extends BaseController { ... } // → BaseController変更が全子に波及 // 委譲パターン (柔軟) class UserController { constructor( private auth: AuthService, private logger: Logger, ) {} handle(req) { this.auth.check(req); this.logger.log("..."); } }
Q.04
リファクタリングするべきかどうか判断できません。
REFACTORING DECISION
A. ANSWER — 「ボーイスカウト・ルール」が定番

「来た時より美しくして去れ」。ボーイスカウトの教えがそのままプログラミングの格言になっています。機能追加・バグ修正で触ったついでに、その周辺だけ少し綺麗にするのが現代的アプローチ。「リファクタリング専用週間」より日常的な小さな改善のほうが効きます。

大規模リファクタリングが必要なのは、「新機能追加に毎回4倍の時間がかかる」「同じバグが繰り返し出る」「新人が1ヶ月経っても全体を把握できない」などの症状が出てから。それまでは小刻みな改善で十分です。

  1. テストを先に書く。リファクタリング前に「動作が変わらない」を保証する網を張る。
  2. 小さなステップで進める。1コミット = 1リファクタリング。大きな書き換えは避ける。
  3. 動作確認をこまめに。各ステップでテストを走らせて緑になることを確認。
  4. 機能追加と混ぜない。リファクタコミットと機能追加コミットは絶対に分ける。
裏技 — 「変更しにくい」が信号
バグ修正や機能追加で「触りたくないコード」に出会ったら、それがリファクタリング候補。恐怖が判断基準。「ここ触ると壊れそう」と感じる箇所こそ、テストを書いてリファクタすべき箇所です。
Q.05
レガシーコードに介入するのが怖いです。どう手をつけたら?
LEGACY CODE INTERVENTION
A. ANSWER — 「Strangler Fig パターン」が王道

レガシーシステムを一気に書き換えるのは失敗の典型例。「Big Bang」リライトはまず破綻します。代わりに使うのがStrangler Fig パターン（イチジクの絞め殺し戦略）。新機能を新システムで作り、既存機能を1つずつ移行する。古いシステムを徐々に「絞め殺して」いきます。

具体的には、新旧の前にプロキシ層を置いて、どちらに振り分けるかを切り替えていきます。1つの機能が移行できたらルーティングを切り替え、旧コードは削除。これを繰り返します。

// Strangler Fig の流れ [フェーズ0] 旧システムのみ Client → LegacySystem [フェーズ1] プロキシ層を追加 Client → Proxy → LegacySystem [フェーズ2] 新機能は新システムへ Client → Proxy ┬→ LegacySystem (既存) └→ NewSystem (新規) [フェーズ3] 既存機能を1つずつ移行 /users → NewSystem ✓ 移行済み /orders → NewSystem ✓ 移行済み /reports → LegacySystem ← 残り [フェーズ最終] 旧システム停止 Client → Proxy → NewSystem (LegacySystemをshutdown) // メリット: // - 常に動いている (リスク分散) // - 小さく刻めるので問題発見しやすい // - 進捗が見える // - 失敗しても戻れる
POINT — テストの網を張ってから
レガシーに手を入れる前に「動作の網」を張ります。E2Eテスト、API契約テスト、ゴールデンマスターテスト（出力比較）など、「変えても壊れていないこと」を機械的に確認できる仕組みが必須。テストなしの介入は自殺行為です。
Q.06
クラス名・関数名・変数名が思いつきません。命名のコツは?
NAMING THINGS
A. ANSWER — 「業務の言葉」をそのまま使います

命名で困ったら、まず業務担当者が使っている言葉を確認します。顧客が「注文確定」と呼ぶものを「OrderFinalize」と呼ぶか「OrderConfirm」と呼ぶか迷ったら、業務側に聞きます。これがユビキタス言語の発想です。

名前付けで最もダメなのは「Manager」「Helper」「Util」「Process」「Data」「Info」などの「中身を説明できない曖昧語」。「何をするものか」が名前から読み取れないのは責務未定義の証拠です。

▲ NG — 曖昧な命名
UserUtil, DataHelper, processData(), handleInfo()

何をするか名前から読めない。命名が曖昧 = 責務も曖昧。
○ OK — 意図が明確な命名
PasswordHasher, OrderConfirmationService, calculateTax(amount)

名前を読むだけで何をするか分かる。コメントも要らないレベル。
裏技 — 「英訳しないで日本語で考える」
最初から英語で命名しようとすると詰まります。まず日本語で「○○を△△する」と書き出し、それを直訳。「顧客の与信を判定する」→ judgeCustomerCredit。業務語が先、英語名は後。
Q.07
例外をどこで捕まえて、どう投げるべきか分かりません。
EXCEPTION DESIGN
A. ANSWER — 「例外は境界で捕まえる」

例外処理の基本ルール:「奥で投げて、外で捕まえる」。ドメイン層で例外を投げ、プレゼンテーション層（Controller / グローバルエラーハンドラ）でまとめてキャッチするのが定石。途中の層で try-catch を散らかすと、見通しが悪くなりエラー情報も失われます。

もう一つの鉄則:「予想できるエラーは例外にしない」。バリデーション失敗のような「業務上ありえる結果」はResult型や戻り値で表現するのが現代的。例外は「ありえない事態」のために予約しておきます。

// 例外設計のレイヤ別役割 [Domain層] ビジネスルール違反を例外で投げる throw new InsufficientStockError(...); [Application層] try-catch しない (基本) ドメイン例外をそのまま伝播させる [Presentation層] グローバルエラーハンドラでキャッチ → ステータスコードとレスポンス変換 if (err instanceof InsufficientStockError) return res.status(409).json({...}); if (err instanceof ValidationError) return res.status(422).json({...}); // 予測可能なエラーは Result 型 type Result<T, E> = | { ok: true; value: T } | { ok: false; error: E }; findUser(id): Result<User, NotFound> { // 見つからないは「例外」ではなく // 「ありえる結果」 }
Q.08
DRYを意識しすぎて、強引な共通化をしてしまいます。
PREMATURE ABSTRACTION
A. ANSWER — 「形が似てるだけ」は共通化しません

DRY原則は強力ですが、「ちょっと似てる」だけで共通化すると地獄を見ます。判断基準は「変更理由が同じか」。コードは似ているが、片方は「税法改正で変わる」、もう片方は「商品マスタの仕様で変わる」なら、共通化してはいけません。

「偶然の重複」と「本質的な重複」の見極めが肝。前者を共通化すると、片方の都合に合わせて変更したら、もう片方が壊れる。「2つのコードが同じ理由で変わる時、初めて共通化する」のが鉄則です。

▲ NG — 形が似てるだけで共通化
購入金額の計算と給与計算が、たまたま「数字×数字+調整」の形が似てるからと共通化。

後で消費税法が変わって購入計算だけ修正したいのに、給与にも影響が出てバグ発生。
○ OK — 文脈別に独立
購入金額計算と給与計算は業務文脈が違うので別関数。一見コードが重複していても、それぞれ独立して進化できる。

"ちょっとした重複は、間違った抽象化よりずっと安い" — Sandi Metz
POINT — 「Rule of Three」も併用
Q.01の「3回ルール」がここでも有効。1回目: 単に書く / 2回目: 重複を許容 / 3回目: 初めて共通化を検討。共通化は「変更理由が同じ」と確信できてから。早すぎる抽象化は、遅すぎる抽象化より高くつきます。
Q.09
チーム内で設計の意見が割れて、決まりません。
CONSENSUS BUILDING
A. ANSWER — 「トレードオフ表」で議論します

設計の議論が空転する最大の原因は「軸を共有していないこと」。Aさんは「性能優先」、Bさんは「保守性優先」、Cさんは「開発速度優先」とそれぞれ別の評価軸で話していると、永遠に合意できません。

解決策は「選択肢×評価軸のトレードオフ表」を作ること。性能/保守性/コスト/学習コスト/拡張性などの軸を立て、各選択肢を点数化します。これで議論が「好み」から「事実」に変わります。

// トレードオフ表の例: キャッシュ層の選定 | | Redis | Memcached | アプリ内 | |-------------|--------|-----------|----------| | 性能 | ◎ | ◎ | ◎ | | 永続性 | ○ | × | × | | 機能数 | ◎ | △ | △ | | 運用コスト | △ | △ | ◎ | | 学習コスト | △ | ○ | ◎ | | 分散対応 | ○ | ○ | × | | 既存スキル | ○ | △ | ◎ | |-------------|--------|-----------|----------| | 合計 | 16 pt | 12 pt | 14 pt | // → 「なんとなく」ではなく // 軸ごとの判断で議論できる // 決まらない時の最終手段: [1] 一番経験ある人が決める (Tech Lead) [2] 戻せる選択肢を選ぶ [3] ADRで「なぜ選んだか」を残す // 「Disagree and commit」 // 反対意見でも一旦合意したら全力で実行 // が大人のプロのやり方
裏技 — タイムボックスを区切る
議論が長引く時は「30分で結論を出す」「決まらなければデフォルト案で進む」と最初に宣言。永遠に議論しても答えは出ません。進めながら調整するほうが、止まって考えるより速いのがソフトウェア開発の鉄則です。
Q.10
設計力を伸ばすにはどう学べばいいですか?
HOW TO GROW
A. ANSWER — 「書く・読む・直す」の3つ

設計力は知識を増やすだけでは伸びません。実装の中で判断を繰り返した経験からしか身につきません。「たくさん書く」「他人のコードを読む」「自分のコードを直す」の3軸を継続することが最短ルートです。

本も大事ですが、読んだ知識は実コードで試さない限り定着しません。「SOLIDを学んだ」より「先週の自分のコードをSOLIDで書き直してみた」のほうが10倍学べます。手と頭の両方を動かすのが鉄則です。

  1. 個人プロジェクトで自由に書く。仕事のコードでは試せないアーキテクチャを試す場所を持つ。
  2. OSSのコードを読む。有名プロダクトのリポジトリは設計の宝庫。最初は分からなくてもOK。
  3. 過去の自分のコードをリファクタする。半年前のコードを読むと粗が見える。それが成長。
  4. レビューで「なぜ?」を聞く。先輩の指摘から学ぶより、自分が「なぜそう書いた?」と問われて答える経験が一番伸びる。
  5. 定番書を3冊読む。『リーダブルコード』『リファクタリング』『達人プログラマー』。読んで「実装」して、また読み返す。
POINT — 設計に「正解」はない
どんなベテランも「これが完璧な設計だ」とは断言できません。文脈、制約、チーム、寿命で最適解は変わります。だから「これが今のベスト」を見つけ続けるしかない。それが設計という営みの面白さでもあります。

△ SERIES — COMPLETE △

ここまで読み進めていただき、ありがとうございました。

本シリーズは「設計とは何か」から始まり、要件、原則、パターン、アーキテクチャ、データ、API、ドキュメント、そして困った時の判断基準まで、設計という営みを9つの巻で旅してきました。

大事なことを最後にもう一度。設計に唯一絶対の正解はありません。プロジェクトの文脈、チームの習熟度、ビジネスの寿命によって、最適解は変わり続けます。本シリーズで紹介した原則・パターン・手法は、あくまで判断の手掛かりです。

大事なのは「原則を知っていること」と「場面に応じて使い分けられること」。SOLIDを完璧に守る必要はないし、デザインパターンを全部覚える必要もありません。判断に詰まった時、何を手掛かりにすればよいかを知っていること。それだけで十分です。

設計は「動くコードを書く前に、どう組み立てるかを決める行為」。最初に時間をかけたぶん、後から数倍の労力で報われます。逆に最初をサボったコードは、半年後に必ず破綻します。

これからの皆さんの設計が、変更に強く、読みやすく、テストしやすく、長く愛されるコードでありますように。困った時はまた本シリーズに戻ってきてください。辞書として使い倒していただけるのが、私たちの願いです。
