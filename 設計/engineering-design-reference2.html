01

要件とは何か

WHAT IS A REQUIREMENT

要件は「お客さんの口から出た言葉」ではありません。顧客が表現した「Want（欲しいもの）」の奥にある「Need（本当に必要なこと）」こそが本物の要件です。ここを取り違えると、動くけど誰も使わないシステムができあがります。

want-vs-need 表面の要望 vs 本当の課題
// 表層と深層
[Want] お客さんが言ったこと ↓ なぜそれが欲しい? [Need] 本当に解決したい課題 ↓ どう実現するか [Solution] 設計

古典的な例:
「もっと速い馬が欲しい」
→ 速く移動したい（Need）
→ 自動車という設計解

言われたまま「速い馬」を作ってはいけません。

// 現場でよくある誤訳の例 [Want] 顧客の言葉] 「注文画面に検索機能つけて」 [NG: 言われたまま作る] 注文画面にキーワード検索ボックスを設置 → 使われない → 「結局、商品IDで探したかった」 [OK: なぜ?を3回問う] Q1: なぜ検索が欲しい? A1: 注文時に商品を素早く特定したい Q2: なぜ素早く? A2: 電話注文中に待たせたくない Q3: 普段どう特定してる? A3: 紙のカタログでID見て入力してる [Need: 本当の要件] 電話応対中に商品IDで即座に 商品を呼び出せる手段が欲しい [Solution: 設計] IDサジェスト + バーコード入力対応
POINT — トヨタの「なぜを5回」
顧客の最初の発言は氷山の一角です。「なぜ?」「具体的には?」「今はどうしてる?」を繰り返し、表面下の本当の課題を掘り起こすのが要件定義の鉄則です。
02

機能要件と非機能要件

FUNCTIONAL & NON-FUNCTIONAL

要件には2種類あります。機能要件は「何ができるか」、非機能要件は「どう動くか」です。後者を疎かにすると、機能は揃っているのに使い物にならないシステムが出来上がります。

functional-vs-nfr 2種類の要件を見分ける
// 非機能要件の代表的カテゴリ
  • 性能レスポンス3秒以内
  • 可用性99.9% / 月
  • セキュリティ個人情報暗号化
  • 拡張性同時1万接続
  • 運用性ログ保管90日
  • 保守性新人が1週間で改修可能

非機能要件は数値で書くのが鉄則。「速い」「安全」では設計できません。

// 機能要件 vs 非機能要件 [機能要件 — 何ができるか] - ユーザーは商品を検索できる - ユーザーはカートに追加できる - ユーザーはクレジットカードで決済できる - 管理者は売上レポートを出力できる [非機能要件 — どう動くか] 性能: 検索: 95%のリクエストが1秒以内 決済: 3秒以内に確定通知 可用性: 稼働率 99.95%（月間ダウン 22分以内） セキュリティ: カード情報は非保持（PCI DSS準拠） パスワードはbcryptでハッシュ化 拡張性: 同時ユーザー 5,000人 ピーク時 200 req/sec // 非機能要件が決まらないと // アーキテクチャは絶対に決まらない
裏技 — ハマりどころ
顧客は機能要件は熱心に語りますが、非機能要件は「当たり前でしょ」と言って明示しません。こちらから「ピーク時のアクセス数は?」「許容できる停止時間は?」と数値で聞き出すのがプロの仕事です。
03

ユースケースの書き方

USE CASE WRITING

ユースケースは「誰が、何をして、何を得るか」を1シナリオとして書き出したものです。機能を箇条書きで並べる代わりに、利用者の動きとして書くのがポイントです。

use-case-template 標準的なユースケース記述
// ユースケースの構成要素
項目内容
IDUC-001 等
名前「商品を購入する」
主アクター誰が使うか
事前条件ログイン済み等
基本フロー正常時の手順
代替フロー異常・分岐
事後条件完了時の状態
// ユースケース記述例 [UC-007] 商品を購入する 主アクター: 会員ユーザー 事前条件: ログイン済み / カートに商品1点以上 基本フロー: 1. ユーザーが「購入する」を押す 2. システムが配送先一覧を表示 3. ユーザーが配送先を選択 4. システムが決済方法を表示 5. ユーザーがクレカ情報を入力 6. システムが決済確定 → 注文番号発行 7. システムがメールで注文確認を送信 代替フロー: 5a. カード認証失敗 → エラー表示、5に戻る 6a. 在庫切れ → 1に戻る、該当商品をカートから除去 事後条件: - 注文レコードが PAID 状態で作成される - 在庫が減算される - ユーザーにメール送信される
POINT — 代替フローこそ宝
基本フローを書くのは誰でもできます。代替フロー（異常系・例外系）をどれだけ洗い出せるかが要件定義の腕の見せどころ。「在庫切れ」「カード失敗」「セッション切れ」を網羅すると、後の手戻りが激減します。
04

ユーザーストーリー

USER STORIES

ユースケースよりもっと軽量・アジャイル向きの表現がユーザーストーリーです。「As a 〜, I want 〜, so that 〜」の定型文で、誰が何のために何をしたいかを1行で表します。

user-story 3部構成の定型文
// ストーリーの公式
As a 〔誰として〕 I want 〔何をしたい〕 so that〔何を得たい〕

so that（目的）を必ず書くのがコツ。これがあると「本当にその機能が要るか」を判断できます。

受け入れ基準（Acceptance Criteria）とセットで管理：

  • Given 〔前提〕
  • When 〔操作〕
  • Then 〔期待結果〕
// ユーザーストーリー実例 [Story] As a 通販サイトの会員として I want 過去の注文履歴を見たい so that 同じ商品を再注文しやすくするため [Acceptance Criteria] Given ユーザーがログイン済みで 過去に3件以上の注文がある When マイページの「注文履歴」を開く Then 注文日の新しい順に表示される 各注文に「再注文」ボタンがある Given 注文履歴が0件の場合 When 注文履歴を開く Then 「まだ注文がありません」と表示 おすすめ商品を表示する // この粒度なら開発者は迷わない。 // QAはそのままテストケースに使える。
TIP — INVEST原則
良いユーザーストーリーは Independent（独立）/ Negotiable（交渉可能）/ Valuable（価値あり）/ Estimable（見積れる）/ Small（小さい）/ Testable（テスト可能）。頭文字でINVEST。これを満たさないストーリーは分割しましょう。
05

制約の見極め

IDENTIFYING CONSTRAINTS

設計の自由度を縛るのが制約です。技術的制約、ビジネス制約、法的制約。これらを把握せずに設計を始めると、後で「その方式じゃダメだった」とちゃぶ台返しを食らいます。

constraints 4種類の制約を洗い出す
// 制約のカテゴリ
  • 技術言語/FW指定、既存DB
    「Javaで作って」「Oracleがある」
  • ビジネス予算/納期/体制
    「3ヶ月で」「メンバー3人」
  • 法的個人情報保護法、業界規制
    「医療系/金融系の規制」
  • 運用既存運用ルール
    「夜間バッチ枠2時間まで」

制約は早い段階で全部書き出す。後から出てくると致命傷。

// 制約のヒアリング例 [技術制約] - 開発言語: Java 21 必須（社内標準） - DB: 既存 PostgreSQL 14 を流用 - インフラ: AWS のみ可（Azure不可） - フロント: React + TypeScript [ビジネス制約] - 予算: 2,000万円 - 期限: 2026/09/30 リリース - 体制: PG 3名 + PL 1名 [法的制約] - 個人情報保護法 準拠 - カード番号は PCI DSS 準拠で非保持 - アクセスログ保管 5年 [運用制約] - メンテ枠: 毎週 日曜 2:00-4:00 - 監視: 既存 Datadog に統合 - デプロイ: 月2回まで // この一覧が無いと設計は始まらない
POINT — 制約は敵ではなく味方
制約があるからこそ選択肢が絞られて、設計が早く決まるのです。「何でも自由に作っていいよ」が一番難しい。制約を恐れず、むしろ積極的にヒアリングしましょう。
06

要件から設計への変換

TRANSLATION TECHNIQUE

要件定義書から設計書へは、機械的な翻訳でかなりの部分が進められます。「名詞」「動詞」「数値」をそれぞれ設計のどの要素に変換するか、定型パターンを覚えておくと作業が早くなります。

translate-pattern 要件文の品詞 → 設計要素
// 翻訳ルール

名詞
（ユーザー、注文、商品）		エンティティ
クラス / テーブル
動詞
（注文する、確定する）		メソッド / API
ユースケース
形容詞
（有効な、未払いの）		状態 / フラグ
列の値
数値
（3秒以内、月10万件）		非機能要件
制約値
「〜の」
（注文の明細）		関連
外部キー
// 要件文を機械的に分解 [要件文] 「会員は、有効なクーポンを使って 商品の購入を割引できる」 [品詞分解] 名詞: 会員 / クーポン / 商品 / 購入 形容詞: 有効な 動詞: 使う / 割引する 関連: 商品の購入 [設計に変換] class Member { } class Coupon { status: CouponStatus; // ACTIVE/EXPIRED } class Product { } class Purchase { product: Product; member: Member; applyCoupon(coupon: Coupon) { if (coupon.status !== "ACTIVE") throw; // 割引適用 } } // 要件文の言葉が、そのまま // クラス名・メソッド名に現れている。 // これが「ユビキタス言語」の原型。
裏技 — ユビキタス言語
要件文に出てくる単語と、コードに出てくる単語を完全に一致させるのがDDD（ドメイン駆動設計）の核心。「顧客が『クーポン』と呼ぶものを、コードでは『Voucher』と呼ぶ」のはNGです。
07

優先順位付け（MoSCoW）

MOSCOW PRIORITIZATION

要件を全部実装しようとすると間に合いません。「絶対必要」と「あったらいい」を分けることが重要です。これを定型化したのがMoSCoW法（Must / Should / Could / Won't）です。

moscow 4段階の優先度
// MoSCoWの定義
  • MMust have
    これが無いとリリース不可 / 全体の60%以下
  • SShould have
    重要だが無くても動く / 20%
  • CCould have
    あれば嬉しい / 20%
  • WWon't have
    今回はやらない（次回検討）

特に重要なのが Won't。「やらないこと」を明示するとスコープが守れます。

// ECサイト リリース初版の優先付け [Must — 必須] - 会員登録 / ログイン - 商品一覧 / 詳細 - カート機能 - クレジット決済 - 注文確認メール送信 [Should — 重要] - 検索機能 - 注文履歴 - パスワードリセット - 在庫管理（管理者画面） [Could — あれば嬉しい] - レビュー投稿 - お気に入り - レコメンド表示 - クーポン適用 [Won't — 今回やらない] - 多言語対応 - ポイント制度 - サブスクリプション - モバイルアプリ // このリストを顧客と握っておくと、 // 途中で「これ追加して」と言われた時に // 「Won'tに分類されてます」と返せる。
