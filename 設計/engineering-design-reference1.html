01

なぜ設計が必要か

WHY DESIGN MATTERS

設計を飛ばして書き始めたコードは、最初の1週間は速いです。問題は1ヶ月後から始まります。修正のたびに別の箇所が壊れ、機能追加に途方もない時間がかかるようになります。

design-or-not 設計あり vs 設計なし
// 開発速度の推移イメージ
設計なし 速 ━━━━━━━╲ ╲___ぐにゃり ╲____ ╲___ ╲____ 設計あり 普通 ━━━━━━━━━━━━━━━━━━━━ （同じペースで進む）

最初は設計なしのほうが速い。しかし変更コストが指数関数的に増えていき、ある時点で逆転します。そして二度と追いつけません。

// 設計なしで書いたコードのなれの果て function processOrder(order) { // 注文処理…のはずが if (order.type == "A") { // 在庫チェックも if (stock[order.item] < 1) { sendEmail(...); // メール送信も log(...); // ログも updateDB(...); // DB更新も // 全部この関数の中 } } else if (order.type == "B") { // 同じような処理がコピペで… } // 800行続く }
POINT — 設計の目的は3つ
①変更容易性（後から直しやすい）、②理解容易性（他人が読める）、③正しさの保証（意図した通りに動く）。この3つを満たすために設計を行います。コードを速く書くためではありません。
02

設計の3つのレイヤ

THREE LAYERS OF DESIGN

設計と一口に言っても抽象度が3段階あります。それぞれ扱う粒度・道具・成果物が違います。混同すると会話が噛み合いません。

design-layers 要件 → 基本 → 詳細
// 抽象度の高い順
  • L1要件設計
    何を作るか / 誰のためか
    → ユースケース、要件定義書
  • L2基本設計（アーキテクチャ）
    どう組み立てるか / 何で作るか
    → アーキ図、画面遷移、API一覧
  • L3詳細設計
    どう書くか / クラス/関数の中身
    → クラス図、ER図、シーケンス図

上から決めて下に降りるのが基本。下から上は決めないのが鉄則。

// 同じ「ログイン機能」を3レイヤで設計 [L1 要件設計] ユーザーは メールとパスワードで ログインできる。 連続失敗5回でロックされる。 [L2 基本設計] 画面: /login (POST) API: POST /api/auth/login DB: users, login_attempts 認証方式: JWT (24h有効) [L3 詳細設計] class LoginUseCase { execute(email, password) { // 1. ユーザー検索 // 2. パスワード照合 // 3. 試行回数チェック // 4. JWT発行 } }
TIP — 役割と巻の対応
本シリーズではVol.2が要件設計、Vol.3〜5が基本設計、Vol.6〜7が詳細設計のうちデータとAPI、Vol.8が成果物（ドキュメント）を担当します。
03

設計プロセスの全体像

THE DESIGN PROCESS

設計は一直線ではなく往復運動です。要件から実装まで一気に降りるのではなく、下のレイヤで気付いたことを上に戻して修正するのが現実的なやり方です。

design-process V字モデル + フィードバック
// 上から下へ、そして戻る
要件定義 ↓ ↑ 基本設計 ← レビューで戻る ↓ ↑ 詳細設計 ← 実装中に矛盾発覚 ↓ ↑ 実装 ↓ ↑ テスト ← 仕様の穴が見つかる

下のフェーズで気づいた問題は、上のフェーズに戻して直すのが正攻法。「もう要件は決まったから」と無視すると、辻褄合わせの汚いコードが量産されます。

// 現実によく起きる手戻り例 [実装中の発覚] 「DBの設計だとN+1問題が出る」 ↓ [詳細設計へ戻す] ER図の関連を見直し ↓ [基本設計へ戻す] 画面の一覧表示方式を変更 ↓ [要件へ戻す] 「そもそも1画面に全部要る?」 を顧客に確認 ↓ [再設計] ページネーション導入で解決 // この往復を恐れないこと。 // 早く戻るほど傷は浅い。
POINT — 設計の鉄則
「実装で気づいた問題を、実装の中だけで解決しない」。原因が要件にあるなら要件に戻る。これができるチームの設計は強くなります。
04

良い設計の特徴

CHARACTERISTICS OF GOOD DESIGN

良い設計の判断基準は感性ではなく客観的な指標で測れます。以下の4つを満たしているかチェックしてみてください。

good-design-check 4つの評価軸
// 良い設計のチェックリスト
  • ★★★ 変更容易性
    新機能追加で触る箇所が少ない
  • ★★★ 理解容易性
    新メンバーが1日で全体像を掴める
  • ★★ テスト容易性
    単体テストが書きやすい構造
  • ★★ 再利用性
    同じ部品を別の場面でも使える

最重要は「変更容易性」。ソフトウェアは変わり続けるものだからです。

// 良い設計の例: 決済方法の追加 interface PaymentMethod { pay(amount: number): Result; } class CreditCard implements PaymentMethod { ... } class BankTransfer implements PaymentMethod { ... } // 新しい決済方法を追加するときは… class PayPay implements PaymentMethod { pay(amount) { // 実装するだけ } } // 既存コードは1行も触らない。 // これが「変更容易」の意味。
裏技 — 設計品質の自己診断
迷ったら「この設計、半年後の自分が読んで5分で理解できるか?」と問うてください。NoならVol.3の設計原則を見直す合図です。
05

悪い設計の兆候

CODE SMELLS

設計が崩れているコードには共通の症状があります。これらを「コードの臭い（Code Smell）」と呼びます。早めに嗅ぎ取れるようになると、リファクタリングのタイミングを逃しません。

code-smells 代表的な7つの兆候
// よく見るコード臭
症状意味
God Class1クラスが何でもやる
Long Method1メソッド100行超
Magic Number数字直書き
Duplicate Codeコピペの群れ
Shotgun Surgery1機能直すのに10箇所触る
Feature Envy他クラスの中身ばかり触る
Dead Code使われていないコード
// God Class の典型例 class UserManager { register() {...} login() {...} sendMail() {...} // メール? generatePDF() {...} // PDF? calculateTax() {...} // 税金? backupDatabase() {...} // DB? // …Userだけじゃないよね? } // Magic Number の例 if (user.age >= 20) { ... } // この 20 は何? // → 成人年齢? 投票年齢? // 改善: 名前を付ける const ADULT_AGE = 20; if (user.age >= ADULT_AGE) { ... }
TIP — 詰まりポイント
「動いてるから直さなくていい」と放置すると、半年後に倍の労力で直すハメになります。コードの臭いに気づいた時が、リファクタリングのベストタイミングです。
06

設計とコードの関係

DESIGN AS CODE

設計とコードは別物ではありません。設計は「書くべきコードの形」を決める行為で、コードはその実体です。クラス名・メソッド名・パッケージ構成、これら全てが設計の表現です。

design-in-code 命名と構造が設計
// パッケージ構成は設計そのもの
src/ ├─ domain/ ← ビジネスルール │ ├─ user/ │ └─ order/ ├─ application/ ← ユースケース │ ├─ register/ │ └─ checkout/ ├─ infrastructure/ ← DB/外部API │ ├─ repository/ │ └─ client/ └─ presentation/ ← UI/Controller ├─ web/ └─ api/

フォルダ構成を見ただけで「このシステムは何をしているか」が分かるのが理想。これが「設計がコードに現れている」状態です。

// 名前で意図が伝わる例 [NG: 意図が伝わらない] class Manager { do(d) { ... } proc(x) { ... } } [OK: 名前が説明になっている] class OrderConfirmationService { confirmOrder(order: Order) { ... } notifyCustomer(c: Customer) { ... } } // クラス名・メソッド名を見るだけで // 何をする部品か分かる。 // これが設計の「コード上の現れ」。
POINT — 設計書は紙の上だけにあらず
クラス名・パッケージ名・関数名が、最も読まれる設計ドキュメントです。Word文書よりコードのほうが正確で最新です。命名に時間をかけることは、設計に時間をかけることと同義です。
