要求工学用語集編

Glossary — Words That Define the Field

要求工学では多くの専門用語が飛び交います。本巻は辞書として開くためのリファレンス。8カテゴリに整理し、各用語は「読み・英語・定義・関連語」のカード形式。本シリーズで登場した用語を網羅しつつ、現場で遭遇する用語も補完しました。分からない言葉に出会ったら、まずここを開く運用です。

01

基本概念

FUNDAMENTAL CONCEPTS

要求工学の土台となる用語。「要求と要件って違うの？」「ステークホルダーとユーザーは？」——この章でまず押さえます。

要求 ようきゅう
REQUEST / NEED
ステークホルダーが「欲しい」「困っている」と表明した内容。主観的・曖昧でも構わない、生の声。
関連: 要件、ニーズ、Voice of Customer
要件 ようけん
REQUIREMENT
要求を分析・整理し、システムが満たすべき条件として明文化したもの。測定可能・検証可能であること。
関連: 要求、仕様、SRS
ステークホルダー
STAKEHOLDER
システムの開発・運用・利用に利害を持つ全ての人。ユーザー、顧客、経営層、運用、法務などを含む。
関連: ユーザー、顧客、Power/Interest Grid
プロダクトオーナー
PRODUCT OWNER (PO)
スクラムにおける要求の最終決定権者。バックログの優先順位を決め、何を作るかに責任を持つ。
関連: PdM、PM、BA
スコープ
SCOPE
プロジェクトで「やる」「やらない」の範囲。スコープが膨張することをスコープクリープと呼ぶ。
関連: Won't (MoSCoW)、In/Out リスト
MVP
MINIMUM VIABLE PRODUCT
価値を検証できる「実用上の最小製品」。すべての機能を盛り込まず、コア価値だけで世に出す。
関連: Lean Startup、プロトタイプ
プロダクトビジョン
PRODUCT VISION
プロダクトが長期的に目指す姿。日々の判断のよりどころとなる「北極星」。
関連: 北極星指標、ロードマップ
要求工学
REQUIREMENTS ENGINEERING
要求の獲得・分析・仕様化・検証・管理を体系的に扱う工学分野。コードを書く前の重要な技術領域。
関連: IEEE 830、ISO/IEC 29148
02

獲得・調査

ELICITATION / RESEARCH

要求を「聞き出す・引き出す」段階の用語。インタビュー、ワークショップ、観察などの技法に関する語彙。

要求獲得 ようきゅうかくとく
REQUIREMENTS ELICITATION
ステークホルダーから要求を引き出す・聞き出す活動の総称。インタビュー、観察等を含む。
関連: インタビュー、ワークショップ
インタビュー
INTERVIEW
1対1で深く話を聞く技法。開いた質問・閉じた質問を使い分け、5whyで深掘りする。
関連: 開いた質問、5why、誘導尋問
ワークショップ
WORKSHOP
複数のステークホルダーが同じ場で議論する場。合意形成と発散・収束を同時に行う。
関連: ファシリテーション、ブレスト
観察
OBSERVATION / CONTEXTUAL INQUIRY
業務現場に出向き、実際の作業を見る技法。本人が言葉にできない「暗黙の作法」を発見できる。
関連: エスノグラフィ、Job Shadowing
ペルソナ
PERSONA
典型ユーザーを1人の具体的な人物として描いたもの。「30代女性」より「田中さん、34歳、子持ち」が強い。
関連: カスタマージャーニー、共感マップ
カスタマージャーニー
CUSTOMER JOURNEY MAP
顧客が製品と関わる全段階（認知→検討→購入→利用→継続）を時系列で可視化したもの。
関連: タッチポイント、ペインポイント
5why ふぁいぶわい
FIVE WHYS
「なぜ？」を5回繰り返して根本原因に迫る技法。トヨタ生産方式由来、要求の深掘りにも有効。
関連: 根本原因分析、なぜなぜ分析
KJ法 けーじぇーほう
AFFINITY DIAGRAM
川喜田二郎が考案した親和図法。発言や付箋を似たもの同士で集めて構造化する整理術。
関連: 親和図、ブレインストーミング
Power/Interest Grid
POWER / INTEREST GRID
ステークホルダーを影響力×関心度の4象限に分け、それぞれに最適な関わり方を決める。
関連: ステークホルダー分析
03

機能要求・ストーリー

FUNCTIONAL / STORIES

アジャイル開発における機能要求の記述法に関する用語。ユーザーストーリー、ユースケース、受入条件など。

機能要求
FUNCTIONAL REQUIREMENT (FR)
システムが「何をするか」を表す要求。「ユーザーは商品を検索できる」など、振る舞いの仕様。
関連: 非機能要求、ストーリー
ユーザーストーリー
USER STORY
「As a 〜, I want 〜, so that 〜」形式で書く、ユーザー視点の機能要求。アジャイルの標準。
関連: Connextra形式、3C、INVEST
3C
CARD / CONVERSATION / CONFIRMATION
ユーザーストーリーの本質。カード（短い文）・会話（議論）・確認（受入条件）の3要素。
関連: ユーザーストーリー
INVEST
INVEST PRINCIPLES
良いストーリーの6条件：Independent / Negotiable / Valuable / Estimable / Small / Testable。
関連: Bill Wake、ユーザーストーリー
受入条件
ACCEPTANCE CRITERIA (AC)
ストーリーの「完成判定基準」。Given/When/Then で書くのが標準。テスト可能であること。
関連: Given/When/Then、BDD
エピック
EPIC
複数のストーリーをまとめた大きなストーリー。1スプリントには収まらないため、実装直前に分割する。
関連: テーマ、ストーリー
テーマ
THEME
複数のエピックを束ねる大目標。半年〜1年単位の戦略的なまとまり。
関連: エピック、ロードマップ
ユースケース
USE CASE
アクターとシステムの詳細なやりとり手順。基本フロー・代替フローを記述。規制業界で多用。
関連: UML、アクター、基本フロー
アクター
ACTOR
ユースケース図に登場するシステムの外部利用者。人間に限らず、外部システムも含む。
関連: ユースケース、UML
Connextra形式
CONNEXTRA FORMAT
「As a [役割], I want [機能], so that [理由]」というユーザーストーリーの標準テンプレート。
関連: ユーザーストーリー
04

非機能要求

NON-FUNCTIONAL

品質・性能・セキュリティ等の用語。SLA/SLO/SLI、ISO/IEC 25010など、現場で頻出する語を整理。

非機能要求
NON-FUNCTIONAL REQUIREMENT (NFR)
システムが「どう動くか」を表す要求。性能・可用性・セキュリティ・使いやすさなど。
関連: 品質要求、ISO/IEC 25010
ISO/IEC 25010
QUALITY MODEL
ソフトウェア品質の国際標準。8品質特性（機能適合性、性能効率性、互換性、使用性、信頼性、セキュリティ、保守性、移植性）。
関連: 旧9126、品質特性
SLA えすえるえー
SERVICE LEVEL AGREEMENT
顧客と取り交わすサービス品質の契約。違反時のペナルティを含む。
関連: SLO、SLI
SLO えすえるおー
SERVICE LEVEL OBJECTIVE
サービス提供側の内部目標値。SLAより厳しく設定し、SLA違反を予防する。
関連: SLA、SRE
SLI えすえるあい
SERVICE LEVEL INDICATOR
実際に計測する指標。「成功リクエストの割合」「P95レイテンシ」など、具体的な数値。
関連: SLO、メトリクス
可用性
AVAILABILITY
システムが使える状態である割合。99.9%なら年間8.76時間の停止許容を意味する。
関連: SLA、稼働率、MTBF
スループット
THROUGHPUT
単位時間あたりの処理可能量。req/s（リクエスト毎秒）、TPS（トランザクション毎秒）などで表す。
関連: レイテンシ、性能要求
レイテンシ
LATENCY
1リクエストの応答時間。平均値ではなくパーセンタイル（P95等）で語るのが現代の標準。
関連: パーセンタイル、性能要求
RTO / RPO
RECOVERY TIME / POINT OBJECTIVE
RTO=障害復旧までの目標時間。RPO=許容されるデータ損失時間。災害対策の指標。
関連: BCP、DR、バックアップ
CIA
CONFIDENTIALITY / INTEGRITY / AVAILABILITY
情報セキュリティの3要素：機密性・完全性・可用性。セキュリティ要求の基本枠組み。
関連: セキュリティ要求
WCAG
WEB CONTENT ACCESSIBILITY GUIDELINES
Webアクセシビリティの国際ガイドライン。レベルA/AA/AAAで適合度を示す。
関連: アクセシビリティ、A11y
SUS
SYSTEM USABILITY SCALE
10問の質問でユーザビリティを0〜100点で数値化する標準手法。68点が業界平均。
関連: 使用性、UX指標
05

分析・モデリング

ANALYSIS / MODELING

要求を整理・構造化・視覚化する技法の用語。ストーリーマッピング、イベントストーミング、ドメインモデルなど。

As-Is / To-Be あずいず／とぅーびー
AS-IS / TO-BE ANALYSIS
業務の現状（As-Is）とあるべき姿（To-Be）を対比し、ギャップを抽出する分析。
関連: 業務分析、BPM
ストーリーマッピング
USER STORY MAPPING
Jeff Patton考案。左→右にユーザーの行動の流れ、上→下に詳細度でストーリーを2次元配置する技法。
関連: バックボーン、ウォーキングスケルトン
インパクトマッピング
IMPACT MAPPING
Gojko Adzic考案。WHY（目標）→ WHO（誰）→ HOW（行動変化）→ WHAT（機能）の階層で機能の根拠を可視化。
関連: アウトカム志向、価値検証
イベントストーミング
EVENT STORMING
Alberto Brandolini考案。業務上のイベントをオレンジ付箋で時系列に並べ、ドメインを発見するワークショップ技法。
関連: DDD、ドメインモデル
コンテキスト図
CONTEXT DIAGRAM
システムを1つの円で表し、外部の関係者・他システムとの相互作用を示す図。要求のスコープを定義。
関連: DFD、システム境界
ドメインモデル
DOMAIN MODEL
業務領域の概念とその関係を表すモデル。業務側の用語で書くのが鉄則。実装やDBではない。
関連: DDD、ユビキタス言語
ユビキタス言語
UBIQUITOUS LANGUAGE
DDDの中心概念。業務・開発・テストの全員が同じ用語を使うことで、認識ズレを防ぐ。
関連: DDD、ドメインモデル
MECE みーしー
MUTUALLY EXCLUSIVE COLLECTIVELY EXHAUSTIVE
「重複なく、漏れなく」整理する原則。コンサル業界由来、要求の分類で頻用。
関連: ロジカルシンキング
バックボーン
BACKBONE
ストーリーマップの最上段。ユーザーが行う「大きな活動」を左から右に並べる骨格。
関連: ストーリーマッピング
06

優先順位・見積もり

PRIORITIZATION / ESTIMATION

「やる/やらない」「どの順で」を決める技法と、見積もりに関する用語。

プロダクトバックログ
PRODUCT BACKLOG
プロダクトに必要な全タスクの優先順位付きリスト。POが管理。DEEP原則に従う「生きているリスト」。
関連: DEEP、PBI、Refinement
DEEP
DETAILED / ESTIMATED / EMERGENT / PRIORITIZED
良いバックログの4性質。適切に詳細化・見積もり済み・常に変化・優先順位付き。
関連: プロダクトバックログ
リファインメント
BACKLOG REFINEMENT
バックログを「Ready」な状態に磨き続ける活動。詳細化・分割・見積もり・優先順位調整。
関連: Grooming（旧称）、DoR
MoSCoW法 もすくわほう
MOSCOW METHOD
要求をMust / Should / Could / Won'tの4分類に振り分ける優先順位付け技法。
関連: 勇気テスト、DSDM
WSJF
WEIGHTED SHORTEST JOB FIRST
SAFe標準の優先順位付け。CoD（遅延コスト）÷ サイズでスコア化、高い順から実施。
関連: CoD、SAFe
カノモデル
KANO MODEL
狩野紀昭考案。機能を当たり前品質・一元的品質・魅力的品質の3カテゴリで分類する満足度モデル。
関連: Delighter、Must-Be
ストーリーポイント
STORY POINT (SP)
作業の相対的なサイズを表す単位。時間ではない。フィボナッチ数列（1,2,3,5,8,13...）で見積もる。
関連: プランニングポーカー、ベロシティ
プランニングポーカー
PLANNING POKER
チーム全員が同時にカードを出して見積もる手法。同調圧力を排除し、認識のズレを発見できる。
関連: ストーリーポイント、Wideband Delphi
ベロシティ
VELOCITY
1スプリントで完了するストーリーポイントの量。リリース予測に使う。個人評価には絶対に使わない。
関連: Capacity、リリース計画
CoD
COST OF DELAY
「実装が遅れることによる損失」。ビジネス価値＋緊急度＋リスク削減/機会創出の合計で表現。
関連: WSJF
DoR / DoD
DEFINITION OF READY / DONE
DoR=ストーリーが「着手可能」と認められる条件。DoD=「完了」と認められる条件。チームで合意。
関連: 受入条件、リファインメント
07

検証・管理

VALIDATION / MANAGEMENT

作った要求の確認・追跡・変更管理に関する用語。

検証 (Verification)
VERIFICATION
「仕様通りに作られているか」を確認する活動。仕様 ⇔ 成果物の整合確認。
関連: 妥当性確認、レビュー
妥当性確認 (Validation)
VALIDATION
「ユーザーが望むものを作っているか」を確認する活動。現実 ⇔ 仕様の整合確認。
関連: UAT、スプリントレビュー
レビュー
REVIEW
文書やコードを他者の目で確認する活動。ピアレビュー・ウォークスルー・インスペクションの3段階。
関連: インスペクション、Fagan法
インスペクション
INSPECTION
Faganが提唱した厳格な公式レビュー。事前準備・公式会議・追跡を含み、欠陥検出率が最も高い。
関連: レビュー、ウォークスルー
プロトタイピング
PROTOTYPING
動くものを早く作って見せる技法。使い捨て型と進化型がある。期待値管理が成否を分ける。
関連: ワイヤフレーム、モックアップ
スプリントレビュー
SPRINT REVIEW
スプリント末に行う動くインクリメントのデモ会。ステークホルダーから直接フィードバックを得る。
関連: Validation、UAT
UAT
USER ACCEPTANCE TEST
ユーザーによる受入テスト。本番投入前の最終確認。ユーザー視点で要求が満たされているか検証。
関連: 妥当性確認、受入条件
変更要求 (CR)
CHANGE REQUEST
既存の要求や仕様への変更依頼。正式な票として記録・分析・判断・追跡を行う。
関連: CCB、影響分析
CCB
CHANGE CONTROL BOARD
変更要求を承認・却下する組織。PO・PM・主要ステークホルダーで構成。
関連: 変更管理、ガバナンス
影響分析
IMPACT ANALYSIS
変更がどこに波及するかを特定する活動。他要求・設計・コード・データ・運用への影響を網羅。
関連: トレーサビリティ、CR
トレーサビリティ
TRACEABILITY
ビジネス目標→ストーリー→テスト→コード→リリースを双方向に追跡できる状態。規制業界では必須。
関連: 影響分析、監査
ベースライン
BASELINE
合意された時点の要求のスナップショット。以後の変更はベースラインからの差分として管理する。
関連: バージョン管理、CR
