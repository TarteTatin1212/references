要求獲得編

Elicitation — Listening to Stakeholders

要求は「自然と湧いて出てくる」ものではありません。誰かに尋ね、観察し、引き出して、初めて手に入ります。本巻ではインタビュー、ワークショップ、観察、アンケート——「聞き出す技術」の全体を扱います。ステークホルダーの言葉を、価値ある要求に変えるのが要求獲得です。

01

要求獲得の全体像

ELICITATION OVERVIEW

要求獲得（Elicitation）は「聞き出す」「引き出す」活動です。技法は1つではなく、状況に応じて使い分け・組み合わせます。ユーザーが明確に言葉にできない「暗黙の要求」を引き出すことが、本当の腕の見せどころ。

elicitation_techniques代表的な6つの獲得技法
// 主要な獲得技法とその特徴
  • インタビュー1対1で深く聞く（個人の本音）
  • ワークショップ複数人で議論（合意形成）
  • 観察現場で見る（暗黙知の発見）
  • アンケート広く浅く集める（傾向把握）
  • 既存資料分析マニュアル・帳票（現状の把握）
  • プロトタイプ動くものを見せる（具体化）

★★ 技法は併用するのが鉄則。インタビューだけで全部把握できると思うのは危険です。

# 状況別の技法選択 def choose_technique(context): if context == "新規プロジェクト": return [ "既存資料分析", # まず読む "キーパーソンIV", # 全体像 "ワークショップ", # 合意 ] elif context == "業務改善": return [ "観察", # 現場へ "インタビュー", # 課題深掘 ] elif context == "既存改修": return [ "アンケート", # 不満収集 "プロトタイプ", # 確認 ] # 1つの技法で済むことはほぼない
POINT —
「ユーザーは欲しいものを言葉にできない」(Henry Ford風の格言)——だから複数の技法で多角的に引き出します。言葉にされない要求こそ、価値が高いことが多い。
02

ステークホルダー分析

STAKEHOLDER ANALYSIS

「誰に聞くか」は「何を聞くか」より重要です。影響力と関心度の2軸でステークホルダーを4象限に分け、それぞれに最適な接し方を決めます。これをPower/Interest Gridと呼びます。

power_interest_grid影響力 × 関心度の4象限
// Power/Interest Grid

高影響・高関心

Key Player
密に連携。最優先で対話。

例：プロダクトオーナー、主要顧客

高影響・低関心

Keep Satisfied
満足させる。重要決定時に確認。

例：経営層、セキュリティ部門

低影響・高関心

Keep Informed
情報を共有。意見は聞く。

例：エンドユーザー、運用担当

低影響・低関心

Monitor
軽く監視。最小限のコミュニケーション。

例：間接的な関係部署

★★★ 「全員と密に対話」は時間が足りません。優先順位を決めて接します。

# ステークホルダーリストの例 stakeholders = [ SH(name="営業部長", influence=5, interest=5, # → Key Player strategy="weekly_1on1"), SH(name="CFO", influence=5, interest=2, # → Keep Satisfied strategy="monthly_report"), SH(name="現場オペレーター", influence=2, interest=5, # → Keep Informed strategy="newsletter, demo"), SH(name="隣接部署", influence=2, interest=2, # → Monitor strategy="mailing_list_only"), ] # 影響力5 × 関心度5 が最重要
裏技 —
「敵対的なステークホルダー」を見つけたら最初に押さえます。後から反対されると致命傷。反対派こそ早期に取り込むのが鉄則です。
03

インタビュー技法

INTERVIEW

インタビューは要求獲得の王道。ただし「ぼんやり聞いて、ぼんやり答えてもらう」では何も得られません。準備・進行・記録の3点を押さえることで、得られる情報の質が劇的に変わります。

interview_technique開いた質問と閉じた質問の使い分け
// 質問の2タイプ

開いた質問 (Open)

「普段、この業務でどんなことに困りますか？」

→ 自由に語ってもらう
→ 暗黙の課題が見える

閉じた質問 (Closed)

「この処理は1日何件くらいですか？」

→ 数字や事実を確定
→ 仕様化の材料

★★★ 最初は開いた質問で文脈を広げ、後半に閉じた質問で事実を固めるのが定石。

5why「なぜ？」を5回繰り返すと根本要因が見える、という技法も併用されます。

# インタビュー進行テンプレート [1] アイスブレイク（5分） # 相手の緊張を解く "今日はお時間ありがとうございます" [2] 全体像の質問（15分） "普段の業務の流れを 教えていただけますか？" [3] 課題の深掘り（20分） "その中で困っていることは？" "なぜそれが起きるんですか？" # 5why "なぜ今のやり方なんですか？" [4] 具体化の質問（15分） "1日何件くらいですか？" "何人で対応していますか？" [5] クロージング（5分） "他に伝えたいことは？" # これが意外と本音が出る
ハマりどころ —
誘導尋問に要注意。「〜って大変ですよね？」と聞くと、相手は「はい」と答えがち。事実と意見を分けて聞くのが鉄則です。「先週は何件処理しましたか？」(事実) →「それは多いと感じますか？」(意見)。
04

ワークショップ

WORKSHOP

複数のステークホルダーを集めて同じテーブルで議論する手法です。インタビューと違って「合意形成」が同時にできるのが強み。ただし準備の質が成否を分けます。アジャイル時代のイベントストーミングやストーリーマッピングもこの仲間です（Vol.5で詳述）。

requirements_workshop複数人で要求を引き出す・合意する
// ワークショップの構成要素
  • ファシリテーター進行役。中立を保つ
  • スクライブ記録係。発言を構造化
  • 参加者5〜8名が最適（多すぎ注意）
  • 時間90〜180分（半日まで）
  • 道具付箋、ホワイトボード、Miro等
  • アウトプット合意事項リスト・未決リスト

★★ 事前にゴールを明文化しないワークショップは、ほぼ確実に脱線します。

# ワークショップ運営の流れ [事前] # ゴール定義（最重要） goal = "新システムのコア機能を 5つに絞り込む" # アジェンダ・参加者・場所 prepare(goal, agenda, members) [当日] opening() # ゴール共有 divergence() # 発散（出す） clustering() # 整理（束ねる） prioritize() # 優先（絞る） closing() # 合意確認 [事後] # 24時間以内に議事録共有 share_minutes() followup_unsolved() # 発散→収束 を意識的に分ける
TIP —
役職の高い人と低い人を同席させると、発言が偏ります。匿名で付箋に書いてから貼る「サイレント・ブレインストーミング」を使うと、誰の意見も平等に出ます。
05

観察と現場訪問

OBSERVATION / CONTEXTUAL INQUIRY

「聞く」だけでは見えないものがあります。現場に行き、実際の作業を観察するのが観察手法（Contextual Inquiry）。ユーザー自身が意識していない「当たり前の手順」「裏マニュアル」「Excelの神シート」が、ここで発見されます。

contextual_inquiry現場の暗黙知を発見する
// 観察で見つかるもの
  • 裏マニュアル正式な手順書にないExcel・付箋
  • 回避策「このボタンは押すとバグるから使わない」
  • 職人技長年の勘・経験で判断している部分
  • 無駄本人が気づいていない冗長作業
  • ピーク月末・期末などの特殊状況
  • コミュニケーション誰が誰に何を聞いているか

★★★ 観察した内容は「本人に確認」します。本人にとっては当たり前すぎて、口頭では出てこない情報が宝の山。

# 観察のチェックリスト @observation class ContextualInquiry: # 観察の基本原則 principles = [ "Master-Apprentice", # 弟子のように学ぶ姿勢 "Interpretation", # 観察 → 解釈 → 確認のループ "Focus", # テーマを明確にして観察 "Partnership", # 評価ではなく協働 ] # 質問例 questions = [ "今、何をなさっていますか？", "なぜそうするんですか？", "いつもこの順番ですか？", "これは何のファイルですか？", ]
現場の声 —
「業務マニュアル通りの仕事をしている人はほぼいない」——観察するとよく分かります。「マニュアル」と「実態」のギャップこそ、新システムが解決すべき課題です。
06

アンケートと既存資料

SURVEY / DOCUMENT ANALYSIS

アンケートは「広く浅く」集めるのに最適。逆にインタビューは「狭く深く」。使い分けと併用が要です。また既存業務マニュアル・帳票・過去のシステム仕様書は「無料の宝の山」。まずここを読まずに獲得を始めるのは時間の無駄です。

survey_and_documents広く集める／既にあるものを使う
// 各手法の使い分け

アンケート

強み: 大人数の傾向把握
弱み: 深い理由が分からない
使う時: 機能の優先度投票、満足度測定
5段階評価＋自由記述が定番

既存資料分析

強み: 無料、客観的、網羅的
弱み: 実態と乖離している場合あり
使う時: プロジェクト初期、業務理解
マニュアル・帳票・規程・過去仕様書

★★ 最初に既存資料を読むことで、インタビューの質問が圧倒的に深くなります。

# 既存資料の分析チェックリスト documents_to_review = [ "業務マニュアル", "申請書・帳票のテンプレート", "組織図", "業務規程", "既存システムの仕様書", "過去の改修履歴", "ヘルプデスクの問い合わせ履歴", "監査報告書", ] # 特に有用なのが # 「ヘルプデスク履歴」 # 困りごとの宝庫 @survey_design questions = { # 5段階評価（量的） "満足度": "1=不満 〜 5=満足", # 自由記述（質的） "改善案": "自由にどうぞ", # 比較順位（優先度） "重要機能TOP3": "3つ選択", }
裏技 —
「ヘルプデスク・コールセンターの問い合わせログ」は金の鉱脈です。ユーザーが本当に困っているポイントが、生の言葉で蓄積されています。社内の人脈で入手できないか試してみましょう。
07

ペルソナの作成

PERSONA

獲得した情報をもとに、典型ユーザー像を1人の人物として描くのがペルソナです。「30代女性のユーザー層」では曖昧ですが、「田中さん、34歳、子育て中、スマホは寝室で使う」だと議論が具体的になります。アジャイルではユーザーストーリーの「As a 〜」の主語になる重要要素。

persona_creation「ユーザー像」を1人の人物として描く
// ペルソナの例（ECサイト）
佐藤 美咲（34歳・女性）
職業: 営業職（フルタイム）
家族: 夫・子1人（4歳）
スマホ: iPhone（通勤中・寝る前に使用）
GOALS
・週末の買い物時間を減らしたい
・子どもの服を手早く揃えたい
FRUSTRATIONS
・サイト内検索で目的の商品が出ない
・レビューが多すぎて読みきれない
TECH SAVVY
中の上（決済アプリは普通に使う）

★★★ ペルソナは2〜4人程度に絞ります。多すぎると焦点がぼけます。

# ペルソナのテンプレート class Persona: name: str # 実名風 age: int occupation: str family: str devices: list[str] # 行動特性 tech_level: str # 低/中/高 usage_context: str # いつ使う # 心理特性 goals: list[str] # 達成したいこと frustrations: list[str] motivations: list[str] # データ起源 sources: list[str] # 何件のインタビューから # 何件のアンケートから # 作ったかを記録 # 想像で作らない # 必ず実データに基づく
ハマりどころ —
「空想ペルソナ」は最悪です。インタビュー・観察データに基づかない、頭の中だけのペルソナは、議論を間違った方向に導きます。「このペルソナは何人の話から作ったか」を必ず添えましょう。
08

獲得結果の整理

CONSOLIDATION

インタビュー・観察・アンケートで集めた情報はそのままでは使えません。整理・構造化して初めて要求として活用できます。ここで使う技法が親和図法（KJ法）と要求の分類。獲得→整理は不可分です。

consolidation生の声 → 構造化された要求
// 親和図法 (KJ法) の流れ
[1] 発言・観察を1枚ずつ付箋に書く ↓ [2] 似たもの同士を集める ↓ [3] グループに名前を付ける ↓ [4] グループ同士の関係を線で結ぶ ↓ [5] 全体構造として図解する 「機能要求」「非機能要求」「業務ルール」 「制約」「願望」などのカテゴリに分かれていく

★★ 整理の段階で重複・矛盾・欠落が見えてきます。これがVol.5「分析・モデリング」への入り口です。

# 要求の典型カテゴリ class RequirementCategory(Enum): # 「何を」やるか FUNCTIONAL = "functional" # → Vol.3 ユーザーストーリー編 # 「どう」やるか（性能・品質） NON_FUNCTIONAL = "non_functional" # → Vol.4 非機能要求編 # 業務ルール・制約 CONSTRAINT = "constraint" # 法律、社内規程、技術制約 # 願望・ニーズ（要件未満） WISH = "wish" # 「あったらいいな」レベル # 当然のもの IMPLICIT = "implicit" # 言われなくても期待される # 全て分類すると、抜けが見える
POINT —
獲得した情報をその日のうちに整理するのが鉄則。記憶が新しいうちなら、付箋の意味も思い出せます。1週間放置すると「これ何の意味だっけ？」状態になります。
NEXT STEP —
獲得した要求の中でも「機能」に関するものは、アジャイル開発ではユーザーストーリーとして記述するのが主流です。Vol.3 ユーザーストーリー編では、INVEST原則・受入条件・ストーリー分割など、現代の機能要求記述の標準技法を扱います。
