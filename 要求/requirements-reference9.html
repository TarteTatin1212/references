Q.01

顧客と何回打ち合わせしても要求が決まりません。どうすればいいですか？

A. ANSWER

「言葉で議論」しているうちは、永遠に決まりません。動くもの・触れるもので議論に切り替えるのが王道です。要求が決まらない最大の原因は、顧客自身も「何が欲しいか」が分かっていないことです。

NG「もう一度仕様書を作り直しますので、来週レビューをお願いします」

→ 何度作っても「ちょっと違う」と言われ続ける
OK「来週、紙のプロトタイプを持ってきます。実際にこの画面で操作する感じを確かめてください」

→ 触ることで、自分が本当に欲しかったものに気づく

具体的な対処手順

  1. 議論の対象を「文書」から「画面」へ変える。紙でもFigmaでも良い
  2. 15分で作れる粗いプロトを持参する（完璧にしない）
  3. 「これでイメージ合ってますか？」と現物を見せる
  4. 修正版を翌日に作って再提示する短サイクルで回す
  5. 3回プロト共有してまだ揉めるなら、上位の関係者を巻き込む
TIP —
「百聞は一見に如かず」は要求工学の鉄則です。「論より実物」。プロトの作成時間は議論時間の1/10で済みます。
Q.02

ステークホルダーが毎回言うことを変えてきます。どう対応すれば？

A. ANSWER

変えてくること自体は悪いことではありません。アジャイルは変化を歓迎するもの。問題は「過去の発言が記録されていない」こと、または「変更の影響が見えない」ことです。

NG「先週はA、今週はB、来週はCですか？！」と苛立つ

→ 関係が悪化する
OK「前回はAでしたね。今回はBに変えるということで合ってますか？ Bにすると工数が+3日になりますが、進めて良いでしょうか？」

→ 事実を記録し、影響を見せて判断を委ねる

具体的な対処手順

  1. 全ての打ち合わせを議事録に残す（Slack・メールで全員に共有）
  2. 決定事項は「誰がいつ、何を決めたか」を明記
  3. 変更時は「変更要求票（CR）」として正式に扱う（Vol.7参照）
  4. 変更の工数・スケジュール影響を毎回見える化する
  5. 「変更はOK、ただしコストは別途」を最初に合意する
現場の声 —
「言った言わない」を防ぐには「言わせる」より「書いてもらう」。Slackで「決定事項：◯◯ ですよね？」と書いて、相手から「はい」のスタンプをもらう。これだけで揉め事が激減します。
Q.03

「適切に」「迅速に」「使いやすく」など曖昧な要求ばかりです

A. ANSWER

曖昧表現はそのまま実装に進むと必ず炎上します。「適切」とは何か、人によって違うからです。必ず数値に変換して合意します。これを「要求の操作可能化」と呼びます。

NG「とにかく速くしてほしい」「使いやすく」
「分かりやすく」「ユーザーフレンドリーに」

→ テスト時に「これでは遅い」と揉める
OK「検索結果はP95で800ms以内に表示」
「新規ユーザーがマニュアルなしで5分以内に注文完了」

→ 達成判定が明確

曖昧表現を数値化する質問テンプレート

  1. 「具体的に何秒以内なら満足ですか？」（速さ）
  2. 「何回クリックなら許容範囲ですか？」（使いやすさ）
  3. 「初めての人がどのくらいで使えればOKですか？」（学習性）
  4. 「満たせないと、どんな問題が起きますか？」（重要度確認）
  5. 合意した数値を受入条件として文書化する
裏技 —
「迅速」「適切」「容易」「効率的」など、曖昧語の禁止語リストをチームで作ります。これらを見つけたら必ず数値化、をルール化すると、自然と要求の質が上がります。
Q.04

営業部と経理部の要求が真っ向対立。どっちを取れば？

A. ANSWER

「どちらを取るか」ではなく「上位目標は何か」に立ち戻ります。対立する要求は、より上位のビジネスゴールから派生しているはず。共通の目的を見つけ、そこに照らして判断します。それでも合意できない場合は「決められる権限を持つ人」に判断を委ねます。

NG「営業部の方が声が大きいから営業を優先」

→ 経理部から後で激しい反対が来る
OK「両者の上位目標は『売上向上』ですね。営業案Aは売上+5%、経理案Bはコスト-3%。投資効果で比較しましょう」

→ 数値で公平に判断

対立時の対処フロー

  1. 両者の要求を文書化して見える化する
  2. それぞれが「なぜそれが必要か」を聞く（5why）
  3. 上位の共通目標を確認する
  4. 共通目標に照らして定量的な比較を行う
  5. それでも決まらないなら権限者（経営層）にエスカレーション
  6. 決定後は「決まった理由」を全員に説明する
POINT —
POの最も大切な仕事は「決めること」です。「両方の意見を反映」は、誰も望まないものを生みます。「決めない」も決断ですが、保留にも期限を設けます。
Q.05

要求が多すぎて整理しきれません

A. ANSWER

要求は「絞り込む」もので、「全部実装する」ものではありません。プロダクトの成功は、機能の数ではなく「価値を出せる機能だけ」に集中することで決まります。MoSCoW・WSJF・カノモデル（Vol.6）を駆使します。

NG「全部やりますので、優先順位は全部Aで」

→ 結果として全部中途半端
OK「v1ではMust 5機能だけ、Should 3機能はv2、Could 10機能はWon't（今回は捨てる）」

→ 必要なものに集中

絞り込みの実践手順

  1. 要求を全部リストアップ（漏れなく書き出す）
  2. カノモデルで「当たり前品質」を最優先（無いと致命傷）
  3. MoSCoWで4分類（Must / Should / Could / Won't）
  4. Mustが工数の60%を超えていないか確認
  5. Won't を明示することで「やらない」を合意する
  6. MustだけでMVPを構成、リリース後にフィードバック
裏技 —
「全部Must」と言われたら勇気テストを使います：「これがないとリリースする勇気はありますか？」「ある」ならShould以下。これだけで2/3はShould以下に降格できます。
Q.06

顧客が「全部Must」と譲りません

A. ANSWER

これはQ.05の派生問題ですが、対処法は別です。「全部Must」と言う顧客の真意は「優先順位を考えるのが面倒」or「やらないと言いたくない」のどちらか。「決める痛みを引き受ける」のがPO・コンサルタントの仕事です。

NG「では全部Mustということで進めます」と引き下がる

→ プロジェクトが破綻する
OK「予算と期日からは、Must10個は不可能です。1個諦めるとしたらどれですか？」
と数字で迫る

→ 嫌でも順位が出る

「全部Must」を崩す技法

  1. 制約条件を示す：「予算◯円、期日◯月では、できるのは5機能までです」
  2. 強制ランキングを使う：「1番大事なのはどれですか？2番は？」
  3. 100点配分法：「100点を配分してください」（全部100点はできない）
  4. 段階リリースを提案：「v1にこれ、v2にこれ、ではどうですか？」
  5. それでも決まらなければ権限者にエスカレーション
現場の声 —
「決める」が嫌な人ほど、「全部やって」と言います。これは判断を委ねているサイン。「では私が決めますね」と引き取って、後で報告するのも一つの手です。
Q.07

ユーザーインタビューしても、当たり障りのない回答ばかりです

A. ANSWER

「あなたの欲しい機能は？」と聞いても、人は「欲しいもの」を上手く言葉にできません。Henry Fordの「もっと速い馬が欲しい」の例の通り。アプローチを「困りごと」「行動観察」に変えます。

NG「どんな機能が欲しいですか？」
「使いやすいシステムって何ですか？」

→ 抽象的な回答しか来ない
OK「先週、業務で一番イライラしたのは何ですか？」
「いつも残業になる日は、何の作業が原因ですか？」

→ 具体的な困りごとが出る

具体的な対処手順

  1. 「欲しいもの」ではなく「困りごと」を聞く
  2. 最近の具体的な事例を引き出す（「先週のこと」など）
  3. 5whyで根本原因まで掘る（Vol.2参照）
  4. 言葉だけでなく現場を観察する（Vol.2参照）
  5. ヘルプデスクの問い合わせ履歴を入手して分析する
  6. 競合製品を使わせて「ここがダメ」を引き出す
TIP —
人は「未来の理想」より「過去の不満」の方が語りやすいもの。「最近、◯◯で困ったことありませんか？」はインタビューの最強の問いです。
Q.08

非機能要求、特に性能の数値をどう決めれば？

A. ANSWER

非機能要求の数値は「業務上の許容範囲」から逆算します。エンジニアの感覚で決めるものではありません。「許容できる遅延は何秒か」「許容できる停止時間は年何時間か」を業務側に問います。

NG「とりあえずレスポンス1秒、稼働率99.9%にしておく」

→ 過剰投資 or 不足のリスク
OK「1日に何件処理しますか？ ピーク時は？ 何秒以上待たされると業務に支障が出ますか？」と業務から逆算

→ 適切な水準が見える

非機能数値を決める質問

  1. 性能：「1日の処理件数は？」「ピーク時は何倍？」「ユーザーが許容する待ち時間は？」
  2. 可用性：「停止できない時間帯は？」「停止すると業務にどんな影響が？」
  3. セキュリティ：「扱うデータの重要度は？」「業界規制は？」（金融はPCI DSS等）
  4. 保守性：「想定運用期間は？」「将来どこを変えたい？」
  5. 業界・規制の業界標準値も参照する（ISO/IEC 25010等）
POINT —
非機能の数値は「コスト直結」です。99.9%と99.99%では、必要な冗長化コストが桁違い。「過剰品質はムダ、不足品質は事故」のバランス感覚が問われます。
Q.09

要求仕様書がすぐ古くなります。どう維持すれば？

A. ANSWER

「完璧な仕様書を作って凍結する」のは、もはや現実的ではありません。アジャイルでは仕様書を「単一の真実の場所（SSoT）」として、コード・チケット・ドキュメントの一貫性を保つ運用に切り替えます。

NG500ページのWord仕様書を作って凍結。
変更があっても更新されない

→ 半年で「実態と違う」状態
OKJira（要求）+ Confluence（補足）
+ コード内コメント（実装）が連携。
受入条件は自動テストとして残る

→ 常に「最新」

仕様書を腐らせない運用

  1. 真実の場所を1つに決める（Jira、Notion、Confluence等）
  2. 仕様書を「ストーリー単位」に細かく分割する
  3. 変更時は必ずSSoTを更新するルールにする（コードと同時にPR）
  4. 受入条件は自動テストとして実装する（BDD）
  5. 古い仕様書は削除またはアーカイブする
  6. 定期的に「実態と乖離していないか」レビューを行う
裏技 —
「Confluenceに書いた」「Notionに書いた」「コードコメントに書いた」が乱立すると同じ情報が複数箇所で食い違います。「これは最新版か？」と毎回疑う癖がついたら危険信号、SSoT運用の見直し時期です。
Q.10

要求工学の仕事が現場で評価されません。コードを書く方が偉い空気があります

A. ANSWER

残念ながら、これは多くの現場で見られる現象です。要求工学の成果は「目に見えにくい」「炎上を防いだ価値は数字に表れにくい」から。しかし、これを覆す3つの戦略があります。

NG「自分はコードを書かない、ただ会議をしているだけ」と卑下する

→ チーム内のポジションを失う
OK「先月、要求の見直しで◯◯のリワークを防ぎ、開発工数を10日削減できました」と数字で語る

→ 価値が見える化される

要求工学の価値を可視化する3戦略

  1. 防いだ手戻りを記録する：「この要求の見直しでXX工数を節約」と毎回ログに残す
  2. 少しでもコードを書く：プロトタイプ、テストコード、自動化スクリプト。手を動かす人と認識される
  3. 失敗事例を社内に共有する：「要求工学を怠ったらこうなった」を発信する（自分のチーム外含む）
  4. 機会があれば顧客との交渉で実力を見せる：価格交渉・契約調整など、エンジニアが苦手な領域で活躍する
  5. 長期的に成果が出る仕事だと割り切る：3年後に「あの時の判断が良かった」と気づかれる
最後に —
要求工学は「目立たないが効く」仕事です。良い要求エンジニアがいるプロジェクトは、静かに成功します。炎上しないこと自体が成果——これを胸に、誇りを持って続けてください。本シリーズの最後にお伝えしたかったのは、まさにこのことです。

★ SERIES — COMPLETE ★

9巻にわたるご精読、お疲れさまでした。

要求工学は、コードを書かない仕事です。打ち合わせと、文書と、ホワイトボードと、そして「言葉」を扱う仕事。地味で、目立たず、すぐには成果が見えません。

けれど、要求工学を真面目にやっているプロジェクトは、静かに成功します。「言われた通りに作ったのに使われない」という最も悲しい失敗を、防ぐことができます。

本シリーズで扱った技法——インタビュー、ユーザーストーリー、INVEST、MoSCoW、WSJF、ストーリーマッピング、SLO、トレーサビリティ——これらは、すべて「人と向き合うため」の道具です。技法を覚えるだけでは足りません。「相手の困りごとに、本気で向き合う姿勢」こそが、要求工学の本質です。

本シリーズが、あなたの現場で「正しいものを、正しく作る」ための一助となれば、これに勝る喜びはありません。次のプロジェクトで、ぜひ実践してみてください。
