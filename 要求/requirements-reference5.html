01

要求分析の役割

WHY ANALYSIS

獲得した要求の山から「本当に必要なもの」「優先すべきもの」「実は不要なもの」をふるい分けるのが分析。ここで重要なのは「視覚化する」こと。テキストの羅列で議論するより、図にした方が桁違いに早く合意に達します。

analysis_role「生の声」→「構造化された要求」
// 分析でやること
  • 分類機能・非機能・制約に分ける
  • 構造化関係性・階層を整理する
  • 優先化必要なもの・後回しを判別
  • 検証矛盾・欠落・重複を発見
  • 視覚化図にして共通認識を作る
  • 仮説化「これで本当に価値が出るか」を問う

★★★ 「分析しない要求」は、そのまま実装すると「言われた通りなのに、誰も喜ばない」システムになります。

# テキスト vs 視覚化の違い # [テキストで議論] 顧客「検索機能、フィルタも、ソート も、それから過去履歴もあって、 お気に入りも、レコメンドも...」 チーム「うーん、優先順位は？」 顧客「全部大事です！」 # → 1時間議論しても決まらない # [視覚化（ストーリーマップ）] チーム「では、左から右の流れで ユーザーがやることを並べましょう」 顧客「商品を探す → 比較する → 買う」 チーム「各ステップで何が必要？」 顧客「探すは検索 + フィルタ、 比較はお気に入り...」 チーム「これを上下で必須/任意に 分けましょう」 # → 30分で優先度が見える # 図にすると議論が前に進む # これが本巻のテーマ
POINT —
分析の道具は「描けるホワイトボード」または「Miro/Figjam」。アジャイル時代の分析は、テキスト文書ではなく付箋の二次元配置で行います。これが「ビジュアルファシリテーション」の本質。
02

As-Is / To-Be 分析

AS-IS / TO-BE

業務システムを作る際、最初にやるべきは「現状（As-Is）」と「あるべき姿（To-Be）」の対比です。これをやらずに「新システムを作ります」と進めると、現場の業務を破壊するシステムが出来上がります。ギャップこそが要求です。

as_is_to_be現状とあるべき姿のギャップ分析
// 業務改善の典型フロー

As-Is（現状）

・紙の伝票に手書き
・Excel転記に1時間
・転記ミスが月10件
・月末締めで残業

→ 現場で本当に起きていること

To-Be（あるべき姿）

・スマホで直接入力
・自動集計（リアルタイム）
・入力時バリデーション
・月末締めも30分

→ 新システム導入後の姿

★★★ ギャップ = 要求の源泉。「To-Be は理想だけど、現実離れしてない？」のチェックも分析の仕事。

# As-Is / To-Be の記述テンプレート @process 経費精算業務 As-Is (現状): step 1: 領収書を集める └ 人: 申請者 └ 時間: 1日5分 × 月20日 └ 問題: 紛失あり step 2: 紙申請書に転記 └ 人: 申請者 └ 時間: 月末2時間 └ 問題: ミスが月数件 step 3: 上長承認 (印鑑) └ 人: 上長 └ 時間: 出社待ち平均3日 └ 問題: リモート時止まる To-Be (改善後): step 1': 領収書をスマホ撮影 └ 即時保存・OCR読取り └ 時間: 30秒/件 step 2': 自動転記 └ システム自動 └ 時間: 0分 └ ミス: 0件 step 3': 電子承認 └ Slack通知 → ワンクリック └ 平均承認時間: 2時間 Gap: ここが要求の源泉
ハマりどころ —
「To-Beだけ書いてAs-Isを書かない」のは典型的な失敗。現状の業務を把握しないまま新システムを作ると、現場が混乱します。「現状の良さを残す」のも改善の一形態です。
03

ストーリーマッピング

STORY MAPPING

Jeff Patton考案のユーザーストーリーを2次元で配置する技法。横軸はユーザーの行動の流れ、縦軸は優先順位。「最初に何をリリースするか（MVP）」が一目で分かるようになります。アジャイル時代の定番中の定番。

story_mapping2次元でストーリーを並べる
// EC サイトのストーリーマップ例

商品を探す
詳細を見る

購入

キーワード検索

★★★ 上から順にリリースする＝「薄く全機能を作って、徐々に厚くする」。

# ストーリーマップの作り方 @workshop 半日 step 1: ユーザーの一連の行動を 左から右に並べる （ジャーニーマップ） 例: 探す → 見る → 買う → 受取 step 2: 各行動の下に 必要な機能を縦に列挙 探す: [検索, フィルタ, ソート, AI] step 3: 縦軸を優先順位で並べる 上に行くほど必須 step 4: 横線で「リリース」を切る Release 1: 最低限の体験 Release 2: 改善・追加 Release 3: 差別化要素 step 5: 各リリースが 「一通り価値を提供できる」 ことを確認 # Miro / FigJam / 模造紙でやる # テキストで作るものではない
裏技 —
ストーリーマップの「一番上のライン」を見て、それだけで顧客にとって価値ある体験になっているかチェック。なっていなければ、まだMVPとして成立していません。「動く骨格」を作る発想です。
04

インパクトマッピング

IMPACT MAPPING

Gojko Adzic考案の「目標から逆算して機能を導く」技法。WHY → WHO → HOW → WHATの4階層で、ビジネス目標から具体的な機能までを1枚の図にまとめます。「作る機能と目標の繋がり」が常に見えるのが最大の利点。

impact_mappingWHY → WHO → HOW → WHAT の階層
// インパクトマップの例
WHY (目標)
月間アクティブユーザー10%増 (3ヶ月で)
WHO (誰)
新規ユーザー
休眠ユーザー
既存ヘビーユーザー
HOW (どうなる)
登録までの離脱を減らす
再ログインのきっかけを作る
使用頻度を上げる
WHAT (機能)
SNS連携ログイン
復活クーポンメール
プッシュ通知
ゲーミフィケーション

★★★ 「この機能、目標とどう繋がる？」を常に問えるのが強み。繋がらない機能は作らない判断ができる。

# インパクトマッピングの問い WHY: なぜこれを作るのか？ → ビジネスゴール → 計測可能な数値目標 例: "3ヶ月で離脱率20%減" WHO: ゴール達成に影響する人は？ → ペルソナ・ステークホルダー 例: 新規 / 休眠 / 既存 HOW: 彼らの行動はどう変わるべき？ → 行動変容の仮説 例: "登録を完了する" "再アクセスする" WHAT: それを促す機能・支援は？ → 実装候補 例: SNSログイン 復活クーポン プッシュ通知 # 最大のメリット # 「目標との繋がりがない機能」 # を堂々と削除できる @anti_feature × 「便利そうだから作る」 × 「競合にあるから作る」 ○ 「この目標に繋がるから作る」
POINT —
「インパクトマップに書かれていない機能は作らない」を原則にすると、機能の増殖（Scope Creep）を防げます。新機能の提案があったら「これはどのWHYに繋がる？」と問うのが鉄則。

イベントストーミング

EVENT STORMING

Alberto Brandolini考案の「業務をイベントの連なりとして可視化する」技法。DDD（ドメイン駆動設計）との相性が抜群で、近年急速に普及しています。付箋の色で要素を区別し、業務全体の流れを浮かび上がらせます。

event_storming業務を「イベント」の連鎖で表す
// イベントストーミングの要素
  • ピンクドメインイベント（〜された）
  • 青コマンド（〜する命令）
  • 黄アクター（誰が）
  • 緑集約・データ（対象物）
  • 紫ポリシー（〜なら〜する）
  • 赤問題・課題（要議論）

注文業務の例：

注文する
→
注文された
→
決済処理
→
決済完了
注文
|
在庫
|
顧客

★★★ 「過去形のイベント」を時系列に並べると、業務の本質が浮かび上がります。

# イベントストーミングの進め方 @workshop 4時間〜半日 step 1: 大きな壁・模造紙を用意 （イベント数: 数十〜100以上） step 2: ドメインイベントを書き出す - 過去形で書く - 「注文された」「決済完了した」 - 順序は気にせず付箋に貼る step 3: 時系列に並べる - 左から右へ時間の流れ - 分岐があれば縦に展開 step 4: コマンド・アクターを追加 - 「誰が」「何をして」 - そのイベントが起こるのか step 5: 集約 (緑) を追加 - 状態を持つ対象物 - 注文・在庫・顧客 など step 6: ポリシー (紫) を追加 - 「Xが起きたら、Yを発動」 - 業務ルール step 7: 問題 (赤) を明示 - 議論が必要な点 - 後でフォローアップ # 出力 = ドメインモデルの素材 # DDD への入り口として最適
現場の声 —
イベントストーミングは「業務の人」「開発の人」が同じ言葉で話せるのが最大の価値。「画面では◯◯」「テーブルでは△△」と言っていた人たちが、「注文された」という共通のイベント名で話せるようになります。

コンテキスト図

CONTEXT DIAGRAM

システムを1つの箱として描き、その周りに外部システム・ユーザーを配置するだけの図。システムの境界（スコープ）を明確にする最重要ツール。「これは我々が作るもの／作らないもの」を一発で示せます。

★★★ コンテキスト図はプロジェクト最初の1枚。これが描けない＝スコープが固まっていません。

# コンテキスト図の役割 明確になること: ✓ 何を作るか（システム本体） ✓ 何を作らないか （外部の既存システム） ✓ 連携相手 （API・データ連携先） ✓ ユーザー種別 （ロール・アクター） よくある失敗: ✗ 内部構造を描いてしまう → コンテキスト図ではない ✗ 外部が多すぎる → スコープが広すぎる兆候 ✗ 矢印に意味が書いてない → どんなデータが流れる？ # シンプルが正義 # A4 1枚に収まる粒度 @notation - 中央の四角 = システム本体 - 周囲の四角 = 外部 - 人型: ユーザー（アクター） - 箱: 外部システム - 矢印 = データ・要求の流れ - ラベルで内容を明示 - 例: "注文情報", "在庫照会" # DFD のレベル0 とも呼ばれる
TIP —
「このAPIは我々が作るんですか？」「この機能は既存システムを使うんですか？」の議論はコンテキスト図1枚で一掃できます。要件定義フェーズ最初に作って全関係者の合意を取るべき図です。

ドメインモデル

DOMAIN MODEL

業務の主要な「もの」と「関係」を表したのがドメインモデル。クラス図に似ていますが、業務理解のための図であり、実装そのままではありません。「ユビキタス言語」を作る道具でもあります。

domain_model業務の「もの」と「関係」を表す
// EC サイトのドメインモデル例
┌─────────┐ ┌─────────┐ │ 顧客 │ 1 * │ 注文 │ │─────────│─────────│─────────│ │ ID │ │ 注文番号 │ │ 氏名 │ →持つ│ 注文日 │ │ 住所 │ │ 合計金額 │ └─────────┘ └─────────┘ │ 1 │ │ * ┌─────────┐ ┌─────────┐ │ 注文明細 │* 1│ 商品 │ │─────────│───│─────────│ │ 数量 │ │ 商品名 │ │ 単価 │ │ 価格 │ └─────────┘ └─────────┘

★★★ 用語の統一に貢献。
「お客さま」「ユーザー」「会員」が混在していた現場が、「顧客」で揃うようになります。

# ドメインモデルの作法 業務側の用語で書く: ✓ "顧客" "注文" "商品" ✗ "User" "Order" "Product" （業務の人と話せない） 実装は後で考える: ✓ "会員ランク": 業務概念 ✗ "member_rank_id": 実装 ✗ "members_table": DB 関係を線で結ぶ: - 1対多 / 多対多 を明示 - 必須 / 任意 を多重度で 業務ルール（不変条件）も併記: - "未成年は購入できない" - "在庫切れ商品は注文不可" # ユビキタス言語との関係 # （DDDの用語） # この図に出てくる名詞・動詞が # 開発・業務全員の「共通語」になる @goal PO「注文が完了した状態とは？」 開発「注文された&決済完了です」 業務「私たちはそれを"確定注文" と呼んでいます」 全員「では "確定注文" に統一」 # これがユビキタス言語の誕生
ハマりどころ —
「ドメインモデル＝DB設計」と勘違いしがち。違います。ドメインモデルは業務の構造であり、DB設計は実装の都合。両者は似ていますが、混同すると業務概念が歪みます。UML編Vol.2のクラス図と併せて読むと理解が深まります。

MECEで構造化

MECE STRUCTURING

要求を整理するときの黄金ルールがMECE（ミーシー）：Mutually Exclusive, Collectively Exhaustive。「重複なく、漏れなく」。コンサルティング由来の概念ですが、要求の構造化に非常に有用。これを意識すると、要求リストが綺麗に整います。

mece_structuring重複なく、漏れなく整理する
// MECE な分類 vs ダメな分類

ダメな分類

・主要機能
・サブ機能
・便利機能
・新機能

→ 「便利」と「新」は重複し得る

MECEな分類

・必須機能 (Must)
・重要機能 (Should)
・あったら機能 (Could)
・今回除外 (Won't)

→ MoSCoW法。1つに割り当たる

★★★ 「両端」「中間」が抜けてないかを確認するのがコツ。
MoSCoWは次巻Vol.6で詳述。

# MECEなチェック観点 @check_mece def validate(categories): # [1] 漏れがないか (Exhaustive) # すべての要求がどこかに入るか for req in all_requirements: assert req in some_category # [2] 重複がないか (Exclusive) # 1つの要求は1カテゴリだけ for req in all_requirements: matched = [ c for c in categories if req in c ] assert len(matched) == 1 # 要求の典型的なMECE分類 分類軸A: 種別 - 機能要求 / 非機能要求 / 制約 分類軸B: 優先度 - Must / Should / Could / Won't 分類軸C: 確実性 - 確定 / 検討中 / 仮説 分類軸D: 関係者 - エンドユーザー / 管理者 / 運用 # 軸を複数持つと多面的に見える
POINT —
MECEは万能ではありません。「ざっくり80%MECE」でも実用上十分です。完璧主義は時間の浪費。「重複・漏れを発見したら、その都度修正する」くらいの運用が現実的。
NEXT STEP —
分析した要求は、最終的に「いつ、誰が、何を作るか」に落とし込まれます。Vol.6 プロダクトバックログ・優先順位編では、MoSCoW法・WSJF・カノモデルなどの優先順位付け技法と、プランニングポーカーによる見積もり、トレーサビリティの確保について扱います。
