非機能要求編

Non-Functional Requirements — Quality Attributes

「速い」「落ちない」「使いやすい」「守られている」——これらは機能では表せない品質の要求です。本巻ではISO/IEC 25010の品質特性を軸に、性能・可用性・セキュリティ・保守性・使用性を体系的に整理。「曖昧な品質要求を数値に変換する」技術が身につきます。

01

非機能要求とは

WHAT IS NFR

機能要求が「何ができるか」を表すのに対し、非機能要求は「どのように動くべきか」を表します。性能、可用性、セキュリティ、使いやすさ——これらが満たされないと、機能が正しくてもシステムは使い物になりません。非機能の見落としは、リリース後の大事故の主因です。

functional_vs_nfr「何を」と「どう」の違い
// 同じ機能でも、NFRで価値が変わる

機能要求

「商品を検索できる」

→ 動けばOK

非機能要求

「検索結果は1秒以内に表示」
「同時1000人が検索可能」
「24時間365日稼働」

→ ここが本番の価値

★★★ 「動くけど遅い」「動くけど落ちる」はユーザーから見れば「動かない」と同じ。これがNFR軽視の代償です。

# NFR の典型カテゴリ class NFRCategory: PERFORMANCE # 性能・速度 AVAILABILITY # 可用性・稼働率 RELIABILITY # 信頼性・MTBF SECURITY # セキュリティ USABILITY # 使いやすさ MAINTAINABILITY # 保守性 PORTABILITY # 移植性 COMPATIBILITY # 互換性 # 他にも SCALABILITY # 拡張性 OBSERVABILITY # 可観測性 ACCESSIBILITY # アクセシビリティ # これらが揃って初めて # 「使える」システムになる # NFR ≠ オプション # NFR = システムの基礎
POINT —
非機能要求は「あってあたりまえ」と思われがち。だからこそ明示的に書いて合意する必要があります。「速くしてほしい」と言われても「何秒？」まで詰めなければ何もできません。
02

ISO/IEC 25010

QUALITY MODEL

非機能要求の国際標準がISO/IEC 25010（旧9126）。8つの品質特性に整理され、それぞれにさらに副特性が定義されています。「品質の地図」として、要求の抜け漏れチェックに使えます。

ISO_IEC_250108つの品質特性
// 8品質特性
機能適合性

Functional Suitability — 正しく機能するか

性能効率性

Performance Efficiency — 速さ・資源効率

互換性

Compatibility — 他システムと共存

使用性

Usability — 学びやすい・使いやすい

信頼性

Reliability — 安定稼働

セキュリティ

Security — 安全性・データ保護

保守性

Maintainability — 直しやすさ

移植性

Portability — 環境変更への対応

★★ 要求定義のチェックリストとして使うのが実用的。「8つ全部に何か書けるか？」と自問すると、抜けが見えます。

# 25010 を要求定義書のテンプレに @nfr_template sections = { "1. 機能適合性": { "正確性": "...required...", "完全性": "...", }, "2. 性能効率性": { "応答時間": "95%tile < 1s", "スループット": "1000 req/s", "資源使用率": "CPU < 70%", }, "5. 信頼性": { "稼働率": "99.9%", "MTBF": "720時間以上", "復旧時間": "RTO 1時間", }, "6. セキュリティ": { "認証": "MFA必須", "暗号化": "TLS1.3以上", "監査": "全操作ログ保持", }, # … 8カテゴリ全部 } # 全項目に「数値 or N/A」を書く # 空欄を作らないのが鉄則
TIP —
25010の「該当なし」を明示するのも立派な要求定義です。「移植性: 単一クラウド前提のため、移植性は要求しない」と書くことで、議論せずに済みます。
03

性能要求

PERFORMANCE

非機能要求の中で最も「数値化しやすく、しなければいけない」のが性能要求。応答時間、スループット、同時接続数をパーセンタイル付きで指定するのが現代の標準。「平均」だけでは尻尾の遅延が見えません。

performance_metrics応答時間・スループット・同時接続
// 主要な性能指標
  • 応答時間1リクエストの所要時間（Latency）
  • スループット単位時間の処理量（req/s, TPS）
  • 同時接続数同時に処理できるユーザー数
  • 資源使用率CPU/メモリ/帯域の消費
  • 起動時間システム立ち上がりの時間

★★★ 平均値ではなくパーセンタイルで語る：

パーセンタイル意味
P50中央値（半分のユーザー）
P9595%のユーザーが体感する上限
P9999%のユーザー（重要）
P99.91000人に1人の最悪値
# 性能要求の書き方 # [悪い例] ✗ "サクサク動くこと" ✗ "1秒以内に応答" # どのリクエスト？平均？ # [良い例] ✓ @PERF-001 検索APIのレスポンスタイムは 通常負荷時（500 req/s）において、 P95 ≤ 800ms P99 ≤ 1500ms ✓ @PERF-002 ピーク時（2000 req/s）でも タイムアウト率 ≤ 0.1% ✓ @PERF-003 バッチ処理は1日100万件を 2時間以内に完了 # 必ず数値+条件で書く # テスト可能性が命
ハマりどころ —
「平均1秒」を達成しても、P99が10秒だと100人に1人が激怒します。「ユーザー体感は最悪値で決まる」のが性能の鉄則。必ずパーセンタイルで合意します。
04

可用性と信頼性

AVAILABILITY / RELIABILITY

「落ちないこと」を表す要求が可用性。SLO/SLA/SLIの3点セットで定義するのが現代流。「99.9%」が何分の停止か、即答できないとSREの現場では話になりません。

SLO_SLA_SLI目標 / 契約 / 計測の3点セット
// SLI / SLO / SLA
  • SLIService Level Indicator — 実際に計測する指標
    例: 成功リクエストの割合
  • SLOService Level Objective — 内部目標値
    例: 99.9%以上を維持する
  • SLAService Level Agreement — 顧客との契約
    例: 99.5%下回ったら返金

SLO > SLAが鉄則。契約より厳しい内部目標を持つことで、SLA違反を予防します。

★★★ 99.9%という数値の重み：

稼働率年間停止許容
99.0%3.65日
99.9%8.76時間
99.99%52.6分
99.999%5.26分
# 可用性要求の例 @AVAIL-001 全体稼働率 SLO: 月間 99.9% 以上 SLA: 月間 99.5% 下回ったら 利用料を10%返金 計測除外: ・事前告知メンテ ・利用者の操作起因 @AVAIL-002 復旧時間 RTO: 1時間以内 (Recovery Time Objective) サービス停止からの復旧時間 RPO: 5分以内 (Recovery Point Objective) 許容されるデータ損失時間 @AVAIL-003 エラーバジェット SLO 99.9% = 月43.2分の停止許容 この時間を「使い切る」前提で リリース速度を調整 # 「100%」は不可能 # どのくらい止まっていいか # を決めるのが要求定義
現場の声 —
「99.999% (ファイブナイン)」が必要な業務は金融取引・通信網・医療機器などごく一部です。Webサービスでは99.9%で十分なことが多い。過剰なSLOは過剰な投資を呼ぶことを覚えておきます。
05

セキュリティ要求

SECURITY

セキュリティ要求は「CIA」の3要素で整理します。機密性（Confidentiality）、完全性（Integrity）、可用性（Availability）。それぞれに具体的な対策要求を紐づけます。「セキュアであること」では仕様化できません。

CIA_triad機密性・完全性・可用性
// CIA 3要素
  • C — 機密性権限のない人に見られない
    対策: 認証・認可・暗号化
  • I — 完全性データが改ざんされない
    対策: 署名・ハッシュ・監査ログ
  • A — 可用性必要な時に使える
    対策: DDoS対策・冗長化・WAF

★★ 業界別に追加要求：

  • 金融PCI DSS, FISC
  • 医療HIPAA, 医療情報安全管理ガイドライン
  • 公共政府情報システムのセキュリティ要件
  • EU圏GDPR（個人情報保護）
# セキュリティ要求の例 @SEC-AUTH-001 認証 ・パスワード 12文字以上 ・MFA必須（管理者は必ず） ・5回失敗で15分ロック ・90日でパスワード変更 # NIST最新では非推奨だが # 監査要件で残る場合も @SEC-AUTHZ-001 認可 ・RBAC（役割ベース） ・最小権限の原則 ・特権操作は二人承認 @SEC-CRYPTO-001 暗号化 ・通信: TLS 1.3以上 ・保管: AES-256 ・鍵管理: KMS / HSM @SEC-LOG-001 監査 ・全認証イベント記録 ・全特権操作記録 ・保管期間 1年 ・改ざん検出機構あり @SEC-PRIV-001 個人情報 ・GDPR準拠 ・忘れられる権利の実装 ・データポータビリティ # 「セキュアに」では実装できない
POINT —
セキュリティ要求は必ず専門家のレビューを受けます。社内のセキュリティチーム、外部監査、ペネトレーションテスター。「自分たちで考えただけ」のセキュリティ要求は危険信号です。
06

使用性 (ユーザビリティ)

USABILITY

「使いやすい」も要求として書けます。ニールセンの5つの品質要素やSUS (System Usability Scale)といった定量化技法が確立しているからです。「直感的」「シンプル」では仕様化不可。

usability_metrics「使いやすさ」も定量化できる
// ニールセンの5要素
  • 学習しやすさ初めての人がすぐ使える
  • 効率慣れた人が素早く使える
  • 記憶しやすさ久しぶりでも思い出せる
  • エラー誤操作が少なく、復帰しやすい
  • 満足度使っていて気持ちが良い

★★ SUS (System Usability Scale) ：10問の質問に5段階で答え、0〜100点でユーザビリティを数値化。68点が業界平均、80点以上で「使いやすい」と評価される。

アクセシビリティも使用性に含まれます。WCAG 2.1の達成レベル（A / AA / AAA）で要求します。

# 使用性要求の例 @USAB-001 学習しやすさ 新規ユーザーが、マニュアル なしで主要機能を5分以内に 完了できること （5名中4名以上が達成） @USAB-002 効率 定常作業（注文登録）を 3クリック以内に完了 @USAB-003 エラー回復 全ての破壊的操作は ・確認ダイアログ表示 ・取り消し可能 （最低10分間Undo可） @USAB-004 満足度 ユーザビリティテスト SUSスコア 75点以上 @A11Y-001 アクセシビリティ WCAG 2.1 レベルAA準拠 ・スクリーンリーダー対応 ・コントラスト比 4.5:1以上 ・キーボード操作のみで全機能 # 「直感的」は要求にならない # 計測可能な指標に変換する
TIP —
使いやすさは「ユーザビリティテスト」で確かめるのが最も確実。5人にやってもらえば、深刻な問題の85%は発見できる、というのが定説です（Nielsen）。
07

保守性と移植性

MAINTAINABILITY / PORTABILITY

「直しやすさ」「移しやすさ」も非機能要求です。これらは運用フェーズに入ってから真価が問われるもの。リリースまで気づかれず、5年後の改修コストを大きく左右します。技術的負債の元凶もここ。

maintainability_portability長期運用を見据えた要求
// 保守性の副特性
  • モジュール性変更の影響範囲が限定される
  • 再利用性共通部品として使い回せる
  • 解析性原因調査がしやすい（ログ・監視）
  • 修正性変更が容易
  • 試験性テストが書きやすい

移植性の副特性:

  • 適応性環境変化への対応
  • 設置性インストール・移設のしやすさ
  • 置換性他システムとの置き換え可能性

★★ これらは「品質ゲート」として開発プロセスに組み込みます。

# 保守性・移植性の要求例 @MAINT-001 コード品質 ・複雑度 (CC) 10以下 ・コードカバレッジ 80%以上 ・SonarQube グレード B以上 # CI で自動検査 @MAINT-002 ドキュメント ・全API に OpenAPI仕様書 ・READMEに開発手順 ・ADR (Architecture Decision Record) を残す @MAINT-003 観測性 ・構造化ログ (JSON) ・分散トレーシング対応 ・メトリクス公開 # Prometheus形式など @PORT-001 環境非依存 ・コンテナ化 (Docker) ・12-Factor App準拠 ・環境変数で設定を分離 @PORT-002 クラウド非依存 ・特定クラウド固有のサービス 使用は最小限 ・必要時に他クラウドへ移行可能 # 5年後の保守性を、今日決める
現場の声 —
「最初は速く作りたい、保守性は後で」——これが技術的負債のスタートライン。返済しない借金は雪だるま式に膨らみます。NFR-保守性を要求段階で明文化することで、開発時に手抜きしにくくなります。
08

アジャイルでのNFR

NFR IN AGILE

「アジャイルではNFRは扱いにくい」とよく言われます。確かに「速くしたい」はストーリーになりにくい。アジャイルでは3つの戦略でNFRを扱います：Definition of Done、NFRバックログ、制約として横断適用。

NFR_in_agileDoD / NFRバックログ / 制約
// アジャイルでの3つの戦略
  • DoDDefinition of Done に組み込む
    毎ストーリーで満たすべき品質基準
  • NFRバックログ専用のNFRストーリーを作る
    「P95レイテンシをXXに改善」など
  • 制約全ストーリー共通の前提
    「セキュリティ基準は常に満たす」

★★★ 最も実用的なのはDoDへの組み込み。「リリース可能」の基準にNFR項目を含めれば、毎回チェックされます。

# DoD への NFR 組み込み例 @definition_of_done checklist = [ # 機能 "受入条件を全て満たす", # テスト "単体テスト カバレッジ ≥ 80%", "E2Eテスト パス", # 性能 "性能劣化が無いことを確認", "P95 レイテンシ < 1s", # セキュリティ "OWASP Top10対応", "依存パッケージ脆弱性ゼロ", # 観測性 "主要操作のログ出力", "エラー監視のアラート設定", # ドキュメント "README更新", "API仕様書更新", # レビュー "コードレビュー完了", "PO の受入確認", ] # 全項目クリアで「Done」
POINT —
NFRは「全員のもの」。機能ストーリーに集中すると、いつのまにかNFRが置き去りになります。DoDに組み込み、毎スプリント自動的にチェックされる仕組みが、アジャイルでNFRを守る最良の方法です。
