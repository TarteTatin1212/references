01

ユーザーストーリーとは

WHAT IS A USER STORY

ユーザーストーリーは「ユーザー視点で書かれた機能の小さな約束」です。詳細な仕様書ではなく、会話のきっかけ。書いた瞬間に完成するものではなく、開発者・PO・ユーザーが議論するための共通言語です。「カード・会話・確認」の3Cが本質。

3C_of_user_storiesCard / Conversation / Confirmation
// ユーザーストーリーの3C
  • Cardカード（短い文章で書かれた約束）
    付箋やチケット1枚分のサイズ感
  • Conversation会話（詳細は対話で決める）
    カードは話のきっかけにすぎない
  • Confirmation確認（受入条件で完了判定）
    「何ができればOKか」を明確化

★★★ 「書いた瞬間に完成」ではないのがストーリー。「議論の出発点」と心得ます。

# 仕様書 vs ユーザーストーリー # [従来の仕様書スタイル] 3.2.1 検索機能 入力欄に文字列を入力し、 検索ボタンを押下することにより、 該当する商品が一覧表示される。 表示形式は別紙参照。 ... # 100ページに及ぶ仕様書 # [ユーザーストーリー] As a 買い物客 として I want 商品をキーワードで検索したい So that 欲しい商品を素早く見つけられる # たった3行 # 詳細は実装直前に対話で決める # 必要十分な情報量
POINT —
ユーザーストーリーは「詳細仕様書の代わり」ではなく「会話を促す仕組み」です。「書きさえすれば伝わる」と思った瞬間に失敗します。動くものを見せて、対話で詳細化するのがアジャイル流。
02

基本構文

STORY TEMPLATE

最も普及している構文がConnextra形式の「As a 〜, I want 〜, so that 〜」。日本語では「〜として / 〜したい / 〜のために」。3要素はユーザー・機能・価値。特に「so that（〜のため）」が抜けがちですが、ここが最も重要です。

connextra_format「として／したい／のため」の3要素
// ストーリーの構造
As a [ユーザー役割]
I want [機能・行動]
so that [理由・価値]

それぞれの要素が答える問い：

  • As a誰のための機能か？（WHO）
  • I want何ができるべきか？（WHAT）
  • So thatなぜそれが必要か？（WHY）

★★★ 「so that」を書いてから「I want」を書くと、本質的に必要な機能が見えてきます。

# 良いストーリーの例 @story US-101 As a 出張の多い営業担当 として I want 領収書をスマホで撮影して 経費精算に登録したい So that 月末に紙の領収書を 大量に処理する手間を減らせる @story US-102 As a 経理担当 として I want 月末に未承認の経費を 一覧で確認したい So that 締め日までに承認漏れがないか 把握できる # 同じ機能でも # 役割が違えば別ストーリー # これが「ユーザー視点」の真意
ハマりどころ —
「As a user」と書いてしまうのが典型的なアンチパターン。「user」とは誰なのか？ 営業？経理？管理者？役割を具体化しないと、その機能が誰のためのものか分からなくなります。
03

INVEST原則

INVEST PRINCIPLES

良いユーザーストーリーが満たすべき6つの性質の頭文字をつなげたのがINVEST。Bill Wakeが提唱した、ストーリーの品質チェックリストです。書いたストーリーがこの6つを満たしているか、毎回確認します。

INVEST_criteria良いストーリーが備える6つの性質
// INVEST の各文字の意味
I
INDEPENDENT
独立している
N
NEGOTIABLE
交渉可能
V
VALUABLE
価値がある
E
ESTIMABLE
見積可能
S
SMALL
小さい
T
TESTABLE
テスト可能
  • I他のストーリーに依存しない
  • N詳細は対話で決められる余地がある
  • Vユーザー・ビジネスに価値がある
  • E大きさを見積もれる程度に明確
  • S1スプリント内で完了する大きさ
  • T完成判定の基準が明確
# INVEST チェック例 @story "商品検索機能を実装する" # [I] 独立？ ✓ ログイン機能と独立に動く # [N] 交渉可能？ ✓ 検索結果の表示形式は議論余地 # [V] 価値がある？ ✓ ユーザーが商品を素早く見つけられる # [E] 見積可能？ ✗ 範囲が広すぎる → "キーワード検索のみ"に絞る # [S] 小さい？ ✗ フィルタ・ソート全部入りは大きい → 別ストーリーに分割 # [T] テスト可能？ ✓ 受入条件を書ける # この場合、分割が必要 # 1スプリントに収まらないなら大きすぎ
POINT —
INVESTで「S」と「I」が特に守られにくい。「全部繋がってる」「一通り作りたい」と言い出すと、すぐ大きくなります。「価値を保ったまま、いかに小さく切るか」が腕の見せどころ。
04

受入条件 (AC)

ACCEPTANCE CRITERIA

ストーリーの「完成判定」を明確にするのが受入条件（Acceptance Criteria）。書き方の標準が「Given / When / Then」形式。テスト可能であることが絶対条件で、後の自動テスト（BDD）にもそのまま使えます。

given_when_then前提・行動・結果の3点セット
// Given / When / Then 構文
Given前提条件（〜の状態で）
When行動（〜したとき）
Then結果（〜となる）

具体例（ログイン機能）:

Given登録済みユーザーがログイン画面を開いている
When正しいメールアドレスとパスワードを入力して「ログイン」を押す
Thenダッシュボード画面に遷移する

★★★ 1つのストーリーに3〜5個の受入条件が目安。多すぎたらストーリー分割の合図。

# 受入条件の実例（複数パターン） @story US-201 ログイン機能 @AC1 正常系 Given: 有効なユーザー登録済み When : 正しい認証情報で送信 Then : ダッシュボードへ遷移 @AC2 パスワード誤り Given: ログイン画面表示中 When : 誤ったパスワードで送信 Then : "認証情報が違います" 表示 And ログイン画面に留まる @AC3 連続失敗時のロック Given: 同じユーザーが5回連続失敗 When : 6回目を試行 Then : "アカウントがロックされました" And 管理者にメール通知 # そのまま自動テストにできる # これがBDDの強み
TIP —
受入条件は「実装方法」を書かないのがコツ。「赤色のボタンを押す」ではなく「ログインを実行する」。WHAT を書く、HOW は書かない。これだけでストーリーがブレなくなります。
05

ストーリーの分割

STORY SPLITTING

大きすぎるストーリーは分割します。分割の鉄則は「価値を保ったまま小さくする」。技術階層で分けるのは絶対NG。SPIDRという覚え方が便利です。Spike / Path / Interface / Data / Rules の5パターン。

SPIDR_split5つの分割パターン
// SPIDR の各文字
  • SSpike — 不確実性を切り出す（調査ストーリー）
  • PPath — 処理経路で分ける（正常系 / 異常系）
  • IInterface — UIで分ける（Web / モバイル）
  • DData — データの種類で分ける（クレカ / 銀振 / コンビニ）
  • RRules — ルールで分ける（無料会員 / 有料会員）

★★★ NG分割の典型：
・「フロントエンド」「バックエンド」で分ける
・「DB設計」「画面実装」「テスト」で分ける
→ 個別では誰の役にも立たない

# 分割の良い例 vs 悪い例 # [元のストーリー] As a 買い物客 I want 商品を購入したい So that 欲しい物が手に入る # ============================ # 悪い分割（技術階層） # ============================ ✗ US-1: DB設計 ✗ US-2: API実装 ✗ US-3: 画面実装 ✗ US-4: テスト # → 1個ずつでは価値ゼロ # ============================ # 良い分割（Data で分割） # ============================ ✓ US-1: クレジットカードで購入 ✓ US-2: コンビニ後払いで購入 ✓ US-3: 銀行振込で購入 # → 各個別で価値が完結 # → リリース順に優先付けできる # → 1つずつデプロイ可能
裏技 —
「分割できない、これ以上分けたら意味がなくなる」と感じたら「ハッピーパスだけ」から始めるのが定石。エラー処理や例外パターンは別ストーリーに切り出す——これだけで多くの場合分割できます。
06

エピックとテーマ

EPIC / THEME

ストーリーは小さいので、「もっと大きな塊」でまとめたいことがあります。それがエピック（大きなストーリー）やテーマ（複数エピックの集まり）。階層構造で整理することで全体感を失わずに済みます。

story_hierarchyテーマ → エピック → ストーリー → タスク
// 階層と粒度の目安
  • テーマ大目標。1年単位
    例：「顧客体験の向上」
  • エピック大ストーリー。複数スプリント
    例：「決済機能の追加」
  • ストーリー1スプリント内
    例：「クレカ払いで購入できる」
  • タスク1日以内
    例：「決済APIの呼び出し実装」

★★ エピックはそのままでは着手不能。実装直前にストーリーに分割します（JIT詳細化）。

# 階層構造の例（ECサイト） @theme 顧客体験の向上 [2025年] │ ├─ @epic 決済機能の刷新 │ ├─ @story クレカで購入 │ ├─ @story コンビニ払いで購入 │ ├─ @story 銀振で購入 │ └─ @story 領収書発行 │ ├─ @epic 検索機能の改善 │ ├─ @story キーワード検索 │ ├─ @story カテゴリ絞り込み │ └─ @story 価格でソート │ └─ @epic モバイル対応 ├─ @story レスポンシブ化 └─ @story プッシュ通知 # 上から下へ詳細化されていく
POINT —
テーマ → エピック → ストーリー → タスク、と「ロケットの段階」のように切り替わります。JIRA・Azure DevOpsなどのツールは、この階層を最初からサポートしています。階層なしで全部「ストーリー」だと、すぐ収拾がつかなくなります。
07

ユースケース記述との違い

VS USE CASE

「ユーザーストーリー」と「ユースケース」は混同されやすいですが、目的も粒度も違います。ストーリーは会話のきっかけ、ユースケースは詳細なシナリオ。どちらが優れているではなく使い分けです。

story_vs_usecase「軽量」vs「詳細」の使い分け
// 比較表

ユーザーストーリー

粒度: 短い（1〜3行）
目的: 会話のきっかけ
記法: As a / I want / So that
適合: アジャイル
詳細は対話・受入条件で補完

ユースケース

粒度: 長い（数ページ）
目的: 詳細シナリオの記述
記法: 基本フロー＋代替フロー
適合: ウォーターフォール／規制業界
画面遷移・例外も明記

★★ 両方使う現場もあります。アジャイルでも「重要機能はユースケースで補強」など。排他的ではないのです。

# 同じ機能を両方で書いてみる # ============================ # ユーザーストーリー版 # ============================ As a 会員 I want パスワードを再設定したい So that 忘れても再ログインできる # ============================ # ユースケース版 # ============================ UC-001 パスワード再設定 アクター: 会員 事前条件: 登録済みメール有効 基本フロー: 1. 会員が「PW忘れ」をクリック 2. システムがメール入力画面を表示 3. 会員がメール入力 4. システムがリセットURLをメール送信 5. 会員がURLクリック 6. 新PW入力 → 完了 代替フロー: 3a. メール未登録 → エラー表示 5a. URL期限切れ → 再発行案内 # 規制業界・大規模システムでは # この詳細さが必要な場合がある
現場の声 —
金融・医療・公共系では「ユースケース必須」のところも多い。一方Web系スタートアップは「ストーリーのみ」が主流。業界文化を見極めて選ぶのが正解です。
08

アンチパターン

ANTI-PATTERNS

「ストーリーは書いてみたけど、なんだか機能しない」——よくある現象です。背景には典型的なアンチパターンがあります。書き方が形式的になりすぎたり、開発者目線で書かれたり、価値が抜けたり。実例で確認しましょう。

story_anti_patternsNG例とOK例の対比
// NG → OK のリファクタ例

NG: 開発者目線

「As a 開発者、I want DBにインデックスを張りたい、so that 検索が速くなる」

→ ユーザー視点になっていない

OK: ユーザー視点

「As a 利用者、I want 検索結果を1秒以内に見たい、so that 待たずに次に進める」

→ 価値が明確

NG: so that が空

「As a 管理者、I want 設定画面を作る」

→ 何のため？が抜けている

OK: 価値が明確

「As a 管理者、I want メール通知の頻度を変えたい、so that 業務量に合わせて運用できる」

→ WHY が明示

# よくあるアンチパターン早見表 @anti_pattern 「全部As a user」 問題: ユーザー像が曖昧で 仕様判断ができない 対策: 具体的な役割名にする @anti_pattern 「so that を省略」 問題: なぜ作るか不明 対策: 価値を書くまで完成としない @anti_pattern 「仕様書化」 問題: 1ストーリーが10ページに 対策: 分割 + 受入条件で対応 @anti_pattern 「受入条件なし」 問題: 完成判定ができない 対策: 必ず Given/When/Then で書く @anti_pattern 「技術タスク」 問題: 「リファクタする」 「DBを設計する」 対策: 価値が説明できないなら バックログから外す （別管理にする） @anti_pattern 「永遠の Epic」 問題: 何年も着手されない 対策: 期限とゴールを切る 不要なら諦めて削除
ハマりどころ —
「リファクタリング」「DB設計」「テスト整備」などの技術タスクは、ユーザーストーリーには向きません。別の「技術的負債バックログ」として管理するのが現実的です。無理にストーリーに押し込むと、価値が説明できず優先度がブレます。
