01

プロダクトバックログ

PRODUCT BACKLOG

プロダクトバックログは「やるべきこと全部、優先順位付きリスト」。Scrumの中心要素であり、要求工学の成果物そのもの。DEEP原則（Detailed appropriately / Estimated / Emergent / Prioritized）に従う「生きているリスト」です。

product_backlog優先順位付きの「やることリスト」
// DEEP 原則
  • DDetailed appropriately — 上位ほど詳細
  • EEstimated — 見積もり済み
  • EEmergent — 常に変化し続ける
  • PPrioritized — 優先順位が付いている

★★★ 「上位は詳細、下位は概略」が肝。全部を詳細化しない。下位は変わる前提です。

バックログの様子:

P13ptログイン機能 ✓詳細・受入条件あり
P25pt商品検索 ✓詳細・受入条件あり
P38ptカート機能 △一部詳細
P4?レコメンド機能（概略のみ）
P5?クーポン（アイデア段階）
# バックログ管理の責任分担 class ProductOwner: # PO の責任 def manages_backlog(self): return [ "優先順位の決定", "何を作る/作らないか", "ストーリーの追加・削除", "ステークホルダーとの調整", ] class DevelopmentTeam: # 開発チームの責任 def supports_backlog(self): return [ "ストーリーの見積もり", "技術的な詳細化", "実現性のアドバイス", ] # PO は「何を作るか」 # 開発は「どう作るか」 # 領域を侵さないのが平和の鍵 # ツール: Jira, Azure DevOps, # Asana, Linear など
POINT —
「誰でも編集できるバックログ」は失敗のもと。POだけが優先順位を変更できるのがScrumの鉄則です。みんなで触ると、いつの間にか「全部1位」になります。
02

リファインメント

BACKLOG REFINEMENT

バックログを「実装可能な状態」に磨き続ける活動がリファインメント（旧称: Grooming）。スプリント時間の約10%を使い、上位ストーリーの詳細化・分割・見積もりを行います。これを怠ると「実装直前に詳細不明で着手不能」になります。

backlog_refinement「Ready」な状態を準備する
// Definition of Ready (DoR)

ストーリーが「着手可能」と見なされる条件：

  • ✓受入条件が書かれている
  • ✓見積もりが済んでいる
  • ✓依存が解消されている
  • ✓UI/UXの方針が決まっている
  • ✓必要なテストデータが揃う見込み
  • ✓チーム全員が内容を理解している

★★ スプリント計画時にReadyでないストーリーは、原則として選ばない。例外を作るとリファインメントが形骸化します。

# リファインメント・ミーティング @meeting Backlog Refinement 頻度: 週1回 / スプリント中間 時間: 1〜2時間 参加者: PO + 開発チーム 議題: [1] 上位ストーリーの詳細化 ・受入条件を埋める ・不明点をPOに確認 [2] 大きすぎるストーリーの分割 ・SPIDR で切る (Vol.3参照) ・1スプリント内に収まる粒度に [3] 見積もり ・プランニングポーカー ・ストーリーポイントで [4] 優先順位の確認・調整 ・新たな情報があれば順序変更 ・捨てるストーリーの判定 [5] 次の2〜3スプリント分が Ready の状態を維持する # 「常に整っているバックログ」 # が POの最大の仕事
ハマりどころ —
「リファインメントの時間がない」と省略すると、スプリント計画が地獄になります。「これは何をどうするんですか？」が連発し、計画会議が長期化。リファインメントを定例化するのが平和の秘訣。
03

MoSCoW法

MoSCoW PRIORITIZATION

最も普及している優先順位付け技法がMoSCoW（モスクワ）法。要求をMust / Should / Could / Won'tの4分類に振り分けます。シンプルですが、「全部Mustになる」のを防ぐのがコツ。各カテゴリの比率を意識します。

moscow_method4分類で「やる/やらない」を決める
// MoSCoW の4分類
M
MUST HAVE
絶対必須。これが無いとリリース不可。
S
SHOULD HAVE
重要。あるべきだが、無くてもリリース可能。
C
COULD HAVE
あったら良い。余裕があれば実装。
W
WON'T HAVE
今回はやらないと明確に決める。

★★★ 目安比率：Must 60% / Should 20% / Could 20%（工数ベース）。Mustが80%超えたら危険信号。

# MoSCoW の運用例 @release_v1.0 must = [ "ログイン・認証", "商品一覧表示", "カート機能", "決済（クレカのみ）", # これが無いとECとして成立せず ] should = [ "お気に入り機能", "レビュー表示", "配送追跡", # あるべきだが、無くてもOK ] could = [ "レコメンド機能", "クーポン", "ポイント", # 余裕があれば ] wont = [ "AIチャットボット", "AR試着", # 今回はやらない、と明示 # 次回以降の候補 ] # Won't を書くのが最重要 # 「やらない」を合意する
裏技 —
「これMust？それともShould？」で揉めたら、「これが無くてもリリースする勇気があるか？」と問います。「ない」ならMust、「ある」ならShould。勇気テストと呼ばれます。
04

WSJF

WEIGHTED SHORTEST JOB FIRST

MoSCoWは「分類」しかできず、Must同士の順序は決められません。そこで使うのがWSJF（重み付き最短ジョブ優先）。「価値 ÷ サイズ」でスコアを出し、高い順に並べます。SAFeで標準採用されている定量的手法です。

WSJF_formula価値 ÷ サイズ = WSJFスコア
// WSJF の計算式

WSJFスコア = CoD ÷ サイズ

CoD（Cost of Delay）= 遅延コスト

  • CoD = ビジネス価値
    　　　 + 時間的重要度
    　　　 + リスク削減/機会創出
  • サイズジョブ・サイズ（≒見積もり）

★★★ 大事だけど大きいより、そこそこ大事で小さいを先にやる、というのがWSJFの本質。

通常はフィボナッチ数列（1, 2, 3, 5, 8, 13, 20）で相対評価。

# WSJF の計算例 @features features = [ { "name": "検索機能", "business_value": 8, "urgency": 5, "risk_reduction": 3, "size": 5, # CoD = 8+5+3 = 16 # WSJF = 16/5 = 3.2 }, { "name": "AIレコメンド", "business_value": 13, "urgency": 3, "risk_reduction": 2, "size": 20, # CoD = 13+3+2 = 18 # WSJF = 18/20 = 0.9 }, { "name": "ログイン改修", "business_value": 5, "urgency": 8, "risk_reduction": 5, "size": 3, # CoD = 5+8+5 = 18 # WSJF = 18/3 = 6.0 ←最優先 }, ] # 結果: ログイン改修 > 検索 > AI # 「大きいAI」より「小さいログイン」
POINT —
WSJFの真価は「議論のフレーム」にあります。「価値は5？8？」と数値で議論することで、感覚論を排除できます。絶対値の正確さより、相対順序が大事です。
05

カノモデル

KANO MODEL

狩野紀昭博士が考案した日本発の品質モデル。機能を「あって当然」「あればあるほど嬉しい」「あると感動」の3軸で分類し、限られた工数で最大の満足度を出す方法を示します。世界でカノモデルとして知られています。

kano_model機能を3カテゴリに分けて満足度を最大化
// カノモデルの3カテゴリ
  • 当たり前品質あって当然、無いと不満
    例: ログインできる、データが保存される
  • 一元的品質あればあるほど満足度UP
    例: 検索が速い、画面がきれい
  • 魅力的品質あると感動、無くても不満なし
    例: ワンクリック購入、AI推薦

★★★ 魅力的品質は時間が経つと一元的→当たり前へ降格します。今の「魅力」は5年後の「当たり前」。だから常に新しい魅力を作り続ける必要があります。

投資配分の鉄則: 当たり前を100%確保→一元を伸ばす→魅力で差別化

# カノモデルでの判定（ECサイト例） def classify_kano(feature): # [当たり前品質] if feature in [ "カートが消えない", "決済が完了する", "商品が画像付きで表示", ]: # 無いと致命的不満 # あっても感動はない return "MUST_BE" # [一元的品質] elif feature in [ "検索の精度", "ページの表示速度", "レビューの数", ]: # 多いほど・速いほど嬉しい return "PERFORMANCE" # [魅力的品質] elif feature in [ "AIによる試着シミュ", "ライブ配信ショッピング", "パーソナルスタイリスト", ]: # 無くて当然、あれば感動 return "DELIGHTER" # Must_Be が崩れると # Delighter があっても # 顧客は去る
現場の声 —
新人POにありがちなのが「全部魅力的品質を狙ってしまう」こと。当たり前品質を疎かにして「ログインができない最先端AI」を作ってしまいます。足元から固めるのがプロ。

ストーリーポイントと見積もり

STORY POINTS

アジャイルの見積もりは時間ではなくストーリーポイント。相対的なサイズで表します。「Aは3pt、Bは5pt、BはAより少し大きい」という具合。絶対値より相対比較の方が、人間は得意なのです。プランニングポーカーで全員一致を目指します。

// プランニングポーカーのカード

100

☕

フィボナッチ数列で構成（不確実性が大きくなるほど数値の精度が落ちる、を表現）

  • ?「分からない」を表明できる
  • ☕「休憩しよう」のサイン
  • ∞「大きすぎ、分割すべき」

★★ 基準ストーリーを最初に決めるのがコツ：「これが5pt」を共有してから見積もる。

# プランニングポーカーの進行 [1] 1ストーリーを読み上げる # PO が説明、質疑応答 [2] 全員、同時にカードを出す # 議論前に独立に判断 # 同調圧力を避ける [3] 最高値と最低値の人が説明 A「2pt: ログインAPIあるから簡単」 B「8pt: 新規DBスキーマ必要」 # 認識の差が見える [4] 議論後、再度カードを出す # 全員が±1以内に収まるまで [5] 最終的に多数決 or 中央値 # 見積もりは「予測」ではなく # 「共通理解の形成」が目的 # ストーリーポイント vs 時間 # 時間: 人によってブレる # 「自分が3日」 # 「彼が5日」 # pt: 個人差がない # 「みんなにとって5pt」
TIP —
「ポーカーって遊んでるみたい...」と思われがちですが、同調圧力を排除する仕組みなのです。先に上司が「3pt」と言うと、皆「3」を出してしまう。同時提示こそが本質です。

ベロシティとリリース計画

VELOCITY / RELEASE PLAN

ベロシティ＝「1スプリントで完了するストーリーポイントの合計」。これを使って「いつ完成するか」を予測します。過去3〜5スプリントの平均を使うのが現実的。個人の評価指標として使わないのが鉄則です。

velocity_release_plan過去の実績から未来を予測
// ベロシティとは

直近のスプリント実績：

  • Sprint 5完了 28pt
  • Sprint 6完了 32pt
  • Sprint 7完了 25pt
  • Sprint 8完了 30pt
  • Sprint 9完了 31pt

平均ベロシティ= (28+32+25+30+31) ÷ 5 = 29.2 pt/sprint

残りバックログ200pt → 約7スプリント後に完了見込み

★★★ ベロシティを人事評価に使うと崩壊します。みんな見積もりを大きくして数字を稼ぐようになるからです。

# リリース計画の作り方 class ReleasePlan: velocity_avg: float # 平均 velocity_min: float # 最悪 velocity_max: float # 最良 def forecast(self, backlog_pts): # 範囲で予測する（点ではない） return { "楽観": backlog_pts/self.velocity_max, "想定": backlog_pts/self.velocity_avg, "悲観": backlog_pts/self.velocity_min, } # 例: 200pt 残り # 楽観: 200/32 = 6.3 sprint # 想定: 200/29 = 6.9 sprint # 悲観: 200/25 = 8.0 sprint # → 「7〜8スプリント後」と # 幅で伝える # 「いつ終わる？」に対して 良い答え: "6.3〜8.0 sprint後" 悪い答え: "7月15日" # ピンポイント予測は精度低い @anti_pattern # 「ベロシティを上げよう！」 → 見積もり水増しが始まる → 数字は上がるが実体は変わらず → 信頼性が崩壊
ハマりどころ —
「ベロシティをチーム間で比較」は厳禁。ストーリーポイントはチームごとの基準で決められるため、別チームの30ptと自チームの30ptは違うものです。同じチームの時系列でしか比較できません。

トレーサビリティ

TRACEABILITY

「この機能はなぜ作ったのか？」を辿れるようにするのがトレーサビリティ。ビジネス目標 → エピック → ストーリー → 受入条件 → テスト → コード、と双方向に追跡可能な状態を保ちます。規制業界では必須、アジャイルでも変更影響分析で大活躍。

traceability_matrix要求と成果物を双方向に紐づける
// トレースの階層
ビジネス目標 (BG-001) ↕ エピック (EP-101) ↕ ユーザーストーリー (US-1011) ↕ 受入条件 (AC-1011-a) ↕ テストケース (TC-1011-a-01) ↕ コード (commit hash) ↕ リリース (v2.3.0)

★★ 前方トレース: 「この要求は何になった？」
後方トレース: 「このコードは何のため？」

活用シーン:

  • 変更時「この要求が変わると、何に影響？」
  • 監査時「この要求がなぜ実装された？」
  • 不具合時「このバグは何の要求に違反？」
# トレーサビリティマトリクスの例 | 要求ID | エピック | ストーリー | テスト | |--------|---------|-----------|--------| | BG-001 | EP-101 | US-1011 | TC-001 | | | | US-1012 | TC-002 | | | EP-102 | US-1021 | TC-003 | | BG-002 | EP-201 | US-2011 | TC-101 | # 双方向リンクの実装 @User_Story US-1011 metadata = { "id": "US-1011", "parent_epic": "EP-101", "business_goal": "BG-001", "acceptance_criteria": [ "AC-1011-a", "AC-1011-b", ], "test_cases": [ "TC-1011-a-01", "TC-1011-a-02", ], "implemented_in": [ "PR-#231", "PR-#234", ], } # JIRA等のツールが # これを自動で繋いでくれる # 規制業界（金融・医療・公共）では # 監査対応で必須
POINT —
トレーサビリティを「ツールに依存させる」のが現実的。手動で表を作ると必ず更新が漏れます。Jira・Azure DevOps・GitLabの連携機能を使い、PRやコミットから自動的にリンクが張られる仕組みを整えます。
