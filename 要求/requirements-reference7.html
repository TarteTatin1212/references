REQUIREMENTS ENGINEERING REFERENCEVOL.07

要求検証・変化への対応編

Validation / Verification / Change Management

作った要求が「本当に正しいか」「実現可能か」「ステークホルダーが望むものか」を確かめる活動が要求検証。そして要求は必ず変わります。本巻ではレビュー・プロトタイピング・スプリントレビューによる検証と、避けられない変更要求への対応を扱います。

01

検証と妥当性確認

VERIFICATION VS VALIDATION

よく混同される2つの言葉：Verification（検証）は「正しく作られているか」、Validation（妥当性確認）は「正しいものを作っているか」。要求工学では両方とも重要ですが、上流ほどValidationが命です。コードは正しいのに使い物にならない——よくある悲劇です。

verification_vs_validation正しく作る／正しいものを作る
// 2つの確認

Verification 検証

「仕様通りに作られているか？」

→ 仕様⇔成果物 の整合
→ レビュー、テスト
→ 開発チームの責任

Validation 妥当性確認

「ユーザーが望むものか？」

→ 現実⇔仕様 の整合
→ プロトタイプ、UAT
→ PO・ユーザーの責任

★★★ 「完璧な仕様書通りに作ったのに、誰も使わない」はValidationを怠った典型例。

# 両者を覚えるフレーズ Verification = "Building the thing RIGHT" # ものを正しく作る Validation = "Building the RIGHT thing" # 正しいものを作る # V字モデルでの位置 要求定義 ←─── Validation ───→ 受入テスト ↓ ↑ 基本設計 ←─── Verification ──→ 結合テスト ↓ ↑ 詳細設計 ←─── Verification ──→ 単体テスト ↓ ↑ └─────── 実装 ───────────┘ # 上流の確認 = Validation # 下流の確認 = Verification
POINT —
アジャイルがValidation を毎スプリント行うのはこのため。完成してから「違いました」では遅すぎる。動くものを見せて、こまめに合意を取るのが鉄則です。
02

レビュー技法

REVIEW TECHNIQUES

要求文書のレビューには3つの段階があります。ピアレビュー（同僚同士）、ウォークスルー（作者主導）、インスペクション（厳格な公式レビュー）。検出効率が異なり、目的に応じて使い分けます。

review_techniques軽量〜重量の3レベル
// 3種のレビュー比較
種別主体厳格さ用途
ピアレビュー同僚低日常の品質維持
ウォークスルー作者中知識共有、相談
インスペクションモデレータ高重要文書の最終確認

★★ インスペクション（Fagan法）は
事前準備＋公式会議＋追跡で、
欠陥検出率が最も高い手法です。

アジャイル現場での実践:
・PRレビュー（コード）
・ストーリーレビュー（要求）
・3アミーゴ（PO・開発・QAの3者会議）

# レビューのチェックリスト # （要求文書向け） @review_checklist [完全性] ☐ 機能要求は網羅されているか ☐ 非機能要求も書かれているか ☐ 例外・エラー処理の記述あるか ☐ 「TBD」「？」が残っていないか [一貫性] ☐ 矛盾する要求がないか ☐ 用語が統一されているか ☐ 他の文書と整合しているか [明確性] ☐ 曖昧な表現がないか （「適切に」「素早く」「使いやすい」） ☐ 数値で表せるものは数値か ☐ 受入条件が書かれているか [実現可能性] ☐ 技術的に実装可能か ☐ 期日・予算内で完了するか ☐ 既存システムと連携可能か [検証可能性] ☐ テストできる形になっているか ☐ 計測可能な指標があるか # この6カテゴリでチェック # 欠落・矛盾・曖昧さを撃退
TIP —
レビューで指摘する時は「人ではなく文章を批判する」のが鉄則。「あなたの書き方が悪い」ではなく「この記述は曖昧です」。建設的な空気を作るのがファシリテータの腕の見せ所。

プロトタイピング

PROTOTYPING

「百聞は一見に如かず」を実践するのがプロトタイピング。文書で議論するより、動くものを見せた方が圧倒的に早く合意に達します。重要なのは「捨てる前提で作る」こと。完璧を求めると本末転倒です。

prototyping_types忠実度別の4タイプ
// プロトタイプの忠実度

手書きスケッチ。
15分で作成。

線画のUI。

UX検証

★★★ 初期は紙プロトで十分。最初から高忠実度のプロトを作るのは時間の浪費です。

# プロトの2つの戦略 [1] 使い捨てプロト (Throwaway Prototype) 目的: 要求の確認 作り方: 安く速く 結末: 検証後に捨てる ツール: Figma, Adobe XD, 紙とペン @注意 「もったいないから本番に流用」 → コードがスパゲッティ化 → 必ず捨てる勇気 [2] 進化型プロト (Evolutionary Prototype) 目的: 段階的に本物に育てる 作り方: 最初から本番品質 結末: そのまま製品になる 例: MVP → β版 → 正式版 # どちらが正解？ # → 状況による # UI/UX検証 = 使い捨て向き # 機能検証 = 進化型向き @wireframe_example ┌─────────────────┐ │ ☰ ロゴ 🔍 [Login] │ ├─────────────────┤ │ │ │ [大バナー画像] │ │ │ ├─────────────────┤ │ [商品1] [商品2] [商品3] │ └─────────────────┘ # これだけで議論が10倍進む
裏技 —
顧客に「これはプロトです、捨てる前提です」と最初に宣言します。これを忘れると「もう動くなら、これで完成でいいよね？」と言われて炎上します。期待値の管理こそプロトの成否を分けます。
04

スプリントレビュー

SPRINT REVIEW

アジャイル開発における最大の要求検証イベントがスプリントレビュー。完成したインクリメントをステークホルダー全員に見せて、フィードバックを集めます。これを毎スプリント（1〜2週間ごと）にやることで、方向ズレが早期に発見できます。

sprint_review最強の要求検証イベント
// スプリントレビューの構成
  • 頻度1〜2週間ごと
  • 時間1〜2時間
  • 参加者PO・開発・ステークホルダー
  • 内容動くものを見せて議論
  • 成果物更新されたバックログ

アンチパターン:

  • ✗PowerPointだけのデモ
  • ✗進捗報告だけで終わる
  • ✗批判禁止の褒め会
  • ✗POだけが質問する

★★★ 「動くもの」を「使う人」に見せるのが本質。これに勝る検証はありません。

# スプリントレビュー進行例 [1] オープニング (5分) ・スプリントゴールの確認 ・完了/未完了の概略 [2] デモ (40分) ・各ストーリーを実機で操作 ・ステークホルダーが触る ・質問・意見をその場で # これが最重要 [3] フィードバック集約 (15分) ・新しい要求 → バックログへ ・優先順位の変更検討 ・市場・競合の変化共有 [4] 次の方針議論 (15分) ・次スプリントのテーマ ・残バックログの見通し [5] クロージング (5分) ・次の予定確認 @key_question PO「これで価値ありますか？」 ユーザー「ここはイマイチで...」 PO「では、こう変えるとどうですか？」 # 即座に方向修正できる # この往復が、ウォーターフォール # では半年に1回しか起きない
POINT —
スプリントレビューを「成果発表会」と勘違いしないこと。本質は「次は何を作るかを決める場」です。完成したものを褒めるより、次の方向を決めることに時間を使いましょう。
05

要求バグの発見

FINDING REQUIREMENT BUGS

コードのバグだけでなく、要求自体にも「バグ」があります。曖昧、矛盾、欠落、過剰——これらはレビューやプロトタイプで早期発見すれば、リリース後の大事故を防げます。代表的な「要求バグ」を知っておきましょう。

requirement_defects要求文書に潜む典型的な欠陥
// 要求バグの典型
  • 曖昧性解釈が複数できる
    「迅速に」「適切に」「使いやすく」
  • 矛盾2つの要求が両立しない
    「ログイン必須」と「ゲスト購入可」
  • 欠落あるべき要求が無い
    エラー時の挙動、上限値
  • 過剰不要な要求が紛れる
    ユーザーが使わない機能
  • 実装混入「どう作るか」が要求に紛れる
    「Reactで作ること」は実装の話
  • 検証不能テストできない
    「ユーザーに愛されること」
# NG → OK のリファクタ例 # [曖昧性] ✗ "システムは高速に応答する" ✓ "P95レスポンス1秒以内" # [矛盾] ✗ R1: "全機能ログイン必須" R2: "ゲストも商品閲覧可" ✓ "R1を改訂: 購入・お気入り機能のみログイン必須" # [欠落] ✗ "ユーザーは商品を登録できる" # 削除は？更新は？権限は？ ✓ "CRUD一式の権限を明記" # [過剰] ✗ "アバター画像のフィルタは 20種類を実装" # 使うユーザーいるの？ ✓ "使用頻度上位3種に絞る" # [実装混入] ✗ "ReactとTypeScriptで実装" # 要求ではなく技術選定 ✓ "ブラウザで動作する画面を提供" # 実装は別文書で # [検証不能] ✗ "ユーザーに愛される画面" ✓ "SUS 75点以上を維持"
ハマりどころ —
「曖昧表現の禁止語リスト」を作るのが効果的。「迅速」「適切」「効率的」「ユーザーフレンドリー」「直感的」——これらを見つけたら必ず数値に変換するルールにします。
06

変更要求 (CR)

CHANGE REQUEST

要求は必ず変わります。ビジネス環境、ユーザーの理解、競合の動き——すべてが変化の源です。「変更を防ぐ」のではなく「変更を制御する」のが現代の要求管理。アジャイルでは変更を歓迎するのが基本姿勢です。

change_request_flow変更要求の処理フロー
// CR の標準フロー
[1]要求の受付 — ステークホルダーから提起
[2]記録 — CR番号を振り、Jira等に登録
[3]影響分析 — 工数・スケジュール・他要求への影響
[4]判断 — 承認 / 却下 / 保留（CCBで決定）
[5]反映 — バックログ・仕様書へ反映
[6]通知 — 関係者へ周知
[7]追跡 — 実装・テスト完了まで追う

★★★ CCB (Change Control Board) は変更を承認する組織。PO・PM・主要ステークホルダーで構成。

# 変更要求票（CR Sheet）の例 @change_request CR-2025-042 基本情報: 起票者: 営業部 田中 起票日: 2025-04-15 対象: 検索機能 (US-101) 変更内容: 現状: キーワード検索のみ 変更後: カテゴリ絞り込み追加 理由: 顧客から「探しにくい」の声多数 競合A社にも同等機能あり 緊急度: 中（次リリースまでに） 重要度: 高（売上に直結） 影響分析: ├ 工数: +5pt（+1スプリント） ├ 既存機能: 検索APIに変更必要 ├ 他要求: US-105 と整合確認 └ NFR: P95 ≤ 1s を維持要 判断: CCB承認 / 2025-04-18 → 次スプリントに組込 → 元計画のUS-201を1スプリント後ろ倒し トレース: 関連: US-101, US-105, NFR-002 実装: PR-#456
現場の声 —
「現場の口頭依頼で仕様変更」が最悪パターン。必ずCR票を起こすことで、トレーサビリティと責任の所在が保たれます。アジャイルでも「Slackで言った」だけでは不十分。バックログに登録することが「正式受付」の証です。
07

影響分析

IMPACT ANALYSIS

変更要求が来たときに最初にやるべきは「影響範囲の特定」です。1箇所の変更が、どこまで波及するか。トレーサビリティ（Vol.6参照）が活躍する場面です。「軽い変更だと思ったら全システム改修だった」を防ぎます。

impact_analysis変更の波及範囲を特定する
// 影響分析の観点
  • 他要求関連する要求は？依存関係は？
  • 設計アーキテクチャ・DB設計に影響は？
  • コード修正箇所は？テスト範囲は？
  • データ既存データのマイグレーション必要？
  • UI/UX画面遷移・操作性に影響は？
  • 運用監視・バックアップ手順に影響は？
  • 他システム連携先APIへの影響は？
  • 法令規制・契約に抵触しない？

★★ 「変更工数の見積もり」より「影響範囲の特定」が先。影響を知らずに工数は出せません。

# 影響分析テンプレート @impact_analysis CR-2025-042 [直接影響] US-101 検索機能 → 受入条件の追加 → 工数 +5pt → 検索APIの拡張 [間接影響] US-105 検索履歴 → カテゴリも履歴に保存 → 工数 +2pt NFR-002 性能要求 → カテゴリ絞り込みで P95 ≤ 1s が維持可能か → 要負荷試験 [他システム影響] 商品マスタAPI → カテゴリ情報を含む APIに拡張要 → 商品マスタチーム調整要 [データ影響] 既存商品(100万件)に カテゴリ未設定多数 → データクレンジング要 → 業務側で約2週間 [テスト影響] 影響範囲のリグレッション → E2Eテスト 15シナリオ追加 [総工数] 開発: 7pt 業務: 2週間 テスト: 3日 # これを見てCCBが判断
裏技 —
「影響なし」と判断する時こそ要注意。「本当に影響ないか、もう一段考える」習慣を。「これくらいなら大丈夫」が後で大規模リグレッションを呼ぶことは枚挙にいとまがありません。
08

変化を歓迎する文化

EMBRACE CHANGE

アジャイルマニフェストの一節：「計画に従うことよりも、変化への対応を」。これは「無計画でいい」ではなく「計画を持ちつつ、変化にも対応できる構え」を意味します。変化を恐れず、しかし無秩序にもならない——絶妙なバランスが求められます。

embrace_change変化を制御するチームの作法
// 変化に強いチームの特徴
  • 小さく作る変更コストが低い設計
  • 自動テスト変更後の安全網
  • CI/CD素早い反映と検証
  • 透明性変更経緯がいつでも見える
  • 短サイクル1〜2週間のスプリント
  • プロト文化動かして合意
  • 心理的安全変更提案ができる空気

★★★ 「変化への対応」=「ノーガード戦法」ではない。変化を吸収できる仕組みを整えるのが本質。

# 変化を歓迎するチームの # 「変えない」ものリスト @stable_foundation 変えない: ・プロダクトビジョン ・コア価値（POが守る） ・品質基準（DoD） ・チームの文化・ルール ・継続的改善の姿勢 変える: ・個別の要求・優先順位 ・実装方法 ・スコープ ・スケジュール ・体制 # 「軸はぶれない、枝葉は柔軟」 # これが変化に強いチーム @bad_anti_pattern 「言われたら何でも対応」 → スコープクリープ → 永遠に完成しない 「絶対に変えない」 → 古い要求に縛られる → 価値を失う @good_pattern 「変えるべきか、毎週問い直す」 「変えるなら、ちゃんと変える」 「やらないなら、ちゃんと断る」 # これが プロの要求管理
POINT —
変化に対する答えは3つ：受け入れる / 後回しにする / 断る。「断る」のも立派な選択肢です。何でも受け入れていたら、本当に大事なものが作れません。POの最も大切な仕事は「やらない」と言うことかもしれません。
