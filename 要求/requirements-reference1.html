01

要求工学とは何か

WHAT IS RE

要求工学（Requirements Engineering、RE）とは、システムが満たすべき「望ましさ」を発見し、整理し、合意し、維持する一連の活動です。プログラミングの前段に位置し、「何を作るか」を決める仕事の総称。コードのバグより、要求のズレの方がはるかに高くつきます。

requirements_engineering「何を作るか」を決める工学
// 要求工学が扱う3つの問い
  • WHO誰のために作るのか
    ユーザー、顧客、ステークホルダー
  • WHAT何を作るのか
    機能、画面、振る舞い、性能
  • WHYなぜ作るのか
    目的、価値、ビジネスゴール

★★★ 3つ目のWHYが抜けると、いつの間にか「誰も使わない機能」が出来上がります。

# 要求工学の定義（IEEE 610.12） requirement := { "who": # 利害関係者 [user, customer, biz, ops], "what": # 機能・品質 [function, performance, security, usability], "why": # 価値・目的 [goal, problem, value] } # この3つの整合が取れて # 初めて「要求」になる def is_valid_requirement(r): return r.who and r.what and r.why
POINT —
要求工学は「正しいものを作る」ための工学です。「ものを正しく作る」のはコーディングの仕事。順番を逆にすると、いくら綺麗なコードを書いても誰の役にも立ちません。
02

要求と要件の違い

REQUEST VS REQUIREMENT

日本語では似ているこの2語、実は明確な使い分けがあります。要求（Request）はユーザーの「やりたいこと・困りごと」の生の声、要件（Requirement）はそれをシステムが満たすべき条件として整理した結果。要求→分析→要件、という流れです。

request → requirement生の声を、システムが満たすべき条件に変える
// 同じ事柄を2つの粒度で書く

要求（Request）

「もっと早く検索結果が出てほしいなぁ」

←ユーザーの素直な声
曖昧・主観的・感情込み

要件（Requirement）

「検索ボタン押下から1秒以内に最初の結果を返すこと」

←測定可能・検証可能
客観的・具体的

★★ 「要件」はテスト可能でなければなりません。

# 要求 → 分析 → 要件の流れ # [1] 要求（Stakeholder Request） user_says = "検索が遅い、なんとかして" # [2] 分析（質問で深掘り） # - どのくらい遅い？ # - 何件くらいの検索？ # - 何秒なら満足？ # [3] 要件（System Requirement） SR-001: 検索APIは、10万件規模の データに対して、レスポンスを 1.0秒以内に返却すること （測定: 95パーセンタイル） # テストできる = 要件として成立
現場の声 —
厳密に区別する現場もあれば、「要件定義書」と一括りに呼ぶ現場もあります。本シリーズでは用語を厳密に使い分けつつ、現場の慣用も併記していきます。
03

なぜ上流が重要か

COST OF CHANGE

「要求のミスは下流で直すほど高くつく」——これは現場の鉄則です。設計フェーズで気づけば数時間、テスト直前なら数日、リリース後なら数十倍〜数百倍のコストがかかります。だからこそ上流に時間を投じる価値があるのです。

cost_of_change_curve下流で直すほど指数関数的に高くつく
// 修正コストの目安（Boehm の研究）
要求段階で発見 ┃■ ×1 設計段階で発見 ┃■■■ ×3-6 実装段階で発見 ┃■■■■■■ ×10 テストで発見 ┃■■■■■■■■■■■■■■ ×15-40 リリース後発見 ┃■■■■■■■■■■■■■■■■■■■■■■■■■■■■ ×40-100+ └───────────────────→ 修正コスト

★★★ 「要求の見直しに1日使う」のは、後工程で「10日のやり直し」を防ぐ最高の投資です。

# 同じ「ボタンの位置を変える」修正 if phase == "requirements": # レビュー会で口頭修正 cost = 1 # hour elif phase == "design": # 設計書と画面遷移図を更新 cost = 4 elif phase == "implementation": # コード書き直し + 単体テスト cost = 16 elif phase == "testing": # 結合テストやり直し cost = 40 elif phase == "production": # リリース、調整、顧客対応… cost = 200+
TIP —
「とりあえず作ってから考える」が許されるのは、使い捨てのプロトタイプか、超小規模な機能だけです。本番システムでこれをやると、ほぼ確実に炎上します。
04

アジャイルにおける要求

REQUIREMENTS IN AGILE

「アジャイルでは要求定義はしない」——これは大きな誤解です。アジャイルでも要求は徹底的に扱います。違うのは「最初に全部固める」のではなく、「常に整理し続ける」という姿勢。プロダクトバックログはまさに「生きている要求リスト」なのです。

waterfall vs agile要求を扱う「タイミング」が違う
// 要求活動のタイミング比較

ウォーターフォール

最初に全要求を固める
↓
設計・実装・テスト
↓
完成時に検証

変更コストが高い

アジャイル

概略を決めて開始
↓
イテレーションごとに
要求を詳細化・追加・変更
↓
動くものでフィードバック

変化を前提に設計

★★★ アジャイルは「要求を捨てる」のではなく「要求と継続的に向き合う」方式です。

# アジャイルにおける要求の扱い class ProductBacklog: items = [] # ユーザーストーリーの集合 def add_story(self, story): # いつでも追加可能 self.items.append(story) self.re_prioritize() def refine(self, story): # 実装直前に詳細化 story.acceptance_criteria = ... story.estimated_points = ... # 「最初に全部書く」のではなく # 「やる直前に詳しく書く」(JIT) def sprint_planning(backlog): for story in backlog.top(10): if not story.is_ready(): backlog.refine(story)
POINT —
本シリーズはアジャイル寄りで書いていますが、要求工学の基本概念は開発手法に依存しません。インタビュー、優先順位付け、検証——どの手法でも必要な技術です。
05

主要な活動の地図

RE ACTIVITIES MAP

要求工学は大きく5つの活動に分かれます。獲得・分析・仕様化・検証・管理。これは一直線ではなく、何度も行き来する反復的なプロセス。本シリーズはVol.2以降でこの5つを順に深掘りしていきます。

5_core_activities獲得 → 分析 → 仕様化 → 検証 → 管理
// 5つの活動とそれぞれの問い
  • 獲得Elicitation
    「何が欲しい？何が困っている？」を引き出す
  • 分析Analysis
    「本当に必要なものは何？」を整理・構造化する
  • 仕様化Specification
    「決まったこと」を文書・モデルで残す
  • 検証Validation
    「これで本当に良いか？」を確かめる
  • 管理Management
    「変化」を追跡し続ける

★★★ 5つは独立した工程ではなく、同時進行で何度もループします。

# 要求工学の典型的なループ while project.is_active(): # [1] 獲得 — Vol.2 raw = elicit(stakeholders) # [2] 分析 — Vol.5 structured = analyze(raw) # [3] 仕様化 — Vol.3,4,6 spec = specify(structured) # ├ 機能要求 (Vol.3) # ├ 非機能要求 (Vol.4) # └ 優先順位 (Vol.6) # [4] 検証 — Vol.7 feedback = validate(spec) # [5] 管理 — Vol.7後半 manage_changes(feedback)
TIP —
「要求定義は一回だけやれば終わり」ではありません。プロジェクトが続く限り、ずっと続く活動です。アジャイルでは特に、この感覚が重要です。
06

ステークホルダーの全体像

STAKEHOLDERS

要求は「人」から出てくるものです。誰の声を聞くべきか、誰が決定権を持つか、誰が反対するか——これを把握せずに要求工学は始まりません。ステークホルダーは「ユーザーだけ」ではない、というのが最初に押さえるべきポイント。

stakeholder_categories「使う人」だけがステークホルダーではない
// 典型的なステークホルダー
  • エンドユーザー実際に画面を触る人
  • 顧客システムにお金を払う人（=ユーザーとは限らない）
  • プロダクトオーナー何を作るか決める権限を持つ人
  • 業務担当者業務知識を持つ人（SME）
  • 運用担当本番で動かす人
  • セキュリティ部門守るべき基準を持つ人
  • 法務・コンプラ「やってはいけない」を知る人
  • 経営層投資判断する人
  • 他システム担当連携先・依存先の人
# ステークホルダーマップの例 class Stakeholder: name: str role: str influence: int # 1-5 影響力 interest: int # 1-5 関心度 # 4象限で扱い方を変える def strategy(s: Stakeholder): if s.influence >= 4 and s.interest >= 4: return "密に連携 — Key Player" elif s.influence >= 4: return "満足させる — Keep Satisfied" elif s.interest >= 4: return "情報共有 — Keep Informed" else: return "監視のみ — Monitor" # 詳細は Vol.2 で深掘り
ハマりどころ —
「ユーザーの声を聞いて作ったのに、稟議が通らない」——これは経営層・予算決裁者を最初に押さえなかった典型例。声の大きいユーザーだけ見ていると、こうなります。
