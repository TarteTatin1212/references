02

そもそも「層を分ける」とは何か

WHAT IS LAYERING

「Service層」を語る前に、「層（レイヤー）」という考え方そのものを見直します。これはソフトウェア設計史の中で繰り返し発明された根源的なアイデアです。

layering-concept 層の本質
// 層とは「役割の境界」

層（レイヤー）とは「責任の境界」です。建物が階で分かれているように、ソフトウェアも役割ごとに横の線を引く。各層は「上の層が呼ぶサービス」を提供し、「下の層が提供するサービス」を使う── この一方向の依存関係が層の核心です。

身近な例：インターネット通信。あなたがブラウザで「YouTube」と打った時、内部では4つの層が動いています：

  1. アプリケーション層（ブラウザ）
  2. トランスポート層（TCP）
  3. ネットワーク層（IP）
  4. 物理層（電気信号）

ブラウザはTCPの中身を知らない。TCPはIPの中身を知らない。各層が独立して進化できる── これが層の威力です。

Webアプリも全く同じ。Controllerは「HTTPの世界」、Serviceは「業務の世界」、Mapperは「DBの世界」── 同じ思想がここでも働いています。

// 「層」の本質的なルール // ルール①: 一方向の依存 [上の層] ↓ 知っている／呼ぶ [下の層] ↑ 知らない／呼ばれるだけ // 下の層は上の層を知らない // → 下の層は独立して進化できる // ルール②: 各層は責任が違う HTTP受付 ← Controllerの仕事 業務処理 ← Serviceの仕事 DBアクセス ← Mapperの仕事 // それぞれが「何を担当しているか」 // が明確になっている // ルール③: 隣接層しか呼ばない Controller → Service → Mapper Controllerが直接Mapperを呼ぶのは 「層を飛ばす」アンチパターン （Vol.3まではそうしていた） // インターネットを支える4層 ┌──────────────────┐ │ アプリケーション層 │ HTTPなど ├──────────────────┤ │ トランスポート層 │ TCP/UDP ├──────────────────┤ │ ネットワーク層 │ IP ├──────────────────┤ │ 物理層 │ Ethernet等 └──────────────────┘ // 4層のおかげで: // ・無線LANもLANケーブルも // 上の層を変えずに使える // ・HTTPS化してもTCPは無修正 // Web開発の3層も全く同じ思想
アナロジー ── レストランの厨房

高級レストランの厨房は厳密に層が分かれています：
ホール係（お客対応） → シェフ（料理の組み立て） → 素材係（食材の準備）。
ホール係はお客への対応を覚え、シェフはレシピを覚え、素材係は野菜の切り方を覚える。誰も他人の仕事を覚えない── これが分業の力。

もしホール係がいきなり鍋を振り始めたら？ あるいはシェフが客席で接客を始めたら？ 店は回らなくなる。Web開発もまったく同じ。各層が自分の専門に集中するから、複雑なシステムが回ります。

POINT ──

層は「物理的な分離」ではなく「論理的な役割分担」です。同じパソコンの上で動いていても、Controllerクラス・Serviceクラス・Mapperインタフェースとして役割が分けてある── これだけで層は成立します。「クラスを分けるだけで層になる」のがソフトウェア設計の不思議で美しいところです。

03

Controllerだけで全部やってはダメか?

WHY NOT FAT CONTROLLER

素朴な疑問です。Vol.3まではControllerがMapperを直接呼んで動いていました。なぜわざわざService層を入れるのか? 「動くからいいじゃん」では何が不味いのか、具体的に見ます。

fat-controller-trap Fat Controller の悪夢
// Service無しの未来予想

Vol.3のTaskControllerはシンプルでした。でも Vol.5以降で機能を追加していくと、Service無しならController内に業務ロジックが詰め込まれていくことになります。

3〜4ヶ月後の Controller の姿（想像）：

  • 新規登録メソッド ── 重複チェック、空欄チェック、保存処理
  • 編集メソッド ── 存在チェック、権限チェック、更新処理
  • 削除メソッド ── 関連レコード確認、ソフトデリート判定
  • 完了切替メソッド ── 状態遷移ルール、関連通知処理
  • 一括処理メソッド ── トランザクション制御、エラー集約

一つのControllerクラスが数百行〜千行に膨らみ、HTTPの世界の話と業務ロジックが混在する。これを「Fat Controller」アンチパターンと呼びます。読みにくく、変更しづらく、テストできない負の遺産になります。

// もしService無しで Vol.5〜Vol.10 を作ったら… @Controller public class TaskController { private final TaskMapper mapper; // 一覧 @GetMapping("/tasks") public String list(Model m) { ... } // 登録 @PostMapping("/tasks") public String create(TaskForm f) { // 入力検証 if (f.getTitle() == null) {...} // 重複チェック List<Task> existing = mapper.findByTitle(...); if (!existing.isEmpty()) {...} // エンティティ変換 Task task = new Task(); task.setTitle(f.getTitle()); ... // 保存 mapper.insert(task); // ログ出力 log.info("task created"); return "redirect:/tasks"; } // 編集 @PostMapping("/tasks/{id}") public String update(...) { // 存在チェック・権限チェック // 値変換・保存処理… // (さらに30行) } // 削除・完了切替… // あと200行続く } // 1ファイル 500行超え // HTTPの話と業務ロジックが混在 // 「修正したい場所」を探すだけで一苦労

Fat Controller の何が問題か、具体的に：

読みにくい
HTTPの話（URL、リダイレクト、Model）と業務の話（重複チェック、保存ルール）が混在する。「このメソッドは何をしているか」を1分で把握できない。
変更が他に波及しやすい
「ログ出力のフォーマット変えて」と頼まれた時、全メソッドを修正することになる。本来1箇所で済むはずの変更が、Controller全体に散らばる。
テストできない
Controllerの業務ロジックを単体テストするには、HTTPリクエスト・Modelの偽物・DBを全部用意する必要がある。「業務ルールだけ」をテストする手段がない。
同じ業務ロジックを再利用できない
たとえば「Webからの登録」と「バッチ処理からの登録」で同じ業務ロジックを使いたい時、Controllerから抜き出せないのでコピペになる。同じバグが2箇所に発生する恐怖。
✗ Fat Controller

・1ファイル数百行
・HTTPと業務が混在
・変更コスト爆発
・テスト困難
・再利用不可
・誰も触りたくない

○ Thin Controller + Service

・Controller数十行
・業務はServiceに集約
・変更範囲が明確
・Service単体でテスト可能
・別の入口（API/バッチ）からも呼べる
・読みやすい

POINT ── 「動く」と「保守できる」は違う

Service無しでも動くコードは書けます。でも「半年後に自分が読んで理解できるか」「他人がバグ修正できるか」「業務ルールが変わった時に対応できるか」── これらの問いに答えられるのが「保守できるコード」。Service層は「動く」を「保守できる」に変える装置です。

04

役割の階層 ── HTTP / 業務 / DB

THREE WORLDS

3層アーキテクチャの本質は「3つの世界が存在する」と理解することです。それぞれの世界には固有の語彙があり、関心事が違う。混ぜると思考が混乱します。

three-worlds-of-code 3つの世界とその語彙
// 3つの世界、それぞれの語彙
🌐
HTTPの世界（Controller）
語彙：URL、HTTPメソッド、リクエスト、レスポンス、ステータスコード、Cookie、ヘッダー、リダイレクト、Model、テンプレート
関心事：「どのURLでどんな入力を受け、何を返すか」
⚙
業務の世界（Service）
語彙：タスクを取得する、登録する、検証する、状態遷移、業務ルール、整合性チェック、通知、トランザクション
関心事：「業務として何をすべきか、どんなルールがあるか」
🗄
DBの世界（Mapper）
語彙：SELECT、INSERT、テーブル、カラム、JOIN、インデックス、トランザクション分離レベル
関心事：「データをどう永続化し、どう取り出すか」

注意：トランザクションは両方の世界に現れるのがポイント。業務の世界では「意味的な単位」（送金処理全体）、DBの世界では「技術的な単位」（START TRANSACTION〜COMMIT）。Serviceが両者をつなぐ役目を果たします。

// 世界ごとの語彙の違い（具体例） // HTTPの世界での表現 GET /tasks POST /tasks HTTP 200 OK HTTP 404 Not Found <a href="/tasks/1/edit"> Model.addAttribute("tasks", ...) return "tasks/list"; // 業務の世界での表現 findAll() / register() / complete() タスクを完了状態にする 重複したタイトルは登録できない 未完了タスクは削除前に警告 1日のタスク数は100件まで // DBの世界での表現 SELECT id, title FROM tasks INSERT INTO tasks VALUES(...) UPDATE tasks SET done = true DELETE FROM tasks WHERE id = ? CREATE INDEX idx_tasks_done // 「業務ルール」はどの世界に書くか？ // 例: 「タイトル重複は不可」 // ✗ Controllerに書くと… @PostMapping public String create(...) { if (mapper.findByTitle(...)) {...} // ← 業務ルールがHTTPの世界に } // ✗ Mapperに書くと… @Insert("INSERT ... ") // SQLレベルで重複チェックは不自然 // ○ Serviceに書くと… public void register(Task task) { if (mapper.findByTitle(...)) { throw new DuplicateException(...); } mapper.insert(task); } // 業務ルールが業務の世界に // 正しい場所に正しいコード
アナロジー ── 会社の組織

会社には営業（HTTP相当：顧客と直接話す）、企画（業務相当：判断する）、経理（DB相当：記録する）がいます。

営業マンが顧客から契約の話を聞いた時、その場で会社の方針を決めるのは越権です。「持ち帰って検討します」と一旦企画に渡す。企画が判断し、経理に記録を依頼する── これが組織の正しい流れ。

もし営業マンが全部一人でやろうとしたら（Fat Controller）、会社のルールがバラバラになり、組織として機能しません。役割を分けて、それぞれの専門に集中する── ソフトウェアも組織と同じ原理で動きます。

POINT ── 「コードを読む時、3つの世界のどれを読んでいるか意識する」

業務コードを読む時、「今読んでいる行はどの世界の話か」を意識すると、構造が立体的に見えてきます。「これはHTTP」「これは業務」「これはDB」── この切り替え意識が、設計判断ができるエンジニアの基礎力です。

05

「変わる理由」で分ける思考法

SINGLE RESPONSIBILITY

「役割で分ける」と言いましたが、もう一段深い原則は「変わる理由で分ける」です。これは単一責任の原則（SRP）と呼ばれるソフトウェア設計の鉄則。

TERM CARD
単一責任の原則（SRP）
Single Responsibility Principle
定義
「クラスは、たった1つの理由でしか変更されるべきでない」（ロバート・C・マーティン）。逆に言えば、変わる理由が違うものは違うクラスにすべき。
言い換え
「同じタイミングで変わるもの同士を集める／違うタイミングで変わるものは分ける」。これが「責任」の意味。
適用例
Controller ── HTTPの仕様変更（URL設計・新エンドポイント追加）で変わる
Service ── 業務ルール変更（重複チェック追加・ルール変更）で変わる
Mapper ── DB構造変更（カラム追加・テーブル変更）で変わる
この「変わる理由が3つ違う」からこそ、3層に分ける合理性が生まれる。
嬉しさ
① 変更の影響範囲が限定される（HTTPの仕様変更でMapperに手を入れる事故が起きない）
② テストが楽（責任が1つだけなら検証も1観点）
③ 並列開発できる（チーム内で「私は業務、君はHTTP」と分担可能）
change-reasons 変更要望ごとの修正範囲
// 「変更要望」と「修正範囲」のマトリクス

3層構成（Service有り）の場合、変更要望によって修正範囲がきれいに分かれます：

変更要望 Controller Service Mapper
URL変更
(/tasks → /todo-list)		 ✓ ─ ─
業務ルール追加
(重複チェック)		 ─ ✓ ─
テーブル変更
(カラム追加)		 ─ ─ ✓
画面デザイン変更 ✓ ─ ─

各行に1つだけ ✓ が立つのが理想形。これが達成されていれば、修正範囲が予測できる＝レビューしやすい＝バグが減る。

// SRP適用前 vs 適用後 // ✗ SRP違反（Fat Controller） class TaskController { void create(...) { // HTTPの判断 // 業務ルール // DB操作 } } // 変更要望が来た時: // 「URL変更」も、「業務ルール変更」も、 // 「テーブル変更」も、全部この // クラスをいじることになる。 // → 「同じファイルが何度も変わる」 // ○ SRP遵守（3層） class TaskController { // HTTPの世界だけ } class TaskService { // 業務の世界だけ } interface TaskMapper { // DBの世界だけ } // 変更要望が来た時: // 該当する層のクラスだけ修正。 // 他の層は手を触れない。 // → 「修正範囲が予測可能」 // SRPの威力 // 業務ルールが3年で5回変わったとする // Fat Controller: 5回ともクラス全体をいじる // → コミット履歴が複雑、衝突しやすい // 3層構成: 5回ともServiceだけいじる // → 履歴が単純、レビューが楽、衝突しにくい
POINT ── 「同時に変わるか？」を問う

クラス設計に迷ったら「これとこれは同じタイミングで変わるか？」と自問してください。YESなら同じクラスに、NOなら別クラスに。たったこの問い1つで、SRPに沿った設計判断ができます。「役割」より「変わる理由」を基準にすると、迷いが減ります。

06

@Transactional が何を守るのか

WHAT @TRANSACTIONAL PROTECTS

本編で出てきた@Transactionalをもう一段深く見ます。Vol.4では「読み取り専用」なのでありがたみが薄いですが、本質はVol.5の登録機能で発揮されます。先取りで理解しておきましょう。

transaction-essence トランザクションの本質
// トランザクションが守るもの

トランザクションとは「全部成功するか、全部失敗するか」の保証です。中途半端な状態（一部成功・一部失敗）を絶対に作らない仕組み。

古典的な例：銀行送金。「Aさんの口座から1万円減らす」「Bさんの口座に1万円足す」の2操作。片方だけ成功したら世界からお金が消える（または増える）── 絶対に許されない。だから2操作を「1つの単位」として扱う必要がある。これがトランザクション。

TODOアプリでも同じ。例えばVol.5以降で「タスクを登録 + 通知ログを記録」のような複合操作が出てきた時：

  • INSERT 1: tasks テーブルに登録
  • INSERT 2: notification_logs テーブルに記録

1だけ成功して2が失敗したら── タスクは登録されたのに通知ログがない不整合状態に。@Transactionalがついていれば、2の失敗を検知して1も自動的に巻き戻す（ロールバック）。不整合は決して残らない。

// @Transactional 無しの恐怖（Vol.5予告） public void register(Task task) { // ① タスク登録 taskMapper.insert(task); // ↑成功 // ② 通知ログ記録 logMapper.log("task created"); // ↑失敗（DB障害など） // 結果: tasksには新タスクがあるが // logには何もない // → データ不整合 } // @Transactional ありの安心 @Transactional // この目印を付けるだけ public void register(Task task) { taskMapper.insert(task); logMapper.log("task created"); // ②で例外発生 } // ↓ // Springが例外を検知 // → ①のINSERTを自動ロールバック // → tasksテーブルも元の状態に // → データ整合性が保たれる // Vol.4でreadOnly=trueを付ける意味 @Service @Transactional(readOnly = true) public class TaskService { ... } // Vol.4の意図: // 「このクラスのメソッドは全部読み取り」 // と宣言しておく // Vol.5以降で書き込みメソッドが // 加わったら、そのメソッドだけ // @Transactional で上書きする // → 書き込みメソッドが「目立つ」 // → 「ここは書き込みだぞ」と意識できる // → うっかり読み取りメソッドから // 書き換える事故を防げる
アナロジー ── レジ袋の中身チェック

スーパーのレジで「3点お買い上げ」と言われた時、店員さんは3点全部が袋に入っているかを確認します。もし1点でも入れ忘れがあったら、レジ打ち全部やり直し（ロールバック）。「2点だけ袋詰めして退店」は絶対NG。

トランザクションも同じ。「複数のDB操作」を1つの単位として扱い、全部揃わなければ全部やり直す。@Transactionalはこの「全部か無しか」のルールをコードに宣言する目印です。

POINT ── 「Service層 = トランザクション境界」

業務開発の鉄則：「トランザクションはService層で開始・終了する」。Controllerやmapperに@Transactionalを付けることもできますが、業務的に1つの単位はService層のメソッドが表現するから── トランザクションも自然にそこで定義する。「業務処理 = 1トランザクション」という対応がきれいに成立します。

07

DIの再訪 ── 3つのBeanがどう繋がっているか

DI REVISITED — 3 BEANS

Fat Service / Thin Controller パターン

FAT SERVICE / THIN CONTROLLER

業務開発でよく語られる「Fat Service / Thin Controller」という設計指針があります。「Controllerは薄く、Serviceは厚く」という意味。なぜこのバランスが推奨されるのか、根拠を整理します。

balance-pattern 層ごとの「太さ」のバランス
// 推奨される「太さ」のバランス
[Controller] ← 薄い（Thin） │ HTTPの受付・委譲のみ ↓ [Service] ← 厚い（Fat） │ 業務ロジックの本体 ↓ [Mapper] ← 薄い（Thin） SQLとマッピングのみ

各層のあるべき太さ：

薄
Controller ── Thin（薄い）
仕事：リクエスト受領／引数の検証（バリデーション開始）／Serviceに委譲／レスポンス整形
書くべきでないもの：業務ロジック、複雑な計算、DB操作の組み立て
厚
Service ── Fat（厚い）
仕事：業務ルールの判断／複数のMapperの組み合わせ／トランザクション制御／例外の業務的変換
業務ロジックの「本体」が住む場所
薄
Mapper ── Thin（薄い）
仕事：SQLの発行／結果のオブジェクトマッピング
書くべきでないもの：業務判断、複数テーブルにまたがる業務処理
// 推奨される実装パターン（Vol.5以降の予告） // === Controller（薄い） === @PostMapping("/tasks") public String create( @Valid TaskForm form, BindingResult br) { if (br.hasErrors()) { return "tasks/new"; } taskService.register(form); // 委譲 return "redirect:/tasks"; } // 10行程度。HTTPの作法だけ // === Service（厚い） === @Transactional public void register(TaskForm form) { // 業務ルール: タイトル重複チェック if (taskMapper.existsByTitle( form.getTitle())) { throw new DuplicateTaskException(...); } // 業務ルール: 締切日の検証 if (form.getDeadline() .isBefore(LocalDate.now())) { throw new InvalidDeadlineException(...); } // フォーム → エンティティ変換 Task task = new Task(); task.setTitle(form.getTitle()); ... // 保存 taskMapper.insert(task); // 関連処理 notificationService.sendCreated(task); } // 業務ロジックが集約されている // === Mapper（薄い） === @Insert("INSERT INTO tasks (...) " + "VALUES (...)") void insert(Task task); // 1メソッド = 1 SQL

なぜこのバランスなのか、3つの理由：

ロジックの「再利用性」
業務ロジックがServiceに集まると、同じServiceを別のController（Web画面用、API用、バッチ用）から呼べる。Controllerに書いてしまうとWebからしかBQI業務が呼べない。
変更時の影響範囲
業務ルールが変わった時、Service内の修正で済む。Controllerに分散していると、「あちこち触らないといけない」状態になる。
テストの容易さ
Service単体のテストにはHTTPが要らない。new TaskService(mockMapper)でインスタンス化してメソッドを直接呼べる。Controllerに業務ロジックがあると、テストにHTTPサーバが必要になる。
POINT ── 「業務ロジックの居場所」を作る思想

Service層の本当の役目は「業務ロジックの居場所を作る」こと。居場所があると、業務ロジックは自然にそこに集まります。居場所がないと、業務ロジックはControllerに散らばるか、Mapperにこぼれ落ちるか── どちらも悪い形になります。Vol.4で薄いServiceを先に作っておいたのは、来るべき業務ロジックの居場所を先回りで用意した、ということなのです。

テスタビリティ ── 何が嬉しいか

TESTABILITY

Service層を作ると「テストしやすい」とよく言われます。具体的に何がどう嬉しいのか、業務開発のテスト戦略と合わせて見ます。

testability-benefit Service層のテスト容易性
// 「業務ロジックだけ」をテストできる

業務開発では「業務ルールが正しく実装されているか」を検証する単体テストが重要です。例えば「タイトル重複チェック」が正しく動くかを確認したい。

Service層がない場合── 業務ロジックがControllerにあるので、テストには「HTTPサーバを起動」「HTTPリクエストを送る」「レスポンスを検証」という大掛かりな仕掛けが必要。1ケース実行に数秒かかる。

Service層がある場合── new TaskService(mockMapper)で偽物のMapperを渡してインスタンス化し、メソッドを直接呼ぶ。HTTPもDBも不要。1ケース実行はミリ秒。100ケース動かしても1秒で終わる。

この差は単なる時短ではなく、「テストを書く意欲」を左右する。テストが速いと開発者はテストを書きたくなる → コード品質が上がる、という好循環が生まれます。

POINT: Mock（モック）とは「偽物」のこと。本物のMapperをDBに繋ぐ代わりに、「findByTitle("foo")が呼ばれたら空リストを返す」と指示された通りに動く偽物を作ってテストに使う。これによりDBなしでService単体のロジックを検証できる。
// テストの書き方の対比（イメージ） // === Service層なし（NG） === @SpringBootTest class TaskControllerTest { @Autowired private MockMvc mvc; @Test void タイトル重複なら登録不可() { // 重い: HTTPリクエスト組み立て mvc.perform(post("/tasks") .param("title", "既存タイトル")) // HTTPレスポンスを検証 .andExpect(status().is4xxClientError()); // 実行時間: 数秒（Spring起動含む） // 必要なもの: DB or Mock DataSource } } // === Service層あり（OK） === class TaskServiceTest { @Test void タイトル重複なら例外() { // 偽物のMapperを用意 TaskMapper mockMapper = mock(...); when(mockMapper.existsByTitle("foo")) .thenReturn(true); // Service単体で組み立て TaskService service = new TaskService(mockMapper); // メソッドを直接呼んで例外を検証 assertThrows( DuplicateTaskException.class, () -> service.register(...) ); // 実行時間: ミリ秒 // 必要なもの: なし（HTTPもDBも不要） } } // この軽さが「テスト書く意欲」を支える
アナロジー ── 車のエンジン単体テスト

新しい車のエンジンを開発する時、車を1台組み立ててから動作確認するのは効率が悪いです。だからエンジン単体を試験台に載せて、車体やタイヤなしでエンジン部分だけを検証する。

Service層のテストも同じ。業務ロジックだけを取り出して、HTTPやDBという「車体やタイヤ」を外して検証する。必要な部分だけに集中できるから、テストが速くて確実。

POINT ── 「テスタブル設計」は将来への投資

Vol.4ではテストコードは書きませんが、テスタビリティを意識した設計を体に染み込ませることが重要です。「Service層を作る ＝ テストできる設計を選ぶ」と覚えてください。本気の業務開発ではJUnitやMockitoでService層に単体テストを書くのが標準。テストのないコードはレビューを通らない現場も多いです。

なぜVol.4でServiceを入れるか

WHY VOL.4 IS THE RIGHT TIME

Service層はもっと早く入れるべきだったのでは? あるいはもっと後でも良かったのでは? 教材設計上の「時期の判断」を共有します。

timing-decision 「いつService層を入れるか」の設計判断
// 候補となるタイミング3つ
✗ Vol.1〜Vol.2 から導入

プロジェクト作成と同時にServiceを作る。
欠点：用語が多すぎて初学者の認知負荷が爆発。「動くもの」より「概念」が先行し、挫折リスクが高い。

○ Vol.4 で導入（本書の選択）

Controller・Mapper・Entityで「動くもの」を完成させた後、業務ロジックが入る前夜に追加。
利点：必要性が腹落ちしやすく、認知負荷も適度。

✗ Vol.6以降に先送り

登録・更新・削除を全部Controllerに書いてから、後で分離。
欠点：Fat Controllerの悪夢を一度作ってからリファクタリングするので、「同じ作業の二度手間」が発生。教材として効率悪い。

※ 実務での選択

実務ではプロジェクト最初からService層を作るのが標準。本書は「必要性を体感させてから導入」という教材的判断でVol.4配置。

Vol.4配置の意図：Vol.3まで「Service無しでも動く」を体験し、Vol.4で「これからService層が必要になる」と納得して導入する。「ありがたみを実感してから道具を渡す」という教材設計です。

// 学習者の認知発達と教材構成 Vol.1: プロジェクト作成 // 動くものへの興奮 Vol.2: DB繋ぎ込み // データが見える喜び Vol.3: Entity + Thymeleaf // 構造化への進化 Vol.4: Service層 ← ★ココ // 業務開発の標準形に到達 Vol.5〜: 登録・更新・削除・例外 // Serviceに業務ロジックが集まる // → ありがたみが体感できる // 実務との違い // 実務での進め方: Day 1: プロジェクト雛形作成 // → 最初からControllerもServiceも // MapperもEntityも空のクラスで配置 Day 2〜: 業務要件を順次実装 // 各層に書き込んでいく // 本書（教材）での進め方: Vol.1〜3: 必要最小限で動かす // 「動く」を最優先 Vol.4: 構造を整える // 「保守できる」へ進化 Vol.5〜: 業務ロジック追加 // 整えた構造の上に積み上げる // 教材は「必要性の体感」を重視 // 実務は「最初から正解の形」を採用 // どちらも正しい、目的が違うだけ
POINT ── 「学習教材」と「実務」のギャップ

本書はあえてVol.3まで Service 無しで進めましたが、実務ではプロジェクト最初からService層を作るのが普通です。実務に行ったら「Service無しの選択肢」は基本ありません。教材は「段階的な理解」のために緩い設計から始め、必要性を体感した瞬間に正しい設計に進化する── という構成を取っているだけです。実務では迷わず最初からService層を作りましょう。

自分で書くための判断基準

JUDGEMENT CRITERIA

最後に、「自分でSpring Boot アプリを設計する時」に役立つ判断基準をまとめます。本補足の集大成として、明日から使える意思決定フローを提示します。

decision-flow 設計判断のフローチャート
// 「このロジックはどこに置く?」フロー
そのロジックは「HTTPの世界」のもの?
YES（URL設計、リダイレクト、Modelの組み立て、リクエスト検証）
→ Controllerに書く
NO → 次の質問へ
そのロジックは「DBアクセスの世界」のもの?
YES（SQL、テーブル操作、結果のオブジェクトマッピング）
→ Mapperに書く
NO → 次の質問へ
そのロジックは「業務ルール」のもの?
YES（重複チェック、状態遷移、業務的な検証、複数Mapperの組み合わせ）
→ Serviceに書く
NO → 別の置き場所（Utility、Component等）を検討
そのロジックは「複数のテーブル操作」を含む?
YES → 必ずServiceに書く（トランザクション境界として）
NO → Mapperのみで完結可能
POINT: 迷ったら「Serviceに書く」のが基本。後から「これはControllerでも良かった」と気付いて移すのは簡単。逆は難しい。Service寄りに倒すと安全です。
// 業務開発の「鉄板テンプレート」 // === Controller === @Controller / @RestController public class XxxController { private final XxxService service; // コンストラクタ注入 public XxxController(...) {...} @GetMapping / @PostMapping public String xxx(...) { // 1. 入力検証 // 2. service.xxx() に委譲 // 3. 結果をModelに載せる // 4. テンプレート名を返す } } // === Service === @Service @Transactional(readOnly = true) public class XxxService { private final XxxMapper mapper; // 他のMapperやServiceも依存可能 public XxxService(...) {...} // 読み取り（クラス指定が効く） public List<Xxx> findAll() { return mapper.findAll(); } // 書き込み（メソッド指定で上書き） @Transactional public void register(...) { // 業務ルール検証 // 必要な変換 // mapperの呼び出し } } // === Mapper === @Mapper public interface XxxMapper { @Select("...") List<Xxx> findAll(); @Insert("...") void insert(Xxx x); // 各メソッド = 1 SQL // 業務判断は持たない } // この形を覚えれば、明日から // 新しいエンティティでも同じパターンで // 書ける

新しい機能を作る時のチェックリスト：

  • そのテーブル用のEntityクラスを作った（com.example.todo.entity）
  • そのテーブル用のMapperインタフェースを作った（com.example.todo.mapper、@Mapper付き）
  • 業務ロジック用のServiceクラスを作った（com.example.todo.service、@Service＋@Transactional付き）
  • HTTPアクセス用のControllerクラスを作った（com.example.todo.controller、@Controllerまたは@RestController付き）
  • 各クラスはprivate final + コンストラクタ注入で依存を受け取っている
  • Controllerに業務ロジックを書いていない
  • Mapperに業務判断を書いていない
  • Service に HTTP関連のコード（Model等）を書いていない
  • 書き込みメソッドには @Transactional が付いている
POINT ── 「迷ったらService」を覚える

新しい処理を書く時、「Controllerに書こうか、Serviceに書こうか」と迷ったらServiceを選ぶのが鉄則。Controllerに置いて困るケースは多いが、Serviceに置いて困るケースはほとんどない。「迷ったらService」── これだけ覚えておけば、業務開発の99%は乗り切れます。
