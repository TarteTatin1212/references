03

用語カード：@Delete

TERM — @DELETE

CRUDのD（Delete）を担うアノテーション。@Select / @Insert / @Update と同じ仲間で、書き方も同じ。

TERM CARD #01
@Delete
DELETE文を発行するMyBatisアノテーション
定義
MyBatisのアノテーション。Mapperインタフェースのメソッドに付けて、DELETE文の発行を宣言する。@Select / @Insert / @Update と同じ作法。
仕組み
メソッドが呼ばれると、MyBatisがSQLの#{プロパティ名}部分を引数の値で置き換え、PreparedStatementで実行。返り値は普通voidかint（影響行数）。
典型例
@Delete("DELETE FROM tasks WHERE id = #{id}")
Long型の引数を取り、その値でidを絞り込む。
WHERE句忘れは大事故── 全レコード削除になる
CRUDの仲間
@Select ── SELECT文（読み取り）
@Insert ── INSERT文（登録）
@Update ── UPDATE文（更新）
@Delete ── DELETE文（削除）★Vol.7で登場
delete-sql DELETE文の典型と危険性
// DELETE文の使い方と注意

典型パターン：IDで1件削除

DELETE FROM tasks WHERE id = #{id}

これしか書かないのがVol.7の方針。シンプル。

⚠ WHERE句忘れの恐怖（UPDATEより重大）

DELETE FROM tasks（WHERE句なし）を実行すると全レコード削除。本番DBでやると復元不可能な大事故。UPDATEの「全レコード更新」よりも遥かに重大。「DELETE文を書く時は WHERE 句から書く」癖を必ずつけましょう。

💡 別冊予告：論理削除という選択肢

本格的な業務システムでは、「物理削除しない」運用が主流です。DELETEの代わりに UPDATE で deleted_at = NOW() を立てる方式（論理削除）。「削除されたように見せて、データは残す」── 履歴管理・誤削除復元・監査に重要。詳しくはVol.7 別冊で。

// @Delete の基本 @Delete("DELETE FROM tasks " + "WHERE id = #{id}") void delete(Long id); // MyBatisが裏で発行するSQL // Service側で: taskMapper.delete(2L); // 実際のSQL: DELETE FROM tasks WHERE id = 2; // 1行が削除される // ★絶対NG★ WHERE句忘れ @Delete("DELETE FROM tasks") void deleteAll(); // → 全レコード消失！ // → バックアップないと復元不可 // → 本番ならクビ案件 // 引数の渡し方 // パターンA: プリミティブ型を直接 @Delete("DELETE FROM tasks " + "WHERE id = #{id}") void delete(Long id); // 引数1つなので名前不問 // #{id} は #{x} でも動く // パターンB: オブジェクトを渡す @Delete("DELETE FROM tasks " + "WHERE id = #{id}") void delete(Task task); // task.getId() を #{id} で取る // 削除の影響行数チェック @Delete("DELETE FROM tasks WHERE id = #{id}") int delete(Long id); // 戻り値 = 削除された行数 // 1 → 成功 // 0 → 該当なし（既に削除済み等） // Vol.7では void で OK // シンプルさ優先
POINT ── 「削除は不可逆」

DELETE文を実行すると、レコードは完全に消失します。バックアップがなければ復元不可能。UPDATEなら後で値を書き直せるけれど、DELETEは取り返しがつかない。だから確認ダイアログでユーザーに念押しするのが業界の慣習。本格的な業務システムでは論理削除（DELETEの代わりにフラグを立てる）が主流── これは別冊で深掘ります。

用語カード：部分UPDATEとフォーム送信

TERM — PARTIAL UPDATE

Vol.6の編集機能では全項目UPDATEを使いました。Vol.7の完了切替は「1項目だけ更新」── 部分UPDATEパターンの初登場です。

TERM CARD #02
部分UPDATE
特定カラムだけを更新する設計
定義
UPDATE文で特定のカラムだけを更新するパターン。Vol.6の全項目UPDATEとの対比で「1項目だけのUPDATE」と捉えてよい。
典型例
@Update("UPDATE tasks SET done = #{done} WHERE id = #{id}")
done 列だけ更新、title や他のカラムは触らない。
適する場面
① 1クリックボタンで状態切替（完了切替、いいね、フォロー）
② ステータス変更のみ（注文ステータスの進行）
③ パスワード変更のみ
共通点：1つの操作で1項目だけ変える場面
全項目UPDATEとの違い
全項目：編集画面で全フィールド扱う → 1つのupdate()メソッドで対応
部分：機能ごとに別UPDATE → 専用メソッドを作る（updateDone()など）
共存可能：同じMapperに両方あってOK
partial-update-form 部分UPDATE + 小さなフォーム
// list.html に置く「小さなフォーム」

HTMLの制約で、「リンクをクリックでPOST送信」は基本的にできません（GETになる）。POSTを送るには <form> 要素が必要です。

そこで、list画面の各行に「ボタン1個の小さなフォーム」を置くのがVol.7の方針。完了切替も削除も、それぞれ独立した小さなフォームになります。

このフォームの特徴：

  • 入力欄なし（IDはURLに含まれる）
  • ボタンのみ（クリックで即送信）
  • action にIDを含めた動的URL
  • method="post" を必ず指定
POINT：1つのテーブル行に2つのフォーム（完了切替・削除）を並べる。フォームの中に他のフォームを入れることはできないので、同じ td内に2つのform要素を並列に置く形になります。
// 部分UPDATEのMapperメソッド // Vol.6: 全項目UPDATE @Update("UPDATE tasks SET " + "title = #{title}, " + "done = #{done} " + "WHERE id = #{id}") void update(Task task); // Vol.7: 部分UPDATE（doneのみ） @Update("UPDATE tasks SET " + "done = #{done} " + "WHERE id = #{id}") void updateDone(Task task); // title 等は触らない // done だけ書き換える // 「小さなフォーム」の典型 // list.html の各行に置くフォーム <!-- 完了切替ボタン --> <form th:action="@{/tasks/{id}/toggle(id=${task.id})}" method="post" style="display:inline"> <button type="submit"> 完了切替 </button> </form> <!-- 削除ボタン --> <form th:action="@{/tasks/{id}/delete(id=${task.id})}" method="post" style="display:inline" onsubmit="return confirm('削除しますか?')"> <button type="submit"> 削除 </button> </form> // なぜリンクではダメか // もしリンクで実装したら… <a href="/tasks/2/toggle"> 完了切替 </a> // → ブラウザは GET /tasks/2/toggle を送る // → Springは @GetMapping を探す // → @PostMapping しか定義してないので // 405 Method Not Allowed エラー // → だからフォームで POST を送る必要あり // 「display:inline」の意味 // formは標準で block 要素 (改行する) // テーブルの1セル内に複数form置きたい // → display:inline で横並びに // 簡易的にstyleを直書きしているが // 本来はCSSファイルにまとめるべき
POINT ── 「ボタン = 小さなフォーム」

「リンクでPOST送信」は標準HTMLでは不可能。JavaScriptで実現する方法もありますが、シンプルに小さなフォームを置くのが業界の定番です。「アクションを起こすボタン = 小さなフォーム」と覚えておくと、Webアプリ開発の見通しがよくなります。

05

用語カード：JavaScript confirm() による削除確認

TERM — CONFIRM DIALOG

削除は不可逆なので、ユーザーに「本当に削除しますか?」と確認を取るのが業界の慣習。最もシンプルな方法はJavaScript の confirm() です。

TERM CARD #03
JavaScript confirm()
ブラウザ標準の確認ダイアログ
定義
JavaScript の組み込み関数 confirm(message)。OK / キャンセル の2択ダイアログを表示し、ユーザーの選択結果をtrue / false で返す。
使い方
HTMLのフォームで onsubmit="return confirm('...')" と書くと、送信前に確認ダイアログが出る。OKならtrue → 送信実行、キャンセルならfalse → 送信中止。
嬉しさ
① JavaScriptを1行書くだけで確認機能が実現
② 業界標準のシンプル実装
③ 誤クリックを防げる
限界
① ダイアログのデザインを変えられない（ブラウザ標準のまま）
② 翻訳できない（OKボタンは英語のまま）
③ モバイルでは控えめなUX
本格的なアプリではカスタムモーダルに置き換える（jQuery UI、Bootstrap modal、React コンポーネント等）
confirm-usage confirm() の使い方
// confirm() の仕組み

フォーム送信時の流れ：

送信ボタンクリック時の挙動
  1. ユーザーが「削除」ボタンクリック
  2. ブラウザが<form>のonsubmit属性を実行
  3. confirm("削除しますか?")でダイアログ表示
  4. OKならtrue、キャンセルならfalseが返る
  5. return の値が true なら送信実行
  6. return の値が false なら送信中止

注意点：

  • onsubmit 属性は <form> 要素に付ける（button ではない）
  • return を必ず書く── 書かないと送信中止できない
  • シングルクォート（'）とダブルクォート（"）の使い分けに注意
  • メッセージは具体的に（「削除しますか?」より「このタスクを削除しますか?」がベター）
// confirm() の基本パターン <form action="/tasks/2/delete" method="post" onsubmit="return confirm('削除しますか?')"> <button type="submit">削除</button> </form> // 動作確認 // クリック時の流れ: [ユーザー] 「削除」ボタンクリック ↓ [ブラウザ] onsubmit 実行 confirm("削除しますか?") 呼び出し ↓ [ダイアログ] ┌──────────────┐ │ 削除しますか? │ │ [OK] [キャンセル]│ └──────────────┘ ↓ [OK] → confirm() が true を返す → return true → フォーム送信実行 [キャンセル] → confirm() が false を返す → return false → フォーム送信中止 // よくあるバグ // ✗ return がない <form onsubmit="confirm('削除?')"> // → ダイアログは出るが // キャンセル押しても送信される // → 削除実行される（最悪） // ✓ return を必ず書く <form onsubmit="return confirm('削除?')"> // → キャンセルで送信中止 // メッセージのクオート問題 // HTML属性はダブルクォート、 // JS文字列はシングルクォートで囲む onsubmit="return confirm('削除?')" // ↑ HTML属性内 // ↑ JS文字列 // 同じクォートが入れ子になると // パースエラーになるので注意 // 日本語の「'」は普通のクォートと区別 // されないので、メッセージ内の引用符は // 全角を使うと安全
POINT ── 「シンプルな確認で十分」

本書ではconfirm()でシンプルに済ませますが、本格的なアプリではモーダルダイアログ（Bootstrap、Modal、React/Vueのコンポーネント等）を使うのが主流。デザインを統一できる、メッセージを翻訳できる、アクセシビリティ対応しやすい── という利点があります。学習用にはconfirm()で十分、「シンプルなツールで本質を学ぶ」のがVol.7の方針です。

// 手順

toggleDone()の業務ロジック：

  1. findById() で現在のTaskを取得
  2. 現在の done 値を反転（!task.getDone()）
  3. updateDone() でDBに反映

「Service層が業務判断する」本質的な例。Controllerは「toggleしたい」と頼むだけ、Serviceが「現在の状態を見て反転する」という業務的な判断を行います。これがFat Service / Thin Controllerパターン。

delete()はシンプルで、Mapperにそのまま投げるだけ。業務判断は不要。

// 修正後の TaskService.java（全体） package com.example.todo.service; import java.util.List; import org.springframework.stereotype.Service; import org.springframework.transaction .annotation.Transactional; import com.example.todo.entity.Task; import com.example.todo.form.TaskEditForm; import com.example.todo.form.TaskForm; import com.example.todo.mapper.TaskMapper; @Service @Transactional(readOnly = true) public class TaskService { private final TaskMapper taskMapper; public TaskService(TaskMapper taskMapper) { this.taskMapper = taskMapper; } // 一覧（Vol.4） public List<Task> findAll() { return taskMapper.findAll(); } // 新規登録（Vol.5） @Transactional public void register(TaskForm form) { Task task = new Task(); task.setTitle(form.getTitle()); taskMapper.insert(task); } // 1件取得（Vol.6） public Task findById(Long id) { return taskMapper.findById(id); } // 全項目更新（Vol.6） @Transactional public void update(TaskEditForm form) { Task task = new Task(); task.setId(form.getId()); task.setTitle(form.getTitle()); task.setDone(form.getDone()); taskMapper.update(task); } // 完了切替（Vol.7で追加） @Transactional public void toggleDone(Long id) { Task task = taskMapper.findById(id); task.setDone(!task.getDone()); taskMapper.updateDone(task); } // 削除（Vol.7で追加） @Transactional public void delete(Long id) { taskMapper.delete(id); } } // toggleDone() のロジックの妙 // なぜ「findById → 反転 → updateDone」と // 2回もDBアクセスするのか? // 答え: 「現在のdone値」を知るため // 別案: フォームで現在値を渡す // （Vol.6の編集と同じ方式） // 今回 findById で取りに行ったのは // 「URLパラメータだけで切替したい」 // → フォームに状態を埋め込まずに済む // 業務的な判断を Service に集約 // Controllerは「toggleしたい」とだけ言う taskService.toggleDone(id); // Serviceが業務的な処理を担当 // 1. 現状取得 // 2. 反転判断 // 3. 更新 // → これが Service層の役目 // 「業務ロジックの居場所」 // （Vol.4 ADDで学んだ）
POINT ── 「@Transactional の本領発揮」

toggleDone()は1メソッド内で2つのDBアクセス（findById + updateDone）。@Transactionalがついているので、2つのSQLが1つのトランザクションとして扱われます。万一findById後にupdateDoneで例外が出ても、トランザクション全体が安全にロールバックされる── これがVol.4 ADDで予告した@Transactional の本領です。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

Vol.7のハマりは「フォーム/POST」「動的URL」「JavaScript」の3軸に分布。Vol.6までで馴染んだ問題が中心です。

⚠ ハマり 1 ── 完了切替ボタンを押すと 405 Method Not Allowed
SYMPTOM ボタンクリックで HTTP 405 エラー画面が表示される
CAUSE list.htmlで <a href="..."> リンクで実装してしまった、またはmethod="post" を書き忘れた。サーバが GET /tasks/{id}/toggle を受けるが、@PostMapping しか定義していないので 405。
FIX <form method="post"> + <button> の構造に修正。アンカータグ（<a>）ではなくフォームを使う。
⚠ ハマり 2 ── 削除ボタンを押しても確認ダイアログが出ない
SYMPTOM 確認なしで即削除される。
CAUSE onsubmit 属性を<button>側に書いてしまった。onsubmitは<form>要素にしか効かない。あるいは return を書き忘れて、ダイアログ表示後に必ず送信されている。
FIX <form ... onsubmit="return confirm('...')"> とformタグに付ける、かつreturnを必ず書く。
⚠ ハマり 3 ── 削除しても「該当行が消えない」
SYMPTOM 削除ボタンを押すとリダイレクトはされるが、一覧から消えていない。コンソールには DELETE のSQLが流れている。
CAUSE WHERE句のIDが取れていない。th:action の動的URL指定で (id=${task.id}) を忘れた、または {id} のままでURLに展開されていない。SQLが DELETE FROM tasks WHERE id = NULL となり該当0件。
FIX th:action="@{/tasks/{id}/delete(id=${task.id})}" のパス変数指定 (id=${task.id}) を必ず書く。
⚠ ハマり 4 ── 完了切替で「null cannot be cast to Boolean」
SYMPTOM 完了切替時に NullPointerException または ClassCastException が出る
CAUSE task.getDone() が null を返している（DBで done カラムが NULL になっている）。!null はキャストエラーになる。
FIX DBのデータを確認し、done が NULL のレコードを UPDATE tasks SET done = false WHERE done IS NULL; で修正。または Service の toggleDone() で null チェックを入れる（task.getDone() == null ? true : !task.getDone()）。
⚠ ハマり 5 ── ボタンが縦に並んでしまう
SYMPTOM 「編集」「完了切替」「削除」が縦に並ぶ。同じ行内に並べたい。
CAUSE <form> はHTML標準で block要素。各フォームが改行を作る。
FIX <form style="display:inline"> をすべてのフォームに付ける。あるいは CSS ファイルに form.action-form { display: inline; } を書く。
POINT ── 「Vol.7はHTMLとSpringの境界線」

Vol.7のハマりは「HTMLの仕様」と「Springのルーティング」の境界線で起きがちです。POSTがGETになってしまう、動的URLが展開されない、フォームの入れ子NG等── HTMLの基本を押さえれば避けられます。困った時は「ブラウザの開発者ツール」で実際のリクエストを確認してみましょう。Vol.3 ADD で学んだ 9ステップ思考フロー をここでも活用できます。
