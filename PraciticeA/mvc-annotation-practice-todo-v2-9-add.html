02

そもそも例外とは何を表すのか

WHAT EXCEPTION REPRESENTS

「例外」という言葉は多義的です。プログラミング用語、業務用語、自然言語── それぞれ違う意味を持ちます。業務開発で扱う「例外」を正確に定義するところから始めます。

what-exception-represents 例外の3つの側面
// 例外が表す3つのもの
「想定外」の状態
通常のフローでは起きないはずの状態。例えば「ファイルが見つからない」「ネットワーク切断」「DBアクセス失敗」── システムの正常な動作の外側で起きる事態。
「処理の続行不可」状態
今のメソッドがこれ以上処理を続けられない状態。「呼んだ側に判断を委ねる」必要がある時に例外を投げる。「続けられないからあなたに返します」というメッセージ。
「業務ルール違反」の表現
業務的に許容できない事象。「在庫がないのに注文を受け付けようとした」「権限のないユーザーが操作した」── これらは技術的には可能でも、業務的に弾く必要がある。例外で「これは業務違反です」と表現する。

Vol.9で作った TaskNotFoundException は、3番（業務ルール違反）と1番（想定外）が混じった例。「存在するはずのIDだが、実際には存在しなかった」という業務的な不整合を表現しています。

// 例外が表すもの：3パターン // === パターン1: 想定外の状態 === try { FileReader reader = new FileReader("data.txt"); } catch (FileNotFoundException e) { // 「ファイルがあるはず」だったのに無い // → 想定外 } // IOException, SQLException 等の // 「環境起因」の例外もここ // === パターン2: 続行不可 === public Task findById(Long id) { Task task = mapper.findById(id); if (task == null) { // 「Task を返す」契約を満たせない // → 呼び出し側に判断を委ねる throw new TaskNotFoundException(id); } return task; } // Vol.9の TaskNotFoundException はこれ // === パターン3: 業務ルール違反 === public void placeOrder(Long productId, int qty) { Product p = mapper.findById(productId); if (p.getStock() < qty) { // 業務的に弾く必要あり throw new InsufficientStockException( p.getName(), qty, p.getStock()); } if (!user.hasRole("BUYER")) { // 権限不足 throw new UnauthorizedException(); } } // 3パターンの共通点 // いずれも: // ・「通常のフローでは処理できない」 // ・「上の階層に判断を委ねる」 // → 例外という仕組みは // 「コミュニケーション手段」 // 下位レイヤーから上位レイヤーへ // 「困った」を伝える
アナロジー ── 部下から上司への報告

例外は「部下が上司に困りごとを報告する」ような仕組み。
通常業務：部下が仕事を完遂して結果を上司に渡す（戻り値）
困った時：「自分では判断できない事態が起きました」と上司に報告（例外）

上司は判断します── 別の方法を試すか、自分の上司に相談するか、お客様に謝るか。「判断できない事態を上に上げる」のがシステムの健全な動き。

Vol.9で作った仕組みは、Service（部下）が「タスクが見つかりません」と Controller（上司）に報告し、Controller も判断できないので Spring（さらに上の上司）に上げ、最終的に GlobalExceptionHandler（経営層）が会社の方針として「404画面を出す」と決める── そういう構造です。

POINT ── 「例外は通信手段」

例外を「エラー」と訳すと誤解しやすい。例外は「上のレイヤーへの通信手段」です。「ここでは判断できないから、判断できる上の人に渡す」── これが例外の本質。「下位の責任」を「上位の責任」に切り替える瞬間に例外を投げます。この見方ができると、例外設計の判断が一気にしやすくなります。

03

用語カード：検査例外 vs 非検査例外

TERM — CHECKED VS UNCHECKED

Javaの例外を2種類に分ける言語仕様。これを理解しないと、業務コードで何度もコンパイルエラーに苦しみます。業界の主流は非検査例外── その理由も含めて整理します。

TERM CARD #01
検査例外 vs 非検査例外
Java言語仕様による例外の2分類
検査例外（Checked Exception）
Exception の直接の子クラス（RuntimeExceptionを除く）。呼び出し側で必ず try-catch または throws 宣言が必要。コンパイラがチェックする。
例：IOException、SQLException、InterruptedException
非検査例外（Unchecked Exception）
RuntimeException とその子クラス、または Error 系。try-catch も throws も強制されない。コンパイラはチェックしない。
例：NullPointerException、IllegalArgumentException、TaskNotFoundException（Vol.9で作成）
違い
検査例外：「呼び出し側は対応必須」と言語が強制
非検査例外：「呼び出し側の判断に委ねる」（強制しない）
業界の主流
業務アプリでは非検査例外（RuntimeException系）が圧倒的多数派。
理由：
① コードがシンプルになる（throws宣言不要）
② Spring / Spring Boot が非検査例外を前提に設計されている
③ @Transactional はデフォルトで非検査例外のみロールバック
本書での方針
業務例外はすべて RuntimeException 継承。シンプルで実用的。
checked-vs-unchecked 2種類の例外の対比
// Java の例外階層図
Throwable
├── Error             ← 致命的、扱わない（JVMが死ぬ系）
│   ├── OutOfMemoryError
│   └── StackOverflowError
└── Exception          ← アプリで扱う例外

        ├── NullPointerException

        └── TaskNotFoundException  ★Vol.9

検査例外の不便さ：

検査例外を投げるメソッドを呼ぶには、必ずtry-catchまたは throws 宣言が必要。これがすべての呼び出しに伝播するので、コードが汚くなります。

検査例外（強制が辛い）

あらゆる中間メソッドが throws SQLException を書く羽目に。
SQLException が漏れ出ると、上位の業務メソッドのシグネチャが醜くなる。

非検査例外（業界の選択）

呼び出し側は判断できる時だけcatchすれば良い。中間メソッドが何も書かない（暗黙の透過）。
業務メソッドのシグネチャがすっきり。

POINT：Spring の JdbcTemplate や MyBatis は内部的にSQLExceptionを非検査例外でラップします。だから業務コードに throws SQLException が出ません。フレームワークレベルで「非検査例外への変換」が行われています。
// 検査例外の例（不便さ） // === IOException は検査例外 === // メソッドAが投げると… public void readFile() throws IOException { new FileReader("x.txt"); // IOException可能性 } // メソッドAを呼ぶBも throws 必須 public void B() throws IOException { readFile(); } // Bを呼ぶCも throws 必須 public void C() throws IOException { B(); } // → throws SQLException がコード中に // ずっと付いて回る // → 業務メソッドが「技術的な詳細」 // で汚れる // 非検査例外の例（シンプル） // === TaskNotFoundException は非検査 === public Task findById(Long id) { Task task = mapper.findById(id); if (task == null) { throw new TaskNotFoundException(id); } return task; } // ↑ throws 宣言なし // 呼ぶ側も追加で何もしない @GetMapping("/tasks/{id}/edit") public String editForm(...) { Task task = service.findById(id); // throws 不要 // try-catch 不要 // 例外は勝手に伝播する } // ↑ シンプル！ // 結論：業務開発の選択 // 業務例外 → 非検査例外（RuntimeException） public class 業務例外 extends RuntimeException { ... } // メリット: // ・コードがシンプル // ・@Transactional が自動ロールバック // ・Spring エコシステムと整合 // 検査例外は… // 入出力・通信などの // 「外部リソース起因」の例外でのみ使う // （IOException, SQLException） // 業務ロジックには非検査例外を使う // ← Vol.9の方針
POINT ── 「非検査例外がデフォルト」

業務開発では「迷ったら非検査例外（RuntimeException 継承）」と覚えてください。検査例外を使うのは「呼び出し側に絶対対処させたい例外」のみ── これは実務では稀です。Spring が前提とする世界観に合わせるのが、業務コードを書く上での最短ルートです。

04

用語カード：業務例外 vs システム例外

TERM — BUSINESS VS SYSTEM EXCEPTION

もう1つの例外の分類軸。「業務的に起こりうる事象」と「システム的なトラブル」── これを意識すると、エラー画面の出し分けやログ出力の判断がしやすくなります。

TERM CARD #02
業務例外 vs システム例外
例外の「種類」による2分類
業務例外（Business Exception）
業務ルール違反を表す例外。「想定範囲内」のエラー。
例：在庫不足、権限不足、重複登録、必須項目未入力、業務上の不整合
ユーザー操作に起因することが多い。
システム例外（System Exception）
システム的なトラブルを表す例外。「想定外」のエラー。
例：DBダウン、ネットワーク切断、ファイルアクセス失敗、メモリ不足
環境・インフラに起因することが多い。
扱い方の違い
業務例外：
・ユーザーに分かりやすいメッセージを見せる
・「あなたの操作を見直してください」と促す
・ログレベル：WARN または INFO

システム例外：
・「システムエラーが発生しました」と詳細を隠す（セキュリティ）
・管理者・運用チームに通知する
・ログレベル：ERROR（詳細スタックトレース付き）
Vol.9 での例
業務例外：TaskNotFoundException（存在しないタスクへのアクセス）
システム例外：NullPointerException、SQLException（DB接続失敗）等
→ GlobalExceptionHandler で404と500に振り分け
本書のフォールバック
@ExceptionHandler(Exception.class) が未分類の全例外を捕まえる。これは「最後の砦」であり、業務的に分類できなかった想定外を全部システム例外扱いにする発想。
business-vs-system 2つの例外の振り分け方
// 業界での扱い方の差

業務例外への対応

  • ユーザーに丁寧にメッセージ：「在庫が不足しています」
  • 具体的な解決策を示す：「数量を減らしてください」
  • 該当画面に戻す：操作を訂正できるように
  • ログレベルは WARN：頻発するけど致命的ではない

システム例外への対応

  • 抽象的なメッセージ：「システムエラーが発生しました」
  • 内部情報を隠す：スタックトレース等は絶対にユーザーに見せない（セキュリティ）
  • 管理者に通知：メール、Slack、監視システム経由
  • ログレベルは ERROR：詳細スタックトレース全部記録

振り分けの実装：

@ControllerAdvice
public class GlobalExceptionHandler {

  // 業務例外
  @ExceptionHandler(BusinessException.class)
  public String handleBusiness(...) {
    return "error/business";  // 業務向けメッセージ
  }

  // システム例外（フォールバック）
  @ExceptionHandler(Exception.class)
  public String handleSystem(...) {
    log.error(...);  // 詳細記録
    return "error/500";  // 抽象的メッセージ
  }
}
// 業務例外の典型 // === 在庫不足 === public class InsufficientStockException extends RuntimeException { ... } // 投げる場所: if (stock < quantity) { throw new InsufficientStockException(...); } // ユーザーへの表示: // 「在庫が不足しています」 // 「数量を減らしてください」 // === 権限不足 === public class UnauthorizedException extends RuntimeException {} if (!user.hasRole("ADMIN")) { throw new UnauthorizedException(); } // ユーザーへの表示: // 「この操作には権限が必要です」 // システム例外の典型 // === DB接続失敗 === // Spring が DataAccessException として // 包んで投げる（自動） // この例外は… // ・特定のハンドラーで受けない // ・@ExceptionHandler(Exception.class) が受ける // → 500画面 // ユーザーへの表示: // 「システムエラーが発生しました」 // （DB接続失敗とは絶対に書かない） // === NullPointerException === // 開発者のミス由来 // 同じく 500 画面に // 「予期しないエラー」と表示 // 業務例外の判断基準 // 「これは業務的に起こりうる?」 // ・YES → 業務例外 // ・NO → システム例外 // 例: // ・「在庫切れ」→ 起こりうる → 業務例外 // ・「DB接続不可」→ 起こらないはず // → システム例外 // ・「存在しないIDアクセス」→ 起こりうる // → 業務例外（Vol.9） // 例外クラスの分け方（業務的） com.example.todo.exception/ ├── BusinessException ← 抽象親 │ ├── TaskNotFoundException ★Vol.9 │ ├── InsufficientStockException │ └── UnauthorizedException └── (システム例外は標準のものを使う) // 共通の親クラスを作っておくと // @ExceptionHandler(BusinessException.class) // で一括処理できる
POINT ── 「振る舞いで分類する」

業務例外とシステム例外の区別の意義は「振る舞い」にあります。業務例外はユーザーの操作で起きるので丁寧に説明。システム例外はインフラ起因なので詳細を隠して管理者に通知。同じ「例外」でも扱い方が180度違う。Vol.9のGlobalExceptionHandlerの2つのメソッドはこの分類を実装したものです。

05

例外の責任の所在（誰が投げて誰が掴む?）

RESPONSIBILITY — WHO THROWS, WHO CATCHES

例外設計の核心問題。「どの層で投げて、どの層で受けるか?」── これを間違えると、3層アーキテクチャの分離が崩壊します。正しい場所を把握しましょう。

throw-catch-layers 層ごとの責任分担
// 各層の責任
Mapper層：投げる側（業務例外は投げない）
DBアクセスでの技術的な例外は投げるが（自動）、業務的な例外は投げない。「レコードが0件」は事実をそのまま返す（null や 空Listを返す）。
業務判断は Service層に委ねる。
Service層：業務例外を投げる場所
業務ルール違反・業務的な不整合を見つけたら、独自例外を投げる。「業務として続行できない」という判断はここで下す。
Vol.9で findById に null チェック + throw を追加したのはこれ。
Controller層：基本的に素通り
try-catch は普通書かない。Service が投げた例外をそのまま上位へ伝搬させる。Controllerは「業務知識」を持たないので、例外を扱う立場でもない。
GlobalExceptionHandler層：最終的に掴む場所
@ControllerAdvice + @ExceptionHandler ですべての例外を集約。ここで「どの画面を出すか」「どうログに残すか」を決める。

特殊：Service層内でcatchするケース

業務的にリカバーできる例外なら、Service内でcatchして処理を続行することもある。「例外を上に投げるのではなく、代替の値を使う」── これは例外を例外として扱わない判断。

// 各層の典型的な書き方 // === Mapper層（業務例外は投げない） === @Mapper public interface TaskMapper { @Select("SELECT ... WHERE id = #{id}") Task findById(Long id); // 該当なし → null を返す // 例外は投げない // 「DBの事実」を伝えるだけ } // === Service層（業務例外を投げる） === @Service public class TaskService { public Task findById(Long id) { Task task = mapper.findById(id); if (task == null) { // 業務判断：これは業務的に問題 throw new TaskNotFoundException(id); } return task; } } // === Controller層（基本素通り） === @Controller public class TaskController { @GetMapping("/tasks/{id}/edit") public String editForm( @PathVariable Long id, ...) { Task task = service.findById(id); // try-catch しない // 例外は GlobalExceptionHandler が掴む ... } } // === GlobalExceptionHandler === @ControllerAdvice public class GlobalExceptionHandler { @ExceptionHandler(TaskNotFoundException.class) public String handleTaskNotFound(...) { // 例外をここで掴む // 404画面に振り分け return "error/404"; } } // 特殊：Service内でcatchするケース @Service public class NotificationService { public void sendNotification(Task task) { try { emailSender.send(task); } catch (MailException e) { // メール送信失敗 // → 例外を投げない // → ログだけ残す // → 本処理は続行 log.error("通知送信失敗", e); } } } // メール送信失敗で全体を止めない // 例外を「飲み込んで」処理続行 // ※ 適切に使えば有用 // ※ 安易に飲み込むと事故の元
POINT ── 「責任の所在を明確に」

例外設計の鉄則は「責任の所在の明確化」。Mapperは事実を返すだけ、Serviceが業務判断、Controllerは素通り、Handlerが最終処理── この4層の責任を意識すれば、コードがすっきり書けます。「どこに try-catch を書くべきか?」と迷ったら、「この層がこの例外を掴むのは妥当か?」と問い直しましょう。

06

例外メッセージの設計（ユーザー向けと開発者向け）

MESSAGE DESIGN

例外にはメッセージが付きます。Vol.9で "Task not found: id = 9999" としたメッセージ── 実はこれ、ユーザー向けと開発者向けの両方を兼ねていて、本来は分けるべきです。

message-design 2種類のメッセージを分離する
// メッセージの2つの読み手
ユーザー向けメッセージ

画面に表示する。
業務的に分かりやすく。
専門用語を避ける。
解決策を示すのが理想。

例：「指定されたタスクが見つかりません。一覧から選び直してください。」

開発者向けメッセージ

ログに残す。
技術的に正確に。
具体的なID・値を含める。
調査のヒントになる情報。

例："Task not found: id = 9999, attempted by userId=42"

Vol.9 の現状：例外メッセージ "Task not found: id = 9999" をそのまま画面に表示しているので、開発者向けメッセージがユーザーに見えています。これは本来はNG。学習用なのでシンプルにしているだけ。

改善版の設計

例外クラスに「ユーザー向け」「開発者向け」両方のメッセージを持たせる：

public class TaskNotFoundException
    extends RuntimeException {

  private final Long taskId;

  public TaskNotFoundException(Long id) {
    super("Task not found: id = " + id);  // 開発者向け
    this.taskId = id;
  }

  // ユーザー向けは別途定義
  public String getUserMessage() {
    return "指定されたタスクが見つかりません";
  }
}

同じ例外でも、ユーザーと開発者では必要な情報が違う。ユーザーは「何を直せばよいか」、開発者は「何が起きたか正確に」を知りたい。1つのメッセージで両方を満たそうとすると、どちらにも中途半端になります。業務開発ではメッセージを2系統に分けるのが理想です。

try-catch を使うべき場面と使わないべき場面

WHEN TO USE TRY-CATCH

「try-catchはどこに書くべきか?」── 新人が最も悩む例外設計の問い。使うべき場面と使わないべき場面を整理します。

try-catch-when 使う場面と使わない場面
// try-catchを使うべき4つの場面
例外から本当に復帰できる場面
「失敗してもリカバーできる」場合のみ catch。
例：3回までリトライする、別の経路を試す、デフォルト値で続行
意味のあるリカバリーロジックを書ける時だけ catch する価値あり。
例外を意図的に飲み込む場面
「例外が起きても本処理は続行したい」場合。
例：通知メール送信失敗 → 本処理（注文受付）は成功扱い、ログだけ残す
★ 慎重に使う。安易に飲み込むと事故の元。
例外を変換する場面
低レベル例外を業務例外に包んで再スロー。
例：IOException を DocumentReadException でラップ
★ 呼び出し側に「業務的な意味」を伝えるためにラップする。
例外をログに残す前提で再スローする場面
catch → log.error(...) → 再スロー。
「ここを通った記録を残したい」が、処理は止める時。
※ GlobalExceptionHandler でログを書くなら不要。

try-catchを使わないべき場面

  • 意味なく catch する：「コンパイラに言われたから書く」程度の catch は害悪
  • catch して何もしない（例外を握りつぶす）：バグの元凶
  • catch して同じメッセージで投げ直す：何の付加価値もない
  • Controllerで業務例外を全部 catch する：GlobalExceptionHandler の役割を奪う
// 使うべき場面の例 // === 場面1: リトライ === public void callExternalApi() { for (int i = 0; i < 3; i++) { try { api.call(); return; // 成功 } catch (ApiException e) { log.warn("Retry {}", i); Thread.sleep(1000); } } throw new ApiException("3回失敗"); } // === 場面2: 意図的に飲み込む === public void registerUser(UserForm form) { User user = userService.save(form); // 通知メール送信 try { mailService.sendWelcome(user); } catch (MailException e) { // メール失敗でも登録は成功扱い log.error("メール失敗", e); // 飲み込む } return; // 登録成功 } // === 場面3: 例外の変換 === public Document readDocument(Path p) { try { return reader.read(p); } catch (IOException e) { // 低レベル → 業務的な例外に変換 throw new DocumentReadException( "文書を読めません: " + p, e); // 元の例外を cause に保持（重要） } } // === 使わないべき例 === // ✗ NG: 握りつぶす try { doSomething(); } catch (Exception e) { // 何もしない } // → エラーが見えなくなる // → 最悪のアンチパターン // ✗ NG: 意味なく投げ直す try { doSomething(); } catch (Exception e) { throw new Exception(e.getMessage()); } // → 元の cause を失う // → スタックトレースが切れる // ✗ NG: Controllerで全catch @PostMapping("/tasks") public String create(...) { try { service.create(form); } catch (Exception e) { return "error"; } } // → GlobalExceptionHandlerの役割を奪う // → 例外処理の集約が崩れる // 業務開発の経験則 // try-catchを書く前に問う: // 1. 本当に対応すべきか? // 2. ここで対応する権限/知識があるか? // 3. もっと上位で扱う方が良くないか? // 多くの場合、答えは「上に投げる」 // → GlobalExceptionHandlerに任せる
POINT ── 「try-catch は最後の手段」

新人エンジニアは「例外＝try-catch」と思いがちですが、業務開発ではtry-catch を書かない方が圧倒的に多い。例外を上に投げる方が美しいのが Spring 開発の世界観です。「本当に try-catch が必要か?」と問い直して、不要なら書かないのが上達のコツ。Vol.9 で実装した「Service で投げる、Controller は素通り、Handler で受ける」パターンは、この思想の体現です。

業務例外 vs システム例外でログレベルを変える

業務例外 → WARN レベル

「起こりうるエラー」なので、警告レベル。
・スタックトレースは不要（または簡潔）
・「何が」「誰が」に絞った情報

システム例外 → ERROR レベル

「想定外」なので、エラーレベル。
・スタックトレース全部記録
・調査に必要な情報を全部残す

自分で例外設計するためのチェックリスト

本補足の集大成。新しい業務例外を設計する時のチェックリスト。明日から使える実務テンプレートです。

// 設計時の判断項目

事前判断（設計）

  • この異常は業務例外かシステム例外か（業務的に起こりうるか）
  • 非検査例外（RuntimeException）にする予定か（業界推奨）
  • 例外名は業務的に分かりやすい名前か（〜Exception で終わる）
  • 例外にどの情報を持たせるか（ID、コード、現状値等）
  • ユーザー向けメッセージと開発者向けメッセージは分離するか
  • どの層で投げるか（基本は Service層）
  • どの層で受けるか（基本は GlobalExceptionHandler）
  • どのエラー画面に対応させるか

実装

  • com.example.todo.exception パッケージに例外クラスを作る
  • RuntimeException を継承する
  • コンストラクタで必要な情報を受け取る
  • super() でメッセージを設定する
  • Service層で適切なタイミングで throw する
  • Controller にはtry-catch を書かない（GlobalExceptionHandler に任せる）

  • 対応するエラー画面テンプレートを作成する

  • 該当する異常ケースで例外が発生する

  • 対応するエラー画面が表示される
  • 画面に業務的なメッセージが表示される（技術詳細は隠す）
  • 「次のアクション」が画面に提示される
  • コンソールに適切なログが出力される
  • Whitelabel Error Page が出ない
