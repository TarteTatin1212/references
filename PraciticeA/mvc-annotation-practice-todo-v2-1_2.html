📚 必要な前提知識

PREREQUISITES

本編で学んだ内容のうち、Vol.1に直接関わる部分を軽くおさらいします。さらにv2で初登場の「JarとWarの違い」もここで押さえます。

key-concepts 本編からの復習 ＋ v2の新概念
// 知っておきたい5つ
CONCEPT 1
Spring Boot とは
「設定済みのSpring」。本来は複雑な設定が必要なところを、賢いデフォルトで埋めてくれる仕組み。
CONCEPT 2
Spring Tools 4
Eclipseに入れるSpring専用拡張。Eclipse 2025系には標準で同梱されていることが多い。
CONCEPT 3
@RestController
「このクラスはWeb受付で、戻り値をそのままレスポンスに」という付箋。動作確認の小さなAPIに使う。
CONCEPT 4
@GetMapping
「GETリクエストが来たらこのメソッドを呼ぶ」。URLとメソッドを結びつける付箋。
★ v2新概念
JarとWarの違い
Jar＝Tomcatを内蔵した実行ファイル（java -jarで動く）。War＝外部のTomcatに配備するファイル。業務では今もWarが多い。両方とも開発中はRun As → Spring Boot Appで動作。
// なぜWar形式を選ぶのか // ── Jar形式（旧版Aで採用） ── - 内蔵Tomcatで動く - java -jar todo.jar で起動 - マイクロサービス・クラウド向き - スタートアップ・新規開発で人気 // ── War形式（v2で採用） ── - 外部Tomcatに配備可能 - todo.war ファイルを Tomcat に投入 - SI現場・既存システム連携で定番 - 業務開発で圧倒的多数派 // ※ 開発中は同じ // Jar も War も： // - Eclipseで Run As → Spring Boot App // - 内蔵Tomcatで起動 → localhost:8080 // - DevToolsの自動再起動も同じ // 違いは「ビルド成果物の形」だけ。 // 業務開発に近い形を体験するため、 // v2ではWarを採用しました。 // pom.xmlでの違い // Jar の場合： <packaging>jar</packaging> // or 省略 // War の場合： <packaging>war</packaging> // + 内蔵Tomcatの依存を「provided」に： <dependency> <groupId>org.springframework.boot</groupId> <artifactId> spring-boot-starter-tomcat </artifactId> <scope>provided</scope> </dependency> // → ウィザードでWarを選べば // 自動でこれらが設定される。
POINT ──

「マイクロサービス/クラウドネイティブ＝Jar」「SI/エンタープライズ＝War」がざっくりした使い分け。本書はSI現場での業務開発を想定しているのでWarを選びますが、どちらか優劣はありません。両方使えると業務で困らないのが現実です。

// War版の特徴

Packaging=Warを選ぶと、Eclipseが自動で：

  • SpringBootServletInitializerを継承させる
  • configure()メソッドをオーバーライド
  • 外部Tomcat配備に備える

Jar版との違い：

Jar版（旧A）：
public class TodoApplication {
War版（v2）：
public class TodoApplication
  extends SpringBootServletInitializer {
大事な点：開発中はどちらも同じ動き。Run As → Spring Boot Appで内蔵Tomcatが起動します。
// TodoApplication.java（War版・自動生成） package com.example.todo; import org.springframework.boot.SpringApplication; import org.springframework.boot.autoconfigure .SpringBootApplication; import org.springframework.boot.builder .SpringApplicationBuilder; // ★ import org.springframework.boot.web.servlet .support.SpringBootServletInitializer; // ★ @SpringBootApplication public class TodoApplication extends SpringBootServletInitializer { // ★ // 外部Tomcat配備時の設定 @Override protected SpringApplicationBuilder configure(SpringApplicationBuilder app) { return app.sources( TodoApplication.class); } // 開発中の起動（内蔵Tomcat） public static void main(String[] args) { SpringApplication.run( TodoApplication.class, args); } } // ───────────────────── // このクラスは自動生成されたまま // 触る必要はありません。 // // SpringBootServletInitializer により // 外部Tomcat配備時にも // Springが正しく起動できる仕組み。
POINT ──

多くの依存に<version>が無いのに気づきましたか？　これはspring-boot-starter-parentがバージョン管理してくれているから。これは超便利。ライブラリの互換性に悩まされない、現代Spring Bootの大発明です。MyBatisだけはparentで管理されないので、明示しています。

🛠️ Lombokをセットアップ

SETUP LOMBOK FOR ECLIPSE

プロジェクトができたら、次はLombokのセットアップ。pom.xmlに依存を入れただけではEclipseのエディタが認識しません。これがプロジェクト作成時の最大のハマりどころ。

lombok-install 公式インストーラの実行
// インストール手順
1
FIND JAR
Mavenが既にDLしたlombok.jarを使うのが手軽。場所：%USERPROFILE%\.m2\repository\org\projectlombok\lombok\1.18.38\lombok-1.18.38.jar
2
RUN INSTALLER
そのjarをダブルクリックするとGUIインストーラが起動（出ない場合はjava -jar lombok-1.18.38.jar）
3
SELECT IDE
EclipseまたはSpring Toolsを自動検出。リストになければ「Specify location」でEclipseのインストールフォルダを指定
4
INSTALL
「Install / Update」をクリック → 「Quit Installer」で閉じる
5
RESTART
Eclipseを完全に再起動（重要：再起動しないと反映されない）
6
VERIFY
Help → About Eclipse の末尾に「Lombok v1.18.xx ...」が表示されれば成功
// インストーラが行うこと // eclipse.ini の末尾に追記される行： -javaagent:lombok.jar // ───────────────────── // これがLombokを動かす実体。 // JVMを起動するときにLombokを // エージェントとして注入する設定。 // 動作確認の方法 // 1. Help → About Eclipse → Installation Details // 末尾に「Lombok v1.18.38 ...」が表示されること // 2. 後でTaskクラスを作った時に // @Data を付けたフィールドの // getter/setterが「未定義」と // 赤線にならないこと // 3. Outlineビューに自動生成された // getXxx() / setXxx() が見える
POINT ──

Lombokは「コンパイル時にコードを注入する」という珍しい仕組みで動いています。だから専用インストーラでEclipse本体に組み込む必要があります。pom.xmlの依存追加だけでは不十分、というのが初学者が必ず最初にハマるポイント。

📂 クラス作成の具体手順

CREATING CLASSES IN ECLIPSE

v2の新セクションです。「Eclipseでクラスを作る」という基本動作を、毎巻必ず確認します。新規クラス・パッケージ・インターフェイスの作り方を覚えれば、迷うことが減ります。

class-creation Eclipseでのクラス作成の基本動作
// 3つのパターンを覚える
A. 新しいパッケージを作る
  1. Package Explorerでcom.example.todoを右クリック
  2. New → Package を選択
  3. Name欄にcom.example.todo.controllerと入力（フルパス）
  4. Finish をクリック
B. 新しいクラスを作る
  1. 作成したいパッケージを右クリック（例: controller）
  2. New → Class を選択
  3. Name欄にHelloControllerと入力（最初は大文字）
  4. その他のチェックは全部外す（mainメソッド等は不要）
  5. Finish をクリック
  6. 作成されたファイルにコードを書く
C. 新しいインターフェイスを作る（Vol.4で登場）
  1. パッケージを右クリック
  2. New → Interface を選択（Classじゃない！）
  3. Name欄にインターフェイス名を入力（例: TaskMapper）
  4. Finish をクリック
// よく出る疑問：なぜインターフェイス？ // クラスとインターフェイスの使い分け // ── クラス（class）── // 「実体を持つもの」を表現 // // 例: TaskController // - メソッドの中身を自分で書く // - フィールドを持てる // - newして使う public class HelloController { public String hello() { return "Hello"; // 実装あり } } // ── インターフェイス（interface）── // 「メソッドの形だけ」を宣言 // // 例: TaskMapper（Vol.4で登場） // - メソッドの中身は書かない // - SQLというヒントだけ与える // - 実装はMyBatisが自動生成 public interface TaskMapper { @Select("SELECT * FROM tasks") List<Task> findAll(); // 実装なし } // MyBatisが裏で： // public class TaskMapperImpl // implements TaskMapper { // ...SQLを実行する実装... // } // を自動生成してくれる。 // 本書での使い分け早見表 クラス: - Controller（@Controller等） - Service（@Service） - Entity（@Data付きデータクラス） - 起動クラス（@SpringBootApplication） インターフェイス: - Mapper（@Mapper） - ※ ほぼMapperだけ覚えればOK // ───────────────────── // 迷ったら： // 「メソッドの中身を自分で書く」→ class // 「SQLだけ書いて中身は任せたい」→ interface
POINT ──

本書では「Mapperだけがインターフェイス」と覚えてOK。それ以外（Controller/Service/Entity）は全部クラスです。「自分で実装書く＝class」「SQLだけ宣言＝interface」と理解すれば迷いません。Vol.4で実際に作ります。

✏️ HelloControllerを書く

WRITE HELLO CONTROLLER

クラス作成の手順がわかったら、いよいよコードを書きます。動作確認用の最小のController。Section 6の「B. 新しいクラスを作る」手順をそのまま実行してみてください。

HelloController.java 動作確認用の最小Controller
// 作成手順（Section 6を参照）
1
PACKAGE
A手順でcom.example.todo.controllerを作成
2
CLASS
B手順でHelloController.javaを作成
3
WRITE
右のコードをそのまま書く（コピペでOK）
4
SAVE
Ctrl+Sで保存

コードの読み方：

  • @RestController ── クラスにつける付箋（Web受付＋JSON返却）
  • @GetMapping("/") ── このメソッドはGET /で呼ばれる
  • 戻り値の文字列がそのままレスポンスボディになる
v2の方針：import文を全部書きます。コピペすれば赤線が出ないように、必要なimportは省略せず明示します。
// HelloController.java（完成版・import全明示） package com.example.todo.controller; // ↓ import文は省略せず全部書きます import org.springframework.web.bind .annotation.GetMapping; import org.springframework.web.bind .annotation.RestController; @RestController public class HelloController { @GetMapping("/") public String hello() { return "Hello, TODO!"; } } // ───────────────────── // たった10行。これが // 「動くSpring Bootアプリ」の最小形。 // 自動生成されているTodoApplication.java（War版） package com.example.todo; import org.springframework.boot.SpringApplication; import org.springframework.boot.autoconfigure .SpringBootApplication; import org.springframework.boot.builder .SpringApplicationBuilder; import org.springframework.boot.web.servlet .support.SpringBootServletInitializer; @SpringBootApplication public class TodoApplication extends SpringBootServletInitializer { @Override protected SpringApplicationBuilder configure(SpringApplicationBuilder app) { return app.sources( TodoApplication.class); } public static void main(String[] args) { SpringApplication.run( TodoApplication.class, args); } } // ↑ こちらは触らない // ↑ Spring Bootの起動エンジン（War版）
POINT ──

TodoApplicationは触らずに済ませるのが基本。@SpringBootApplicationの中で@ComponentScanが動き、同じパッケージ以下を自動スキャンします。だからcontrollerパッケージは必ずcom.example.todoの下に置くこと。

🚀 ブラウザで動かして確認

RUN AND VERIFY IN BROWSER

v2の新セクションです。各巻の終盤に必ず「ブラウザで動かして確認」を入れます。「コードを書いただけで終わらせない」── これがv2の最大の改善です。

run-and-check 起動と動作確認の手順
// 起動の流れ
起動手順
  1. Package ExplorerでTodoApplication.javaを右クリック
  2. Run As → Spring Boot App を選択
  3. コンソールにStarted TodoApplication in N secondsが出るのを待つ
  4. ブラウザで http://localhost:8080/ を開く
  5. 「Hello, TODO!」が表示されることを確認

確認ポイント：

  • コンソールにERROR（赤）が出ていない
  • "Started TodoApplication in N seconds" が出る
  • ブラウザで Hello, TODO! が表示される

起動を止めるには：

  • コンソールビューの赤い四角ボタン（Terminate）をクリック
  • またはCtrl+F2（Eclipse標準）
✅ ここまで動けばVol.1クリア！　War形式なのに「外部Tomcatを用意してない」と心配しなくて大丈夫。内蔵Tomcatで動いてます。

🚨 ハマりどころと対処

COMMON PITFALLS

Vol.1で初学者がほぼ確実にぶつかる5つの罠を紹介します。事前に知っておけば、慌てずに対処できます。「ハマる前提で構える」のが業務開発のメンタリティ。

common-pitfalls 5大トラブルと対処法
// よくあるエラー一覧
PITFALL 1
Lombok未インストール
pom.xmlに入れただけでは動かない。Eclipseで@Dataを付けた瞬間、getter/setterが「未定義」と赤線になる。Section 5の手順を完了させること。
PITFALL 2
Javaバージョン不一致
UnsupportedClassVersionError。Window→Preferences→Java→Installed JREsでJDK 17を指定。
PITFALL 3
ポート8080使用中
Port 8080 was already in use。前回の起動が残っているか、他アプリが占有。Eclipseのコンソールで赤い四角でstopするか、application.propertiesにserver.port=8081。
PITFALL 4
パッケージ階層がズレてる
HelloControllerをcom.example.todo.controllerの下に置かないと検出されない。TodoApplicationより上の階層に置かない。
PITFALL 5
v2新規：Warなのに起動しない
scope=providedのTomcatが効きすぎてMVNからの依存解決に問題が出ることが稀にある。プロジェクト右クリック→Maven→Update Project（Alt+F5）で大体治る。

Eclipseで「赤い波線」が消えない時は、プロジェクト右クリック→Maven → Update Project（Alt+F5）かProject → Clean...でだいたい治ります。困ったらCleanとUpdate。これが詰まりポイントの万能薬です。
