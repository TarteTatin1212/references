03

用語カード：Service層

TERM CARD — SERVICE LAYER

本日の主役Service層。深い「なぜ」は別冊に譲り、本編では「役割と居場所」を最小限に押さえます。

TERM CARD #01
Service層
業務ロジックを置く層
定義
業務ロジック（ビジネスロジック）を置くための層。Controllerと Mapper の間に位置し、両者を仲介する。
構造
普通のクラスに@Serviceを付ける。フィールドにMapperを持ち、コンストラクタで受け取る。メソッドで「業務上の操作」を提供する（例：findAll()、register(Task)等）。
対比
Controller ── HTTPの世界（URL、HTTPメソッド、画面）
Service ── 業務の世界（タスクを取得する／登録する／検証する等）
Mapper ── DBの世界（SQL、テーブル）
Serviceは「業務語彙でメソッドを並べる場所」。
嬉しさ
① 業務ロジックの居場所が明確になる
② Vol.5以降の登録・更新・バリデーションを書く時にここに集まる
③ テストしやすい（Controllerを通さずService単体で検証可能）
④ 業界標準の構造になる
◯ Vol.3まで：Service無し
Controller → Mapper → DB

・URLを受ける場所(Controller)が
　Mapperを直接呼ぶ
・業務ロジックを書く場所がない
・登録機能を作る時、「重複チェック」
　などのロジックを置く場所に困る
◎ Vol.4以降：Service有り
Controller → Service → Mapper → DB

・業務語彙でService.find(),
　Service.register() のメソッドを並べる
・複雑な処理(重複チェック等)も
　Serviceに自然に書ける
・業界標準の形になる
POINT ──

「今はService層が薄い（mapperを呼ぶだけ）」と感じても問題ありません。Vol.5以降、登録機能や検証ロジックが入ると、Serviceの中身が太っていくのが体感できます。「先に居場所を作っておく」のが今巻の主旨です。深い理由は別冊で。

04

用語カード：@Service

TERM CARD — @SERVICE

Service層クラスに付けるアノテーション。@Controllerや@Mapperと同じ仲間で、Springに「これはService層のBeanです」と伝える目印です。

TERM CARD #02
@Service
「これはService層」をSpringに宣言
定義
Springのアノテーション。クラスに付けると「Service層のBeanとしてDIコンテナに登録せよ」とSpringに命令する。
結果
起動時、Springが@Serviceを見つけてインスタンスを生成しDIコンテナに登録。他のクラス（Controller等）がコンストラクタで受け取れるようになる。
仲間
@Controller ── Web受付
@RestController ── REST API受付
@Service ── 業務ロジック
@Repository ── DBアクセス
@Component ── その他のBean
全部同じ仲間（DIコンテナに登録される）だが、役割を明示するため使い分ける。
嬉しさ
① クラスを見ただけで役割が分かる
② 将来的に役割別の処理（ロギング・例外変換等）を一括で適用しやすい
③ 業務標準で誰もがこのパターンで書く
POINT ──

@Serviceと@Componentは機能的にはほぼ同じです。「@Serviceを使う＝Service層」という意味的なラベルとして扱われます。Mapperの@Mapperは特殊（MyBatisが動的プロキシで実装を作る）ですが、@Serviceは普通のクラスをDIコンテナに登録するだけ。シンプルです。

05

用語カード：@Transactional

TERM CARD — @TRANSACTIONAL

トランザクション境界を宣言するアノテーション。Vol.4の段階では読み取り専用なので「ありがたみ」は薄いですが、Vol.5以降の登録・更新で真価を発揮します。

TERM CARD #03
@Transactional
トランザクション境界の宣言
定義
メソッド（またはクラス）に付けると、その処理全体を1つのトランザクションとして扱う。途中で例外が起きたら自動的にロールバック（DB変更を巻き戻し）。
構造
クラスに付けると全メソッドに適用。メソッドに付けるとそのメソッドだけ。属性で挙動を変えられる：
readOnly = true ── 読み取り専用（軽量モード）
readOnly = false（デフォルト）── 読み書き両用
嬉しさ
① 「途中で失敗したらDBは元の状態」が保証される
② 複数のINSERT/UPDATEを全部成功 or 全部失敗にできる（原子性）
③ アノテーション1個で済む（自分でcommit/rollbackを書かなくて良い）
Vol.4での使い方
クラスに@Transactional(readOnly = true)を付け、クラス全体を読み取り専用トランザクションにする。Vol.5で書き込みメソッドが増えた時、そのメソッドだけ@Transactional（書き込み有効）に上書きする定番パターン。
why-readonly なぜ最初はreadOnly = true？
// readOnly = true の意味

readOnly = trueは「このメソッドはDBを変更しません」とフレームワークに伝える宣言です。これによりSpring/MyBatis/JDBCドライバが軽量モードで動き、余計な書き込み準備をスキップします。

業務開発の定番パターン：

  1. クラスに@Transactional(readOnly = true)を付ける
  2. 書き込みメソッドだけ@Transactional（readOnly無し）で上書き

この形なら「うっかり読み取りメソッドからDBを書き換える」事故を防げます。書き込みメソッドだけ明示的にreadOnly=falseになるので、開発者が「ここは書き込み」と意識できる。

// Vol.4: 全メソッドが読み取り @Service @Transactional(readOnly = true) public class TaskService { // 読み取り専用モード public List<Task> findAll() { return taskMapper.findAll(); } } // Vol.5以降のイメージ（先取り） @Service @Transactional(readOnly = true) public class TaskService { // 読み取り（クラス指定が効く） public List<Task> findAll() { ... } // 書き込み（メソッド指定で上書き） @Transactional public void register(Task task) { taskMapper.insert(task); } } // この「全体readOnly + 必要箇所書き込み」 // が業務の鉄板パターン
POINT ──

Vol.4で@Transactionalを付ける本当のメリットはVol.5以降に表れます。今は「とりあえずService層は@Transactionalを付けるもの」と覚えれば十分です。深い動作原理（プロキシによる自動コミット/ロールバック）は別冊で扱います。

// src/main/java/com/example/todo/service/TaskService.java package com.example.todo.service; import java.util.List; import org.springframework.stereotype.Service; import org.springframework.transaction .annotation.Transactional; import com.example.todo.entity.Task; import com.example.todo.mapper.TaskMapper; @Service @Transactional(readOnly = true) public class TaskService { private final TaskMapper taskMapper; // コンストラクタ注入（Vol.3と同じ作法） public TaskService(TaskMapper taskMapper) { this.taskMapper = taskMapper; } public List<Task> findAll() { return taskMapper.findAll(); } } // 構造を読み解く // ① クラス宣言 @Service // DI登録 = Service層 @Transactional(...) // 全メソッドが読み取り public class TaskService { ... } // ② 依存（Mapper）の保持 private final TaskMapper taskMapper; // final ＝ 後から差し替え不可 // ③ コンストラクタで受け取り public TaskService(TaskMapper tm) { this.taskMapper = tm; } // Vol.3のControllerと全く同じ作法 // ④ 業務メソッド public List<Task> findAll() { return taskMapper.findAll(); } // 今は単に委譲するだけ // Vol.5以降で「業務ロジック」が入る
POINT ── 「Vol.3と同じ作法」

TaskServiceの構造（final フィールド + コンストラクタ注入）は、Vol.3のTaskControllerと全く同じ作法です。これがSpringのDI（依存注入）の基本テンプレート── 一度覚えれば、Controller、Service、Mapper、その他あらゆるBean間の依存関係が同じ書き方で組めます。パターンの繰り返しこそが業務開発の基礎体力です。

// 4箇所の変更点
変更内容

  2. フィールドの型：TaskMapper → TaskService
  3. コンストラクタの引数：TaskMapper → TaskService

// 修正前（Vol.3のTaskController.java） package com.example.todo.controller; import java.util.List; import org.springframework.stereotype .Controller; import org.springframework.ui.Model; import org.springframework.web.bind .annotation.GetMapping; import com.example.todo.entity.Task; import com.example.todo.mapper.TaskMapper; // ← @Controller public class TaskController { private final TaskMapper taskMapper; // ← public TaskController( TaskMapper taskMapper) { // ← this.taskMapper = taskMapper; } @GetMapping("/tasks") public String list(Model model) { List<Task> tasks = taskMapper.findAll(); // ← model.addAttribute("tasks", tasks); return "tasks/list"; } } // 修正後（Vol.4のTaskController.java） package com.example.todo.controller; import java.util.List; import org.springframework.stereotype .Controller; import org.springframework.ui.Model; import org.springframework.web.bind .annotation.GetMapping; import com.example.todo.entity.Task; import com.example.todo.service.TaskService; // ★変更 @Controller public class TaskController { private final TaskService taskService; // ★変更 public TaskController( TaskService taskService) { // ★変更 this.taskService = taskService; } @GetMapping("/tasks") public String list(Model model) { List<Task> tasks = taskService.findAll(); // ★変更 model.addAttribute("tasks", tasks); return "tasks/list"; } } // 残りは全部Vol.3と同じ
POINT ── Controllerは「Mapperを知らない」

修正後のTaskControllerにはTaskMapperのimport文がありません。Controllerは「ServiceがDBを扱う詳細を知らない」状態になりました。これが「責務の分離」の具体的な形── Controllerは「HTTPの世界」、Serviceが「業務の世界」、Mapperが「DBの世界」と、知識の領域が分かれたのです。

08

3層構成での処理フロー

3-LAYER FLOW

変更後の/tasksへのアクセスは、Controller → Service → Mapper → DB → 戻り、と3層を往復します。流れを図と時系列で押さえます。

request-flow リクエストから画面表示まで
// 処理の流れ
[ブラウザ] ↓ GET /tasks 【Controller層】 TaskController.list(model) ↓ taskService.findAll() ↓ 【Service層】 ← ★Vol.4で追加★ TaskService.findAll() ↓ taskMapper.findAll() ↓ (@Transactional 開始) ↓ 【Mapper層】 TaskMapper.findAll() ↓ @Select の SQL を発行 ↓ 【MySQL】 SELECT ... FROM tasks ... ↑ 結果3行 【Mapper層】 ↑ List<Task> に変換 【Service層】 ↑ そのまま返す ↑ (@Transactional コミット) 【Controller層】 ↑ Model.addAttribute ↑ return "tasks/list" [Thymeleaf] ↓ list.html を描画 [ブラウザ] 表組み表示
// 各層の責務（あなたが書いたコード） // ① Controller層 @GetMapping("/tasks") public String list(Model model) { List<Task> tasks = taskService.findAll(); // 依頼 model.addAttribute("tasks", tasks); return "tasks/list"; // 画面指定 } // HTTPの世界に専念 // ② Service層 @Transactional(readOnly = true) public List<Task> findAll() { return taskMapper.findAll(); } // 業務の世界（今はシンプル） // ③ Mapper層 @Select("SELECT id, title, done, " + "created_at FROM tasks " + "ORDER BY id") List<Task> findAll(); // DBの世界（SQL） // 各層が知っていること // Controller // 知る: TaskService, Model, URL // 知らない: SQL, TaskMapper // Service // 知る: TaskMapper, トランザクション // 知らない: HTTP, Model // Mapper // 知る: SQL, テーブル構造 // 知らない: HTTP, 業務ロジック // ★これが「責務の分離」の具体形 // 各層が必要なものだけを知っている
POINT ──

3層構成の本質は「各層が知らなくていいことを知らないでいい」こと。ControllerはSQLを知らない、ServiceはHTTPを知らない、MapperはModelを知らない。知識が局所化することで、修正が他層に波及しにくくなり、テストもしやすくなります。この感覚は別冊でさらに深掘りします。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

Vol.4で起きやすいエラーを「症状→原因→対処」の因果で整理します。Vol.3 ADDで身につけた3点読み＋9ステップ思考フローを使う場面です。

⚠ ハマり 1 ── No qualifying bean of type 'TaskService'
SYMPTOM アプリ起動時、コンソールに UnsatisfiedDependencyException や No qualifying bean of type 'com.example.todo.service.TaskService' available が出て起動失敗
CAUSE TaskServiceクラスに@Serviceを付け忘れている。Springは@Serviceを見つけてBean登録するので、付いていないとTaskControllerがTaskServiceを要求しても見つからない。
FIX TaskServiceクラス宣言の直前に @Service を追加。import org.springframework.stereotype.Service; も忘れずに。保存すれば自動再起動して復旧。
⚠ ハマり 2 ── パッケージが com.example.todo の下に無い
SYMPTOM @Serviceは付けているのに、ハマり1と同じ No qualifying bean エラーが出る
CAUSE パッケージを com.example.todoservice（todoとserviceがくっつく）や com.example.service（todoが抜ける）のように作ってしまった。@ComponentScanの探索範囲はcom.example.todoの下なので、範囲外のクラスは見つからない。
FIX パッケージ名を確認し、com.example.todo.service（ピリオド区切り、todoの下にservice）になっていることを確認。違う場合は正しい場所に作り直す。
⚠ ハマり 3 ── @Transactionalのimportが違う
SYMPTOM コンパイルは通るが、トランザクション機能が効かない／Vol.5以降で書き込み処理を入れた時に巻き戻しが効かない
CAUSE import jakarta.transaction.Transactional; を選んでしまった（Vol.3 ADDで学んだ「同名異クラスの罠」）。Springの@Transactionalとは別物で、Spring経由のトランザクション管理は機能しない。
FIX import org.springframework.transaction.annotation.Transactional; に修正。パッケージ名が org.springframework で始まっているかを必ず確認する。
⚠ ハマり 4 ── TaskControllerでTaskMapperのimportが残る
SYMPTOM 動作はするが、TaskControllerに使われていないimport文（TaskMapper）が残っており、エディタに黄色の警告が出る
CAUSE Vol.3でControllerに書いたimport com.example.todo.mapper.TaskMapper; を消し忘れた。コードは動くが無駄な依存が残った状態。
FIX 該当のimport行を手動で削除するか、Ctrl+Shift+O で未使用importを自動削除。クリーンに整える。
⚠ ハマり 5 ── 「動かないのに見た目は同じ」と勘違い
SYMPTOM ブラウザを再読み込みしても変化なし。「動いてる」と勘違い。
CAUSE DevToolsが再起動していない（保存ミスや、再起動完了前に再読み込み）。古いコードで動いているだけで、新Service層は動いていない。
FIX コンソールでStarted TodoApplicationのログを必ず確認してから、ブラウザをF5で再読み込み。再起動完了の証拠を見届ける癖をつける。
