01

この巻の地図

WHAT / HOW / WHY

編集機能の流れを把握しましょう。Vol.5の登録機能とそっくり同じパターンに、「既存値の事前ロード」と「URLからのID取得」が加わるだけです。

WHAT
何を作るか
編集機能の一連の流れ。一覧の各行の「編集」リンク → 編集画面（既存値が表示済み） → 値を変更して「更新」ボタン → DB更新 → 一覧へ自動遷移。
HOW
どう作るか
① TaskEditFormクラスを作る（id・title・done）
② Mapperに@Select byIdと@Updateを追加
③ ServiceにfindById()とupdate()を追加
④ Controllerに編集GETと更新POSTを追加（@PathVariable）
⑤ edit.htmlを新規作成、list.htmlに「編集」リンク追加
WHY
なぜそうするか
編集は業務開発で最も頻繁に作る機能。一度パターンを掴めば応用が利く。URLにIDを含める設計と既存値を埋めたフォーム表示は、RESTful設計の中核思想です（深掘りは別冊）。

03

用語カード：@PathVariable

TERM — @PATHVARIABLE

Vol.6で初登場する@PathVariable。「URLの一部を引数として受け取る」アノテーションです。/tasks/2/edit の「2」をJavaの変数に取り出すための仕組み。

TERM CARD #01
@PathVariable
URLパス変数の値を引数で受け取る
定義
Springのアノテーション。@GetMapping("/tasks/{id}/edit") のように URLパスに置いた{name} の部分を、メソッド引数にバインドする。
仕組み
SpringがURLを照合する時、{id} の位置に来た値（例：2）を抽出し、引数の型に自動変換して渡す。Long型なら数値に、String型なら文字列に。
書き方
@GetMapping("/tasks/{id}/edit") と@PathVariable Long id の 名前を一致させるのが基本。違う名前にしたい時は @PathVariable("id") Long taskId と明示。
使いどころ
① 編集画面（どのレコードを編集するか）
② 削除処理（どのレコードを削除するか）
③ 詳細表示（どのレコードを表示するか）
共通点：「特定のレコードに対する操作」でIDをURLで指定する場面
pathvariable-mechanism URLからJavaへの値の流れ
// パス変数の動き

具体例： ブラウザが GET /tasks/2/edit を送信した時：

  1. Springがルーティング表を引く
  2. /tasks/{id}/edit パターンにマッチ
  3. {id} の位置から "2" を抽出
  4. 引数の型 Long に変換 → 2L
  5. メソッドに渡す
POINT：型変換も自動。{id}に「abc」のような数値変換できない値が来たら400 Bad Requestを自動で返す。

名前の一致が鍵：

@GetMapping("/tasks/{id}/edit")
public String editForm(
    @PathVariable Long id,  // ← 名前一致
    Model m) { ... }

{id} と@PathVariable Long id の名前が同じだから自動で繋がる。違う名前にしたい場合は明示的に指定。

// 基本パターン @GetMapping("/tasks/{id}/edit") public String editForm( @PathVariable Long id, // ← 名前一致 Model model) { // id にURLから取った値が入る // GET /tasks/2/edit なら id = 2L } // 名前を変えたい場合 @GetMapping("/tasks/{id}/edit") public String editForm( @PathVariable("id") Long taskId, Model model) { // taskId にURLから取った値が入る } // 複数のパス変数 @GetMapping("/users/{userId}/tasks/{taskId}") public String view( @PathVariable Long userId, @PathVariable Long taskId) { // 2つとも自動で取り出される } // 型変換も自動 @GetMapping("/tasks/{id}") public String view( @PathVariable Long id) {...} // GET /tasks/abc → 400 Bad Request // （abcはLongに変換不可） // String を使えば任意の値受付け @GetMapping("/tasks/{id}") public String view( @PathVariable String id) {...} // クエリ文字列との対比 // パス変数（推奨、業界標準） GET /tasks/2/edit @PathVariable Long id // クエリ文字列（古い形） GET /tasks/edit?id=2 @RequestParam Long id // RESTful設計ではパス変数が標準 // URLの読みやすさが違う
POINT ── 「URLにIDを含める」のがREST流

業界標準のRESTful URL設計では、「リソースのID」はURLパスに含めるのが鉄則。/tasks/2/edit は「タスクの2番を編集」とURLから読み取れるのが利点。クエリ文字列（?id=2）よりも意味が明確で、ブックマークしやすく、SEOにも良い。深掘りは別冊で。

04

用語カード：@Update

TERM — @UPDATE

Vol.5で覚えた@Insertの仲間。UPDATE文を発行するMyBatisのアノテーションです。書き方はほぼ同じ。

TERM CARD #02
@Update
UPDATE文を発行するMyBatisアノテーション
定義
MyBatisのアノテーション。Mapperインタフェースのメソッドに付けて、UPDATE文の発行を宣言する。@Select・@Insertと同じ作法。
仕組み
メソッドが呼ばれると、MyBatisがSQLの#{プロパティ名}部分を引数オブジェクトの値で置き換え、PreparedStatementで実行。返り値は普通voidかint（影響行数）。
典型例
@Update("UPDATE tasks SET title = #{title}, done = #{done} WHERE id = #{id}")
Task型の引数を取り、そのtitle・done・idを使う。
WHERE句忘れに注意── 忘れると全レコード更新の事故になる。
仲間
@Select ── SELECT文（読み取り）
@Insert ── INSERT文（登録）
@Update ── UPDATE文（更新）
@Delete ── DELETE文（削除、Vol.7で扱う）
update-sql-patterns UPDATE文のパターン
// UPDATEの2つのパターン

パターンA：全項目更新（Vol.6で採用）

UPDATE tasks
SET title = #{title}, done = #{done}
WHERE id = #{id}

編集可能な全項目を一度に更新。シンプル。Vol.6のように編集Formで全項目を扱う場面に向く。

パターンB：部分更新（Vol.7で予告される）

UPDATE tasks
SET done = #{done}
WHERE id = #{id}

特定の項目のみを更新。Vol.7の「完了切替」のように、1項目だけ変えたい場面で使う。

使い分け：両方とも有効な設計。プロジェクトの方針による。深掘りは別冊で。

// @Update の基本 @Update("UPDATE tasks SET " + "title = #{title}, " + "done = #{done} " + "WHERE id = #{id}") void update(Task task); // MyBatis が裏で発行するSQL // Service側で: Task task = new Task(); task.setId(2L); task.setTitle("買い物リスト更新"); task.setDone(true); taskMapper.update(task); // 実際にMySQLに送られるSQL: UPDATE tasks SET title = '買い物リスト更新', done = true WHERE id = 2; // 1行が更新される // ★絶対に忘れてはいけないWHERE句★ // WHERE句なしのUPDATE文… @Update("UPDATE tasks SET done = true") void updateAllDone(); // → 実行すると 全タスクの done が true に // → 通常のレコード単位の更新では // 絶対にWHERE id = #{id} を入れる // 返り値で影響行数を取れる @Update("UPDATE tasks ...") int update(Task task); // 戻り値 = 更新された行数 // 1 → 成功（1行更新） // 0 → 該当レコードなし // 業務上、これでチェックする派もある // Vol.6 では void で OK // シンプルさを優先
POINT ── 「WHERE句忘れの恐怖」

UPDATE文で最も恐ろしいのはWHERE句忘れ。UPDATE tasks SET done = trueを実行すると全レコードの done が trueになる。本番DBでやらかすと取り返しがつかない。だから@Updateを書く時は必ずWHERE句から先に書く癖をつけましょう（実務で広く推奨されるテクニック）。

05

用語カード：編集Formと「既存値の初期表示」

TERM — PRE-FILLED EDIT FORM

編集機能と新規登録の最大の違いは、画面表示時に既存値を入力欄に埋めておくこと。この「事前ロード」がVol.6の核心アイデアです。

TERM CARD #03
編集Formの「既存値の初期表示」
Controller が事前にDB値を Form に詰める
定義
編集画面を表示する時、Controllerが先にDBから現在の値を取得し、Form にコピーしてからModelに載せるパターン。Thymeleafがその Form の値を th:field 経由でinputに表示する。
新規登録との違い
新規登録（Vol.5）：new TaskForm() で空っぽのFormを渡す → 入力欄も空っぽ
編集（Vol.6）：DBから取った値で埋めた Form を渡す → 入力欄に既存値が表示される
必要なメソッド
① MapperにfindById()（1件取得）
② ServiceでTask → TaskEditFormに値を詰め替え
③ ControllerでそれをModelに載せる
④ テンプレートでth:object＋th:fieldを使って表示
嬉しさ
① ユーザーが「今の値が何か」を見ながら変更できる
② 変えたくない項目はそのまま送られる（再入力不要）
③ 誤入力の確認がしやすい
prefill-pattern 事前ロードのコードパターン
// 編集画面表示時のControllerの仕事

4ステップで構成されます：

  1. URLからIDを取り出す（@PathVariable）
  2. DBから既存Taskを取得（taskService.findById()）
  3. Taskの値を新しいTaskEditFormにコピー
  4. Modelに載せて、edit.htmlを返す

この4ステップを終えると、Thymeleafがレンダリング時に th:field="*{title}" を見て value="現在のタイトル" 属性を生成し、inputに既存値が表示される。

// 編集画面表示時のControllerコード @GetMapping("/tasks/{id}/edit") public String editForm( @PathVariable Long id, Model model) { // ① DBから既存Taskを取得 Task task = taskService.findById(id); // ② TaskEditForm を作って // Taskの値を詰める TaskEditForm form = new TaskEditForm(); form.setId(task.getId()); form.setTitle(task.getTitle()); form.setDone(task.getDone()); // ③ Modelに載せる model.addAttribute("taskEditForm", form); // ④ edit.html を返す return "tasks/edit"; } // 結果：edit.htmlがレンダリングされる // テンプレート側で… <input type="text" th:field="*{title}"> // Thymeleafが生成するHTML： <input type="text" id="title" name="title" value="買い物リスト"> // ↑ 既存値が value 属性に // → ブラウザに既存値が表示される // 新規登録との対比 // === Vol.5 新規登録 === @GetMapping("/tasks/new") public String newForm(Model m) { m.addAttribute("taskForm", new TaskForm()); // ↑ 空っぽ return "tasks/new"; } // 入力欄も空で表示 // === Vol.6 編集 === @GetMapping("/tasks/{id}/edit") public String editForm(...) { Task task = taskService.findById(id); TaskEditForm form = new TaskEditForm(); // DBの値を詰める ↓ form.setId(task.getId()); form.setTitle(task.getTitle()); form.setDone(task.getDone()); m.addAttribute("taskEditForm", form); return "tasks/edit"; } // 入力欄に既存値が表示
POINT ── 「Form は表示用と入力用の両刀」

Form クラスは「画面表示用の値の入れ物」と「画面入力受付の容器」の両方を兼ねています。新規登録時は空っぽで作って表示し、編集時は既存値を詰めて表示する── どちらも同じ Form クラスで対応できるのが Thymeleaf + Spring の威力。表示と入力で別クラスを作る必要はないのが省力化のポイントです。

Mapperに findById・update を追加

ADD MAPPER METHODS

既存のTaskMapperに2メソッド追加します。「IDで1件取得」と「更新」── SQLが2つ増えるだけのシンプルな修正です。

add-mapper-methods Mapperを修正する
// 手順
TaskMapper.java を修正
  1. com.example.todo.mapper.TaskMapper.java を開く
  2. 既存の findAll() と insert() の後に findById() と update() を追加
  3. Ctrl+Shift+O でimport補完（@Updateのimport追加）
  4. Ctrl+S で保存

追加するSQLの解読：

findById：

SELECT id, title, done, created_at
FROM tasks
WHERE id = #{id}

findAllにWHERE句を追加した形。#{id}に引数のIDが入る。

update：

UPDATE tasks
SET title = #{title}, done = #{done}
WHERE id = #{id}

created_atは更新対象外。WHERE句は絶対必須。

// 修正後の TaskMapper.java（全体） package com.example.todo.mapper; import java.util.List; import org.apache.ibatis.annotations.Insert; import org.apache.ibatis.annotations.Mapper; import org.apache.ibatis.annotations.Select; import org.apache.ibatis.annotations.Update; // ★追加 import com.example.todo.entity.Task; @Mapper public interface TaskMapper { // 一覧（Vol.3） @Select("SELECT id, title, done, " + "created_at FROM tasks " + "ORDER BY id") List<Task> findAll(); // 登録（Vol.5） @Insert("INSERT INTO tasks " + "(title, done, created_at) " + "VALUES (#{title}, false, NOW())") void insert(Task task); // 1件取得（Vol.6で追加） @Select("SELECT id, title, done, " + "created_at FROM tasks " + "WHERE id = #{id}") Task findById(Long id); // 更新（Vol.6で追加） @Update("UPDATE tasks SET " + "title = #{title}, " + "done = #{done} " + "WHERE id = #{id}") void update(Task task); } // SQLの実行イメージ // findById の動き Task task = mapper.findById(2L); // MyBatisが流すSQL: SELECT id, title, done, created_at FROM tasks WHERE id = 2; // 結果（1行）が Task に変換される // update の動き Task t = new Task(); t.setId(2L); t.setTitle("新タイトル"); t.setDone(true); mapper.update(t); // MyBatisが流すSQL: UPDATE tasks SET title = '新タイトル', done = true WHERE id = 2;
POINT ── 「#{id}」の命名規則

SQL内の#{}プレースホルダの中身は引数オブジェクトのプロパティ名と一致させます。findById(Long id)のように引数が単一の場合、引数名そのままで参照できる（#{id}）。update(Task task)のようにオブジェクトを渡す場合は、そのオブジェクトのプロパティ名（#{title}、#{done}、#{id}）で参照。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

Vol.6のハマりは「動的URL」「checkbox」「@PathVariable」の3軸に集中。Vol.3 ADDの9ステップ思考フローで淡々と解きましょう。

⚠ ハマり 1 ── 編集リンクをクリックすると 404
SYMPTOM 一覧の「編集」リンクをクリックすると 404 Not Found。URLバーは /tasks/{id}/edit（{id}のまま）になっている。
CAUSE list.html の th:href が "/tasks/{id}/edit" のように静的に書かれている。ThymeleafのURL式（@{...}）とパス変数指定 (id=${task.id}) がない。
FIX th:href="@{/tasks/{id}/edit(id=${task.id})}" の形式に修正。@{...}でURL式を開き、(id=...)で動的値を指定する。
⚠ ハマり 2 ── 編集画面で既存値が表示されない
SYMPTOM 編集画面の入力欄が空っぽで表示される。
CAUSE Controllerの editForm() で、findById()から取得したTaskの値をTaskEditFormに詰めるのを忘れている。または new TaskEditForm() をそのままModelに載せている。
FIX form.setId(task.getId())、form.setTitle(task.getTitle())、form.setDone(task.getDone()) の3行を必ず書く。Vol.4 ADDの「Service層のFat / Thin」を意識すれば、こうした詰め替え処理は本来Serviceに置くべきだが、本巻ではControllerで簡潔に。
⚠ ハマり 3 ── 更新後にどのレコードが更新されたか不明
SYMPTOM 更新後、別のレコードが更新されている、またはNullPointerExceptionが起きる。
CAUSE edit.html のhidden inputで id を保持していない。送信時に id が null になり、SQLがWHERE id = NULLを実行（該当0件、エラー）。
FIX edit.html に <input type="hidden" th:field="*{id}"> を入れる。これで送信時に name=id&value=2 が送られ、TaskEditForm.id にセットされる。
⚠ ハマり 4 ── 完了チェックを外しても false にならない
SYMPTOM 編集画面で完了のcheckboxを外して更新ボタンを押しても、done が true のままになる。
CAUSE checkbox を th:field="*{done}" ではなく、name="done" 等で手書きしている。HTMLのcheckboxは外した時に何も送信しない仕様で、Thymeleafの_done仕掛けが効いていない。
FIX checkbox を必ず th:field="*{done}" で書く。Thymeleafが裏で _done hidden を自動生成し、「外した時は false」を保証する。
⚠ ハマり 5 ── @PathVariable で id が null
SYMPTOM Controllerに渡される id が常に null または例外
CAUSE @GetMapping("/tasks/{id}/edit") のパス変数名 {id} と、@PathVariable Long id の引数名が一致していない（例：{taskId} と id）。
FIX パス変数名と引数名を同じにする。違う名前にしたい場合は @PathVariable("id") Long taskId と明示。
POINT ── 「動的URLとchecbox特有のクセ」

Vol.6特有のハマりは「動的URL（{id}記法）」「checkboxのhidden仕掛け」「@PathVariableの名前一致」に集中。これらはVol.7（完了切替・削除）でも使う基礎。本巻でしっかり身につけておくと、Vol.7以降が楽になります。
