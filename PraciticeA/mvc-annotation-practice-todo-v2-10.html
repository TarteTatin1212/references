vol-10-flow ログが業務システムで果たす役割
// ログがある世界とない世界の違い

ログがない世界：

  • 本番でエラー → 何が起きたか分からない
  • 「うまく動かない」報告 → 再現できない
  • 処理時間が遅い → どこが遅いか分からない
  • セキュリティ調査 → 誰がいつ何をしたか分からない

ログがある世界：

  • 本番でエラー → ログを見れば スタックトレース付きで原因が分かる
  • 「うまく動かない」報告 → 該当時刻のログから流れを再現
  • 処理が遅い → 各層の処理時間ログでボトルネック特定
  • セキュリティ調査 → アクセスログで監査

用語カード：SLF4Jと Logger

TERM — SLF4J & LOGGER

Java界の標準的なログ抽象化レイヤー。「どのログライブラリを使うかは差し替え可能」にする仕組みです。Spring Boot は内部で SLF4J + Logback を採用しています。

TERM CARD #01
SLF4Jと Logger
Java界の標準ログAPI
SLF4Jとは
Simple Logging Facade for Java の略。「ログAPIの統一インタフェース」。
実際にログを書き出す処理はLogback / Log4j2 / java.util.logging 等の実装が担当する。
SLF4J は「窓口」、実装は「裏方」。
Loggerとは
ログを書き出すためのオブジェクト。クラスごとに1つ持つのが慣習。
メソッド：log.trace()、log.debug()、log.info()、log.warn()、log.error()
Spring Bootでの扱い
Spring Boot はSLF4J + Logback をデフォルトで使う。
spring-boot-starter 依存に含まれるので追加設定不要。
Vol.1の時点ですでに使える状態（コンソールに出ているSpringのログがそれ）。
なぜ抽象化するか
① 実装を後から差し替えられる（Logback → Log4j2 等）
② ライブラリ間で衝突しない（SLF4J経由で統一）
③ 業界標準として既知の挙動
Lombokとの連携
Lombok の @Slf4j アノテーションを付けると、自動で private static final Logger log = LoggerFactory.getLogger(クラス名.class); を生成。
手書きと同等のコード。
slf4j-usage Loggerの基本的な使い方
// 3つの書き方を比較

パターン1：手書き（フルバージョン）

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class TaskService {

  private static final Logger log =
    LoggerFactory.getLogger(TaskService.class);

  public Task findById(Long id) {
    log.info("Finding task: id={}", id);
    ...
  }
}

パターン2：Lombok版（推奨）

import lombok.extern.slf4j.Slf4j;

@Slf4j  // ★これだけ
public class TaskService {

  public Task findById(Long id) {
    log.info("Finding task: id={}", id);
    ...
  }
}

Lombok の @Slf4j はパターン1と完全に同じコードを裏で生成します。3行を1行に短縮。本書ではLombok版を採用します。

「log」という変数名

@Slf4j は変数名を必ず log と命名します。「log.info(...)」と書けるのはこのため。
手書きパターンでも、業界の慣習として変数名はlog または logger にします。

// SLF4J + Logback のエコシステム [あなたのコード] ↓ log.info("Hello"); ↓ [SLF4J API] ↓ 抽象化レイヤー ↓ 実装を呼び出す [Logback] ↓ 実装 ↓ ・コンソール出力 ↓ ・ファイル書き込み ↓ ・ローテーション ↓ ・フォーマット制御 [出力] // pom.xml の依存（自動） // spring-boot-starter に内包される: <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter</artifactId> </dependency> // → SLF4J + Logback が自動的に使える // → 追加依存は要らない // Lombokの @Slf4j の魔法 // Lombokが裏で生成しているコード: private static final Logger log = LoggerFactory.getLogger(TaskService.class); // 各メソッドの使い分け（プレビュー） log.trace("Very detailed"); // 滅多に使わない log.debug("Debug info"); // 開発時調査 log.info("Task created"); // 業務イベント log.warn("Stock low"); // 注意事項 log.error("DB failed", e); // 致命的エラー // 詳しいレベルの使い分けは次セクション // import文（Lombok版） import lombok.extern.slf4j.Slf4j; @Slf4j public class YourClass { ... } // ★ lombok.extern.slf4j.Slf4j // の方を選ぶ（org.slf4j.Slf4j ではない）
POINT ── 「SLF4J が業界標準」

Java界の業務開発では「ログ＝SLF4J」と思って差し支えないほど普及しています。「System.out.println」を業務コードに書くのは絶対NG。理由は① 出力先を制御できない、② ログレベルがない、③ 本番で困る。System.out.printlnが残っているコードはレビューで即指摘される── 業界の常識として覚えてください。

用語カード：ログレベル

TERM — LOG LEVELS

ログには5段階のレベルがあります。「どんな出来事をどのレベルで記録するか」の判断が、業務開発の品質を決めます。Vol.9でWARN と ERROR の使い分けに少し触れましたが、ここで全レベルを整理します。

TERM CARD #02
ログレベル
出来事の重要度を5段階で表現
TRACE
最も詳細なトレース情報。変数の値、ループ内の動き、メソッドの入退場。
普段は使わない。徹底的なデバッグ時のみ。
DEBUG
開発時の調査用。条件分岐の進路、計算過程の中間値、外部API のリクエスト内容など。
本番では出さない（量が多すぎ、性能影響あり）。
INFO
業務イベントを記録。タスク登録完了、ユーザーログイン、注文確定など、「業務として起きた重要な出来事」。
本番でも出す。監査ログとしても重要。
WARN
注意事項・業務例外。業務ルール違反、リトライ発生、廃止予定APIの使用など。
「問題ありそうだが致命的ではない」事象。Vol.9 の業務例外はここ。
ERROR
致命的なエラー。DB接続失敗、想定外の例外、システムが続行できない事態。
運用チームへの通知が必要なレベル。Vol.9 のシステム例外（フォールバック）はここ。
階層関係
TRACE < DEBUG < INFO < WARN < ERROR の順で重要度が上がる。
logging.level=INFO と設定すると、INFO以上（INFO/WARN/ERROR）が出力され、TRACE/DEBUG は出ない。
log-level-usage 各レベルの使い分け（業務開発の慣習）
// 業務シーンとログレベルの対応

業務イベント（INFO）

  • タスクを登録した、更新した、削除した
  • ユーザーがログイン/ログアウトした
  • 外部APIを呼んだ（成功）
  • バッチ処理が開始/完了した

注意事項（WARN）

  • 業務例外（在庫不足、権限不足、TaskNotFound 等）
  • リトライが必要だった（一時的なエラー）
  • 非推奨APIが呼ばれた
  • キャッシュミスが頻発した

致命的エラー（ERROR）

  • DB接続失敗
  • NullPointerException 等の予期しない例外
  • 外部APIが呼べない（タイムアウト等）
  • システムが続行不可能な事態
POINT：レベルを間違えると運用チームが大混乱します。「業務例外を ERROR で出すと、夜中に運用チームが叩き起こされる」── これは実際の現場あるある。レベル選択は運用への影響を考えて決めましょう。
// 各レベルの実例 // === TRACE（超詳細） === log.trace("Loop iteration: i={}, value={}", i, value); // 普段は使わない // === DEBUG（開発調査） === log.debug("Cache hit for key: {}", key); log.debug("API request body: {}", body); // 開発時に「ここを通った」確認 // 本番では出さない // === INFO（業務イベント） === log.info("Task created: id={}, title={}", task.getId(), task.getTitle()); log.info("User login: userId={}", userId); log.info("Batch started: {}", jobName); log.info("Batch completed: {} items", count); // 業務として記録すべき出来事 // 監査ログ・分析にも使える // === WARN（業務例外） === log.warn("Task not found: id={}", id); log.warn("Insufficient stock: " + "product={}, requested={}, available={}", name, qty, stock); log.warn("API retry: attempt={}", attempt); // 起こりうる注意事項 // 致命的ではない // === ERROR（致命的エラー） === log.error("Database connection failed", e); // 第2引数に例外を渡すと // スタックトレースが自動で出る log.error("Unexpected error in batch", e); // 運用チームへ通知すべきレベル // 夜中でも叩き起こしてOK（時に） // レベルのフィルタリング // application.properties で: logging.level.com.example.todo=INFO // → TRACE, DEBUG は出ない // → INFO, WARN, ERROR が出る logging.level.com.example.todo=DEBUG // → DEBUG, INFO, WARN, ERROR が出る // → TRACE のみ出ない // 一般的なレベル設定 // 開発環境: logging.level.com.example=DEBUG // テスト環境: logging.level.com.example=INFO // 本番環境: logging.level.com.example=INFO logging.level.org.springframework=WARN // → 業務ログは出す、Spring内部のINFOは抑制
POINT ── 「レベル設計はチームで決める」

「何を INFO、何を WARN、何を ERROR にするか」── これはチームで統一すべきこと。バラバラだと、本番でアラートが大量に発生したり、逆に重要なエラーが見逃されたりする。本書では「業務イベント=INFO、業務例外=WARN、システム例外=ERROR」の方針を採用。これが業界の最も一般的なパターンです。

用語カード：構造化ログ（プレースホルダ）

TERM — STRUCTURED LOG & PLACEHOLDER

SLF4J の強力な機能「プレースホルダ {}」。文字列連結より速く・読みやすく・安全な、業界標準のログ記述スタイルです。

TERM CARD #03
構造化ログ（プレースホルダ）
{}で動的な値を埋め込む
書き方
メッセージに {} を埋め込み、引数で値を渡す。
log.info("Task created: id={}, title={}", id, title);
文字列連結との違い
遅延評価：ログレベルでフィルタされて出力されない場合、文字列構築コストが発生しない。
文字列連結だと、"... " + obj.toString() + " ..." が毎回実行される（フィルタされても）。
例外を渡すパターン
最後の引数に例外を渡すと、スタックトレースが自動で出力される。
log.error("Failed: id={}", id, e);
★ 例外は{}に対応させない。最後の追加引数として扱われる。
業務での意義
① パフォーマンス：ログ未出力時のコスト削減
② 可読性：メッセージと値が分離されて読みやすい
③ 解析性：ログ集約システムでパラメータ化でき、検索が容易
placeholder-vs-concat プレースホルダ vs 文字列連結
// 推奨と非推奨の対比

✓ 推奨：プレースホルダ

log.info("Task created: id={}, title={}",
         task.getId(), task.getTitle());

// 出力:
// INFO  Task created: id=42, title=買い物リスト

✗ 非推奨：文字列連結

log.info("Task created: id=" + task.getId() +
         ", title=" + task.getTitle());

// 同じ出力だが、内部の挙動が違う

なぜプレースホルダの方が速い?

ログレベル設定が INFO の場合：

  • DEBUG ログは出力されない（フィルタで除外される）
  • 文字列連結の場合、連結処理は実行される（その後で「出力しない」と判断）
  • プレースホルダの場合、連結処理がスキップされる

ループ内でDEBUGログを書くようなケースで、この差が大きな性能差になります。

例外を渡す特別なパターン

log.error("Failed to find task: id={}", id, e);
//                                        ↑ 例外
// id は {} に置換される
// e は最後の引数として「例外扱い」される
// → スタックトレースが自動で出る
// プレースホルダの基本 log.info("Hello, {}", name); // 出力: Hello, Taro // 複数のプレースホルダ log.info("User {}, age {}, role {}", name, age, role); // 出力: User Taro, age 30, role ADMIN // 例外との組み合わせ log.error("DB error in user: {}", userId, e); // 出力: // ERROR DB error in user: 42 // java.sql.SQLException: ... // at com.example... // at ...（スタックトレース全部） // パフォーマンスの違い // === 文字列連結（非推奨） === for (int i = 0; i < 10000; i++) { log.debug("Item: " + items.get(i).toString()); // ↑ 毎回 toString() が呼ばれる // ログレベルが INFO なら出力されないが // toString は実行されている } // → 10000回の無駄なtoString // === プレースホルダ（推奨） === for (int i = 0; i < 10000; i++) { log.debug("Item: {}", items.get(i)); // ↑ SLF4Jが「DEBUG出力するか?」を判定 // → 出力しないなら toString は呼ばれない } // → DEBUG無効時はゼロコスト // 高コストな計算がある場合の対策 // たとえば JSON シリアライズが重い場合: if (log.isDebugEnabled()) { log.debug("Heavy data: {}", mapper.writeValueAsString(data)); } // isDebugEnabled() で先にチェック // → 重い処理を完全にスキップ可能 // 別冊で詳しく解説 // 構造化ログとは（本質的な意味） // プレースホルダを使うと: // 「メッセージのテンプレート」と // 「値」が分離される // ログ集約システム（Datadog等）では: // 同じテンプレートのログを // 「同じイベント」としてグルーピング // → 「Task created」イベントが // 何回起きたか集計可能 // これが「構造化ログ」の真価 // （別冊で深掘る）
POINT ── 「{} が業界標準」

SLF4Jで文字列連結を書くのは『コードのにおい』。レビューで指摘される典型例です。「常にプレースホルダ {} を使う」と覚えてください。性能・可読性・解析性、すべての面で優れています。Vol.9 ADD の Section 08 で予告した「ログ出力」がここで本格的に学べます。

各層に Logger を追加する

ADD LOGGERS

最初の作業は各クラスへのLogger追加。@Slf4j を付けるだけ。続いて業務イベントのログ出力を仕込みます。Service層・Controller層の主要メソッドに INFO ログを入れていきます。

step-service-logger TaskService にロガーを追加
// 手順
TaskService.java を修正
  1. com.example.todo.service.TaskService.java を開く
  2. クラスの上に @Slf4j アノテーションを追加
  3. Ctrl+Shift+O で import 補完
    （lombok.extern.slf4j.Slf4j を選ぶ）
  4. 主要メソッドに log.info(...) を追加
  5. Ctrl+S で保存

追加するログの種類：

  • register：INFO「タスク登録: title=...」
  • update：INFO「タスク更新: id=..., title=...」
  • toggleDone：INFO「完了切替: id=..., done=...」
  • delete：INFO「タスク削除: id=...」
  • findById のnull時：WARN（既に Vol.9 で例外を投げているので、ハンドラー側で記録）
POINT：findAll や findById の正常系には INFO ログを付けないのが業界の慣習。一覧表示など頻繁に呼ばれるメソッドに INFO を入れるとログが溢れる。「状態を変える操作」のみ INFO で記録するのが基本です。

ログレベルを環境ごとに制御する仕組み。application.properties に数行追加するだけで、「開発時はDEBUG、本番時はINFO」のような切替ができます。

設定項目の解説：

  • logging.level.com.example.todo=DEBUG
    自分のパッケージ配下のログをDEBUG以上で全部表示（開発時の設定）
  • logging.level.org.springframework=WARN
    Springフレームワーク内部のINFOを抑制（うるさいので）
  • logging.pattern.console=...
    ログの表示フォーマットをカスタマイズ（任意）

階層的なログレベル設定

パッケージ名をパスのように指定して、細かい粒度で制御できます：

# パッケージ全体
logging.level.com.example.todo=DEBUG

# 特定の層だけ INFO に下げる
logging.level.com.example.todo.controller=INFO
logging.level.com.example.todo.service=DEBUG

調査時に特定の層だけ詳細ログを出す等、柔軟に切替できます。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

Vol.10のハマりは「import 誤り」「Lombok 効かない」「ログ出ない」「文字列連結」「設定ミス」の5軸。シリーズ最後のハマり集です。

⚠ ハマり 1 ── @Slf4j を付けてもログが出ない
SYMPTOM @Slf4j アノテーションを付け、log.info(...) と書いてもコンパイルは通るが、実行してもログが出ない。
CAUSE ① import が org.slf4j.Slf4j になっている（存在しないクラスか、または別物）
② Lombokがインストールされていない（Eclipse の場合 lombok.jar 実行が必要）
③ ログレベル設定がより高いレベルになっている（INFO書いてるのにWARN設定等）
FIX ① import を lombok.extern.slf4j.Slf4j に修正
② Eclipse のメニュー ヘルプ → Eclipse Marketplace から Lombok を再インストール、または java -jar lombok.jar 実行
③ application.properties で logging.level.com.example.todo=DEBUG に変更
⚠ ハマり 2 ── プレースホルダの数と引数の数が合わない
SYMPTOM ログに {} がそのまま出る、または引数が欠落している
CAUSE ① log.info("a={}, b={}", a) ── プレースホルダ2個に対し引数1個（bが欠落）
② log.info("a={}", a, b) ── プレースホルダ1個に対し引数2個（bが捨てられるが、最後の引数が例外なら例外扱いされて混乱）
FIX プレースホルダの数と引数の数を正確に一致させる。例外を渡したい時は、プレースホルダの数 = 「例外を除いた引数の数」にする。
⚠ ハマり 3 ── ログに「null」が表示される
SYMPTOM ログに title=null や id=null と表示される
CAUSE 引数オブジェクトが本当に null。プレースホルダは null を "null" と文字列化するため。
FIX 原因はログではなくコード。なぜ null が来たかをログから逆算して、本来の処理を見直す。ログは事実を映す鏡。
⚠ ハマり 4 ── 文字列連結でレビュー指摘される
SYMPTOM log.info("Task: " + task.getId()) のように書いて、レビューで「プレースホルダを使って」と指摘される
CAUSE 文字列連結はパフォーマンス的にも可読性的にも不利。業界の標準スタイルではない。
FIX log.info("Task: {}", task.getId()) に書き換える。「SLF4J使うなら必ず {}」と覚える。
⚠ ハマり 5 ── パスワードや個人情報がログに出てしまう
SYMPTOM log.info("User form: {}", userForm) のようにForm全体を出力したら、ログにパスワードや個人情報がそのまま残る
CAUSE Lombokの @Data で生成された toString() が全フィールドを出力するため。
FIX ① パスワードフィールドに @ToString.Exclude アノテーションを付ける
② ログには必要な項目だけ明示的に出力（{} でフィールドを個別指定）
③ 個人情報・機密情報は絶対にログに残さない（業界の鉄則）
POINT ── 「ログ設計はセキュリティでもある」

ログには「絶対に残してはいけない情報」があります。パスワード、クレジットカード番号、マイナンバー、医療情報、住所、電話番号等。ログは長期保管されることが多く、漏洩リスクが高い場所です。「何をログに残すか」はセキュリティ設計の一部として考えましょう。詳しくは別冊（Vol.10 ADD）で。
