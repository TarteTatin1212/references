01

この補足の目的

WHAT / HOW / WHY

エラーは避けるものではなく解くものです。Vol.3のハマりどころセクションが「事典」だとすれば、本補足は「事件簿」── 実際に起きた1件を最初から最後まで丁寧に追います。

WHAT
何を扱うか
Vol.3で/tasksを開いたら出た500エラー1件。「Error instantiating class org.springframework.scheduling.config.Task」という独特なメッセージから、真犯人を突き止めて修正する全過程。
HOW
どう解説するか
① 探偵モードでログを推理
② 「3点読み」でスタックトレースを切り抜く
③ 真犯人の動機と手口を明らかにする
④ 修正・確認・再発防止まで完走
WHY
なぜこの補足が必要か
エラー対応の力は「実例1個を深く解く」方が、「100個を浅く眺める」より身につくから。一度事件解決の手応えを経験すれば、別のエラーが来ても同じ手順で解ける。
POINT ──

エラーに対する反射的な恐怖こそが初学者最大の壁です。「赤い文字＝怖い」ではなく「赤い文字＝ヒントの宝庫」と思えるようになると、開発体験が一変します。この補足を読み終えた時、あなたはこの種のエラーを「むしろ歓迎」できる状態になっています。

02

症状を確認する

OBSERVE THE SYMPTOM

事件現場の確認から始めます。ブラウザに何が表示されているか、コンソールに何が出ているか── 推理する前に事実を集めます。

two-witnesses ブラウザとコンソールの証言
// 証人1：ブラウザの画面
Whitelabel Error Page This application has no explicit mapping for /error, so you are seeing this as a fallback. Sun May 17 19:05:00 GMT+09:00 2026 There was an unexpected error (type=Internal Server Error, status=500). ### Error querying database... ### The error may exist in com/example/todo/mapper/ TaskMapper.java (best guess) ### The error may involve com.example.todo.mapper. TaskMapper.findAll ### The error occurred while handling results ### SQL: SELECT id, title, done, created_at FROM tasks ORDER BY id ...

ブラウザからの第一情報：
・ステータスコードは500（Internal Server Error）
・404ではない── URLは正しく見つかった
・サーバー側で処理中に何かが壊れた

// 証人2：コンソールの抜粋 // 起動部分 Started TodoApplication in 0.271 sec // アクセス時に出たログ DEBUG c.e.todo.mapper.TaskMapper.findAll ==> Preparing: SELECT id, title, done, created_at FROM tasks ORDER BY id ==> Parameters: ERROR ... Servlet.service() ... threw exception org.mybatis.spring .MyBatisSystemException: ### Error querying database. Cause: org.apache.ibatis.reflection .ReflectionException: Error instantiating class org.springframework.scheduling .config.Task with invalid types (Runnable) or values (1). ### The error occurred while handling results

この時点で分かる事実：

起動は成功している
Started TodoApplication in 0.271 secondsが出ているので、Bean生成・DI・ルーティング登録は完了。Vol.1のような「起動失敗」ではない。
SQLは正しく実行されている
==> Preparing: SELECT id, title, done, created_at FROM tasks ORDER BY idがログに出ている。これはMyBatisがSQLをMySQLに投げて成功した証拠。Vol.2の「ORDERD BY」のようなSQL文法エラーではない。
エラーは「結果を扱う段階」で起きている
### The error occurred while handling results── 結果を扱う最中にエラーが起きた、と明記されている。つまりSQL実行は成功、その後のJava変換でこける。これが今回の事件の第一の手がかり。
POINT ── 似た「500」でも違うエラー

Vol.2の事件（ORDERD BY）も500だったが、ログの様相が違います：
・Vol.2：BadSqlGrammarException ── SQLを発行する前にこけた
・今回：MyBatisSystemException + ReflectionException ── SQLは成功、結果を扱う段階でこけた
同じ500でも原因のレイヤーが違うので、ログの読み方も変わります。これが「ログを読む価値」です。

03

「3つの場所」を読むメソッド

THE 3-POINT READING METHOD

Vol.3のハマりどころセクションで紹介した「3点だけ読めばOK」メソッドを、今回の事件に本格適用します。長いスタックトレースに圧倒されないための道具です。

3-point-method どこを読めば原因が分かるか
// MyBatisエラーの3点読み
一番上の例外名
「何の種類のエラーか」を5秒で把握。BadSqlGrammarExceptionならSQL文法、MyBatisSystemException + ReflectionExceptionならJavaクラスのインスタンス化失敗。
### SQL: 行
「実際にDBに送られたSQL」を確認。このSQLがコンソールに出ている＝SQLは無事に走ったこと。SQLが出ていなければSQL作成段階でこけている。
### The error... 系の行
MyBatisが親切に書いてくれる4つの「###」行：
・### Error querying database ── 種別
・### The error may exist in ── ファイル
・### The error may involve ── メソッド
・### The error occurred while ── タイミング（これが重要）
// 100行のスタックトレースの中で // 読むべき場所は実は10行以下 org.mybatis.spring.MyBatisSystemException: // ← ①例外名 ### Error querying database. Cause: org.apache.ibatis.reflection .ReflectionException: // ← ①根本原因 Error instantiating class org.springframework.scheduling // ← ★最重要★ .config.Task // 真犯人の名前 with invalid types (Runnable) or values (1). ### The error may exist in com/example/todo/mapper/TaskMapper.java // ← ファイル ### The error may involve com.example.todo.mapper.TaskMapper.findAll // ← メソッド ### The error occurred while handling results // ← ③タイミング ### SQL: SELECT id, title, done, created_at FROM tasks ORDER BY id // ← ②走ったSQL // 残りの at xxx.xxx(...) は // 9割が無視してOK // 自分のコード（com.example.todo.*）の // 行があれば、そこは要チェック
POINT ──

スタックトレースは「ライブラリの内部処理を通った全経路」を全部記録しているので長くなります。でも実際にあなたが書いたコードは数ファイルだけ。「com.example.todo.*」を含む行と「### で始まる行」だけ拾えば、99%の情報が手に入ります。

04

このエラーの3点を読む

APPLY THE METHOD

前節のメソッドを今回の事件に適用します。3つの場所からどんな手がかりが見つかるか、探偵モードで読み解きます。

手がかり① 一番上の例外名
MyBatisSystemException → さらにCauseに ReflectionException。
意味の翻訳： MyBatisがJavaのリフレクション（クラスを動的に扱う仕組み）を使った時に失敗した。SQL文法エラーではない── これだけで「SQLの書き間違い」の線は消える。
手がかり② ### SQL:
SELECT id, title, done, created_at FROM tasks ORDER BY id
意味の翻訳： このSQLは正しい文法で、しかも実際にMySQLに送られた。さらにコンソールのPreparing:ログにも出ているので、MySQLからの応答も返ってきた。つまりDB操作自体は無事。
手がかり③ ### The error occurred while handling results
意味の翻訳： エラーは「結果（ResultSet）をJavaオブジェクトに詰める段階」で発生した。SQL実行 → 結果取得 → ★ここで失敗 → Controllerに届かず。
結論： DBから取った結果を Task クラスに変換しようとして、何かが起きている。
手がかり④ 例外メッセージの核心
Error instantiating class org.springframework.scheduling.config.Task with invalid types (Runnable) or values (1).
これが事件の核心。MyBatisはTaskクラスを作ろうとして、そのコンストラクタにIDの値「1」を渡そうとしたが、コンストラクタはRunnable型を期待していて型が合わなかった── そう叫んでいる。
the-smoking-gun 決定的証拠
// この1行が全てを語る
Error instantiating class org.springframework.scheduling .config.Task with invalid types (Runnable) or values (1).

何かおかしいことに気付きましたか？ MyBatisが作ろうとしているTaskクラスのパッケージ名を見てください：

org.springframework.scheduling.config.Task

これはSpring Frameworkが内部で持っているTaskクラス（スケジューラの設定用）。── あなたが作ったTaskクラスのパッケージは何でしたか？

com.example.todo.entity.Task

名前は同じ「Task」だが、別物。MyBatisは間違ったTaskを作ろうとして爆発しています。なぜそんなことが起きるのか？── 次のセクションで真犯人を絞り込みます。

// 2つのTaskクラス // ① あなたが作ったクラス（Vol.3で） package com.example.todo.entity; @Data public class Task { private Long id; private String title; private Boolean done; private LocalDateTime createdAt; } // シンプルなデータ用クラス // 引数なしコンストラクタを持つ // ② Spring Framework内部のクラス package org.springframework .scheduling.config; public abstract class Task { private final Runnable runnable; public Task(Runnable runnable) { this.runnable = runnable; } ... } // スケジューラ用の抽象クラス // Runnableを取るコンストラクタのみ // MyBatisがやろうとしたこと // SQLの1行目： id=1, title=... // MyBatisの思考： // 「Taskクラスを作る」 // 「コンストラクタを呼ぶ」 // 「第一引数に値 1 を渡そう」 // でも参照しているTaskは // Springの方だった // Springの Task(Runnable) // に Long の 1 は渡せない // → 型不一致で爆発
POINT ──

エラーメッセージの「クラスのフルパッケージ名」は絶対に流し読みしてはいけない場所です。「Task」と書かれていても、パッケージ名まで合わせて初めて同じクラス。java.util.Dateとjava.sql.Date、java.util.Listとjava.awt.List── 同名異クラスの罠はJava特有のあるあるです。

05

真犯人を特定する

IDENTIFY THE CULPRIT

MyBatisが間違ったTaskを作ろうとしている── ということは、誰かが間違ったTaskを指定したはずです。真犯人は誰か？ そしてどこに証拠が残っているか？

whodunit 真犯人の追跡
// MyBatisはどうやってTaskを知ったか

MyBatisが「このSELECT結果をTaskクラスに詰める」と判断する根拠はどこか？ ── Mapperの返り値の型です。

@Select(...)
List<Task> findAll();
       ↑
   この「Task」をMyBatisが読む

そしてこのコードの中の「Task」がどのパッケージのものかを決めるのが── import文です。

// TaskMapper.java の冒頭
package com.example.todo.mapper;

import java.util.List;
import org.apache.ibatis.annotations.*;
import ??? .Task;  ← 怪しいのはここ

この「???.Task」の正体を確認すれば、犯人が確定します。

// あなたのTaskMapper.java を開いて確認 package com.example.todo.mapper; import java.util.List; import org.apache.ibatis.annotations.Mapper; import org.apache.ibatis.annotations.Select; // ← ここの行を見てください import org.springframework.scheduling .config.Task; // ↑ おそらくこうなっている // 本来は： // import com.example.todo.entity.Task; @Mapper public interface TaskMapper { @Select("SELECT id, title, done, " + "created_at FROM tasks " + "ORDER BY id") List<Task> findAll(); // ↑ この「Task」が // import文によって // Springのものを指している } // 同じことが TaskController.java // でも起きている可能性 // もしTaskControllerでも // 間違ったimportを入れていたら // そちらも直す必要がある // Eclipseで両ファイルの // import 部分を確認すること

確認すべき手順：

真犯人の証拠を発見する
  1. Eclipseで TaskMapper.java を開く
  2. ファイル冒頭のimport文の集まりを見る
  3. import ___ .Task; の行を探す
  4. そのパッケージ名を確認する
  5. org.springframework.scheduling.config.Task ならこれが真犯人
  6. 同じことを TaskController.java でも確認する
POINT ── importは「クラスの本籍地」

Javaのソースコード内でTaskと書いた時、それがどのパッケージのTaskを指すかはimport文によって決まります。複数の同名クラスが存在する場合、importで指定したパッケージのものが選ばれる── これがJavaの名前解決のルール。importは「住所録」のようなもので、同姓同名がいた時にどちらを呼ぶかを決める仕組みです。

06

なぜ間違ったimportが入ったか

WHY THE WRONG IMPORT

真犯人が「間違ったimport」と分かりました。でもなぜそうなったのか？ 動機（メカニズム）を理解すれば、再発防止に繋がります。

eclipse-mechanism Eclipseの自動import機能の落とし穴
// 容疑者：Ctrl+Shift+O

Vol.3の手順でCtrl+Shift+O（Organize Imports）を使いました。これは赤線で示された未importクラスを自動で追加してくれる便利機能ですが、同名クラスが複数ある場合に問題が起きます。

Eclipseの動き：

候補が1つだけなら → 自動で追加
Stringのように同名クラスが1つしかなければ、Eclipseは勝手にimportを追加して終わり。
候補が複数あれば → 選択ダイアログ
Taskのように同名クラスがクラスパス上に複数ある場合、Eclipseは「Choose Imports」ダイアログを出してあなたに選ばせる。
問題：選択肢を流し読みする
ダイアログには com.example.todo.entity.Task と org.springframework.scheduling.config.Task の両方が表示される。急いでいるとデフォルトで選ばれている方を確認せずEnterを押してしまう。これが今回の犯行手口。
真相： Eclipse は悪意なく選択肢を提示しただけ。選択を間違えたのはあなた（または初期値のまま放置）。これは「誰でもハマるあるあるミス」で、業務経験豊富な開発者でも稀に犯します。
// Eclipseに表示されるダイアログ ┌─ Organize Imports ────────────┐ │ │ │ The choice for class "Task": │ │ │ │ ○ com.example.todo.entity │ │ .Task │ │ │ │ ● org.springframework │ ← デフォルトで │ .scheduling.config.Task │ こちらが選択 │ │ │ [ Back ] [ Next ] [ Finish ] │ └────────────────────────────┘ // 急いでFinishを押すと… // 間違ったTaskがimportされる // なぜSpringのTaskが先頭に来るか // Eclipseの並び順は // アルファベット順や // 「お気に入りパッケージ」順など // 環境によって変わる // 偶然、Springが先に出るケースが // 多い理由： // ・Springは依存に多く含まれる // ・パッケージ階層が短い // ・既知の優先度が設定されている // 真犯人の動機（まとめ） 事件： TaskMapperのimport間違い ↓ 動機： Eclipse自動importで Springの方を選択した ↓ 原因： ・候補ダイアログ確認不足 ・デフォルト値のまま確定 ↓ 結果： MyBatisが間違ったTaskを インスタンス化しようとして 型不一致で爆発
POINT ── 「便利機能の罠」

IDEの自動補完・自動importは素晴らしい機能ですが、「ノールックで使うと事故る」という諸刃の剣でもあります。経験を積むほど「自動補完したものをチラ見する習慣」が身につきます。「IDEは助手であって、責任者はあなた」── これがエディタとの正しい付き合い方です。

07

修正手順

FIX IT

原因が確定したので、修正に入ります。手順を間違えず、最短で復旧させます。

fix-steps 修正手順
// 修正手順
A. TaskMapper.javaを開く
  1. パッケージ・エクスプローラーで com.example.todo.mapper > TaskMapper.java をダブルクリック
  2. ファイル冒頭のimport群を表示
B. 間違ったimportを修正
  1. import org.springframework.scheduling.config.Task; の行を見つける
  2. その行を選択して削除（または直接書き換え）
  3. import com.example.todo.entity.Task; に修正
  4. Ctrl+S で保存
C. TaskController.javaも確認
  1. TaskController.javaを開く
  2. 同じく import ...Task; の行を確認
  3. もしorg.springframework.scheduling.config.Taskを指しているなら同様に修正
  4. 正しければそのまま
DevToolsが効いているため、保存した時点でアプリは自動再起動します。コンソールに Started TodoApplication が再度出るまで待ちましょう。

修正のついでに、使っていないimportが残っていないかも確認すると良いです。Ctrl+Shift+Oを正しいimport状態から実行すると、不要なimport（黄色の警告）も自動的に削除されてクリーンになります。今度はダイアログが出ても候補は1つだけなので安全です。

動作確認 ── 直ったことを確かめる

VERIFY THE FIX

修正したら、本当に直ったかを確かめます。「動いた！」で終わりではなく、意図通りの動作になっているかまで見届けます。

verify-checklist 動作確認の手順
// 段階的に確認していく
A. コンソールで再起動を確認
  1. 保存後、コンソールに Restarting due to ... のログが出る
  2. 続いて Started TodoApplication in X.XXX seconds が再度出る
  3. 赤いERRORログが新しく出ていないか確認

  • 3件のタスク行が並んでいる

エラーが消えても意図通りに動いているとは限りません。例えば「全件取得」を期待しているのに1件しか出ない、「タイトル」列が空欄で表示される── これらは表面的にはエラーなし。だから動作確認は「結果の中身まで」確認します。「★がついた行」「件数」「列の値」── 細部までチェックする癖をつけましょう。

09

再発防止 ── import確認の習慣

PREVENTION HABITS

同じ罠に二度ハマらないために、習慣レベルの予防策を身につけます。意識ではなく行動で防ぐのがコツです。

prevention 3つの防御テクニック
// 再発防止 3 strategies
習慣① import後にチラ見する
Ctrl+Shift+Oを押した後、必ずファイル冒頭のimport群を1秒だけ眺める。com.example.todo.*が想定通り入っているか、見慣れないパッケージ名が混ざっていないか確認。これは2秒の習慣で1時間のデバッグを防ぎます。
習慣② 候補ダイアログを「読む」
「Choose Imports」ダイアログが出たら、慌てずに全候補のパッケージ名を読む。com.example.todo.entity.Taskのように自分のプロジェクトのパッケージを選ぶのが原則。「短いから」「上にあるから」で選ばない。
習慣③ お気に入りパッケージを設定する
Eclipseの設定で「お気に入りパッケージ」を登録できます。com.example.todo.entityを登録しておけば、候補で優先表示されるようになる。環境側の対策で人的ミスを減らす。

COMMON DUPLICATE TRAPS

// よくある衝突パターン
✗ Date の罠

java.sql.Date （DB用、時刻部分なし）
java.util.Date （汎用、時刻あり）
業務開発では LocalDate / LocalDateTime を使うのが今の正解

✗ List の罠

java.util.List （コレクション）
java.awt.List （GUI部品、ほぼ使わない）
意図せずGUIの方を入れると意味不明なエラー

✗ Service の罠

org.springframework.stereotype.Service
jakarta.ws.rs.Path... .Service 等
SpringのServiceを入れるべき場面でJAX-RS系を入れる事故

✗ Transactional の罠

org.springframework.transaction.annotation.Transactional
jakarta.transaction.Transactional
機能が微妙に違うので注意。SpringのDIで動かすなら必ずSpring版

共通の見分け方： パッケージ名に「org.springframework.*」が入っていたらSpring製、「java.*」ならJava標準、「jakarta.*」ならJakarta EE。自分のプロジェクトのはcom.example.todo.*。

// クラス名の出所を見分ける手がかり // 1. com.example.* → 自分のプロジェクト import com.example.todo.entity.Task; // ← Vol.3で作ったやつ // 2. java.* → Java標準ライブラリ import java.util.List; import java.time.LocalDateTime; // ← 基本これらは安全 // 3. javax.* / jakarta.* → Java EE系 import jakarta.servlet.http.HttpServlet; import jakarta.persistence.Entity; // ← Spring Boot 3+では jakarta // 4. org.springframework.* → Spring本体 import org.springframework.stereotype.Controller; import org.springframework.web.bind.annotation.*; // ← MVCの定番 // 5. org.apache.ibatis.* → MyBatis import org.apache.ibatis.annotations.Mapper; import org.apache.ibatis.annotations.Select; // ← DB窓口の定番 // 6. lombok.* → Lombok import lombok.Data; import lombok.AllArgsConstructor; // ← ボイラープレート消し // 警戒すべきimport // org.springframework.scheduling.* // スケジューラ用、エンティティとは別物 // java.awt.* // GUI用、Web開発では普通使わない // javax.* (Spring Boot 3+で警戒) // Spring Boot 3はjakarta.*へ移行済み // 古い記事のjavax.*をコピーすると // 互換性で問題発生
POINT ── 「パッケージ名を読む」習慣

クラス名だけ見て判断するのではなく、パッケージ名の prefix まで読む習慣をつけましょう。「org.springframework なら Spring 製」「java.* なら標準ライブラリ」と即座に分類できると、エラー解析の速度が劇的に上がります。パッケージ名は「クラスの本籍地」── 同姓同名を見分けるための重要情報です。

DEBUGGING THOUGHT FLOW

debugging-flow エラー解決の思考フロー
// 9ステップ汎用フロー
症状を「事実」として記録する
ブラウザに何が出ているか／コンソールに何が出ているか／いつから出始めたか── 感情を交えず事実だけを集める。
エラーの「種類」を5秒で判定
・404＝URLが見つからない
・500 + 起動失敗＝設定やBean生成の問題
・500 + 起動成功＝処理中の問題（今回はこれ）
3点読みでログを切り抜く
① 一番上の例外名 → エラーの種別
② ### SQL: → SQLが走ったか
③ ### The error occurred while ... → どの段階で起きたか
エラーメッセージを精読する
クラス名・パッケージ名・引数の型など、メッセージに含まれる固有名詞を全て読む。「違和感」（パッケージが想定外、型が想定外）を見つけたら、それが手がかり。
「真犯人」を絞り込む
違和感の出所はどのコードか？ そのコードを書いた時に何が起きたか？ ── 因果関係を遡る。
仮説を立てて検証する
「○○が原因なら、こうなっているはず」と予測を立てて、実際のコードを確認。一致すれば原因確定。
修正する
最小限の変更で修正。関係ない箇所をついでに触らないのが鉄則（副作用を防ぐため）。
動作確認
エラーが消えただけでなく、結果の中身まで確認。
再発防止策を考える
「なぜこのミスを犯したか」を冷静に分析し、習慣やツール設定で防ぐ仕組みを作る。
