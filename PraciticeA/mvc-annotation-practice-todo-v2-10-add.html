02

そもそもログとは何のため

WHAT IS LOG FOR

ログは「過去の事実を残す」仕組みですが、「何のために残すのか」を整理すると目的が見えてきます。実は4つの目的があります。

log-purposes ログを残す4つの目的
// ログの4つの目的
トラブル調査
最も基本的な目的。「本番でエラーが出た」「ユーザーから問題報告」── ログを見て原因を特定する。スタックトレース、リクエスト内容、処理順序などが手がかり。
監査（オーディット）
「いつ・誰が・何をしたか」を証跡として残す。金融、医療、官公庁系のシステムでは法的要件になることも。「このユーザーが10時に削除した」が記録されていることが重要。
性能分析
「どの処理がどれくらい時間がかかったか」を計測。ボトルネック特定、遅延検知、キャパシティプランニングに使う。Vol.10本編の「実行時間ログ」がこれ。
業務分析
「どの機能がよく使われているか」「離脱率はどうか」「どのページから入って何を購入するか」── ビジネス分析の素材としてログを使う。マーケティングや経営判断に直結。

業務システムでは、これら4つの目的すべてが同時に求められます。ログ設計は「誰が何のために読むか」を意識する必要があります。

// 4つの目的別ログ例 // === 1. トラブル調査用 === log.error("DB connection failed: url={}", url, e); log.warn("Retry attempt: {}/{}", attempt, maxRetries); // 開発者・運用が読む // 例外詳細、状態、コンテキスト // === 2. 監査用 === log.info("AUDIT: User {} deleted task {} at {}", userId, taskId, timestamp); log.info("AUDIT: Permission changed by admin {}", adminId); // セキュリティ担当・コンプライアンスが読む // 5W1H（誰がいつ何をどこで）が重要 // === 3. 性能分析用 === log.info("PERF: {} ms, query={}, rows={}", elapsed, queryName, rowCount); log.info("API: {} responded in {} ms", endpoint, elapsed); // 性能エンジニア・SREが読む // 数値が中心 // === 4. 業務分析用 === log.info("BUSINESS: Order placed: " + "product={}, amount={}, channel={}", productId, amount, channel); log.info("BUSINESS: User journey: {} -> {}", fromPage, toPage); // マーケティング・経営層が読む // 集計・分析しやすい形 // 同じ「ログ」でも目的が違う // → 出力先を分けることもある // ・アプリログ：トラブル調査用 // ・監査ログ：別ファイル＋長期保管 // ・アクセスログ：性能分析用 // ・イベントログ：業務分析用 // 「すべて1つのログに混ぜる」より // 「目的別に分ける」のが本格運用
アナロジー ── ビルの監視カメラ

ログは「ビルの監視カメラ」のようなもの。
・セキュリティ警備員が見る（トラブル調査）
・監査法人が見る（コンプライアンス）
・施設管理が見る（混雑分析）
・マーケティングが見る（来訪者分析）

1つのカメラ映像を複数の人が違う目的で見る。映像の保管期間・解像度・閲覧権限も目的によって違う。ログも同じ。「誰が何のために見るか」を考えて設計するのが業務開発のリアルです。

POINT ── 「ログは複数の読者を持つ」

「ログは1つだけ書いて終わり」ではなく、「複数の読者・複数の目的」を持つことを意識しましょう。Vol.10 本編で書いた INFO/WARN/ERROR は「トラブル調査と業務イベント記録」の2目的を兼ねています。実務では監査・性能・業務分析用に別ロガーを用意することも珍しくありません。

03

観測可能性（Observability）── 3つの柱

OBSERVABILITY — THREE PILLARS

現代の業務開発で必修概念となった「観測可能性」。「システムの内部状態を、外部からの観測だけで推測できる度合い」のこと。ログ・メトリクス・トレーシングの3要素で実現します。

CONCEPT
観測可能性（Observability）
業務開発の必修概念
定義
「システムの内部状態を、外部からの観測だけでどれだけ推測できるか」を示す概念。
高い観測可能性 = 問題が起きた時に原因をすぐ特定できる。
3つの柱
① ログ（Logs）── 何が起きたか
② メトリクス（Metrics）── どれくらい起きているか
③ トレーシング（Traces）── どう繋がっているか
ログ
個別イベントの記録。Vol.10で学んだ内容。「何が起きたか」を時系列で残す。
メトリクス
時系列の数値データ。「リクエスト数」「レスポンスタイム」「エラー率」「CPU使用率」など。集計・グラフ化に向く。
トレーシング
1リクエストの旅を追跡。マイクロサービス環境で重要。「リクエストAが、サービスB→C→Dと回って、どこで何msかかったか」を可視化。
業界での扱い
現代の業務システムは「観測可能性ファースト」で設計する。SRE（Site Reliability Engineering）の核心概念。
three-pillars 3つの柱の使い分け
// 3つの柱で何が分かるか

ログ（Logs）── 「何が起きたか」

2024-01-15 14:30:25 INFO  Task registered: id=42
2024-01-15 14:30:30 WARN  Task not found: id=99
2024-01-15 14:30:32 ERROR DB connection failed

個別の「事件」が時系列に並ぶ。詳細情報が豊富。

メトリクス（Metrics）── 「どれくらい起きているか」

request_count        2,450 / sec
response_time_p99    230 ms
error_rate           0.5%
cpu_usage            42%
memory_usage         1.2 GB / 4 GB

集計値・統計値。グラフで時系列推移を見る。異常検知に強い。

トレーシング（Traces）── 「どう繋がっているか」

Request "POST /tasks" (250ms)
├── Controller (5ms)
├── Service.register (240ms)
│   ├── Mapper.insert (180ms)  ★ボトルネック
│   └── EmailService.send (55ms)
└── Response (5ms)

1リクエストの内部構造を可視化。マイクロサービス間の連携を追跡。

3つを使い分けるシナリオ

  1. メトリクスで「エラー率が上昇」を検知
  2. ログで「具体的にどんなエラーか」を調査
  3. トレースで「どの段階で失敗しているか」を特定
  4. → 原因解明 → 修正
// 3つの柱の実装例 // === ① ログ（Vol.10で学んだ） === log.info("Task registered: id={}", taskId); // → 「何が起きたか」 // === ② メトリクス（Micrometer等） === @RestController public class TaskController { private final MeterRegistry meter; @PostMapping("/tasks") public String create(...) { meter.counter("task.created").increment(); // → 「タスク登録回数」をカウント // Prometheus等で時系列グラフ化 ... } } // === ③ トレーシング（Spring Cloud Sleuth等） === // 自動的に「Trace ID」が各リクエストに付く // ログにも自動で出る: INFO [task-app,b3a1c5d2,e8f7g6h5] Task created // ↑ Trace ID と Span ID // → 複数サービス間でリクエストを追跡可能 // 3つを統合するツール [アプリ] ├ ログ → Fluentd → Elasticsearch（ELK） ├ メトリクス → Prometheus → Grafana └ トレース → Jaeger / Zipkin ↓ [統合ダッシュボード] Datadog / New Relic / Dynatrace 等 // 一画面で全部見られる // → 観測可能性の実現 // 観測可能性の評価 // 「観測可能性が高い」システム: // ・問題が起きた瞬間に検知される // ・5分以内に原因が特定できる // ・修正後の効果が定量的に分かる // 「観測可能性が低い」システム: // ・ユーザーから問い合わせて初めて気づく // ・原因調査に数時間かかる // ・直したつもりが直っていない // 本シリーズで学んだログは「第一の柱」 // Vol.10で学んだのは観測可能性の入り口 // ・メトリクスは Spring Boot Actuator で // ・トレーシングは Spring Cloud Sleuth で // それぞれ次のステップで学ぶ価値あり
POINT ── 「観測可能性は業務エンジニアの必修」

2020年代の業務開発では「観測可能性」を知らないと話が通じません。SRE（Site Reliability Engineering）、DevOps、クラウドネイティブ── どの文脈でも観測可能性が前提です。Vol.10 で学んだログは第一の柱。残りのメトリクス・トレーシングも、業務開発の次のステップとして必ず触れることになります。

用語カード：ログレベルの哲学

TERM — LOG LEVEL PHILOSOPHY

Vol.10本編で5段階のログレベルを学びました。でも実務でのレベル選択はもっと深い議論があります。「業界の流派」「夜中起こされる基準」「廃止予定の議論」── 哲学的な側面を見ていきます。

TERM CARD
ログレベルの哲学
業界の暗黙的なルール
レベル選択の「3つの問い」
① 本番でも出すべきか?
② 運用チームに通知すべきか?
③ 夜中でも対応すべきか?
この3つにYesがいくつ付くかで、レベルが決まります。
YesがゼロのDEBUG/TRACE
本番では出さない。開発時のみ。
Yesが1のINFO
本番でも出すが、通知や対応は不要。業務記録。
Yesが2のWARN
本番で出す、監視対象だが、夜中の即対応は不要。業務例外はここ。
Yesが3のERROR
本番で出す、通知、夜中でも対応。システム例外はここ。
レベルの誤用
「ERROR乱発」── 業務例外も全部ERRORにすると、本物のERRORが埋もれる
「INFO過剰」── 全メソッドに INFO 入れるとログ量が爆発
「WARN曖昧」── 何が WARN か基準がないとチーム内で揺れる
log-level-realworld 業界の実例で見るレベル選択
// 「夜中起こされる」基準で考える

業務開発のチームでは、ログレベルは「運用負荷」と直結します。ERROR が大量に出るシステムは、夜中に運用チームが叩き起こされる原因に。だから「ERRORの基準」をチームで真剣に議論します。

業界の典型的なルール

ERRORの基準（厳格派）

「夜中に起こされる価値があるか」で判断。
・DB完全ダウン → ERROR
・外部API3回連続失敗 → ERROR
・予期しない NullPointer → ERROR
・データ不整合 → ERROR

WARNの基準

「気になるが緊急ではない」。
・業務例外（在庫不足等） → WARN
・リトライで回復 → WARN
・廃止予定APIの使用 → WARN
・性能劣化（しきい値超） → WARN

アラート設定との関係

# Datadog等の典型的なアラート設定

ERROR が 5分間で 5件以上
  → Slack通知 + メール
  → On-Call担当者に電話

WARN が 1時間で 100件以上
  → Slackのみ
  → 翌朝レビュー

INFO は通知なし
  → ダッシュボードで可視化

この設定の意味は、「ERRORを乱発するとチームが壊れる」ということ。本物のERRORだけにERRORを出すのが、チームを守る設計です。

業界あるある：新人エンジニアが「念のためERROR」で書いて、夜中にチーム全員が叩き起こされ、「これ業務例外じゃん…」と発覚するパターン。レベル選択は真剣に。
// 実例：レベル判断のシナリオ // === シナリオ1：ユーザー登録 === // 正常完了 log.info("User registered: id={}", userId); // → INFO（業務イベント） // 重複メールでエラー log.warn("Duplicate email: {}", email); // → WARN（業務例外、起こりうる） // DB書き込み失敗 log.error("User save failed", e); // → ERROR（システム異常） // === シナリオ2：外部API呼び出し === // 1回目失敗 → リトライ log.warn("API call retrying: attempt={}", n); // → WARN（自動回復可能） // 3回連続失敗 log.error("API call failed permanently", e); // → ERROR（回復不能） // === シナリオ3：性能 === // 通常の応答時間 log.debug("Request: {} ms", elapsed); // → DEBUG（普段は出さない） // しきい値超過 if (elapsed > 3000) { log.warn("Slow request: {} ms", elapsed); } // → WARN（要注意） // タイムアウト log.error("Request timeout", e); // → ERROR（システム異常） // アンチパターン // ✗ NG: 業務例外をERRORで出す log.error("Task not found"); // → 夜中にチーム叩き起こされる // → 本物のERRORが埋もれる // ✗ NG: System.out.println を残す System.out.println("Debug: ..."); // → コードレビューで即指摘 // → ログレベル制御不可 // → 本番でも出続ける // ✗ NG: 例外を catch して無視 try { ... } catch (Exception e) { // 何もしない } // → ログさえ出ない // → エラーが見えなくなる // → 最悪のアンチパターン // 業界の合言葉 // 「ERRORは少なく、WARNは適切に、 // INFOは業務的に、DEBUGは詳細に」 // 4レベルの基準を統一すれば // チーム運用が楽になる
POINT ── 「ERRORは『叫び』として使う」

ERROR は「夜中に運用チームを叩き起こすほどの叫び」。乱発すると「狼少年効果」で誰も反応しなくなります。「本当のERRORだけにERROR」── これが業務開発の鉄則。レベル選択は運用への愛情でもあります。

用語カード：構造化ログ vs プレーンログ

TERM — STRUCTURED VS PLAIN LOG

本編でプレースホルダ {} を学びました。これを発展させた「構造化ログ」は、JSON形式でログを出力して機械的に解析できるようにする仕組み。本格的な運用環境では必須です。

TERM CARD
構造化ログ vs プレーンログ
ログ形式の2つの流派
プレーンログ
従来の「人間が読む」ログ形式。
2024-01-15 INFO Task registered: id=42, title=買い物
メリット：そのまま読める / デメリット：機械的な集計が難しい
構造化ログ
JSON形式等で、キー・バリューとして出力するログ。
{"time":"2024-01-15","level":"INFO","event":"task.registered","id":42,"title":"買い物"}
メリット：機械解析が容易 / デメリット：人間が読みにくい
業界の主流
大規模な業務システムは「構造化ログ」が主流。理由：ログ集約システム（ELK / Datadog 等）で検索・集計するため。
実装方法
① logback-spring.xml で JSON フォーマッタを設定
② Logstash Logback Encoder 等のライブラリを使う
③ Spring Boot 3.x なら logging.structured.format プロパティで切替可能
MDC（Mapped Diagnostic Context）
各リクエストに「文脈情報」を載せるSLF4Jの仕組み。
MDC.put("userId", "42") でセットすれば、全ログに自動で userId=42 が付く。
構造化ログと組み合わせると「ユーザーごとに全ログ追跡」ができる。
plain-vs-structured プレーンと構造化の対比
// 同じイベントを2形式で表現

プレーンログ（Vol.10本編で出力される形式）

2024-01-15 14:30:25.123 INFO  [http-nio-8080-exec-1]
c.e.t.s.TaskService - Task registered: title=買い物リスト

構造化ログ（JSON形式）

{
  "timestamp": "2024-01-15T14:30:25.123Z",
  "level": "INFO",
  "thread": "http-nio-8080-exec-1",
  "logger": "com.example.todo.service.TaskService",
  "message": "Task registered: title=買い物リスト",
  "title": "買い物リスト",
  "event": "task.registered"
}

構造化ログの威力

ログ集約システムで以下のクエリが可能：

  • event:task.registered ── タスク登録イベントだけ抽出
  • level:ERROR AND service:payment ── 決済サービスのエラーだけ
  • userId:42 ── ユーザー42の全ログ追跡
  • response_time:>3000 ── 3秒以上かかったリクエスト

プレーンログだとgrep頼みで難しいこんな解析が、構造化ログならSQLライクに簡単。これが業界が構造化ログに移行した理由です。

// MDC（Mapped Diagnostic Context）の使い方 @Slf4j @Component public class RequestLoggingFilter implements Filter { @Override public void doFilter( ServletRequest req, ServletResponse res, FilterChain chain) throws IOException, ServletException { // リクエストごとに一意なIDを生成 String requestId = UUID.randomUUID().toString(); MDC.put("requestId", requestId); try { chain.doFilter(req, res); } finally { MDC.clear(); // 必ずクリア } } } // MDC設定後のログ // アプリ内のどこでログを書いても // 自動で requestId が付く INFO [requestId=abc-123] c.e.t.s.TaskService - Task registered: id=42 INFO [requestId=abc-123] c.e.t.m.TaskMapper - SQL: INSERT INTO tasks ... クラウド時代の業務開発では「構造化ログ」が標準です。AWS CloudWatch、Datadog、Splunk── どれも構造化ログを前提に設計されています。Vol.10 本編のプレースホルダ {} は構造化ログへの第一歩。次のステップで JSON出力 と MDC を学べば、現代的なログ運用ができるようになります。

06

機微情報のマスキング設計

SENSITIVE DATA MASKING

ログ設計のセキュリティの側面。Vol.10本編のハマり5で触れた「個人情報・機密情報を絶対にログに残さない」仕組みを深掘ります。業務開発者の必須スキルです。

masking-design 機微情報を守る3つのアプローチ
// 守るべき情報の代表例

絶対にログに出してはいけない情報

  • パスワード（生文字・ハッシュ問わず）
  • クレジットカード番号、CVV
  • マイナンバー、運転免許番号
  • 医療情報、健康診断結果
  • セッショントークン、APIキー
  • 個人を特定する情報（住所、電話番号、生年月日）

条件付きで残せる情報

  • ユーザーID（内部ID）── 個人特定情報と紐づけなければOK
  • メールアドレス── 一部マスクして残す（foo@***）
  • IPアドレス── 監査目的なら残す、保管期間に注意

3つのマスキング戦略

@ToString.Exclude で toString() から除外
Lombok の @Data で生成される toString() から、機微フィールドを除外する。
log.info("User: {}", user) で書いても、パスワードは出ない。
マスキング関数で値を加工
maskEmail("foo@example.com") → "f**@example.com"。
一部だけ残すことで、「同じユーザーか」は分かるが、個人特定はできない状態にする。
フィルタで自動マスキング
Logback の正規表現フィルタ等で、ログ出力時に自動でパターンを置換。「クレジットカード番号らしき16桁」を見つけたら****-****-****-1234に書き換え。
実装のうっかりミスを防ぐ最後の砦。
業界の事故事例：本番ログにパスワードがそのまま残っていて、ログ閲覧権限のある開発者全員に知られる事故が実際にあります。「ログは社内に流通する文書」と思って設計しましょう。
// === 戦略1: @ToString.Exclude === @Data public class UserForm { private String email; @ToString.Exclude // ★追加 private String password; @ToString.Exclude // ★追加 private String creditCard; } // ログ出力: log.info("User: {}", userForm); // 出力: // User: UserForm(email=foo@example.com) // → password も creditCard も出ない // === 戦略2: マスキング関数 === public class LogMasker { public static String maskEmail(String email) { if (email == null) return null; int at = email.indexOf('@'); if (at < 2) return "***"; return email.charAt(0) + "***" + email.substring(at); } public static String maskCard(String card) { if (card == null || card.length() < 4) return "****"; return "****-****-****-" + card.substring(card.length() - 4); } } // 使い方: log.info("Login attempt: email={}", LogMasker.maskEmail(email)); // 出力: // Login attempt: email=f***@example.com // === 戦略3: Logback フィルタ === // logback-spring.xml で: <appender name="CONSOLE" class="..."> <filter class="ch.qos.logback...MaskingFilter"> <pattern> \d{4}-\d{4}-\d{4}-\d{4} </pattern> <replacement>****-****-****-****</replacement> </filter> </appender> // → ログ出力時に自動でカード番号をマスク // → アプリ側のミスを防ぐ最後の砦 // 業務開発のチェックリスト // Form/Entity の機微フィールドに: @ToString.Exclude // ログ出力前にマスキング関数を通す: log.info("...", LogMasker.mask(value)); // 最後の砦としてLogbackフィルタ // → 多層防御 // 「ログを社外に出す」ケースも考える // クラウドサービスのログ: // → AWS CloudWatch → AWS（第三者）が見る // → Datadog → Datadog社のサーバに送信 // 「自社内」だけの問題ではない // → 外部サービスとの契約を確認 // → GDPR/個人情報保護法も視野に
アナロジー ── 書類のシュレッダーとマスキングテープ

機微情報の扱いは「書類処理」に似ています。
・シュレッダー（@ToString.Exclude）── 該当箇所を最初から除去
・マスキングテープ（maskEmail等）── 該当箇所を黒塗りで残す
・受付フィルタ（Logbackフィルタ）── 外に出る前に最終チェック

どれか1つだけだと「うっかり通過」するので、多層防御が業務システムの基本。「本番ログにパスワードが流れる事故」は今でも実際に起きています。

POINT ── 「ログは社内文書」

ログは「社内に流通する文書」です。閲覧権限のある開発者、運用チーム、外部委託業者、監査人── 様々な人の目に触れます。「自分しか見ない」と思い込まないのが業務開発のリアル。「機微情報は最初からログに入れない設計」が、業務開発者の必須スキルです。

LOG AGGREGATION WORLD

本番システムでは、ログは個別のサーバに散らばっているのではなく、1か所に集めて検索するのが標準。ログ集約システムがそれを担当します。業界で使われる代表的なツールを見ていきます。

本番システムの典型構成：

[Webサーバ1] [Webサーバ2] [Webサーバ3]
     ↓           ↓           ↓
    各サーバが個別にログを書く
     ↓           ↓           ↓
   /var/log/    /var/log/    /var/log/

            [統合ダッシュボード]
            検索・集計・グラフ化

代表的なログ集約ツール

ELK スタック（Elasticsearch + Logstash + Kibana）
オープンソースの王道。Elasticsearch（検索エンジン）+ Logstash（収集）+ Kibana（可視化）の3点セット。
自社でホスティング可能でコストコントロールしやすい。本格的な業務システムで広く採用。
Datadog
クラウドの統合監視サービス。ログ + メトリクス + トレースを1つのダッシュボードで。
SaaSなので運用が楽。料金は高めだが、開発生産性は抜群。スタートアップ〜大企業まで広く採用。
AWS CloudWatch Logs
AWS の標準ログサービス。AWS上のシステムなら自動でログを集められる。
CloudWatch Logs Insights で SQL ライクな検索が可能。
Splunk
エンタープライズの定番。大企業、金融・官公庁系で広く採用。

Grafana Loki
Prometheus同様のラベル指向の比較的新しいログ集約。
Grafanaと組み合わせて使う。コスト効率が良い。

08

本番運用のリアル（ローテーション・保管期間・コスト）

PRODUCTION REALITY

業務開発者が遭遇する「ログの本番運用」のリアル。ログファイルが肥大化、保管コスト爆発、法令での保管期間要件── 知っておくべき「現場の常識」です。

production-reality 本番運用での3つの課題
// 本番運用で立ちはだかる3つの壁
ログローテーション
ログを1ファイルに書き続けると、サイズが無限に増える。GB級になるとファイルアクセスが遅くなる、ディスクが溢れる。
対策：日次/サイズで切り替え＋古いログを圧縮＋一定期間で削除。
保管期間とコスト
「どれくらいログを残すか」は業務要件とコストのトレードオフ。
・トラブル調査用：2週間〜1ヶ月
・監査要件：3〜7年（金融・医療は法令で決まる）
・分析用：1〜2年
ログ集約サービスの料金はデータ量に比例するので、無計画だと月100万円超えも。
パフォーマンス影響
ログ出力もI/O処理。ループ内で大量にDEBUGログ → 性能劣化。同期書き込み → リクエスト遅延。
対策：非同期Appender＋ログレベルの適切な制御＋isDebugEnabled()。

ローテーション設定の例

application.properties で簡単に設定可能：

# ファイル出力先
logging.file.name=logs/app.log

# ローテーション設定
logging.logback.rollingpolicy.file-name-pattern=
  logs/app-%d{yyyy-MM-dd}.%i.log.gz
logging.logback.rollingpolicy.max-file-size=100MB
logging.logback.rollingpolicy.total-size-cap=10GB
logging.logback.rollingpolicy.max-history=30
  • max-file-size=100MB：100MB超えたら新ファイル
  • max-history=30：30日分保持、古いものから削除
  • total-size-cap=10GB：全体で10GBを超えないように制限
  • .gz：圧縮して保存（容量節約）

ログとパフォーマンス（isDebugEnabled() の話）

LOG & PERFORMANCE

ログは処理コストを伴います。何気なく書いたログが性能ボトルネックになることも。本編で触れたプレースホルダの恩恵と、その更に上の対策を見ます。

log-performance パフォーマンスを意識したログ
// ログのコスト構造

ログ1行あたりの処理：

  1. レベル判定（フィルタ通過チェック）
  2. 引数の文字列化（toString() 呼び出し）
  3. メッセージ組み立て
  4. ファイル/コンソールへの書き込み（I/O）

通常は問題になりませんが、「ループ内で大量にログ」や「巨大オブジェクトのtoString()」では性能問題になります。

プレースホルダの恩恵（再確認）

// ◎ プレースホルダ
log.debug("Item: {}", item);
// → DEBUG無効時は item.toString() が呼ばれない

// ✗ 文字列連結
log.debug("Item: " + item);
// → DEBUG無効でも item.toString() は呼ばれる

isDebugEnabled() の使い所

「引数の準備自体に時間がかかる」場合は、プレースホルダだけでは対策しきれません。引数を作る処理が毎回実行されるからです。
そんな時は isDebugEnabled() で明示的にスキップします。

// ◎ 重い処理を伴うログ
if (log.isDebugEnabled()) {
  String json = objectMapper.writeValueAsString(largeObj);
  log.debug("Data: {}", json);
}

// → DEBUG有効時のみ JSON 化処理が走る
// → DEBUG無効時はゼロコスト

ループ内ログの注意

// 1000万件ループ
for (Item item : items) {  // 10,000,000 件
  process(item);
  log.info("Processed: {}", item.getId());  // 毎回出力
}
// → ログ書き込みI/Oが性能ボトルネック

// 改善：サマリーログ
int count = 0;
for (Item item : items) {
  process(item);
  count++;
  if (count % 10000 == 0) {
    log.info("Progress: {}/{}", count, items.size());
  }
}
log.info("Complete: {} items", count);
// → ログ量を1/10000に削減
// パフォーマンス最適化の段階 // === レベル1：文字列連結（最悪） === for (int i = 0; i < 1000000; i++) { log.debug("i=" + i + ", item=" + items.get(i).toString()); // 毎回 toString と文字列連結が走る // DEBUG無効でも } // → 100万回の無駄な処理 // === レベル2：プレースホルダ（標準） === for (int i = 0; i < 1000000; i++) { log.debug("i={}, item={}", i, items.get(i)); // DEBUG無効時は toString が呼ばれない } // → DEBUG無効ならゼロコスト // → これだけでも大きく改善 // === レベル3：重い処理を伴う場合 === for (int i = 0; i < 1000000; i++) { if (log.isDebugEnabled()) { String json = mapper .writeValueAsString(items.get(i)); // JSON化は重い処理 log.debug("i={}, data={}", i, json); } } // → DEBUG無効時はJSON化さえスキップ // → 完全にゼロコスト // === レベル4：ループ自体を見直す === int count = 0; for (Item item : items) { process(item); count++; if (count % 10000 == 0) { log.info("Progress: {}/{}", count, items.size()); } } log.info("Complete: {} items", count); // → ログ量を 1/10000 に削減 // → 必要な情報は失わない // 業務開発の判断指針 // 通常のメソッド呼び出し: // プレースホルダで十分 // ループ・頻繁な処理: // isDebugEnabled() + サマリー化 // 重い toString / JSON化: // isDebugEnabled() でガード // 注意：意味のないisDebugEnabled // ✗ NG：軽い処理に isDebugEnabled は不要 if (log.isDebugEnabled()) { log.debug("i={}", i); } // → 何の最適化にもならない // → むしろコードが汚くなる // → プレースホルダだけで十分 // ◎ OK：重い引数準備がある時のみ if (log.isDebugEnabled()) { // JSON化、巨大オブジェクトのtoString等 log.debug("...", heavyExpression); }
POINT ── 「過剰最適化は害悪」

パフォーマンス最適化は「ボトルネックを特定してから対処」が原則。「念のため isDebugEnabled()」は害悪です。コードが汚くなるだけ。「測定して、本当に遅い箇所だけ最適化する」のがプロの流儀。Vol.10 本編のプレースホルダを正しく使えていれば、99%のケースで性能問題は起きません。
