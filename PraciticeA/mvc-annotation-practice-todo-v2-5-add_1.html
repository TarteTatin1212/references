そもそも「フォーム送信」とは何が起きているか

WHAT HAPPENS ON SUBMIT

「登録ボタンを押す」── たった1秒の操作の裏側で、ブラウザとサーバがHTTPプロトコルを使って細かい会話をしています。仕組みをイベントの連鎖として追います。

submit-mechanism 送信ボタンクリック時の連鎖反応
// 「登録」ボタンクリックの瞬間に起きること
ブラウザがform要素を見る
ボタンが押されたformの action 属性（送信先URL）と method 属性（HTTPメソッド）を読む。Vol.5なら action="/tasks" と method="post"。
入力値を集める
form内の全ての入力要素（input, textarea, select等）のname属性と入力値のペアを収集。title=新しいタスク のような形式に整形。
HTTPリクエストを組み立てる
POSTの場合：リクエストボディに title=新しいタスク 形式（URL-encoded）で詰め、Content-Type は application/x-www-form-urlencoded。
GETの場合：URLの末尾に ?title=新しいタスク としてクエリ文字列にする。
サーバに送信
TCP/IPでサーバ（localhost:8080）にリクエスト送信。サーバが受信。
Springがリクエストを解析
DispatcherServletがHTTPメソッドとURLを見て、@PostMapping("/tasks") のメソッドにマッチ。リクエストボディの title=... を解析して、TaskForm のフィールドに詰める。
レスポンスを返す
create() メソッドが処理を実行し、リダイレクト指示（HTTP 302）をブラウザに返す。
// 実際のHTTPリクエスト（送信ボタンクリック時） POST /tasks HTTP/1.1 Host: localhost:8080 Content-Type: application/x-www-form-urlencoded Content-Length: 32 Cookie: JSESSIONID=... title=%E6%96%B0%E3%81%97%E3%81%84%E3%82%BF%E3%82%B9%E3%82%AF // ↑ "新しいタスク" を URL-encoded したもの // HTMLフォームと送信内容の対応 <!-- このHTML --> <form action="/tasks" method="post"> <input type="text" name="title" value="新しいタスク"> <button type="submit">登録</button> </form> // が、送信ボタンクリックで: POST /tasks ↑ action 属性 ↑ method 属性 body: title=新しいタスク ↑ name属性 = 入力値 // なぜ name 属性が重要か // name="title" の場合 // → 送信データ: title=新しいタスク // → サーバは「titleパラメータ」と認識 // → TaskForm.setTitle() が呼ばれる // name属性なし、または name="memo" だと // → TaskForm.title に値が入らない！ // name属性は「サーバへのキー」 // Vol.5で th:field="*{title}" を書くと // 自動で name="title" が出力される
アナロジー ── 郵送の伝票

フォーム送信は郵便での書類送付に似ています。
action属性 = 宛先住所、method属性 = 配送方法（速達か普通か）、name属性 = 書類の項目名、入力値 = 書類の中身。送信ボタンは「ポストに投函」。郵便屋さん（HTTPプロトコル）が宛先まで届け、受け取った側が項目名で書類を整理する── これがフォーム送信の本質です。

POINT ── 「ブラウザは賢い郵便屋」

ブラウザはHTMLフォームを見て、HTTPリクエストに変換するエンジン。あなたが書いた<form>と<input>の組み合わせから、適切なHTTPリクエストを自動的に組み立てて送信してくれます。Springはその逆をやる── 受け取ったHTTPリクエストをJavaオブジェクトに復元する。両者の境界がHTTPプロトコルです。

03

なぜGETとPOSTを分けるのか

WHY GET AND POST

HTTPには8種類のメソッドがありますが、Webアプリ開発で9割使うのはGETとPOSTです。なぜ分かれているのか、その性質の違いを深掘ります。

get-vs-post-essence GETとPOSTの本質的な違い
// 性質の違い一覧
観点 GET POST
意図 情報を取得する 情報を送信・変更する
パラメータの置き場所 URL末尾（?key=value） リクエストボディ
冪等性 冪等（何回叩いても同じ） 非冪等（叩くたび新タスク追加）
ブックマーク可能 OK NG
URL履歴に残る パラメータごと残る URLのみ（値は残らない）
リロード時の挙動 そのまま再取得 警告「再送しますか?」

核心は「冪等性」。GETは何度叩いてもサーバの状態は変わらない（一覧を取るだけ）。POSTは叩くたびに変更が発生する（新タスクが増える）。この性質の違いから、ブラウザの扱いや業界慣習の違いが派生します。

// 冪等性（idempotency）とは // GETの場合（冪等） GET /tasks ← 1回目: 一覧3件取得 GET /tasks ← 2回目: 一覧3件取得 GET /tasks ← 3回目: 一覧3件取得 ↑ サーバ状態は変わらない ↑ 結果も毎回同じ // POSTの場合（非冪等） POST /tasks (title=A) ← 1回目: 1件追加(計4件) POST /tasks (title=A) ← 2回目: 1件追加(計5件) POST /tasks (title=A) ← 3回目: 1件追加(計6件) ↑ 叩くたび状態変化 ↑ 結果も毎回違う // 業界の鉄則 // 鉄則1: 副作用ある操作は POST // INSERT/UPDATE/DELETE → POST // SELECT → GET // 鉄則2: GETで状態を変えてはいけない // /tasks/delete?id=1 のような // GETでDELETEする設計は // 業界では大NG // 鉄則3: 機密情報は POST // パスワード等 → URLに乗せない // GETだとブラウザ履歴/サーバログに残る // 「冪等性違反」の実例 // もし「削除」をGETにしたら… <a href="/tasks/delete?id=1"> 削除 </a> // 問題1: Googleのクローラーが // 全リンク辿ってDELETE実行 // 問題2: ブラウザのプレビュー機能で // マウスホバーだけで削除実行 // 問題3: 戻るボタンで再実行 // → 必ず POST で実装する
POINT ── 「設計の原則」と「実装の手段」

GETとPOSTは単に「HTTPメソッドの違い」ではなく、「Webアプリ設計の根本原則」を体現しています。「読み取りはGET、書き込みはPOST」── これを徹底するだけで、検索エンジン対策、リロード対策、セキュリティ対策が同時に成立します。Vol.6で削除機能を作る時も必ずPOSTで実装します（GETでの削除はあり得ない）。

04

なぜTaskFormとTaskを分けるのか

FORM vs ENTITY

本編で「画面用とDB用は別物」と書きましたが、深い理由は3つあります。それぞれ業務開発の実例と照らし合わせて見ます。

three-reasons 分離する3つの理由
// 分離の3つの根拠
理由① 「画面に必要な項目」と「DBに必要な項目」が違う
DBにはあるけど画面に不要なもの：id（自動採番）、created_at（自動設定）。
画面にあるけどDBに不要なもの：「パスワード確認入力」「利用規約同意チェックボックス」「reCAPTCHAトークン」。
両者を1クラスにすると、どちらかが歪む。
理由② バリデーションは「画面入力」に対するもの
@NotBlank「タイトルは必須」というルールは画面入力に対する制約。Entityに付けると、DBから読み取った値にもチェックがかかって変な挙動になる。
バリデーションは Form の責任── これがVol.8の伏線。
理由③ Mass Assignment（一括代入）脆弱性の防止
もしControllerが直接Taskを引数で受けたら── 悪意あるユーザーが普通のフォームに「id=999」「done=true」を細工して送信すると、それらの値もJavaオブジェクトに入ってしまう（編集権限が破られる）。
Formに「画面用フィールドだけ」を定義すれば、フォーム入力できる項目が物理的に制限される── これがセキュリティ防御になる。
// 理由③ Mass Assignment攻撃の例 // === もしFormを使わなかったら === @PostMapping("/tasks") public String create(Task task) { taskMapper.insert(task); } // 通常のフォーム送信: // title=新タスク // → task.title に "新タスク" が入る // 攻撃者がブラウザのDevToolsで // フォームを編集: // title=ハック&done=true&id=999 // → task.title = "ハック" // → task.done = true (本来は false) // → task.id = 999 (本来DB採番) // → 不正なデータがDBに混入 // === Formを使うと防げる === public class TaskForm { private String title; // titleのみ } @PostMapping("/tasks") public String create(TaskForm form) { taskService.register(form); } // 攻撃者が title=ハック&done=true&id=999 送信 // → form.title = "ハック" だけ入る // → done と id は無視される // （Formに対応するフィールドがない） // Service側で安全に値を選んでTaskにコピー: public void register(TaskForm form) { Task task = new Task(); task.setTitle(form.getTitle()); // done と id は Service が制御 taskMapper.insert(task); } // → 攻撃者は title しか送れない // → 不正フィールドは入らない // 業務開発の鉄則 // 鉄則: 外部からの入力は // 必ず専用のFormで受け取る // 「Entityを直接受ける」設計は // セキュリティ的にも保守的にも // アウト
アナロジー ── 受付窓口と内部書類

役所の窓口を想像してください。「住民票の請求書」（Form）には名前と住所欄しかなく、利用者はその欄しか書けません。役所内部では「住民票」（Entity）に受付番号、職員サイン、発行日等の項目が追加される── これが業務の正しい流れ。

もし窓口の請求書に「受付番号」「職員サイン」欄まで載っていたら？ 悪意ある人が偽の番号やサインを書いて偽造申請できる。「外部に出す書類」と「内部の正式書類」を分けるのは、行政も Web アプリも同じ思想です。

POINT ── 「分離」は将来への投資

「今は項目が同じだから一緒で良くない?」── これに対する業界の答えは「ダメ。最初から分けろ」です。Vol.8のバリデーション、Vol.9の例外処理を経た時、分離していたコードと統合していたコードでは保守性が天と地。「最初の少しの面倒」が「後の大きな安心」を生むのが、業務開発の鉄則です。

05

用語カード：DTO

TERM — DATA TRANSFER OBJECT

業界で頻出するDTOという用語。Form もこの DTO の一種です。データの流れを整理するために役立つ概念を押さえます。

TERM CARD
DTO（Data Transfer Object）
データを運ぶための入れ物クラス
定義
「層と層の間でデータを運ぶための入れ物」となるクラスの総称。フィールドとgetter/setterだけを持ち、業務ロジックは持たないのが特徴。
構造
普通のJavaクラス（POJO）。Lombokの@Dataでgetter/setterを自動生成するのが業界の慣習。アノテーションは付かない（Bean登録不要）。
種類
Form ── 画面 → Controller の入口で使う DTO（本書のTaskForm）
View Model ── Controller → 画面 の出口で使う DTO
Request DTO ── REST APIの受信用 DTO
Response DTO ── REST APIの返却用 DTO
共通点：層間でデータを運ぶ「入れ物」
違い：どの層からどの層へ運ぶか
Entity との違い
EntityはDBテーブルに対応する「正式な業務データ」。DTOはそれを切り取って運ぶ箱。Entityはアプリ内で長く生き、DTOは1リクエスト内で短命。
嬉しさ
① 層の独立性（画面の要件変更でEntityが影響を受けない）
② セキュリティ（不要な項目を外部から受け取らない）
③ 柔軟性（複数Entityをまとめた画面用DTOを作れる）
dto-flow データの流れの中でのDTO
// データの流れと各「入れ物」の役割
[Browser] ↓ POST title=新タスク ↓ [Controller] ↓ TaskForm を受け取る ↓ (Form = 入口用DTO) ↓ [Service] ↓ TaskForm → Task に変換 ↓ (業務ロジック) ↓ [Mapper] ↓ Task で INSERT ↓ (Entity = 業務データ) ↓ [DB] ↓ tasks テーブル 逆方向（取得）: [DB] → Mapper → Service → Controller → Browser Task(Entity) Task(Entity) または TaskViewModel(出口用DTO)

Vol.5の現状：取得時はTask（Entity）をそのまま画面に渡している。これは小規模なら問題ないが、業務規模が大きくなると出口用DTO（TaskViewModelなど）を別に作ることもある。Vol.5は「Form だけ分離」でシンプルさを優先しています。

// 業界でよく見る DTO の例 // === 入口用 DTO（Form） === // Vol.5で作成 @Data public class TaskForm { private String title; // バリデーション用フィールド等 } // === 出口用 DTO（View Model） === // Vol.5では作っていないが… @Data public class TaskListItem { private Long id; private String title; private String doneLabel; // "完了" or "未完了" の文字列 private String createdAtJp; // "2026年5月18日 15:30" 形式 } // → 画面表示用の整形済みデータ // → Thymeleafで条件分岐や整形を // 書かなくて良くなる // === Request DTO（REST API） === // API化したら… @Data public class CreateTaskRequest { private String title; private String dueDate; } // === Response DTO（REST API） === @Data public class TaskResponse { private Long id; private String title; private Boolean done; } // === Entity（DB ↔ 業務） === @Data public class Task { private Long id; private String title; private Boolean done; private LocalDateTime createdAt; } // Entityはアプリの中核 // DTOはEntityを「切り取って」運ぶ箱
POINT ── 「DTOは1回切り」

DTOは1リクエスト内で使われて捨てられるのが普通です。Entityは「業務の中核データ」なのでアプリ内で長く生きますが、DTOは「その瞬間にデータを運ぶための紙袋」。運び終わったらゴミ箱でOK。この使い分け感覚が、業界でDTOを語る時の前提です。

06

PRGパターンの本質

PRG IN DEPTH

本編で「POST後はリダイレクト」と覚えました。なぜそうするのか、仕組みのレベルで深掘ります。「リロードで二重登録」が起きる本当の原因を見ます。

prg-essence PRGの仕組み
// ブラウザの「リロード」の正体

ブラウザのリロード（F5）は「直前のリクエストを再実行する」動作です。直前がGETならGETを再実行、POSTならPOSTを再実行。

POST後にHTMLを返す設計では、ブラウザのURL欄はPOST /tasksのまま。リロードすると同じPOSTを再送信してしまう── これが二重登録の正体。

PRGの解決策：POST処理後、サーバはHTML（200 OK）ではなく、リダイレクト指示（302 Found）を返す。ブラウザはこれを受けて自動的に別のURLにGETを投げ直す。結果、URL欄がGET /tasksになり、リロードしてもGETの再実行になるだけで安全。

POINT：「Post → Redirect → Get」の3ステップで、「ユーザーがリロードした時の最後のリクエスト」がGETになるのが PRG の核心。
// PRGなしの場合（HTTPの会話） [Browser] [Server] POST /tasks (title=A) →→→→→→ create()処理 1件INSERT実行 一覧HTMLを返す ←←←←←← HTTP 200 OK + HTMLボディ URL欄: /tasks (POSTのまま) 画面表示: 一覧 (4件) ★ユーザーがF5 POST /tasks (title=A) ← 同じPOST再送！ →→→→→→ create()処理 ★1件INSERT実行 (2件目の同じタスク) 一覧HTMLを返す ←←←←←← HTTP 200 OK 画面表示: 一覧 (5件) → 同じタスクが2件登録！ // PRGありの場合 [Browser] [Server] POST /tasks (title=A) →→→→→→ create()処理 1件INSERT実行 リダイレクト指示 ←←←←←← HTTP 302 Found Location: /tasks (HTMLボディなし) // ブラウザが自動でGET実行 GET /tasks →→→→→→ list()処理 SELECT発行 一覧HTMLを返す ←←←←←← HTTP 200 OK + HTMLボディ URL欄: /tasks (GETに変わった) 画面表示: 一覧 (4件) ★ユーザーがF5 GET /tasks ← GETを再実行 →→→→→→ list()処理 SELECT発行 一覧HTMLを返す ←←←←←← HTTP 200 OK 画面表示: 一覧 (4件) → 何度F5しても件数は変わらない

PRGが解決する3つの問題：

リロードによる二重登録
これがPRGの主目的。決済処理など重要な場面では必須。
ブラウザ「戻る」での再送信警告
PRGなしだと「戻る」で「フォームを再送信しますか?」警告が出る。ユーザー体験を損なう。PRGありなら警告なしで普通に戻れる。
URLの整合性
ユーザーが「URLをコピペして友達に送る」場合、PRGなしだと POST /tasks がURL欄にあって意味不明。PRGありなら GET /tasks（一覧画面のURL）でブックマーク可能。
POINT ── 「PRG＝Webアプリの大前提」

PRGは「選択肢の1つ」ではなく「Webアプリの大前提」です。業務系のレビューでは「POSTの後にredirectしてない」はほぼ100%指摘されます。return "redirect:/xxx";の1行を忘れないだけで、品質が大きく変わります。

07

「同じURL」で違う処理を呼び分ける仕組み

SAME URL, DIFFERENT HANDLERS

Vol.5では GET /tasks（一覧）と POST /tasks（登録）が同じURLですが別の処理になりました。Springはどうやって見分けているのか── ルーティングの仕組みを解明します。

routing-mechanism Springのルーティング表
// アプリ起動時に作られる対応表

Springは起動時、@Controllerのクラスをスキャンし、各メソッドのアノテーション（@GetMapping、@PostMapping等）を読み取って対応表を作ります。

対応表のキーは「（HTTPメソッド, URLパス）」のペア。値は「呼び出すControllerメソッド」。同じURLパスでもHTTPメソッドが違えば別のキーとして登録されます。

核心：URLだけでなく「HTTPメソッド + URL」でルーティングする。だからGET /tasksとPOST /tasksを別物として扱える。
// Vol.5完成時のルーティング表（イメージ） URL ROUTING TABLE ────────────────────────────────── キー 処理 ────────────────────────────────── (GET, "/") HelloController.hello() (GET, "/tasks") TaskController.list() (GET, "/tasks/new") TaskController.newForm() (POST, "/tasks") TaskController.create() ────────────────────────────────── // 同じ "/tasks" でも // GET と POST で別エントリー // リクエスト処理の流れ [Browser] ↓ POST /tasks (title=A) [Tomcat] ↓ リクエスト受信 ↓ DispatcherServlet に渡す [Spring DispatcherServlet] ↓ HTTPメソッド: POST ↓ URLパス: /tasks ↓ 対応表検索 ↓ → (POST, "/tasks") を引く [HandlerMapping] ↓ TaskController.create() ↓ にマッチ [Argument Resolver] ↓ メソッドの引数 TaskForm を見て ↓ リクエストボディから値抽出 ↓ TaskForm インスタンスを作って ↓ title フィールドに値をセット [TaskController] ↓ create(taskForm) 呼び出し ↓ taskService.register(taskForm) ↓ return "redirect:/tasks" [ViewResolver] ↓ "redirect:" 検出 ↓ HTTP 302 + Location: /tasks ↓ をレスポンスに [Browser] ↓ レスポンス受信 ↓ Location ヘッダーを見て ↓ GET /tasks を自動実行
POINT ── 「アノテーションは目印」(Vol.2 ADDの再確認)

@GetMappingや@PostMappingはSpringに「これが何の処理か」を教える目印。これを起動時に集めて対応表を作る── このパターンはVol.2 ADDで学んだ「アノテーションは目印」の原則そのものです。新しい機能を追加する時も同じ：適切な目印を貼る → Springが自動で対応表に登録 → リクエストが来たら呼ばれる。この感覚があれば、Vol.6以降の機能追加も同じパターンで応用できます。

th:object と th:field の魔法

TH:OBJECT & TH:FIELD MAGIC

本編の new.html で書いた th:object="${taskForm}" と th:field="*{title}"── たった2つの属性で、JavaオブジェクトとHTMLフォームが繋がる仕組みを解明します。

th-attrs-mechanism Thymeleafが生成するHTML
// Thymeleafの内部動作

Thymeleafはサーバ側でHTMLを動的に生成するテンプレートエンジン。th:から始まる属性を見て、実際にブラウザに送るHTMLを作り変えます。th:objectとth:fieldはその魔法の一種です。

th:object="${taskForm}" の役割
「このフォーム内では、Modelに載っているtaskFormを基準にしてth:fieldを解釈する」とThymeleafに伝える。form タグ全体のコンテキスト宣言。
th:field="*{title}" の役割
「このinputはtaskForm.titleと紐付ける」と宣言。*は「th:objectで指定したオブジェクトの」の略記。Thymeleafがこれを見てid・name・value の3属性を自動生成。
生成されるHTMLの3点セット
id="title"（label の for と紐付け用）
name="title"（フォーム送信時のキーに使われる）
value="${taskForm.title}"（編集時に既存値を表示）
この3つを手動で書くのは面倒。th:field1つで済むのが魔法。
// あなたが書いた new.html <form th:action="@{/tasks}" th:object="${taskForm}" method="post"> <input type="text" th:field="*{title}"> </form> // Thymeleafが生成するHTML <form action="/tasks" method="post"> <input type="text" id="title" // 自動生成 name="title" // 自動生成 value=""> // 自動生成 </form> // taskForm.title が null なので value="" // 編集画面なら value="既存値" になる // もしth:fieldを使わなかったら <input type="text" id="title" name="title" th:value="${taskForm.title}"> // id, name, th:value を全部手で書く必要あり // しかも編集画面で値を保つには // th:value を必ず付ける（忘れがち） // th:field の真の威力 // 編集画面で使うと… // Controller側で: TaskForm form = new TaskForm(); form.setTitle("既存タイトル"); model.addAttribute("taskForm", form); // Thymeleafが自動で: <input value="既存タイトル"> // 編集画面で既存値が表示される // → Vol.6 編集機能で活きる仕掛け
POINT ── 「name 属性が連携の鍵」

HTMLフォームとSpringの連携はname属性を介して行われます。フォーム送信時、ブラウザはname=値の形でデータを送り、Springはnameを見てJavaオブジェクトの対応フィールドに値を詰める。th:fieldはこのname属性を自動生成してくれる便利機能。「name属性こそがフォームの連携の核心」と覚えておけば、混乱した時に立ち戻れます。

@ModelAttribute はどう動くか

HOW @MODELATTRIBUTE WORKS

Vol.5本編で create(TaskForm taskForm) と書いたら、引数の taskForm に勝手にフォーム値が入っていた── この魔法の正体を解明します。

model-attribute-mechanism 暗黙の @ModelAttribute
// Springのデータバインディング

Spring MVCは、Controllerメソッドの引数の型を見て、「何をどこから取り出して渡すか」を判断します。これを引数リゾルバ（Argument Resolver）と呼びます。

POSTメソッドの引数に普通のクラス（TaskFormのような）が現れたら、Springは暗黙的に@ModelAttributeと解釈します。これにより：

  1. new TaskForm()でインスタンスを作る
  2. リクエストパラメータ（title=...）の名前を見て、TaskFormの同名フィールドに値をセット
  3. 完成したTaskFormを引数として渡す

「リクエストパラメータの名前」と「フィールド名」が一致することが鍵。だからth:fieldでname属性をtitleにし、TaskFormにもtitleフィールドがある── これで自動的に繋がります。

// Controllerの引数バインディング // あなたが書いたコード @PostMapping("/tasks") public String create( TaskForm taskForm) { // ← Spring が自動で詰める taskService.register(taskForm); return "redirect:/tasks"; } // Springが裏でやっていること // 1. リクエストパラメータ抽出 String titleParam = request.getParameter("title"); // → "新しいタスク" を得る // 2. TaskFormインスタンス生成 TaskForm form = new TaskForm(); // 3. setter経由で値セット form.setTitle(titleParam); // → リフレクションで setter を呼ぶ // → @Data が生成した setTitle が動く // 4. メソッドに渡す controller.create(form); // あなたは「引数を書く」だけで // 中身が満たされて渡される // 暗黙の @ModelAttribute // 以下の2つは同じ意味 // 省略形（よく使う） public String create(TaskForm form) {...} // 明示形 public String create( @ModelAttribute TaskForm form) {...} // POSTメソッドの普通のクラス引数は // 自動的に @ModelAttribute 扱い // → 省略するのが業界の慣習 // バインディングが失敗するケース // ケース1: setterがない public class TaskForm { private String title; // @Data なし、setTitle なし } // → 値がセットされない（null のまま） // ケース2: フィールド名が違う public class TaskForm { private String taskTitle; // name属性が title だと不一致 } // → 値がセットされない（null のまま） // ケース3: 型が変換できない public class TaskForm { private Integer title; } // → "新しいタスク" を Integer に変換不可 // → BindException 発生
POINT ── 「Springは型と名前で動く」

Springの自動化の仕掛けは「型と名前」を頼りに動きます。引数の型を見て「何を作るか」を決め、リクエストパラメータの名前とフィールドの名前を照合して値を詰める。「規約による設定」と呼ばれるこの設計思想は、命名規則を守ればコードが少なく済む代わりに、名前を1文字でも間違えると静かに失敗する恐ろしさを併せ持ちます。命名統一こそが Spring 業務開発の要です。

リダイレクト vs フォワードの違い

REDIRECT vs FORWARD

Vol.5で "redirect:/tasks" を使いました。実は対になる "forward:/xxx" もあります。2つの「画面遷移」の違いを整理します。

redirect-vs-forward 2つの遷移方法
// 仕組みの違い
→
リダイレクト（redirect）
仕組み：サーバが「次はこのURLに行ってね」とブラウザに指示（HTTP 302）。ブラウザが改めてリクエストを送る。
URL欄：変わる（遷移先のURLに）
HTTPやりとり：2往復
業界用語：「クライアントリダイレクト」とも言う
⇒
フォワード（forward）
仕組み：サーバ内部で別のControllerに処理を渡す。ブラウザは何も知らない。
URL欄：変わらない（元のURLのまま）
HTTPやりとり：1往復
業界用語：「サーバサイドフォワード」とも言う

使い分けの原則：

redirect 使うべき場面

・POST処理の後（PRG）
・処理完了後の画面遷移
・URL変更が必要な時
・業務開発では9割こちら

forward 使うべき場面

・エラー画面表示時（バリデーション失敗 → 同URLでフォーム再表示）
・実は普通のreturn "テンプレート名"もフォワードの一種
・明示的に書くケースは稀

// 仕組みの違い（HTTPやりとり） // === redirect の場合 === [Browser] [Server] POST /tasks →→→→ create() INSERT実行 return "redirect:/tasks" ←←←← HTTP 302 Location: /tasks (URL欄が /tasks に書き換わる) GET /tasks (自動) →→→→ list() SELECT実行 一覧HTMLを返す ←←←← HTTP 200 + HTML // → 2往復 // → URL欄は最終的に /tasks // === forward の場合 === [Browser] [Server] POST /tasks →→→→ create() INSERT実行 return "forward:/tasks" // ↑ サーバ内部で別の処理へ list()が呼ばれる SELECT実行 一覧HTMLを返す ←←←← HTTP 200 + HTML (URL欄は /tasks のまま、POST送信時のまま) // → 1往復 // → URL欄は POST /tasks のまま // → リロードで二重登録の罠 // Springでの書き方 // redirect（推奨） return "redirect:/tasks"; // HTTP 302 を返す // forward（稀） return "forward:/tasks"; // サーバ内で別のハンドラへ // 暗黙のforward（毎回これ） return "tasks/list"; // テンプレート名指定もforwardの一種 // → サーバ内でテンプレートを描画 // 使い分けの実例 // 一覧表示（普通のフォワード） @GetMapping("/tasks") public String list(...) { ... return "tasks/list"; // 暗黙のforward (テンプレート描画) } // 登録処理（明示のリダイレクト） @PostMapping("/tasks") public String create(...) { ... return "redirect:/tasks"; // 明示のリダイレクト (PRG) } // バリデーション失敗時（暗黙のforward） @PostMapping("/tasks") public String create(...) { if (errors) { return "tasks/new"; // 同じURLでフォーム再表示 // → エラーメッセージと共に // → Vol.8 で詳しく扱う } ... }
POINT ── 「書き込み後はredirect、表示はそのままテンプレート名」

業務開発のシンプルなルール：書き込み（INSERT/UPDATE/DELETE）処理を含むメソッドの最後は必ず redirect:。表示専用メソッドの最後はテンプレート名のみ。この使い分けを守るだけで、PRG パターンが自然に成立し、ユーザー体験も保たれます。
