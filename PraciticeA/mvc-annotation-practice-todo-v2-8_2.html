01

この巻の地図

WHAT / HOW / WHY

バリデーションは「3点セット」で構成されます：① Formに制約付与、② Controllerで検証実行、③ テンプレートでエラー表示。この流れに沿って実装します。

WHAT
何を作るか
新規登録と編集の両方に バリデーション を入れる。空欄やタイトル長過ぎを弾く。エラー時は赤字メッセージ表示、入力値は保持。messages.propertiesで日本語化。
HOW
どう作るか
① Form に @NotBlank @Size を付ける
② Controller に @Valid と BindingResult を入れる
③ エラー時はテンプレート名を返す（リダイレクトしない）
④ テンプレートに th:errors を追加
⑤ messages.properties でメッセージ日本語化
WHY
なぜそうするか
「信頼できない入力」から業務システムを守る最初の砦。クライアント側のHTML5バリデーションだけでは不十分（簡単に回避される）。サーバ側で必ず検証が業界鉄則。深掘りは別冊で。
vol-8-flow バリデーションが入ると処理が分岐する
// バリデーション成功時 / 失敗時の2つのフロー
★ 成功時（入力が妥当）★ ブラウザ が Controller に POSTを送る行き 送信: POST /tasks (title="買い物") Spring が Form の 制約を検証行き @NotBlank・@Size をチェック Spring が BindingResult に 「エラーなし」を返す帰り Controller が Service の register() を呼ぶ行き 通常のCRUDフロー（Vol.5と同じ）でDB登録、redirect
★ 失敗時（入力が不正）★ ブラウザ が Controller に POSTを送る行き 送信: POST /tasks (title="" 空欄) Spring が Form の 制約を検証行き Spring が BindingResult に 「@NotBlank違反」を返す帰り Controller が BindingResult に hasErrors() を確認 true が返る → エラーあり Controller が ブラウザ に フォーム画面を返す帰り return "tasks/new" （redirectしない） ブラウザ が ユーザー に エラー入りフォームを表示 入力値は保持、エラーメッセージが赤字で表示
// この巻でやることのコード骨格 // ① TaskForm に制約付与 @Data public class TaskForm { @NotBlank @Size(max = 100) private String title; } // ② Controllerに @Valid と BindingResult @PostMapping("/tasks") public String create( @Valid TaskForm taskForm, BindingResult bindingResult) { if (bindingResult.hasErrors()) { return "tasks/new"; // ↑ redirectせずテンプレートに戻る } taskService.register(taskForm); return "redirect:/tasks"; } // ③ テンプレートにエラー表示 <input th:field="*{title}"> <span th:if="${#fields.hasErrors('title')}" th:errors="*{title}" style="color:red"></span> // ④ messages.properties で日本語化 NotBlank.taskForm.title=タイトルは必須です Size.taskForm.title=タイトルは{2}文字以内です
POINT ── 「redirectしないのが新しい」

これまでのVol.5・6・7では「POSTの後はredirect」が鉄則でした。でもバリデーションエラー時は例外── redirectせず、テンプレート名をそのまま返す。理由は「入力値とエラー情報をModelで運ぶ」必要があるから（redirectすると消える）。「成功時はredirect、エラー時はテンプレート直接」── これがバリデーションを入れたPRGの進化形です。

03

用語カード：Bean Validation

TERM — BEAN VALIDATION

Java界の業界標準のバリデーション仕組み。Java EE / Jakarta EEの仕様で、Spring Bootも採用。「クラスのフィールドにアノテーションで制約を付ける」のが特徴です。

TERM CARD #01
Bean Validation
Java標準のバリデーション仕様（Jakarta Validation）
定義
Java EE / Jakarta EE の標準仕様。クラスのフィールドにアノテーションで制約を宣言し、専用のバリデータが検証実行する仕組み。Spring Boot 3.x では jakarta.validation パッケージを使用。
主な標準アノテーション
@NotNull ── null禁止
@NotBlank ── null・空文字・空白のみ禁止（String用）
@NotEmpty ── null・空コレクション禁止
@Size(min=, max=) ── 長さチェック（String/Collection）
@Min / @Max ── 数値範囲
@Email ── メールアドレス形式
@Pattern(regexp=) ── 正規表現
他にも多数（別冊で網羅）
仕組み
① クラスのフィールドにアノテーションで制約宣言
② バリデータがリフレクションで全フィールドをチェック
③ 違反があれば BindingResult にエラー情報を格納
嬉しさ
① 宣言的：何がOK・NGかがクラス定義から読める
② 業界標準：あらゆるJavaフレームワークで通じる
③ 再利用可能：同じ制約が複数フォームで使える
④ カスタム可能：独自バリデーターも追加できる
bean-validation-example 代表的な制約の組み合わせ
// 制約付きフォームの読み方

このTaskFormを見れば、業務ルールが一目で分かる：

@Data
public class TaskForm {
  @NotBlank          ← 必須（空欄禁止）
  @Size(max = 100)   ← 100文字以内
  private String title;
}

クラスを読むだけで「titleは必須」「100文字以内」という業務ルールが伝わる。コメントを書かなくても制約が自己説明している。これが「宣言的プログラミング」の威力です。

@NotBlank vs @NotNull vs @NotEmpty

入力値 @NotNull @NotEmpty @NotBlank
null NG NG NG
"" (空文字) OK NG NG
" " (空白のみ) OK OK NG
"abc" OK OK OK

テキスト入力には@NotBlankが最適。空白文字だけの入力も拒否してくれる。

// jakarta.validation の代表的アノテーション // === 必須 === @NotNull // null禁止 @NotEmpty // null・空コレクション禁止 @NotBlank // null・空文字・空白禁止 // === サイズ === @Size(min = 1, max = 100) // 文字列やコレクションの長さ // === 数値範囲 === @Min(0) @Max(999) @Positive // 正の数のみ @PositiveOrZero @Negative @NegativeOrZero // === パターン === @Email @Pattern(regexp = "^[0-9]+$") @URL // 別ライブラリ // === 日付 === @Past // 過去日付 @Future // 未来日付 @PastOrPresent @FutureOrPresent // === 真偽値 === @AssertTrue @AssertFalse // 組み合わせ可能 public class UserForm { @NotBlank @Size(min = 2, max = 30) private String name; @NotBlank @Email private String email; @NotNull @Min(18) @Max(120) private Integer age; @Past private LocalDate birthDate; } // クラスを読むだけで業務ルールが分かる
POINT ── 「Bean Validation の歴史」

Bean Validation はJava EE 6 (2009年)から続く業界標準。Spring Boot 3.x では Jakarta EE の jakarta.validation パッケージを使用（旧 javax.validation から名前変更）。15年以上磨かれてきた仕組みなので、安心して使えます。Java界で「バリデーション」と言えばこれです。

用語カード：@Valid と BindingResult

TERM — @VALID & BINDINGRESULT

Controllerでバリデーションを動かす2つの仕掛け。@Validが「検証してくれ」と命令、BindingResultが「結果を受け取る」入れ物。必ずペアで使います。

TERM CARD #02
@Valid と BindingResult
バリデーション実行と結果受取の組み合わせ
@Valid とは
「このFormの制約を検証してください」と Spring に依頼するアノテーション。引数に付ける。
BindingResult とは
バリデーション結果を格納するオブジェクト。hasErrors()でエラー有無を確認、getFieldErrors()で詳細取得可。
ペアで使う理由
@Valid 単独だと、エラー発生時に Spring が例外を投げる（MethodArgumentNotValidException）→ 500エラー画面。
BindingResult を引数に追加すると、Spring が例外を投げず、エラー情報を BindingResult に詰める。Controller側でエラー処理を制御できる。
引数順の重要ルール
BindingResult は @Valid の直後に置く必須ルール。間に他の引数が入ると動かない。
判定パターン
if (bindingResult.hasErrors()) { ... } でエラー時の分岐を作る。エラーならフォーム再表示、エラーなしなら通常のCRUD処理。
valid-bindingresult-usage 正しい使い方
// @Valid と BindingResult のペア

正しい引数順：

public String create(
    @Valid TaskForm taskForm,
    BindingResult bindingResult) {  ← @Validの直後！

⚠ 間違った引数順

public String create(
    @Valid TaskForm taskForm,
    Model model,                  ← 間に他の引数
    BindingResult bindingResult)  ← ここではダメ！

これだと BindingResult が機能せず、エラー時に例外が投げられる。ペアは必ず連続して置くのが鉄則。

正しい引数順（複数引数の場合）

public String update(
    @PathVariable Long id,        ← 他の引数
    @Valid TaskEditForm form,     ← @Valid
    BindingResult bindingResult,  ← その直後！
    Model model) {                ← その後でOK
// 正しい使い方 @PostMapping("/tasks") public String create( @Valid TaskForm taskForm, BindingResult bindingResult) { if (bindingResult.hasErrors()) { // エラーあり → フォーム再表示 return "tasks/new"; } // エラーなし → 通常処理 taskService.register(taskForm); return "redirect:/tasks"; } // BindingResult が無い場合 @PostMapping("/tasks") public String create( @Valid TaskForm taskForm) { // ↑ BindingResult なし // エラー時： // MethodArgumentNotValidException // が投げられる // → 500エラー画面表示 // → ユーザーに優しくない } // BindingResult の便利メソッド bindingResult.hasErrors() // → エラー有無 (boolean) bindingResult.getFieldErrors() // → 全フィールドエラーのリスト bindingResult.hasFieldErrors("title") // → 特定フィールドのエラー有無 bindingResult.getFieldError("title") // → 特定フィールドのエラー情報 // 引数順ルール（厳守） // ✓ OK：@Valid と BindingResult が連続 create(@Valid Form form, BindingResult br, ...) // ✓ OK：他の引数が前後にあるのは平気 update(@PathVariable Long id, @Valid Form form, BindingResult br, Model model) // ✗ NG：@Valid と BindingResult の間に他 create(@Valid Form form, Model model, BindingResult br) // → エラー時例外
POINT ── 「ペアで覚える」

@Valid と BindingResult は「対」で覚えるのが鉄則です。「検証してくれ + 結果受け取る」── 必ずセット。引数の順番も「Validのすぐ後にBindingResult」と暗記。これだけ守れば、業務開発でバリデーションを書く時に迷いません。

05

用語カード：エラー時の挙動（PRGの例外パターン）

TERM — ERROR HANDLING IN PRG

これまでのPRG（POST→Redirect→GET）の例外パターン。バリデーション失敗時はredirectしないのが正解。理由を整理します。

TERM CARD #03
エラー時の挙動（PRGの例外）
バリデーションエラー時は redirect ではなく forward
通常のPRG
POST → 処理 → return "redirect:/..."; → ブラウザが GET を送り直す → 新画面表示
エラー時の例外
POST → 検証失敗 → return "tasks/new";（テンプレート名直接） → 同じ画面を同じURLで表示
★ redirectしないのが特徴
なぜredirectしないか
redirectすると Model の中身が消える。
エラー情報、入力値、BindingResultはすべて1リクエスト内でしか保持されない。redirectすると新しいリクエストになり、これらが全部失われる。
→ エラーメッセージも消える、入力値も消える、ユーザーが最初から入力し直しに
テンプレート直接返す挙動
これは暗黙のforward。サーバ内部でテンプレートをレンダリングしてHTMLを返す。Model の中身は保持されたまま。
業界の通称
「バリデーションエラー時はforward、成功時はredirect」── 業界の常識的パターン。
prg-with-validation バリデーション付きPRGの全体像
// 成功時 vs 失敗時のフロー差
★ バリデーション成功時（PRG通常） Controller が Service に 登録処理を依頼行き Service が DB に INSERT実行行き Controller が ブラウザ に redirect指示を返す帰り return "redirect:/tasks" → HTTP 302 ブラウザ が サーバ に GET /tasks を送信行き（新リクエスト） ★ 新リクエストなのでModel空っぽに（問題なし） 一覧画面表示
★ バリデーション失敗時（PRG例外パターン） Spring が BindingResult に エラー情報を格納帰り 内容: title=「必須項目です」のエラー Controller が BindingResult の hasErrors() を確認 true が返る Controller が ブラウザ に フォームHTMLを返す帰り（同リクエスト内） return "tasks/new" → forward でテンプレート描画 Modelには taskForm（入力値）と BindingResult（エラー）が入っている Thymeleaf が Model から 値を読み取って描画 入力値を value に、エラーを赤字に表示 ブラウザ が ユーザー に エラー入りフォームを表示 「タイトルは必須です」など赤字、入力値は保持されたまま
// Controllerコード対比 // === 成功時（PRG通常）=== if (!bindingResult.hasErrors()) { taskService.register(taskForm); return "redirect:/tasks"; // ↑ 別URLへリダイレクト // → 新リクエスト → Model消える(問題ない) } // === 失敗時（PRGの例外） === if (bindingResult.hasErrors()) { return "tasks/new"; // ↑ テンプレート名を直接返す // → forward でレンダリング // → Modelは保持される // → 入力値もエラー情報も表示できる } // なぜModelが大事か // Spring の Modelの中身（エラー時）: Model{ "taskForm": TaskForm{title=""}, // ↑ ユーザーが入力した値 "org.springframework.validation. BindingResult.taskForm": BindingResult{ errors: [ {field="title", code="NotBlank", message="必須項目です"} ] } } // → Thymeleafがこれを読んで: // 1. inputに value="" を入れる // 2. エラーメッセージを表示 // redirectしてしまうと… // ✗ NG例 if (bindingResult.hasErrors()) { return "redirect:/tasks/new"; // ↑ ブラウザが GET /tasks/new を送る // → 新リクエスト = 新Model // → エラー情報も入力値も消える // → ユーザーが空っぽのフォームを見る // → 何がエラーだったか分からない } // → だから「エラー時はforward」 // が業界の鉄則
POINT ── 「Modelは1リクエスト内のみ」

Spring の Model は1リクエストの中だけ生きる。リクエストが終わるとゴミ箱行き。なので「リダイレクトを挟むとModelが消える」のは Spring の根本仕様。バリデーションエラー時は 同じリクエスト内で処理を終える必要があり、それがreturn "tasks/new"; というテンプレート直接返却なのです。深い仕組みは別冊で。

メッセージキーの命名規則

Spring は以下の優先順位でメッセージを探します：

  1. 制約名.Form名.フィールド名 ← 最も具体的（推奨）
    例：NotBlank.taskForm.title
  2. 制約名.フィールド名 ← フィールドが共通名の時
    例：NotBlank.title
  3. 制約名.型名 ← 型ごとの汎用メッセージ
    例：NotBlank.java.lang.String
  4. 制約名 ← フォールバック
    例：NotBlank

業務開発では最も具体的なパターン（1番）で書くことが多い。「どのフォームのどのフィールド」を特定してメッセージを変えられるから。

// src/main/resources/messages.properties # === TaskForm（新規登録） === NotBlank.taskForm.title=タイトルは必須です Size.taskForm.title=タイトルは{2}文字以内で入力してください # === TaskEditForm（編集） === NotBlank.taskEditForm.title=タイトルは必須です Size.taskEditForm.title=タイトルは{2}文字以内で入力してください // プレースホルダの解説 // {0} = フィールド名 // {1} = max値（@Size の場合） // {2} = max値（{1}=max, {2}=max のことも） // 注: 制約により位置が違うので公式参照 // @Size(max = 100) の場合: // Size.taskForm.title=...{2}文字以内... // → "100" に置換される // 文字化け対策 // messages.properties は // 必ず UTF-8 で保存する // Eclipse の保存時のエンコーディング設定: // Window → Preferences // → General → Workspace // → Text file encoding: UTF-8 // または application.properties に: // spring.messages.encoding=UTF-8 // メッセージキーの優先順位（実例） // @NotBlank が title に付いた時、Springは: 1. NotBlank.taskForm.title ← 探す 2. NotBlank.title ← 無ければ 3. NotBlank.java.lang.String 4. NotBlank ← 最終フォールバック 5. (デフォルト英語メッセージ) ← どれもなければ // 1番にヒットすればそれを使う // 他のアノテーションへの対応 # @Email を使った場合 Email.userForm.email=有効なメールアドレスを入力してください # @Min(18) を使った場合 Min.userForm.age=18歳以上を入力してください # @Pattern を使った場合 Pattern.userForm.phone=半角数字で入力してください // すべて「制約名.Form名.フィールド名」のパターン
POINT ── 「UTF-8で保存」

messages.properties は必ずUTF-8で保存。日本語が文字化けすると"???????"のような表示になります。Eclipse のデフォルトエンコーディング設定を確認しておきましょう（Window → Preferences → General → Workspace → Text file encoding: UTF-8）。プロパティファイルは古い慣習でISO-8859-1だったので、Spring Boot 3.x でもデフォルト挙動を確認すべきです。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

Vol.8のハマりは「依存」「import」「引数順」「redirect」「文字コード」の5軸に分布。すべて初心者の通過儀礼です。

⚠ ハマり 1 ── @NotBlank を付けたのに空欄登録できてしまう
SYMPTOM Formに @NotBlank を付けてもバリデーションが効かない。空欄でも登録成功してしまう。
CAUSE ① pom.xml に spring-boot-starter-validation 依存が無い、または
② Controller に @Valid を書き忘れた、または
③ import を javax.validation.*（旧）にしてしまった（Spring Boot 3.x は jakarta.validation.*）
FIX ① pom.xml に依存を追加してMaven Update
② Controllerの引数に @Valid TaskForm taskForm を確認
③ import文を確認し、jakarta.* に統一
⚠ ハマり 2 ── エラー画面に行ってしまう（MethodArgumentNotValidException）
SYMPTOM 空欄送信するとWhitelabel Error Page (500)が表示される。コンソールに MethodArgumentNotValidException。
CAUSE @Valid はあるが BindingResult が無い、または引数順が間違っている（Validの直後にBindingResultが無い）。Springが例外を投げてしまう。
FIX BindingResult bindingResult を@Validのすぐ後に置く。他の引数（Model等）を間に挟まない。
⚠ ハマり 3 ── エラー時にも一覧画面に遷移してしまう（入力値消失）
SYMPTOM 空欄送信時に一覧画面にリダイレクトされる。エラーメッセージも表示されない。
CAUSE if (bindingResult.hasErrors()) の分岐で return "redirect:/..."; と書いてしまった。redirect は新しいリクエストになり、Model（エラー情報・入力値）が消える。
FIX エラー時は return "tasks/new";（テンプレート名だけ）にする。「エラー時はforward、成功時はredirect」が鉄則。
⚠ ハマり 4 ── エラーメッセージが英語のまま表示される
SYMPTOM バリデーションエラーは表示されるが、メッセージが "must not be blank" のまま日本語にならない。
CAUSE ① messages.properties が存在しない、または
② キーの命名が間違っている（NotBlank.taskform.title のように小文字始まりにしてしまった等）、または
③ src/main/resources ではなく別の場所に置いた
FIX ① ファイルを src/main/resources 直下に置く
② キー名は制約名.Form名（キャメルケース）.フィールド名（例：NotBlank.taskForm.title）
③ アプリ再起動してファイル変更を反映
⚠ ハマり 5 ── 日本語メッセージが文字化けする
SYMPTOM 日本語メッセージが ??????? や 譁?字? のように文字化けする
CAUSE messages.properties がUTF-8以外（Shift-JIS、ISO-8859-1等）で保存されている。Eclipse のデフォルト保存エンコーディングが古い設定のまま。
FIX ① Eclipse 設定で Text file encoding を UTF-8 に変更
② または application.properties に spring.messages.encoding=UTF-8 を追加
③ ファイルを開き直してUTF-8で再保存
POINT ── 「バリデーションは初心者の登竜門」

Vol.8のハマりはすべて初心者必修の経験です。「Validation依存忘れ」「import違い」「引数順違反」「redirect間違い」「文字化け」── どれも実務で遭遇する典型的な落とし穴。一度経験しておくと、業務でも即座に原因がわかるようになります。Vol.3 ADD で学んだ9ステップ思考フローがここでも活躍します。
