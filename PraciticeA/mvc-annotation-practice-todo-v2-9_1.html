03

用語カード：例外（Exception）の基礎

TERM — EXCEPTION BASICS

Java界の例外（Exception）は、メソッドが「異常状態を呼び出し元に伝える仕組み」。エラーが起きた時、戻り値で「-1」「null」を返すのではなく、例外オブジェクトを「投げる」のがJavaの流儀です。

TERM CARD #01
例外（Exception）
異常事態を呼び出し元に伝える仕組み
定義
Java の言語仕様レベルで組み込まれた、メソッドの異常終了を表現する仕組み。throw で投げ、try-catch で受ける。
仕組み
throw された例外は、try-catch で受け止められるまで呼び出し階層を逆方向にどんどん遡る。誰も受け止めなければ最終的にプログラムが終了。
戻り値との違い
戻り値で異常を返す方式（C言語等）：呼び出し元がチェック忘れするとバグ。
例外で投げる方式：チェック忘れても「黙って続行」されない（自動で伝搬する）。安全性が高い。
Javaの例外階層
Throwable ─ Error（システム的致命傷）
Throwable ─ Exception ─ アプリで扱う例外
　Exception ─ RuntimeException（実行時例外）
　Exception ─ それ以外（検査例外、深掘りは別冊で）
業界の使い方
業務アプリで作る独自例外はほぼ全て RuntimeException 継承。「投げるだけで throws 宣言が要らない」のが楽。
exception-basics 例外の基本動作
// 例外が伝搬する仕組み

例外の最大の特徴は「呼び出し階層を遡る」こと。Service で投げた例外が、Service を呼んだ Controller を素通りし、Controller を呼んだ Spring まで届く── これが自動的に起きます。

従来の戻り値方式との対比：

// 戻り値方式（C言語的）
Task task = service.findById(id);
if (task == null) {  // ← チェック忘れがち
  // エラー処理
}

// 例外方式（Java的）
Task task = service.findById(id);
// ← null チェック不要
// 異常時は例外が勝手に上に飛ぶ
// この行に来た時点で task は確実に non-null

戻り値で異常を表すと「チェック忘れ」で事故が起きる。例外は「黙って続行できない」仕組みなので、エラー時の挙動が予測可能になります。

// 基本：throw と try-catch // === 投げる側 === public Task findById(Long id) { Task task = taskMapper.findById(id); if (task == null) { throw new TaskNotFoundException(id); // ↑ 例外を投げる // この後の行は実行されない } return task; } // === 受ける側（try-catch） === try { Task task = service.findById(id); // 正常な場合の処理 } catch (TaskNotFoundException e) { // 例外を受け止めた場合の処理 log.error(e.getMessage()); } // この方式は「ローカルで受ける」 // === Vol.9での方式 === // Controllerでは try-catch しない @GetMapping("/tasks/{id}/edit") public String editForm( @PathVariable Long id, ...) { Task task = service.findById(id); // 例外が起きたら… // このメソッドを突き抜けて // GlobalExceptionHandlerまで届く } // → Controller側はシンプルに保てる // → 例外処理を一元化できる // 例外伝搬の流れ [Mapper] findById(9999) → null を返す | ↑ [Service] findById ↓ if (task == null) { ↓ throw new TaskNotFoundException(9999); ↓ } ↓ ★ 例外を投げる ★ | ↓ メソッドが即座に終了 | [Controller] editForm ↓ try-catch なし | ↓ 素通り（突き抜ける） | [Spring] DispatcherServlet | ↓ @ControllerAdvice をチェック | [GlobalExceptionHandler] ↓ handleTaskNotFound() に到達 ↓ ここで初めて受け止める
POINT ── 「例外は自動エレベーター」

例外は「呼び出し階層を上に向かう自動エレベーター」のようなもの。投げると受け止める誰かに出会うまで勝手に上昇します。Service で投げて Controller を素通り → Spring まで届く── この自動伝搬のおかげで、各層にtry-catchを書かずに済むのが業務開発の王道。Vol.9 ではGlobalExceptionHandler が最終的な受け皿になります。

用語カード：独自例外クラス

TERM — CUSTOM EXCEPTION

業務アプリでは「自分でクラスを作って例外として投げる」のが業界標準。Java標準の例外（IllegalArgumentException等）をそのまま使うこともできますが、業務名を付けた独自例外の方が圧倒的に読みやすい。

TERM CARD #02
独自例外クラス（カスタム例外）
業務名を付けて作る、専用の例外クラス
定義
RuntimeException または Exception を継承して、業務的に意味のある名前を付けた専用の例外クラス。
典型例
業務名 + Exception を付ける：
TaskNotFoundException（タスクが見つからない）
InsufficientStockException（在庫不足）
DuplicateUserException（ユーザー重複登録）
UnauthorizedAccessException（権限不足）
なぜ独自クラスを作るか
① 業務的な意味がクラス名から読める
② 同じ業務エラーに対する処理を1か所にまとめられる
③ ハンドラーで@ExceptionHandler(TaskNotFoundException.class)と例外の型で振り分けできる
④ 追加情報（ID、コード等）をフィールドとして持たせられる
本書での方針
RuntimeException を継承する非検査例外として作る。理由：
・throws 宣言が不要で呼び出し側がシンプル
・Spring / Spring Boot のエコシステムが非検査例外を前提に設計されている
・トランザクション自動ロールバックも非検査例外がトリガー
命名規則
必ず ~Exception で終わる名前にする（Java界の慣習）。
custom-exception-design 独自例外クラスの作り方
// 独自例外の基本パターン

シンプル版（Vol.9で使用）：

public class TaskNotFoundException
    extends RuntimeException {

  public TaskNotFoundException(Long id) {
    super("Task not found: id = " + id);
  }
}

充実版（業務開発では一般的）：

public class TaskNotFoundException
    extends RuntimeException {

  private final Long taskId;  // 追加情報

  public TaskNotFoundException(Long id) {
    super("Task not found: id = " + id);
    this.taskId = id;
  }

  public Long getTaskId() {
    return taskId;
  }
}

追加フィールドを持たせると、ハンドラー側で詳しい情報を使える。「ID〇〇のタスクが見つかりません」のように動的なメッセージが作れる。

// シンプルな独自例外（Vol.9） package com.example.todo.exception; public class TaskNotFoundException extends RuntimeException { public TaskNotFoundException(Long id) { super("Task not found: id = " + id); } } // 使い方 throw new TaskNotFoundException(9999L); // → メッセージ: "Task not found: id = 9999" // 業務的なバリエーション // 在庫不足 public class InsufficientStockException extends RuntimeException { public InsufficientStockException( String productName, int requested, int available) { super(String.format( "%s: 要求 %d、在庫 %d", productName, requested, available)); } } // 利用 if (stock < requested) { throw new InsufficientStockException( name, requested, stock); } // 重複登録 public class DuplicateUserException extends RuntimeException { public DuplicateUserException(String email) { super("User already exists: " + email); } } // 業務例外設計の原則 // 1. 業務名で命名 (~Exception) // 2. RuntimeException を継承 // 3. コンストラクタで情報を受け取る // 4. super() でメッセージを設定 // 5. 必要なら追加フィールド // シンプルが正義
POINT ── 「クラス名で意図を伝える」

独自例外の最大の価値はクラス名そのもの。throw new TaskNotFoundException(id) と書けば、「タスクが見つからないというエラー」が一目瞭然。これが throw new RuntimeException("not found") だったら、何が見つからないのか分からない。業務名を例外クラス名にするのは、コードを「読みやすい文書」に変える秘訣です。

用語カード：@ControllerAdvice と @ExceptionHandler

TERM — @CONTROLLERADVICE & @EXCEPTIONHANDLER

Spring が提供する「全Controller共通の例外処理を1か所に書く」仕組み。@ControllerAdvice でクラスを宣言、@ExceptionHandler でメソッドが受け止める例外の型を指定します。

TERM CARD #03
@ControllerAdvice と @ExceptionHandler
グローバルな例外ハンドリングの仕組み
@ControllerAdvice とは
「このクラスは全ての Controller を補助する」とSpringに宣言するアノテーション。Spring 起動時に検出され、全Controllerの例外を捕捉する役割を担う。
@ExceptionHandler とは
「このメソッドは特定の例外を受け止める」と宣言するアノテーション。@ExceptionHandler(TaskNotFoundException.class)のように受け止める例外の型を指定する。
役割分担
@ControllerAdvice：「このクラスは全体に効く」（クラスレベル）
@ExceptionHandler：「このメソッドはこの例外を処理する」（メソッドレベル）
1ハンドラーに複数メソッド
1つの @ControllerAdvice クラスに、複数の @ExceptionHandler メソッドを置ける。
handleTaskNotFound() ─ TaskNotFoundException用
handleInsufficientStock() ─ InsufficientStockException用
handleException() ─ それ以外の全ての例外用（フォールバック）
マッチング規則
例外発生時、Springが最も具体的な型のハンドラーを探す。TaskNotFoundExceptionが投げられたら、@ExceptionHandler(TaskNotFoundException.class)が優先。なければ親クラスのRuntimeException用、最後にException用フォールバック。
controller-advice-usage グローバルハンドラーの構造
// グローバルハンドラーの基本構造

3つのアノテーションを組み合わせる：

  1. @ControllerAdvice ── クラスに付ける、「全体に効くハンドラーですよ」宣言
  2. @ExceptionHandler(例外.class) ── メソッドに付ける、「この例外を受け止めますよ」宣言
  3. 戻り値はテンプレート名（普通のControllerと同じ）

1クラスに複数の @ExceptionHandler を並べて、例外ごとに違うエラー画面を返せます。「業務例外 → 404画面」「システム例外 → 500画面」のように振り分け。

引数：例外オブジェクト、Model、HttpServletRequest 等を受けられる。普通のControllerメソッドと同じ感覚で書ける。

// グローバルハンドラーの典型構造 package com.example.todo.controller; import org.springframework.ui.Model; import org.springframework.web.bind.annotation .ControllerAdvice; import org.springframework.web.bind.annotation .ExceptionHandler; import com.example.todo.exception .TaskNotFoundException; @ControllerAdvice public class GlobalExceptionHandler { // 業務例外 → 404画面 @ExceptionHandler(TaskNotFoundException.class) public String handleTaskNotFound( TaskNotFoundException e, Model model) { model.addAttribute("message", e.getMessage()); return "error/404"; } // その他全ての例外 → 500画面 @ExceptionHandler(Exception.class) public String handleException( Exception e, Model model) { model.addAttribute("message", "予期しないエラーが発生しました"); return "error/500"; } } // マッチング優先順位 // 例外発生時のSpringの選び方: throw new TaskNotFoundException(id) ↓ 1. @ExceptionHandler(TaskNotFoundException) ← マッチ！ 2. @ExceptionHandler(RuntimeException) 3. @ExceptionHandler(Exception) // → 最も具体的なハンドラーが選ばれる // 業務例外を追加した時の拡張 @ControllerAdvice public class GlobalExceptionHandler { @ExceptionHandler(TaskNotFoundException.class) public String handle1(...) { return "error/404"; } // 例: 在庫不足例外を追加した時 @ExceptionHandler(InsufficientStockException.class) public String handle2(...) { return "error/no-stock"; } // 例: 権限例外を追加した時 @ExceptionHandler(UnauthorizedException.class) public String handle3(...) { return "error/403"; } @ExceptionHandler(Exception.class) public String fallback(...) { return "error/500"; } } // 例外の種類が増えても、ここに追加するだけ // → 例外処理の「窓口」が一元化される
POINT ── 「例外処理の中央集権化」

@ControllerAdvice + @ExceptionHandler の仕組みは「例外処理の中央集権化」。各Controllerで try-catch を書かなくても、1か所のクラスで全ての例外を捕まえられる。「例外が起きたらどうするか」を1か所で管理できる── これが業務開発で評価される設計です。Vol.4で学んだ「Service層で業務ロジック集約」と同じ思想が、例外処理にも適用されています。

作った例外クラスを実際に投げる処理を Service に入れます。findById でnull チェックを入れて、見つからなければ TaskNotFoundException を投げる。これだけ。

  6. Ctrl+S で保存

修正の意義：

これまでは findById がnull を返すと、Controller側で NullPointerException が起きていました（Vol.6 ADDで予告した「存在しないIDで500エラー」の原因）。Vol.9で 業務的な例外 を投げる方式に変えることで、その時点で適切なエラー画面に誘導できるようになります。

POINT：「findById は必ず Task を返す」と保証するメソッドになりました。呼び出し側はnull チェック不要。例外が起きなければ確実に Task が返ってくる── これが例外を使った「事後条件の保証」です。
// 修正後の TaskService.java（findById のみ抜粋） package com.example.todo.service; // ...省略 import com.example.todo.exception .TaskNotFoundException; // ★追加 @Service @Transactional(readOnly = true) public class TaskService { // ...省略 // === 1件取得（Vol.9で修正） === public Task findById(Long id) { Task task = taskMapper.findById(id); if (task == null) { // ★追加 throw new TaskNotFoundException(id); // ★追加 } // ★追加 return task; } // 他のメソッドは変更なし } // 修正の前後比較 // ===== 修正前（Vol.8まで） ===== public Task findById(Long id) { return taskMapper.findById(id); // id=9999 → null を返す // → Controller で NullPointerException } // ===== 修正後（Vol.9） ===== public Task findById(Long id) { Task task = taskMapper.findById(id); if (task == null) { throw new TaskNotFoundException(id); } return task; // id=9999 → 例外を投げる // → GlobalExceptionHandlerが受ける } // 影響範囲 // findById を呼んでいる場所: // 1. TaskController.editForm() // → 編集画面表示 // → 例外発生時は 404 画面に // 2. TaskService.toggleDone() // 内部で this.taskMapper.findById // （TaskMapper を直接呼んでいる） // → 影響は受けない // 注意：Service.findById ではなく // Mapper.findById を直接呼んでいる場合は // この修正の影響を受けない // 別の業務例外設計 // 必要に応じて他にも例外を作れる // 例: タスクが既に完了済みなのに // 完了切替しようとした時 if (task.getDone()) { throw new TaskAlreadyDoneException(id); } // → 業務エラーは独自例外で表現する // （Vol.9では1つに留めるが // 業務では複数作るのが普通）
POINT ── 「事後条件を保証する」

修正後の findById は「正常終了したら必ず非nullのTaskが返る」という事後条件を持ちます。呼び出し側はこれを信頼できる。「nullかも？」と疑う必要がない── これが例外を活用した防御的プログラミングの利点。業務メソッドの「契約」が明確になります。

GlobalExceptionHandler クラスを作る

CREATE GLOBAL EXCEPTION HANDLER

例外の受け手となるGlobalExceptionHandlerクラスを新規作成。@ControllerAdvice付きで、2つのハンドラーメソッドを定義します。

step-create-handler GlobalExceptionHandlerを新規作成
// 手順
GlobalExceptionHandler.java を新規作成
  1. com.example.todo.controller パッケージを右クリック
  2. 新規(N) → クラス
  3. 名前に GlobalExceptionHandler と入力
  4. 完了(F)
  5. 右側のコードをコピペ
  6. Ctrl+Shift+O で import 補完
  7. Ctrl+S で保存

このクラスの構成：

  • クラスに @ControllerAdvice：「全Controller共通の補助役」宣言
  • メソッド1：handleTaskNotFound：TaskNotFoundException 専用、404画面を返す
  • メソッド2：handleException：その他全例外のフォールバック、500画面を返す
  • 引数で例外オブジェクト + Model を受け取る（普通のControllerと同じ）

パッケージの選択

GlobalExceptionHandler はcom.example.todo.controller に置くのが業界の慣習です。「Controller の補助役」という意味合いから、controller パッケージ配下が自然です。com.example.todo.exception に置く流派もあります（例外関連を一箇所にまとめる発想）。本書では controller 配下を採用します。

POINT：パッケージ位置は重要。Spring が @ControllerAdvice を検出するには、メインクラス（@SpringBootApplication）の同じパッケージ階層配下にある必要があります。com.example.todo 配下なら自動検出されます。
// GlobalExceptionHandler.java（全体） package com.example.todo.controller; import org.springframework.ui.Model; import org.springframework.web.bind.annotation .ControllerAdvice; import org.springframework.web.bind.annotation .ExceptionHandler; import com.example.todo.exception .TaskNotFoundException; @ControllerAdvice public class GlobalExceptionHandler { // 業務例外：タスクが見つからない @ExceptionHandler(TaskNotFoundException.class) public String handleTaskNotFound( TaskNotFoundException e, Model model) { model.addAttribute("message", e.getMessage()); return "error/404"; } // その他全ての例外（フォールバック） @ExceptionHandler(Exception.class) public String handleException( Exception e, Model model) { model.addAttribute("message", "予期しないエラーが発生しました"); return "error/500"; } } // 各メソッドの役割 // === handleTaskNotFound === // 受け止める例外: TaskNotFoundException // 返すテンプレート: error/404.html // Modelに渡す情報: 例外のメッセージ // → 業務的なエラー（見つからない等） // は 404 として扱う // === handleException === // 受け止める例外: Exception（全例外） // 返すテンプレート: error/500.html // Modelに渡す情報: 固定メッセージ // → 想定外のエラーは 500 として扱う // → 詳細なエラー内容は表示しない // （セキュリティ的に内部情報を隠す） // マッチング動作 // TaskNotFoundException 発生時: 1. handleTaskNotFound(TaskNotFoundException) → マッチ！ 404画面表示 // NullPointerException 発生時: 1. handleTaskNotFound(TaskNotFoundException) → マッチしない 2. handleException(Exception) → マッチ！ 500画面表示 // なぜModelを使うか？ model.addAttribute("message", ...); return "error/404"; // → これまでのController と同じパターン // → Thymeleafでmessage変数を使える // → 404.htmlに ${message} で埋め込む
POINT ── 「ハンドラーも普通のController」

@ControllerAdvice のクラス内のメソッドは、普通のControllerメソッドとほぼ同じ書き方です。引数でModel受け取り、戻り値でテンプレート名を返す── 違いは「呼ばれるタイミングが例外発生時」であるだけ。「Vol.3で学んだ @Controller の知識がそのまま活きる」── これがSpring Bootの一貫した設計の美しさです。

テンプレートのパスとファイル名

POINT：Spring Boot には「error フォルダ配下に 404.html や 500.html があれば自動でデフォルトエラー画面に使う」仕組みがあります。本書では@ControllerAdvice から明示的に "error/404" を返す方式を採用していますが、自動マッピングも活用すれば、未ハンドル例外でも自動で同じ画面に行きます。

POINT ── 「一覧に戻るリンク」が大事」

エラー画面で一番大事なのは「一覧に戻るリンク」。ユーザーが詰まないようにするのがエラー画面の役目です。何もリンクがない画面だと、ユーザーは「戻る」ボタンを使うしかなくなる── これは業務開発では失格。「エラーは隠さず、次の行動を示す」のが業界標準のUX設計です。Vol.7 ADD で学んだ 「失われた時の代償」 という考え方に通じます。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

Vol.9のハマりは「パッケージ位置」「@ControllerAdvice 検出」「テンプレートパス」「redirectとModel」「親例外の優先順位」の5軸に分布します。

⚠ ハマり 1 ── 例外を投げても 404画面ではなく Whitelabel Error Page
SYMPTOM 存在しないIDアクセスでWhitelabel Error Page（Springデフォルトのエラー画面）が表示される。
CAUSE GlobalExceptionHandler がSpringに検出されていない。最も多いのは、com.example.todo パッケージ階層の外側に置いてしまったケース。Spring Boot は@SpringBootApplication クラスがある階層の同じか配下のパッケージしかコンポーネントスキャンしない。
FIX GlobalExceptionHandler.java をcom.example.todo.controller 等、com.example.todo の配下に置く。アプリ再起動。
⚠ ハマり 2 ── テンプレートが見つからない（TemplateInputException）
SYMPTOM 例外発生時にコンソールに TemplateInputException: Error resolving template [error/404] が出る
CAUSE ① テンプレートを templates/ の直下に置いてしまい templates/error/ サブフォルダがない、または
② ファイル名が 404.html でなく error-404.html 等、
③ return "error/404" の指定とパスが一致していない
FIX src/main/resources/templates/error/404.html という正確なパスに配置。return "error/404"; と完全一致させる。
⚠ ハマり 3 ── 全例外で500画面になる（404が出ない）
SYMPTOM TaskNotFoundException を投げたのに、404 ではなく500画面が表示される
CAUSE ① @ExceptionHandler(TaskNotFoundException.class) の引数指定間違い、または
② handleTaskNotFound メソッドの引数で Exception e と書いてしまった（親クラスで受けると具体的なハンドラーが効かない）
FIX アノテーションの引数を TaskNotFoundException.class と正確に指定。メソッド引数の型も TaskNotFoundException e と一致させる。
⚠ ハマり 4 ── エラー画面に「null」と表示される
SYMPTOM 404画面が表示されるが、「null」や「空欄」と表示される
CAUSE ① GlobalExceptionHandlerで model.addAttribute("message", ...) し忘れた、または
② テンプレート側のキー名が間違っている（${msg} 等）、または
③ ハンドラーメソッドの引数で Model を受け取り忘れている
FIX ハンドラーメソッドで Model model 引数を受け、model.addAttribute("message", e.getMessage()); を確認。テンプレートで ${message} とキー名一致を確認。
⚠ ハマり 5 ── 業務例外が @Transactional 内で発生してDB変更が残る
SYMPTOM Service層で例外を投げる前にDB変更していた場合、それが保存されたままになっている（想定はロールバック）
CAUSE @Transactional は非検査例外（RuntimeException系）でロールバックする仕様。検査例外を投げるとデフォルトではロールバックされない（rollbackFor 指定が必要）。
FIX Vol.9の TaskNotFoundException は RuntimeException 継承なので自動ロールバックされる。業務例外は非検査例外で作るが業界標準（深掘りは別冊）。
POINT ── 「例外処理のハマりはパッケージとパス」

Vol.9のハマりの大半は「Springがクラスを見つけられない」または「テンプレートを見つけられない」に集中します。パッケージ階層とテンプレートのパスを正確に合わせることが重要。困った時は「コンソールのエラーメッセージ」を見ると、何を探していて見つからなかったかが書いてあります。Vol.3 ADD の 9ステップ思考フロー を活用してください。
