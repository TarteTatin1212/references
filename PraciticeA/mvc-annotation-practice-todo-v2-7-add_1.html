02

「状態を変えるアクション」の本質

NATURE OF STATE-CHANGING ACTIONS

完了切替、削除、編集── これら3つの機能には共通点があります。「状態を変える」こと。「単に表示するだけ」の機能と本質的に違う性質を持っています。

action-essence アクションが持つ3つの性質
// 「読むだけ」と「変える」の違い

表示系の機能（一覧、詳細、検索）：何度実行しても影響なし。叩くたびに同じ結果が返る（冪等）。後戻りできる（実害がない）。

アクション系の機能（登録、更新、削除、切替）：叩くたびに世界が変わる。同じボタンを2回押せば2回処理が走る（非冪等）。後戻りできない場合もある（特に削除）。

アクション系の3つの性質

非冪等性
同じ操作を複数回行うと結果が変わる。「いいね」を2回押せばカウントが2増える。「タスク登録」を3回押せば3件登録される。
副作用の発生
DB・ファイル・外部APIなど、システムの状態が変わる。実行後の世界は実行前と違う状態になっている。
影響の波及
他のユーザー・他のシステム・他の機能に連鎖的に影響する。タスク削除すれば、それを参照していた別画面も変わる。送金処理すれば、相手の口座も変わる。
// 「読むだけ」と「変える」のSQL対比 // === 読むだけ系 === SELECT id, title FROM tasks SELECT id, title FROM tasks WHERE id = 2 SELECT COUNT(*) FROM tasks WHERE done = false // 何回叩いてもDBは変わらない // 結果は同じ // → GETで実装してOK // === 状態を変える系 === INSERT INTO tasks (...) VALUES (...) UPDATE tasks SET title = ? WHERE id = ? UPDATE tasks SET done = true WHERE id = ? DELETE FROM tasks WHERE id = ? // 叩くたびにDBが変わる // 連続実行で結果が累積（または消える） // → 必ずPOSTで実装 // 業界の鉄則 // SELECTのみ → GET (idempotent) // INSERT/UPDATE/DELETE → POST (非冪等) // これを破ると恐ろしいことが起きる // （次セクションで詳しく）
アナロジー ── 図書館 vs 銀行

図書館（読むだけ系）：何度も同じ本を読みに行っても、本は減らない。誰かが先に読んでいるからといって、あなたが読めなくなるわけではない。「読む」は副作用がない。

銀行（変える系）：振込ボタンを2回押せば2回送金される。「あ、しまった」と思っても取り戻せない場合がある。だから銀行のATMは確認画面と明確な操作ボタンで慎重に作られている。

Webアプリも同じ。「読む系」と「変える系」では、必要な慎重さがまったく違う。Vol.7の完了切替・削除は後者です。

POINT ── 「状態を変える=慎重に作る」

「読む」機能は失敗しても見直すだけで済む。でも「変える」機能は失敗が世界に残る。だから業務開発では、状態を変える機能には表示系の何倍もの慎重さを払います。POSTで送る、確認ダイアログを出す、@Transactional でくくる、ログを残す── これらは全部「変える側の慎重さ」の表れです。Vol.7の各実装はその慣習に従っています。

03

なぜGETではなくPOSTで状態変更するのか（再訪）

WHY POST FOR STATE CHANGES

Vol.5 ADDでも触れた話を、「もしGETで削除を実装したら何が起きるか」という具体的な悲劇のシナリオで深掘ります。これがGETで削除してはいけない核心的な理由です。

get-delete-disaster GETで削除を実装した悲劇
// 仮想シナリオ：GETで削除を実装したら

もしあなたが GET /tasks/2/delete というGETエンドポイントで削除を実装したら── 業務システムが大事故を起こします。具体的に何が起きるか見てみましょう。

クローラーが全削除
Google・Bing 等の検索エンジンのクローラーはWebサイトを巡回して全リンクを辿る。一覧ページに <a href="/tasks/2/delete"> があればクローラーが削除リクエストを送る。
結果：ある朝、全タスクが消滅。
プリフェッチで誤削除
最近のブラウザは「先読み」機能を持つ。ユーザーがリンクにマウスホバーした瞬間に「先読み」リクエストを送る場合がある。ホバーしただけで削除実行。
結果：マウスを動かしただけでデータが消える。
「戻る」「進む」で再削除
一度削除した後、ブラウザの「戻る」「進む」を使うと同じGETが再実行される。さらに削除しようとする（既に消えているので0件削除になるが、エラー表示が出るかも）。
結果：UXが大混乱。
ブックマーク・URLシェアで誤削除
「あれ、便利だな」と削除URLをブックマークしてしまう。SNSでURLをシェアすると受け取った人が開いた瞬間に削除される。
結果：意図せず削除地獄。
業界では「GETで削除」は「絶対NG」として扱われる。実装してしまったら、レビューで即指摘・差し戻し。本番リリース前に必ず捕まる。
// ★絶対NG★ GETでDELETE // やってはいけない実装 @GetMapping("/tasks/{id}/delete") // ✗ public String delete(...) { taskService.delete(id); return "redirect:/tasks"; } // list.html で: <a href="/tasks/2/delete">削除</a> // このリンクが存在するだけで: // ・クローラーがアクセスしてDELETE実行 // ・ブラウザのプリフェッチでDELETE実行 // ・誤クリック1回で復元不能 // === 正しい実装（POST） === @PostMapping("/tasks/{id}/delete") // ✓ public String delete(...) { taskService.delete(id); return "redirect:/tasks"; } // list.html で: <form method="post" action="/tasks/2/delete"> <button type="submit">削除</button> </form> // POSTなら: // ・クローラーは絶対にPOSTしない // ・プリフェッチもPOSTは送らない // ・URLシェアでも実行されない // ・「明示的なボタン操作」が必須 // 業界のルール（再確認） GET → 読み取り、副作用なし POST → 状態変更、副作用あり DELETE → 状態変更だがHTMLフォーム からは送れない PUT → 同上 // HTMLフォーム制約下では: // 「読み取り = GET, 変更 = POST」 // この鉄則だけ守れば事故は防げる
post-vs-get-flow クローラーの動きの違い（新ルール適用フロー図）
// GETの場合：クローラーが削除を実行
クローラー が サーバ に 巡回リクエストを送る行き 送信内容: GET /tasks サーバ が クローラー に HTMLを返す帰り 内容: 一覧画面のHTML（削除リンク含む）
クローラー が HTMLから全リンクを抽出 発見: <a href="/tasks/1/delete">、<a href="/tasks/2/delete">... クローラー が サーバ に 削除リクエストを送る行き 送信内容: GET /tasks/1/delete サーバ が DB に DELETE SQLを発行行き SQL: DELETE FROM tasks WHERE id = 1 DB が サーバ に 削除完了を返す帰り ★タスク1が消失★ 同じことが /tasks/2/delete、/tasks/3/delete... と全リンクで起きる
// POSTの場合：クローラーは無視する
[クローラー] | ↓ クローラー が サーバ に | 巡回リクエストを送る（行き） | GET /tasks | [サーバ] | ↑ サーバ が クローラー に | HTMLを返す（帰り） | 内容: 一覧HTML | 削除ボタンは <form> 内 | [クローラー] | ↓ クローラーがHTMLを解析 | <form> 要素は通常スキップ | <a href="..."> のみフォロー | ★ /tasks/X/delete は POST のみ ★ クローラーはアクセスしない ★ 削除されない！
POINT ── 「POSTは『明示的な意志表示』」

POSTリクエストは「ボタンを押す」「フォームを送信する」など、明示的なユーザー操作でしか発生しません。GETは「URLを開く」「リンクをクリック」「先読み」「クローラー」など様々な経路で発生します。「ユーザーが意図して操作した時だけ実行したい」処理は、POSTで実装するのが鉄則。これだけ守れば、Vol.7のような状態変更系の機能が事故りません。

04

用語カード：論理削除 vs 物理削除

TERM — LOGICAL VS PHYSICAL DELETE

業務システムで頻出する設計上の選択。「DELETEで本当に消すか」「削除フラグを立てて見えなくするだけか」── どちらを選ぶかは業務要件の根本に関わります。

TERM CARD
論理削除 vs 物理削除
削除の2つのアプローチ
物理削除
DELETE文でレコードを完全に消す方式。Vol.7本編で採用。シンプル、テーブル肥大化なし。復元不可能。
論理削除
DELETE文を使わず、「削除済みフラグ」を立てる方式。deleted_atカラムに削除日時を入れる、または is_deleted フラグを true にする。レコードはDBに残り続けるが、画面に表示しないようにする。
共通点
どちらも「ユーザーから見れば削除された」状態。一覧から消える、検索しても出ない。
違い
物理削除：DBから完全消去、テーブル軽量、復元不可
論理削除：DBに残る、テーブル肥大化、復元可能、削除履歴が残る、すべてのSELECT文に WHERE deleted_at IS NULL 等の条件追加が必要
選び方
業務要件で決める：
物理削除が向く：本当に不要なデータ、個人情報保護（削除請求対応）、シンプルなアプリ
論理削除が向く：監査が必要、誤削除復元したい、削除履歴を辿りたい、本格業務システム
logical-vs-physical 2つの設計の実装比較
// テーブル設計の違い

物理削除のテーブル（Vol.7本編で使用）：

CREATE TABLE tasks (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(100),
  done BOOLEAN,
  created_at DATETIME
);

論理削除のテーブル：

CREATE TABLE tasks (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  title VARCHAR(100),
  done BOOLEAN,
  created_at DATETIME,
  deleted_at DATETIME NULL  -- ★追加
);

deleted_at が NULL = 削除されていない、値あり = 削除済み（その日時）。

論理削除の影響範囲

論理削除を採用すると、すべてのSELECT文に条件が増える：

  • findAll() → WHERE deleted_at IS NULL 必須
  • findById(id) → WHERE id = ? AND deleted_at IS NULL
  • 「削除済み」を見せたい画面 → WHERE deleted_at IS NOT NULL
  • uniqueキー制約に注意（同じtitleを2回登録できなくなる罠）

つまり「削除」だけでなく「読み取り全体」に影響する大きな設計判断。Vol.7では学習しやすさ重視で物理削除を採用しています。

// 物理削除の実装（Vol.7本編で採用） @Delete("DELETE FROM tasks WHERE id = #{id}") void delete(Long id); // SQL実行: DELETE FROM tasks WHERE id = 2; // → レコードが完全消失 // → 復元するにはバックアップが必要 // 論理削除の実装 @Update("UPDATE tasks SET " + "deleted_at = NOW() " + "WHERE id = #{id}") void softDelete(Long id); // SQL実行: UPDATE tasks SET deleted_at = NOW() WHERE id = 2; // → レコードは残る // → deleted_at に日時が入る // 一覧取得もSQLが変わる // 物理削除採用なら（Vol.7まで）: @Select("SELECT ... FROM tasks " + "ORDER BY id") List<Task> findAll(); // 論理削除採用なら: @Select("SELECT ... FROM tasks " + "WHERE deleted_at IS NULL " + "ORDER BY id") List<Task> findAll(); // 全SELECT文に条件追加が必要 // 「削除済み一覧」も作れる @Select("SELECT ... FROM tasks " + "WHERE deleted_at IS NOT NULL " + "ORDER BY deleted_at DESC") List<Task> findDeleted(); // 「ゴミ箱」のようなUI // → 復元機能を作れる // 「復元」機能の実装 @Update("UPDATE tasks SET " + "deleted_at = NULL " + "WHERE id = #{id}") void restore(Long id); // → 物理削除ではこれが不可能 // → 論理削除の大きな利点
アナロジー ── ゴミ箱に入れる vs 焼却炉に投げ込む

論理削除はパソコンの「ゴミ箱に入れる」操作に似ています。デスクトップからは消えるが、ゴミ箱を開けば復元できる。「いつ消したか」もログとして残る。

物理削除は「Shift+Deleteで完全消去」または「焼却炉に投げ込む」操作。復元不可能。スッキリするが、「あれ、やっぱり要るかも」と気付いた時には手遅れ。

業務システムでは「ゴミ箱方式」（論理削除）が安全。Vol.7では学習しやすさ重視で「焼却炉方式」（物理削除）を採用しているだけです。

POINT ── 「業務システムは論理削除が標準」

本格的な業務システム── 顧客管理、受注管理、人事システム── では論理削除が標準。「監査要件」（履歴がないと法規に違反する業界がある）、「誤削除復元」（人間はミスする）、「削除日時の追跡」（いつ消えたかが重要）── これらが理由です。Vol.7のTODOアプリは個人用なので物理削除でOKですが、業務で書く場合は論理削除をデフォルトと考えましょう。

05

削除の不可逆性 ── 復元できない設計の重み

IRREVERSIBILITY OF DELETE

業界経験者が口を揃えて言う言葉── 「DELETE文だけは慎重に書く」。なぜそこまで慎重なのか、「失敗した時の代償」を見ます。

delete-disaster-stories 業界に伝わる削除事故の典型
// よくある削除事故3つ
WHERE句忘れ事故
開発者がコンソールで DELETE FROM customers WHERE まで書いたところで Enter を誤入力。
実行されるSQL: DELETE FROM customers WHERE ── 構文エラーで停止... と思いきや、ある状況では WHERE 句なしの DELETE FROM customers が走ることがある（実装による）。
結果：顧客全データ消失。バックアップから復元するも、最新数時間分のデータは戻らない。
本番DBにテスト用スクリプト誤実行
開発環境でテスト用にDELETE FROM tasks（全削除）を書いたスクリプト。間違って本番環境で実行してしまう。
結果：本番データ全消失。リカバリーに数日。会社が大損害。
カスケード削除の想定外
ユーザーを1人削除したつもりが、外部キー設定でそのユーザーのタスク、メッセージ、ログ全部が連鎖削除される。
結果：「ちょっと整理したかっただけ」が、関係する全データを巻き込む大事故に。

これらの事故は実話ベースです。業界では「DELETE文を書く時は、心臓に手を当てて深呼吸してから書く」と冗談半分でよく言われます。

本番DBで削除する時の業界ベストプラクティス

  1. 事前にSELECTで削除対象を確認する
  2. トランザクション開始してから DELETE 実行
  3. 削除件数を確認
  4. 想定通りならCOMMIT、違ったらROLLBACK
  5. 必ずバックアップを取ってから実行
// 本番DBで安全に削除する手順 // === ステップ1: SELECTで確認 === SELECT COUNT(*) FROM tasks WHERE created_at < '2020-01-01'; // → 削除対象件数を把握 // === ステップ2: トランザクション === BEGIN; -- トランザクション開始 // === ステップ3: DELETE実行 === DELETE FROM tasks WHERE created_at < '2020-01-01'; // 影響行数: 1234 rows affected // === ステップ4: 確認 === SELECT COUNT(*) FROM tasks; // → 残ったレコード数を確認 // === ステップ5: 判断 === // 期待通り？ COMMIT; -- 確定（取り消し不可） // 想定と違う？ ROLLBACK; -- 取り消し（救済可能） // 業界の格言 // 1. DELETE文は WHERE 句から書く // 2. SELECT で確認してから DELETE // 3. トランザクションで囲む // 4. バックアップ前提で実行 // 5. 不安なら誰かに見てもらう // アプリ側の対策 // ✓ 物理削除 → 論理削除に変える // ✓ DELETE実行前に必ずログを残す // ✓ 削除実行できるユーザーを制限 // ✓ 一定件数以上の削除は禁止する // ✓ 重要レコードは「保護フラグ」を持たせる // Vol.7のTODOアプリは学習用なので // シンプルだが、業務系では上記対策が標準
POINT ── 「削除の前に一呼吸」

業界で10年・20年働いている人ほど「DELETE文を恐れる」傾向があります。経験から、削除事故の代償を知っているから。あなたが業務開発の現場に出たら、「DELETE文書く時は集中する」習慣をつけてください。コーヒー片手にDELETE文を書く軽率さは、命取りになります。

06

確認ダイアログの設計と限界

CONFIRM DIALOG LIMITS

Vol.7本編で使った confirm() による確認ダイアログ。これだけで本当に安全か? 業務開発で求められるより厳密な確認UIを見ていきます。

confirm-evolution 確認UIの進化段階
// 確認UIの4段階
Level 1: 確認なし（最悪）
削除ボタンをクリックしたら即削除。ユーザーが気付かないうちにデータが消える。
使い所：絶対にない。学習用にすらNG。
Level 2: confirm() ダイアログ（Vol.7本編）
confirm("削除しますか?")でブラウザ標準のOK/キャンセル。
使い所：個人用アプリ、小規模ツール、社内システム。
限界：デザインが固定、誤クリックは防げない、重要度の伝達が弱い。
Level 3: モーダルダイアログ
Bootstrap / Modal / カスタム要素で独自デザインの確認画面。「削除されます。よろしいですか?」と詳細メッセージ、対象データのプレビュー、明確なボタン2つ。
使い所：本格的な業務システム、一般ユーザー向けサービス。
Level 4: 「タイトルを入力して削除」型
GitHub の Repository 削除のような「対象の名前を入力してください」方式。誤クリックがほぼ不可能。
使い所：超重要データ（顧客削除、リポジトリ削除等）。
例：「タスク『買い物リスト』を削除します。タイトルを入力してください: ___」

重要度に応じて段階を選ぶのが業務開発の流儀。重要なデータほど「面倒な確認」を入れます。

// confirm() の動作を分解
ユーザー が 削除ボタン を クリック ブラウザ が <form> の onsubmit を実行行き 実行内容: return confirm('このタスクを削除しますか?') ブラウザ が ユーザー に ダイアログを表示行き 表示内容: 「このタスクを削除しますか?」 [OK] [キャンセル]
★ ユーザーが OK を押した場合 ユーザー が ダイアログ の OKを押す ダイアログ が onsubmit に true を返す帰り 戻り値: true onsubmit が ブラウザ に true を返す帰り ブラウザ が サーバ に POST送信を実行行き 送信内容: POST /tasks/2/delete

CSRFの予告（Spring Securityへの伏線）

CSRF PREVIEW

Vol.7で「POSTで状態変更」を学びました── でも実は、それだけでは防げない攻撃があります。CSRF（クロスサイトリクエストフォージェリ）と呼ばれる、業務開発者なら必ず知っておくべき脅威です。

csrf-attack CSRF攻撃の仕組み
// CSRF攻撃のシナリオ

CSRFは「他サイトから勝手にPOSTリクエストを送らせる攻撃」です。仕組みを順番に見ます。

前提：あなたはアプリにログイン中
あなたはTODOアプリにログイン済み。ブラウザにセッションCookieが保存されている。
攻撃者の罠：偽サイトに誘導
攻撃者が悪意あるWebサイトを作る。あなたが何かのリンクからそのサイトを開く。
罠の中身：自動POST送信
悪意あるサイトには「ページを開いた瞬間に自動でPOSTを送るスクリプト」が仕込まれている。
送信先: https://your-todo-app.com/tasks/1/delete
ブラウザはあなたのCookie付きでリクエストを送る（同じドメインからのリクエストと区別なし）。
結果：あなたのアプリで削除実行
TODOアプリのサーバは正規のリクエストと判断（Cookieが正しいから）して、削除を実行。
あなたは何もしていないのに、自分のタスクが消える。
これがCSRF攻撃。POSTで実装していても、セッションCookieが自動送信されるのでこの攻撃が成立します。
// CSRF攻撃の例（悪意あるサイトのHTML） // 攻撃者が用意するサイト // https://evil-site.com/cat-pictures.html <!DOCTYPE html> <html> <body> <h1>かわいい猫の写真集</h1> <img src="cat.jpg"> <!-- 隠しフォーム --> <form id="evil" action="https://your-todo.com/tasks/1/delete" method="post" style="display:none"> </form> <script> // ページ表示と同時に自動送信 document.getElementById("evil").submit(); </script> </body> </html> // ユーザーがこのページを開いた瞬間: // 1. JavaScriptが自動でフォーム送信 // 2. ブラウザがPOSTリクエスト送信 // 3. ブラウザは自動でCookieを添付 // 4. TODOサーバは正規ユーザーと認識 // 5. 削除実行 // → ユーザーは何もしていない // 防御策：CSRFトークン // Spring Securityが標準で対応 // 1. サーバが「秘密のトークン」を発行 // フォーム表示時にHTMLに埋め込む <form> <input type="hidden" name="_csrf" value="a8s7df9as8df7a..."> ... </form> // 2. サーバはPOST受信時にトークン照合 // トークンが一致しなければ拒否 // 3. 悪意あるサイトはトークンを知らない // → 攻撃が成立しない // 本書での扱い // Vol.7時点では CSRF 対策を入れていない // → 学習用、ローカル動作のみ // → セキュリティリスクは小さい // 本格的なアプリにする時は: // Spring Security を導入 // （Vol.10付近で軽く触れる予定） // Spring Security は CSRF 対策を // デフォルトでONにする // → ほぼ自動で守られる
POINT ── 「POSTだけでは不十分」

「POSTで実装したから安全」と思いがちですが、それは誤解。POSTはクローラーやプリフェッチを防ぐだけで、CSRF攻撃には別の対策が必要です。業務に出る前に「セキュリティは多層防御」という言葉を覚えてください。POST + CSRFトークン + 認証 + 認可 を組み合わせて、初めて「業務に出せる安全性」になります。Spring Securityがこれをほぼ自動で守ってくれるのがSpring Bootの強み。本書では Vol.10 付近で軽く触れる予定です。

「1クリックUI」vs「フォーム経由」のトレードオフ

ONE-CLICK vs FORM

Vol.7の完了切替は「1クリックで状態変更」。Vol.6の編集は「フォームに遷移して送信」。両者のUXのトレードオフを整理します。業務開発での選択基準が見えてきます。

one-click-vs-form 2つのUIパターンの比較
// 2つのUIパターンの本質的な違い
1クリックUI（Vol.7）

例：完了切替、削除、いいね、フォロー
UX：速い、軽快、操作回数少ない
リスク：誤クリック、確認なし

フォーム経由UI（Vol.6）

例：編集、登録、設定変更
UX：丁寧、変更内容を見直せる
リスク：操作が多い、面倒

選び方の指針

変更が「1項目」かつ「軽い」 → 1クリックUI
完了切替、いいね、ブックマーク、フォロー解除── 1つの状態を反転させるだけの操作は1クリックで快適に。
変更が「多項目」または「重要」 → フォーム経由
編集（複数項目）、設定変更、登録── 「内容を確認してから送信」したい場面はフォーム経由。
変更が「不可逆」 → 確認ダイアログ追加
1クリックUIでも、削除のように戻せない操作にはconfirm()等の確認を入れる。Vol.7の削除はこのパターン。
POINT：「気軽さ」と「慎重さ」のトレードオフ。失った時の代償が大きいほど、慎重なUIにする。
// 業界での選択例 // === 1クリックUI === 完了切替（Vol.7） // 元に戻せるので軽快に 「いいね」ボタン // 連打を許容、何度押してもOK フォロー / フォロー解除 // 即座に反映、簡単に元に戻せる ブックマーク追加 // 軽快さが価値 // === フォーム経由UI === タスク編集（Vol.6） // 複数項目変更、確認したい ユーザー登録 // 多項目、一度しか登録できない パスワード変更 // 慎重に、新旧両方入力 注文確定 // 金銭関わる、明示的な意思 // === 1クリック + 確認ダイアログ === 削除（Vol.7） // 不可逆なので confirm() で確認 アカウント停止 // 重要な状態変更 サブスク解約 // 慎重に止める // 選択フロー if (操作が不可逆) { return "確認ダイアログ必須"; } if (変更項目が多い) { return "フォーム経由"; } if (頻繁に使われる) { return "1クリックUI"; } return "フォーム経由が無難";

UIパターンの選択は「業務開発における品質判断」です。同じ機能でもUI設計が違うとユーザー体験は劇的に変わる。「動く」だけのアプリと「使いやすい」アプリの境目はここにあります。Vol.7では完了切替と削除であえてUIパターンを分けたのは、この判断を実体験してもらうためです。

PRGパターンの再応用（書き込み3兄弟）

PRG REUSED — CRUD WRITE TRIO

Vol.5でPRGパターンを学び、Vol.6・7でも同じパターンを使いました。登録・更新・削除の「書き込み3兄弟」はすべてPOST → DB変更 → redirectの構造。あなたはすでにこの基本パターンをマスターしています。

write-trio-pattern 書き込み3兄弟の共通構造
// 書き込み3兄弟の共通骨格

登録（Vol.5）・更新（Vol.6）・削除（Vol.7）── 機能は違いますが、Controllerの骨格はまったく同じです。

3兄弟の共通パターン

  1. @PostMappingでPOSTを受ける
  2. 必要な引数（フォームデータ または @PathVariable）を受け取る
  3. Serviceのメソッドを呼ぶ（業務ロジックは丸投げ）
  4. return "redirect:/...";でリダイレクト

パターンを意識すると、新しい書き込み機能を作る時も同じ手順で進められます。Vol.8でバリデーションが入っても、この基本骨格は変わりません。

// 書き込み3兄弟の比較 // === 登録（Vol.5）=== @PostMapping("/tasks") public String create( TaskForm taskForm) { taskService.register(taskForm); return "redirect:/tasks"; } // === 更新（Vol.6）=== @PostMapping("/tasks/{id}/edit") public String update( @PathVariable Long id, TaskEditForm taskEditForm) { taskService.update(taskEditForm); return "redirect:/tasks"; } // === 切替（Vol.7）=== @PostMapping("/tasks/{id}/toggle") public String toggle( @PathVariable Long id) { taskService.toggleDone(id); return "redirect:/tasks"; } // === 削除（Vol.7）=== @PostMapping("/tasks/{id}/delete") public String delete( @PathVariable Long id) { taskService.delete(id); return "redirect:/tasks"; } // 共通骨格 @PostMapping("/URL") public String action(...) { taskService.業務メソッド(...); return "redirect:/..."; } // 3〜5行で全てのCRUD書き込みが実装できる // PRGパターンが効く理由 // 全ての書き込み処理が「redirect」で終わる // → リロード時に二重実行されない // → ブラウザ「戻る」で警告なし // → URLが意味的にきれい // 「POSTの後はredirect」は // CRUD書き込みの絶対法則
prg-flow-trio 書き込み3兄弟の共通フロー（新ルール適用）
// どの書き込みも基本フローは同じ
ブラウザ が サーバ に POSTを送る行き 送信: POST /tasks（登録）または /tasks/{id}/edit（更新）または /tasks/{id}/delete（削除） Controller が Service の 業務メソッドを呼ぶ行き 呼ぶ: register(form) / update(form) / delete(id) など Service が Mapper に SQL発行を依頼行き 依頼: insert / update / delete のSQL実行 Mapper が MySQL に SQLを送る行き MySQL が Mapper に 完了応答を返す帰り Mapper が Service に 制御を返す帰り Service が Controller に 制御を返す帰り @Transactional がコミット実行 Controller が ブラウザ に リダイレクト指示を返す帰り レスポンス: HTTP 302 / Location: /tasks
★ ここからは「行き」が再開 ブラウザ が サーバ に GETを自動送信行き 送信: GET /tasks（再表示用） サーバ が ブラウザ に 更新後の一覧HTMLを返す帰り ブラウザ が ユーザー に 画面を表示 表示: 変更が反映された一覧画面
// このパターンを内面化する // 書き込み3兄弟は全て: // 1. POSTを受ける @PostMapping // 2. 必要な情報を受け取る @PathVariable, Form // 3. Serviceに丸投げ service.someMethod(...); // 4. リダイレクト return "redirect:/..."; // 新しい機能を作る時の手順 // 例: 「タスクのアーカイブ」機能を追加 // Mapper: アーカイブ用SQL @Update("UPDATE tasks SET " + "archived = true WHERE id = #{id}") void archive(Long id); // Service: 業務ロジック @Transactional public void archive(Long id) { taskMapper.archive(id); } // Controller: 同じパターン @PostMapping("/tasks/{id}/archive") public String archive( @PathVariable Long id) { taskService.archive(id); return "redirect:/tasks"; } // list.html: 同じパターン <form method="post" th:action="@{/tasks/{id}/archive(...)}"> <button>アーカイブ</button> </form> // パターン認識のおかげで即実装できる
POINT ── 「パターン認識が応用力」

同じパターンを3回（登録・更新・切替・削除）実装したことで、あなたは「書き込み機能の作り方が体に染み込んだ」はず。新しい機能の追加が頼まれた時、「あ、これも書き込み系だな。じゃあパターンに当てはめれば...」と即座に設計できる── これが業務開発で重宝されるスキルです。

アクション系機能のチェックリスト

ACTION CHECKLIST

本補足の集大成。「状態を変えるアクション」を実装する時のチェックリスト。明日から使える実務指針です。

action-checklist アクション機能の実装チェックリスト
// 実装前の設計判断

設計判断

  • その操作は「状態を変える」か（読むだけならGETでOK）
  • 状態変更なら必ずPOSTで実装する
  • 1項目だけ変えるか、複数項目変えるか（→ 1クリックUI vs フォーム）
  • 削除なら物理削除 vs 論理削除を選ぶ
  • 不可逆な操作なら確認ダイアログを入れる
  • 重要な操作ならLevel 3〜4の確認UIを検討する
  • 本番ではCSRF対策（Spring Security）を入れる

実装

  • Mapperに該当のSQL用メソッドを追加（@Update / @Delete）
  • WHERE句を必ず入れる（DELETEは特に注意）
  • Serviceに業務メソッドを追加
  • @Transactionalを付ける（書き込み処理）
  • Controllerに@PostMappingを追加
  • 必要な引数（@PathVariable / Form）を受ける
  • 最後はreturn "redirect:/...";（PRG）
  • テンプレートに<form method="post">のボタンを追加
  • 削除にはonsubmit="return confirm('...')"を入れる（重要な場合）

動作確認

  • ボタンクリックでPOSTリクエストが送られる（GETではない）
  • 確認ダイアログでキャンセル → 何も起きない
  • 確認ダイアログでOK → 処理が実行される
  • 処理後一覧（または別画面）にリダイレクトする
  • リロードで二重実行されない
  • ERRORログが出ていない
  • コンソールに想定通りのSQLが出ている

Vol.5・6・7で「書き込み3兄弟」のすべての実装パターンを身につけました。これらは業務開発のあらゆるアクション機能の基礎。Vol.8以降は、このパターンに「品質」を上乗せする旅です。本補足のチェックリストを「明日からの実務テンプレート」として手元に置いてください。

この補足を読み終えたあなたへ

お疲れさまでした。状態を変えるアクションの設計思想を一通り解明しました。GETではなくPOST、論理削除vs物理削除、削除の不可逆性、確認ダイアログの限界、CSRFという脅威、1クリックUIとフォーム経由のトレードオフ、PRGパターンの再応用── 業務開発者の品質感覚の地図が手に入りました。

「動く」と「業務に出せる」の境目に立つあなたは、Vol.1から始まった旅で本当に多くのことを学びました。CRUD完全制覇は「終わり」ではなく「次の章の始まり」。Vol.8以降で「品質を上げる」3つの工程（バリデーション・例外処理・ログ）が待っています。

次のVol.8ではバリデーションを学びます。「空欄での登録を防ぐ」「文字数制限をかける」「数値範囲をチェック」── あらゆる業務システムで必須の入力検証。@NotBlank、@Size、@Valid、BindingResult── 新しい仲間が登場します。CRUD完成のお祝いをしながら、業務品質への階段を一段ずつ登っていきましょう。
