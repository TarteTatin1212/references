02

そもそもバリデーションとは何を守るのか

WHAT DOES VALIDATION PROTECT

「バリデーション」と言う時、何を何から守っているのか? 表面的な「入力チェック」の奥にある本当の目的を見ます。

what-protects バリデーションが守る3つの層
// バリデーションは「3つの層」を守る
データ整合性
DBに入る値が、テーブル定義（型・サイズ・NULL許可）と整合することを保証。
例：100文字までしか入らないVARCHAR(100)カラムに、101文字を入れようとして失敗する事故を防ぐ。
業務ルール
業務的に意味のある値かをチェック。
例：「タスクのタイトルは必須」「年齢は18歳以上」「メールアドレスは正しい形式」── 業務の暗黙的なルールを明示的に表現する。
セキュリティ
悪意ある入力からシステムを守る。
例：SQLインジェクション、XSS、巨大データ攻撃（DoS）── 不正な入力で破壊・侵入されないようにする。

Vol.8で書いた @NotBlank と @Size(max=100) は、3つの層すべてに同時に効く強力な仕組みです。1行のアノテーションで、データ整合性・業務ルール・セキュリティの一部を守れる。

// 守る3つの層の例 // === 層1: データ整合性 === @Size(max = 100) private String title; // DB: title VARCHAR(100) // → アプリで先に弾けば // DB側のエラーを未然に防げる // === 層2: 業務ルール === @NotBlank private String title; // 業務ルール: タスクは必ずタイトルが要る // → 空欄を弾く // === 層3: セキュリティ === @Size(max = 100) private String title; // 攻撃: 1GBの巨大文字列を送ってメモリ枯渇させる // → サイズ制限で防御 @Pattern(regexp = "^[0-9]+$") private String userId; // 攻撃: userId に SQL を混ぜる // → 数字のみ許可で防御 // （+ PreparedStatementと合わせて多層防御） // バリデーションは「最初の砦」 // アプリの入り口で: // ・データ整合性 // ・業務ルール // ・セキュリティ // を一気にチェック // 通過した値だけが // 業務ロジック（Service層）に届く // → 後続コードが「信頼できる値」を // 前提に書ける（複雑さが減る）
アナロジー ── 空港のセキュリティ検査

空港で搭乗前のセキュリティ検査がありますね。これはバリデーションそのもの。
・身分証チェック（必須項目チェック = @NotNull）
・荷物サイズ制限（@Size）
・金属探知機（不正な値の検出 = @Pattern）
・液体物の制限（業務ルール = @Min @Max）

検査を通過した人だけが搭乗エリアに入れる。アプリでも同じ。バリデーションを通った値だけが業務ロジックに入る。これで内部のコードが「正しい値が来る前提」で書ける。

もしセキュリティ検査がなかったら、搭乗エリア内の警備員が各人を疑い続けないといけない。アプリでも、バリデーションを省くとService層やMapper層で毎回チェックすることになり、コードが破綻します。

POINT ── 「最初に守れば後が楽」

バリデーションを入り口（Controller）で済ませると、後続のService層・Mapper層は信頼できる値だけ扱える。これがバリデーションの本当の価値。「データ整合性 + 業務ルール + セキュリティ」の3層を最初にクリアしておけば、業務ロジックがシンプルに書ける── 業務開発の生産性が大きく変わります。

03

用語カード：信頼境界（Trust Boundary）

TERM — TRUST BOUNDARY

バリデーション設計の核心概念。「どこまでが信頼できて、どこからが信頼できないか」を線引きする考え方です。これを意識すると、バリデーションを書くべき場所が一発で見えるようになります。

TERM CARD
信頼境界（Trust Boundary）
信頼できる領域と信頼できない領域の境目
定義
「ここまでは信頼できる、ここからは信頼できない」という境目のこと。セキュリティとバリデーション設計の根本概念。
典型的な境界
サーバ ⇔ クライアント（ブラウザ）
サーバ側は信頼できる（あなたが書いたコード）
クライアント側は信頼できない（ユーザー側で改変可能）
↑ ここに「境界線」がある
境界の特徴
境界を越える時に必ずチェックが必要。「外から入ってきたデータ」はすべて信頼できないと考える。
Webアプリでの境界
① ブラウザ → サーバ（ユーザー入力）
② 外部API → 自サーバ（外部からのデータ）
③ サーバ → DB（書き込み時、PreparedStatementで防御）
各境界で適切な防御が必要
なぜこの概念が大事か
「どこにバリデーションを書くか」「何をチェックするか」が、境界を意識すると自動的に決まるから。「境界の手前で必ず検証する」のがセキュリティ・バリデーション設計の鉄則。

POINT ── 「境界を越える瞬間に検証」

「データが信頼できない領域から信頼できる領域へ入る瞬間」に必ずバリデーションを置く── これが鉄則。Web アプリならController の入口がその場所。@Valid はまさに境界を越える瞬間の検査です。「境界を意識すれば、いつ何を検証すべきか自動的にわかる」── これが業務開発者の設計感覚です。

04

クライアント側 vs サーバ側 ── どちらに書くべきか

CLIENT VS SERVER VALIDATION

バリデーション設計の最も実務的な問い。HTML5のrequired属性やJavaScriptでのチェック（クライアント側）と、Spring の @Valid（サーバ側）── どちらに書くべきか?

client-vs-server 2種類のバリデーションの役割分担
// 結論：「サーバ側は必須、クライアント側はオプション」
サーバ側（Vol.8で実装）

絶対に書く。信頼境界を越えた値を必ずチェック。
セキュリティの最後の砦。
HTMLフォーム以外（API、curl等）からも検証可能。
業務ルールの定義場所。

クライアント側（オプション）

UX向上のために書く。ユーザーが入力中に即座にフィードバック。
サーバ通信せずに済むので速い。
ただしセキュリティ目的では絶対不可。
ブラウザのDevToolsで簡単に回避できる。

「クライアント側だけ」の罠：

HTML5の required を付けて「これで大丈夫」と思うのは初心者がやりがちな罠。攻撃者はブラウザを使わずにcurl や POSTManで直接サーバにリクエストを送れます。クライアント側の検証は完全に無視されます。

業務開発の慣習

  • サーバ側バリデーション ── 必ず実装（Vol.8で学んだ）
  • クライアント側バリデーション ── UXのため余裕があれば実装
  • 両方ある場合 ── 同じルールを両側に書く（DRY違反だが業務では普通）

Vol.8ではサーバ側のみを実装しました。これが業界標準の最低ラインです。

// クライアント側バリデーションの例 // === HTML5 required === <input type="text" name="title" required maxlength="100"> // ↑ ブラウザで弾く // === JavaScript検証 === <script> form.addEventListener("submit", e => { if (titleInput.value.trim() === "") { e.preventDefault(); alert("タイトル必須"); } }); </script> // ★ 攻撃者がDevToolsで回避 ★ // 1. ブラウザのDevToolsを開く // 2. HTMLの required を削除 // <input ... required> // → // <input ...> // 3. 空欄で送信 // 4. サーバが受信 // → サーバ側にバリデーションがないと // → 空欄が登録される // curl での直接攻撃 curl -X POST "https://your-app.com/tasks" \ -d "title=" // ↑ ブラウザを介さない // ↑ クライアント側検証を完全無視 // → サーバ側バリデーションがないと // → 空欄が登録される // だから「サーバ側必須」 // クライアント側だけだと: // 「正規ユーザーには優しいが // 攻撃者には無防備」 // サーバ側があれば: // 「どんな攻撃にも対応可能」 // 両方書くパターン（推奨） // HTML側（UX）: <input required maxlength="100"> // Java側（セキュリティ）: @NotBlank @Size(max = 100) private String title; // → 同じルールを2か所 // → クライアント側が即座にエラー表示（UX） // → サーバ側が最終防衛（セキュリティ） // 業務開発では両方書くのが標準
POINT ── 「サーバ側は絶対、クライアント側は補強」

「サーバ側があれば、クライアント側はオプション」── これが業界の基本ルール。クライアント側だけでバリデーションを済ませている設計を見たら「危ない」と思うのが業務開発者の感覚です。Vol.8のようにサーバ側 (Spring + Bean Validation) をしっかり実装するのが最重要。クライアント側は「ユーザー体験向上のため」に後から足すもの、と覚えてください。

05

Bean Validation の仕組み（裏側の動き）

HOW BEAN VALIDATION WORKS

本編でアノテーション付けただけで動いたバリデーション。Spring の中では何が起きているのか? 内部動作を解明します。

validation-internals @Valid 実行の内部動作
// バリデーション実行の4段階
フォームバインディング
Spring が HTTPリクエストの値を Java オブジェクト (TaskForm) に詰める。
POST のフィールド title=... を taskForm.setTitle(...) でセット。Vol.5 ADDで学んだ「@ModelAttribute の暗黙動作」。
バリデータの起動
@Valid を見たSpringが、Hibernate Validator（Bean Validation の実装）を起動。リフレクションでクラスのフィールドを走査。
アノテーション検査
各フィールドに付いた制約アノテーション（@NotBlank等）を見つけ、対応する検証ロジックを実行。
@NotBlankならNotBlankValidator.isValid()が呼ばれる、@SizeならSizeValidator...
結果のBindingResult への格納
違反があればBindingResult にエラー情報を追加。フィールド名・違反した制約・デフォルトメッセージ等を格納。
違反がなければ何もしない（BindingResult は空のまま）。

ここまで全自動。あなたが書くのは「アノテーション付ける」「@Valid引数追加」だけ。Hibernate Validatorが裏で重い仕事を全部やってくれます。

// 内部動作の擬似コード class SpringFramework { void handleRequest(Request req) { // === 1. フォームバインディング === TaskForm form = new TaskForm(); form.setTitle(req.getParameter("title")); // ↑ HTTPリクエスト値をJavaに詰める // === 2. @Validを発見 === if (引数に @Valid あり) { BindingResult br = new BindingResult(); // === 3. リフレクションでフィールド走査 === for (Field f : form.getClass().getFields()) { // === 4. アノテーション検査 === if (f.hasAnnotation(NotBlank.class)) { String value = f.get(form); if (value == null || value.trim().isEmpty()) { br.addError(new FieldError( "taskForm", f.getName(), "must not be blank" )); } } if (f.hasAnnotation(Size.class)) { String value = f.get(form); int max = f.getAnnotation(Size.class).max(); if (value != null && value.length() > max) { br.addError(...); } } } // === 5. Controllerに渡す === controller.create(form, br); } } } // バリデーターの「中身」 // Hibernate Validator のNotBlankValidator class NotBlankValidator implements ConstraintValidator<NotBlank, String> { @Override public boolean isValid( String value, ConstraintValidatorContext ctx) { return value != null && !value.trim().isEmpty(); } } // → アノテーションごとに専用のクラスがある // → カスタムバリデーターも同じ方式で作れる // （次セクションで実例）
POINT ── 「リフレクションの威力」

Bean Validation の魔法は「Javaのリフレクション」に支えられています。クラスのフィールドを実行時に走査して、付いているアノテーションを見て、対応する処理を呼ぶ── これがJavaフレームワークの典型的な仕組み。Vol.2 ADDで学んだ「アノテーションは目印」の原則そのもの。Spring の@Autowired、MyBatisの@Mapper、Bean Validation の@NotBlank── すべて同じ思想で動いています。

1つのフィールドに複数のアノテーションを付けるのが業界の流儀。@NotBlank @Size(max=100) @Emailのように積み重ねれば、複雑な業務ルールも宣言的に表現できます。「@で始まる行を読めば、業務ルールが分かる」── これがBean Validation の「自己ドキュメント性」です。

07

カスタムバリデーターの作り方

CUSTOM VALIDATOR

標準アノテーションでは表現できない独自の業務ルール。例えば「禁止ワードを含まない」「会員IDの形式」「カタカナのみ」── これらを自分で実装する方法を見ます。

custom-validator-example 「禁止ワード」検証の自作例
// カスタムバリデーター作成の3ステップ
アノテーションを定義
@interfaceで独自アノテーションを作る。@Constraint でバリデータークラスを紐付け。
バリデーターを実装
ConstraintValidator インタフェースを実装。isValid() メソッドに検証ロジックを書く。
Formに付ける
標準アノテーションと同じように、Formのフィールドに付ける。Spring が自動で検出して動かす。

業務開発でカスタムが必要な場面：

  • 業界固有のID形式（医療コード、製品コード等）
  • 禁止ワードチェック
  • 複雑な営業時間チェック
  • 会員ランクごとの上限チェック
  • DBの状態と連動した検証
// === ステップ1: アノテーション定義 === package com.example.todo.validation; import jakarta.validation.Constraint; import jakarta.validation.Payload; import java.lang.annotation.*; @Constraint(validatedBy = NoNgWordValidator.class) @Target({ElementType.FIELD}) @Retention(RetentionPolicy.RUNTIME) public @interface NoNgWord { String message() default "禁止ワードを含んでいます"; Class<?>[] groups() default {}; Class<? extends Payload>[] payload() default {}; } // === ステップ2: バリデーター実装 === package com.example.todo.validation; import jakarta.validation.ConstraintValidator; import jakarta.validation.ConstraintValidatorContext; public class NoNgWordValidator implements ConstraintValidator<NoNgWord, String> { private static final String[] NG_WORDS = { "禁止1", "禁止2", "禁止3" }; @Override public boolean isValid( String value, ConstraintValidatorContext ctx) { if (value == null) return true; // nullは他の @NotNull が担当 for (String ng : NG_WORDS) { if (value.contains(ng)) { return false; } } return true; } } // === ステップ3: Formに付ける === @Data public class TaskForm { @NotBlank @Size(max = 100) @NoNgWord // ★追加 private String title; } // messages.properties で日本語化 NoNgWord.taskForm.title=不適切な単語が含まれています // これで業界固有ロジックも // 宣言的に書ける // チームで共有可能なルールにすれば // あらゆるFormに簡単に適用できる
POINT ── 「ルールを資産化する」

業務固有のチェックロジックをカスタムバリデーターにすると、「チームの資産」になります。例えば@JapanesePhoneNumberを作れば、すべての電話番号Formで使い回せる。新人が書いたコードでもルールが同じ。バリデーション資産化はチーム開発の品質保証として強力です。

「業務ルール検証」と「単純な入力検証」の境界

BUSINESS RULE VS INPUT CHECK

バリデーションには2種類あります。「単純な入力検証」（空欄禁止、長さ制限等）と「業務ルール検証」（在庫超過の注文不可、重複登録不可等）。書く場所が違うのがポイントです。

business-vs-input 2種類の検証の役割分担
// 2種類のバリデーションを比較
単純な入力検証

場所：Form（Bean Validation）
例：必須、文字数、形式（メール・URL等）
特徴：単一フィールドで完結、DB不要、シンプル
Vol.8で実装

業務ルール検証

場所：Service層
例：重複登録不可、在庫超過、関連レコード必要
特徴：複数項目、DB参照、業務知識必要
Vol.8以降で必要になる

境界の判断基準

単純な入力検証と判定：

  • そのフィールド単体で判定できる
  • DBアクセスが要らない
  • 業務知識不要（誰でも理解できる）

業務ルール検証と判定：

  • 複数フィールドを見て判定する
  • DBの状態を参照する必要がある
  • 業務知識が必要（「在庫」「権限」等の概念）
POINT：両方を1か所に書こうとしない。Form（入口）で単純検証、Service（業務層）で業務検証── 役割分担が大切。
// === Form（単純検証） === @Data public class OrderForm { @NotNull private Long productId; @NotNull @Min(1) @Max(100) private Integer quantity; // 入力検証だけ // 「productIdが何か」「在庫があるか」 // はここでは判定しない } // === Service（業務ルール検証） === @Service public class OrderService { @Transactional public void placeOrder(OrderForm form) { // ===== 業務ルール検証 ===== // 1. 商品が存在するか Product p = productMapper.findById(form.getProductId()); if (p == null) { throw new BusinessException("商品が存在しません"); } // 2. 在庫があるか if (p.getStock() < form.getQuantity()) { throw new BusinessException("在庫不足"); } // 3. 1日上限を超えていないか int todayOrders = orderMapper.countToday(); if (todayOrders >= 10) { throw new BusinessException("1日10件まで"); } // ===== 業務処理 ===== // 注文登録、在庫減算、メール送信... } } // 業務ルール例 // ・「同じメールアドレスでは登録できない」 // → DB参照必要 → Service // ・「会員ランクA以上のみ購入可能」 // → ユーザー情報参照 → Service // ・「24時間以内に同じ商品を5個以上注文不可」 // → 履歴参照 → Service // ・「メールアドレスが正しい形式」 // → 単一項目 → Form (@Email) // ・「年齢は18歳以上」 // → 単一項目 → Form (@Min) // 例外設計の予告（Vol.9） // 業務ルール違反は // 「例外を投げる」のが業界標準 // Vol.9で @ControllerAdvice を使って // この例外を画面表示するフローを学ぶ
POINT ── 「役割分担で複雑さを管理」

「すべてFormに書けば良いのでは?」と思うかもしれませんが、業務ルールはDB参照や複数項目の組合せが必要で、Form だけでは表現できません。Form = 入口の門番、Service = 業務の判事── 役割分担を守ると、コードがシンプルで保守しやすくなります。Vol.9で業務例外の扱い方を学ぶことで、この役割分担がさらに明確になります。

// エラー発生から画面表示までの完全フロー
★ シナリオ：ユーザーが空欄でPOST送信 ブラウザ が Spring に POSTリクエストを送る行き 送信: POST /tasks (title="")
★ ステップ1: フォームバインディング Spring が TaskForm を インスタンス化 Spring が TaskForm に setTitle("") を呼ぶ行き taskForm.title = "" の状態

POINT ── 「内部を知ると応用が利く」

このフローを頭で再生できるようになると、バリデーションのトラブルシューティングが楽になります。「エラーメッセージが出ない → BindingResultが空? Modelに入ってない?」「日本語にならない → codes配列に該当キーがない?」── 原因の当たりがつくのは内部理解の力です。

多言語化と messages.properties の深掘り

I18N WITH MESSAGES.PROPERTIES

messages.properties は多言語対応のための仕掛けでもあります。日本語・英語・中国語などを切り替えられる仕組みを覗きます。

i18n-deep 多言語対応の仕組み
// 多言語化（i18n）の仕組み

i18n = Internationalization（国際化）の略。i から n までの間に18文字あるので「i18n」。

Spring の仕組み：言語ごとに messages.properties ファイルを用意すれば、自動で切り替わる。

  • messages.properties ── デフォルト
  • messages_ja.properties ── 日本語
  • messages_en.properties ── 英語
  • messages_zh.properties ── 中国語
  • messages_ko.properties ── 韓国語

言語の判定：ブラウザの Accept-Language ヘッダーから自動判別。あるいは ?lang=en パラメータで切替可能（LocaleResolver 設定が必要）。

プレースホルダの使い方

メッセージに動的な値を埋め込む：

Size.taskForm.title=タイトルは{2}文字以内です

{0}、{1}、{2}... に制約の値が入る。@Size(max=100)なら{2}が100に置換される。

// 多言語対応の例 // === messages.properties (デフォルト) === NotBlank.taskForm.title=Title is required Size.taskForm.title=Title must be within {2} characters // === messages_ja.properties (日本語) === NotBlank.taskForm.title=タイトルは必須です Size.taskForm.title=タイトルは{2}文字以内で入力してください // === messages_en.properties (英語) === NotBlank.taskForm.title=Title is required Size.taskForm.title=Title must be {2} chars or less // === messages_zh.properties (中国語) === NotBlank.taskForm.title=标题为必填项 Size.taskForm.title=标题不能超过{2}个字符 // ブラウザの言語が日本語の場合 // Accept-Language: ja // → messages_ja.properties を選ぶ // → 日本語メッセージ表示 // 切替の仕組み（高度） // LocaleResolver の Bean を定義すると // より細かい制御が可能 @Configuration public class WebConfig implements WebMvcConfigurer { @Bean public LocaleResolver localeResolver() { SessionLocaleResolver r = new SessionLocaleResolver(); r.setDefaultLocale(Locale.JAPAN); return r; } @Bean public LocaleChangeInterceptor localeChangeInterceptor() { LocaleChangeInterceptor i = new LocaleChangeInterceptor(); i.setParamName("lang"); return i; } } // → ?lang=en でURL から切替可能に // 業務開発での使い分け // 国内向けサービス: // → messages.properties（日本語固定）でOK // グローバルサービス: // → 各言語のpropertiesファイル必須 // → 翻訳者と協業する場面も // Vol.8では日本語化のみ // → 多くの業務でこれが現実的 // 注意点：UTF-8 // 全ファイル UTF-8 で保存 // application.properties に: spring.messages.encoding=UTF-8 // → 文字化けを防ぐ
POINT ── 「i18nは最初から考える」

多言語対応が必要かどうかはプロジェクト開始時に決めるもの。後から導入するのは大変です。「最初から messages.properties で書く習慣」をつけておけば、後で多言語化が必要になった時に追加するだけで済みます。「日本語をコードに直書きしない、messages.properties に書く」── これが業務開発のベストプラクティスです。

自分でバリデーションを設計するためのチェックリスト

CHECKLIST

本補足の集大成。新しいFormを作る時のチェックリスト。明日から使える実務テンプレートです。

validation-checklist バリデーション設計チェックリスト
// 設計時の判断項目

事前判断（設計）

  • この入力は必須か任意か
  • 文字数制限は？（DB定義と合っているか）
  • 形式制限はあるか？（メール、URL、電話番号等）
  • 数値範囲はあるか？（年齢、価格、数量等）
  • 業務ルールはあるか？（DB参照が必要なものは Service に書く）
  • カスタムバリデーターが必要か？（業界固有のルール）
  • クライアント側にも書くか？（UX向上のため）

実装

  • pom.xml に spring-boot-starter-validation が入っているか確認
  • Form クラスのフィールドに適切なアノテーションを付ける
  • import は jakarta.validation.*（javax.* ではない）
  • Controllerの引数に @Valid を付ける
  • Controller の引数に BindingResult を @Valid の直後に置く
  • エラー時は return "テンプレート名";（redirectしない）
  • 成功時は return "redirect:/...";（PRG）
  • テンプレートに th:if="${#fields.hasErrors('...')}" + th:errors を追加
  • messages.properties を作成（UTF-8で保存）
  • キー名は制約名.Form名.フィールド名のパターン

動作確認

  • 空欄送信で必須エラーが出る
  • 長文送信でサイズエラーが出る
  • エラー時に入力値が保持される
  • エラー時にURL がフォームのまま（redirect していない）
  • 正常入力時のみ DB登録 + redirect
  • メッセージが日本語で表示される（文字化けなし）
  • 業務ルール違反は Service で例外を投げる（Vol.9 で扱う）
// バリデーション スターターテンプレート // === 1. Form === package com.example.app.form; import jakarta.validation.constraints.*; import lombok.Data; @Data public class XxxForm { @NotBlank @Size(max = 100) private String name; @NotBlank @Email private String email; @NotNull @Min(0) @Max(100) private Integer quantity; // 業務ルール（DB参照が必要等）は // ここに書かない → Service層に書く } // === 2. Controller === @PostMapping("/xxx") public String create( @Valid XxxForm form, BindingResult bindingResult) { // ↑ @Validの直後！ if (bindingResult.hasErrors()) { return "xxx/new"; // ↑ redirectしない } service.register(form); return "redirect:/xxx"; } // === 3. テンプレート === <input type="text" th:field="*{name}"> <span th:if="${#fields.hasErrors('name')}" th:errors="*{name}" style="color:red"></span> // === 4. messages.properties === NotBlank.xxxForm.name=名前は必須です Size.xxxForm.name=名前は{2}文字以内で入力してください Email.xxxForm.email=有効なメールアドレスを入力してください NotNull.xxxForm.quantity=数量は必須です Min.xxxForm.quantity=数量は{1}以上を入力してください Max.xxxForm.quantity=数量は{1}以下を入力してください // このパターンで業務の80%は対応可能
POINT ── 「明日からの実務テンプレート」

このチェックリストとスターターテンプレートを「実務のおまもり」として手元に置いてください。新しいFormを設計する時、項目を1つずつチェック・コピペすれば、業務品質のバリデーションが短時間で組めます。「動くもの」を超えて、「業務で評価されるもの」を作れるエンジニアの土台です。
