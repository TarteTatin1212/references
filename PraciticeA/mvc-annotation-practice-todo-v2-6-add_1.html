02

そもそも「編集」とは何が起きているか

WHAT HAPPENS ON EDIT

「タスクを編集する」── たった1つの操作の裏側で、2つのHTTPリクエストと2回のDB操作と1回のリダイレクトが連鎖しています。編集ライフサイクルとして整理します。

edit-lifecycle 編集の2段階構造
// 編集は「2段階」で構成されている

編集機能の本質は、「既存の状態をユーザーに見せ、変更させて、新しい状態に置き換える」プロセス。Vol.5の新規登録（1段階）と違って、2段階のやりとりが必要です。

第1段階：状態の取得と提示
GET /tasks/{id}/edit
サーバはDBから現在の状態を取得し、フォームに埋めた状態でユーザーに見せる。ユーザーは「現在こうなっている」を確認しながら変更できる。
第2段階：状態の置き換え
POST /tasks/{id}/edit
ユーザーが変更を加えて送信。サーバは新しい状態でDBを更新し、結果を確認する画面（一覧）にリダイレクト。

新規登録との対比：

新規登録（Vol.5）

第1段階：空のフォームを見せる（DBアクセスなし）
第2段階：新レコードを作る（INSERT）
シンプル、過去の状態を意識しない

編集（Vol.6）

第1段階：現在の値を取得してフォームに埋める（SELECT）
第2段階：レコードを更新する（UPDATE）
複雑、過去の状態を見せる必要がある

// 編集ライフサイクル全体（HTTPの会話） [Browser] [Server] // === 第1段階：状態取得 === GET /tasks/2/edit →→→→→→ editForm(id=2) ↓ SELECT WHERE id=2 ↓ Task取得 ↓ TaskEditFormに詰める ↓ Modelに載せる ↓ edit.htmlレンダリング ←←←←←← HTTP 200 OK + フォームHTML （既存値で埋まっている） [ユーザーが画面を見て変更] // === 第2段階：状態適用 === POST /tasks/2/edit title=新タイトル done=true id=2 (hidden) →→→→→→ update(id=2, form) ↓ Form → Task変換 ↓ UPDATE WHERE id=2 ↓ @Transactionalで ↓ コミット ←←←←←← HTTP 302 Found Location: /tasks // === リダイレクト（PRG） === GET /tasks →→→→→→ list() ↓ SELECT全件 ↓ list.htmlレンダリング ←←←←←← HTTP 200 OK + 一覧HTML （変更が反映されている） // 1つの「編集」操作で: // HTTP 3往復、DB 3クエリ // → 裏では多くの処理が連鎖
アナロジー ── 書類の修正申請

役所での書類修正を想像してください。
第1段階：窓口で「現在登録されている内容を見せてください」と請求 → 役所が現状の控えを出してくれる。あなたはそれを見ながら修正を書き込む。
第2段階：修正済みの書類を窓口に提出 → 役所が記録を上書き。
新規申請（Vol.5の新規登録）は「白紙の用紙」から始めますが、修正申請（Vol.6の編集）は「既存の控えを見ながら」変更点を書き込む。これが事前ロードの本質です。

POINT ── 「編集 = 状態の置き換え」

編集は本質的に「現在の状態を新しい状態に置き換える」操作です。新規登録（無から有を作る）とは性質が異なります。「過去の状態を見せる」「変更を受け付ける」「新しい状態にする」── この3つを意識すると、編集機能のあらゆる側面が説明しやすくなります。

03

なぜURLにIDを含めるのか

WHY ID IN URL

/tasks/2/edit というURLにIDを含める設計。/tasks/edit?id=2 でもいいはずなのに、なぜパスに含めるのか── 業界の選択の理由を解明します。

id-in-path-vs-query URLパス vs クエリ文字列
// 2つの設計の比較

同じ「タスク2を編集」を表す2つのURL：

観点 パスに含める
/tasks/2/edit		 クエリ文字列
/tasks/edit?id=2
読みやすさ 「タスク2の編集」と読める 「編集にid=2を渡す」と読める
階層構造 明確（リソース／ID／操作） フラット
ブックマーク 特定のレコードへ直リンク 同上だが見にくい
SEO 検索エンジンに評価されやすい クエリは軽視されがち
業界標準 RESTful、推奨 古い形式、非推奨

結論：「リソースのID」はURLパスに含めるのが業界の鉄則。/tasks/2/editは「タスクという資源の、IDが2のものに対する、編集操作」をURLそのものが表現している。

// URL設計の進化 // === 古い形式（PHP時代） === /tasks.php?action=edit&id=2 // 問題点: // ・ファイル名が露出 // ・action パラメータが混在 // ・読みにくい // === クエリ文字列形式 === /tasks/edit?id=2 // 改善点: // ・ファイル名は消えた // 残る問題点: // ・「タスクの編集」感はあるが、 // IDがクエリ扱いで弱い // === RESTful形式（推奨） === /tasks/2/edit // 利点: // ・「タスクの2番の編集」と読める // ・階層構造が美しい // ・拡張しやすい // 拡張例 // このRESTful設計だと、こんなURLも // 自然に作れる: /users/123/tasks/2/edit // → ユーザー123のタスク2の編集 /projects/45/users/123/tasks/2 // → プロジェクト45の、ユーザー123の、 // タスク2 // クエリ文字列だとこうなる: /edit?type=task&id=2&user=123&project=45 // ↑ 何が主役か分からない、読みにくい // 業界の鉄則 // ルール1: リソースIDはパスに含める // /tasks/2 ✓ // /tasks?id=2 ✗ // ルール2: クエリ文字列は補助的に使う // /tasks?status=done ✓ (絞り込み) // /tasks?id=2 ✗ (特定のリソース) // ルール3: 動詞はパス末尾 // /tasks/2/edit ✓ // /tasks/edit/2 △ (順序が違和感)
POINT ── 「URLは設計言語」

URLは単なる「ページの場所」ではなく、システムの構造を表現する言語です。/tasks/2/edit というURLは、「tasksというリソース集合があり、その中の2番目を編集する」とURLだけで意味が伝わる。良いURL設計はシステムの「読みやすさ」の根本。次セクションのRESTful URL設計はこの考え方を体系化したものです。

04

用語カード：RESTful URL設計

TERM — RESTFUL URL DESIGN

業界で頻出する「RESTful」という用語。これは「リソース指向のURL設計原則」です。Vol.6で使った/tasks/{id}/editはこの思想に従っています。

TERM CARD
RESTful URL設計
リソース指向のURL命名原則
定義
「システムが扱う各種データをリソースとして捉え、URLはリソースの識別に専念し、操作はHTTPメソッドで表現する」設計原則。Roy Fielding（HTTPの設計者の1人）が提唱したREST原則の一部。
核となる発想
URLは「動詞」ではなく「名詞」。/tasksは「タスク（達）」というリソース集合を指す名詞。「何をするか」（取得・登録・更新・削除）はHTTPメソッド（GET/POST/PUT/DELETE）が表現する。
典型パターン
GET /tasks ── 一覧取得
GET /tasks/2 ── 1件取得
POST /tasks ── 新規作成
PUT /tasks/2 ── 更新（全項目）
PATCH /tasks/2 ── 更新（部分）
DELETE /tasks/2 ── 削除
URLは同じ、HTTPメソッドが変わるのが特徴
本書での扱い
HTMLフォームはGET/POSTしか送れない制約があるため、本書ではRESTfulに近いが完全なRESTではない形式を採用：
GET /tasks/{id}/edit ── 編集画面表示
POST /tasks/{id}/edit ── 更新実行（本来はPUT）
POST /tasks/{id}/delete ── 削除（本来はDELETE）
API化する時はPUT/DELETEを使う方針へ移行できる
嬉しさ
① URL設計の指針が明確（迷わない）
② API化が容易（REST APIへのスムーズな移行）
③ 業界標準なので開発者間で通じる
④ SEO・キャッシュ・ブックマークと相性が良い
restful-rules RESTful URL設計の原則
// 設計の3原則
URLは名詞、操作は動詞（HTTPメソッド）
/tasks（名詞）に対してGET（取得）POST（追加）DELETE（削除）等のHTTPメソッド（動詞）を適用。
×：/getTasks、/createTask
○：/tasksに各HTTPメソッド
階層は「親リソース／ID／子リソース」
データの包含関係がURL階層に現れる。
/projects/45/tasks/2 = プロジェクト45のタスク2
/users/123/tasks = ユーザー123のタスク一覧
URLを見ただけでデータ構造が想像できるのが理想。
複数形 + ID で「集合とその要素」
/tasks（集合） → /tasks/2（要素）
常に複数形を使うのが慣習。/task/2より/tasks/2。
「タスクという集合があり、その中の2番」と読める。
POINT：完全なRESTを実現するにはPUT/DELETEメソッドが必要。HTMLフォームの制約上、Vol.6ではPOSTで代用しているが、思想は同じ。
// RESTful URL設計の典型例 // === タスク管理 === GET /tasks 一覧取得 GET /tasks/2 1件取得 POST /tasks 新規作成 PUT /tasks/2 全項目更新 PATCH /tasks/2 部分更新 DELETE /tasks/2 削除 // === HTMLフォーム制約下での代替 === GET /tasks 一覧取得 GET /tasks/2/edit 編集フォーム表示 POST /tasks 新規作成 POST /tasks/2/edit 更新実行（PUT代替） POST /tasks/2/delete 削除実行（DELETE代替） // 「URLパスに動詞が残る」のが妥協点 // API化（REST API）すれば本来の形に // 入れ子になったリソース // プロジェクトとタスクの関係 GET /projects/45/tasks → プロジェクト45のタスク一覧 GET /projects/45/tasks/2 → プロジェクト45のタスク2 POST /projects/45/tasks → プロジェクト45に新タスク追加 // URLからデータ構造が読み取れる // NG例（よくある間違い） // 動詞をURLに含める GET /getTasks POST /createTask POST /deleteTask?id=2 // → 操作がURLに混入 // → 業界標準から外れる // 単数形を使う GET /task/2 // → 業界では複数形が標準 // IDがクエリ文字列 GET /tasks?id=2 // → 特定リソースはパスで指定 // 良いURL設計の判定基準 // チェックポイント: // ① URLに動詞が含まれていないか // ② リソースは複数形で表現しているか // ③ 階層関係が自然に表現されているか // ④ URLだけ見て何のページか分かるか // ⑤ HTTPメソッドで操作が分かれているか // 全部 YES なら RESTful 度が高い
POINT ── 「RESTful設計は資産になる」

RESTful設計に従ったURLは業界標準なので、他の開発者がコードを読んだ時に意図が伝わる。チーム開発・引き継ぎ・新人教育で大きな資産になります。逆に独自URL設計のコードは「なぜこの形なのか」を毎回説明する必要があり、保守コストが高い。「業界標準に従う」のは生産性の高い選択です。

05

@PathVariableの仕組み

HOW @PATHVARIABLE WORKS

本編で「URLから値を取り出すアノテーション」と説明しましたが、Springの中でどう動いているのかを解明します。仕組みが見えれば、ハマった時に自力で解決できるようになります。

pathvariable-internals @PathVariableの内部動作
// Spring内部での処理の流れ

Vol.5 ADDで学んだ「Springのルーティング表」を思い出してください。Springは起動時に「（HTTPメソッド, URLパターン）→ メソッド」の対応表を作ります。@PathVariableはそのパターンマッチングの延長で動きます。

URLパターンの登録
起動時、Springは @GetMapping("/tasks/{id}/edit") を見て、
「このパターンに来たリクエストは editForm() へ送る」と記録。
{id} は正規表現でいう ([^/]+) のような「任意の文字列」マッチを表す。
リクエスト到来時のマッチング
ブラウザが GET /tasks/2/edit を送ると、Springはルーティング表のパターン /tasks/{id}/edit とマッチさせる。
マッチ成功時、{id} の位置にあった値「2」を抽出。
型変換
抽出した値「2」は文字列。引数の型 Long に変換するため、Springが内蔵のコンバータを使って "2" → 2L に変換。
変換不能な値（例：「abc」を Long へ）の場合、400 Bad Requestを自動で返す。
引数バインディング
変換済みの値を、@PathVariable が付いた引数に渡す。
「パス変数名と引数名が一致」を頼りに紐付け。違う名前にしたい時は @PathVariable("id") Long taskId と明示。
// 内部動作のイメージ（疑似コード） // 起動時: class RoutingTable { Map<Pattern, Method> routes; void register() { // @GetMapping("/tasks/{id}/edit") を読む Pattern p = new Pattern( "/tasks/(?<id>[^/]+)/edit"); routes.put(p, editFormMethod); } } // リクエスト到来時: class DispatcherServlet { void handle(HttpRequest req) { // 1. URLマッチング String url = req.getURL(); // /tasks/2/edit Match match = routes.find(url); if (match != null) { // 2. 変数抽出 String idStr = match.getGroup("id"); // idStr = "2" // 3. 型変換 Long id = Long.parseLong(idStr); // id = 2L // 4. メソッド呼び出し match.invoke(id, model); } } } // @PathVariable vs @RequestParam // URLからの値取得には2種類ある // === @PathVariable === // URLパスから取る @GetMapping("/tasks/{id}") public show(@PathVariable Long id) {...} // → /tasks/2 の「2」を取る // === @RequestParam === // クエリ文字列から取る @GetMapping("/tasks") public list( @RequestParam String status) {...} // → /tasks?status=done の「done」を取る // 使い分けの原則 // @PathVariable: 「どのリソース」 // → /tasks/{id} // → 必須かつ「特定」を表す // @RequestParam: 「どんな条件で」 // → /tasks?status=done&page=2 // → 任意かつ「絞り込み・修飾」 // この使い分けが業界標準
POINT ── 「アノテーションは目印（再確認）」

Vol.2 ADDで学んだ「アノテーションは目印」の原則がここでも生きています。@PathVariableは「この引数にURLから値を入れて」という目印。Springが起動時にこの目印を集め、リクエスト時にその指示に従って値を埋める── 仕掛けの本質は「目印を見て自動で動く」です。Spring Bootのあらゆる機能がこの原則で動いていることを意識すると、新しい機能も理解しやすくなります。

06

編集Formを「事前に埋める」設計判断

PRE-FILL DESIGN DECISION

編集機能の核心アイデアは「Controller が事前にDB値を Form に詰めてからテンプレートを描画する」こと。これは業務開発の意思決定として、いつ・なぜ・どう実装するかを掘ります。

prefill-when-and-why 事前ロードのいつ・なぜ・どう
// 事前ロードの意義

事前ロードしない場合のシナリオ：

編集画面を「空のフォーム」で見せたら、ユーザーは「現在の値を全部入力し直す」羽目になる。1文字だけ修正したいのに全項目入力── これはUXの大失敗。事前ロードがあって初めて、編集機能が「実用的な道具」になる。

事前ロードしないほうが良いケースもある：

  • パスワード変更：現在のパスワードを画面に表示してはいけない
  • クレジットカード番号変更：セキュリティ上、再入力させる
  • 機密性の高い項目：マスク表示（「****」）にする

判断基準：「その項目を見せても良いか」と「変更する時に再確認が必要か」の2軸で決める。

// 事前ロードのコード分担 // === Controller でやる（Vol.6の方式） === @GetMapping("/tasks/{id}/edit") public String editForm( @PathVariable Long id, Model model) { Task task = taskService.findById(id); // ↓ Controllerで詰め替え TaskEditForm form = new TaskEditForm(); form.setId(task.getId()); form.setTitle(task.getTitle()); form.setDone(task.getDone()); model.addAttribute("taskEditForm", form); return "tasks/edit"; } // シンプルだが「Fat Controller」傾向 // === Service でやる（より業務的） === // Service層に詰め替えメソッド追加 public TaskEditForm getEditForm(Long id) { Task task = taskMapper.findById(id); TaskEditForm form = new TaskEditForm(); form.setId(task.getId()); form.setTitle(task.getTitle()); form.setDone(task.getDone()); return form; } // Controllerはスッキリ @GetMapping("/tasks/{id}/edit") public String editForm( @PathVariable Long id, Model model) { model.addAttribute("taskEditForm", taskService.getEditForm(id)); return "tasks/edit"; } // 業務開発ではこちらが推奨 // 但しシンプルなアプリでは前者でもOK // 「事前ロードしない」例 // パスワード変更フォーム @GetMapping("/account/password") public String passwordForm(Model m) { // 空のFormを渡す（既存値を埋めない） m.addAttribute("passwordForm", new PasswordForm()); return "account/password"; } // ユーザーが現在のパスワードを // 再入力する // 機密項目のマスク表示 // クレカ番号: "****-****-****-1234" // (最後の4桁だけ見せる) form.setCardNumberMasked( "****-****-****-" + task.getCardNumber().substring(12)); // 表示用と入力用を分けるなど工夫
POINT ── 「UXとセキュリティのバランス」

事前ロードはUX向上の強力な武器。でもセキュリティと相反する場面がある。UX重視か、セキュリティ重視か── 業務開発では項目ごとに判断します。本書では「項目すべてを安全に表示できる」前提で全部事前ロードしていますが、実務では「この項目は見せて良いか」を逐一考えるのが基本です。

UPDATEのSQLパターン

UPDATE SQL PATTERNS

Vol.6で書いたUPDATE tasks SET title = #{title}, done = #{done} WHERE id = #{id}── これは「全項目更新パターン」です。他にもいくつかパターンがあり、業務に合わせて選びます。

update-patterns 3つのUPDATEパターン
// 主な3パターン
全項目更新（Vol.6で採用）
編集可能な全項目を毎回更新。SQL固定でシンプル。
適する場面：通常の編集画面（全項目を1画面で扱う）
欠点：変えていない項目も毎回UPDATEされる
部分更新（特定項目のみ）
特定の1項目だけ更新。Vol.7で「完了切替」に採用予告。
適する場面：ステータス変更、ON/OFF切替、いいねボタン
欠点：項目数だけSQLが増える
動的SQL（変更があった項目のみ）
ユーザーが実際に変更した項目だけ UPDATE する。
適する場面：項目数が多く、毎回全項目を流したくない大規模システム
欠点：SQL構築が複雑（MyBatisの動的SQL機能を使う）

業務開発での選び方：

  • シンプルアプリ：A（全項目更新）でOK
  • 1クリック操作：B（部分更新）
  • 項目が多い・性能重要：C（動的SQL）
// パターンA：全項目更新（Vol.6） @Update("UPDATE tasks SET " + "title = #{title}, " + "done = #{done} " + "WHERE id = #{id}") void update(Task task); // 利点: シンプル、SQL固定 // 欠点: 変えてない項目も書き込み // パターンB：部分更新（Vol.7予告） @Update("UPDATE tasks SET " + "done = #{done} " + "WHERE id = #{id}") void updateDone(Task task); // 完了状態だけ更新する専用メソッド // 利点: 意図が明確、軽い // 欠点: 項目別にメソッドが増える // パターンC：動的SQL @UpdateProvider( type = TaskSqlProvider.class, method = "buildUpdate") void updateDynamic(Task task); // 別のJavaクラスでSQLを動的構築 class TaskSqlProvider { public String buildUpdate(Task t) { StringBuilder sql = new StringBuilder("UPDATE tasks SET "); if (t.getTitle() != null) { sql.append("title = #{title}, "); } if (t.getDone() != null) { sql.append("done = #{done}, "); } sql.append(" WHERE id = #{id}"); return sql.toString(); } } // 変更項目だけUPDATE // → 上級者向け、本書では扱わない // Vol.6 と Vol.7 の役割分担 // Vol.6 編集機能 → パターンA // 全項目を編集する画面なので // 全項目を一度に更新するのが自然 // Vol.7 完了切替 → パターンB // doneだけを切り替える操作なので // 1項目だけ更新するのが自然 // 「画面の意図」と「SQLパターン」を // 合わせるのが業務開発のセオリー
POINT ── 「画面の意図とSQLを合わせる」

UPDATE文のパターンは「画面の意図」と一致させるのが鉄則。編集画面で全項目を扱うなら全項目更新、1ボタンで状態切替するなら部分更新。画面の意図に合っていないSQLを書くと、意図しない副作用が起きがち。Vol.7の完了切替でパターンBの実例を見ることになります。

編集後のリダイレクト先の選び方

REDIRECT TARGET CHOICE

Vol.6本編では更新後にreturn "redirect:/tasks";で一覧に戻す選択をしました。実は他にも選択肢があり、業務によって使い分けます。

redirect-options 編集後の遷移先 3パターン
// 編集後のリダイレクト先 3候補
一覧画面へ戻す（Vol.6で採用）
return "redirect:/tasks";
適する場面：複数レコードを順次編集したい、全体の状況を確認したい
UX特徴：「変更が一覧に反映されているか」を即座に確認できる
業務系での出現頻度：高い（最も一般的）
詳細画面へ遷移
return "redirect:/tasks/" + id;
適する場面：そのレコードを詳しく見たい、関連情報を確認したい
UX特徴：編集対象の更新後の姿を集中して見られる
業務系での出現頻度：中（詳細画面がある場合）
編集画面を再表示
return "redirect:/tasks/" + id + "/edit";
適する場面：複数項目を連続して直したい、長文編集など
UX特徴：「保存して続行」系のフロー
業務系での出現頻度：低（特殊な業務でのみ）
POINT：選び方の指針は「ユーザーは次に何をしたいか」。一覧確認なら A、関連情報確認なら B、続けて編集なら C。

応用：ユーザー選択型。Gmail のような業務系では「保存して閉じる」「保存して次へ」「保存して新規」のような複数ボタンを置き、押されたボタンで遷移先を分けるパターンもある。

// 3パターンの実装比較 // === パターンA：一覧へ戻す === @PostMapping("/tasks/{id}/edit") public String update(...) { taskService.update(form); return "redirect:/tasks"; // シンプル、最も一般的 } // === パターンB：詳細画面へ === @PostMapping("/tasks/{id}/edit") public String update( @PathVariable Long id, ...) { taskService.update(form); return "redirect:/tasks/" + id; // 編集したレコードの詳細を表示 } // === パターンC：編集画面再表示 === @PostMapping("/tasks/{id}/edit") public String update( @PathVariable Long id, ...) { taskService.update(form); return "redirect:/tasks/" + id + "/edit"; // 編集を続けたい人向け } // === 応用：複数ボタンで分岐 === // edit.html： // <button name="action" value="list">保存して一覧</button> // <button name="action" value="continue">保存して続行</button> @PostMapping("/tasks/{id}/edit") public String update( @PathVariable Long id, @RequestParam String action, ...) { taskService.update(form); if ("continue".equals(action)) { return "redirect:/tasks/" + id + "/edit"; } return "redirect:/tasks"; } // ボタンによって遷移先が変わる // → 高機能なUI // 各遷移先のUX特性 A. 一覧: 全体俯瞰、複数編集向け B. 詳細: 1件集中、関連情報確認 C. 編集再: 連続編集、長文向け // 業務要件に合わせて選択
POINT ── 「リダイレクト先 ≒ UX設計」

編集後のリダイレクト先は細かい話に見えて、UXに大きな影響を与えます。「ユーザーは何をしたくて編集したのか」を考えて選ぶのが業務開発のセオリー。「とりあえず一覧に戻す」はデフォルトとして悪くないですが、ユーザーリサーチで他の選択肢が出てきたら柔軟に変える価値があります。

「楽観的ロック」の予告

OPTIMISTIC LOCK PREVIEW

Vol.6で作った編集機能には1つ大きな問題が潜んでいます── 「2人のユーザーが同時に同じレコードを編集したらどうなる?」。本格的な業務システムで必ず立ちはだかる同時編集問題を予告します。

concurrent-edit-problem 同時編集問題と解決策
// 同時編集問題のシナリオ
[時刻 t1] Aさんが編集画面を開く GET /tasks/2/edit → タイトル「牛乳」を表示 [時刻 t2] Bさんも同じレコードの編集画面を開く GET /tasks/2/edit → タイトル「牛乳」を表示 [時刻 t3] Aさんが「牛乳→豆乳」に変更して送信 POST /tasks/2/edit (title=豆乳) → DBは「豆乳」に更新 [時刻 t4] Bさんが「牛乳→お米」に変更して送信 POST /tasks/2/edit (title=お米) → DBは「お米」に更新 ★ 結果：Aさんの変更が消えた！ Bさんは「豆乳」になっていたことを知らない

これが「Lost Update（更新喪失）問題」。少人数の社内ツールなら起きないかもしれないが、本格的な業務システムや一般公開サービスでは必ず起きる。

2つの解決策：

楽観的ロック

「衝突は稀」と楽観視。バージョン番号を持たせ、更新時にチェック。衝突したらエラーを返す。
性能◎、衝突時の対処はユーザー任せ。

悲観的ロック

「衝突は起きる」と悲観視。編集中はDBにロックをかけて他人を排除。
確実だが性能△、デッドロック懸念。

業務開発では「楽観的ロック」が圧倒的多数派。Webアプリのスケーラビリティと両立する。

// 楽観的ロックの実装イメージ（予告） // === Entityにバージョン番号を追加 === @Data public class Task { private Long id; private String title; private Boolean done; private LocalDateTime createdAt; private Integer version; // ★追加 } // === DDLにカラム追加 === ALTER TABLE tasks ADD COLUMN version INT DEFAULT 0; // === UPDATE時のバージョンチェック === @Update("UPDATE tasks SET " + "title = #{title}, " + "done = #{done}, " + "version = version + 1 " + "WHERE id = #{id} " + "AND version = #{version}") int update(Task task); // ↑ 影響行数を返す // === Service層で衝突検知 === @Transactional public void update(TaskEditForm form) { Task task = ...; int updated = taskMapper.update(task); if (updated == 0) { // バージョンが合わなかった = 他人が先に更新 throw new OptimisticLockException( "他のユーザーが先に変更しました"); } } // 仕組み // t1: Aさんが編集開始 (version=0 を取得) // t2: Bさんが編集開始 (version=0 を取得) // t3: Aさんが送信 // UPDATE ... WHERE id=2 AND version=0 // → 1行更新成功、versionが1に // t4: Bさんが送信 // UPDATE ... WHERE id=2 AND version=0 // → 0行更新（versionが既に1） // → 例外をスロー // → ユーザーに警告して再編集を促す // 本書での扱い // Vol.10付近で本格的に実装する予定 // （TODOアプリは小規模なので // 実用上の必要性は低い） // だが「思想を知っているか」が // 業務開発者の必須要件
POINT ── 「業務システムには必ず必要」

楽観的ロックは「本格的な業務システムでは必須」です。社内システムは複数人が同時操作するし、顧客向けサービスは何千人が並行で動く。「同時編集が起きない前提」のコードは本番で破綻します。Vol.6時点では「衝突対策を後回しにできる学習用アプリ」ですが、実務に出たら真っ先に検討すべき項目です。
