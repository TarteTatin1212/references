03

役割を分ける思考法

DECOMPOSING ROLES

「なぜControllerとMapperの2つに分かれているのか？」── 1つのクラスに全部書いてもいいのでは？ という素朴な疑問に答えます。役割を分ける思考こそ、コードを書く前に必要な準備運動です。

why-split なぜ分けるのか
// 「DBからブラウザ」のやることリスト

「DBからブラウザに出す」をやるには、最低でも次の5つの仕事が必要です：

  1. HTTPリクエストを受ける
  2. そのURLが何を求めているか判断する
  3. DBにSQLを発行する
  4. DBの結果をJavaの形に変換する
  5. 結果をHTTPレスポンスとして返す

これを1つのクラスに詰め込むことも技術的には可能です。しかし、現場ではほぼ全てのプロジェクトで役割ごとにクラスを分ける。なぜでしょうか？

// 役割を「誰がやるか」で分けると… // ① HTTPを受ける ──┐ // ② URL判断 ───┤ → Controller // ⑤ レスポンス返す ─┘ // ③ SQL発行 ─────┐ // ④ 結果変換 ─────┤ → Mapper (MyBatis任せ) // もし1クラスに詰め込んだら… public class EverythingController { @GetMapping("/raw") public Object everything() { // HTTPの判断 // ↓ // DB接続を直接開く Connection conn = ...; // ↓ // PreparedStatement作る PreparedStatement ps = ...; // ↓ // SQL実行 ResultSet rs = ps.executeQuery(); // ↓ // ResultSetをMapに変換 // ↓ // JSONに整形 return ...; } } // 100行超のメソッドが完成 // 読めない、テストできない、 // 修正もできない悪夢

なぜ分けるのか、3つの理由：

役割が違うものは、変わるタイミングも違う
Controllerは「URL設計」「HTTPの作法」が変わると変わる。Mapperは「テーブル構造」「SQL」が変わると変わる。変わる理由が違うものは分けておくと、片方を直してももう片方に影響しない。
専門家を割り当てられる
Controllerには「Spring MVCの作法」に強い人を、Mapperには「SQLとDB」に強い人を担当させられる。一人で全部できなくても、分業すれば回せる。
フレームワーク（Spring/MyBatis）が役割を区別する
Springは@Controllerが付いたクラスを「Web受付」として扱い、MyBatisは@Mapperが付いたインタフェースを「DB窓口」として扱う。役割をアノテーションで宣言することで、フレームワークが面倒を見てくれる。
roles-in-vol2 Vol.2のコードを役割で読み直す
// 役割カード
受
Controller（受付係）
仕事：HTTPリクエストを受け取り、誰に何を依頼するか判断し、結果を返す。
Vol.2では RawTaskController が担当。
伝
Mapper（DB伝令係）
仕事：DB専用の窓口。「これ持ってきて」と頼まれたらSQLを発行して結果を返す。
Vol.2では RawTaskMapper が担当（インタフェースのみ、実装はMyBatis任せ）。
蔵
MySQL（記録の倉庫）
仕事：データを保管し、SQLで問い合わせられたら答える。
Vol.2では tasksテーブルが該当。
// 役割と実装の対応 // ┌───────────────┐ // │ 受付係(Controller) │ // │ RawTaskController │ // └────────┬──────┘ // │ 依頼 // ┌────────▼──────┐ // │ 伝令係(Mapper) │ // │ RawTaskMapper(IF) │ // └────────┬──────┘ // │ SQL // ┌────────▼──────┐ // │ 倉庫(Database) │ // │ MySQL の tasks │ // └───────────────┘ // 自分で考える手順 // 「DBの何かをブラウザで見せる」と // 言われたら、まず役割を割り振る： // 1. URLを受ける誰か // → Controllerを作る // 2. DBに問い合わせる誰か // → Mapper(interface)を作る // 3. テーブルから取る方法 // → @SelectでSQLを書く // この3点が決まれば // コードの骨格は決まる
POINT ──

「役割で分ける」という考え方は、現場では「関心の分離（Separation of Concerns）」と呼ばれる設計の鉄則です。Vol.2のControllerとMapperの2分割は、その最小例。これからVol.4でService層を入れて3分割になりますが、考え方は同じ── 「変わる理由が違うものは分ける」です。

04

なぜ @RestController なのか

WHY @RESTCONTROLLER

Controllerには2種類あります── @Controllerと@RestController。なぜVol.2では後者を選んだのか。「目的に合ったアノテーションを選ぶ」という発想を身につけます。

controller-types 2つのControllerの違い
// 自分への問い：何を返したい？

「DBの中身を最速で見たい」── このVol.2の目的に対して、選択肢は2つ：

選択肢A: @Controller

HTMLページを返す用。テンプレートファイル（list.html等）を別途用意し、Modelにデータを載せる必要あり。

必要なもの: Controller + Model + テンプレート

選択肢B: @RestController

データ（JSON）を返す用。テンプレート不要。返り値オブジェクトがそのままJSONに変換される。

必要なもの: Controllerのみ

Vol.2の目的は「最速で動くものを見たい」── そこに「テンプレートを作る手間」は余計。だから @RestController を選ぶのが自然な判断です。Vol.3で表組み画面を作る段になって初めて、@Controllerに切り替えます。

// @RestController の意味 @RestController public class RawTaskController { @GetMapping("/raw") public List<Map<String,Object>> rawList() { return mapper.findAllRaw(); // ↑ この返り値が // 自動でJSONに変換される // [{"id":1,...},...] } } // もし @Controller だったら… @Controller public class RawTaskController { @GetMapping("/raw") public String rawList(Model model) { model.addAttribute( "tasks", mapper.findAllRaw()); return "tasks/raw"; // ↑ templates/tasks/raw.html // が必要になる（追加作業） } } // Vol.2では「画面に見せる」 // よりも「中身を見る」が目的 // だから @RestController が正解
TERM CARD
@RestController
REST API用のController宣言
定義
@Controller + @ResponseBody の合成アノテーション。「このクラスはWeb受付」かつ「メソッドの返り値をHTTPレスポンスのボディに直接書く」を同時に宣言。
結果
メソッドがListやMapを返すと、SpringがJacksonというライブラリで自動的にJSONに変換してブラウザに送る。テンプレートは介在しない。
使いどころ
① REST APIエンドポイント／② JavaScriptが裏で呼ぶAPI／③ 動作確認用のお手軽な可視化（Vol.2はコレ）
POINT ──

アノテーションは「フレームワークへの宣言」です。@RestControllerと書くことで、あなたはSpringに「このクラスをWeb受付として扱え、かつ返り値はそのままレスポンスにしろ」と命令しているわけです。アノテーション選びは「自分が何をしたいかをフレームワークに伝える行為」。要件を見たら、それに合うアノテーションを選ぶのが第一歩。

05

なぜ「フィールド+コンストラクタ」なのか

WHY FIELD + CONSTRUCTOR

Vol.2のControllerにはprivate final RawTaskMapper mapper;というフィールドがあり、コンストラクタで受け取っています。なぜこの形なのか？ 「依存注入（DI）」というSpringの中核思想がここに詰まっています。

why-this-shape なぜこのコードの形なのか
// 自分への問い：Controllerは何が必要？

Controllerは単独では仕事ができません。なぜなら、HTTPを受けるだけが仕事で、DBにアクセスする能力は持っていないから。だからMapperを「持って」おく必要がある。

「持っておく」とは、Javaではフィールドに保存しておくこと。だから：

private final RawTaskMapper mapper;

というフィールド宣言が必要になります。「mapperという名前でMapperを覚えておく場所」を作ったわけです。

では、そのフィールドに誰が値を入れるのか？── ここでコンストラクタの出番です。

// フィールド + コンストラクタ の意味 public class RawTaskController { // ① フィールド：覚えておく場所 private final RawTaskMapper mapper; // ② コンストラクタ：受け取る入口 public RawTaskController( RawTaskMapper mapper) { this.mapper = mapper; // ↑ 渡されたmapperを // フィールドに保存 } // ③ メソッド：保存したmapperを使う @GetMapping("/raw") public List<...> rawList() { return mapper.findAllRaw(); // ↑ this.mapper を使う } } // 誰がコンストラクタを呼ぶ？ // あなたではない。Spring本体です。 // Springが起動時に： // 1. RawTaskMapperを作る（用意する） // 2. RawTaskControllerを作る時に // そのMapperを渡す // 3. Controllerはmapperを保持する // これが「依存注入(DI)」 // = 必要なものを外から注入する
アナロジー ── 寿司職人と仕入れ業者

寿司職人（Controller）は魚を捌くプロです。でも魚を仕入れる能力は持っていない。だから店には仕入れ業者（Mapper）が必要。職人は「業者を覚えておくメモ帳」を持ち、注文が来たら業者に魚を頼みます。

では業者を誰が職人に紹介する？── 店長（Spring）です。店を開く時、店長が「この業者から仕入れて」と紹介する。職人は「ありがとう、メモした」と業者の連絡先を覚えます。これがコンストラクタ注入の正体です。

TERM CARD
依存注入（Dependency Injection / DI）
Springの中核思想
定義
クラスが必要とする他のクラス（＝依存）を、自分で作るのではなく、外から受け取る仕組み。
構造
① 必要なものをフィールドに宣言
② コンストラクタで受け取る
③ Springが起動時に自動でコンストラクタを呼んで注入
嬉しさ
① クラス同士が緩く結合される（差し替えやすい）
② テスト時に偽物を注入できる
③ クラスは「使う側」に集中でき、「作る責任」を持たなくて良い

「private final」と書く意味：

private ── 外から触らせない
mapperはこのクラスの内部状態。外のクラスが勝手に書き換えると挙動が壊れる。だからprivateで隠す。
final ── 一度入れたら変えない
コンストラクタで受け取ったmapperを後から差し替えないと保証する。finalを付けると、変数の再代入が禁止される。これにより「このControllerはずっとこのMapperを使う」という不変条件が手に入る。
POINT ──

「必要なものはフィールドに宣言してコンストラクタで受け取る」── これはSpring業務開発の鉄則テンプレートです。Vol.4で@Serviceクラスを作る時も、Vol.6以降で複数のService/Mapperを組み合わせる時も、全く同じ形で書きます。一度この形を覚えれば、応用できます。

なぜ @GetMapping("/raw") でURLが繋がるのか

WHY @GETMAPPING WORKS

ブラウザでhttp://localhost:8080/rawを開くと、rawList()メソッドが呼ばれます。なぜそうなるのか？ アノテーションが「フレームワークへの目印」として働く仕組みを解き明かします。

routing-mechanism URLからメソッドへの紐付け
// 仕組み：Springの「ルーティング」

アプリ起動時、Springは@Controllerや@RestControllerが付いたクラスを全部探し出します。そして各メソッドの@GetMappingや@PostMappingを見て、URL → メソッドの対応表を作ります。

対応表ができたら、SpringはHTTPリクエストを待ち受けます。リクエストが来たら、URLとHTTPメソッドで対応表を引いて、該当メソッドを呼びます。

因果関係： @GetMapping("/raw") という宣言 → 起動時にSpringが対応表に登録 → ブラウザからGET /rawが来た時、その対応表を引いてrawList()を呼び出す。
// 起動時に作られる対応表（イメージ） URL ROUTING TABLE ───────────────────────────── GET / → HelloController.hello() GET /raw → RawTaskController.rawList() GET /favicon.ico → (静的リソース) ───────────────────────────── // リクエスト処理の流れ // ブラウザ： GET /raw HTTP/1.1 Host: localhost:8080 ↓ // Spring DispatcherServlet： // 1. URL = /raw を受信 // 2. METHOD = GET を確認 // 3. ルーティング表を引く // 4. RawTaskController#rawList() // にマッチ // 5. そのメソッドを呼ぶ ↓ // あなたのコード： @GetMapping("/raw") public ... rawList() { return mapper.findAllRaw(); } ↓ // レスポンス返却

アノテーションの3つの構成要素：

@GetMapping = 「GETリクエストを受ける」の宣言
HTTPにはGET / POST / PUT / DELETEなどのメソッドがある。@GetMappingは「GETだけ受ける」という指定。POSTで来たら無視（404）。「表示は GET、登録は POST」が定番。
("/raw") = どのパスを担当するかの宣言
"/raw"と書けばURLの/rawを担当。"/tasks"なら/tasks。あなたが好きに決めて良い── ただし他のメソッドと被ってはダメ。
メソッド名は自由
rawList()でもfetchAll()でもhelloWorld()でも動く。Springはアノテーションで判断するので、メソッド名はあなたのために分かりやすく付ければよい。
POINT ──

「アノテーションは目印」と覚えてください。@GetMappingは「これがGETの受付窓口だよ」とSpringに教えるシール。Springは起動時に全クラスをスキャンしてこのシールを集め、対応表を作る。あなたの仕事は「適切な位置にシールを貼る」こと。これがSpring業務開発の第一原理です。

なぜMapperは interface なのか

WHY MAPPER IS INTERFACE

Mapperはpublic classではなくpublic interfaceです。中身（実装）を書いていないのに、なぜ動くのか？ ── これがMyBatisの魔法の核心です。

interface-magic 実装なしで動く仕組み
// 違和感の正体

普通のJavaなら── interfaceを使うには必ず実装クラスを書く必要があります：

interface Animal { void cry(); }

// 実装クラスがないと使えない
class Dog implements Animal {
  public void cry() { ... }
}

ところがMapperは違います。RawTaskMapperにimplementsするクラスを誰も書いていない── にも関わらず、Controllerから普通に呼び出せる。これは誰かが裏で実装クラスを作っているということです。

その「誰か」が── MyBatisです。

// あなたが書くコード @Mapper public interface RawTaskMapper { @Select("SELECT * FROM tasks " + "ORDER BY id") List<Map<String,Object>> findAllRaw(); } // メソッドの中身を書いていない！ // それなのに動く。なぜ？ // MyBatisが裏で生成するコード（イメージ） public class RawTaskMapper$$Generated implements RawTaskMapper { private SqlSession session; @Override public List<Map<String,Object>> findAllRaw() { // 1. @Selectから SQL を取得 String sql = "SELECT * FROM tasks " + "ORDER BY id"; // 2. DB接続を借りる Connection conn = ...; // 3. SQL実行 ResultSet rs = ...; // 4. ResultSetをListMapに変換 List<...> result = ...; // 5. 返す return result; } } // これを MyBatis が自動で作る // あなたは書かなくて良い

なぜinterfaceが採用されているのか、3つの理由：

interfaceは「契約」だけを記述する
interfaceは「このメソッドが存在し、こういう型を返す」という約束ごとだけを書くもの。実装の詳細は書かない。MyBatisはあなたが書いた契約（メソッド名・引数・返り値・@Select）から、自動的に実装を生成できる。これが「書かなくて済む」を可能にする鍵。
Javaの仕組み「動的プロキシ」を使う
Javaには「実行時に interface の実装クラスを動的に生成する」機能（Dynamic Proxy）が標準で備わっている。MyBatisはこれを使って、起動時にRawTaskMapperの偽物の実装クラスを作り、Springに渡している。Controllerが受け取るmapperは、実はその動的に作られた偽物です。
interfaceなら何度でも作り直せる
もしクラスとして実装が固定されていたら、テスト時に偽物のMapperに差し替えるのが難しい。interfaceならテスト用の実装クラスを別途用意して差し替えられる。柔軟性のためでもある。
アナロジー ── 注文書と業者

あなたが「魚を仕入れる注文書のテンプレート」を作るとします── 「魚種・量・到着日を書く欄」がある書面です。これがinterface。

実際にその注文書を埋めて魚を持ってくる業者は別に契約します。書面を渡せば、業者は中身を読み取って魚を仕入れて持ってくる。あなたは業者がどうやって仕入れるかは知らなくていい── 注文書のフォーマットさえ決めれば、結果が手に入る。

@Mapper付きinterface = 注文書のフォーマット。MyBatis = それを読んで仕事する業者。

TERM CARD
@Mapper
MyBatisへの「これは私のクライアント」宣言
定義
MyBatisに「このinterfaceの実装を起動時に生成して、Springに登録しろ」と命令するアノテーション。
結果
起動時、MyBatisが@Mapperを見つける → 動的プロキシで実装クラス生成 → SpringのDIコンテナに登録 → 他のクラスがコンストラクタで受け取れるようになる。
注意
必ずinterfaceに付ける。classに付けても効かない。これは「interfaceでないと動的プロキシが作れない」というJavaの制約。
POINT ──

「書かなくていいコードは書かない」── これがMyBatisの設計思想です。DB接続の細かい処理、PreparedStatementの作成、ResultSetのループ、Mapへの変換── これら全てを毎回書くのは時間の無駄。だからMyBatisが自動で生成する。あなたは「何が欲しいか（SQL + 返り値の型）」を宣言するだけ。「How」ではなく「What」を書く── これが現代のフレームワークの賢い使い方です。

08

なぜ @Select だけでSQLが動くのか

WHY @SELECT WORKS

Mapperのメソッドに@Select("SELECT * FROM tasks ORDER BY id")と書くだけで、本当にDBにSQLが飛んでいる。このマジックの正体を見ます。

select-mechanism @SelectがSQL実行に化ける流れ
// メソッド呼び出しからSQL実行まで

Controllerがmapper.findAllRaw()と書くと、裏では連鎖反応が起きます。一つずつ追います。

[1] Controllerが呼ぶ mapper.findAllRaw() ↓ [2] mapperの正体は動的プロキシ 実装クラス（MyBatis製）が この呼び出しを横取り ↓ [3] プロキシがメソッドを調査 "findAllRaw" メソッドに @Selectアノテーションが 付いているか？ ↓ YES [4] @Selectの中のSQL文字列を取得 "SELECT * FROM tasks ORDER BY id" ↓ [5] DataSourceから接続を借りる (Vol.2のproperties設定で MySQLに繋がる) ↓ [6] PreparedStatementで実行 ↓ [7] ResultSetを受け取る ↓ [8] 返り値の型を見て変換 List<Map<String,Object>> なので、各行をMapに変換 ↓ [9] List を Controller に返す
// あなたが書くコード @Mapper public interface RawTaskMapper { @Select("SELECT * FROM tasks " + "ORDER BY id") List<Map<String,Object>> findAllRaw(); } // たったこれだけ。 // 残り全てはMyBatisが面倒見る。 // もし MyBatis がなかったら… public List<...> findAllRaw() { try (Connection conn = dataSource.getConnection(); PreparedStatement ps = conn.prepareStatement( "SELECT * FROM tasks ORDER BY id"); ResultSet rs = ps.executeQuery()) { List<Map<String,Object>> result = new ArrayList<>(); ResultSetMetaData meta = rs.getMetaData(); int cols = meta.getColumnCount(); while (rs.next()) { Map<String,Object> row = new HashMap<>(); for (int i = 1; i <= cols; i++) { row.put(meta.getColumnName(i), rs.getObject(i)); } result.add(row); } return result; } catch (SQLException e) { throw new RuntimeException(e); } } // 20行を超える地獄 // これを毎回書く時代もあった
TERM CARD
@Select
「このメソッドはこのSELECTを実行する」宣言
定義
MyBatisのアノテーション。「このメソッドが呼ばれたら、中のSQL文字列を実行して、結果を返り値の型に変換せよ」という命令。
使い分け
@Select ── SELECT文／@Insert ── INSERT文／@Update ── UPDATE文／@Delete ── DELETE文。Vol.5以降で全部登場します。
結果型の決定
返り値の宣言（List<Map>とか）を見て、MyBatisが結果の入れ物の形を決める。Listなら複数行、単一オブジェクトなら1行、intならCOUNT結果、など。
POINT ──

「SQLを直接書く」のがMyBatisの方針です。JPA/Hibernateのような「SQLを書かないでもOK」のライブラリと違って、MyBatisはSQLを隠さない。これは「SQLは書いた方が見通しが良い」「最適化が容易」という思想。業務開発、特にSI現場でMyBatisが好まれる理由がここにあります。

09

なぜ List<Map<String,Object>> なのか

WHY LIST OF MAP

Vol.2の返り値はやたら複雑なList<Map<String, Object>>でした。なぜこんな型なのか？ Vol.3でList<Task>に置き換えた理由も含めて整理します。

type-breakdown 型を一段ずつ解きほぐす
// 入れ子になっている型を一段ずつ読む

List<Map<String, Object>>は一見複雑ですが、外側から1つずつ読むと簡単です：

外側の List ── 「複数の行を入れる箱」
SELECTの結果は複数行。だから「複数」を表現するListで囲む。
中身の Map ── 「1行を表す」
DBの1行は「id=1, title=牛乳, done=false, created_at=...」のようなキーと値のペア。これはMapの典型的な用途。
Map<String, Object> ── 「列名（String）→ 値（Object）」
キーは「カラム名（文字列）」、値は「何でも入る箱（Object）」。なぜObjectか── 列によって型が違うから。idは数値、titleは文字列、doneは真偽値…。共通の親クラスとしてのObjectで受けるのが最小ルール。
// 型を分解して読む List<Map<String, Object>> ─┬─ ─┬─ ─┬───── ─┬─── │ │ │ │ │ │ │ └ 値（何でも入る） │ │ └─ キー（列名） │ └─ 1行を表すMap └── 複数行を表すList // 実体イメージ [ { "id": 1, "title": "牛乳", "done": false, "created_at": "2026-..." }, { "id": 2, "title": "パン", "done": true, "created_at": "2026-..." }, ... ] // ↑ これがJSONとしてブラウザに行く // なぜVol.3でこれを捨てるか // Object は何でも入る = 何が入ってるか // 取り出すまで分からない Object raw = row.get("title"); // ↑ String? Integer? null? String title = (String) raw; // ↑ キャストが必要で危険 // Task クラスなら型が確定 Task t = ...; String title = t.getTitle(); // ↑ キャスト不要・安全

「なぜVol.2はあえてこの型を使ったか」── 理由は2つ：

エンティティクラスを作る前でも動かせるから
Map<String, Object>はどんなテーブルの結果でも受けられる万能型。Taskクラスをまだ作っていない状態で「とりあえずDBの中身を見たい」目的を達成できる。これがVol.2で「動くものを最速で見せる」戦略にハマる。
「型安全でないと困る」を体感させるため
Vol.3でTaskに置き換えた時、「ああ、Mapだとここが弱かったんだな」と差を体感できる。不便を一度経験させてから便利を提示する── これが教材的な意図です。
アナロジー ── ダンボール箱と専用ケース

List<Map<String, Object>>は「何でも入る大きなダンボール箱」。とりあえず何でも放り込めるけど、取り出す時に「これ何の道具だっけ？」と確認しないといけない。

List<Task>（Vol.3）は「ハンマー専用のケース」。ハンマー以外は入らないけど、取り出した瞬間にハンマーだと分かる。試作段階では汎用の箱、本格運用には専用ケース── これが型の使い分けです。

POINT ──

「初期段階では緩い型 → 後で厳密な型に置き換える」── これは現場でもよくある進化パターンです。プロトタイプをMapで素早く作り、要件が固まったらエンティティクラスを作って差し替える。Vol.2→Vol.3の流れはまさにこの実務的進化を体験させるための設計です。

END-TO-END FLOW

ここまでの全要素を1つのストーリーに繋ぎます。アプリ起動の瞬間からブラウザに表示される瞬間まで、Vol.2のコードがどう連鎖して動くかを追体験します。

startup-phase 起動時に何が起きるか
// フェーズ1：アプリ起動（リクエスト前）
SpringApplication.run() でブート
TodoApplicationのmainメソッドからSpringApplication.run()が呼ばれる。これが全ての出発点。
@ComponentScan が動く
@SpringBootApplicationに含まれる@ComponentScanがcom.example.todo以下を全部スキャン。@Controller、@RestController、@Service、@Component等のシールが貼られたクラスを全部見つける。
Mapper探索（MyBatis）
@Mapperが貼られたinterfaceを全部見つける。動的プロキシで実装クラスを生成。DIコンテナに登録。
Bean生成とDI
RawTaskMapperのプロキシを先に生成。次にRawTaskControllerを生成しようとして、コンストラクタにRawTaskMapperが必要と判明。すでに作ってあるプロキシを引数として渡す。これでControllerが完成。
ルーティング表の構築
全Controllerの全メソッドを調査して、@GetMapping等からURL→メソッドの対応表を作る。/raw → RawTaskController#rawList()がここで登録される。
Tomcat起動
内蔵Tomcatがポート8080でリッスン開始。これで待ち受け完了。コンソールにStarted TodoApplication in X.XXX secondsが出る。

request-phase リクエストが来てから返るまで
// フェーズ2：リクエスト処理
ブラウザがGET /raw を送信
あなたがhttp://localhost:8080/rawを開いた瞬間、ブラウザはGET /raw HTTP/1.1というHTTPリクエストを送る。
DispatcherServlet が受信
Tomcatが受け取り、SpringのDispatcherServlet（全てのHTTPリクエストの中央受付）に渡す。
ルーティング表を引く
URL=/raw、メソッド=GETでルーティング表検索。RawTaskController#rawList()がヒット。

業務開発の「定型パターン」を覚える：

  • 「DBから取ってブラウザに出す」要件 → Controller + Mapper の2点セット
  • JSON返却なら @RestController、HTMLページなら @Controller
  • Mapper は必ず interface、@Mapper を付ける
  • SQL は @Select アノテーションに直接書く
  • 依存（Mapper）は private final + コンストラクタ で受け取る
  • URL は @GetMapping("/...") で決める
  • パッケージは必ず com.example.todo（または各プロジェクトのルート）の下に置く
POINT ── 最後に一言

「書くべきコードは決まっている。あなたは要件をそのコードに翻訳するだけ。」
Spring業務開発では、パターンが9割です。新しい要件が来ても、「Controller + Mapper + テーブル」「@RestController + @Mapper + @Select」「フィールド + コンストラクタ」── このテンプレートを組み合わせるだけ。このパターンが手に馴染めば、業務コードは怖くなくなります。Vol.3以降で同じパターンが何度も登場するので、繰り返しの中で身体で覚えていきましょう。
