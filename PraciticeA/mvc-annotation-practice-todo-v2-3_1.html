01

この巻の地図

WHAT / HOW / WHY

v2の新セクション。各巻の冒頭で「何を作るか」「どう作るか」「なぜそうするか」を最初に整理します。手を動かす前に全体像を掴むことで、各セクションの位置づけが明確になります。

WHAT
何を作るか
表組みのTODO一覧画面。Vol.2の/rawはJSON文字列の塊でしたが、Vol.3末では/tasksでブラウザにHTMLの表が表示されます。中身はTaskクラスで型安全。
HOW
どう作るか
① Taskクラスを作る（Lombokの@Data）
② MapperをList<Task>を返すよう書き換える
③ Controllerを@Controllerに変えてModelにデータを載せる
④ Thymeleafテンプレートで表組みを描画
WHY
なぜそうするか
Map<String,Object>はキー名のタイポでも実行時まで気付かない危険な構造。エンティティクラスならコンパイル時に検出でき、補完も効く。これが業務開発の標準的な作りです。
vol-3 transformation Vol.2末 → Vol.3末への進化
// Vol.2末（現状）
[Browser] /raw ↓ JSONベタ表示 [RawTaskController] ↓ Map<String,Object> [RawTaskMapper] ↓ SELECT * FROM tasks [MySQL] × キー名の保証なし × 補完が効かない × HTMLじゃない
// Vol.3末（目標）
[Browser] /tasks ↓ 表組みのHTML [TaskController] ↓ Model.addAttribute [Thymeleafテンプレート] ↓ List<Task> [TaskMapper] ↓ Task型で取得 [MySQL] ○ 型安全 ○ 補完が効く ○ HTML描画
// この巻でやること（順番） // ① Taskクラス作成 package com.example.todo.entity; @Data public class Task { private Long id; private String title; private Boolean done; private LocalDateTime createdAt; } // ② Mapper置き換え package com.example.todo.mapper; @Mapper public interface TaskMapper { @Select("SELECT ... FROM tasks ORDER BY id") List<Task> findAll(); } // ③ Controller置き換え package com.example.todo.controller; @Controller public class TaskController { @GetMapping("/tasks") public String list(Model model) { ... } } // ④ Thymeleafテンプレート作成 src/main/resources/templates/tasks/list.html // ⑤ rawパッケージ削除 // 仮実装を撤去して完成形へ
POINT ──

このVol.3は「v2の真骨頂」とも言える巻です。Mapで雑に動かす→エンティティで型安全に整えるという流れは、業務開発で「プロトタイプ→本実装」の進化を体験する縮図。最後まで進めれば「業務でもこう書く」という形が手に入ります。

rawパッケージの正体と引退

RETIRE THE SCAFFOLDING

Vol.2で唐突に登場したrawパッケージ。これは標準用語ではなく、本書独自の便宜的な命名でした。意味と引退の段取りを明確にしておきます。

what-is-raw 仮実装の正体
// rawパッケージとは何だったか

「生（なま）のデータ＝型なしのMap<String, Object>で扱う仮置きのコード」を入れたパッケージです。一般的には「仮実装（scaffold）」「プロトタイプ」「動作確認用コード」と呼ばれるものに相当します。

なぜVol.2で使ったか── 「DBから取ったデータがブラウザに見える」体験を最速で届けるため。本物のTaskクラスやService層を作る前に、骨抜きで動かす必要がありました。

どこへ消えるか── このVol.3の最後で完全に削除します。本物のentity／mapper／controllerパッケージに置き換わります。

// 現在のプロジェクト構造 todo/ └ src/main/java/com/example/todo/ ├ TodoApplication.java ├ controller/ │ └ HelloController.java └ raw/ // ★Vol.3で削除予定★ ├ RawTaskMapper.java // Map版Mapper └ RawTaskController.java // JSON返却Controller // Vol.3終了時の構造（目標） todo/ └ src/main/java/com/example/todo/ ├ TodoApplication.java ├ controller/ │ ├ HelloController.java │ └ TaskController.java // ★NEW ├ entity/ │ └ Task.java // ★NEW └ mapper/ └ TaskMapper.java // ★NEW // rawパッケージは消滅 // 本物の3層構造に置き換わる
POINT ──

本書のような「動くものを早く見せたい」教材では、仮実装→本実装の置き換えは頻出パターンです。実務でもプロトタイプ（PoC）→本番実装という流れは普通にあるので、「仮で雑に作って、後で型安全に整える」段取りは現場感覚としても自然です。

03

用語カード：エンティティ

TERM CARD — ENTITY

v2の新セクション。専門用語を「定義／構造／対比／嬉しさ」の4観点で解説します。最初の用語はエンティティ── これからずっと使う基本概念です。

TERM CARD #01
エンティティ（Entity）
DB の 1 行を表すクラス
定義
データベースの1行（1レコード）に相当するデータをまとめて持つJavaクラス。tasksテーブルの1行 ↔ Taskクラスのインスタンス1個、という対応関係。
構造
テーブルのカラム名と同じ名前のフィールドを持つ。各フィールドにgetter/setterを用意（Lombokで自動生成）。普通はビジネスロジックを持たず、データを入れる箱に徹する。
対比
Map<String, Object>と比較：
Mapの場合── map.get("titel")とタイポしてもコンパイル成功、実行時にnullで気付く。
エンティティの場合── task.getTitel()と書いたら赤線が出てコンパイル不可。エディタが守ってくれる。
嬉しさ
① 型安全（タイポを実行前に検出）
② 補完が効く（Ctrl+Spaceでフィールド一覧）
③ 意図が読める（Taskと書けば誰でもタスクのデータと分かる）
④ refactor安全（フィールド名変更が一括で済む）
✗ NG: Map で扱う
Map<String, Object> row = ...;
String title = (String) row.get("titel");
// ↑ タイポ ↓ コンパイル通る
// 実行時に title = null
// 画面に「null」と出てから気付く
○ OK: エンティティで扱う
Task task = ...;
String title = task.getTitel();
// ↑ 赤線が即座に出る
// "The method getTitel() is undefined"
// 書いた瞬間に気付ける
TIP ──

業務開発では「テーブル名は複数形（tasks）／エンティティ名は単数形（Task）」が定番。テーブルは「複数の行が入る箱」、エンティティは「その1個」なのでこの命名は理にかなっています。

Taskクラスを作る

CREATE TASK ENTITY CLASS

いよいよ本物のTaskエンティティを作ります。entityパッケージを新規作成し、その下にTaskクラスを配置します。

step-by-step Eclipseでの具体手順
// クラス作成の具体手順
A. entityパッケージを作る
  1. 左のパッケージ・エクスプローラーでsrc/main/java > com.example.todoを右クリック
  2. 新規(N) → パッケージ
  3. 名前に com.example.todo.entity と入力
  4. 完了(F)をクリック
B. Taskクラスを作る
  1. 作ったentityパッケージを右クリック
  2. 新規(N) → クラス
  3. 名前に Task と入力
  4. 完了(F)をクリック
  5. 右のコード欄の内容をクラスにコピペ
  6. Ctrl+Shift+O でimport自動補完
  7. Ctrl+S で保存

POINT: インターフェイスとクラスを間違えないこと。エンティティはクラス。@Mapperを付けるMapperはインターフェイス。

// src/main/java/com/example/todo/entity/Task.java package com.example.todo.entity; import java.time.LocalDateTime; import lombok.Data; @Data public class Task { private Long id; private String title; private Boolean done; private LocalDateTime createdAt; } // ───────────────────── // たったこれだけ。 // @Dataの力でgetter/setterは // 全部自動生成される。 // ───────────────────── // テーブルとクラスの対応 // MySQLのtasksテーブル +----+--------+------+------------+ | id | title | done | created_at | +----+--------+------+------------+ | 1 | 牛乳 | 0 | 2026-05-17 | | 2 | パン | 1 | 2026-05-17 | +----+--------+------+------------+ // ↕ 自動対応 Task task = new Task(); task.setId(1L); task.setTitle("牛乳"); task.setDone(false); task.setCreatedAt(...); // テーブル1行 = Taskインスタンス1個
POINT ──

フィールド名はcreatedAt（キャメルケース）で、テーブルのカラム名created_at（スネークケース）と形が違うことに注目。これはVol.2で設定したmap-underscore-to-camel-case=trueのおかげで自動変換されます。だから両方の流儀でJava側・DB側がきれいに書けます。

05

用語カード：Lombok

TERM CARD — LOMBOK

Vol.1から依存に入れていたLombok。ついに登場です。先ほどのTaskクラスで使った@DataがLombokの代表機能。面倒なボイラープレートコードを自動生成してくれる魔法のライブラリです。

TERM CARD #02
Lombok
ボイラープレート撲滅ライブラリ
定義
Javaクラスにアノテーションを付けるだけで、getter/setter/toString/equals/hashCodeなどを自動生成してくれるライブラリ。コンパイル時にバイトコードを直接書き換える仕組み。
構造
主なアノテーション：
@Data ── 全部入り（getter+setter+toString+equals+hashCode）
@Getter／@Setter ── 個別
@NoArgsConstructor ── 引数なしコンストラクタ
@AllArgsConstructor ── 全フィールド引数コンストラクタ
@Builder ── Builderパターンを自動実装
対比
Lombokなし── 4フィールドのクラスでもgetter/setterを書くと40行以上に膨らむ。
Lombokあり── @Dataの1行で同じ機能。コード量1/5以下。
嬉しさ
① コードが激減（本質的なフィールド定義だけ書けばよい）
② 修正漏れ防止（フィールド追加時にgetter追加し忘れる事故なし）
③ 読みやすい（クラスの意図が一目で分かる）
④ 業務でも標準（多くのプロジェクトで採用）
@data-magic @Dataが裏で生成するコード
// @Dataがやってくれること

@Dataを1行付けると、コンパイル時に下記のコードが自動的に追加された状態になります（実際のソースには書かれませんが、バイトコードに展開されます）。

これにより、エディタの補完にもgetTitle()やsetDone()がちゃんと出てきます。Lombokは「書かなくても、書いたのと同じ」を実現する仕組み。

注意── Eclipseで使うにはLombokプラグインのインストールが必要。Vol.1のプロジェクト作成時にPleiadesに入っていればOKです。入っていない場合はlombok.jarを実行してEclipseに紐付けます。
// 実際のソース（あなたが書く） @Data public class Task { private Long id; private String title; private Boolean done; private LocalDateTime createdAt; } // Lombokが裏で展開するコード（イメージ） public class Task { private Long id; private String title; private Boolean done; private LocalDateTime createdAt; // ↓ 全部自動生成 ↓ public Long getId() { return id; } public void setId(Long id) { this.id = id; } public String getTitle() { return title; } public void setTitle(String title) { this.title = title; } public Boolean getDone() { return done; } public void setDone(Boolean done) { this.done = done; } // ...createdAtのgetter/setter // ...toString() // ...equals() // ...hashCode() } // ↑ これを書かずに済む！
TIP ──

Lombokが本当に裏で生成しているか確認したければ、Eclipseでアウトラインビュー（ウィンドウ → ビューの表示 → アウトライン）を開いてください。Taskクラスを選ぶと、ソースに書いていないはずのgetId()やsetTitle()がメソッド一覧に並んでいるのが見えます。これがLombokの仕事です。

Mapperを本物に置き換える

REPLACE MAPPER WITH ENTITY VERSION

Vol.2のRawTaskMapper（Map<String, Object>を返すもの）を、型安全なTaskMapper（List<Task>を返すもの）に置き換えます。返り値の型が変わるだけで、書き心地が劇的に良くなります。

step-by-step TaskMapper作成手順
// 手順
A. mapperパッケージを作る
  1. src/main/java > com.example.todoを右クリック
  2. 新規(N) → パッケージ
  3. 名前に com.example.todo.mapper と入力
  4. 完了(F)
B. TaskMapperインターフェイスを作る
  1. 作ったmapperパッケージを右クリック
  2. 新規(N) → インターフェイス（クラスではない）
  3. 名前に TaskMapper と入力
  4. 完了(F)
  5. 右のコードをコピペして Ctrl+Shift+O でimport補完
  6. Ctrl+S で保存
⚠ ハマりどころ予告── 「クラス」と「インターフェイス」を間違えないこと。Mapperは必ずインターフェイスです。実装クラスはMyBatisが裏で自動生成します。
// src/main/java/com/example/todo/mapper/TaskMapper.java package com.example.todo.mapper; import java.util.List; import org.apache.ibatis.annotations.Mapper; import org.apache.ibatis.annotations.Select; import com.example.todo.entity.Task; @Mapper public interface TaskMapper { @Select("SELECT id, title, done, created_at " + "FROM tasks ORDER BY id") List<Task> findAll(); } // 何が変わったか（RawTaskMapperと比較） // 旧版（RawTaskMapper / Vol.2） List<Map<String, Object>> findAllRaw(); // 新版（TaskMapper / Vol.3） List<Task> findAll(); // 違いは「返り値の型」だけ // でも、その差は大きい： // Map版: row.get("titel") タイポ無検出 // Task版: task.getTitel() コンパイル不可
POINT ──

SQLの結果カラム名はcreated_at（スネークケース）、TaskクラスのフィールドはcreatedAt（キャメルケース）。名前が違うのにどうやって対応する？ Vol.2のapplication.propertiesに書いたmybatis.configuration.map-underscore-to-camel-case=trueが、MyBatisに「スネーク→キャメル変換しろ」と指示しています。裏で文字列変換が走って自動マッピングされる、という因果です。

07

Controllerを本物に置き換える

REPLACE CONTROLLER WITH MVC VERSION

Vol.2のRawTaskControllerは@RestControllerでJSONを返していました。Vol.3では@Controllerに変えてThymeleafテンプレートを描画するよう書き直します。MVCのCの本領発揮です。

@restcontroller-vs-@controller 2つのControllerアノテーションの違い
// @RestController と @Controller の違い
@RestController

返り値をJSON文字列として返す。Vol.2のRawTaskControllerはこれ。API用。

return tasks;
// ↓ ブラウザに表示
[{"id":1,...},{"id":2,...}]
@Controller

返り値をテンプレート名として扱う。HTML画面用。Modelにデータ載せて渡す。

return "tasks/list";
// ↓ ブラウザに表示
<table>...HTMLが描画される...

因果関係： @Controllerがついたメソッドが"tasks/list"を返すと、Spring MVCはtemplates/tasks/list.htmlを探してThymeleafで描画します。これが「ViewResolver」という仕組みの働きです。

// src/main/java/com/example/todo/controller/TaskController.java package com.example.todo.controller; import java.util.List; import org.springframework.stereotype.Controller; import org.springframework.ui.Model; import org.springframework.web.bind.annotation.GetMapping; import com.example.todo.entity.Task; import com.example.todo.mapper.TaskMapper; @Controller public class TaskController { private final TaskMapper taskMapper; // コンストラクタ注入（DI） public TaskController(TaskMapper taskMapper) { this.taskMapper = taskMapper; } @GetMapping("/tasks") public String list(Model model) { // ① DBからTask一覧を取得 List<Task> tasks = taskMapper.findAll(); // ② Modelに載せる（HTMLに渡すデータ） model.addAttribute("tasks", tasks); // ③ テンプレート名を返す return "tasks/list"; // → templates/tasks/list.html が描画される } } // 処理の流れ（仕組み） GET /tasks ↓ TaskController.list() が呼ばれる ↓ taskMapper.findAll() でDB問合せ ↓ List<Task> 取得 model.addAttribute("tasks", tasks) ↓ return "tasks/list" ↓ Spring MVCのViewResolverが templates/tasks/list.html を探す ↓ Thymeleafがテンプレートを処理 ↓ HTMLがブラウザに返る
POINT ──

「コンストラクタ注入」（final ＋ コンストラクタ）は業務開発でもっとも推奨される依存注入の書き方。@Autowiredを書く必要すらありません（Springが自動でコンストラクタ呼び出し）。final付きなので「後から差し替え不可＝不変」が保証されます。詳しい仕組みはVol.4のService層で深掘りします。

08

用語カード：Thymeleaf

TERM CARD — THYMELEAF

Vol.1から依存に入れていたThymeleaf（タイムリーフ）がついに登場。Spring Boot公式が推奨するテンプレートエンジンです。これまでJSON返却だった画面が、ここからHTMLに進化します。

TERM CARD #03
Thymeleaf（テンプレートエンジン）
HTML+データ → 動的HTMLの生成器
定義
HTMLテンプレートにJavaのデータを埋め込んで、最終的なHTMLをサーバー側で生成する仕組み。Spring Bootの標準推奨。
構造
テンプレートファイル（.html）に専用の属性を書く：
th:text ── テキストとして表示
th:each ── リストをループして繰り返し描画
th:if／th:unless ── 条件分岐
${...} ── Modelに載せたデータを参照
対比
JSP（旧来の仕組み）── HTMLの中にJavaコードが混在し、読みづらい。
Thymeleaf── 普通のHTMLとして開ける。th:属性は無視されるので、デザイナーがブラウザでプレビューできる。
嬉しさ
① サーバー側でHTML完成（SEO/初期表示に強い）
② Javaの値を安全に埋め込める（XSS自動エスケープ）
③ 普通のHTMLとして読める（プレビュー可能）
④ Spring Bootとの統合が完璧（設定不要）
how-it-works テンプレートが画面になる仕組み
// テンプレート＋データ＝HTML
[1] テンプレート（list.html） <tr th:each="task : ${tasks}"> <td th:text="${task.title}"></td> </tr> ＋ [2] Modelに載ったデータ tasks = [Task(牛乳), Task(パン)] ↓ Thymeleaf [3] 生成されるHTML <tr><td>牛乳</td></tr> <tr><td>パン</td></tr> ↓ ブラウザへ [4] 表として表示される

因果関係： Controllerがmodel.addAttribute("tasks", tasks)とデータを載せる → Thymeleafがテンプレート中の${tasks}でそれを参照 → th:eachでリストをループ → HTMLが完成、という流れです。

// 主なThymeleaf属性の早見表 // テキスト表示 <span th:text="${task.title}"></span> // → タスクのタイトルが表示される // 繰り返し <tr th:each="task : ${tasks}"> <td th:text="${task.title}"></td> </tr> // → リストの数だけ<tr>が増える // 条件分岐 <span th:if="${task.done}">✓</span> <span th:unless="${task.done}">-</span> // → done=trueなら✓、falseなら- // 属性に変数埋め込み <a th:href="@{/tasks/{id}(id=${task.id})}"> // → href="/tasks/1" のようにURL生成 // 連結 <td th:text="|ID: ${task.id}|"> // → "ID: 1" のように|で挟むと連結
TIP ──

Thymeleafテンプレートは必ずsrc/main/resources/templates/配下に置きます。これはVol.1で出ていた警告Cannot find template location: classpath:/templates/の原因の場所。Vol.3でこのフォルダにファイルを作れば、警告も自動的に消えます。

POINT── <html xmlns:th="http://www.thymeleaf.org"> のxmlns:th宣言は必ず。これがあるからThymeleafはth:属性を認識します。呪文として最初に書くと覚えてください。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

v2の新セクション。Vol.3でよく起きるエラーを「症状→原因→対処」の因果で結びます。エラー文のどこを見ればよいかも併記。

⚠ ハマり 1 ── ホワイトラベル500: Error querying database
SYMPTOM ブラウザに「Whitelabel Error Page (type=Internal Server Error, status=500)」、ログに BadSqlGrammarException や ### SQL: の行が出る
CAUSE @Select内のSQLにタイポがある（ORDERD BYなど）。MySQLが文法エラーで拒否 → MyBatisが例外 → Spring MVCが500を返す、という因果連鎖。ログの### SQL:に実際に実行されたSQLが出るので、まずそこを読む。
FIX TaskMapperの@Select内SQLを修正。本書の正解は "SELECT id, title, done, created_at FROM tasks ORDER BY id"。DevToolsが自動再起動するのでブラウザ再読込で復帰。
⚠ ハマり 2 ── 404: /tasks にアクセスしてもページが見つからない
SYMPTOM ブラウザに「Whitelabel Error Page (type=Not Found, status=404)」、コンソールに例外スタックトレースは無し
CAUSE そのURLを担当するControllerが見つからない。考えられる原因は3つ：① @GetMapping("/tasks")のパス文字列にタイポ／② @Controllerアノテーションを書き忘れた／③ TaskControllerをcom.example.todoパッケージ外に作ってしまい、@ComponentScanに拾われていない。
FIX ① "/tasks" のスペル確認／② @Controllerがついているか確認／③ パッケージ階層がcom.example.todo.controllerになっているか確認。
⚠ ハマり 3 ── 500: Could not resolve view with name 'tasks/list'
SYMPTOM コンソールログに Could not resolve view with name 'tasks/list' in servlet with name 'dispatcherServlet'
CAUSE Thymeleafがテンプレートファイルを見つけられない。Controllerが"tasks/list"を返したので、Springはsrc/main/resources/templates/tasks/list.htmlを探したが無かった、または場所が違う。
FIX ファイルの場所を確認： src/main/resources/templates/tasks/list.html。「templatesフォルダ直下にtasksフォルダ、その中にlist.html」という3階層が正解。resources/tasks/list.html（templatesが抜けている）など要注意。
⚠ ハマり 4 ── 画面に「null」と表示される列がある
SYMPTOM テーブルは表示されるが、特定の列（例えば「作成日時」）にnullと出る
CAUSE DBカラム名 → Javaフィールド名の自動変換が効いていない。例えばcreated_atカラムをcreatedAtフィールドに対応させる設定が抜けている。具体的にはVol.2のapplication.propertiesにmap-underscore-to-camel-case=trueがない、またはタイポしている。
FIX application.propertiesに mybatis.configuration.map-underscore-to-camel-case=true を追加。アプリ再起動後、再度/tasksを開く。
⚠ ハマり 5 ── getter not found / 補完にgetTitle()が出ない
SYMPTOM Taskクラスに@Dataを付けたのに、task.getTitle()が赤線になる、または補完候補に出てこない
CAUSE EclipseにLombokプラグインが入っていない。pom.xmlにlombok依存があってもMavenはコンパイル時に解決するだけ。Eclipseのエディタが@Dataを「理解」するには別途プラグインが必要。
FIX Pleiades版なら最初から入っているはず。入っていない場合は~/.m2/repository/org/projectlombok/lombok/X.X.X/lombok-X.X.X.jarをダブルクリックで実行 → Eclipseインストールパスを指定 → Eclipse再起動。
POINT ── エラーログの読み方

長いスタックトレースは「3点だけ」読めばOKです：
① 一番上の例外名（例：BadSqlGrammarException ＝ SQL文法エラー）
② ### SQL: 行（実際に走ったSQL）
③ ### The error may exist in 行（原因のファイル）
残りの at xxx.xxx(...) はライブラリ内部の経路なので、自分のコード（com.example.todo.*）以外は気にしなくて大丈夫です。
