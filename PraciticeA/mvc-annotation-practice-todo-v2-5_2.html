03

用語カード：フォームクラスとEntityの分離

TERM — FORM vs ENTITY

Vol.5で初登場するフォームクラス（Form）。すでにTask（Entity）があるのに、なぜわざわざ別のクラスを作るのか── 「画面用」と「DB用」を分ける思想を押さえます。

TERM CARD #01
フォームクラス（Form Class）
画面入力を受け取る専用クラス
定義
HTMLフォームから送信されるデータを受け取るための専用クラス。com.example.todo.formパッケージに置く。
構造
Entityと似ているが、画面入力に対応するフィールドだけを持つ。普通のJavaクラスで、@Dataを付けてgetter/setterを自動生成。
対比
Task（Entity） ── テーブルtasksと1対1対応。id、title、done、createdAt等を持つ。
TaskForm（Form） ── 入力フォームと1対1対応。Vol.5ではtitleのみ。
共通点：両方ともデータの入れ物（POJO）
違い：何の世界を表現するかが違う
嬉しさ
① 画面入力項目とDB項目が独立して進化できる
② DBにない項目（パスワード確認用など）も気軽にFormに追加できる
③ バリデーション（Vol.8）はFormに付けるのが自然
④ セキュリティ面で安全（DBの全項目を入力受け付けする事故を防げる）
why-separate なぜ分けるか具体例
// 同じ「Task」概念でも世界が違う

Vol.5時点では、TaskFormはとてもシンプル：

@Data
public class TaskForm {
  private String title;  // 入力項目はこれだけ
}

「これってTaskと同じじゃないの？」── と思うかもしれません。今は確かにそう。でも将来：

  • Vol.8でバリデーションが入る → Formに@NotBlankを付ける（Entityには付けたくない）
  • Vol.6以降で編集機能を作る → 編集用Formは登録Formと違うかもしれない（id必要等）
  • 実務でよくある：パスワード確認欄、利用規約同意チェックなどDBには保存しない項目がフォームに必要になる

この変化が起きた時、EntityとFormが分かれていれば片方ずつ進化できる。一緒だと両方に影響が出てぐちゃぐちゃに。「未来の柔軟性のため、最初から分ける」のが業務開発の鉄則です。

// 画面とDBの「世界」の違い // === Task (Entity) === // DBテーブル tasks と1対1対応 @Data public class Task { private Long id; // DBで採番 private String title; private Boolean done; // DB状態 private LocalDateTime createdAt; // DB自動 } // === TaskForm (Form) === // 新規登録フォームと1対1対応 @Data public class TaskForm { private String title; // 入力したいのはタイトルだけ // id は不要（DBが自動採番） // done は不要（初期状態は false） // createdAt は不要（DBが NOW()） } // 変換は Service の仕事 @Transactional public void register(TaskForm form) { Task task = new Task(); task.setTitle(form.getTitle()); // done と createdAt は SQL 側で設定 taskMapper.insert(task); } // Form → Entity の変換 // これが Service の「業務ロジック」
POINT ── 「DTO」という総称

Form と Entity に加えて、業界では DTO（Data Transfer Object） という総称も使われます。「データを運ぶための入れ物クラス」の総称で、Form は「画面からServiceへの DTO」とも言えます。これら「データの入れ物クラス」の使い分けは別冊で深掘りします。

04

用語カード：@PostMapping

TERM — @POSTMAPPING

Vol.3で覚えた@GetMappingのPOST版。表示はGET、登録はPOST── これがHTTPの基本作法です。

TERM CARD #02
@PostMapping
POSTリクエストの受付窓口
定義
HTTPのPOSTメソッドでやってきたリクエストを処理するメソッドに付けるアノテーション。@GetMappingと全く同じ作法でURLを指定。
仕組み
Springが起動時に (HTTPメソッド, URL) → メソッド の対応表を作る。@GetMapping("/tasks")と@PostMapping("/tasks")は同じURLでも別の処理として登録される。
使い分け
GET ── データを表示する（一覧、フォーム表示）。何度叩いても影響なし
POST ── データを変更する（登録、更新、削除）。叩くたびに影響あり
業界の原則： GETでDBを変更してはいけない／POSTで表示専用はおかしい
嬉しさ
① HTTPメソッドで処理の意図が明確になる
② ブラウザの戻るボタンでPOSTが二重実行されない（警告が出る）
③ 同じURLでもメソッドで処理を分けられる
get-vs-post GETとPOSTの使い分け
// Vol.5で使う3つのエンドポイント

Vol.5完成時点でアプリは3つのエンドポイントを持ちます：

メソッド URL 処理内容
GET /tasks 一覧表示（Vol.3で作成）
GET /tasks/new 新規登録フォーム表示
POST /tasks 新規登録実行

注目すべきは GET /tasks と POST /tasks が同じURLだが別の処理であること。SpringはHTTPメソッドで処理を分けるので、これが成立します。

// 3つのエンドポイントの実装 @Controller public class TaskController { ... // 一覧（既存） @GetMapping("/tasks") public String list(Model m) { m.addAttribute("tasks", taskService.findAll()); return "tasks/list"; } // 登録フォーム表示（新規） @GetMapping("/tasks/new") public String newForm(Model m) { m.addAttribute("taskForm", new TaskForm()); return "tasks/new"; } // 登録実行（新規） @PostMapping("/tasks") public String create( TaskForm taskForm) { taskService.register(taskForm); return "redirect:/tasks"; } } // 同じURL「/tasks」でも // GET: list() を呼ぶ // POST: create() を呼ぶ // Springが HTTPメソッドで判断
POINT ── 「/tasks/new」のような新規作成専用URL

業界の慣例では 「/{資源}/new」 が「新規作成フォーム」を意味します。/tasks/new = 「新しいタスクを作るフォーム」。これはRESTful URL設計の標準パターンで、慣れるとURLを見ただけで意図が分かるようになります。Vol.6以降、/tasks/{id}/edit（編集フォーム）等の慣例URLも登場します。

05

用語カード：PRGパターン

TERM — POST-REDIRECT-GET

POST処理の後、必ず別のGETにリダイレクトする設計パターン。「リロードで二重登録」を防ぐWeb開発の定番作法です。

TERM CARD #03
PRGパターン（Post-Redirect-Get）
POST後は必ずリダイレクトする
定義
「POST処理を受けたサーバは、レスポンスとしてHTMLを返さず、別のGETエンドポイントへのリダイレクトを返す」設計パターン。「Post → Redirect → Get」の3ステップから命名。
解決する問題
POST直後にブラウザでリロードすると、POSTが再送信される問題。リダイレクト先のGETに飛ばすことで、リロード時の挙動が「一覧画面を再取得する」になり、二重登録が起きない。
Vol.5での実装
POST /tasks 処理の最後に return "redirect:/tasks"; と書く。SpringがHTTP 302リダイレクトレスポンスを返し、ブラウザが自動的にGET /tasksを再リクエストする。
嬉しさ
① 二重登録の事故を防ぐ（クレジットカード決済等で重大）
② ブラウザの戻るボタンで「フォーム再送信しますか？」警告が出なくなる
③ URLの整合性が保たれる（登録後のURLが一覧画面のURL）
why-prg PRGなしの悪夢
// PRG無しの場合のシナリオ
【PRG 無し （ダメな実装）】 1. ユーザーがフォームを送信 POST /tasks (タイトル: "牛乳") ↓ 2. サーバが登録 → 一覧HTMLを直接返す レスポンス: 一覧画面のHTML ↓ 3. ブラウザのアドレス欄: /tasks (POSTのURLのまま) ↓ 4. ★ユーザーがリロード(F5) ↓ 5. ブラウザ:「POSTを再送信しますか?」 ↓ OK 6. POST /tasks (タイトル: "牛乳") ← 再送信 ↓ 7. ★同じタスクが二重登録される★ 【PRG あり （正しい実装）】 1. ユーザーがフォームを送信 POST /tasks (タイトル: "牛乳") ↓ 2. サーバが登録 → リダイレクトを返す レスポンス: HTTP 302 → /tasks ↓ 3. ブラウザが自動で GET /tasks ↓ 4. サーバが一覧HTMLを返す ↓ 5. ブラウザのアドレス欄: /tasks (GETのURLになっている) ↓ 6. ユーザーがリロード(F5) ↓ 7. GET /tasks 再実行 (何も問題なし) ↓ 8. ★同じ一覧が再表示されるだけ★
// Springでの実装 // === PRGなし（ダメ） === @PostMapping("/tasks") public String create(...) { taskService.register(form); return "tasks/list"; // ↑ テンプレート名を直接返す // → 一覧HTMLが直接送られる // → URLは POST /tasks のまま // → リロードで二重登録 } // === PRGあり（正解） === @PostMapping("/tasks") public String create(...) { taskService.register(form); return "redirect:/tasks"; // ↑ "redirect:" プレフィックスが鍵 // → HTTP 302 を返す // → ブラウザが /tasks を再リクエスト // → URLが GET /tasks に変わる // → リロードしても再送信されない } // redirect: の書き方バリエーション return "redirect:/tasks"; // 一覧へ return "redirect:/tasks/new"; // 新規へ return "redirect:/"; // ルートへ // "redirect:" + 任意のURLパス // テンプレート名ではなくリダイレクト先
POINT ── 「書き込み処理の後は必ずリダイレクト」

業務開発の絶対的ルール：「INSERT/UPDATE/DELETEの後は必ずリダイレクトで返す」。これを守るだけで、ユーザーが知らずに二重登録する事故を完全に防げます。return "redirect:/..."; の1行を忘れないだけで、品質が劇的に上がります。深い原理は別冊で。

07

Mapperに insert を追加する

ADD @INSERT TO MAPPER

Vol.3で作った TaskMapper に登録用メソッドを1つ追加します。@Selectと並んで@InsertもMyBatisの基本アノテーションです。

add-insert-method Mapperを修正する
// 手順
TaskMapper.java を修正
  1. com.example.todo.mapper.TaskMapper.java を開く
  2. 既存の @Select メソッド findAll() の下に新しい @Insert メソッドを追加
  3. Ctrl+Shift+O でimport補完（@Insert のimportが追加される）
  4. Ctrl+S で保存

SQLの解読：

INSERT INTO tasks (title, done, created_at)
VALUES (#{title}, false, NOW())
  • (title, done, created_at) ── 値を入れるカラム名のリスト
  • VALUES (...) ── 対応する値のリスト
  • #{title} ── 引数オブジェクトのtitleプロパティを動的に埋め込む（プレースホルダ）
  • false ── 完了フラグ。新規登録時は未完了固定
  • NOW() ── MySQL関数。現在日時を返す
// 修正後の TaskMapper.java（全体） package com.example.todo.mapper; import java.util.List; import org.apache.ibatis.annotations.Insert; import org.apache.ibatis.annotations.Mapper; import org.apache.ibatis.annotations.Select; import com.example.todo.entity.Task; @Mapper public interface TaskMapper { // 一覧取得（Vol.3で作成） @Select("SELECT id, title, done, " + "created_at FROM tasks " + "ORDER BY id") List<Task> findAll(); // 新規登録（Vol.5で追加） @Insert("INSERT INTO tasks " + "(title, done, created_at) " + "VALUES (#{title}, false, NOW())") void insert(Task task); } // SQLの動き // Service側で: Task task = new Task(); task.setTitle("牛乳"); taskMapper.insert(task); // MyBatisが実際に流すSQL: INSERT INTO tasks (title, done, created_at) VALUES ('牛乳', false, NOW()); // #{title} は task.getTitle() の値 // SQLインジェクション対策で // PreparedStatement になっている // （安全！） // 返り値が void である理由 void insert(Task task); // ↑ 返り値なし // INSERT は「成功 or 例外」の二択 // 成功時は特に返すものがない // 失敗時は例外でServiceに伝わる // （応用：1件INSERTした影響行数を // int で受け取る派もあるが、 // 業務的にあまり使われない）
POINT ── 「#{}」と「${}」の違い

MyBatisには #{...} と ${...} の2種類のプレースホルダがあります。#{...}はPreparedStatement（SQLインジェクション対策あり、ユーザー入力に使ってOK）。${...}は文字列置換（対策なし、ユーザー入力には使うな）。業務開発では#{...}のみ使うのが鉄則です。

Serviceに register を追加する

ADD REGISTER TO SERVICE

Vol.4のService層がついに業務ロジックを持つ瞬間です。@Transactional(readOnly = true) をクラスに付けた状態で、書き込みメソッドだけを @Transactional で上書きします。

add-register-method Serviceを修正する
// 手順
TaskService.java を修正
  1. com.example.todo.service.TaskService.java を開く
  2. 既存の findAll() の下に新しい register() メソッドを追加
  3. メソッドに @Transactional を付ける（クラスのreadOnly=trueを上書き）
  4. Ctrl+Shift+O でimport補完（TaskFormのimportが追加される）
  5. Ctrl+S で保存

register()メソッドが行うこと：

  1. TaskFormを受け取る（Controllerから渡される）
  2. 新しいTaskインスタンスを作る
  3. FormのtitleをTaskにコピー
  4. Mapperに保存依頼

@Transactionalの効果：このメソッドの処理全体が1つのトランザクションになり、途中で例外が出たら自動でロールバックされる。

// 修正後の TaskService.java（全体） package com.example.todo.service; import java.util.List; import org.springframework.stereotype.Service; import org.springframework.transaction .annotation.Transactional; import com.example.todo.entity.Task; import com.example.todo.form.TaskForm; // ★追加 import com.example.todo.mapper.TaskMapper; @Service @Transactional(readOnly = true) public class TaskService { private final TaskMapper taskMapper; public TaskService(TaskMapper taskMapper) { this.taskMapper = taskMapper; } // 一覧取得（Vol.4で作成） public List<Task> findAll() { return taskMapper.findAll(); } // 新規登録（Vol.5で追加） @Transactional // ★書き込み有効に上書き public void register(TaskForm form) { Task task = new Task(); task.setTitle(form.getTitle()); taskMapper.insert(task); } } // クラス指定 vs メソッド指定の上書き @Service @Transactional(readOnly = true) public class TaskService { // クラスの readOnly=true が効く public List<Task> findAll() {...} // メソッド指定で上書き → 書き込み有効 @Transactional public void register(...) {...} } // ルール: メソッド指定がクラス指定より優先 // register() の業務ロジック public void register(TaskForm form) { // ① 新しいEntityインスタンス作成 Task task = new Task(); // ② FormからEntityへ値コピー task.setTitle(form.getTitle()); // id、done、createdAtはSQL側で設定 // ③ Mapperに保存依頼 taskMapper.insert(task); // 例外が出たら@Transactionalで自動ロールバック // 正常終了なら自動コミット }
POINT ── 「Service層の役目」が見えてくる

このVol.5の register() メソッドがService層の役目を体現しています：「Form（画面の世界） → Task(業務の世界） → Mapper呼び出し（DBの世界）」の橋渡し。Vol.4で作っておいた「業務ロジックの居場所」に、業務ロジックが入った瞬間です。Vol.8でバリデーション、Vol.9で例外処理が入ると、ここはさらに育っていきます。

ハマりどころ（因果で説明）

PITFALLS — CAUSE & EFFECT

Vol.5は修正・新規・テンプレートと作業範囲が広いので、ハマりやすいポイントも多めです。Vol.3 ADDの9ステップ思考フローを活かす場面。

⚠ ハマり 1 ── フォーム送信したら 404
SYMPTOM フォームに入力して「登録」を押すと、ブラウザに 404 Not Found が表示される
CAUSE @PostMapping("/tasks") を付け忘れ、または @GetMapping のままになっている。送信先（POST /tasks）に対応するメソッドが存在しないので404。
FIX create() メソッドのアノテーションが @PostMapping("/tasks") になっていることを確認。@GetMappingと間違えやすいので注意。
⚠ ハマり 2 ── 登録ボタンを押すとフォーム画面が再表示される
SYMPTOM 登録ボタンを押すと、一覧に戻らずに同じフォーム画面に戻る。DBは更新されている。
CAUSE create() の戻り値が return "tasks/new"; や return "tasks/list"; になっている。"redirect:" プレフィックスが無いのでテンプレート名として扱われている。
FIX return "redirect:/tasks"; に修正。「POSTの後はredirect」が業務の鉄則。
⚠ ハマり 3 ── BindingResult exception または Model has no taskForm
SYMPTOM /tasks/new を開くと java.lang.IllegalStateException: Neither BindingResult nor plain target object for bean name 'taskForm' available as request attribute 等のエラー
CAUSE newForm() で model.addAttribute("taskForm", new TaskForm()) を忘れている、または属性名が "taskForm" 以外になっている。new.html の th:object="${taskForm}" が参照先を見つけられない。
FIX newForm() でModelに "taskForm" という名前で空のTaskFormを載せる。属性名・引数名・th:object すべてが「taskForm」で統一されているか確認。
⚠ ハマり 4 ── INSERT時に「Field 'created_at' doesn't have a default value」
SYMPTOM 登録ボタンを押すと500エラー。コンソールに Field 'created_at' doesn't have a default value や類似のSQLエラー
CAUSE @Insert のSQLで created_at を指定していない、または NOW() を書き忘れている。MySQLの設定によってはNULL不可で挿入エラーになる。
FIX SQLを INSERT INTO tasks (title, done, created_at) VALUES (#{title}, false, NOW()) に修正。3つのカラムすべてに値を指定する。
⚠ ハマり 5 ── TaskFormのimportが間違っている
SYMPTOM Eclipse上で TaskForm に赤線、または cannot find symbol エラー
CAUSE com.example.todo.form.TaskForm のimportが無い、またはVol.3 ADDで学んだ「同名異クラスの罠」で別のTaskFormを誤って選んだ。
FIX Service と Controller の冒頭で import com.example.todo.form.TaskForm; が入っているか確認。Ctrl+Shift+O で候補が出たら、必ず com.example.todo.form のものを選ぶ。
⚠ ハマり 6 ── @Transactional が反映されない
SYMPTOM register実行時に Connection is read-only 等の例外が出る
CAUSE register() メソッドに @Transactional（メソッド指定）を付け忘れ、クラスの @Transactional(readOnly = true) のみが効いている → 書き込み禁止モード。
FIX register() の直前に @Transactional を付ける（属性なしでOK、デフォルトが書き込み有効）。
POINT ── 「ハマり多発はチャンス」

Vol.5はファイルを6つ触る大型回。1〜2個はハマる前提で進めてください。ハマったらVol.3 ADDの9ステップ思考フロー（症状→種別→3点読み→精読→真犯人→仮説→修正→確認→再発防止）で淡々と解く。解いた回数だけ、業務開発の即戦力に近づきます。
