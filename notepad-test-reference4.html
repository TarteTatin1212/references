01

Controllerテストとは

WHAT IS CONTROLLER TEST

Controllerの仕事は「URLを受けて、正しい画面とデータを返す」こと。それを確かめるには、実際に擬似HTTPリクエストを送って、返ってきたビュー名やステータスを検証します。

@WebMvcTest + MockMvc Web層だけを試す
// リクエストの流れ
MockMvc　擬似HTTPで GET /memos を送る
↓
NoteController　テスト対象（本物）
↓ 呼ぶ
NoteService　@MockitoBeanで偽物に
POINT ── Serviceは偽物でいい

Controllerテストで確かめたいのは「URLとビューの対応」「Serviceを正しく呼ぶか」。Serviceの中身が正しいかはVol.2で済んでいる。だからここではServiceを偽物にして、Controllerの責務だけに集中します。

// テスト対象：NoteController（おさらい） @GetMapping("/memos") public String list(Model model) { model.addAttribute("notes", noteService.findAll()); return "memos/memos"; } // ─── ここで確かめたいこと ─── // ① GET /memos が成功するか（status） // ② ビュー名が "memos/memos" か // ③ Modelに "notes" が入っているか // ④ Serviceの findAll を呼んだか // ※ findAll が本当にDBから取れるかは // Vol.2/3で確認済み。ここでは見ない。 // ─── @WebMvcTest の範囲 ─── // Controller・フィルタ等のWeb層だけ起動。 // Service/Mapper/DBは読み込まない。 // → @SpringBootTest より軽い。
02

骨格と@MockitoBean

SKELETON

Controllerテストの基本形です。@WebMvcTest で対象Controllerを指定し、MockMvc を注入。依存するServiceは @MockitoBean で偽物にします。

NoteControllerTest.java 骨格
// @MockitoBean とは

Vol.2の @Mock はMockitoだけのMock。一方 @MockitoBean はSpringの管理する偽物です。

@WebMvcTest はSpringの世界で動くので、Controllerが @Autowired するServiceもSpring管理の偽物でないと差し込めません。それが @MockitoBean です。

⚠ @MockBean は非推奨

以前は @MockBean でしたが、Spring Boot 3.4以降は非推奨。このアプリは3.5系なので、新しい @MockitoBean を使います。

// src/test/java/jp/example/controller/ // NoteControllerTest.java package jp.example.controller; import static org.mockito.Mockito.*; import static org.springframework.test.web .servlet.request.MockMvcRequestBuilders.*; import static org.springframework.test.web .servlet.result.MockMvcResultMatchers.*; import org.junit.jupiter.api.Test; import org.springframework.beans.factory .annotation.Autowired; import org.springframework.boot.test .autoconfigure.web.servlet.WebMvcTest; import org.springframework.test.context .bean.override.mockito.MockitoBean; import org.springframework.test.web .servlet.MockMvc; import jp.example.service.NoteService; @WebMvcTest(NoteController.class) class NoteControllerTest { @Autowired MockMvc mockMvc; // 擬似HTTP @MockitoBean NoteService noteService; // Spring管理の偽物 // ↓ ここに各テストを書く }

一覧（GET /memos）

LIST

最初のテスト。GET /memos を擬似的に送り、成功するか・正しいビュー名か・Modelにデータが入るかを確かめます。ビュー名は実コードどおり memos/memos。

GET /memos 一覧の検証
// 3つの検証ポイント
  • status().isOk()：200 が返るか
  • view().name("memos/memos")：ビュー名が正しいか
  • model().attributeExists("notes")：Modelに notes があるか
POINT ── andExpect を重ねる

mockMvc.perform(...) でリクエストを送り、.andExpect(...) をいくつも重ねて検証します。「成功して、このビューで、このデータがある」を一気に確かめられる。ビュー名は実ファイルと完全一致させること（memos/memos）。

// 一覧テスト @Test void 一覧_memos画面を返す() throws Exception { // given：Serviceが空リストを返すと仕込む when(noteService.findAll()) .thenReturn(List.of()); // when & then：GET /memos を送って検証 mockMvc.perform(get("/memos")) .andExpect(status().isOk()) .andExpect( view().name("memos/memos")) .andExpect( model().attributeExists("notes")); // Serviceが呼ばれたかも確認 verify(noteService).findAll(); } // perform(get("/memos")) // = GET /memos を擬似的に送る // andExpect(...) // = 返り値が期待通りか検証（重ねられる） // throws Exception は MockMvc のお約束。 // perform がチェック例外を投げるため。

DETAIL

パス変数つきのURL /memos/view/{id} を検証します。idが正しくServiceに渡るか、ビュー名が memos/view か、Modelに note が入るかを確かめます。

GET /memos/view/1 パス変数の検証
// 実行結果（イメージ）

POINT ── model().attribute で中身も検証

attributeExists("note") は「あるか」だけ。attribute("note", 期待値) なら中身まで検証できます。Serviceが返すNoteを仕込んでおき、それがModelに乗るかを確かめます。ビュー名は memos/view（実ファイルと一致）。

// 詳細テスト @Test void 詳細_view画面を返す() throws Exception { // given：id=1 のNoteを返すと仕込む Note note = new Note(); note.setId(1L); note.setTitle("買い物"); when(noteService.findById(1L)) .thenReturn(note); // when & then mockMvc.perform(get("/memos/view/1")) .andExpect(status().isOk()) .andExpect( view().name("memos/view")) .andExpect( model().attribute("note", note)); // idが正しく渡ったか verify(noteService).findById(1L); } // get("/memos/view/1") の "1" が // @PathVariable で id=1L として渡る。 // verify(...).findById(1L) でそれを確認。

登録とリダイレクト

CREATE & REDIRECT

POSTでの登録はリダイレクトが絡みます。POST /memos で登録後、redirect:/memos へ飛ぶ。このリダイレクト先を検証する書き方を学びます。

POST /memos 登録とリダイレクト検証
// リダイレクトの検証

登録が成功すると、Controllerは "redirect:/memos" を返します。これはビュー名ではなくリダイレクト指示。

なので検証は view().name(...) ではなく、status().is3xxRedirection()（リダイレクトのステータス）と redirectedUrl("/memos")（飛び先）を使います。

POINT ── POSTにはパラメータを付ける

登録にはフォームの入力が要ります。.param("title", "買い物") でPOSTパラメータを付けます。タイトルは必須（@NotBlank）なので、正常系では必ず入れること。入れ忘れるとバリデーションで弾かれ、新規画面に戻ってしまいます（その検証はVol.5で）。

// 登録テスト（正常系） @Test void 登録_成功で一覧へリダイレクト() throws Exception { // when & then：POST /memos mockMvc.perform(post("/memos") .param("title", "買い物") .param("content", "牛乳")) .andExpect( status().is3xxRedirection()) .andExpect( redirectedUrl("/memos")); // Serviceのregisterが呼ばれたか verify(noteService) .register(any()); } // is3xxRedirection() … リダイレクトのstatus // redirectedUrl("/memos") … 飛び先の検証 // register(any()) … 「何らかのNoteForm」で // registerが呼ばれたか。引数の中身は // Vol.2のServiceテストで検証済みなので // ここでは any() で十分。

削除（POST）

DELETE

削除もPOSTで、リダイレクトします（Vol.4のフロントで学んだ通り、削除はGETでなくPOST）。POST /memos/delete/{id} を検証します。

POST /memos/delete/1 削除の検証
// 削除テストのポイント

削除は「正しいidでServiceのdeleteを呼び、一覧へリダイレクトする」が確かめたいこと。

verify(noteService).delete(1L) で、URLのidが正しくServiceに渡ったかを確認します。

TIP ── GETでは弾かれるはず

このControllerの delete は @PostMapping です。試しに get("/memos/delete/1") を送ると、405 Method Not Allowed になるはず。「削除はPOSTのみ」という設計を、テストで保証することもできます（余裕があれば）。

// 削除テスト @Test void 削除_一覧へリダイレクト() throws Exception { // when & then：POST /memos/delete/1 mockMvc.perform( post("/memos/delete/1")) .andExpect( status().is3xxRedirection()) .andExpect( redirectedUrl("/memos")); // 正しいidでdeleteが呼ばれたか verify(noteService).delete(1L); } // ─── 編集（update）も同様の形 ─── // post("/memos/edit/1") // .param("id", "1") // .param("title", "新タイトル") // → is3xxRedirection // → redirectedUrl("/memos") // → verify(noteService).update(any())

例外時の挙動を記録

EXCEPTION BEHAVIOR

存在しないメモを開くと NoteNotFoundException が飛び、GlobalExceptionHandler が error/404 を返します。ここでこのアプリの「クセ」をテストで正直に記録します。

404画面だがステータス200 現状の挙動を固定
// 重要な事実
⚠ ステータスは 200 のまま

GlobalExceptionHandler の handleTaskNotFound は error/404 を返しますが、@ResponseStatus が無いためHTTPステータスは200（OK）です。「404画面なのにステータスは200」という状態。

POINT ── 現状をテストで固定する意味

これは直すべき点（本来404を返すべき）ですが、今は「現状の挙動」をテストで記録します。リファクタリングで @ResponseStatus(NOT_FOUND) を足したら、このテストを200→404に書き換える。テストが「いつ挙動を変えたか」の記録になります。

// 例外時の挙動を記録するテスト @Test void 詳細_存在しないと404画面_ただし200() throws Exception { // given：Serviceが例外を投げると仕込む when(noteService.findById(99L)) .thenThrow( new NoteNotFoundException(99L)); // when & then mockMvc.perform(get("/memos/view/99")) // 現状：ステータスは200のまま .andExpect(status().isOk()) // 404用のビューが返る .andExpect( view().name("error/404")) // message属性にエラー文が入る .andExpect( model().attributeExists("message")); } // ─── リファクタ後の姿（伏線）─── // @ResponseStatus(HttpStatus.NOT_FOUND) // を例外 or ハンドラに付けたら… // status().isOk() → status().isNotFound() // にこのテストを書き換えることになる。 // thenThrow(...) … モックに「呼ばれたら // 例外を投げろ」と指示する書き方。

ハマりどころ

PITFALLS

Controllerテスト特有の詰まりポイントを3つ。Web層ならではの罠です。

よくある詰まり 3つの罠
// 詰まりポイント
  • @MockitoBean を忘れる：Serviceが用意されず「Bean が無い」エラー。@WebMvcTestはService実体を読まない
  • ビュー名のタイプミス：memos/memo など微妙な違いで赤に。実ファイル名と完全一致させる
  • throws Exception 忘れ：perform はチェック例外を投げる。メソッドに付け忘れるとコンパイルエラー
POINT ── @WebMvcTest はWeb層だけ

@WebMvcTest はServiceもMapperも読み込みません。だから依存するServiceは必ず @MockitoBean で用意する。これを忘れると「依存Beanが見つからない」で起動に失敗します。Web層だけを切り出すのが@WebMvcTestの本質です。

// 罠①：@MockitoBean 忘れ // NG：NoteService の偽物が無い @WebMvcTest(NoteController.class) class NoteControllerTest { @Autowired MockMvc mockMvc; // → 起動時「NoteService が無い」エラー } // OK：@MockitoBean で用意 @MockitoBean NoteService noteService; // 罠②：ビュー名のタイプミス // 実ファイル：templates/memos/memos.html // 実コード ：return "memos/memos" view().name("memos/memo") // ✗ 末尾s抜け view().name("memos/memos") // ✓ 完全一致 // 罠③：throws Exception 忘れ // perform はチェック例外を投げるので void test() throws Exception { // ← 必須 mockMvc.perform(...); }
