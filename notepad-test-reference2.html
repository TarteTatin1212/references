01

なぜモックにするか

WHY MOCK

NoteService をテストしたいのに、中で NoteMapper（本物のDB接続）を呼びます。本物を使うとDBの準備が要る・遅い・壊れやすい。そこでMapperを偽物に差し替えます。

mock = 偽物 依存を差し替える
// 本物 vs モック
NoteService（テスト対象・本物）
↓ 依存
NoteMapper（モック＝偽物に差し替え）
「findById(1)が来たらこのNoteを返せ」
とこちらで指示できる
POINT ── テストしたいのは Service のロジック

確かめたいのは「Serviceが正しく振る舞うか」。Mapperが本当にDBから取れるかは別の話（Vol.3で確かめる）。だからここではMapperは「指示通りに返す偽物」で十分。Serviceのロジックだけに集中できます。

// テスト対象：NoteService（おさらい） public Note findById(Long id) { Note note = noteMapper.findById(id); if (note == null) { throw new NoteNotFoundException(id); } return note; } // ─── ここで確かめたいこと ─── // ① Mapperがnoteを返したら、そのまま返すか // ② Mapperがnullを返したら、例外を投げるか // ②の「nullのとき」を本物のDBで再現するのは // 面倒（存在しないデータを用意する必要）。 // モックなら "nullを返せ" と一言で指示できる。 // ─── モックの利点 ─── // ・DB不要（接続も準備もいらない） // ・Spring起動不要（一瞬で終わる） // ・「異常な状況」も自由に作れる
02

テストクラスの骨格

SKELETON

Mockitoを使うテストの基本形です。@Mock で偽物を作り、@InjectMocks でそれをテスト対象に注入します。3つのアノテーションを押さえましょう。

NoteServiceTest.java 骨格と3つの注釈
// 3つのアノテーション
注釈意味
@ExtendWith
(MockitoExtension.class)		Mockitoを有効化
@Mock偽物を作る（Mapper）
@InjectMocks偽物を対象に注入（Service）
POINT ── フィールド注入でも刺さる

NoteService は @Autowired private NoteMapper（フィールド注入）です。@InjectMocks はこのフィールドに直接モックを差し込めます。だから現状コードのままテストできます。Spring起動は不要（@SpringBootTest は使わない）。

// src/test/java/jp/example/service/ // NoteServiceTest.java package jp.example.service; import static org.assertj.core.api.Assertions.*; import static org.mockito.Mockito.*; import org.junit.jupiter.api.Test; import org.junit.jupiter.api.extension.ExtendWith; import org.mockito.InjectMocks; import org.mockito.Mock; import org.mockito.junit.jupiter.MockitoExtension; import jp.example.entity.Note; import jp.example.exception.NoteNotFoundException; import jp.example.form.NoteForm; @ExtendWith(MockitoExtension.class) class NoteServiceTest { @Mock NoteMapper noteMapper; // 偽物 @InjectMocks NoteService noteService; // 偽物を注入 // ↓ ここに各テストメソッドを書く }
03

Given-When-Then

GIVEN-WHEN-THEN

テストは3段構成で書くと読みやすくなります。Given（準備）→ When（実行）→ Then（検証）。この型を体に入れると、どんなテストも迷わず書けます。

G-W-T テストの型
// 3段構成
GIVEN（準備）モックの振る舞いを決める。「findById(1)が来たらこのNoteを返せ」
WHEN（実行）テスト対象のメソッドを呼ぶ。service.findById(1)
THEN（検証）結果が期待通りか確かめる。assertThat(...)
TIP ── コメントで区切るだけでいい

特別な書き方は不要。コメントで // given // when // then と区切るだけで、テストの意図が一目で伝わります。「何を準備し、何を実行し、何を確かめたか」が読む人に伝わるのが大事です。

// 型のイメージ @Test void テスト名() { // given（準備） // モックに「こう来たらこう返せ」を仕込む when(noteMapper.findById(...)) .thenReturn(...); // when（実行） // テストしたいメソッドを呼ぶ Note result = noteService.findById(...); // then（検証） // 結果が期待通りか確かめる assertThat(result)...; } // この3段を意識するだけで // 「何をテストしているか」が明確になる。
04

findById 正常系

findById ── SUCCESS

最初の本物のテスト。Mapperがメモを返したら、Serviceはそのまま返す——この当たり前を確かめます。Given-When-Thenの型で書きます。

findById_存在すれば返す 正常系
// 実行結果（イメージ）
JUnit
NoteServiceTest
✓ findById_存在すればNoteを返す
Tests: 1 passed ✓
POINT ── assertThat の使い方

assertThat(実際の値) に続けて .isEqualTo(期待) や .isSameAs(...) で検証。ここでは「返ってきたNoteが、モックに仕込んだNoteと同じか」「titleが期待通りか」を確かめます。1つのテストで1つのことを確かめるのがコツです。

// findById 正常系 @Test void findById_存在すればNoteを返す() { // given：idが1のNoteを用意して仕込む Note stub = new Note(); stub.setId(1L); stub.setTitle("買い物"); stub.setContent("牛乳"); when(noteMapper.findById(1L)) .thenReturn(stub); // when：ServiceのfindByidを呼ぶ Note result = noteService.findById(1L); // then：返り値を検証 assertThat(result.getTitle()) .isEqualTo("買い物"); assertThat(result.getContent()) .isEqualTo("牛乳"); } // when(...).thenReturn(...) // = 「findById(1L)が来たらstubを返せ」 // という、モックへの指示（スタブ化）。
05

findById 異常系

findById ── EXCEPTION

テストの真価は異常系にあります。Mapperがnullを返したら、Serviceは NoteNotFoundException を投げる。これを確かめます。例外のテストには専用の書き方があります。

assertThatThrownBy 例外が出ることを検証
// 例外検証の考え方

「例外が投げられること」自体が期待する結果です。普通の assertThat では書けないので、assertThatThrownBy を使います。

これは「この処理を実行したら例外が出るはず」を検証する専用メソッド。例外の型とメッセージまで確かめられます。

POINT ── メッセージまで検証できる

例外クラスは NoteNotFoundException、メッセージは "Note not found: id = 99"（実コードで確認済み）。だから .hasMessage("Note not found: id = 99") まで書けます。「正しい例外が、正しいメッセージで」出ることを保証します。

// findById 異常系 @Test void findById_無ければ例外を投げる() { // given：Mapperがnullを返すよう仕込む when(noteMapper.findById(99L)) .thenReturn(null); // when & then：例外が出ることを検証 assertThatThrownBy(() -> noteService.findById(99L)) .isInstanceOf( NoteNotFoundException.class) .hasMessage( "Note not found: id = 99"); } // assertThatThrownBy( () -> 危険な処理 ) // .isInstanceOf(例外の型) ← 型を検証 // .hasMessage(メッセージ) ← 文言を検証 // () -> ... はラムダ式。 // 「この処理を実行したら」という意味。 // ─── これがリファクタの安全網になる ─── // 将来 findById の中身を変えても、 // 「nullなら例外」が保たれているか // このテストが自動で見張ってくれる。
06

register と変換の検証

ArgumentCaptor

register は Form→Entity を手動コピーして insert を呼びます。「正しくコピーされたEntityがMapperに渡ったか」を ArgumentCaptor で捕まえて確かめます。

ArgumentCaptor<Note> 渡された引数を捕獲
// なぜCaptorが要るか

register は戻り値がありません（void）。だから「結果」を直接見られない。

代わりに「Mapperのinsertに、どんなNoteが渡されたか」を捕まえれば、Form→Entityの変換が正しいか確かめられます。それが ArgumentCaptor の役目です。

POINT ── 「渡された引数」を捕まえる罠

ArgumentCaptor はモックに渡された引数を横取りして保管する仕掛け。verify(mock).insert(captor.capture()) で捕まえ、captor.getValue() で中身を取り出す。voidメソッドの「中で何が起きたか」を検証する定番です。

// register の変換を検証 import org.mockito.ArgumentCaptor; @Test void register_FormをEntityに変換して保存() { // given：登録するフォーム NoteForm form = new NoteForm(); form.setTitle("買い物"); form.setContent("牛乳とパン"); // when：登録を実行 noteService.register(form); // then：insertに渡されたNoteを捕獲 ArgumentCaptor<Note> captor = ArgumentCaptor.forClass(Note.class); verify(noteMapper).insert(captor.capture()); Note passed = captor.getValue(); assertThat(passed.getTitle()) .isEqualTo("買い物"); assertThat(passed.getContent()) .isEqualTo("牛乳とパン"); } // verify(mock).insert(captor.capture()) // = 「insertが呼ばれた。その引数を捕獲」 // captor.getValue() で中身を取り出す。 // → Form→Entityのコピーが正しいか検証。
07

findAll・delete の委譲確認

VERIFY DELEGATION

findAll と delete はMapperにそのまま処理を任せるだけ。こういう「委譲するだけ」のメソッドは、正しくMapperを呼んだかを verify で確かめます。

verify(mock).method() 呼び出しの確認
// verify とは

verify(mock).メソッド(引数) は「そのメソッドが、その引数で呼ばれたか」を確かめます。

delete は戻り値なし。中身も「Mapperを呼ぶだけ」。だから「正しいidでdeleteが呼ばれたか」を verify で確認すれば十分です。

TIP ── 「何を確かめるか」を選ぶ

findAllは「Mapperの結果をそのまま返すか」を検証。deleteは「正しいidでMapperを呼んだか」を検証。メソッドの性質に応じて、確かめる対象を選びます。やみくもに全部verifyしない。

// findAll：結果をそのまま返すか @Test void findAll_Mapperの結果を返す() { // given List<Note> stubList = List.of(new Note(), new Note()); when(noteMapper.findAll()) .thenReturn(stubList); // when List<Note> result = noteService.findAll(); // then：件数を検証 assertThat(result).hasSize(2); } // delete：正しいidで呼ばれたか @Test void delete_正しいidでMapperを呼ぶ() { // when noteService.delete(5L); // then：delete(5L)が1回呼ばれたか verify(noteMapper).delete(5L); } // verify はデフォルトで「ちょうど1回」を検証。 // verify(mock, times(2)) で回数指定も可能。

ハマりどころ

PITFALLS

Mockitoの単体テストで初学者が必ず踏む罠を3つ。先に知っておけば、赤いエラーに慌てずに済みます。

よくある詰まり 3つの罠
// 詰まりポイント
  • @ExtendWith の付け忘れ：これが無いと @Mock/@InjectMocks が動かず NullPointer に
  • UnnecessaryStubbing：when で仕込んだのに使わないと怒られる。使う分だけ仕込む
  • @SpringBootTest を使ってしまう：単体テストには重すぎ。Mockitoだけで十分
POINT ── 単体テストは軽く保つ

Service単体テストの長所は「速い・Spring不要・DB不要」。@SpringBootTest を付けるとアプリ全体が起動して台無しに。@ExtendWith(MockitoExtension.class) だけで、一瞬で終わる軽いテストになります。

// 罠①：@ExtendWith 付け忘れ // NG：これだと @Mock が null のまま class NoteServiceTest { ... } // OK：必ず付ける @ExtendWith(MockitoExtension.class) class NoteServiceTest { ... } // 罠②：UnnecessaryStubbing // 仕込んだのに使わないと: // → UnnecessaryStubbingException when(noteMapper.findById(1L)) // 使わないなら .thenReturn(stub); // 消す // 「使う分だけ仕込む」が基本。 // 罠③：重い注釈を使わない // NG：単体テストには過剰 @SpringBootTest // OK：軽量 @ExtendWith(MockitoExtension.class)
