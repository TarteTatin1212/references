バリデーションの仕様確認

VALIDATION RULES

テストの前に「何が正しい仕様か」を確定させます。実際の NoteForm を見ると、制約が付くのはtitle だけ。content には何もありません。ここを正確に押さえます。

NoteForm の制約 title のみ・content無し
// フィールドごとの制約
フィールド制約
title@NotBlank（必須）
@Size(max=30)
content制約なし（空でもOK）
POINT ── 仕様を先に確定する

テストは「仕様が正しく実装されているか」を確かめるもの。だからまず仕様を明文化します。「titleは必須・30字以内、contentは自由」。これがテストの合格基準になります。実コードを確認した事実そのままです。

// テスト対象：NoteForm（おさらい） @Data public class NoteForm { @NotBlank // 空・空白だけはNG @Size(max = 30) // 30文字まで private String title; private String content; // 制約なし } // ─── 確かめたいこと ─── // ① title 空 → NG（@NotBlank違反） // ② title 31文字 → NG（@Size違反） // ③ title 30文字 → OK（ちょうど境界） // ④ title 普通の文字列 → OK // ⑤ content 空 → OK（制約なし） // @NotBlank は「null・空文字・空白のみ」を // すべてNGにする（@NotNullより厳しい）。
02

Validatorで直接テスト

VALIDATOR

バリデーションはControllerを通さず、単体でテストできます。Validator を直接使い、フォームに違反がいくつあるかを調べる方法です。速くて確実です。

Validator#validate 制約違反を直接調べる
// 仕組み

Validator にフォームを渡すと、違反の集合（Set）が返ります。違反が無ければ空、あれば中身が入る。

「空のはず（違反なし）」「1件あるはず」を assertThat(violations) で確かめます。Spring起動もControllerも不要。

TIP ── ValidatorFactory で生成

Validation.buildDefaultValidatorFactory() から Validator を取り出します。@BeforeEach で各テスト前に用意しておくと使い回せます。これは純粋なJavaのテスト——Springの世界に入らないので一瞬で終わります。

// NoteFormTest.java の骨格 package jp.example.form; import static org.assertj.core.api.Assertions.*; import jakarta.validation.*; import org.junit.jupiter.api.*; import java.util.Set; class NoteFormTest { private Validator validator; @BeforeEach void setup() { ValidatorFactory f = Validation .buildDefaultValidatorFactory(); validator = f.getValidator(); } // ↓ ここに各テストを書く } // validate(form) が返す Set の中身を見る: // 空 → 違反なし（合格データ） // 要素あり → 違反あり（不合格データ）

境界値テスト（30文字）

BOUNDARY TEST

バリデーションテストの核心は境界値。「30文字はOK、31文字はNG」という境目を狙い撃ちします。真ん中の値より、境界こそがバグの出る場所です。

@Size(max=30) の境界 29/30/31を試す
// 境界値の考え方
title結果
空 ""NG
空白 " "NG
1文字OK
30文字OK
31文字NG
POINT ── 境界の「両側」を試す

「30はOK・31はNG」のように境界のすぐ両側をテストします。もし実装が <30（29まで）のように1ずれていたら、この2つのテストが捕まえます。真ん中の15文字を試しても境界バグは見つからない。境界こそ狙う価値があります。

// 境界値テスト @Test void title_空はNG() { NoteForm f = new NoteForm(); f.setTitle(""); Set<ConstraintViolation<NoteForm>> v = validator.validate(f); // 違反が1件以上あるはず assertThat(v).isNotEmpty(); } @Test void title_30文字はOK() { NoteForm f = new NoteForm(); f.setTitle("あ".repeat(30)); // 30文字 Set<ConstraintViolation<NoteForm>> v = validator.validate(f); // 違反なし（空）のはず assertThat(v).isEmpty(); } @Test void title_31文字はNG() { NoteForm f = new NoteForm(); f.setTitle("あ".repeat(31)); // 31文字 assertThat(validator.validate(f)) .isNotEmpty(); } // "あ".repeat(30) で30文字の文字列を生成。 // 境界ちょうど＝30 と 越えた＝31 を試す。
04

content は制約なし

content ── NO CONSTRAINT

忘れてはいけないのが「制約が無いこと」も仕様だという点。content は空でも長くてもOK。これを明示的にテストしておくと、誤って制約を足したときに気づけます。

content 自由 制約なしを保証する
// なぜ「制約なし」もテストするか

「title だけ入れて content は空」のフォームが合格することを確かめます。これは「content に制約が無い」という仕様の証明です。

OKtitle="買い物" / content="" → 違反0
OKtitle="買い物" / content=長文 → 違反0
POINT ── 仕様の「保険」になる

もし将来うっかり content に @NotBlank を足してしまったら、このテストが赤くなって教えてくれます。「contentは自由」という決定を、テストが見張る。「制約が無いこと」も立派な仕様です。

// content は制約なしを保証 @Test void content_空でもOK() { NoteForm f = new NoteForm(); f.setTitle("買い物"); // titleは入れる f.setContent(""); // contentは空 Set<ConstraintViolation<NoteForm>> v = validator.validate(f); // titleが正しければ違反なし assertThat(v).isEmpty(); } @Test void content_長文でもOK() { NoteForm f = new NoteForm(); f.setTitle("買い物"); f.setContent("あ".repeat(1000)); assertThat(validator.validate(f)) .isEmpty(); } // content に @Size 等が無いので、 // どんなに長くても違反にならない。 // これが「現状の仕様」。
05

不正入力で画面に戻る

INVALID → BACK TO FORM

Validator単体だけでなく、Controllerを通したときの挙動も確かめます。不正な入力でPOSTすると、リダイレクトせず新規画面に戻る。これをMockMvcで検証します。

POST /memos（不正） 画面に戻る検証
// 正常 vs 不正の分岐
正常（titleあり）register実行 → redirect:/memos
不正（title空）registerせず → memos/new に戻る
POINT ── 不正系こそ verify(never())

不正入力では register が呼ばれてはいけない。これを verify(noteService, never()).register(any()) で確かめます。「呼ばれないことの検証」も立派なテスト。Controllerの if (bindingResult.hasErrors()) が効いている証拠になります。

// 不正入力テスト（Controller経由） // ※ NoteControllerTest に追記（@WebMvcTest） @Test void 登録_title空ならnewに戻る() throws Exception { // when & then：titleを空でPOST mockMvc.perform(post("/memos") .param("title", "") .param("content", "本文")) // リダイレクトせず画面表示（200） .andExpect(status().isOk()) .andExpect( view().name("memos/new")) // titleにエラーが付いている .andExpect(model() .attributeHasFieldErrors( "noteForm", "title")); // registerは呼ばれていないはず verify(noteService, never()) .register(any()); } // attributeHasFieldErrors(オブジェクト名, // フィールド名) で「そのフィールドに // エラーがある」ことを検証。 // never() で「一度も呼ばれていない」を検証。

例外ハンドラの振り分け

@ControllerAdvice

後半は例外。GlobalExceptionHandler は2種類の例外を振り分けます。NoteNotFoundException → 404画面、その他すべて → 500画面。この振り分けを検証します。

例外の振り分け 404 と 500
// 振り分けの構造
例外が発生
↓ @ControllerAdvice が捕まえる
NoteNotFound
Exception
→ error/404
その他の
Exception
→ error/500
POINT ── 専用が優先、その他は受け皿

Springは「より具体的な例外ハンドラ」を優先します。NoteNotFoundException 専用のハンドラがあればそちらが、無ければ Exception の受け皿が捕まえる。だから特定の例外は404、想定外は全部500に振り分けられます。

// テスト対象（おさらい） @ControllerAdvice public class GlobalExceptionHandler { @ExceptionHandler( NoteNotFoundException.class) public String handleTaskNotFound(...) { model.addAttribute("message", ...); return "error/404"; } @ExceptionHandler(Exception.class) public String handleException(...) { model.addAttribute("message", ...); return "error/500"; } } // ─── テストでの検証方針 ─── // ① Serviceが NoteNotFoundException // → error/404 に飛ぶ // ② Serviceが その他の例外 // → error/500 に飛ぶ // どちらも MockMvc + @WebMvcTest で // Serviceに thenThrow を仕込んで検証する。

ステータス200問題の記録

STATUS 200 ISSUE

Vol.4でも触れた「404画面なのにHTTPステータスは200」問題。500画面も同じです。これを現状の事実として両方テストし、リファクタリングの的を明確にします。

404も500もステータス200 現状を正直に記録
// 実行結果（イメージ）
JUnit
例外振り分けテスト
✓ NotFound_error404_status200
✓ その他例外_error500_status200
Tests: 2 passed ✓
POINT ── 「直すべき点」を明確に残す

本来 404画面は HTTPステータスも404、500画面は500を返すべき。でも現状は両方200。これをテストで記録しておけば、リファクタリングで @ResponseStatus を足すとき、「どのテストを200→404/500に書き換えるか」が一目瞭然。テストが改善の的になります。

// 404の振り分けテスト @Test void NotFound例外_error404画面_status200() throws Exception { when(noteService.findById(99L)) .thenThrow( new NoteNotFoundException(99L)); mockMvc.perform(get("/memos/view/99")) .andExpect(status().isOk()) // 現状200 .andExpect(view().name("error/404")); } // その他例外の振り分けテスト @Test void その他例外_error500画面_status200() throws Exception { when(noteService.findAll()) .thenThrow( new RuntimeException("想定外")); mockMvc.perform(get("/memos")) .andExpect(status().isOk()) // 現状200 .andExpect(view().name("error/500")); } // ─── リファクタ後（伏線）─── // @ResponseStatus を足したら: // isOk() → isNotFound() / is5xxServerError() // にこれらを書き換える。

ハマりどころ

PITFALLS

バリデーション・例外テストの詰まりポイントを3つ。検証の対象を間違えやすい領域です。

よくある詰まり 3つの罠
// 詰まりポイント
  • @NotBlank と @NotNull の混同：@NotBlankは空白だけもNG。@NotNullはnullだけNG（空文字は通る）
  • @Valid の付け忘れ：Controllerの引数に @Valid が無いとバリデーションが動かない。実コードは付いている
  • 例外テストで実DBを期待：例外振り分けは@WebMvcTest（モック）で。本物のServiceは不要
POINT ── @NotBlank の意味を正確に

@NotBlank はnull・空文字・空白のみを全部NGにします（文字列専用・最も厳しい）。@NotEmpty は空白を通す、@NotNull はnullだけ見る。実コードは @NotBlank なので、テストでは「空白だけの文字列」もNGになることを確認できます。
