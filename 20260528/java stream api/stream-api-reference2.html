01

インターフェースの復習

INTERFACE REVIEW

まずは土台の土台。インターフェースは「やることリスト（約束）」です。中身は書かず、「このメソッドを必ず用意してね」と決めるだけ。実装する側がその約束を守ります。

interface = 約束ごと中身は実装クラスが書く
// 関係を図にすると
INTERFACE 約束Greeting
▼ implements
CLASS 実装中身を書く

約束（メソッド名）を決め、実装クラスが中身を埋める。

// インターフェースの基本
// 約束だけ書く（中身なし）
interface Greeting {
  String say();   // 中身なし
}

// 実装クラスが中身を書く
class Hello implements Greeting {
  public String say() {
    return "こんにちは";
  }
}
POINT —インターフェースは「メソッド名と戻り値だけ決めて、中身は実装者に任せる」仕組みです。この「中身は後で」という性質が、後のラムダ式に効いてきます。
02

関数型インターフェースとは

FUNCTIONAL INTERFACE

本題です。関数型インターフェース＝「中身を書くべきメソッドが、ちょうど1個だけ」のインターフェース。これだけ。さっきの Greeting も、メソッドが say() 1個だけなので、立派な関数型インターフェースです。

抽象メソッドが1個 = 関数型★★★ 最重要の定義
// 関数型かどうかの判定
メソッド1個 → 関数型 ◯
メソッド2個 → 関数型 ✕
メソッド0個 → 関数型 ✕

「中身を書く必要があるメソッド（抽象メソッド）」の数で決まります。

// これは全部 関数型インターフェース
// 1個だけ → OK
interface Greeting {
  String say();
}

interface Calc {
  int apply(int a, int b);
}

// 2個 → 関数型ではない
interface NG {
  void open();
  void close();  // 2個目でアウト
}
POINT —なぜ1個が大事か。メソッドが1個だけなら「このインターフェースの仕事＝そのメソッド」と一意に決まるからです。だからラムダ式という「中身だけの短い書き方」が使えるようになります。
03

@FunctionalInterface

THE ANNOTATION

関数型インターフェースには @FunctionalInterface という目印（アノテーション）を付けられます。「これは関数型ですよ」と宣言する札です。付けると、うっかりメソッドを2個書いた時にコンパイラが教えてくれます。

@FunctionalInterface付けると安全になる目印
// アノテーションの効果
あり：安全メソッドが2個になったらコンパイルエラーで警告してくれる
なし：危険2個になっても気づかず、後でラムダが使えず混乱

付けるのは任意。でも付けると間違いに早く気づけます。

// 札を付けておく
@FunctionalInterface
interface Calc {
  int apply(int a, int b);
}

// もし間違えて2個書くと…
@FunctionalInterface
interface Bad {
  void a();
  void b();  // ← ここで赤エラー！
}
// 「抽象メソッドが複数」と怒られる
TIP —@FunctionalInterface は必須ではありません。なくても1メソッドなら関数型として機能します。でも「これはラムダで使う前提だよ」という意思表示になるので、自作する時は付けるのが定番です。
04

なぜ「1メソッド」なのか

WHY ONLY ONE

少し立ち止まって考えます。なぜメソッドが1個だと、ラムダ式という短い書き方が許されるのか。理由は「迷わないから」です。メソッドが1個なら、書いた中身がどのメソッドのものか一目瞭然です。

1個だから省略できる考え方の核心
// 1個なら迷わない
メソッドが1個
→ 中身を書けば、どのメソッドかは明白
だから名前を省略できる
→ 中身（処理）だけ書けばいい
それがラムダ式
→ (a, b) -> a + b
// 「中身だけ」で済む理由
// Calc は apply 1個だけ
interface Calc {
  int apply(int a, int b);
}

// 中身を書く時、どうせ apply
// なので名前を書く必要がない

// → 引数と処理だけでOK
(a, b) -> a + b
  ↑ これがラムダ式の正体
POINT —関数型インターフェースとラムダ式はセットで覚えるのが鉄則です。「メソッド1個の約束（関数型インターフェース）」があるから、「中身だけ書く短い記法（ラムダ式）」が成立する。この因果関係がVOL.4で効いてきます。
05

匿名クラスで実装してみる

ANONYMOUS CLASS

ラムダ式の「前の時代」の書き方も見ておきましょう。匿名クラスです。これはクラス名を付けずにその場で実装する書き方。冗長ですが、ラムダ式がこれを短くしたものだと分かると理解が深まります。

匿名クラス = 名前なし実装ラムダの一歩手前
// 実行結果
8

3 + 5 が計算され、8 が表示されます。
長いけれど、ちゃんと動きます。

// 匿名クラスで Calc を実装
Calc calc = new Calc() {
  @Override
  public int apply(int a, int b) {
    return a + b;
  }
};

System.out.println(
  calc.apply(3, 5)  // → 8
);
TIP —この書き方、正直めんどくさいですよね。new Calc(){...}、@Override、public int apply…。やりたいのは「a + b」だけなのに。この「めんどくさい」を解決するのがラムダ式です。
06

ラムダへの橋渡し

BRIDGE TO LAMBDA

この巻のゴールです。匿名クラスから「飾り」をどんどん削ると、ラムダ式になります。同じ処理が、こんなに短くなる。その変化を目で追ってみてください。

削って削って → ラムダ式次巻の予告編
// 削れるもの
new Calc() → 型は分かるので不要
@Override → 1個しかないので不要
public int apply → 名前も不要
残るのは引数と中身だけ
// 同じ計算が、これだけに
// 匿名クラス（長い）
Calc c1 = new Calc() {
  public int apply(int a, int b){
    return a + b;
  }
};

// ラムダ式（短い！）
Calc c2 = (a, b) -> a + b;

// やってることは全く同じ
