LAMBDA EXPRESSION

01

ラムダ式の基本形

BASIC FORM

ラムダ式の形はたった1つ。「引数 → 処理」です。(引数) -> { 処理 }。この -> がラムダの目印。「左の引数を受け取って、右の処理をする」と読みます。

(引数) -> { 処理 }★★★ これが全ての基本
// 構造を分解
(a, b) 受け取る引数
-> 「これを受けて、こうする」
{ return a + b; } 実際の処理

左が入口、右が中身。アローで繋ぐだけ。

// もっとも省略していない形
// 2つの数を足すラムダ式
(int a, int b) -> {
    return a + b;
}

// 読み方：
// 「a と b を受け取って
//   a + b を返す」
POINT —まずはこの完全形を覚えてください。ここから「省略できるものを削る」のがラムダ上達の道。いきなり短い形を覚えると、なぜ短くできるのか分からず混乱します。遠回りが近道です。
02

アローの読み方

READING THE ARROW

前巻の匿名クラスと並べると、ラムダ式が「何を省いたものか」が一目で分かります。ラムダ式は匿名クラスから、決まりきった部分を全部消したもの。それだけです。

匿名クラス → ラムダ式同じ意味・違う見た目
// 何が消えたか
new 型名(){ } → 消える
public 戻り値 メソッド名 → 消える
残るのは引数と処理だけ
// 比べてみる
// 匿名クラス
new Calc() {
  public int apply(int a,int b){
    return a + b;
  }
}

// ラムダ式（同じ意味）
(a, b) -> a + b

// 「飾り」を全部消すと
// 引数と処理だけが残る
TIP —ラムダ式が突然出てきて戸惑ったら、「これは匿名クラスの省略形だ」と思い出してください。難しい新概念ではなく、「めんどくさい書き方を短くしただけ」。気持ちがラクになります。
03

省略ルール

OMISSION RULES

ここがこの巻の核心です。ラムダ式はさらに3つの省略ができます。①引数の型 ②引数1個ならカッコ ③1行なら波カッコとreturn。順番に削っていきます。

3段階で削る★★★ 省略の手順
// 削れるもの3つ
1引数の型（int等）
2引数1個なら( )
31行なら{ } と return

型は文脈から分かるので省略OK。Javaが自動で補ってくれます。

// 段階的に削っていく
// 完全形
(int a, int b) -> { return a+b; }

// ①型を削る
(a, b) -> { return a + b; }

// ③1行なら{}とreturn削る
(a, b) -> a + b

// 完成！すっきり
POINT —型を省略できるのは、渡す先の関数型インターフェースから型が決まっているから。前巻で学んだ Function<String,Integer> のような型指定が効いているわけです。土台がここで活きてきます。
04

引数1個なら、カッコも省略

SINGLE ARGUMENT

引数がちょうど1個のときは、引数を囲む ( ) も省略できます。(s) -> ... が s -> ... に。ストリームでいちばんよく見る形です。

s -> s.length()よく見るあの形の正体
// カッコ省略の条件
引数1個：省略OKs -> s.length()
引数0個や2個：カッコ必須() -> ... / (a, b) -> ...

「1個だけ」が省略の条件です。

// 引数1個のラムダ
// カッコあり
Function<String,Integer> f1 =
    (s) -> s.length();

// カッコなし（1個なのでOK）
Function<String,Integer> f2 =
    s -> s.length();

// f1 も f2 も全く同じ意味
TIP —引数が0個のときは () -> ... と、空カッコが必須です（前巻のSupplierがこれ）。「1個だけ省略OK、0個と複数はカッコ必要」と覚えてください。1個は特別扱いです。
05

関数型と組み合わせる

WITH FUNCTIONAL INTERFACE

前巻の4大インターフェースに、実際にラムダ式を入れてみます。「箱（インターフェース）」に「中身（ラムダ）」を入れる。これでようやく1セットになります。

箱にラムダを入れる土台パートの総まとめ
// 実行結果
true
HELLO

Predicateで判定、Functionで変換。
箱とラムダが噛み合っています。

// 4大インターフェース × ラムダ
// 判定（Predicate）
Predicate<String> notEmpty =
    s -> !s.isEmpty();
System.out.println(
  notEmpty.test("hi"));  // true

// 変換（Function）
Function<String,String> up =
    s -> s.toUpperCase();
System.out.println(
  up.apply("hello")); // HELLO
POINT —これで土台が完成しました。「関数型インターフェース（箱）＋ラムダ式（中身）」——この組み合わせが、ストリームのfilterやmapの正体です。Vol.7からのストリーム本編で、この知識をフル活用します。
06

ハマりどころ

COMMON PITFALLS

初学者がラムダ式でつまずく代表例を3つ。先に知っておけば回避できます。特に「複数行のとき」「returnの有無」は混乱の元です。

よくある間違い先回りして潰す
// つまずきポイント
複数行なのに{ }を省いた
{ }ありなのにreturnを忘れた
{ }なしなのにreturnを書いた
// NG例とOK例
// NG：1行だけど return と {} が
//      ちぐはぐ
s -> return s.length();  // エラー

// OK：1行 → {} も return も無し
s -> s.length();

// OK：複数行 → {} と return 必要
s -> {
  int n = s.length();
  return n * 2;
}
POINT —鉄則は「{ } を書いたら return も書く。{ } を書かないなら return も書かない」。この2つはセットです。波カッコの有無で「returnが要る／要らない」が決まる、と覚えれば9割のエラーは防げます。
NEXT STEP — ラムダ式、自在に書けるようになりましたか？「引数 → 処理」と3つの省略ルールが身につけば完璧です。次のVol.5「メソッド参照・前編」では、ラムダ式をさらに短くする「メソッド参照」に進みます。s -> s.length() が String::length になる、あの省略記法。まずは仕組みとstaticメソッド参照からです。
