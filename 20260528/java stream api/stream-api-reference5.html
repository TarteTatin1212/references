01

メソッド参照とは

WHAT IS IT

メソッド参照は「ラムダ式が、既存メソッドを1個呼ぶだけ」のときに使える、もっと短い書き方です。クラス名::メソッド名 という形。:: が目印です。

クラス名::メソッド名ラムダの究極の省略形
// 短くなる流れ
s -> s.length()
▼ メソッド参照に
String::length

引数すら書かなくてよくなります。

// ラムダ → メソッド参照
// ラムダ式
Function<String,Integer> f1 =
    s -> s.length();

// メソッド参照（同じ意味）
Function<String,Integer> f2 =
    String::length;

// :: は「このメソッドを使う」の印
POINT —メソッド参照は新しい機能ではありません。ラムダ式の「さらなる省略形」です。だから、無理に使う必要はなし。「ラムダで書けるものが、たまたまもっと短く書ける」くらいの気軽さでOKです。
02

なぜ短くできるのか

WHY SHORTER

なぜ引数を省略できるのか。それは「受け取った値を、そのままメソッドに渡すだけ」だからです。s -> s.length() は「sを受け取って、sのlengthを呼ぶ」。受け取ってそのまま渡すなら、引数の名前すら不要です。

「そのまま渡す」だけなら省ける仕組みの理解
// 省略できる条件
受け取ったsを…
そのままs.length()に使うだけ
→ sという名前が不要になる
→ String::length で済む
// 「ただ渡すだけ」を見抜く
// このラムダは…
s -> s.length()
// 「sを受け取り、sのメソッドを
//  呼ぶだけ」

// だから名前を省ける
String::length

// 逆に加工が入ると省略できない
s -> s.length() * 2
// ← これはメソッド参照にできない
TIP —見分け方の鉄則：「ラムダの中身が、メソッドを1個呼ぶ”だけ”か？」。それだけならメソッド参照にできます。* 2 のような追加の処理があると、ラムダのままにするしかありません。

4つの種類（全体像）

FOUR TYPES

メソッド参照には4種類あります。この前編で1つ、後編（Vol.6）で残り3つを扱います。まずは全体像だけつかんでください。完璧に覚える必要はありません。

メソッド参照 4種前編=1つ / 後編=3つ
// 4種類のマップ
①static クラス::staticメソッド ←この巻
②インスタンス（型） 型::メソッド Vol.6
③インスタンス（特定） 変数::メソッド Vol.6
④コンストラクタ クラス::new Vol.6
// 見た目の例（雰囲気だけ）
① static メソッド参照
Integer::parseInt

② 型のインスタンスメソッド参照
String::toUpperCase

③ 特定インスタンスのメソッド参照
System.out::println

④ コンストラクタ参照
ArrayList::new
POINT —4種類あると聞くと身構えますが、書き方の見た目はどれも ○○::△△ で同じです。違うのは「::の左に何が来るか」だけ。今は「4つあるんだな」で十分。この巻では①に集中します。
04

staticメソッド参照

STATIC METHOD REF

1つ目、staticメソッド参照です。クラス名::staticメソッド名。staticメソッド（インスタンスを作らず呼べるメソッド）を参照します。★★★ いちばん分かりやすい型です。

Integer::parseIntクラス名::staticメソッド
// 実行結果
123

文字列の "123" が、数値の 123 に変換されました。
Integer.parseInt を参照しています。

// staticメソッドを参照
// ラムダ式
Function<String,Integer> p1 =
    s -> Integer.parseInt(s);

// static メソッド参照（同じ意味）
Function<String,Integer> p2 =
    Integer::parseInt;

System.out.println(
  p2.apply("123"));  // → 123
TIP —parseInt は Integer.parseInt(...) のようにクラス名から直接呼べるメソッド（static）。だから Integer::parseInt と書けます。「クラス名.メソッド名で呼べるもの」がstaticメソッド参照の対象です。
05

実例で確認

IN PRACTICE

少し先取りして、ストリームの中で使ってみます。文字列のリストを、数値のリストに変換する例。map の中にstaticメソッド参照を入れると、驚くほどすっきりします。

ストリーム × static参照現場でよく使う形
// 実行結果
[1, 2, 3]

文字列リスト ["1","2","3"] が
数値リスト [1,2,3] になりました。

// 文字列リスト → 数値リスト
List<String> strs =
    List.of("1", "2", "3");

List<Integer> nums = strs.stream()
    .map(Integer::parseInt)
    .toList();

System.out.println(nums);
// → [1, 2, 3]

// map(s -> Integer.parseInt(s))
// と書くより、ずっと短い！
POINT —.map(Integer::parseInt) は「各要素をparseIntで変換する」と読めます。ラムダより主語がはっきりして読みやすい。これがメソッド参照が好まれる理由です。現場でも頻出します。
06

使える/使えないの判断

CAN I USE IT?

最後に判断基準です。「迷ったらラムダ式のままでOK」。メソッド参照は“短くできるときのボーナス”であって、必須ではありません。とはいえ見分けられると気持ちいいので、目安を示します。

メソッド参照にできる条件無理しなくていい
// 判断フロー
ラムダの中身はメソッド1個呼ぶだけ？
YES → メソッド参照にできる
計算や条件が混じってる？
YES → ラムダのままが正解
// できる・できないの例
// ◯ メソッド1個だけ → 参照OK
s -> Integer.parseInt(s)
   → Integer::parseInt

// ✕ 計算が混じる → ラムダのまま
s -> Integer.parseInt(s) + 1

// ✕ 条件分岐がある → ラムダのまま
s -> s.isEmpty() ? 0 : 1
POINT —無理にメソッド参照に変えようとして時間を溶かすのは本末転倒。「短く書けたらラッキー、書けなければラムダでOK」。読みやすさが第一です。IDEが「メソッド参照にできるよ」と提案してくれることも多いので、それに乗るのも手です。
