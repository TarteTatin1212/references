INTERFACE — a promise of abilities / chapter 5-6

塊②「抽象化」の後半です。interface（インターフェース）は、抽象クラスとよく似た仲間。でも役割が少し違います。キーワードは「できることの約束」。「このメソッドを持っていますよ」という宣言だけを集めたものです。implements で「その約束を守ります」と表明します。前巻のabstractと比べながら読むと、両者の違いがくっきり見えてきます。

01

「できることの約束」という考え方

A PROMISE

インターフェースは「これができますよ」という約束のリストです。中身は書きません。「泳げる」「飛べる」のように、能力の名前だけを並べます。それを守ると約束したクラスは、必ずその能力を実装します。

interface = contract 能力の約束、それがインターフェース
// たとえ話：資格・免許

「泳げる」という資格を考えてください。魚も人もカエルも、種類はバラバラ。でも「泳げる」という資格を持つ点では同じです。

インターフェースは「血のつながり（継承）」ではなく「持っている資格」でクラスをまとめる仕組みです。

// 「泳げる」という約束 interface Swimmable { void swim(); // 中身なし＝約束だけ } // この約束を守るクラスは、 // 種類が何であれ swim() を持つ。 // → 魚でも人でも「泳げる仲間」
02

interfaceの書き方

DEFINE INTERFACE

class の代わりに interface と書きます。中のメソッドはすべて中身なし（; で終わる）が基本。abstract を付ける必要すらありません（自動でそうなる約束だから）。とてもシンプルです。

interface { } メソッド名だけを並べる
// ルール
  • class ではなく interface
  • メソッドは中身なし・; 終わり
  • abstract や public は省略可（自動付与）

「やることリスト」だけを書く、と覚えればOKです。

// インターフェースの定義 interface Flyable { void fly(); int maxHeight(); // 複数並べてもOK } // 全部「中身なし」。 // これが Flyable という資格の中身。
03

implements で約束を守る

IMPLEMENTS

クラスがインターフェースの約束を守ると宣言するのが implements（インプリメンツ＝「実装する」）です。extends が「継ぐ」なら、implements は「約束を守る」。約束したメソッドは必ず中身を書く義務が生まれます。

class ... implements ... 約束を守ると表明する
// 流れ
Flyable（約束）
fly();
▲ implements
Bird
fly(){ "羽ばたく" }

Bird は「飛べる」と約束したので、fly() の中身を必ず書きます。

// implements で実装 class Bird implements Flyable { @Override public void fly() { System.out.println("羽ばたいて飛ぶ"); } @Override public int maxHeight() { return 100; } } // 約束した2つを両方書いた → 完成
POINT
インターフェースのメソッドを実装する時は public を付けます（インターフェース側が自動でpublic扱いなので、実装側も合わせる必要があるため）。付け忘れると地味にハマる定番ポイントです。
04

複数のインターフェースを実装できる

MULTIPLE IMPLEMENTS

ここがインターフェース最大の強み。クラスの継承（extends）は1つしかできないのに、インターフェースはいくつでも implements できます。「泳げて、かつ飛べる」を同時に表現できるわけです。

implements A, B, C 複数の資格を同時に持てる
// カモは泳げて飛べる
Swimmable
swim();
Flyable
fly();
▲ 両方 implements
Duck
swim() も fly() も持つ
// カンマ区切りで複数実装 class Duck implements Swimmable, Flyable { @Override public void swim() { System.out.println("スイスイ"); } @Override public void fly() { System.out.println("バサバサ"); } } // 両方の約束を守れば、両方の仲間になれる
TIP
クラスの多重継承（親を2つ持つこと）はJavaでは禁止されています。混乱のもとだからです。でもインターフェースならいくつ持ってもOK。これがインターフェースが重宝される大きな理由です。
05

06

なぜ便利なのか

WHY USEFUL

インターフェースの本当のうれしさは「中身を気にせず、能力だけで扱える」点です。「飛べるもの」をまとめて扱う時、それが鳥か飛行機かを気にしなくていい。これが現場のコードを柔軟にします。

polymorphism 能力でまとめて扱う
// 中身を気にしない

Flyable 型の変数には、Flyable を実装したものなら何でも入れられます。鳥でも飛行機でも。

使う側は fly() が呼べることだけ知っていればOK。中身の違いに振り回されないのが利点です。

// 能力でまとめて扱う Flyable[] things = { new Bird(), new Airplane() // 種類はバラバラ }; for (Flyable f : things) { f.fly(); // 鳥も飛行機も同じ呼び方で飛ばせる } // → "羽ばたいて飛ぶ" / "エンジンで飛ぶ"
07

おまけ：defaultメソッド

DEFAULT METHOD

「インターフェースは中身なし」と言いましたが、実は例外があります。default を付けると中身付きのメソッドも書けます。「実装しなくても使える共通機能」を用意したい時に便利です。最初は「そういうのもある」程度でOK。

default method 中身を持てる例外ルール
// どんな時に使う？

実装するクラスが「書いても書かなくてもいい」共通処理を、インターフェース側で用意できます。

実装側で書かなければ default の中身が使われ、書けばそちらが優先されます。

// default メソッド interface Greetable { void name(); // 普通の約束 default void hello() { // 中身つき！ System.out.println("こんにちは"); } } // implements 側は hello() を書かなくても // "こんにちは" が使える
POINT
これだけ覚えれば初心者卒業：インターフェースは「能力の約束」、implementsは「約束を守る宣言」、複数持てるのが強み。defaultは後回しでも全く問題ありません。
