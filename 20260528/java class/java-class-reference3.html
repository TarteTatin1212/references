抽象クラス編

ABSTRACT CLASS — incomplete blueprint / chapter 3-4

ここから塊②「抽象化」に入ります。最初の主役はabstract（アブストラクト）。日本語で「抽象的な」という意味です。「中身を書かないメソッド」と「new できないクラス」という、ちょっと不思議な存在が登場します。でも安心してください。「未完成の設計図」というたとえ一発で腑に落ちますよ。

こんな悩みから 抽象メソッド 抽象クラス 継承して完成させる new できない理由 普通のメソッドと混ぜる いつ使うか
01

こんな悩みから始まります

THE PROBLEM

動物クラスを考えます。犬も猫も鳥も「鳴く」けれど、鳴き方はバラバラ。親クラスの Animal に cry() を書きたいけれど、「親の時点では中身を決められない」。この悩みを解決するのが abstract です。

motivation 親では中身を決められないメソッド
// 困りごと

Animal を継承する犬・猫は、それぞれ違う鳴き方をします。

Animal
cry() ← 中身は？
▲
Dog
ワン
Cat
ニャー
// 親で書こうとすると… class Animal { void cry() { // ここに何を書く？？ // 犬かもしれないし猫かもしれない // 親の時点では決められない！ } } // → 「中身は子に任せる」と宣言したい // それが abstract メソッド
02

抽象メソッド（abstractメソッド）

ABSTRACT METHOD

中身（{ } の処理）を書かないメソッドを、抽象メソッドと呼びます。abstract を付けて、{ } の代わりに ; で終わらせるのがルール。「やることはあるけど、中身は子クラスに任せる」という宣言です。

abstract method 中身のない、宣言だけのメソッド
// 普通のメソッドとの違い
普通のメソッド
void cry() { ... } ← 中身あり、{ } で終わる
抽象メソッド
abstract void cry(); ← 中身なし、; で終わる

中身を書かない＝子に「必ず作ってね」と義務付ける、という意味になります。

// 抽象メソッドの書き方 abstract void cry(); // ↑キーワード ↑中身なし、セミコロンで終わり // これは「鳴くという行動はある。 // でも中身は子クラスが書くこと」 // という約束を表します
03

抽象クラス（abstractクラス）

ABSTRACT CLASS

抽象メソッドを1つでも持つクラスは、クラス自体にも abstract が必要です。これを抽象クラスと呼びます。「未完成のメソッドがあるから、このクラスはまだ未完成ですよ」という印です。

abstract class 未完成の設計図
// たとえ話：未完成の設計図

家の設計図に「ここに窓を付ける（種類は施主が選ぶ）」と書いてある状態。枠は決まっているが、一部は空欄。

この設計図のままでは家は建ちません（＝new できない）。空欄を埋めた完成版の設計図（子クラス）が必要です。

// 抽象クラスの完成形 abstract class Animal { // 抽象メソッド（中身は子に任せる） abstract void cry(); } // ↑ class の前にも abstract が必要 // もし abstract を付け忘れると… // 「抽象メソッドがあるのにクラスが // abstract じゃない」とエラーになる
04

継承して完成させる

EXTEND & COMPLETE

抽象クラスは extends で継承します。継承した子クラスは抽象メソッドの中身を必ず書かなければなりません。書けば完成、書かなければエラー。この「強制力」が abstract の最大の価値です。

extends 子クラスが空欄を埋める
// 完成までの流れ
Animal（abstract）
abstract cry();
▲ extends
Dog
cry(){ "ワン" }

Dog が cry() の中身を書いたので、Dog は完成＝new できるクラスになりました。

// 子クラスで中身を実装（オーバーライド） class Dog extends Animal { @Override void cry() { System.out.println("ワン"); // 中身を書いた } } class Cat extends Animal { @Override void cry() { System.out.println("ニャー"); } } // Dog も Cat も new できる！
POINT
抽象メソッドの中身を書き忘れると、子クラスもまだ未完成扱いになりコンパイルエラーになります。これは欠点ではなく「書き忘れを防いでくれる安全装置」です。
05

抽象クラスは new できない

CANNOT INSTANTIATE

抽象クラスの大きな特徴が「直接 new できない」こと。未完成の設計図から家が建たないのと同じです。new できるのは、空欄を埋めた子クラスだけ。これも仕様であり、わざとそうなっています。

no instantiation 未完成だから直接は作れない
// new できる / できない
NG
new Animal(); ← 抽象クラスは作れない
OK
new Dog(); ← 完成した子クラスはOK

「Animal という抽象的な動物」は世の中に存在しません。存在するのは犬や猫といった具体的な動物だけ。コードもそれを表しています。

// 実際のコード Animal a = new Animal(); // ❌ コンパイルエラー Animal dog = new Dog(); // ⭕ OK dog.cry(); // → ワン Animal cat = new Cat(); // ⭕ OK cat.cry(); // → ニャー // 型は Animal、中身は Dog や Cat。 // これがポリモーフィズムの土台です
06

普通のメソッドも混ぜられる

MIXING METHODS

抽象クラスは「中身あり」と「中身なし」のメソッドを混ぜて持てます。共通の処理は親に書いてしまい、バラバラな部分だけ abstract にする。これが抽象クラスの一番おいしい使い方です。

concrete + abstract 共通処理は親、違う部分だけ子に任せる
// 役割分担

「寝る」は全動物共通だから親に書く。「鳴く」はバラバラだから子に任せる。

親に書く（共通）
sleep() ← 全動物「Zzz」で同じ
子に任せる（バラバラ）
cry() ← 動物ごとに違う
// 混在させた抽象クラス abstract class Animal { // 中身あり：共通処理 void sleep() { System.out.println("Zzz..."); } // 中身なし：子に任せる abstract void cry(); } // Dog は sleep() をそのまま使い、 // cry() だけ自分で書けばよい
07

いつ使うか

WHEN TO USE

最後に「結論」です。抽象クラスは「共通の親を作りたいけれど、一部だけは子ごとに変えたい」時に使います。共通部分の使い回し＋バラバラ部分の強制、この2つを同時に叶えたい時の道具です。

summary 抽象クラスの使いどころ まとめ
// 結論

こういう時に抽象クラス：

  • 複数のクラスに共通の親を持たせたい
  • でも一部のメソッドは子ごとに違う
  • その違う部分を「書き忘れさせたくない」

3つ揃ったら abstract の出番です。

// 抽象クラスの3点セット（暗記不要・感覚でOK） // ① abstract class で宣言 abstract class Base { // ② 共通処理は普通に書く void common() { ... } // ③ 子に任せる部分は abstract abstract void custom(); }
こんな時は抽象クラス不要
共通の親がいらない、単独のクラスで済む → 普通のクラスでOK。
こんな時に抽象クラス
「テンプレートを親で用意し、細部だけ子に埋めさせる」設計をしたい時。
