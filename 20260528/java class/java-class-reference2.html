INITIALIZERS — instance & static / chapter 1-2

フィールドに最初の値を入れる作業が初期化（イニシャライズ）です。コンストラクタは知っていますね。実はコンストラクタ以外にも初期化を書ける場所があります。この巻ではその2つ——インスタンスイニシャライザとスタティックイニシャライザ——を学びます。さらに「どの順番で実行されるか」を整理すれば、初期化まわりのバグはほぼ怖くなくなります。

初期化のおさらい インスタンスイニシャライザ なぜ使うのか スタティックイニシャライザ なぜ使うのか 実行順序の整理 ハマりどころ
01

初期化のおさらい

REVIEW

まず復習です。フィールドに値を入れる方法を、知っているものから並べてみます。「宣言と同時」と「コンストラクタ」の2つはもう使えますね。今回はここに2つ追加します。

新スタティックイニシャライザ static { ... }

02

インスタンスイニシャライザ

INSTANCE INITIALIZER

クラスの中に、ただ{ } だけのブロックを書きます。これがインスタンスイニシャライザです。インスタンスを作るたびに（new するたびに）実行されます。コンストラクタの兄弟だと思ってください。

{ } block new のたびに走る初期化ブロック
// 動かすと…

右のコードで new Sample() を2回呼ぶと、ブロックも2回実行されます。

// 出力結果 インスタンスイニシャライザ実行
コンストラクタ実行
インスタンスイニシャライザ実行
コンストラクタ実行

毎回「ブロック → コンストラクタ」の順で走るのがポイントです。

// インスタンスイニシャライザ class Sample { { // ← これがインスタンスイニシャライザ System.out.println("インスタンスイニシャライザ実行"); } Sample() { System.out.println("コンストラクタ実行"); } } public static void main(String[] args) { new Sample(); new Sample(); // 2回 new した }
03

なぜ使うのか

WHY USE IT

「コンストラクタがあるのに、なぜブロックが必要？」と思いますよね。正直、使う場面はそんなに多くありません。でも便利なケースがひとつあります。コンストラクタが複数あって、共通の初期化を1か所にまとめたい時です。

use case 複数コンストラクタの共通処理をまとめる
// 比較してみる
まとめない場合
コンストラクタが3つあると、共通の初期化コードを3回コピペ。修正漏れの温床に。
ブロックにまとめる
共通処理を { } に1回書けば、どのコンストラクタでも必ず実行される。
// 共通処理をブロックに集約 class User { List<String> logs; { // どのコンストラクタでも必ず走る logs = new ArrayList<>(); logs.add("生成されました"); } User() { } User(String name) { } // どちらも logs は準備済み }
POINT
実務では「コンストラクタに直接書く」または「メソッドに切り出して呼ぶ」ことの方が多いです。インスタンスイニシャライザは「読めればOK」のテーマ。無理に使わなくても大丈夫です。
04

スタティックイニシャライザ

STATIC INITIALIZER

こちらは static { } と書きます。クラスが最初に使われる時、たった一度だけ実行されます。何回 new しても、再実行されません。static フィールド（クラス全体で共有する変数）の準備に使います。

static { } block 最初の一度だけ走る初期化ブロック
// 動かすと…

new Config() を何回呼んでも、static { } は1回しか走りません。

// 出力結果（new を3回しても） スタティック初期化（1回だけ）
コンストラクタ実行
コンストラクタ実行
コンストラクタ実行
// スタティックイニシャライザ class Config { static String appName; static { // 最初の一度だけ System.out.println("スタティック初期化（1回だけ）"); appName = "MyApp"; } Config() { System.out.println("コンストラクタ実行"); } }
05

なぜ使うのか

WHY USE IT

スタティックイニシャライザはインスタンスイニシャライザより出番が多いです。「複雑な準備が必要な共有データ」を一度だけ作るのに向いています。たとえば計算で求める定数表や、設定の読み込みなどです。

use case 一度だけ作ればよい共有データの準備
// 典型例

単純な代入なら宣言と同時でOK。でもループや計算が必要な準備は static { } の出番です。

右の例は「0〜9を2乗した表」を一度だけ作って、全インスタンスで共有しています。何度 new しても表は作り直されません。

// 計算が必要な共有データの準備 class Squares { static int[] table = new int[10]; static { for (int i = 0; i < 10; i++) { table[i] = i * i; // 0,1,4,9,16... } } // table はプログラム中ずっと使い回せる }
06

実行順序の整理

EXECUTION ORDER

ここがこの巻のいちばん大事なところです。4つの初期化が、どの順で実行されるか。この順番さえ覚えれば、「なぜか変数が空っぽ」系のバグはほぼ解決できます。

order.rule 初期化が走る順番（鉄則）
// 実行される順番
1static { } と static変数（最初の一度だけ）
2{ } とインスタンス変数（new のたびに）
3コンストラクタ Sample()

static が最優先で一度だけ → 残りは new ごとに上から、と覚えましょう。

// 全部入りで確認 class Demo { static { System.out.println("1: static"); } { System.out.println("2: instance"); } Demo() { System.out.println("3: constructor"); } } // new Demo(); new Demo(); の出力 ↓ // 1: static ← 一度だけ // 2: instance ← 1回目 // 3: constructor // 2: instance ← 2回目（static は出ない） // 3: constructor
POINT
覚え方は「スタ → インスタ → コンスト」。スタティックが先頭で1回きり、あとはインスタンス用が new のたびに繰り返されます。

ハマりどころ

COMMON PITFALLS

初学者が初期化で詰まる定番のポイントを2つ紹介します。どちらも「順番」を意識すれば防げます。

pitfall.guide よくあるミスと回避法
// ハマりどころ①

まだ準備されていない変数を使ってしまう。上から順に走るので、下で宣言した変数を上のブロックで使うとエラーになります。

NG
ブロックで使う変数が、ブロックより下に書いてある
OK
使う変数は、使う場所より上で宣言する
// ハマりどころ②：staticの中でインスタンス変数 class Bad { int count = 0; static { count = 5; // ❌ エラー！ // static の中では // インスタンス変数は使えない // （まだ存在しないから） } } // → static { } では static 変数だけ触る
詰まりポイント
static { } の中でインスタンス変数（static なしの変数）を触ろうとしてコンパイルエラー。
鉄則
static の世界 と インスタンスの世界 は別物。static ブロックからは static のものだけ触れる。
