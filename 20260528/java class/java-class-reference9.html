Q.01

abstractクラスとinterface、結局どっちを使えばいい？

// 判断基準
A. ANSWER

迷ったらまずinterfaceです。共通の処理やフィールドを親に持たせたい時だけabstractクラスを選びます。

interface を選ぶ
「○○できる」という能力でまとめたい／複数の能力を組み合わせたい
abstract を選ぶ
共通の処理やフィールドを親で用意したい／血のつながりで結びたい
// 判断のコツ // ① 共通のフィールドやメソッドの中身を // 親に書きたい？ // YES → abstract クラス // NO → interface // ② 複数の親を持たせたい？ // YES → interface（複数implements可） // NO → どちらでもOK // 迷ったら interface でだいたい正解
TIP
実はabstractクラスとinterfaceは併用できます。共通処理はabstractで親に置き、能力はinterfaceで足す——という設計が定番です。
Q.02

「Animal is abstract; cannot be instantiated」と怒られます

// 原因と対処
A. ANSWER

抽象クラスを直接 new しようとしたのが原因です。抽象クラスは未完成の設計図なので、new できるのは継承した子クラスだけ。

「Animal」型の変数に、子クラス（Dog や Cat）を入れる形にしましょう。

// NG / OK Animal a = new Animal(); // ❌ NG // ↑ 抽象クラスは new 不可 Animal a = new Dog(); // ⭕ OK // 型は Animal、中身は具体的な Dog // interface も同じ。new はできない Swimmable s = new Swimmable(); // ❌ Swimmable s = new Duck(); // ⭕
TIP
戻る先：Vol.3「抽象クラス編」。「new できない理由」のセクションを再読すると腑に落ちます。
Q.03

interfaceをimplementsしたのに「実装されていない」と怒られる

// よくある2パターン
A. ANSWER

原因はほぼ2つです。①メソッドを1つでも実装し忘れ、②publicを付け忘れ。とくに②は地味にハマります。

NG
void hello() { … }（publicなし）
OK
public void hello() { … }
// publicを忘れがち interface Greeting { void hello(); } class JP implements Greeting { void hello() { // ❌ public なし System.out.println("やあ"); } } // 正しくは： public void hello() { ... } // ⭕
TIP
interfaceのメソッドは自動的にpublic扱い。実装側で「より狭く」できないので、必ず public を付けます。戻る先：Vol.4。
Q.04

内部クラスを new しようとするとエラーになる

// 書き方が独特
A. ANSWER

内部クラス（static無し）の生成は独特の書き方です。外側を先にnewしてから .new。new Outer.Inner() ではダメ。

staticネステッドなら普通に new できますが、内部クラスはこのルールに従う必要があります。

// NG / OK new Outer.Inner(); // ❌ NG Outer o = new Outer(); Outer.Inner i = o.new Inner(); // ⭕ // staticネステッドなら普通でOK new Outer.Nested(); // ⭕（static付きクラス）
TIP
そもそも独立して使いたいなら最初から static を付けてネステッドにするのがラクです。戻る先：Vol.5・Vol.7。
Q.05

匿名クラスで「';' expected」と出る

// 定番のハマり
A. ANSWER

匿名クラスの閉じ } の後にセミコロンを忘れています。匿名クラスは「new式の文」なので、}; で終わるのが正解です。

// 1文字違いの罠 // ❌ NG（セミコロンなし） Greeting g = new Greeting() { public void hello() { ... } } // ⭕ OK Greeting g = new Greeting() { public void hello() { ... } }; // ← この ; が必要！
TIP
匿名クラスは「クラス定義」ではなく「式」です。だから文末の ; が要る、と覚えましょう。戻る先：Vol.6。
Q.06

フィールドを使うと NullPointerException が頻発します

// 初期化のタイミング
A. ANSWER

「フィールドが初期化される前に使ってしまった」のが定番原因です。Vol.2の実行順序を思い出しましょう。スタ→インスタ→コンスト、の順です。

特に static { } の中でインスタンス変数を触ろうとすると「まだ存在しない」状態になります。

// 順番を意識する class Sample { List<String> logs; // ❌ コンストラクタで初期化忘れ Sample() { } void add(String s) { logs.add(s); // 💥 logs が null } } // ⭕ 宣言と同時に初期化 List<String> logs = new ArrayList<>();
TIP
戻る先：Vol.2「実行順序の整理」。「スタ→インスタ→コンスト」を口に出すだけで、初期化バグはぐっと減ります。
Q.07

enumの比較は == と equals のどっち？

// enumの比較
A. ANSWER

enumは == でOKです。文字列だと == ではダメで equals() が必要ですが、enumは選択肢が1つずつ唯一の存在なので== で安全に比較できるのです。

⭕ enum
sig == Signal.RED
⚠️ String
str == "abc" はNG。equalsを使う
// enumの比較 Signal sig = Signal.RED; if (sig == Signal.RED) { // ⭕ OK System.out.println("赤！"); } // equals() でも動くが、== で十分 if (sig.equals(Signal.RED)) { ... } // ⭕ OKだが冗長
TIP
enumとswitchの組み合わせも超便利。switch(sig)でケースに値だけ書けば、自動でenumの値として扱われます。戻る先：Vol.8 Part1。
Q.08

staticメソッドから普通のフィールドが触れない

// staticの世界の壁
A. ANSWER

これも仕様です。static はクラス全体のもの、普通のフィールドはインスタンスごとのもの。どのインスタンスのフィールドかをstatic側からは決められないので触れないのです。

// staticから普通のフィールドはNG class Sample { int count = 0; // インスタンス変数 static void show() { System.out.println(count); // ❌ } } // 解決策： // ① count に static を付ける、または // ② show() を普通のメソッドにする
TIP
Vol.2の「staticの世界とインスタンスの世界は別物」がここでも効きます。staticブロックの中でも同じ制約があります。
Q.09

recordの値を後から変えたいのですが…

// recordはイミュータブル
A. ANSWER

recordは「中身を変えられない」のが特徴です（イミュータブル）。値を変えたい時は新しいrecordを作り直します。「壊して作る」のが record の流儀です。

// recordは作り直す record Point(int x, int y) {} Point p = new Point(3, 5); p.x = 10; // ❌ 変更不可 // ⭕ 新しい record を作る Point p2 = new Point(10, p.y()); // xだけ変えた新しい Point ができる
TIP
変更が頻繁にあるデータにはrecordではなく普通のクラスを使いましょう。recordは「変わらない値の組」を表すのに向いています。戻る先：Vol.8 Part3。
