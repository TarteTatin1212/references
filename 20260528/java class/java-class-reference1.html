なぜクラスを深掘りするのか

WHY GO DEEPER

基本のクラスだけでもプログラムは書けます。でも現場のコードを読むと、見慣れない書き方が次々出てきます。その「見慣れない書き方」の正体が、この14テーマです。読めるようになれば世界が一気に広がります。

class++ 基本クラスの「次」に何があるか
// イメージ：道具箱が増える
これまで
class / フィールド / メソッド / コンストラクタ / 継承
これから
abstract / interface / 内部クラス / enum / record / annotation …

同じ「クラス」という言葉でも、目的に応じたいろんな種類・書き方がある、というのが今回のテーマです。

// 現場のコードでよく見る光景 public abstract class Animal { ... } // 抽象クラス？ public interface Runnable { ... } // インターフェース？ button.addClickListener(new Handler() { ... // この { } は何者？ }); enum Color { RED, GREEN, BLUE } // enum？ record Point(int x, int y) {} // record？
POINT
今回のゴールは「全部を完璧に使いこなす」ことではありません。まずは読めて、なんとなく意図がわかる状態を目指します。厳密な細部はあとから何度でも辞書として開けばOKです。

塊① 初期化の仕組み

INITIALIZERS

オブジェクトが作られる時、フィールドに最初の値を入れる作業を初期化と言います。コンストラクタはもう知っていますね。実はそれ以外にも初期化を書ける場所があるのです。

{ } と static { } コンストラクタ以外の初期化スポット
// たとえ話

引っ越しを想像してください。

インスタンス初期化
部屋を借りるたびに毎回やる準備（カーテンを付ける等）
スタティック初期化
マンション全体で一度だけやる準備（共用ロビーの設営）

「毎回」か「一度だけ」かが分かれ目です。

// Vol.2 で詳しくやります（予告編） class Sample { // インスタンスイニシャライザ { System.out.println("作るたびに実行"); } // スタティックイニシャライザ static { System.out.println("最初の一度だけ実行"); } }
04

塊② 抽象化の仕組み

ABSTRACTION

プログラムが大きくなると「共通の決まりごと」を作りたくなります。abstractクラスとinterfaceは、その決まりごとを表現する道具です。「中身は後で、形だけ先に決める」のがキーワードです。

abstract / interface 未完成の設計図・できることの約束
// たとえ話
抽象クラス
「動物」という未完成の型。犬や猫が中身を完成させる
インターフェース
「泳げる」という約束。魚も人も守れば名乗れる

どちらも「これを継ぐ／守るなら、このメソッドは必ず用意してね」と強制する仕組みです。

// Vol.3・Vol.4 で詳しくやります（予告編） abstract class Animal { abstract void cry(); // 中身なし＝子に丸投げ } interface Swimmable { void swim(); // 「泳げる」という約束 } class Dog extends Animal implements Swimmable { // cry() と swim() を必ず書く義務が発生 }
05

塊③ ネストされたクラス

NESTED CLASSES

クラスの中に、さらにクラスを書く——それがネスト（入れ子）です。「え、そんなことできるの？」と思いますよね。できます。そして種類が4つあるので、ここが今回いちばんボリュームのある塊です（Vol.5〜7）。

nested classes ×4 クラスの中のクラス、4タイプ
// 4種類の早見
1内部クラス：外側にべったり寄り添うクラス
2ローカルクラス：メソッドの中だけで使うクラス
3匿名クラス：名前すら付けない使い捨てクラス
4staticネステッド：外側と距離を置いた独立寄りのクラス
// 入れ子のイメージ class Outer { // 外側 class Inner { // 中のクラス（内部クラス） } static class Nested { // staticネステッド } void method() { class Local {} // ローカルクラス } }
TIP
ネストされたクラスは「使い分けの理由」が分かりにくいテーマです。Vol.5〜7では「いつ・なぜ使うか」を必ずセットで説明します。丸暗記しなくて大丈夫。
06

塊④ 特殊なクラス

SPECIAL CLASSES

最後の塊は、特定の目的に特化した「便利な専用クラス」たちです。enum・annotation・record。どれも「普通のクラスで書くと面倒なことを、短く安全に書く」ための仕組みです。

enum / annotation / record 目的特化型のクラス
// それぞれの一言まとめ
enum（列挙型）
「選択肢が決まってる値」を安全に表す（曜日・信号の色など）
annotation
コードに付ける「付箋・ラベル」。@Override など
record
「データを入れる箱」を一行で作れる新しいクラス
// Vol.8 で詳しくやります（予告編） enum Signal { RED, YELLOW, GREEN } @Override // これがアノテーション public String toString() { ... } record Point(int x, int y) {} // たった1行でデータクラス完成
