01

ResourceBundleとは

WHAT IS ResourceBundle

ResourceBundleは「言語ごとのプロパティファイルの束（bundle）を、Localeに応じて自動で選ぶ係」です。あなたはgetString("welcome")と書くだけ。どの言語のファイルを読むかはJavaが判断します。

java.util.ResourceBundle 「キーで取り出す」のは同じ、選ぶのは自動
// 同じコード、Localeで結果が変わる
ja_JP→ようこそ
en_US→Welcome
fr_FR→Bienvenue
コードは bundle.getString("welcome") の一行だけ。if文ゼロでこの切り替えが実現します。
イメージ
// Localeを指定してbundleを取得
ResourceBundle b =
  ResourceBundle.getBundle(
      "messages", Locale.JAPAN);

// キーで取り出す（Propertiesと同じ感覚）
String msg = b.getString("welcome");
System.out.println(msg); // ようこそ

// Locale.US に変えれば "Welcome" になる
// → どのファイルを読むかはJavaが選ぶ
// → これがResourceBundleの正体
POINT —
Vol.2のPropertiesは「1つのファイルを自分で読む」。ResourceBundleは「複数の言語ファイルから正しい1つを自動で選ぶ」。Propertiesの進化版と捉えると分かりやすいです。
02

ファイル命名規則

NAMING CONVENTION

ResourceBundleの命とも言えるのが、ファイル名のルールです。基本名_言語_国.properties。この名前の付け方さえ守れば、Javaが正しく言語を選んでくれます。逆にここを間違えると一切動きません。

basename_language_country.properties 基本名は共通、末尾でLocaleを表す
// messages という基本名のファイル群
messages.properties既定（保険）
welcome=Welcome
messages_ja.properties日本語
welcome=ようこそ
messages_en.properties英語
welcome=Welcome
messages_fr.properties仏語
welcome=Bienvenue
命名のルール
// 形：基本名_言語コード_国コード

messages           ← 基本名（共通）
messages_ja        ← 日本語
messages_en        ← 英語
messages_en_US     ← 英語(米国)まで指定
messages_en_GB     ← 英語(英国)

// 重要ポイント
// ・全ファイルで「キーは共通」にする
//   welcome というキーは全言語に存在させる
// ・値だけを各言語に翻訳する
// ・基本名（_なし）は最後の保険になる
TIP —
全ファイルでキーの顔ぶれを必ず揃えるのが鉄則。messages_ja には welcome があるのに messages_en に無い、という状態だと英語環境でエラーになります。「翻訳するのは値だけ、キーは共通」と覚えましょう。
03

getBundleの使い方

USING getBundle

使うのはResourceBundle.getBundle(基本名, Locale)の1行。基本名には言語の部分（_ja など）を含めないのがコツです。あとは取り出すだけ。Propertiesより圧倒的に楽です。

getBundle(baseName, locale) 基本名は "messages"、_ja は付けない
// 実行結果（Localeを切り替え）
JAPAN→ようこそ / 保存
US→Welcome / Save
FRANCE→Bienvenue / Enregistrer
同じ main メソッド。Localeを差し替えるだけで全メッセージが切り替わります。
BundleExample.java
import java.util.Locale;
import java.util.ResourceBundle;

public class BundleExample {
  public static void main(String[] a) {
    print(Locale.JAPAN);
    print(Locale.US);
    print(Locale.FRANCE);
  }

  static void print(Locale loc) {
    // 基本名は "messages"（_ja は書かない！）
    ResourceBundle b =
      ResourceBundle.getBundle("messages", loc);

    String w = b.getString("welcome");
    String s = b.getString("button.save");
    System.out.println(w + " / " + s);
  }
}
POINT —
いちばんの間違いが基本名に _ja を付けてしまうこと。getBundle("messages", ...) が正解で、getBundle("messages_ja", ...) は誤り。「基本名＝共通部分だけ」と肝に銘じてください。
04

探索のしくみ

SEARCH & FALLBACK

「messages_fr_CA（カナダ仏語）が無かったらどうなる？」——Javaは細かい指定から大まかな指定へ、順番に探して見つかったものを使います。この“だんだん大ざっぱに探す”動きを知っておくと安心です。

フォールバック 細かい→大まか→既定 の順に探す
// fr_CA を要求した時の探索順
① messages_fr_CA（カナダ仏語）
↓ 無ければ
② messages_fr（仏語）
↓ 無ければ
③ messages（既定の保険）
↓ それも無ければ
エラー（MissingResourceException）
だから「保険ファイル」が大事
// 探索の考え方
// 「できるだけピッタリの言語を使いたい。
//  無ければ少しずつ妥協していく」

// messages_fr_CA を探す
//   → 無い
// messages_fr を探す
//   → あった！ これを使う

// ポイント：
// 「_なし」の messages.properties を
// 必ず1つ用意しておくと、
// どんなLocaleでも最低限ここに着地できる。
// → 落ちない保険として超重要
TIP —
現場の鉄則は「基本名ファイル（_なし）を必ず英語で用意する」。想定外の言語環境で動いても、英語に着地してアプリが落ちません。保険ファイルは英語、が世界的な定番です。
05

実践：多言語メッセージ

HANDS-ON

ここまでを組み合わせて、「ユーザーのLocaleで挨拶を出すアプリ」を完成させます。ファイルの準備からコードまで、ひと通りの流れを通しで見れば全体像が腑に落ちます。

多言語あいさつアプリ 3ファイル + 1クラスで完成
// 用意するファイル（src/resources）
greet.properties保険=英
hello=Hello
bye=Goodbye
greet_ja.properties日
hello=こんにちは
bye=さようなら
greet_fr.properties仏
hello=Bonjour
bye=Au revoir
GreetApp.java
import java.util.*;

public class GreetApp {
  public static void main(String[] a) {
    // 今のPC設定で挨拶（getDefault）
    greet(Locale.getDefault());

    // 言語を明示して切り替え
    greet(Locale.FRANCE);
    greet(Locale.ENGLISH);
  }

  static void greet(Locale loc) {
    ResourceBundle b =
      ResourceBundle.getBundle("greet", loc);
    System.out.println(
      loc + ": "
      + b.getString("hello") + " / "
      + b.getString("bye"));
  }
}
// 出力例（日本のPCの場合）：
// ja_JP: こんにちは / さようなら
// fr_FR: Bonjour / Au revoir
// en: Hello / Goodbye

これだけ覚えれば多言語対応は卒業です。あとは言語を足したいときに greet_de.properties のようにファイルを増やすだけ。コードは一切変えなくていい——これが国際化を正しく設計した時のご褒美です。

06

ハマりどころ

COMMON PITFALLS

ResourceBundleの定番事故をまとめます。多くはファイル名・置き場所・キーの不一致が原因。先に知っておけば、エラーメッセージを見た瞬間に原因が分かるようになります。

MissingResourceException 対策 この例外を見たら、まずここを疑う
// 原因と対策
エラーの原因
  • 基本名に _ja を付けた
  • ファイルが置き場に無い
  • キーのスペルミス
  • 保険ファイルが無い
確認すること
  • 基本名は共通部分だけか
  • クラスパス上にあるか
  • 全言語でキーを揃えたか
  • _なしを1つ用意したか
安全に取り出すテク
// キーが無いと例外で落ちる。心配なら確認：
if (b.containsKey("welcome")) {
    String w = b.getString("welcome");
}

// 文字化けする場合（古い環境）：
// 今のJavaは .properties を UTF-8 で
// 読むので基本は化けません。
// もし化けたらファイルの保存文字コードを
// UTF-8 にし直すのが第一手。

// どのキーがあるか一覧したい時：
for (String key : b.keySet()) {
    System.out.println(key);
}
