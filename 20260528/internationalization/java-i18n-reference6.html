01

なぜ文章を組み立てるのか

WHY MESSAGEFORMAT

「ようこそ、田中さん」のような文章。"ようこそ、" + name + "さん" と文字列を + でつなぐと、国際化で破綻します。言語によって語順が変わるからです。これを解決するのがMessageFormatです。

文字列連結の限界 言語で語順が違う＝+ でつなぐと詰む
// 言語で語順が変わる例
日本語→田中さん、3件あります
英語→Tanaka has 3 items
名前と数の位置が言語で逆転します。"名前"+"has"+数 のように決め打ちで連結すると、語順の違う言語に対応できません。
なぜ + 連結ではダメか
// ✕ 連結方式
String msg = name + "さん、"
             + count + "件あります";
// 日本語専用。英語版を作ろうとすると
// 語順が違ってコードを書き換えるハメに

// ○ MessageFormat方式
// 文章のひな型を「外」に置き、
// {0} {1} という穴に値を差し込む
//   日本語: {0}さん、{1}件あります
//   英語  : {0} has {1} items
// → ひな型を差し替えるだけで多言語化
POINT —
国際化では「文章を + でつながない」が鉄則。文章はひとつのまとまり（ひな型）として外部ファイルに置き、穴に値を差し込む。語順の違いはひな型側で吸収できます。Vol.3のResourceBundleと相性抜群です。
02

MessageFormatの基本

MessageFormat BASICS

{0} {1} という番号付きの穴（プレースホルダ）を文章に空けておき、そこへ値を順番に差し込みます。使うのはMessageFormat.format(ひな型, 値...)の一発。とてもシンプルです。

MessageFormat.format(pattern, args...) {0} に1個目、{1} に2個目…が入る
// 穴に値が入るイメージ
ひな型：{0}さん、{1}件あります
値："田中"（→{0}）、3（→{1}）
結果：田中さん、3件あります
{0} は1番目の値、{1} は2番目の値。番号で対応づけられます。
MsgExample.java
import java.text.MessageFormat;

String pattern = "{0}さん、{1}件あります";

String msg = MessageFormat.format(
    pattern,
    "田中",  // {0} に入る
    3        // {1} に入る
);
System.out.println(msg);
// → 田中さん、3件あります

// 同じ番号は何度でも使える
MessageFormat.format(
    "{0}、こんにちは {0}！", "佐藤");
// → 佐藤、こんにちは 佐藤！
TIP —
穴は0から始まる番号です。1番目の値が {0}、2番目が {1}。プログラミングではおなじみの「0始まり」。同じ番号を文中で何度でも使えるのも便利なポイントです。
03

型つきプレースホルダ

TYPED PLACEHOLDERS

MessageFormatの真価はここから。穴に「数値」「通貨」「日付」の型を指定すると、差し込みと同時に国ごとのフォーマットまでやってくれます。Vol.4・Vol.5の内容が、文章の中に自動で組み込まれるイメージです。

{0,number}, {1,date}, {2,number,currency} 穴の中で型と書式を指定できる
// 同じひな型、Localeで自動整形
JAPAN→在庫1,234件・￥5,678
US→在庫1,234件・$5,678.00
穴に {n,number} や {n,number,currency} と書くと、桁区切り・通貨記号がLocale任せに。文章の中で国際化が完結します。
TypedExample.java
import java.text.MessageFormat;
import java.util.Locale;

String p = "在庫{0,number}件・{1,number,currency}";

// Locale付きの MessageFormat を作る
MessageFormat mf =
    new MessageFormat(p, Locale.JAPAN);

String msg = mf.format(
    new Object[]{ 1234, 5678 });
System.out.println(msg);
// → 在庫1,234件・￥5,678

// 日付の型もある
// "{0,date,long}" → 2024年3月5日 のように
// 日付をLocaleの形式で差し込める
POINT —
型つき穴を使うときはLocale付きの new MessageFormat(pattern, locale) で作るのがコツ。これで {0,number,currency} がそのLocaleの通貨記号で整形されます。「文章の組み立て」と「数値・日付の国際化」が一度に片付くのが最大の魅力です。
04

ResourceBundleと合体

COMBINING WITH BUNDLE

いよいよ集大成。文章のひな型をResourceBundle（言語別ファイル）に置き、MessageFormatで穴を埋める。これが国際化アプリの王道パターンです。Vol.3とVol.6が合体して、真の多言語メッセージが完成します。

Bundle × MessageFormat ひな型は言語別ファイル、穴埋めはMessageFormat
// 言語別のひな型ファイル
msg_ja: info={0}さん、{1}件あります
msg_en: info={0} has {1} items
↓ 同じコードで…
日本語環境 → 田中さん、3件あります
英語環境 → Tanaka has 3 items
王道パターン
import java.text.MessageFormat;
import java.util.*;

// ① ユーザーのLocaleでBundleを取得
ResourceBundle b =
    ResourceBundle.getBundle("msg", loc);

// ② ひな型をキーで取り出す
String pattern = b.getString("info");

// ③ MessageFormatで穴を埋める
MessageFormat mf =
    new MessageFormat(pattern, loc);
String msg = mf.format(
    new Object[]{ "田中", 3 });

System.out.println(msg);
// Localeに応じて言語も語順も自動で変わる
// → これが国際化の完成形！
TIP —
この「Bundleでひな型を選び、MessageFormatで埋める」三段構えが、実務での国際化メッセージの定番です。これまで学んだ全部（Locale・Properties・Bundle・数値/日付フォーマット）が一本につながる瞬間。ここが本シリーズのゴール地点です。
困った時の国際化Q&A
TROUBLESHOOTING — Q.01 〜 Q.08

全6巻を通して初学者が必ずぶつかる詰まりどころを、Q&A形式で総ざらいします。エラーメッセージや症状から逆引きできるよう、「こうなったらここを見る」視点でまとめました。

Q.01MissingResourceException が出ます
A. ANSWER —

ResourceBundleで最頻出のエラーです。9割は「ファイルが見つからない」か「基本名の間違い」。次の順で確認します。

✕ よくある原因getBundle("messages_ja", loc)
// 基本名に _ja を付けてしまった
○ 正しい形getBundle("messages", loc)
// 基本名は共通部分だけ

それでも出る場合は、ファイルがクラスパス上にあるか（src/main/resources など）、拡張子が .properties か、保険の messages.properties が1つあるかを順に確認しましょう。

TIP —
まず「_なしの基本名ファイル」を英語で1つ用意。これだけで多くの環境で落ちなくなります。
Q.02日本語が文字化けします
A. ANSWER —

原因はファイルの保存文字コードか読み込み時の指定のどちらか。現在のJavaはプロパティを基本UTF-8で読むので、まずファイル自体をUTF-8で保存し直すのが第一手です。

✕ 化ける状況ファイルがShift_JIS保存
// 読み手はUTF-8前提 → ズレる
○ 対策// エディタでUTF-8保存
// Reader版loadで明示も可

古い環境で直接読むときは InputStreamReader に StandardCharsets.UTF_8 を指定して load(reader) すると確実です。

Q.03パーセントが100倍ズレます
A. ANSWER —

getPercentInstance は渡した数を100倍して % を付ける仕様です。25%を出したいなら 25 ではなく 0.25 を渡します。

✕ 2500%になるp.format(25)
// → 2,500%
○ 25%になるp.format(0.25)
// → 25%

「割合は 1.0 が100%」と覚えるのが事故防止のコツ。率は0〜1の世界、と頭を切り替えましょう。

Q.04通貨表示にしたら金額が変わった気がします
A. ANSWER —

これは誤解です。getCurrencyInstance は表示の見た目を変えるだけで、為替の換算はしません。同じ数字に通貨記号を付けているだけです。

✕ 誤解US通貨で表示=ドル換算された
// 数値は変わっていない
○ 正しい理解表示の習慣だけ変わる
// 換算は自分で計算が必要

円1000を「$1,000.00」と表示しても、それは1000という数にドル記号を付けただけ。実際の換算は為替レートを使って別途計算します。

Q.05日付が変な値になります（月のはずが分に）
A. ANSWER —

パターン記号の大文字小文字の取り違えが原因です。月は大文字M、分は小文字m。ここが入れ替わると意図しない値になります。

✕ 月のつもりが分ofPattern("yyyy/mm/dd")
// mm は「分」
○ 正しい月指定ofPattern("yyyy/MM/dd")
// MM が「月」

「月＝大M、分＝小m」。さらに dd（日）と DD（年内通算日）も別物です。迷ったらFormatStyleに任せるのが安全。

Q.06区切り付きの "1,234" を数値にできません
A. ANSWER —

Integer.parseInt("1,234") はカンマを理解できず例外になります。区切り付き文字列は NumberFormat.parse で戻します。

✕ 落ちるInteger.parseInt("1,234")
// NumberFormatException
○ 成功nf.parse("1,234")
// → 1234

表示も入力も同じ NumberFormat で揃えるのが安全。formatで出した形式は、同じLocaleのparseで素直に戻せます。

Q.07英語版を足したらコードまで直すことに…
A. ANSWER —

それは文章を + で連結しているサインです。文字列連結は語順の違う言語に対応できません。MessageFormat ＋ ひな型の外出しに切り替えましょう。

✕ 連結方式name + "さんは" + n + "件"
// 言語ごとにコード修正
○ ひな型方式"{0}さんは{1}件"
// 翻訳はファイル側だけ

正しく設計すれば言語追加でコードは一切変わりません。ファイル（ひな型）を足すだけ。これが国際化の本来の姿です。

Q.08結局、何から手をつければいい？
A. ANSWER —

迷ったら次の順番です。この流れがそのまま国際化対応の手順書になります。

① コード内の日本語を全部キーに置き換える（Vol.2）
② 言語別ファイルを命名規則で用意する（Vol.3）
③ 日付・数値は決め打ちせずLocaleで整形（Vol.4・5）
④ 文章は連結せずMessageFormatのひな型に（Vol.6）
POINT —
一言でまとめると「言葉とデータをコードの外に出し、整形はJavaに任せる」。これさえ守れば、国際化はもう怖くありません。
