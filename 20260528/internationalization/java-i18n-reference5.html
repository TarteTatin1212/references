01

数値の国差を知る

NUMBER DIFFERENCES

まず何が国によって違うのかを押さえます。桁区切り、小数点、通貨記号の位置——この3点が主にズレます。日本とドイツでは小数点と桁区切りが真逆。これを手作業で合わせるのは無謀です。

同じ 1234567.89 の見え方 区切り・小数点・通貨位置が国で変わる
// 国ごとの表記
国数値金額
日本1,234,567.89￥1,234,568
米国1,234,567.89$1,234,567.89
独国1.234.567,891.234.567,89 €
仏国1 234 567,891 234 567,89 €
独：小数点が「,」桁区切りが「.」。仏：桁区切りが空白。通貨記号は前にも後ろにも来ます。
手作業は破綻する
// 自前でやろうとすると…
// 「3桁ごとにカンマ」を書いたつもりが
// ドイツでは「.」、フランスでは空白…
// 小数点も国で入れ替わる。地獄。

// しかも通貨記号は
//   ￥1,234（前）
//   1.234 €（後ろ）
// と位置まで変わる。

// → 自分でやらない。NumberFormatに任せる。
//   これがこの巻の結論です。
POINT —
数値の国際化で絶対にやってはいけないのが「カンマを自分で入れる」こと。String.replace で区切りを足すようなコードは、別の国で必ず破綻します。整形はNumberFormatに丸投げが鉄則です。
02

NumberFormatの基本

NumberFormat BASICS

数値整形の主役NumberFormatです。使い方は日付と同じ発想。「Localeを指定して整形係を取得 → format で文字列化」。getNumberInstance(Locale)で桁区切り付きの数値整形係が手に入ります。

NumberFormat.getNumberInstance(locale) 桁区切りや小数点をLocaleで自動調整
// 1234567.89 を各国で整形
JAPAN→1,234,567.89
US→1,234,567.89
GERMANY→1.234.567,89
FRANCE→1 234 567,89
同じ数。Localeを変えるだけで区切りと小数点が国仕様に。
NumberExample.java
import java.text.NumberFormat;
import java.util.Locale;

double n = 1234567.89;

// Localeを指定して整形係を取得
NumberFormat jp =
    NumberFormat.getNumberInstance(Locale.JAPAN);
System.out.println(jp.format(n));
// → 1,234,567.89

NumberFormat de =
    NumberFormat.getNumberInstance(Locale.GERMANY);
System.out.println(de.format(n));
// → 1.234.567,89

// 小数桁数も調整できる
jp.setMaximumFractionDigits(1);
System.out.println(jp.format(n)); // 1,234,567.9
TIP —
setMaximumFractionDigits（最大小数桁）やsetMinimumFractionDigits（最小小数桁）で小数の見せ方を細かく調整できます。「常に小数2桁で揃えたい」なら最小・最大とも2に設定。地味ですが現場で頻出のテクです。
03

通貨フォーマット

CURRENCY FORMAT

お金の表示はgetCurrencyInstance(Locale)。通貨記号・記号の位置・小数桁を、その国のお金の習慣に合わせて全部やってくれます。円なら小数なし、ドルなら小数2桁、と通貨ごとの違いも自動です。

NumberFormat.getCurrencyInstance(locale) 記号も桁数も位置も、その国の通貨仕様で
// 1234567.89 を通貨表示
JAPAN→￥1,234,568
US→$1,234,567.89
GERMANY→1.234.567,89 €
円は小数なしで四捨五入、ドルは小数2桁、ユーロは記号が後ろ。通貨ごとの常識をJavaが知っています。
CurrencyExample.java
import java.text.NumberFormat;
import java.util.Locale;

double price = 1234567.89;

NumberFormat jp =
    NumberFormat.getCurrencyInstance(Locale.JAPAN);
System.out.println(jp.format(price));
// → ￥1,234,568（円は小数なし）

NumberFormat us =
    NumberFormat.getCurrencyInstance(Locale.US);
System.out.println(us.format(price));
// → $1,234,567.89

// 注意：Localeが決めるのは「表示の習慣」。
// 数値を実際に円→ドル換算するわけではない！
// 為替変換は別途自分で行う。
POINT —
超重要な勘違いポイント。getCurrencyInstance は「表示の仕方」を変えるだけで、為替の換算はしません。1234567.89 がドル表示で $1,234,567.89 になっても、それは「同じ数字にドル記号を付けた」だけ。円→ドルの換算は自分で計算が必要です。
04

パーセント表示

PERCENT FORMAT

割合の表示はgetPercentInstance(Locale)。ここにも初学者が必ず引っかかる罠があります。「0.25 を渡すと 25% になる」——つまり100倍してから%を付ける動きをします。最初は必ず驚きます。

NumberFormat.getPercentInstance(locale) 0.25 → 25%。渡す数は「割合（1.0=100%）」
// 入力 → 出力の対応
渡す値出力
0.2525%
0.550%
1.0100%
0.0758%（四捨五入）
「25%にしたいなら 25 ではなく 0.25 を渡す」。ここを取り違えると 2500% になります。
PercentExample.java
NumberFormat p =
    NumberFormat.getPercentInstance(Locale.JAPAN);

// 0.25 を渡すと 25% になる（100倍される）
System.out.println(p.format(0.25)); // 25%
System.out.println(p.format(1.0));  // 100%

// 小数の%も出したい場合
p.setMaximumFractionDigits(1);
System.out.println(p.format(0.075)); // 7.5%

// よくある事故：
// p.format(25) → 2500% になってしまう
// 「割合は 0〜1 で渡す」と覚える
TIP —
覚え方は「割合は 1.0 が100%」。テストの点数80点なら 80 ではなく 0.8 を渡す。"率"は0〜1の世界と頭を切り替えれば、定番事故「桁が100倍ズレる」を回避できます。
05

文字列を数値に戻す

PARSING NUMBERS

フォームに入力された「1,234,567」のような区切り付き文字列を数値に戻すのもNumberFormatの仕事。parseを使います。Integer.parseIntはカンマ付き文字列で失敗するので、ここでもNumberFormatが頼りになります。

numberFormat.parse(text) 区切り付き文字列 → 数値。Localeの区切りを理解
// なぜ parseInt ではダメか
Integer.parseInt
  • "1,234" で失敗
  • カンマを理解しない
  • 例外で落ちる
NumberFormat.parse
  • "1,234" → 1234
  • 国の区切りを理解
  • 独語の "1.234" も処理
ParseNumberExample.java
import java.text.NumberFormat;
import java.util.Locale;

NumberFormat jp =
    NumberFormat.getNumberInstance(Locale.JAPAN);

// 区切り付き文字列を数値に
Number n = jp.parse("1,234,567");
System.out.println(n); // 1234567

// ドイツ語の "1.234,5" もLocaleが合えばOK
NumberFormat de =
    NumberFormat.getNumberInstance(Locale.GERMANY);
Number g = de.parse("1.234,5");
System.out.println(g); // 1234.5

// parse は ParseException を投げ得るので
// try-catch（または throws）が必要
POINT —
区切り付きの数字をそのまま parseInt に渡してはいけません。"1,234" は NumberFormatException で落ちます。表示はNumberFormatでformat、入力はNumberFormatでparse。入口と出口を同じ道具で揃えるのが安全です。
06

ハマりどころ

COMMON PITFALLS

数値フォーマットの定番事故をまとめます。「%の100倍」「通貨=為替の誤解」「自前カンマ」が御三家。日付編と同じく、「自分で整形しない・Localeに任せる」が一貫した解決策です。

あるある事故と対策 困ったら「整形係に任せたか？」を疑う
// 詰まりポイント
やりがちな失敗
  • %で桁が100倍ズレる
  • 通貨表示で換算した気に
  • カンマを自分で挿入
  • 区切り付きをparseInt
対策
  • 率は0〜1で渡す
  • 換算は別途自前で
  • 整形はNumberFormat
  • 入力もNumberFormat.parse
3つの整形係まとめ
// この巻の3兄弟を整理：

// ① ただの数値（桁区切り）
NumberFormat.getNumberInstance(loc);

// ② 通貨（記号・桁数つき）
NumberFormat.getCurrencyInstance(loc);

// ③ パーセント（100倍して%）
NumberFormat.getPercentInstance(loc);

// どれも使い方は同じ：
//   Localeを渡す → format で表示
//   → parse で文字列を数値に戻す
// 「get○○Instance(Locale)」の形を覚える
