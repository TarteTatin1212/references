02

なぜテストを書くのか

WHY TEST

テストは「壊れていないことの証明書」です。特にこの後リファクタリングを控えているので、テストがあれば「内部を変えても動きは同じ」を機械的に確認できます。

value-of-test テストが何を守るか
// テストが無いとどうなる

コードを直すたびに、全画面を手でポチポチ確認することになります。一覧を直したら詳細も新規も編集も削除も、毎回全部。

しかも「直したつもりが別の場所を壊していた」に気づけない。これがバグの温床です。

POINT ── テストは未来の自分への保険

テストを一度書いておけば、コマンド1つで全機能を一瞬で確認できます。「直した→テスト緑→安心して次へ」。特にリファクタリングでは「動きを変えずに内部だけ改善」を保証する命綱になります。

// テストがある開発の流れ // ① コードを直す // ② mvn test を実行 // ③ 全部緑（PASS）→ 壊していない！安心 // 赤（FAIL）→ どこを壊したか即わかる // ─── 手動確認 vs テスト ─── 手動: 一覧開く→詳細開く→編集する→ 保存する→削除する→…（毎回全部） 時間かかる・見落とす・面倒 テスト: mvn test （数秒で全部チェック） 速い・見落とさない・何度でも // ─── 特にリファクタリングで効く ─── // 「フィールドインジェクションを // コンストラクタ注入に変えたい」 // → 変えてテスト緑なら、動きは同じと証明 // → 安心して内部を綺麗にできる
03

テストピラミッド

TEST PYRAMID

テストには種類があり、「どれをどれだけ書くか」には定石があります。それを表したのがテストピラミッド。下ほど数多く・速く、上ほど少なく・本物に近い。

test-pyramid どれをどれだけ書くか
// テストピラミッド
E2E（少）
結合テスト
単体テスト（多・速い）
下ほど数が多く・速い／上ほど少なく・本物に近い
TIP ── なぜピラミッド型？

上の層（本物に近いテスト）は遅くて壊れやすい。アプリ全部を起動するので時間がかかり、ちょっとした変更でも失敗しがち。だから速くて安定した単体テストを土台に厚く、本物に近いテストは要所だけ薄く——が効率的なバランスです。

// 3種類のテスト // ── 単体テスト（Unit）── // 部品1つを単独で試す。 // 例：Serviceのロジックだけ（DBなし） // 速い・たくさん書ける・安定 // ── 結合テスト（Integration）── // 複数の部品を繋いで試す。 // 例：MapperとDB、ControllerとWeb層 // やや遅い・本物に近い // ── E2E（End to End）── // アプリ全体を起動して通しで試す。 // 例：ブラウザ操作を自動化 // 遅い・本物そのもの・壊れやすい // このシリーズで書くのは主に // 単体（Service/Form）と // 結合（Mapper/Controller）。 // E2Eは扱わない（小規模アプリには過剰）

何をテストするか（3層）

WHAT TO TEST

このメモ帳アプリは3層構造です。それぞれ役割が違うので、確かめ方も変わります。まず各層が何を担当しているか整理しましょう。

3-layers 層ごとに手法が変わる
// 3層と確かめ方
Controller
URLを受け、画面を返す
→ @WebMvcTest + MockMvc
↓ 呼ぶ
Service
業務ロジック・変換
→ Mockito 単体テスト
↓ 呼ぶ
Mapper
DBアクセス（SQL）
→ @MybatisTest（DB）
POINT ── 層ごとに「狙い」が違う

Controllerは「正しいURLで正しい画面が返るか」、Serviceは「ロジックが正しいか」、Mapperは「SQLが意図通り動くか」。狙いが違うから手法も違う。それぞれに最適な道具を使い分けます。

// このアプリのテスト対象 jp.example/ ├ controller/ │ ├ NoteController // Web層 │ │ list/view/create/ │ │ editForm/update/delete │ └ GlobalExceptionHandler // 例外集約 ├ service/ │ └ NoteService // 業務ロジック │ findAll/register/ │ findById/update/delete ├ mapper/ │ └ NoteMapper // DBアクセス │ findAll/insert/ │ findById/update/delete └ form/ ├ NoteForm // 入力(必須・30字) └ NoteEditForm // ─── 確かめたいことの例 ─── // ・正しいURLで正しい画面が返るか // ・FormからEntityへ正しく変換するか // ・存在しないidで例外が出るか // ・バリデーションが効くか // ・SQLが意図通り動くか

置き場所と命名

PLACE & NAMING

テストコードは本体とは別のフォルダに置きます。場所と名前のルールを最初に押さえると、後でテストが増えても迷子になりません。

src/test/java テストの置き場所
// 置き場所のルール

本体は src/main/java、テストは src/test/java。パッケージ構成は本体と同じにします。

NoteService（本体）のテストは NoteServiceTest。「対象クラス名＋Test」が定番の命名です。

POINT ── 本体と同じパッケージに置く

テストを本体と同じパッケージ名にすると、本体のpackage-private（公開していない）なメソッドにもアクセスできて便利。src/test/java 配下に jp.example.service を作り、そこに NoteServiceTest を置きます。

// フォルダ構成 src/ ├ main/java/jp/example/ │ ├ controller/NoteController.java │ ├ service/NoteService.java │ └ mapper/NoteMapper.java │ └ test/java/jp/example/ // ★テスト ├ controller/ │ └ NoteControllerTest.java ├ service/ │ └ NoteServiceTest.java ├ mapper/ │ └ NoteMapperTest.java └ form/ └ NoteFormTest.java // ─── 命名のルール ─── // クラス：対象クラス名 + Test // NoteService → NoteServiceTest // メソッド：何を保証するかを日本語で // findById_存在すればNoteを返す() // findById_無ければ例外を投げる() // → テスト名がそのまま仕様書になる

TIP ── @Test と assertThat

@Test を付けたメソッドが1本のテスト。中で assertThat(実際).isEqualTo(期待) と書き、実際の値が期待通りかを確かめます。一致すれば緑、違えば赤。これがテストの最小単位です。

// src/test/java/jp/example/SmokeTest.java package jp.example; import static org.assertj.core.api.Assertions.*; import org.junit.jupiter.api.Test; class SmokeTest { @Test void 一たす一は二になる() { // 実行 int result = 1 + 1; // 検証：result は 2 のはず assertThat(result).isEqualTo(2); } } // ─── 実行方法 ─── // Eclipse：クラスを右クリック // → Run As → JUnit Test // コマンド：mvn test // ─── 緑を確認したら ─── // このSmokeTestは役目を終えたので // 消してOK（動作確認用の捨てテスト）。 // 「テストが走る環境」が整った証拠。

現状コードを対象にする理由

PREMISE

このシリーズはリファクタリング前の「今のコード」をそのままテストします。あえて綺麗にする前に固める。その理由を押さえておきましょう。

test-then-refactor 先にテスト、後でリファクタ
// なぜ現状コードを固めるか

今のコードには「直したい点」がいくつかあります（フィールドインジェクション、例外ハンドラが200のまま等）。でもいきなり直すのは危険です。

先に「今の動き」をテストで記録しておけば、リファクタリングで動きが変わってしまったらテストが赤くなって教えてくれる。これが安全網になります。

POINT ── テストは「現状の写真」

テストは今の振る舞いを写真に撮るようなもの。リファクタリング後に同じ写真（テスト）を見て、ズレていなければ「動きは変えていない」と分かる。だから綺麗にする前に撮っておくのが鉄則です。

// 現状コードの「直したい点」（伏線） // このシリーズでは直さず、テストで固定。 // 次の「リファクタリング編」で直す。 // ① フィールドインジェクション @Autowired private NoteService noteService; // → コンストラクタ注入に変えたい // ② 例外ハンドラがステータス200のまま // → @ResponseStatus で本物の404に // ③ NoteForm と NoteEditForm が似ている // → 共通化の余地 // ④ Controllerの new NoteForm() // → @ModelAttribute で消せる // ─── 順序が大事 ─── // 1. 今の動きをテストで固定（このシリーズ） // 2. テストに守られながらリファクタ（次） // 3. テスト緑のまま＝動きは不変と証明
