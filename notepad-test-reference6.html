POINT ── 2つの実行方法

コマンドなら mvn test。Eclipseならプロジェクトを右クリック → Run As → JUnit Test、または src/test を右クリックで一括実行。どちらでも全テストが一度に走り、結果がまとめて出ます。日常的には頻繁に回します。

結果の読み方

READ RESULTS

テストが赤くなったときこそ本番。慌てず、どのテストが・何を期待して・実際は何だったかを読みます。エラーメッセージは原因を教えてくれます。

失敗の読み方 expected vs actual
// 失敗メッセージの構造

AssertJの失敗は「期待（expected）」と「実際（actual）」を並べて教えてくれます。

「期待は "買い物" なのに、実際は null だった」のように。この2つの差が、バグの正体への最短ルートです。

POINT ── まず「どこで」「何が違うか」

赤いテストを見たら、①どのテストメソッドが ②expectedとactualは何かを読む。前にフロントで学んだログの読み方と同じで、長い出力に圧倒されず核心の数行を見ます。テストが失敗を具体的に指し示してくれるのが、手動確認との決定的な違いです。

// 失敗例の読み方 // 赤くなったテスト： [ERROR] NoteServiceTest.findById_存在すれば返す // AssertJ のメッセージ： org.opentest4j.AssertionFailedError: expected: "買い物" but was: null // 読み解き： // 「買い物」が返るはずが、nullだった // → モックの仕込み忘れ？ // → getTitleの実装ミス？ // 原因を絞り込める // ─── 失敗の3パターン ─── // ① 値が違う（expected ≠ actual） // → ロジックかテストの期待値を見直す // ② 例外が出た（想定外のException） // → スタックトレースの一番上を読む // ③ 起動失敗（Bean不足など） // → @MockitoBean忘れ等の設定ミス

テストは仕様書になる

TEST AS SPEC

テストにはもう一つの顔があります。「このアプリは何ができるのか」を語る仕様書です。テスト名を読むだけで、アプリの振る舞いが分かる——そんな書き方を心がけます。

テスト名＝仕様 読めば仕様が分かる
// テスト名を並べると…

findById_存在すればNoteを返す
findById_無ければ例外を投げる
title_空はNG
title_30文字はOK
title_31文字はNG
登録_成功で一覧へリダイレクト

POINT ── 名前がドキュメントになる

テスト名を日本語で「何を保証するか」に書くと、一覧がそのまま仕様書になります。「このアプリはtitleが空だと登録できない」——コードを読まなくても、テスト名から分かる。コメントは古くなるが、テストは古くなれば赤くなる。だから常に正しい仕様書です。

// 良いテスト名 vs 悪いテスト名 // ✗ 悪い：何を確かめるか不明 @Test void test1() { ... } @Test void testFindById() { ... } // ✓ 良い：仕様が読み取れる @Test void findById_存在すればNoteを返す() @Test void findById_無ければ例外を投げる() // ─── @DisplayName も使える ─── @Test @DisplayName("タイトルが31文字なら登録できない") void title_31文字はNG() { ... } // → テスト結果に日本語の説明が出る // → 非エンジニアにも伝わる仕様書に // テストは「動く仕様書」。 // 仕様が変わればテストが赤くなるので、 // 仕様書とコードが食い違わない。
04

どこまでテストするか

HOW MUCH

テストは多いほど良い——わけではありません。費用対効果を考えます。「壊れたら困るところ」「間違えやすいところ」を厚く、当たり前すぎる部分は薄く。

cost vs value テストの優先順位
// 優先度の考え方
対象優先
分岐・条件（例外/バリデーション）高
変換ロジック（Form→Entity）高
単純な委譲（findAll等）中
getter/setter（Lombok生成）低（不要）
POINT ── 「間違えやすい所」を狙う

@Data が生成する getter/setter をテストしても意味は薄い（ライブラリが保証済み）。逆に条件分岐・境界値・変換・例外は人が間違えやすく、壊れたら影響が大きい。限られた時間を、価値の高いテストに使うのが賢いやり方です。100%を目指す必要はありません。

// テストすべき / しなくていい // ─── 厚くテストすべき ─── // ・if分岐（findById の null判定） // ・境界値（@Size(max=30) の30/31） // ・変換（NoteForm → Note のコピー） // ・例外（NoteNotFoundException） // → 人が間違えやすく、影響大 // ─── 薄くてよい / 不要 ─── // ・getId() / setId()（Lombok生成） // ・フレームワークの機能そのもの // （Springが@GetMappingを正しく // 処理するか、等はテスト不要） // ─── カバレッジの罠 ─── // 「カバレッジ100%」が目的化すると、 // 無意味なテストが増える。 // 数字より「重要な所を守れているか」。

リファクタへの橋渡し

TO REFACTORING

テストが揃った今、ついに安全にリファクタリングができます。第1巻から伏線として置いてきた「直したい点」に、テストという守りを背負って挑めます。

test → refactor 次フェーズへ
// これからの流れ
今ここ：テストで現状を固定（完了）
↓
次：テストに守られながらリファクタ
↓
テスト緑のまま＝動きは不変と証明
POINT ── 赤くなったら立ち止まる

リファクタ中にテストが赤くなったら、それは「動きを変えてしまった」サイン。意図通りならテストを書き換え（例：ステータス200→404）、意図しないならコードを戻す。テストが「やりすぎ」を検知してくれます。

// 第1巻で置いた伏線＝リファクタ候補 // ① フィールドインジェクション @Autowired private NoteMapper ...; // → @RequiredArgsConstructor + final // （コンストラクタ注入）に変える // テストはそのまま緑のはず // ② 例外ハンドラがステータス200 // → @ResponseStatus(NOT_FOUND) を足す // Vol.4/5のテストを 200→404 に書き換え // ③ insert に採番キー取得が無い // → @Options(useGeneratedKeys...) を足す // Vol.3のテストを id 取得方式に改善できる // ④ NoteForm / NoteEditForm の重複 // → 共通化を検討（継承 or 統合） // どれも「テストが緑か」で // 安全性を確認しながら進められる。

これからの習慣

HABITS

テストは一度書いて終わりではなく、開発に組み込む習慣です。最後に、これから先ずっと役立つ3つの習慣を渡します。

good habits 続けるコツ
// 3つの習慣
  • こまめに実行：コードを変えたら mvn test。赤くなったらすぐ気づける
  • バグを直す前にテスト：バグを再現するテストを先に書く。直ったら緑になる＝再発防止
  • 機能追加にはテストも：新機能を足すとき、そのテストも一緒に。守りを増やしながら進む
POINT ── 「バグ修正はテストから」

バグを見つけたら、まずそのバグを再現する（＝失敗する）テストを書く。それからコードを直し、テストが緑になれば修正完了。さらにそのテストが残るので、同じバグの再発を永久に防げます。これはプロが必ずやる習慣です。

// バグ修正の正しい順序 // ① バグを発見 // 「title=null のメモで一覧が落ちる」 // ② 先に「落ちることを示すテスト」を書く @Test void title_nullでも一覧は落ちない() { // この時点では赤（バグがあるから） } // ③ コードを直す // ④ テストが緑になる＝修正完了 // ⑤ テストが残る＝再発防止 // 将来また同じバグを入れたら // このテストが赤くなって教えてくれる // テストは「書いて終わり」ではなく // 開発のリズムに組み込むもの。
