01

順序という問題

QUESTION OF ORDER

TDDが特殊なのは、新しい技術を導入したからではなく「やる順番を逆にした」だけであることです。同じ部品（実装とテスト）を使って、置く順序を変える。それだけで結果が劇的に変わる現象を、ここから読み解きます。

order matters 同じ要素でも順序が違えば結果が違う
// 順序の対比
CONVENTIONAL — 従来
実装
↓
テスト
↓
動いた / 失敗
VS
TDD — テスト先行
テスト
↓
実装
↓
動いた / 失敗

要素は同じ。順序だけが違う。この微小な変更がもたらす大きな違いを、以降のセクションで分解していきます。

// 順序の違いが // 何を変えるのか 考え始める起点が変わる ・従来: 実装方法から考える ・TDD : 期待する振る舞いから考える 完了の合図が変わる ・従来: 動いた気がする ・TDD : テストが通った テストの存在意義が変わる ・従来: 動作チェックの道具 ・TDD : 設計を導く仕様書 // 同じ作業の積み重ねでも // 出来上がるソフトウェアは別物
POINT
順序が違えば、「何を考えながらコードを書くか」が変わります。考える順序が変わると、設計の選び方が変わる。設計が変わると、最終的な出来上がりが変わる。これがTDDの「順序の魔法」の正体です。
02

「後で書く」が招くもの

CONSEQUENCES OF LATE TESTS

実装が終わってからテストを書こうとすると、時間の経過とともに「書きにくくなる」「書かなくなる」力学が働きます。これは怠惰の問題ではなく、構造的に起きる現象です。順を追って見ていきましょう。

test-last problems 時間軸で何が起きるか
// 「後で書く」を選んだ場合の時間軸
0:00
実装を書き始める
0:30
動いた、気持ちが良い
0:45
テストを書こうと思うが面倒
1:00
別タスクの依頼が来る
翌日
コードの細部を忘れる
数日後
テストを書こうとして書きにくいことに気付く
数週間
結局書かないまま次へ
// なぜ書きにくくなるのか ① 実装を先に作ると テストしやすい構造にならない 例: 一つの関数に多くの責任 → 入力が複雑 → モックすべき箇所が多い → テストが書けない ② 動いている安心感が テストの動機を弱める ③ 後付けテストは 「実装に合わせて書く」ことに → 本来の仕様の検証ではなく → 実装の追認になる // この3つが連動して // テスト不在のコードが量産される
注意
「テストは書こうと思っているがいつも後回しになる」のは個人の性格や意志の問題ではなく、順序が「実装→テスト」になっている時点で発生する構造的な現象です。順序を変えない限り、意志力でこの流れに抗うのは困難です。
03

「先に書く」が変えるもの

SHIFTS BY TEST-FIRST

テストを先に書くと、対比的に何が変わるのか。Vol.3で見た現象が「自然と」発生するようになります。意志力ではなく、順序がそれを引き出します。

test-first shifts 逆にすると何が逆になるか
// 4つの軸での比較
観点
変化
テスト網羅
低い → 高い
テスト容易性
後付け困難 → 設計に組込み
仕様の明確化
頭の中 → コードとして残る
動かない時間
長い → 短い

どれも順序を変えただけで自然に発生する変化です。チームに意志や規律を要求するのではなく、構造で実現する。

// 順序を変えると // 自然に起こること A. 必ずテストが書かれる → 実装の前提条件だから B. テストしやすい設計になる → テストを書ける形を選ぶから C. 仕様が先に明確になる → 何を作るかを定義する作業 そのものになるから D. 動く状態がほぼ常に保たれる → 小さく回す前提だから E. リファクタリングに踏み込める → 安全網があるから // これらは「副作用」だが // TDDの最大の価値かもしれない
POINT
Vol.3の本編で「TDDは設計を導く」と表現したのは、この構造的な変化のことです。設計を導いてくれるのは、TDDという技法そのものではなく「順序の逆転」。これが分かると、TDDという言葉の中身が見えてきます。
04

テストファーストの因果連鎖

CHAIN OF EFFECTS

なぜ「順序を変えるだけ」で多くのことが連鎖的に変わるのか。それは1つの変化が次の変化を引き起こし、それがまた次を引き起こすからです。連鎖の各段階を順番に辿ってみましょう。

causal chain テストファーストが引き起こす連鎖反応
// 因果の連鎖
  1. テストを先に書くと決める
  2. テストが書けない実装は選べなくなる
  3. 必然的にテストしやすい設計を選ぶ
  4. テストしやすい設計＝責務が分離された設計
  5. 責務が分離されたコードは変更しやすい
  6. 変更しやすいコードは改善を積み重ねやすい
  7. 結果として長期に強いソフトウェアになる
// 連鎖の出発点と終着点 出発点: テストを先に書く という意思決定 ↓ 連鎖 短期: ・実装にテストが付く ・動かない時間が減る 中期: ・設計の質が上がる ・責務分離が進む ・依存関係が整理される 長期: ・リファクタリングできる ・変更を恐れなくなる ・レガシー化を遅らせる // 「テストを書く理由」が // 検証から設計、設計から保守へ // と意味を広げていく
この連鎖の意味
TDDの提唱者ケント・ベックが「TDDの恩恵で一番大きいのは設計への影響」と語ったのは、この連鎖を念頭においたものです。テストはきっかけにすぎず、本当の価値はその先にある設計の質に現れます。
05

リファクタリングとの関係

RELATION TO REFACTORING

リファクタリング（振る舞いを変えずにコードを整える作業）は、テストの存在を前提とする活動です。テストがあるかどうかで、コード変更の意味そのものが変わります。

refactoring & tests テストの有無が変更の意味を変える
// 左右の対比
テストなし
コードを書き換える
↓
動くかどうか祈る
↓
壊れていないか不明
VS
テストあり
コードを書き換える
↓
テストを走らせる
↓
壊れていないと確信

コード変更の不確実さを、テストが確実さに変換します。これが「変更可能性」の源泉です。

// 時間軸で見る変更コスト テストがない世界: 1ヶ月後の変更 → 怖い 半年後の変更 → 触りたくない 1年後の変更 → 触れない ↓ レガシーコード化 テストがある世界: 1ヶ月後の変更 → 普通 半年後の変更 → 普通 1年後の変更 → 普通 ↓ 継続的に改善可能 // テストの本当の価値は // 「将来の自分」を助けること
用語の整理
リファクタリング＝外から見た振る舞いを変えずに、内部構造を整える作業。Martin Fowlerの著書『Refactoring』（1999）で広まった概念。
レガシーコード＝古いコードのこと。Michael Feathersの定義では「テストがないコード」を指す。テストがあれば古くてもレガシーではない、という見方。

設計への副次効果

SIDE-EFFECT ON DESIGN

TDDの「設計への影響」はやや抽象的なので、具体的に何が起きるかを見ましょう。テストを先に書くと、「使う側の視点」が先に来るため、自然と扱いやすいAPIが生まれます。

合わせる

内部はそれに合わせて

// 同じ機能 / 違う設計 // === 実装ファーストの典型 === calc.setBasePrice(1000); calc.setDiscountRate(0.1); calc.setTaxRate(0.1); calc.setCurrency("JPY"); calc.calculate(); calc.getResult(); // → 内部状態が外に漏れた設計 // → 順序を間違えると壊れる // === テストファーストの典型 === price({ base: 1000, discount: 0.1, tax: 0.1, }); // → 一度に必要な情報を渡す // → 順序の概念がない // → 使う側が幸せ
POINT
テストコードは「最初の使用者」になります。書きづらいテストは、使いづらいコードの兆候。テストファーストはこの違和感を「実装する前に」察知できる仕組みなのです。

TDD・テストファースト・BDD

TDD vs TEST-FIRST vs BDD

「TDD」「テストファースト」「BDD」は似ているようで指す範囲が異なる用語です。混乱しやすいので、ここで整理しておきます。本編Vol.3で出会った用語の位置関係を地図化します。

terms map 3つの用語の関係
// 3つの違い
用語
焦点
Test-First
テストを先に書く、ただそれだけ
TDD
Test-First + Red-Green-Refactorサイクル
BDD
TDDの拡張、振る舞いを自然文で書く

BDD（Behavior-Driven Development、振る舞い駆動開発）はTDDから派生した手法。技術者でない人にも仕様が読める形を志向します。

// 包含関係 ┌────────────────────┐ │ Test-First │ │ 「先にテストを書く」 │ │ │ │ ┌──────────────┐ │ │ │ TDD │ │ │ │ Test-First + │ │ │ │ RGRサイクル │ │ │ │ │ │ │ │ ┌─────────┐ │ │ │ │ │ BDD │ │ │ │ │ │ 自然文化 │ │ │ │ │ └─────────┘ │ │ │ └──────────────┘ │ └────────────────────┘ // 大きい円 ⊃ 中の円 ⊃ 小さい円 // 一番外側がゆるい概念 // 一番内側が具体的な手法
用語の整理
Test-First（テストファースト）＝テストを実装より先に書くというルール。最も広い概念。
TDD＝Test-FirstにRed-Green-Refactorサイクルを加えたもの。Kent Beck (1999)。
BDD（Behavior-Driven Development）＝TDDをさらに発展させ、「Given-When-Then」など自然言語に近い記法で振る舞いを記述するスタイル。Dan North (2006)。
