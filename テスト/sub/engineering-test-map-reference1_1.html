01

テストのなかった時代

BEFORE 1970s

1950〜60年代、ソフトウェアはごく小さなプログラムで、書いた本人がその場で動かして確認していました。「テストする」という独立した行為や仕事はまだ存在しません。動かしてみて変な結果が出たら、その場で直す。これが原始的な姿でした。

prehistory テストが行為として独立する前の時代
// 時間軸 — 出発点
1950s プログラム黎明期 紙テープ・パンチカードで命令を入力。1つのプログラムは数十〜数百行
1960s 「バグ」という言葉が定着 プログラムの欠陥を見つけて直すこと=デバッグ。テスト=デバッグだった
〜1970 書いた人が動かして確認 プログラムは小さく、作者が全体を把握できた
// この時代の「テスト」 1. プログラムを書く ↓ 2. 動かしてみる ↓ 3. 変な結果が出る ↓ 4. 原因を探して直す ← デバッグ ↓ 5. もう一度動かす // テストとデバッグの // 区別はなかった
この時代の用語
バグ（bug）＝プログラムの誤り。語源は1947年、初期のコンピュータに飛び込んだ蛾（虫=bug）が原因で動作不良が起きた逸話から。デバッグ（debug）＝バグを取り除く作業。「動かない原因を探して直す」こと。
02

ソフトウェアが大きくなった

SOFTWARE GROWS

1970年代、コンピュータの普及と共にソフトウェアは急速に大規模化しました。一人では全体を把握できない規模の開発が当たり前になり、書いた本人が動かして確かめるだけでは品質を保てなくなります。ここで「品質」を別の人・別の工程で守る必要が出てきました。

software crisis 規模拡大 → 品質を守る別の仕組みが必要に
// 規模の変化
1960s 数百〜数千行 一人で全体を理解できる
1970s 数万〜数十万行 複数人で分担。誰も全体を知らない
1980s 数百万行 大企業の基幹システム。失敗が会社存続に直結

ソフトウェア危機と呼ばれた時代。プロジェクトが頻繁に遅延・予算超過・品質崩壊を起こした

// 変化の構図 規模小 規模大 ───── ───── 作者 把握できる 把握できない 責任 一人 分散 バグ すぐ気づく 見つからない 品質 自然に保てる 仕組みが必要 ↓ 求められたもの: ・規律ある開発工程 ・品質を担保する仕組み ・専門の「検証する人」
POINT
テストという仕事は、規模が個人の理解を超えた時に必要になったのです。小さなプログラムなら不要だったものが、大きくなった瞬間に不可欠になる。この因果が出発点です。
03

工程の最後にテストする時代

WATERFALL ERA

1980〜90年代、大規模開発を進めるための方法として「ウォーターフォール開発」が定着します。要件定義 → 設計 → 実装 → テスト → リリースと工程を一方向に流す進め方です。「テスト」が独立した工程として明確に区切られ、専門の「テスター」という職業も生まれました。

waterfall テストが独立した工程として確立した時代
// ウォーターフォールの流れ
要件定義
▶
設計
▶
実装
▶
テスト
▶
リリース

滝（waterfall）が上から下に流れるように、前の工程は完了させてから次へ進む。テストは実装後にまとめて行う独立工程として位置づけられた。

プログラマ
テスター
機能を実装する
実装されたものを検証する
// この時代の問題点 実装が終わってからテストを始める ↓ 重大なバグは数ヶ月後に発覚 ↓ 原因が設計レベルだと 最初からやり直し ↓ プロジェクト遅延・予算超過 // 用語 ウォーターフォール開発: 工程を一方向に流す進め方 テスト工程: 実装の後に置かれる検証フェーズ テスター: 検証を専門に行う担当者
用語の整理
ウォーターフォール開発＝開発を「要件→設計→実装→テスト→リリース」の一方向に進める方式。各工程を完了させてから次へ進むため、後戻りが難しい。現在でも基幹システムや組み込み開発で採用されていますが、後述するアジャイル開発と対比される伝統的な手法です。
04

バグ発見が遅れるほど高くつく

COST OF LATE BUGS

1981年、研究者バリー・ベームが衝撃的なデータを発表しました。バグは発見が遅れるほど修正コストが指数関数的に増える。設計段階で見つかれば1の手間が、リリース後に発覚すると100倍以上になるという事実です。これが現代のテスト戦略すべての出発点になりました。

cost curve 後の工程で見つかるほど修正コストは跳ね上がる
// 修正コストの曲線

Boehm（ベーム）曲線として知られる、ソフトウェア工学の最も重要な経験則の一つ。

// なぜ後ほど高くつくのか 設計段階で発見 → 設計書を直すだけ → コストは小さい 実装段階で発見 → コードを直す → やや手間 テスト工程で発見 → 設計まで戻る場合あり → 結構な手戻り 本番リリース後に発見 → 緊急対応 → 既に動いている他機能への影響 → ユーザーへの謝罪 → 信頼の失墜 → コストは何倍にも膨らむ
この曲線から導かれる原則
「テストはなるべく早い段階で・なるべく頻繁に」。この鉄則が、後のTDD（テスト先行）・CI/CD（自動化）・テストピラミッド（高速な単体テストを多く）といった現代のテスト思想すべての根っこにあります。
05

「先にテストを書く」発想

TDD APPEARS

1999年、ソフトウェア開発者ケント・ベックが革命的な手法を提唱しました。それがXP（エクストリームプログラミング）と、その中核技法であるTDD（テスト駆動開発）です。「実装してからテスト」という当たり前の順序を逆転させ、「テストを先に書いてから実装する」という発想を打ち出しました。

test first 順序を逆転させたパラダイムシフト
// 順序の逆転

BEFORE — 従来

実装
▶
テスト

AFTER — TDD

テスト
▶
実装
▶
改善

先にテストを書くことで、「何を作るか」が明確になってから「どう作るか」を考える順序に。設計の質も同時に上がる効果が知られている。

// 同じ時期に登場した動き 1999 XP / TDD (Kent Beck) 2001 アジャイルソフトウェア 開発宣言 // 共通する思想 ・短いサイクルで作って試す ・変化を恐れず受け入れる ・動くソフトウェアを重視 ・自動テストが前提 // この流れが現代に直結している
用語の整理
XP（エクストリームプログラミング）＝1999年提唱の開発手法。良い習慣を「極端に（extreme）」まで徹底するという思想。
TDD（Test-Driven Development、テスト駆動開発）＝テストを先に書き、それを通すコードを書き、改善する、を繰り返す手法。本編Vol.3で詳しく扱います。
アジャイル開発＝短いサイクルで作って試して直すを繰り返す進め方の総称。ウォーターフォールと対比される。
06

自動化と継続的な検証

CI/CD ERA

2000年代以降、自動テストの普及と共にCI/CD（継続的インテグレーション／継続的デリバリー）が広がりました。誰かがコードを変更するたびに、自動でテストが走り、合格すれば自動で公開準備まで進む。テストが「時々やるイベント」から「常に回っている仕組み」へと変わった時代です。

continuous テストが日常的なバックグラウンド業務へ
// 検証頻度の変化
〜1990s 数ヶ月に1度 リリース直前にまとめてテスト
2000s 毎日（夜間ビルド） 毎晩自動でビルドとテストを実行
2010s〜 毎コミット（CI） 変更するたびに自動テスト。数分でフィードバック
現在 本番反映まで自動（CD） テスト合格→自動デプロイ。1日に何十回も更新
// 自動化サイクル 開発者がコード変更 ↓ git push ↓ CIサーバが起動 ↓ 自動でビルド ↓ 自動でテスト全実行 ↓ 合格? ──── 失敗 ──→ 通知・ブロック ↓ 成功 CDで自動デプロイ ↓ 本番環境に反映 // 人間の手はほぼ介在しない
用語の整理
CI（Continuous Integration、継続的インテグレーション）＝変更を頻繁に統合し、その都度自動でビルド・テストを行う仕組み。
CD（Continuous Delivery / Deployment、継続的デリバリー／デプロイメント）＝CIの後、自動で配布・公開まで進める仕組み。
DevOps＝開発（Dev）と運用（Ops）を一体化させる文化・実践。CI/CDはその中核技術。本編Vol.9で詳しく扱います。
07

今、テストが仕事である理由

WHY TESTING MATTERS NOW

ここまでの流れを一本の物語として振り返ります。テストは時代の必然として生まれ、規模拡大・コスト構造・思想転換・自動化という4つの波を経て、今の形になりました。これは「単に良い習慣」ではなく、現代の開発で避けて通れない構造的な要請です。

why now 私たちがテストを学ぶ構造的な理由
// 全体の時間軸（まとめ）
~1970 テスト=デバッグ 小規模・個人完結
1970-80s 規模拡大の波 品質を守る仕組みが必要に
1980-90s 工程化の波 テストが独立した仕事に
1999- 思想転換の波 TDD・アジャイルで先行化
2000s- 自動化の波 CI/CDで常時実行へ
現在 エンジニアリングの一部 書ける人=実装と並ぶ必須スキル
// 4つの波 ① 規模が個人を超えた → 仕組みでの品質保証が必要 ② 修正コストの曲線 → 早期発見が経済合理性を持つ ③ 開発の高速化 → ウォーターフォールでは追いつかない ④ 信頼性=事業価値 → ソフトウェアが社会の基盤に ↓ 結論: テストは「やった方が良いこと」 ではなく、 「やらないと成り立たないこと」 になった
本編Vol.1とのつながり
本編Vol.1で「テストとは何か」「なぜ書くのか」「ピラミッド構造」を学びました。地図編Vol.1で学んだ時間軸の物語を重ねると、本編で出てきた一つ一つの内容が「歴史の必然として今そこにある」と見えてきます。ピラミッド構造もTDDもCI/CDも、すべて4つの波の答えとして生まれてきたものです。
