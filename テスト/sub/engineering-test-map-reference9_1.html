01

テスト頻度の歴史的変化

FREQUENCY EVOLUTION

テストの実行頻度は、過去70年で劇的に変化しました。1960年代は「リリース前にまとめて」だったものが、現代では「毎コミットごと」。この頻度の変化が、ソフトウェア開発のあらゆる側面を変えていきました。

frequency テスト実行頻度の歴史的推移
// 頻度の推移
1960s
数ヶ月に1度
1980s
毎月
1990s
週次
2000s
毎日 (夜間)
2010s
毎コミット
2020s
継続的

60年で頻度は数百万倍に。これは技術の進歩だけでなく、ソフトウェアに対する社会の要求の変化でもある。

// なぜ頻度が変わったか 1960s: テスト=リリース前の儀式 メインフレーム時代 ソフトを「焼き付けて」出荷 間違うとやり直しが大事 1980s: 月次リリース体制 企業システム 月1の定期メンテ枠でテスト 1990s: 週次の動作確認 PCソフト・Webの登場 リリース間隔が短くなる 2000s: 夜間ビルドの自動化 CruiseControl, Hudson登場 毎晩テスト実行が当たり前に 2010s: 毎コミットの自動実行 Jenkins, Travis CI隆盛 分単位のフィードバック 2020s: 常時実行 + 即デプロイ GitHub Actions時代 検証 = 環境の一部
この変化の意味
検証は「特別な活動」から「日常の背景」へと変化しました。誰かが意識して「テストする」のではなく、コミットすれば自動で検証されてしまう環境が当たり前に。これは品質保証のあり方そのものを変えました。
02

手作業時代の課題

MANUAL ERA

自動化される前、テストは人が画面を触って確かめる仕事でした。専門のテスターが画面を操作し、結果をメモし、不具合を報告する。この時代の課題が、後の自動化への動機を形作りました。

manual era 手作業によるテストの限界
// 手作業の構造的問題
問題結果
遅い1機能の確認に数時間〜数日
不安定人によって判定基準が違う
再現困難同じ操作を毎回再現できない
退屈同じ操作を繰り返す精神的負担
スケールしない機能が増えても人は増やせない
抜け落ちる変更時に確認漏れが必発
// 手作業時代の典型的1日 09:00 リリース予定機能のリスト確認 10:00 機能Aの操作手順書を読む 10:30 画面で操作 「ボタンが反応した」 「画面遷移OK」 「保存OK」とメモ 12:00 ランチ 13:00 機能Bの手順書 ... 17:00 機能Eまで完了 18:00 報告書をまとめる // 1日で5機能のテスト // 100機能あれば20営業日 // = 1ヶ月の検証期間 ↓ 問題: ・リリース間隔が縮まると追いつかない ・「いつも通り動く」の確認に 多大な人手を投入 ・退屈で離職率も高い ↓ 「同じことを繰り返すなら 機械にやらせるべき」
用語の整理
手動テスト（manual test）＝人が手作業で行うテスト。
リグレッションテスト（regression test）＝既存機能が壊れていないか確認するテスト。手作業時代は最も時間を食う仕事の代表。
テスト工程＝開発工程の一区切りとしてテストを行うフェーズ。手作業時代の象徴的な仕組み。
03

スクリプト化の時代

SCRIPTING ERA

1990年代後半から、テストを「スクリプトとして書く」動きが広まりました。JUnit（1997年頃）、Selenium（2004年）など、現代でも使われるツールの原型が次々と登場。「テスト=コード」という発想の革命でした。

scripting テストを「書く」時代の幕開け
// スクリプト化の歴史
1997 JUnit (Java) Kent Beck・Erich Gammaが作成。「単体テストをコードに書く」の原点
1999 XP / TDD テストを「先に書く」発想が広まる
2004 Selenium ブラウザ操作の自動化。E2Eテストの先駆け
2009 Hudson / Jenkins CIサーバの定番。OSSとして急速に普及
// テスト=コード革命の本質 それまで: テスト = 手順書 「ボタンAを押し、画面Bを確認」 人が読んで実行 ↓ スクリプト化以降: テスト = 実行可能なコード プログラムが代わりに実行 ↓ 何が変わったか: ① 何度でも同じ実行ができる → 再現性 ② 速い → 人手の数百倍以上 ③ 履歴が残る → バージョン管理可能 ④ コードと一緒に進化する → 仕様変更にテストも追随 ↓ 「テストもまたソースコード」 この発想の転換が革命的
スクリプト化の意義
手作業からスクリプトへの移行は、「テストの価値が記録され、累積するようになった」転換点。一度書けば資産になる。書いた人が辞めても残る。これにより「テストの数」が組織の財産になりました。
04

CI登場と頻度革命

CI REVOLUTION

2000年代、「テストはあるが、走らせるのは時々」という状態から、「変更のたびに自動で走る」という発想が登場します。これがCI（継続的インテグレーション）の本質。頻度を上げることが本質的価値であると気づいた瞬間でした。

CI revolution 頻度こそが価値を生む
// CIのサイクル
commit
▶
build & test
▶
notify
fix
▶
commit
▶
...再び

サイクルが数分単位で回り続ける。問題は数時間後ではなく数分後に発覚する。

// なぜ「頻度」が本質なのか 問題発見の早さ: 発見が早い → 直すコストが安い (地図編Vol.1 Boehm曲線) 毎日テスト → 1日分の変更だけ調査 毎月テスト → 1ヶ月分の変更を調査 → 調査範囲が劇的に変わる 変更の心理的負担: 「次のテストはいつ?」 ↓ 「翌月」 → 不安・慎重・遅い ↓ 「即時」 → 安心・大胆・速い → 開発文化そのものが変わる 統合の痛み: まれにしかやらない統合 → 大量の競合・衝突 頻繁にやる統合 → 小さな衝突を都度解消 → 統合が「痛くない」状態に
頻度がもたらす変化
CIは「テストの内容」を変えたのではなく、「テストの頻度」を変えたイノベーションです。同じテストでも、月1回走らせるのと毎コミット走らせるのでは、得られる価値が桁違いに違う。頻度こそが質を変える—これは多くの分野で観察される現象です。
05

CDと「常に出荷可能」

CD AND ALWAYS-SHIPPABLE

CIが「テストを常に回す」だとすると、CDは「常に出荷可能な状態を保つ」仕組みです。さらに進めば「自動で出荷する」（継続的デプロイメント）まで行きます。テストと出荷が一体化した世界に到達しました。

CD CDが意味する2つの段階
// CDの2つのレベル
段階意味
Delivery常に「出荷できる状態」にしておく
Deployment合格したら自動で「出荷してしまう」

頻度の極致: 1日に何十回もリリース、というのが現代のWebサービスでは普通。Amazon・Netflix等は1秒に複数回デプロイすると言われる。

// 「出荷頻度」の歴史 1990s: CDをCD-ROMに焼いて出荷 半年に1度 / 年1回 2000s: Webサービス登場 数週間に1度のリリース 2010s: CI/CD普及 日次〜週次リリース 現在: Amazon: 1秒に複数回 Netflix: 1日数千回 Google: 継続的に ↓ これが可能になった条件: ・自動テストが網羅的にある ・CIで常に検証されている ・問題があれば即ロールバック ・「壊れない仕組み」が前提 ↓ テストへの投資が ビジネス速度を決める時代
この帰結の意味
リリース頻度=ビジネス価値の差別化要因になりました。「テストがしっかりある会社は、競合より速く改善できる」。これがもはや事業優位性の源泉です。テストは「品質を守る」だけでなく「速度を生む」道具へと進化したのです。
06

DevOpsという文化

DEVOPS CULTURE

CI/CDの普及と並行してDevOpsという概念が登場しました。開発（Dev）と運用（Ops）の垣根を取り払い、両者が一体となってソフトウェアを動かし続ける文化です。自動化はDevOpsの基盤であり、テスト自動化はその中核に位置します。

DevOps 開発と運用の融合
// 従来 vs DevOps
観点従来
役割開発と運用が別チーム
引き継ぎ開発→運用への「お渡し」
障害責任運用が対応・原因は開発に問い合わせ
変更月次・四半期単位
観点DevOps
役割一体のチーム
引き継ぎ「お渡し」がない・全員が両方
障害責任開発・運用共同で対応
変更日次・常時
// DevOpsを支える要素 ① Infrastructure as Code サーバ構築もコードで管理 Terraform, Ansible ② Containerization Docker・Kubernetes 本番と同じ環境がどこでも ③ CI/CD GitHub Actions・GitLab CI 常時自動化 ④ Automated Testing ピラミッド全層の自動テスト これがすべての前提 ⑤ Monitoring & Observability Prometheus・Grafana・Datadog 本番の状態が常に見える ↓ 自動テストは DevOpsの基盤として 不可欠な存在 // テストがない=DevOpsはできない
用語の整理
DevOps＝開発と運用を一体化する文化・実践の総称。2009年頃から広まる。
Infrastructure as Code＝インフラ設定をコードで管理する手法。
SRE（Site Reliability Engineering）＝Google発の運用方法論。DevOpsを工学的に体系化したもの。
可観測性 / Observability＝システムの内部状態を外から推測できる性質。ログ・メトリクス・トレースが3本柱。
07

「環境」としてのテスト

TEST AS ENVIRONMENT

ここまでの時間軸を振り返ると、テストは「イベント（時々やる行為）」から「環境（常にそこにあるもの）」へと変質しました。空気のように存在し、意識されなくなった時こそ、テストが最も価値を発揮している瞬間です。

test as environment 「環境」としてのテストへ
// 変質の構造
  1. 初期: テスト=儀式（リリース前の特別な作業）
  2. 中期: テスト=イベント（決まったタイミングで実行）
  3. 後期: テスト=習慣（毎日・毎コミット）
  4. 現代: テスト=環境（意識せず常に走っている）

現代のエンジニアは「テストを走らせる」とは言わない。「テストが走る」と言う。主語が人から仕組みに変わった。

// 「環境」化することの意味 儀式の時代: 「来月のテスト工程に向けて準備を」 → 大ごと・特別 イベントの時代: 「金曜にテストを走らせます」 → スケジュールされた予定 習慣の時代: 「コミット時に毎回走ります」 → ルーチン 環境の時代: 「テストはいつも背景で走っている」 → 空気のように存在 ↓ この変化が示すもの: 検証は「特別な営み」ではなく 「ソフトウェア開発という活動の 一部、当たり前の構成要素」 になった ↓ テストがある = 当たり前 CIが回る = 当たり前 これが現代の基準点
この巻のまとめ
地図編Vol.1で「テストはどこから来たか」と歴史を辿り、地図編Vol.9で「テストはどこへ向かったか」を振り返りました。儀式→イベント→習慣→環境という変質の旅。本編Vol.9のCI/CDは、この長い旅の到達点として位置づくのです。テスト自動化を実装する時、自分が立っている地点の歴史的意味も意識しておくと、より深い理解になります。
