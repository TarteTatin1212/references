01

依存という概念

DEPENDENCY

依存（dependency）とは、「あるコードが、別の何かに頼って動いている」状態のことです。ある関数がDBの存在に頼っている、あるクラスが現在時刻に頼っている、など。依存があると、その「頼り先」が変わるとコードも影響を受けます。

dependency 何に頼って動いているか
// 依存の可視化
注文サービス
↓ 依存
在庫リポジトリ
↓ 依存
DBドライバ
↓ 依存
本物のデータベース

「注文サービス」をテストしたいのに、その下に積み重なった依存をすべて用意しないと動かない。これがテストの壁。

// コードでの依存の表れ方 class OrderService { place(itemId) { // ↓ 依存: 在庫リポジトリ const stock = repo.find(itemId); // ↓ 依存: 現在時刻 const now = new Date(); // ↓ 依存: メール送信 mailer.send("完了"); // ↓ 依存: ランダム値 const id = Math.random(); } } // この関数は // ・repo // ・Date // ・mailer // ・Math.random // に依存している
用語の整理
依存（dependency）＝あるコードが、その動作のために別の何かを必要としている関係。依存先（dependee）＝頼られている側。依存元（depender）＝頼っている側。
02

純粋な関数 vs 不純な関数

PURE vs IMPURE

依存の有無で関数は2種類に分けられます。純粋関数（pure function）と不純な関数（impure function）です。純粋関数は外の影響を受けず、引数だけから結果を決めます。不純な関数は外に頼ったり、外に影響を残します。この区別がテストの容易さを大きく変えます。

purity 純粋／不純という重要な分類
// 純粋 vs 不純
PURE — 純粋
IMPURE — 不純
同じ入力 → 必ず同じ出力
同じ入力 → 出力が変わりうる
外の世界に影響しない
外の世界に影響を残す
例: add(1,2)=3
例: new Date().now()
テスト超容易
テストに工夫が必要

純粋関数はテストの天国。引数を渡して結果を見るだけで完結する。不純な関数は事情によって結果が変わるので、その「事情」を制御する必要がある。

// === 純粋関数 === function add(a, b) { return a + b; } // add(1, 2) は 何度呼んでも 3 // 外の世界とは無関係 function discount(price, rate) { return price * (1 - rate); } // 引数 → 結果 のみ。純粋 // === 不純な関数 === function greet() { const h = new Date().getHours(); return h < 12 ? "おはよう" : "こんにちは"; } // 結果が時間帯で変わる // 同じ引数(なし)でも結果が違う function save(user) { db.write(user); // 外に影響 } // 副作用 = 外の世界の変更
用語の整理
純粋関数（pure function）＝入力だけで出力が決まり、外に影響を残さない関数。
副作用（side effect）＝関数が、戻り値を返すこと以外に外の世界に与える影響。DB書き込み、ファイル出力、画面表示、メール送信などすべて副作用。
参照透過性（referential transparency）＝同じ引数なら同じ結果を返す性質。純粋関数の数学的な性質を指す。
03

なぜ不純が問題か

PROBLEM OF IMPURITY

不純な関数は「テスト可能性」を3つの側面で損なうからです。再現性・速度・独立性。この3つが揃わないと、自動テストの本来の価値は得られません。

why impurity hurts 3つの損なわれる性質
// 不純が損なう3つの性質
  1. 再現性 同じテストを再度走らせても結果が違うかもしれない
  2. 速度 DB・ネット越しの呼び出しが遅くする
  3. 独立性 外部の状態に左右されるため、他テストや環境と干渉

地図編Vol.2の「速い・安定・独立」という単体テストの3条件を、不純性は丸ごと崩します。だから工夫が必要になる。

// 不純が引き起こす具体的問題 [A] 時刻依存 test("朝の挨拶", () => { expect(greet()).toBe("おはよう"); }); // 夜に実行 → ✗ FAIL // テストが時刻次第で結果が変わる [B] DB依存 test("ユーザー取得", () => { const u = findUser(1); expect(u.name).toBe("佐藤"); }); // DBの中身次第で結果が変わる // 他テストが書き換えると壊れる [C] ネットワーク依存 test("API応答", () => { expect(fetchUser()).toMatchObject(...); }); // API停止中 → ✗ FAIL // ネット不通 → ✗ FAIL
この対立軸の重要性
「純粋ならテスト容易、不純ならテスト困難」。この対立軸はソフトウェア設計の至るところで顔を出します。関数型プログラミング、ヘキサゴナルアーキテクチャ、クリーンアーキテクチャ、いずれも「純粋なロジック」を中心に置く設計を志向しています。
04

偽物に置き換えるという発想

SUBSTITUTION

「不純な依存があるからテストできない」を解決する発想が、「依存先を偽物に置き換える」です。これがテストダブルの存在理由です。本物の代わりに「制御可能な偽物」を差し込めば、不純さの問題がそっくり消えます。

substitution 不純な依存を、制御可能な偽物に置き換える
// 置き換えの構造
// 本番
対象→本物のDB
// テスト時
対象→偽物のDB
対象から見ると「同じインターフェース」なので、置き換えても気付かず動く

対象側のコードは変えない。差し込まれる依存先だけを偽物に切り替える。これが「置き換え」の発想です。

// 置き換えで失われる性質と // 取り戻される性質 本物のDB依存: ✗ 遅い ✗ 不安定 ✗ 副作用 ✗ 不可制御 ↓ 偽物に置換 偽物のDB依存: ✓ 速い (メモリ内) ✓ 安定 (外部影響なし) ✓ 副作用なし ✓ 制御可能 (好きな結果を仕込める) // 対象は「DB呼び出し」を // しているつもりで動く // 偽物が代わりに応答する
POINT
「ダブルを使う」は手法であって目的ではないのがポイント。目的は「対象を不純な依存から切り離して、純粋な関数のように扱えるようにする」こと。本編Vol.4でMock/Stubの種類を学んだのは、この目的を達成する手段です。
05

5種類のダブルの位置関係

FIVE DOUBLES MAP

本編Vol.4で出てきた5種類のダブルを、軸の上に並べると関係が見えてきます。「実装の本格度」と「検証の関心」という2軸で位置づけることで、なぜ5種類あるのかが分かります。

five on axes 2軸で見る5種類の位置関係
// 位置関係
種類実装の本格度主な関心
Dummyほぼゼロ形を埋める
Stub小 (固定値返すだけ)返り値を制御
Spy本物に近い呼ばれたか記録
Mock小〜中呼ばれ方を検証
Fake大 (動く軽量実装)本物の振る舞い

本格度の左右軸: Dummy/Stub/Mock が「最小限の偽物」、Fakeが「動く実装」。Spyはその間。関心の上下軸: 状態検証(Stub/Fake)系か、振る舞い検証(Mock/Spy)系か。

// 軸での位置関係 実装ゼロ ────────── 本格実装 ↑ ↑ Dummy Stub Spy Fake ↑ ↑ 状態を 本物に近い 仕込む 動き Mock ↓ 振る舞いを 検証 // 縦軸: 何を関心にするか 状態検証 (State) ↑ Stub / Fake ↓ 振る舞い検証 (Behavior) ↑ Mock / Spy
用語の整理
状態検証（state verification）＝処理が終わった後の「状態」（変数の値・DBの中身など）を見て検証する方式。Stub/Fakeと相性が良い。
振る舞い検証（behavior verification）＝処理中に「どう呼ばれたか」「何回呼ばれたか」を見て検証する方式。Mock/Spyと相性が良い。
クラシシスト派＝状態検証重視の流派。モッキスト派＝振る舞い検証重視の流派。テスト界の宗教論争として知られる。
06

DIは「置き換え可能性」

DI = SWAPPABILITY

依存性注入（DI）とは、技術的には「依存先を、外から差し込めるようにする」設計のことです。これを「置き換え可能性」という視点で捉えると、その意義がクリアになります。本物も偽物も、同じ場所に差し込めるからこそテストが書けます。

DI as swappability 置き換え可能性こそDIの本質
// DIなし vs DIあり
DIなし
内部で new
↓
固定の依存
↓
置き換え不可
↓
テスト困難
VS
DIあり
外から受け取る
↓
差し替え可能
↓
本物⇔偽物 OK
↓
テスト容易

DIは「テストのためだけ」ではない。設計の柔軟性そのものが向上する。本番/開発/テストで違う実装を差し込めるのは、運用上も大きな価値。

// DIが実現する3つの世界 本番環境: new OrderService( new PostgresRepo(), new SmtpMailer() ); 開発環境: new OrderService( new SqliteRepo(), new ConsoleMailer() // 画面に出すだけ ); テスト: new OrderService( new InMemoryRepo(), // Fake { send: jest.fn() } // Mock ); // 同じ「OrderService」が // 3つの世界で適切に動く // → これがDIの本当の価値
この見方の価値
DIを「テストのための仕組み」と捉えると小さく見えるが、「置き換え可能性を設計に組み込む」と捉えると大きく見える。後者の視点でコードを書けると、自然とテスト容易性も上がります。「テストのために」ではなく「柔軟性のために」DIを使う、というのが現場の感覚に近いです。
07

偽物を使う・使わないの判断軸

WHEN TO FAKE

どんな場面でも偽物を使うべき、ではありません。偽物を使うコストと、本物を使うコストを秤にかけて決めます。判断軸を整理しておくと、本編Vol.4の「やりすぎ問題」を避けられます。

when to use 本物 vs 偽物の判断基準
// 偽物を使うべきケース
  1. 遅い DB・HTTP・ファイルI/O
  2. 副作用が困る 本物のメール・本物の決済
  3. 再現したい エラー応答・タイムアウト
  4. 準備が大変 認証・初期データ作成
// 本物を使うべきケース ① 同じプロセス内のクラス → DIで切れる程度なら本物で良い ② 純粋な計算ロジック → 偽物にする意味がない ③ 結合テスト・E2Eテスト → そもそも本物を組合せて試す層 ④ 価値ある検証になるとき → DB制約・SQL構文・索引性能など // 判断の原則 偽物コスト > 本物コスト → 本物を使う 偽物コスト < 本物コスト → 偽物を使う // 単純な経済原理で考える
この巻の結論
偽物は「不純さからの避難場所」です。本物が遅い・不安定・副作用ありの時に逃げ込む場所。逆に言えば、本物に問題がなければ偽物に逃げる必要はない。「全てを偽物に」も「全てを本物で」も極端で、現場では境目を見極める判断力が求められます。
