01

ソフトウェアは層で出来ている

LAYERED STRUCTURE

まず前提として、現代のソフトウェアは層構造で組み上がっています。小さな関数→クラス→モジュール→サービス→システム、と部品が積み上がる構造。これをレイヤードアーキテクチャと呼びます。テストの粒度はこの構造に対応します。

layers 部品が積み上がる階層構造
// 5層の組み上がり
SYSTEMシステム全体 / 完成品
MODULEモジュール / 機能群
CLASSクラス / 部品
FUNCTION関数 / メソッド
STATEMENT文 / 行

下から上へ部品が組み合わさって大きくなる。各層には独自の役割と境界がある。

// 層と粒度の対応 STATEMENT if文・代入・式 → テスト対象にはならない FUNCTION add(), discount(), greet() → 単体テスト の対象 CLASS OrderService, UserRepository → 単体 or 結合のどちらでも MODULE 注文管理機能, 認証機能 → 結合テスト の対象 SYSTEM ECサイト全体 → E2Eテスト の対象 // テストの粒度は // この階層に従って決まる
用語の整理
レイヤードアーキテクチャ＝ソフトウェアを層に分けて構築する設計。よく見るのは「プレゼンテーション層 / アプリケーション層 / ドメイン層 / インフラ層」の4層。
モジュール＝関連するクラス群を1つにまとめた単位。「注文モジュール」「認証モジュール」など機能単位の塊。
関心の分離＝層ごとに役割を明確に分けるという設計原則。
02

組み上がりの上下軸

VERTICAL ASSEMBLY

層は下から上へ積み上がります。下層は小さく・抽象度低く・数が多い。上層は大きく・抽象度高く・数が少ない。この性質は地図編Vol.2のテストピラミッドと驚くほど似ています。テストの粒度はソフトウェアの構造そのものを反映しているのです。

vertical axis 下から上への組み上がり
// 各層の性質
層数大きさ
関数数百〜数千1〜数十行
クラス数十〜数百100〜数百行
モジュール数個〜数十千〜数千行
サービス1〜数個万〜数十万行
システム1全体

下に行くほど数が多いから単体テストは数が多くなる。上に行くほど数が少ないからE2Eテストも少なくなる。この対応こそピラミッド型の根拠。

// ピラミッドの正体 ソフトウェア構造 テスト粒度 ───────── ───────── STATEMENT FUNCTION 数千 → UNIT 数千 CLASS 数百 MODULE 数十 → INTEG 数十 SERVICE 数個 SYSTEM 1 → E2E 数件 ↓ ピラミッド型は 構造の自然な反映 「下が広く、上が狭い」 // 部品の数 = テストの数 // → ピラミッドが必然的に発生
この対応の意味
地図編Vol.2では「ピラミッドにすると効率が良いから」と説明しました。今回はもう一歩深い理由が見えました。「ソフトウェアの構造自体がピラミッド型だから、テストもピラミッド型になる」。下層の部品数が多いからこそ、下層のテストも多くなるのです。
03

境目が生まれる場所

WHERE JUNCTIONS APPEAR

層と層のつなぎ目、それが「境目（interface）」です。部品Aと部品Bが情報をやり取りする場所。バグが集中する場所として知られ、結合テストが見張るべき領域そのものです。

junctions 部品と部品のつなぎ目に注目
// 典型的な境目
部品 A
Controller
(HTTP受付)
境目
部品 B
Service
(業務ロジック)
部品 A
Service
境目
部品 B
Repository
(DB操作)

境目で起きる典型バグ:
・データ形式の不一致
・null許容性の食い違い
・文字コード変換ミス
・例外の取り扱い相違

// 境目バグの実例 部品A (Controller): 受け取ったメールを そのまま渡す → "FOO@BAR.com" 部品B (Service): メールで重複チェック 小文字化してDB問い合わせ → "foo@bar.com" 部品C (Repository): DBには大文字のまま保存されていた → "FOO@BAR.com" // 単体テストでは見えない // それぞれの部品は「正しく」動く // 結合してみると // 境目の処理がズレているのが発覚 ↓ これが結合テストの仕事 「個別では正しいが組合せると壊れる」
なぜ境目にバグが集中するか
「2人以上の作者が関わるから」。部品Aを作った人と部品Bを作った人の暗黙の前提が食い違うと、境目で問題が露呈します。1人で全部書いていても、書いた時期がズレれば過去の自分の前提を忘れる。これは構造的に避けられない。だから境目を狙ったテストが必要なのです。
04

結合テストはどこを見るか

WHAT INTEGRATION SEES

結合テストが見るのは「境目の正常な振る舞い」です。本編Vol.6では具体的な書き方を学びましたが、地図編で改めて整理すると、結合テストは「単体では見えない領域を狙い撃つ」専用の道具だと分かります。

what to test 結合層が見るべき固有のもの
// 結合層固有の検証対象
対象説明
SQLの構文本物のDB上での実行可能性
外部API連携HTTP往復・JSONマッピング
トランザクションロールバック・コミットの境界
制約UNIQUE・FK・NOT NULL
マッピングDTO ⇔ Entity ⇔ DB
シリアライズJSON ⇔ オブジェクト

いずれも「本物の何か」を組み合わせないと検証できないもの。だから結合テストでなければ意味がない。

// 何を結合テストで見るかの基準 ① 単体で検証できないか? yes → 単体で書く ② その「組合せ」固有の問題か? yes → 結合テストの出番 ③ E2Eで検証する方が自然か? yes → E2Eに任せる ↓ 結合テストは 「単体でもE2Eでもなく」 「境目を狙い撃つ層」 という位置づけ // 範囲を欲張ると // E2Eと区別がつかなくなる // 狭くしすぎると // 単体テストと区別がつかない 「ちょうど境目だけ」 が結合テストの真価
POINT
本編Vol.6で「Testcontainersで本物DBを起動」「WireMockで外部APIを擬似化」という手段を学びました。これらが必要になる理由は、まさに「境目だけを正確に見る」ためです。本物のSQL構文や本物のHTTPやり取りを検証する必要があるから、ダブルだけでは足りない。
05

統合戦略の時間軸

INTEGRATION OVER TIME

部品を組み合わせていく手順にも歴史があります。ビッグバン統合から始まり、トップダウン・ボトムアップ・サンドイッチ・継続的統合へと進化してきました。今の主流であるCI（継続的統合）はこの進化の到達点です。

integration history 統合手順の時代変遷
// 統合戦略の進化
ビッグバン統合 最後に全部一気に組合せる。失敗多発の悪しき手法
トップダウン統合 上層から組合せる。スタブで下層を代用
ボトムアップ統合 下層から組合せる。ドライバで上層を代用
サンドイッチ統合 両端から中央へ向かって組合せる
継続的統合 (CI) 変更ごとに自動で組合せ&テスト。現代の主流
// 戦略ごとの特徴 ビッグバン: 全部書く → 最後に組合せ → どこが原因か特定不能 → 失敗の典型 トップダウン: 上から組み立て 下はStub(空っぽ)で代用 → 動く形が早く見える ボトムアップ: 下から組み立て 上はDriver(呼び出し役)で代用 → 基礎が固まる 継続的統合 (CI): 小さな変更を頻繁に統合 自動テストが常に回る → 問題の早期発見 → 統合の痛みを分散 // 詳細は本編Vol.9で扱う
用語の整理
スタブ（stub）＝統合段階で、まだ実装されていない下層の代わりに置く仮実装。本編Vol.4のテストダブルと同じ言葉だが、文脈が違う。
ドライバ（driver）＝下層を呼び出す上層が未完成の時、代わりに呼び出してくれる仮の上層。
継続的統合 / CI＝変更を毎回統合し自動テストを走らせる開発スタイル。本編Vol.9で詳述。
06

マイクロサービスと統合

MICROSERVICES

現代のソフトウェアはマイクロサービス（多数の小サービスの集合体）として構築されることが増えています。「境目」がプロセス内ではなくネットワーク越しに広がり、結合テストの意味も変わります。契約テストという新しい概念も登場しました。

microservices 境目がネットワーク越しに広がる時代
// 一枚岩 vs マイクロサービス
観点違い
境目プロセス内 → ネットワーク越し
境目の数数十 → 数百〜
失敗モード例外 → タイムアウト・部分障害
結合テスト1プロセス内 → 複数サービス間
新たな技法契約テスト (Contract Test)

サービス数が増えると全部を起動して結合テストは事実上不可能。代わりに「契約テスト」が広まっている。

// 契約テストの発想 問題: サービスAがサービスBを呼ぶ 10サービスあると組合せ爆発 全部起動してテストは無理 解決: AとBの「契約」を文書化 AはBの仕様に従って呼ぶか検証 BはAの期待通りに応答するか検証 両者が契約を満たしていれば 本物を組合せなくても動くはず // 代表ツール // Pact, Spring Cloud Contract // 「結合テストの代替」というより // 「境目を細分化して検証する」 // 新しいアプローチ
用語の整理
マイクロサービス＝小さな独立サービスの集合体としてシステムを構築するアーキテクチャ。1サービス1機能が原則。
契約テスト（Contract Test）＝「サービス間の取り決め（契約）」を満たしているかを検証するテスト。両端を実際に組合せず、契約に対する適合を検証する。
Pact＝契約テストのデファクトツール。「コンシューマ駆動契約テスト」を実現する。
07

ピラミッドの中で結合層が持つ意味

ROLE IN THE PYRAMID

地図編Vol.2のテストピラミッドで「中段」に位置した結合テスト。今や、その役割がより鮮明になります。「単体では見えず、E2Eでは遅すぎる」境目専門の層として、ピラミッドのバランスを保つ要となっています。

role 中間層は単に「中間」ではない
// 各層の役割の再整理
  1. UNIT 個々の部品の正しさを検証
  2. INTEGRATION 部品同士の境目を検証
  3. E2E 部品が組み上がったシステム全体の振る舞いを検証

結合層がなければ、単体ですり抜けたバグはE2Eまで見つからない。E2Eは遅いので発見が遅れる。だから結合層が必要。

// 結合層の存在価値 単体で見つからない ↓ 結合層で捕まえる ↓ ここで捕まえないと… ↓ E2Eまで見つからない ↓ E2Eで失敗が出る頃には ↓ 原因特定に時間がかかる ↓ 地図編Vol.1のBoehm曲線 「発見が遅いほど高コスト」 ↓ 結合層は 単体とE2Eの 「セーフティネット」 // 数は少なくても、ないと困る層
この層の本質
結合テストは「単体テストと E2Eテストの間にある検証の隙間を埋める」役割。多すぎるとピラミッドが崩れ、少なすぎると境目バグの発見が遅れる。「境目を狙い撃つ専門部隊」として、ちょうど良い数だけ書く、これが現場の知恵です。
