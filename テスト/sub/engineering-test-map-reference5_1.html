01

入力空間という考え方

INPUT SPACE

入力空間（input space）とは、「ある関数が取りうる入力の集合すべて」のことです。引数が1つなら1次元、2つなら2次元、N個なら N次元の空間として考えます。テスト設計は、この空間から「どの点を試すか」を選ぶ営みです。

input space 入力の集合をひとつの空間として捉える
// 入力空間の大きさ
boolean
2
1〜100の整数
100
char (256)
256
int (32bit)
42億
long (64bit)
1844京
string
∞

たった1つの引数でこの広さ。複数引数では掛け算で膨れ上がる。

// 入力空間の例 function add(a, b) {...} // 入力空間 = // aのとりうる値 × // bのとりうる値 // もしa,bがint型なら // 42億 × 42億 // = 約1844京通り // 1ナノ秒で試しても // 585年かかる // 「全部試す」は // 物理的に不可能 // → 何らかの方法で // この空間を絞る必要
用語の整理
入力空間（input space）＝関数が取りうる入力の全体集合。N次元空間＝N個の引数を持つ関数の入力空間。各引数が1つの軸。網羅性（coverage）＝この空間をどれだけカバーしたかという度合い。
02

組合せ爆発

COMBINATORIAL EXPLOSION

入力空間は「引数の数」と「各引数のとりうる値」の掛け算で決まります。引数が増えるたびに、空間は急激に膨張します。これを組合せ爆発と呼びます。テスト設計が「全部」を諦め、「賢く絞る」方向に進化した最大の理由がこれです。

explosion 引数が増えると指数的に膨らむ
// 組合せ数の増え方
引数の数 (各3値)組合せ数
1個3
2個9
3個27
5個243
10個59,049
20個約34億

3Nで増える。たった3値しか取らないパラメータでも、20個並べば手に負えなくなる。

// 現実の組合せ爆発 ECサイトの購入処理: ・会員種別 (3値) ・支払方法 (5値) ・配送方法 (4値) ・キャンペーン (10値) ・配送先地域 (47値: 都道府県) ・商品種別 (20値) ... 単純な掛け算: 3 × 5 × 4 × 10 × 47 × 20 = 564,000通り // 全網羅は事実上不可能 // だから絞り込みの戦略が必要 // ・同値分割 // ・境界値 // ・ペアワイズ // ・etc.
この事実が示すもの
「全部試したい」は人間の本能ですが、組合せ爆発の前では成立しません。テスト技法の歴史は、「諦め方の進化」とも言えます。何を諦めるかを賢く決める技術、それが設計技法なのです。
03

「同じ振る舞い」という仮定

EQUIVALENCE ASSUMPTION

全網羅を諦めた時、頼りになる仮定があります。「似た入力は似た振る舞いをする」という経験則です。年齢15歳と16歳は同じ「未成年」として扱われるはず。この仮定が正しいと信じることで、無限の入力空間を有限の「グループ」に分割できます。

key assumption テスト設計を支える唯一の前提
// 仮定の中身
  1. 入力空間に「同じ振る舞いをする領域」がある
  2. その領域内の代表1点を試せば、領域全体を試したことになると仮定する
  3. 領域の境目で振る舞いが変わる
  4. 境目の前後を試せば、領域の切り替えを検証したことになる

この仮定が成り立つから、同値分割と境界値が意味を持つ。仮定が崩れる時はバグが潜んでいるサイン。

// 仮定が崩れる例 // 仕様: 0〜17歳はNG // 仮定: 1〜17歳のどれを試しても同じ // テスト: 15歳 → ✓ NG // しかし実装に隠れたバグがあると… function classify(age) { if (age === 13) return "OK"; // バグ if (age < 18) return "NG"; ... } // 15歳のテストでは見つからない // → 仮定が局所的に崩れている // だから100%安全ではない // だが、現実的には十分機能する // テスト設計は「確率を上げる」 // 営みである
POINT
テストは「バグがないこと」を証明できません。Dijkstra（ダイクストラ）の名言「テストはバグの存在を示せるが、不在を示せない」がこの限界を表しています。テスト設計はあくまで「見つかる確率を上げる」営み、と理解しておくと過信を避けられます。
04

絞り込みの3層構造

THREE-LAYER REDUCTION

テスト設計技法は「3層の絞り込み」として整理できます。入力空間を最初は大きな「群」に、次に「代表」に、最後に「狙い目」に絞り込む流れです。本編Vol.5の技法はすべてこの3層のどこかに位置づけられます。

three layers 絞り込みは3段階で進む
// 3層の絞り込み
LAYER 1
∞
無限の入力空間
LAYER 2
N群
同値クラスに分割
LAYER 3
N点
代表値 + 境界値

無限 → 数十群 → 数十点。この圧縮こそテスト設計の核心。

// 各層で起きること LAYER 1: 入力空間 ∞通りの入力 ↓ // この層では考えるだけで途方に暮れる LAYER 2: グループ化 同値分割 ↓ 「同じ振る舞いの群」に仕分け 例: NG群, OK群, シニア群 ↓ N群に圧縮 LAYER 3: 代表選定 各群から代表値1つ + 境界値の前後 + 異常値 ↓ 最終的に数十点のテストケース ↓ ∞ から 数十 へ これがテスト設計
この構造の価値
技法を覚える前に、「自分は今、無限から有限に圧縮しているのだ」と意識すること。3層構造を頭に入れておけば、本編で出てきた様々な技法が「同じ目的に向かう違う道具」として整理できます。
05

技法と入力空間の対応関係

TECHNIQUES MAPPED

本編Vol.5で出てきた各技法は、「入力空間のどんな性質に焦点を当てるか」で分類できます。すべての技法が同じ問題を解いているわけではなく、それぞれが空間の異なる側面を切り取ります。

techniques map 技法が攻めている空間の側面
// 技法と焦点
技法焦点
同値分割空間を群に分ける
境界値群の境目を狙う
デシジョン条件の組合せを網羅
状態遷移時間軸を加えた空間
ペアワイズ多次元空間を効率圧縮
異常系空間の「外」を狙う
// 視覚化 — どこを攻めるか 同値分割: [ A ][ B ][ C ] ↑ ↑ ↑ 各群から1点 境界値: [ A | B | C ] ↑↑ ↑↑ 境目の前後 デシジョン: 条件×条件の網 全マスを埋める 状態遷移: 状態 + 操作 + 時系列 空間に時間軸を追加 ペアワイズ: 多次元空間 ↓ 圧縮 すべてのペアだけ網羅 3因子以上は妥協 異常系: 仕様外の空間 null, 型違い, 範囲外
この対応の意味
技法は「相互排他的」ではなく「補完的」です。同値分割で大枠 → 境界値で精度向上 → 異常系で隙を埋める、という組み合わせが現場の定番。1つの技法で全部やろうとしないのがコツです。
06

網羅率という幻想

COVERAGE ILLUSION

テスト網羅率（カバレッジ）は便利な指標ですが、誤解しやすい数字でもあります。「100%カバレッジ＝バグゼロ」ではありません。網羅と入力空間の関係を整理しておくと、Vol.8で出てくるカバレッジ議論への準備になります。

coverage myth カバレッジは入力空間を覆い尽くしていない
// 何をカバレッジは測っているか
人の錯覚
「100%カバレッジ」
=
「すべての入力を試した」
↑ 違う
≠
実際の意味
「100%カバレッジ」
=
「コードの全行を1度通った」
入力空間とは別の話

カバレッジは「コード視点」の網羅、テスト設計は「入力視点」の網羅。両方が必要で、片方では足りない。

// 100%カバレッジでもバグるコード function divide(a, b) { return a / b; // 1行 } test("6/2=3", () => expect(divide(6,2)).toBe(3)); // カバレッジ 100% // しかし divide(1, 0) でバグる // 入力空間視点では // b = 0 のケースが抜けている // = 同値分割の不足 // カバレッジは // 「テストが弱い場所」は教えてくれる // 「テストが十分」は保証しない // → 詳しくは本編Vol.8
2つの網羅性
コードカバレッジ（コードの何行を実行したか）と入力カバレッジ（入力空間のどこを試したか）は別物です。両者を分けて考えるのが、カバレッジ盲信から抜け出す第一歩。Vol.8でさらに深く扱います。
07

テスト設計の本質

ESSENCE

ここまで読み解いた構造をまとめます。テスト設計の本質とは「無限の入力空間を、限られた時間と労力で、できる限り高い確率でバグを見つけられるように切り取る」営みです。完璧は望めない。だからこそ知恵を絞る。

essence テスト設計とは何だったか
// 因果の総まとめ
  1. ソフトウェアの入力空間は無限
  2. 全部は試せない（組合せ爆発）
  3. 「似た入力は似た振る舞い」と仮定する
  4. 空間を群に分割する
  5. 各群の代表値と境目を選ぶ
  6. 残りは異常系・組合せで補強
  7. 結果として少数で高確率の検出を実現
// テスト設計は何ではないか NOT: 完璧を目指す営み 全網羅を達成する技 バグの不在を証明する手段 // テスト設計は何であるか IS: 限られた資源で 高い確率でバグを見つける営み 「諦め方の設計」 経験と理論で 最も効率的な点を選ぶ技 ↓ 技法を覚える前に この本質を押さえる
この巻のまとめ
本編Vol.5で「同値分割」「境界値」「デシジョンテーブル」を学んだ時、それぞれが個別の技に見えたかもしれません。地図編Vol.5を通して見ると、すべてが「無限を有限に圧縮する」という同じゴールに向かう、異なる切り口の道具だと分かります。これが「点」を「面」として理解する、ということです。
