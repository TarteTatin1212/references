01

「大きさ」とは何か

WHAT IS TEST SIZE

テストの「大きさ」とは、1つのテストが一度に検証する範囲の広さのことです。1つの関数だけを試すのが「小さい」テスト、画面操作からデータベース更新まで全部通して試すのが「大きい」テスト。この大きさの違いが、テスト全体の戦略を決めます。

test size 範囲の広さで分類される
// テストの粒度 — 3つの層
E2E大きい / 少ない
INTEGRATION中くらい
UNIT小さい / 多い

この3層構造をテストピラミッドと呼びます。提唱者はMike Cohn（2009年頃）。今では世界中の現場で標準的な考え方になっています。

// なぜ「層」で考えるのか 検証範囲が違うと ↓ かかる時間が違う ↓ 扱える数が違う ↓ 壊れやすさも違う ↓ 役割分担が必要 // 1種類で全部やろうとすると // どこかで必ず破綻する // → 異なる粒度を組み合わせる // → これが「ピラミッド」の発想
用語の整理
粒度（grain size）＝テストが扱う対象の細かさ。「単体テスト」「結合テスト」「E2Eテスト」は粒度による分類。
テストピラミッド＝3層の理想的な比率を「下が広く上が狭い三角形」として示す概念。Mike Cohn著『Succeeding with Agile』（2009）が出典。
02

単体テスト・最小の層

UNIT LAYER

ピラミッドの最下層・底辺の広い部分が単体テストです。1つの関数・1つのメソッドだけを切り出して試します。本編Vol.2で書き方を学んだのがこの層。速い・多い・壊れにくいのが特徴です。

unit layer 関数1つを単独で試す層
// 単体テスト層の性質
項目性質
対象関数・メソッド・クラス
速度1件 数ミリ秒
数数百〜数千件
外部依存原則なし
失敗時原因が一目で分かる

速いから多く書ける、範囲が狭いから原因が特定しやすい。この2つが単体テストの最大の価値です。

// 単体テストの例 // 「価格計算」関数だけを試す function calc(price, rate) { return price * (1 - rate); } // テストはこの関数 だけ を呼ぶ test("30%引き", () => { expect(calc(1000, 0.3)) .toBe(700); }); // DB? いらない // 画面? いらない // ネットワーク? いらない // 純粋にロジックだけを検証 // → だから 速い・多い・壊れにくい
POINT
単体テストが多く書けることは、変更への安心感を生む。リファクタリングや機能追加のたびに数千件のテストが数秒で回って「OK」と言ってくれる、この体験こそが現代開発の基盤です。
03

結合テスト・中間の層

INTEGRATION LAYER

ピラミッドの中段が結合テスト（インテグレーションテスト）です。複数の部品を組み合わせた時の挙動を試します。単体では問題なくても、組み合わせると不具合が出る場面はよくあります。「境目で起きるバグ」を見つける層です。

integration layer 部品の組合せ・境目を試す層
// 結合テスト層の性質
項目性質
対象複数クラス + DB / 外部API
速度1件 0.1〜数秒
数数十〜数百件
外部依存DB等の本物 or 模擬
見つかるもの境目のバグ・接続不整合

結合の境目＝サービスとリポジトリ、コントローラとサービス、アプリとDB、など。単体テストでは見えない問題がここで発覚します。

// 結合テストの例 // 「注文サービス + DB」を組合せて試す class OrderService { place(userId, itemId) { const stock = repo.find(itemId); if (stock.qty < 1) throw ...; repo.decrement(itemId); return repo.createOrder(...); } } // テストは実DB(テスト用)と一緒に動かす test("注文するとDBの在庫が減る", () => { service.place("u1", "i1"); const after = repo.find("i1"); expect(after.qty).toBe(9); }); // 本編Vol.6で詳しく扱う
用語の整理
結合テスト / 統合テスト（integration test）＝複数のモジュールやコンポーネントを統合して行うテスト。日本語では「結合」と「統合」がほぼ同義で使われます。
境目（インターフェース）＝部品と部品が情報をやり取りする接点。バグが集中しやすい場所として知られています。
04

E2Eテスト・最上の層

E2E LAYER

ピラミッドの最上層・最も狭い部分がE2Eテスト（End-to-End、エンドツーエンド）です。ユーザーが画面を操作するのと同じように、システム全体を通して動かす検証です。遅い・少ない・壊れやすいのが特徴で、ピラミッドの頂上に置かれる理由がここにあります。

E2E layer 画面操作からDBまで全体を貫く層
// E2Eテスト層の性質
項目性質
対象システム全体・画面から DB まで
速度1件 数秒〜数十秒
数数件〜数十件
外部依存すべて本物に近い構成
見つかるものユーザー体験の崩れ・全体の破綻

壊れやすいのは、関係する部品が多すぎて1つでも変われば影響を受けるから。「フレーキー（不安定）」と呼ばれる失敗が起きやすい層です。

// E2Eテストの例 // ブラウザを自動操作して全体を確認 test("購入フロー", async () => { await page.goto("/shop"); await page.click("商品A"); await page.click("カートに入れる"); await page.click("購入"); await page.fill("#card", "4242..."); await page.click("確定"); expect(await page.title()) .toBe("購入完了"); }); // 1テストで数十秒 // 画面・API・DB・決済 全て通る // 本編Vol.7で詳しく扱う
用語の整理
E2Eテスト（End-to-End Test）＝システムの入口から出口まで（端から端まで）を通して行うテスト。
フレーキーテスト（flaky test）＝同じコード・同じ環境でも、走るたびに結果が変わってしまう不安定なテスト。E2Eで起きやすい現象。
UIテスト＝画面操作を伴うテスト。E2Eと重なる場面が多い。
05

なぜピラミッド型なのか

WHY PYRAMID

「下が広く上が狭い」三角形になる理由は、3つの軸（速度・コスト・壊れにくさ）が層ごとに変わるからです。底辺の単体テストは速くて安くて壊れにくい、頂上のE2Eは遅くて高くて壊れやすい。この性質の差が、自然とピラミッド型を要請します。

why pyramid 3つの軸で見る各層の特性
// 3つの軸 — 上に行くほど…
SPEED — 速度
UNIT 速いE2E 遅い
COST — 維持コスト
UNIT 安いE2E 高い
STABILITY — 安定性
UNIT 安定E2E 不安定
// なぜ下の層を厚くするのか ① 速い → 数千件でも数秒で回る → 毎コミット実行できる ② 安い → 維持コストが低い → 量を増やせる ③ 壊れにくい → 外部依存がないから → フレーキーになりにくい ↓ 結論: 単体テストを多く積むのが 経済的に最も合理的
この層構造の意義
ピラミッド型は「速く回せる検証を多く・遅い検証を少なく」という経済原理を視覚化したもの。地図編Vol.1のBoehm曲線（早期発見ほど安い）と同じ思想です。両方を重ねて理解すると、なぜ業界全体がこの形に向かったかが見えてきます。
06

各層の比率の理屈

RATIO BY LAYER

各層をどれくらいの比率で書けばいいのか。よく言われる目安は「単体70% / 結合20% / E2E 10%」です。これは絶対の数字ではなく目安ですが、根拠を理解しておくと自分のプロジェクトに合わせて調整できます。

ratio 層別比率の根拠
// 目安比率(7:2:1)
E2E約10%
INTEGRATION約20%
UNIT約70%

チーム・プロジェクトの性質により調整は必要。データ処理が多いなら単体多め、UIが複雑ならE2E多め、など。

// 比率を導く因果 何を最大化したいか → 「短い時間で多くの欠陥を見つける」 ↓ どの層が効率的か 単体 : 速度 × 数 = 高効率 結合 : 速度 × 数 = 中効率 E2E : 速度 × 数 = 低効率 ↓ 結論: 高効率な層を多く 低効率な層を最小限 = ピラミッド型 // 数字より「形」を覚えるのが大事
POINT
7:2:1は業界で多く語られる典型値であって、絶対の正解ではありません。重要なのは「下を厚く、上を薄く」という形そのもの。比率の数字を暗記するより、なぜその形になるかを理解する方が現場で役立ちます。

アンチパターン

ANTI-PATTERNS

ピラミッドが崩れた形には失敗の予兆があります。代表的なのは「逆ピラミッド」と「アイスクリームコーン」。形を見るだけで、そのプロジェクトのテスト戦略の問題点が分かります。

anti-patterns 崩れた形が示す危険信号
// NGパターン1 — 逆ピラミッド
E2E ばかり多い
INTEGRATION 少し
UNIT ほぼなし
// NGパターン2 — アイスクリームコーン
手動テスト 大量
E2E
INTEG
UNIT
// 逆ピラミッドの症状 ・テスト時間が異常に長い ・直してもすぐ別の所が壊れる ・テストがしょっちゅう失敗する ・原因特定に時間がかかる // なぜ起きるか 最初に画面テストから書く ↓ 単体テストを後回し ↓ 気付くと逆ピラミッドが完成 // アイスクリームコーンの症状 ・人手によるテストが膨大 ・リリース直前に1週間テスト工程 ・自動化したいが手が回らない // 共通する処方箋 底辺(単体)を厚くする方向に 少しずつ移行する
用語の整理
逆ピラミッド（inverted pyramid）＝E2E多め・単体少なめになっている崩れた形。テストが遅く・脆くなる。
アイスクリームコーン（ice cream cone）＝上に大きな「手動テスト」が乗った形。自動化の遅れを意味する。
テストハニカム / トロフィー型＝近年提唱される代替モデル。結合テストを厚くする発想。プロジェクト特性によりピラミッド以外の形が適する場合もある。
