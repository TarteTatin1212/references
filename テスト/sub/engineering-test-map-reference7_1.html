01

2つの視点

TWO PERSPECTIVES

ソフトウェアを検証する時、視点が2つあります。内側から見る視点（inside-out）と外側から見る視点（outside-in）。同じシステムを別の角度から眺めるこの2つの視点が、テストの粒度の本質的な違いを作ります。

two views 内側と外側、視点で世界が変わる
// 内側 vs 外側
INSIDE-OUT — 内側から
開発者の視点
↓
部品ごとに見る
↓
正しく実装されているか
vs
OUTSIDE-IN — 外側から
ユーザーの視点
↓
全体として見る
↓
正しい結果が得られるか

内側 = 部品が仕様通り作られているかを問う視点。
外側 = ユーザーが望む結果が得られるかを問う視点。

// 同じシステムを2つの視点で見る 内側からの問い: ・このクラスは仕様通りか? ・このメソッドは正しい値を返すか? ・このSQLは適切か? ・このAPIは要件を満たすか? ↓ 答える層 単体テスト, 結合テスト 外側からの問い: ・ユーザーは目的を達成できるか? ・画面は意図通り見えるか? ・操作の流れは破綻していないか? ↓ 答える層 E2Eテスト, UIテスト // テストの粒度の違いは // 視点の違いに対応している
POINT
地図編Vol.2のテストピラミッドは「粒度の上下軸」でしたが、別の見方では「視点の左右軸」でもあります。下層=内側、上層=外側。同じ階層構造を2つの軸で理解すると、テスト戦略がより立体的に見えてきます。
02

内側から見る検証 (単体・結合)

INSIDE-OUT VIEW

内側から見る検証は、「開発者がシステムを部品に分解した目線」です。コードの構造に沿ってテストを書きます。関数1つ、クラス1つ、サービス層、リポジトリ層。コードの理解者だけが書ける検証です。

inside-out 開発者の視点に立つ検証
// 内側視点の検証範囲
UI / 画面
INSIDE-OUT
API
Service
DB

ピラミッドの下部から中部にあたる層。コードに対応した粒度でテストを書く。

// 内側視点のテスト例 // 個別クラスの責務を確認 test("DiscountCalc#apply", () => { const calc = new DiscountCalc(); expect(calc.apply(1000, 0.1)) .toBe(900); }); // 内部構造に立脚した検証 test("OrderRepo がSQLでINSERTを発行", () => { service.place(...); expect(jdbc).toHaveExecuted("INSERT INTO orders"); }); // 「コードを書いた人にしか分からない」 // 視点の検証 // 観点: // ・各部品は仕様通りか // ・契約 (interface) を満たすか // ・境目で正しく繋がるか
用語の整理
ホワイトボックステスト＝コードの中身を見ながら設計するテスト。内側視点の典型。「実装の知識を前提とする」テスト。
構造ベーステスト＝同義。プログラムの構造（分岐・パス）に沿って設計する。
開発者テスト＝開発者が自分のコードに対して書くテスト。多くは内側視点。
03

外側から見る検証 (E2E・UI)

OUTSIDE-IN VIEW

外側から見る検証は、「ユーザーがシステムをひとつの箱として使う目線」です。中身は問わない、結果だけを見る。「目的を達成できるか」が唯一の関心事です。コードの内部を知らなくても書ける検証です。

outside-in ユーザーの視点に立つ検証
// 外側視点の検証範囲
OUTSIDE-IN
UI / 画面
API
Service
DB

ピラミッドの最上層にあたる層。中身を貫通して「結果」だけを見る。

// 外側視点のテスト例 // 内部構造に踏み込まない test("ユーザーが商品を購入できる", async () => { await page.goto("/shop"); await page.click("text=本"); await page.click("text=購入"); await page.fill("#card", "..."); await page.click("text=確定"); await expect(page.getByText("購入完了")) .toBeVisible(); }); // 観点: // ・ユーザーは何を見るか // ・操作は迷わずできるか // ・目的の結果に到達できるか // 内部の実装は知らない // 知らなくても書ける // → 利用者の立場で検証
用語の整理
ブラックボックステスト＝中身を見ずに「入力と出力」だけで検証するテスト。外側視点の典型。
振る舞いベーステスト＝同義。仕様で定義された振る舞いに沿って設計する。
受け入れテスト＝ユーザーや顧客の視点で「これで納品して良いか」を判定するテスト。外側視点の代表。
04

視点が変わると何が変わるか

CONSEQUENCES OF PERSPECTIVE

視点が変わるだけで、テストの性質が連鎖的に変化します。書きやすさ・速さ・壊れやすさ・実装変更への耐性。すべてが内側／外側で逆方向に振れます。これを理解しておくと、なぜE2Eが「遅くて脆い」のかが必然として見えます。

consequences 視点の違いが生む連鎖的な性質差
// 視点 → 性質の対応
観点内側 / 外側
速度速い / 遅い
必要部品少ない / 全部
失敗特定容易 / 困難
実装変更耐性低い / 高い
仕様変更耐性高い / 低い
ビジネス価値間接的 / 直接的

おもしろい点: 「実装変更耐性」と「仕様変更耐性」が逆。実装をいじっても外側テストは壊れない。仕様が変わると外側テストは壊れやすい。

// 実装変更 vs 仕様変更 実装変更: 関数を分割・統合 クラスをリファクタ SQLを最適化 ↓ 影響を受けやすい 内側テスト (詳細に依存) 仕様変更: 画面のレイアウト変更 項目の追加・削除 操作フローの変更 ↓ 影響を受けやすい 外側テスト (画面に依存) // テストはどちらかに脆い // 完全無敵のテストはない // → だから両方必要 // → だからピラミッド
この事実の意味
テストには「何に脆いか」のトレードオフがある。内側テストは実装の変更に脆く、外側テストは仕様の変更に脆い。どちらか片方だけだと盲点が生まれる。両方を組み合わせるからこそ全体が守られるのです。
05

E2Eが脆い構造的理由

WHY E2E IS FRAGILE

E2Eテストはなぜ壊れやすいのか。よく「不安定だから」「待ち時間が」と語られますが、もっと根本の理由は「依存する部品が圧倒的に多い」ことにあります。外側から見るということは、内側のすべてに依存するということです。

why fragile E2Eが壊れやすい構造的理由
// 依存の連鎖
  1. E2Eテストは画面・API・DB・外部すべてに依存する
  2. そのどれか1つでも変われば、テストに影響しうる
  3. 変更を加える機会は無数にある (人・時間・要因)
  4. 結果、テストが壊れる確率は単体テストより遥かに高い

単体テストの依存=1 〜 数個。E2Eの依存=数十〜数百。壊れる確率は依存数の積。

// 確率モデルで理解する 1つの部品が壊れない確率: 仮に 99% 単体テストの安定性: 依存 = 1部品 安定確率 = 0.99 ≒ 99%安定 結合テストの安定性: 依存 = 数部品 安定確率 = 0.99^5 ≒ 95%安定 E2Eテストの安定性: 依存 = 数十部品 + 環境 安定確率 = 0.99^50 ≒ 60%安定 // 個々の部品の質を上げても // 依存数が多いほど // 全体の安定性は下がる // これがE2Eが脆い構造的理由
この構造の意味
E2Eテストの脆さは「気をつければ直る」性質のものではない。依存が多いという構造そのものの帰結。だから本編Vol.7で見た「フレーキー対策」は「壊れにくくする」ではなく「壊れた時の被害を最小にする」方向の工夫が中心になります。「E2Eは少なく書く」が現場の知恵である理由がここにあります。
06

視点の混在というアンチパターン

MIXING PERSPECTIVES

視点の概念が分かると、テストで起きるアンチパターンの多くが「視点の混在」として説明できます。「内側でテストすべきものを外側で」「外側で確かめるべきものを内側で」やってしまう失敗です。

mixing 視点の使い分けを誤るアンチパターン
// 2つの典型的混乱
混乱A — 外側で詳細を見る
E2Eで全バリデーション網羅
↓
遅い・脆い・冗長
↓
単体でやればよかった
×
混乱B — 内側で全体を見る
単体でユーザーフロー検証
↓
実装の擬似に頼る
↓
E2Eでやればよかった

視点を間違えると、「適切な層で取れるはずの効果」を取り損なう。視点の使い分けこそテスト戦略の核。

// 混乱Aの実例 // E2Eで全部やろうとした失敗 test("メールが空でエラー表示", ...); test("メール形式不正でエラー", ...); test("メール長すぎでエラー", ...); test("パスワード空でエラー", ...); ... // 各テスト 30秒 // 50ケース = 25分 // 単体なら 数秒で同じ網羅 // → バリデーションは内側で // 混乱Bの実例 // 単体テストで「画面遷移」を検証 test("購入後に完了画面", () => { const mockRouter = mock(); service.complete(...); expect(mockRouter.push) .toHaveBeenCalledWith("/done"); }); // ルーターをモック // 本物のブラウザ遷移を試していない // → これはE2Eでやるべき
処方箋
「視点を意識して書く」のが処方箋。書きながら「自分は今、内側を見ているか / 外側を見ているか」を自問する習慣を持つと、適切な層を選べるようになります。これがテスト戦略の実践的な判断軸です。

役割分担の地図

DIVISION OF ROLES

最後に、視点という軸でテストの役割分担を整理します。「何を内側で・何を外側で」の判断が一目で分かる対応表。これを頭に入れておくと、新機能を作る時のテスト戦略がすぐ立てられます。

division of roles 何を内側で、何を外側で見るか
// 役割分担
検証したいもの適切な視点
計算ロジック内側 (単体)
バリデーション内側 (単体)
SQL・DB操作内側 (結合)
外部API連携内側 (結合)
画面の見え方外側 (UI)
ユーザーフロー外側 (E2E)
ハッピーパス外側 (E2E)
// 視点による役割分担の3原則 ① 内側で取れるものは内側で 速い・安定・大量に書ける ② 外側でしか取れないものは外側で ユーザー体験・全体の通り ③ 同じものを両方で書かない 時間と労力の無駄 ↓ 結果としてピラミッドが完成 外側 (E2E) 少数・厳選 ↑ 内側 (結合) 中程度 ↑ 内側 (単体) 大量 // 視点という軸で整理しても // ピラミッドが必然的に出てくる
この巻のまとめ
地図編Vol.2でピラミッドを「粒度」で、地図編Vol.6で「構造」で読み解き、地図編Vol.7では「視点」で読み解きました。同じ結論に3つの違う角度から到達する—これは知識が立体化している証です。テスト戦略を理解する上で、視点の左右軸を持つことは大きな助けになります。
