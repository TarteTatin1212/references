01

単体テストとは

WHAT IS UNIT TEST

単体テスト（ユニットテスト）とは、関数1つ・メソッド1つ・クラス1つといった「最小の部品」を単独で動かして検証するテストのことです。外部のDBやネットワーク・他のクラスには極力依存させず、対象だけを試します。

unit test 最小の部品を単独で検証する
// 単体テストの3条件
1. 範囲が狭い関数・メソッド単位
2. 速い1件あたりミリ秒オーダー
3. 独立している他のテストと干渉しない

単体テストは数千件あっても数秒で終わるのが理想。これがあるからこそ「何度も繰り返し走らせる」運用が成り立ちます。

// 単体テストの典型例 function discount(price, rate) { if (rate < 0 || rate > 1) { throw new Error("rate out of range"); } return price * (1 - rate); } // この関数だけを単独で試す test("10%引き", () => { expect(discount(1000, 0.1)).toBe(900); }); test("範囲外でエラー", () => { expect(() => discount(100, 2)) .toThrow(); });
POINT
「単体（unit）」の正確な範囲は議論が分かれますが、現場では「1つの関数 / 1つのクラスのpublicメソッド」を1単位とすることが多いです。DBやAPIに繋がない・外部要因に左右されない、これが単体テストの大事な条件です。
02

AAAパターン徹底

ARRANGE-ACT-ASSERT

Vol.1で触れたAAAパターン（Arrange-Act-Assert）を、しっかり身につけましょう。準備・実行・検証の3ブロックを視覚的に分けて書くことで、後から読む人が一瞬で意図を理解できます。これが現場の定番です。

AAA pattern 3ブロックを意識して書く
// 3ブロックを分ける
ARRANGEテスト対象・データを用意
↓ 空行を1つ入れる
ACT対象の処理を1回だけ呼ぶ
↓ 空行を1つ入れる
ASSERT期待結果を検証

ACTは1回だけが鉄則。1テストに複数のActionが入ると「何を検証しているか」が曖昧になります。

test("在庫を減らすと残数が更新される", () => { // --- Arrange --- const stock = new Stock(10); // --- Act --- stock.consume(3); // --- Assert --- expect(stock.remaining).toBe(7); }); // 別名: Given-When-Then // 構造は同じ、呼び方が違うだけ // Given = Arrange // When = Act // Then = Assert
POINT
AAAをきっちり分けたテストは自然にドキュメントになります。「このテストは何を準備し、何をして、何を確認しているか」が一読で分かるからです。これが品質の高いテストの第一歩。
03

テストの命名規則

TEST NAMING

テスト名は失敗した時に「何が壊れたか」を一瞬で示す役割を持ちます。「test1」「testAdd」のような曖昧な名前では役割を果たせません。現場で広く使われる命名パターンを覚えましょう。

naming 良い名前 = 失敗ログだけで原因が見える
// 3つの定番命名パターン
パターン1 / 日本語自然文「在庫が0の時、購入はエラーになる」
パターン2 / メソッド_条件_結果「purchase_stockZero_throwsError」
パターン3 / should〜when〜「shouldThrow_whenStockIsZero」

チーム内で1つに統一するのが鉄則。混在は読みづらさの元です。

// === 悪い例 === test("test1", () => { ... }); test("add", () => { ... }); test("正常系", () => { ... }); // → 失敗しても何が壊れたか不明 // === 良い例 === test("在庫0で購入するとエラー", ...); test("割引率が範囲外なら例外", ...); test("空文字列はfalseを返す", ...); // JUnit流 (Java) @Test void purchase_stockZero_throwsError() { ... } // RSpec流 (Ruby) — describe/contextで階層化 // describe '#purchase' do // context '在庫が0の時' do // it 'エラーになる' do ...
鉄則
テスト名を読むだけで「何をテストしているか」が分かれば合格。テスト100個並んだ時、名前一覧がそのまま仕様書になるのが理想形です。
04

アサーションの基本

ASSERTIONS

アサーション（assertion）とは「これが期待値です」と検証するための命令です。「等しい」「含む」「真である」「例外を投げる」など、フレームワークごとに豊富な種類が用意されています。代表的なものを押さえましょう。

assertions 期待値と実際値を比較する命令群
// 代表的なアサーション (Jest)
関数意味
toBe厳密な等価（===）
toEqualオブジェクトの中身が等しい
toBeTruthy真と評価される
toBeNullnullと等しい
toContain配列・文字列が要素を含む
toThrow例外を投げる
toBeGreaterThanより大きい
toHaveLength長さが一致
// 値の比較 expect(1 + 2).toBe(3); // オブジェクトの比較 // ※toBeだと別インスタンスなので失敗 expect({ id: 1, name: "X" }) .toEqual({ id: 1, name: "X" }); // 配列に含まれるか expect([1,2,3]).toContain(2); // 例外の検証 expect(() => divide(1, 0)) .toThrow("divide by zero"); // JUnit (Java) assertEquals(3, result); assertTrue(list.contains(item)); assertThrows(Exception.class, () -> ...); // pytest (Python) — assertだけ assert result == 3 assert item in list
ハマりどころ
toBe と toEqual の違いは初学者の定番の詰まりポイント。プリミティブ値（数値・文字列）は toBe、オブジェクトや配列は toEqual を使うのが基本ルールです。
05

テストの独立性

TEST INDEPENDENCE

すべてのテストは互いに独立していなければなりません。テストAの実行結果がテストBの結果を変えるような状態は最悪です。実行順序を変えても並列で走らせても、同じ結果が得られる。これが現場の絶対条件です。

independence テストは互いに干渉してはいけない
// 独立性を壊す典型例
NG — 状態を共有してしまう let cart = new Cart();
test("追加",()=>{
  cart.add(A); // ← 残る
})
test("空のはず",()=>{
  expect(cart.size).toBe(0); // 失敗
})
OK — 各テスト内で作り直す test("追加",()=>{
  const cart = new Cart();
  cart.add(A);
})
test("空",()=>{
  const cart = new Cart();
  expect(cart.size).toBe(0); // OK
})
// 独立性が壊れる4つの要因 ① モジュールスコープの変数を共有 → describe外でlet宣言は危険 ② 外部DB・ファイルへの書き込み → 残ったデータが次のテストに影響 ③ シングルトン・静的フィールド → 状態がテスト間で持ち越される ④ 実行順序への暗黙の依存 → 「Aの後にB」前提のテストは禁 // 解決の鉄則 テスト関数の中で全て用意して テスト関数の中で使い捨てる
なぜ独立性が大事か
独立していないと、テストが失敗しても原因がそのテストか前のテストか分からなくなります。さらに並列実行ができないので時間も食う。良いテスト=独立したテスト、これは譲れない条件です。
06

セットアップとティアダウン

SETUP & TEARDOWN

同じ準備を毎テストで書くのは冗長です。フレームワークには「各テストの前後に自動で実行される」仕組みが用意されています。これを使えば、独立性を保ちながら準備の重複を防げます。

setup hooks 前処理・後処理を共通化する仕組み
// 4つのフック
フック実行タイミング
beforeAllすべてのテストの前に1度
beforeEach各テストの直前ごと
afterEach各テストの直後ごと
afterAllすべてのテストの後に1度

重い準備（DB接続など）はbeforeAll、状態のリセットはbeforeEach、と使い分けます。

describe("カート機能", () => { let cart; beforeEach(() => { // 各テスト直前に新しく作る cart = new Cart(); }); afterEach(() => { // 各テスト直後に後片付け cart.clear(); }); test("追加できる", () => { cart.add({ id: 1 }); expect(cart.items.length).toBe(1); }); test("初期状態は空", () => { expect(cart.items.length).toBe(0); }); }); // JUnit (Java) の対応 // @BeforeEach @AfterEach // @BeforeAll @AfterAll
TIP
beforeEachは使いすぎ注意。前処理が複雑になるとテストを読むだけでは挙動が掴めなくなります。「準備が複雑＝テストしづらい設計」のサインかもしれません。
07

パラメタライズドテスト

PARAMETERIZED TEST

同じ検証ロジックを入力値だけ変えて何度も流したい場面はよくあります。これを「パラメタライズドテスト」と呼びます。テーブル形式で入出力を並べ、1行ごとに1テストが自動生成されるイメージです。

parameterized 複数ケースを表形式でまとめて検証
// 同じ検証を多数の入力で
入力 price入力 rate期待値
10000.1900
5000.2400
2000200
10001.00

表で書くと仕様が一目で見える。レビューもしやすい。境界値テスト（Vol.5）と相性抜群です。

// Jest — test.each test.each([ [1000, 0.1, 900], [500, 0.2, 400], [200, 0, 200], [1000, 1.0, 0], ])("discount(%i, %f) = %i", (price, rate, expected) => { expect(discount(price, rate)) .toBe(expected); } ); // JUnit (Java) — @ParameterizedTest @ParameterizedTest @CsvSource({ "1000, 0.1, 900", "500, 0.2, 400", }) void discountTest(int p, double r, int e) { assertEquals(e, discount(p, r)); }
POINT
パラメタライズドテストは境界値テスト・同値分割（Vol.5で学習）と組み合わせるのが定番。仕様の一覧表をそのままテストにできるので、現場で多用されます。
08

失敗メッセージの読み方

READING FAILURES

テストが失敗した時、フレームワークは「期待値 vs 実際値」を分かりやすく表示してくれます。この読み方を覚えれば、原因特定の速度が飛躍的に上がります。初学者がまず慣れるべき第一歩です。

failure output 失敗ログの解釈
// 典型的な失敗メッセージ
✗ 在庫を減らすと残数が更新される

  Expected: 7
  Received: 10

  at stock.test.js:14:33

// 解釈
// 期待: 7 / 実際: 10
// 14行目33文字目で失敗

ポイントは3つだけ: ①テスト名 ②期待値と実際値 ③ファイル・行番号

// 読み解きの手順 1. テスト名を見る → 何の動作を検証していたか 2. Expected と Received を比較 → 差分から「何が」ずれてるか 3. 行番号にジャンプ → そのテストの場所を確認 4. Arrange / Act を再読 → 入力データに問題は? 5. 実装側のコードを確認 → ロジックのバグを修正 6. 再実行 → ✓ PASS
超便利
多くのテストランナーにはウォッチモード（jest --watch等）があります。ファイル保存のたびに関連テストが自動再実行されるので、失敗→修正→再実行のサイクルが超高速になります。
