01

テストという仕事

WHAT IS TESTING

エンジニアリングテストとは、ソフトウェアが「期待通りに動くこと」を確認する作業です。単に画面を触って確かめるのではなく、検証手順そのものをコードとして書き残し、何度でも再実行できる形にします。これが「自動テスト」と呼ばれる現代の主流です。

testing 「動くはず」を「動いた」に変える作業
// 手動テストと自動テスト
手動テスト人が画面を操作して確認
↓ 毎回手間がかかる
自動テストコードが代わりに検証
↓ 一度書けば何度でも
本シリーズの対象自動テストを中心に学ぶ
// 期待: 1 + 2 は 3 になるはず function add(a, b) { return a + b; } // 自動テスト test("1 + 2 = 3", () => { expect(add(1, 2)).toBe(3); }); // 実行すると… // ✓ PASS 1 + 2 = 3
POINT
「テスト」と「動作確認」は別物です。動作確認はその場限り、テストは記録に残り何度も再利用されます。この違いがすべての出発点です。
02

なぜテストを書くのか

WHY WE WRITE TESTS

テストを書く理由は一つではありません。バグの早期発見・変更への安心感・仕様の記録という3つの価値が同時に得られます。書く手間以上のリターンがあるから、現場ではテストが定着しています。

value of testing テストが現場にもたらす3つの価値
// 3つの価値
1. 早期発見バグを書いた直後に気づける
2. 安心感コードを書き換えても壊れていないか即確認
3. 仕様書テストコード自体が「何ができるか」の説明になる
↓
開発速度を長期で上げる投資
// テストがない世界 // → コード変更 // → 画面で動作確認 // → 別の機能が壊れていないか心配 // → 全機能を手で再確認…ツライ // テストがある世界 // → コード変更 // → npm test (自動実行) // → ✓ 132 passed / ✗ 0 failed // → 壊れていない、安心して進める
TIP
「テストを書く時間がない」と感じる時こそ書くべきタイミングです。テストがない状態で機能を増やすと、いずれ変更が怖くて触れないコードが出来上がります。これを「レガシーコード」と呼びます。
03

テストピラミッド

TEST PYRAMID

テストは粒度によって大きく3つに分かれます。単体テスト・結合テスト・E2Eテストです。この3層を底辺の広いピラミッドの形で組み合わせるのが鉄則で、これを「テストピラミッド」と呼びます。

test pyramid 下が広く・上が狭いのが理想形
// ピラミッド構造
E2E
INTEGRATION
UNIT
UNIT（単体）: 関数1つを単独で検証。高速・大量。
INTEGRATION（結合）: 複数部品を組合せて検証。中速・中量。
E2E: 画面操作を通した全体検証。低速・少量。
// 各層の比較 速度 数 範囲 UNIT 速 多 狭い INTEG 中 中 中 E2E 遅 少 広い // 理想の比率（目安） UNIT : 70% INTEG : 20% E2E : 10% // 逆ピラミッドはアンチパターン // → E2Eばかり書いても遅くて壊れやすい
POINT
なぜピラミッド型にするのか・各層が具体的に何をするのかは、地図編Vol.2「テストの大きさを読み解く編」で構造として詳しく解説します。本編Vol.2以降で各層を順に学んでいきます。
04

自動テストの基本構造

AAA PATTERN

自動テストは言語やフレームワークが違っても、「準備 → 実行 → 検証」の3ステップで書く形が共通です。これを「AAAパターン（Arrange-Act-Assert）」または「Given-When-Then」と呼びます。最初に覚えるべき型です。

AAA pattern テストコードの黄金フォーマット
// 3ステップの構造
A — ARRANGE / 準備テスト対象や入力データを用意
↓
A — ACT / 実行テスト対象の関数や処理を呼び出す
↓
A — ASSERT / 検証結果が期待通りかチェック

別名:「Given-When-Then」。同じ構造を別の言葉で呼ぶだけです。

test("カートに商品を追加できる", () => { // === Arrange (準備) === const cart = new Cart(); const item = { id: 1, name: "本" }; // === Act (実行) === cart.add(item); // === Assert (検証) === expect(cart.items.length).toBe(1); expect(cart.items[0].name).toBe("本"); });
TIP
AAAパターンはほぼ全ての自動テストで使える共通フォーマットです。これを意識して書くと、テストコードが他人にも読みやすくなります。Vol.2でこの型を徹底的に身につけます。
05

主要なテストフレームワーク

FRAMEWORKS

自動テストは「テストフレームワーク」と呼ばれる支援ツールを使って書きます。言語ごとに定番が決まっており、本シリーズで扱う言語別の代表を押さえておきましょう。書き方の細部は違っても、考え方はどれも共通です。

major frameworks 言語ごとの「定番」を覚えれば十分
// 言語別 定番フレームワーク
  • JUnitJava★★★ Java界の事実上の標準
  • JestJavaScript★★★ Reactとも相性◎
  • VitestJavaScript★★ Jest互換で高速
  • pytestPython★★★ シンプルで人気
  • RSpecRuby★★ Rails開発の定番
  • Go testGo★★ 標準ライブラリ内蔵
  • XCTestSwift★★ Apple標準
// ── JUnit (Java) ── @Test void addTest() { assertEquals(3, add(1, 2)); } // ── Jest (JS) ── test("add", () => { expect(add(1, 2)).toBe(3); }); // ── pytest (Python) ── def test_add(): assert add(1, 2) == 3 // 書き方は違っても // 「準備→実行→検証」の型は同じ
POINT
どれか一つを深く学べば、他のフレームワークもすぐ理解できます。本シリーズでは主にJUnitとJestのコード例で説明しますが、考え方はそのままpytestやRSpecにも通用します。
06

はじめてのテスト

HELLO TEST

実際に1つだけテストを書いて、最初の感触をつかみましょう。「足し算が正しく動くか」を確認するテストです。たった数行ですが、これがエンジニアリングテストの原型です。

hello, test 最小単位のテストを動かしてみる
// 実行結果のイメージ
$ npm test

PASS ./add.test.js
add()
  ✓ 1 + 2 = 3 (2 ms)
  ✓ 負の数も計算できる (1 ms)
  ✗ 0との足し算

Tests: 1 failed, 2 passed
Time: 0.421 s

✓ がPASS（成功）、✗ がFAIL（失敗）。失敗があれば直して再実行します。

// add.js export function add(a, b) { return a + b; } // add.test.js import { add } from "./add"; describe("add()", () => { test("1 + 2 = 3", () => { expect(add(1, 2)).toBe(3); }); test("負の数も計算できる", () => { expect(add(-1, -2)).toBe(-3); }); });
POINT
describe でテストのまとまりを作り、test または it で1つのケースを書く。この入れ子構造はほぼ全フレームワークで共通の形式です。
