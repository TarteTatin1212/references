01

カバレッジとは

WHAT IS COVERAGE

カバレッジ（coverage、網羅率）とは「テストがコードのどれだけを実行したか」を測る指標です。書いたテストの「量」ではなく「届く範囲」を見える化します。複数の測り方があり、どれを使うかで意味も精度も変わります。

coverage テストが触れた範囲を測る指標
// カバレッジの種類
種類何を測る
ライン実行された行
ブランチ条件分岐の各方向
条件各条件式の真偽
MC/DC各条件の独立性 (厳格)
関数呼び出された関数
パス実行経路の全パターン

下に行くほど精度が高いが、達成も困難。実用上は「ライン+ブランチ」の組合せが定番。

// カバレッジ測定の流れ ① コードに計測コードを埋め込む (テストランナーが自動でやる) ② テストを実行する どの行・分岐が通ったか記録 ③ レポート生成 通った/通らなかった箇所を可視化 ↓ // 主なツール JaCoCo — Java Istanbul — JavaScript (Jestに同梱) coverage.py — Python SimpleCov — Ruby go cover — Go (標準) // 実行例 $ jest --coverage $ pytest --cov $ mvn test (JaCoCo)
POINT
カバレッジは「テストが弱い場所」を教えてくれる強力なヒント。ただし「カバレッジが高い=品質が高い」ではない点に注意。地図編Vol.8で「測る」とは何かを構造的に読み解きます。
02

ラインカバレッジ

LINE COVERAGE

ラインカバレッジは最も基本的で、最も使われる指標です。「コードの全行のうち、テスト実行で通った行は何%か」を測ります。シンプルで分かりやすく、すべてのカバレッジツールが必ず対応しています。

line coverage 実行された行の割合
// 色分け表示の例
SOURCE: discount.js
1│ function discount(p, r) { 2│ if (r < 0 || r > 1) { 3│ throw new Error("range"); 4│ } 5│ return p * (1 - r); 6│ }

緑=通った、赤=通っていない。
6行中4行通過 → ライン率 67%

// JSのカバレッジ出力例 % Statements Branches Functions Lines discount.js 67.0% 50.0% 100% 67.0% // テストが通った行と通らない行を // HTML/JSONで詳細レポート // HTMLレポート例 // coverage/index.html を開くと // → ファイル一覧 // → クリックで行レベル可視化 // CI連携で閾値チェック // jest.config.js { coverageThreshold: { global: { lines: 80, functions: 80, branches: 75, } } } // 閾値割れでビルド失敗にできる
POINT
ラインカバレッジは「測りやすさ」と「分かりやすさ」が魅力。ただしこれだけでは不十分なケースがあります。次に紹介するブランチカバレッジと組み合わせるのが定番です。
03

ブランチカバレッジ

BRANCH COVERAGE

ブランチ（分岐）カバレッジは「if文の真の側・偽の側、両方を通ったか」を測ります。ライン100%でも、分岐の片側しか試していないことは普通にあります。これを見抜くのがブランチカバレッジの仕事です。

branch coverage 条件分岐の両方向を網羅
// ライン100%でもブランチ50%
テスト: max(3,5)のみ
1│ function max(a, b) { 2│ if (a > b) return a; 3│ return b; 4│ }

全行通過しているが、「a>bがtrueの場合」が試されていない。ライン=100% / ブランチ=50%

ライン
100%
ブランチ
50%
// ブランチが見るもの if (a > b) { ... } // ブランチは2つ // ① a > b が true の場合 // ② a > b が false の場合 // 両方通って初めて 100% // 三項演算子も同じ return a > b ? a : b; // 2ブランチ // switch文 switch (kind) { case "A": ...; // ブランチ1 case "B": ...; // ブランチ2 default: ...; // ブランチ3 } // → 各caseを試さないと未達
POINT
「ライン100%だから安心」は典型的な錯覚。ブランチカバレッジまで見て初めて「分岐の取りこぼし」を発見できます。本気で品質を測るなら、最低でもブランチカバレッジまで見るのが現場の常識です。
04

条件カバレッジ・MC/DC

CONDITION & MC/DC

複合条件（&& や || で複数の真偽を組み合わせる式）は、ブランチカバレッジでも見落としが出ます。条件カバレッジとMC/DCはこの隙間を埋める指標。航空宇宙・医療など極めて高い信頼性が必要な現場で使われます。

condition coverage 複合条件の各要素を網羅
// 例: if (a > 0 && b > 0)
a>0b>0結果
truetruetrue
truefalsefalse
falsetruefalse
falsefalsefalse

条件カバレッジ: 各条件 (a>0, b>0) が両方の真偽を取ること
MC/DC: さらに「各条件が独立して結果に影響する」ことを確認

// ブランチカバレッジでは見落とす例 if (a > 0 && b > 0) { ... } // テスト1: a=5, b=5 → true // テスト2: a=-1, b=5 → false // ブランチカバレッジ 100% // (true側もfalse側も通った) // しかし条件カバレッジは不足 // → b≤0 のケースが試されていない MC/DC (Modified Condition / Decision Coverage): 各条件が単独で結果を決められる ことを示す厳格な指標 ・航空宇宙: DO-178B/C 規格 ・医療機器: IEC 62304 これらの規格で要求される // 一般のWebアプリでは過剰 // 命に関わるソフトで採用される水準
TIP
一般的なWebアプリでMC/DCは過剰です。「ライン+ブランチで十分」と覚えておけば現場の8割は問題ありません。MC/DCは「人命が関わるソフトウェア」の世界の話、と頭の片隅に置いておけば十分です。
05

カバレッジレポートの読み方

READING REPORTS

カバレッジツールはHTMLレポートを出します。ファイル単位の数字 → 行レベルの色分けと掘り下げられる構造。読み方の基本を覚えれば、品質改善のホットスポットがすぐ見つかります。

reading reports レポートから改善ポイントを読む
// カバレッジレポートのサマリ
SUMMARY
File%Stmts%Branch%LinesUncovered
order.js92.585.792.542-45
payment.js68.042.168.010-30
user.js95.088.095.067
cart.js100100100-
ALL87.372.487.3-

payment.jsが弱いと一目でわかる。10-30行が未カバー → ここから着手。

// レポートを読む3つの観点 ① 全体率より個別率 全体87%でも特定ファイルが 30%なら、そこが弱点 ② ライン率よりブランチ率 ブランチが大きく下回るなら 分岐の見落としあり ③ 重要度で重み付け payment.js が低い → 致命的 util.js が低い → 影響小 ↓ // 改善の優先順位を決める 最優先: ビジネス重要 + 低カバレッジ 後回し: あまり呼ばれない + 低カバレッジ // 数字を眺めるだけでは無意味 // 「重要なところが弱いか」を見る
POINT
レポートを毎日眺めるのは現実的ではありません。CIに組み込んで「閾値を割ったらビルド失敗」にすると自動で品質を守れます。さらに「PRで下がったら警告」のような差分チェックも便利です。
06

ミューテーションテスト

MUTATION TESTING

カバレッジの根本的な弱点は「実行しただけ」を測る点です。実行されても検証が甘ければ意味がない。これを解決するのがミューテーションテスト。「コードに変異を加えて、テストが気づくか」を測る究極の指標です。

mutation testing コードを壊してテストの真価を測る
// ミューテーションの仕組み
変異の例意味
> → >=境界をずらす
+ → -演算を変える
true → false真偽逆転
0 → 1定数を変える
削除文を消す

自動で数百〜数千の変異を作り、テストを走らせる。テストが落ちれば「気づいた=殺した」と言う。

// ミューテーションテストの実行 // 元のコード function isAdult(age) { return age >= 18; } // 自動生成される変異 Mutant 1: age > 18 Mutant 2: age <= 18 Mutant 3: age == 18 Mutant 4: age >= 19 ... // テストを走らせる // Mutant 1: 失敗を捕まえる ✓殺害 // Mutant 2: 失敗を捕まえる ✓殺害 // Mutant 3: 通ってしまう ✗生存 // Mutant 4: 失敗を捕まえる ✓殺害 // Mutation Score = 殺害率 // (殺害 / 全変異) // 生存した変異 = テストの穴 // 代表ツール // Stryker (JS/TS) // PIT (Java) // mutmut (Python)
超便利
ミューテーションテストは「ライン100%でも生存変異があれば、それはテストの穴」と教えてくれます。カバレッジでは見えなかった「実行はしているが検証していない」コードを暴く強力な道具。実行時間がかかるので、CIではなく週次/月次で回すのが現実的です。
07

適切な目標値

TARGET COVERAGE

「カバレッジは何%が良いか」は誰もが気になる疑問です。100%は現実的ではないし、追うべきでもない。プロジェクトの性質と限界費用効果で決まる、というのが現場の答えです。

target プロジェクトに応じた目標値
// 目安値
プロジェクトライン目安
趣味・実験気にしない
業務Webアプリ70〜80%
OSSライブラリ85〜95%
金融・決済95%以上
航空宇宙・医療MC/DC 100%

一般的なWebアプリでは「ライン80%+ブランチ70%」が現実的な落とし所。100%は手段の目的化。

// なぜ100%を目指してはいけないか 80% → 90%: 改善コスト = 中 バグ検出効果 = 中 → 投資対効果◎ 90% → 95%: 改善コスト = 高 バグ検出効果 = 小 → 投資対効果△ 95% → 100%: 改善コスト = 膨大 バグ検出効果 = ほぼゼロ → 投資対効果✗ ↓ // 最後の数%は // ・エラーハンドリング // ・到達困難な分岐 // ・自動生成コード // に消費される // その時間で別の検証をした方が // 全体の品質は上がる
鉄則
「カバレッジ100%を目指す」は宗教。手段が目的化した状態です。意味のないテストでカバレッジを稼ぐと、テストコードが負債になります。「重要な部分の検証密度」と「全体のカバレッジ率」の両立が現場の知恵。
08

カバレッジの罠

PITFALLS

最後にカバレッジ運用でよく陥る罠をまとめます。数字に振り回されると、テストの本来の目的を見失います。罠を知っておけば回避できます。

pitfalls カバレッジ運用のアンチパターン
// 4つの罠
罠1 — カバレッジ目当てのテスト 検証のない実行だけのテスト。assertが無い・弱い
罠2 — 100%強要 手段が目的化し、価値のないテストが量産される
罠3 — 数字盲信 80%の意味を理解せず、見せかけの安心感
罠4 — 重要度無視 コアロジックが30%でユーティリティが100%
// 罠1: 検証のないテスト // NG test("カバレッジ稼ぎ", () => { process(input); // → 実行はしている // → だが何も検証していない // → カバレッジは上がる // → 品質は上がっていない }); // OK test("正しく動く", () => { const result = process(input); expect(result).toEqual(expected); expect(db.find(...)).toBeDefined(); }); // 罠2の対策 // 強制せず推奨レベルに留める // 100%でなく80%目標 // 一律ではなくファイル別に // 罠4の対策 // 重要モジュールは高めの閾値 // 単なるDTOは閾値除外
結論
カバレッジは「便利な道具」だが「目的ではない」。テストの目的は「バグを見つける」ことで、カバレッジは目的に近づいているかを示す道しるべの1つに過ぎません。地図編Vol.8で「測る」という行為そのものの本質を読み解きます。
