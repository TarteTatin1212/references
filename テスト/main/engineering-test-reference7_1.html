01

E2Eテストとは

WHAT IS E2E

E2Eテスト（End-to-End test、エンドツーエンド）とは、ユーザーが画面を操作する流れを再現してシステム全体を検証するテストです。ボタンをクリック・フォームに入力・画面遷移・結果表示まで、人が行う一連の操作をプログラムで自動再現します。

E2E test 画面操作の一連の流れを自動化
// E2Eが触れるもの
領域触れるか
画面 / DOM○ 本物の画面を操作
JavaScript○ 実ブラウザで動かす
API / サーバ○ 通信が発生
DB○ 実DBが更新される
外部サービス場合による (擬似化が多い)

「ユーザーが操作する画面から、DBが更新されるまで」を貫いて検証する。本物に最も近いテスト層。

// E2Eテストの典型構造 test("ログインから商品購入まで", async () => { // 画面を開く await page.goto("/login"); // 入力 — 本物のフォームに入力 await page.fill("#email", "u@x.com"); await page.fill("#pass", "secret"); // クリック — 本物のクリックイベント await page.click("button[type=submit]"); // 画面遷移を待つ await page.waitForURL("/dashboard"); // 表示の検証 await expect(page.getByText("ようこそ")) .toBeVisible(); }); // 1つのテスト = 数秒〜数十秒 // 画面・API・DB全部を貫通
POINT
E2Eテストの強みは「ユーザー視点での確からしさ」。実際にユーザーが行う操作で、システム全体が正しく動くかを確認できます。単体や結合では絶対に見つからない「画面で見たら違和感」系のバグを捕まえる層です。
02

主要ツールの比較

E2E TOOLS

E2Eツールはここ数年で大きく変わりました。古参のSelenium・現代のPlaywright/Cypress。それぞれ思想と得意分野が違います。新規プロジェクトでは2025年現在Playwrightが人気の選択肢です。

tools 代表ツールの特徴比較
// 主要3ツール
ツール特徴
Selenium★★ 老舗。多言語対応。動作が遅め
Cypress★★★ JS界で人気。デバッグ体験が良い
Playwright★★★ Microsoft製。多ブラウザ・速い・新しい
Puppeteer★ Chrome専用。シンプル用途向き

2025年の新規採用ならPlaywrightが第一候補。多言語対応（JS/TS/Python/Java/C#）、自動待機、トレース機能など現代的な要素を備える。

// 各ツールの書き味 // === Selenium === WebDriver driver = new ChromeDriver(); driver.get("https://example.com"); driver.findElement(By.id("email")) .sendKeys("u@x.com"); // やや冗長 // === Cypress === cy.visit("/"); cy.get("#email").type("u@x.com"); cy.contains("ログイン").click(); // チェーン記法が独特 // === Playwright === await page.goto("/"); await page.fill("#email", "u@x.com"); await page.click("text=ログイン"); // シンプル + 自動待機
TIP
既存プロジェクトはSeleniumベースが多いです。「ある=Selenium、ない=Playwright」くらいの感覚で選んでよいでしょう。本書では現代的な書き味のPlaywrightを中心に説明します。
03

はじめてのE2E (Playwright)

FIRST E2E

Playwrightで実際に1つE2Eテストを書いてみましょう。題材は「ログイン → ダッシュボード表示」。シンプルですが、E2Eテストの基本構造をすべて含んでいます。

first E2E ログイン→ダッシュボードの最小フロー
// 操作する画面のイメージ
https://app.example.com/login

ログイン

  1. 1ログイン画面を開く
  2. 2メールとパスワードを入力
  3. 3ログインボタンをクリック
  4. 4ダッシュボードへ遷移するのを確認
  5. 5「ようこそ」が表示されている
// tests/login.spec.ts import { test, expect } from "@playwright/test"; test("ログイン成功", async ({ page }) => { // ① 画面を開く await page.goto("/login"); // ② フォームに入力 await page.getByLabel("Email") .fill("u@x.com"); await page.getByLabel("Password") .fill("secret"); // ③ ボタンクリック await page.getByRole("button", { name: "ログイン" }).click(); // ④ 画面遷移を待つ await page.waitForURL("/dashboard"); // ⑤ 表示の検証 await expect(page.getByText("ようこそ")) .toBeVisible(); }); // 実行: npx playwright test // → ブラウザが起動して自動で操作
超便利
Playwrightにはコード自動生成機能があります。npx playwright codegenで実際に画面を操作すると、対応するテストコードが生成されます。最初の取っ掛かりに最適です。
04

要素の特定方法

SELECTORS

E2Eで最重要かつ最ハマりどころが「要素の特定方法」です。「クリックするボタン」「入力するフォーム」を、どうやってプログラムから指し示すか。悪い特定方法を使うと、デザイン変更のたびにテストが壊れます。

selectors 要素を見つける方法と優先順位
// 優先順位 (Playwright推奨)
方法優先度
getByRole★★★ アクセシビリティ視点
getByLabel★★★ フォーム入力に最適
getByText★★ 表示文字で検索
data-testid★★ テスト専用属性
id / class△ 変更されやすい
XPath× DOM構造に依存。最悪

ユーザーが画面で識別する方法に近い順に選ぶのが鉄則。スタイル変更に強くなる。

// === 良い例 === await page.getByRole("button", { name: "購入する" }).click(); await page.getByLabel("メールアドレス") .fill("u@x.com"); await page.getByText("購入完了"); await page.getByTestId("checkout-btn").click(); // === 悪い例 === // CSSのクラス頼り (デザイン変更で壊れる) await page.click(".btn-primary.large"); // XPath (DOM構造に密着) await page.click("/html/body/div[2]/form/button"); // 表示文字の部分一致 (曖昧) await page.click("text=する"); // → これらは「変更耐性」が低い // → リファクタの度に壊れる
鉄則
「ユーザーが画面で識別する方法」でセレクタを書くのが現代の正解。アクセシビリティ属性（role/label）を優先する習慣をつけると、テストもアクセシビリティも同時に良くなる。一石二鳥です。
05

待機戦略

WAITING STRATEGY

E2Eでもっとも詰まりやすいのが「待ち」の問題です。ページ読み込み、API応答、アニメーション。画面が「準備できる」までの待ち時間は予測できません。これを正しく扱わないとフレーキーテストの温床になります。

waiting 非同期処理の待ち方
// 3種類の待ち方
方法性質
固定待ち× sleep(3000) 等。最悪
暗黙待ち○ ツールが自動で待つ (Playwright標準)
明示待ち○ 「特定条件まで待つ」を書く ★定番

固定待ちは絶対NG。短すぎると失敗、長すぎると遅い。条件ベースの待機が正解。

// === NG: 固定待ち === await page.click("button"); await page.waitForTimeout(3000); // NG // 3秒で終わる保証なし // 速くても3秒待つ // === OK: 暗黙待ち (Playwright) === await page.click("button"); await page.getByText("完了").click(); // → 要素が表示されるまで自動で待つ // → デフォルト30秒のタイムアウト // === OK: 明示待ち === await page.click("button"); // URL変化を待つ await page.waitForURL("/done"); // 特定の文言が出るまで待つ await expect(page.getByText("完了")) .toBeVisible({ timeout: 10000 }); // APIの応答を待つ await page.waitForResponse("/api/orders");
鉄則
sleep()やsetTimeout()はE2Eテスト最大の悪です。これを書いた瞬間、テストは脆くなります。「何かが満たされるまで」という条件で待つのが、唯一の正解。
06

Page Objectパターン

PAGE OBJECT

E2Eテストが増えてくると「同じ操作を書くのが煩雑」になります。Page Objectパターンは、画面の操作を1つのクラスにまとめる定番手法。テストコードと画面操作の知識を分離することで、保守性が大きく上がります。

page object 画面ごとに操作をクラス化する
// 改善の効果
BEFORE — 直書き 各テストで毎回
await page.fill("#email", ...)
await page.fill("#pass", ...)
await page.click("button")
↓
同じ操作の繰り返し
画面変更時に全テスト修正
AFTER — Page Object 画面の操作を1クラスに集約
await loginPage.login(email, pass)
↓
1つのメソッド呼び出しで完結
画面変更時はクラスだけ修正
// pages/LoginPage.ts export class LoginPage { constructor(private page: Page) {} async goto() { await this.page.goto("/login"); } async login(email: string, pass: string) { await this.page.getByLabel("Email").fill(email); await this.page.getByLabel("Password").fill(pass); await this.page.getByRole("button", { name: "ログイン" }).click(); } } // テスト側がすっきり test("ログイン成功", async ({ page }) => { const login = new LoginPage(page); await login.goto(); await login.login("u@x.com", "secret"); await expect(page).toHaveURL("/dashboard"); }); // 画面が変わった → LoginPageだけ修正 // → 数十のテストは無修正で動く
POINT
Page ObjectはE2Eテストにおけるリファクタリングの基本形。最初は直書きでよいですが、3つ以上のテストで同じ画面を触り始めたら導入を検討します。「保守性とテスト容易性」のために投資する価値があります。
07

フレーキーテスト対策

FLAKY TESTS

フレーキーテストとは「同じコードでも実行のたびに成否が変わる」不安定なテストのこと。E2Eで最も悩ましい現象です。原因を体系的に押さえ、それぞれの対策を知ることで、悪夢のフレーキー地獄を防げます。

flaky 不安定なテストの原因と対策
// 不安定の典型原因
原因対策
固定待ちの不足条件待ちに置き換える
テスト間の干渉各テストでデータを準備・破棄
並列実行で衝突テストごとに別ユーザー・別データ
外部APIの不安定WireMockで擬似化
時刻依存時刻を固定する
ランダムな順序ソートしてから検証
// フレーキー対策の実例 // NG: テスト間で同じユーザー test("登録1", ()=>register("佐藤")); test("登録2", ()=>register("佐藤")); // → 並列実行で重複エラー // OK: 一意なデータを使う test("登録", ()=>{ const name = "佐藤_" + Date.now(); register(name); }); // NG: テストの順序依存 test("作成", ()=>...); test("上で作ったものを更新", ()=>...); // → 順序が変わると壊れる // OK: 各テストで完結 test("更新", ()=>{ // テスト内で作成→更新→検証 }); // 再実行は最後の手段 // 「失敗したら2回まで再実行」は // 根本解決を後回しにする悪習
注意
フレーキーが放置されると「失敗してもとりあえず再実行」という文化が広まり、本当のバグも見落とされるようになります。フレーキーは見つけたらすぐ直すのが鉄則。「再実行3回まで許容」のような設定は、根本対策の代わりにしないこと。
08

E2Eを書く適量

HOW MUCH E2E

E2Eテストは強力だがコストが高い。すべての機能をE2Eで検証しようとすると、ピラミッドが崩れて遅く脆いテスト群が出来上がります。何をE2Eで検証すべきかの判断軸を押さえましょう。

how much E2Eを書く・書かないの判断
// E2Eで書くべきもの
カテゴリ例
クリティカルパスログイン・購入・支払い等
主要フロー登録 → 利用開始までの導線
UI依存の検証レイアウト・遷移・表示
結合の最終確認本番に近い形で動くか

目安は「失敗したら事業が止まるフロー」を中心に。細かい分岐や境界値はE2Eには向かない（単体や結合に任せる）。

// E2Eで書かないもの × 細かいバリデーション → 「メール形式エラー」「文字数制限」 → 単体で書く方が速い × エッジケースの大量検証 → 同値分割・境界値は単体で × 個々のUIコンポーネント → Storybook + コンポーネントテストで × 計算ロジック → 純粋関数として切り出し単体で // E2Eで書くもの ○ 「ユーザーが一連のことを達成できる」 ○ 「画面遷移が破綻していない」 ○ 「主要なジャーニーが動く」 // 目安: 全体テストの 10%以下 // 数件〜数十件くらい
結論
E2Eは「最後の砦」として最小限に。「すべての機能を画面操作で網羅する」のはアンチパターンです。各機能のバリエーションは単体・結合で網羅し、E2Eは「ユーザーが達成したい主要ゴール」に絞るのが現場の知恵。地図編Vol.7で「視点の違い」を読み解くと、この棲み分けの理由がクリアになります。
