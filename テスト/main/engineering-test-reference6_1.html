01

結合テストとは

WHAT IS INTEGRATION TEST

結合テスト（integration test）とは、複数のクラス・モジュール・サービスを組み合わせた状態で動作を検証するテストです。単体テストでは個々の部品の正しさを確かめましたが、組み合わせた時の振る舞いはまた別問題です。「境目で起きるバグ」を捕まえる層です。

integration test 複数部品の境目を検証する
// 単体 vs 結合
観点単体結合
対象1部品複数部品
外部切り離す本物を含むことも
速度ミリ秒0.1〜数秒
見つけるものロジックバグ境目のバグ
典型対象関数サービス+DB

単体は「ロジックの正しさ」、結合は「組み合わさった時の正しさ」を見る。役割が違います。

// 結合テストで初めて見つかる例 // 単体テストでは全部 ✓ service.create() ✓ // 単体OK repo.save() ✓ // 単体OK mailer.send() ✓ // 単体OK // ところが結合させてみると… test("ユーザー登録の一連の流れ", () => { service.register("佐藤", "x@y.z"); // 期待: DBに保存されている const u = repo.findByEmail("x@y.z"); expect(u).toBeDefined(); }); // ✗ FAIL // 原因: service内でemailを小文字化 // しているが、repo側は大文字保存 単体ではどちらも通っていた 結合で初めて発覚するタイプのバグ
POINT
結合テストは「境目（インターフェース）の整合性」を確かめる層。データ形式の取り違え、null許容性のずれ、文字コードの違いなど、個別では正しい部品が組合せると壊れる典型バグはここで捕まえます。
02

統合範囲の決め方

SCOPE OF INTEGRATION

「結合」と一口に言っても範囲は様々です。2クラスの結合〜全システムの結合まで、どこをどう組み合わせるかで性質が変わります。範囲の選び方に正解はなく、目的に応じた使い分けが必要です。

scope どこまでを「結合」と呼ぶか
// 範囲のパターン
UI / 画面
A. 狭い結合
Controller
Service
B. 中程度
Repository
DB

範囲を広くするほど現実に近いが、遅くて壊れやすくなる。目的に合わせて選ぶ。

// 範囲別の典型 A. 2クラス結合 Service + Repository(モック) // 高速 / 限定的 B. サービス+DB Service + 本物DB(テスト用) // よく使う / 中速 C. API層〜DB Controller + Service + Repo + DB // 広い / リクエスト単位 D. 外部含む + 外部API (実 or 模擬) + メール + ストレージ // E2Eに近い // 現場で最多はB // 「サービス〜DB」
POINT
結合テストは「範囲を欲張ると遅く・脆くなる」のが鉄則。範囲を絞った小さな結合テストを多めに書き、E2Eに近い広いものは最小限に。これがピラミッドの「結合層」のあり方です。
03

DBとの結合テスト

DB INTEGRATION

現場で最も書かれる結合テストが「サービス + 本物のDB」の組み合わせです。SQLが正しいか、トランザクションが効くか、制約に引っかからないか。これらは本物のDBを使わないと検証できません。

DB integration 本物のDBを使った検証
// なぜ本物のDBが必要か
確かめたいことモックでは無理
SQL構文の正しさ本物が解釈しないとわからない
制約 (FK, UNIQUE)DBエンジン依存
トランザクションACID挙動の検証
性能・索引実データでないと意味なし
方言差PostgreSQL vs MySQLの違い
// 結合テストの典型 (Java) @SpringBootTest @Transactional // 各テストで自動ロールバック class OrderServiceTest { @Autowired OrderService service; @Autowired OrderRepository repo; @Test void 注文すると履歴に保存される() { // Arrange var userId = "u1"; var itemId = "i1"; // Act — 本物のDBに書く service.place(userId, itemId); // Assert — 本物のDBから読む var orders = repo.findByUser(userId); assertThat(orders).hasSize(1); assertThat(orders.get(0).itemId) .isEqualTo(itemId); } }
POINT
DB結合テストの絶対鉄則は「本番と同じDBエンジンを使う」こと。「テストはSQLite、本番はPostgreSQL」のような構成は、方言差で動作が変わるので避けます。次に紹介するTestcontainersがこの問題を解決します。
04

Testcontainers

TESTCONTAINERS

Testcontainersは「テストのたびにDockerでDBコンテナを立ち上げる」ライブラリです。本番と同じPostgreSQL/MySQLをテスト時に瞬時に起動し、終了時に破棄。環境差分の悩みを完全に消す現代的な解決策です。

Testcontainers 本物DBを使い捨てで起動
// 仕組みの全体像
テスト開始
▶
DockerでDB起動
スキーマ初期化
▶
テスト実行
テスト終了
▶
コンテナ破棄

実行のたびクリーンな本物DBが手に入る。共有DBでの汚染やローカル環境差の悩みから解放される。

// Testcontainers (Java) @Testcontainers @SpringBootTest class DbIntegrationTest { @Container static PostgreSQLContainer<?> db = new PostgreSQLContainer<>("postgres:16") .withDatabaseName("test") .withUsername("u") .withPassword("p"); @DynamicPropertySource static void props(DynamicPropertyRegistry r) { r.add("spring.datasource.url", db::getJdbcUrl); r.add("spring.datasource.username", db::getUsername); r.add("spring.datasource.password", db::getPassword); } @Test void 本物のPostgresで動く() { ... } } // 実行時に毎回 docker run postgres:16 // → 本番と同じ環境で検証
超便利
TestcontainersはDB以外にも使えます。Redis、Kafka、Elasticsearch、Selenium、果ては独自のDockerイメージまで。「外部依存があるならとりあえずTestcontainers」が今や定番。Java/Python/JS等多言語に対応しています。
05

外部APIとの結合

EXTERNAL API

外部API（決済・地図・SaaS）との結合は本物を使えないのが普通です。本物にアクセスすると料金がかかったり、レート制限に当たったり、安定しなかったり。代わりに「擬似サーバ」を立てる手法が使われます。

external API 擬似サーバで本物のHTTPやり取りを再現
// 4つのアプローチ
手法性質
WireMockJavaで人気の擬似HTTPサーバ
MSWJS界の擬似HTTP (Mock Service Worker)
nockNode.js用のHTTPインターセプト
Sandbox環境外部サービス提供のテスト用環境

ポイントは「HTTPの往復までは本物に近い」こと。クライアント側のシリアライズ・エラーハンドリングを実際に走らせて検証できる。

// WireMock (Java) の例 @Test void 外部APIから情報を取得する() { // 擬似応答を仕込む stubFor(get(urlEqualTo("/users/1")) .willReturn(aResponse() .withStatus(200) .withBody("{\"name\":\"佐藤\"}"))); // 本物のHTTPクライアントを使う var result = client.fetchUser(1); assertThat(result.name).isEqualTo("佐藤"); } // MSW (JavaScript) const server = setupServer( http.get("/users/1", () => HttpResponse.json({ name: "佐藤" }) ) ); // テスト側のコードは本物のfetch() // → HTTPの動作全体が検証される
POINT
擬似サーバを使うと「エラー応答」「タイムアウト」「不正なJSON」といった異常系も自由に再現できます。本物のサンドボックスでは困難な状況を作れるのが擬似サーバの利点。
06

結合テストの独立性

INDEPENDENCE

結合テストは外部状態（DB等）を持つため、独立性の確保が単体テストより難しくなります。テストAで書き込んだデータがテストBに影響しないように、明確な戦略が必要です。

independence DB状態の汚染を防ぐ
// 独立性確保の戦略
戦略説明
ロールバック各テスト後にトランザクション巻戻し ★定番
テーブルtruncate各テスト前後で全テーブル空に
DBごと再作成Testcontainersで新規起動
名前空間分離テストごとにユーザー/スキーマを分ける

現場で多用されるのは「ロールバック」。@Transactionalで囲むだけで実現できる手軽さが魅力。

// 戦略1: ロールバック (推奨) @SpringBootTest @Transactional // テスト後にロールバック class OrderServiceTest { ... } // 戦略2: truncate @BeforeEach void cleanDb() { jdbc.execute("TRUNCATE orders, users CASCADE"); } // 戦略3: Testcontainersで毎回起動 // 一番安全だが時間がかかる // テスト数が多いと辛い // アンチパターン @Test void 前のテストの結果を前提に() { // "test1で作ったuserを使う" 等 // → テスト順序に依存 // → 並列実行で壊れる // → 絶対NG }
鉄則
結合テストでも「各テストは単独で完結」が大原則。前のテストの状態を引き継いではいけません。独立性を犠牲にしてテストを速くするのは本末転倒です。
07

トランザクション戦略

TRANSACTION STRATEGY

結合テストでDBを使う時、トランザクション境界の扱い方がしばしば問題になります。テスト用のトランザクションと、業務コード内のトランザクションが干渉して予期せぬ挙動を引き起こすことがあります。

transactions テストとビジネスのトランザクション境界
// よくある落とし穴
パターン問題
テスト全体を1Tx本番のコミット挙動が再現できない
commit済を見るテスト後にデータが残る
Tx伝播の誤解REQUIRES_NEWで意図と違う挙動

特に「非同期コミット後の状態を検証」したい時は、@Transactionalでは見られないので別アプローチが必要。

// パターンA: 簡単・推奨 @Transactional @Test void test1() { service.create(...); // 同じTx内で検証 var u = repo.find(...); assertThat(u).isNotNull(); } // テスト終了でロールバック // パターンB: コミット動作も含める @Test void test2() { // テストにはTransactionalを付けない service.create(...); // 別Txで検証 (実コミット後を見る) var u = repo.find(...); assertThat(u).isNotNull(); // 手動でクリーンアップが必要 repo.deleteById(u.id); } // → 場面に応じて使い分け
TIP
多くの場合はパターンA（@Transactional付与）で十分です。ただし「実コミット後のイベント発火」「非同期処理の結果」を検証したい時はパターンBが必要。使い分けの判断力が現場で問われます。
08

結合テストの落とし穴

PITFALLS

結合テストは強力ですが、使い方を誤ると単体テストの代わりにしてしまう失敗が頻発します。よくある落とし穴と、それを避けるための原則を押さえましょう。

pitfalls 陥りがちな失敗パターン
// 4つの落とし穴
落とし穴処方箋
単体の代わり単体テストを十分書いた上で結合を追加
過剰な結合範囲を最小限に保つ
テスト時間爆発結合は数を絞る、並列実行
共有DB汚染Testcontainersで分離
// 結合テストを書く前のチェック ① 同じことを単体で検証できるか? → できるなら単体で書く ② 結合でしか見つからないバグか? → 「DBの制約」「SQL構文」「Tx」 → これらが対象なら結合が適切 ③ 結合範囲は最小か? → 余分な部品を含めない ④ 独立性は保たれているか? → ロールバックや初期化を仕込む // 結合テストは「結合でしか // 見つからないもの」だけに使う // ピラミッドの「中間層」として // 適切な分量を保つ
結論
結合テストは「単体では見えないものを見る」ための専用ツール。万能薬ではありません。地図編Vol.6で「部品から製品への組み上がり」という構造視点から、結合テストがピラミッドの中間に位置する理由を読み解きます。
