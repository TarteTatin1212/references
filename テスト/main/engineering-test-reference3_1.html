01

TDDとは何か

WHAT IS TDD

TDD（Test-Driven Development、テスト駆動開発）とは、「失敗するテストを先に書き、それを通すコードを書く」という手順を繰り返す開発手法です。1999年にKent Beckが体系化しました。実装→テスト という順序を、テスト→実装 に反転させる発想です。

test-driven development テストが実装を駆動する開発スタイル
// 順序の対比
従来
TDD
1. 実装を書く
1. テストを書く
2. テストを書く
2. 実装を書く
3. 動作確認
3. 改善する

「テストが 設計と実装を駆動する」というのがTDDの本質。テスト=検証の道具、ではなく、テスト=作り方の指針、という捉え方です。

// TDDの基本リズム ① 何ができれば良いかをテストで書く → まだ実装がないので失敗する ② テストを通す最小限のコードを書く → テストが成功する ③ 重複や読みづらさを整える → テストは通ったままにする ④ ①へ戻る (次の小さな機能) // このリズムを1サイクル数分で回す // 1日に何十回も回るのが理想
POINT
TDDはテストを書く技法ではなくコードを書く技法です。「テストの書き方が上手くなる」のではなく、「テストを起点にすることで実装の質が上がる」のが目的。ここを取り違えると、TDDの効果は薄くなります。
02

Red-Green-Refactorサイクル

RED-GREEN-REFACTOR

TDDの中核となるのがRed→Green→Refactorの3ステップサイクルです。Red＝失敗、Green＝成功、Refactor＝整理。この3つを必ずこの順番で、何度も繰り返すのがTDDの作法です。

RGR cycle TDDの心臓部となる3ステップ
// 3つの色とサイクル
RED失敗
▶
GREEN成功
▶
REFACTOR整理

Red = テストランナーが赤く失敗を示す状態。
Green = 緑のPASSが並ぶ状態。
Refactor = テストを通したままコードを整える。

1失敗するテストを書く
2最短で通る実装を書く
3テストを通したまま整理
→ もう一周、別の小さなテストへ
// ── Step 1: RED ── test("add(1,2)=3", () => { expect(add(1, 2)).toBe(3); }); // → add未定義 → ✗ FAIL (Red) // ── Step 2: GREEN ── function add(a, b) { return 3; // 最短で通す } // → ✓ PASS (Green) // ── Step 3: REFACTOR ── function add(a, b) { return a + b; // 本来の形へ } // → ✓ PASS のまま (Green維持) // 次のテストへ: // add(-1, -2) など
鉄則
Greenが先、Refactorは後。動かないコードを綺麗にしようとしてはいけません。「とにかく通す→そのあとで整える」という順序を守ることで、常に動く状態を保ちながら進めます。これがTDDの安全網です。
03

ベイビーステップの原則

BABY STEPS

TDDでは歩幅を極端に小さくします。これを「ベイビーステップ」と呼びます。「数行書いて即テスト」を繰り返すスタイル。一見遠回りに見えますが、常に動く状態を保てるので、結果的に手戻りが減り早く進めます。

baby steps 小さく進む方が速い、というパラドクス
// 大きな歩幅 vs 小さな歩幅
大きな歩幅
ベイビーステップ
機能をまるごと実装
1機能を細かい単位に分割
最後にテスト
各単位ごとにテスト
失敗時に原因不明
失敗時は直前変更が原因
デバッグ大変
デバッグ最小

サイクル1回あたり数分〜10分が目安。これより長いなら、ステップが大きすぎるサイン。

// ベイビーステップで進める例 // 「FizzBuzz」を作る場合 [1] 1を渡したら "1" を返す [2] 2を渡したら "2" を返す [3] 3を渡したら "Fizz" を返す [4] 5を渡したら "Buzz" を返す [5] 6を渡したら "Fizz" [6] 10を渡したら "Buzz" [7] 15を渡したら "FizzBuzz" // 各 [N] でRed-Green-Refactorを1周 // → 1周3〜5分 // → 7周で完成 // 「全部一気に作る」よりずっと安定
超便利
ベイビーステップを徹底すると、失敗した時の原因が必ず「直前の変更」に限定されます。デバッグの探索範囲が極小化されるので、結果的に時間効率が上がる。これがTDDの「遠回りに見えて近道」というパラドクスの正体です。
04

三角測量

TRIANGULATION

三角測量（Triangulation）はTDDの中核技法のひとつ。具体例を複数並べることで、一般化された実装を導き出す方法です。1ケースだけだと「即値返し」で通ってしまいますが、複数ケースを並べることで自然と汎化が要求されます。

triangulation 複数の具体例から一般則を導く
// 三角測量の流れ
11つ目のテストを書く
2即値を返して通す（Fakeする）
32つ目のテストを書く
4即値では通らなくなる
5仕方なく一般化したコードを書く

2点が決まれば直線が引ける、3点が決まれば曲線が見えてくる。具体例を増やすことで実装の輪郭が浮かび上がります。これが三角測量の発想です。

// ── 1点目 ── test("add(1,2)=3", () => expect(add(1,2)).toBe(3)); // 実装(Fake) function add(a, b) { return 3; } // ✓ PASS (でも明らかに嘘) // ── 2点目 ── test("add(2,3)=5", () => expect(add(2,3)).toBe(5)); // 即値では通らない → 一般化が必要 function add(a, b) { return a + b; // 仕方なく汎化 } // ✓ PASS / ✓ PASS // このとき初めて「一般則」が見える
POINT
三角測量は「最初から汎用コードを書こうとしない」ための装置でもあります。具体から始めて一般化させるので、過剰設計（オーバーエンジニアリング）を防げます。
05

リファクタリングの自信

CONFIDENT REFACTORING

リファクタリングとは「振る舞いを変えずにコードを綺麗にする」作業のことです。テストがあれば、リファクタリングしても壊れていないか即確認できます。TDDで積み上げたテスト群は、後から振り返ると変更への安全網として機能します。

refactoring テストがある世界の特権
// リファクタリングの定義
DEFINITION外から見た振る舞いを変えずに、内部構造を整えること
REQUIRESテストがGreenな状態で開始
PROCEEDS小さく変更→テスト→変更→テスト
ENDSテストが通ったまま終わる

テストがない状態でコードを書き換えるのはリファクタリングではなく「祈り」と言われます。動くかどうかを祈っているだけ、という意味です。

// リファクタリングの例 // === BEFORE === function tax(p, k) { if (k == 1) return p * 1.10; if (k == 2) return p * 1.08; return p; } // テスト ✓ ✓ ✓ 通っている // === AFTER === const RATES = { STANDARD: 0.10, REDUCED: 0.08, TAX_FREE: 0, }; function tax(price, kind) { return price * (1 + RATES[kind]); } // テスト ✓ ✓ ✓ 同じく通る // → 振る舞いを保ちつつ整形完了
POINT
TDDで積み上げたテストの価値は「将来のリファクタリングを可能にする」ことにあります。書いた瞬間の検証だけでなく、半年後・1年後に安心して変更するための保険として効いてきます。
06

TDDの3つの法則

THREE LAWS OF TDD

Robert C. Martin（通称Uncle Bob）は、TDDを純化した3つの法則を提唱しました。厳格に守ると窮屈ですが、TDDの本質を理解するには最良の指針です。

3 laws Uncle Bobによる厳格な定式化
// 3つの法則
LAW 1失敗するテストを書くまで、実装コードを書いてはいけない
LAW 2失敗を発生させる以上のテストを書いてはいけない（コンパイルエラーも失敗のうち）
LAW 3失敗中のテストを通す以上の実装を書いてはいけない

要するに「常にテストが先、常に最小限」。これを徹底するとサイクルが極めて短くなります。

// 3つの法則を破るとどうなるか 違反 1: テストなしに実装を書き始める → 後付けテストになりがち → 結局TDDではない 違反 2: あれもこれもテストに書く → 一気に大量の失敗 → どこから直すか分からない 違反 3: 失敗以上のコードを実装 → テスト範囲を超えた実装 → 検証されない部分が生まれる // 法則を守ると // → 自然にベイビーステップになる // → 自然に必要な実装だけになる
TIP
現場では厳格適用しないことが多いです。研修や練習では3法則を意識し、業務では「テストファースト＋ベイビーステップ」程度に緩めるのが現実的なバランスです。
07

はじめてのTDD実演

TDD HANDS-ON

実際の流れを一気に追ってみましょう。題材は「FizzBuzz」。1〜数字を渡して、3の倍数は"Fizz"、5の倍数は"Buzz"、両方は"FizzBuzz"を返す古典的な練習問題です。TDDのリズムが体に伝わるはずです。

FizzBuzz with TDD 7サイクルで完成させる
// サイクルごとの進化
11→"1"　即値で通す
22→"2"　文字列化に汎化
33→"Fizz"　3の分岐追加
45→"Buzz"　5の分岐追加
56→"Fizz"　倍数判定に汎化
610→"Buzz"　同上
715→"FizzBuzz"　優先順位調整

1サイクル≒3〜5分。7サイクル=約30分で完成。各段階でテストが必ずGreenを保つ。

// ──── サイクル1 ──── test("1→1", ()=>expect(fb(1)).toBe("1")); function fb(n) { return "1"; } // Green // ──── サイクル3 ──── test("3→Fizz", ()=>expect(fb(3)).toBe("Fizz")); function fb(n) { if (n === 3) return "Fizz"; return String(n); } // ──── サイクル7・最終形 ──── function fb(n) { let s = ""; if (n % 3 === 0) s += "Fizz"; if (n % 5 === 0) s += "Buzz"; return s || String(n); } // 全7テスト ✓ ✓ ✓ ✓ ✓ ✓ ✓
POINT
FizzBuzzのTDD演習は初学者の登竜門です。文章だけで分かった気にならず、実際に手を動かして7サイクル回してみてください。「テストが先」のリズム感は、書いて体得する以外に道がありません。
08

現場でのTDDの実情

TDD IN PRACTICE

現場では厳格なTDDを徹底するチームばかりではないのが実情です。状況に応じてグラデーションがあります。100%TDDをやるべきかは議論が分かれますが、「テストファースト」の意識を持つこと自体は普遍的に推奨されます。

in practice どこまでTDDをやるかの現実解
// TDD実践度のグラデーション
100% TDD常にテスト→実装の順を徹底
↑ 厳格
テストファースト新機能はまずテストから
並走テストと実装を行き来する
テストラスト実装後にテストを書く
↓ 緩い
テストなし動作確認だけ

多くの現場は「テストファースト〜並走」のあたり。場面で使い分けるのが現実的です。

// TDDが特に効く場面 ★★★ 仕様が明確なロジック (計算・変換・バリデーション) ★★★ リファクタリング前後 (テストが安全網に) ★★ バグ修正 (再現テスト→修正の順) ★★ ライブラリ・API設計 (使う側の視点で書ける) // TDDが効きづらい場面 △ UI / 見た目の調整 △ 試行錯誤中の探索的コード △ 仕様未確定の領域 // → 使い分けが大切
結論
TDDは万能ではないが、強力な道具です。すべてに適用しようとせず、効く場面で確実に使う。これが業務で長く付き合うコツです。「バグ修正は必ず再現テストから書く」だけでも、品質改善効果は絶大です。
