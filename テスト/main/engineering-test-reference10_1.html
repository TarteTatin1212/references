Q.01
QUESTION

「テストを書く時間がない」と言われます

A. ANSWER その認識は順序が逆です。テストを書かないから時間が足りなくなります。最初は確かに2割ほど遅く感じますが、3ヶ月後には逆転します。「全部書く」のではなく「重要な部分から書き始める」のが現実的なアプローチです。
no time 時間がないからこそ書く
// 段階的に始める手順
  1. 1新規コードだけ書く (古いのは触らない)
  2. 2バグ修正時は再現テストから書く
  3. 3主要な分岐だけまず網羅
  4. 4カバレッジ目標は設けない (最初)
  5. 53ヶ月後に変化を測る
// 「時間がない」の正体 実際にかかる時間: 実装: 1h テスト: 30min ← 一見余分 しかし後で起きること: リファクタリングで全テスト走らせる → 30秒で確認完了 テストなしでリファクタ → 全画面手動確認 = 2h 3ヶ月後の累計: テストあり: 開発時間 +20% テストなし: 開発時間 +50〜100% ↓ テストは時間を奪うのではなく 未来の時間を稼ぐ 投資
TIP
最初の3ヶ月はテストの効果を実感しにくいのが普通。3ヶ月以降に「テストがあって助かった」体験が積み重なります。諦めずに続けるのが最大のコツです。
Q.02
QUESTION

テストが多すぎて実行が遅い

A. ANSWER 並列化・分割・適切な粒度の見直しの3手で大半は解決します。CIで5分超えたら最適化のサイン。10分を超えたら「人が待たなくなる」分水嶺なので、本格的な対策を始めます。
too slow 3つの手で実行時間を短縮
// 短縮の3手
  1. 1シャーディング (4並列で1/3〜1/4)
  2. 2変更影響範囲のみ実行 (test-impact)
  3. 3E2E → 結合 → 単体への置換
  4. 4テスト前の準備をbeforeAllで共有
  5. 5遅いテストの個別最適化
優先順位 まず「並列化」
次に「分割」
最後に「個別最適化」
// 改善の典型例 Before: E2E 50ケース → 30分 結合 200ケース → 8分 単体 1000件 → 2分 合計 40分 After (シャーディング 4並列): E2E → 8分 結合 → 2分 単体 → 30秒 合計 8〜9分 ← 主要時間=E2E After (E2Eの一部を結合に置換): E2E 20ケース → 5分 結合 220ケース → 2分 単体 1000件 → 30秒 合計 5分 ← 大幅短縮 // 多くの場合 E2Eが過剰 // 下層に移すと劇的に速くなる
POINT
遅いと感じたら最初に確認すべきはピラミッドの形。E2Eが多すぎる場合がほとんど。「結合 or 単体に置き換えられないか」を1テストずつ問い直してみましょう。
Q.03
QUESTION

フレーキーテストが多発して困っている

A. ANSWER フレーキーは放置すると組織全体の感覚を麻痺させる最大の毒です。「再実行で消えるからOK」の文化が根付くと、本物のバグも見落とされます。必ず根治を目指す姿勢が現場文化の生命線です。
flaky 根治への手順
// 対処の順序
  1. 1フレーキーを検疫（quarantineタグ）
  2. 2本流のCIから外す
  3. 3原因分類: 待ち / 順序 / 環境 / 外部
  4. 4原因別に根治
  5. 5修正後に検疫解除
禁止 無条件のリトライ常用
retries: 5 のような設定
→ 根治を放棄する宣言
// 原因別の処方箋 原因A: 固定待ち NG sleep(3000) OK waitFor(...条件...) 原因B: テスト順序依存 NG 前テストの状態を前提 OK 各テスト内で完結 原因C: 共有データ衝突 NG 固定ID (user_1) OK 一意ID (user_Date.now()) 原因D: 外部API依存 NG 本物のAPI OK WireMockで擬似化 原因E: 時刻依存 NG new Date() OK 注入された時計を使う
鉄則
フレーキー対応のメトリクスを常時可視化すること。「先月フレーキーが30件 → 今月50件」という傾向が見えれば、組織として手を打てる。見えないものは管理できない。
Q.04
QUESTION

モックを使うべきか、本物を使うべきか分からない

A. ANSWER 判断基準は「対象がプロセスを越えるか」「副作用があるか」「不可制御か」の3つです。1つでも該当すればダブル化を検討。同じプロセス内の純粋なクラスなら、本物を使う方が自然です。
mock or real 判断基準の早見表
// 判断フロー
  1. Q1プロセスを越える? → YES なら検討
  2. Q2副作用がある? → YES なら検討
  3. Q3不可制御 (時刻等)? → YES なら検討
  4. Q4テストが遅すぎる? → YES なら検討
  5. →すべてNOなら本物を使う
迷ったら 「Stub」から始める
複雑になったらMock検討
動く実装が欲しい→Fake
// 判断の実例 外部API: ・プロセス越える ✓ ・不可制御 ✓ → ダブルにする DB: ・プロセス越える ✓ ・副作用あり ✓ → 結合テストなら本物 (Testcontainers) → 単体テストならダブル 同じプロセス内のクラス: ・プロセス越えない ・副作用ない → 本物を使う → DIで差し込めば十分 時刻 / ランダム: ・不可制御 ✓ → ダブル必須 → 「時刻を返す関数」を注入 メール送信: ・副作用あり ✓ → ダブル → 実送信は事故の元
TIP
過剰モックは設計の悪臭。1テストで4個以上Mockが要るなら、対象クラスが他に依存しすぎている可能性大。テストが書きづらい=設計が窮屈、というシグナルとして受け取りましょう。
Q.05
QUESTION

カバレッジは何%を目指すべきですか

A. ANSWER 一般的なWebアプリなら「ライン80% + ブランチ70%」が現実的な目安。ただし「目標値を組織で強制」しないのが重要です。グッドハートの法則どおり、目標化した瞬間に意味を失います。「観察対象」として使うのが正解。
coverage target プロジェクト別の目安
// プロジェクト別の目安
  1. A業務Webアプリ → ライン70〜80%
  2. BOSSライブラリ → 85〜95%
  3. C金融・決済 → 95%以上
  4. D航空・医療 → MC/DC 100%
  5. E個人プロジェクト → 気にしない
アンチパターン 一律「100%」を強制
→ 価値のないテストの量産
→ 全体の品質が下がる
// カバレッジとの健全な付き合い やるべきこと: ・閾値「下限」を設定 (例: 70%) ・PRで下がったら警告 ・重要モジュールに高い閾値 ・トレンドを見る やってはいけないこと: ・「100%でないとマージ不可」 ・チーム評価指標に使う ・assertのないテストを書く ・カバレッジ目当てのテスト ↓ 数字は鏡である 磨くべきは映る対象 // ライン80%は到達点ではない // 道半ばの観察ポイント // その先は質を上げる方向へ
POINT
カバレッジを上げるより、ミューテーション殺害率を上げる方が品質に直結します。「ライン80%・ミューテーション60%」を目指すと、量と質のバランスが整います。詳細は地図編Vol.8を参照。
Q.06
QUESTION

テストが書きづらいコードに直面した

A. ANSWER 「書きづらい」は設計のサインです。コードが悪いから書きづらい。リファクタリングしてからテストを書くのが王道。テスト容易性=良い設計と捉えると、改善の方向が見えてきます。
untestable 書きづらさの原因と対処
// 書きづらさの原因
  1. ①関数が大きすぎる → 分割
  2. ②依存が多すぎる → 整理 / DI
  3. ③内部状態が複雑 → 純粋関数に
  4. ④副作用とロジック混在 → 分離
  5. ⑤private多用 → 公開IFを再設計
// よくある「書きづらい」コード NG 大きな関数 + 副作用 function processOrder(id) { const u = db.find(id); const p = u.points * 0.1; db.update(u.id, ...); mailer.send(...); log.info(...); } // → テスト書こうとすると // モック4個・準備が膨大 OK 分離した設計 function calcPoints(u) { // 純粋 return u.points * 0.1; } function processOrder(deps, id) { const u = deps.repo.find(id); const p = calcPoints(u); deps.repo.update(u.id, p); deps.mailer.send(u, p); } // calcPointsは純粋関数なのでテスト楽 // processOrderはDI済みなので楽
POINT
「テストが書きにくい」と感じたら、その感覚を大事に。テストファースト（地図編Vol.3）の本質はここにあります。テストが書きやすい形を選ぶことで、自然と良い設計に近づきます。
Q.07
QUESTION

古いコード(レガシー)にテストを追加したい

A. ANSWER レガシーコードは「特性化テスト」から始めます。「正しい振る舞い」を知らなくても、「現状の振る舞い」を記録するテストを書き、それを安全網としてリファクタリングする手法。マイケル・フェザーズの古典『レガシーコード改善ガイド』が必読です。
legacy code レガシーへのテスト追加手順
// 攻略の手順
  1. 1変えたい場所を「シーム」と呼ぶ
  2. 2シームの周辺を特性化テストで固定
  3. 3テスト緑のまま小さくリファクタ
  4. 4依存を切り離せる形に整える
  5. 5本来のテストを追加
  6. 6特性化テストは役目を終えたら削除
// 特性化テスト (characterization test) // 古い関数 — 中身がよく分からない function mysteryCalc(x, y, z) { // 200行の謎ロジック } // 「現状の振る舞い」を記録する test("mysteryCalc(1,2,3)", () => { // 実際に動かして結果を記録 expect(mysteryCalc(1,2,3)).toBe(42); }); test("mysteryCalc(5,5,0)", () => { expect(mysteryCalc(5,5,0)).toBe(0); }); // ↑ 「正しい」とは言っていない // 「今こう動く」と言っているだけ // 安全網ができたので // 中をリファクタしても // 振る舞いが変わらないと確認できる
POINT
レガシーへのテスト追加は時間がかかります。1日10分でもよいので「触れた箇所にテストを残す」を続けると、半年後には景色が変わっています。「ボーイスカウト・ルール」(来た時より美しく)を意識しましょう。
Q.08
QUESTION

チームの他のメンバーがテストを書きません

A. ANSWER 「書け」と言うのではなく、「書かないとマージできない」仕組みにするのが現場の知恵です。文化を変えるのは難しいですが、構造を変えるのは比較的容易。CIで強制し、PRレビューで触れる、自分が手本を示すの3点セットが効きます。
team culture 構造でテスト文化を作る
// 文化醸成の3手
  1. 1CIで「テストなしの新機能は不可」
  2. 2PRテンプレに「テスト有無」欄
  3. 3レビューで必ず触れる
  4. 4自分のPRで模範を示す
  5. 5テストで救われた事例を共有
効くアプローチ 理屈で説得 → 効果薄
仕組みで強制 → 効果大
成功体験を語る → 文化が育つ
// 段階的な仕組み導入 第1段: 観察 カバレッジを計測するだけ まだ強制はしない 数値を見える化 第2段: 推奨 PRテンプレに「テスト追加した?」 レビュアーが必ず確認 第3段: 軽い強制 「新規ファイルは80%以上」 既存は緩く 第4段: 全面強制 「PRで下がったらマージ不可」 ブランチ保護で構造化 ↓ 急に上のレベルに飛ばない 反発を生まないペースで // 半年〜1年かけて文化を育てる
TIP
最強の説得材料は「テストのおかげで救われた」事例。リファクタリングで全テストが赤くなり、本番事故を未然に防いだ、といった具体的なエピソードがチームの意識を変えます。事例を意識的に集めて共有する習慣を持ちましょう。
Q.09
QUESTION

TDDを始めたいが、どう始めればよいか

A. ANSWER 最初は「全部TDD」ではなく「ロジックの濃い部分だけTDD」から始めます。計算・変換・バリデーションのような純粋関数が向いています。FizzBuzz・BowlingGame等の練習問題で体に手順を染み込ませるのが最短ルート。
starting TDD TDDの始め方
// 最初の30日
  1. Day1FizzBuzzをTDDで(本編Vol.3)
  2. Day2-3その他の練習問題を3個
  3. 1週目業務で純粋関数だけTDD
  4. 2週目サービス層もTDDで
  5. 3週目バグ修正は必ず再現テストから
  6. 4週目振り返って続ける範囲を決める
// TDDが向く・向かない場面 ★★★ 強く向く ・計算ロジック ・データ変換 ・バリデーション ・パーサ・フォーマッタ ・状態遷移ロジック ★★ 向く ・API定義・ハンドラ ・ビジネスルール ・バグ修正 ★ 向きづらい ・UI / 見た目の調整 ・探索的なプロトタイプ ・仕様未確定の領域 ↓ 最初は★★★に絞る 体得してから広げる // 1日30分のTDD練習を1ヶ月 // → リズムが体に入る
POINT
TDDは「読んで理解する」より「やって体得する」もの。FizzBuzlやBowlingGame、Tofuなどの古典的演習を実際に書いてみるのが最短。書いた手の感覚が後で効いてきます。
Q.10
QUESTION

テストの失敗メッセージが読み解けません

A. ANSWER まず「3つの情報だけ拾う」習慣を作りましょう。①テスト名 ②期待値と実際値の差分 ③スタックトレースの先頭行。これだけで原因の8割は特定できます。全部読もうとしないのがコツです。
reading failures 失敗ログを読み解く順序
// 読み方の5ステップ
  1. 1テスト名を読む(何の検証だったか)
  2. 2Expected vs Received を比較
  3. 3スタックトレース先頭行へ
  4. 4該当行のコードを確認
  5. 5該当テストのArrangeを再読
慣れのコツ 毎回同じ順で読む
1回目で全て分かろうとしない
差分が大きい時はメモ
// 典型的な失敗メッセージ ✗ 在庫を減らすと残数が更新される ↑① Expected: 7 Received: 10 ↑②差分 at stock.test.js:14:33 ↑③ここを見る at describe (...) // 解釈 // 「在庫が10残っている」 // → 減算ロジックが効いていない // → 14行目を確認 // → consume()内のバグ発見 // 全部読まなくてよい // ① テスト名 // ② 期待値/実際値 // ③ 場所 // この3つで8割解決
超便利
Playwright/Cypressにはスクリーンショット・ビデオ録画があります。E2E失敗時はログだけでなく「実際の画面」を見る習慣をつけると一目で分かる失敗が多い。trace: onを有効にしましょう。
