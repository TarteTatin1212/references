01

設計技法が必要な理由

WHY DESIGN MATTERS

プログラムの入力の組み合わせは事実上無限です。年齢を受け取る関数なら整数だけで数十億通り、文字列なら無限大。すべて試すのは不可能なので、「少数の代表ケースで多くを網羅する」技法が必要になります。

why design 全部試せない、だから設計する
// 全部試そうとすると…
入力組合せ数
boolean2通り
1〜100の整数100通り
int型 全体約42億通り
文字列事実上 無限
2引数の組合せ掛け算で爆発

1回のテストが1ミリ秒でも、42億通りで約49日。組合せが増えればさらに不可能になる。

// 全部試すは無理 → どうする? 考え方の転換: 全部試そうとしない ↓ 「同じグループに属する入力は 同じ振る舞いをするはず」 ↓ 各グループの代表者だけ試す ↓ これがテスト設計技法 // 設計技法の役割 ① 入力空間を意味あるグループに分ける ② 各グループの代表を選ぶ ③ バグが集中しやすい場所を狙う ④ 結果として少数で多くを網羅
POINT
テスト設計は「全網羅」ではなく「賢い絞り込み」。これがVol.5全体の通奏低音です。地図編Vol.5で「全部試せない」という前提から構造を読み解きますので、合わせて読むと技法の意義が腑に落ちます。
02

同値分割法

EQUIVALENCE PARTITIONING

同値分割法（equivalence partitioning）はテスト設計の基本中の基本。入力空間を「同じように振る舞うであろう」グループ（同値クラス）に分けて、各グループから1つだけ代表を選びます。これだけで膨大な入力を1〜数件のテストに圧縮できます。

equivalence partitioning 入力空間を仕分けて代表で試す
// 例: 年齢の同値分割

仕様: 0〜17歳=NG、18〜64歳=OK、65歳以上=シニア

クラス A0 〜 17
NG
クラス B18 〜 64
OK
クラス C65 〜
シニア

3クラスなので3ケースで代表検証。「12歳・35歳・75歳」を選べば全クラス網羅。実際にはマイナス値・小数・文字列等の不正値クラスもカバーする。

// 同値分割で書くテスト describe("年齢区分", () => { // クラスA: 0〜17歳 (NG) test("15歳はNG", () => { expect(classify(15)).toBe("NG"); }); // クラスB: 18〜64歳 (OK) test("35歳はOK", () => { expect(classify(35)).toBe("OK"); }); // クラスC: 65歳以上 test("75歳はシニア", () => { expect(classify(75)).toBe("シニア"); }); // 不正値クラス test("-1は例外", () => { expect(()=>classify(-1)).toThrow(); }); });
POINT
同値分割の絶対鉄則は「有効同値クラスだけでなく無効同値クラスも分ける」こと。正常系だけでなく、マイナス値・null・型違い等もクラスとして洗い出します。これを怠ると異常系の検証が抜け落ちます。
03

境界値分析

BOUNDARY VALUE ANALYSIS

境界値分析は「クラスとクラスの境界」に着目する技法です。バグは境界に潜む、というのが鉄則。17歳と18歳、64歳と65歳のような境目で切り替わる値を狙い撃ちで検証します。同値分割と組み合わせるのが定番です。

boundary values 境目の値を狙い撃ちする
// 境界値の見つけ方
17
(NG) 18
(OK) 64
(OK) 65
(シニア)

各境目で「境界の前後2点」を試す。17/18, 64/65。これだけで境界バグの大半を捕まえられる。

// 境界値テストの定番 // 仕様: 18歳以上64歳以下がOK test.each([ // 下限境界 [17, "NG"], // 境界の外 [18, "OK"], // 境界そのもの // 上限境界 [64, "OK"], // 境界そのもの [65, "シニア"], // 境界の外 // 極端な境界 [0, "NG"], // 最小値 [-1, null], // 最小値の外 ])("年齢 %i → %s", (age, expected) => { expect(classify(age)).toBe(expected); }); // 境界バグの典型 // if (age > 18) → 18歳がNGになる // if (age >= 18) → 正しい // この差を捕まえるのが境界値
なぜ境界にバグが集中するか
> と >= の取り違え、-1 や +1 のずれ、off-by-oneと呼ばれる古典的バグが境界で起きるからです。境界値テストは費用対効果が抜群です。
04

デシジョンテーブル

DECISION TABLE

複数の条件が組み合わさって結果が決まるロジックにはデシジョンテーブルが向いています。条件と結果を表形式で網羅し、抜けや矛盾を視覚的に発見します。仕様レビューにも使える強力な技法です。

decision table 複合条件を表で網羅する
// 例: 送料判定 (会員 × 金額)
条件 / ケース1234
会員であるYYNN
5,000円以上YNYN
結果送料無料送料500送料500送料800

条件2つ → 2² = 4ケース。すべての組み合わせを表に並べて、結果が抜けなく定義されているか確認。仕様の穴も見つかる。

// テーブルからテストへ test.each([ // 会員, 金額, 期待 [true, 6000, 0], // case1 [true, 3000, 500], // case2 [false, 6000, 500], // case3 [false, 3000, 800], // case4 ])("会員%s/%i円 → %i円", (member, amount, expected) => { expect(shipping(member, amount)) .toBe(expected); } ); // デシジョンテーブルの利点 // ① 仕様の網羅性が一目で分かる // ② 抜けがあれば表で気づく // ③ 矛盾も発見しやすい // ④ テストにそのまま落とせる
注意
条件が増えると2のN乗で爆発します。条件5つ=32ケース、条件10個=1024ケース。実用上は、関連の薄い条件は別テーブルに分けるか、後述のペアワイズ法と組み合わせます。
05

状態遷移テスト

STATE TRANSITION

オブジェクトが状態を持つ場合に向く技法です。ATM・申請ワークフロー・ゲームのキャラクター等、「今の状態と入力で次の状態が決まる」もの全てに使えます。状態と遷移を図にして網羅します。

state transition 状態と遷移を表・図で網羅
// 例: 注文のステータス遷移
現状態操作次状態
新規支払支払済
新規キャンセル取消
支払済発送発送済
支払済返金取消
発送済受取完了
取消—(終了)
完了—(終了)

表にすると不正遷移も発見しやすい。「発送済から取消可能?」「完了から再支払?」など、仕様に明記されていない遷移を洗い出せる。

// 正常遷移のテスト test("新規→支払→発送→完了", () => { const o = new Order(); expect(o.status).toBe("新規"); o.pay(); expect(o.status).toBe("支払済"); o.ship(); expect(o.status).toBe("発送済"); o.receive(); expect(o.status).toBe("完了"); }); // 不正遷移のテスト (これも重要!) test("完了から再支払は不可", () => { const o = ...; // 完了状態 expect(() => o.pay()).toThrow(); }); // 状態を持つコードは // 「許される遷移」と「許されない遷移」 // 両方を必ずテストする
POINT
状態遷移は「不正な操作で壊れない」ことの検証が最重要。「完了済を返金」「キャンセル済を再支払」など、ユーザーが想定外の操作をしても安全か、ここで確かめます。
06

ペアワイズ法

PAIRWISE TESTING

条件・パラメータが多すぎて全網羅が現実的でない時に使う、賢い絞り込み技法です。「すべてのペアの組合せだけは網羅する」という発想で、ケース数を爆発的に減らせます。経験的にバグの大半が2つのパラメータの相互作用で起きることから生まれました。

pairwise / all-pairs 2因子の組合せだけを網羅する
// 圧縮の威力
条件値の数
OS3 (Win, Mac, Linux)
ブラウザ4 (Chrome, FF, Safari, Edge)
言語3 (ja, en, zh)
ユーザー種別3 (Free, Pro, Admin)

全網羅 = 3×4×3×3 = 108通り。
ペアワイズ = 約 12通りに圧縮可能 (約9割削減)。

// ペアワイズの原理 前提: バグの多くは2つの因子の組合せで発生 3つ以上の組合せが原因のバグは 経験的に稀 ↓ 戦略: すべての「2因子の組合せ」を 少なくとも1度はカバーする // ツール // 手で組むのは無理 // 専用ツールで自動生成 // Microsoft PICT // AllPairs (Python等) // PictMaster (Excel) // 入力: 因子と水準 // 出力: 必要最小のケース表 // これをテストに流すだけ
超便利
ペアワイズはツールで自動生成するのが定番です。手作業は非現実的。「組合せ爆発」の救世主として、Webテスト・設定パターンの多いミドルウェアテスト等で愛用されます。
07

異常系・例外パスの洗い出し

ERROR PATHS

初学者がよく忘れるのが異常系のテスト。バグは正常系より異常系に潜むことが多いです。「想定外の入力」「想定外のタイミング」を意図的に洗い出す視点を持ちましょう。

error paths 異常系の体系的な洗い出し
// 異常系のチェック観点
カテゴリ例
null/未定義null, undefined, 空文字, 空配列
型違い数値期待に文字列、文字列期待に数値
範囲外負数、0、最大値超過
大量データ100万件、10MBの文字列
並行同時更新、競合
外部失敗API停止、タイムアウト、500エラー
重複同じIDで2回作成、二重送信
// 異常系テストの定番セット describe("ユーザー登録 異常系", () => { test("名前がnullで例外", () => { expect(()=>register(null)).toThrow(); }); test("名前が空文字で例外", () => { expect(()=>register("")).toThrow(); }); test("名前が長すぎて例外", () => { const long = "a".repeat(10000); expect(()=>register(long)).toThrow(); }); test("重複登録で例外", () => { register("佐藤"); expect(()=>register("佐藤")) .toThrow("duplicate"); }); test("DB接続失敗時の挙動", () => { repo.save = ()=>{throw new Error()}; expect(()=>register("X")).toThrow(); }); });
鉄則
1機能につき「正常系1〜2件 + 異常系3〜5件」が目安。異常系の方が多くなるのが普通です。「とにかく動く」だけのテストは不十分。「壊れない」「壊れた時の振る舞い」も検証するのが現場の品質基準です。
08

技法の組み合わせ方

COMBINING TECHNIQUES

技法は単独で使うより組み合わせるのが定番です。現場では「同値分割で大枠 → 境界値で狙い撃ち → 異常系を追加」という流れがほぼ毎回使われます。場面ごとの最適解を覚えましょう。

combining 技法の使い分けと組合せ
// 場面別の最適技法
場面使う技法
入力1つ同値分割 + 境界値
複合条件デシジョンテーブル
状態あり状態遷移
多因子ペアワイズ
全領域+ 異常系の洗い出し

同値分割+境界値はほぼ全ての関数に効く基本セット。これだけマスターしても現場の8割は対応できる。

// 典型的な組合せの実例 // 仕様: 商品割引 // 会員 & 5000円以上 → 20%引 // 会員 → 10%引 // 非会員 & 10000円〜→ 5%引 // それ以外 → 割引なし // STEP 1: デシジョンテーブル // (会員Y/N) × (金額帯3つ) = 6ケース // STEP 2: 境界値を追加 // 4999/5000, 9999/10000 // STEP 3: 異常系 // マイナス金額・null会員 // 合計 6 + 4 + 3 = 13ケース // 全網羅(無限)を13ケースに圧縮 // これが「賢い絞り込み」の威力
結論
テスト設計技法は「全部覚える必要はない」。同値分割と境界値だけ徹底できれば、現場の大半は戦えます。残りは必要に応じて引き出します。技法はあくまで道具、目的は「少ない数で品質を守る」ことです。
