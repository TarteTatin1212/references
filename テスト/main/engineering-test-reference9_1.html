01

CI/CDとは

WHAT IS CI/CD

CI（継続的インテグレーション）は「変更があるたびに自動でビルド・テスト」する仕組み、CD（継続的デリバリー／デプロイメント）は「テスト合格したら自動で配布・公開」する仕組みです。両者は連続した一つのパイプラインとして運用されます。

CI/CD 変更から本番反映までの自動化
// 全体のパイプライン
PUSHcommit
▶
CIbuild + test
▶
STAGING配備
▶
CD本番

人手はほぼ介在しない。コミットからリリースまで自動化されている状態が現代の標準。

// CI/CDの意味 CI = Continuous Integration 継続的インテグレーション 小さな変更を頻繁に統合 毎回ビルド&テストを自動実行 壊れた所はすぐ通知 CD = Continuous Delivery 継続的デリバリー リリース可能な状態を保ち続ける ボタン1つで本番反映できる CD (alt) = Continuous Deployment 継続的デプロイメント さらに進めて、本番反映も自動 合格したら勝手にリリース // 「テストが書かれている」は前提 // 「テストが回り続ける」が次のレベル
POINT
地図編Vol.1で「テストの自動化」が現代の必然と読み解きました。CI/CDはその実装形態です。書いたテストはCI/CDに乗せて初めて、毎コミットの安心感を生む「生きた検証」になります。
02

GitHub Actionsの基本

GITHUB ACTIONS

現代のCI/CDサービスの代表がGitHub Actionsです。GitHubリポジトリと統合され、設定ファイル1つで動き出します。他にもGitLab CI・CircleCI・Jenkinsなどありますが、考え方はほぼ共通。GitHub Actionsを押さえれば応用が利きます。

GitHub Actions 事実上の標準CIサービス
// 用語と構造
用語意味
Workflow1つのYAMLファイル単位
Job並列で動かせる単位
StepJobの中の1コマンド
Trigger起動条件 (push, PR等)
Runner実行するVM環境
Action再利用可能な部品

.github/workflows/ci.yml に置けば自動認識。

# .github/workflows/ci.yml name: CI on: push: branches: [main] pull_request: jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: "20" - run: npm ci - run: npm test - run: npm run build # これだけでCIが動く
超便利
GitHub ActionsはPublicリポジトリなら無料で使えます（実行時間制限あり）。テストを書いたらすぐCIに乗せる習慣が、現場では当たり前。OSS活動の入り口としてもおすすめです。
03

テストパイプラインの設計

PIPELINE DESIGN

CIパイプラインは「早く失敗する」ように設計します。軽いチェックを先に、重いテストを後に。途中で失敗したら即停止。無駄な実行を避けながら、問題を最速で見つけるのがコツです。

pipeline design 速く失敗するステージ構成
// 5ステージ構成の例
1Lint
▶
2Build
▶
3Unit
4Integ
▶
5E2E

速いものから順に実行。Lint 数秒 → Unit 数十秒 → E2E 数分。途中で落ちたら後ろは実行されない。

// 「速く失敗する」設計 [1] Lint (数秒) 構文・スタイル違反 ここで弾けば残り全部スキップ [2] Build (数十秒) コンパイル・型チェック ここで弾ければユニットも不要 [3] Unit Test (数分) 論理的なバグを検出 ここで弾ければ統合は不要 [4] Integration (数分) 境目のバグを検出 [5] E2E (数十分) 最終確認 ↓ 全てが PASS → デプロイへ // なぜこの順か // 安いものから先に走らせれば // 失敗時の総待ち時間が最小
鉄則
「早く失敗する（fail fast）」は CI設計の絶対原則。Lintで明らかな問題があるのに、わざわざE2Eまで走らせるのは時間の浪費。順序が品質と効率の両方を決めます。
04

並列実行とシャーディング

PARALLELISM

テストが増えるとCI時間が伸びていきます。並列実行でテストを分散させ、シャーディングで1つのジョブをさらに細分化することで、CI時間を劇的に短縮できます。

parallelism テストを分散して時間を短縮
// 並列実行のイメージ (4分散)
直列
↓ 4並列に分散
Shard 1
Shard 2
Shard 3
Shard 4

理論上は1/4の時間。実際は分散の準備時間で 1/3 程度。

# シャーディング (Playwright) jobs: test: strategy: matrix: shard: [1, 2, 3, 4] runs-on: ubuntu-latest steps: - run: | npx playwright test \ --shard=${{ matrix.shard }}/4 # 4つのVMで並列実行 # 1/4ずつのテストを担当 # Jestの場合 run: | jest --shard=${{ matrix.shard }}/4 # 結果は最後にマージ
TIP
並列化は「テストの独立性」が前提。互いに依存するテストは並列化できません。Vol.2で学んだ独立性が、ここで真価を発揮します。「独立性を確保したからこそ並列化できる」という連鎖です。
05

キャッシュとアーティファクト

CACHE & ARTIFACTS

CIは毎回まっさらな環境で実行されるため、依存パッケージのダウンロードに時間を取られるのが課題。キャッシュで繰り返しを省き、アーティファクトで実行結果を保存・受け渡しすることで、効率と可視性が大幅に上がります。

cache & artifacts 毎回の繰り返しを賢く省く
// キャッシュ対象
対象効果
node_modulesnpm ci を高速化
~/.m2Mavenの依存を保持
~/.gradleGradleの依存
~/.cache/pipPythonの依存
Docker layersイメージビルド

キャッシュキーはロックファイルのハッシュを使うのが定石。package-lock.json が変わったらキャッシュ無効化。

# 依存パッケージのキャッシュ - uses: actions/cache@v4 with: path: node_modules key: ${{ hashFiles('package-lock.json') }} # テスト結果をアーティファクトに - uses: actions/upload-artifact@v4 if: always() with: name: test-results path: | coverage/ test-results/ playwright-report/ # 失敗時もアップロード # → 後から確認できる
POINT
アーティファクトにはテスト結果・カバレッジレポート・スクリーンショット・E2Eのトレースなどを残します。失敗時こそ重要なので、if: always()を必ず指定しましょう。
06

失敗の通知と可視化

NOTIFICATIONS

CIは「失敗をすぐ伝える」仕組みがあって初めて価値を発揮します。失敗が放置されると、誰も赤を見ても気にしなくなり、CIは形骸化します。通知設計・バッジ・ダッシュボードを組み合わせて、現状が常に見える状態を作りましょう。

visibility 失敗を可視化して放置を防ぐ
// 通知と可視化の手段
手段使い方
Slack通知失敗時にチャンネルへ
メール担当者個別に
READMEバッジ常時ステータス表示
PR上の表示マージ前にチェック
ダッシュボード傾向の追跡

build passing build failing coverage 78%
README上にバッジで現状を一目で

# Slack通知 (失敗時のみ) - name: Notify on failure if: failure() uses: slackapi/slack-github-action@v1 with: payload: | { "text": "🔴 CI失敗", "branch": "${{ github.ref }}", "url": "${{ github.run_url }}" } # PR画面に表示するチェック on: pull_request: branches: [main] # ブランチ保護ルールと組合せ # → 「CIが緑じゃないとマージ不可」 # → 失敗放置を構造的に防ぐ
鉄則
「mainブランチは常に緑」を守るのが現場の聖域。赤いまま放置されたCIは、もう誰も信用しません。「赤になったら即直す」文化を作り、「マージ前に緑」のブランチ保護ルールで構造的に強制します。
07

フレーキーへの対処

FLAKY IN CI

本編Vol.7でフレーキーテストの対策を学びました。CI環境では本番サーバで初めて表面化するフレーキー問題があります。リトライ戦略・検疫・モニタリング、それぞれの使い分けを押さえましょう。

flaky handling CI上でのフレーキー対処
// 3つの対処レベル
段階方法
応急処置リトライで一時凌ぎ
隔離flakyタグで検疫 (quarantine)
根治テスト/プロダクトコードを直す

リトライは「最後の手段」。常用するとCIの信頼性が崩壊する。本来は根治が王道。

# リトライ設定 (Playwright) # playwright.config.ts retries: process.env.CI ? 2 : 0 # Jestの場合 # jest.config.js testRunner: "jest-circus/runner" jest.retryTimes(2) # フレーキー検疫 test.describe.configure({ tag: '@flaky' }) # 普段はスキップ、別ジョブで監視 run: | npx playwright test \ --grep-invert "@flaky" # → CIの安定性を保ちつつ # → 問題は別管理で解決を目指す
注意
リトライに頼ると「再実行で消えたから無視」という悪習が広まります。リトライ回数・再実行率は必ずメトリクスとして可視化し、増えてきたら根本対処に動く。これがCIを健全に保つ鉄則です。
08

CI/CDの育て方

GROWING CI/CD

CI/CDは最初から完璧を目指さず、少しずつ育てるもの。「とりあえずテストが動く」から始めて、徐々にステージを増やし、通知を整え、並列化を入れていく。段階的成熟のロードマップを持っておきましょう。

growing 段階的に成熟させていく
// 成熟度のロードマップ
Lv.1テストが動くだけ
Lv.2PR毎に自動実行
Lv.3ブランチ保護で強制
Lv.4カバレッジ計測・閾値
Lv.5並列化で高速化
Lv.6通知・可視化
Lv.7CDで自動デプロイ

一度に全部やろうとせず、下のレベルを固めてから次へ。

// 各レベルの判断基準 Lv.1〜2: npm test が動く → 自動化へ PRに連携 → これだけで価値大 Lv.3〜4: 「赤でマージ禁止」を仕組み化 カバレッジ計測を始める Lv.5: CIが5分超えたら並列化検討 シャーディング・キャッシュ Lv.6: 失敗の検知漏れがあれば通知強化 ダッシュボード導入 Lv.7: 自動デプロイは 「即時ロールバック可能」が前提 ↓ 急いでLv.7に行こうとしない 下が緩いと上が崩れる // 1段ずつ確実に
結論
CI/CDは「インフラ整備」です。テストを書く労力と同じくらい、CI/CDの整備にも投資する価値があります。「テストが書かれている」だけでなく「テストが走り続けている」状態こそが、現代開発の到達点。地図編Vol.9で「自動化の流れ」を時間軸で読み解きます。
