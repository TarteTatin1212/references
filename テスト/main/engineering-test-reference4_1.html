01

テストダブルとは

WHAT IS TEST DOUBLE

テストダブル（test double）とは、本物の代わりにテストで使う「偽物」の総称です。映画のスタント代役（stunt double）から取った名前で、Gerard Meszarosが命名しました。DB、外部API、時刻、ファイル、メール送信など、テストで本物を使うと困るものを置き換えます。

test double 本物の振りをする代役オブジェクト
// 何を「置き換える」のか
DB 実DBの代わりにメモリ上の偽物
外部API ネット越しの呼び出しを擬似応答に
時刻 「今」を固定値にする
ランダム 予測できない値を予測可能に
メール送信 実送信せず記録だけ
決済 実カード処理せず成功を装う
// 本物との置き換え図 テスト対象 ↓ 依存 本物のDB ← 遅い・準備が大変 本物のAPI ← ネット必要・課金 本物の時計 ← 結果が時刻依存 本物の決済 ← 実際に課金される ↓ 置き換え テスト対象 ↓ 依存 DBダブル ← 一瞬・どこでも動く APIダブル ← 自由に結果を作れる 時計ダブル ← 「2024-01-15」に固定 決済ダブル ← 成功 / 失敗を選べる
POINT
テストダブルは「テストできない」を「テストできる」に変える道具です。実際の業務コードはほぼ必ず何かに依存しているので、ダブルを使いこなせるかどうかでテスト可能な範囲が大きく変わります。
02

なぜダブルを使うのか

WHY USE DOUBLES

ダブルが必要な理由は4つの「困った事情」に集約されます。遅い・不安定・副作用・不可制御。本物がこのうち1つでも持っていれば、ダブルへの置き換えを検討します。

why 本物が持つ4つの困った性質
// 4つの困った事情
事情例
遅いDB問い合わせ、HTTP通信
不安定外部APIの調子、ネット切断
副作用本物のメール送信、本物の決済
不可制御現在時刻、ランダム値、エラー再現

特に「不可制御」が重要。エラーケースを試したくても、本物のAPIにエラーを返してもらうのは難しい。ダブルなら自由自在です。

// ダブルなしのテスト test("決済が失敗したらキャンセル", () => { service.order(...); // 失敗を起こすには… // 本物のカード会社にエラー // を返してもらう必要 // → そんなの無理 }); // ダブルありのテスト test("決済が失敗したらキャンセル", () => { payment.charge = () => { throw new Error("declined"); }; const result = service.order(...); expect(result.status).toBe("cancelled"); }); // 自分でエラーを起こせる // → テストできる範囲が広がる
POINT
ダブルの真価は「ハッピーパス以外を試せること」。正常系だけでなく、エラー・タイムアウト・部分失敗を自由に再現できるのが最大の価値です。
03

5種類の分類(Meszaros)

FIVE TYPES

テストダブルには5つの種類があります。Meszarosの分類が業界標準です。実は「Mock」という言葉が広く使われがちですが、厳密にはMockは5つのうちの1つでしかありません。役割が違うので、適切に使い分けると有用です。

5 types Meszarosによる5分類
// 5種類の比較
種類役割
Dummy引数を埋めるだけの空っぽ。中身は使われない
Stub決まった値を返す。問い合わせ系
Spy呼び出しを記録する。後から確認
Mock「こう呼ばれるはず」を事前定義して検証
Fake軽量な動く実装。例: メモリ上DB

「Mock」は本来狭い意味だが、現場では総称的に使われがち。フレームワークが jest.fn() や @Mock を「mock」と呼ぶ影響もあり。

// 比較イメージ Dummy: const u = new User(null, null); // nullは使われないがコンパイル用 Stub: repo.find = () => return { id:1 }; // 問い合わせると決まった値を返す Spy: const spy = jest.fn(); ...あとで... expect(spy).toHaveBeenCalled(); Mock: expect(notifier) .toReceive("send") .with("hello"); // 事前に期待を定義 Fake: class InMemoryRepo { // メモリ上に動く実装 }
注意
現場ではこの5分類を厳格に使い分けるチームは少なく、「Stub」と「Mock」だけは区別するのが現実的なライン。残り3つは「ダブルの一種」と理解しておけば十分です。
04

Stubの使い方

USING STUB

Stub（スタブ）は「決まった値を返す」偽物です。テスト対象が情報を「問い合わせる」相手として使います。最も使用頻度が高く、まずこれから慣れるのがオススメです。

stub 問い合わせに決まった答えを返す
// Stubの典型構造
テスト対象→Stub: repository
「IDを渡してユーザー情報を取得」を問い合わせる
Stub→テスト対象
あらかじめ仕込んだUser{name:"佐藤"}を返す

ポイントは「返り値」を制御すること。呼ばれたかどうかは気にしない。

// Jest — 単純なStub test("ユーザー名を表示する", () => { const repo = { find: () => ({ name: "佐藤" }), }; const view = new UserView(repo); expect(view.render(1)) .toBe("こんにちは佐藤さん"); }); // jest.fn().mockReturnValue で楽に const repo = { find: jest.fn().mockReturnValue({ name: "佐藤", }), }; // 引数で返り値を変える repo.find.mockImplementation((id) => { if (id === 1) return { name: "佐藤" }; if (id === 2) return { name: "鈴木" }; return null; });
超便利
Stubを使えば異常系のテストが格段に書きやすくなります。「エラーを返すStub」「nullを返すStub」「空配列を返すStub」を仕込めば、本物では再現困難な状況も自由に作れます。
05

Mockの使い方

USING MOCK

Mock（モック）は「呼び出されたかどうか」「どんな引数で呼ばれたか」を検証する偽物です。Stubが「問い合わせ系」だったのに対し、Mockは「指示系」を検証します。メール送信・ログ出力・通知など、呼ばれたこと自体が重要な場面で使います。

mock 「呼ばれたか」を検証する
// MockとStubの違い
STUB
返り値に注目
「何を返すか」を仕込む
呼ばれたか問わない
MOCK
呼び出しに注目
「どう呼ばれたか」を検証
返り値は気にしない場合も

英語では "State verification" vs "Behavior verification" と呼び分けられる。状態検証 vs 振る舞い検証。

// Mockの典型例 test("購入時にメールが送られる", () => { const mailer = { send: jest.fn(), // Mock }; const service = new OrderService(mailer); service.purchase({ user: "u1", item: "X" }); // 呼ばれたかを検証 expect(mailer.send).toHaveBeenCalled(); // 引数まで検証 expect(mailer.send).toHaveBeenCalledWith( "u1@example.com", "ご購入ありがとうございます" ); // 回数まで検証 expect(mailer.send).toHaveBeenCalledTimes(1); });
使い分けの目安
対象オブジェクトが情報を「取りに行く」相手 → Stub。情報を「渡す」相手・通知する相手 → Mock。これがざっくりした使い分けの指針です。
06

Spy・Fake・Dummy

SPY, FAKE, DUMMY

残り3種類もざっくり押さえましょう。Spyは「実物に記録機能を追加」、Fakeは「動く軽量実装」、Dummyは「形だけ」。それぞれ向く場面が異なります。

spy fake dummy 残り3種類のダブル
// 3種類の役割
Spy 本物の動きはそのまま、ただし呼び出し履歴を記録。後から「何回呼ばれたか」を検証できる
Fake 本物の代わりに動く軽量実装。例: メモリ上で動くRepository、テスト用の偽SMTPサーバ
Dummy 中身が使われないので、引数を埋めるためだけの空のオブジェクト

現場での出現頻度はStub > Mock > Spy > Fake > Dummy。Dummyはほぼ意識せず使っていることも多い。

// Spy — 本物 + 記録 const realLogger = new Logger(); const spy = jest.spyOn(realLogger, "info"); service.doSomething(realLogger); expect(spy).toHaveBeenCalledWith("started"); // 本物のLoggerはそのまま動く // 加えて記録もされる // Fake — メモリ上で動くDB class InMemoryUserRepo { constructor() { this.data = new Map(); } save(u) { this.data.set(u.id, u); } find(id) { return this.data.get(id); } } // 本物のDBと同じインターフェース // メモリ上で動くので超高速 // Dummy — 形だけ const dummyUser = new User(null, null); service.process(dummyUser, 10); // userの中身は使われない // 引数のために形だけ用意
TIP
Fakeは特に強力です。「メモリ上で動くDB」「テスト用の偽SMTPサーバ」のように、本物と同じインターフェースで動くため、StubやMockより自然なテストが書けます。複雑なテストでは積極的に検討する価値があります。
07

依存性注入(DI)

DEPENDENCY INJECTION

テストダブルを使うには、「外から依存を差し込める」設計が必要です。これを依存性注入（Dependency Injection、DI）と呼びます。複雑なフレームワークを使わなくても、コンストラクタや引数で渡すだけで実現できます。

DI ダブルを差し込めるようにする設計
// DIの効果
DIなし
内部でnew
↓
置き換え不可
↓
テスト困難
DIあり
外から受け取る
↓
置き換え可能
↓
テスト容易

DIはテスト容易性を実現する設計原則。本番では本物を、テストではダブルを、同じ場所に差し込みます。

// === DIなし(NG) === class OrderService { constructor() { this.mailer = new SmtpMailer(); // ↑ 内部でnew = 置換不可 } } // === DIあり(OK) === class OrderService { constructor(mailer) { this.mailer = mailer; // ↑ 外から受け取る = 置換可 } } // 本番 const service = new OrderService( new SmtpMailer() ); // テスト const service = new OrderService( { send: jest.fn() } // Mock );
POINT
DIはSpringやDIコンテナを使うことだけを指す言葉ではありません。「外から渡す」というだけのシンプルな概念です。コンストラクタ・関数引数で渡すだけでも立派なDI。これだけでテスト容易性は大きく向上します。
08

やりすぎ問題

MOCK OVERUSE

テストダブルは強力な道具ですが、使いすぎは害になります。何でもかんでもMock化すると、「実装の写経テスト」になり、リファクタリング耐性を失います。現場で頻発する典型的なアンチパターンを押さえましょう。

overuse 過剰なMockが招く問題
// 3つのアンチパターン
過剰Mock すべての依存をMockに。実装の細部を写経したテストになり、リファクタで全部壊れる
連鎖Mock a.getB().getC().getD() のような深い連鎖を全てMock化。複雑で読めない
戻り値の型偽装 Mockが本物と異なる型を返してしまい、本番で型エラー発覚

目安: 1テストで2〜3個までのMock。それ以上は設計の見直し時かも。

// === NG: 過剰Mock === test("処理", () => { const a = mock(); const b = mock(); const c = mock(); const d = mock(); const e = mock(); // 5個もMock? // → 対象が依存しすぎ // → 設計の問題 }); // === NG: 実装写経 === test("処理", () => { service.do(); expect(a.step1).toHaveBeenCalled(); expect(a.step2).toHaveBeenCalled(); expect(a.step3).toHaveBeenCalled(); // → 内部手順を全部検証 // → リファクタで全壊 }); // === OK: 結果を検証 === test("処理", () => { const result = service.do(); expect(result).toEqual(...); // → 内部手順は不問 // → 振る舞いだけ確認 });
鉄則
Mockは「外との境目」だけに使うのが現場の知恵。内部のクラス同士までMock化し始めたら、設計を疑うサインです。地図編Vol.4で「依存」という概念を構造的に整理するので、合わせて読むと使い分けが見えてきます。
