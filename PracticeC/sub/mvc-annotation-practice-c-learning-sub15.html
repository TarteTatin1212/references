02

Cで学んだことの全体像

THE BIG PICTURE

15巻の技術は、実は「データとどう向き合うか」という一本の軸で繋がっています。その全体像を描きます。

big-picture データのライフサイクル
// データと向き合う4つの局面
局面1
設計する（どう持つか）
2テーブル設計・正規化・外部キー。「データをどんな形で保存するか」── すべての土台。
局面2
取得する（どう取るか）
JOIN・resultMap・N+1対策・検索・動的SQL。「必要なデータを正しく速く取り出す」。
局面3
加工する（どう見せるか）
集計・HAVING・時系列・Service層のロジック・DTO変換。「生データを意味ある情報に変える」。
局面4
守る（どう正しく保つか）
トランザクション・外部キー制約。「データの整合性を二重に守る」。

この4局面こそ、あらゆる業務システムに共通するデータとの向き合い方です。

// データのライフサイクル // ━━━ 設計 ━━━ // 「どんな形で持つか」 subjects ──< study_records （1対多・外部キー） // S1, M1, M14 // ━━━ 取得 ━━━ // 「必要な分を正しく速く」 SELECT ... JOIN ... WHERE ... （resultMap / 動的SQL / N+1対策） // M2-8 // ━━━ 加工 ━━━ // 「意味ある情報に」 GROUP BY / HAVING / 集計 → Service で画面用に変換 → DailyStatView / SubjectStatView // M9-12 // ━━━ 守る ━━━ // 「正しさを保証」 @Transactional（アプリ層） FOREIGN KEY（DB層） // M13-14 // ━━━ そして統合 ━━━ // すべてが1つのアプリに // M15 // この流れは、学習記録アプリに // 限らない。どんなシステムも同じ。 // → だから「応用が効く」
POINT ──

個々の技術（JOIN、HAVING、@Transactional…）は「道具」に過ぎません。本当に大切なのは「データとどう向き合うか」という考え方です。Cで学んだのは、実は「データを設計し・取得し・加工し・守る」という一貫した思考。この4局面を意識すれば、新しい技術も「どの局面の道具か」で理解できます。

06

2つの習慣の意味

TWO HABITS

Cシリーズが特に大切にしてきた2つの習慣──「動かしながら学ぶ」と「なぜを問う」。なぜこの2つが重要なのか、最後に深く考えます。

two-habits-meaning 2つの習慣が持つ力
// なぜこの2習慣か
習慣1
動かしながら学ぶ
「分かったつもり」を防ぐ。コードは動かすと必ず現実を教えてくれる。エラーも成功も、すべてが具体的なフィードバック。抽象を具体で確かめる習慣。
習慣2
なぜを問う
「暗記」を「理解」に変える。「こう書けば動く」で止めず「なぜ動くか」を問えば、応用できる知識になる。未知の問題も推理で解ける。

この2つは補い合う。動かして「何が起きるか」を知り、なぜを問うて「どうしてか」を理解する。具体と抽象を往復することで、学びが深く根を張ります。

// 2つの習慣の相乗効果 // ─── 動かすだけ（なぜを問わない）─── // 「コピペしたら動いた」 // → 似た問題に応用できない // → 少し違うと手が止まる // ─── 考えるだけ（動かさない）─── // 「理屈は分かった気がする」 // → 実際に書くと動かない // → 細部の落とし穴に気づけない // ─── 両方やる（Cの方法）─── // 1. まず動かす → 現実を見る // 2. なぜ動くか問う → 理解する // 3. 少し変えて動かす → 確かめる // 4. また問う → 深める // → 具体と抽象の往復 // → 揺るがない理解 // ─── 例：N+1問題（Vol.5）─── // 動かした：SQLログに大量のクエリ // → 「現実」を目撃 // 問うた：なぜ増える?（別冊S5） // → ループ内クエリと「理解」 // だから：別の場面でも // 「あ、これN+1だ」と気づける // これが応用力の正体
POINT ──

「動かしながら学ぶ」と「なぜを問う」── この2つは、プログラミングだけでなくあらゆる学びに通じる普遍的な姿勢です。新しい言語、新しいフレームワーク、未知の業務ドメイン── 何に出会っても、「まず触って、なぜを考える」。この習慣さえあれば、あなたはどんな技術も自力で習得できます。それこそがCシリーズが最も伝えたかったことです。
