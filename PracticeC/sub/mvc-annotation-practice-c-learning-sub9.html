SQLの集計関数は本編で扱った5つ以外にも多数。業務で出会う代表的なものを整理。

all-functions 主要集計関数
// 集計関数一覧
関数意味業務例
COUNT(*)行数注文件数
COUNT(DISTINCT col)重複除いた個数ユニークユーザー数
SUM(col)合計売上合計
AVG(col)平均平均評価
MAX(col)最大値最高得点

業務で最頻出：COUNT / SUM / AVG / MAX / MIN の5つ。COUNT(DISTINCT) も重要。

// 業務での集計例 // ─── ユニーク数 ─── // 「今月学習した日数」 SELECT COUNT(DISTINCT studied_on) FROM study_records WHERE studied_on >= '2026-05-01'; // ─── 最大・最小 ─── // 「最長学習時間と最短」 SELECT MAX(minutes) AS longest, MIN(minutes) AS shortest FROM study_records WHERE subject_id = 1; // ─── 連結 ─── // 「Javaのメモ一覧をカンマ区切り」 SELECT s.name, GROUP_CONCAT(r.memo SEPARATOR ', ') AS all_memos FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id WHERE s.id = 1 GROUP BY s.id, s.name; // ─── 統計 ─── // 「学習時間のばらつき」 SELECT subject_id, AVG(minutes) AS mean, STDDEV(minutes) AS sd FROM study_records GROUP BY subject_id; // ─── 複合 ─── // 「学習効率レポート」 SELECT subject_id, COUNT(*) AS sessions, SUM(minutes) AS total_min, AVG(minutes) AS avg_min, MAX(minutes) AS longest, MIN(minutes) AS shortest FROM study_records GROUP BY subject_id;
03

WHERE と HAVING の使い分け

WHERE vs HAVING

集計クエリで「絞り込み」をする時、WHERE と HAVING の使い分けが問われます。Vol.11 で本格的に扱いますが、ここで予告。

where-vs-having 2つの絞り込みの違い
// 違いの本質
WHERE
集計「前」の絞り込み
JOINや各行をフィルタした後でGROUP BYに渡す。「5月の記録だけ集計したい」など、個別の行を絞り込む場合。
HAVING
集計「後」の絞り込み
GROUP BY で集計した結果に対する絞り込み。「合計時間が60分以上の科目だけ」など、集計値を条件にする場合。

判断軸：「集計関数を使った条件か？」 → YES なら HAVING、NO なら WHERE。

// WHERE と HAVING の例 // ─── WHERE：個別行の絞り込み ─── SELECT s.id, s.name, SUM(r.minutes) AS total FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id WHERE r.studied_on >= '2026-05-01' AND r.studied_on <= '2026-05-31' GROUP BY s.id, s.name; // → 「5月の記録だけを対象に集計」 // → 集計前の絞り込み // ─── HAVING：集計値の絞り込み ─── SELECT s.id, s.name, SUM(r.minutes) AS total FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id GROUP BY s.id, s.name HAVING total >= 60; // → 「合計が60分以上の科目だけ」 // → 集計後の絞り込み // ─── 両方使う ─── SELECT s.id, s.name, SUM(r.minutes) AS total FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id WHERE r.studied_on >= '2026-05-01' GROUP BY s.id, s.name HAVING total >= 120; // → 「5月の記録のみを集計し、120分以上の科目」 // ─── SQL 実行順序（重要） ─── // 1. FROM / JOIN ← テーブル結合 // 2. WHERE ← 個別行のフィルタ // 3. GROUP BY ← グループ化 // 4. HAVING ← 集計値のフィルタ // 5. SELECT ← 列の選択 // 6. ORDER BY ← 並び替え // 7. LIMIT ← 件数制限 // 「WHERE → GROUP BY → HAVING」 // の順番を意識すると間違わない
POINT ──

「WHERE に集計関数を書きたい」と思った時はHAVINGです。WHERE SUM(...) > 100 は SQL エラー。逆に「WHERE で十分なのに HAVING を使う」のはパフォーマンスロス（集計前の絞り込みの方が処理する行が少なくて済む）。「できるだけ早い段階で絞る」のがSQL最適化の鉄則。

04

NULL と COALESCE の扱い

NULL HANDLING

集計クエリでNULL は避けられない問題。LEFT JOIN すると子テーブル側が NULL になり、SUM(NULL) は NULL になります。業務での処理パターンを整理。

null-handling NULL処理の3パターン
// NULL処理の3パターン
パターン1
COALESCE で代替値
COALESCE(val, default)。val が NULL なら default を返す。業務で最頻出。複数引数も可：COALESCE(a, b, c) で「最初の非NULL」。
パターン2
IFNULL（MySQL固有）
2引数版の COALESCE。IFNULL(val, 0)。COALESCE と動作はほぼ同じ。標準SQLは COALESCE。
パターン3
NULL のままJavaへ→ Optional / null チェック
DTO 側で null を許容し、画面で th:text="${row.total ?: 0}" 等で処理。業務では避けがち（複雑になる）。

推奨：SQL の COALESCE で 0 にするのが最も安全＆シンプル。

// NULL関連の実例 // ─── COALESCE の使い方 ─── // 基本 COALESCE(SUM(minutes), 0) // SUM が NULL なら 0 // 複数候補 COALESCE(nickname, real_name, '匿名') // nickname→ NULL なら real_name // それも NULL なら '匿名' // 文字列連結での NULL 防御 COALESCE(memo, '') // memo が NULL でも空文字 // ─── 業務での集計テンプレート ─── SELECT s.id, s.name, COUNT(r.id) AS cnt, // NULLを数えない COALESCE(SUM(r.minutes), 0) AS total, // NULL を 0 に COALESCE(AVG(r.minutes), 0) AS avg, // 同上 COALESCE(MAX(r.minutes), 0) AS longest FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id GROUP BY s.id, s.name // → 全列で NULL を排除 // → Java側で安心して受けられる // NULL の罠 // 1. NULL の演算 // NULL + 1 = NULL （0ではない） // NULL = NULL = NULL（true ではない） // 2. NULL の比較 // WHERE col = NULL ← 必ず false（要注意） // WHERE col IS NULL ← 正しい書き方 // 3. COUNT の挙動 // COUNT(*) : 全行を数える // COUNT(col) : col が NULL でない行 // COUNT(1) : COUNT(*) と同じ
POINT ──

NULLの扱いは「DB の三大難所」のひとつと言われます（他に「タイムゾーン」「文字コード」）。業務では「NULLを許容しない設計」か「明示的に処理する設計」のどちらかを徹底するのが安全。「曖昧なNULL」が一番危ない。

GROUP BY の応用

GROUP BY VARIATIONS

GROUP BY は1列以上を指定できます。応用パターンを紹介。

groupby-app 複数キー・関数式での GROUP BY
// 応用パターン
パターンA
複数キーでグループ化
「科目×日付」「ユーザー×月」など、2つ以上の軸で集計。Vol.12 で「日別×科目」のクロス集計に発展。
パターンB
関数式でグループ化
GROUP BY DATE(created_at) で日単位、GROUP BY MONTH(created_at) で月単位。時系列集計でよく使う。
パターンC
CASE 式でグループ化
GROUP BY CASE WHEN minutes >= 60 THEN '長' ELSE '短' END。カテゴリ化集計。業務でよく使う。
// 応用パターンの例 // ─── パターンA: 複数キー ─── // 「科目別・月別の合計時間」 SELECT s.name AS subject_name, DATE_FORMAT(r.studied_on, '%Y-%m') AS month, SUM(r.minutes) AS total FROM subjects s JOIN study_records r ON s.id = r.subject_id GROUP BY s.name, DATE_FORMAT(r.studied_on, '%Y-%m') ORDER BY month, subject_name; // 結果: // subject_name | month | total // Java | 2026-05 | 375 // Spring Boot | 2026-05 | 255 // SQL | 2026-05 | 30 // ─── パターンB: 関数式 ─── // 「日別の合計時間（学習日ベース）」 SELECT studied_on, SUM(minutes) AS daily_total FROM study_records WHERE studied_on >= '2026-05-08' GROUP BY studied_on ORDER BY studied_on; // → Vol.12 でこの形を使う // ─── パターンC: CASE ─── // 「長時間／短時間の本数」 SELECT CASE WHEN minutes >= 60 THEN '長時間' ELSE '短時間' END AS duration_type, COUNT(*) AS count FROM study_records GROUP BY CASE WHEN minutes >= 60 THEN '長時間' ELSE '短時間' END; // 結果: // duration_type | count // 長時間 | 6 // 短時間 | 3
06

集計クエリのパフォーマンス

PERFORMANCE

集計クエリは「全件を計算する」性質上、テーブルが大きくなると遅くなります。パフォーマンス改善の3つの定石を押さえます。

aggr-perf 集計の高速化
// 高速化の3定石
定石1
GROUP BY 列にインデックス
GROUP BY subject_id なら subject_id 列にインデックスがあると速い。頻繁に集計に使う列はINDEX化。
定石2
WHERE で集計対象を絞る
「過去1ヶ月だけ」など期間を絞れば、集計対象の行が激減。WHERE → GROUP BY の順で処理される性質を活かす。
定石3
集計結果のキャッシュ
同じ集計を頻繁に表示するなら、結果をキャッシュするかサマリーテーブルを作る。大規模システムでは必須。

学習記録アプリは小規模なので定石1だけ意識すれば十分。大規模化したら定石2・3を検討。

// インデックス設計の例 // 集計クエリでよく使う列を INDEX化 CREATE INDEX idx_subject_id ON study_records(subject_id); CREATE INDEX idx_studied_on ON study_records(studied_on); // 複合インデックス（より高速） CREATE INDEX idx_subject_studied ON study_records(subject_id, studied_on); // ─── 検証手法 ─── // EXPLAIN で実行計画を見る EXPLAIN SELECT subject_id, SUM(minutes) AS total FROM study_records WHERE studied_on >= '2026-05-01' GROUP BY subject_id; // 結果に注目： // ・type が ALL → 全件スキャン（遅い） // ・type が ref → インデックス使用（速い） // ・rows: 検査する行数の予測 // ・Extra: Using filesort などの情報 // サマリーテーブル戦略（大規模） // 集計結果を別テーブルに保存 CREATE TABLE subject_stats_daily ( subject_id INT, date DATE, total_minutes INT, record_count INT, PRIMARY KEY (subject_id, date) ); // バッチで毎日更新 // 画面では集計せず、このテーブルを SELECT // → 高速だが整合性管理が必要 // → 中・大規模プロジェクトの定番手法

DTO 設計のお作法

DTO DESIGN

DTO は画面ごとに増えていく性質があります。業務で「混乱しないDTO設計」のお作法を整理。

dto-design DTO命名と分離のコツ
// DTOの3原則
原則1
「画面1個」=「DTO1個」を基本に
画面ごとに DTO を作る。「使い回し」を控える。再利用しようとして変更したら、別画面が壊れる事故が多い。
原則2
命名は「対象＋用途」
SubjectStat（科目統計）、DailyTotal（日別合計）、UserRanking（ユーザーランキング）。「何の何か」を一目で。
原則3
不変オブジェクト化を検討
DTO は読み取り専用なので、@Value（Lombok）や Java 14+ の record で不変にすると安全。Cでは @Data で十分。

Cの15巻で増える DTO：SubjectStat（Vol.9）、DailyTotal（Vol.12）、RankingDto（任意）など。

// DTO命名・パッケージ構成 // ─── 推奨パッケージ構造 ─── com.example.todo/ ├ entity/ // DBテーブル対応 │ ├ Subject.java │ └ StudyRecord.java ├ form/ // 画面入力 │ └ StudyRecordForm.java ├ dto/ // 画面出力（読み取り） │ ├ SubjectStat.java │ ├ DailyTotal.java │ └ Ranking.java ├ mapper/ ├ service/ └ controller/ // ─── 役割の明確な分離 ─── // Entity: DBに対応する型 // ↔ DBのテーブルと1対1 // Form: 画面から来る入力 // ↔ HTMLフォームと対応 // ↔ バリデーション持つ // DTO: 画面に渡す出力 // ↔ 任意の形（集計・複合） // ↔ 読み取り専用が原則 // DTO のアンチパターン // ─── ✗ NG: 万能DTO ─── @Data public class DataDto { private Integer id; private String name; private Integer count; private Integer total; private Integer average; private LocalDate date; private String category; // ... 何でも入る } // → 何の DTO か分からない // → どのフィールドがnullなのか不明 // ─── ✓ OK: 用途別DTO ─── @Data public class SubjectStat { ... } @Data public class DailyTotal { ... } @Data public class UserRanking { ... } // → 名前で用途が分かる // → 各DTOのフィールドが意味を持つ

集計のハマりどころ

AGGREGATION PITFALLS

集計クエリで業務でよくある罠を4つ。本編Vol.9 で扱わなかった発展的な内容を含みます。

aggr-pitfalls 集計の罠
// よくある罠
PITFALL 1
JOIN で重複行が生まれて集計が膨らむ
原因：1対多×1対多 のJOIN でデカルト積が発生し、SUM が水増しされる。
対処：DISTINCT を使うか、サブクエリで先に集計してからJOIN。
PITFALL 2
日付の月またぎ集計の誤り
原因：「今月の合計」のつもりが、月またぎでズレる。
対処：WHERE created_at >= '2026-05-01' AND created_at < '2026-06-01' の半開区間。S7で学んだ通り。
PITFALL 3
タイムゾーンを無視した日別集計
原因：UTCで保存しているのにJSTで集計したい時、変換忘れで日付が1日ズレる。
対処：CONVERT_TZ を使うか、Java側で計算前にタイムゾーン考慮。S7参照。
PITFALL 4
平均の計算がリアルタイムでズレる
原因：「過去30日の平均」を見る画面で、日付が変わると毎日数値が変動。キャッシュ更新タイミングと不整合。
対処：「集計対象期間を明示」「キャッシュ有効期限を短く」など。
// PITFALL 1: 重複行による水増し // ─── 業務シナリオ ─── // 「ユーザーの記録合計 + タグ情報」を集計 // ─── ✗ NG: 単純JOIN ─── SELECT u.id, SUM(r.minutes) AS total FROM users u LEFT JOIN study_records r ON r.user_id = u.id LEFT JOIN record_tags t ON t.record_id = r.id // ← 多対多 GROUP BY u.id; // → 1記録に3タグ付いてたら // SUM が 3倍に膨らむ // → 業務上アウト // ─── ✓ OK: サブクエリで先に集計 ─── SELECT u.id, COALESCE(s.total, 0) AS total FROM users u LEFT JOIN ( SELECT user_id, SUM(minutes) AS total FROM study_records GROUP BY user_id ) s ON s.user_id = u.id; // → タグの結合は別クエリで処理 // PITFALL 2: 月またぎの落とし穴 // ─── ✗ NG ─── WHERE studied_on BETWEEN '2026-05-01' AND '2026-05-31' // → 5/31 23:59:59 のレコードが入る // かどうかDB依存 // ─── ✓ OK: 半開区間 ─── WHERE studied_on >= '2026-05-01' AND studied_on < '2026-06-01' // ─── DATE 型なら BETWEEN OK ─── WHERE studied_on BETWEEN '2026-05-01' AND '2026-05-31' // DATE型は日単位なので OK // DATETIME型は半開区間が必須 // → 統一した書き方が安全
