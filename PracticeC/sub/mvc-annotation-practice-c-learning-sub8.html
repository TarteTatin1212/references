ALL DYNAMIC SQL TAGS

MyBatis の動的SQLタグは主要6種類。本編Vol.8 では <if> と <where> を扱いました。残り4つも整理します。

all-tags 主要6タグ
// 主要6タグ
<if test="...">
条件式が真ならタグ内容を出力（Vol.8で習得済）
<where>
WHERE句を動的に組み立て・先頭のAND/OR自動除去（Vol.8で習得済）

<foreach> で IN 句を組み立てる

FOREACH FOR IN-CLAUSE

業務でよく使う動的SQL第3位が <foreach>。配列やリストから IN (1,2,3) のような句を組み立てます。S5別冊で予告したN+1対策のIN句一括取得でも必須。

foreach-usage <foreach> の基本
// 4つの属性
属性意味
collection反復対象（引数名 or "list"/"array"）
item各要素の参照名
separator要素間の区切り文字
open / close全体の囲み文字

IN句の定型：open="(" close=")" separator="," のセット。

// <foreach> の典型例 // ─── 複数科目で絞り込み ─── // Mapper interface List<Subject> findByIds( @Param("ids") List<Integer> ids); // XML <select id="findByIds" resultType="Subject"> SELECT * FROM subjects WHERE id IN <foreach item="id" collection="ids" open="(" close=")" separator=","> #{id} </foreach> </select> // ─── 呼び出し ─── List<Integer> ids = List.of(1, 2, 5); mapper.findByIds(ids); // ─── 生成されるSQL ─── SELECT * FROM subjects WHERE id IN (?, ?, ?) // Parameters: 1, 2, 5 // ─── 注意：collection が空のとき ─── // ids が空リスト → IN () になる // → MySQLでは構文エラー // → 呼び出し側で「空なら呼ばない」 // または <if> で囲んで防ぐ // IN句でN+1対策（S5予告の実装） // 親を取得 List<Subject> subjects = mapper.findAll(); // id を集める List<Integer> ids = subjects.stream() .map(Subject::getId) .toList(); // 子を一括取得（SQL 1回） List<StudyRecord> allRecords = studyRecordMapper.findBySubjectIds(ids); // → 親1回 + 子1回 = 計2回 // → JOIN方式と並ぶ N+1 対策の王道
04

<choose> / <set> / <trim>

CHOOSE / SET / TRIM

残り3つのタグを紹介。業務では特定の場面で力を発揮します。

choose-tag <choose> ── 排他的分岐
// choose / when / otherwise

複数の条件を排他的に評価。Javaの switch-case または if/else if/else 相当。

使い所：「ソート順を切り替え」「ステータス別の絞り込み」など。

// <choose> の例：ソート切替 <select id="findSorted" ...> SELECT * FROM study_records <choose> <when test="sortBy == 'date'"> ORDER BY studied_on DESC </when> <when test="sortBy == 'minutes'"> ORDER BY minutes DESC </when> <otherwise> ORDER BY id </otherwise> </choose> </select> // ─── <if> の連続 vs <choose> ─── // ─── <if> の連続：複数条件を同時に評価 ─── <if test="sortBy == 'date'"> ... </if> <if test="sortBy == 'minutes'"> ... </if> // → 排他にならない、両方真なら両方出る // ─── <choose>：上から評価、最初に真の一つだけ ─── // → switch-case 的に安全
set-tag <set> ── UPDATE用
// <set> の役割

UPDATE 文の SET 句を動的化。末尾の余分なカンマを自動除去するのが嬉しい点。Vol.13 で本格的に使う予定。

業務での使用場面：「変更があったフィールドだけ更新」のパッチ的UPDATE。

// <set> の例：部分更新 <update id="updatePartial"> UPDATE study_records <set> <if test="studiedOn != null"> studied_on = #{studiedOn}, </if> <if test="minutes != null"> minutes = #{minutes}, </if> <if test="memo != null"> memo = #{memo}, </if> </set> WHERE id = #{id} </update> // ─── 生成されるSQL（メモだけ指定の場合）─── UPDATE study_records SET memo = ? WHERE id = ? // → 末尾の「,」を <set> が自動除去 // 全フィールド更新する場合は // 普通の UPDATE SET ... で十分。 // <set> は「部分更新」専用ツール。
trim-tag <trim> ── 万能版
// <trim> とは

<where> や <set> の汎用版。任意の文字列を前後で除去・追加できる。「内部的には <where> も <trim> で実装されている」。

使い所：標準タグでは表現しきれない複雑なケース。業務でも稀。

// <trim> の例 // <where> を <trim> で実装すると <trim prefix="WHERE" prefixOverrides="AND |OR "> ... </trim> // → <where> と同じ動作 // <set> を <trim> で実装すると <trim prefix="SET" suffixOverrides=","> ... </trim> // → <set> と同じ動作 // ─── 知識として知っておけば十分 ─── // 業務では <where> / <set> で大体足りる

SQL生成の透明性とログの読み方

TRANSPARENCY & LOG READING

動的SQLは便利な反面、「実際にどんなSQLが生成されたか」が見えにくいのが弱点。これを克服するログの読み方を整理します。

read-logs SQLログから生成SQLを読む
// 標準ログの読み方

MyBatis標準ログはパラメータが ? で表示される。Parameters の行で値を確認し、頭の中で組み立てる。

p6spy（S5で紹介）を使うと、パラメータ展開済みのSQLがそのまま出る。業務での生産性向上手段。

POINT ──

動的SQLが複雑になると、「自分が書いたXMLからどんなSQLが生まれるか」を頭で組み立てるのが大変になります。p6spy のようなパラメータ展開ツールを業務プロジェクトで導入する価値は大きい。「動かない時のデバッグ時間」が圧倒的に短くなる。

ON句 vs WHERE句（伏線回収）

ON vs WHERE

本編Vol.8 で残した「LEFT JOIN + WHERE の罠」の決着。「記録のない科目を残したい時」と「消したい時」で、JOIN条件の書く場所が変わる。

on-vs-where ON句とWHERE句の違い
// 違いの本質
ON句
JOIN 時の条件
「どう繋ぐか」を指定。LEFT JOIN なら「右側にない時も左側は残す」が保証される。
WHERE句
結合後のフィルタ
「結合した結果から何を残すか」を指定。左側だろうと条件偽なら消える。

原則：「子テーブルの条件は ON 句」「親テーブルの条件は WHERE 句」。

// 具体例で比較 // ─── ① 子の条件を WHERE に置く（Vol.8 の方式）─── SELECT s.name, r.studied_on FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id WHERE r.studied_on >= '2026-05-10' AND r.studied_on <= '2026-05-15' // 結果： // Java | 2026-05-10 // Java | 2026-05-12 // Java | 2026-05-14 // Spring | 2026-05-11 // Spring | 2026-05-13 // Spring | 2026-05-15 // HTML/CSS や JavaScript は消える // なぜなら r.studied_on が NULL → 条件偽 // ─── ② 子の条件を ON に移す ─── SELECT s.name, r.studied_on FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id AND r.studied_on >= '2026-05-10' AND r.studied_on <= '2026-05-15' // 結果： // Java | 2026-05-10 // Java | 2026-05-12 // Java | 2026-05-14 // Spring | 2026-05-11 // Spring | 2026-05-13 // Spring | 2026-05-15 // SQL | (NULL) ← 範囲外で NULL // HTML/CSS | (NULL) ← 元から記録なし // JS | (NULL) ← 元から記録なし // すべての科目が表示される（記録は NULL あり） // どっちを選ぶか // ─── WHERE 句方式（Vol.8 の方式）─── // 「期間内に記録がある科目だけを見たい」 // → 業務的に「実績フォーカス」 // → 学習記録アプリでは妥当な動作 // ─── ON 句方式 ─── // 「全科目を表示して、期間内記録だけ並べたい」 // → 業務的に「俯瞰フォーカス」 // → ダッシュボードでは好まれる // ─── どっちが「正解」？─── // 要件次第。両方とも有効な設計。 // 「ユーザーがどう見たいか」で決める。
POINT ──

「ON句 vs WHERE句」は SQL の理解度を測る代表的な質問です。技術面接でもよく聞かれます。「両方説明できて、要件次第で選べる」と答えられれば、SQL 中級者として認められる目安。本巻で押さえた感性は、転職活動でも武器になります。

07

動的SQLのアンチパターン

ANTI-PATTERNS

動的SQLは強力だが使い方を誤ると保守不能になります。業務でやってはいけない4パターンを整理。

anti-patterns 4つのアンチパターン
// やってはいけない4つ
ANTI 1
条件分岐が10個以上の巨大動的SQL
「とりあえず1つのSQLにすべて押し込む」型。SQLの動作が予測不可能になり、デバッグ不能化。
対処：5条件を超えたら用途別にメソッドを分けるか、別の検索エンジン（Elasticsearch等）を検討。
ANTI 2
${...} で文字列を直接埋め込む
SQLインジェクションの温床。${tableName} 等で動的に表を切り替えるテクは便利だが、ユーザー入力を絶対渡さない。
対処：必ず #{...}（バインドパラメータ）を使う。
ANTI 3
<if> ネストの深さ4階層以上
条件の中の条件の中の条件…が深くなると、もはや読めない。動的SQLは「フラットに書く」のが鉄則。
対処：複雑な分岐は Service 層で if し、複数の Mapper を呼び分ける。
ANTI 4
SELECTカラムを動的に変える
<if> で SELECT 句を切り替える設計。戻り値の型が変動するので resultMap も壊れる。
対処：必要なカラムは常に取得し、Java側で取捨選択。
// ANTI 2: ${} の罠 // ─── ✗ NG: ユーザー入力を ${} で展開 ─── <select id="searchUnsafe"> SELECT * FROM users WHERE name = '${name}' </select> // ユーザー入力 name に // "alice' OR '1'='1" を入れると… // 生成SQL: SELECT * FROM users WHERE name = 'alice' OR '1'='1' // → 全ユーザー情報が漏洩 // → SQLインジェクション攻撃 // ─── ✓ OK: #{} を使う ─── <select id="searchSafe"> SELECT * FROM users WHERE name = #{name} </select> // #{} は PreparedStatement のバインドパラメータ // 値はエスケープ＆型変換されて安全 // #{} vs ${} の使い分け // ─── #{} = バインドパラメータ ─── // ・通常はこちらを使う // ・値が安全にエスケープされる // ・WHERE id = #{id} // ─── ${} = 文字列直接置換 ─── // ・SQL構造を変える時のみ使う // ・絶対にユーザー入力を渡さない // ・ORDER BY ${sortColumn} (列名) // ↑ ホワイトリストで検証必須 // ─── 鉄則 ─── // 「迷ったら #{}」 // ${} を使う前に // ・本当に必要か？ // ・入力源は安全か？ // ・列名/テーブル名なのか値なのか？ // を必ず確認
POINT ──

SQLインジェクションは業務システムで最も多いセキュリティ脆弱性のひとつ。#{} を常用すれば自動的に防げるのがMyBatisの良いところ。${} を使うのは「動的にテーブル名や列名を変える」「ORDER BYの列名指定」などSQL構造の動的化でだけ、と覚えておくと安全です。

パフォーマンスとセキュリティ

PERFORMANCE & SECURITY

動的SQLは「便利」と「安全/速度」の両立が大事。最後に2つの観点で押さえます。

perf-sec 業務での注意点
// パフォーマンス観点
PERF 1
プレースホルダ ? でクエリキャッシュ活用
DB のSQL実行プランキャッシュは「同じSQL構造（パラメータ部分が ?）」なら再利用する。#{} 使用なら自動でこの恩恵。
PERF 2
条件追加時の INDEX 設計
「期間絞り込み」を追加したら studied_on 列にインデックスが必要。新しい検索条件＝新しい INDEX 検討。
PERF 3
条件が全部偽の時の動作
全 <if> が偽だと WHERE なし＝全件取得。件数次第で重い。デフォルト条件を入れるか、件数制限する。
// セキュリティ観点
SEC 1
#{} 厳守でSQLインジェクション防止
前セクションの通り #{} 使用が鉄則。
SEC 2
${} を使うならホワイトリスト検証
列名指定など ${} がどうしても必要な時、受け入れ可能な値のリストでチェック。
// ホワイトリスト検証の例 // ─── ✗ NG: そのまま埋め込み ─── @GetMapping("/list") public String list( @RequestParam String sortBy) { // sortBy をそのまま XML へ service.findSorted(sortBy); } // XML ORDER BY ${sortBy} // → 攻撃者が「id; DROP TABLE users;」を送ると… // ─── ✓ OK: 許可リストで検証 ─── @GetMapping("/list") public String list( @RequestParam String sortBy) { // 許可された値のみ通す List<String> allowed = List.of("id", "name", "createdAt"); if (!allowed.contains(sortBy)) { sortBy = "id"; // 既定値に } service.findSorted(sortBy); } // ─── パフォーマンス改善ヒント ─── // よく使う検索パターン: // 1. 期間 + 科目で絞り込み // 2. 1ヶ月分の一覧 // → これらに対応する INDEX: CREATE INDEX idx_studied_on ON study_records(studied_on); CREATE INDEX idx_subject_id_studied_on ON study_records(subject_id, studied_on); // 業務システムでは： // 検索条件を増やすたびに // 「この条件で INDEX 効く？」を確認する習慣
