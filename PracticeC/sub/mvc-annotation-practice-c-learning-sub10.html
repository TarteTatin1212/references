01

この別冊のテーマ

THEME

統計画面・ダッシュボードは「数字を並べるだけ」では業務で評価されません。「見ただけで意味が伝わる」のが理想。本巻ではエンジニアが押さえておくべき UX の基本を整理します。

why-ux-matters なぜエンジニアもUXを知るべきか
// エンジニアがUXを知る理由
理由1
画面の品質を自分で判断できる
「デザイナー任せ」では業務で困る場面が多い。「これで分かりやすいか？」を自問できる感性が必要。
理由2
仕様書を読み解く深さが上がる
「合計を表示」「割合を出す」── 同じ仕様でも10通りの表現がある。設計者の意図を推し量れる。
理由3
ユーザーフィードバックを活かせる
「分かりにくい」と言われた時、「どこをどう直せばいいか」の選択肢を持てる。
理由4
技術的な意思決定が正しくなる
「ここはサーバ側で集計？クライアント側で計算？」── UX を踏まえると正解が見える。

02

ダッシュボード設計の原則

DASHBOARD PRINCIPLES

業務ダッシュボードの設計には4つの確立された原則があります。Vol.10 で作った画面も、これらを意識すると一段良くなる。

4-principles 設計の4原則
// ダッシュボード4原則
原則1
「最重要」が最上部・最大表示
「全体合計」「今月の数字」など、ユーザーが最初に見たい数字を画面の一番目立つ位置に。フォントサイズで強弱を付ける。
原則2
1画面 = 1テーマ
「科目別」「日別」「タグ別」を1画面に混在させない。テーマごとにページを分ける。「あれもこれも」は分かりにくくなる元凶。
原則3
階層構造を持たせる
「全体合計 → 各カテゴリ → 詳細」のようなドリルダウン構造。視線の流れに沿う。
原則4
「比較対象」を必ず示す
「6時間勉強した」だけでは多いか少ないか分からない。「先月比 +30%」「目標達成率 80%」など比較を入れて初めて意味が出る。
// 原則を Vol.10 の画面に当てはめる // ─── 現在の画面 ─── [ 科目別 学習統計 ] 全体合計: 10時間 30分 // ← 原則1 OK（最上部・目立つ） Java 6時間 15分 (59%) [████████████░░░░░░░░] 5件 / 平均 75分 Spring Boot 4時間 15分 (40%) [█████████░░░░░░░░░░░] 3件 / 平均 85分 ... // ─── どこが良いか ─── // 原則1: 「全体合計」が一番上 ✓ // 原則2: 「科目別」だけ ✓ // 原則3: 「全体 → 各科目」階層 ✓ // 原則4: 「割合」が比較対象 ✓ // ─── 改善ポイント ─── // 原則4を強化： // ・「先週比」「目標時間 vs 実績」 // ・「学習日数 / 月の日数」 // → さらに「比較対象」を増やす余地あり // Vol.12 で「日別推移」を追加すると // 比較がさらに豊富になる
POINT ──

「1画面に詰め込みすぎ問題」は業務でよく起きます。「あの数字も見たい・この比較も欲しい」と要望が積み重なり、画面がカオス化する。「1画面1テーマ」を守る勇気が、結果的にユーザーを助けます。

03

情報密度のバランス

INFORMATION DENSITY

画面に「どれだけ情報を詰めるか」は永遠の課題。「余白を取りすぎても・詰めすぎても」見にくい。バランス感覚を整理。

density-balance 情報密度のスペクトラム
// 密度の3段階
低密度
「広告風」── 余白多め・大文字
「1画面1KPI」型。スマートフォン向け、エグゼクティブ向け。業務管理系にはやや物足りない。
中密度
「業務ダッシュボード風」── バランス重視
複数の指標を一目で把握。Vol.10 のカード型UI はこのゾーン。業務での標準。
高密度
「Excel風」── 表形式・小さい文字
大量データを一覧。分析担当者向け。一般ユーザーには情報過多。

判断軸：ユーザーの「画面の前で過ごす時間」。短時間なら低密度、長時間使うなら高密度可。

// 密度の例 // ─── 低密度（広告風）─── ╔════════════════════════════╗ ║ ║ ║ 今月の学習時間 ║ ║ ║ ║ 10時間 30分 ║ ║ ║ ║ 先月比 +25% ║ ║ ║ ╚════════════════════════════╝ // ─── 中密度（Vol.10）─── [ 全体合計: 10時間 30分 ] [ Java 6時間 15分 (59%) ] [ ████████████░░░░░░░░ ] [ 5件 / 平均 75分 ] [ Spring Boot 4時間 15分 (40%) ] [ █████████░░░░░░░░░░░ ] [ 3件 / 平均 85分 ] // ─── 高密度（Excel風）─── 科目 件数 合計 平均 最大 最小 標準偏差 Java 5 375 75.0 120 45 28.6 Spring 3 255 85.0 120 45 30.0 SQL 1 30 30.0 30 30 0.0 HTML 0 0 0.0 - - - // ─── 結論 ─── // Vol.10 は「中密度」を目指した設計 // → 大半の業務ユーザーに最適
POINT ──

業務開発で「情報密度をどうするか」迷ったら、「中密度から始める」が安全。ユーザーから「もっと情報が欲しい」と言われたら密度を上げ、「ごちゃごちゃする」と言われたら下げる。始点として中密度が一番フィードバックを得やすい。

視覚化の作法

VISUALIZATION CRAFT

「グラフ」「進捗バー」「色」など視覚要素には、それぞれ得意な見せ方があります。「何を伝えたいか」に応じて使い分けます。

viz-types 視覚化要素の使い分け
// 視覚化要素 6種
要素得意な見せ方
棒グラフカテゴリ別の比較
折れ線グラフ時系列の推移
円グラフ構成比（カテゴリ数 4以下）
進捗バー「達成率」「割合」のシンプル表現

05

色の意味

COLOR SEMANTICS

色は「意味を持つ情報」です。色の選び方ひとつで、ユーザーの理解速度が変わる。エンジニアが押さえるべき色の基本ルール。

color-semantics 色が持つ業界共通の意味
// 業界共通の色の意味
色意味用途
緑正常・成功正常稼働・OK・達成
黄/橙注意・警告警告・要確認
赤異常・エラー失敗・危険・停止
青情報・中性通常情報・リンク
グレー無効・非アクティブ休止・データなし

Vol.10 でやったこと：記録なし科目をグレー（opacity 0.5）に。これは「非アクティブ」を表す典型例。業界の慣習に従っている。

// 色使いの3原則 // ─── 原則1: 色に意味を持たせる ─── // ✓ OK： // 緑＝順調、赤＝遅延、黄＝注意 // ✗ NG： // 科目別に色を変える // → なぜその色か根拠なし // → ユーザーが意味を推測しにくい // ─── 原則2: 色だけに依存しない ─── // 色覚多様性（色覚異常）の方は // 緑と赤の区別がつきにくい // ✓ OK： // 色 + テキスト/アイコン で表現 // 「✓ OK」「⚠ 警告」「✗ NG」 // ✗ NG： // 色だけで「OK/NG」を表現 // ─── 原則3: 色数を絞る ─── // ✓ OK： // 1〜3色のアクセント色 + グレー系 // （Cのモスグリーン＋アクセント） // ✗ NG： // 「ロゴの色」「ボタンの色」 // 「グラフの色」が全部違う // → 統一感がなくなる // ─── Cでの実践 ─── // 全15巻でカラーパレット統一 // 本編：モスグリーン // 別冊：セージグレー // → 「シリーズ感」が保たれている理由
POINT ──

「色覚多様性」は世界人口の約8%（男性の約12%、女性の約0.5%）と言われます。業務システムを作るなら必ず考慮すべき配慮。「色だけに頼らない」がアクセシビリティの基本。テキストとアイコンも併用するのが鉄則です。

06

「数字」から「物語」へ

FROM NUMBERS TO STORIES

統計画面の最終的なゴールは「数字を見せる」ではなく「物語を伝える」こと。ユーザーが画面を見て、「次に何をするか」が分かる状態を作るのが理想です。

numbers-to-story 数字を物語化する3段階
// 物語化の3段階
段階1
生データ表示（最低限）
「Java: 375 分」だけ。これでは多いか少ないか分からない。Vol.9 はここに近い。
段階2
比較・割合を添える（業務標準）
「Java: 6時間 15分（全体の 59%）」のように、比較対象・割合・分かりやすい単位を添える。Vol.10 はここ。
段階3
解釈・示唆を出す（上級）
「先週比 +20% で順調に学習中」「Spring Boot を増やすと達成バランスが良いかも」など、システムからの提案。AI/ML を絡めることも。Cの範囲外。

段階2まで実現できれば、業務ダッシュボードとしては及第点。段階3 は中級〜上級の領域。

// 物語化の実例 // ─── 段階1: 数字だけ（Vol.9）─── 科目 件数 合計 平均 Java 5 375 75 Spring 3 255 85 SQL 1 30 30 // ─── 段階2: 比較・割合（Vol.10）─── 全体合計: 10時間 30分 Java 6時間 15分 (59%) ████████████░░░░░░░░ 5件 / 平均 75分 Spring Boot 4時間 15分 (40%) █████████░░░░░░░░░░░ 3件 / 平均 85分 // ─── 段階3: 解釈・示唆（業務上級）─── 全体合計: 10時間 30分 ↗ 先週比 +25% で順調に上昇中 ⭐ 学習スタイル：「集中型」 （少数の科目に時間を投入） 提案： ・「Spring Boot」も並行で進めると バランスが良くなりそうです ・5/14 が最長学習日（120分）でした // ─── ポイント ─── // 段階3 を実現するには： // 1. 比較データ（履歴）の蓄積 // 2. ルールベースの解釈ロジック // 3. （理想）機械学習による提案 // Cでは段階2を目標 // 業界では Saas 系で段階3 が普及中
POINT ──

「数字だけのダッシュボード」が業務でよく作られますが、ユーザーは数字を解釈する手間を嫌います。「これは多いの？少ないの？」「先月と比べてどう？」── これらをシステム側で答えてあげるのがUX向上の核心。「データはあるのに使えていない」のは業務システムの典型的な失敗パターンです。

07

Mapper用DTO vs 画面用DTO 再論

DTO LAYERING REVISITED

本編Vol.10 で「DTOを2つに分けた」件、UX観点から再整理。「画面が増えても破綻しない設計」の核心です。

dto-layering DTO分離の業務価値
// 「画面用DTO」の業務価値
価値1
画面が変わってもDBは無影響
「カード型UI」「グラフ表示」「PDF出力」── 画面ごとに別の View DTO を作る。Mapper はそのまま使える。
価値2
同じDBデータから多様な見せ方
SubjectStat から、SubjectStatView（カード）、SubjectChartView（グラフ）、SubjectExportView（CSV出力）などを作る。Mapper 1個に View 多数。
価値3
画面固有の表示ロジックを集約
「N時間M分」「+25%」「⭐順調」など、表示用文字列を View に持たせる。Service が責任を持つ。Template はただ表示するだけ。
価値4
テストしやすい
「Service の変換ロジック」だけを単体テスト可能。DBもHTMLも要らない。
// 業務での View DTO 多様化 // ─── 同じ Mapper 出力から複数 View ─── // Mapper List<SubjectStat> stats = subjectMapper.findSubjectStats(); // Service が画面別に変換 public List<SubjectStatView> findSubjectStatViews() { // Vol.10: カード型UI 用 // → totalTimeText / percentage 付き } public List<SubjectChartData> findChartData() { // 将来: グラフ表示用 // → 数値だけ / フォーマットなし } public List<SubjectExportRow> findExportRows() { // 将来: CSV出力用 // → カンマ区切り済み / エスケープ済み } // ─── 3画面・3 View・1 Mapper ─── // ・Mapper は変更なし // ・XML / SQL は変更なし // ・新画面追加 = 新 View + 新変換メソッド // → 「最小限の変更」で機能拡張 // 業務でのアンチパターン // ─── ✗ NG: Mapper戻り値を直接画面に渡す ─── // Service: return mapper.findSubjectStats(); // → 画面側で「N時間M分」を計算 // → Thymeleaf に複雑なロジックが入る // → テストしにくい // ─── ✗ NG: Mapper戻り値を画面用に流用 ─── // SubjectStat に totalTimeText を追加 // → Mapper 出力に画面用フィールドが混入 // → 「DBから取った」と「画面用に計算した」 // が同じ DTO に混在 → 役割が曖昧 // ─── ✓ OK: 役割で分ける ─── // Mapper用 = DBの素材 // View用 = 料理済みの画面用

統計画面の罠

DASHBOARD PITFALLS

業務での統計画面でやりがちな4つの罠。Vol.10 で押さえた設計と対比して見ます。

dashboard-pitfalls 統計画面の罠
// 4つの罠
PITFALL 1
「単位がない数字」だけを並べる
原因：「375」「10.5」とだけ表示。「分？時間？個数？」
対処：必ず「分」「時間」「件」「%」を併記。Vol.10 では「N時間 M分」で対応済み。
PITFALL 2
「ゼロの場合」を考慮しない
原因：データが0件の時に画面がエラー、または不気味な空欄。
対処：「0件」「記録なし」を明示。Vol.10 では薄いカード（opacity 0.5）で表現。
PITFALL 3
過剰な小数点表示
原因：「59.2727...%」「375.0 分」など。
対処：業務に応じた精度に丸める。「分」レベルなら小数不要、お金なら2桁、科学計算なら桁が増える。
PITFALL 4
PDF や CSV エクスポートで崩れる
原因：画面向けの表示文字列を CSV にもそのまま出力。「6時間 15分」を Excel で扱えない。
対処：エクスポート用は別 View DTO。生数値で出す。前セクションの実装パターン。
// PITFALL 1 の比較 // ─── ✗ NG ─── 科目 合計 平均 Java 375 75 Spring 255 85 // → 「375 って何の単位？」 // ─── ✓ OK ─── 科目 合計時間 平均 Java 6時間15分 1時間15分 Spring 4時間15分 1時間25分 // → 単位 + 業務に最適な単位 // PITFALL 3 の対処 // ─── ✗ NG: 過剰な小数 ─── Java 平均: 75.0000 分 割合: 59.27272727272727% // ─── ✓ OK: 業務に応じて丸める ─── Java 平均: 1時間15分 // 整数 割合: 59% // 整数 // 業務の精度別ガイド: // ・時間（時:分） → 整数 // ・割合 → 整数 or 小数1桁 // ・金額（円） → 整数 // ・金額（ドル） → 小数2桁 // ・科学計算 → 桁数大 // → Vol.10 で setScale / RoundingMode // を意識したのはこれ
