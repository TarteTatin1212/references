02

「魔法に見える」処理の正体

DEMYSTIFY THE MAGIC

resultMap の「魔法感」は、実は3つの異なる処理が組み合わさったものです。それぞれは普通のJavaコードで実現できる仕組み。

three-magics 3つの「魔法」
// 魔法に見える3つの処理
魔法1
「property="name" と書けば setName が呼ばれる」
正体：リフレクション。実行時にクラス情報を取り出して setName() を呼ぶ。Javaの標準機能。
魔法2
「DBのDATE型がJavaのLocalDateになる」
正体：TypeHandler。型変換専用のクラスが存在し、ResultSet から取った値を Java の型に変換する。
魔法3
「11行の冗長な行が5個のオブジェクト + 子Listになる」
正体：グループ化アルゴリズム。<id> の column の値で行をグループ化する、ループとMapを使った標準処理。

結論：全部「普通のJava処理」の組み合わせ。魔法ではない。

// 概念的に「自分で書く」とどうなるか // resultMap がやっていることを // Javaで「自分で書く」とこうなる： Map<Integer, Subject> map = new LinkedHashMap<>(); // JOIN結果の11行をループ for (ResultRow row : resultSet) { Integer sid = row.getInt("subject_id"); // 既にこのSubjectがあれば取り出し、 // なければ新規作成 Subject s = map.computeIfAbsent(sid, k -> { Subject ns = new Subject(); ns.setId(sid); ns.setName(row.getString("subject_name")); ns.setColor(row.getString("color")); ns.setRecords(new ArrayList<>()); return ns; }); // 子の id がNULLなら子なし行 Integer rid = row.getInt("record_id"); if (rid != null) { StudyRecord r = new StudyRecord(); r.setId(rid); r.setSubjectId(sid); r.setStudiedOn(row.getDate("studied_on")); r.setMinutes(row.getInt("minutes")); r.setMemo(row.getString("memo")); s.getRecords().add(r); } } return new ArrayList<>(map.values()); // → resultMapはこの処理を自動化 // → XML側で「設計」を書くだけで // 自動的に組み立ててくれる

MyBatisの内部処理フロー

INTERNAL PROCESSING FLOW

MyBatisが subjectMapper.findAllWithRecords() の呼び出しを受けてから、結果リストを返すまでの7ステップを順に追います。

flow-7-steps 処理の7段階
// 7ステップ
1
プロキシ経由でメソッド呼び出し
SubjectMapper はインターフェイス。MyBatisが動的プロキシを生成し、メソッド呼び出しを受け取る。
2
XMLからSQLとresultMapを取得
起動時にパース済みの findAllWithRecords 定義を引く。SQL本体と resultMap が手に入る。
3
SQLを実行（JDBC経由）
PreparedStatement を作って実行。ResultSet（結果のテーブル）が返る。
4
ResultSetHandler が行ごとに処理開始
11行を1行ずつ取り出す。各行が「列名→値」のマップとして扱える状態に。
5
TypeHandler が型変換
studied_on 列の値は SQL の DATE。Java側の LocalDate に変換するのが TypeHandler の仕事。
6
リフレクションでオブジェクト構築
new Subject() → リフレクションで setId(...)、setName(...) 等を順次呼び出し。
7
グループ化と List 構築
<id> の値で「同じ親に纏める」処理。<collection> の指示で子を追加。最終的に List<Subject> が完成。
// シーケンス図的イメージ [Service] ↓ findAllWithRecords() [SubjectMapper] // インターフェイス ↓ [MyBatisProxy] // ★Step 1 ↓ [XMLConfigCache] // ★Step 2 ↓ SQL + resultMap 取得 [SqlSession] // ★Step 3 ↓ SQL 実行 [JDBC/ResultSet] // 11行 ↓ [ResultSetHandler] // ★Step 4〜7 ↓ 各行を処理 ├─ TypeHandler (型変換) ├─ Reflector (setter呼出) └─ ResultContext (グループ化) ↓ List<Subject> ↓ [Service] // この流れを知ると： // ・「namespace 不一致」→ Step 2 で失敗 // ・「TypeHandler not found」→ Step 5 で失敗 // ・「ReflectionException」→ Step 6 で失敗 // ・「件数おかしい」→ Step 7 のグループ化 // と原因の場所が即推理できる
04

リフレクションの役割

REFLECTION

リフレクションは「実行時にクラスの情報を取り出して操作する」Javaの標準機能。resultMap の property="name" から setName(...) への変換は、リフレクションで実現されています。

reflection-basics リフレクションの動き
// リフレクションでできること
  • "Subject" という文字列から Class<?> を取り出す
  • そのクラスのフィールド一覧を取得
  • 「property="name"」から setName(String) メソッドを見つける
  • そのメソッドを引数付きで呼び出す

普通のJavaコードでは obj.setName("Java") と書きますが、リフレクションでは「メソッドを動的に探して呼ぶ」ことができます。

便利だが代償

リフレクションは強力だが遅い。MyBatisは結果をキャッシュすることで実用速度を保っています。1回目は遅くても、2回目以降は爆速。

// リフレクションで setter を呼ぶ例 // ─── 通常のJavaコード ─── Subject s = new Subject(); s.setName("Java"); // ─── リフレクション版（同じ動作）─── Class<?> clazz = Class.forName("...Subject"); Object obj = clazz .getDeclaredConstructor() .newInstance(); Method setter = clazz .getMethod("setName", String.class); setter.invoke(obj, "Java"); // ─── MyBatisは何をしているか ─── // XML: <result property="name" .../> // ↓ // 1. type="Subject" でクラス取得 // 2. property="name" から // "setName" のメソッド名を作る // （Bean命名規則） // 3. メソッドを invoke で呼ぶ // ─── 知っておくべき事実 ─── // resultMap で property="name" と書いた時: // ・getter は無くてもいい // ・setter は必須（リフレクションで呼ぶ） // ・Lombok の @Data があれば // setterが自動生成されるのでOK
POINT ──

「@Dataなのに setter が見つからないエラー」が出る時は、Lombokの設定が効いていない可能性があります。Eclipse なら 「Project → Clean」でビルド再生成、起動し直しで解決することが多いです。リフレクションは「実体としての setter」を探すので、コンパイル時に生成されていないと失敗します。

TypeHandler の役割

TYPE HANDLER

SQLの値とJavaの値は、そもそも型が違います。VARCHAR→String、INT→Integer、DATE→LocalDate──この変換をしているのがTypeHandler。

typehandler TypeHandler の動き
// 標準的な対応
DB側の型Java側の型TypeHandler
VARCHAR / CHARStringStringTypeHandler
INTInteger / intIntegerTypeHandler
BIGINTLong / longLongTypeHandler
DECIMALBigDecimalBigDecimalTypeHandler
DATELocalDateLocalDateTypeHandler
DATETIMELocalDateTimeLocalDateTimeTypeHandler
TINYINT(1)BooleanBooleanTypeHandler

結論：MyBatisは標準の型変換を最初から大量に持っています。学習記録アプリの全列はこれで賄えます。自前のTypeHandlerが必要になる場面はほぼない。

// 学習記録アプリでのTypeHandler使用箇所 // study_records テーブル CREATE TABLE study_records ( id INT, // → Integer subject_id INT, // → Integer studied_on DATE, // → LocalDate minutes INT, // → Integer memo VARCHAR(200), // → String created_at DATETIME // → LocalDateTime ); // 各列で TypeHandler が暗黙に動く // 失敗パターン // ── 例1：型ミスマッチ ── // DB側: VARCHAR / Java側: LocalDate // → 変換失敗で例外 // → 「Could not convert...」エラー // ── 例2：DATE → Date (古い) ── private Date studiedOn; // java.util.Date // 動くが古い書き方 // → 必ず LocalDate を使う（Vol.2でも強調） // ── 例3：DATE → String ── private String studiedOn; // 動く（自動で文字列化）が // 「日付として比較・計算」できない // → 業務的にはアウト
06

グループ化アルゴリズム

GROUPING ALGORITHM

resultMapの最も重要な処理がここ。<id> 要素の column の値で行をグループ化します。「同じidの行は同じオブジェクトに纏める」のがロジックの心臓。

grouping グループ化の動作
// アルゴリズムの流れ
  1. 結果セットの各行を順に処理
  2. その行の <id> column の値を取得
  3. 既に同じ値で作ったオブジェクトがあるかマップで検索
  4. あればそのオブジェクトに子だけ追加／なければ新規作成して登録
  5. 子側の <id> column が NULL なら子は追加しない
  6. 全行処理後、登録順を保ったListを返す
なぜ ORDER BY が重要か

JOIN結果が「親IDでソート」されていれば、グループ化がスムーズです。並び順次第で同じ親の行が散らばるとMyBatisは正しく集約できないため、SQL側で ORDER BY s.id, ... を入れます。

// 11行を5オブジェクトに変換する流れ // 入力（JOIN後の11行）： sid=1, sname=Java, rid=1 sid=1, sname=Java, rid=2 sid=1, sname=Java, rid=3 sid=1, sname=Java, rid=4 sid=1, sname=Java, rid=5 sid=2, sname=Spr, rid=6 sid=2, sname=Spr, rid=7 sid=2, sname=Spr, rid=8 sid=3, sname=SQL, rid=9 sid=4, sname=HTML, rid=NULL sid=5, sname=JS, rid=NULL // 処理ループ： 行1: sid=1 → 新規作成、Subject{id:1, ...} 子: rid=1 → 追加 → records=[1] 行2: sid=1 → 既存利用 子: rid=2 → 追加 → records=[1,2] 行3: sid=1 → 既存利用 子: rid=3 → 追加 → records=[1,2,3] 行4: sid=1 → 既存利用 子: rid=4 → 追加 → records=[1,2,3,4] 行5: sid=1 → 既存利用 子: rid=5 → 追加 → records=[1,2,3,4,5] 行6: sid=2 → 新規作成、Subject{id:2, ...} 子: rid=6 → 追加 → records=[6] 行7〜8: sid=2 既存利用 → records=[6,7,8] 行9: sid=3 → 新規作成、records=[9] 行10: sid=4 → 新規作成 子: rid=NULL → 追加しない → records=[] 行11: sid=5 → 新規作成 子: rid=NULL → 追加しない → records=[] // 結果： [ Subject{id=1, name=Java, records=[5件]}, Subject{id=2, name=Spr, records=[3件]}, Subject{id=3, name=SQL, records=[1件]}, Subject{id=4, name=HTML, records=[]}, Subject{id=5, name=JS, records=[]} ] // → これがVol.2で「2回SQL」で作っていた // オブジェクトと完全に同じ形

内部を意識すべきハマりどころ

INTERNAL PITFALLS

内部処理を知ると、「なぜこういう書き方をすべきか」が見えてきます。表面的なルールではなく原理で覚えられる。

internal-pitfalls 内部理解で防げる罠
// 内部視点での罠
PITFALL 1
setter が無いプロパティを書く
@Data 無しのクラスで private final フィールドを書いた等。リフレクションで setter を呼ぶのが原理なので、setter 無し＝即失敗。
PITFALL 2
no-arg コンストラクタが無い
MyBatisは new Subject() でインスタンス化する。引数なしのデフォルトコンストラクタが必須。
@Data があれば自動だが、@Builder だけだと無くなることがあるので注意。
PITFALL 3
JOIN結果が ORDER BY されていない
グループ化アルゴリズムは順序に依存しない（LinkedHashMapで対応）が、結果のList順序はSQL順になる。「IDが順番に並んでほしい」なら ORDER BY s.id を必ず入れる。
PITFALL 4
resultMap を再帰的に使う複雑性
resultMap は他のresultMapを extends や association で参照できる。複雑になると追えなくなるので、Cでは深追いしない（Bで扱う予定）。
// PITFALL 1 を内部視点で見る // ─── ✗ NG ─── public class Subject { private final Integer id; // final → setterが作れない } // MyBatisが Step 6 でやる処理： // 1. new Subject() ← コンストラクタが要る // 2. setId(1) ← setterが要る // → final で setter 無し // → ReflectionException // ─── ✓ OK ─── @Data public class Subject { private Integer id; } // Lombok @Data が生成するもの： // ・引数なしコンストラクタ ✓ // ・setId(...) ✓ // ・getId() ✓ // → リフレクションで全て呼べる // PITFALL 2 を内部視点で見る // ─── ✗ NG（@Builder のみ）─── @Builder public class Subject { private Integer id; } // → 引数なしコンストラクタが生成されない // → new Subject() でエラー // ─── ✓ OK ─── @Data @NoArgsConstructor // 明示 @AllArgsConstructor @Builder public class Subject { private Integer id; }
