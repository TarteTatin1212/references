01

この別冊のテーマ

THEME

本編Vol.1では「subjectsテーブルを作りました」と結果だけ提示しました。でも本来、業務開発で大事なのは「テーブルをどう分けるか」の判断です。「これは1テーブルでいい？」「いや2テーブルに分けるべき？」── この判断軸を持つのが本巻のゴールです。

why-design-matters 設計判断が大事な理由
// 設計を間違えるとどうなるか
BEFORE
テーブル分けを誤る
「タスクと科目を一緒のテーブルにしよう」と1つのtasksテーブルに混ぜると、科目だけ更新したい時に全タスクが影響を受ける構造になります。
AFTER
適切に分ける
「性質が違うものは別テーブル」と分けておけば、それぞれ独立して変更できる。後から困らない設計になります。
// この巻で答える3つの問い // 問1: 1対多 / 多対多 / 1対1 は // どう見分けるか？ // 問2: 外部キーは「親」と「子」の // どっちに置くべきか？ // 問3: テーブルや列をどう命名すべきか？ // これに答えられるようになれば // 業務でDBを「読む」「書く」「設計する」 // 全部が一段レベルアップします。
POINT ──

業務開発で「最初の1ヶ月で一番ぶつかる壁」は、実はコードよりもテーブル設計の読み解きです。既存システムを引き継いだ時、ER図を見て「なぜこの構造？」と分かるかどうかで、その後の仕事の早さが3倍変わります。

ER図のミニマム ── 線と矩形だけ

ER DIAGRAM ESSENTIALS

ER図（Entity Relationship Diagram）にはたくさんの記法がありますが、業務でよく見るのは「矩形」と「線」だけのシンプル版です。線の先端の形でだいたい関連の種類が分かります。

er-basics ER図の最低限の読み方
// 線の先端の意味（鳥の足記法）
記号意味
―|必ず1個
―○0個か1個
―〈1個以上（多）
―○〈0個以上（多 or 無し）

カラスの足のように見えるので「鳥の足記法（Crow's Foot）」と呼ばれます。業務でも最も普及しています。

03

三種の関連の見分け方

THREE RELATIONSHIPS

テーブル同士の関連は基本的に3種類あります。「左の1件に対して右は何件あるか／右の1件に対して左は何件あるか」を両方向で考えるのがコツ。

relationship-types 1対多／1対1／多対多
// 3種類の関連
1対多（最頻出）
subjects ── study_records
「1つの科目に対して複数の学習記録」「1つの学習記録は必ず1つの科目に属する」。外部キーは「多」側に置く。
1対1（稀）
users ── user_profiles
「1人のユーザーに対して1個だけのプロフィール」「1個のプロフィールは1人だけのユーザーに紐づく」。実は1テーブルにまとめてもいい場面が多い。
多対多（要中間テーブル）
tasks ── tags
「1つのタスクに複数のタグ」「1つのタグに複数のタスク」。これは中間テーブル（task_tags）が必要。Cでは扱わない（Bで登場予定）。
// 見分けの定型問 // 「左1件に対して右は何件？」 // 「右1件に対して左は何件？」 // この2つを必ず両方向で問う。 // ─── 学習記録の例 ─── // 1つのsubject（Java）に対して // study_records は何件？ // → 何十、何百件もありうる（多） // 1つのstudy_recordに対して // subject は何件？ // → 必ず1件（どの科目か明確） // → 1 : 多 = 1対多 ✓ // ─── 反例：もし「1記録が複数科目」だったら？ ─── // 例：「複数科目を同時並行で勉強」を許す // → 1記録に複数subjectが付く // → これは「多対多」になる // → 中間テーブル record_subjects が必要 // Cでは「1記録 = 1科目」と単純化 // → 1対多 で済む
POINT ──

「これは1対多？多対多？」を迷ったら、「業務上、1件に複数の○○を許すか？」を関係者（業務担当やPM）に確認するのが鉄則。技術判断ではなく業務判断です。1件1科目で良いなら1対多、1件に複数科目を許すなら多対多になります。

04

外部キーはどちら側に置く？

WHICH SIDE HOLDS THE FK

1対多の関係で、外部キー（subject_id）を親側に置くのか／子側に置くのか──これは絶対のルールがあります。間違えるとデータが壊れます。

fk-direction 外部キーの置き場所
// 鉄則：外部キーは「多」側に置く
✗ NG
親に置く

subjects テーブルに record_id を持たせる。
問題：1科目に複数記録があるのに、列は1個しか入らない。

✓ OK
子に置く

study_records テーブルに subject_id を持たせる。
これが正解：各記録が「どの科目に属するか」を1個の値で示せる。

イメージ：「子は親を1人指せる」けれど「親は子を何人指すか分からない」。指せる側に矢印（外部キー）を持たせるのが原則。

// NG設計と OK設計の比較 // ─── ✗ NG（親に置く）── CREATE TABLE subjects ( id INT, name VARCHAR(50), record_id INT // ← 1個しか入らない ); // Java科目に学習記録が10件あったら？ // → 9件は記録できない…破綻 // ─── ✓ OK（子に置く）── CREATE TABLE subjects ( id INT, name VARCHAR(50) ); CREATE TABLE study_records ( id INT, subject_id INT, // ← 各記録が1個指す studied_on DATE, minutes INT ); // Java科目に10件記録があるなら // → 10行 すべて subject_id=1 を持つ // → これで自然に1対多が表現できる

命名規則の業界相場

NAMING CONVENTIONS

テーブル名と列名の付け方には業界の相場があります。プロジェクトごとに違いますが、Spring Boot＋MyBatis界隈の標準はだいたい以下です。Cでもこれに従います。

naming-conventions 業界の命名相場
// 命名のお作法
対象規則例
テーブル名複数形・小文字・snake_casesubjects
study_records
主キー列idid
外部キー列{単数形}_idsubject_id
日付列_on / _atstudied_on（日付）
created_at（日時）
フラグ列is_{形容詞}is_active
// 命名でよくある迷い // Q. テーブル名は単数形？複数形？ // A. 複数形が業界主流 // （「複数行を持つ集合」だから） subjects // ✓ よい study_records // ✓ よい subject // 単数形派もあるが少数 // Q. 主キーは id？ subject_id？ // A. 主キーは「id」だけが業界主流 // （外部キー側で subject_id と // 明確に書き分けられる） subjects.id // ← 主キー study_records.subject_id // ← 外部キー // Q. 日付の列名は？ // A. _on（日付）/ _at（日時） studied_on DATE // 学習した日 created_at DATETIME // レコード作成時刻 // ─── Java側の命名 ─── // DBの snake_case ↔ Javaの camelCase // → application.properties で自動変換 study_records.subject_id ↔ subjectId study_records.studied_on ↔ studiedOn
TIP ──

「主キーは id 派」と「主キーは {table}_id 派」は実は宗教戦争があります。Cでは「id派」を採用。理由は外部キー側で「subject_id＝subjects表のidを指す」と明確に書き分けられるから。プロジェクトに参加した時はそのプロジェクトの流儀に合わせるのが鉄則です。

06

物理キー vs 自然キー

SURROGATE KEY vs NATURAL KEY

主キーには2つの流派があります。物理キー（代理キー）＝意味のない連番ID、自然キー＝業務的に意味のある値（メールアドレス・社員番号など）。Cは物理キー派です。

key-philosophy 主キー設計の二大流派
// 2つの流派の比較
流派A
物理キー（推奨）

意味のない連番（1, 2, 3...）。
長所：変更されない／高速／業務変更に強い。
短所：人間にとって意味が読み取れない。

流派B
自然キー

業務的に意味ある値（社員番号など）。
長所：人間に読みやすい。
短所：業務変更で値が変わると地獄。外部キーに使うと連鎖更新が大変。

業界標準は物理キー派。AUTO_INCREMENT の数字IDが入る理由はこれです。

// 自然キーで失敗する典型例 // ─── 自然キー設計 ─── CREATE TABLE users ( email VARCHAR(100) PRIMARY KEY, name VARCHAR(50) ); CREATE TABLE posts ( id INT, user_email VARCHAR(100) // ← FK ); // ある日、ユーザーがメアド変更… UPDATE users SET email = 'new@example.com' WHERE email = 'old@example.com'; // → posts の user_email も全部更新が必要 // → 何万行も巻き込まれる // → 業務停止する可能性 // ─── 物理キー設計（推奨）─── CREATE TABLE users ( id INT AUTO_INCREMENT PRIMARY KEY, email VARCHAR(100), name VARCHAR(50) ); // メアド変更しても id は不変 // → 外部キー（user_id）への影響ゼロ
07

NULL を許す／許さない

NULL OR NOT NULL

列を作る時、毎回「NOT NULL を付けるべきか」を考えます。原則は「迷ったらNOT NULL」。NULL を許す列はバグの温床になりやすい。

null-policy NULL設計の判断軸
// NOT NULL を付けるべきかの判断
Q. その列に値が入らない瞬間がある？
ない → NOT NULL
ある → NULL許容を検討
Q. その列の意味は「不在」と「値が無い」を区別する？
区別しない → NOT NULL + DEFAULT
区別する → NULL許容
Q. 後から値が決まるか？
同時に決まる → NOT NULL
後から決まる → NULL許容
// 学習記録アプリでの NULL 設計 CREATE TABLE subjects ( id INT NOT NULL, // 必須 name VARCHAR(50) NOT NULL, // 必須 color VARCHAR(7), // NULL OK created_at DATETIME NOT NULL // 必須 DEFAULT CURRENT_TIMESTAMP ); // 判断の理由 // ・id : 主キーは必ず存在 → NOT NULL // ・name : 科目名のない科目はない → NOT NULL // ・color : 後で設定でもよい → NULL OK // ・created_at : 作成時に必ず決まる // → NOT NULL + DEFAULT // ─── NULL の罠 ─── // NULLは「等しい」で比較できない SELECT * FROM subjects WHERE color = NULL; // → ↑これは絶対 0 件返る // 正しくは ↓ SELECT * FROM subjects WHERE color IS NULL;
POINT ──

業務開発の「初日の地雷ベスト3」に必ず入るのが「NULL の比較ミス」です。= NULL ではなく IS NULL。SQLの基礎ですが、毎日のように事故が起きます。覚えておいてください。

08

設計のハマりどころ

DESIGN PITFALLS

テーブル設計でよくある「初心者がやりがち」な落とし穴を、最後にまとめます。

design-pitfalls 初心者が踏みがちな地雷
// よくある地雷
PITFALL 1
1つのテーブルに何でも入れる
subjectsテーブルに「最後の学習日」や「合計時間」を持たせる。派生情報は持たせないのが鉄則。集計はクエリで導出する。
PITFALL 2
外部キー制約を忘れる
DBレベルで FOREIGN KEY 制約を貼らないと、子に存在しない親IDを入れられる。整合性が破綻する。
PITFALL 3
列名がDBとJavaでバラバラ
DBが study_minutes / Javaが time。意味と名前を揃えるのが大原則。map-underscore-to-camel-case でDB study_minutes ↔ Java studyMinutes に揃える。
PITFALL 4
列の型が曖昧
「日付」を VARCHAR で持つ。型は意味を持つ。日付なら DATE、日時なら DATETIME。検索や集計の効率が変わる。
// PITFALL 1 の悪い例とその修正 // ✗ NG ── 派生情報をテーブルに持つ CREATE TABLE subjects ( id INT, name VARCHAR(50), total_minutes INT, // ← 派生 last_studied_on DATE // ← 派生 ); // 問題：study_records への登録ごとに // total_minutes を更新する必要 // → 漏れたらデータ不整合 // ✓ OK ── 派生は SELECT で導出 CREATE TABLE subjects ( id INT, name VARCHAR(50) ); // 合計時間が欲しい時は SELECT subject_id, SUM(minutes) FROM study_records GROUP BY subject_id; // で都度計算 → 不整合の心配ゼロ // ─── ただし業務では ─── // 集計性能が問題になったら // あえて「派生を持つ」設計もある // = サマリーテーブル // （これはBで触れる予定）
