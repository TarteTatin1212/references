02

SQLの論理実行順序

LOGICAL EXECUTION ORDER

SQL は「書く順序」と「実行される順序」が違う。これが全ての鍵。SELECT を最初に書くのに、実行は後半。この論理実行順序を理解すれば全てが繋がります。

exec-order 書く順序 vs 実行順序
// 論理実行順序（7段階）
1FROM/JOINテーブルを読み・結合する
2WHERE個別行をフィルタ（集計前）
3GROUP BY行をグループにまとめる
4HAVINGグループをフィルタ（集計後）
5SELECT列を選ぶ・集計関数を評価
6ORDER BY並び替える
7LIMIT件数を絞る

注目：WHERE（2番目）は GROUP BY（3番目）より前。だから WHERE の時点ではまだ集計されていない。

// 書く順序 vs 実行順序 // ─── 人が書く順序 ─── SELECT subject_id, SUM(minutes) // 1番目に書く FROM study_records // 2番目 WHERE studied_on >= '2026-05-01'// 3番目 GROUP BY subject_id // 4番目 HAVING SUM(minutes) >= 60 // 5番目 ORDER BY SUM(minutes) DESC; // 6番目 // ─── DBが実行する順序 ─── 1. FROM study_records // テーブルを読む 2. WHERE studied_on >= '2026-05-01' // 5月以降の行だけ残す（集計前） 3. GROUP BY subject_id // 科目ごとにグループ化 4. HAVING SUM(minutes) >= 60 // 合計60分以上のグループだけ（集計後） 5. SELECT subject_id, SUM(minutes) // 列を選ぶ 6. ORDER BY SUM(minutes) DESC // 並び替え // ─── ここが核心 ─── // WHERE（2番目）の時点では // GROUP BY（3番目）がまだ実行されてない // → SUM はまだ計算されていない // → WHERE で SUM は使えない！ // HAVING（4番目）の時点では // GROUP BY（3番目）が済んでいる // → SUM が計算済み // → HAVING で SUM が使える！
POINT ──

この論理実行順序は、SQL の全ての挙動を説明する万能の鍵です。「なぜ SELECT の別名（AS）が WHERE で使えないのか」「なぜ ORDER BY では AS が使えるのか」── 全部この順序で説明できます（ORDER BY は SELECT より後なので AS を知っている）。困ったらこの順序を思い出す。

03

なぜWHEREで集計関数が使えないか

WHY WHERE CAN'T USE AGGREGATES

本編Vol.11 でも触れた「WHERE SUM(...) >= 60 はエラー」の真の理由を、実行順序から説明します。これが理解できればHAVING を完全マスター。

why-where-fails エラーの原理
// なぜエラーになるか

SQL実行順序で WHERE（2番目）は GROUP BY（3番目）より前。

WHERE が評価される時点では、まだ行が個別に存在している状態。「グループ」も「集計値（SUM）」もまだ生まれていない。

だから WHERE SUM(minutes) >= 60 と書くと、「まだ存在しない SUM を参照しようとした」のでエラーになる。

時系列で考える

WHERE は「集計が始まる前」の関所。一行ずつ通過させるかどうかを判断する。集計値はまだ計算されていないので、SUM を条件にできない。HAVING は「集計が終わった後」の関所。集計済みの値で判断できる。

// 時系列で見る // ─── 元データ ─── // id | subject_id | minutes // 1 | 1 | 60 // 2 | 1 | 90 // 3 | 2 | 45 // ─── WHERE の時点（集計前）─── // 行は個別のまま： // id | subject_id | minutes // 1 | 1 | 60 // 2 | 1 | 90 // 3 | 2 | 45 // この時点で「SUM」は存在しない // → WHERE で SUM は参照不可能 // ─── GROUP BY の後（集計済み）─── // subject_id | SUM(minutes) // 1 | 150 // 2 | 45 // この時点で「SUM」が確定 // → HAVING で SUM を参照可能 // ─── 結論 ─── // ✗ WHERE SUM(minutes) >= 60 // → SUM がまだ無い → エラー // ✓ HAVING SUM(minutes) >= 60 // → SUM が計算済み → OK // ─── 個別の minutes は WHERE で使える ─── // ✓ WHERE minutes >= 60 // → 個別の行の minutes 列 → OK // （これは集計関数ではなく素の列） // 「個別の列」と「集計関数」を区別
04

使い分けの判断フロー

DECISION FLOW

実務で「WHERE か HAVING か」迷った時の判断フローを整理します。たった1つの問いで決まります。

decision-flow 1問で決まる判断
// 判断は1問だけ
Q. その条件は「集計値」を使う？
YES（SUM/COUNT/AVG等を条件に）
→ HAVING
NO（個別の列を条件に）
→ WHERE

原則の補足：両方必要なら両方使う。「WHEREで早く絞り、HAVINGで集計後に絞る」が最速。

// 判断フローの実例 // ─── ケース1: 「5月の記録だけ」─── // 条件: studied_on（個別の列） // → 集計値じゃない → WHERE WHERE studied_on >= '2026-05-01' // ─── ケース2: 「合計60分以上の科目」─── // 条件: SUM(minutes)（集計値） // → 集計値 → HAVING HAVING SUM(minutes) >= 60 // ─── ケース3: 「3件以上記録がある科目」─── // 条件: COUNT(*)（集計値） // → 集計値 → HAVING HAVING COUNT(*) >= 3 // ─── ケース4: 「60分以上の記録が対象」─── // 条件: minutes（個別の列） // → 集計値じゃない → WHERE WHERE minutes >= 60 // ─── ケース5: 「5月の記録で、合計120分以上」─── // 条件1: studied_on（個別）→ WHERE // 条件2: SUM（集計値）→ HAVING // → 両方使う WHERE studied_on >= '2026-05-01' // 個別 GROUP BY subject_id HAVING SUM(minutes) >= 120 // 集計 // ─── 紛らわしいケース ─── // 「minutes が 60以上の記録だけを集計」 // → 個別の minutes で絞る → WHERE // 「合計が 60以上の科目だけ表示」 // → 集計の SUM で絞る → HAVING // 同じ「60」でも意味が全然違う！
05

パフォーマンス比較

PERFORMANCE

WHERE と HAVING は「処理する行数」が違うのでパフォーマンスに差が出ます。「できるだけ早く絞る」が鉄則。

perf-compare 早く絞るほど速い
// なぜ WHERE が速いか

WHERE は集計前（GROUP BYより前）に絞り込む。絞り込んでから集計するので、集計対象が少なくて済む。

HAVING は集計後に絞り込む。全部集計してから捨てるので、無駄な集計が発生する場合がある。

原則

「WHEREで絞れる条件は WHEREで絞る」。集計値の条件だけ HAVING に回す。両方使えるケースで HAVING に寄せると遅くなる。

// パフォーマンスの違い // ─── 想定: 100万行のテーブル ─── // うち5月のデータは 1万行 // ─── ✓ 良い: WHERE で先に絞る ─── SELECT subject_id, SUM(minutes) FROM study_records WHERE studied_on >= '2026-05-01' // 1万行に絞る GROUP BY subject_id // 1万行を集計 HAVING SUM(minutes) >= 60; // 処理の流れ： // 100万行 → WHERE → 1万行 → 集計 // → 集計対象は1万行だけ → 速い // ─── ✗ 悪い: HAVING で日付を絞ろうとする ─── // （そもそも書けないが、概念的に） // もし全件集計してから日付で絞ると： // 100万行 → 集計 → その後絞る // → 集計対象が100万行 → 遅い // ─── 鉄則 ─── // 1. 個別行で絞れるものは WHERE // → 集計前に行数を減らす // 2. 集計値の条件だけ HAVING // → これは HAVING でしか書けない // ─── 検証 ─── // EXPLAIN で実行計画を確認 EXPLAIN SELECT subject_id, SUM(minutes) FROM study_records WHERE studied_on >= '2026-05-01' GROUP BY subject_id HAVING SUM(minutes) >= 60; // ・WHERE がインデックスを使えているか // ・rows（処理行数）が小さいか

サブクエリとの関係

SUBQUERIES

HAVING でできることの一部はサブクエリでも書けます。両者の関係と使い分けを整理。発展的な内容ですが、業務で役立つ知識です。

subquery-relation HAVING とサブクエリ
// サブクエリでの代替

「集計結果を絞り込む」は、HAVING でもサブクエリでも書ける。

HAVING：シンプル・読みやすい。同じ集計に対する絞り込み。

サブクエリ：柔軟。「集計結果をさらに別の条件で処理」など複雑なケース。

使い分け

シンプルな「集計値の閾値」ならHAVING。「集計結果を別テーブルと結合」「集計のさらに集計」など複雑ならサブクエリ。Vol.11 のケースは HAVING で十分。

// HAVING vs サブクエリ // ─── HAVING版（Vol.11採用・シンプル）─── SELECT subject_id, SUM(minutes) AS total FROM study_records GROUP BY subject_id HAVING SUM(minutes) >= 60; // ─── サブクエリ版（同じ結果）─── SELECT * FROM ( SELECT subject_id, SUM(minutes) AS total FROM study_records GROUP BY subject_id ) AS agg WHERE agg.total >= 60; // サブクエリで一度集計し、 // 外側の WHERE で絞る // （集計済みなので WHERE で total 使える） // ─── サブクエリが有利なケース ─── // 「合計が平均以上の科目」 // （集計の集計が必要） SELECT subject_id, total FROM ( SELECT subject_id, SUM(minutes) AS total FROM study_records GROUP BY subject_id ) AS agg WHERE total >= ( // 全科目の平均合計 SELECT AVG(t) FROM ( SELECT SUM(minutes) AS t FROM study_records GROUP BY subject_id ) AS inner_agg ); // → HAVING だけでは書きにくい // → サブクエリの出番 // ─── 結論 ─── // 単純な閾値: HAVING // 複雑な集計の集計: サブクエリ // Cの範囲では HAVING で十分
07

複雑な集計条件

COMPLEX CONDITIONS

HAVING には複数の条件や複数の集計関数を組み合わせられます。業務でよく使うパターンを整理。

complex-having 複雑な HAVING
// HAVING の応用パターン
パターン1
複数条件（AND / OR）
HAVING SUM(minutes) >= 60 AND COUNT(*) >= 2。「合計60分以上 かつ 2件以上」。
パターン2
複数の集計関数を組み合わせ
HAVING AVG(minutes) >= 60。「1回あたり平均60分以上の科目」。
パターン3
範囲指定
HAVING SUM(minutes) BETWEEN 60 AND 300。「合計が60〜300分の科目」。

注意：複雑になりすぎたら、Service層のロジックに分けることも検討。「SQLに何でも詰め込まない」バランス感覚。

// 複雑な HAVING の例 // ─── 複数条件（AND）─── SELECT subject_id, SUM(minutes) AS total, COUNT(*) AS cnt FROM study_records GROUP BY subject_id HAVING SUM(minutes) >= 60 AND COUNT(*) >= 2; // 「合計60分以上 かつ 2件以上」 // ─── 平均で絞る ─── SELECT subject_id, AVG(minutes) AS avg FROM study_records GROUP BY subject_id HAVING AVG(minutes) >= 60; // 「1回平均60分以上の集中科目」 // ─── 範囲指定 ─── SELECT subject_id, SUM(minutes) AS total FROM study_records GROUP BY subject_id HAVING SUM(minutes) BETWEEN 60 AND 300; // 「合計60〜300分の科目」 // ─── 動的SQLで複雑条件を組み立て ─── // （Vol.8の動的SQLとの合わせ技） GROUP BY s.id, s.name <if test="minMinutes != null or minCount != null"> HAVING 1=1 <if test="minMinutes != null"> AND COALESCE(SUM(r.minutes), 0) >= #{minMinutes} </if> <if test="minCount != null"> AND COUNT(r.id) >= #{minCount} </if> </if> // HAVING 1=1 のトリック： // 後続の AND を綺麗に繋ぐため // （<where>のHAVING版がないので手動） // ─── やりすぎ注意 ─── // 条件が増えすぎたら // Service層でフィルタも検討 // 「SQLとJavaの役割分担」
POINT ──

HAVING 1=1 のトリックは、動的SQL で HAVING に複数の <if> を繋ぐ時の定番技。<where> タグは WHERE 専用で HAVING には使えないため、常に真の 1=1 を先頭に置いて、後続の AND を綺麗に繋ぎます。業務でも見かける手法です。

08

ハマりどころ集

PITFALLS

HAVING / WHERE まわりの業務でよくある罠を4つ。原理を理解していれば回避できます。

pitfalls よくある罠
// よくある罠
PITFALL 1
SELECT の別名を WHERE で使う
原因：SELECT SUM(m) AS total ... WHERE total > 60。WHERE は SELECT より前なので total を知らない。
対処：WHERE では元の式を書く、または HAVING / サブクエリを使う。
PITFALL 2
HAVING で個別列を絞ろうとする
原因：HAVING studied_on >= '2026-05-01'。動くこともあるが、本来は WHERE の役割。
対処：個別列は WHERE。「早く絞る」原則にも反する。
PITFALL 3
GROUP BY なしで HAVING を使う
原因：GROUP BY がないと全体が1グループ扱い。HAVING は使えるが意図しない動作になりがち。
対処：HAVING は基本 GROUP BY とセット。
PITFALL 4
NULL を含む集計の HAVING で混乱
原因：HAVING SUM(minutes) >= 60 で、SUM が NULL の行が消える（NULL は条件偽）。
対処：COALESCE(SUM(minutes), 0) で NULL を 0 に。本編Vol.11 で採用済み。
// PITFALL 1 の詳細 // ─── ✗ NG ─── SELECT subject_id, SUM(minutes) AS total FROM study_records WHERE total >= 60 // エラー GROUP BY subject_id; // ERROR: Unknown column 'total' // WHERE（2番目）は // SELECT（5番目）より前なので // total という別名をまだ知らない // ─── ✓ OK: HAVING で元の式 ─── SELECT subject_id, SUM(minutes) AS total FROM study_records GROUP BY subject_id HAVING SUM(minutes) >= 60; // HAVING（4番目）も SELECT より前だが // 集計値そのものは使える // （多くのDBで別名 total も HAVING で使える） // ─── 別名が使える場所・使えない場所 ─── // WHERE : ✗ 別名不可（SELECTより前） // GROUP BY: △ DB依存 // HAVING : △ DB依存（MySQLは使える） // ORDER BY: ✓ 別名OK（SELECTより後） // 「実行順序」で全部説明できる： // SELECT より後の ORDER BY だけが // 確実に別名を使える
