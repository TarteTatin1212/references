01

この別冊のテーマ

THEME

外部キー制約は「データの正しさを DB が守る」強力な仕組み。しかし削除の挙動（特に CASCADE）は事故の元にもなります。業務での「賢い削除設計」を学びます。

why-delete-design 削除設計の重要性
// 削除が難しい理由
理由1
取り返しがつかない
削除したデータは基本戻らない。「うっかり削除」が大事故に直結。
理由2
関連データへの波及
親を消すと子も消える（CASCADE）。連鎖の範囲を正しく把握しないと予想外の被害。
理由3
監査・履歴の要求
業務では「誰がいつ消したか」「過去にどんなデータがあったか」を残す要求が多い。
理由4
法令・コンプライアンス
「一定期間データを保持する義務」「個人情報は削除する義務」など、法的要件も絡む。

02

参照整合性の設計思想

REFERENTIAL INTEGRITY

参照整合性とは「参照は必ず実在する対象を指す」という原則。外部キー制約はこれを DB が保証する仕組み。なぜ重要かを設計思想から理解します。

integrity-philosophy なぜ参照整合性が大事か
// 参照整合性が守る3つのこと
守ること1
孤児データを作らない
「存在しない科目を参照する記録」を防ぐ。集計やJOINで不可解な動作を起こさない。
守ること2
アプリのバグから守る
アプリ側のバグで不正なデータを書こうとしても、DBが最後の砦として拒否。
守ること3
データの信頼性を担保
「このデータベースの参照は全て正しい」と保証される。分析・移行・連携が安全に。
「DBで守るか、アプリで守るか」論争

「外部キー制約を使わず、アプリ側でチェックすればいい」という考え方もあります。マイクロサービスや超大規模では外部キーを使わないことも。但し「DBが守る方が確実」が一般論。Cでは外部キーを使う標準的な設計を学びます。

// 参照整合性の3つの保証 // ─── 保証1: INSERT時 ─── // 子を追加する時、親が存在するか確認 INSERT INTO study_records (subject_id, ...) VALUES (999, ...); // → subject_id=999 が subjects になければ拒否 // ─── 保証2: UPDATE時 ─── // 子の外部キーを変更する時も確認 UPDATE study_records SET subject_id = 999 WHERE id = 1; // → 999 が存在しなければ拒否 // ─── 保証3: DELETE時 ─── // 親を消す時、子をどうするか制御 DELETE FROM subjects WHERE id = 1; // → ON DELETE の設定に従う // ─── 外部キーなしの悲劇（実例）─── // 外部キーがないシステムで起きたこと： // 1. アプリのバグで subject_id=0 の記録が大量発生 // 2. 集計画面で「不明な科目」が表示される // 3. 原因調査に何日もかかる // 4. データ修正の特別バッチを作る羽目に // → 外部キーがあれば「最初から防げた」 // ─── 設計思想 ─── // 「不正なデータは、そもそも // 保存させない」 // 「後で掃除する」より // 「最初から汚さない」が鉄則
03

CASCADEの罠と論理削除

CASCADE TRAP & SOFT DELETE

本編で採用した CASCADE は便利ですが業務では危険な面も。代わりに論理削除（ソフトデリート）が主流です。両者を深く比較します。

cascade-vs-soft 物理削除と論理削除
// 物理削除 vs 論理削除
物理削除
DELETE文でデータを消す（Vol.14本編）
DBから本当に消える。CASCADE で連鎖削除。シンプルだが復元不可。
論理削除
deleted_at 等で「削除済み」を記録
データは残し、「削除フラグ」で見えなくする。復元可能・履歴が残る。業務の主流。
観点物理論理
復元不可可能
履歴残らない残る
実装簡単やや複雑
データ量減る増え続ける

業務システムは論理削除が主流。「消したつもりが取り返せない」リスクを避けるため。

// 論理削除の実装 // ─── 列を追加 ─── ALTER TABLE subjects ADD COLUMN deleted_at DATETIME DEFAULT NULL; // ─── 「削除」= UPDATE ─── UPDATE subjects SET deleted_at = NOW() WHERE id = 1; // → 物理的には残っている // → deleted_at に日時が入った // ─── 「表示」は未削除のみ ─── SELECT * FROM subjects WHERE deleted_at IS NULL; // → 削除済みは出てこない // ─── 「復元」も簡単 ─── UPDATE subjects SET deleted_at = NULL WHERE id = 1; // → 復活！ // ─── CASCADE の罠（実例）─── // ECサイトで「商品カテゴリ」を削除したら // CASCADE で： // ・カテゴリ内の全商品が消えた // ・その商品の全注文履歴も消えた // ・売上データが吹き飛んだ // → 「カテゴリ1つ」のつもりが大惨事 // ─── 論理削除なら ─── // カテゴリに deleted_at をセットするだけ // → 商品も注文も無事 // → 表示だけ消える // → 間違えても復元できる // ─── Cの方針 ─── // 学習のため物理削除＋CASCADEを学んだ。 // 実務では「論理削除を検討する」 // 引き出しを持っておく。
POINT ──

「CASCADE は本当に必要か」を常に問うこと。「親子が完全に運命共同体」（注文と注文明細など）なら CASCADE が適切。しかし「記録・履歴として価値がある」データは、安易に CASCADE で消すと後悔します。迷ったら RESTRICT か論理削除。「消せる」より「消えない」方が安全な場面が業務では多い。

削除設計の判断フロー

DELETE DESIGN FLOW

「どの削除方式を選ぶか」を整理した判断フロー。業務でこの判断を求められた時の指針になります。

delete-decision 削除方式の選び方
// 判断のステップ
Q1. 復元・履歴が必要?
YES → 論理削除
NO → 次へ（物理削除）
Q2. 子データは親と運命共同体?
YES → ON DELETE CASCADE
NO → 次へ
Q3. 子があれば削除を止めたい?
YES → ON DELETE RESTRICT
子を残したい → SET NULL
// 業務での具体例 // ─── 例1: 注文と注文明細 ─── // 注文を消したら明細も不要 // → 運命共同体 → CASCADE // （ただし注文は普通「論理削除」が多い） // ─── 例2: 顧客と注文履歴 ─── // 顧客を消しても注文履歴は残したい // （売上分析・法的保持義務） // → RESTRICT or 論理削除 // ─── 例3: 部署と社員 ─── // 部署を消しても社員は残る // （別部署に異動するまで未所属） // → SET NULL（部署なし状態に） // ─── 例4: 学習記録アプリ（Vol.14）─── // 科目を消したら記録も不要 // （個人の学習記録・履歴義務なし） // → CASCADE（本編で採用） // ※ 実務なら論理削除も検討 // ─── 判断の優先順位 ─── // 1. まず「復元・履歴が要るか」 // → 要るなら論理削除一択 // 2. 物理削除なら「連鎖の妥当性」 // → CASCADE / RESTRICT / SET NULL // 3. 迷ったら安全側（RESTRICT/論理）
05

複合外部キーとインデックス

COMPOSITE FK & INDEX

外部キーには複数列にまたがる「複合外部キー」もあります。また外部キー列にはインデックスが重要。発展的だが実務で役立つ知識。

composite-index 複合外部キーとINDEX
// 押さえるべき2点
複合外部キー
複数列で1つの外部キー
親の主キーが複合キー（複数列）の場合、子も同じ列数で参照。「テナントID＋ユーザーID」など。
外部キーとINDEX
外部キー列にはインデックスを
外部キー列はJOINや参照チェックで頻繁に使われる。インデックスがないと遅い。MySQLは外部キー作成時に自動でインデックスを作ることが多い。
削除時のパフォーマンス

CASCADE 削除で大量の子を消す時、子テーブルの外部キー列にインデックスがないと非常に遅くなる。「親1件削除に何分もかかる」事故も。外部キー列のインデックスは必須と考える。

// 複合外部キーとインデックス // ─── 複合外部キーの例 ─── // 親：複合主キー CREATE TABLE enrollments ( tenant_id INT, user_id INT, ... PRIMARY KEY (tenant_id, user_id) ); // 子：複合外部キーで参照 CREATE TABLE submissions ( id INT PRIMARY KEY, tenant_id INT, user_id INT, ... FOREIGN KEY (tenant_id, user_id) REFERENCES enrollments(tenant_id, user_id) ); // → 2列セットで参照整合性を保証 // ─── 外部キー列のインデックス ─── // study_records.subject_id は // 外部キー → 自動でインデックスが付く // （MySQL の場合） // 確認： SHOW INDEX FROM study_records; // → subject_id のインデックスがある // ─── なぜインデックスが要るか ─── // 1. JOIN が速くなる // （subjects と study_records の結合） // 2. CASCADE 削除が速くなる // （親削除時に子を探すのに使う） // 3. 参照チェックが速くなる // （INSERT時の親存在確認） // → 外部キー = 実質インデックス必須

正規化との関係

NORMALIZATION

外部キーは「正規化」の産物。S1 で学んだ「テーブルを分ける」設計が、外部キーで結ばれる。データ設計の全体像を俯瞰します。

normalization 正規化と外部キー
// 正規化のおさらい

正規化とは「データの重複をなくし、1つの事実を1箇所に持つ」設計。S1 で学んだ「subjects と study_records を分ける」がまさにこれ。

テーブルを分けると、関連を表すために外部キーが必要になる。「正規化 → テーブル分割 → 外部キーで結ぶ」が一連の流れ。

非正規化の例
1テーブルに全部詰める
「記録テーブルに科目名も色も持つ」→ 科目名変更時に全記録を更新する羽目に。重複・不整合の温床。
正規化の例
科目と記録を分け、外部キーで結ぶ
科目情報は subjects に1箇所。記録は subject_id で参照。科目名変更は1箇所だけ。Cの設計。
// 正規化と外部キーの流れ // ─── 非正規化（悪い例）─── study_records: id | subject_name | subject_color | minutes 1 | Java | red | 60 2 | Java | red | 90 3 | Java | red | 120 // 問題： // ・"Java" "red" が何度も重複 // ・色を変えたい時、全行を UPDATE // ・打ち間違いで "java" "Java" が混在 // ─── 正規化（良い例・Cの設計）─── subjects: // 科目は1箇所 id | name | color 1 | Java | red study_records: // 参照で繋ぐ id | subject_id | minutes 1 | 1 | 60 2 | 1 | 90 3 | 1 | 120 // 利点： // ・"Java" は1箇所だけ // ・色変更は subjects の1行だけ // ・subject_id で確実に参照 // ─── 外部キーが繋ぐ ─── study_records.subject_id → subjects.id を参照 // 正規化でテーブルを分けたから // 外部キーで「関連」を表現する // ─── 設計の流れ（C全体）─── // S1: 1対多を見極める（正規化） // M1: 2テーブルを作る // M4: JOIN で関連を取得 // M14: 外部キーで整合性を保証 // → データ設計の一貫した物語

制約の運用とマイグレーション

OPERATION & MIGRATION

外部キー制約は「運用」も大事。本番DBへの制約追加、スキーマ変更（マイグレーション）の注意点を整理します。

operation 制約の運用知識
// 運用の注意点
注意1
既存データへの後付けは要注意
本編でも触れた通り、孤児データがあると制約追加が失敗。本番では事前にデータクレンジング。
注意2
マイグレーションツールで管理
スキーマ変更は Flyway / Liquibase 等でバージョン管理。「いつ・誰が・何を変えたか」を記録。手動 ALTER は避ける。
注意3
外部キーの一時無効化
大量データ投入時など、一時的に制約チェックを止めることがある（FOREIGN_KEY_CHECKS=0）。慎重に。
本番への適用は段階的に

本番DBへのスキーマ変更は「テスト環境で検証 → ステージング → 本番」と段階的に。外部キー追加はテーブルロックが発生しうるので、アクセスの少ない時間帯に行うのが定石。

// マイグレーションの考え方 // ─── Flyway のマイグレーションファイル例 ─── // V1__create_subjects.sql CREATE TABLE subjects (...); // V2__create_study_records.sql CREATE TABLE study_records (...); // V3__add_foreign_key.sql ALTER TABLE study_records ADD CONSTRAINT fk_subject FOREIGN KEY (subject_id) REFERENCES subjects(id) ON DELETE CASCADE; // V4__add_last_studied_on.sql（Vol.13） ALTER TABLE subjects ADD COLUMN last_studied_on DATE; // ─── 利点 ─── // ・スキーマ変更が「履歴」として残る // ・チーム全員が同じDB構造になる // ・「いつ何を変えたか」が明確 // ・新環境構築が自動化される // ─── 手動 ALTER の問題 ─── // 「開発者Aだけが手で ALTER した」 // → 開発者Bの環境と構造が違う // → 「自分の環境では動くのに」問題 // ─── Cでの方針 ─── // 学習なので手動 ALTER で進めた。 // 実務では「マイグレーションツール」 // を使うと知っておく。 // ─── 外部キーチェックの一時停止 ─── // （大量データ投入時など特殊ケース） SET FOREIGN_KEY_CHECKS = 0; // ... 大量 INSERT ... SET FOREIGN_KEY_CHECKS = 1; // ⚠ 整合性チェックを止めるので // 投入データが正しい前提でのみ使う

外部キーのハマりどころ

FK PITFALLS

外部キー制約まわりで業務でよくある罠を4つ。本編の内容と合わせて押さえます。

fk-pitfalls 外部キーの罠
// よくある罠
PITFALL 1
CASCADE の連鎖を把握していない
原因：A→B→C と多段で CASCADE があると、Aを消すとC まで連鎖。
対処：CASCADE の連鎖範囲を図で把握。深い連鎖は危険。
PITFALL 2
テストデータの削除順序でエラー
原因：RESTRICT で、親（subjects）を子（records）より先に消そうとした。
対処：削除は子→親の順。または CASCADE。
PITFALL 3
型の不一致で外部キーが作れない
原因：親の id が INT なのに子の subject_id が BIGINT 等。
対処：親子の型・文字コードを完全一致させる。
PITFALL 4
制約違反例外をユーザーに見せてしまう
原因：RESTRICT で削除拒否された時、生の例外がエラー画面に出る。
対処：例外をキャッチし「記録があるため削除できません」と分かりやすく表示。
// PITFALL 4: ユーザーフレンドリーな処理 // ─── ✗ NG: 生の例外を見せる ─── @PostMapping("/subjects/{id}/delete") public String delete( @PathVariable Integer id) { subjectService.delete(id); // RESTRICT で例外 → 500エラー画面 // → ユーザーには意味不明 return "redirect:/subjects"; } // ─── ✓ OK: 例外を捕まえて伝える ─── @PostMapping("/subjects/{id}/delete") public String delete( @PathVariable Integer id, RedirectAttributes ra) { try { subjectService.delete(id); } catch ( DataIntegrityViolationException e) { ra.addFlashAttribute("error", "この科目には記録があるため" + "削除できません。"); } return "redirect:/subjects"; } // → ユーザーに分かる言葉で伝える // → RedirectAttributes でメッセージ表示 // ─── 削除順序の罠（PITFALL 2）─── // RESTRICT の場合の正しい削除順： // 1. 先に子を消す DELETE FROM study_records WHERE subject_id = 1; // 2. 次に親を消す DELETE FROM subjects WHERE id = 1; // （CASCADE なら親だけ消せば連鎖）
