02

UTC vs JST の世界観

UTC vs JST

タイムゾーンは日付処理の最初の判断。「世界標準時（UTC）」と「日本時間（JST）」のどちらを基準にするか、で設計が変わります。

timezone-basics タイムゾーンの基礎
// タイムゾーンの基本
UTC
協定世界時（Coordinated Universal Time）
世界共通の基準時刻。タイムゾーンの「ゼロ点」。各国の時刻はUTCからのオフセットで表現される。
JST
日本標準時（Japan Standard Time）
UTC +9時間。日本の時刻。サマータイムなし。
原則
国際展開ならUTC基準
DBにはUTCで保存し、画面表示時にユーザーのタイムゾーンに変換するのが現代の主流。「保存はUTC、表示はローカル」。
例外
国内専用ならJST直接でOK
日本人だけが使うシステムなら、DBにもJSTで保存して問題なし。学習記録アプリはこちら。
// 同じ瞬間を異なる表現で見る // ある瞬間：日本で 2026-05-20 09:00 // ─── 表現1: ローカル時刻 + TZ ─── 2026-05-20T09:00+09:00 // → 2026年5月20日 9:00、UTC+9（日本） // ─── 表現2: UTC ─── 2026-05-20T00:00Z // → UTCで 0:00 (Z = ゼロタイム) // 同じ瞬間だが表現が異なる // ─── 落とし穴 ─── // 例: 「日本で2026-05-20の出来事」 // ・JST 9:00 を UTC で保存 → 2026-05-20 00:00 UTC // → 表示時にJSTに戻すと 2026-05-20 9:00 ✓ // ・JST 5:00 を UTC で保存 → 2026-05-19 20:00 UTC // → 日付だけ取り出すと 5-19 になる… // → 「日本では5-20の出来事」のはずが // UTCでは5-19扱いになる罠 // → これがタイムゾーン地獄の入り口 // 学習記録アプリの判断 // 対象：日本のユーザーのみ // → JST 直接で OK // → 「学習日」は LocalDate（時刻なし） // → DBは DATE 型 // → タイムゾーン考慮は実質不要 // ─── 将来海外展開する場合は ─── // LocalDate → LocalDateTime / Instant // DATE → DATETIME / TIMESTAMP // に変えていく必要あり
POINT ──

「とりあえずタイムゾーン考慮しなくていいや」と思って国内専用で作ると、後から国際展開する時にデータマイグレーションが地獄になります。新規開発時に「将来の海外展開可能性」を関係者に確認し、可能性ありなら最初からUTC基準で設計するのが鉄則。

03

DATE / DATETIME / TIMESTAMP の使い分け

DB TYPE CHOICE

MySQL の日付関連型は3つ：DATE、DATETIME、TIMESTAMP。業務での使い分けを整理します。

db-types 3つの型の違い
// MySQLの日付型
型範囲用途
DATE1000-01-01 〜 9999-12-31日付のみ（時刻不要）
DATETIME1000-01-01 〜 9999-12-31日時（タイムゾーン非対応）
TIMESTAMP1970-01-01 〜 2038-01-19日時（UTC自動変換対応）
使い分けの原則：
  • 「学習日」「誕生日」など日付だけ → DATE
  • 「予定開始時刻」「ログ記録時刻」など日時 → DATETIME
  • 「レコード作成時刻」などUTC基準で残したい → TIMESTAMP
// 学習記録アプリでの選択 CREATE TABLE study_records ( id INT PRIMARY KEY, subject_id INT, studied_on DATE, // ✓ 日付のみ minutes INT, memo VARCHAR(200), created_at DATETIME // ✓ レコード作成時刻 DEFAULT CURRENT_TIMESTAMP ); // なぜこの選択か： // ・studied_on は「Java を 5月20日 勉強した」 // → 時刻不要、日単位の集計で十分 // → DATE が最適 // ・created_at は「いつ記録登録したか」 // → 監査・履歴用 // → DATETIME（時刻まで欲しい） // → 国内専用なら DATETIME でOK // → 国際展開なら TIMESTAMP の方が安心 // TIMESTAMP の特徴と落とし穴 // 特徴1: 2038年問題 // TIMESTAMP の上限は 2038-01-19 03:14:07 UTC // → それ以降のシステム稼働は危険 // → 長期運用なら DATETIME を選ぶ判断もある // 特徴2: タイムゾーン自動変換 // MySQL内部では UTC で保存 // 取得時にセッションのタイムゾーンに変換 // → 「DBサーバの設定でタイムゾーンが変わると // 返ってくる値が変わる」という地雷 // ─── 推奨の現代的選択 ─── // 「日付のみ」 → DATE // 「日時・国内のみ」→ DATETIME // 「日時・国際的」 → DATETIME + 別途タイムゾーン列 // または UTC で保存し型は DATETIME

Java側の型選択

JAVA TYPE CHOICE

Java 8 以降の java.time パッケージには多数の日付型があります。それぞれの使い分けを整理。

java-types java.time の主要型
// 主要な6つの型
型表すもの用途
LocalDate日付（年月日）誕生日・学習日
LocalTime時刻（時分秒）営業時間など
LocalDateTime日時（TZなし）国内向け日時
ZonedDateTime日時＋TZ多TZ対応のイベント
InstantUTCの瞬間ログ・監査・国際API
Duration / Period期間計算結果

使い分け：「日付だけ」なら LocalDate、「日時」なら LocalDateTime、「グローバル」なら Instant。

// 学習記録アプリでの選択 public class StudyRecord { private Integer id; private Integer subjectId; private LocalDate studiedOn; // 学習日 private Integer minutes; private String memo; private LocalDateTime createdAt; // 作成時刻 } // DB↔Java の対応： // DATE ↔ LocalDate // DATETIME ↔ LocalDateTime // TIMESTAMP ↔ Instant or LocalDateTime // 古い型を使わない理由 // ─── ✗ 使ってはいけない型 ─── java.util.Date // 古い・混乱の元 java.util.Calendar // 古い・複雑 java.sql.Date // JDBC専用・避ける java.sql.Timestamp // 同上 // これらの問題： // ・スレッドセーフでない // ・タイムゾーン管理が曖昧 // ・APIが直感的でない（月が0始まり等） // ・mutable（変更可能）で副作用が出る // ─── ✓ Java 8以降の現代的型 ─── java.time.LocalDate java.time.LocalDateTime java.time.Instant java.time.ZonedDateTime // メリット： // ・スレッドセーフ（不変オブジェクト） // ・タイムゾーン扱いが明確 // ・直感的なAPI // ・計算メソッドが豊富 // ─── Cでの方針 ─── // 全15巻で java.time のみ使用 // java.util.Date は一切登場しない
POINT ──

レガシーコードに java.util.Date が混じっていることはよくあります。新規コードでは絶対に使わない。既存コードで使われていても、触る機会があれば java.time に置き換えるのが現代の方針。これは業界共通のコンセンサスです。

05

業務での落とし穴3選

FIELD DISASTERS

日付処理の業務での災厄事例を3つ紹介。「そんなのある？」と思える内容ですが、すべて実話ベースです。

disasters 業務での災厄
// 落とし穴3選
DISASTER 1
「月末跨ぎで請求が消えた」
月締めの請求処理で、「翌月1日 00:00 までの取引を当月分」とすべきところ、「当月末 23:59:59 まで」と書いてしまった。うるう秒のせいで2分の取引が当月にも翌月にも入らず消えた。年に1度の伝説的バグ。
対策：境界条件は <（未満）で書く。WHERE t.created_at >= '2026-05-01' AND t.created_at < '2026-06-01' が定石。
DISASTER 2
「翌日が誕生日のはずなのに今日が誕生日扱い」
海外サーバ（UTC）に日本人ユーザーの誕生日を保存。アメリカ深夜に日本人がアクセス → JSTでは翌日、UTCではまだ今日 → 翌日が誕生日のはずなのに今日が誕生日として表示。
対策：日付比較はユーザーのタイムゾーンで判定。「日本人ユーザー＝JSTで判定」を関数化。
DISASTER 3
「3月1日のはずなのに2月29日と表示」
「現在から3年後の月日」を計算するロジック。3年後がうるう年ではない時、2024-02-29 → 2027-02-29 として計算するも、その日は存在せず例外。あるいは無理やり 02-28 にして「3月1日」が消える。
対策：LocalDate の plusYears(3) はうるう年補正してくれる。手動計算ではなく java.time の標準メソッドを使う。
// DISASTER 1 の正しい書き方 // ─── ✗ NG: BETWEEN 使用 ─── SELECT * FROM transactions WHERE created_at BETWEEN '2026-05-01 00:00:00' AND '2026-05-31 23:59:59'; // 問題： // 23:59:59.001 〜 23:59:59.999 の取引が漏れる // マイクロ秒精度の場合さらに細かく抜ける // ─── ✓ OK: 半開区間 ─── SELECT * FROM transactions WHERE created_at >= '2026-05-01' AND created_at < '2026-06-01'; // → 「5月開始から6月開始の手前まで」 // → ぴったり1ヶ月分を漏れなく取得 // DISASTER 3 の正しい書き方 // ─── ✗ NG: 手動計算 ─── int year = today.getYear() + 3; int month = today.getMonthValue(); int day = today.getDayOfMonth(); LocalDate threeYearsLater = LocalDate.of(year, month, day); // 2024-02-29 + 3年 → LocalDate.of(2027,2,29) // → 例外 DateTimeException // ─── ✓ OK: 標準メソッド ─── LocalDate threeYearsLater = today.plusYears(3); // 2024-02-29 + 3年 → 2027-02-28 // （標準メソッドが自動補正） // ─── 教訓 ─── // 日付計算は「自分で頑張らない」 // java.time の標準メソッドを使う
POINT ──

「日付処理のバグ」は「年に数回しか起きないが、起きた時にデータ整合性を破壊する」のが特徴です。月末・年末・うるう年・サマータイム切替日── これらの「特殊な日」すべてを念頭においたテストが業務開発では必要。「2月29日に動くか」を意識的にテストするだけでも、品質が一段上がります。

ハマりどころ集

PITFALLS

日付処理の追加ハマりどころ4選。本編Vol.7では触れなかった「業務寄り」の罠です。

extra-pitfalls 業務寄りの罠
// 業務寄りの罠
PITFALL 1
JST と UTC の境界での集計ズレ
原因：DBに UTC で保存、画面で JST 表示。「5月20日の合計」を集計すると、UTC では 5月20日 0:00〜5月21日 0:00（JST 9:00〜翌日 9:00）の範囲。
対処：集計時に JST に変換した日付でグループ化。DATE_FORMAT(CONVERT_TZ(created_at, 'UTC', 'Asia/Tokyo'), '%Y-%m-%d') など。
PITFALL 2
うるう年判定を自前で書く
原因：「4年に1度」と思って year % 4 == 0 で判定する。これは間違い（100で割れて400で割れない年はうるう年ではない）。
対処：Year.isLeap(year) や LocalDate 標準メソッドを使う。
PITFALL 3
古い API と新しい API の混在
原因：レガシーコードに java.util.Date、新コードに LocalDate。変換しながら共存するが、変換時のバグが頻発。
対処：境界に変換ヘルパを置き、全コードでどちらか統一する移行計画を立てる。
PITFALL 4
DST（サマータイム）の存在を忘れる
原因：海外サーバ運用で、年2回サマータイム切替日に時刻が1時間ズレる。その1時間は2回存在する／存在しない。
対処：日本にはDSTがないが、海外のシステムでは ZonedDateTime を使い、タイムゾーン情報を必ず持たせる。
// PITFALL 2: うるう年判定の罠 // ─── ✗ NG: 自前判定 ─── boolean isLeapYear = (year % 4 == 0); // → 2100年も「うるう年」と判定する // → 実際の2100年はうるう年ではない // ─── 正しいルール ─── // ① 4で割り切れる → うるう年候補 // ② ただし 100で割り切れる → 平年 // ③ ただし 400で割り切れる → うるう年 // 例: // 2024: 4で割れる → うるう年 ✓ // 2100: 4で割れる、100でも割れる、400では割れない → 平年 // 2000: 4で割れる、100で割れる、400でも割れる → うるう年 ✓ // ─── ✓ OK: 標準メソッド ─── import java.time.Year; boolean isLeap = Year.isLeap(2100); // → false（正しい） boolean isLeap2 = Year.isLeap(2000); // → true（正しい） // ─── 教訓 ─── // 「自分でカレンダーを実装しない」 // java.time を信じて使う // PITFALL 4: DST の罠 // ─── アメリカでの典型例 ─── // 2026-03-08 2:00 AM 米国DST 開始日 // 1:59:59 → 3:00:00 にジャンプ // → 2:00〜2:59 は「存在しない時刻」 // 2026-11-01 2:00 AM 米国DST 終了日 // 1:59:59 → 1:00:00 に戻る // → 1:00〜1:59 は「2回存在する時刻」 // 国内アプリには影響なし // 海外展開する時は要注意
