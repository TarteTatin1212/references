02

「関連を取得する」とは何か

WHAT DOES "FETCH RELATED" MEAN

そもそも「関連オブジェクトを取得する」とはどういうことか。Java側の表現に立ち戻って整理します。

fetch-meaning 取得の本質
// Java側の「関連を持つ」の意味

S1で学んだ通り、DBでは 外部キーは「多」側に置かれます。一方Java側では、画面に出しやすいよう親に子のリストを持たせるのが自然な設計でした。

「関連を取得する」とは、この「親に子のリストを詰める」処理のことを指します。

3つの戦略の本質は、この「詰める処理」を
① いつやるか
② 何回SQLを発行するか
③ どれだけメモリに保持するか
── の組み合わせ違いです。
// 「関連を取得する」のイメージ // ─── 取得前 ─── Subject { id: 1, name: "Java", records: null // 空の入れ物 } // ─── 取得後（関連を詰めた状態）─── Subject { id: 1, name: "Java", records: [ // ← ここに詰める StudyRecord(id=1, ...), StudyRecord(id=2, ...), StudyRecord(id=3, ...) ] } // 戦略の違いは「詰め方の違い」だけ // 結果のオブジェクト構造は同じ。
03

戦略1: 都度取得（Vol.2の方式）

STRATEGY 1: FETCH-PER-PARENT

本編Vol.2でやった方式です。親を取った後、ループで各親に対して子を取る。最も素朴な戦略。

strategy-1 都度取得の特徴
// 都度取得
STRATEGY 1
都度取得（Fetch-per-parent）

親を1回取得→各親に対して子を1回ずつ取得→Service層で合体。

✓ 実装が直感的・分かりやすい
✓ 子を取得するSQLが単純
✓ 親と子のSQLを別Mapperに分離できる
✗ SQLが (1+N) 回発行される（N+1問題）
✗ DBとの通信回数が多い → 遅延の原因

業務での適用シーン：親の件数が常に少ない場合（5〜10件くらいまで）。「設定マスタ」「カテゴリ一覧」のような、件数が予測可能な小規模データ。

// SQL発行回数の計算 // 親=N件 として // ① 親の取得 SELECT * FROM subjects; // → 1回 // ② 各親に対して子の取得 for (each subject) { SELECT * FROM study_records WHERE subject_id = ?; } // → N回 // = 合計（1 + N）回 // N=5 なら 6回 // N=100 なら 101回 // N=10000 なら 10001回 // → Nに比例して悪化する // これが「N+1問題」 // 業務で最も警戒される性能課題のひとつ
POINT ──

本編Vol.2で「あえて」この方式を採用したのは教育的意図です。N+1問題は「知ってから直す」ほうが理解が深まる。Vol.5で本格的に対策を学びますが、その前提として「これは問題だ」と気付ける感性を養うのが今の段階です。

04

戦略2: 一括取得（JOIN）

STRATEGY 2: EAGER JOIN

1回のSQLで親と子の全部を取得する方式。SQLが JOIN を使う形になります。Vol.4でresultMap + collection を使って実装します。

strategy-2 一括取得の特徴
// 一括取得（JOIN）
STRATEGY 2
一括取得（Eager Join）

親テーブルと子テーブルを JOIN し、1回のSQLで全部取得。MyBatis では resultMap + collection で親子を組み立てる。

✓ SQLは常に1回（N+1問題なし）
✓ DB通信が最小化
✓ 一覧画面で必ず子も必要なケースに最適
✗ JOIN結果の行が多くなる（重複あり）
✗ 子を一切使わない時もメモリに読み込む
✗ resultMapの書き方が初見ややこしい

業務での適用シーン：「親一覧で必ず子の概要も見せる」画面。本編Vol.2のような科目別記録一覧はまさにこれが理想。

// 一括取得のSQL（イメージ） SELECT s.id AS subject_id, s.name, s.color, r.id AS record_id, r.studied_on, r.minutes, r.memo FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id ORDER BY s.id, r.studied_on; // 結果（行が冗長になる） // ────────────────────────── // sid|name |color |rid|studied |min // ────────────────────────── // 1 |Java |#c43030| 1 |05-08 | 60 // 1 |Java |#c43030| 2 |05-10 | 60 // 1 |Java |#c43030| 3 |05-12 | 90 // 2 |Spr |#5a8a4a| 6 |05-11 | 45 // 4 |HTML |#c4862e|null|null |null // ────────────────────────── // MyBatisの resultMap が裏で // この冗長な行を Subject オブジェクト // + 子の List に組み立て直す // → 詳細はVol.4 / S4で
POINT ──

「JOIN で1回SQLが正義」と単純化しがちですが、業務では「常に良い」わけではない。子のデータが大量になる場合、JOIN結果が爆発的に増え、メモリ・ネットワークの両方を圧迫します。件数と用途を見て選ぶのが鉄則。

05

戦略3: 遅延ロード

STRATEGY 3: LAZY LOAD

子は最初取得せず、必要になった時だけ取得する方式。「必要になるかどうか分からない」「ユーザーが詳細を見る時だけ取れば良い」という場合に有効。

strategy-3 遅延ロードの特徴
// 遅延ロード
STRATEGY 3
遅延ロード（Lazy Load）

親だけ取得 → 子のリストは空のまま → アクセスされた時に初めて子を取得する。

✓ 初回ロードが軽い
✓ 子を使わないケースで無駄ゼロ
✓ 大規模データに強い
✗ アクセスの度にSQLが発生
✗ 意図しないN+1問題が起きやすい
✗ Hibernate/JPA 側の機能で、MyBatisでは標準ではない

業務での適用シーン：詳細画面で「クリックして展開」「タブを切り替えた時に取得」など、UIのアクションをトリガーにした取得。

// 遅延ロードのイメージ // ─── 一覧画面（親だけ取得）─── List<Subject> subjects = mapper.findAll(); // この時点で records は空（null） // ─── 詳細画面でアクセスされたら ─── Subject s = ...; s.getRecords(); // ↑ ここで初めてSQL発行 // (JPAのHibernateならProxy機構で実現) // ─── MyBatisでは ─── // 遅延ロードは公式機能としてある // （fetchType="lazy"）が、 // 実務では明示的なService分離（ // findSubject() と findRecords()） // の方が読みやすいことが多い // ─── 隠れた罠 ─── // 「ループの中で getRecords() を呼ぶ」 // → 結果的にN+1問題が出現 // 遅延ロードは便利だが「意図しない // アクセス」を生み出しやすい
TIP ──

Cでは遅延ロードは扱わない方針です。理由は「素朴な3層設計＋明示的なService分離」の方が初学者には「いつSQLが発行されるか」が見えるので学びやすいから。遅延ロードはJPA中心の世界で本格活用される機能なので、JPA編が登場した時に深掘りすべきテーマです。

業務での選択基準

CHOOSING IN PRACTICE

3戦略を整理しましたが、業務で「どう選ぶか」こそが本題。3つの判断軸で決めます。

selection-criteria 選び方の3軸
// 3つの判断軸
Q1. その画面で子データを必ず使う？
必ず使う → 一括取得（JOIN）
たまに使う → 都度取得 or 遅延
Q2. 親の件数は予測できる？多い？
少数で予測可能（〜10件） → 都度取得でもOK
多い・可変（数百〜） → 一括取得 or ページング必須
Q3. 子の量はどれくらい？
少ない → 一括取得で問題なし
多い（親1件あたり数千行） → 一括取得すると爆発する → 別画面で見せる
// 学習記録アプリの判断 // ─── 一覧画面（subjects + records）─── // Q1: 子を使う？ → 使う（一覧表示する） // Q2: 親の件数？ → 5〜数十（少ない） // Q3: 子の量？ → 各科目で数十〜数百 // → 結論：一括取得（JOIN）が最適 // → Vol.4 でこれに進化させる // ─── 統計画面（合計時間のみ）─── // Q1: 子の生データ使う？ → 使わない（合計だけ） // → 結論：JOINではなく集計クエリ（GROUP BY） // → Vol.9 で扱う // ─── 詳細画面（特定科目の記録のみ）─── // Q1: 親は1件だけ // → 結論：findSubject() + findRecords() の2回でOK // → 「都度取得」がそのまま使える

性能とメモリのトレードオフ

PERFORMANCE vs MEMORY

関連オブジェクト取得は「性能（速さ）」と「メモリ消費」のトレードオフになります。両方を最大化することはできない。「何を優先するか」が設計判断です。

tradeoff 3戦略のトレードオフ
// 戦略ごとの特性
戦略SQL回数メモリ遅延感
都度取得1+N回中悪い
一括取得1回大良い
遅延ロード必要時のみ小（初期）初期良／アクセス時に悪化リスク

結論：「DB通信を減らす」「メモリを減らす」のどちらかを選ぶ。両方は基本的に取れない。業務ではDB通信を減らす（=一括取得）が選ばれることが多い。

// なぜDB通信を減らすが優先か // DB通信の典型的なコスト： // ① ネットワーク往復 // 同一ホストなら1msオーダー // 別ホスト/AWSなら10〜数十ms // ② DBサーバの解析・実行 // 毎回パース・プランニング・実行 // ③ コネクションプールの専有 // 接続数が有限なので影響大 // SQL 100回 = 数百ミリ秒〜数秒 // SQL 1回 = 数ミリ秒 // → DB通信を減らす方が体感への影響大 // 一方メモリ： // 最近のサーバは16〜64GB // JOINで多少増えても許容範囲 // だから「迷ったらJOIN」が業界の傾向
POINT ──

「早すぎる最適化は諸悪の根源」（Donald Knuth）という有名な言葉があります。設計判断は「測ってから決める」のが鉄則。まず動かして、ボトルネックを測定して、必要なら直す──この順序を守るのが、業務開発の最良の進め方です。Vol.2が「あえて都度取得」だったのは、この精神に基づきます。

戦略選択のハマりどころ

PITFALLS

戦略選択でやりがちな3つの失敗。経験者から伝授される、現場の「あるある」です。

selection-pitfalls 戦略選びで失敗するパターン
// 戦略選びの罠
PITFALL 1
「とりあえずJOIN」で大規模化
最初は問題なかったJOINが、データ蓄積で結果セット爆発。プロジェクト後半で性能問題が表面化する典型。
対策：本気の業務システムならページング前提で設計する。
PITFALL 2
「とりあえず都度取得」で本番障害
テスト環境では10件で1+10=11回。本番では1000件で1+1000=1001回。テスト環境の少データで気づけないのが厄介。
対策：開発時から本番相当のデータ件数でテストする習慣を。
PITFALL 3
遅延ロードを過信して意図せぬN+1
JPAで List<Parent> parents = repo.findAll() → for (p : parents) p.getChildren() ── 知らずにN+1が発生。
対策：JPAでもJOIN FETCH を明示的に使うのが基本。
// 性能問題の検出方法 // ① SQLログを常に見る // logging.level.com.example.mapper=DEBUG // → 開発中、常にSQL発行を観察 // ② スロークエリログを設定 // MySQL側で「1秒以上のSQLを記録」 // → 本番運用で性能劣化を検出 // ③ 負荷テスト // JMeter / Gatling 等で本番相当を模擬 // → リリース前にボトルネック発見 // ─── 一般原則 ─── // 「気付けないものは直せない」 // → 観測の仕組みを最初から仕込む // 本編Vol.2でSQLログを見るのも、 // この観測習慣を身につけるため。
