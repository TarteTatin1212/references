02

欠損補完の戦略

GAP FILLING STRATEGY

本編Vol.12 では「0埋め」をしました。しかし業務では「どう埋めるか」に複数の選択肢があります。データの性質で使い分けます。

fill-strategies 欠損補完の4戦略
// 補完の4戦略
戦略1
ゼロ埋め（Vol.12採用）
記録なし＝0。「学習時間」「売上」など「やってない＝0」が自然なデータに最適。
戦略2
前値保持（直前の値を引き継ぐ）
「在庫残高」「口座残高」など「変化がなければ前と同じ」データ。欠損日は前日の値。
戦略3
線形補間（前後から推定）
「気温」「センサー値」など連続的に変化するデータ。前後の値から中間を推定。
戦略4
欠損のまま（NULL/null表示）
「測定できなかった」ことに意味がある場合。グラフで線を切るなどの表現。

判断軸：「そのデータで欠損は何を意味するか」。学習時間なら「やってない＝0」が自然。

// 戦略別のコード例 // ─── 戦略1: ゼロ埋め（Vol.12）─── int v = map.getOrDefault(d, 0); // 学習時間：やってない → 0分 // ─── 戦略2: 前値保持 ─── Integer v = map.get(d); if (v == null) { v = lastValue; // 前日の値を引き継ぐ } else { lastValue = v; } // 在庫残高：動きがなければ前日と同じ // ─── 戦略3: 線形補間 ─── // 前後の実測値から中間を計算 // 例: 5/8=10, 5/11=40 で 5/9,5/10 欠損 // → 5/9=20, 5/10=30 と推定 // （気温などの連続データ） // ─── 戦略4: 欠損のまま ─── Integer v = map.get(d); // null のまま // グラフで線を切る // 「データなし」を明示 // ─── 学習記録アプリの選択 ─── // 「サボった日 = 0分」が自然 // → 戦略1（ゼロ埋め）が正解 // → 習慣の途切れも可視化できる
POINT ──

「欠損をどう埋めるか」を間違えると分析が嘘になる。例えば在庫残高を「ゼロ埋め」すると「在庫が突然消えた」ように見える。逆に学習時間を「前値保持」すると「ずっと勉強し続けた」ように見える。データの意味に合った補完を選ぶのが鉄則です。

03

カレンダーテーブル方式

CALENDAR TABLE

本編では「Javaで欠損日を埋めた」が、業務ではSQLだけで完結させる方法もあります。それがカレンダーテーブル。大規模・複雑な時系列で有効。

calendar-table SQL で欠損日を埋める
// カレンダーテーブルとは

「全ての日付を持つテーブル」を用意し、集計結果と LEFT JOIN する。記録のない日もカレンダー側に存在するので結果に出る。

2つの作り方：

  • 物理テーブルを事前に作る（calendarテーブル）
  • 再帰CTEで動的に生成（MySQL 8.0+）
Java方式 vs SQL方式

Java方式（Vol.12）：シンプル・小規模向け・補完ロジックが柔軟。
SQL方式（カレンダーテーブル）：大規模・複雑な集計・SQL内で完結。学習記録アプリはJava方式で十分。

// カレンダーテーブル方式 // ─── 方法1: 物理テーブル ─── CREATE TABLE calendar ( d DATE PRIMARY KEY ); // 事前に日付を大量投入しておく // INSERT INTO calendar VALUES // ('2026-05-01'), ('2026-05-02'), ... // 集計と LEFT JOIN SELECT c.d AS studiedOn, COALESCE(SUM(r.minutes), 0) AS total FROM calendar c LEFT JOIN study_records r ON c.d = r.studied_on WHERE c.d BETWEEN '2026-05-08' AND '2026-05-15' GROUP BY c.d ORDER BY c.d; // → カレンダー側の全日付が残る // → 欠損日も 0 で出る // → Java の補完が不要に // ─── 方法2: 再帰CTE（MySQL 8.0+）─── WITH RECURSIVE dates AS ( SELECT DATE('2026-05-08') AS d UNION ALL SELECT d + INTERVAL 1 DAY FROM dates WHERE d < '2026-05-15' ) SELECT dates.d AS studiedOn, COALESCE(SUM(r.minutes), 0) AS total FROM dates LEFT JOIN study_records r ON dates.d = r.studied_on GROUP BY dates.d ORDER BY dates.d; // → テーブル不要・SQLだけで日付生成 // ─── どちらを選ぶか ─── // Java方式: 学習記録アプリ（小規模）✓ // 物理テーブル: 大規模・頻繁な集計 // 再帰CTE: モダンDB・中規模

移動平均と累積

MOVING AVG & CUMULATIVE

時系列の「傾向」を見るには、生データだけでは不十分。移動平均でノイズを消し、累積で総量を見る。業務分析の定番テクニック。

moving-cumulative 傾向を見る2手法
// 2つの分析手法
手法1
移動平均（ノイズを消す）
「直近7日の平均」を毎日計算。日々のばらつきを平滑化し、「上昇傾向か下降傾向か」を見やすくする。株価チャートでお馴染み。
手法2
累積（総量を見る）
「今月の累計学習時間」を日々加算。「目標まであとどれくらいか」が見える。進捗管理に最適。

ウィンドウ関数（OVER）を使うとSQLで計算可能。MySQL 8.0+ で利用可。Cの範囲外ですが知識として。

// ウィンドウ関数での計算（参考） // ─── 7日移動平均 ─── SELECT studied_on, SUM(minutes) AS daily, AVG(SUM(minutes)) OVER ( ORDER BY studied_on ROWS BETWEEN 6 PRECEDING AND CURRENT ROW ) AS moving_avg_7d FROM study_records GROUP BY studied_on ORDER BY studied_on; // 各日について「直近7日の平均」を計算 // → 日々のブレが平滑化される // ─── 累積（ランニングトータル）─── SELECT studied_on, SUM(minutes) AS daily, SUM(SUM(minutes)) OVER ( ORDER BY studied_on ) AS cumulative FROM study_records GROUP BY studied_on ORDER BY studied_on; // 各日について「その日までの累計」 // → 総量の推移が見える // ─── Java で計算する場合（Cの範囲）─── // 累積はループで簡単に計算できる int cumulative = 0; for (DailyStatView v : views) { cumulative += v.getTotalMinutes(); v.setCumulative(cumulative); } // → ウィンドウ関数を使わずとも // Java のループで実現可能 // 移動平均も同様に // 直近N件を平均すればよい
POINT ──

ウィンドウ関数（OVER）は時系列分析の強力な武器ですが、SQLの中ではやや高度。学習記録アプリの規模なら、累積や移動平均は Java のループで計算する方が分かりやすい。「SQLでやるべきか・Javaでやるべきか」は、複雑さと規模で判断。Cでは Java 寄りで十分です。

集計の粒度（日/週/月）

GRANULARITY

時系列集計は「どの単位でまとめるか」を選びます。日別・週別・月別── 見たい期間で粒度を変えるのが業務の定石。

granularity 粒度の使い分け
// 粒度と表示期間の関係
粒度適した表示期間
時間別1日〜数日
日別1週間〜2ヶ月
週別2ヶ月〜1年
月別1年〜数年

原則：「表示する点が20〜60個」くらいになる粒度を選ぶ。1年を日別（365点）で見ると細かすぎ、月別（12点）が適切。

// 粒度別の GROUP BY // ─── 日別（Vol.12）─── GROUP BY studied_on // studied_on が DATE 型ならそのまま // ─── 週別 ─── GROUP BY YEARWEEK(studied_on) // 年と週番号でグループ化 // ─── 月別 ─── GROUP BY DATE_FORMAT(studied_on, '%Y-%m') // "2026-05" でグループ化 // ─── 曜日別（時系列ではないが分析的）─── GROUP BY DAYOFWEEK(studied_on) // 「何曜日に勉強しがちか」 // ─── 時間帯別（DATETIME の場合）─── GROUP BY HOUR(created_at) // 「何時に記録しがちか」 // ─── 業務での粒度切り替え UI ─── // よくある実装： // 「日」「週」「月」のタブ切り替え @RequestParam(defaultValue = "day") String granularity // granularity で GROUP BY を切り替え // （<choose> などの動的SQLで・S8参照）

時系列のパフォーマンス

PERFORMANCE

時系列データは「蓄積し続ける」性質上、放置すると膨大になります。パフォーマンスの3つの定石を整理。

ts-perf 時系列の高速化
// 高速化の3定石
定石1
日付列にインデックス
studied_on や created_at にインデックス。期間絞り込みと集計が速くなる。時系列の最重要対策。
定石2
期間で必ず絞る（WHERE）
「全期間集計」は避け、「直近30日」など期間限定。S11 の「早く絞る」原則。
定石3
古いデータのアーカイブ
時系列は無限に増える。古いデータは別テーブルや別ストレージに退避。アクティブなテーブルを小さく保つ。

学習記録アプリは小規模なので定石1だけ意識すれば十分。大規模化したら定石2・3。

// 時系列の INDEX 設計 // ─── 日付列に INDEX ─── CREATE INDEX idx_studied_on ON study_records(studied_on); // 「日別集計」「期間絞り込み」が速くなる // ─── 複合 INDEX（科目×日付）─── CREATE INDEX idx_subject_studied ON study_records(subject_id, studied_on); // 「特定科目の日別推移」に効く // ─── 期間で絞る（定石2）─── SELECT studied_on, SUM(minutes) FROM study_records WHERE studied_on >= '2026-05-01' // 必ず絞る AND studied_on < '2026-06-01' GROUP BY studied_on; // 全期間ではなく月単位など // ─── アーカイブ戦略（定石3）─── // アクティブ: study_records（直近1年） // アーカイブ: study_records_archive（過去） // 定期バッチで古いデータを移動 // → アクティブテーブルが小さく保たれる // → 日々の集計が速い // ─── 学習記録アプリでは ─── // 個人利用の記録なので // 数年分でも数千行程度 // → INDEX があれば全く問題なし

集計の前計算（サマリー）

PRE-COMPUTATION

大規模な時系列で究極の高速化が「前計算」。集計を毎回するのではなく、事前に計算して保存しておく。S9 でも触れたサマリーテーブルの応用。

pre-compute 前計算の考え方
// 前計算とは

「画面表示のたびに集計」は、データが増えると重くなる。「1日1回バッチで集計し、結果を保存」しておけば、画面は保存済みの結果を読むだけで高速。

リアルタイム集計
毎回 SQL で集計
常に最新・実装シンプル。但しデータ増で遅くなる。Cの学習記録アプリはこれ。
前計算
バッチで集計→サマリーテーブル保存
画面は高速。但し「最新でない」リスクと、バッチ運用の複雑さ。大規模向け。

トレードオフ：「鮮度」と「速度」。リアルタイム性が必要なら都度集計、速度優先なら前計算。

// 前計算の実装イメージ // ─── サマリーテーブル ─── CREATE TABLE daily_summary ( summary_date DATE PRIMARY KEY, total_minutes INT, record_count INT, updated_at DATETIME ); // ─── バッチで集計＆保存（1日1回）─── // Spring の @Scheduled で定期実行 @Scheduled(cron = "0 0 1 * * *") // 毎日 AM1:00 に実行 public void aggregateDaily() { // 前日分を集計して daily_summary に保存 // （INSERT or UPDATE） } // ─── 画面表示は高速 ─── SELECT summary_date, total_minutes FROM daily_summary WHERE summary_date >= '2026-05-01' ORDER BY summary_date; // → 集計済みなので一瞬で返る // → study_records を毎回集計しない // ─── トレードオフ ─── // メリット: 画面が高速 // デメリット: // ・「今日のデータ」は翌朝まで反映されない // ・バッチの運用・監視が必要 // ・サマリーと実データの整合性管理 // ─── Cでの方針 ─── // 学習記録アプリは小規模 // → リアルタイム集計で十分 // → 前計算は「知識として」知っておく
POINT ──

「前計算（サマリーテーブル）」は大規模システムの定番ですが、「最初から導入するのは過剰設計」のことが多い。まずはリアルタイム集計で作り、「遅くなってきたら前計算を検討」が正しい順序。「早すぎる最適化は諸悪の根源」── これはエンジニアリングの格言です。

時系列のハマりどころ

TIME SERIES PITFALLS

時系列データで業務でよくある罠を4つ。本編Vol.12 で押さえた内容と対比します。

ts-pitfalls 時系列の罠
// よくある罠
PITFALL 1
欠損日を消したまま分析する
原因：SQL結果をそのまま使い、記録のない日が抜ける。「毎日やっている」ように錯覚。
対処：欠損日を埋める（Vol.12 の核心）。「ないことも情報」。
PITFALL 2
タイムゾーンで日付がズレる
原因：UTC保存のデータを日別集計する時、JSTとの境界で日付がずれる。
対処：集計前にタイムゾーン変換（S7参照）。studied_on は DATE なので Cでは問題なし。
PITFALL 3
月別集計で月の途中まで含めてしまう
原因：「今月」の範囲を BETWEEN 月初 AND 今日 としたが、月末の比較対象が揃わない。
対処：「完了した月だけ」集計するか、「経過日数で正規化」する。
PITFALL 4
LocalDate の不変性を忘れる
原因：d.plusDays(1) の戻り値を代入し忘れて無限ループ（Vol.12 でも警告）。
対処：d = d.plusDays(1)。java.time は全て不変。
// PITFALL 1: 欠損で錯覚する例 // ─── 欠損を消したまま ─── // 05-08 60分 // 05-10 90分 ← 05-09 が抜けてる // 05-12 120分 ← 05-11 が抜けてる // グラフにすると「ずっと右肩上がり」 // に見える → 実際はサボった日がある // ─── 欠損を埋めた（Vol.12）─── // 05-08 60分 // 05-09 0分 ← サボりが見える // 05-10 90分 // 05-11 45分 // 05-12 120分 // → 習慣の途切れが正しく見える // PITFALL 3: 月途中の罠 // ─── 今日が 5/15 の場合 ─── // ✗ NG: 単純比較 // 4月（30日分）: 600分 // 5月（15日分）: 350分 // → 「5月は減った」と誤解 // （まだ月の半分なのに） // ✓ OK: 1日あたりで正規化 // 4月: 600/30 = 20分/日 // 5月: 350/15 = 23分/日 // → 「5月の方がペースが速い」と正しく判断 // 「期間の長さ」を揃えて比較するのが鉄則
