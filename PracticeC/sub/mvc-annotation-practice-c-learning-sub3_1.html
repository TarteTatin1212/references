ANNOTATION vs XML ── HOW TO CHOOSE

本編Vol.3で2つの書き方を体験した今、当然湧いてくる疑問が「結局どっちを使うべき？」です。
別冊S3では、「いつアノテーションを使い／いつXMLを使うか」という使い分けの判断軸を、業界の流れ・現場の規約・チーム開発の観点で整理します。
結論を先に言うと：シンプルなSQLはアノテーション・複雑/動的なSQLはXML・混在を恐れない。
この判断ができるようになれば、「実装の道具を選ぶセンス」が一段上がります。

01

この別冊のテーマ

THEME

技術選択の場面では、必ず「どっちを使うか」が問われます。本巻で扱うのは「アノテーションかXMLか」ですが、根底にあるのは「実装の道具を選ぶセンス」です。これはCの全15巻を通じて磨くべき能力。

why-choose 選択の意義
// 「どっちでもいい」は嘘

初学者向け解説では「どっちでもいいです」と片付けられがち。でも実際にプロジェクトに入ると、規約として片方に統一されているか、SQLの性質に応じて使い分けているのが普通です。

「どっちでもいい」と「どっちを選ぶか」は別の問いです。前者は技術的に等価という意味、後者は業務上の判断。

この別冊が答えるのは 「業務上の判断」 の問い。
技術的にはどちらも動くという前提で、「より良い選択」を学びます。
// この巻で答える4つの問い // 問1: それぞれの長所・短所は？ // → SQLの可読性・型安全・保守性で比較 // 問2: いつどっちを使う？ // → SQLの複雑度・動的SQLの有無で判断 // 問3: 業界のトレンドはどう？ // → 「アノテ派」「XML派」「混在派」の現状 // 問4: プロジェクト規約はどう決めるべき？ // → チーム規模・スキル・既存資産で決まる // 結論を先に： // ・短いSQL → アノテーション // ・複雑SQL → XML // ・動的SQL → 必ずXML // ・混在を恐れない

改めて2つの書き方

TWO WAYS

同じSQLをアノテーションとXMLの両方で書いた場合の見た目の対比を改めて確認します。何が異なり、何が同じかを視覚的に把握するのが本セクションの目的。

side-by-side 同じSQLを2通りで書く
// 同じものを2通りで
要素アノテーションXML
SQLの実体Java内の文字列XML内のテキスト
SQL → メソッド対応@Select 直下id 属性
戻り値型指定Javaメソッドの戻り値resultType 属性
パラメータ@Param + #{...}同上
動的SQL不可（文字列連結のみ）<if> <where> <foreach>
編集の閾値Javaコンパイル必要XML編集→DevToolsで反映
// 同じ findAll() を2通りで // ─── ① アノテーション版 ─── @Mapper public interface SubjectMapper { @Select("SELECT * FROM subjects " + "ORDER BY id") List<Subject> findAll(); } // ─── ② XML版 ─── @Mapper public interface SubjectMapper { List<Subject> findAll(); } <mapper namespace="...SubjectMapper"> <select id="findAll" resultType="...Subject"> SELECT * FROM subjects ORDER BY id </select> </mapper> // 結果は完全に同じ // 違いは「どこに書くか」だけ // → どっちを選ぶかが判断対象
03

アノテーションの長所と短所

ANNOTATION PROS & CONS

アノテーションSQLは「シンプル・1ファイル完結・読みやすい」のが長所。一方で「動的SQLが書けない・SQLが長くなると地獄」が短所。

annotation-analysis アノテーション派の評価
// 長所と短所
✓ PROS
長所

1. シンプル・1ファイルで完結
JavaとSQLが隣にあるので、認知負荷が低い。

2. 短いSQLには圧倒的に向く
5〜10行のSQLならアノテーションが最も読みやすい。

3. 学習コストが低い
初学者にも「メソッドの上のSQL」とすぐ理解できる。

4. IDEの補完が効きやすい
Javaの世界内に閉じているので、IDEのリファクタ機能と相性◎。

✗ CONS
短所

1. 動的SQLが書けない
文字列連結だけでは <if> 系の条件分岐を表現できない。

2. 長いSQLは可読性が地獄
20行を超えると + 結合だらけで読めない。

3. SQLフォーマットが崩れる
キーワード強調が効かない・SQLツールにコピペしにくい。

4. SQL修正のたびに再コンパイル
XMLなら自動リスタート可能だが、Javaは要再ビルド。

// アノテーションが地獄になる例 @Select( "SELECT s.id, s.name, s.color, " + "r.id AS rid, " + "r.studied_on, r.minutes, r.memo " + "FROM subjects s " + "LEFT JOIN study_records r " + "ON s.id = r.subject_id " + "WHERE s.id = #{subjectId} " + " AND r.studied_on BETWEEN #{from} " + " AND #{to} " + "ORDER BY r.studied_on DESC" ) List<Subject> findInPeriod(...); // このSQLは何の処理？ // → "+" だらけで読めない // → SQLツールにコピペするには整形が必要 // → そして「期間が指定されない場合」の条件分岐を // 入れたい時に詰む（動的SQL書けない） // XMLなら： <select id="findInPeriod" ...> SELECT s.id, s.name, s.color, r.id AS rid, r.studied_on, r.minutes FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id <where> s.id = #{subjectId} <if test="from != null"> AND r.studied_on >= #{from} </if> </where> </select> // → 読みやすい・条件も柔軟
04

XMLの長所と短所

XML PROS & CONS

XML Mapperは「複雑SQL・動的SQL・SQL職人による保守」に向く。一方で「ファイル分散・初期設定の手間」が短所。

xml-analysis XML派の評価
// 長所と短所
✓ PROS
長所

1. 動的SQLが書ける
<if> <where> <foreach> で条件分岐・ループが自在。Vol.6で活用。

2. SQL本来の形で書ける
キーワード強調・整形がそのまま。SQLツールへのコピペも容易。

3. resultMap で複雑なマッピング
JOIN結果のオブジェクト組み立て（Vol.4でやる）。

4. SQL専門家・DBAが書ける
Javaを知らなくてもXMLとSQLだけで完結。大規模プロジェクトでは強み。

✗ CONS
短所

1. ファイルが分散する
Java と XML を行き来する手間。「あれ、このSQLどっちにある？」が起きる。

2. namespace / id ミスが致命的
typo 1文字で動かない。エラーも分かりにくい。

3. XMLの構文が冗長
DOCTYPE宣言・タグ閉じ・属性引用符など、ボイラープレート多い。

4. 初期設定が必要
mapper-locations の設定や type-aliases の準備など。

// XMLが本領発揮するパターン // 例1: 複雑な JOIN + resultMap <resultMap id="subjectWithRecords" type="Subject"> <id property="id" column="subject_id"/> <result property="name" column="name"/> <collection property="records" ofType="StudyRecord"> <id property="id" column="rid"/> <result property="studiedOn" column="studied_on"/> </collection> </resultMap> // → アノテーションでは表現困難 // 例2: 動的な絞り込み（検索） <select id="search"> SELECT * FROM study_records <where> <if test="subjectId != null"> AND subject_id = #{subjectId} </if> <if test="from != null"> AND studied_on >= #{from} </if> </where> </select> // → これがXML Mapperの真の強み // → Vol.6 検索で使う
05

使い分けの判断軸

DECISION CRITERIA

業務での判断は3つの軸で決まります。「SQLの複雑度」「動的SQLの有無」「保守の主体」。

decision-criteria 3つの判断軸
// 3つの判断軸
Q1. SQLは何行くらい？
1〜5行 → アノテーション
10行以上 or JOIN → XML
Q2. 条件分岐や動的な部分がある？
固定SQL → どちらでも可
動的SQL必須 → 絶対XML
Q3. このSQLは誰が保守する？
Java開発者 → アノテーションで近接配置
SQL専門家/DBA → XMLで分離

結論：3軸を見て総合判断。「全てXML」「全てアノテーション」と画一化する必要はない。

// 学習記録アプリでの判断 // StudyRecordMapper.findBySubjectId @Select("SELECT * FROM study_records " + "WHERE subject_id = #{subjectId} " + "ORDER BY studied_on") // Q1: 3行 → アノテーションでも読める // Q2: 固定SQL → アノテーションOK // Q3: Java開発者が保守 // → 結論：アノテーションのまま // ───────────────── // SubjectMapper.findAllWithRecords (Vol.4で作る) // SELECT で 8列の JOIN を書く // + resultMap でオブジェクト組み立て // Q1: JOIN + 15行以上 → XML向き // Q2: 今のところ固定だが将来動的化の可能性 // Q3: 結合パターン理解にSQL知識必要 // → 結論：XML一択 // ───────────────── // 検索機能 (Vol.6で作る) // 「科目絞り込み」「期間絞り込み」を組み合わせ // Q1: SQL自体は短いが // Q2: 動的SQL必須 // → 結論：XML一択（<if><where>が必要）
POINT ──

「混在を恐れない」のがプロの態度。1プロジェクト内で「SubjectMapperはXML / StudyRecordMapperはアノテーション」のような混在はむしろ普通です。重要なのは「なぜこっちを選んだか」がチームで共有されていること。

業界トレンドの読み方

INDUSTRY TRENDS

SQL書き方の業界トレンドは時代と共に変わってきました。「歴史を知る」と、なぜ今の状態があるか分かります。

trend-history トレンドの変遷
// 時代ごとのトレンド
〜2010年頃
XML黄金期
MyBatis（旧iBATIS）はXML一択時代。SQLを「DBAが書く設定ファイル」として扱う思想が主流。
2010〜2015年頃
アノテーション登場・コード派の台頭
「XMLは冗長」という反動でアノテーションSQLが好まれる時期。Spring DataやJPAも普及。
2015〜2020年頃
JPA / Hibernate の本格台頭
「SQLを書かない世界」を志向するJPAが人気。一方で複雑クエリは @Query で書く必要があり「結局SQL」と回帰も。
2020年頃〜現在
適材適所・混在許容
「JPAでもMyBatisでも目的に合えばOK」「SQLは隠さない方が良い」というプラグマティックな空気。SpringプロジェクトではMyBatis派が再評価される動き。
// 業界トレンドの現在地（2026年時点） // ─── 日本の業務システム ─── // MyBatis + XML Mapper が依然強い // 理由： // ・SIerに古くからXML文化が根強い // ・SQL職人が独立した存在として残る // ・JPAの「魔法感」を嫌う声 // ─── ベンチャー / スタートアップ ─── // JPA / Spring Data が好まれる傾向 // ・素早く開発したい // ・SQLを書かない世界を志向 // ─── 大規模Webサービス ─── // MyBatis（XML中心）か独自ORM // ・性能チューニング必須 // ・SQLを完全制御したい // Cでは「MyBatis + 混在」を採用 // → 業界の主流に合わせつつ // → 両方の感覚を身につける設計

プロジェクト規約の決め方

PROJECT CONVENTIONS

プロジェクトに参加した時、または立ち上げる時、「SQL書き方規約」をどう決めるか。実務でよくあるパターンを紹介します。

conventions よくある規約パターン
// 規約の例
パターン1
全部XML
SQL専門家がいる大規模システムで多い。統一感は最大だが、簡単なSQLでも冗長になる。
パターン2
全部アノテーション
小規模・Spring Data寄りのチームで稀に見る。SQLが簡単な時に最速だが、複雑化で苦しむ。
パターン3
単純SELECT のみアノテーション、それ以外XML
最も多い実用的パターン。「動的SQLが必要な瞬間」を境界にする。Cで採用する方針もこれ。
パターン4
INSERT/UPDATE/DELETE はアノテーション、SELECT は XML
CRUD操作の種類で分ける流派。「データを書く操作は単純」「読む操作は複雑」という業務感覚を反映。
// 規約決定の3つのポイント // ① チームのSQL熟練度 // → 全員 Java エンジニアならアノテ寄り // → DBA / SQL職人がいるなら XML 中心 // ② 既存資産 // → 既存プロジェクトに XML が大量にある // なら、新規部分も XML 寄せる // → 一貫性は超重要 // ③ 業務の複雑度 // → 単純CRUD中心ならアノテ寄り // → 集計・検索・統計が多いなら XML 中心 // ─── Cでの規約 ─── // 「単純SELECT のみアノテーション // 動的/複雑SQL は XML」 // 具体的に： // ・StudyRecordMapper.findBySubjectId → アノテ // ・SubjectMapper.findAllWithRecords → XML // （Vol.4でJOIN+resultMap） // ・検索メソッド → XML（Vol.6で動的SQL） // ・集計メソッド → XML（Vol.9でGROUP BY）
POINT ──

規約は「正解」より「合意」が大事。チーム内で同じ判断ができる状態が最重要です。「Aさんはアノテで、Bさんはxmlで似たSQLを書く」 が一番悪い。明文化されたガイドラインを最初に作るのが、品質の土台です。

08

使い分けのハマりどころ

PITFALLS

使い分けで起きやすい4つの罠。実務での「あるある」です。

choosing-pitfalls 使い分けの罠
// 罠
PITFALL 1
「最初はアノテで…」で長期化
プロトタイプ的にアノテーションで書き始め、機能追加でSQLがどんどん長くなる。気付いた時にはXMLに移行できないほどコードが増える。
対策：10行を超えたら即XMLに移す判断を。
PITFALL 2
XMLとアノテで同じSQLを二重定義
本編Vol.3でも触れた起動エラー。移行時に削除し忘れ。
対策：「移行＝旧を即削除」のセットで作業。
PITFALL 3
同種SQLが両方の書き方で混在し統一感喪失
「findByIdはアノテで」「findByEmailはXMLで」という同じ性質のSQLが分裂。新人が混乱する。
対策：同種SQLは同じ方式に統一するという規約を加える。
PITFALL 4
動的SQLが必要になって慌てる
最初固定SQLで作ったが、「検索機能足して」と言われ <if> を入れたくなり、アノテで詰む。
対策：将来動的化の可能性を最初に検討。可能性ありなら最初からXML。
// PITFALL 1 の典型シナリオ // ─── 最初 ─── @Select("SELECT * FROM tasks " + "WHERE id = #{id}") Task findById(@Param("id") Integer id); // ─── 1ヶ月後（機能追加）─── @Select("SELECT t.*, u.name AS owner_name " + "FROM tasks t " + "LEFT JOIN users u " + " ON t.owner_id = u.id " + "WHERE t.id = #{id}") Task findById(...); // ─── 3ヶ月後（さらに機能追加）─── @Select("SELECT t.*, u.name AS owner_name, " + "COUNT(c.id) AS comment_count, " + "GROUP_CONCAT(tg.name) AS tags " + "FROM tasks t " + "LEFT JOIN users u " + " ON t.owner_id = u.id " + "LEFT JOIN comments c " + " ON c.task_id = t.id " + "LEFT JOIN task_tags tt " + " ON tt.task_id = t.id " + "LEFT JOIN tags tg " + " ON tg.id = tt.tag_id " + "WHERE t.id = #{id} " + "GROUP BY t.id, u.name") Task findById(...); // → もはや読めない // → でも変更履歴が大量にあり今更XML化しづらい // → 教訓: 「気付いた時には遅い」
