この別冊のテーマ

THEME

@Transactional は「付けるだけ」で動く反面、「なぜか効かない」トラブルが頻発します。原因はほぼ「プロキシの仕組み」と「例外の種類」。本巻でこの2大原因を完全に理解します。

why-deep 深く理解する価値
// 「効かない」の主な原因
原因1
例外の種類（チェック例外）
デフォルトではRuntimeException系のみロールバック。チェック例外は素通り。
原因2
プロキシ（同一クラス内呼び出し）
同じクラスの this.method() 呼び出しでは効かない。AOPプロキシの仕組み。
原因3
private メソッド
@Transactional は public でしか効かない。
原因4
例外の握りつぶし（try-catch）
catch して再スローしないと、Spring が例外を検知できない。

02

ロールバックされる例外の条件

ROLLBACK RULES

本編で「例外でロールバック」と学びましたが、厳密には「どんな例外でも」ではありません。Spring のデフォルトルールを正確に理解します。

rollback-default デフォルトのロールバックルール
// 例外の種類とロールバック
例外の種類デフォルトで
RuntimeException
（非チェック例外）		ロールバックする
Errorロールバックする
チェック例外
（Exception系）		ロールバックしない

意外な落とし穴：IOException や SQLException などのチェック例外はデフォルトでロールバックされない。「例外が出たのにコミットされた」事故の原因No.1。

なぜこの仕様か

歴史的経緯で、「チェック例外は『想定内の事態』、非チェック例外は『想定外の異常』」という思想。想定外（バグ等）ならロールバック、想定内（業務エラー）なら呼び出し側が処理、という考え方。但し実務では「全例外でロールバック」が安全なことが多い。

// ロールバック条件の制御 // ─── デフォルト ─── @Transactional public void method() { // RuntimeException → ロールバック ✓ // IOException → ロールバックしない ✗ } // ─── 全例外でロールバック（推奨）─── @Transactional(rollbackFor = Exception.class) public void method() throws Exception { // あらゆる例外でロールバック ✓ } // ─── 特定の例外を除外 ─── @Transactional( noRollbackFor = SomeException.class) public void method() { // SomeException ではロールバックしない } // ─── 実務の鉄則 ─── // 「想定外の状況では全部ロールバック」 // が最も安全 // → rollbackFor = Exception.class // を付けるプロジェクトが多い // ─── 本編 Vol.13 の実験 ─── // throw new RuntimeException(...) // → RuntimeException なのでデフォルトで // ロールバックされた // もし throw new Exception(...) なら // → デフォルトではロールバックされず // INSERT が残ってしまう（要注意）
03

プロキシの仕組み

PROXY MECHANISM

@Transactional の「効かない」問題の根本原因がこれ。Spring は「プロキシ」という代理オブジェクトを使ってトランザクションを管理しています。仕組みを理解すれば全ての謎が解けます。

proxy-how プロキシが包む仕組み
// プロキシとは

Spring は @Transactional の付いたクラスを「プロキシ（代理人）」で包みます。あなたが呼ぶのは本物ではなくプロキシ。

プロキシが「メソッド呼び出しの前後でトランザクション処理」を挟みます。

  1. 呼び出し → プロキシが受ける
  2. プロキシがトランザクション開始
  3. 本物のメソッドを実行
  4. プロキシがコミット or ロールバック
AOP（アスペクト指向）

この「メソッドの前後に処理を挟む」技術をAOPと呼びます。@Transactional は AOP で実装されている。「本来の処理に横断的な関心事（トランザクション）を織り込む」仕組み。

// プロキシの動作イメージ // ─── あなたが書くコード ─── @Service @Transactional public class StudyRecordService { public void register(...) { // DB操作 } } // ─── Spring が裏で作るもの（概念）─── class StudyRecordService$$Proxy extends StudyRecordService { private StudyRecordService real; @Override public void register(...) { // ① トランザクション開始 txManager.begin(); try { real.register(...); // 本物を呼ぶ // ② コミット txManager.commit(); } catch (RuntimeException e) { // ③ ロールバック txManager.rollback(); throw e; } } } // ─── あなたが @Autowired で受け取るのは ─── @Autowired private StudyRecordService service; // → 実はプロキシが注入されている // → service.register() を呼ぶと // プロキシ経由でトランザクション管理 // ─── 重要な帰結 ─── // トランザクションが効くのは // 「プロキシ経由で呼ばれた時」だけ。 // プロキシを経由しない呼び出しでは // トランザクションが効かない。 // → 次セクションの「同一クラス内」問題
04

同一クラス内呼び出しの罠

SELF-INVOCATION TRAP

本編 TROUBLE 2 で触れた「同一クラス内で効かない」問題。プロキシの仕組みを理解した今なら、その理由が分かります。

self-invocation なぜ自己呼び出しで効かないか
// 自己呼び出しの問題

同じクラス内で this.otherMethod() と呼ぶと、プロキシを経由しません。this は本物のオブジェクトを指すので、プロキシのトランザクション処理が挟まらない。

結果、otherMethod() に @Transactional を付けても効かない。

解決策

① 別の Service クラスに分ける（最も一般的）。② self-injection（自分自身をプロキシ経由で注入）。③ メソッドを呼び出し元に統合。①が最もクリーン。

// 自己呼び出しの罠 // ─── ✗ NG: 同一クラス内呼び出し ─── @Service public class MyService { public void outer() { // this 経由 = プロキシを通らない inner(); // = this.inner() } @Transactional public void inner() { // ← この @Transactional は効かない！ // outer から呼ばれると // プロキシを経由しないため } } // ─── ✓ OK: 別クラスに分ける ─── @Service public class OuterService { @Autowired private InnerService innerService; public void outer() { // 別クラス = プロキシ経由 innerService.inner(); // 効く！ } } @Service public class InnerService { @Transactional public void inner() { // プロキシ経由で呼ばれるので効く ✓ } } // ─── 本編 Vol.13 はなぜ大丈夫だった？─── // Controller → Service.register() // Controller は別クラス // → プロキシ経由で register が呼ばれた // → @Transactional が効いた ✓ // 「外部から呼ばれる public メソッド」に // @Transactional を付けるのが基本
05

トランザクションの伝播

PROPAGATION

伝播（propagation）は「トランザクションが入れ子になった時のルール」。Service が別の Service を呼ぶ時、トランザクションをどう扱うか。普段はデフォルトで十分ですが、知っておくと役立ちます。

propagation 主要な伝播タイプ
// 主要な propagation
REQUIRED
既存があれば参加・なければ新規（デフォルト）
最も一般的。呼び出し元にトランザクションがあればそれに乗る、なければ新しく作る。普段はこれ。
REQUIRES_NEW
常に新しいトランザクション
呼び出し元と独立したトランザクション。「ログだけは必ず残したい」など、親がロールバックしても残したい処理に。
SUPPORTS
あれば参加・なければ非トランザクション
トランザクションがあれば乗る、なくても動く。読み取り系で使うことも。
NESTED
入れ子（セーブポイント）
親の中に部分的なロールバックポイントを作る。高度な制御。

普段は REQUIRED（デフォルト）で十分。REQUIRES_NEW は「独立させたい時」に使う。

// 伝播の使用例 // ─── デフォルト（REQUIRED）─── @Transactional public void parentMethod() { serviceA.methodA(); // 同じトランザクション serviceB.methodB(); // 同じトランザクション // → A も B も親と運命を共にする // → どれか失敗で全部ロールバック } // ─── REQUIRES_NEW（独立）─── @Transactional( propagation = Propagation.REQUIRES_NEW) public void writeLog() { // 親と独立したトランザクション // 親がロールバックしても // このログは残る } // ─── 使い分けの典型例 ─── // 「注文処理」（親・REQUIRED） // ・在庫を減らす // ・注文を記録 // → どれか失敗で全部取り消し // 「監査ログ」（REQUIRES_NEW） // ・「誰が何をしようとしたか」 // → 注文が失敗しても // 「試みた」記録は残したい // ─── Cでの方針 ─── // 学習記録アプリではデフォルト // （REQUIRED）で十分。 // 伝播は「知識として」押さえる。

分離レベル

ISOLATION LEVELS

分離レベル（isolation）は ACID の I（Isolation）の具体的な制御。「複数のトランザクションが同時実行された時、互いにどこまで見えるか」を決めます。S9 で触れた ACID の深掘り。

isolation 4つの分離レベル
// 分離レベルと防げる問題
レベル防げる問題
READ UNCOMMITTED（ほぼ何も防げない）
READ COMMITTEDダーティリード
REPEATABLE READ＋ 反復不能読み取り
SERIALIZABLE＋ ファントムリード（完全分離）

下に行くほど厳格だが遅い。MySQL のデフォルトは REPEATABLE READ、PostgreSQL は READ COMMITTED。

3つの問題

ダーティリード：未コミットの変更を読む。反復不能読み取り：同じSELECTが2回で違う結果。ファントムリード：途中で行が増減する。同時実行が多いシステムで問題になる。

// 分離レベルの指定 // ─── 指定方法 ─── @Transactional( isolation = Isolation.READ_COMMITTED) public void method() { ... } // ─── 3つの問題（同時実行で起きる）─── // 【ダーティリード】 // TxA: UPDATE（まだコミットしてない） // TxB: その未確定の値を読む // TxA: ロールバック // → TxB は「存在しなかった値」を読んだ // 【反復不能読み取り】 // TxA: SELECT → 100 // TxB: UPDATE して COMMIT → 200 // TxA: 同じ SELECT → 200 // → 同じトランザクション内で値が変わった // 【ファントムリード】 // TxA: SELECT COUNT → 5件 // TxB: INSERT して COMMIT // TxA: 同じ SELECT COUNT → 6件 // → 行数が変わった（幻の行） // ─── 実務の方針 ─── // ほとんどの場合 DB のデフォルトで十分 // 厳格にするほど性能が落ちる // 「同時に同じデータを更新する」 // クリティカルな処理だけ // 分離レベルを上げることを検討 // ─── Cでの方針 ─── // 個人利用の学習記録アプリは // 同時実行がほぼないので // デフォルトで全く問題なし // 「知識として」理解しておく

readOnly 属性

READONLY ATTRIBUTE

@Transactional(readOnly = true) は「読み取り専用」の宣言。参照系のメソッドに付けるとパフォーマンス最適化になります。実務でよく使う属性。

readonly readOnly の効果
// readOnly の効果
効果1
最適化のヒント
DBやORMに「このトランザクションは書き込みしない」と伝え、最適化を促す。フラッシュ処理のスキップなど。
効果2
意図の明示
コードを読む人に「これは参照専用」と伝わる。可読性向上。
効果3
誤った書き込みの抑止
設定によっては、readOnly内で書き込むとエラー。意図しない更新を防ぐ安全装置。

使い分け：参照系メソッドは readOnly = true、更新系はデフォルト（false）。クラスにデフォルトを設定し、更新系で上書きするパターンも。

// readOnly の使い方 // ─── 参照系メソッド ─── @Transactional(readOnly = true) public List<Subject> findAll() { return subjectMapper.findAll(); // 読み取りだけ → readOnly } // ─── 更新系メソッド ─── @Transactional // readOnly = false（デフォルト） public void register(...) { studyRecordMapper.insert(...); // 書き込みあり → readOnly にしない } // ─── クラスデフォルト + メソッド上書き ─── @Service @Transactional(readOnly = true) // クラス全体は読み取り public class SubjectService { public List<Subject> findAll() { // クラスの readOnly=true を継承 } @Transactional // メソッドで上書き（false） public void register(...) { // 更新系はここで readOnly=false に } } // ─── 実務の定番パターン ─── // 「参照系は readOnly=true、 // 更新系はデフォルト」 // → 大半が参照系のシステムで // パフォーマンス向上 // ─── Cでの方針 ─── // 必須ではないが、付けると // 「意図が明確」になる良い習慣
