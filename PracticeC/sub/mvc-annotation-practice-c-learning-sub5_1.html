N+1の検出方法

HOW TO DETECT N+1

N+1問題は「動作上はバグではない」ので、能動的に探さないと気付けません。発見方法は5パターンあります。

detection-methods 5つの検出手法
// 検出方法5つ
METHOD 1
SQLログ目視確認（最も基本）
本編Vol.5でやった方法。logging.level.com.example.todo.mapper=DEBUG でMyBatisのSQLログを出す。1リクエストでSQLが何本流れたか数える。
METHOD 2
p6spy（プロキシ・データソース）
JDBCドライバをラッピングして全SQLをログ出力するツール。実行時間も計測できる。詳細は次セクション。
METHOD 3
DBサーバ側のクエリログ
MySQLの general_log や slow_query_log を有効化。DB側から「1リクエストで何本SQLが来たか」を観測する。
METHOD 4
SQLカウントテスト
テストコードで「1リクエスト = SQL n本以下」を assertion する。CIで自動検出できる。大規模プロジェクトで採用される手法。
METHOD 5
APMツール（New Relic・Datadog 等）
本番運用中に遅いリクエストを自動検出。SQL本数・実行時間も可視化。商用環境での発見手段。

業界の検出ツール

DETECTION TOOLS

N+1検出に使われる代表的なツール3つを紹介します。名前を知っているだけでも、プロジェクトに入った時に「あ、これか」と気付ける。

industry-tools 代表的なツール3つ
// 業界ツール
TOOL 1
p6spy
JDBCドライバのプロキシとして動作。全SQL（パラメータ展開済み）と実行時間をログ出力。Spring Boot との連携が容易で日本のSI現場では定番。
TOOL 2
datasource-proxy
p6spyと似たプロキシ系ツール。スロークエリ閾値やSQLカウント取得のAPIが豊富で、テストでの利用に向く。
TOOL 3
Hibernate Statistics
JPA/Hibernate内蔵の統計機能。「N+1が何回起きたか」「キャッシュヒット率」などを取得できる。JPA派の現場で必須。MyBatisでは使わない。
// p6spy の設定例（参考） // pom.xml に依存追加 <dependency> <groupId>com.github.gavlyukovskiy</groupId> <artifactId>p6spy-spring-boot-starter</artifactId> <version>1.9.0</version> </dependency> // application.properties decorator.datasource.p6spy.enable-logging=true decorator.datasource.p6spy.logging=slf4j // ─── 出力例 ─── // （パラメータが ? ではなく実値で展開される） #1715923200000 | took 3ms | statement | SELECT * FROM subjects ORDER BY id | SELECT * FROM subjects ORDER BY id #1715923200003 | took 1ms | statement | SELECT * FROM study_records WHERE subject_id = ? ORDER BY studied_on | SELECT * FROM study_records WHERE subject_id = 1 ORDER BY studied_on // → MyBatis標準ログより詳細 // → DBツールにそのままコピペできる // → 開発効率が一段上がる // 注意：Cではこのツールは導入しない // （現場知識として「あるよ」を伝えるだけ）
POINT ──

p6spy は「業務開発を10倍ラクにするツール」と言われます。SQL のパラメータが ? ではなく実値で展開されるので、DBツールに即コピペ・即再実行できる。プロジェクトに参加した時、もし「p6spy 入ってない」と言われたら導入を提案する価値あり。

業務での災厄事例集

FIELD DISASTERS

N+1問題が引き起こした実際の業務災厄を、典型的な4パターンで紹介。これらは業界で何度も繰り返されている話。

disaster-cases 業務での災厄パターン
// 業務災厄
DISASTER 1
「リリース後に画面が30秒返らない」
ECサイトの商品一覧。開発時はデータ5件で爆速。本番でデータが10万件になり、商品ごとに「在庫」「画像」「タグ」を別SQLで取得 → 30万SQL／リクエスト → タイムアウト連発。
対策：JOIN+resultMap で1リクエスト1〜3SQLに削減
DISASTER 2
「DB接続プールが枯渇してサーバ全停止」
APIサーバが1リクエストで100SQLを発行 → 同時アクセス10ユーザーで1000接続が同時に必要 → コネクションプールの上限（50）を超えてシステム全停止。
対策：N+1対策によりリクエストあたりSQL数を激減
DISASTER 3
「ピーク時にDBサーバCPU 100%」
人気のレポート画面で、各行ごとに集計SQL発行。ピーク時の同時アクセスでDBサーバが悲鳴。アプリ側ではなくDB側がボトルネック化。
対策：集計を1SQL化（GROUP BY活用、Vol.9で扱う）
DISASTER 4
「ローカルでは動くがAWSで激遅」
開発はDBがローカルでSQL 1往復 1ms。本番はDBがAWS RDSで1往復 20ms。100SQL × 20ms = 2秒の遅延。「動くのに遅い」状態でユーザーから苦情。
対策：ネットワーク遅延を踏まえた本番想定の負荷試験
// DISASTER 1 の典型コード // 商品一覧Service（NG実装） public List<ProductDto> findAll() { List<Product> products = productMapper.findAll(); // → 10万件返る for (Product p : products) { Stock stock = stockMapper.findByProductId(p.getId()); List<Image> images = imageMapper.findByProductId(p.getId()); List<Tag> tags = tagMapper.findByProductId(p.getId()); ... } // → 商品ごとに3〜4SQL // → 30〜40万SQL/リクエスト } // ─── 開発環境では...─── // 商品 = 5件 // SQL = 5*4 + 1 = 21回 // 1リクエスト 200ms // → 「速い」と感じる // ─── 本番では...─── // 商品 = 100,000件 // SQL = 400,001回 // 1リクエスト 数十秒〜タイムアウト // → 「使えない」 // ─── 教訓 ─── // 「動く」と「本番で使える」は別物 // テスト環境の少データに油断しない
POINT ──

業務開発の「あるある」として：「急いでリリースしたら、翌日からサポート対応で1ヶ月吹き飛ぶ」というパターンが本当によくあります。N+1問題はその常連犯人。リリース前にSQLログを30秒見るだけでも、何百万円分のサポートコストを防げます。

05

対策3パターン

3 SOLUTION PATTERNS

N+1対策は3つの確立されたパターンがあります。場面に応じて使い分けます。

solution-patterns 3つの対策パターン
// 対策パターン
PATTERN 1
JOIN + resultMap（本編Vol.4の手法）
最も基本。1回のJOINで親子を取得。MyBatis標準。軽中規模なら第一選択。
PATTERN 2
IN句での一括取得
親の id を集めて WHERE id IN (1,2,3,...) で子を一括取得 → Javaコード側で親に振り分け。JOIN結果が爆発しやすい場面で有効。
PATTERN 3
バッチサイズ指定（JPA/Hibernate固有）
@BatchSize(size=N) で「N件まとめてIN句で取りに行く」設定。JPA派の常用テクニック。MyBatisでは使わない。

結論：MyBatisでは「PATTERN 1（JOIN）」を基本、件数や複雑さに応じて「PATTERN 2（IN句）」。

// PATTERN 2: IN句一括取得（イメージ） // ─── Service層 ─── List<Subject> subjects = subjectMapper.findAll(); // SQL ① → 5件取得 List<Integer> subjectIds = subjects .stream() .map(Subject::getId) .toList(); List<StudyRecord> allRecords = studyRecordMapper.findBySubjectIds(subjectIds); // SQL ② → 1回でまとめて取得 // WHERE subject_id IN (1,2,3,4,5) // Java側で振り分け Map<Integer, List<StudyRecord>> bySubject = allRecords.stream() .collect(Collectors .groupingBy(StudyRecord::getSubjectId)); subjects.forEach(s -> s.setRecords( bySubject.getOrDefault(s.getId(), List.of()))); // → SQL は計 2回 // → JOINと違って結果セットが爆発しない // ─── XMLマッパー側 ─── <select id="findBySubjectIds" resultType="StudyRecord"> SELECT * FROM study_records WHERE subject_id IN <foreach item="id" collection="list" open="(" close=")" separator=","> #{id} </foreach> </select> // → <foreach> は Vol.6 動的SQLで詳しく
どっちを選ぶか

JOIN派 vs IN句派は時に宗教戦争になります。判断軸は「JOIN結果の行数が爆発するか」。親100件×子10件=1000行ならJOINでも問題ない。親100件×子1000件=10万行になるとIN句の方が安全。件数を見て選ぶのが現場の鉄則。

JPAでも起きる同じ問題

SAME PROBLEM IN JPA

JPA/Hibernate（Spring Data JPA）でも同じN+1問題が発生します。むしろJPAの方が発生しやすい側面もある。技術が変わっても本質は同じ。

jpa-n-plus-1 JPAでのN+1
// JPAで起こる典型例
JPA罠1
@OneToMany(fetch = LAZY) のループアクセス
親をリスト取得 → ループで parent.getChildren() → 各呼び出しで子のSQL発行 → 結果N+1。JPAの最頻出パターン。
JPA罠2
@OneToMany(fetch = EAGER) でも危険
EAGER にすると毎回JOINになる。「子の数が大きい時」「親を全件取る時」に結果セットが爆発し、メモリ枯渇。
JPA罠3
findAll() の見た目に騙される
repository.findAll() は一見「全部取る」が、関連エンティティはLAZYなので後で別SQLになる。コードが綺麗に見えるだけに気付きにくい。

JPA対策：JOIN FETCH 句を明示する／@EntityGraph を使う／@BatchSize でIN句一括化。

// JPA罠1の典型コード @Entity public class Subject { @Id private Long id; @OneToMany(mappedBy = "subject", fetch = FetchType.LAZY) private List<StudyRecord> records; } // Service層 List<Subject> subjects = repo.findAll(); // ↑ ここで SELECT * FROM subjects (1回) for (Subject s : subjects) { int count = s.getRecords().size(); // ↑ 各反復でSQL発行！ // SELECT * FROM study_records WHERE... // 「LAZY だから後で取りに行く」 } // → 1+N回SQL = N+1問題 // JPA対策: JOIN FETCH @Repository public interface SubjectRepository extends JpaRepository<Subject, Long> { @Query("SELECT DISTINCT s FROM Subject s " + "LEFT JOIN FETCH s.records") List<Subject> findAllWithRecords(); } // → 1回のJOINで親子取得 // → MyBatisの resultMap + JOIN と本質同じ // ─── 技術が変わっても本質は同じ ─── // MyBatis: // <resultMap> + <collection> + JOIN // JPA: // @Query + JOIN FETCH // 生JDBC: // PreparedStatement + JOIN文字列 // すべて「1回でJOINして取る」

業界の検出文化

DETECTION CULTURE

N+1検出は個人の技からチーム文化へと進化してきました。業界全体での「N+1を生まない仕組み」がどう運用されているかを紹介します。

culture-practices 現場の慣習
// 業界での慣習
PRACTICE 1
コードレビューでSQLログ確認
PRレビュー時に「このコードでSQL何本出る？」と必ず質問する文化。レビュアー側がN+1を見つける目を持つ。
PRACTICE 2
SQL カウントの assertion 必須化
主要画面のSQL本数上限をテストで保証。一覧画面は3本以下、詳細画面は5本以下、など。CIで自動検証。
PRACTICE 3
本番相当データでの定期検証
月1回、本番相当のデータ量で各画面をアクセス → SQL本数と応答時間を計測。劣化の早期検出。
PRACTICE 4
スロークエリ通知の自動化
DB側でスロークエリ閾値（例：100ms以上）を設定 → ログ収集ツール → Slack通知。本番で出てから即対応できる体制。

「N+1 を見つける目」は、エンジニアとしての市場価値を直接的に押し上げるスキルです。面接や転職活動で「N+1問題について説明してください」と聞かれた時、本編Vol.5での体験を語れるなら、「実戦感のある人」として評価されます。理屈だけでなく「やった経験」が大事。

N+1対策のハマりどころ

PITFALLS

N+1対策そのものにも落とし穴があります。「直したつもりがまた別の問題を作る」の典型例を整理。

solution-pitfalls 対策の罠
// 対策の罠
PITFALL 1
JOIN で結果セットが爆発
親1000件×子100件 = 10万行のJOIN結果。メモリを大量消費＋ネットワーク帯域を食う。直したのに別の遅さが出る。
対策：IN句一括取得（PATTERN 2）に切り替える。
PITFALL 2
JOIN FETCH で DISTINCT 忘れ（JPA）
JPAで JOIN FETCH すると親が子の数だけ重複。重複した親リストが返ってきて画面表示がおかしくなる。
対策：SELECT DISTINCT s FROM ... と DISTINCT を付ける。
PITFALL 3
不要なEAGER化
「N+1怖いから全部EAGER」とした結果、使わない関連まで毎回取得。性能劣化。
対策：必要な画面でだけ JOIN FETCH を明示。デフォルトは LAZY のまま。
PITFALL 4
JOINだらけで保守性低下
複雑なJOINに次々resultMapを書くと、SQLが読めなくなる。性能と引き換えに保守性を失う。
対策：性能と読みやすさのバランス。本当に必要な時だけJOIN化。
// PITFALL 1: JOIN爆発の事例 // 親 1,000件 × 子平均 100件 // ─── JOIN方式 ─── SELECT ... FROM parents p LEFT JOIN children c ON p.id = c.parent_id; // 結果セット: 100,000行 // 各行: 親の列＋子の列を全部含む // → ネットワーク転送量が大きい // → アプリ側のメモリ消費も大きい // → SQL自体は速いがトータルで遅い // ─── IN句方式 ─── SELECT * FROM parents; // → 1,000行 SELECT * FROM children WHERE parent_id IN (1,2,3,...,1000); // → 100,000行（重複なし） // 合計: 1,000 + 100,000 = 101,000行 // JOIN: 100,000行と同等 // だがJOIN行と違って親列は重複しない // → 転送量は同等以下 // → メモリ効率は良い // ─── 結論 ─── // 親が多くて子も多い場合は IN句方式の方が有利 // PITFALL 2: DISTINCT忘れ（JPA） @Query("SELECT s FROM Subject s " + "LEFT JOIN FETCH s.records") List<Subject> findAllWithRecords(); // 結果: Subject 1(records 3件) が3個 // Subject 2(records 2件) が2個 ... // → 画面に Subject が重複表示 @Query("SELECT DISTINCT s FROM Subject s " + "LEFT JOIN FETCH s.records") // ↑ DISTINCT で重複除去
