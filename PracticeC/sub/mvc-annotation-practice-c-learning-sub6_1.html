この別冊のテーマ

THEME

「URLなんて適当でも動く」と思いがちですが、業務開発ではURL設計は最重要です。URLはユーザーが直接目にする「インターフェース」であり、システムの構造を表すドキュメントでもある。Vol.6 で書いた ?subjectId=1 の裏にある考え方を整理します。

why-url-design URL設計が大事な理由
// URL設計が問われる4つの場面
場面1
ユーザーが目にする
URLは画面上部・ブックマーク・リンクで必ず見えるもの。「謎の文字列」よりも意味のあるURLが信頼感を与える。

02

URLはインターフェース

URLS ARE UI

画面のボタンや入力欄を「インターフェース」と呼ぶように、URLも「ユーザーがシステムと対話する手段」の一つです。URL設計はUI設計。

url-as-ui 良いURLと悪いURL
// 良いURL vs 悪いURL
✓ GOOD
良いURL

/subjects?subjectId=1

意味が読める／短い／パラメータ名と値が一致／RESTful

✗ BAD
悪いURL

/sbj.aspx?q=1&t=js&_r=8932

意味不明／拡張子付き／略語／謎の数値

「URLを見ただけで何ができる画面か分かる」のが理想。

// 実例：良いURL集 // ─── 一覧画面 ─── GET /subjects GET /products GET /users // ─── 詳細画面 ─── GET /subjects/1 GET /products/laptop-2024 GET /users/123 // ─── 絞り込み ─── GET /subjects?subjectId=1 GET /products?category=electronics GET /users?role=admin // ─── 検索 ─── GET /search?q=Java // ─── ページング ─── GET /subjects?page=2&size=20 // 共通の特徴： // ・小文字 // ・複数形（リソースの集合） // ・拡張子なし（.html とか付けない） // ・パラメータ名が意味を持つ
03

クエリストリングの命名

QUERY STRING NAMING

URLの ? 以降をクエリストリングと呼びます。?subjectId=1&size=20 のように key=value を & で繋げる構造。命名のお作法があります。

naming-rules 命名のお作法
// 命名のルール
原則説明
小文字優先subjectId はOK（キャメル）。Subject_Id は避ける。大文字スタートは特にNG。
意味を込めるid より subjectId、q より keyword。短さよりも明確さ。
キャメル or スネークsubjectId（キャメル）／subject_id（スネーク）。プロジェクト内で統一。
値はURLエンコード日本語や記号はエンコード必須。Springが自動で処理してくれることが多い。
真偽値は明示?active=true や ?completed=1。?on や ?yes は曖昧。

本編Vol.6で採用：キャメルケース（subjectId）。Java側のフィールド名と揃える方針。

// 命名の例（業界相場） // ─── 単数（特定リソース）─── ?id=123 ?subjectId=1 ?userId=5 // ─── 複数（複数指定）─── ?ids=1,2,3 ?tags=java,spring // ─── 検索 ─── ?keyword=Java ?q=Java // 短縮形（Google等で定番） // ─── 範囲 ─── ?from=2026-01-01&to=2026-12-31 ?minPrice=1000&maxPrice=5000 // ─── ページング ─── ?page=2&size=20 // ─── ソート ─── ?sort=createdAt,desc ?sort=name,asc // ─── フィルタ（真偽）─── ?active=true ?archived=false
POINT ──

URLパラメータ命名は「プロジェクトの規約に揃える」のが最重要。新規プロジェクトを立ち上げる場合は最初にパラメータ命名規約を決める。既存プロジェクトに入る場合は既存の命名を踏襲する。「統一感」が品質の根幹。

04

RESTfulな検索URL

RESTFUL SEARCH URLS

RESTful（レストフル）とは「リソースをURLで表現し、HTTPメソッドで操作する」設計思想。業界で広く採用されています。検索URLもRESTfulの観点で考えると美しく整います。

restful-basics RESTful の基本
// RESTfulの中心思想
原則1
URLは「リソース」を表す
/subjects は「科目集合」、/subjects/1 は「特定の科目」。名詞で表現（動詞ではない）。
原則2
HTTPメソッドが「動作」を表す
GET = 取得、POST = 新規作成、PUT/PATCH = 更新、DELETE = 削除。同じURLでもメソッドで意味が変わる。
原則3
階層関係をパスで表現
/subjects/1/records = 「科目1の学習記録集合」。親子関係を表現。
原則4
クエリストリングは「条件」
「絞り込み」「並び替え」「ページング」など、同じリソースに対する条件指定に使う。
// RESTful URLの例 // ─── GET（取得）─── GET /subjects // 科目一覧 GET /subjects/1 // 科目1の詳細 GET /subjects?active=true // 絞り込み // ─── POST（新規作成）─── POST /subjects // 新しい科目を作る // ─── PUT / PATCH（更新）─── PUT /subjects/1 // 科目1を全更新 PATCH /subjects/1 // 科目1を部分更新 // ─── DELETE（削除）─── DELETE /subjects/1 // 科目1を削除 // ─── 階層関係 ─── GET /subjects/1/records // 科目1の学習記録一覧 POST /subjects/1/records // 科目1に記録を追加 // ─── 本編Cでの採用 ─── // Vol.6: GET /subjects?subjectId=1 // ↑ 一覧画面に条件で絞り込み // Vol.7以降: POST /records で記録登録 // PUT / PATCH は使わずPOSTで統一 // （Aと同じ方針）
なぜRESTfulは便利か

RESTfulに従うと、URLを見ただけで動作が予測できる。GET /subjects/1 は「科目1を取得するんだな」と即座に分かる。チーム開発で「次に作るべきURLが自然と決まる」のも大きな利点。「業界共通の文法」を共有しているのと同じ。

ページングと並び順の予告

PAGINATION & SORTING

Cでは扱いませんが、業務開発で必ず登場する「ページング」「並び順」のURL設計も触れておきます。Vol.6の @RequestParam がそのまま使える発展形。

pagination-sort ページング・並び順の業界定石
// ページングの2大流派
流派A
ページ番号方式
?page=2&size=20。業務管理画面で定番。「2ページ目、20件ずつ」を表現。
流派B
オフセット方式
?offset=20&limit=10。「先頭から20件飛ばして10件」。API向け。

並び順は ?sort=createdAt,desc が一般的。複数項目なら ?sort=priority,desc&sort=name,asc。

業務でよくあるURL

/subjects?subjectId=1&page=2&size=20&sort=createdAt,desc
「科目1の記録、2ページ目、20件ずつ、作成日時降順」── これ全部 @RequestParam で受け取れます。

// 業務想定のURL設計 // ─── 単純絞り込み（Vol.6で実装済み）─── GET /subjects?subjectId=1 // ─── 期間絞り込み（Vol.7・Vol.8で扱う）─── GET /records?from=2026-05-01&to=2026-05-31 // ─── ページング ─── GET /records?page=2&size=20 // ─── 並び順 ─── GET /records?sort=studiedOn,desc // ─── 複合 ─── GET /records ?subjectId=1 &from=2026-05-01 &to=2026-05-31 &page=1 &size=10 &sort=studiedOn,desc // Controller側 @GetMapping("/records") public String list( @RequestParam(required = false) Integer subjectId, @RequestParam(required = false) @DateTimeFormat(iso = ISO.DATE) LocalDate from, @RequestParam(required = false) @DateTimeFormat(iso = ISO.DATE) LocalDate to, @RequestParam(defaultValue = "1") Integer page, @RequestParam(defaultValue = "20") Integer size, ...) { ... } // @DateTimeFormat はVol.7で扱う // ページングはCでは深追いしない
06

ブックマーカブル / シェアラブル

BOOKMARKABLE & SHAREABLE

「ブックマーカブル」とは「URLをブックマークすれば後で同じ画面に戻れる」性質のこと。「シェアラブル」とは「URLをコピペで送れば相手が同じ画面を見られる」性質。現代のWebアプリで最重要な品質特性です。

bookmarkable ブックマーカブルの効果
// なぜブックマーカブルか
EFFECT 1
ユーザーが「お気に入り」できる
「Java の学習記録だけ表示する画面」をブックマーク → 翌日も即座にその画面を開ける。UXが圧倒的に良い。
EFFECT 2
URLが情報共有手段になる
「この検索結果見て」とSlackやメールでURL送信。受け取った人が同じ画面を再現できる。
EFFECT 3
ブラウザの戻る・進むが効く
検索条件を変えながら戻る／進むで履歴をたどれる。「あの条件何だったっけ」をブラウザが覚えていてくれる。
EFFECT 4
SEOに有利
クローラーがURLパラメータごとに別ページとして扱う → 検索エンジンに各条件の画面がインデックスされる。

本編Vol.6 で method="get" にしたのはこのため。条件がURLに残るのでブックマーカブルになる。

// ブックマーカブルにならない例 // ─── ✗ POST フォームで検索 ─── <form method="post" action="/subjects/search"> <select name="subjectId">...</select> <button>絞り込む</button> </form> // 送信後のURL： // /subjects/search // → 条件がURLに残らない // → ブックマークしても再現できない // → 共有しても相手は条件を知れない // ─── ✓ GET フォームで検索 ─── <form method="get" action="/subjects"> <select name="subjectId">...</select> <button>絞り込む</button> </form> // 送信後のURL： // /subjects?subjectId=1 // → 条件がURLに残る // → ブックマーク・共有・履歴 全部OK // ─── 実例 ─── // Amazonの検索結果URLが長いのは // 「全条件を URL に含めて共有可能にする」 // 設計の現れ
POINT ──

「ブックマーカブル」は「ユーザーがそのアプリを愛するか」に直結する品質特性です。検索結果やフィルタ条件をブックマークできるアプリは、「使いやすい」と感じられる。最近の業務アプリでも、SPA（React/Vue）でもURL同期を必ず実装するのは、この思想の延長線上。

GET vs POST の使い分け

GET vs POST

HTTPメソッドの中で、GETとPOSTは最も使われる2つ。「画面を表示する」のはGET、「データを変更する」のはPOSTが原則ですが、判断基準を整理します。

get-vs-post 使い分けの判断軸
// 使い分けの3軸
Q1. データを「読む」か「変更する」か？
読む（取得） → GET
変更する（作成・更新・削除） → POST
Q2. URLに条件を残したい？
残したい（ブックマーカブル） → GET
残したくない → POST
Q3. パラメータに機密情報を含む？
含む（パスワード等） → POST
含まない → GETでもOK

本編Vol.6で採用：検索なのでGET。条件をURLに残してブックマーカブルにする。

// 使い分けの実例 // ─── GET の場面 ─── GET /subjects // 一覧表示 GET /subjects?subjectId=1 // 絞り込み GET /subjects/1 // 詳細表示 GET /records/edit/5 // 編集画面表示 // ─── POST の場面 ─── POST /records // 記録登録 POST /records/5 // 記録更新 POST /records/5/delete // 削除 // ─── GET と POST の組み合わせ ─── // 編集画面 = GET（フォーム表示） GET /records/edit/5 ↓ ユーザーが値を変更 ↓ 送信ボタン // 更新処理 = POST（変更を反映） POST /records/5 ↓ 成功 // 一覧画面に戻る = GET（画面表示） GET /records // → これが PRG パターン // （A v2 Vol.5別冊で深掘り済み） // GET の制約 // 1. URLの長さに制限あり // ブラウザ: 2,000〜8,000文字程度 // → 大量データの送信には不向き // 2. パラメータが履歴に残る // → パスワード送信NG // 3. キャッシュされる // → 結果が常に同じであるべき // POST はこれらの制約なし // だが「URLに残らない」のが両刃の剣
08

URL設計のハマりどころ

PITFALLS

URL設計でやりがちな4つの罠を紹介。実務に出てから後悔しないよう、最初に意識しておきたい点。

url-pitfalls URL設計の罠
// 罠
PITFALL 1
動詞をURLに含めてしまう
/getSubjects、/createUser のような動詞URLはRESTful違反。
対処：URLは名詞。動詞はHTTPメソッドで表す。GET /subjects、POST /users。
PITFALL 2
単数形と複数形の混在
/subject/1 と /subjects が混ざる。
対処：業界相場は複数形で統一。/subjects/1 として複数形のリソース集合の中の1要素を指す。
PITFALL 3
パスパラメータとクエリストリングの混在
/subjects/1?subjectId=2 のように同じものを2箇所で指定。
対処：「リソース特定はパス／条件指定はクエリ」と役割を明確に分ける。
PITFALL 4
POST で検索フォームを実装
本編Vol.6で説明した「ブックマーカブルでない問題」。Vol.6 で GET にしたのはこれを避けるため。
対処：検索は GET。「フォームは全部 POST」と思い込まない。
// PITFALL 1 の比較 // ─── ✗ NG: 動詞URL ─── GET /getSubjects POST /createSubject POST /updateSubject/1 POST /deleteSubject/1 // ─── ✓ OK: RESTful ─── GET /subjects POST /subjects PUT /subjects/1 DELETE /subjects/1 // PITFALL 3 の比較 // ─── ✗ NG ─── GET /subjects/1?subjectId=2 // → どちらが正？ // → 1 と 2 どちらが優先？ // ─── ✓ OK パターン1 ─── GET /subjects/1 // → 科目1の詳細を表示 // ─── ✓ OK パターン2 ─── GET /subjects?subjectId=1 // → 科目1で絞り込んだ一覧 // 違いを明確に。本編Vol.6はパターン2。 // パターン1は将来「個別詳細画面」を作る時に使う
