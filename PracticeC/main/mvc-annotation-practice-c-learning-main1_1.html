04

subjectsテーブルを設計する

DESIGN THE TABLE

学習記録アプリの「親テーブル」になる subjects を設計します。子テーブル study_records はVol.5で追加するので、Vol.1ではまだ1テーブルのままです。

table-design subjects テーブルの設計
// 列の設計
列名型意味
idINT主キー（自動採番）
nameVARCHAR(50)科目名（必須）
colorVARCHAR(7)カラーコード（任意・#xxxxxx）
created_atDATETIME作成日時

tasksテーブルと同じくらいシンプルです。Vol.5で study_records を追加し、その時に subjects の id を外部キーとして使います。

// DDL（CREATE TABLE文） CREATE TABLE subjects ( id INT NOT NULL AUTO_INCREMENT, name VARCHAR(50) NOT NULL, color VARCHAR(7), created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP, PRIMARY KEY (id) ); // ─── ポイント ─── // ・id は AUTO_INCREMENT で自動採番 // ・name は NOT NULL（科目名なしの科目はない） // ・color は NULL OK（あとから設定でもよい） // ・created_at は DEFAULT で「今」が自動入る
POINT ──

「id 列の命名は subject_id ではなく id」というのが Cの方針です。tasksテーブルも id でした。理由はVol.5で外部キーを置く時、子テーブル側で subject_id と明確に書き分けられるからです。詳しくは別冊S1で。

create-subject-entity ① Subject エンティティを作る
// 手順（Eclipse操作）
  1. Package Explorer で com.example.todo.entity を右クリック
  2. New → Class を選択

なぜEntityはクラス？

Entityは「データを入れる箱」です。フィールド（id, name など）を持ち、Lombokがgetter/setterを自動生成します。具体的なデータを持つ「もの」なのでクラスで作ります。

// Subject.java package com.example.todo.entity; import java.time.LocalDateTime; import lombok.Data; @Data public class Subject { private Integer id; private String name; private String color; private LocalDateTime createdAt; } // ─── ポイント ─── // ・import文は全部書く // ・@Data ひとつで getter/setter/ // toString/equals/hashCode が揃う // ・created_at（DB） ⇄ createdAt（Java） // は application.properties の // map-underscore-to-camel-case で // 自動マッピング（Aで設定済み）
create-subject-mapper ② SubjectMapper（インターフェイス）を作る
// 手順（Eclipse操作）
  1. Package Explorer で com.example.todo.mapper を右クリック
  2. New → Interface を選択（★Classじゃない）
  3. Name欄に SubjectMapper と入力
  4. 他のチェックは全部外して Finish
  5. 開いたファイルに右のコードを書く
なぜMapperはインターフェイス？

Mapperは「SQLを実行する命令」だけを書く場所です。実装はMyBatisが裏で自動生成してくれるので、自分で書くのは「メソッドの形」だけ。実装なしで成立するためインターフェイスにします。

// SubjectMapper.java package com.example.todo.mapper; import java.util.List; import org.apache.ibatis.annotations.Mapper; import org.apache.ibatis.annotations.Select; import com.example.todo.entity.Subject; @Mapper public interface SubjectMapper { @Select("SELECT * FROM subjects ORDER BY id") List<Subject> findAll(); } // ─── ポイント ─── // ・interface宣言（class じゃない） // ・@Mapper で MyBatis に見つけてもらう // ・findAll() に { } はない // → 実装は MyBatis が自動生成
create-subject-service ③ SubjectService を作る
// 手順（Eclipse操作）
  1. Package Explorer で com.example.todo.service を右クリック
  2. New → Class
  3. Name欄に SubjectService
  4. Finish
  5. 右のコードを書く
Vol.1のServiceは「素通し」

業務ロジックがまだないので、ServiceはMapperを呼ぶだけの「素通し（パススルー）」です。形だけ作っておくのがVol.1の役割。Vol.4以降で本物のロジックが入ります。

// SubjectService.java package com.example.todo.service; import java.util.List; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Service; import org.springframework.transaction.annotation.Transactional; import com.example.todo.entity.Subject; import com.example.todo.mapper.SubjectMapper; @Service @Transactional public class SubjectService { @Autowired private SubjectMapper subjectMapper; public List<Subject> findAll() { return subjectMapper.findAll(); } } // ─── ポイント ─── // ・@Service で DI 対象に // ・@Transactional はクラスに付ける // → 全メソッドにトランザクション // ・現時点では Mapper を呼ぶだけ
create-subject-controller ④ SubjectController を作る
// 手順（Eclipse操作）
  1. Package Explorer で com.example.todo.controller を右クリック
  2. New → Class
  3. Name欄に SubjectController
  4. Finish
  5. 右のコードを書く
// SubjectController.java package com.example.todo.controller; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Controller; import org.springframework.ui.Model; import org.springframework.web.bind.annotation.GetMapping; import com.example.todo.service.SubjectService; @Controller public class SubjectController { @Autowired private SubjectService subjectService; @GetMapping("/subjects") public String list(Model model) { model.addAttribute("subjects", subjectService.findAll()); return "subjects/list"; } } // ─── ポイント ─── // ・@Controller（@RestController じゃない） // ・/subjects URL を担当 // ・"subjects/list" を返す // → templates/subjects/list.html

templates の階層

tasksは templates/tasks/list.html に置きました。subjectsも同じくサブフォルダに置きます。Controllerが返す "subjects/list" がこのパスに対応します。

// templates/subjects/list.html <!DOCTYPE html> <html xmlns:th="http://www.thymeleaf.org"> <head> <meta charset="UTF-8"> <title>科目一覧</title> </head> <body> <h1>科目一覧</h1> <table border="1"> <tr> <th>ID</th> <th>科目名</th> <th>カラー</th> </tr> <tr th:each="s : ${subjects}"> <td th:text="${s.id}"></td> <td th:text="${s.name}"></td> <td th:text="${s.color}"></td> </tr> </table> </body> </html> // ─── ポイント ─── // ・th:each="s : ${subjects}" // → モデル属性 subjects を順に s に代入 // ・th:text で値を埋め込み // ・Aで作ったtasks/list.htmlと瓜二つ

ハマりどころと対処

TROUBLESHOOTING

Vol.1で実際に起きやすいトラブル4選です。原因と対処をペアで把握しておくと、詰まる時間が激減します。

troubles 起きやすいトラブル4選
// よくあるトラブル
TROUBLE 1
Whitelabel Error Page が出る
原因：templates/subjects/list.html のパスが間違っている。
対処：Controllerの戻り値 "subjects/list" と、ファイルパス templates/subjects/list.html が一致しているか確認。
TROUBLE 2
Table 'subjects' doesn't exist
原因：DDLを実行し忘れ／別のDBに作ってしまった。
対処：MySQLで USE todo_db; SHOW TABLES; して subjects があるか確認。
TROUBLE 3
Subject に赤線「import 'Subject' (...)」
原因：Mapper / Service / Controller で Subject を使っているがimport不足。
対処：Ctrl + Shift + O で一括解決。
TROUBLE 4
画面は出るが表が空
原因：INSERT文を実行し忘れ。
対処：MySQLで SELECT COUNT(*) FROM subjects; が 0 なら INSERT文を再実行。
// デバッグの3点読み // A v2 Vol.3 別冊で習った // 「ログの3点読み」の復習： // ① エラーメッセージの先頭1行 Error resolving template [subjects/list], template might not exist or might not be accessible // ② Caused by: の連鎖を末尾まで // → 真因はだいたい末尾の Caused by // ③ どのファイルの何行目で起きたか at com.example.todo.controller .SubjectController.list (SubjectController.java:22) // この3点を必ず見る。 // スタックトレース全部読まなくていい。
