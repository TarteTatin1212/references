04

Mapper用DTO と 画面用DTO の使い分け

TWO LAYERS OF DTO

Vol.10 のキーとなる設計判断です。「Mapper が返すDTO」と「画面に渡すDTO」を分けることで、各層の責務が明確になります。

two-dto-design 2つのDTO
// 2つのDTOの違い
DTO 1
SubjectStat（Mapper用・既存）
DBの集計結果そのまま。totalMinutes は分単位の整数。avgMinutes は BigDecimal。「DB の素材」。
DTO 2
SubjectStatView（画面用・新規）
画面に表示するための加工済みデータ。「6時間 15分」のフォーマット文字列、percentage など。「料理された結果」。

分ける利点：

  • Mapper の責任は「DBから取る」だけに集中
  • Service が「画面用に加工する」役割を持つ
  • 画面が変わっても Mapper を触らずに済む
  • 「関心の分離」がシステム全体で実現される
// 2つのDTOの中身 // ─── Mapper用（既存・Vol.9 で作成）─── package com.example.todo.dto; @Data public class SubjectStat { private Integer subjectId; private String subjectName; private Integer recordCount; private Integer totalMinutes; private BigDecimal avgMinutes; } // ─── 画面用（★Vol.10 で新規）─── package com.example.todo.dto; @Data public class SubjectStatView { private Integer subjectId; private String subjectName; private Integer recordCount; // 既存（Mapper用 と共通） private Integer totalMinutes; private BigDecimal avgMinutes; // ★ 画面用に加工された値 private String totalTimeText; // "6時間 15分" private String avgTimeText; // "1時間 15分" private Integer percentage; // 59（割合） } // ─── ポイント ─── // ・既存フィールド + 表示用フィールド // ・元の数値も残すと汎用性が高い // ・percentage は Integer（小数点不要） // → 進捗バーの幅指定で使う
POINT ──

「Mapper戻り値をそのまま画面に渡せばいいのでは？」と思うかもしれません。小規模なら確かにそれで OK。しかし業務システムが成長すると、「同じデータを違う形で表示する複数の画面」が増えてきます。その時に「Mapper用は1個、画面用は画面の数だけ」という設計が活きる。Vol.10でその第一歩を踏み出します。

05

Service層に入れる業務ロジック

SERVICE BUSINESS LOGIC

Vol.10 の Service に入れる3つの業務ロジックを整理します。これが Cで初めて「Service の本領」を体感する場面です。

service-logic Service に入る3つの加工
// 加工処理3つ
LOGIC 1
分 → 「N時間M分」のフォーマット
375 分を "6時間 15分" に。0 分は "0分"、60 分なら "1時間" など、業務的に自然な表記。
LOGIC 2
全体合計の計算
全科目の totalMinutes を合計。stream().mapToInt(...).sum() で1行。
LOGIC 3
各科目の割合（パーセンテージ）計算
「subjectMin / totalMin * 100」を Integer で。全体が 0 のときの割り算エラーを防ぐ。
// 加工メソッド3つの実装イメージ // ─── LOGIC 1: 分→時:分 変換 ─── private String formatMinutes(int min) { if (min <= 0) return "0分"; int h = min / 60; int m = min % 60; if (h == 0) return m + "分"; if (m == 0) return h + "時間"; return h + "時間 " + m + "分"; } // 入力 → 出力 例: // 0 → "0分" // 30 → "30分" // 60 → "1時間" // 90 → "1時間 30分" // 375 → "6時間 15分" // ─── LOGIC 2: 全体合計 ─── int grandTotal = stats.stream() .mapToInt(SubjectStat::getTotalMinutes) .sum(); // ─── LOGIC 3: パーセンテージ ─── private Integer calcPercentage( int subj, int total) { if (total <= 0) return 0; return (int) Math.round( (double) subj / total * 100); } // 注意： // ・total が 0 ならゼロ除算エラー → 0% を返す // ・double 計算してから round で整数化 // ・(int) キャストで Integer に
POINT ──

「全体が 0 の時にどうするか」── 業務開発で絶対避けたいゼロ除算エラー。「全データが 0 件」「初期表示でデータがまだない」など、「合計 0」になる場面は意外と多い。if (total <= 0) return 0; を必ず入れる。「予想外の入力」を想定するのが業務開発の感性です。

// SubjectService.java（追加部分） import java.util.ArrayList; import com.example.todo.dto.SubjectStat; import com.example.todo.dto.SubjectStatView; // 他のimportは省略 @Service @Transactional public class SubjectService { // 既存メソッド省略 ... // ★ Vol.10で追加：画面用に加工して返す public List<SubjectStatView> findSubjectStatViews() { List<SubjectStat> stats = subjectMapper.findSubjectStats(); // 全体合計 int total = stats.stream() .mapToInt(SubjectStat::getTotalMinutes) .sum(); // 各科目を画面用に変換 List<SubjectStatView> views = new ArrayList<>(); for (SubjectStat s : stats) { SubjectStatView v = new SubjectStatView(); v.setSubjectId(s.getSubjectId()); v.setSubjectName(s.getSubjectName()); v.setRecordCount(s.getRecordCount()); v.setTotalMinutes(s.getTotalMinutes()); v.setAvgMinutes(s.getAvgMinutes()); v.setTotalTimeText( formatMinutes(s.getTotalMinutes())); v.setAvgTimeText( formatMinutes( s.getAvgMinutes().intValue())); if (total > 0) { int pct = (int) Math.round( (double) s.getTotalMinutes() / total * 100); v.setPercentage(pct); } else { v.setPercentage(0); } views.add(v); } return views; } // ヘルパ：分 → 「N時間M分」 private String formatMinutes(int min) { if (min <= 0) return "0分"; int h = min / 60; int m = min % 60; if (h == 0) return m + "分"; if (m == 0) return h + "時間"; return h + "時間 " + m + "分"; } }

ハマりどころと対処

TROUBLESHOOTING

Vol.10 でよくある5つのハマりどころ。Service 層の業務ロジック導入で初めて出会う罠が中心。

troubles 起きやすいトラブル
// よくあるトラブル
TROUBLE 1
ArithmeticException: / by zero
原因：全体合計が 0 なのにパーセンテージ計算でゼロ除算。
対処：if (total > 0) で囲む。または三項演算子で安全に。業務開発の鉄則「ゼロ除算チェック」。
TROUBLE 2
パーセンテージの合計が 100% にならない
原因：Math.round で各科目を四捨五入した結果、合計が 99% や 101% になる。
対処：許容するか、最大値の科目だけ調整して 100% に揃える。Vol.10 では許容する方針。
TROUBLE 3
進捗バーの幅が反映されない
原因：th:style の式の書き方ミス、'width: ' の : 後ろのスペース欠落など。
対処：'width: ' + ${row.percentage} + '%' の形を守る。シングルクォート＋プラス連結。
TROUBLE 4
avgMinutes が BigDecimal で intValue() できない
原因：BigDecimal の intValue() は小数点以下切り捨て。「平均 89.5 分」が「平均 1時間 29分」になり違和感。
対処：setScale(0, RoundingMode.HALF_UP).intValue() で四捨五入してから整数化。
TROUBLE 5
th:class で動的にクラス切り替えが効かない
原因：th:class の式の括弧ミス、文字列リテラルのクォート忘れ。
対処：th:class="${row.totalMinutes > 0 ? 'card' : 'card empty'}" の三項演算子形式を守る。
