04

動的SQL とは

WHAT IS DYNAMIC SQL

MyBatis の動的SQLは、XML内で <if> や <where> といったタグを使ってSQLを動的に組み立てる機能。条件によって WHERE 句や AND を出したり消したりできます。

dynamic-basics 動的SQLの基本タグ
// 主要4タグ
TAG 1
<if test="...">
条件式が true ならタグの中身を出力。test 属性にOGNL式を書く。
TAG 2
<where>
中身がある時だけ WHERE を付ける。先頭の余分な AND や OR も自動除去。動的SQL最重要タグ。
TAG 3
<set>
UPDATE 文用。SET 句の末尾カンマを自動除去。Vol.13 編集機能で本格的に使う予定。
TAG 4
<foreach>
コレクションをループ展開。IN (1,2,3) のような複数値処理。本巻では予告のみ。
// 動的SQLの動き方 // ─── 入力パラメータが各種ある状態 ─── subjectId = 1 from = 2026-05-10 to = 2026-05-15 // ─── XML側の動的SQL ─── SELECT ... FROM subjects s LEFT JOIN study_records r ... <where> <if test="subjectId != null"> AND s.id = #{subjectId} </if> <if test="from != null"> AND r.studied_on >= #{from} </if> <if test="to != null"> AND r.studied_on <= #{to} </if> </where> // ─── 実際に生成されるSQL（条件全部あり）─── SELECT ... FROM subjects s LEFT JOIN study_records r ... WHERE s.id = ? AND r.studied_on >= ? AND r.studied_on <= ? // ─── 条件なしの場合 ─── SELECT ... FROM subjects s LEFT JOIN study_records r ... // → WHERE句自体が出ない！ // ─── subjectId だけの場合 ─── SELECT ... FROM subjects s LEFT JOIN study_records r ... WHERE s.id = ? // → <where> が先頭の AND を自動削除 // 1つのSQLが状況により形を変える
POINT ──

<where> タグの賢いところ：「内容が空ならWHERE自体を出力しない」＋「先頭の余分なANDやORを自動除去」。これがないと WHERE AND s.id = ? のような文法エラーが頻発します。<where> は動的SQLの命綱。

検索条件の設計

SEARCH CONDITION DESIGN

3つの検索条件「科目」「開始日」「終了日」をどのように組み合わせるかを整理します。各条件は独立に有無を選べるのがポイント。

condition-design 3条件の組み合わせ
// 8通りの組み合わせ
科目開始日終了日意味
---全件
有--その科目すべて
-有-開始日以降の全科目
--有終了日以前の全科目
-有有期間内の全科目
有有-その科目で開始日以降
有-有その科目で終了日以前
有有有すべて指定

8通りすべてを1つのSQLでカバーするのが動的SQLの威力。<if> 3個＋<where> 1個で実現。

// 統合する SQL の構造 <select id="searchWithRecords" resultMap="subjectWithRecords"> SELECT s.id AS subject_id, s.name AS subject_name, s.color AS color, s.created_at AS subject_created_at, r.id AS record_id, r.subject_id AS record_subject_id, r.studied_on AS studied_on, r.minutes AS minutes, r.memo AS memo, r.created_at AS record_created_at FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id <where> <if test="subjectId != null"> AND s.id = #{subjectId} </if> <if test="from != null"> AND r.studied_on >= #{from} </if> <if test="to != null"> AND r.studied_on <= #{to} </if> </where> ORDER BY s.id, r.studied_on </select> // ─── ポイント ─── // ・各 <if> の先頭に「AND」を必ず付ける // ・<where> が最初の AND を除去してくれる // ・resultMap は既存の subjectWithRecords を再利用 // ・OGNL式 「!= null」 で値の有無を判定 // ・from / to は LocalDate（型変換は TypeHandler）
日付絞り込みの落とし穴

「期間内の記録だけ」を絞り込む時、LEFT JOIN との組み合わせに注意。r.studied_on >= ? を WHERE 句に書くと、記録のない科目（HTML/CSS 等）が消えます（LEFT JOINの結果で r.studied_on が NULL になり、条件が偽になるため）。
Vol.8では意図的にこの仕様とし、「期間絞り込みすると記録のない科目は消える」動作を採用します（業務的にも妥当）。「記録のない科目も常に表示したい」場合は ON 句に条件を入れる別設計が必要。別冊S8で深掘りします。

08

ハマりどころと対処

TROUBLESHOOTING

動的SQLは便利な分、ハマりどころも多い。よくあるトラブル6選を整理します。

troubles 起きやすいトラブル
// よくあるトラブル
TROUBLE 1
Parameter 'subjectId' not found
原因：Mapper メソッドの引数が複数あるのに @Param 不足。
対処：全引数に @Param("...") を付ける。引数1個なら不要だが、Vol.8 のように3つあるなら必須。
TROUBLE 2
テスト条件で「null」が誤判定される
原因：<if test="subjectId != null"> と書くべきところ <if test="subjectId"> 等。OGNL式の文法ミス。
対処：!= null を明示。文字列なら != null and subjectId != '' など。
TROUBLE 3
syntax error near 'AND'
原因：<where> を使わず素のWHERE句で書いた時、最初の <if> が false になり WHERE AND ... になってしまう。
対処：必ず <where> タグで囲む。
TROUBLE 4
期間絞り込みすると記録なし科目が消える
原因：LEFT JOIN + WHERE 句の組み合わせ。期間条件が「子テーブルが NULL」を弾く。
対処：本巻では意図的にこの挙動とする。「記録のない科目も表示したい」場合は ON 句に条件を入れる。別冊S8で深掘り。
TROUBLE 5
日付パラメータが LocalDate に変換できない
原因：Controller の @RequestParam に @DateTimeFormat 付け忘れ（Vol.7 と同じ）。
対処：@DateTimeFormat(iso = DateTimeFormat.ISO.DATE) を全日付パラメータに付ける。
TROUBLE 6
「条件クリア」しても条件が残る
原因：「条件クリア」リンクの URL が間違っていて、パラメータを残したまま。
対処：<a th:href="@{/subjects}"> でパラメータなしのURLに飛ばす。
