04

外部キー制約とは

WHAT IS A FOREIGN KEY

外部キー（FOREIGN KEY）は「あるテーブルの列が、別のテーブルの主キーを参照する」制約。「子テーブルの値は、必ず親テーブルに存在する」を DB が保証します。

fk-concept 外部キーの仕組み
// 親子の参照関係
subjects（親）
id（主キー）, name, color, ...
▲ 参照
study_records（子）
id, subject_id（外部キー）, ...

study_records.subject_id は subjects.id を参照。外部キー制約があると、存在しない subject_id の記録は作れない。

孤児データ（orphan）の防止

外部キーがないと、「親が存在しない子」（孤児データ）が生まれうる。「subject_id = 999（存在しない）の記録」など。外部キー制約はこれをDBレベルで防ぐ。

// 外部キーが守ること // ─── 外部キーなしの世界（危険）─── // study_records に好きな subject_id を入れられる INSERT INTO study_records (subject_id, studied_on, minutes) VALUES (999, '2026-05-20', 60); // → subject_id=999 の科目は存在しない！ // → でも INSERT できてしまう // → 「孤児データ」発生 // 後で集計すると： // 科目名が出ない・JOIN で消える等の不具合 // ─── 外部キーありの世界（安全）─── ALTER TABLE study_records ADD CONSTRAINT fk_subject FOREIGN KEY (subject_id) REFERENCES subjects(id); // 同じ INSERT を試すと： INSERT INTO study_records (subject_id, ...) VALUES (999, ...); // ERROR 1452: // Cannot add or update a child row: // a foreign key constraint fails // → DB が拒否！ // → 孤児データを作れない // ─── 外部キーが保証する2方向 ─── // 【INSERT/UPDATE 時】 // 子に「存在しない親」を参照させない // 【DELETE 時】 // 親を消す時、子をどうするか // → ON DELETE で制御（次セクション）
05

ON DELETE の挙動

ON DELETE BEHAVIOR

外部キーの核心。「親を削除した時、子をどうするか」を ON DELETE で指定します。3つの主要な挙動を業務要件で選びます。

on-delete-types 3つの挙動
// ON DELETE の3つの挙動
CASCADE
親を消すと子も消える
「科目を削除 → その記録も全部削除」。「親がなくなった子は不要」な場合。学習記録アプリに適する。
RESTRICT
子があれば親を消せない
「記録がある科目は削除拒否」。「うっかり消して大量データ消失」を防ぐ。安全重視。多くのDBのデフォルト。
SET NULL
親を消すと子の外部キーを NULL に
「科目削除 → 記録は残るが subject_id が NULL」。記録を残したいが親は切る場合。列が NULL 許容である必要。

Vol.14 では CASCADE を採用。「科目を消したら、その学習記録も不要」が自然なため。

// 3つの挙動を比較 // 前提：Java（id=1）に5件の記録がある // 「Java を削除」を試みる // ─── CASCADE ─── FOREIGN KEY (subject_id) REFERENCES subjects(id) ON DELETE CASCADE; DELETE FROM subjects WHERE id = 1; // → Java が削除される // → Java の記録5件も自動削除 // → 「親子まとめて消える」 // ─── RESTRICT ─── FOREIGN KEY (subject_id) REFERENCES subjects(id) ON DELETE RESTRICT; DELETE FROM subjects WHERE id = 1; // ERROR 1451: // Cannot delete or update a parent row // → 記録があるので削除拒否 // → 「先に記録を消せ」と言われる // ─── SET NULL ─── FOREIGN KEY (subject_id) REFERENCES subjects(id) ON DELETE SET NULL; DELETE FROM subjects WHERE id = 1; // → Java が削除される // → 記録5件は残るが subject_id = NULL // → 「親なし記録」になる // （subject_id が NULL 許容の必要あり） // ─── 業務での選び方 ─── // CASCADE: 親子が運命共同体 // （注文と注文明細、など） // RESTRICT: 誤削除を防ぎたい // （顧客と注文履歴、など） // SET NULL: 子を孤児として残す // （担当者と案件、など）
POINT ──

CASCADE は強力だが危険でもあります。「科目を1つ消したら、知らないうちに数百件の記録が消えた」という事故も。業務では「本当に親子が運命共同体か」を慎重に判断。迷ったら RESTRICT（安全側）にして、「先に子を消す」運用にすることも多い。削除は取り返しがつかないので慎重に。

既存データに注意

外部キー制約を後から追加する時、既存データに孤児（存在しない subject_id）があると失敗します。まず孤児がないか確認。テーブル作成時に最初から制約を付けるのが理想ですが、Vol.14 では学習のため後付けします。

// 外部キー制約の追加SQL // ─── まず孤児データを確認 ─── SELECT * FROM study_records r WHERE NOT EXISTS ( SELECT 1 FROM subjects s WHERE s.id = r.subject_id ); // → 0件なら OK（孤児なし） // ─── 外部キー制約を追加 ─── ALTER TABLE study_records ADD CONSTRAINT fk_subject FOREIGN KEY (subject_id) REFERENCES subjects(id) ON DELETE CASCADE; // ─── 確認 ─── SHOW CREATE TABLE study_records; // → CONSTRAINT fk_subject FOREIGN KEY ... // ON DELETE CASCADE が表示される // ─── もし RESTRICT にしたいなら ─── // ON DELETE RESTRICT に変える // （Vol.14 では CASCADE で進める） // ─── 制約を外したい時（参考）─── // ALTER TABLE study_records // DROP FOREIGN KEY fk_subject;

@Transactional との連携

削除中に何らかのエラーが起きた場合、@Transactional があればロールバックされます。Vol.13 の知識がここでも活きる。「削除＋他の処理」を組み合わせる時、トランザクションで守るのが安全。

// SubjectService.java（追加部分） @Service @Transactional public class SubjectService { @Autowired private SubjectMapper subjectMapper; // 既存メソッド省略 ... // ★ Vol.14で追加 public void delete(Integer id) { subjectMapper.deleteById(id); // → subjects から削除 // → 外部キー CASCADE で // 関連する study_records も削除 } } // ─── ポイント ─── // ・Service は subjects の削除を指示するだけ // ・関連記録の削除は DB が自動で行う // ・「DBに任せる」のが外部キーの利点 // ・Javaで「先に子を消して...」を // 書かなくて済む

削除は POST で（GET厳禁）

削除のような「状態を変える操作」は必ず POST。GET で削除できると、URLを踏むだけ・クローラーのアクセスだけで消えてしまう大事故に。A v2 で学んだ「副作用のある操作は POST」の鉄則。

// SubjectController.java（追加部分） import org.springframework.web.bind.annotation.PathVariable; import org.springframework.web.bind.annotation.PostMapping; // 他のimportは省略 @Controller public class SubjectController { @Autowired private SubjectService subjectService; // 既存メソッド省略 ... // ★ Vol.14で追加 @PostMapping("/subjects/{id}/delete") public String delete( @PathVariable Integer id) { subjectService.delete(id); // PRG パターン：削除後リダイレクト return "redirect:/subjects"; } } // ─── ポイント ─── // ・@PostMapping（GET不可） // ・@PathVariable で URL から id 取得 // ・redirect で PRG パターン // （リロードで二重削除を防ぐ）

ハマりどころと対処

TROUBLESHOOTING

外部キー制約は「制約違反エラー」が頻発します。よくある6つのトラブルを押さえます。

troubles 起きやすいトラブル6選
// よくあるトラブル
TROUBLE 1
外部キー追加時に "constraint fails"
原因：既存データに孤児（存在しない subject_id）がある。
対処：制約追加前に孤児データを削除 or 修正。NOT EXISTS で確認。
TROUBLE 2
RESTRICTで科目削除がエラーになる
原因：記録がある科目を削除しようとした（RESTRICT の正常動作）。
対処：先に記録を削除する、CASCADE に変える、または「削除不可」とユーザーに通知。
TROUBLE 3
CASCADEで意図せず大量データ消失
原因：CASCADE の連鎖を理解せず、親を消したら子が大量に消えた。
対処：CASCADE は慎重に。確認ダイアログ・論理削除（後述）も検討。
TROUBLE 4
削除ボタンが GET になっている
原因：<a href> で削除を実装。GET なので危険。
対処：<form method="post"> で POST にする。
TROUBLE 5
SET NULL なのに NOT NULL 列
原因：ON DELETE SET NULL なのに外部キー列が NOT NULL。NULL にできず矛盾。
対処：SET NULL を使うなら列を NULL 許容にする。
TROUBLE 6
制約名の重複でALTERが失敗
原因：同じ制約名（fk_subject）が既にある。
対処：別名にする、または既存制約を DROP してから追加。
// TROUBLE 1: 孤児データの処理 // ─── エラー ─── ALTER TABLE study_records ADD CONSTRAINT fk_subject ...; // ERROR 1452: // Cannot add or update a child row // → 既存データに孤児がある // ─── 対処：孤児を確認 ─── SELECT * FROM study_records r WHERE NOT EXISTS ( SELECT 1 FROM subjects s WHERE s.id = r.subject_id ); // ─── 孤児を削除 or 修正 ─── // 削除する場合： DELETE FROM study_records WHERE subject_id NOT IN ( SELECT id FROM subjects ); // その後、制約追加が成功する // 補足：論理削除という選択肢 // 「物理削除」（DELETE）の代わりに // 「論理削除」を使う設計もある // deleted_at カラムを追加し、 // DELETE せず UPDATE で「削除日時」を記録 UPDATE subjects SET deleted_at = NOW() WHERE id = 1; // 表示時は WHERE deleted_at IS NULL で除外 // メリット： // ・誤削除しても復元できる // ・履歴が残る // デメリット： // ・全クエリに条件追加が必要 // ・データが増え続ける // → 業務システムでは論理削除が主流 // → Cでは物理削除で学ぶ（基礎として）
