複数テーブル更新と
@Transactional応用

TRANSACTION ── ALL OR NOTHING

Vol.13は「トランザクション応用」のマイルストーン。Cの「完成フェーズ」の幕開けです。
これまで何気なく付けていた @Transactional。Vol.13 では「1つの操作で複数テーブルを更新する」場面で、その真の力を体験します。
キーワードは「全部成功するか、全部失敗するか」。途中でエラーが起きたら全ての変更を取り消す（ロールバック）。これがデータ整合性を守る業務システムの心臓部です。

💡 赤線が出たら

Vol.13では org.springframework.transaction.annotation.Transactional を確認します。
@Transactional には2つの同名アノテーションがあります（Spring のと jakarta/javax のと）。Spring 版を選ぶのが定石。「Import 'Transactional' (org.springframework.transaction.annotation)」を選択。
意図的に例外を投げる場面で RuntimeException を使うので、その import も。

04

トランザクションとは

WHAT IS A TRANSACTION

トランザクションとは「分割できない一連の処理のまとまり」。複数のDB操作を「全部成功 or 全部失敗」のどちらかにする仕組み。中途半端な状態を許さない。

acid トランザクションの ACID 特性
// ACID 特性
A
Atomicity（原子性）
「全部 or 何もなし」。複数操作が分割不可能なひとまとまり。Vol.13 の主役。
C
Consistency（一貫性）
トランザクション前後でデータの整合性が保たれる。制約違反する状態にはならない。
I
Isolation（分離性）
複数のトランザクションが互いに干渉しない。同時実行されても矛盾しない。S13 で深掘り。
D
Durability（永続性）
コミットされた変更は確実に保存される。障害が起きても失われない。

Vol.13 の焦点は A（原子性）。「全部成功 or 全部失敗」が業務で最も重要。

// トランザクションの身近な例 // ─── 銀行振込（古典的な例）─── 振込: A口座 → B口座 へ 1万円 ① A口座から 1万円 引く ② B口座へ 1万円 足す // もし ① だけ成功して ② で障害が起きたら？ // → A口座から消えたお金が B口座に入らない // → 1万円が「消滅」する大事故！ // トランザクションがあれば： // ② が失敗したら ① も取り消す // → お金は消えない // ─── 学習記録アプリの例 ─── 記録登録: ① study_records に記録を INSERT ② subjects の最終学習日を UPDATE // もし ② が失敗したら？ // → 記録はあるのに最終学習日が古いまま // → データ不整合 // トランザクションがあれば： // ② 失敗 → ① も取り消し // → 「記録なし・最終学習日も変化なし」 // → 整合性が保たれる // ─── コミットとロールバック ─── // コミット（commit）: // 変更を「確定」する // → DB に永続化 // ロールバック（rollback）: // 変更を「取り消す」 // → 開始前の状態に戻す
spring-transaction Spring の @Transactional
// Spring が自動でやってくれること

Spring は @Transactional の付いたメソッドに対して、自動的にトランザクションを管理します。

  1. メソッド開始時にトランザクション開始
  2. メソッドが正常終了したらコミット
  3. メソッド内で実行時例外が出たらロールバック
「宣言的トランザクション」

自分で commit() や rollback() を書かず、アノテーションを付けるだけで管理される。これを「宣言的トランザクション」と呼びます。Spring の大きな利点のひとつ。

// @Transactional の動き @Transactional public void register(StudyRecordForm form) { // ↓ ここでトランザクション開始（Spring） studyRecordMapper.insert(record); // ① INSERT subjectMapper.updateLastStudiedOn( form.getSubjectId(), form.getStudiedOn()); // ② UPDATE // ↓ メソッド正常終了 → コミット（Spring） } // ─── 例外が起きた場合 ─── @Transactional public void register(StudyRecordForm form) { studyRecordMapper.insert(record); // ① if (something) { throw new RuntimeException("エラー"); // ↑ ここで例外 } subjectMapper.updateLastStudiedOn(...); // ② // → ② に到達せず例外 // → Spring が自動ロールバック // → ① の INSERT も取り消される } // ─── 重要な注意（S13で詳説）─── // 自動ロールバックは // 「RuntimeException 系」がデフォルト。 // チェック例外は別途設定が必要。

ロールバックを体験する

EXPERIENCE ROLLBACK

Vol.13 の目玉。意図的に例外を起こして、ロールバックの動作を自分の目で確認します。「百聞は一見にしかず」。

rollback-demo わざとエラーを起こす実験
// 実験の流れ
  1. register メソッドで ① INSERT の後に
  2. わざと throw new RuntimeException() を入れる
  3. 記録登録を実行 → 例外発生
  4. DB を確認 → INSERT した記録が無い！
  5. = ロールバックされた証拠
  6. 次に @Transactional を外して同じ実験
  7. 今度は INSERT した記録が残る（ロールバックされない）
  8. = @Transactional の効果を実感
実験は学びの宝庫

「@Transactional あり/なし」で挙動が変わるのを自分の目で見ると、トランザクションの理解が一気に深まります。実験後は必ず例外を入れたコードを元に戻すこと。

// ロールバック実験のコード // ─── 実験用：わざと例外を起こす ─── @Transactional public void register(StudyRecordForm form) { // ① INSERT（成功する） StudyRecord record = new StudyRecord(); record.setSubjectId(form.getSubjectId()); record.setStudiedOn(form.getStudiedOn()); record.setMinutes(form.getMinutes()); record.setMemo(form.getMemo()); studyRecordMapper.insert(record); // ★ 実験：わざと例外 throw new RuntimeException( "ロールバック実験"); // ② ここには到達しない // subjectMapper.updateLastStudiedOn(...) } // ─── 結果の比較 ─── // 【@Transactional あり】 // 例外発生 → ロールバック // → DB を見ると INSERT が無い // → SELECT COUNT(*) は増えていない // 【@Transactional なし（実験）】 // 例外発生してもロールバックされない // → DB を見ると INSERT が残っている // → 中途半端なデータが発生！ // ─── 実験後 ─── // throw を削除して正常なコードに戻す // @Transactional は付けたままにする

2つのMapperを1メソッドで呼ぶ

これが「複数テーブル更新」の核心。studyRecordMapper.insert() と subjectMapper.updateLastStudiedOn() を同じメソッド内で呼ぶ。@Transactional があるので、この2つは1つの取引として扱われます。

// StudyRecordService.java（改修後） package com.example.todo.service; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Service; import org.springframework.transaction.annotation.Transactional; import com.example.todo.entity.StudyRecord; import com.example.todo.form.StudyRecordForm; import com.example.todo.mapper.StudyRecordMapper; import com.example.todo.mapper.SubjectMapper; @Service @Transactional // ← この宣言が今巻の主役 public class StudyRecordService { @Autowired private StudyRecordMapper studyRecordMapper; @Autowired private SubjectMapper subjectMapper; // ★追加 public void register(StudyRecordForm form) { // ① 記録を INSERT StudyRecord record = new StudyRecord(); record.setSubjectId(form.getSubjectId()); record.setStudiedOn(form.getStudiedOn()); record.setMinutes(form.getMinutes()); record.setMemo(form.getMemo()); studyRecordMapper.insert(record); // ② 科目の最終学習日を UPDATE ★追加 subjectMapper.updateLastStudiedOn( form.getSubjectId(), form.getStudiedOn()); // ① と ② は1トランザクション // 両方成功 → コミット // どちらか失敗 → 両方ロールバック } } // ─── ポイント ─── // ・SubjectMapper を新たに注入 // ・register が2テーブルを更新 // ・@Transactional でひとまとまりに // ・例外時は全部取り消される

ハマりどころと対処

TROUBLESHOOTING

トランザクションは「効かない」トラブルが多い。@Transactional が効かない典型パターンを押さえます。

troubles 起きやすいトラブル6選
// よくあるトラブル
TROUBLE 1
チェック例外でロールバックされない
原因：Spring の自動ロールバックはRuntimeException系のみがデフォルト。IOException などのチェック例外ではロールバックされない。
対処：@Transactional(rollbackFor = Exception.class) を指定。S13 で詳説。
TROUBLE 2
同一クラス内のメソッド呼び出しで効かない
原因：同じクラスの別メソッドを this.method() で呼ぶと @Transactional が効かない（プロキシの仕組み）。
対処：別のService に分けるか、self-injection。S13 で詳説。
TROUBLE 3
try-catch で例外を握りつぶす
原因：メソッド内で try-catch し、例外を外に投げないと Spring が例外を検知できずロールバックされない。
対処：catch 内で再スロー、または catch しない。
TROUBLE 4
@Transactional が private メソッドで効かない
原因：@Transactional はpublic メソッドでしか効かない（プロキシの制約）。
対処：public にする。
TROUBLE 5
import が jakarta 版になっている
原因：自動 import で jakarta.transaction.Transactional を選んだ。Spring 版と挙動が微妙に違う。
対処：org.springframework.transaction.annotation.Transactional に統一。
TROUBLE 6
最終学習日が NULL のまま
原因：updateLastStudiedOn の呼び出し忘れ、または WHERE 条件のミス。
対処：register に ② の更新があるか、subjectId が正しいか確認。
// TROUBLE 3: 例外の握りつぶし // ─── ✗ NG: catch で握りつぶす ─── @Transactional public void register(...) { studyRecordMapper.insert(record); try { subjectMapper.updateLastStudiedOn(...); } catch (Exception e) { // 例外をログに出すだけで何もしない e.printStackTrace(); } // → 例外が外に出ない // → Spring がロールバックを判断できない // → INSERT だけ残る（不整合） } // ─── ✓ OK: 再スローする ─── @Transactional public void register(...) { studyRecordMapper.insert(record); try { subjectMapper.updateLastStudiedOn(...); } catch (Exception e) { // ログを出してから再スロー throw new RuntimeException(e); } // → 例外が外に出る // → Spring がロールバック } // ─── ✓ さらにシンプル：catch しない ─── @Transactional public void register(...) { studyRecordMapper.insert(record); subjectMapper.updateLastStudiedOn(...); // 例外はそのまま伝播 // → Spring が自動ロールバック } // 「不要な try-catch を書かない」が安全
