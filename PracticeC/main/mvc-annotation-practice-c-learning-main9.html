集計関数とGROUP BYの基本

AGGREGATE FUNCTIONS

集計関数と GROUP BY 句はセットで覚えます。「何をグループ化し、何を集計するか」を意識すれば必ず書ける。

aggr-functions 5つの基本集計関数
// 主要集計関数
関数意味NULL の扱い
COUNT(*)行数を数えるNULL も数える
COUNT(col)NULLでない値の数NULLは除外
SUM(col)合計NULLは無視
AVG(col)平均NULLは分母にも入らない
MAX(col) / MIN(col)最大／最小NULL以外で判定

業務でよく使う：COUNT(*)（件数）、SUM(amount)（合計金額）、AVG(score)（平均点）など。

// 集計関数の具体例 // 単純な集計（全体の合計） SELECT COUNT(*) AS total_count, SUM(minutes) AS total_minutes, AVG(minutes) AS avg_minutes, MAX(minutes) AS max_minutes, MIN(minutes) AS min_minutes FROM study_records; // 結果（全体）: // total_count | total_minutes | avg | max | min // 9 | 615 | 68.3| 120 | 30 // ─── GROUP BY で「科目ごとに」集計 ─── SELECT subject_id, COUNT(*) AS cnt, SUM(minutes) AS total, AVG(minutes) AS avg FROM study_records GROUP BY subject_id; // 結果（科目別）: // subject_id | cnt | total | avg // 1 | 5 | 375 | 75.0 // 2 | 3 | 255 | 85.0 // 3 | 1 | 30 | 30.0 // → HTML/CSS と JavaScript (subject_id=4,5) // が消える！ // （study_records に記録がないため）
POINT ──

GROUP BY は「同じ値の行をひとまとめにする」働き。集計関数と組み合わせて「グループごとの集計値」を得ます。SELECT に書ける列は「GROUP BYに含まれる列」か「集計関数」のみ ── これは集計クエリの絶対ルールです。

left-join-aggr LEFT JOIN で「記録なし科目」も残す
// LEFT JOIN + COALESCE

記録なし科目（HTML/CSS, JavaScript）も「0件・0分」として並べたい。Vol.4 で学んだ LEFT JOIN と、新しい関数 COALESCE を組み合わせる。

COALESCE(値, 代替値)：NULL の時に代替値を返す関数。SUM が NULL になる時に 0 へ。

// LEFT JOIN + GROUP BY + COALESCE SELECT s.id AS subjectId, s.name AS subjectName, COUNT(r.id) AS recordCount, COALESCE(SUM(r.minutes), 0) AS totalMinutes, COALESCE(AVG(r.minutes), 0) AS avgMinutes FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id GROUP BY s.id, s.name ORDER BY totalMinutes DESC // ─── 結果 ─── // subjectId | subjectName | recordCount | total | avg // 1 | Java | 5 | 375 | 75.0 // 2 | Spring Boot | 3 | 255 | 85.0 // 3 | SQL | 1 | 30 | 30.0 // 4 | HTML/CSS | 0 | 0 | 0.0 // 5 | JavaScript | 0 | 0 | 0.0 // ─── 仕組み ─── // 1. LEFT JOIN で全科目を残す // → HTML/CSS の r.id, r.minutes は NULL // 2. COUNT(r.id) は NULL を数えない // → HTML/CSS は recordCount = 0 // 3. SUM(r.minutes) は NULL のみだと NULL // → COALESCE で 0 に置き換え // 4. AVG も同様 // LEFT JOIN + 集計 + COALESCE = 業務の鉄板

DTO の役割と作り方

DTO INTRODUCTION

集計結果は「テーブル1対1」ではない──Entity では表現できません。新しく DTO（Data Transfer Object）クラスを作ります。

dto-vs-entity DTO と Entity の違い
// 役割の違い
Entity
DBのテーブル1対1のクラス
「Subject ↔ subjects テーブル」「StudyRecord ↔ study_records テーブル」。DBの実体を表現。
DTO
画面や API のための任意の形のクラス
「SubjectStat = 科目名+件数+合計+平均」など、「画面に出したい形」。Entity と直接対応しない。

使い分けの原則：「テーブルそのまま」なら Entity、「集計や複数テーブル組合せ」なら DTO。

パッケージで明確に分離

本巻で初めて com.example.todo.dto パッケージを作成します。Entity は entity、DTOは dto。パッケージ名で「役割が違う」と一目で分かる設計です。

// SubjectStat DTOの例 package com.example.todo.dto; import java.math.BigDecimal; import lombok.Data; @Data public class SubjectStat { private Integer subjectId; private String subjectName; private Integer recordCount; private Integer totalMinutes; private BigDecimal avgMinutes; } // ─── ポイント ─── // ・テーブル「subject_stats」は存在しない // = 画面用に作ったクラス // ・avgMinutes は BigDecimal // SQLの AVG は小数返すので Integer不可 // 実務では BigDecimal が安全 // ・@Data で getter/setter 自動生成 // ─── DTO の数だけ画面の自由度が広がる ─── // ・ランキング画面 → RankingDto // ・日別集計画面 → DailyStatDto // ・概要ダッシュボード → DashboardDto // ... // → Vol.10〜12で複数のDTOが登場予定

BigDecimal を選んだ理由

SQLの AVG は小数を返すので、Java の Integer では受けられません。Double でも動くが、金銭計算や統計値では誤差が出ない BigDecimal が安全。業務開発のお作法。

// SubjectStat.java（新規） package com.example.todo.dto; import java.math.BigDecimal; import lombok.Data; @Data public class SubjectStat { private Integer subjectId; private String subjectName; private Integer recordCount; private Integer totalMinutes; private BigDecimal avgMinutes; } // ─── ポイント ─── // ・com.example.todo.dto パッケージ // ・@Data で getter/setter 自動 // ・avgMinutes は BigDecimal // ・全フィールドはSQL列の AS と対応 // subjectId ↔ subjectId // subjectName ↔ subjectName // recordCount ↔ recordCount // totalMinutes ↔ totalMinutes // avgMinutes ↔ avgMinutes

ハマりどころと対処

TROUBLESHOOTING

集計クエリ初登場の巻なので、SQL 構文のハマりどころが多めです。よくあるトラブル6選。

troubles 起きやすいトラブル6選
// よくあるトラブル
TROUBLE 1
「Column 's.name' must appear in the GROUP BY clause」
原因：SELECTに s.name を書いたが GROUP BY に入れてない。
対処：GROUP BY s.id, s.name のようにSELECTの非集計列を全部書く。
TROUBLE 2
記録なし科目が出てこない
原因：INNER JOIN にしてしまった、または LEFT JOIN したのに WHERE r.id IS NOT NULL 等を書いてしまった。
対処：LEFT JOIN を使い、子側の条件は ON句に置く（Vol.8 別冊S8の話）。
TROUBLE 3
SUM の結果が null になる
原因：記録なし科目で SUM(NULL) = NULL になる。
対処：COALESCE(SUM(r.minutes), 0) でNULLを0に置換。Java側で Integer として安全に受けられる。
TROUBLE 4
avgMinutes が null や変な値
原因：DTOの型を Integer にしているが、AVG は小数を返すので変換失敗。
対処：DTOの avgMinutes は BigDecimal または Double に。
TROUBLE 5
マッピング不一致
原因：SQL の AS と DTO のフィールド名がズレている。
対処：両方を完全に同じにする。AS totalMinutes ↔ private Integer totalMinutes など。
TROUBLE 6
「/stats」を開くと 404
原因：StatsController を作成済みでもアプリ再起動を忘れた／パッケージ違い／@Controller 忘れ。
対処：再起動し、コンソールに「Mapped GET /stats」のログが出るか確認。
