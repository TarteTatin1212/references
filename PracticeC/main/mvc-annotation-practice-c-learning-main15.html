04

完成したアーキテクチャ

FINAL ARCHITECTURE

15巻で築いた層構造（レイヤードアーキテクチャ）を俯瞰します。各層の役割分担こそが、保守しやすいシステムの土台です。

architecture 層構造の全体像
// 各層の役割
Controller
リクエストの受付と画面の選択
URLとメソッドのマッピング・パラメータ受取・Modelへの詰め込み。薄く保つ。
Service
業務ロジックの中心
データの加工・トランザクション管理・複数Mapperの調整。業務の頭脳。
Mapper
DBアクセス
SQL の実行。アノテーション or XML。DBとの境界。
Entity / DTO / Form
データの器
Entity=テーブル / DTO=画面出力 / Form=画面入力。役割で使い分け。
// 完成したパッケージ構造 com.example.todo/ ├ controller/ │ ├ SubjectController // 科目・記録一覧 │ ├ StudyRecordController // 記録登録 │ └ StatsController // 統計画面 ├ service/ │ ├ SubjectService // 科目の業務 │ ├ StudyRecordService // 記録の業務 │ └ StatsService // 統計の業務 ├ mapper/ │ ├ SubjectMapper // 科目のDB │ └ StudyRecordMapper // 記録のDB ├ entity/ │ ├ Subject // subjects テーブル │ └ StudyRecord // study_records テーブル ├ dto/ │ ├ SubjectStat // 集計（Mapper用） │ ├ SubjectStatView // 集計（画面用） │ ├ DailyStat // 日別（Mapper用） │ └ DailyStatView // 日別（画面用） └ form/ └ StudyRecordForm // 記録入力 // ─── データの流れ ─── リクエスト ↓ Controller（受付） ↓ Service（業務ロジック・@Transactional） ↓ Mapper（SQL） ↓ Database（外部キー制約） // 各層が「自分の仕事」に集中 // = 関心の分離 // = 変更に強いシステム
POINT ──

この層構造こそ、15巻で最も大切な学びかもしれません。「Controller は薄く・Service に業務ロジック・Mapper は SQL だけ」── この役割分担を守れば、機能が増えても破綻しない。「どこに何を書くか」が決まっているから、チーム開発でも迷わない。これがプロの設計です。
