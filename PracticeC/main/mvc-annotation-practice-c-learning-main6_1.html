04

@RequestParam の使い方

@REQUESTPARAM USAGE

@RequestParam の主要オプションを整理。「必須/任意」「初期値」「名前指定」の3つを押さえれば実用十分です。

requestparam-options @RequestParam の3つの使い方
// 3つのパターン
PATTERN 1
必須パラメータ（required=true・既定）
パラメータが無いと 400エラー。「絶対必要なパラメータ」に使う。Vol.6では使わない。
PATTERN 2
任意パラメータ（required=false）
パラメータが無ければ null が入る。Vol.6で使う方式。「絞り込みなしの状態も許す」設計。
PATTERN 3
初期値あり（defaultValue="..."）
パラメータが無ければ既定値が入る。「並び順 = updated_at が既定」など。ページングでよく使う（Vol.6では未使用）。

結論：「絞り込みなしも許す」設計なら required = false が定石。

// パターン別のコード // PATTERN 1: 必須 public String show( @RequestParam Integer id) { ... } // /subjects → 400エラー（id 不足） // /subjects?id=1 → OK // PATTERN 2: 任意（Vol.6で使う） public String list( @RequestParam(required = false) Integer subjectId) { ... } // /subjects → OK（subjectId=null） // /subjects?subjectId=1 → OK（1） // PATTERN 3: 初期値 public String list( @RequestParam(defaultValue = "10") Integer limit) { ... } // /subjects → OK（limit=10） // /subjects?limit=20 → OK（20） // ─── 名前指定 ─── // URL側とJava側で名前を変えたい時 public String list( @RequestParam(name = "q", required = false) String keyword) { ... } // /subjects?q=Java → keyword="Java" // ─── 型変換 ─── // Spring が自動で文字列→Integer等 // "1" → Integer 1 // "abc" → 400エラー（変換失敗）
05

設計：メソッドを分ける方針

DESIGN ── SPLIT METHODS

Vol.6では「全件取得」と「1科目取得」を別メソッドに分けます。@RequestParam の値の有無でService側で分岐するシンプル設計。動的SQLは Vol.8 で本格化します。

design-strategy 2メソッド分割設計
// 全体の流れ
  1. Controller が @RequestParam Integer subjectId を受け取る
  2. subjectIdがnullなら findAllWithRecords() を呼ぶ（Vol.4既存）
  3. subjectIdが値ありなら findWithRecordsBySubjectId(subjectId) を呼ぶ（新規）
  4. 結果Listを "subjects" としてmodel追加（型は同じ List<Subject>）
  5. HTMLは現状のまま動く（テンプレート変更最小）
なぜメソッドを分けるのか

シンプルだから。動的SQLを使えば1つにまとめられるが、初学者には「メソッド名と動作が一致」している方が読みやすい。Vol.8で動的SQLを学んだ後、「これを統合してみよう」とリファクタリングする伸びしろを残します。

// 分岐の流れ // Controller @GetMapping("/subjects") public String list( @RequestParam(required = false) Integer subjectId, Model model) { List<Subject> subjects; if (subjectId == null) { subjects = service.findAllWithRecords(); } else { subjects = service .findWithRecordsBySubjectId(subjectId); } model.addAttribute("subjects", subjects); ... } // Service public List<Subject> findAllWithRecords() { return mapper.findAllWithRecords(); } public List<Subject> findWithRecordsBySubjectId(Integer id) { return mapper .findWithRecordsBySubjectId(id); } // XMLマッパー側で SELECT は2つ存在 <select id="findAllWithRecords" ...> SELECT ... FROM subjects s LEFT JOIN study_records r ON ... ORDER BY s.id, r.studied_on </select> <select id="findWithRecordsBySubjectId" ...> SELECT ... FROM subjects s LEFT JOIN study_records r ON ... WHERE s.id = #{subjectId} // ← 違いはここだけ ORDER BY s.id, r.studied_on </select>

modify-xml ① SubjectMapper.xml にSELECTを追加
// 手順

resultMap の再利用

Vol.4で定義した <resultMap id="subjectWithRecords"> はそのまま再利用できるのがMyBatisの利点。1つのresultMapを複数のSELECTで共有できるので、保守性が高い。

// SubjectMapper.xml（追加部分のみ） <!-- 既存の findAllWithRecords の後に追加 --> <select id="findWithRecordsBySubjectId" resultMap="subjectWithRecords"> SELECT s.id AS subject_id, s.name AS subject_name, s.color AS color, s.created_at AS subject_created_at, r.id AS record_id, r.subject_id AS record_subject_id, r.studied_on AS studied_on, r.minutes AS minutes, r.memo AS memo, r.created_at AS record_created_at FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id WHERE s.id = #{subjectId} ORDER BY s.id, r.studied_on </select> // ─── ポイント ─── // ・findAllWithRecords とほぼ同じSQL // ・違いは WHERE s.id = #{subjectId} のみ // ・resultMap は既存を再利用 // ・#{subjectId} は @Param 不要 // （メソッド引数が1個なので自動認識） // 「コピペで似たSQLが2つ並ぶ」のは // Vol.8で動的SQLを学ぶと解消される伏線

セレクトボックスに何のデータを渡すか

HTMLのセレクトボックスに「すべての科目名」を表示するため、全科目リストを別途渡す必要があります。model.addAttribute("allSubjects", ...) で findAll() の結果を渡します。業務ではこれを「マスタ一覧」と呼びます。

// SubjectController.java（改修後・全体） package com.example.todo.controller; import java.util.List; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Controller; import org.springframework.ui.Model; import org.springframework.web.bind.annotation.GetMapping; import org.springframework.web.bind.annotation.RequestParam; import com.example.todo.entity.Subject; import com.example.todo.service.SubjectService; @Controller public class SubjectController { @Autowired private SubjectService subjectService; @GetMapping("/subjects") public String list( @RequestParam(required = false) Integer subjectId, Model model) { List<Subject> subjects; if (subjectId == null) { subjects = subjectService.findAllWithRecords(); } else { subjects = subjectService .findWithRecordsBySubjectId(subjectId); } model.addAttribute("subjects", subjects); // セレクトボックス用の全科目リスト model.addAttribute("allSubjects", subjectService.findAll()); // 選択中の科目IDを画面に保持 model.addAttribute("selectedSubjectId", subjectId); return "subjects/list"; } @GetMapping("/subjects/legacy") public String listLegacy(Model model) { // Vol.5の内容（省略） } }

form の method は GET

検索フォームは method="get" が定石。URLパラメータとして送られるので、結果ページをブックマークしたりURLをコピペで共有できる。「ブックマーカブル」と呼ぶ重要な特性。詳細は別冊S6で。

ハマりどころと対処

TROUBLESHOOTING

Vol.6で起きやすいトラブル6選。@RequestParam 絡みと、Thymeleafの th:selected 絡みが多い。

troubles 起きやすいトラブル
// よくあるトラブル
TROUBLE 1
Required parameter 'subjectId' is not present（400エラー）
原因：@RequestParam に required = false を付け忘れた。
対処：@RequestParam(required = false) Integer subjectId と書く。
TROUBLE 2
Failed to convert value of type 'String' to required type 'Integer'
原因：URLに ?subjectId=abc など数値変換できない値が来た。
対処：ユーザー入力経路（セレクトボックス）では起きないが、URL直書きで発生。実務では「文字列で受けて自分で変換」する設計に変えることも。
TROUBLE 3
セレクトボックスが「選択中」にならない
原因：th:selected の式が間違い／selectedSubjectId を model に追加し忘れ。
対処：th:selected="${s.id == selectedSubjectId}" の形を確認。コントローラ側で必ず model.addAttribute("selectedSubjectId", subjectId) する。
TROUBLE 4
「すべての科目」を選んでも絞り込まれたまま
原因：<option value=""> の value を空文字にし忘れ → URLに ?subjectId=（空）が付くがControllerでは null にならないことがある。
対処：option の value を空文字にし、Springが自動的に空文字を null に変換するのを期待。または「すべて」のとき form をリセット。
TROUBLE 5
th:action で /subjects に飛ばない
原因：action="/subjects" と書いた（th:action ではなく素のaction）。
対処：th:action="@{/subjects}" と書く。@{...} は context-path を考慮するThymeleaf構文。
TROUBLE 6
画面に「セレクトボックス＋ボタン」が出ない
原因：form タグ内に <button> を入れ忘れ／name="subjectId" を付け忘れ。
対処：form 内に submit型のbuttonがあり、selectのname属性が "subjectId" になっているか確認。
// TROUBLE 4 の詳細 // HTML側 <select name="subjectId"> <option value="">すべての科目</option> // ↑ value="" が重要 ... </select> // 「すべての科目」を選んで送信 // → URL: /subjects?subjectId= // → クエリは subjectId="" となる // Springの自動変換 // 空文字 → Integer は通常 null になる // が、設定や Spring Bootバージョンで挙動が違う // ─── 安全対策（推奨） ─── // Controller側で明示的に分岐 if (subjectId == null) { // 全件取得 } // 上記で動かない場合、文字列で受けて手動変換も検討 // TROUBLE 5 の対処 // ─── ✗ NG ─── <form action="/subjects"> // → contextパスが付かない // → 動く時もあるが脆い // ─── ✓ OK ─── <form th:action="@{/subjects}"> // → Thymeleafがcontextパスを補完 // → どんなデプロイ環境でも安全
