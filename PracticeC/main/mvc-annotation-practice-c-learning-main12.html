時系列集計の考え方

TIME SERIES AGGREGATION

時系列集計は「日付でグループ化」するだけ。SQL自体はシンプル。難しいのは「結果の見せ方」と「欠損日」です。

timeseries-sql 日別集計SQL
// 日別集計のSQL

「日付でグループ化して合計」── これだけ。studied_on が DATE 型なので、そのまま GROUP BY できます。

DATETIME の場合は工夫が必要

studied_on が DATE 型ならそのまま GROUP BYできます。もし DATETIME なら、時刻まで含むので「同じ日でも別グループ」になる。その場合は GROUP BY DATE(created_at) のように日付部分だけ取り出す。Cの studied_on は DATE なので問題なし。

// 日別集計SQL と結果 // ─── SQL ─── SELECT studied_on AS studiedOn, SUM(minutes) AS totalMinutes, COUNT(*) AS recordCount FROM study_records GROUP BY studied_on ORDER BY studied_on; // ─── 結果（記録のある日だけ）─── // studiedOn | totalMinutes | recordCount // 2026-05-08 | 60 | 1 // 2026-05-10 | 90 | 1 // 2026-05-11 | 45 | 1 // 2026-05-12 | 120 | 2 // 2026-05-14 | 80 | 1 // 2026-05-15 | 60 | 1 // ─── 注目：05-09 と 05-13 が無い ─── // 記録のない日は SQL結果に出ない // study_records にその日の行がないから // → 「6日分」しか返ってこない // → でも画面では「8日連続」で見せたい // → 欠損日を埋める処理が必要（次セクション） // ─── 期間で絞る場合 ─── SELECT studied_on AS studiedOn, SUM(minutes) AS totalMinutes FROM study_records WHERE studied_on >= #{from} AND studied_on <= #{to} GROUP BY studied_on ORDER BY studied_on; // 「5月の日別」など期間限定集計も可能 // WHERE で集計前に絞る（S11の原則）
05

欠損日をどう扱うか

HANDLING MISSING DAYS

時系列集計の最大の難所。「記録のない日」は SQL結果に出てこないので、Java側で連続した日付を生成して埋める。これが時系列の定石です。

fill-missing 欠損日を埋める2つの方法
// 欠損日を埋める2方法
方法A
Java で日付を生成して埋める（採用）
SQLの結果を Map<LocalDate, Integer> にし、開始日から終了日まで Java でループ。Map にない日は 0分とする。シンプルで分かりやすい。
方法B
SQL で日付マスタを作って LEFT JOIN
「カレンダーテーブル」や再帰CTEで全日付を生成し、集計結果と LEFT JOIN。高度だがSQLが複雑。大規模向け。

Vol.12 では方法Aを採用。「SQLはシンプルに・Javaで補完」の方針。学習記録アプリの規模なら十分。

方法Aの流れ

① SQLで「記録のある日」だけ取得 → ② Map<LocalDate, Integer> に変換 → ③ 開始日〜終了日を plusDays(1) でループ → ④ 各日について Map から値を引く（なければ 0）→ ⑤ 連続した DailyStatView リストを作る。

// 方法A：Javaで欠損日を埋める // ─── ① SQLで記録のある日だけ取得 ─── List<DailyStat> stats = mapper.findDailyStats(); // → 6日分（05-09, 05-13 は無い） // ─── ② Map に変換（日付→分）─── Map<LocalDate, Integer> map = new HashMap<>(); for (DailyStat s : stats) { map.put(s.getStudiedOn(), s.getTotalMinutes()); } // ─── ③〜⑤ 連続した日付でループ ─── LocalDate start = LocalDate.of( 2026, 5, 8); LocalDate end = LocalDate.of( 2026, 5, 15); List<DailyStatView> views = new ArrayList<>(); LocalDate d = start; while (!d.isAfter(end)) { // Map にあればその値、なければ 0 int min = map.getOrDefault(d, 0); DailyStatView v = new DailyStatView(); v.setStudiedOn(d); v.setTotalMinutes(min); views.add(v); d = d.plusDays(1); // 翌日へ } // → 8日分の連続したリストが完成 // → 05-09, 05-13 は 0分で埋まる // ─── getOrDefault が決め手 ─── // map.getOrDefault(key, 0) // = キーがあれば値、なければ 0 // → 欠損日を綺麗に 0 埋め
POINT ──

「記録のない日も見せる」のは、学習記録アプリでは重要なUX。「5月9日はサボった」が見えることで、ユーザーは習慣の途切れを自覚できる。欠損日を消してしまうと「毎日やっている」ように錯覚する。「ないことを見せる」のも情報です。

なぜ Service を分けるか

統計ロジックが増えてきたので、SubjectService から統計を独立させて StatsService に集約します。「責務でクラスを分ける」のリファクタリング。1クラスが大きくなりすぎるのを防ぐ業務の定石。

// StatsService.java（新規・主要部分） package com.example.todo.service; import java.time.LocalDate; import java.util.*; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Service; import org.springframework.transaction.annotation.Transactional; import com.example.todo.dto.DailyStat; import com.example.todo.dto.DailyStatView; import com.example.todo.mapper.StudyRecordMapper; @Service @Transactional public class StatsService { @Autowired private StudyRecordMapper studyRecordMapper; private static final int GOAL = 60; // 1日の目標(分) public List<DailyStatView> findDailyStatViews( LocalDate start, LocalDate end) { // ① SQLで記録のある日だけ取得 List<DailyStat> stats = studyRecordMapper.findDailyStats(); // ② Map に変換 Map<LocalDate, Integer> map = new HashMap<>(); for (DailyStat s : stats) { map.put(s.getStudiedOn(), s.getTotalMinutes()); } // ③ 連続した日付でループ・欠損日埋め List<DailyStatView> views = new ArrayList<>(); LocalDate d = start; while (!d.isAfter(end)) { int min = map.getOrDefault(d, 0); DailyStatView v = new DailyStatView(); v.setStudiedOn(d); v.setTotalMinutes(min); v.setDateText( String.format("%02d-%02d", d.getMonthValue(), d.getDayOfMonth())); v.setTimeText(formatMinutes(min)); // バー幅: 120分を100%とする v.setBarWidth( Math.min(100, min * 100 / 120)); v.setAchieved(min >= GOAL); views.add(v); d = d.plusDays(1); } return views; } // 期間達成率（達成日 / 全日数） public int calcAchievementRate( List<DailyStatView> views) { if (views.isEmpty()) return 0; long achievedDays = views.stream() .filter(DailyStatView::isAchieved) .count(); return (int) (achievedDays * 100 / views.size()); } private String formatMinutes(int min) { if (min <= 0) return "0分"; int h = min / 60, m = min % 60; if (h == 0) return m + "分"; if (m == 0) return h + "時間"; return h + "時間 " + m + "分"; } }

ハマりどころと対処

TROUBLESHOOTING

Vol.12 でよくある5つのハマりどころ。時系列特有の罠が中心です。

troubles 起きやすいトラブル
// よくあるトラブル
TROUBLE 1
記録のない日が表示されない
原因：SQL結果をそのまま画面に出した。欠損日埋め処理を入れ忘れ。
対処：Service で start〜end をループし getOrDefault で 0埋め。これが今巻の核心。
TROUBLE 2
無限ループになる
原因：while ループで d = d.plusDays(1) を書き忘れた。LocalDate は不変なので plusDays の戻り値を代入しないと進まない。
対処：d = d.plusDays(1)（代入を忘れない）。
TROUBLE 3
Map のキーで日付が一致しない
原因：DB から来た LocalDate と生成した LocalDate の比較。LocalDate は equals で正しく比較できるが、LocalDateTime だと時刻差で一致しない。
対処：studied_on は DATE 型なので LocalDate で受ければOK。
TROUBLE 4
バー幅が100%を超えて崩れる
原因：120分を100%とする計算で、120分超のデータがあると幅が100%超に。
対処：Math.min(100, ...) で上限を100%に制限。
TROUBLE 5
達成率が0%や変な値
原因：日数が0でゼロ除算、または整数除算で小数が消える。
対処：if (views.isEmpty()) return 0。achievedDays * 100 / size の順（先に掛ける）。
