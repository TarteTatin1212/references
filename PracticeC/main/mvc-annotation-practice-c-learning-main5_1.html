04

N+1問題の正体を整理

WHAT IS N+1

「N+1問題」とは：親を取得する1回のSQL（=「1」）に対し、N件の親それぞれに対して子取得のSQLを呼ぶ（=「N」）ことで、合計 1+N 回 のSQLが発行される現象。

n-plus-1-definition N+1問題の正式な定義
// なぜ「N+1」と呼ぶか
「1」の部分
親をまとめて取る最初のクエリ
SELECT * FROM subjects など。1回だけ。
「N」の部分
親の件数N件に対して、それぞれ子を取る
親が5件なら子のSELECTが5回。親が100件なら100回。
合計
1 + N 回のSQL発行
親の件数が増えると線形にSQL回数が増える。これが「災厄」と呼ばれる所以。

なぜ問題か：DB通信のコストはSQLの「中身の重さ」より「往復回数」で大きく決まる。1回で取れることを100回に分けて取るのは、明らかな無駄。

// 件数別の SQL 回数（理論値） 科目 5件: 1 + 5 = 6 回 科目 10件: 1 + 10 = 11 回 科目 50件: 1 + 50 = 51 回 科目100件: 1 + 100= 101 回 科目500件: 1 + 500= 501 回 // SQL 1回 ≒ 1〜数ms（ローカル） // SQL 1回 ≒ 10〜数十ms（リモートDB） // ─── 体感での違い ─── // 5科目 → 30ms ユーザー気付かず // 50科目 → 300ms やや遅く感じる // 500科目→ 3000ms 明らかに遅い・苦情来る // ─── 同じ画面を JOIN で取る場合 ─── 科目 5件: 1 回（5〜10ms） 科目 10件: 1 回（5〜10ms） 科目 50件: 1 回（10〜20ms） 科目100件: 1 回（20〜40ms） 科目500件: 1 回（50〜100ms） // 件数による劣化が「ゆるやか」 // = 「スケーラブル」な実装
POINT ──

N+1問題は「動作上はバグではない」のが厄介。テストは通る・画面も表示される・データも正しい。「ただ遅い」だけ。だから開発時には気付きにくく、本番でユーザー数が増えてから「重い」と苦情が来る。SQLログを開発時から見る習慣が唯一の予防策。

SubjectServiceとControllerの改修

MODIFY SERVICE & CONTROLLER

2ファイルを改修します。新規メソッドの追加と新規エンドポイント。

modify-service ① SubjectService に Legacyメソッドを追加
// 手順
  1. SubjectService.java を開く
  2. 既存の findAllWithRecords() の下に findAllWithRecordsLegacy() を追加

Vol.2 で書いたコードをそのまま復活させるだけ。StudyRecordMapper の @Autowired はVol.4から残っているので、追加注入は不要。

// SubjectService.java（改修後・全体） package com.example.todo.service; import java.util.List; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Service; import org.springframework.transaction.annotation.Transactional; import com.example.todo.entity.Subject; import com.example.todo.entity.StudyRecord; import com.example.todo.mapper.SubjectMapper; import com.example.todo.mapper.StudyRecordMapper; @Service @Transactional public class SubjectService { @Autowired private SubjectMapper subjectMapper; @Autowired private StudyRecordMapper studyRecordMapper; public List<Subject> findAll() { return subjectMapper.findAll(); } // Vol.4の JOIN 方式（既存） public List<Subject> findAllWithRecords() { return subjectMapper.findAllWithRecords(); } // ★ Vol.5で追加：Vol.2のループ方式 // （比較・教育用に意図的に残す） public List<Subject> findAllWithRecordsLegacy() { List<Subject> subjects = subjectMapper.findAll(); for (Subject s : subjects) { List<StudyRecord> records = studyRecordMapper .findBySubjectId(s.getId()); s.setRecords(records); } return subjects; } }
modify-controller ② SubjectController に /subjects/legacy を追加
// 手順
  1. SubjectController.java を開く
  2. 既存の list(...) の下に listLegacy(...) を追加
  3. 戻り値の HTML は同じ "subjects/list"
  4. 保存
同じテンプレートを使う理由

モデル属性名 subjects と Subject オブジェクトの構造が両方の方式で同じなので、テンプレート側は何も変えなくて良い。「同じ画面・違う中身」を実体験するための仕掛けです。

// SubjectController.java（改修後・全体） package com.example.todo.controller; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Controller; import org.springframework.ui.Model; import org.springframework.web.bind.annotation.GetMapping; import com.example.todo.service.SubjectService; @Controller public class SubjectController { @Autowired private SubjectService subjectService; // 既存：Vol.4の1回JOIN方式 @GetMapping("/subjects") public String list(Model model) { model.addAttribute("subjects", subjectService .findAllWithRecords()); return "subjects/list"; } // ★ Vol.5で追加：Vol.2のループ方式 @GetMapping("/subjects/legacy") public String listLegacy(Model model) { model.addAttribute("subjects", subjectService .findAllWithRecordsLegacy()); return "subjects/list"; // ↑ テンプレートは同じ } } // ─── ポイント ─── // ・URLが違うだけで同じ画面が出る // ・SubjectService の呼び分けで切替 // ・コンソールログだけが違う

ハマりどころと他のN+1パターン

PITFALLS & OTHER N+1 PATTERNS

Vol.5自体の実装ハマりどころは少ないですが、「他の場面で出会うN+1パターン」を知っておくと、業務開発で未来の自分を救えます。

other-patterns よくあるN+1パターン4選
// N+1の登場場面
PATTERN 1
ループ内でMapperを呼ぶ（今回の典型）
for ループで親を回し、中で子用のMapperを呼ぶ。最も古典的。Vol.5で体験済み。
PATTERN 2
Stream API 内でMapperを呼ぶ
subjects.stream().map(s -> mapper.findRecords(s.getId())).collect(toList())。一見モダンだが中身は同じN+1。
PATTERN 3
JPAの遅延ロード（lazy）誤用
JPA で parent.getChildren() をループ中に呼ぶ。Proxyが裏でSQLを発行し、結果N+1。JPA最大の罠。
PATTERN 4
複数のサービスがそれぞれ取りに行く
マイクロサービス時代の派生形。1リクエストの裏で、各サービスがDBを叩く。「分散N+1」。設計者が気付きにくい。
// PATTERN 2: Stream APIの罠 // ─── ✗ NG ─── List<Subject> subjects = subjectMapper.findAll(); return subjects.stream() .map(s -> { s.setRecords( studyRecordMapper .findBySubjectId(s.getId())); return s; }) .collect(Collectors.toList()); // 「Stream API使ったから速い」 // と勘違いしがちだが、 // 中身は forループと完全に同じ。 // N+1問題は解消されない。 // → 直し方：JOINクエリを使うこと自体 // (= 本編Vol.4 の手法） // PATTERN 3: JPAの遅延ロード罠 @Entity public class Subject { @OneToMany(fetch = FetchType.LAZY) private List<StudyRecord> records; } // 一覧画面 for (Subject s : subjects) { s.getRecords().size(); // ↑ ここで初めて子SQLが走る！ // 親N件あれば N回 SQL } // 直し方： FETCH JOIN を明示 @Query("SELECT s FROM Subject s " + "LEFT JOIN FETCH s.records") List<Subject> findAllWithRecords(); // 結局のところ、MyBatisでもJPAでも // 「親と子をJOINで一発取得」 // が王道の解決策。
POINT ──

N+1問題は「使う技術が変わっても本質は同じ」です。MyBatisでも JPA/Hibernate でも Spring Data JPA でも、「ループ中に親→子のSQLを呼ぶ」パターンは即座にN+1。逆に「JOINで一発取得する」のがどの技術でも王道の解決策。「N+1のセンサー」を一度養えば、技術が変わっても通用します。
