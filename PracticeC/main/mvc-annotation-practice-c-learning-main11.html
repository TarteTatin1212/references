04

HAVING の基本

HAVING BASICS

HAVING は「GROUP BY で集計した結果に対する絞り込み」。WHERE が「集計前」なら、HAVING は「集計後」。SQL の実行順序を理解すると腑に落ちます。

having-mechanism HAVING の動き
// SQL の実行順序（再掲・重要）
  1. FROM / JOIN ── テーブル結合
  2. WHERE ── 個別行のフィルタ
  3. GROUP BY ── グループ化
  4. HAVING ── 集計値のフィルタ
  5. SELECT ── 列の選択
  6. ORDER BY ── 並び替え

WHERE は GROUP BY の前、HAVING は後。だから HAVING では SUM や COUNT の結果を条件にできる。

覚え方

「WHERE → グループ化 → HAVING」。WHERE は「材料を選ぶ」、HAVING は「料理を選ぶ」。材料の段階ではまだ料理（集計値）が存在しないので、料理を条件にできるのは HAVING だけ。

// HAVING の基本構文 // ─── 例1: 合計時間で絞る ─── SELECT subject_id, SUM(minutes) AS total FROM study_records GROUP BY subject_id HAVING SUM(minutes) >= 60; // → 合計60分以上の科目だけ // ─── 例2: 件数で絞る ─── SELECT subject_id, COUNT(*) AS cnt FROM study_records GROUP BY subject_id HAVING COUNT(*) >= 3; // → 3件以上記録がある科目だけ // ─── 例3: WHERE と HAVING 併用 ─── SELECT subject_id, SUM(minutes) AS total FROM study_records WHERE studied_on >= '2026-05-01' // 集計前 GROUP BY subject_id HAVING SUM(minutes) >= 120; // 集計後 // → 「5月の記録で集計し、120分以上の科目」 // ─── ✗ NG: WHERE で集計関数 ─── SELECT subject_id, SUM(minutes) FROM study_records WHERE SUM(minutes) >= 60 // ← エラー！ GROUP BY subject_id; // ERROR: Invalid use of group function // → WHERE は集計前なので SUM がまだない
having-coalesce LEFT JOIN + HAVING の注意点
// LEFT JOIN との組み合わせ

Vol.9 で学んだ通り、LEFT JOIN すると記録なし科目は SUM が NULL になります。HAVING SUM(...) >= 60 だとNULL は条件偽なので、記録なし科目は自動的に消えます。

閾値0以上の時に記録なし科目（0分）を含めたい場合は COALESCE を使う。Vol.11 では閾値1以上が前提なので、記録なし科目は自然に除外されます。

NULL と比較の落とし穴

NULL >= 60 は true でも false でもなく NULL（=偽扱い）。これは S9 で学んだ「NULL の三値論理」。COALESCE(SUM(...), 0) >= 60 と書けば、NULL を 0 に変換してから比較するので安全です。

// LEFT JOIN + HAVING の挙動 // ─── COALESCE なし ─── SELECT s.id, s.name, SUM(r.minutes) AS total FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id GROUP BY s.id, s.name HAVING SUM(r.minutes) >= 60; // 結果： // Java 375 ✓ // Spring 255 ✓ // （SQL 30 は < 60 で除外） // （HTML/CSS は SUM=NULL → 条件偽で除外） // （JavaScript も同様に除外） // ─── COALESCE あり（Vol.11採用）─── SELECT s.id, s.name, COALESCE(SUM(r.minutes), 0) AS total FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id GROUP BY s.id, s.name HAVING COALESCE(SUM(r.minutes), 0) >= #{minMinutes}; // minMinutes = 60 の結果： // Java 375 ✓ // Spring 255 ✓ // SQL 30 ✗ (60未満) // HTML/CSS 0 ✗ (60未満) // JS 0 ✗ (60未満) // minMinutes = 0 の結果（全件表示と同じ）： // 全科目が 0 以上 → 全部表示 // → COALESCE で予測可能な動作に
05

SQLで絞るか・Javaで絞るか

FILTER IN SQL OR JAVA

「集計結果の絞り込み」はSQLの HAVING でも、Java側の filter でもできる。どちらを選ぶか、業務での判断軸を整理します。

sql-vs-java 2つのアプローチの比較
// 2つのアプローチ
アプローチA
SQL の HAVING で絞る（Vol.11採用）
DBが絞り込んでから返す。「不要なデータをそもそも取得しない」ので転送量が少ない。大量データで有利。
アプローチB
Java の Stream.filter で絞る
全件取得後、Java側で .filter(s -> s.getTotal() >= 60)。柔軟だが全件取得する。複雑な条件には向く。
観点SQL(HAVING)Java(filter)
転送量少ない多い
速度（大量）速い遅い
条件の柔軟性SQLの範囲何でも書ける
可読性SQLに慣れが必要Javaで分かりやすい

原則：「DBでできることはDBで」。シンプルな閾値なら HAVING。

// 2つのアプローチのコード // ─── アプローチA: SQL HAVING（採用）─── // XML GROUP BY s.id, s.name <if test="minMinutes != null"> HAVING COALESCE(SUM(r.minutes), 0) >= #{minMinutes} </if> // Service return subjectMapper .findSubjectStats(minMinutes); // → DBが絞り込んだ結果が返る // ─── アプローチB: Java filter ─── // Service List<SubjectStat> all = subjectMapper.findSubjectStats(); if (minMinutes != null) { all = all.stream() .filter(s -> s.getTotalMinutes() >= minMinutes) .toList(); } // → 全件取得後にJavaで絞る // ─── どちらを選ぶか ─── // 学習記録アプリは小規模 // → どちらでも性能差はほぼない // 教育的には HAVING を学ぶ価値が高い // → Vol.11 では HAVING を採用 // 業務の大規模システムでは // → 「DBで絞る」が基本 // → 全件取得は避ける
POINT ──

「DBでできることはDBで」は業務開発の重要な原則。理由は「DBは絞り込み・集計に最適化されている」から。Java側で全件取得してから filter すると、大量の不要データを転送＋メモリに展開することになる。Vol.5 の N+1 と同じく、「無駄な転送を避ける」のがパフォーマンスの鍵です。

HAVING の位置

SQLの構文順序は GROUP BY → HAVING → ORDER BY。<if> で囲んだ HAVING をこの位置に入れます。順序を間違えると構文エラー。

HAVING で絞り込むと、残った科目だけの合計になります。「全体に対する割合」を出すなら、絞り込み前の全体合計を別途取得する設計も考えられます。Vol.11 ではシンプルに「絞り込み後の合計」を基準にします（割合の意味が変わる点はハマりどころで補足）。

// SubjectService.java（改修部分） // ★ Vol.11で改修：引数追加 public List<SubjectStatView> findSubjectStatViews(Integer minMinutes) { List<SubjectStat> stats = subjectMapper.findSubjectStats(minMinutes); // 全体合計（絞り込み後の合計） int total = stats.stream() .mapToInt(SubjectStat::getTotalMinutes) .sum(); // 以下、Vol.10と同じ変換ロジック List<SubjectStatView> views = new ArrayList<>(); for (SubjectStat s : stats) { SubjectStatView v = new SubjectStatView(); v.setSubjectId(s.getSubjectId()); v.setSubjectName(s.getSubjectName()); v.setRecordCount(s.getRecordCount()); v.setTotalMinutes(s.getTotalMinutes()); v.setAvgMinutes(s.getAvgMinutes()); v.setTotalTimeText( formatMinutes(s.getTotalMinutes())); v.setAvgTimeText( formatMinutes( s.getAvgMinutes().intValue())); if (total > 0) { int pct = (int) Math.round( (double) s.getTotalMinutes() / total * 100); v.setPercentage(pct); } else { v.setPercentage(0); } views.add(v); } return views; } // formatMinutes() は Vol.10 のまま

ハマりどころと対処

TROUBLESHOOTING

Vol.11 でよくある5つのハマりどころ。HAVING の位置・WHERE との混同・割合の意味が中心。

troubles 起きやすいトラブル
// よくあるトラブル
TROUBLE 1
Invalid use of group function
原因：WHERE SUM(...) >= 60 と書いた。WHERE は集計前なので集計関数を使えない。
対処：HAVING に書く。「集計関数を条件にするなら HAVING」。
TROUBLE 2
HAVING の位置がおかしくて構文エラー
原因：HAVING を ORDER BY の後ろや GROUP BY の前に書いた。
対処：順序は GROUP BY → HAVING → ORDER BY。この位置を守る。
TROUBLE 3
Parameter 'minMinutes' not found
原因：Mapperメソッドの引数に @Param("minMinutes") がない。XMLの <if test="minMinutes != null"> で名前を解決できない。
対処：@Param("minMinutes") を付ける。
TROUBLE 4
割合の合計が100%にならない（絞り込み後）
原因：HAVING で絞ると、残った科目だけで割合計算するため、「全体に対する割合」ではなくなる。
対処：仕様として許容するか、絞り込み前の全体合計を別途取得。Vol.11 では「絞り込み後の合計が基準」と割り切る。
TROUBLE 5
空文字の閾値で 400 エラー
原因：閾値欄を空のまま送信すると ?minMinutes= となり、Integer 変換に失敗することがある。
対処：required = false に加え、空文字を null とみなす処理。または条件クリアリンクで対応。
