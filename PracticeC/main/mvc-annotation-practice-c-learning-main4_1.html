04

LEFT JOIN の動きを理解する

UNDERSTAND LEFT JOIN

JOIN（結合）は「2つのテーブルを関連で繋いで取得する」SQL構文。Vol.4ではLEFT JOINを使います。「左側のテーブルを基準に、右側のテーブルから一致する行を引っ張ってくる」のが LEFT JOIN。

join-basics LEFT JOIN の基本
// JOIN の3種類（基本だけ）
INNER JOIN
両方にデータがある行だけ
記録のない科目（HTML/CSS, JavaScript）は結果に出ない。今回は使わない。
LEFT JOIN
左テーブル基準・右が無くても左を返す
記録のない科目も結果に残る（右側列は NULL）。今回はこれを使う。
RIGHT JOIN
右テーブル基準
LEFT の鏡像。実務ではあまり使わない（LEFT で書く方が読みやすい）。

結論：本編Vol.4ではLEFT JOIN。HTML/CSSとJavaScriptの「（記録なし）」表示を成立させるため。

// LEFT JOIN の SQL SELECT s.id AS subject_id, s.name AS subject_name, s.color, s.created_at AS subject_created_at, r.id AS record_id, r.subject_id AS record_subject_id, r.studied_on, r.minutes, r.memo, r.created_at AS record_created_at FROM subjects s LEFT JOIN study_records r ON s.id = r.subject_id ORDER BY s.id, r.studied_on // ─── 結果イメージ ─── // sid|sname|color |rid |stud |min // ───────────────── // 1 |Java |#c43030| 1 |2026-05-08| 60 // 1 |Java |#c43030| 2 |2026-05-10| 60 // 1 |Java |#c43030| 3 |2026-05-12| 90 // 1 |Java |#c43030| 4 |2026-05-14|120 // 1 |Java |#c43030| 5 |2026-05-16| 45 // 2 |Spr |#5a8a4a| 6 |2026-05-11| 45 // 2 |Spr |#5a8a4a| 7 |2026-05-13|120 // 2 |Spr |#5a8a4a| 8 |2026-05-15| 90 // 3 |SQL |#2e6090| 9 |2026-05-09| 30 // 4 |HTML |#c4862e|null|null |null // 5 |JS |#d6b020|null|null |null // 親の行が子の数だけ「冗長に重複」 // 子のない親は1行だけで右側がNULL
POINT ──

LEFT JOIN の重要ポイント：「親側の id（subjects.id）が同じ値の行は、Java側では1つのSubjectオブジェクトに纏める」べき。これがresultMapで「親のidをグループ化キー」に使う理由です。

05

resultMap と collection の構造

RESULTMAP & COLLECTION

JOINで得た冗長な行を、MyBatisが「親オブジェクト + 子のList」に組み立て直してくれる仕組みが <resultMap> + <collection> です。resultMapはCの中で最も重要な概念。

resultmap-structure resultMap の構造
// resultMap の構成要素
PART 1
<resultMap id="..." type="...">
idはこのマップの名前（XML内で使う識別子）。typeは組み立てる対象クラスのフルパス。
PART 2
<id property="..." column="..."/>
グループ化のキー。同じ column の値の行は、同じ Subject にまとめられる。これが超重要。
PART 3
<result property="..." column="..."/>
通常の列マッピング。Javaのプロパティ名 ↔ SQLの列名。
PART 4
<collection property="..." ofType="...">
親に紐づく子のリストの組み立て指示。ofTypeはリストの中身の型。
// resultMap の完成形（次セクションで実装） <resultMap id="subjectWithRecords" type="com.example.todo.entity.Subject"> <!-- PART 2: 親の主キー（グループ化キー）--> <id property="id" column="subject_id"/> <!-- PART 3: 親の通常プロパティ --> <result property="name" column="subject_name"/> <result property="color" column="color"/> <result property="createdAt" column="subject_created_at"/> <!-- PART 4: 子のリスト --> <collection property="records" ofType="com.example.todo.entity.StudyRecord"> <id property="id" column="record_id"/> <result property="subjectId" column="record_subject_id"/> <result property="studiedOn" column="studied_on"/> <result property="minutes" column="minutes"/> <result property="memo" column="memo"/> <result property="createdAt" column="record_created_at"/> </collection> </resultMap>
なぜ AS でエイリアスを付けるのか

JOIN すると subjects.id と study_records.id の列名が衝突します。created_at も両テーブルにあって衝突する。これを AS subject_id／AS record_id などで別名にして区別します。
resultMap の column="..." はこのエイリアス側を指定。列名衝突対策がJOIN実装の超ハマりどころです。

grouping-mechanism グループ化の動き（重要）
// MyBatisが裏で何をしているか

JOINで得た冗長な行（Java=5行、Spring Boot=3行、SQL=1行、HTML=1行 NULL、JS=1行 NULL の合計11行）を、MyBatisは以下のルールで「親オブジェクト + 子List」に組み立て直します：

  1. 各行を順番に見る
  2. <id> の column（subject_id）の値でグループ化
  3. 同じ subject_id の行は同じSubjectにまとめる
  4. 各行から <collection> の指示で StudyRecord を作って親の records に追加
  5. 子側の <id>（record_id）が NULL なら子は追加しない（記録なし科目）
// グループ化のイメージ // ── 入力：JOIN後の11行 ── sid=1, sname=Java, rid=1, stud=05-08 sid=1, sname=Java, rid=2, stud=05-10 sid=1, sname=Java, rid=3, stud=05-12 sid=1, sname=Java, rid=4, stud=05-14 sid=1, sname=Java, rid=5, stud=05-16 sid=2, sname=Spr, rid=6, stud=05-11 sid=2, sname=Spr, rid=7, stud=05-13 sid=2, sname=Spr, rid=8, stud=05-15 sid=3, sname=SQL, rid=9, stud=05-09 sid=4, sname=HTML, rid=null sid=5, sname=JS, rid=null // ── 出力：5つのSubjectオブジェクト ── [ Subject{id=1, name="Java", records=[5件]}, Subject{id=2, name="Spring Boot", records=[3件]}, Subject{id=3, name="SQL", records=[1件]}, Subject{id=4, name="HTML/CSS", records=[]}, Subject{id=5, name="JavaScript", records=[]} ] // → これがVol.2で「2回SQL」で作っていた // オブジェクトと同じ形！ // → MyBatisがJOIN結果から組み立てた

ハマりどころと対処

TROUBLESHOOTING

Vol.4はresultMap初登場なのでハマりどころが多めです。よくあるトラブル6選を整理しました。

troubles 起きやすいトラブル6選
// よくあるトラブル
TROUBLE 1
列名衝突エラー（Column 'id' in field list is ambiguous）
原因：JOIN先のテーブルにも id／created_at がある。
対処：必ず AS でエイリアス。s.id AS subject_id／r.id AS record_id。SELECT句に列名を全部書き出し、AS を付けるのが鉄則。
TROUBLE 2
子のリストが全部同じ件数になる
原因：<id property="id" column="subject_id"/> を書き忘れ／間違い。MyBatisがグループ化できず、全行を別オブジェクト扱いする。
対処：resultMap の中の <id> 要素を必ず書く。column は親の主キー（subject_id）に対応するエイリアス。
TROUBLE 3
記録なし科目に「記録1件（中身NULL）」が出る
原因：<collection> 内の <id property="id" column="record_id"/> 不足／記載漏れ。MyBatisが「子のidがNULLなら子は無し」と判定できない。
対処：<collection> 内にも必ず <id>。
TROUBLE 4
画面のメモが文字化けする
原因：MySQLの文字コード設定不一致。
対処：application.properties の spring.datasource.url に ?useUnicode=true&characterEncoding=utf8 を付ける。A v2 で設定済みのはず。
TROUBLE 5
resultMap 'subjectWithRecords' not found
原因：<select resultMap="..."> のID指定 typo／同じXML内に定義していない。
対処：id="subjectWithRecords" と resultMap="subjectWithRecords" が完全一致するか確認。
TROUBLE 6
アプリ起動でクラッシュ
原因：XML構文エラー／DOCTYPE typo。
対処：起動時のコンソールエラーの末尾「Caused by:」を探す。XMLの行番号が出るのでその行を確認。Eclipse の Problems ビューも有効。
// TROUBLE 1 の典型出力 ERROR 1052 (23000): Column 'id' in field list is ambiguous // → SELECTで id と書いたが // s.id か r.id か曖昧 // ── 直し方 ── SELECT // ✗ ダメ id, // ✓ OK s.id AS subject_id, r.id AS record_id, ... // TROUBLE 2 のチェックポイント // resultMap で <id> を書き忘れた場合： // MyBatisは「行ごとに別のSubject」と判定 // → 5+3+1+1+1 = 11個のSubjectが返る // → 各 records は 1件ずつ // 正しく書いた場合： // 同じ subject_id の行は同じSubject // → 5個のSubjectが返る // → Java=5件, Spr=3件, ... // 「件数がおかしい」と感じたら // まず <id> をチェック
TIP ──

resultMap デバッグの定番テクニック：「まず resultType="java.util.HashMap" でJOIN結果をそのまま見る」── これでJOIN自体が動くか確認できます。動けばresultMapに移行、動かなければSQLを直す。切り分けを段階的にするのが詰まらないコツ。
