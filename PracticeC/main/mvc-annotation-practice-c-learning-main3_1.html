XML Mapper初登場
SELECTを移そう

INTRODUCING XML MAPPER ── MIGRATE THE SELECT

Vol.3は「XML Mapper 導入」の獲得です。
A v2 と本編Vol.1・Vol.2 では、SQLはすべてアノテーション（@Select("...")）で書いてきました。Vol.3では、そのSQLを「XMLファイル」に移し替える体験をします。
「動くものを保ちながら書き場所を変える」──これがいわゆるリファクタリング。同じ画面が動き続ければ成功です。Vol.4で resultMap + collection を書くための「箱」の準備がこの巻の役割。

💡 赤線が出たら

XMLファイルでの赤線は Ctrl + Shift + O では消えません。XMLは構文エラーが地雷の主犯。
typo・閉じタグ抜け・属性の引用符抜けがよくある原因。エディタの赤線・コンソールのエラーログ両方を見ましょう。Eclipseの Problems ビュー（Window → Show View → Problems）が便利。

01

この巻でやること（ミッション）

MISSION

Vol.3の主役は「XML Mapper」。SQLをJavaのアノテーション内（文字列）から、専用のXMLファイルに移します。動作は変わらず、画面も同じ──しかしこの「箱」を用意することで Vol.4 以降のJOINや動的SQLが書けるようになります。

vol.3-mission この巻の3つのゴール
// 3つのゴール
GOAL 1
XML Mapper が起動できる環境を作る
application.properties に mybatis.mapper-locations を追加して、Spring Boot に XML の在りかを教える。
GOAL 2
SubjectMapper.xml にSELECTを移す
これまで @Select("SELECT * FROM subjects ORDER BY id") だったSQLを、XMLの <select id="findAll"> に書き直す。
GOAL 3
同じ画面が動くことを確認する
/subjects を開いて、Vol.2 と全く同じ画面が表示されればOK。「動くまま中身を変える」リファクタリングの成功。
// この巻の Before / After // ─── Before ─── // SubjectMapper.java @Mapper public interface SubjectMapper { @Select("SELECT * FROM subjects " + "ORDER BY id") List<Subject> findAll(); } // ─── After ─── // SubjectMapper.java @Mapper public interface SubjectMapper { List<Subject> findAll(); // ↑ アノテーション削除！ } // SubjectMapper.xml（新規） <mapper namespace="...SubjectMapper"> <select id="findAll" resultType="Subject"> SELECT * FROM subjects ORDER BY id </select> </mapper>
POINT ──

「動くまま中身を変える」のがリファクタリングの神髄です。Vol.3 で大事なのは「機能を増やさない・新しいことをしない」という制約を守ること。新機能を増やすと、動かなくなった時の原因切り分けが難しくなります。1度に1つの変更──これが現場の鉄則。

03

必要な前提知識

PREREQUISITES

Vol.3は「MyBatisの2つの書き方を切り替える」が中心。XML自体は中学校レベルの構文知識で読めますが、MyBatis特有のルールが3つあります。

prerequisites 前提となる知識と新しい3ルール
// 前提知識
知識該当箇所
Mapper interface / @Select の動きA v2 Vol.3
application.properties の役割A v2 Vol.2
1対多取得（基礎）本編 M2
3つの取得戦略別冊 S2
新しく覚える3つのルール：
  1. namespace：XMLとJavaのMapperインターフェイスをフルパスで紐付ける
  2. id：<select id="..."> はインターフェイスのメソッド名と完全一致
  3. resultType：1行が何のクラスに対応するかを指定
// 3つのルールの紐付け関係 // ── Java側 ── package com.example.todo.mapper; @Mapper public interface SubjectMapper { // メソッド名 List<Subject> findAll(); // 戻り値の中身の型 } // ── XML側 ── <mapper namespace="com.example.todo.mapper.SubjectMapper"> // ↑ Javaのフルパス <select id="findAll" // ↑ Javaのメソッド名と完全一致 resultType="com.example.todo.entity.Subject"> // ↑ List<Subject> の中身の型のフルパス SELECT * FROM subjects ORDER BY id </select> </mapper> // → 3点が揃って初めて // MyBatis が「これは findAll() の実装だ」 // と認識できる
POINT ──

XML Mapper の「namespace = JavaのMapperインターフェイスのフルパス」は最も間違えやすいルールです。typoでもダメ、パッケージ違いでもダメ。後でハマりどころでも触れますが、ここを最初に意識しておくと事故を防げます。

04

application.properties を設定する

CONFIGURE PROPERTIES

XMLファイルをSpring Bootに認識させるために、application.properties に1行追加します。これだけ。

configure-properties 設定の追加
// 手順
  1. src/main/resources/application.properties を開く
  2. 末尾に1行追加：mybatis.mapper-locations=classpath:mybatis/*.xml
  3. 保存（Ctrl + S）
この1行の意味

classpath:mybatis/*.xml = 「クラスパス上の mybatis フォルダの中にある .xml ファイル全て」を MyBatis に読み込ませる指定。
Spring Boot は src/main/resources 配下を「クラスパス」として扱うので、resources/mybatis/SubjectMapper.xml が対象になります。

// application.properties（改修後・全体） # DB接続（Aで設定済み） spring.datasource.url=jdbc:mysql://localhost:3306/todo_db spring.datasource.username=root spring.datasource.password=パスワード spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver # MyBatis（Aで設定済み） mybatis.configuration.map-underscore-to-camel-case=true logging.level.com.example.todo.mapper=DEBUG # ★ Vol.3で追加：XML Mapper の場所 mybatis.mapper-locations=classpath:mybatis/*.xml // ─── ポイント ─── // ・1行追加するだけ // ・classpath:はSpring Bootの「リソース」を指す // ・*.xml はワイルドカード（全XMLファイル） // ・複数フォルダなら ; で区切る or 別行で書く
05

SubjectMapper.xml を作る

CREATE THE XML

いよいよ初のXML Mapperを書きます。Vol.1で書いた @Select のSQLを、そのまま XML に移すだけ。SQL自体は1文字も変えません。

create-xml SubjectMapper.xml の中身
// XML Mapper の3つの要素
PART 1
DOCTYPE 宣言
XML自体の文法ルールを示す「魔法のおまじない」。毎回コピペでOK。覚える必要なし。
PART 2
<mapper namespace="...">
JavaのMapperインターフェイスのフルパスを書く。これでXMLとJavaがリンクされる。
PART 3
<select id="..." resultType="...">
idはメソッド名、resultTypeは1行を詰めるクラス。本体にSQLを書く。
// SubjectMapper.xml（全体） <?xml version="1.0" encoding="UTF-8"?> <!-- ★ PART 1: DOCTYPE 宣言（コピペ）--> <!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN" "http://mybatis.org/dtd/mybatis-3-mapper.dtd"> <!-- ★ PART 2: namespace --> <mapper namespace="com.example.todo.mapper.SubjectMapper"> <!-- ★ PART 3: select 要素 --> <select id="findAll" resultType="com.example.todo.entity.Subject"> SELECT * FROM subjects ORDER BY id </select> </mapper> // ─── ポイント ─── // ・SQL自体は @Select の中身と同じ // ・namespace と id で Java と繋がる // ・resultType = List<Subject> の中身 // ・DOCTYPE は1文字でも変えると動かない
TIP ──

「resultType と resultMap」は別物です。resultTypeは「単純に1行を1クラスに詰める」シンプル指定。resultMapは「列とプロパティの対応を細かく書く」高度な指定。Vol.4で resultMap を使ってJOINを表現します。今のVol.3では resultType で十分。

06

ファイル作成の手順と Mapperから @Select 削除

CREATE & MODIFY

Eclipseで2つの操作をします。XMLファイルの新規作成と、SubjectMapper.javaの @Select を削除。順番は「XMLを先に作る」のが鉄則──逆だと一瞬アプリが壊れます。

create-mybatis-folder ① mybatis フォルダを作る
// 手順（Eclipse操作）
  1. Package Explorer で src/main/resources を右クリック
  2. New → Folder
  3. Folder name 欄に mybatis と入力
  4. Finish
なぜ「mybatis」フォルダ？

これは慣習的な命名です。application.properties の mybatis.mapper-locations の指定と一致していれば、名前は何でもOK（sql でも mapper でも）。ただし業界的に mybatis が一番見かけます。

// 作業前後のフォルダ構造 // ─── 作業前 ─── src/main/resources/ ├ application.properties ├ static/ └ templates/ └ subjects/ └ list.html // ─── 作業後 ─── src/main/resources/ ├ application.properties ├ mybatis/ // ★ 新規フォルダ ├ static/ └ templates/ └ subjects/ └ list.html // 次の手順でこの mybatis/ の中に // SubjectMapper.xml を作る
create-xml-file ② SubjectMapper.xml を新規作成
// 手順（Eclipse操作）
  1. 作成した mybatis フォルダを右クリック
  2. New → File（★Folderじゃない）
  3. File name 欄に SubjectMapper.xml と入力
  4. Finish
  5. 開いたファイルに、前のセクション「PART 1〜3」のXMLを書く
  6. 保存（Ctrl + S）
なぜ「SubjectMapper.xml」？

慣習的に「Javaのインターフェイス名と完全一致させる」のが業界相場。SubjectMapper.java ↔ SubjectMapper.xml。
こうすると「どのXMLがどのインターフェイスのSQLか」一目で分かる──保守性が爆上がりします。

// 完成した SubjectMapper.xml（再掲） <?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE mapper PUBLIC "-//mybatis.org//DTD Mapper 3.0//EN" "http://mybatis.org/dtd/mybatis-3-mapper.dtd"> <mapper namespace="com.example.todo.mapper.SubjectMapper"> <select id="findAll" resultType="com.example.todo.entity.Subject"> SELECT * FROM subjects ORDER BY id </select> </mapper> // 注意： // この時点ではまだ SubjectMapper.java の // @Select も生きている // → 起動すると // 「同じ findAll が二重定義」エラー！ // → 次のステップで @Select を削除
modify-mapper-java ③ SubjectMapper.java から @Select を削除
// 手順
  1. SubjectMapper.java を開く
  2. @Select(...) 行を削除（メソッド宣言だけ残す）
  3. 不要になった import org.apache.ibatis.annotations.Select; を Ctrl + Shift + O で整理
  4. 保存
重要：SQLは2箇所にあってはダメ

XMLとアノテーション両方に同じ findAll のSQLを書くと、MyBatis が「どっち使えばいいの？」と起動エラーを出します。移したらアノテーション側を消すのがリファクタリングの作法。

// SubjectMapper.java（改修後） package com.example.todo.mapper; import java.util.List; import org.apache.ibatis.annotations.Mapper; // import @Select 不要なので削除 import com.example.todo.entity.Subject; @Mapper public interface SubjectMapper { // @Select は削除した List<Subject> findAll(); // メソッド宣言だけ残す } // ─── Before / After 比較 ─── // Before @Select("SELECT * FROM subjects ORDER BY id") List<Subject> findAll(); // After List<Subject> findAll(); // ↑ SQL の実体は XML に移動
other-mappers StudyRecordMapper は今回は触らない
// ポイント

Vol.3 ではSubjectMapper だけをXMLに移します。StudyRecordMapper はアノテーションのまま放置。これでも問題なく動きます。

理由は2つ：

  • 1度に1つだけ変える（リファクタリングの鉄則）
  • 「アノテーションとXMLの混在」が現場でも普通にあると体験
// Vol.3 完了後の Mapper の状態 // SubjectMapper.java + SubjectMapper.xml @Mapper public interface SubjectMapper { List<Subject> findAll(); // XML側で実装 } // StudyRecordMapper.java（アノテーションのまま） @Mapper public interface StudyRecordMapper { @Select("SELECT * FROM study_records " + "WHERE subject_id = #{subjectId} " + "ORDER BY studied_on") List<StudyRecord> findBySubjectId( @Param("subjectId") Integer subjectId); } // → アノテーション+XML の混在は問題なし // → 現場でもこういう状態は普通 // → 「使い分け」の話は別冊S3で深掘り

08

ハマりどころと対処

TROUBLESHOOTING

XML Mapper 初登場の巻だけあって、地雷が多めです。よくあるトラブル6選をまとめます。

troubles 起きやすいトラブル6選
// よくあるトラブル
TROUBLE 1
Invalid bound statement (not found)
原因：XMLとJavaが繋がっていない。
対処：以下4点を確認。
① application.properties の mybatis.mapper-locations の指定
② XML の namespace が com.example.todo.mapper.SubjectMapper と完全一致
③ <select id="findAll"> がメソッド名と完全一致
④ XMLファイルが src/main/resources/mybatis/ 直下にある
TROUBLE 2
Type 'Subject' is ambiguous
原因：resultType="Subject" と短く書いたが、MyBatisが見つけられない。
対処：フルパスで書く：resultType="com.example.todo.entity.Subject"。
または別冊S3で扱う type-aliases 機能を使う。
TROUBLE 3
Method 'findAll' is defined twice
原因：Java側に @Select、XML側に <select id="findAll"> が両方残っている。
対処：SubjectMapper.java の @Select(...) 行と、不要な import を削除する。
TROUBLE 4
XML構文エラー（Premature end of file 等）
原因：閉じタグ抜け／引用符抜け／DOCTYPE typo。
対処：Eclipseの Problems ビューを開く。XMLファイルにマウスを置くと該当行が示される。DOCTYPE は1文字も変えずコピペが鉄則。
TROUBLE 5
XMLを修正しても反映されない
原因：起動中のアプリは XML を読み直さない（コンパイル時の値で固定）。
対処：アプリを停止→再起動。または DevTools を有効にして自動リスタート。
TROUBLE 6
XML を作ったが空のフォルダになる
原因：New → File ではなく New → Class を選んでしまった等。
対処：拡張子 .xml のファイルが resources/mybatis/ 直下にあるか Package Explorer で確認。
// TROUBLE 1 の典型出力 org.apache.ibatis.binding .BindingException: Invalid bound statement (not found): com.example.todo.mapper .SubjectMapper.findAll // ─── 原因チェックの定型 ─── // ① XMLは読み込まれているか？ // Spring Bootの起動ログに「Parsed mapper file」 // の行があるか探す（DEBUGログ要有効） // ② namespaceは正確か？ <mapper namespace= "com.example.todo.mapper.SubjectMapper"> // ↑ メソッド名「findAll」までは入れない // ↑ クラス名で終わる // ③ id はメソッド名と完全一致？ <select id="findAll"> // ↑ findAll() と完全一致 // ↑ FindAll, find_all はダメ // TROUBLE 5 への根本対策 # application.properties に追加すれば # XML編集時に自動でリスタート spring.devtools.restart.enabled=true # Aで設定済みのはず
