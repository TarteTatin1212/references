★日付ハマり3選★（先回り）

3 DATE PITFALLS ── PREVIEW

本巻の最重要セクション。コードを書く前に、業務でよくある日付のハマりどころ3つを先に見ます。「これを知らずに書いて詰まる」のと「知ってから書く」のは、雲泥の差です。

date-mines 日付3大地雷
// 3つの地雷
⚠ 地雷1 ── 古い型 java.util.Date を使う
症状：java.util.Date を Form のフィールドに使うと、フォーマット指定が複雑になり、TypeHandler が混乱、表示が壊れる。
原因：Javaには「古い日付型（Date, Calendar）」と「新しい日付型（LocalDate, LocalDateTime）」が併存。古い方は混乱の元。
対策：必ず java.time.LocalDate（日付のみ）または java.time.LocalDateTime（日時）を使う。java.util.Date はCの15巻で一切使わない。
⚠ 地雷2 ── @DateTimeFormat を付け忘れる
症状：フォーム送信時に「Failed to convert value of type String to LocalDate」エラー。HTMLから来た文字列 "2026-05-20" を Java の LocalDate にうまく変換できない。
原因：Springは「どのフォーマットで来るか」を明示されていないと変換できない。
対策：Form のフィールドに @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) を付ける。これで「ISO形式（YYYY-MM-DD）でやり取りする」と Spring に伝わる。
⚠ 地雷3 ── Thymeleafで日付の初期値表示が崩れる
症状：編集画面で既存の日付を表示しようとすると "2026-05-20T00:00:00" のような変な形式で出る、または空欄になる。
原因：Thymeleafは LocalDate を自動的に良い形式で出してくれるが、th:value を使う時の書き方に注意が必要。
対策：th:value="${form.studiedOn}" で LocalDate をそのまま渡せばOK。Vol.7では新規登録なので空欄でOK。編集機能は将来作る時に注意。
// 3つの地雷を回避する型・記述 // ─── 地雷1: 型の選択 ─── // ✗ NG private java.util.Date studiedOn; private java.sql.Date studiedOn; private String studiedOn; // ✓ OK private java.time.LocalDate studiedOn; // ↑ DBが DATE 型なら LocalDate // ↑ DBが DATETIME 型なら LocalDateTime // ─── 地雷2: @DateTimeFormat の付与 ─── // ✗ NG public class StudyRecordForm { private LocalDate studiedOn; // → 「Failed to convert」エラー！ } // ✓ OK public class StudyRecordForm { @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) private LocalDate studiedOn; } // ─── 地雷3: Thymeleaf の書き方 ─── // 新規登録（値なし） → そのままでOK <input type="date" th:field="*{studiedOn}"> // 編集（値あり）の時の注意 → 将来課題 // LocalDate なら自動で YYYY-MM-DD 形式で出る // LocalDateTime だと注意が必要
POINT ──

この3つの地雷を「最初に知っている」と知らないでハマる人と比べて、圧倒的に学習効率が違います。本編Vol.7のコードでは、3つすべてを自然にクリアしている形で書きます。読みながら「あ、あの地雷を踏まないように書いてあるな」と気付けたら、あなたのレベルは1段上です。

05

LocalDate × type=date × @DateTimeFormat

THE GOLDEN COMBO

日付処理の「黄金の組み合わせ」は、HTML側の <input type="date">、Java側の LocalDate、Spring側の @DateTimeFormat の3点セット。この3点が揃って初めて、すべてが噛み合います。

golden-combo 3つの役割
// 3点セットの役割
役割1
<input type="date"> ── HTML側
ブラウザが日付ピッカー（カレンダー）を出してくれる。送信時の値の形式は必ず YYYY-MM-DD（ISO 8601形式）。
役割2
LocalDate ── Java側
「時刻なし・日付のみ」を表す Java 標準クラス。java.time パッケージ。比較・計算・フォーマットが全て揃っている。
役割3
@DateTimeFormat ── Spring側
HTMLから来た文字列 "2026-05-20" を LocalDate に自動変換するための指示。iso = ISO.DATE で「ISO形式」を指定。

これ全部が「ISO 8601形式 YYYY-MM-DD」で統一されているのが美しい。3点共通の言語として ISO形式が活躍します。

// 黄金の組み合わせの動き // ① ブラウザで日付を選択 <input type="date" name="studiedOn"> // → ユーザーがカレンダーで「2026年5月20日」を選ぶ // ② フォーム送信 POST /records Content-Type: application/x-www-form-urlencoded studiedOn=2026-05-20&minutes=60&... // → 値は YYYY-MM-DD 形式で送られる // ③ Springが文字列 → LocalDate に変換 public class StudyRecordForm { @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) private LocalDate studiedOn; // ↑ ここで "2026-05-20" → LocalDate.of(2026,5,20) } // ④ Service → Mapper → DB @Insert("INSERT INTO study_records " + "(subject_id, studied_on, minutes, memo) " + "VALUES (#{subjectId}, #{studiedOn}, " + "#{minutes}, #{memo})") void insert(StudyRecord record); // MyBatis の TypeHandler が LocalDate → DATE 変換 // （S4で学んだ通り） // ⑤ DBに保存される study_records.studied_on = '2026-05-20' // → 4段階で型が変換されながら受け渡されている // → 各段階の「設定」がきちんとあれば全部自動

なぜEntityと別物

A v2 Vol.5別冊で学んだ通り、FormとEntityは別の関心です。Formは「画面から来る入力」を表現し、Entityは「DBに対応するデータ」を表現。フィールドは似ていますが、役割が異なるので別クラスにします。

// StudyRecordForm.java（新規） package com.example.todo.form; import java.time.LocalDate; import org.springframework.format.annotation.DateTimeFormat; import lombok.Data; @Data public class StudyRecordForm { private Integer subjectId; @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) private LocalDate studiedOn; private Integer minutes; private String memo; } // ─── ポイント ─── // ・LocalDate（java.util.Date ではない） // ・@DateTimeFormat で形式指定 // ・@Data で getter/setter 自動生成 // ・地雷1・2 を回避済み // 全フィールドの型： // subjectId : Integer（科目ID） // studiedOn : LocalDate（学習日） // minutes : Integer（学習時間 分） // memo : String（任意メモ）

Service の役割：Form → Entity 変換

Service で FormをEntityに詰め替えします。「Mapperに渡すのはEntity」の鉄則を守るため。今回は単純な詰め替えですが、業務ではここに業務ロジックが入ってきます（Vol.13で深掘り）。

// StudyRecordService.java（新規） package com.example.todo.service; import org.springframework.beans.factory.annotation.Autowired; import org.springframework.stereotype.Service; import org.springframework.transaction.annotation.Transactional; import com.example.todo.entity.StudyRecord; import com.example.todo.form.StudyRecordForm; import com.example.todo.mapper.StudyRecordMapper; @Service @Transactional public class StudyRecordService { @Autowired private StudyRecordMapper studyRecordMapper; public void register(StudyRecordForm form) { StudyRecord record = new StudyRecord(); record.setSubjectId(form.getSubjectId()); record.setStudiedOn(form.getStudiedOn()); record.setMinutes(form.getMinutes()); record.setMemo(form.getMemo()); studyRecordMapper.insert(record); } } // ─── ポイント ─── // ・Form → Entity の詰め替え // ・Mapper には Entity を渡す // ・@Transactional でロールバック保証 // ・id/created_at は DB側で自動付与

ハマりどころと対処

TROUBLESHOOTING

Vol.7の地雷地帯。セクション04の「3大地雷」以外にも、細かな罠がいくつかあります。

troubles 起きやすいトラブル6選
// よくあるトラブル
TROUBLE 1
Failed to convert value of type 'String' to LocalDate
原因：@DateTimeFormat 付け忘れ（地雷2）。
対処：Form の studiedOn フィールドに @DateTimeFormat(iso = DateTimeFormat.ISO.DATE) を付ける。
TROUBLE 2
DateTimeFormat の import が間違い
原因：Eclipse の自動 import で org.joda.time.format.DateTimeFormat（古い Joda-Time）を選んでしまった。
対処：org.springframework.format.annotation.DateTimeFormat に書き換える。
TROUBLE 3
submit すると 400 エラー
原因：必須項目が空のまま送信／日付欄に何も入っていない。
対処：日付欄に値を入れる。Vol.10でバリデーションを学べば、適切なエラーメッセージを出せるようになる。
TROUBLE 4
INSERT は走るが画面に出ない
原因：DBには登録されているが、リダイレクト先 /subjects のJOIN取得でN+1の比較画面（/subjects/legacy）に行ってしまった等のミス。
対処：URLを直接 /subjects にアクセスして確認。
TROUBLE 5
th:object と form をネストし忘れ
原因：th:field を th:object の外で使うとエラー。
対処：<form th:object="${form}">...</form> の中に th:field 入りの input を入れる。
TROUBLE 6
セレクトボックスが空
原因：Controllerで allSubjects を model 追加し忘れ。
対処：newForm() メソッドに model.addAttribute("allSubjects", subjectService.findAll()) を入れる。
