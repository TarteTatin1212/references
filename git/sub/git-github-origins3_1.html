01

「ブランチ」という比喩

METAPHOR OF BRANCH

Gitに限らず、ソフトウェアの世界には「ブランチ(枝)」という言葉が頻繁に登場します。 なぜ「枝」なのか? この比喩はどこから来たのか? 実は木という形が、人類が「分岐」を理解する最古の構造の一つだったのです。

branch / branch metaphor
ブランチ / 分岐の比喩
「木の枝のように、一つの幹から枝分かれする」イメージ。版管理に限らず、データ構造、議論の分岐、人類の系図など広く使われる古い比喩
tree / tree structure
ツリー / 木構造
「一つの根から枝分かれしていく構造」。コンピュータ科学では基本的なデータ構造の一つで、ファイルシステムも家系図もこの形をしている
fork / branching
フォーク / 枝分かれ
「分岐する瞬間そのもの」。「fork」は元々「(道や川の)分岐点」を指す英単語。プロセスもプロジェクトも、ここから「枝分かれ」する

人類は古くから「枝」で考えてきた

家系図、系統樹、論理学の分類木、組織図——人間は「分岐するもの」を木の形で描く強い習性があります。これは偶然ではありません。私たちが日常的に目にしている自然物の中で、「一つの根から無限に枝分かれする」構造として、木が最も身近で分かりやすいからです。

17世紀のスコラ哲学では、概念を分類する時に「ポルピュリオスの樹」と呼ばれる図がよく使われました。19世紀、ダーウィンが「種の起源」で進化を説明する時、彼が描いたのも一本の枝分かれする木でした。「すべての生物は共通の祖先から枝分かれした」というイメージを、彼は紙の上で木として可視化したのです。

コンピュータ科学への移植

1950-60年代、コンピュータ科学が学問として成立する時、「木構造(tree structure)」は基本中の基本のデータ構造として定着しました。ファイルシステムは木構造。ディレクトリの中にディレクトリがあり、その中にファイルがある——これは一本の幹から枝分かれする形そのものです。

探索木(search tree)、構文木(syntax tree)、決定木(decision tree)——コンピュータ科学のあらゆる分野で「木」は使われます。プログラマにとって、「木」は最も馴染みのある抽象構造の一つになりました。

「我々は世界を分類し、整理する時、自然と木の形に描く。これはおそらく、人間の認知の根本的な性質なのだろう」 — Lakoff & Johnson "Metaphors We Live By"

版管理に「ブランチ」が来た日

1970年代、最初の版管理ツール SCCS が登場した時、すでに「枝」という比喩が使われていました。ある版から派生して、独立に進化する別の系列——これを「ブランチ」と呼ぶのは、ほとんど自明の選択でした。プログラマの頭の中に既に「木」のイメージが染み付いていたのです。

ただし、初期のツールでは「ブランチ」は例外的な操作でした。製品版の保守のために、リリース済みの版から別系統を立てる——これはチームで合意して行う重大な手続きで、日常的には起きないことでした。「ブランチを切る = 何か特別なことをする」という時代が長く続いたのです。

「軽量ブランチ」が変えたもの

この常識を破ったのが、後に Git が採用する「軽量ブランチ(lightweight branch)」という発想です。ブランチを例外的な手続きではなく、日常的な作業単位に変えてしまった。git switch -c feature/new と打つだけで、誰でも一瞬で新しい枝を作れる——この変化は、開発文化そのものを変える革命でした。

本巻のテーマは、この革命がいかに可能になったのかを辿ることです。「ブランチ」という言葉の重みを軽くした技術的な発明、そしてそれが解放した思考の自由——一つ一つ見ていきましょう。

∴ POINT

「ブランチ」という言葉は、何百年も前から人類が分岐を理解する道具だった。Git が革命的だったのは、この比喩を再発明したのではなく、「比喩通りに、軽やかに使える」道具を作ったことです。言葉が現実に追いついた瞬間と言えるかもしれません。

02

旧来ツールの重いブランチ

HEAVYWEIGHT BRANCHES

Git 以前のツールでは、ブランチを切るのは大事業でした。 なぜ Subversion 等のツールでブランチが重かったのか? その答えは、彼らがブランチを「ファイルやディレクトリのコピー」として実装していたという、 根本的な設計上の決断にあります。

このセクションの用語
copy / file copy
コピー / ファイル複製
「元のファイル群をもう一式、別の場所に複製すること」。古いツールでは、ブランチを作る = ファイル全部をコピーすることだった
cheap copy
チープコピー / 安価な複製
「実際にはファイルを複製せず、参照だけを作る賢い複製」。Subversion が採用した工夫だが、それでも操作は重かった
heavyweight
ヘビーウェイト / 重量級
「コストが大きく、気軽にやれない」操作の性質。Git 以前のブランチはこれ。物理的なコストと心理的なコストの両方が重かった
なぜブランチが重かったか PRE-GIT WORLD

● 権限設定の更新が必要
● 誰がいつ作ったかを記録
● 後の整理(削除)も大事業

"In Subversion, creating a branch was an event." — SVN時代の典型的な感想

SVN はブランチを「ディレクトリ」と見なした

Subversion(SVN)の設計者たちは、ブランチを「ファイル群のコピー」として実装しました。具体的には、リポジトリ内に trunk/(本流)、branches/(枝)、tags/(印)という特別なディレクトリを置く慣習を作ったのです。

新しいブランチを作るには、svn copy trunk branches/feature-x という具合に、trunk の中身をまるごと branches の下にコピーする。つまり「ブランチを作る」は「ファイルを複製する」とほぼ同義だったのです。

「cheap copy」の工夫

もちろん、本当にファイルを物理的に全コピーしていたら、大規模リポジトリでは時間もディスクも持ちません。SVN はこれを賢く回避していて、cheap copy(チープコピー)という技術を使っていました。これは「新しい場所に参照だけを作り、実体は元のファイルを使い回す」工夫で、いわばシンボリックリンクに近い発想です。

これによって、ブランチ作成自体はファイルサイズに関係なく速くなりました。しかし、それでもディレクトリ構造を作る、メタデータを更新する、リポジトリ全体のリビジョン番号を進める——これらの処理が必要で、Git の軽量ブランチに比べれば桁違いに重い操作でした。

心理的な重み

技術的なコストよりも大きかったのが、心理的なコストです。SVN でブランチを作ることは、リポジトリ全体の構造に変更を加える行為でした。branches/ ディレクトリは「みんなが見る公共の場所」。あなたが個人的な実験のためにブランチを作っても、それは全員から見える。

その結果、開発者は「ブランチを作ることに躊躇する」習性が育ちました。「大きな機能のために、チームで合意してから作るもの」という感覚です。日常的な思いつきや実験のためにブランチを切るなど、考えもしなかった——それが SVN 時代の感覚でした。

「SVN では、ブランチは『重大な決断』だった。Git では、ブランチは『今日のお昼に何を食べるか』くらいの軽さで決められる」 — SVN から Git に移行した多くのエンジニアの感想

マージはさらに地獄だった

ブランチ作成より、もっと大変だったのがマージでした。SVN 初期のバージョンでは、マージ操作には「どこからどこまでをマージするか」を人間が手動で指定する必要がありました。バージョン番号を覚えておいて、 svn merge -r 100:200 のように指定する。指定を間違えると、変更が二重に適用されたり、抜け落ちたりする悪夢が起きました。

SVN 1.5 以降、これはマージ追跡(merge tracking)機能で改善されましたが、根本的な「ファイル複製ベースのブランチ」というモデルがある限り、Git の軽快さには程遠かったのです。「SVN ではマージしたくない」が現場の合言葉でした。

その結果、ブランチが使われなくなる

ブランチ作成もマージも重いと、開発者の行動はどう変わるか。答えは単純で、「ブランチを使わなくなる」のです。皆が trunk(本流)で直接作業し、互いに変更を上書きしないようコミュニケーションする——そんな運用が一般的でした。

これは Linus が「SVN はダメだ」と切り捨てた根本的な理由の一つでもあります。ブランチが使えないということは、並行開発の自由を制限されるということだからです。次の節で、彼がどうやってこれを解決したかを見ていきます。

∴ INSIGHT

SVN 等のブランチが重かったのは、それを「ファイル群のコピー」として実装したからです。物理的なコストは工夫で減らせたものの、概念モデル自体が重く、結果として心理的にも重い操作になってしまった。道具のコストは、技術と概念の両方に宿るのです。

03

軽量ブランチの発明

LIGHTWEIGHT BRANCHES

Git が革命的だったのは、「ブランチ = ポインタ1つ」と再定義したことです。 具体的には、「あるコミットの ID を記録した41バイトのテキストファイル」。 これだけで、SVN が大事業だった操作が一瞬になりました。 設計思想を変えれば、コストは劇的に減る——その教科書のような事例です。

このセクションの用語
pointer / reference
ポインタ / 参照
「実体を指す『矢印』のような小さなデータ」。実体ではなく、実体の在り処を持つ。Git のブランチの正体はこれ
ref / refs
レフ / 参照群
「Git で使う『名前付きの参照』全般」。ブランチもタグも、本質的には全部 ref。.git/refs/ という場所に保存される
commit hash / commit ID
コミットハッシュ / コミットID
「あるコミットを一意に特定する40文字の文字列」。ブランチは、この ID を1つ書いたファイルとして存在している
41バイトの革命 BRANCH = ONE FILE
// ブランチの実体
[ ファイルシステム上 ] .git/ ├─ HEAD ← 現在地 ├─ refs/ │ ├─ heads/ │ │ ├─ main │ │ ├─ feature │ │ └─ develop │ └─ tags/ │ └─ v1.0.0 └─ objects/ [ main ファイルの中身 ] a3f2c1d8b7e5a92... ↑ たった40文字 + 改行 = 41バイトのテキスト! これがブランチの「正体」
確認してみる

$ cat .git/refs/heads/main
a3f2c1d8b7e5a92f...

$ ls -la .git/refs/heads/main
... 41 bytes ...

"A branch in Git is just a tiny file containing a SHA-1 hash." — Pro Git book

発想の転換

Linus の天才性は、ブランチを「ファイル群のコピー」ではなく「コミットへの参照」と定義し直したことにあります。SVN は「ブランチ = ファイルの集まり」と考えていました。Git は「ブランチ = ある時点を指す1個の印」と考えたのです。

この発想転換の意味は深い。「最新のコミット」だけを記憶しておけば、そこから過去のすべての履歴を辿れる——なぜなら、各コミットは自分の親コミットを覚えているからです。終端の1点を覚えれば、線全体が分かる。ブランチを表現するのに、終端の1点で十分なのです。

refs/heads/ という発明

具体的に Git リポジトリの中を覗いてみましょう。.git/refs/heads/ というディレクトリがあります。ここに、あなたのリポジトリの全ブランチが、それぞれ1つの小さなテキストファイルとして存在しています。

たとえば .git/refs/heads/main というファイルを開くと、中身は a3f2c1d8b7e5a92f... という40文字のハッシュ値が1行書かれているだけ。これが main ブランチの「正体」です。改行を含めても41バイト——フロッピーディスクの時代の感覚で言えば、無視できるほど小さい。

新しいブランチを作るとは、このような小さなファイルを1つ作るだけの操作です。「ブランチを切る」が「ファイルを1個作る」と等価——だから一瞬で終わるのです。

「Git のブランチは、ファイルの集合ではなく、ある時点へのラベルである」 — Git の内部設計を語る典型的な解釈

HEAD という指揮棒

もう一つ重要なファイルが .git/HEAD です。これも単純なテキストファイルで、中身は ref: refs/heads/main のように、「今、自分はどのブランチにいるか」を示しているだけ。

git switch develop でブランチを切り替える操作は、本質的には .git/HEAD の中身を ref: refs/heads/develop に書き換えるだけです。これに付随してワーキングディレクトリのファイルが入れ替わりますが、概念的な操作は「1個のテキストファイルを更新する」ことに集約されます。

ブランチ削除が一瞬な理由

削除も同じく、ものすごく簡単です。git branch -d feature という操作は、本質的には .git/refs/heads/feature という1つのファイルを削除する操作です。中身のコミットや、過去の履歴が消えるわけではない——ただ「ポインタ」が消えるだけです。

これは重要な性質です。ブランチを消しても、そこから辿れたコミット群は Git のオブジェクトストア(.git/objects/)にしばらく残ります。一定期間が経つとガベージコレクションで消えますが、それまでは git reflog 等で復元可能。「軽く作って、軽く消せる」が成立しているのです。

「ファイルシステムを使う」という素朴な選択

もう一つ感心するのは、Git がこれらを普通のファイルシステム上の普通のファイルとして実装していることです。専用のデータベースも、複雑なバイナリ形式も使っていない。cat や ls で中身を見られる。これも UNIX 哲学の表れです。「テキストファイルでできることは、テキストファイルで」という、極めて素朴な選択。

その結果、Git の内部は「ハッキングしやすい」性質を持っています。何かおかしいと思ったら、ファイルを直接覗ける。場合によっては手で編集することすらできる。これが「Git は透明な道具」と呼ばれる理由でもあります。

∴ INSIGHT

Git の軽量ブランチは、「ブランチとは何か」の定義そのものを変えた発明です。SVN が「ファイル群のコピー」と考えていたものを、Git は「1個のテキストファイル」に再定義した。抽象を変えれば、コストは劇的に変わる——ソフトウェア設計の真髄が、ここに凝縮されています。

04

HEAD という哲学

PHILOSOPHY OF HEAD

Git に HEAD という概念があるのを知っていますか? リファレンス本編で「現在地を指すポインタ」と学びましたが、 これは単なる技術的な仕組みではなく、「『今』をデータとして扱う」という、 かなり深い哲学的選択の表れです。

このセクションの用語
HEAD
ヘッド / 先頭
「いま自分が立っている地点」を指す特別な参照。通常はあるブランチを指していて、そのブランチの先端のコミットが「現在地」になる
detached HEAD
デタッチト・ヘッド / 切り離されたヘッド
「どのブランチにも属さない、宙ぶらりんの状態」。コミット ID を直接指定して移動した時に起きる。慌てる必要はないが、そのままコミットすると迷子になる
symbolic ref
シンボリックレフ / 名前による参照
「他の参照を指す参照」。HEAD は普通、特定のブランチ名を指している。直接コミット ID を指す形式も可能(これが detached の状態)
「今」を表現する HEAD AS DATA
// HEAD の2つの状態
[ 通常状態: ブランチを指す ] HEAD ↓ "ref: refs/heads/main" ↓ main ↓ a3f2c1d... (コミット) [ detached HEAD: 直接指す ] HEAD ↓ "a3f2c1d..." (コミットID) * この場合、コミットしても どのブランチにも属さない
確認してみる

$ cat .git/HEAD
ref: refs/heads/main

$ git checkout a3f2c1d
Note: switching to 'a3f2c1d'.
You are in 'detached HEAD' state.

$ cat .git/HEAD
a3f2c1d8b7e5a92f...

「今」を変数として扱う

多くのソフトウェアでは、「現在の状態」は暗黙的に管理されます。エディタなら「いま開いているファイル」、ブラウザなら「いま表示しているページ」——これらは内部的にどこかで覚えられていますが、ユーザーが直接アクセスできる概念ではありません。

Git は違います。「現在いる場所」を HEAD という名前のある変数として明示的に扱います。ユーザーは git log HEAD や git show HEAD~3 のように、「今」を起点にした操作を自由に行えるのです。

HEAD の柔軟性

HEAD の表現力は思ったより豊かです:

● HEAD = 現在のコミット
● HEAD~1 または HEAD^ = 1つ前のコミット
● HEAD~5 = 5つ前のコミット
● HEAD@{1} = 1操作前の HEAD(reflog ベース)
● HEAD@{yesterday} = 昨日の HEAD

「現在地」を起点に、時間軸・操作軸の両方で過去を辿れる。これは「今」を一級市民として扱う設計の力です。

「Git は『今』を抽象化した。これによって、過去への参照が無限に組み立てられるようになった」 — Git の設計を分析した解釈の一つ

detached HEAD の意味

リファレンス本編で「detached HEAD」(切り離された HEAD)という状態を扱いました。これは初学者を一番驚かせる概念ですが、HEAD の設計を理解すると自然に納得できます。

通常、HEAD はブランチを指している(間接参照)。ref: refs/heads/main のように。これだと「いま main ブランチにいる」ということになります。

でも、HEAD はコミット ID を直接指すこともできる。たとえば git checkout a3f2c1d のように特定のコミットを指定すると、HEAD は a3f2c1d という値そのものを持つようになります。これが detached HEAD の状態です。

なぜ「迷子」になるのか

detached HEAD でコミットすると、新しいコミットがどのブランチにも属さない状態で誕生します。例えるなら、住所のない建物を建てたようなものです。建物自体は存在しますが、地図上には載っていない——だから他のブランチに移動した瞬間、たどり着く手段がなくなる。これが「迷子のコミット」と呼ばれる現象です。

解決は単純で、git switch -c new-branch でその場所からブランチを切る。これによってコミットに住所(ブランチ名)が与えられ、世界に正式に存在することになる。ブランチとは、コミットに住所を与える仕組みでもあるのです。

「移動可能な現在地」という発想

HEAD の真の哲学は、「現在地は固定されない」ということです。あなたは過去のコミットに「移動」できる(detached HEAD)。別のブランチに「移動」できる。マージで複数の歴史を「合流」できる。コミット ID を書き換えれば「歴史を改変」することすらできる。

これは時間を扱う非常に柔軟なモデルです。プログラム本体は時間の中で動いていますが、その「どこから見るか」を、HEAD という1つの変数で表現している。Git は「視点」を明示的なデータにした道具なのです。

哲学としての HEAD

HEAD の設計には、もう少し深い思想が見えます。「今」を不可侵の絶対座標ではなく、可動な相対座標として扱う——これは現代的な時間観です。私たちが日常で「今」と呼んでいるものは、実は常に「何かを基準にした今」であり、視点が変われば変わる。

Git はこの観念をデータ構造として実装しました。「今いるブランチ」「今の HEAD」「今のステージング」——これらは全て、「あなたの視点」を構成する変数です。視点を変えれば、見えるものも、できる操作も変わる。Git の操作とは、視点の操作であると言えるかもしれません。

∴ INSIGHT

HEAD は単なる技術的概念ではなく、「視点を一級市民にする」哲学の表現です。これによって、過去の任意の地点から世界を見て、操作することが可能になった。Git は時間を扱う道具であると同時に、視点を扱う道具でもあるのです。

05

マージという数学

MATHEMATICS OF MERGE

Git のマージは、実は「数学的な操作」です。 Fast-forward と 3-way マージの違いには、明確な数学的理由があります。 そして 3-way マージというアルゴリズムは、Git が発明したものではなく、 1980年代から練り上げられてきた集合論的な解でした。

このセクションの用語
3-way merge
スリーウェイマージ / 3点合流
「共通の祖先と、2つの変更を見比べてマージする方式」。各変更が共通祖先からどう変わったかを見ることで、賢く統合できる
common ancestor / merge base
共通祖先 / マージベース
「2つのブランチが最後に共有していたコミット」。3-way マージで「基準点」として使われる
fast-forward
ファストフォワード / 早送り
「片方のブランチが、もう片方の延長線上にある時の特殊な合流」。マージコミットを作らず、ポインタを前進させるだけで済む
合流のアルゴリズム 3-WAY MERGE
// Fast-forward の構造
[ FF が可能な状況 ] main: A───B ↑ feature: A───B───C───D ↑ main は feature の祖先 → main を D まで進めるだけ 結果: main: A───B───C───D
// 3-way merge の構造
[ 3-way が必要な状況 ] main: A───B───X───Y feature: └─C───D 共通祖先 = B (merge base) main側変更: B → Y feature側変更: B → D 3点を見比べて合流 main: A───B───X───Y───M └─C───D────┘ (Mがマージコミット)

マージとは「2つの変更の統合」

マージという操作を抽象的に言えば、「同じ起点から派生した2つの変更を、1つに統合する」ことです。プログラマには馴染みのある操作ですが、これは数学的・論理的にはなかなか難しい問題を含んでいます。

たとえば、共通の出発点から、ある人は A 行目を変えた。別の人は B 行目を変えた。これを統合するには「A 行目の変更と B 行目の変更、両方を反映した結果」を作ればよい。これは比較的シンプルです。

しかし、もし「同じ行に対して、両方が違う変更をした」としたら? ここで初めてコンフリクトが発生します。次節で詳しく扱いますが、コンフリクトの本質は「機械的に決められない選択」が現れる瞬間なのです。

Fast-forward の条件

マージの中でも特殊で美しいのが、Fast-forward(早送り)と呼ばれるケースです。これが起きる条件は明確で、「合流先のブランチが、合流元の祖先である」こと。つまり、片方のブランチが、もう片方の延長線上にある状況です。

この場合、本当の意味で「マージ」する必要はありません。合流先のポインタを、合流元の最新コミットまで進めるだけで済みます。これが Fast-forward という名前の由来です——テープを早送りするように、ポインタが進む。

具体的には、main ブランチで作業せず、ずっと feature ブランチだけで開発していたなら、feature を main にマージする時、main には新しいコミットが何もないので、Fast-forward が成立します。マージコミットすら作られない——「合流」ではなく「移動」なのです。

3-way merge の発見

Fast-forward が使えない状況——両方のブランチに独立した変更があるケース——では、3-way merge という別のアルゴリズムが必要になります。これは Git の発明ではなく、1980年代に diff3 というツールで定式化された古典的な手法です。

3-way の「3」は、「3つの状態を見比べる」から来ています:

① 共通祖先(merge base): 2つのブランチが最後に同じだった点
② ブランチA の最新
③ ブランチB の最新

この3点を見比べることで、「A がどう変えたか」「B がどう変えたか」を独立に判定できる。同じ場所に違う変更が来ていなければ、自動で統合できる。これが 3-way merge の仕組みです。

「diff3 という古いツールが、3-way merge の数学的基礎を作った。Git はその上に立っている」 — マージアルゴリズム史の一般的な見解

共通祖先を探すアルゴリズム

3-way merge の鍵は、「共通祖先をどう特定するか」です。Git は、コミット履歴を辿って「2つのブランチの両方から到達可能な、最も新しいコミット」を探します。これをmerge baseと呼びます。

履歴が DAG(有向非巡回グラフ)構造になっているので、これは「最大共通祖先」を求める典型的なグラフ問題です。Git は効率的なアルゴリズムでこれを高速に求めています。あなたが git merge を打った瞬間、内部でこの計算が走っているのです。

マージコミットの特殊性

3-way merge の結果として生まれるマージコミットは、Git の中で唯一「2つの親を持つコミット」です。普通のコミットは親が1つだけ——直前のコミットを指しています。マージコミットだけが例外で、「これとこれを合流させた」という事実を、2本の親リンクで表現します。

これによって、後から履歴を辿る時、「ここでマージが起きた」ことが明確に分かります。git log --graph で表示される枝分かれ図は、マージコミットの「2親構造」を視覚化したものです。「合流の歴史」が、データとして残る——これはチーム開発の透明性にとって極めて重要な性質です。

マージという協調の儀式

マージは技術的な操作であると同時に、協調の儀式でもあります。「私の変更とあなたの変更を、ここで合わせる」——これは社会的な意味を持つ瞬間です。GitHub の Pull Request も、本質はこの「合流の儀式」を可視化したものに他なりません。

3-way merge という数学的仕組みは、こうした「合流の必然性と困難さ」を、ソフトウェア工学の問題として体系化したものです。50年に近い研究の蓄積の上に、現在の Git のマージが成り立っているのです。

∴ INSIGHT

マージは単なる「ファイルの結合」ではなく、数学的に定義された統合操作です。Fast-forward と 3-way merge の違い、共通祖先の特定アルゴリズム、マージコミットの2親構造——これら全てが、「変化を統合する」という普遍的問題への、人類の知恵の蓄積です。

06

コンフリクトの本質

NATURE OF CONFLICTS

コンフリクトは初学者を最も怖がらせる現象ですが、本質を理解すると怖くなくなります。 コンフリクトとは「Git が壊れた」ではなく、「機械では決められない選択がある」と Git が言っているだけ。 本節では、その情報学的な本質を解き明かします。

このセクションの用語
conflict / merge conflict
コンフリクト / マージ衝突
「同じ場所に対して、2つの異なる変更が来た時に発生する競合」。機械的には決められないので、人間に判断を仰ぐ
conflict marker
コンフリクトマーカー / 衝突印
「コンフリクトが起きた場所をファイル内に示す目印」。<<< === >>> の3記号で構成される
resolution / conflict resolution
レゾリューション / 衝突解決
「人間が判断してコンフリクトを解消すること」。マーカーを取り除いて正しい形にし、git add で解決を宣言する
情報の限界としてのコンフリクト WHY MACHINE CAN'T DECIDE
// 3-way merge と判断の限界
[ 共通祖先 ] function greet() { return "Hello"; } [ main側の変更 ] function greet() { return "Hello, World"; } [ feature側の変更 ] function greet() { return "Hi, World"; } ↓ 機械的に判断不能 [ コンフリクト発生 ] function greet() { <<<<<<< HEAD return "Hello, World"; ======= return "Hi, World"; >>>>>>> feature }
コンフリクトが起きない例

● 違うファイルへの変更
● 違う行への変更
● 同じ行でも、片方は無変更

→ これらは自動マージで解決

コンフリクトはエラーではない

初心者の多くが、コンフリクトを「Git のエラー」「やってはいけないこと」と誤解します。しかし、これは誤解です。コンフリクトは「Git が正しく動作している」証拠であり、むしろ Git が賢いからこそ発生する現象です。

機械的な自動マージで全てを処理してしまう設計だと、「機械が間違った選択をしても気づけない」ことになります。Git はそうではなく、「機械では判断できない場面は、必ず人間に判断を仰ぐ」設計を選びました。これは設計者の誠実さの現れです。

「同じ場所、違う変更」

コンフリクトが発生する条件は、極めて明確です。3-way merge の3点を見比べた時、同じ場所(同じ行とその周辺)に対して、両方のブランチが異なる変更を加えていた——この時にコンフリクトが起きます。

例えるなら、家族で買い物リストを共同編集している状況です。元のリストに「牛乳」とあった。あなたは「低脂肪牛乳」に書き換えた。同時に、家族は「豆乳」に書き換えた。これを統合する時、システムは「結局どっちにすればいいのか」を判断できません。両方の意図を尊重する自動解はないのです。

コンフリクトマーカーの読み方

Git がコンフリクトを検出すると、ファイルにコンフリクトマーカーを埋め込みます:

<<<<<<< HEAD から ======= までが 現在のブランチ(HEAD 側)の変更。
======= から >>>>>>> feature までが 取り込もうとしているブランチの変更。

つまり、Git は「こちらにはこれが、あちらにはあれが書いてある。どちらが正しいか、あなたが決めてくれ」と言っているのです。あなたの仕事は、マーカーを取り除いて、「正しい最終的な形」を書くこと——それが解決(resolution)です。

「コンフリクトとは、機械が解けない問題を、人間に正しく譲渡する仕組みである」 — ある熟練エンジニアの教え

第3の選択肢の発見

コンフリクト解決の時、ありがちな判断は「こっちかあっちか、どちらを取るか」です。が、優れた解決は第3の道を見つけることもあります。

先ほどの例なら、main は「Hello, World」、feature は「Hi, World」。そもそも、両方とも「挨拶」を変えたかったのではないか? もしかして真の意図は「もっと丁寧に」だったのかもしれない。それなら「Good morning, World」が正解かもしれない——

これはコードでも同じです。コンフリクトは、しばしば「2人の開発者が、同じ問題を別の角度から解こうとしていた」ことを示しています。それを統合する時、両者の意図を汲んだ新しい解を作れることもある。コンフリクトは、創造的な機会でもあるのです。

情報学的な視点

もう少し抽象的な視点で捉えると、コンフリクトは「並行操作の情報的限界」を示しています。並行して進む2つの作業が、最後に統合される時——両者の意図が完全に独立であれば、機械的に統合可能です。両者が同じ場所に重なれば、追加情報(=人間の判断)が必要になる。

これは、Git に限らず分散システム全般で起きる本質的な問題です。データベースの分散レプリケーション、複数編集者によるドキュメント同時編集——どこにも「同じ場所への並行変更」の問題は付き纏います。Git のコンフリクトは、その典型的・教科書的な例なのです。

恐れるのは無知の所産

コンフリクトを恐れる気持ちは、それが「分からないもの」だから生まれます。仕組みが分かれば、コンフリクトは怖くない。「2つの変更が同じ場所にぶつかった、人間が判断する」——それだけのことだと分かれば、淡々と解決できます。

むしろ、コンフリクトを「Git との対話」と捉える人もいます。Git が「ここをどうしますか?」と聞いてきた。あなたは選択肢を理解し、判断する。判断したら git add で「これで決まり」と告げ、git commit で確定する——これは、機械と人間の役割分担の、極めて美しい例だと言えます。

∴ INSIGHT

コンフリクトはエラーではなく、機能です。Git が「機械的には判断できない」と正しく伝えてくれる仕組みであり、人間の判断を尊重する設計の現れ。コンフリクトを恐れるな、それは Git からの対話の招きである——この視点を持てた瞬間、Git との関係が変わります。

07

ブランチモデルの進化

EVOLUTION OF MODELS

Git が軽量ブランチを実現したことで、現場では「ブランチをどう使うか」の流儀が次々と発明されました。 Git Flow、GitHub Flow、Trunk-based Development—— これらは技術ではなく、チームの社会的な合意のパターンです。

このセクションの用語
Git Flow
ギットフロー(モデル名)
「複数のブランチを役割分担させた、しっかりした運用モデル」。2010年に Vincent Driessen が提案。リリース・開発・本番を厳密に分ける
GitHub Flow
ギットハブフロー(モデル名)
「main と機能ブランチだけのシンプルなモデル」。GitHub が2011年に提唱。継続的にデプロイする Web サービス向け
Trunk-based Development
トランクベース開発
「ほぼ1本のブランチ(trunk = main)で開発する流儀」。短命なブランチだけ作り、1日以内に main へ統合する。大規模・高頻度デプロイ向け
3つの主要モデル SOCIAL CONVENTIONS
// モデルの位置づけ
Git Flow (2010)

● 厳密な役割分担
● 製品の正式リリース重視
● パッケージソフト向け

GitHub Flow (2011)

● シンプル(main + feature)
● 継続的デプロイ向け
● Webサービス・SaaS向け

Trunk-based (2010s後半)

● ほぼ1本の main で開発
● 短命な機能ブランチのみ
● 大規模・高頻度デプロイ向け

軽量ブランチが解放したもの

SVN 時代、ブランチは「特別な操作」でした。誰もブランチを多用しないので、運用モデルもシンプルでした——「本流で開発し、リリース時に枝を切る」程度。

Git が軽量ブランチを実現すると、状況は一変します。「機能ごとに枝を切る」「個人実験で枝を切る」「修正のために枝を切る」——あらゆる作業がブランチを伴う日常になりました。すると今度は、「ブランチをどう使えば混乱しないか」という新しい問題が浮上したのです。

Git Flow の誕生(2010年)

2010年、オランダのエンジニア ヴィンセント・ドリッセンが、ブログ記事 "A successful Git branching model" を公開します。彼が提案したのは、5種類のブランチを役割で分ける厳密なモデルでした:

● main: 本番にリリースされた状態
● develop: 次のリリースに向けた統合先
● feature/*: 個別機能の開発
● release/*: リリース準備中
● hotfix/*: 緊急修正

これは、当時のソフトウェア開発の典型——「数ヶ月に一度の正式リリース」を前提とした設計でした。記事は瞬く間に世界中で読まれ、Git Flow は「ベストプラクティス」として広く採用されました。

GitHub Flow の対抗(2011年)

Git Flow から1年後、GitHub 自身が異なるモデルを提唱します。GitHub の創業者 スコット・チャコンのブログ記事で、後に "GitHub Flow" と呼ばれるシンプルなモデルです:

● main は常にデプロイ可能な状態
● 機能ごとに main から枝を切る
● PR でレビュー
● マージしたら即デプロイ

これだけです。develop も release もありません。シンプルさが特徴でした。これは、「継続的にデプロイする Web サービス」には完璧にフィットしました。月に1回ではなく、1日に何度もデプロイする時代——重い手続きはもう要らないのです。

Trunk-based Development の台頭

2010年代後半、Google や Facebook のような巨大な開発組織から、もう一つの流儀が広まります。Trunk-based Development(トランクベース開発)です。これは GitHub Flow をさらに極端にして、「ブランチは1日以内に main へ統合する」を徹底するもの。

この方式の特徴は、「長く生きるブランチを許さない」こと。長期ブランチがあると、後でマージが大変になる(コンフリクトが大量発生する)からです。代わりに、未完成の機能は「フィーチャーフラグ」という仕組みで制御し、コードはマージしつつ、ユーザーには見せない方式を取ります。

Google は内部的にこの方式を採用しており、毎日数万件のコミットが、ほぼリアルタイムで main に統合されると言われています。これは Git Flow の対極にある運用です。

「Git Flow は『製品』のためのモデル、GitHub Flow は『サービス』のためのモデル、Trunk-based は『規模』のためのモデルだ」 — 業界の通説的な解釈

なぜモデルが分裂したのか

3つのモデルが共存しているのは、「現代のソフトウェア開発が一様ではない」からです。半年に一度パッケージをリリースするオフィスソフト、毎日新機能をデプロイする Web サービス、数万人のエンジニアが同時に動く巨大組織——それぞれに最適な運用は違います。

面白いのは、これら全てが「同じ Git の上で実現される」ことです。Git は特定の運用モデルを強制しません。軽量ブランチという技術的な土台の上に、各組織が自分たちに合った社会的な合意を築く。Git は「運用を選ばせる柔軟性」を提供する道具だと言えます。

道具と文化の循環、再び

本シリーズ Vol.1 で見たように、Git は「道具と文化が互いを生み出す」循環の中で発展してきました。ブランチモデルもまた、その循環の一例です。軽量ブランチという技術が、新しい運用モデルを可能にし、運用モデルが新しいツール(GitHub、GitLab、Bitbucket)を要求し、ツールがさらに新しい運用モデルを支える——終わらない循環。

あなたが現場で「うちのチームは GitHub Flow です」「Git Flow を採用しています」と聞く時、それは「単なる技術選択」ではなく、「文化的な選択」でもあるのです。チームの規模、リリース頻度、顧客との関係——これら全てが背景にある。道具は、それを使う人々の社会を映すのです。

∴ INSIGHT

ブランチモデルは技術ではなく、社会的な合意です。Git Flow、GitHub Flow、Trunk-based——それぞれは、特定の組織と時代の合理性を反映した運用の結晶。あなたの現場のモデルは、あなたの現場の社会を映している。それを意識して使えば、より良い選択ができるはずです。

08

自由としてのブランチ

BRANCH AS FREEDOM

Git の軽量ブランチが本当にもたらしたものは何か? 速さや効率ではありません——それは「実験する自由」でした。 気軽に失敗できる、気軽に捨てられる、気軽にやり直せる。 この心理的安全性こそが、Git が開発文化に与えた最大の贈り物です。

このセクションの用語
psychological safety
サイコロジカル・セーフティ / 心理的安全性
「失敗を恐れずに行動できる環境の性質」。組織心理学の用語だが、開発ツールの設計にも当てはまる重要な概念
throwaway branch

  • 「これ動くかな…」

よくある軽量ブランチの使い方

● spike/xxx — 技術調査用
● experiment/xxx — 実験用
● poc/xxx — 概念実証用
● scratch/xxx — 落書き用

→ どれも捨てる前提の「使い捨て」

"Branching is freedom." — Linus Torvalds 風の格言として広まる言葉

「失敗できる」という贅沢

軽量ブランチがもたらした最大の革命は、技術的なものではなく心理的なものです。具体的に言えば、「失敗を恐れずに試せる」環境の創出です。

SVN 時代、開発者は本流で慎重に作業していました。なぜなら、大きな変更を加えて失敗した時、取り消すコストが大きすぎるからです。「あれ、これダメだった…どこから戻せばいいんだ」が、現実の悪夢でした。

Git では違います。git switch -c experiment で枝を切り、好きなだけ実験する。うまくいったら main にマージ、ダメなら git branch -D experiment で枝ごと削除。「失敗が消える」のです。これは思考の自由を解放しました。

心理的安全性という概念

組織心理学の世界に、「心理的安全性(psychological safety)」という概念があります。これは、ハーバード大学のエイミー・エドモンドソン教授が提唱した、「失敗や異論を表明しても安全だと感じられる環境の性質」です。チームの生産性に決定的な影響を与える要素として、近年注目されています。

軽量ブランチは、技術によって心理的安全性を提供する仕組みだと言えます。「失敗できる、捨てられる、やり直せる」——この三拍子が揃っていることが、開発者の創造性に直接影響します。Git 以前の世界では、開発者は無意識にリスク回避的になっていた。Git は、その重しを取り除いたのです。

「ブランチを切るコストがゼロに近づくと、思考のコストもゼロに近づく」 — Git 時代の開発文化を表す典型的な感想

使い捨てブランチの文化

軽量ブランチが定着すると、面白い文化が生まれました。「使い捨てブランチ」です。spike(スパイク = 短い調査)、experiment(実験)、poc(proof of concept = 概念実証)、scratch(落書き)——どれも「作っては捨てる」前提のブランチに付ける慣習的な名前です。

これらのブランチは、本流に統合される予定がありません。「思考のメモ」のような存在です。何かを試してみる。動くか確認する。動かないなら、何が悪かったか考える。納得したら、ブランチごと削除する——これは「コードを書きながら考える」新しいスタイルの誕生でした。

リファクタリングという文化

もう一つ、軽量ブランチが解放した重要な営みがリファクタリングです。リファクタリングは「動作を変えずにコードの構造を改善する」作業ですが、これには大きな勇気が要ります。「改善のつもりが、壊してしまったらどうしよう」という不安が付き纏うからです。

軽量ブランチがあれば、この不安は劇的に軽減されます。refactor/restructure-auth ブランチを切り、自由に大改造する。テストが全部通れば、main にマージ。通らなければ、改良するか、ブランチごと捨てる。「壊しても元に戻せる」という安心感が、大規模な改善を可能にします。

Martin Fowler の名著 "Refactoring"(1999年)が広く読まれるようになったのも、Git の普及と無縁ではありません。道具が文化を支え、文化が道具を活かす——この循環は、ここでも見られます。

並行思考の解放

もう一つ、軽量ブランチが可能にした重要なことが、「複数のアイデアを並行に試す」習慣です。1つの問題に対して、解法 A と解法 B と解法 C を思いついた時、SVN 時代なら「どれか1つだけ選んで進める」しかなかった。

Git では、3つのブランチを切って、それぞれで試す。全部試してから比較し、最良のものを採用する。これは科学的な仮説検証に似ています。「コードによる思考実験」と言ってもいい。Git は、この並行思考を技術的に支えるインフラを提供したのです。

「ブランチ = 自由」というスローガン

本巻全体を貫く問いは、「ブランチがなぜ Git を革命的にしたのか」でした。答えは技術的なものを超えて、哲学的です。

軽量ブランチは、開発者から「変化することへの恐れ」を取り除きました。失敗を恐れずに、大胆な変更を試みる。複数の方向性を並行に探る。気になることがあれば、すぐに実験する。納得できなければ、迷わず捨てる——これらすべてが、ブランチが軽くなったことで可能になりました。

Git が生んだのは、単なる版管理ツールではなく、「変化に対する自由」を支えるインフラだったのです。あなたが今、何の躊躇もなく git switch -c と打てる、その指の動きの中に、20年に渡る思考の自由化の歴史が宿っているのです。

∴ FINAL THOUGHT

軽量ブランチが解放したのは、速度ではなく自由です。失敗できる自由、捨てられる自由、やり直せる自由——これらの自由が、現代の開発文化の前提を形作りました。道具の革新は、しばしば心の革新でもあるのです。
