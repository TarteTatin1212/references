01

GitHub Actionsとは

WHAT IS ACTIONS

GitHub Actionsは「イベントをきっかけに、何かを自動実行する」仕組み。 pushされたらテスト、PR作成されたらlint、タグが切られたらリリース—— こうした「人間がやらなくていい作業」をGitHubが代わりにやってくれるのがActionsの本質です。

github actions overview ★★★ 自動化の入口
// 動作イメージ
[ 開発者 ] git push ↓ [ GitHub ] push イベント発生 ↓ [ Actions が反応 ] ワークフローを起動 ↓ [ Runner (仮想マシン) ] 1. リポジトリをチェックアウト 2. 依存関係インストール 3. テスト実行 4. ビルド 5. 結果を通知 ↓ [ PR画面 ] ✓ All checks passed
できること(代表例)

◯ push時に自動テスト
◯ PRに自動lint/format
◯ タグでリリース自動化
◯ 定期実行(cron)
◯ Slack通知・Issue自動作成
◯ デプロイ(AWS/Vercel等)

# --- 料金 (2026年5月時点) --- パブリックリポジトリ: 完全無料 プライベートリポジトリ: Free: 月2,000分まで無料 Pro: 月3,000分 Team: 月3,000分 Enterprise: 月50,000分 # 個人開発なら気にしなくてOK! # --- 用語の整理 --- Workflow 自動化の全体定義(YAML 1ファイル) Event ワークフローを起動するきっかけ Job Workflow内の実行単位 Step Job内の個別コマンド Runner 実行する仮想マシン Action 再利用可能な部品 # --- 置き場所 --- my-repo/ └─ .github/ └─ workflows/ ├─ ci.yml ← これらが ├─ deploy.yml 全部読まれる └─ release.yml # 拡張子は .yml or .yaml
POINT —

Actionsは「マーケットプレイス」から既製のActionを取り入れて、ほぼノーコードで強力な自動化が組めるのが魅力。actions/checkout や actions/setup-node など、定番の部品を組み合わせるだけで多くのことができます。

02

ワークフローの基本構造

WORKFLOW STRUCTURE

ワークフローはYAMLファイルで書きます。 構造はname / on / jobs の3階層。 一度覚えると、どのワークフローも同じパターンで読み解けます。

.github/workflows/ci.yml ★★★ YAML構造の理解
// 階層イメージ
Workflow (ファイル) ├─ name : ワークフロー名 ├─ on : トリガー └─ jobs ├─ build (Job 1) │ ├─ runs-on : 実行環境 │ └─ steps │ ├─ Step 1 │ ├─ Step 2 │ └─ Step 3 └─ test (Job 2) ├─ runs-on └─ steps └─ ...
YAML 書き方の鉄則

◯ インデントはスペース2個(タブ禁止)
◯ key: value の形
◯ リストは - で始める
◯ コメントは #

注意 —

YAMLはインデントズレでよく死ぬ。エディタの「YAML対応」を有効に。

name: CI on: push: branches: [main] pull_request: branches: [main] jobs: build: runs-on: ubuntu-latest steps: - name: Checkout uses: actions/checkout@v4 - name: Setup Node.js uses: actions/setup-node@v4 with: node-version: '20' - name: Install run: npm ci - name: Test run: npm test # --- 各セクションの意味 --- name: Actions画面に表示される名前 on: 何をきっかけに起動するか jobs: 実際の処理を書く runs-on: 実行する仮想マシンの種類 steps: ジョブ内で順に実行する処理 uses: 既製のActionを呼ぶ run: シェルコマンドを実行
TIP —

YAMLが苦手ならGitHubの「Actions」タブからテンプレートを選ぶのが最速。「New workflow」で言語別のテンプレが大量に出てきます。最初はそこをベースに、必要に応じて削ったり足したりで十分。ゼロから書くのは玄人になってからでOK。

03

最初のCI設定

FIRST CI SETUP

実際に最初のCIを動かしてみます。 やることは「ファイルを置いてpushするだけ」。 push後、リポジトリのActionsタブを見ると、走っているワークフローが見られます。

first ci workflow ★★★ Hello CI
// 実行結果のイメージ
✓ build — ubuntu-latest 42s
✓ Checkout 3s
✓ Setup Node.js 5s
✓ Install 18s
✓ Test 16s
POINT —

PR画面の下部にも緑のチェックが表示されます。これが全部緑にならないとマージしないのが現代の流儀。

# --- ステップ1: ファイル作成 --- $ mkdir -p .github/workflows $ touch .github/workflows/ci.yml # --- ステップ2: 中身を書く --- name: CI on: [push, pull_request] jobs: hello: runs-on: ubuntu-latest steps: - run: echo "Hello, Actions!" # --- ステップ3: コミット & push --- $ git add .github/workflows/ci.yml $ git commit -m "ci: 初回ワークフロー追加" $ git push # --- ステップ4: 確認 --- GitHub Repository → Actions タブ → 走っているワークフローを確認 クリックするとログが見られる: Run echo "Hello, Actions!" Hello, Actions! # これだけ! 最初の自動化完成! # --- バッジを README に貼る --- README.md に↓を貼ると CIステータスが表示される: ![CI](https://github.com/USER/REPO/actions/workflows/ci.yml/badge.svg)
TIP —

READMEに「ビルドステータスバッジ」を貼っておくと、リポジトリの第一印象が一気にプロっぽくなります。緑のpassingバッジを維持するのは、エンジニアとしてのこだわりの一つ。

04

トリガーの種類

EVENT TRIGGERS

on: で指定する「何をきっかけに動かすか」には、たくさんの種類があります。 push / pull_request / schedule / workflow_dispatch —— 用途に応じて適切に絞り込むのが、無駄な実行を減らすコツ。

on: ... ★★ トリガー一覧
// 主なトリガー
トリガー意味
pushブランチへpush時
pull_requestPR作成・更新時
schedule定期実行(cron)
workflow_dispatch手動実行
releaseリリース作成時
issuesIssue作成時等
workflow_run他のWFが完了時
あるある —

絞り込みなしの on: push だと全ブランチ全pushで動く。無料枠を一気に消費するので注意!

# --- 単純な書き方 --- on: push # --- 複数イベント --- on: [push, pull_request] # --- 詳細指定 (一番よく使う) --- on: push: branches: [main, develop] paths: - 'src/**' - 'package.json' pull_request: branches: [main] paths-ignore: - 'docs/**' - '**.md' # --- 定期実行 (cron形式) --- on: schedule: - cron: '0 9 * * 1-5' # 平日9時(UTC)に実行 = 日本時間18時 # --- 手動実行 (神機能) --- on: workflow_dispatch: inputs: environment: type: choice options: [staging, production] # Actions画面に「Run workflow」ボタンが出る # --- タグpush時 (リリース用) --- on: push: tags: ['v*'] # v1.0.0 のようなタグで起動
鉄則 —

「実行しなくていいものは実行しない」が無料枠節約のコツ。docs更新だけのpushでテストを回す意味はありません。paths-ignore で '**.md' を除外、branches を必要なものに絞る——この2つだけでも実行回数を大幅削減できます。

05

ジョブとステップ

JOBS & STEPS

1つのワークフローには複数ジョブを並べられます。 ジョブ間はデフォルトで並列実行、needs: で依存関係を作れば直列。 Matrix機能を使えば「複数Node.jsバージョン × 複数OS」みたいな組合せテストも一発。

jobs & steps ★★ 並列実行の構造
// 並列と直列
[ 並列実行 (デフォルト) ] lint ─┐ │ test ─┼─→ 全部終わったら完了 │ build ─┘ [ 直列実行 (needs: 指定) ] lint ─→ test ─→ build ─→ deploy [ Matrix (組み合わせ) ] Node 18 ↗ test → Node 20 ↘ Node 22 → 1ジョブで3並列実行
POINT —

ジョブごとに別の仮想マシンが立ち上がる。ファイル共有は artifacts 経由。

# === 並列ジョブの例 === jobs: lint: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm run lint test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm test # 上の2つは同時に走る # === 直列ジョブ === jobs: test: runs-on: ubuntu-latest steps: [...] deploy: needs: test runs-on: ubuntu-latest if: github.ref == 'refs/heads/main' steps: [...] # test成功 + mainブランチ時のみdeploy # === Matrix (神機能) === jobs: test: runs-on: ${{ matrix.os }} strategy: matrix: os: [ubuntu-latest, macos-latest] node: [18, 20, 22] steps: - uses: actions/setup-node@v4 with: node-version: ${{ matrix.node }} - run: npm test # → 2 OS × 3 Node = 6並列実行
裏技 —

if: github.ref == 'refs/heads/main' のような条件式を使えば、ジョブやステップを「mainブランチの時だけ実行」のように制御できます。ステージング・本番でジョブを分ける時に必須の機能。

06

シークレット管理

SECRETS

APIキー、デプロイ用トークン、DBパスワード——外部に漏れたら死亡の情報は絶対にYAMLに直書きしない。 GitHubには専用のSecrets機能があります。 暗号化されて保存され、ワークフロー内では${{ secrets.NAME }} で参照できます。

github secrets ★★★ 機密情報の安全管理
// 設定の流れ
設定手順

① リポジトリ → Settings
② 左メニュー Secrets and variables → Actions
③ New repository secret
④ Name: AWS_ACCESS_KEY
⑤ Value: 機密情報を貼る
⑥ Add secret

Scope (適用範囲)

◯ Repository secrets: そのリポジトリ専用
◯ Environment secrets: 環境(staging/prod)別
◯ Organization secrets: 組織全体共通

注意 —

Secretsはログにも「***」と伏字になる。安心して使えます。

# --- 悪い例: 直書き --- env: API_KEY: "sk_live_xyz123..." # ✗ コードに残る = リポジトリに残る = 漏れる # --- 良い例: Secrets --- env: API_KEY: ${{ secrets.API_KEY }} # ◯ 暗号化された値が読み込まれる # --- 使用例 --- name: Deploy on: push: branches: [main] jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: AWS設定 uses: aws-actions/configure-aws-credentials@v4 with: aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY }} aws-secret-access-key: ${{ secrets.AWS_SECRET_KEY }} aws-region: ap-northeast-1 - name: デプロイ run: aws s3 sync ./dist s3://my-bucket # --- 漏れたらどうする? --- 1. 該当キーを即座にローテート 2. Settings → Secrets → 該当を更新 3. 関連サービスでも無効化 4. git log でリポジトリ内も確認
鉄則 —

機密情報をうっかりコミットしてpushした場合、リポジトリから消しても履歴に残るので必ず漏れます。即座にそのキーを無効化(ローテート)するのが正解。git rebaseで歴史を書き換えても、もう手遅れ。「直書きしない」が最強の防衛策です。

07

実用CI/CD例

REAL EXAMPLES

現場でよく使われる実用パターンを3つ紹介します。 Node.jsアプリのテスト、Pythonの自動lint、タグでのリリース自動化。 これらをコピペベースで改造すれば、たいていの現場で通用します。

practical workflows ★★★ 実戦テンプレ集
// 用途別の選び方
パターン①: Node.js テスト

PR時に lint + test を回す、超定番。React/Vue/Express何でも使い回せる。

パターン②: Python lint

ruff/blackで自動フォーマット確認。Pythonプロジェクトのお約束。

パターン③: タグでリリース

v1.0.0 のようなタグをpushすると、自動でGitHub Releaseを作成。

TIP —

本物のプロジェクトの .github/workflows/ を覗いてみるのが最強の教科書。OSSはほぼ全部見られます。

# === ① Node.js CI === name: CI on: pull_request: branches: [main] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' - run: npm ci - run: npm run lint - run: npm test - run: npm run build # === ② Python lint === name: Lint on: [push, pull_request] jobs: lint: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: '3.12' - run: pip install ruff black - run: ruff check . - run: black --check . # === ③ タグでリリース === name: Release on: push: tags: ['v*'] jobs: release: runs-on: ubuntu-latest permissions: contents: write steps: - uses: actions/checkout@v4 - name: Create Release uses: softprops/action-gh-release@v2 with: generate_release_notes: true # git tag v1.0.0 && git push --tags # これだけで Release ページに自動追加!
POINT —

cache: 'npm' のようなキャッシュ機能を必ず使うこと。これだけで2回目以降の実行が劇的に高速化します。npm install が30秒→3秒に。地味だけど効果絶大。

08

失敗時のデバッグ

DEBUGGING

Actionsはローカルで再現しづらく、失敗時のデバッグが地味に難しい。 ログを開いて、原因の切り分け方を知っておくのが重要。 定番のデバッグ手順と、ローカルで再現するためのactというツールを紹介します。

debug actions ★★ 詰まった時の救済
// よくある失敗パターン
① YAML構文エラー

原因: インデントズレ、コロン抜け
確認: Actionsタブで赤い「invalid workflow」

② 依存関係エラー

原因: package-lock.jsonが古い
解決: ローカルで npm install して再push

③ Secretsが読めない

原因: 名前ミス、scope違い
解決: Settings → Secrets を再確認

④ 権限不足

原因: permissions指定漏れ
解決: permissions: を追加

# === ログを開く === Actions タブ → 失敗したワークフロー # → 赤いジョブをクリック # → 失敗ステップを展開 # → エラーメッセージを読む # === デバッグ用フラグ === # Secrets に追加するとログが詳細化: ACTIONS_RUNNER_DEBUG = true ACTIONS_STEP_DEBUG = true # 値はどちらも "true" # === デバッグ用 echo === steps: - name: Debug run: | echo "Branch: $GITHUB_REF" echo "Actor: $GITHUB_ACTOR" echo "PWD: $(pwd)" env ls -la # === Re-run (再実行) === 失敗画面右上の "Re-run jobs": - Re-run all jobs - Re-run failed jobs - Re-run with debug logging ←神機能 # === ローカル実行: act === $ brew install act # または scoop / apt $ act # .github/workflows を Docker で実行! # pushしなくてもCI動作を試せる # 特定のジョブだけ $ act -j test # --- 公式ドキュメントが詳しい --- # docs.github.com/actions
最後に —

Actionsは「最初は誰でもハマる」世界。YAMLの細かい構文、コンテキスト変数、permissionsの仕様——一気に覚えようとせず、動くワークフローをコピペして少しずつ改造するのが王道です。GitHub公式ドキュメントも非常に充実しているので、迷ったら必ずそちらへ。
