Q.01

編集をやっぱり全部やめたい。コミット前なら戻せる?

RESTORE
ANSWER

コミット前なら戻せます。状況によって2パターン:

① まだ git add していない時

git restore <file> または git restore . (全部)

② git add 済みの時

先に git restore --staged <file> でステージから降ろし、続けて git restore <file>

✗ NG: 何も考えず rm

新規ファイルが惜しい時にこれをやると復元不能。

◯ OK: restore

追跡中のファイルを安全に戻せる。

TIP —

未追跡ファイル(??表示)は restoreで消えない。git clean -fd で削除可能。こちらは復元不能なので慎重に。

# --- ケース①: addしてない時 --- $ git status Changes not staged for commit: modified: src/app.js modified: README.md # 全部捨てる $ git restore . # 特定だけ $ git restore src/app.js # --- ケース②: addしちゃった時 --- $ git status Changes to be committed: modified: src/app.js # まずステージから降ろす $ git restore --staged src/app.js # それから変更も戻す $ git restore src/app.js # --- 未追跡ファイルを消したい時 --- $ git clean -fd ↑ 警告なしで削除 事前に -n でドライラン推奨: $ git clean -fdn # 何が消えるか先に確認
Q.02

間違った内容でコミットしちゃった。取り消したい!

RESET / AMEND
ANSWER

まだpushしていなければ、自由に取り消せます。やりたいことで使い分け:

① 直前のコミットを修正したい

git commit --amend — メッセージ修正、ファイル追加が可能

② コミットだけ取り消したい(変更は残す)

git reset --soft HEAD~1

③ コミットも変更も全部消したい

git reset --hard HEAD~1 ⚠ 慎重に

注意 —

push済みのコミットを reset で巻き戻すと、後でpushに --force が必要になり、チーム事故の元。push済みなら git revert を使う(Q.03)。

# --- ① 直前を修正 --- # メッセージだけ書き直し $ git commit --amend -m "feat: 正しい説明" # ファイル追加忘れたので追加 $ git add forgotten.js $ git commit --amend --no-edit ↑ メッセージ変えない # --- ② コミットだけ取消(変更残す) --- $ git reset --soft HEAD~1 $ git status Changes to be committed: modified: ... # 変更は残ってる → やり直しできる # --- ③ 完全に消す --- $ git reset --hard HEAD~1 # 慎重に! ファイル変更も消える # --- 複数コミットまとめて取消 --- $ git reset --hard HEAD~3 ↑ 3つ消す # --- やってしまった→Q.06 で救出 ---
Q.03

push済みのコミットに間違いがある!どうすれば?

REVERT
ANSWER

共有済みなら絶対に git revert。reset --hardしてpush --forceは他人のローカルを壊す世紀の事故になります。

revertの仕組み

過去のコミットを「打ち消す内容の新しいコミット」を作る。履歴は改変されないので安全。

✗ 絶対NG (チーム事故)

git reset --hard HEAD~1
git push --force

◯ 正解

git revert HEAD
git push

鉄則 —

「pushしたら過去は変えない」。これがチーム開発の絶対法則。例外なし。

# --- 直前のpush済みコミットを取消 --- $ git revert HEAD # エディタ開く → 保存 → 新コミット作成 [main b7e5a92] Revert "feat: ..." $ git push # --- 特定コミットを取消 --- $ git log --oneline a3f2c1d 良いコミット 9b8e4a2 問題のコミット ← これを消したい 5d6f3a8 その他 $ git revert 9b8e4a2 # --- マージコミットを取消 --- $ git revert -m 1 <merge-commit> ↑ -m 1 = mainを残す # --- 複数まとめて --- $ git revert a3f2c1d^..9b8e4a2 # --- revertコミット自体をrevert --- # 「取消を取消」も可能。元の変更が復活 $ git revert <revert-commit-hash>
Q.04

.env をうっかりコミット&pushしてしまった!

SECURITY ⚠
ANSWER

業界全体の悲劇です。順序が超重要:

緊急対応の手順

① 該当APIキーを即座に無効化(ローテート)
② .gitignore に .env を追加
③ git rm --cached .env でGit管理から除外
④ コミット&push
⑤ 必要なら git filter-repo で履歴から完全削除

最重要 —

履歴から消しても「漏れた事実」は消えない。GitHubは多数のbotが常時スキャンしており、push直後に外部に複製される可能性も。「キーローテートが最優先」これを忘れずに。

# === ① キーを即座に無効化 === # AWS / Stripe / Firebase 等の # 管理画面で該当キーをローテート # これが最優先!! # === ② .gitignore 更新 === $ echo ".env" >> .gitignore # === ③ Git管理から除外 === $ git rm --cached .env $ git commit -m "chore: .env を追跡解除" $ git push # これで「現在」からは消える # しかし「過去の履歴」には残っている! # === ④ 履歴からも完全削除 === # git filter-repo (推奨ツール) $ pip install git-filter-repo $ git filter-repo --path .env --invert-paths # 履歴が書き換わる → 強制push必要 $ git push --force # --- チーム共有リポジトリでは要相談 --- # 履歴書き換えはチームの同意必須 # --- 再発防止 --- # .env.example だけコミット # pre-commit hookで検出する
Q.05

気付いたら間違ったブランチで作業してた!

STASH / CHERRY-PICK
ANSWER

よくあるやつ。状況で対応が変わります:

① まだコミットしていない

git stash で退避 → 正しいブランチに switch → git stash pop

② すでにコミットしてしまった

正しいブランチに cherry-pick で移植 → 元ブランチで reset --hard HEAD~1

POINT —

業務開発で一番頻繁にやらかすミス。状況の判断ができれば慌てる必要なし。「stashとcherry-pickの使い方」が分かっていれば余裕。

# === ケース①: 未コミットの時 === $ git status On branch main ← あれ、mainで作業してた! modified: src/app.js # 退避 $ git stash # 正しいブランチへ $ git switch feature/x # 取り出す $ git stash pop # → 復活! 作業続行 # === ケース②: コミット済 === # 状態確認 $ git log --oneline a3f2c1d あ、これmainに乗ってる! 9b8e4a2 既にあるコミット $ git branch * main ← ここに乗ってる feature/x ← ここに移したい # ① 正しいブランチへ移動 $ git switch feature/x # ② コミットを取り込む $ git cherry-pick a3f2c1d # ③ 元ブランチから消す $ git switch main $ git reset --hard HEAD~1 # まだpush前なら ↑ で安全 # push済みなら revert を使う(Q.03)
Q.06

reset --hard でコミット消した!何とかして!

REFLOG ⚠
ANSWER

慌てるな。90%以上のケースで救出可能。Gitの最終救済策git reflogを使います。

救出の3ステップ

① git reflog で操作履歴を確認
② 戻したいコミットのハッシュをコピー
③ git reset --hard <hash> で復活

reflogが救えるもの

◯ reset --hard で消した
◯ rebase で書き換えた
◯ branch -D で消した
◯ commit --amend の前の状態
✗ 未コミットの変更だけは救えない

保持期間 —

reflogはデフォルト90日残る。それまでなら冷静に対処可能。

# === 救出の流れ === # ① reflog で操作履歴を見る $ git reflog a3f2c1d HEAD@{0}: reset: --hard HEAD~3 9b8e4a2 HEAD@{1}: commit: feat: 大事な機能 5d6f3a8 HEAD@{2}: commit: fix: 修正 2c1b0e9 HEAD@{3}: commit: 序章 ↑ ここまで戻したい! # ② 戻す $ git reset --hard 9b8e4a2 HEAD is now at 9b8e4a2 feat: 大事な機能 # 復活!! # === 別ブランチに保管する派 === $ git branch recovered 9b8e4a2 失ったコミットを新ブランチに保存 # 後で確認してからmergeすれば安全 # === 削除済みブランチの復活 === $ git reflog --all | grep "feature/x" a3f2c1d ... checkout: moving from feature/x $ git switch -c feature/x a3f2c1d # ブランチ復元成功 # === 時間指定でも探せる === $ git reflog "HEAD@{1 hour ago}" $ git reflog "HEAD@{yesterday}"
Q.07

マージしたらコンフリクトで真っ赤!どうすればいい?

CONFLICT
ANSWER

コンフリクトは敵じゃなくGitからの助け要請。手順は決まっています:

解消の5ステップ

① git status で衝突ファイル特定
② エディタで開く
③ <<< === >>> マーカーを見つけて、正しい形に書き換え
④ マーカー行も削除
⑤ git add → git commit

諦めて中止したい

git merge --abort でマージ前の状態に戻る。

TIP —

VS Code等のエディタにはコンフリクト解消用UIがある。「Accept Current / Accept Incoming / Accept Both」ボタンが出る。マウスで選ぶだけ。

# --- マージしたらコンフリクト --- $ git merge feature/x Auto-merging src/app.js CONFLICT (content): Merge conflict in src/app.js Automatic merge failed; fix conflicts and then commit the result. # ① どこが衝突したか確認 $ git status Unmerged paths: both modified: src/app.js # ② ファイルを開く function greet() { <<<<<<< HEAD return "Hello, World"; ======= return "Hi, World"; >>>>>>> feature/x } # ③ 正しい形に編集 (マーカーも消す) function greet() { return "Hello, World"; // 採用 } # ④ 解決を宣言 $ git add src/app.js $ git commit # マージコミットメッセージは自動生成 # --- 諦める --- $ git merge --abort # マージ前の状態に戻る # --- 片方をまるごと採用 --- $ git checkout --ours src/app.js $ git checkout --theirs src/app.js # rebase中なら $ git rebase --continue $ git rebase --abort
Q.08

git push したら拒否された!エラーがよく分からない

PUSH ERROR
ANSWER

pushエラーは4パターン。エラーメッセージで切り分けます:

① non-fast-forward

リモートに新コミットあり → git pull してから再push

② Authentication failed

パスワード認証は廃止 → PAT または SSH鍵を使う(Vol.4)

③ src refspec does not match

そのブランチが存在しない、コミット未作成。git logとgit branchで確認

④ Permission denied / GH006

書き込み権限がない、またはブランチ保護に弾かれている

鉄則 —

エラーメッセージのhint: から始まる行に解決策が書いてあることが多い。まずそこを読む。

# === ① non-fast-forward === $ git push ! [rejected] main -> main (non-fast-forward) hint: Updates were rejected because the hint: remote contains work that you do not hint: have locally. # → pull してから push $ git pull $ git push # === ② 認証エラー === remote: Support for password authentication was removed on August 13, 2021. fatal: Authentication failed # → PAT or SSH鍵を使う (Vol.4) # === ③ refspec エラー === error: src refspec main does not match any # 確認 $ git log # コミットある? fatal: your current branch ... # → まだ1つもコミットしてない $ git commit -m "init" $ git push -u origin main # === ④ 権限・保護エラー === remote: error: GH006: Protected branch update failed # → ブランチ保護に弾かれている PR経由でマージするのが正解 # === 切り分けの定番 === $ git remote -v # URL確認 $ git branch -vv # 追跡確認 $ ssh -T git@github.com # 接続確認
Q.09

「detached HEAD」って出てきた!怖い!

DETACHED HEAD
ANSWER

怖くない。ただの「ブランチに紐づかない場所にいる」状態です。

発生する原因

ブランチ名でなくコミットハッシュで直接checkoutすると発生。
例: git checkout a3f2c1d

解決法

① 何もせず帰る場合: git switch <ブランチ名>
② この場所からブランチを切る場合: git switch -c new-branch

POINT —

detached HEAD状態でコミットを作っても、ブランチで保護されていない「迷子のコミット」になります。Gitに自動回収されると消えるので、残したいなら必ずブランチを切ること。

# --- 状況確認 --- $ git status HEAD detached at a3f2c1d ↑ ブランチでなくハッシュを指している nothing to commit, working tree clean # --- 抜け出す --- # ① そのまま帰る (何も保存しない) $ git switch main Previous HEAD position was a3f2c1d ... Switched to branch 'main' # ② この場所からブランチを作る $ git switch -c my-investigation Switched to a new branch 'my-investigation' # --- detached HEADで誤ってコミット --- $ git commit -m "何かした" [detached HEAD b7e5a92] 何かした # 慌ててブランチ化! $ git switch -c save-my-work # これで安全 # --- うっかり消えても reflog で救える --- $ git reflog b7e5a92 HEAD@{0}: commit: 何かした a3f2c1d HEAD@{1}: checkout: ... $ git switch -c recovered b7e5a92
Q.10

リモートとローカルが滅茶苦茶で、もう諦めたい…

RESET TO REMOTE
ANSWER

「もうリモートを正として、ローカルをリモートに完全一致させる」のが最強の脱出技。

最終手段の手順

① git fetch origin
② git reset --hard origin/main(リモート完全一致)
③ git clean -fd(余計なファイル削除)

それでもダメな時の最終奥義

別フォルダにクローンし直し: cd .. && git clone ...。
手元の作業はバックアップしてから。

最後に —

Gitは「壊れたら再クローンすれば良い」が究極の安心感。リポジトリの本体はリモートにあるので、ローカルは何度でも作り直せます。潔く諦めるのも技術。

# === 強制的にリモートに揃える === # ① 最新を取得 $ git fetch origin # ② ローカルmainをリモートに完全一致 $ git switch main $ git reset --hard origin/main # ③ 追跡されてないファイルも消す $ git clean -fd # これでクリーンスタート! # === 残したい変更がある場合 === # まず退避してから $ git stash push -u -m "念のため退避" # または別ブランチで保管 $ git switch -c backup-current $ git add -A $ git commit -m "WIP backup" $ git switch main $ git reset --hard origin/main # === 最終奥義: 再クローン === $ cd .. $ mv my-project my-project-broken $ git clone <url> my-project $ cd my-project # 古いフォルダから必要なものだけ # 手動コピー → 落ち着いて作業 # === 大事なこと === # Gitは何度でもやり直せる。 # 焦るな、落ち着いて、まずは status。
