01

Pull Requestとは

PULL REQUEST

通称「PR(プルリク)」。ある作業ブランチの変更を、「mainに取り込んでください」とお願いする仕組みです。 単なるマージではなく、レビュー・議論・CI実行を経て統合するための一連のワークフロー。 現代のGitHub開発では、これがあらゆる作業の中心になります。

pull request workflow ★★★ チーム開発の中心
// PR の全体フロー
[ 開発者 ] 1. ブランチ切る → コミット → push [ GitHub ] 2. PR 作成 [ チーム ] 3. レビュー (コメント・修正依頼) [ 開発者 ] 4. 指摘反映 → 追加push [ CI/CD ] 5. テスト・lint 自動実行 [ 承認後 ] 6. マージ → ブランチ削除 [ 完了 ] 7. main が更新される
PRがあるとできること

◯ 変更内容を一覧で見られる
◯ 行ごとにコメントが付く
◯ 議論履歴が残る
◯ CIで自動チェック
◯ 承認制で品質を担保
◯ Issueと紐付け可能

# --- PR完成形のイメージ --- PR #42 feat: ログイン機能を実装 ┌─ 変更ファイル ─────────────┐ src/login.js +85 src/auth.js +23 tests/login.spec +40 └──────────────────────────┘ ┌─ チェック結果 ─────────────┐ ✓ ESLint passed ✓ Unit tests passed (52/52) ✓ Build successful └──────────────────────────┘ ┌─ レビュー ──────────────────┐ ✓ @hanako approved 🗨 @taro commented on line 23 └──────────────────────────┘ ┌─ ブランチ ──────────────────┐ feature/login → main └──────────────────────────┘ # これが「PR」の正体
POINT —

PRは単なるマージ機能ではなく「議論の場」。コードの良し悪しだけでなく、設計の妥当性・命名・テスト方針までをチーム全体で議論する場です。これを通じてチームの技術レベルが上がっていきます。

02

PRの作成手順

CREATE PR

実際にPRを作る流れを順を追って見ていきます。 「ブランチ push → GitHubで作成」の2ステップが基本。 PRのタイトルと説明文は、レビュアーへの最初の印象を決める重要なポイントです。

create pull request ★★★ PR作成の流れ
// PR画面のイメージ
● Open #42
feat: ログイン機能を実装
taro wants to merge 5 commits into main from feature/login

## 概要
ログインフォームと認証ロジックを実装。

## 変更点
- ログインフォームUI追加
- JWTトークン発行ロジック
- バリデーション処理

## テスト
- 単体テスト追加(52件全パス)

## 関連Issue
Closes #38

定番テンプレ —

「概要 / 変更点 / テスト / 関連Issue」の4つを書けばレビュアーが助かります。

# --- ① ブランチで作業 --- $ git switch -c feature/login # ファイル編集 → コミット # --- ② リモートに push --- $ git push -u origin feature/login Branch 'feature/login' set up to track... remote: remote: Create a pull request for 'feature/login': remote: https://github.com/team/app/pull/new/feature/login remote: # ↑ GitHubがPR作成URLを教えてくれる # --- ③ ブラウザでPR作成 --- GitHubページに行くと: 「Compare & pull request」ボタン が黄色で表示されている # --- ④ 入力項目 --- タイトル: feat: ログイン機能を実装 本文: (テンプレに沿って) レビュアー: @hanako @ichiro 担当者: @taro ラベル: enhancement, frontend マイルストン: v1.2 # --- ⑤ Draft PR --- 完成前でも「Draft」状態で作れる → レビュー対象外、CIだけ動かす → 進捗共有に便利
TIP —

PRのタイトルは「動詞: 何を」の形が定番(コミットメッセージと同じ)。「fix: 404エラーを修正」「feat: パスワード強度メーター追加」など。大量のPRが流れる現場では、タイトルだけで内容が分かるかが死活問題です。

03

レビューを受ける

RECEIVING REVIEW

PRを出したら「レビューを受ける側」になります。指摘をどう受け止め、どう反映するか—— ここでチームの一員としての姿勢が試されます。反映の作法を押さえておきましょう。

handle review ★★★ 指摘への対応
// 指摘への対応パターン
① 修正が必要 → 反映してpush

「ありがとうございます、修正しました」+ Resolve conversation ボタンで解決マーク。

② 同意できない → 議論する

「この方が○○の理由で良いと考えました。いかがでしょうか?」と理由を添えて返信。

③ あとで対応 → Issue化

本PRのスコープ外なら「別Issue化します」と返して切り出す。

鉄則 —

レビューは「あなた個人への攻撃」ではない。コードに対する意見。感情で返さず、技術で返す。

# --- 指摘を反映する流れ --- # ① レビューコメントを読む "ここはnullチェックがないので エラーになりそうです" - @hanako # ② ローカルで修正 $ git switch feature/login # ファイル編集... # ③ コミット (修正であることを明示) $ git add . $ git commit -m "fix: nullチェックを追加" # ④ プッシュ $ git push # → PR画面に自動反映される CIも再実行される # ⑤ GitHubでコメント返信 「ご指摘ありがとうございます、 修正しました(commit a3f2c1d)」 → Resolve conversation で解決 # --- 再レビュー依頼 --- PR画面右側 Reviewers の レビュアー名の横の 🔄 アイコンを クリック → 再レビューがリクエストされる # --- 細かいfixupコミットを整理 --- $ git rebase -i HEAD~5 完成前にコミットを綺麗にまとめると吉 (push --force-with-lease で反映)
あるある —

レビュー指摘の「typoとか細かい指摘でいちいち修正コミットが量産される」問題。これはgit commit --fixup + git rebase -i --autosquash でまとめるか、マージ時に「Squash and merge」(本巻 05)を選べば解決します。

04

レビューする側

DOING REVIEW

レビューする側の作法も重要。 「指摘の質」と「言葉遣い」がチームの空気を決めます。 GitHubのレビュー機能には3段階の判定があり、これも使い分けが大事。

code review ★★★ レビューの作法
// レビュー3つの判定
判定意味
Approve承認 (マージOK)
Request changes修正必須
Commentコメントのみ (賛否なし)
コメントの定番プレフィックス

nits: 些細な指摘 (任意対応)
must: 必須対応
imo: 個人的意見
q: 質問
praise: 良いと思った箇所

鉄則 —

レビューは「人ではなくコードに対して」。「○○さんのコード変」ではなく「この実装は△△の理由で改善できそう」。

# --- 良いレビュー例 --- // must: ここはnullチェックが必要です。 // user が undefined の時に落ちます。 // nits: 変数名を userName に揃えると // 他のメソッドと統一されて読みやすいです // q: ここで try-catch しているのは // 何か想定エラーがありますか? // praise: この設計とても良いですね! // 後で参考にさせてください # --- 避けるべきレビュー --- // これはダメです。 // 「何が」「なぜ」が抜けてる // こんなコード書く人初めて見た // 人格攻撃NG // 直してください。 // どう直す? 議論の余地は? # --- レビュー操作の手順 --- ① PR画面 → Files changed タブ ② 行をクリックで + ボタン → コメント ③ "Start a review" で複数まとめる ④ "Review changes" で判定送信 ⑤ Approve / Request changes / Comment いずれかを選んで Submit # --- Suggestion 機能 --- 変更案を直接コードで提案できる: ```suggestion const user = data?.user ?? null; ``` PR作成者は1クリックで取り込める!
現場で —

レビューは「翌営業日まで」に応えるのが目安。長く放置するとPRが積もって誰も嬉しくない。「1日に1回はPRを見る時間」を確保する人が、信頼されるエンジニアになります。

05

マージの3パターン

3 MERGE STRATEGIES

GitHubのマージボタンには3つの選択肢があります。 Create a merge commit / Squash and merge / Rebase and merge。 それぞれ履歴の残り方が違い、チーム方針で決まっていることが多い項目です。

merge strategies ★★ マージ方式の選択
// 3方式の比較
方式履歴適性
Merge commitそのまま全部残る記録重視
Squash and merge1コミットに集約クリーン派
Rebase and merge一直線に並べる線形履歴派
[ Merge commit ] main ●─●─●─────● \ / feature ●─● [ Squash and merge ] main ●─●─●─●' ↑1コミットに集約 [ Rebase and merge ] main ●─●─●─●─● ↑featureが一直線に
現場の主流 —

多くの現場でSquash and mergeが定番。PR単位で履歴がスッキリ、レビューしやすい。

# --- ① Create a merge commit --- 最も標準的な方式。 マージコミットが必ず作られる。 main: ●─●─● ●←merge \ / feature: ●─●─●─●─ 長所: いつ何が入ったか明確 短所: 履歴が複雑になりがち # --- ② Squash and merge --- feature の全コミットを1つに集約。 中の細かい履歴は消える。 main: ●─●─●─●' ← featureの全変更が 1コミットに 長所: メインの履歴が綺麗 短所: 細かい開発過程が消える # --- ③ Rebase and merge --- featureを main の先に並べ替えて 一直線にする。 main: ●─●─●─●─●─●─● ↑ feature の各コミット 長所: 履歴が綺麗な一直線 短所: マージした事実が消える # --- どれを選ぶ? --- 迷ったら Squash and merge が無難。 細かい修正コミットを残したくない時に最適。 # 設定で常にSquashを強制 Repo → Settings → General → Allow Squash and merge のみ ON
POINT —

マージ方式は「個人の好み」ではなくチームの規約。配属されたらまず確認しましょう。Squash派が圧倒的多数ですが、Linuxカーネルなど大規模OSSではMerge commit派も。「あえて選ばない」のがプロです。

06

Issueの使い方

ISSUES

Issueは「課題管理ツール」です。バグ報告、機能要望、タスク管理、ディスカッション—— あらゆる「やること・議論すること」をIssueとして起票しておくのが、現代のGitHub開発の流儀。

github issues ★★ プロジェクト管理の中心
// Issue の典型例
● Open #38
[Bug] ログイン後に画面が真っ白になる
Reported by @taro · Assigned to @hanako

## 再現手順
1. ログイン画面で正しい情報を入力
2. ログインボタンを押す
3. 画面が真っ白になる

## 期待する動作
ダッシュボード画面が表示される

## 環境
Chrome 124, macOS 14.4

活用シーン

◯ バグの追跡 (再現手順・環境)
◯ 機能リクエスト
◯ タスク管理 (誰が何をいつまで)
◯ 質問・議論

# --- Issue の使い方 --- ① リポジトリの Issues タブ → New issue ② テンプレが用意されていれば選択 (Bug report / Feature request 等) ③ 必須項目を埋める ④ ラベル・担当者・マイルストンを設定 # --- ラベルの定番 --- bug バグ報告 enhancement 機能改善 documentation ドキュメント good first issue 初心者歓迎 help wanted 助けが欲しい question 質問 # --- PR と Issue の連携 --- PR本文に下記キーワードを書くと マージ時に Issue が自動でクローズされる: Closes #38 Fixes #38 Resolves #38 # --- Issue検索の便利フィルタ --- is:open is:issue label:bug is:closed assignee:@me is:open milestone:"v1.2" is:open no:assignee # --- 高度な機能 --- # GitHub Projects: かんばん化 # Issue Forms: 入力フォームで起票 # Discussions: スレッド型の議論用
鉄則 —

「Issueを書ける人」は重宝されます。再現手順・期待する動作・実際の動作・環境——この4点をきちんと書けるだけで「仕事ができる人」評価。逆に「動きません。直してください」だけのIssueはチームから嫌われます。

07

PR/Issueテンプレート

TEMPLATES

リポジトリにテンプレートファイルを置いておくと、新しいPRやIssueを作る時に自動でひな型が表示されます。 チーム全体で記述粒度を揃えるための重要な仕組み。 特定のフォルダにMarkdownを置くだけ。

.github/templates ★★ 記述ルールの統一
// 配置場所
my-repo/ ├─ .github/ │ ├─ PULL_REQUEST_TEMPLATE.md │ └─ ISSUE_TEMPLATE/ │ ├─ bug_report.md │ ├─ feature_request.md │ └─ config.yml └─ src/ これだけで自動適用! PR作成画面 / Issue作成画面で 自動的にひな型が読み込まれる
効果

◯ 必要情報が漏れない
◯ レビュー効率アップ
◯ 新メンバーが書き方に迷わない
◯ チェックリストで抜け漏れ防止

# === PULL_REQUEST_TEMPLATE.md === ## 概要 <!-- このPRで何を変えたか --> ## 変更点 - - - ## 動作確認 - [ ] ローカルで動作確認済み - [ ] 単体テスト追加 - [ ] レビュアー向け確認手順を記載 ## 関連Issue Closes # ## スクリーンショット (UI変更時) | Before | After | |--------|-------| | | | # === ISSUE_TEMPLATE/bug_report.md === --- name: バグ報告 about: 不具合を報告する labels: bug --- ## 再現手順 1. 2. 3. ## 期待する動作 ## 実際の動作 ## 環境 - OS: - ブラウザ: - アプリバージョン: # この内容で起票画面に自動展開!
TIP —

テンプレートは「最初の数行だけ」でもOK。最初から完璧を目指さず、運用しながら育てるのがコツ。「このリポジトリのPRはこう書いて」とドキュメント代わりにもなる、地味だけど効果絶大の仕組みです。

08

GitHub CLI (gh)

GITHUB CLI

gh はGitHub公式のコマンドラインツール。 ターミナルから直接PR作成・レビュー・Issue操作ができ、ブラウザを開かずに作業が完結します。 使いこなせば作業効率が劇的に上がる、現代エンジニア必携のツール。

gh command ★★ ターミナルが完結環境に
// よく使うghコマンド
用途コマンド
認証gh auth login
PR作成gh pr create
PR一覧gh pr list
PR表示gh pr view
PRチェックアウトgh pr checkout 42
Issue作成gh issue create
リポジトリclonegh repo clone
インストール —

macOS: brew install gh / Windows: scoop or winget / Linux: 各種パッケージ。

# --- 認証 (一度きり) --- $ gh auth login ? What account do you want to log into? > GitHub.com ? What is your preferred protocol? > SSH # 対話形式で設定完了 # --- PR作成 (神機能) --- $ gh pr create ? Title: feat: ログイン機能を実装 ? Body (e to launch editor) ? What's next? Submit Creating pull request for feature/login into main in team/app https://github.com/team/app/pull/42 # ブラウザ不要でPR完成! # --- 短縮形 --- $ gh pr create --fill ↑ コミットメッセージから自動生成 # --- PR一覧 --- $ gh pr list #42 feat: ログイン open taro #41 fix: 404エラー open hana #40 docs: README更新 merged ichiro # --- 他人のPRをローカルで試す --- $ gh pr checkout 42 # そのブランチに切替完了 # --- PRをブラウザで開く --- $ gh pr view --web # --- リポジトリ作成 --- $ gh repo create my-app --public ローカル → リモート作成 → push まで一気に
最後に —

ghを使い始めると「GitHubのWeb画面を開く回数が激減」します。gh pr create --fill で30秒、gh pr checkout で他人のPRも瞬時に試せる。「PR作るの面倒だなあ」と思うことが無くなり、自然にPRを切り出す頻度が増えていきます。
