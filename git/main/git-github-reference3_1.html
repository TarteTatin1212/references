ブランチとマージ編

BRANCHING & MERGING

ブランチはGitの真骨頂です。「機能ごとに分岐して並行開発、出来上がったら統合する」—— この仕組みがあるからこそ、現代の大規模開発が成り立っています。 本巻ではブランチの作り方・切替・統合・削除までを通しで身につけます。

01

ブランチとは何か

WHAT IS A BRANCH

ブランチは「履歴の枝分かれ」です。 本流(main)からひょいと別の道を作り、そこで自由に作業し、終わったら本流に戻して統合する—— この「並行する平行世界」を低コストで作れるのがGitの最大の特徴。

branch concept ★★★ 必修コンセプト
// 履歴の枝分かれイメージ
[feature/login] ●─●─● / ●───●───●───● ← main \ ●─● [feature/signup] 各ブランチが独立した "並行する作業空間" mainに影響を与えずに 新機能の実験・開発ができる
POINT —

ブランチの実体は「あるコミットを指す軽量なポインタ」。コピーじゃないので、何個作っても容量はほぼゼロ。

# 現在のブランチ一覧 $ git branch * main # * 印が現在地 feature/login fix/header-bug # --- なぜブランチを使うのか? --- # × 直接 main で作業する場合 main: 完成 → バグ追加 → 修正 → 完成 途中の壊れた状態が公開される 他の人の作業をブロックする # ◯ ブランチで作業する場合 main: 完成 ───────── マージ → 完成 \ / feature: 新機能を開発 いくら壊しても本流に影響なし # これが現代の開発の常識
鉄則 —

業務開発では「mainに直接コミットしない」のがお約束。何か作業するときは、必ず作業用のブランチを切る。これが事故を防ぐ最大のコツです。

02

ブランチを作る

CREATE BRANCH

新しいブランチを作るのは1コマンドで一瞬です。 名前のつけ方には現場の慣習があり、それに従うとチームに馴染みやすくなります。「種類/内容」の形が定番です。

git branch <name> ★★★ 作業の始まり
// ブランチ名の定番パターン
プレフィックス用途
feature/新機能の追加
fix/ / bugfix/バグ修正
hotfix/緊急の本番修正
refactor/リファクタリング
docs/ドキュメント関連
chore/雑務(依存更新等)
名前の例 ◯

feature/user-login
fix/header-overflow
hotfix/payment-500-error
docs/update-readme

避けたい名前 ✗

test
taro
new-branch
aaa

# 作るだけ (切り替えない) $ git branch feature/login # 一覧確認 $ git branch * main feature/login # 作られた # --- 作って即切替 (定番) --- $ git switch -c feature/login Switched to a new branch 'feature/login' # 古い書き方 (今でも動く) $ git checkout -b feature/login # --- 特定のコミットから分岐 --- $ git switch -c hotfix/bug a3f2c1d ↑ ここから派生 # --- リモートブランチから作る --- $ git switch -c feature/x origin/feature/x
POINT —

ブランチ名に日本語やスペースは入れないのが鉄則。半角英数とハイフン・スラッシュのみ。チームで命名規則が決まっていることも多いので、最初に確認しましょう。

03

ブランチを切り替える

SWITCH BRANCH

ブランチを切り替えると、作業ディレクトリのファイルがそのブランチの内容に丸ごと入れ替わります。 これがGitの「タイムマシン感」を最も実感できる瞬間。 git switch(新)とgit checkout(旧)、どちらも知っておきます。

git switch <name> ★★★ 平行世界への移動
// 切替で起きること
[ main ブランチ ] my-app/ ├─ login.js ├─ signup.js └─ README.md (古い版) ↓ git switch feature/login [ feature/login ブランチ ] my-app/ ├─ login.js (修正された版) ├─ signup.js ├─ login-form.js (新規) └─ README.md (古い版) ファイルの内容が変わる! 場合によっては 新しいファイルも現れる
ハマりどころ —

未コミットの変更があると切替を拒否されることがあります。 → git stash で一時退避(Vol.5)。

# 既存のブランチに切替 $ git switch feature/login Switched to branch 'feature/login' # 1つ前のブランチに戻る $ git switch - Switched to branch 'main' # --- switch vs checkout --- # 新 (Git 2.23〜推奨) $ git switch main # 旧 (今でも有効) $ git checkout main checkoutは機能が多すぎて分割された: switch — ブランチ切替専用 restore — ファイル復元専用 # --- リモートブランチへ切替 --- $ git fetch $ git switch feature/x # 自動で追跡ブランチを作成 # --- 強制切替 (危険・要注意) --- $ git switch -f main ↑ 未コミット変更を破棄
TIP —

業務中はswitchを使う方が安全。checkoutは「ファイル復元」も兼ねるため、間違えると変更を消し飛ばす事故が起きがち。switch / restore を分けて使うのが現代流です。

04

HEADを理解する

HEAD POINTER

HEAD はGitで頻出する概念で、「現在地を指すポインタ」のこと。 ふだんはあるブランチを指していて、そのブランチが指しているコミットが「今いる場所」になります。 エラーメッセージや高度な操作で必ず出てくるので、ここで掴んでおきましょう。

HEAD ★★ 現在地の指し手
// HEADの位置イメージ
[HEAD] ↓ [main] ↓ ●───●───●───● ↓ switch feature [HEAD] ↓ [feature] ↓ ●───●───●───● \ ●───● HEAD = 「今あなたがいる場所」 通常はブランチ名を指す
HEAD表記の覚え方

HEAD = 現在のコミット
HEAD~1 = 1つ前のコミット
HEAD~3 = 3つ前のコミット
HEAD^ = 親コミット (= HEAD~1)

# HEADが指すコミットを確認 $ cat .git/HEAD ref: refs/heads/main # ↑ HEADはmainブランチを指している $ git log -1 HEAD commit a3f2c1d8... (HEAD -> main) # --- HEAD相対の参照 --- $ git show HEAD # 最新 $ git show HEAD~1 # 1つ前 $ git show HEAD~5 # 5つ前 # --- detached HEAD状態 --- $ git checkout a3f2c1d You are in 'detached HEAD' state. # ↑ ブランチでなくコミットを直接指す そこでコミットしても "宙ぶらりん" # 元に戻る $ git switch main # --- HEAD移動の確認 --- $ git reflog a3f2c1d HEAD@{0}: checkout: main 9b8e4a2 HEAD@{1}: checkout: ...
POINT —

detached HEAD という警告は初学者を一番驚かせるメッセージ。「ブランチに紐づかない宙ぶらりんの状態」のことで、慌てず git switch <ブランチ名> で戻れば問題ありません。

05

Fast-forwardマージ

FAST-FORWARD MERGE

マージには2種類あって、まず一つ目が Fast-forward(早送り)マージ。 分岐元から本流に新しいコミットがない場合、ブランチのポインタをスーッと前に進めるだけ。マージコミットすら作られません。

git merge (fast-forward) ★★★ 一直線な統合
// Fast-forwardの起きる条件
[ マージ前 ] [main] ↓ ●───●───● \ ●───●───● ↑ [feature] ↓ git switch main ↓ git merge feature [ マージ後 ] [main] [feature] ↓ ●───●───●───●───●───● mainポインタが featureの位置までスーッと 前進するだけ
条件 —

分岐後、main側に新しいコミットが追加されていないこと。

# Fast-forward マージの流れ # 1. mainに戻る (取り込み先へ移動) $ git switch main # 2. 最新化 (リモートがあるなら) $ git pull # 3. feature を取り込む $ git merge feature/login Updating 9b8e4a2..a3f2c1d Fast-forward src/login.js | 12 ++++++++++++ 1 file changed, 12 insertions(+) # ↑ "Fast-forward" の文字が目印 # --- Fast-forwardを禁止する --- $ git merge --no-ff feature/login ↑ 強制的にマージコミットを作る 履歴上、何を統合したか明示できる # --- 設定でデフォルト変更 --- $ git config --global merge.ff false
現場で —

「履歴をシンプルに保ちたい」派はFast-forward歓迎。「いつどの機能が入ったか明示したい」派は--no-ffを強制。チームの方針で決まります。GitHubのPRマージ画面でも選択可能(Vol.6)。

06

3-wayマージ

THREE-WAY MERGE

両方のブランチに変更があるとFast-forwardは使えず、Gitは「3-wayマージ」を実行します。 分岐元(共通祖先)、main側の変更、feature側の変更——3点を見比べて統合し、専用のマージコミットを作ります。

git merge (3-way) ★★★ 分岐の合流
// 3-wayマージの構造
[ マージ前 ] [main] ↓ ●───●───●───● \ ●───●───● ↑ [feature] ↓ git merge feature [ マージ後 ] [main] ↓ ●───●───●───●───● ← マージコミット \ / (親が2つ!) ●───●───● ↑ [feature] 3点(共通祖先・main・feature)を 比較して統合
注意 —

同じ箇所を両方で編集しているとコンフリクト発生(Vol.5で詳説)。

# 3-way マージの流れ $ git switch main $ git merge feature/login Merge made by the 'ort' strategy. src/login.js | 12 ++++++++++++ src/auth.js | 3 +++ 2 files changed, 15 insertions(+) # ↑ "Merge made by..." の表示 = 3-wayマージが行われた合図 # --- マージコミットの中身 --- $ git log --graph --oneline * e9f8d27 Merge branch 'feature/login' |\ | * a3f2c1d fix: 認証バグ修正 | * 9b8e4a2 feat: ログインフォーム * | 5d6f3a8 docs: README更新 |/ * 2c1b0e9 共通祖先のコミット # マージコミットは "親が2つ" # →分岐の合流地点になる # --- マージを取り消したい時 --- $ git merge --abort ↑ コンフリクト中なら中止できる
POINT —

3-wayマージは履歴に「ここで合流した」という事実を残すのが特徴。後から「いつ何が取り込まれたか」を追いやすくなります。逆に、履歴のグラフが入り組んで見えるのが弱点。状況に応じて Fast-forward / 3-way / Rebase(Vol.5) を使い分けるのが上達のポイントです。

07

ブランチの削除と整理

DELETE & CLEANUP

マージし終わったブランチはこまめに消すのが鉄則です。 放置するとブランチ一覧が「死んだブランチ」だらけになって、現役のブランチが埋もれてしまいます。 削除コマンドには安全モードと強制モードがあり、状況で使い分けます。

git branch -d / -D ★★ 整理整頓
// 削除の前後
削除前
* main
feature/login
feature/signup (merged)
fix/header-bug (merged)
experiment-old (未マージ)
削除後 (整理済)
* main
feature/login
鉄則 —

マージ済みは -d で、未マージを消すなら -D(強制)。大文字Dは慎重に。

# --- 削除の基本 --- # 安全削除 (マージ済みのみ消せる) $ git branch -d feature/login Deleted branch feature/login. # 未マージのを消そうとすると… $ git branch -d experiment error: The branch 'experiment' is not fully merged. # 強制削除 (本当にいい時だけ!) $ git branch -D experiment # --- マージ済みブランチを列挙 --- $ git branch --merged main feature/login feature/signup fix/header-bug # ↑これらはまとめて消してOK # --- リモートのブランチを消す --- $ git push origin --delete feature/x # --- 一括クリーンアップ --- $ git branch --merged main | grep -v main \ | xargs git branch -d # mainとマージ済みを一気に削除
TIP —

消したブランチも30〜90日は復元可能(git reflog で履歴が残るため)。なので「とりあえず消す」でほぼ問題ありません。気になる人は git tag で目印を打ってから削除すると安心。

08

ブランチ戦略の基礎

BRANCHING STRATEGY

チームでブランチをどう使い分けるかを決めた「お作法」がブランチ戦略です。 代表的なものを2つ知っておけば、たいていの現場で通用します。GitHub FlowとGit Flow——名前が似てて紛らわしいですが別物。

github-flow vs git-flow ★★ 2大流派
// 2つの戦略の比較
▼ GITHUB FLOW (シンプル派)

使うブランチ: main + feature
流れ: main から feature を切り、PRでmainにマージ。
向いてる: Webサービス、SaaS、継続デプロイ。
長所: 覚えやすい、運用が軽い。

▼ GIT FLOW (本格派)

使うブランチ: main / develop / feature / release / hotfix
流れ: develop に機能を集積、releaseで仕上げ、mainがリリース版。
向いてる: パッケージソフト、バージョン管理が厳密な製品。
長所: 役割が明確。短所: 複雑で重い。

結論 —

迷ったらGitHub Flow。世界の多くのチームがこちら。

# === GitHub Flow の典型 === # 1. mainから機能ブランチを切る $ git switch main $ git pull $ git switch -c feature/login # 2. コミットを重ねる $ git commit -m "..." # 3. プッシュしてPR (Vol.6) $ git push origin feature/login # 4. レビュー → main にマージ (GitHub上で操作) # 5. ローカル整理 $ git switch main $ git pull $ git branch -d feature/login # これを繰り返すのが # GitHub Flow の本体 # --- 詳しくは Vol.8 で ---
POINT —

「ブランチ戦略はチームごとに違う」が現場の真実。新しい現場に入ったらまずブランチ戦略を確認するのが一流のお作法です。Git Flow派、GitHub Flow派、独自カスタム派、すべてあります。
