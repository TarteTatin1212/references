01

git restore — 変更の破棄

RESTORE

まだコミットしていない変更を「なかったことに」するコマンド。 ワーキングディレクトリの編集を捨てる、ステージから降ろす——どちらもrestoreでできます。 commit前の取り消しならコレと覚えればOK。

git restore ★★★ コミット前の取消
// どこを取り消すか
コマンド効果
restore file.jsワーキングの変更を破棄
restore --staged file.jsステージから降ろす
restore .すべての変更を破棄
restore -s HEAD~1 file.js1つ前の状態に戻す
⚠ 危険 —

restore で破棄した変更はほぼ復元不能(ステージに乗せていなければ)。実行前に git diff で内容確認を。

# --- ワーキングの変更を破棄 --- $ git status modified: src/app.js ↑ さっき編集したけどやっぱり戻したい $ git restore src/app.js $ git status nothing to commit, working tree clean ↑ 編集前に戻った # --- ステージから降ろす --- $ git add src/app.js $ git status Changes to be committed: modified: src/app.js ↑ 間違えてaddした! $ git restore --staged src/app.js ↑ ワーキングの変更は残る ステージから降ろすだけ # --- 全部まとめて取消 --- $ git restore . # --- 過去のコミットからファイル復元 --- $ git restore -s HEAD~3 src/app.js ↑ 3つ前のコミットの状態に # 旧式 (今も動く) $ git checkout -- src/app.js
POINT —

Git 2.23以降は checkout -- file でなく restore が推奨です。「変更の取り消しは restore」と覚えれば、ブランチ切り替えの switch と用途が分かれて分かりやすい。

02

git reset — 履歴の巻き戻し

RESET

コミット済みの履歴を巻き戻す強力なコマンド。 3つのモード(--soft / --mixed / --hard)があり、それぞれ影響範囲が違うのがポイント。 使いどころを間違えると変更が吹き飛ぶので、違いの理解必須。

git reset ★★★ 3モードを使い分け
// 3モードの影響範囲
モードHEADステージワーキング
--soft戻すそのままそのまま
--mixed
(デフォルト)		戻す戻すそのまま
--hard戻す戻す破棄
使い分けの目安

soft: コミットだけ取り消して再コミットしたい時
mixed: ステージも含めて状態を戻したい時(普通)
hard: ファイルごと完全に消したい時(要注意)

⚠ 厳重注意 —

--hard は未コミットの変更を全消去。実行前に必ず git status で確認!

# --- soft: コミットだけ取消 --- $ git reset --soft HEAD~1 # 直前のコミットがほどけて # 変更はステージに残る (再コミット可) # --- mixed (デフォルト) --- $ git reset HEAD~1 # コミット取消 + ステージ降ろす # 変更はワーキングに残る # --- hard: 完全削除 (危険) --- $ git reset --hard HEAD~1 # 直前のコミット + 変更全部消える # 戻すなら git reflog (本巻 08) # --- 特定のコミットまで戻す --- $ git reset --hard a3f2c1d # --- pushしたコミットを戻したら? --- resetは履歴を改変するため、 既にpush済みなら push --force が必要。 → チーム開発では事故の元。 → そんな時は revert (本巻 03) を使う
鉄則 —

git reset --hard を打つ前に「3秒待ってstatus確認」。これだけで多くの事故が防げます。やってしまっても git reflog(本巻08)で復活できることが多いので、慌てずに。

03

git revert — 安全な取消

REVERT

「過去のコミットを打ち消す新しいコミット」を作るコマンド。 履歴を改変しないので、共有済み(push済み)のコミットを取り消す時はこれ一択。 resetが「過去を消す」なら、revertは「過去にダメ出ししただけ」。

git revert ★★★ push済みの取消
// reset vs revert
[ reset の場合 ] ●───●───●───● ↑ 最新 $ git reset --hard HEAD~1 ●───●───● (1つ消える) [ revert の場合 ] ●───●───●───● ↑ 最新 $ git revert HEAD ●───●───●───●───●' ↑ "取消コミット"が追加
どっちを使う?

個人のローカル(未push)→ reset でOK
共有済み(pushした)→ 必ず revert
mainブランチ→ 例外なく revert

POINT —

revertは履歴を書き換えないので、他人とのコンフリクトを起こさない。チーム開発の救世主。

# --- 直前のコミットを取消 --- $ git revert HEAD # コミットメッセージ編集画面が開く Revert "feat: ログイン機能追加" This reverts commit a3f2c1d8... # 保存して閉じると新コミット作成 [main b7e5a92] Revert "feat: ..." # --- 特定のコミットを取消 --- $ git revert a3f2c1d # --- 複数まとめて取消 --- $ git revert a3f2c1d..b7e5a92 # --- マージコミットを取消 --- $ git revert -m 1 <merge-commit> ↑ -m 1 = mainブランチ側を残す # --- コミットなしで変更だけ取込 --- $ git revert -n <commit> まとめて1コミットにしたい時 # --- 取消の取消 --- $ git revert <revert-commit> revertコミット自体をrevertすれば 元の変更が復活
現場で —

本番障害を起こしたコミットの巻き戻しは必ずrevert。「reset --hardしてpush --force」は、他のメンバーのローカルを壊す世紀の事故になります。共有済みのものは revert、これは鉄則中の鉄則。

04

git stash — 一時退避

STASH

作業途中で「ちょっとだけ別ブランチに行きたい」時に使う一時退避コマンド。 変更を引き出しにしまっておくイメージ。後で取り出せます。 緊急のバグ対応で頻出する超便利機能です。

git stash ★★★ 作業の一時退避
// よくある使用シーン
シーン: 緊急対応

feature/x で開発中 → 本番障害発生 → mainで緊急修正したい → でもまだコミットできる状態じゃない → stashで退避!

[ 開発中 ] feature/x で編集中 src/app.js (未コミット変更あり) ↓ git stash [ 退避完了 ] ワーキング: クリーンな状態 stash@{0}: 退避された変更 ↓ 別作業 (緊急修正等) ↓ feature/x に戻ってきた ↓ git stash pop [ 復元 ] src/app.js (変更が戻った)
# --- 退避 --- $ git stash Saved working directory and index state WIP on feature/x: a3f2c1d ... # 名前付きで退避 (分かりやすい) $ git stash push -m "login form WIP" # --- 退避一覧 --- $ git stash list stash@{0}: On feature/x: login form WIP stash@{1}: WIP on main: a3f2c1d ... # --- 取り出す --- $ git stash pop ↑ 最新を取り出して、stashから消す # 特定のを指定して取り出し $ git stash pop stash@{1} # 取り出すけどstashにも残す $ git stash apply # --- 中身を確認 --- $ git stash show stash@{0} $ git stash show -p stash@{0} ↑ 差分も表示 # --- 不要なstashを削除 --- $ git stash drop stash@{0} $ git stash clear # 全削除 # --- 新規ファイルも含める --- $ git stash -u ↑ Untrackedも退避対象に
あるある —

stashにため込んで「あれ、何を退避したっけ?」状態になりがち。これを防ぐには git stash push -m "用途" で必ず名前を付ける。長期間放置するくらいなら、いったんコミットしておくのも手です。

05

git rebase — 履歴の整形

REBASE

分岐したブランチを「最新のmainからやり直したことにする」強力なコマンド。 マージとの違いは「履歴が一直線になる」こと。 対話モード(-i)を使えば、コミットの並べ替え・結合・編集まで自由自在。

git rebase ★★★ 履歴を一直線に
// merge vs rebase
[ merge の場合 ] ●───●───●───● (main) \ \ ●───●───●' (merge commit) → 分岐が履歴に残る [ rebase の場合 ] ●───●───●───● (main) \ ●'──●' (再生成) → 履歴が一直線に
rebase のお約束

◯ 個人ブランチでは自由に使う
✗ 共有ブランチでやると大事故
→ 「自分しか触らないブランチ」が原則

⚠ 黄金律 —

push済みのコミットを rebase してはいけない。他人のローカルとの整合性が崩れます。

# --- 基本: mainの最新を取り込む --- $ git switch feature/x $ git rebase main First, rewinding head to replay your work on top of it... Applying: feat: ログイン追加 Applying: fix: バリデーション # 履歴が main の先に並び直る # --- 対話モード (神機能) --- $ git rebase -i HEAD~5 ↑ 直近5コミットを対象 # エディタが開く: pick a3f2c1d feat: ログイン追加 pick 9b8e4a2 fix: typo修正 pick 5d6f3a8 fix: typo修正2 pick 2c1b0e9 feat: ログアウト追加 pick b7e5a92 typo修正 # 操作キーワード: pick そのまま採用 reword メッセージ書き換え edit コミット内容を修正 squash 直前のコミットに結合 fixup squashと同じ(メッセージ捨) drop 削除 # 例: typo修正コミットをまとめる pick a3f2c1d feat: ログイン追加 fixup 9b8e4a2 fixup 5d6f3a8 pick 2c1b0e9 feat: ログアウト追加 fixup b7e5a92 # --- 途中で中止 --- $ git rebase --abort
裏技 —

PR出す前に git rebase -i でコミットを綺麗に整形するのがプロのお作法。「typo修正」「ちょっと修正」みたいなコミットを fixup でまとめて、レビュアーに読みやすい履歴を渡す。これができると評価が一段上がります。

06

コンフリクトの解消

RESOLVING CONFLICTS

同じファイルの同じ行を両方で編集すると発生するのがコンフリクト。 Gitが「どっちが正解か判断できないので、人間が決めて」と頼ってくる状態です。 慌てる必要なし、手順を踏めば必ず解決できます。

conflict resolution ★★★ 避けられぬ通過儀礼
// コンフリクトマーカー
function greet() { <<<<<<< HEAD return "Hello, World"; ======= return "Hi, World"; >>>>>>> feature/x } ↑ HEAD側 (現在のブランチ) ↓ 取り込み元 (feature/x側) これを人間が判断して、 正しい形に書き換える
解消の流れ

① git status で衝突ファイルを確認
② エディタで開く
③ <<< === >>> マーカーを見つける
④ 正しい形に書き換え(マーカーも消す)
⑤ git add で「解決済み」を宣言
⑥ git commit または rebase --continue

# --- マージ時にコンフリクト発生 --- $ git merge feature/x Auto-merging src/app.js CONFLICT (content): Merge conflict in src/app.js Automatic merge failed; fix conflicts and then commit the result. # --- どこが衝突したか確認 --- $ git status Unmerged paths: (use "git add ..." to mark resolution) both modified: src/app.js # --- ファイルを開いて修正 --- # <<< === >>> マーカーを消し、 # 正しい内容に整える # --- 解決を宣言 --- $ git add src/app.js $ git commit # マージコミットメッセージが自動生成 # --- rebase中なら --- $ git add src/app.js $ git rebase --continue # --- マージを諦めて中止 --- $ git merge --abort # --- 便利: マージツール --- $ git mergetool # VS Code、IntelliJ等の統合可 # 片側だけを採用 $ git checkout --ours file $ git checkout --theirs file
鉄則 —

コンフリクトを見てパニックにならない。手順は決まっています。①statusで確認 → ②マーカーを見つけて修正 → ③addでcontinue。それだけ。「コンフリクトは敵じゃない、Gitが助けを求めているだけ」と思えば気が楽です。

07

git cherry-pick — 特定コミットの取込

CHERRY-PICK

別のブランチから「特定のコミット1個だけ」を今のブランチに取り込むコマンド。 「あのバグ修正だけ先に本番ブランチに当てたい」みたいな時の救世主。 名前の通り、サクランボを摘むように美味しい部分だけ持ってきます。

git cherry-pick ★★ ピンポイント取込
// 動作イメージ
[ feature/x ] ●───A───B───C───D ↑ bug fix これだけ main に欲しい! [ main ] ●───●───●───● $ git switch main $ git cherry-pick A [ 取込後 ] ●───●───●───●───A' ↑ コピーされたコミット
よくある使い道

① 本番ブランチに緊急修正だけ当てたい
② 他人のPRから1コミットだけ拝借
③ 間違って消したコミットの復活

注意 —

cherry-pickは「新しいコミット」として作られるため、ハッシュは変わります。

# --- 1コミットを取込 --- $ git switch main $ git cherry-pick a3f2c1d [main b7e5a92] fix: 認証バグ修正 Date: Wed May 13 10:23:45 2026 1 file changed, 3 insertions(+) # --- 複数コミットを取込 --- $ git cherry-pick a3f2c1d 9b8e4a2 # 範囲指定 (..は除く、^..は含む) $ git cherry-pick a3f2c1d^..9b8e4a2 # --- コミットせず変更のみ取込 --- $ git cherry-pick -n a3f2c1d ↑ ステージに乗るだけ 自分でメッセージ書いて commit # --- コンフリクト発生時 --- $ git cherry-pick a3f2c1d CONFLICT (content): ... $ git status # 確認 # 解消 → add $ git cherry-pick --continue # 諦める $ git cherry-pick --abort
現場で —

cherry-pickは便利ですが多用は禁物。「同じ修正が複数ブランチにある」状態は履歴が辿りにくくなる原因です。本来はマージで解決するのが基本、cherry-pickは「どうしてもこの1個だけ」の最終手段と心得ましょう。

08

reflog — 最後の救済

REFLOG

git reflog はあなたの「HEAD移動の全履歴」。 reset --hard で消したコミットも、間違ってrebaseで失ったコミットも、ここから復活できることが多い。 「Gitに最終救済策あり」と覚えておけば、いざという時に冷静でいられます。

git reflog ★★★ Gitの命綱
// reflogの中身
$ git reflog a3f2c1d HEAD@{0}: commit: feat: 追加 9b8e4a2 HEAD@{1}: reset: --hard HEAD~1 5d6f3a8 HEAD@{2}: commit: fix: 修正 ← これ 2c1b0e9 HEAD@{3}: checkout: from main ... ここから HEAD@{2} を救出可能! $ git reset --hard 5d6f3a8
reflogが救うもの

◯ reset --hard で消えたコミット
◯ rebase で書き換える前のコミット
◯ ブランチを削除してしまった
◯ amendで上書きされた古いコミット
✗ 未コミットの変更(これだけは救えない!)

保持期間 —

reflog はデフォルト90日残る。それまでなら救出可能。

# --- reflog を確認 --- $ git reflog a3f2c1d HEAD@{0}: commit: feat: 追加 9b8e4a2 HEAD@{1}: reset: --hard HEAD~1 5d6f3a8 HEAD@{2}: commit: fix: 修正 2c1b0e9 HEAD@{3}: checkout: ... # 「あれ?5d6f3a8 を消しちゃった!」 # --- 救出: コミットを取り戻す --- $ git reset --hard 5d6f3a8 HEAD is now at 5d6f3a8 fix: 修正 # または別ブランチに復元 $ git branch recovered 5d6f3a8 # --- 特定ブランチのreflog --- $ git reflog show main $ git reflog show feature/x # --- やった、復活できた! --- # --- reflog の覚え方 --- HEAD@{0} 最新 HEAD@{1} 1つ前 HEAD@{2} 2つ前 HEAD@{n} n操作前 # 時刻指定もできる $ git reflog "HEAD@{1 hour ago}" $ git reflog "HEAD@{yesterday}"
最後に —

Git歴の長い人は「reflog があるから安心して壊せる」と言います。失敗を恐れずに reset --hard や rebase を使えるのは、この命綱があるから。reflog は Git からの最大の贈り物です。
