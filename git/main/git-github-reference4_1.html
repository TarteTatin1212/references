01

リモートの概念

REMOTE CONCEPT

リモートとは「ネットワークの向こうにあるリポジトリ」のこと。 GitHubはその代表的な置き場所です。 あなたのPCのリポジトリ(ローカル)と、GitHub上のリポジトリ(リモート)が「ペア」になって、双方向にやり取りするイメージを持ってください。

local ⇄ remote ★★★ 全体像
// 双方向のやり取り
[ あなたのPC ] ローカルリポジトリ .git/ src/ README.md │ │ push (送る) ↓ ↑ pull (取る) │ [ GitHub ] リモートリポジトリ github.com/taro/app ↑↓ [ 同僚のPC ] ローカルリポジトリ (同じものを持っている)
主な操作と方向

push = ローカル → リモート (送信)
fetch = リモート → ローカル (取得のみ)
pull = リモート → ローカル + マージ
clone = リモートを丸ごと複製(初回)

# 接続されているリモートを確認 $ git remote origin # URLも表示 $ git remote -v origin https://github.com/taro/app.git (fetch) origin https://github.com/taro/app.git (push) # --- 「origin」って何? --- 単なる「リモートに付けた名前(ニックネーム)」 慣習として最初のリモート = origin 名前は自由に変えられる # --- リモートとの関係 --- ローカルブランチ: main リモートブランチ: origin/main ↑ "origin"上の"main" # 2つは別物 # push/pull で同期する
POINT —

リモートは1つに限定されない。たとえばオープンソースに貢献する時は「origin(自分のフォーク)」と「upstream(本家)」の2つを登録するのが定番。複数リモートの活用はVol.8で詳説します。

02

git remote — リモートの管理

REMOTE COMMANDS

リモートの追加・確認・変更・削除を行う基本コマンド群です。 URL変更(例:HTTPSからSSHに切替)はよくある操作なので、set-urlは特に覚えておくと便利。

git remote ... ★★ リモート操作の一覧
// よく使うサブコマンド
コマンド意味
remote -vURL含めて一覧
remote addリモートを追加
remote removeリモートを削除
remote rename名前を変更
remote set-urlURLを変更
remote show origin詳細を表示
あるある —

「あれ、いつの間にかpushできなくなった?」 → URLが古いままのことが多い。remote -v で確認する癖を。

# --- 追加 --- $ git remote add origin https://github.com/taro/app.git # --- 確認 --- $ git remote -v origin https://... (fetch) origin https://... (push) # --- 詳細 --- $ git remote show origin * remote origin Fetch URL: https://... Push URL: https://... HEAD branch: main Remote branches: main tracked feature tracked # --- URLを変更 (HTTPS→SSH等) --- $ git remote set-url origin git@github.com:taro/app.git # --- 名前を変える --- $ git remote rename origin upstream # --- 削除 --- $ git remote remove origin (リモート側は消えない、登録解除のみ) # --- 複数リモート登録の例 --- $ git remote add origin git@github.com:taro/app $ git remote add upstream git@github.com:org/app ↑ 自分のフォーク ↑ 本家
TIP —

リモートの削除は remove だけど、リネームは rename(removeじゃない)。地味に紛らわしいので注意。

03

git push — ローカルからリモートへ

PUSH

ローカルでコミットした内容をリモート(GitHub)に送るコマンドです。 pushしないとGitHub上には何も反映されません。 「コミット」と「push」は別物——これは初学者が一番ハマるポイント。

git push ★★★ リモートへ反映
// 基本パターン
初回 (-u が必要)

git push -u origin feature/x

2回目以降 (シンプル)

git push

別ブランチへ送る

git push origin feature/x

[ Local ] [ Remote ] main: ●─●─● main: ● ↑ HEAD │ │ push ↓ main: ●─●─● main: ●─●─● (送られて同期)
# --- 初回 push (上流設定) --- $ git push -u origin main ↑ "upstream"を設定 Enumerating objects: 3, done. Writing objects: 100% (3/3) To github.com:taro/app.git * [new branch] main -> main Branch 'main' set up to track remote branch 'main' from 'origin'. # 以降は引数なしでOK $ git push # --- 拒否されるケース --- # ① リモートに新しいコミットがある時 $ git push ! [rejected] main -> main (non-fast-forward) hint: Updates were rejected because... # 解決: 先にpullで取り込む $ git pull $ git push # ② 強制push (危険・要注意) $ git push --force-with-lease ↑ 他人のコミットがあれば拒否 --forceより安全 # --- リモートのブランチを消す --- $ git push origin --delete feature/x
鉄則 —

git push --force は「他人のコミットを消し飛ばす」核ミサイル級の操作。チーム開発で軽率にこれをやると信頼を失います。どうしても必要な時は --force-with-lease を使うのが現代の流儀です。

04

git fetch と git pull

FETCH & PULL

リモートの最新情報を取ってくる2つのコマンド。 違いを一言で: fetchは「取ってくるだけ」、pullは「取ってきて即マージ」。 安全派は fetch → 確認 → merge、お手軽派は pull 一発。

fetch vs pull ★★★ 取得の2系統
// 動作の違い
動作fetchpull
リモート確認◯◯
ダウンロード◯◯
自動マージ✗◯
事故リスク低中
[ fetch のイメージ ] Remote: ●─●─●─● (最新) ↓ fetch Local: ●─●─● origin/main更新 ローカルmainは無変化 → 確認してからmergeで反映 [ pull のイメージ ] Remote: ●─●─●─● (最新) ↓ pull Local: ●─●─●─● ローカルmainも更新 (fetch + merge を一発)
# --- fetch (取得のみ) --- $ git fetch From github.com:taro/app 9b8e4a2..a3f2c1d main -> origin/main # 取れたものを確認 $ git log --oneline origin/main a3f2c1d 同僚のコミット (まだローカル未反映) 9b8e4a2 自分の最後のコミット # 差分を確認 $ git diff main origin/main # 取り込むならmerge $ git merge origin/main # --- pull (取得 + マージ) --- $ git pull remote: Enumerating objects... Updating 9b8e4a2..a3f2c1d Fast-forward # pull = fetch + merge を一発 # --- pull のリベース版 --- $ git pull --rebase ↑ 履歴が一直線になる(Vol.5) # --- 設定でデフォルト変更 --- $ git config --global pull.rebase true
POINT —

業務開発では「朝一番にpull」がお約束。これを怠ると、自分のpush時にコンフリクトの嵐に巻き込まれます。「迷ったらまずfetch、安全なものはpull」と覚えましょう。

05

認証 (HTTPS + PAT)

AUTHENTICATION

2021年8月以降、GitHubはHTTPSでのパスワード認証を廃止しました。 今はPersonal Access Token (PAT) という「使い捨てパスワード」を使うのが基本。 pushで「Authentication failed」と出る初学者の99%がここに引っかかります。

personal access token ★★★ HTTPS認証の壁
// PAT発行の手順
STEP

① GitHub にログイン
② 右上のアバター → Settings
③ 左サイドバー最下部 Developer settings
④ Personal access tokens → Tokens (classic)
⑤ Generate new token (classic)
⑥ Note(用途名) / Expiration / Scope(repo必須)
⑦ 発行されたトークンを必ずコピー
⚠ 1度しか表示されない!

注意 —

PATは「パスワード相当の機密情報」。コードに書いてpushしたら即座に無効化してください。GitHubが検知して自動失効させることも。

# --- 実際のpush認証 --- $ git push Username: taro Password: # ←ここにPATを貼る! GitHubパスワードではない # --- 認証情報をキャッシュ --- # macOS (キーチェーン) $ git config --global \ credential.helper osxkeychain # Windows (資格情報マネージャー) $ git config --global \ credential.helper manager # Linux (一定時間メモリ保持) $ git config --global \ credential.helper "cache --timeout=3600" # --- よくあるエラー --- remote: Support for password authentication was removed on August 13, 2021. fatal: Authentication failed # → パスワードでなくPATを使う # --- PATの代わりに gh コマンド --- $ gh auth login # GitHub CLI を使えば対話的に設定可能
裏技 —

もっと楽な方法もあります。GitHub CLI (gh) をインストールして gh auth login を一度実行すれば、以降のpush/pullは認証を意識せずに使えます。初学者には超おすすめ。

06

SSH鍵での認証

SSH KEY AUTH

PATの代わりにSSH鍵を使う方法もあります。 一度設定すればパスワード/トークン入力が一切不要になり、長期運用に最適。 Web系エンジニアの多くはこちらを採用しています。

ssh key authentication ★★★ 玄人の選択
// SSH鍵のしくみ
[ あなたのPC ] ~/.ssh/ ├─ id_ed25519 秘密鍵 (門外不出) └─ id_ed25519.pub 公開鍵 (GitHub登録用) 公開鍵を GitHub に登録すると… [ push/pull 時 ] ローカル ──── 秘密鍵で署名 ────→ GitHub ↓ 公開鍵で検証 ←──────┘ ↓ "本人確認OK!" パスワード/PAT入力が不要に
注意 —

秘密鍵(.pubなしの方)は絶対に外に出さないこと。これが漏れたら別人になりすませます。

# --- ① 鍵を生成 --- $ ssh-keygen -t ed25519 -C "taro@example.com" Enter file in which to save the key: (Enterで標準パス) Enter passphrase: ******** (任意の合言葉, 空Enterでも可) # --- ② 公開鍵を表示してコピー --- $ cat ~/.ssh/id_ed25519.pub ssh-ed25519 AAAAC3...XYZ taro@example.com ↑ この行ぜんぶコピー # --- ③ GitHub に登録 --- GitHub → Settings → SSH and GPG keys → New SSH key → 貼り付け → Add # --- ④ 接続テスト --- $ ssh -T git@github.com Hi taro! You've successfully authenticated, but GitHub does not provide shell access. # --- ⑤ リモートURLをSSHに変更 --- $ git remote set-url origin \ git@github.com:taro/app.git # 以降、認証不要でpush/pull可能! $ git push # パスワードもPATも聞かれない
TIP —

鍵にはパスフレーズを設定するのがセキュリティ的に推奨。「毎回入力は面倒」と思うかもしれませんが、ssh-agentに登録すれば一度きりでOK。会社PCでは特に強く推奨です。

07

追跡ブランチ

TRACKING BRANCHES

ローカルブランチが「リモートのどのブランチと組になっているか」の関係を「追跡ブランチ(upstream)」と言います。 これが設定されていると、git push や git pull を引数なしで叩けるようになります。 地味に重要。

tracking / upstream ★★ ローカルとリモートの紐付け
// 追跡関係の図
[ Local ] [ Remote ] main ←─→ origin/main feature/x ←─→ origin/feature/x experiment (未追跡) ─ 追跡されているブランチは push/pull が引数なしでOK 未追跡だと毎回フルパス指定が必要: git push origin experiment
POINT —

git push -u の -u は「--set-upstream(追跡関係を設定)」の略。初回pushでは必ず付ける。

# 追跡関係の確認 $ git branch -vv * main a3f2c1d [origin/main] update feature 9b8e4a2 [origin/feature: ahead 2] ... local-only 5d6f3a8 just local ↑ どのリモートと組になっているか表示 # [origin/main] = 追跡中 # [ahead 2] = リモートより2コミット先 # --- 追跡を後から設定 --- $ git branch --set-upstream-to=origin/main または短縮: -u origin/main # --- pushと同時に設定 --- $ git push -u origin feature/x # --- 追跡を解除 --- $ git branch --unset-upstream # --- リモートを丸ごと最新化 + 不要な追跡を削除 --- $ git fetch --prune GitHub側で消されたブランチ追跡を整理 # 自動で prune するように設定 $ git config --global fetch.prune true
あるある —

「pushしてって言われたけど、なんか上手くいかない…」 → 大抵 -u を付けずに初回pushしようとしている。新しいブランチを初めて送る時は git push -u origin <branch>。これだけ覚えれば十分。

08

リモートのトラブル対処

TROUBLESHOOTING

リモート操作で遭遇する「よくあるエラー4つ」と対処法をまとめます。 どれも一度はハマるやつばかり。慌てず原因を切り分けるのがコツ。

remote troubles ★★ よくあるエラー集
// エラー一覧
① Authentication failed

原因: パスワード認証は廃止。
解決: PAT または SSH鍵を使う(本巻 05〜06)。

② Updates were rejected (non-fast-forward)

原因: リモートに自分が知らないコミットがある。
解決: 先に git pull。

③ src refspec ... does not match any

原因: 存在しないブランチをpushしようとした。
解決: ブランチ名のタイポ確認、コミットを1つは作る。

④ remote: Permission denied

原因: 書き込み権限がないリポジトリ。
解決: 自分のリポジトリかフォーク先か確認。

# ① 認証エラー fatal: Authentication failed for ... → PATを使い直す or SSH切替 # ② 競合push ! [rejected] main -> main (non-fast-forward) error: failed to push some refs to ... hint: Updates were rejected because the hint: remote contains work that you do not hint: have locally. → git pull してから git push # ③ refspec エラー error: src refspec main does not match any → コミットがないか、ブランチ名間違い $ git log # コミットあるか? $ git branch # 名前合ってる? # ④ 権限エラー remote: Permission to xxx/yyy.git denied fatal: unable to access ... → 別人のリポジトリにpushしようとしている 自分のフォークを作るか権限を貰う # --- 切り分けの定番 --- $ git remote -v # URL確認 $ git status # 状態確認 $ git branch -vv # 追跡確認 $ ssh -T git@github.com # 接続確認
鉄則 —

リモートでエラーが出たらまずエラーメッセージを丁寧に読む。Gitのエラーメッセージは「ヒント付き」で意外と親切。hint: から始まる行に解決策が書かれていることがほとんどです。
