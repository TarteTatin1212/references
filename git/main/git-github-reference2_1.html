01

git init — リポジトリを作る

INITIALIZE

既存のフォルダをGitで管理されるリポジトリに変えるコマンドです。 実行すると .git という隠しフォルダが生まれ、そこにすべての履歴が記録されていきます。

git init ★★★ 最初の一手
// init で起きること
[ 実行前 ] my-app/ ├─ index.html └─ style.css ↓ git init [ 実行後 ] my-app/ ├─ .git/ ← 誕生! │ ├─ HEAD │ ├─ config │ ├─ objects/ │ └─ refs/ ├─ index.html └─ style.css
注意 —

.git を誤って消すと履歴が全部消えます。ホームディレクトリで git init してしまう事故も多発、要注意。

# 新規プロジェクトを始める $ mkdir my-app && cd my-app $ git init Initialized empty Git repository in /Users/taro/my-app/.git/ # 既存フォルダを途中から管理することも可 $ cd existing-folder $ git init # --- ブランチ名を指定して初期化 --- $ git init -b main ↑ 最初から main で作る # --- やってしまった時の取り消し --- $ rm -rf .git ↑ これで完全にGit管理から外れる # --- 確認 --- $ ls -la drwxr-xr-x .git/ # ある = Git管理下
POINT —

1つのプロジェクトに1つのリポジトリが原則。サブフォルダにさらに git init するとネストして混乱の元になります。

02

git clone — リポジトリを複製する

CLONE

GitHubなどに置かれたリモートリポジトリを自分のPCに丸ごとコピーするコマンドです。 新しい職場で「このリポジトリから始めて」と渡されたら、まずこれを叩きます。

git clone <url> ★★★ チーム参加の第一歩
// clone で取得されるもの
取得される内容

① 最新のファイル全部
② 過去すべての履歴
③ リモートの登録情報 (origin)
④ すべてのブランチ情報

github.com/team/web-app │ │ git clone ↓ [ あなたのPC ] web-app/ ├─ .git/ (履歴も全部) ├─ src/ ├─ package.json └─ README.md
TIP —

cloneはinit + remote add + pull を一気にやる便利コマンド。

# HTTPSでクローン (基本) $ git clone https://github.com/team/web-app.git Cloning into 'web-app'... remote: Counting objects: 421, done. Receiving objects: 100% (421/421) # SSHでクローン (認証設定後) $ git clone git@github.com:team/web-app.git # フォルダ名を指定して取得 $ git clone https://github.com/team/web-app.git my-folder # 特定のブランチだけ取得 $ git clone -b develop ... # 履歴を浅くする (容量節約) $ git clone --depth 1 ... ↑ 最新コミットだけ取得
ハマりどころ —

大規模リポジトリ(GBクラス)を --depth なしでcloneすると30分待たされることも。読むだけなら --depth 1、開発するなら通常cloneと使い分けます。

03

git status — 状態を確認する

STATUS

今どのファイルが、どのエリアに、どんな状態でいるかを表示する、Git操作中の「中継カメラ」です。 何か操作するたびにこれを叩くのが上達への近道。git statusは10秒に1回叩いていいくらいの存在です。

git status ★★★ 一番打つコマンド
// ファイル状態の種類
?? Untracked — まだGit管理外
M Modified — 変更済み、未ステージ
A Added — ステージ済み、コミット待ち
· Clean — 変更なし、最新状態
D Deleted — 削除した
R Renamed — リネームした
POINT —

色で判断: 赤=Untracked/Modified、緑=Staged。

$ git status On branch main Your branch is up to date with 'origin/main'. Changes to be committed: (use "git restore --staged ..." to unstage) modified: src/app.js new file: src/util.js Changes not staged for commit: (use "git add ..." to update) modified: README.md deleted: old.txt Untracked files: (use "git add ..." to include) temp.log # --- 短縮表示 --- $ git status -s M src/app.js A src/util.js M README.md ?? temp.log ↑左:ステージ 右:ワーキング
鉄則 —

「迷ったらstatus」。何か変なエラーに遭遇したら、まず git status。今どの状態にいるか把握すれば、次にやるべきことが見えてきます。

04

git add — ステージに乗せる

ADD / STAGE

変更を「次のコミットに含める対象」として宣言するコマンド。 ワーキングディレクトリの変更をステージングエリアに移動します。「準備が整った変更だけを選んでまとめる」ためのGit独自の仕組みです。

git add <files> ★★★ コミットの前準備
// add のパターン
書き方意味
git add file.txt1ファイルだけ
git add src/フォルダ配下全部
git add .カレント以下すべて
git add -A削除も含めて全部
git add *.jsパターンマッチ
git add -p変更箇所ごとに対話的選択
裏技 —

git add -p は神機能。同じファイル内の変更を、「この部分はコミットに含める、こっちは別コミット」とパッチ単位で振り分けできます。

# 基本: ファイル指定 $ git add src/app.js # 複数ファイル $ git add src/app.js README.md # すべてのファイル変更 $ git add -A # 拡張子で指定 $ git add "*.js" # --- 対話モード (おすすめ) --- $ git add -p src/app.js @@ -1,3 +1,5 @@ function hello() { + console.log("debug"); return "hi"; + // TODO: cleanup } Stage this hunk [y,n,q,a,d,e,?]? # y=入れる n=飛ばす s=さらに分割 # --- やり直したい時 --- $ git restore --staged <file> ↑ ステージから降ろす
ハマりどころ —

git add . でうっかり機密ファイルや巨大ファイルをステージしてしまう事故が頻発。これを防ぐのが .gitignore(Vol.8で詳説)です。

05

git commit — 履歴に刻む

COMMIT

ステージに乗せた変更を「正式な履歴」として記録するコマンド。 一度コミットすれば、原則としてその状態にはいつでも戻せるようになります。これがGitの強みの本体です。

git commit -m "message" ★★★ 履歴への刻印
// コミットメッセージの書き方
良いメッセージ ◯

fix: ログイン時の404エラーを修正
feat: パスワード強度メーターを追加
docs: READMEのセットアップ手順を更新

悪いメッセージ ✗

修正
バグ直した
asdfg
WIP

定番 —

「動詞 + 何を + なぜ(必要なら)」の形。これだけで未来の自分とチームに親切。

# 基本 $ git commit -m "fix: ログインエラーを修正" [main a3f2c1d] fix: ログインエラーを修正 2 files changed, 5 insertions(+), 2 deletions(-) # --- 詳しく書きたい時 --- $ git commit # エディタが開く fix: ログイン時の404エラーを修正 セッショントークンの検証ロジックに バグがあり、未認証時に正しい404でなく 500を返していたのを修正。 #1234 を参照。 # --- add + commit を同時に --- $ git commit -am "update README" ↑ 既存ファイルだけ自動add (新規ファイルは対象外!) # --- 直前のコミットを修正 --- $ git commit --amend ↑ メッセージ修正 / 変更追加
POINT —

コミットは「論理的に意味のある単位」で区切るのが鉄則。「機能追加」「バグ修正」「リファクタ」を1コミットに混ぜない。後でレビューする人(未来の自分含む)が泣きます。

06

git log — 履歴を見る

LOG

過去のコミット履歴を新しい順に表示します。「いつ・誰が・何を変えたか」を遡れるのがGitの真価。 オプションが豊富で、ここを使いこなせるかが中級者の境目です。

git log ★★★ 履歴を辿る
// よく使うオプション
オプション効果
--oneline1行表示
--graphブランチを線で表示
-n 5最新5件だけ
--all全ブランチ含む
--author="Taro"作者で絞り込み
--since="1 week"期間で絞り込み
-- file.js特定ファイルの履歴
-p差分も表示
定番の組み合わせ —

git log --oneline --graph --all。これをエイリアス登録しておくのが鉄板。

# 普通のlog $ git log commit a3f2c1d8... (HEAD -> main) Author: Taro Yamada <taro@...> Date: Wed May 13 10:23:45 2026 fix: ログインエラーを修正 commit 9b8e4a23... Author: Hanako Suzuki <hana@...> Date: Tue May 12 15:10:02 2026 feat: パスワード強度メーター追加 # --- 1行表示 (おすすめ) --- $ git log --oneline a3f2c1d fix: ログインエラーを修正 9b8e4a2 feat: パスワード強度メーター 5d6f3a8 docs: README更新 # --- グラフ表示 --- $ git log --oneline --graph --all * a3f2c1d fix: ログイン | * b7e5a92 feat: 通知機能 |/ * 9b8e4a2 feat: パスワード強度 * 5d6f3a8 docs: README更新 # 抜けるには q キー
TIP —

~/.gitconfig に alias.lg = log --oneline --graph --all --decorate を仕込んでおくと、git lg でいつでも美しいログ表示。現場のGit使いはほぼ全員入れています。

07

git diff — 差分を見る

DIFF

「何が変わったか」を行単位で表示するコマンド。 コミット前のチェック、コミット間の比較、ブランチ間の比較など、用途は無数にあります。 +(追加)と -(削除)で表示されるのが特徴。

git diff ★★★ 変更内容の確認
// 何と何を比べているか
コマンド比較対象
git diffワーキング ⇔ ステージ
git diff --stagedステージ ⇔ 直近コミット
git diff HEAD全変更 ⇔ 直近コミット
git diff a..bコミットa ⇔ コミットb
git diff main現在のブランチ ⇔ main
git diff -- file特定ファイルだけ
POINT —

3つのエリアの「どこの差を見たいか」でオプションが変わります。これが混乱の元なので、上の表は手元に置いておくと安心。

# 基本: 未ステージの変更を見る $ git diff diff --git a/app.js b/app.js index a3f2c1d..b7e5a92 100644 --- a/app.js +++ b/app.js @@ -3,7 +3,9 @@ function login(user) { if (!user) { return null; } + console.log("debug:", user); return user.token; - // old code } # --- ステージ済みの差分 --- $ git diff --staged # --- 統計のみ表示 --- $ git diff --stat src/app.js | 5 ++++- README.md | 2 +- 2 files changed, 5 insertions(+), 2 deletions(-) # --- ブランチ間比較 --- $ git diff main..feature/login
TIP —

コミット前に必ず git diff --staged で「今コミットしようとしている内容」を見直す習慣を。console.log の消し忘れやデバッグ用コードの混入を防げます。

08

日常ワークフロー

DAILY WORKFLOW

ここまでの7コマンドを実際の作業の流れの中でどう使うか。 朝の業務開始から、夕方の終業まで——典型的な1日のGit操作を時系列で並べてみます。

a-day-with-git 現場での実際の流れ
// 1日のGitフロー
09:00 出社git pull で最新を取得 (Vol.4)
↓
09:30 作業開始ファイルを編集
↓
10:30 一段落git status で状態確認
↓
10:31git diff で変更内容確認
↓
10:32git add で必要なものだけステージ
↓
10:33git commit -m で記録
↓
12:00 昼前git log でその日の作業を眺める
↓
18:00 終業git push でリモートに上げる (Vol.4)
# === 典型的な1日 === # [朝] 最新を取得 $ git pull # [作業] ファイル編集... # [区切り] 状態確認 $ git status -s M src/login.js ?? src/util.js # [区切り] 差分確認 $ git diff src/login.js # [区切り] 必要な分だけステージ $ git add src/login.js # [区切り] コミット $ git commit -m "fix: 認証バグ修正" # この流れを 1日に 3〜10回 回す # それがエンジニアの普通! # [終業] リモートに反映 $ git push # [振り返り] 1日の成果 $ git log --oneline --since="today"
鉄則 —

「小さく、早く、頻繁にコミット」。1日1コミットは少なすぎ。機能単位・修正単位でこまめに記録するのが、後で履歴をたどるときに圧倒的に便利です。
