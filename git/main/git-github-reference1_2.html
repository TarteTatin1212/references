01

バージョン管理とは

VERSION CONTROL

一言で言えば「ファイルの履歴を全部保存しておく仕組み」です。 「やっぱり前の状態に戻したい」「誰がいつ何を変えたかを知りたい」「複数人で同じファイルを編集したい」—— こうしたあるあるの困りごとを全部解決してくれます。

version-control なぜ必要か
// バージョン管理がない世界
report.docx report_v2.docx report_v2_最新.docx report_v2_最新_修正.docx report_v2_最新_修正_本当.docx report_最終版_FINAL.docx report_最終版_FINAL_2.docx
あるある —

こうなった経験、ありませんか? 「最終版」が一番古いという悲劇も。

# Gitがある世界 $ git log --oneline a3f2c1d レビュー指摘を反映 9b8e4a2 序章を加筆 5d6f3a8 図表を差し替え 2c1b0e9 初稿完成 # すべてが1つのファイルで管理され、 # いつでも過去の状態に戻れる
POINT —

バージョン管理は「タイムマシン付きのファイル管理」と覚えればOK。コードでも文書でも、変化するものすべてに有効です。

02

GitとGitHubの違い

GIT vs GITHUB

初学者が一番最初に混乱するポイントです。 結論から言うと、Git=ツール、GitHub=サービス。 料理で言えば「包丁(Git)」と「セントラルキッチン(GitHub)」のような関係です。

git ≠ github 混同しないこと
// 役割の違い
GIT (ツール)
  • あなたのPCにインストール
  • 履歴を記録する本体
  • コマンドラインで動く
  • 無料・オフライン動作
  • 2005年 Linus Torvalds 作
GITHUB (サービス)
  • Web上のホスティングサービス
  • Gitリポジトリを保管
  • ブラウザで操作・閲覧
  • 無料枠あり (一部有料)
  • 2008年〜 Microsoft傘下
# Git だけでも履歴管理はできる $ git init $ git commit -m "changes" # でも… 同僚に共有したい! # PCが壊れたら全部消える! # そこで GitHub の出番 $ git push origin main ↑ GitHub に送る # GitHub以外にも # ・GitLab # ・Bitbucket # といった選択肢があります
TIP —

面接で「GitとGitHubの違いを説明して」は鉄板の質問。「Gitは分散型バージョン管理システムというツール本体で、GitHubはそのGitリポジトリをホスティングするWebサービス」と答えれば満点です。

03

インストールと初期設定

INSTALL & CONFIG

GitはOSによってインストール方法が違います。 インストール後、最初に必ずやることは名前とメールアドレスの設定。これがコミットの「署名」になります。

install OSごとの導入手順
// OS別インストール
WINDOWS

git-scm.com から「Git for Windows」を落としてインストール。Git Bashという黒い画面が付いてきます。

macOS

ターミナルで brew install git 一発。Homebrewが入っていなければ先にそれを導入。

LINUX (Ubuntu系)

sudo apt install git でOK。たいていは最初から入っています。

# インストール確認 $ git --version git version 2.43.0 # --- 初期設定 (一度だけ) --- # 名前 (コミットに表示される) $ git config --global user.name "Taro Yamada" # メールアドレス $ git config --global user.email "taro@example.com" # デフォルトブランチ名を main に $ git config --global init.defaultBranch main # 設定確認 $ git config --list
POINT —

GitHubで使うメールアドレスと同じものを設定すること。これが揃っていないと、GitHub上で「アバターが出ない」「貢献度カウンタに反映されない」といった地味な不都合が起きます。

04

3つのエリア

THREE TREES

Gitの仕組みを理解する最重要コンセプトがこれ。ファイルは ワーキングディレクトリ → ステージング → リポジトリ の3段階を経て記録されます。 このイメージが頭に入っていれば、Gitコマンドは8割理解できたと言ってもいいくらいです。

working / staging / repository ★★★ 必修コンセプト
// 3つのエリアの流れ
① WORKING DIRECTORY あなたが普段編集している作業フォルダ。
エディタで開くあの場所
↓ git add
② STAGING AREA 「次にコミットしたい変更」を一時的に置く場所。
荷物を運ぶ前の積み込み場
↓ git commit
③ REPOSITORY (.git) 履歴として正式に記録される場所。
ここに入ったら戻せる
# たとえばこんな流れ # 1. ファイルを編集 $ echo "Hello" > hello.txt # 2. 状態を確認 $ git status Untracked files: hello.txt # まだ①の段階 # 3. ステージに上げる $ git add hello.txt $ git status Changes to be committed: new file: hello.txt # ②へ移動 # 4. コミットして記録 $ git commit -m "add hello" [main a3f2c1d] add hello # ③に正式記録
POINT —

なぜステージングが挟まるのか? 答えは「コミット単位を自分で選べるから」。10ファイル変更しても、「この3つだけ先にコミット」が可能。これがgit の柔軟さの源です。

05

はじめてのコミット

HELLO COMMIT

理屈はここまで。実際に手を動かして、「git の世界の Hello World」であるはじめてのコミットをしてみます。 たった5コマンドで、あなたのフォルダがリポジトリになります。

first-commit 手を動かす最初の儀式
// ファイル状態の遷移
my-project/ ├─ .git/ ← git init で誕生 │ └─ (内部の管理ファイル群) └─ hello.txt ← あなたが作るファイル [ git init ] ↓ [ ファイル作成 ] ↓ [ git add . ] ↓ [ git commit -m "..." ] ↓ ✓ 最初のコミットが 履歴に刻まれる
注意 —

.git フォルダは絶対に手で触らないこと。中身が壊れるとリポジトリが死にます。

# ① 新しいフォルダを作る $ mkdir my-project $ cd my-project # ② リポジトリとして初期化 $ git init Initialized empty Git repository in .../my-project/.git/ # ③ ファイルを作る $ echo "Hello, Git!" > hello.txt # ④ ステージに乗せる (. は全ファイル) $ git add . # ⑤ コミット! $ git commit -m "first commit" [main (root-commit) a3f2c1d] first commit 1 file changed, 1 insertion(+) create mode 100644 hello.txt # ⑥ 履歴を確認 $ git log
TIP —

コミットメッセージは未来の自分への手紙。「修正」「変更」のような中身のない言葉は避け、「ログインフォームのバリデーションを追加」のように「何をやったか」が一目でわかる文に。詳しくはVol.8で扱います。

06

GitHubに繋ぐ

CONNECT TO GITHUB

ローカルのリポジトリをGitHubに「アップロード」します。これで、PCが壊れてもコードは消えず、世界中どこからでもアクセスでき、他人と共有もできるようになります。 ここから本格的にGitの威力が発揮されます。

push to github ローカル → クラウド
// 全体イメージ
[ あなたのPC ] ┌──────────────┐ │ my-project │ │ .git/ │ └──────┬───────┘ │ git push ↓ ┌──────────────────┐ │ github.com/ │ │ taro/my-project │ └──────────────────┘ ↑ git pull │ [ 同僚のPC ] ┌──────────────┐ │ my-project │ └──────────────┘
手順 (GitHub側)

① github.com にログイン
② 右上 + → New repository
③ リポジトリ名を入力 → Create
④ 表示されるURLをコピー

# ローカル → GitHub に繋ぐ # ① リモートを登録 $ git remote add origin https://github.com/taro/my-project.git # ↑名前 ↑URL # ② ブランチ名を main に $ git branch -M main # ③ プッシュ (アップロード) $ git push -u origin main Enumerating objects: 3, done. Writing objects: 100% (3/3), done. To https://github.com/taro/my-project.git * [new branch] main -> main # これでGitHub上に表示される! # --- 別PCから取ってくるなら --- $ git clone https://github.com/taro/my-project.git
ハマりどころ —

2021年以降、GitHubはパスワード認証を廃止しています。pushしようとして「Authentication failed」で固まる初学者が大量発生中。解決策は「Personal Access Token (PAT)」か「SSH鍵」。詳しくはVol.4で。

学習の進め方

HOW TO STUDY

最後に、このシリーズを最も効率よく自分のものにする使い方を提案します。 Gitは「読むだけでは絶対に身につかない」技術。手を動かす前提で進めてください。

how-to-master 挫折しないための鉄則
// 推奨スタイル
段階やること
1 一読該当の巻をざっと眺める
2 写経コマンドを自分のPCで実行
3 改変少し条件を変えて試す
4 失敗わざとエラーを起こして観察
5 復習3日後に該当章を見返す
鉄則 —

練習用リポジトリを1つ作って、そこで自由に壊す。本番リポジトリで実験すると事故ります。

# 推奨: 練習用フォルダを用意 $ mkdir git-playground $ cd git-playground $ git init # ここで好きなだけ実験 # 失敗してもダメージなし # --- このシリーズの読み方 --- ★★★ ぜったい押さえる ★★ 業務で頻出 ★ 余裕があれば # マーク付きの項目を優先! # --- つまずいたら --- → Vol.9 (困った時のGit編) を引く → エラーメッセージで検索 → git help <command> を見る # Gitは公式ドキュメントが優秀!
最後に —

Gitは「最初の山が高い」けれど、一度越えれば一生使える武器になります。今この瞬間つまずいても、3週間後には平気で git rebase -i を叩いているはず。焦らず、毎日少しずつ。
