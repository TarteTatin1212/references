01

.gitignore — 除外設定

GITIGNORE

「Gitに管理させたくないファイル」を指定する設定ファイル。 node_modules、ビルド成果物、ローカル設定ファイル、機密情報—— こうしたものを誤ってコミットしないための防波堤です。

.gitignore ★★★ プロジェクトの第一歩
// よく除外するもの
種類例
依存関係node_modules/
ビルド成果物dist/ build/
ログ*.log
環境変数.env
IDE設定.vscode/ .idea/
OS生成物.DS_Store Thumbs.db
キャッシュ__pycache__/
POINT —

.gitignore はプロジェクト最初に書く。あとから追加しても既に追跡されてるファイルは消えないので二度手間。

# --- 書き方の基本 --- # ディレクトリを除外 (末尾に/) node_modules/ dist/ build/ # ファイル名で除外 *.log .env # 特定の階層のみ /config.local.js # ルートのみ **/temp/ # どの階層も # 除外の除外 (! で打ち消し) *.log !important.log # これだけは含める # コメント (#) # 設定ファイル .vscode/ # --- 言語別テンプレ --- # Node.js node_modules/ npm-debug.log* .env* dist/ # Python __pycache__/ *.py[cod] *.egg-info/ .venv/ .pytest_cache/ # Java *.class target/ .gradle/ # --- 既に追跡されてる場合 --- $ git rm --cached .env ↑ 追跡解除 (ファイルは残す) $ git commit -m "chore: .envを追跡解除" # --- 一発生成: gitignore.io --- # https://www.toptal.com/developers/gitignore # 「Node, VSCode, macOS」とか指定で # 完成形のテンプレが生成される
あるある —

「うっかり .env をコミットしてしまった!」。これは業界全体の悲劇です。多くの企業がこれで本番のAPIキーを漏洩してきました。プロジェクト開始時に真っ先に .gitignore を整える、これだけで人生が変わります。

02

.gitattributes — 属性設定

GITATTRIBUTES

.gitattributesは「ファイルごとの扱い方」をGitに教えるファイル。 特に重要なのが改行コードの統一です。 WindowsとMac/Linuxが混在するチームで、これがないと毎回全行が変更扱いになる事故が起きます。

.gitattributes ★★ 改行コードの平和を守る
// よくある設定パターン
改行コード問題

Windows: CRLF (\r\n)
Mac/Linux: LF (\n)
→ 違うOSで触ると毎行が差分扱いになる悲劇

解決

.gitattributes で「リポジトリ内はLFに統一」を宣言。WindowsでもチェックアウトでLFに変換される。

POINT —

地味だけどこれがあるのとないので、チーム開発の快適度が雲泥。プロジェクト開始時にセットでおく。

# === .gitattributes 標準形 === # テキストは全てLFに統一 * text=auto eol=lf # バイナリは触らない *.png binary *.jpg binary *.pdf binary *.zip binary # Windows固有のスクリプト *.bat text eol=crlf *.cmd text eol=crlf # 差分表示の言語ヒント *.tsx diff=tsx *.md diff=markdown # --- linguist (言語判定の調整) --- # GitHubの言語統計に出さない docs/* linguist-documentation vendor/* linguist-vendored # 自動生成物 *.lock linguist-generated dist/* linguist-generated # --- 関連設定 --- # Git本体での改行設定 $ git config --global core.autocrlf input Windows: コミット時にCRLF→LF変換 チェックアウト時はそのまま $ git config --global core.autocrlf false Mac/Linux: 何もしない (.gitattributes が効くなら不要) # --- 既存リポジトリに後から導入 --- $ git add .gitattributes $ git add --renormalize . $ git commit -m "chore: 改行コードを統一"
TIP —

「PRを出したら何故か全ファイルが変更扱いになってる」——これは99%改行コード問題。.gitattributes で * text=auto eol=lf を入れて --renormalize でクリーンに。OS混在チームの必須教養です。

03

ブランチ保護ルール

BRANCH PROTECTION

mainブランチに「直pushされたら困る」「レビューなしでマージされたら困る」—— こうした事故を防ぐのがブランチ保護ルール。 GitHubのSettings画面で設定するだけで、ルール違反のpushが機械的に拒否されます。

branch protection ★★★ mainを守る盾
// 設定できるルール
推奨ルール (定番)

◯ PR必須 (直push禁止)
◯ 承認数 1以上
◯ CIパス必須
◯ 会話の解決必須
◯ linear historyを強制
◯ force push禁止
◯ 削除禁止

適用範囲

Branch name pattern:
main — 厳密
release/* — ワイルドカード
* — 全ブランチ

注意 —

管理者(Admin)も例外なく適用にする「Do not allow bypassing」をONにしておくのが鉄壁。

# --- 設定の手順 --- リポジトリ → Settings → Branches → Add branch ruleset (推奨) または Add classic protection rule # --- 設定項目の例 --- Branch name pattern: main [✓] Require a pull request before merging [✓] Require approvals: 1 [✓] Dismiss stale approvals on new commits [✓] Require review from Code Owners [✓] Require status checks to pass [✓] Require branches to be up to date Status checks: - test - lint - build [✓] Require conversation resolution [✓] Require linear history (merge commitを禁止、squash/rebase推奨) [✓] Restrict deletions [✓] Block force pushes [✓] Do not allow bypassing the above settings # --- 試しに直pushしてみる --- $ git switch main $ git commit -m "直接修正" $ git push remote: error: GH006: Protected branch update failed remote: At least 1 approving review is required. # → 守られた!
鉄則 —

新規リポジトリを作ったら「まずブランチ保護」。設定3分でチームのコード品質が一段上がります。「レビューなしのmain直push」が物理的に不可能になる安心感は、一度味わうと手放せません。

04

コミットメッセージ規約

COMMIT CONVENTION

現代の業界標準はConventional Commits。 type(scope): subject という形で書く規約で、 覚えれば履歴がそのまま「変更履歴 (CHANGELOG)」になる強力さがあります。

conventional commits ★★★ 業界標準の書き方
// type 一覧
type意味
feat新機能追加
fixバグ修正
docsドキュメントのみ
style整形(動作変更なし)
refactorリファクタ
testテスト追加・修正
chore雑務(依存更新等)
perfパフォーマンス改善
ciCI関連
◯ 良い例

feat(login): JWT認証を実装
fix(api): 404レスポンス修正
docs: README更新

✗ 悪い例

修正
update
WIP
色々直した

# === 基本形 === type(scope): subject [body] [footer] # --- 各要素の意味 --- type 必須 (feat/fix/docs/...) scope 任意 影響範囲 subject 必須 50字以内、現在形 body 任意 詳細説明 footer 任意 Breaking change / Issue参照 # === 例 === feat(auth): JWTトークン認証を追加 ログインAPIにJWT発行ロジックを実装。 有効期限は1時間、リフレッシュトークン は別エンドポイントで発行する設計。 Closes #42 # === Breaking Change === feat!: User APIのレスポンス形式変更 BREAKING CHANGE: /users の戻り値が 配列から { data: [...] } 形式に変更。 クライアント側の修正が必要。 # === 自動化との連携 === "feat:" → minor (1.X.0) "fix:" → patch (1.0.X) "feat!:" → major (X.0.0) # semantic-release などのツールが # コミットメッセージから自動で: ・バージョン番号を上げる ・CHANGELOGを生成 ・タグを打つ ・リリースを作る
TIP —

コミット規約を強制したいならcommitlint + Huskyを導入。規約違反のコミットを物理的にできなくする仕組みです。最初は窮屈に感じますが、慣れると「全員が同じ書き方」になり、履歴を見るだけで何が起きたか分かる至福の状態に。

05

CODEOWNERS — 自動レビュアー指定

CODEOWNERS

「このフォルダの変更は、この人のレビューを必須に」を設定するファイル。 PRが作成されると、変更ファイルのオーナーが自動でレビュアーに指定されます。 大規模リポジトリでは必須の仕組み。

CODEOWNERS ★★ 責任分担を明示
// 配置場所と動作
my-repo/ ├─ .github/ │ └─ CODEOWNERS ← ここに置く └─ src/ PRが作成されると… 変更ファイル: src/api/login.js ↓ CODEOWNERSを参照 ↓ src/api/* → @backend-team ↓ 自動的にレビュアー指定 "backend-team の承認が必須" 状態に
活用例

◯ フロントエンド/バックエンドで分業
◯ 重要ファイルに必ずシニアが入る
◯ ドキュメントはテックライターが見る
◯ インフラは SRE チームが見る

POINT —

ブランチ保護のRequire review from Code Ownersと組み合わせると強制力が生まれる。

# === .github/CODEOWNERS === # 書式: pattern @user1 @user2 @team # --- 全体のデフォルトオーナー --- * @taro @hanako # --- ディレクトリ指定 --- /src/api/ @backend-team /src/ui/ @frontend-team /docs/ @tech-writers # --- ファイル種別 --- *.js @js-experts *.ts @ts-experts *.css @design-team # --- 重要ファイル --- /package.json @taro @senior-dev /.github/ @devops-team /Dockerfile @devops-team /.env.example @taro # --- 後から書いた方が優先 --- *.md @tech-writers /CHANGELOG.md @release-team ↑ CHANGELOG.md は tech-writers でなく release-team が指定される # --- 否定はできない (除外不可) --- # --- チーム指定 --- @org/team-name の形式 例: @mycompany/backend # --- 動作確認 --- 新しいPRを作成 ↓ Reviewers 欄に自動で @backend-team が入っていれば成功 # --- 注意点 --- パターンマッチに引っかからないファイルは レビュアー指定なし。 → 最初の "* @default" で全体を担保。
現場で —

CODEOWNERS は「責任の所在をコードで宣言する」仕組み。新人が入ってきた時に「このフォルダは誰に聞けばいいの?」が一目で分かる、地味だけど効果絶大なドキュメント代わりにもなります。

06

タグとリリース

TAGS & RELEASES

特定のコミットに「これがv1.0.0」と印を付けるのがタグ。 そのタグをもとに、GitHubでリリースページを作成すれば、配布物として一般公開できます。 バージョン番号はセマンティックバージョニングに従うのが業界の流儀。

tags & releases ★★ バージョンの管理
// セマンティックバージョニング
v 1 . 2 . 3 │ │ │ │ │ └─ PATCH │ │ バグ修正 │ │ (動作互換あり) │ │ │ └───── MINOR │ 機能追加 │ (互換あり) │ └───────── MAJOR 互換性の破壊 (BREAKING CHANGE)
アノテートタグ vs 軽量タグ

アノテート (-a) = メッセージ・署名付き(推奨)
軽量 = 単なる別名

POINT —

リリース用タグは必ずアノテートで。「誰がいつなぜ切ったか」が残る。

# --- タグを作る --- # アノテートタグ (推奨) $ git tag -a v1.0.0 -m "Release v1.0.0" # 軽量タグ $ git tag v1.0.1 # 過去のコミットにタグ $ git tag -a v0.9.0 a3f2c1d -m "..." # --- タグの確認 --- $ git tag v0.9.0 v1.0.0 v1.0.1 $ git show v1.0.0 # --- リモートに送る --- # 個別 $ git push origin v1.0.0 # まとめて $ git push --tags # --- タグの削除 --- $ git tag -d v1.0.0 # ローカル $ git push origin --delete v1.0.0 # リモート # --- GitHub Release作成 --- GitHubのRelease機能: リポジトリ → Releases → Draft new release → Tag指定 (新規 or 既存) → タイトル・本文 → Generate release notes (自動生成) → Publish release # gh コマンドで一発 $ gh release create v1.0.0 \ --generate-notes # --- バージョン番号の決め方 --- v1.0.0 → v1.0.1 (バグ修正) v1.0.1 → v1.1.0 (機能追加) v1.1.0 → v2.0.0 (破壊的変更) # まだ正式版前 = 0.x.y
TIP —

GitHubのRelease機能は 「Generate release notes」 ボタンで前回タグからの差分をまとめてくれます。コミット規約 + ConventionalCommits を併用すれば、ほぼ何もせずに綺麗なリリースノートが完成。これがプロのリリースワークフローです。

07

フォーク開発フロー

FORK WORKFLOW

OSSへの貢献や、書き込み権限のないリポジトリへの提案で使うのがフォーク。 本家リポジトリを自分のアカウントに丸ごとコピーして、そこから改造・PR—— この流れを身につけると、世界中のOSSに貢献する道が開けます。

fork workflow ★★ OSS貢献の入口
// 全体の流れ
[ GitHub上 ] 本家リポジトリ github.com/upstream/lib ↓ Fork ボタン 自分のフォーク github.com/taro/lib [ ローカル ] $ git clone (自分のフォーク) $ git remote add upstream (本家) ↓ ブランチを切る → 変更 → push ↓ [ GitHub ] 自分のフォーク → 本家 へ Pull Request → レビュー → マージ → OSS貢献者デビュー!
POINT —

2つのリモートを登録するのがポイント。origin = 自分のフォーク、upstream = 本家。

# --- ステップ① フォーク作成 --- 本家リポジトリの右上「Fork」ボタンをクリック → 自分のアカウントにコピーが作成される # --- ステップ② クローン --- $ git clone git@github.com:taro/lib.git $ cd lib # --- ステップ③ upstream を登録 --- $ git remote add upstream \ git@github.com:original/lib.git $ git remote -v origin git@github.com:taro/lib.git upstream git@github.com:original/lib.git # --- ステップ④ 機能ブランチ --- $ git switch -c feature/my-fix # 編集 → commit # --- ステップ⑤ プッシュ --- $ git push -u origin feature/my-fix # --- ステップ⑥ PR --- GitHub: 自分のフォーク → "Compare & pull request" base repository: original/lib base: main head repository: taro/lib compare: feature/my-fix # --- ステップ⑦ 本家更新の取り込み --- $ git fetch upstream $ git switch main $ git merge upstream/main $ git push origin main # 自分のフォークのmainを本家に同期 # --- 短縮版 (gh使用) --- $ gh repo fork original/lib --clone $ cd lib $ gh pr create
最後に —

気になるOSSにtypo修正PRを一つ送ってみるのが、フォークフローを覚える最高の練習。「自分のPRが世界中で使われるソフトにマージされた」瞬間は、エンジニア人生の中でも忘れがたい体験になります。

チームのお作法まとめ

新しいリポジトリを始めるとき・新しいチームに入るとき、確認しておくべき項目のチェックリスト。 全部押さえれば「Gitまわりで事故るチーム」から抜け出せます。

team checklist ★★★ 配属時の確認事項
// 新プロジェクト立ち上げチェック
▼ ファイル整備

[ ] .gitignore
[ ] .gitattributes (改行コード統一)
[ ] README.md
[ ] LICENSE
[ ] .github/PULL_REQUEST_TEMPLATE.md
[ ] .github/CODEOWNERS

▼ GitHub設定

Q. 誰がレビュアーになる? Q. レビュー対応の目安時間は? # ④ マージ Q. Squash / Merge / Rebase どれを使う? Q. マージ前のCIは何が必須? Q. 古いブランチは自動削除? # ⑤ CI/CD Q. テストは誰が書く? Q. 本番デプロイのトリガーは? Q. 失敗時の対応フローは? # ⑥ 緊急時 Q. 本番障害時のhotfix手順は? Q. revertの判断基準は? Q. ロールバックの仕組みは? # === 押さえれば「強い人」 === これら全部に答えられるエンジニアは チームから絶大に信頼される。 逆にこれを新人向けに整備した文書を 書ける人は、まさに「テックリード」。
POINT —

これら全部を「リポジトリのREADMEとCONTRIBUTING.md」に書き残しておくと、新メンバーが入ってきた時に迷わない最強のチームになります。「習慣を仕組みで担保する」のがチーム開発の極意。
