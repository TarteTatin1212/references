01

直感に反する事実

COUNTERINTUITIVE TRUTH

「同時に多くこなせば、たくさん終わる」──これは直感的には正しく見えます。しかし現実は逆です。「同時に1つに集中すれば、結果的にもっと早く・もっと多く終わる」。これは経験則ではなく数学的事実です。なぜ我々の直感は外れるのか、そこから話を始めます。

counter.intuition 直感が外れる理由
// 「忙しいのに進まない」の正体
「もっと働けば、もっと終わる」── これは間違っている
— D. REINERTSEN, 製品開発フロー研究
  • 稼働率100% → 必ずキューが滞留する（数学的事実）
  • 稼働率80%でも、待ち時間は急増（待ち行列理論）
  • 「忙しさ」と「成果」は別物
  • 「全員フル稼働」が破滅への道

※ 工場・サービス業・道路交通でも全く同じ現象が観測される普遍法則。

// 直感が外れる原因 @原因1: 線形に考える 「同時に2倍やれば 2倍終わる」 → これは線形思考 現実は非線形 キューが指数的に増える @原因2: 切替コストを忘れる タスク切替コスト = 15〜30分 10タスク同時 → 切替が頻繁に発生 → 実作業時間が削られる @原因3: 個人視点で考える 自分1人なら理解できる → システム全体の挙動は別 // 例: 高速道路 道路に車が少ない: 皆スイスイ走れる 道路が80%混雑: まだ流れる 道路が90%混雑: 渋滞発生 急に流速ダウン 道路が100%: 完全停止 「全員乗れた」が 「誰も進まない」 // ソフト開発も同じ WIP上限なし = 全員フル稼働 = 皆忙しいが進まない
POINT —
「100%稼働」「全員フル活用」は、経営層が好む KPI ですが、科学的には壊滅的。稼働率を意図的に80%程度に保つのがフロー最大化のポリシーです。「遊んでる時間」は無駄ではなく、フローを保つための余白。これを上層部に説明できることが、アジャイル熟達者の証。★★★
02

リトルの法則

LITTLE'S LAW

1961年、MIT のJohn Little がたった1つの数式を証明しました。L = λW。これが「リトルの法則」。製造業・サービス業・コンピュータシステム・道路交通──あらゆる「流れ」を支配する驚くべき普遍法則です。

law.little L = λW の意味
// リトルの法則
L = λ × W
FUNDAMENTAL LAW OF QUEUING
  • L = システム内のアイテム数（WIP）
  • λ = 到着率（単位時間あたりの新規流入）
  • W = 滞在時間（リードタイム）

式を変形すると：
リードタイム = WIP ÷ スループット

「分布の形・処理順序・分散・到着間隔──これらに関係なく、定常状態であれば必ず成立する」
— J. LITTLE, 1961
// 具体的な計算例 @チームの状況 WIP (進行中) = 20 件 スループット = 5 件/週 @リトルの法則で計算 リードタイム = WIP / スループット = 20 / 5 = 4週 // WIP を半分にすると? @WIPを10件に制限 WIP = 10 スループット = 5 件/週 (変わらない) リードタイム = 10 / 5 = 2週 → 半分になる // 何が起きたか 同じスループットのまま リードタイムが半減 → 顧客満足度UP → 機能リリース速度UP // しかも実際は… WIP削減でスループットも上がる ↓ ✓ 切替コスト減 ✓ 集中力UP ✓ コンフリクト減 ✓ レビュー待ち減 → スループット8 件/週へ リードタイム = 10 / 8 = 1.25週 → 3分の1以下に // これがWIP制限の威力
TIP —
リトルの法則は「物が流れるあらゆる場所」で成立します。スーパーのレジ・空港の入国審査・コールセンター・道路交通。ソフトウェア開発も例外ではない──このシンプルな事実が、カンバンの理論的根拠です。「WIP制限がフローを速くする」のは経験則ではなく、1961年から数学的に証明された事実です。★★★
03

なぜ常に成立するのか

WHY ALWAYS HOLDS

リトルの法則の驚くべき点は「ほぼ何の前提も置かない」こと。タスクの大きさ・処理順序・到着間隔──全てがバラバラでも成立します。これは「保存則」に近い、システムの原理的性質。なぜ無条件で成立するのかを直感的に理解しておきます。

law.proof 法則が常に成立する理由
// 直感的説明
時間 → カード1: ●━━━━━━━● カード2: ●━━━━━━━● カード3: ●━━━━━━━● カード4: ●━━━━━━━● │ ↓ この時点での ↓ WIP = 4 ↓ それぞれの ↓ 滞在時間 (W)

全アイテムの合計滞在時間を集計すると：

総滞在時間 = WIP × 平均滞在 = 完了数 × 平均滞在

※ これは保存則。どう測っても辻褄が合う。

// 法則の数学的優美さ @前提が極めて少ない ✗ タスクサイズ均一 ✗ 到着間隔均一 ✗ 処理速度均一 これらいずれも不要 @必要な前提は1つだけ 「定常状態」であること = 入る量と出る量が 長期的に等しい // なぜ前提が緩いか L = λW は 時間平均の関係を述べる どんなにバラついても 長期で見ると 平均は安定する // 例: スーパーのレジ 10時に客10人 / 退店3人 11時に客5人 / 退店8人 … 1日平均で見ると 入る量 = 出る量 → 法則成立 @同様に成立する場所 ✓ 工場の生産ライン ✓ コールセンター ✓ 空港の入国審査 ✓ ATM の列 ✓ Web サーバーのリクエスト処理 ✓ カンバンボード // 重要な含意 「特殊な前提があるのでは?」 → ない 「ソフト開発は違うのでは?」 → 違わない どんな現場でも 必ず成立する
POINT —
「うちは特別だからリトルの法則は当てはまらない」──現場でよく聞く反論ですが、これは間違い。リトルの法則は「特別な状況」を許容しない、究極の一般化された法則。例外は存在しません。WIP を増やすとリードタイムが延びるのは、業種・現場・規模に関係なく成立します。★★★
04

待ち行列理論

QUEUING THEORY

リトルの法則は待ち行列理論（Queuing Theory）という大きな数学分野の一部です。稼働率（utilization）と待ち時間（wait time）の関係を扱うこの理論は、「稼働率を上げるほど待ち時間が指数的に増える」という極めて非直感的な事実を教えてくれます。

queue.theory 稼働率と待ち時間の関係
// 稼働率と待ち時間（M/M/1モデル）
稼働率 ρ相対待ち時間
50%1.0（基準）
70%2.3
80%4.0
90%9.0
95%19.0
99%99.0

※ 稼働率を上げると待ち時間は指数的に爆発。これが「忙しいのに進まない」の正体。

// 待ち行列理論の基本式 M/M/1 モデルでの平均待ち時間: W = ρ / (1 - ρ) × 1/μ ρ = 稼働率 μ = 処理速度 @ρ が 1 に近づくと… 分母 (1-ρ) が 0 に近づく → W が無限大に発散 // 視覚化 待ち時間 │ │ ↗ │ / │ / │ / │ / │_/ 0%──50%──80%──95%──稼働率 80%超で急激に 待ち時間が増える // なぜ稼働率を上げてはダメか @変動性の影響 現実のタスクは サイズも到着もばらつく 稼働率が高いと 少しの変動で キューが溜まる @回復不能 一度溜まったキューは なかなか減らない 「均衡」に時間がかかる // アジャイルでの含意 稼働率 70〜80%を狙う ✓ 余白が変動を吸収 ✓ 緊急対応の余力 ✓ 学習・改善の時間 ✓ 持続可能なペース 稼働率 100%は ✗ 短期は速く見える ✗ 中長期で破綻 ✗ 燃え尽きの罠 // 結論 余白こそ価値である
TIP —
Donald Reinertsen の『The Principles of Product Development Flow』（2009）は、待ち行列理論を製品開発に応用した名著。「変動性を許容する設計」「キューを管理する経済」など、現代カンバン理論の数学的基盤を提供しています。★★★
05

マルチタスクの数学的害

MULTITASKING IS HARMFUL

「マルチタスクは効率を下げる」──これは心理学・脳科学の常識になっていますが、ソフトウェア開発でも数学的に証明可能です。タスクを並行進行させると、「切替コスト」が積算し、全体のリードタイムが線形以上に増えます。

multitask.cost マルチタスクの数学的害
// 単独 vs 並行
[単独実行] タスクA: ━━━━━━━━━ (5日) タスクB: ━━━━━━━━━ (5日) タスクC: ━━━━━━━━━ (5日) 完了タイミング: A=5日, B=10日, C=15日 平均完了: 10日 [並行実行(コンテキスト切替なし)] タスクA: ━━━━━━━━━━━━━━━━━━━━━━ (15日) タスクB: ━━━━━━━━━━━━━━━━━━━━━━ (15日) タスクC: ━━━━━━━━━━━━━━━━━━━━━━ (15日) 完了タイミング: 全部15日 平均完了: 15日 ← 50%遅い

※ 切替コストを入れるとさらに悪化。実態は20%増し以上。

// 切替コストの実測値 @心理学研究 Rubinstein, Meyer, Evans (2001) の実験: 「タスク切替で40%の 時間が失われる」 @プログラミング特有 コンテキスト復帰に さらに時間を要する 「あれ、どこまでやったっけ」 @実用的な観測 並行タスク数を増やすと リードタイムは急増 並行1: 5日 並行2: 12日 (2.4倍) 並行3: 21日 (4.2倍) 並行4: 35日 (7.0倍) // 数学モデル @単純加算モデル 実時間 = 純作業 × n @切替コストモデル 実時間 = 純作業 × n + 切替 × (n-1) @脳の限界モデル 実時間 = 純作業 × n × (1 + c)^n c = 切替の累積効果 → 並行数が増えると 指数的に悪化 // ソフト開発の現実 「複数案件兼任」 → 切替で集中力消費 → コンテキスト復帰で時間 → 結局どれも進まない 「全員フル稼働」 → 余裕なし → 緊急対応で他案件停滞 → 連鎖的遅延 // 鉄則 "One Piece Flow" 1つずつ終わらせる これが最も速い
POINT —
「1つずつ終わらせる方が、結果的に多く終わる」──これは経験則ではなく、数学的に証明される事実。"Stop Starting, Start Finishing"（始めるのをやめ、終わらせるのを始めよ）は、待ち行列理論の必然です。新人に教える時は「数学だから守れ」と説明できます。★★★
06

プッシュ型 vs プル型

PUSH VS PULL

WIP制限を理解する上で重要なのが「プッシュ型」と「プル型」の対比。プッシュ型＝上流が下流に押し付ける。プル型＝下流が必要に応じて引く。トヨタ生産方式の本質はこのプル型への転換にあり、カンバンもまたプル型システムです。

flow.pull プッシュとプルの構造的違い
// プッシュ型 vs プル型
[プッシュ型] 上流 →→→→→ 中流 →→→→→ 下流 ●●●●●● ●●● ● 過剰 滞留 閑古鳥 作業押付けで 下流が処理しきれず溜まる [プル型] 上流 ←←←←← 中流 ←←←←← 下流 ● ●● ●● 要求あり時のみ流す 下流の処理能力に合わせる
側面プッシュプル
主導上流下流
WIP溜まる制限される
変動対応難しい容易
例従来工場トヨタ・カンバン
// プル型がなぜ優れるか @変動への適応 下流が要求した分だけ → 過剰生産なし → 在庫が積み上がらない @フィードバックループ 下流の状態を 上流が見ながら動く → 全体最適に向かう @「カンバン」が情報媒体 カンバンが渡されたら作る 渡されなければ作らない → シンプルな仕組み // ソフト開発での適用 @プッシュ的な現場 PM「来週までにこれやれ」 → 開発者が処理しきれない → タスクが溜まる → 「忙しいけど進まない」 @プル的な現場 開発者「次のタスク取る」 → WIP上限に達すれば停止 → 上流に「詰まってる」が シグナルとして伝わる → 全体が落ち着く // WIP制限はプル型の道具 WIP上限 = プル型を保証する仕組み ↓ WIPが上限なら新規着手不可 ↓ 完了するまで誰も流せない ↓ 完了が速くなる ↓ 全体が速くなる // トヨタの教え 「必要なものを、必要な時に、 必要なだけ作る」 Just In Time の核心
TIP —
プッシュ型の典型が「マネージャがタスクを割り当てる」運用。プル型は「開発者が自分でタスクを引く」。後者の方が遥かに健全です。「タスクを取りに来ない人」「割り当てがないと動かない人」はプル型に向きません。プル型は自律的なチームを前提とします。★★
07

局所最適と全体最適

LOCAL VS GLOBAL OPTIMIZATION

最後に「局所最適 vs 全体最適」の問題。「個々の人を最大効率にすれば、全体も最速になる」は典型的な間違い。Eliyahu Goldrattが著書『The Goal』で示したこの逆説は、現代の組織論の中心的テーマです。

optimization.global 部分最適化の罠
// 工場の例え（『The Goal』）
「個々のマシンの稼働率を上げても、ボトルネック以外の改善は全体には貢献しない」
— E.M. GOLDRATT, "THE GOAL"
  • 各工程が個別に100%稼働 → 在庫が溜まる
  • ボトルネック以外を高速化 → 全体は変わらない
  • 全体最適には「制約理論」が必要
「全体の流速は、最も遅い工程によって決まる」
— THEORY OF CONSTRAINTS（制約理論）
// 部分最適化の例 @「Aさんを最大効率に」 Aさんが速く作る → Bさんの所に溜まる → Bさんが詰まる → 全体は変わらない @「全員フル稼働」 各人忙しい → でも完成しない → 「やってる感」だけ // 制約理論の処方箋 1. ボトルネックを特定 2. ボトルネックを活用 (最大限働かせる) 3. 他工程を従属させる (ボトルネックに合わせる) 4. ボトルネックを強化 (能力を増やす) 5. 次のボトルネックを探す // アジャイル現場の含意 @「全員フル活用」志向 経営層がやりがち → 局所最適の罠 @「ボトルネック中心」志向 本物のアジャイル → QAが詰まってるなら 全員でQAを助けに行く → 自分の専門外OK → これが「全体最適」 // 鉄則 "The goal is to make the team productive, not the individuals" 個人の生産性ではない チームの生産性が目的 // WIP制限はこの考えの実装 WIP制限が 個人最適化を阻む ボトルネック中心の動きを誘発 → 全体最適へ自然に向かう // 結論 WIP制限は単なるルールではない 組織の哲学を変える装置
POINT —
『The Goal』（1984）は工場の現場長を主人公にした経営小説で、制約理論を物語形式で学べる名著。アジャイル熟達者の必読書の1つです。「個別最適化の幻想」を脱却するには、この本の物語を頭に焼き付けるのが一番。「ハーバート（鎖の最も弱い輪）」の比喩は一度読むと忘れません。★★★
