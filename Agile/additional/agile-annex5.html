なぜTDDはテストから書くのか

ANNEX VOL.05 — TEST FIRST AS A DESIGN TOOL

本館 Vol.5 で見たTDD。「テストを先に書く」という業界の常識を覆した手法。実装してからテストを書いた方が普通では？という疑問は当然です。本巻はその答えをKent Beck の原典・認知科学・哲学から掘り下げます。「TDDは設計の道具」──この一文の意味が、本巻を読み終える頃に腹落ちするはずです。

01

TDDの誕生と Smalltalk

BIRTH OF TDD

TDDの起源は意外と古く、1990年代初頭に遡ります。Kent Beck が Smalltalk コミュニティで実践していた「Test First Programming」が原型。「これは新発明ではなく、古い実践の再発見だ」とBeckが書いています。NASA や IBM が1960年代から類似の手法を実践していた事実があるのです。

history.tdd TDDの起源と Kent Beck
// TDDの系譜
1960s
NASA Mercury 計画：各小ステップにテストを書いていた
1990
Kent Beck、Smalltalk でTest First 実践開始
1996
C3 プロジェクトで XP の中核として位置づけ
1998
Kent Beck & Erich Gamma がJUnit を開発
2002
Beck『Test Driven Development by Example』刊行
2010s
業界標準化。ほぼ全ての言語に xUnit 系フレームワーク
「これは私の発明ではない。古いプログラマたちが実践していた知恵を、現代に再導入しただけだ」
— KENT BECK, "TDD by Example"
// NASA の古い実践 1960年代 Mercury 計画: 「先に期待する出力を テストに書く」 「実装はテストを満たす ことを目的に書く」 ↑ TDD の本質と同一 // なぜ忘れられたか @1970-80年代 ソフトが大規模化 → テストは「最後の工程」 に押しやられた → 開発者と分離 @1990年代 Kent Beck が再発見 ペアプロ・XPと統合 → 業界全体へ波及 // JUnit の影響 JUnit (1998): Java 用のテストフレームワーク Kent Beck と Erich Gamma が共著 xUnit ファミリーへ: PyUnit, RUnit, NUnit Mocha, Jest, RSpec → 全言語へ展開 // 結論 TDDは Beck の「発明」ではなく 忘れられた知恵の再導入 そして体系化・普及 これがTDD史の本質
POINT —
Kent Beck の名著『Test Driven Development by Example』（2002）は、TDDを学ぶ全ての人が読むべき原典。多通貨計算を題材に、TDDの一連のサイクルを実演する形式で、200ページほどの薄い本ですが密度が高い。翻訳版『テスト駆動開発』も入手可能です。★★★
02

「設計の道具」という主張

TDD AS A DESIGN TOOL

Kent Beck が繰り返し強調するのは「TDDはテスト技法ではなく、設計技法だ」という主張。この一文がTDDの本質を捉えています。テストを先に書くと「テスト可能なコードを書かざるを得なくなる」。そして「テスト可能なコード＝モジュラーな良い設計」になるのです。

tdd.design TDDが導く設計品質
// TDDが促す設計の特性
  • 疎結合 ── 単体でテストするには分離されている必要がある
  • 高凝集 ── 1つのことだけする関数のほうがテストしやすい
  • 明確なインターフェイス ── 引数と戻り値が明確でないとテスト書けない
  • 副作用の局在化 ── グローバル状態が多いとテストが難しい
「TDDはコードのデザインを駆動する。テストはコードを良い形に導くフィードバック装置だ」
— KENT BECK, "TDD by Example"

※ 「テスト後付け」だと、テストしにくいコードでも「動いてる」と通せてしまう。

// テストしにくいコードの例 // グローバル状態に依存 function processOrder() { const user = currentUser; // テスト時に user 差替不能 } // 多すぎる責務 function checkout() { validate(); calcTax(); saveDB(); sendEmail(); updateStock(); log(); // 何をテストする? } // 副作用ベタ書き function getUser(id) { return db.query(...); // DB なしでテスト不能 } // TDDが導く改善 // 依存注入 function processOrder(user, db) { // 引数で受ける → 差替可 } // 責務分離 class Checkout { validate(order) {...} calcTax(order) {...} execute(order) { // validate と calcTax // を呼ぶだけ } } // 関数の純粋化 function calculateTax(price) { return price * 0.1; // 副作用なし // テストしやすい }
POINT —
「テストが書きにくい」と感じる時、それは「設計が悪い」のシグナルです。「テストを書きやすくするにはどう設計を変えるか？」を考えることで必然的に良い設計に至る。TDDが「設計の道具」と呼ばれる所以です。テストを書く前に設計を考えるのは順序が逆──TDDではテストが設計を導きます。★★★
03

認知科学から見た意味

COGNITIVE PERSPECTIVE

TDDがなぜ機能するか、認知科学的にも説明できます。「ゴールから逆算する」という思考順序が、人間の脳に「集中の構造」を与える。これはSMARTゴールや逆引き計画法と同じ原理です。

cognition.goal-first ゴール先行思考の認知効果
// 2つの思考順序
思考実装先テスト先（TDD）
出発点「何を作る？」「どう完成と判定？」
視点実装者目線利用者目線
判断主観的客観的
過剰実装起きやすい起きにくい
YAGNI意識しないと逸脱自然に守れる

※ 「完成の定義」を先に書くことで、無駄な道に逸れにくくなる。

// 実装先の典型的な迷走 @「とりあえず書く」 function checkout() { // どこから書こう... // 課税は先?後? // エラー処理は? // ログは入れる? → 詰まる → 過剰実装 @「念のため入れとく」症候群 完成判定がないので 「あれもこれも」と 機能を盛りがち → YAGNI 違反 // テスト先の集中力 @明確なゴール 「税込価格を返す」 ← これだけ → 余計な機能を考えない @順次的な視点切替 Step 1: 利用者として 「期待する振る舞いは?」 Step 2: 実装者として 「最小限でどう実装?」 → 視点の役割分担 // SMARTゴール理論との一致 Specific(具体的) Measurable(測定可) Achievable(達成可) Relevant(関連あり) Time-bound(期限あり) ↑ 心理学・経営学で 効果実証されたゴール設定 テストはまさにこれを コードに対して行うもの // 結論 TDDは認知科学的に 「集中の仕掛け」として 機能する
TIP —
心理学者Locke & Latham のゴール設定理論は「具体的で挑戦的なゴール」が成果を最大化することを実証しています。TDDの「失敗するテストを最初に書く」は、このゴール設定理論のコードレベルでの実装と見ることができます。★★
04

テスト可能性が強制する設計

TESTABILITY DRIVES DESIGN

TDDのコアな主張は「テスト可能性（Testability）が設計品質と相関する」こと。これは経験的に観察される事実です。SOLID原則の主要部分は、実は「テスト可能性を担保する」ための原則と読むこともできます。

testability.solid SOLIDとテスト可能性の関係
// SOLID原則とテスト可能性
原則テストへの効果
Single Responsibilityテストが小さく書ける
Open/Closed既存テストを壊さず拡張
Liskov Substitutionモック差替が安全
Interface Segregation必要分だけモックすれば良い
Dependency Inversion依存を差替えてテスト可能

※ TDDを実践すると、結果的にSOLID準拠の設計に近づく。

// テスト可能性の3要素 @1. 観察可能性 (Observability) 振る舞いを確認できるか? ✗ 内部状態を確認する手段なし ✓ 戻り値・公開状態で確認可 @2. 制御可能性 (Controllability) 入力を操作できるか? ✗ ハードコードされた値 ✓ 引数・依存注入で差替可 @3. 分離可能性 (Isolability) 単独で動かせるか? ✗ 他クラスと密結合 ✓ 1つだけで動作可能 // 依存性逆転の例 @テストしにくい (直接依存) class OrderService { submit(order) { // 直接 DB に書く Database.save(order); } } // テストで DB 必須 @テストしやすい (DI) class OrderService { constructor(repo) { this.repo = repo; } submit(order) { this.repo.save(order); } } // テストでモック差替可 // この変換を強制するのが TDD テストから書くから 最初から DI 設計になる
POINT —
「テストしやすいコード」を追求すると、結果的にSOLID準拠の設計になる。逆に「テスト後付け」だと、テストしにくいコードでも動いていれば通せてしまう。TDDは設計品質を強制する──これがBeckらの主張の核心です。★★★
05

フィードバックループの最短化

SHORTEST FEEDBACK LOOP

TDDは「フィードバックを最短にする」戦略でもあります。「数日かけて実装してからテスト」では、間違いに気づくのが遅すぎる。「数分のサイクル」で動かしては失敗、動かしては成功──これが学習速度を最大化します。

feedback.loop フィードバックループの距離
// フィードバックの遅延コスト
検出時点修正コスト
TDDサイクル中（数分）1
同じセッション内（1時間）5
翌日10
レビュー時20
QAでの発見50
本番リリース後100+

※ Capers Jones や Steve McConnell らによる業界統計。早期発見ほど指数的に安い。

// なぜ早期発見が安いか @文脈がまだ頭にある 数分前なら 「あ、ここがダメだ」 → 即修正 数日後なら 「何書いたんだっけ?」 → 思い出すコスト @依存コードがまだ少ない 早期なら 影響範囲が小さい 後期なら 別箇所も依存してる → 連鎖的修正 @心理的負担が違う 即発見: 「次行こう」 本番後発見: 「障害対応...」 「報告書...」 「再発防止策...」 // TDDのループ構造 ┌─→ Red (失敗テスト書く) │ │ │ ↓ 数分 │ Green (実装) │ │ │ ↓ 数分 │ Refactor (改善) │ │ └──────┘ 1サイクル: 5〜15分 // 学習速度の最大化 短いループを高速で回す → 学習の機会が増える → 設計が洗練される → バグが早期発見される // 鉄則 "Make the feedback loop as short as possible"
裏技 —
フィードバックループを短く保つには「テスト実行が速いこと」が前提。テストに10分かかったら、TDDは成立しません。「ユニットテストは1秒以内」「全テストでも10秒以内」が目安。テストが遅くなったら、それ自体がチーム改善のシグナルです。★★
06

テストは実行可能な仕様書

TESTS AS LIVING SPECIFICATION

TDDの副次効果として、「テストが仕様書になる」という現象があります。ドキュメントは古びますが、テストは常に動いて検証される。「Living Documentation（生きたドキュメント）」──これがTDD実践の現場で起きる驚きの効果です。

docs.living テストが仕様書になる
// ドキュメント vs テスト
側面普通のドキュメントテストコード
正確性すぐ古びる動かないと気づく
具体性抽象的具体的
網羅性主要ケースのみエッジケースまで
更新後回しになる必須
信頼性低い高い（動いてる）
「仕様書は嘘をつくが、テストは嘘をつかない」
— アジャイル界の格言
// テストが仕様書になる例 @仕様書での記述 「ユーザーがログインに 失敗した場合、エラー メッセージを表示する」 → 曖昧: - 何回失敗で? - どんなメッセージ? - ロックは? @テストでの記述 describe('ログイン失敗', () => { it('3回失敗で30分ロック', () => { for (let i=0; i<3; i++) { login('wrong'); } expect(login('right')) .toThrow('Locked'); }); }); // テストが語ること ✓ 3回までは失敗OK ✓ 4回目はロック ✓ 30分後に解除 ✓ 動作する保証付き // BDDの源流 Behavior Driven Development Dan North 提唱 TDDのテストを仕様風に書く 例: Cucumber, RSpec Given ログイン画面 When ID/PWを誤入力 Then エラーメッセージ表示 // 結論 テストは 「動く仕様書」であり 「形式知の保存装置」 新メンバーが入った時 仕様書ではなくテストを読めば コードの振る舞いが分かる
POINT —
新メンバーが入った時に「READMEを読め」ではなく「テストを読め」と言える現場は強い。テストが網羅的なら、それが最も正確で最新のドキュメントになります。「テスト = 仕様書」として運用すると、コメントの陳腐化問題が消えます。★★★
07

TDD is Dead 論争

"TDD IS DEAD" CONTROVERSY

2014年、Ruby on Rails の作者DHH（David Heinemeier Hansson）が「TDD is Dead」と宣言し、アジャイル界が大揺れしました。Kent Beck・Martin Fowler との「TDDイズデッド対談」はYouTubeで現存。TDDの過剰適用への警鐘として、この論争は重要な意味を持ちます。

debate.tdd-dead 2014年の歴史的論争
// DHHの主張と反論
「TDDの厳格な信奉者は、テストのためにコードを歪めている。これは本末転倒だ」
— DAVID HEINEMEIER HANSSON, 2014
  • DHH：「TDDは内向きのテスト」になりやすく、過剰なモック地獄を生む
  • DHH：「コードを歪めて100%カバー」するのは本末転倒
  • Beck：「TDDは万能ではない、文脈で選べ」が本来の姿勢
  • Fowler：「テストファースト vs テストアフター」より目的を見よ

※ 「TDD is Dead」シリーズ対談は YouTube で公開され続けている重要資料。

// 論争の構造 @DHHが批判したもの ✗ テスト原理主義 100%カバレッジ強要 ✗ 大量のモック テスト用に内部構造を露出 ✗ システムテスト軽視 ユニットテストばかり @DHHが擁護したもの ✓ システムテスト E2Eで動作を確認 ✓ 判断のあるテスト 全部書くのではない @Beck の応答 「私はTDDを万能と 言ったことはない」 「文脈で適切に選択する」 「過剰適用は問題」 @Fowler の整理 - ユニットテスト 小さく素早く - 統合テスト 接続を確認 - システムテスト 全体を確認 ピラミッド型バランスが理想 // 論争から得た教訓 TDDは道具 目的化してはダメ 状況で使い分ける テストの種類のバランス ユニット偏重は危険 E2E も入れる 原理主義の罠 「100%カバレッジ」は 必ずしも品質を意味しない // 現在の主流 「適切な所でTDD」 「適切な所でテストアフター」 両方をバランス良く
POINT —
「TDD原理主義」はBeck本人も否定する立場。「常に必ずテストファースト」ではなく、「文脈に応じて使い分ける」のが熟練者の姿勢。プロトタイピング段階・探索的コーディング・UI調整など、TDDが不向きな局面もある。道具は道具、目的化しない──このバランス感覚が大事です。★★★
NEXT STEP —
TDDが「設計の道具」であり、認知科学・フィードバック理論・仕様化のすべてに根ざした合理的な実践だと見てきました。最後に「TDD is Dead」論争で万能視への警鐘も加えました。次の別館 Vol.6 は本館 Vol.6 のカンバンに対応し、「なぜWIP制限がフローを速くするのか」をリトルの法則から数学的に解き明かします。「直感に反するが数学的に証明可能」な現象です。
