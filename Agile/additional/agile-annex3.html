なぜScrumはこの形なのか

ANNEX VOL.03 — THE ORIGIN OF SCRUM / FROM RUGBY TO SOFTWARE

Scrumの3-5-3構造はどこから来たのか。実は1986年・日本人の手による論文がScrumの遺伝子です。野中郁次郎・竹内弘高がHarvard Business Reviewに発表した「The New New Product Development Game」。富士ゼロックス・ホンダ・キヤノン・NEC等の日本企業の新製品開発の研究から、Scrumの原型は生まれました。本巻はその系譜を辿ります。

01

1986年・運命の論文

THE 1986 PAPER

1986年1月・Harvard Business Review。野中郁次郎（一橋大）と竹内弘高（一橋大）が"The New New Product Development Game"を発表します。当時、日本企業は「製品開発の速さ」で世界を席巻していました。富士ゼロックス・ホンダ・キヤノン・NEC・エプソン──彼らの新製品開発の秘密を経営学者として解き明かしたのがこの論文。Scrumの遺伝子がここに眠っています。

history.1986-paper 伝説の論文の背景
// 論文の概要
「日本の新製品開発は、リレーのバトンパスではなく、ラグビーのスクラムのように行われている。フィールドを前後しながらボールをチーム全体で運ぶのだ」
— I. NONAKA & H. TAKEUCHI, HBR 1986（意訳）
  • 研究対象：富士ゼロックス・ホンダ・キヤノン・エプソン・NEC・ブラザー工業
  • 当時：日本製品の品質と速度が米国を圧倒していた時代
  • 米国経営学者：「なぜ日本だけこんなに速いのか」を研究
  • 結論：開発手法そのものが根本的に違うと発見

※ 米国型「リレー方式（フェーズごとの引き継ぎ）」と、日本型「スクラム方式（並行協働）」の対比が論文の骨格。

// 論文の発見：日米の対比 @米国型: リレー方式 企画 → 設計 → 製造 →↓ →↓ →↓ 人A 人B 人C バトン渡しで完了 シーケンシャル @日本型: スクラム方式 ┌──────────────┐ │企画担当 │ │設計担当 │ ←同時並行 │製造担当 │ 議論しつつ │マーケ担当 │ 進む └──────────────┘ オーバーラップ // なぜ日本型が速かったか 情報の流動性 全員が全工程の状況把握 → 手戻りが少ない 暗黙知の共有 対面・会話・観察 → 文書化されない知恵 が伝わる 自己組織化 上から指示されず チーム自身で動く // 衝撃 この論文が米国経営学に "日本式"の発見をもたらす → 後にトヨタ生産方式と並び 研究対象となる
POINT —
野中・竹内はその後、『知識創造企業』（1995, "The Knowledge-Creating Company"）でSECIモデルを提唱。世界の経営学に多大な影響を与えました。Scrum は単なる開発手法ではなく、日本の組織論・知識論の伝統を継ぐものなのです。★★★
02

なぜ「スクラム」なのか

WHY "SCRUM"

論文タイトルにある「ラグビーのスクラム」。これがそのままScrumの名前になりました。ラグビーのスクラムとは、両チーム8人ずつが組み合って力を出し合う集団的押し合い。なぜこの比喩なのか。1人の英雄でも、並んでバトン渡しでもなく、密集して同時に力を出す──この姿が日本式開発の本質だったのです。

metaphor.rugby なぜラグビーが比喩になったか
// ラグビーのスクラム
  • 両チーム8人が密集して組む
  • 誰か1人ではなく全員で押す
  • 同時に力を発揮する
  • ボールはチーム全体のもの
  • 方向と進路はチーム内で決まる
「（米国の）リレー方式では遅すぎる。スクラム方式は密集と密接でこそ機能する」
— H. TAKEUCHI, 後年の講演（意訳）

※ なお、英語の "scrum" は名詞・動詞両方に使われ、「集まる」「群がる」のニュアンスも持つ。

// 比喩としての適合性 @野球の比喩 打者・投手の1対1構造 → 個人英雄中心 → 米国的価値観 @リレーの比喩 バトン渡し → シーケンシャル → 1人ずつ完結 → ウォーターフォール的 @ラグビーの比喩 全員で密集して動く → 同時並行 → 全員が全責任 → 日本型開発に最適 // この比喩は何を伝えるか "All for one One for all" ↑ これが Scrum の精神 // Jeff Sutherland の証言 1993年、Easel社で 最初のScrum実装時 野中・竹内論文を再読 → そのまま「Scrum」と命名 // なぜ Scrum が魅力的だったか 短い・響きが良い スポーツの躍動感 集団性のニュアンス
TIP —
Jeff Sutherland は著書『SCRUM 仕事が4倍速くなる"世界標準"のチーム戦術』で、論文との出会いを詳細に語っています。「私はこの論文を10回以上読み返した。日本式開発の凝縮されたエッセンスがここにあった」と。1986年の野中・竹内論文がなければ、現代のScrumは存在しなかったのです。★★★
03

6つの特徴と現代Scrum

SIX FEATURES MAPPED

野中・竹内論文は、日本式開発の6つの特徴を抽出しました。これを現代Scrumと並べると、驚くほど一致します。Sutherland と Schwaber は単に「ラグビー」という比喩を借りたのではなく、6つの特徴をすべてScrumに組み込んでいたのです。

structure.six-features 1986論文の6特徴と現代Scrum
// 6つの特徴 → 現代Scrumへの対応
不安定性
→
挑戦的なスプリントゴール
自己組織化チーム
→
開発者の自己管理
フェーズの重複
→
設計・実装・テスト並行
多重学習
→
レビュー・レトロ・リファイン
微細な制御
→
デイリースクラム
学習の組織転移
→
DoD・コミュニティ・LeSS
// 特徴1: Built-in Instability 「経営者は明確な目標を示す。 が、達成方法は自由に」 → スプリントゴールはPOが示し How は開発者が決める という Scrum の原則 // 特徴2: Self-organizing 「小さな自律的チーム が新製品を作る」 → Scrum チーム = 3〜10人の自己管理チーム // 特徴3: Overlapping Phases 「フェーズの境界が ぼかされる」 → 設計と実装が並行 ウォーターフォール否定 // 特徴4: "Multilearning" 「複数レベルで 学習が起きる」 → レトロ＋レビュー＋ バックログリファインメント // 特徴5: Subtle Control 「微細な制御を保ちつつ 自律を尊重する」 → デイリーで進捗共有 ただし指示はしない // 特徴6: Transfer of Learning 「学びを組織に広げる」 → DoDの組織化 実践共有のコミュニティ
POINT —
Scrumは「米国人が考案したフレームワーク」と思われがちですが、その骨格はほぼ全て日本企業の実践から抽出されたものです。Sutherland と Schwaber が果たした役割は、「日本式開発を学び、ソフトウェア領域に翻訳し、ルール化した」ことにある──これがScrum史の正しい理解です。★★★
04

なぜ役割は3つなのか

WHY THREE ROLES

Scrumには3つの役割しかありません。PO・SM・開発者。「プロジェクトマネージャ」も「アーキテクト」も「QAリード」もない。これは偶然ではなく、責任を3つの独立した軸に分けた意図的な設計です。「何を作るか」「どう作るか」「うまく作れる環境か」──この3軸が独立しているからこそScrumは機能します。

structure.three-roles 3つの責任軸の分離
// 3つの責任軸
軸役割責任
WHATプロダクトオーナー何を作るか・優先順位
HOW開発者どう作るか・実装
HOW WELLスクラムマスター仕組みが機能するか
「これらの役割を1人が兼ねると、責任の境界がぼやけ、いずれかの軸が抜け落ちる」
— SCRUM GUIDE, 解釈

※ 3軸の独立がScrumの肝。1人が兼ねるとScrum-but になる。

// 3軸の独立性の効果 @もしPOがいなければ… → 何を作るか不明 → 開発者が「考える」 → 顧客視点が抜ける → 技術的に面白いが 使われないものを作る @もしSMがいなければ… → プロセス改善の主体不在 → 障害が放置される → POと開発者の対立 が解消されない → チーム機能不全 @もし開発者がいなければ… → 言うまでもなく そもそも作れない // 古典的な「2人1組」 との対比 2人1組（PM+開発リーダ）: PMが全責任を負う → 過負荷 → 失敗時の責任集中 Scrumの3軸: 3つの責任が 並立して独立 → 単一障害点なし → 互いに牽制 // もう1つの設計意図 PO ↔ 開発者: 創造的緊張 SM: その緊張を健全に保つ → 三角形の安定構造
TIP —
「PO兼SM」「PO兼開発者」は最初の混乱期によく見られるアンチパターン。3軸が独立してこそ機能するので、無理にでも分離するのが鉄則。とくにSMが開発者を兼任すると、改善の声を上げにくくなり、Scrumが急速に形骸化します。★★★
05

なぜタイムボックスなのか

WHY TIME-BOXED

Scrumの全イベントには時間制限（タイムボックス）があります。スプリント・プランニング・デイリー・レビュー・レトロ──全て時間で区切られた箱。なぜか。パーキンソンの法則──「仕事は与えられた時間を満たすまで膨張する」を逆手に取った設計です。箱を小さくすれば、仕事はその大きさに収まる。

structure.timebox タイムボックスの心理学的設計
// パーキンソンの法則
「仕事の量は、その仕事に充てられる時間を満たすまで膨張する」
— CYRIL N. PARKINSON, 1957
  • 会議を1時間予約 → 1時間使い切る
  • 会議を15分予約 → 15分で終わる
  • これは人間の心理として観察される現象

Scrumはこの法則を逆利用。タイムボックスを意図的に短く設定することで、議論の濃度を上げる。

// 各イベントのタイムボックス設計理由 @デイリー: 15分 「立つ＋15分」で 雑談混入を防ぐ → 同期だけに集中 @プランニング:最大8h 「1ヶ月スプリントなら8h」 → 1日で終わらせる → 余韻なく実行に入れる @レビュー: 最大4h 「動くものを見せる」 → 長くなるなら 分けるべき症状 @レトロ: 最大3h 「全員で話す」 → 3hを超えると疲労 @スプリント: 1〜4週 「短いほど学習が速い」 // もしタイムボックスがないと… プランニング: 5時間 → 細かい議論が延々 → 結論出ず デイリー: 1時間 → 報告会化 → 全員が疲弊 レトロ: 半日 → 議論が拡散 → 改善行動に至らない // 結論 時間の箱を切る 議論は濃くなる アクションが出やすい これがScrum設計の知恵
POINT —
タイムボックスは「終わらなくても切る」のが鉄則。「議論が途中だから延長」を許すと、すぐ崩壊します。「次回までの宿題」「別途打ち合わせ」に逃がす運用が定番。時間管理は規律の証であり、規律こそアジャイルの真髄です。★★★
06

なぜ1〜4週なのか

WHY 1 TO 4 WEEKS

スプリント期間は「1ヶ月以下」と Scrumガイドに明記されています。なぜ1ヶ月以下なのか。なぜ「2週間がベストプラクティス」と言われるのか。これには人間の認知限界・市場の変化速度・チームの集中力が関係しています。

structure.sprint-length スプリント期間の最適解
// 期間別の特徴
期間長所短所
1週超高速学習儀式比率高い
2週バランス最良—
3週余裕あり変化対応がやや遅い
4週大きな機能軌道修正が遅い

※ 業界統計では2週間スプリントが約7〜8割。これが「ちょうどいい」とされる定番。

// なぜ「1ヶ月以下」が上限か @認知限界の壁 人間は1ヶ月以上先を 詳細に予測できない → 計画精度が落ちる @市場変化の壁 現代の市場は 1〜2ヶ月で前提変化 → 1ヶ月超えると古びる @チームの集中限界 4週連続のスプリントは 振り返りが遠すぎる → 燃え尽き・惰性 // なぜ2週間がベストプラクティス? 儀式 vs 開発の比率: 1週: 儀式比率高い (20%) 2週: 適切 (10%) 3週: 偏り (7%) 4週: 振り返り間隔が長い 心理的サイクル: 人は「2週ペース」で リズムを作りやすい (ジムやサウナ通いも 2週間サイクル定着しやすい) 学習サイクル: PDCA を回す最小単位 として2週間が観察される // 4週から2週への移行 多くのチームが経験する 「4週で始めて2週に短縮」 → 短い方が早く学べる ことを体験する
裏技 —
迷ったら「2週間」から始めること。これが業界の経験則。慣れたら1週間に短縮することで学習速度を上げられる。逆に4週間にすると、ほぼ確実に「2週目に手戻り発覚」のパターンに陥ります。短くする勇気はScrum熟達の指標です。★★

Scrum Guide の変遷

EVOLUTION OF THE SCRUM GUIDE

Scrum Guide は2010年に初版が出され、2020年版が最新。10年間で5回改訂されています。改訂の歴史を見ると、Scrumがどう進化したか・何を捨てたかがわかります。これを知らずに「Scrum」を語ると、古い理解で議論することになりかねません。

// 2020年版の主な変更点 @大幅簡素化 19p → 14p 「軽量さ」を体現 例：「開発チーム」削除 → スクラムチーム1つに @プロダクトゴール導入 プロダクトの長期目標 バックログを束ねる 「北極星」になる @役割名の変更 "Development Team" → "Developers" チーム内チームを否定 @3つのコミットメント プロダクトゴール スプリントゴール Done の定義 // 削除されたもの 2020年版で消えた語 ・「3つの質問」 (デイリースクラム) → 形式に縛られない設計に ・「サーバントリーダー」 (SM) → 「真のリーダー」に ・「Sprint Cancellation」 の詳細記述 // 進化の方向 具体化 → 抽象化 指示的 → 原則的 細則重視 → 柔軟性重視 (原文の精神に近づく方向)
POINT —
古い書籍やネット記事には「3つの質問をデイリーでする」等の旧版ベースの説明が残ります。2020年版では削除されているので、最新Scrumガイドで確認すること。Scrumは生きている文書です。原典をたまに読み直すのが熟達者の習慣。★★★

Scrumの遺伝子を辿りました。日本企業の組織論にルーツがあること、3-5-3の構造に明確な設計意図があること──これを知った上で本館 Vol.3 を読み直すと、Scrumガイドの一行一行が異なる重みで読めるはずです。次の本館 Vol.4 では、Scrumを実用化する燃料──ユーザーストーリーと見積もりに進みます。バックログを「動かす」ための技術です。
