コンウェイの法則

CONWAY'S LAW

1968年、Melvin Conwayが発表した論文に、ソフトウェア工学の最も重要な法則の1つがあります。「システムの設計は、それを設計した組織のコミュニケーション構造を模倣する」──これがコンウェイの法則。スケーリングを語る上で、絶対に避けて通れない真実です。

law.conway 組織がアーキテクチャを決める
// コンウェイの法則
「システムを設計する組織は、必然的にそのコミュニケーション構造を模倣した設計を生み出す」
— MELVIN CONWAY, 1968

具体例：

  • 4つの部署 → 4層アーキテクチャに
  • 3チームに分割 → 3つのサービスに
  • フロント部とバック部分離 → API境界が生まれる
  • 地理的に分離 → 別システム化する

※ この法則は50年以上経つ今も成立。組織がシステムを無意識に決める。

// コンウェイの逆設計 @伝統的アプローチ まずアーキテクチャを設計 → それに合わせて組織配置 @インバース・コンウェイ 欲しいアーキテクチャ → それに合う組織を作る 例: マイクロサービス志向 → 小さな自律チームを作る → 自然と疎結合に // スケーリング時の重要性 @モノリス→マイクロサービス 伝統: 「アーキテクチャを刷新」 組織は変えない → 失敗する 旧来構造に引き戻される 正解: 「組織を分割」 → 自然とマイクロサービス化 → アーキテクチャが追従 // 法則を逆手に取る Inverse Conway Maneuver (逆コンウェイ戦略) 目的のアーキテクチャを得るため 先に組織を変える → コンウェイの法則を利用 // 実例 @Amazon Jeff Bezos の 「2ピザルール」 → 小さなチーム → マイクロサービス化 @Netflix 各機能チームが 自律的サービス担当 → 完全な疎結合化 @Spotify Squad モデル → 各 Squad がサービス保有 // 結論 組織を変えずに アーキテクチャは変わらない スケーリングは 組織設計と技術設計の 不可分な問題
POINT —
「マイクロサービス化したのに、結局モノリスに戻る」現場の正体はコンウェイの法則違反。組織がモノリス的（部署別・階層的）なら、どれだけ技術を変えてもシステムはモノリスに引き寄せられる。アジャイル変革は組織変革そのもの──ここを腹に落とすことが、スケーリング成功の出発点です。★★★
02

ダンバー数の制約

DUNBAR'S NUMBER

人類学者Robin Dunbarの研究が、組織設計に重要な示唆を与えます。「人間が安定した社会関係を保てる人数は約150人」──これがダンバー数。霊長類の脳の大きさと群れサイズの相関から導かれた数。「組織が150人を超えると質的に変化する」事実が、スケーリングの根本的難しさを説明します。

research.dunbar 人間の関係性の認知限界
// ダンバーの階層構造
規模関係性
5人親密圏（家族・親友）
15人シンパシー圏（深い友人）
50人友人圏
150人知人・安定した関係
500人名前と顔が一致
1500人顔は知ってる
「人間の脳のサイズから推定される関係性の上限は、約150人である」
— R. DUNBAR, 1992
// アジャイル組織への含意 @1チーム (〜10人) Scrumの推奨範囲 全員が深い関係 → 信頼で動く @10〜50人 (2〜5チーム) 顔と名前が一致 → LeSS の範囲 → シンプル運用可能 @50〜150人 Tribe（Spotify）規模 関係性は維持可能 → ART (SAFe) もこの範囲 @150人超え 個別の関係性は崩壊 形式的ルールが必須 → スケーリングフレームワーク が必要になる地点 // 歴史的な実証 古代の集落: 100〜200人で分裂 ダンバー数で説明可 軍隊の編成: ローマ百人隊 = 80人 現代の中隊 = 100〜150人 → ダンバー数に近い 企業の組織: 150人を超えると 雰囲気が劇的に変化 → 「会社が変わった」感 // なぜ150人で変質するか コミュニケーションパス: 150人 × 149 / 2 = 11,175 パス → 全員を知ることは不可能 → 階層・部署が必須に → 形式的ルールが増える @結果 ✗ 信頼が薄まる ✗ ルールで動く ✗ 階層が深まる ✗ アジャイルが難しくなる // 対応戦略 小さく保つ 150人以下に分割 「会社内会社」構造 → ダンバー範囲を維持 階層化を受け入れる 超えるなら サブ組織を作る → Tribe / Area / Division
TIP —
Spotify モデルの Tribe（40〜150人）はダンバー数に基づいて設計されています。「1つのTribeは150人を超えない」という制約は、生物学的・人類学的に正しい設計判断。「組織サイズに自然な限界がある」と知っておくと、スケーリング設計の指針になります。★★
03

ブルックスの法則

BROOKS'S LAW

1975年、Frederick P. Brooksが著書『人月の神話』で示した有名な命題。「遅れているプロジェクトに人を追加すると、さらに遅れる」──直感に反するこのブルックスの法則は、ソフトウェア開発の根本的な難しさを示しています。スケーリングの根本的制約です。

law.brooks 人を増やせば進むという幻想
// ブルックスの法則
「Adding manpower to a late software project makes it later.」
（遅延中のソフトウェアプロジェクトに人を追加するとさらに遅れる）
— F.P. BROOKS, "THE MYTHICAL MAN-MONTH", 1975
  • 教育コスト ── 新人を教える時間で既存メンバー停止
  • コミュニケーションパス爆発 ── n人で n(n-1)/2 パス
  • 分割不能タスク ── 「9人で1ヶ月作っても、1人で9ヶ月かかる仕事は短縮不能」
// 法則の3つの理由 @理由1: 教育コスト 新人を即戦力化するには 既存メンバーが教える必要 → 既存メンバーの時間消費 → 短期的に生産性ダウン @理由2: コミュニケーション爆発 チームサイズ N パス数 = N(N-1)/2 5人: 10パス 10人: 45パス 20人: 190パス 50人: 1,225パス → 増員ほど指数的増加 @理由3: 分割不能タスク Brooks の名言: 「9人の妊婦を集めても 1ヶ月で赤ちゃんは生まれない」 ソフトには 本質的に並列化できない部分がある → 人数を増やしても短縮不能 // スケーリングへの含意 「人を増やせば成果が増える」 は幻想 真の課題: ✓ 人を増やしながら 生産性を維持する仕組み ✓ コミュニケーションコストを 抑える組織設計 // 解決策（現代版） @小チーム×多数 10人×10チーム = 100人 各チーム独立 → コミュニケーション局所化 @明確な境界線 チーム間はAPI契約のみ 内部実装は独立 → コミュニケーション簡略化 @非同期コミュニケーション ドキュメント主体 会議を最小化 → スケール耐性向上 // 古典に学ぶ 『人月の神話』(1975) は 50年経った今も名著 ソフトウェア工学の本質を 最も正確に語る古典
POINT —
Brooks自身が後の改訂版で「ブルックスの法則は単純すぎた」と認めましたが、本質は変わりません。「人月（Man-Month）」という単位が幻想だ、という主張は正鵠を射ています。「人と月の交換可能性」を前提とした見積もりは、ほぼ確実に失敗します。★★★

LeSS vs SAFe 思想対立

LESS VS SAFE PHILOSOPHY

本館 Vol.8 で扱ったLeSSとSAFe。両者は単なる方法論の違いではなく、根本的に異なる哲学に基づいています。「シンプルさを守る」LeSS、「既存組織との整合性」を取るSAFe。この対立を理解すると、フレームワーク選びが本質的に変わります。

philosophy.less-vs-safe 2大フレームワークの思想的対立
// 哲学的対立
軸LeSSSAFe
哲学「Scrumを大きく」「企業に合わせる」
原則シンプルさ最優先包括的網羅
役職最小化多階層化
既存組織変革する温存する
導入難易高い（文化変革）低い（既存に乗る）
批判される点「現実的でない」「アジャイル風」
「SAFeは大企業のための"アジャイル風"アプローチ。本物のアジャイルとは異なる」
— LeSS派の典型的批判
// 対立の本質 @LeSSの主張 「アジャイルは 組織を変えるもの」 既存階層を温存したまま アジャイルは不可能 → 既存組織と戦う覚悟が必要 @SAFeの主張 「現実の企業は 一夜にして変わらない」 既存組織と整合性ある 段階的アプローチが必要 → 移行期の現実解 // 両者の支持基盤 LeSS支持層: ✓ アジャイル原理主義 ✓ Scrum熟練者 ✓ スタートアップ ✓ 比較的小規模組織 SAFe支持層: ✓ 大企業の経営層 ✓ コンサル ✓ 規制業界 ✓ 既存階層を残したい組織 // 業界の評価 @LeSSへの批判 ✗ 理想主義すぎる ✗ 現実の大企業に 導入困難 ✗ コンサル支援少ない @SAFeへの批判 ✗ 階層が多すぎ ✗ 「アジャイル」と呼べるか ✗ "Dark SAFe"と揶揄 // 本質的な問い スケーリング = 「アジャイルを大きくする」? それとも 「既存組織に合わせる」? → 答えは組織によって異なる // 折衷的解 「段階的に LeSS 方向へ」 最初は SAFe風 徐々にシンプル化 最終的に LeSS風へ → 多くの組織の現実的進化 // 結論 両派の対立は 「組織変革の覚悟」に 帰着する
裏技 —
Craig Larman は「SAFe is a Cargo Cult」と痛烈に批判する一方、SAFe支持者は「LeSS は理想論」と反論します。この対立は「アジャイルの本質をめぐる議論」であり、業界の健全な緊張関係。どちらが正しいかではなく、「自組織は何を選ぶか」を意識的に決めることが大事です。★★★

Team Topologies

TEAM TOPOLOGIES

近年の組織設計論の決定版として注目されているのが『Team Topologies』（Skelton & Pais, 2019）。4種類のチーム型と3種類のインタラクションモードでスケール組織を設計する理論。「コンウェイの法則を逆手に取る」具体的処方箋として、多くの企業が採用しています。

org.team-topologies 4種類のチームと3つの相互作用
// 4つのチームタイプ
  • Stream-aligned Team ── 価値の流れに沿った主役チーム
  • Enabling Team ── 他チームの能力を高めるコーチ役
  • Complicated Subsystem Team ── 高度な専門領域担当
  • Platform Team ── 共通基盤を提供

そして3つのインタラクションモード：

  • Collaboration ── 密接に共働
  • X-as-a-Service ── サービスとして提供
  • Facilitating ── 一時的に支援
// 各チームタイプの役割 @Stream-aligned Team 「価値を流す主役」 顧客機能を作るチーム 自律的に動ける 全体の過半数を占める @Enabling Team 「コーチ役」 特定領域の専門知識 Stream-aligned を支援 期限付きで関わる @Complicated Subsystem 「専門家集団」 ML、暗号、決済など 深い専門性が必要な領域 少数精鋭 @Platform Team 「共通基盤」 内部向けプロダクト 他チームの生産性向上 API/SDKを提供 // 重要な原則 認知負荷の管理 各チームの認知負荷を 許容範囲内に保つ 認知負荷オーバー → 効率激減 → 不健全な状態 Fast Flow 価値の流れを速くする Stream-aligned が 他チームに依存せず 完結できる構造 // 設計例 @典型的な構成 (50人規模) Stream-aligned 5チーム × 6人 = 30 Platform 1チーム × 6人 = 6 Enabling 2チーム × 4人 = 8 Complicated Subsystem 1 × 6 = 6 合計約 50 人 // 適用の効果 @明確化 各チームの責任が明確 「これはどこの担当?」消失 @フロー高速化 Stream-aligned が 独立して動ける → 価値提供が速い @認知負荷適切化 各チームの担当範囲が 人間の認知に収まる // 採用企業 ING (蘭金融大手) Cargill, Comcast 日本でもメルカリ等が採用
POINT —
『Team Topologies』は近年最も影響力のある組織設計書。コンウェイの法則を逆手に取り、「欲しいシステム構造」から「組織構造」を逆算する方法論を提示。SAFeやLeSSの「枠組み」とは別軸の補完的理論として、現代のスケーリングで必須の知識です。★★★

文化が戦略を食う

CULTURE EATS STRATEGY

Peter Drucker の格言「Culture eats strategy for breakfast（文化は朝食に戦略を食う）」。どんなに完璧なアジャイル戦略を立てても、組織文化に逆らえば敗北するという真実。スケーリング失敗の大半は技術ではなく文化の問題です。

culture.eats-strategy 文化と変革の関係
// 文化が戦略を食う構造
「Culture eats strategy for breakfast.」
（文化は朝食に戦略を食う）
— PETER DRUCKER（とされる）
  • 戦略：意識的・短期に変えられる
  • 文化：無意識的・長期でしか変わらない
  • 両者が衝突 → 必ず文化が勝つ
  • 戦略は文化に従属する

※ 実は Drucker 本人の発言記録は曖昧。だがメッセージは業界に深く浸透している。

// 文化抵抗の典型例 @「アジャイル導入」と決定 経営層が決定 研修実施 ツール導入 → 形は揃う @しかし文化が変わらない ✗ 失敗を許さない雰囲気 ✗ 上意下達の慣習 ✗ 個人ノルマで評価 ✗ 部署間の対立構造 @結果 アジャイルが形骸化 「やってる体」になる 数年後、元の姿に戻る // 文化変革に必要なもの @時間 最低 3〜5年 即効性を求めると失敗 @経営層のコミット 本物の変革意志 「やれ」ではなく 「自分も変わる」 @象徴的アクション 目に見える変化 例: 役職の廃止 例: フロア・席の再配置 例: 評価制度の刷新 @心理的安全性 失敗を許す 本音を言える 実験を歓迎する // Schein の組織文化モデル 3つの層: 1. 人工物 (見える) 服装・空間・話し方 2. 価値観 (意識的) 理念・方針・スローガン 3. 基本的前提 (無意識) 「当たり前」とされる信念 ↑ ここが本物の文化 // 変革の階層 @表層変革 ツール導入 研修実施 プロセス変更 → 効果は限定的 @深層変革 価値観の刷新 基本的前提の変更 → 数年単位 → 強い意志が必要 // 結論 スケーリングは 文化変革プロジェクト これを軽く見ると ほぼ確実に失敗する
TIP —
Edgar Schein の『組織文化とリーダーシップ』は組織文化研究の古典。「文化は時間をかけて作られ、時間をかけてしか変わらない」という基本前提を理解せず、「3ヶ月でアジャイル変革」を約束するコンサルは怪しい。「変革は5年計画」と腹を括る経営層こそ本物です。★★★
07

ソシオテクニカル・システム

SOCIO-TECHNICAL SYSTEMS

最後に最も本質的な概念を。ソシオテクニカル・システム──ソフトウェア開発は「人と技術が織りなす一体のシステム」であり、両者を分離して扱うことは不可能、という思想。1950年代に英国Tavistock研究所が提唱した古典が、現代のスケーリング理論の底流を流れています。

theory.socio-technical 人と技術の不可分性
// ソシオテクニカルの含意
「組織は社会的システムと技術的システムの相互作用の上に成り立つ。一方だけを最適化しても全体は改善しない」
— TAVISTOCK INSTITUTE, 1950s
  • 技術だけ変える → 人がついていけない・崩壊
  • 組織だけ変える → 技術が制約となり頓挫
  • 両方同時に変える必要がある

※ コンウェイの法則・Team Topologies・DevOps──全てこの思想の流れにある。

// 4つの組み合わせ @パターン1: 技術△ 人△ → 何も変わらない 伝統的な企業 @パターン2: 技術○ 人△ → 一時的に進む だが長続きしない 「マイクロサービス化したが 組織は旧来のまま」 → 結局モノリスに戻る @パターン3: 技術△ 人○ → 限定的な改善 「アジャイルチーム」できるが 技術的負債で制約 → 速度が出ない @パターン4: 技術○ 人○ → 真の変革 Eliteチーム 持続的改善 → DORA高パフォーマンス層 // 歴史的視点 1950s: Tavistock 研究所 炭鉱の事例研究 「機械化だけで改善せず」 → 社会システムも変える必要 1968: コンウェイの法則 同じ洞察をソフトウェアに 2010s: DevOps運動 Dev と Ops の壁を消す これも同じ思想 2019: Team Topologies ソシオテクニカルの具現化 // アジャイル全体の系譜 ソシオテクニカル思想 │ ├ Scrum (人の組織化) ├ XP (技術プラクティス) ├ Kanban (フロー設計) ├ DevOps (技術-運用統合) ├ Team Topologies (構造化) └ ... // 結論 「アジャイル」と「DevOps」と 「スケーリング」は 同じ問いへの異なる答え その問いとは: 「人と技術を どう統合するか?」 これがソフトウェア開発の 最も根本的な問題 // 別館9巻への橋 ではなぜ アジャイルは失敗するのか? 答え: 多くの組織が ソシオテクニカル視点を 持たないから 詳細は次の Vol.9 で
POINT —
「人と技術を統合する」視点こそ、本シリーズ全体を貫く底流。Scrum（人）・XP（技術）・カンバン（フロー）・CI/CD（自動化）・DevOps（文化）・Team Topologies（構造）──すべてがソシオテクニカル思想の異なる表現です。これに気づくと、アジャイル全体の森が見えるようになります。★★★

スケーリングの本質的難しさを辿りました。コンウェイ・ダンバー・ブルックス・Drucker・Tavistock──50〜70年前の知見が、今もアジャイル組織論を支えています。次の別館 Vol.9 はシリーズ最終巻「なぜアジャイルは失敗するのか」。Cargo Cult Agile・「やってる感」症候群・これからのアジャイル──業界の影の部分に光を当て、シリーズ全体を締めくくります。
