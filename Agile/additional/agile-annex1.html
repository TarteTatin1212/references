AGILE REFERENCE VOL.01 / 09 ANNEX — WHY

なぜアジャイルは生まれたのか

ANNEX VOL.01 — THE BIRTH OF AGILE / HISTORICAL ORIGINS

本館 Vol.1 で見た「アジャイル」という言葉。当たり前のように使っていますが、その誕生はわずか2001年です。なぜこの時代に生まれたのか。それ以前は何が問題だったのか。スノーバードの山荘で何が起きたのか。本巻は歴史の縦糸を辿り、アジャイル誕生の必然性を解きほぐします。これを知ると、宣言の68語が違って見えてきます。

01

1968年「ソフトウェア危機」の宣告

THE SOFTWARE CRISIS

アジャイル誕生の33年前、すでに業界は危機を宣告されていました。1968年、ドイツ・ガルミッシュで開催されたNATO主催のソフトウェア工学会議。ここで初めて「ソフトウェア危機（Software Crisis）」という言葉が公式に使われます。要件が膨らみ、納期が遅れ、品質は劣化する──現在も続く業界の宿痾が、半世紀前に言語化されていたのです。

history.crisis 1968年・NATO会議という分水嶺
// 危機の現象
1950s
ハードが主役。ソフトはおまけ扱い。個人芸で書かれていた
1960s
大規模化が進む。OS/360 開発が予算3倍・人月数千の大炎上
1968
NATO会議で「ソフトウェア危機」宣告。「ソフトウェア工学」誕生
1975
F.P.ブルックス『人月の神話』刊行。OS/360の教訓を世に

※ この時代の課題は「いかに大規模開発を予測通り進めるか」──ここから工学的アプローチが花開きます。

// 危機の症状（1960年代後半） @大規模化 10人の開発 → 100人の開発 → コミュニケーション 爆発的増加 → 全員の認識を揃えるのが 至難の業 @複雑化 単純な計算 → OS・DB・GUI → 想定外の挙動 → デバッグ困難 @納期遅延 「見積もり 6ヶ月」 → 実際 18ヶ月 → 信頼の崩壊 @品質劣化 完成しても動かない リリース後にバグ続出 // 出典 NATO Conference on Software Engineering 1968, Garmisch, Germany P. Naur & B. Randell 編
POINT —
「ソフトウェア工学」という用語が生まれたのもこの会議です。「土木工学・機械工学のような体系」をソフトウェアにも与えようとした努力の出発点。ウォーターフォールはこの努力の一形態として登場したのであって、最初から「悪者」だったわけではないのです。★★★
02

ウォーターフォール誤読史

THE MISREAD OF ROYCE 1970

アジャイルの仮想敵とされるウォーターフォール開発。その元祖とされるWinston W. Royce の 1970年論文には、驚くべき事実が隠されています。Royce 自身が論文の中で「この単純なモデルは危険であり、失敗する」と明言していたのです。ウォーターフォールは反面教師として提示された図──それが世界中で正反対に読まれた歴史こそ、アジャイル誕生の遠因です。

history.royce Royce 1970論文の本当の主張
// 論文の本当の構造
  • 論文冒頭：単純な線形モデルを提示
  • 続く一節：「このやり方はリスクが高い」
  • 論文後半：反復・先行プロトタイプ・顧客関与を提案
  • 結論：「私が主張するのは反復型だ」
「私はこの方式（最初に示した線形モデル）を信じない。失敗を招くから」
— W.W. ROYCE, 1970（意訳）

※ 業界は最初の図だけを引用し、Royce の警告を無視して「これがベストプラクティスだ」と広めた。

// 論文タイトル "Managing the Development of Large Software Systems" W.W. Royce, 1970 IEEE WESCON // 論文の主張ステップ Step1. 単純なモデルを図示 要件→設計→実装→ テスト→運用 (これが後の"ウォーターフォール") Step2. 「これは危険」と明言 "risky and invites failure" Step3. 改善案を提示 ✓ 各工程の反復 ✓ 顧客の関与 ✓ 先行プロトタイプ ✓ 事前設計 ✓ テスト計画の早期 // 何が起きたか 論文の1枚目の図だけが 独り歩きし、Royce の警告は 忘れられた // 結果 30年間、業界は "単純化されたモデル"を 正解と信じ込んだ
TIP —
Craig Larman & Victor Basili の論文「Iterative and Incremental Development: A Brief History」(2003)はこの誤読史を詳細に追跡した名著。反復型開発は1950年代から実践されていたこと、ウォーターフォールは「正解として提示されたものではない」ことを綿密に検証しています。★★
03

1990年代の軽量手法群

LIGHTWEIGHT METHODS ERA

1990年代、現場の不満が臨界点に達します。分厚い手順書ベースの開発手法（CMM・ISO等）への反動として、現場発の「軽量手法（Lightweight Methods）」が次々と誕生。Scrum・XP・Crystal・DSDM・FDD・ASD──これらが後の宣言の母体になります。「アジャイル」という言葉が生まれる前から、思想は各地で芽吹いていたのです。

history.lightweight 1990年代に各地で生まれた軽量手法
// 軽量手法の系譜
1986
野中・竹内"The New New Product Development Game" HBR
1993
Jeff Sutherland、最初のScrum実装（イーゼル社）
1994
DSDM 発表（英国・反復型）
1995
Schwaber & Sutherland、Scrum論文OOPSLA発表
1996
Kent Beck、C3プロジェクトでXP確立
1997
Alistair Cockburn、Crystalシリーズ提唱
1999
Beck『Extreme Programming Explained』刊行
1999
Jim Highsmith『Adaptive Software Development』刊行
// 軽量手法たちの共通項 @反復型 数週間で動くものを出す ↑ ウォーターフォールへの反動 @人間中心 プロセスより個人を重視 ↑ CMM等への反動 @顧客近接 契約より協調 ↑ 受託地獄への反動 @変化歓迎 事前計画より適応 ↑ 大規模設計への反動 // それぞれの特徴 Scrum: チームの動かし方 XP: 技術プラクティス Crystal: チーム規模別調整 DSDM: 反復+優先順位制御 FDD: フィーチャー単位開発 ASD: 適応的開発・推測〜学習 // 共通の悩み 「同じ思想なのに ばらばらに呼ばれ ばらばらに広まり 互いに対立して見える」 → 統一の必要性が 業界内で高まっていた
POINT —
特に1996年のC3プロジェクト（クライスラー社の給与システム）でKent Beckが実践したXPは、業界に衝撃を与えました。「動くものを毎週リリース」「テストを先に書く」「ペアプログラミング」──全部が当時の常識を覆す実践でした。★★★
04

2001年・スノーバード会議の真実

SNOWBIRD 2001 — WHAT REALLY HAPPENED

2001年2月11日〜13日、米国ユタ州スノーバードのスキーリゾート。17人の開発者・著作家が集まりました。目的は「軽量手法の共通点を見出すこと」。スキーをし、議論し、ホワイトボードに殴り書きをし、2日半で68語の宣言を仕上げる──業界の歴史を変えた奇跡の3日間です。

history.snowbird 2001年2月、ユタの山荘で起きたこと
// スノーバード会議・3日間の記録
Day 1
17人が到着・自己紹介。「我々の共通点は何か」を議論。各自の手法を持ち寄る
Day 2
午前はスキー。午後に集中議論。「価値観」を4つに整理することで合意
Day 3
文言を磨き上げ、宣言完成。会議後に「12原則」をメールで策定
「私たちは皆、お互いに敵対していた競合する手法の提唱者でした。それでも、共通の核心があると信じていました」
— ROBERT C. MARTIN, 後年の回想

※ 招集を主導したのは Robert C. Martin（Uncle Bob）。彼の呼びかけで集まった。

// 出席した17人 @Scrum 系 Ken Schwaber Jeff Sutherland Mike Beedle @XP 系 Kent Beck Ward Cunningham Ron Jeffries James Grenning Robert C. Martin @DSDM 系 Arie van Bennekum @FDD 系 Jon Kern @Crystal 系 Alistair Cockburn @ASD 系 Jim Highsmith @ Pragmatic 系 Andrew Hunt Dave Thomas @その他 Brian Marick Martin Fowler Steve Mellor // 全員白人男性17名 これが後年「多様性なし」 と批判される論点に
裏技 —
会議の費用は17人が自腹でした。誰かのスポンサーがついた訳ではなく、純粋に「業界を良くしたい」という有志の集まり。会議は2日半・宣言の作成は数時間と言われています。偉大な文書ほど短く、早く生まれる──これも一つの真実です。★★
05

「軽量」から「アジャイル」へ — 言葉の選定

FROM LIGHTWEIGHT TO AGILE

実はスノーバード会議に集った人々は、自分たちを「軽量手法（Lightweight Methods）」の提唱者と呼んでいました。しかし会議で議論するうち、「軽量」という言葉が誤解を生むと気づきます。「軽い＝中身が薄い＝劣る」と読まれてしまう。代わりに選ばれたのが"Agile"──「機敏」「適応性が高い」を意味する言葉でした。

history.naming なぜ「アジャイル」が選ばれたか
// 候補と却下理由
候補意味却下理由
Lightweight軽量軽い＝劣ると誤読される
Adaptive適応的Highsmithの手法名と重複
Iterative反復的すでに使われ陳腐化
Pragmatic実用的抽象度が高すぎる
Agile機敏—（採用）
「『軽量』は『重要でない』と聞こえる。我々がやっていることは重要だ。『アジャイル』が正しい」
— MIKE BEEDLE, 当時の発言（意訳）
// "Agile" の語源と含意 語源: ラテン語 "agere" = "行動する・動かす" 英語の意味: ✓ 機敏な ✓ 素早く動ける ✓ 柔軟に対応できる ✓ 軽快な // なぜこの語が刺さったか @軽さ + 機動性 単に軽いのではなく 動ける軽さを示唆 @適応性 状況に応じて 形を変えられる @誇り 「軽量」と違って 能力の高さを示す // 選定後の影響 「アジャイル開発宣言」 「アジャイル原則」 「アジャイル手法」 「アジャイルコーチ」 「アジャイル組織」 → "Agile" が形容詞・名詞化 業界の共通語彙へ
POINT —
名前は運動の運命を決めます。もし「軽量開発宣言」と命名されていたら、これほど普及しなかったでしょう。"Agile"という一語が持つ機敏さと前向きさのニュアンスが、業界・経営層・経営学にまで広がる土台を作りました。命名の力です。★★★

宣言の17人 — その後の道

THE 17 AUTHORS — THEIR PATHS

スノーバードの17人は、その後それぞれの道を歩みました。Scrum派・XP派・職人主義派に分かれ、時に協調し、時に対立し、アジャイルというムーブメントを育てつつ揉ませてきたのです。彼らの「その後」を知ると、現在のアジャイル界の派閥構造が理解できます。

07

なぜ宣言は時を超えたか

WHY THE MANIFESTO ENDURES

宣言から25年経った現在も、一字一句変えられていません。これは奇跡的なことです。技術書なら数年で陳腐化する業界で、宣言だけは変化を超越して残った。なぜか。答えは宣言の構造にあります。具体的手法を書かず、価値観だけを提示したからこそ、時代を超えられたのです。

history.endurance 宣言が古びない構造的理由
// 宣言が残る3つの理由
  • 抽象度が高い — 具体的手法を書かなかった
  • 普遍的な人間性に依拠 — 対話・協調・適応は不変
  • "より価値を置く"形式 — 一方を否定せず、優先順位を示すだけ
「アジャイル宣言は10年経っても20年経っても古びない。なぜならそれは『道具』について書いていないからだ」
— MARTIN FOWLER, 10周年ブログ（意訳）

※ 一方で「手法」は変化を続けている。Scrum Guide は2020までに5回改訂・SAFeは大規模化、Heart of Agile は削ぎ落とし──運動は生きている。

// 宣言の構造的強さ @抽象 vs 具体 宣言：抽象的価値観 手法：具体的やり方 → 手法は変わるが 宣言は不変 @排他 vs 比較 「Aを使え」 → Aが古くなれば終わる 「Bより A」 → 関係性は不変 // 宣言の構文設計 A よりも B に価値を置く ↑ 比較構造のため 時代に依存しない // 25年間で変わったもの ハードウェア性能 ×1000倍 言語・フレームワーク 数百 ツール 数千 // 変わらないもの ✓ 人と人の対話の価値 ✓ 動くものへの信頼 ✓ 顧客との協調 ✓ 変化への適応 // 結論 ソフトウェア開発は 人間の営みである ゆえに人間が変わらない限り 宣言も色褪せない
POINT —
本館 Vol.2 で見る「4つの価値・12原則」は、この抽象度の高さこそが武器。手法と違って具体性を切り捨てたからこそ普遍を獲得したのです。これは哲学・宗教経典・憲法と同じ構造。「具体性」と「永続性」はトレードオフ──宣言は永続を選んだ文書です。★★★
NEXT STEP —
歴史の縦糸が見えたところで、次の別館 Vol.2 では宣言と12原則の構造を解剖します。「左より右」という構文は何を意味するのか。4つの価値はなぜ"4つ"なのか。12原則を3グループに整理する視点。本館 Vol.2 で「何を書いてあるか」を学んだあなたに、別館 Vol.2 では「なぜそう書かれたか」を提供します。
