01

ユーザーストーリーとは

WHAT IS A USER STORY

ユーザーストーリーは「機能仕様」ではありません。「誰が」「何を」「なぜ」したいかを3要素で書く、短い物語。これがバックログ項目の標準フォーマットになりました。機能発想から価値発想への転換──それがストーリーの本質です。

story.definition ストーリーの基本フォーマット
// 標準テンプレート
USER STORY
〜として（誰が）
〜したい（何を）
なぜなら〜だから（なぜ）
EXAMPLE 1
通販利用者として
注文履歴を1年分見たい
なぜなら前回購入したものを再度買いたいから
EXAMPLE 2
管理者として
ユーザーを一括停止したい
なぜなら不正利用時に即座に対応したいから
// なぜストーリーを書くか @機能仕様の問題 「履歴画面を作る」 → 何のためか不明 → 開発者が困る → ユーザー視点が抜ける @ストーリーの効果 「履歴を見たい = 再購入のため」 → 目的が明確 → 設計判断ができる → ユーザー視点で語れる // ストーリーの三要素 As a <persona>, I want <goal>, so that <reason>. // 日本語訳 〜として、 〜したい、 なぜなら〜だから。 // 「なぜなら」が最重要 "so that" 節は 仕様変更時の判断軸になる → 削るとストーリーが死ぬ // 発祥 1990年代後半 XP コミュニティ Kent Beck らが提唱 インデックスカードに書く →「3x5カードに収まる粒度」
POINT —
最重要なのは「なぜなら」の部分。これがあると、開発中に判断に迷ったとき「目的に照らしてどちらが良いか」を即判断できます。「なぜなら」を省くストーリーは骨抜きのストーリーです。★★★
02

INVEST原則

INVEST PRINCIPLE

良いストーリーはINVESTを満たします。Independent・Negotiable・Valuable・Estimable・Small・Testable──Bill Wake が2003年に提唱した6つの頭文字。「このストーリーは大丈夫か？」を診断するチェックリストとして今も現役です。

story.invest 良いストーリーの6条件
// INVEST チェック
頭字意味悪い例
IIndependent（独立）他依存ばかり
NNegotiable（交渉可）詳細を固めすぎ
VValuable（価値あり）裏方すぎる
EEstimable（見積可）大きすぎ・不明瞭
SSmall（小さい）1スプリント超え
TTestable（テスト可）受け入れ基準なし
// 各原則の意味 @Independent (独立) 他のストーリーと 独立して着手できる → 順序の制約最小化 @Negotiable (交渉可能) 詳細は実装直前に決める → 早期固定を避ける → カードに細部は書かない @Valuable (価値あり) ユーザーに価値を届ける → 「内部リファクタ」は 本当はストーリーではない @Estimable (見積可能) チームが時間を見積もれる → 不明瞭なものは まずスパイク調査 @Small (小さい) 数日で実装可能 → 1スプリント内完了 @Testable (テスト可能) 完成判定できる → 受け入れ基準必須 // アンチパターン例 "システムを改善する" → V × E × S × T 全滅 "DBを移行する" → V （価値が遠い） → 技術タスクではある
TIP —
INVESTで1つでも欠けるストーリーは要再考。特にE（見積可能）とS（小さい）を満たさないものは、分割するか、スパイク（調査用の小タスク）として独立させるのが定番。★★
03

受け入れ基準

ACCEPTANCE CRITERIA

ストーリーは「これが満たされたら完成」という条件、すなわち受け入れ基準（Acceptance Criteria）とペアで存在します。これがないストーリーは完成の定義が曖昧になり、永遠に終わらない地獄を生みます。受け入れ基準の書き方には「Given-When-Then」形式が定番です。

story.acceptance Given-When-Then 形式
// 受け入れ基準の書き方
USER STORY
通販利用者として、注文履歴を1年分見たい
ACCEPTANCE CRITERIA
Given ログイン済みであり、過去注文がある場合
When 「注文履歴」をタップ
Then 直近1年の注文がカード形式で表示される
CRITERIA 2
Given 過去注文が一件もない場合
When 「注文履歴」をタップ
Then 「履歴がありません」とおすすめ商品が表示される

※ 1ストーリーに2〜5個の基準が定番。これがそのままテストケースになる。

// Given-When-Then の構造 Given <前提条件> When <ユーザー操作> Then <期待される結果> // なぜこの形式が良いか @明確さ 状態 / 動作 / 結果 3要素に分かれる → 曖昧さがなくなる @テスト容易性 そのままテストコードに 変換可能 → BDD (Behavior Driven Development) と親和 @対話の促進 顧客と一緒に書ける 「この場合どうなる？」 → エッジケース発掘 // 派生:Cucumber 等のツール Feature: 注文履歴表示 Scenario: 過去注文あり Given ログイン済み When 履歴をタップ Then 1年分が表示 // アンチパターン "使いやすいこと" → 主観的 / 測定不可能 → 基準として失格 "バグがないこと" → 当然 / 不要 // 鉄則 受け入れ基準は 客観的に検証できる ものに限る
POINT —
受け入れ基準は「DoD（Doneの定義）」とは別物です。受け入れ基準はストーリー固有の完成条件、DoDはチーム全体で全ストーリー共通の品質基準。両方を満たして初めて「Done」になります。混同しないこと。★★★
04

3Cの法則

THREE C'S OF STORIES

ユーザーストーリーには3つのCがあります。Card・Conversation・Confirmation──Ron Jeffries が提唱した3要素。ストーリーは「カードに書かれた文字列」だけではない。会話と確認の仕組みを含めて初めてストーリーになる、という重要な視点です。

story.3c 3Cの構造
// 3つのCの役割
┌──────────────────────────┐ │ C1: Card (カード) │ │ 短い文字列 │ │ 会話のきっかけ │ │ 詳細は書かない │ └──────────────────────────┘ │ ↓ ┌──────────────────────────┐ │ C2: Conversation (会話) │ │ POと開発者で話す │ │ 詳細はここで詰める │ │ 暗黙知を共有 │ └──────────────────────────┘ │ ↓ ┌──────────────────────────┐ │ C3: Confirmation (確認) │ │ 受け入れ基準で検証 │ │ Doneを判定 │ └──────────────────────────┘ 3つそろって1つのストーリー
// なぜカードだけではダメか @紙きれの限界 カードに書ける情報量は 限られる → 全部書こうとすると 仕様書化する → 重くなり、変更に弱い @会話で補う POと開発者の対話 「これってどうすべき？」 「こういう場合は？」 → 暗黙知が共有される @確認で完成 受け入れ基準を満たしたか POが判定 → Done が確定する // アンチパターン "完璧なカード"症候群 カードに詳細を全部書く → 仕様書になる → 会話が省略される → ストーリーが死ぬ "会話なし"症候群 カードだけ渡される 「あとはよろしく」 → 開発者が想像で作る → 結果がPO の想定と かけ離れる // 鉄則 Card: 短く Conversation: たっぷり Confirmation: 明確に 3つの均衡が肝
裏技 —
JIRA等のツール時代は「カード」が消えやすいのが落とし穴。ツールに長文を書けるからといって書きすぎると、「対話」の儀式が消えて仕様書文化に逆戻り。意図的にカード本文を短く保つのがアジャイル熟達者の習慣です。★★
05

ストーリーポイント

STORY POINTS

アジャイルの最大の発明と言われるのがストーリーポイント。「時間」ではなく「規模感」で見積もるという発想の転換です。「このストーリーは3ポイント」と言われても何時間かわからない──それでいい、というのが革命的な点。なぜこれが機能するのか、見ていきましょう。

story.points なぜ時間じゃないのか
// 時間 vs ポイント
側面時間見積もりポイント見積もり
単位時・日抽象的数値
属人性あり（早い人/遅い人）なし（チームの相対）
当たりやすさ低い比較的高い
感情的圧力強い（締切意識）弱い
使用尺度連続値フィボナッチ

※ 1, 2, 3, 5, 8, 13, 21, 40, 100 — フィボナッチ数列を使う。

// ストーリーポイントの考え方 @相対見積もり 「あのストーリーが3なら こっちは5」 → 絶対値は不要 → 比較は容易 @複合的な指標 ポイント = 規模感 + 複雑度 + 不確実性 → 単純な時間ではない // なぜフィボナッチ? 1, 2, 3, 5, 8, 13, 21... ✓ 大きくなるほど誤差大 → 数値も粗くなって整合 ✓ 「5か6か」迷わない (6は無いので) ✓ 直感に合う // 各値の目安 1点: 一瞬で終わる 2点: 数時間 3点: 半日〜1日 5点: 1〜2日 8点: 2〜3日 13点: 大きい・分割検討 21点: 必ず分割 // よくある勘違い "1点 = 1時間" → 時間に変換すると ポイントの意味喪失 → 絶対NG // 結論 ポイントは 時間に翻訳しない
POINT —
上層部から「1ポイントは何時間？」と聞かれたら、それはアジャイル理解の浅さのサイン。「ベロシティとして観測されますが、固定換算はしません」と説明すること。固定換算した瞬間にポイントの意味は死にます。これは別館 Vol.4 で深掘りします。★★★
06

プランニングポーカー

PLANNING POKER

複数人で見積もるとき、声の大きい人の意見に引きずられる──これが業界の長年の問題でした。これを解決するのがプランニングポーカー。James Grenning が2002年に発案し、Mike Cohn が広めた手法。カードを一斉公開することで、独立した判断を引き出します。

estimation.poker ポーカーの進め方
// プランニングポーカー手順
STEP 1 POが1ストーリー紹介 「注文履歴を1年分表示」 STEP 2 開発者が質問・議論 仕様の不明点を解消 STEP 3 全員が同時にカード提示 ──────────────── [3] [3] [5] [8] [3] ──────────────── バラついた！ STEP 4 最高と最低が理由を説明 「8の人」: 認証連携で複雑 「3の人」: シンプルだと思った → 新情報共有される STEP 5 再投票 ──────────────── [5] [5] [5] [5] [5] ──────────────── 一致した → 5点で確定
// なぜポーカーが優れるか @独立判断 全員が同時に出す → 他人に流されない → 個々の見立てが現れる @議論の起点 バラついた所が そのまま議論ポイント → 認識ズレが見つかる @楽しさ ゲーム的要素 → 単調にならない → 全員参加が維持される // よくあるトラブル 先輩の声に合わせる → 「先輩が3なら3」 → 独立判断にならない → ポーカーの意義喪失 対策: 経験少ない人から 理由を語る 議論が永遠 → タイムボックス (1ストーリー5分等) 単位の混乱 「これって時間？」 → 都度教育 // ツール 物理カード Mountain Goat Software 等 オンライン PlanITPoker, Scrum Poker Slack 連携 PokerBot 等
TIP —
リモートワーク時代はオンライン版が便利。PlanITPokerやScrum Poker Onlineはブラウザだけで動き、無料で使えます。「みんなで一斉に出す」「最大と最小が説明する」を守れば、物理かオンラインかは問いません。★★
07

ベロシティ

VELOCITY

ベロシティ＝「1スプリントで完了したポイント合計」。これがアジャイルの進捗指標です。日々の進捗報告でも、ガントチャートでもない。実績ポイントの推移で、チームの本当のキャパシティを観測します。「予測ではなく観測」──これがベロシティ思考の核です。

estimation.velocity ベロシティの計測と活用
// ベロシティの推移例
スプリント 完了ポイント ───────────────────────── Sprint 1 18 pt Sprint 2 22 pt Sprint 3 25 pt Sprint 4 28 pt Sprint 5 27 pt Sprint 6 30 pt ───────────────────────── 最近3回の平均: 28 pt → 次スプリントは 25〜30ptを目安に積む ━━━━━━━━━━━━━━━━━━━━━ ベロシティは 「目標」ではなく「観測」 ━━━━━━━━━━━━━━━━━━━━━
  • 初回は当てずっぽうでよい
  • 3スプリント目から安定し始める
  • 5〜10スプリントで真の値が見える
// ベロシティの活用 @スプリント計画 「平均28pt なら 今回も25〜30ptを上限に」 → 詰め込み過ぎ防止 @リリース予測 残100pt / 平均28pt = 約4スプリント → 予測の根拠を持つ @チーム健康診断 ✓ 安定 → 健全 ✗ 乱高下 → 問題あり ✗ 漸減 → 燃え尽き兆候 // よくある罠 "ベロシティ目標化" 「来期は30→40に！」 → ポイント水増しが横行 → 信頼性崩壊 → 絶対NG "他チームと比較" 「あのチームは50なのに」 → ポイントは相対値 → チーム間比較は無意味 → 絶対NG // 鉄則 ベロシティは ✓ 自チームの観測ツール ✗ KPI / 目標 / 評価指標 // 別の指標との関係 バーンダウンチャート: スプリント内残量推移 バーンアップチャート: 累積完了量推移 両方ともベロシティと併用
POINT —
ベロシティを「個人の評価」「チーム間比較」「経営目標」に使うのは絶対NG。これをやるとポイント水増しが即発生し、見積もりが信頼できなくなります。「ベロシティは観測値であって目標ではない」──経営層への説明が、PO/SMの大事な仕事です。★★★
08

ストーリー分割

STORY SPLITTING

大きすぎるストーリーはスプリントに入りません。8点を超えるストーリーは分割を検討、13点なら必ず分割──これが定番。しかし「大きいから2つに切る」では不十分。分割パターンを知ると、ストーリー分割が機械的にできるようになります。

story.splitting ストーリー分割の定番パターン
// 分割の定番8パターン
  • ワークフロー：処理の段階で分ける（登録→確認→完了）
  • 業務ルール：単純ケース→例外ケース
  • 努力レベル：MVP→ベター→ベスト版
  • データ種別：単一データ→複数データ対応
  • データ操作：CRUD で分ける（読→書→更新→削除）
  • 機能レベル：MUST→SHOULD→COULD
  • パフォーマンス：機能版→最適化版
  • ハッピーパス：正常系→エッジケース
// 具体例：注文機能の分割 大ストーリー (13pt) 「ユーザーとして注文したい」 // パターン: ワークフロー分割 @分割後1 (3pt) カートに商品を追加 → 単独で動くか @分割後2 (3pt) カート画面で確認 @分割後3 (3pt) 決済情報を入力 @分割後4 (5pt) 注文確定 // パターン: 努力レベル分割 @MVP (3pt) クレカ決済のみ対応 @ベター (5pt) + 電子マネー対応 @ベスト (8pt) + 銀行振込・コンビニ決済 // 重要：垂直分割 水平分割: UI 層・ロジ層 ↑ NG（動かないものになる） 垂直分割: UI〜DB まで ↑ OK（小さく動く） // 鉄則 分割後の各ストーリーは それぞれ価値を持つ INVEST のVを維持
裏技 —
SPIDR パターン（Spike・Path・Interface・Data・Rules）も有名な分割技法。Mike Cohn が体系化したもの。「困ったらまずパス（業務フロー）で分けろ」が現場の経験則です。水平分割（DB層だけ・UI層だけ）は禁忌──動くものにならないストーリーは価値を生みません。★★★
