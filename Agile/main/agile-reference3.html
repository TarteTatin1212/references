Scrum編

SCRUM — ROLES, EVENTS, ARTIFACTS

アジャイルの代名詞、それがScrum。世界で最も普及したアジャイルフレームワークで、「3-5-3」と呼ばれる構造（3つの役割・5つのイベント・3つの成果物）でできています。本巻はこの3-5-3を網羅し、「なぜスプリントは時間で区切るのか」「なぜ役割は3つなのか」といった現場の疑問を解いていきます。

01

Scrum全体像（3-5-3）

SCRUM AT A GLANCE

Scrumを語る合言葉が「3-5-3」。3つの役割・5つのイベント・3つの成果物。Scrumガイドはたった14ページの薄い文書で、これだけのルールしか書かれていません。シンプルだが軽くはない──これがScrumの本質です。

scrum.3-5-3 Scrumフレームワークの全要素
// Scrumの構成要素
┌─────────────────────────────────┐ │ 3つの役割 (Accountabilities) │ │ ◇ プロダクトオーナー (PO) │ │ ◇ スクラムマスター (SM) │ │ ◇ 開発者 (Developers) │ ├─────────────────────────────────┤ │ 5つのイベント (Events) │ │ ◇ スプリント (1〜4週) │ │ ◇ プランニング (8h以内) │ │ ◇ デイリースクラム (15min) │ │ ◇ スプリントレビュー (4h以内)│ │ ◇ レトロスペクティブ (3h以内)│ ├─────────────────────────────────┤ │ 3つの成果物 (Artifacts) │ │ ◇ プロダクトバックログ │ │ ◇ スプリントバックログ │ │ ◇ インクリメント │ └─────────────────────────────────┘
// Scrumガイドの定義 Scrum is "a lightweight framework that helps people, teams and organizations generate value through adaptive solutions for complex problems." // 軽量フレームワーク 軽い ≠ 楽 シンプル ≠ 簡単 // 公式ドキュメントの厚さ Scrum Guide 2020: 14p → ルールはわずか → 実践は奥深い // 3つの柱（Three Pillars） @透明性: 全員が見える @検査: 定期的に確認 @適応: その場で調整 // 5つの価値 確約・集中・公開 尊敬・勇気
POINT —
Scrumガイドは2020年版が最新で、Ken SchwaberとJeff Sutherlandの共著。日本語版も公式に翻訳されています。無料で読める14ページなので、一度は原典を通読することを強く推奨します。★★★
02

プロダクトオーナー

PRODUCT OWNER (PO)

POは「何を作るか」を決める役割。プロダクトの価値を最大化する責任を負い、1人で担います（委員会ではない）。POが優柔不断だとチーム全体が止まる──Scrumで最も負担の重いポジションです。

role.product-owner POの責任範囲と典型行動
// POの責任マップ
PROD. OWNER (1人) │ ├─ プロダクトゴール設定 │ 「3ヶ月後にこうなる」 │ ├─ バックログ管理 │ ├ 項目の作成 │ ├ 並び替え（優先順位） │ └ 透明性の確保 │ ├─ 価値の最大化 │ 「投資対効果を最大に」 │ └─ ステークホルダー対応 上司・営業・顧客と調整 ━━━━━━━━━━━━━━━━━━━━━ 決断する人 断る勇気を持つ人 ━━━━━━━━━━━━━━━━━━━━━
// POの1スプリント スプリント開始前: バックログを整える 優先順位を確定 プランニング: 「これとこれをやって」 と開発者に伝える スプリント中: 開発者の質問に即答 バックログを継続整備 次スプリント以降の準備 レビュー: ステークホルダー招集 受け入れ判定 // 良いPOの条件 @決められる 優柔不断は致命傷 @断れる 「やらない」が言える @近い チームと密に対話 @詳しい プロダクト・市場・顧客 // アンチパターン 兼任PO: 不在で停滞 委員会PO: 決まらない 伝書鳩PO: 上の意向待ち
裏技 —
POが「他部署と兼任」になると9割失敗します。POの仕事はフルタイム級。日本の現場では「業務側マネージャがPO兼任」が多く、これがアジャイル失敗の最大要因になりがちです。専任POを置けるかが成否を分けます。★★★
03

スクラムマスター

SCRUM MASTER (SM)

SMは「チームのコーチ」。プロジェクトマネージャでも進捗管理係でもありません。Scrumが正しく機能しているかを守り、チームを支援し、障害を取り除くのがSMの仕事。「サーバントリーダー」と呼ばれる独特な立ち位置です。

role.scrum-master SMの本当の仕事
// SMがやること・やらないこと
SMがやること
  • イベント進行のファシリ
  • 障害（impediment）除去
  • Scrumの教育
  • チームの自己組織化支援
  • 組織への変革働きかけ
SMがやらないこと
  • タスク割り当て
  • 進捗管理
  • 個人評価
  • 仕様決定
  • 命令・指示

※ SMは「上司」ではなく「コーチ」。

// SMの典型的な1日 朝: デイリーを進行 阻害要因をメモ 日中: 阻害要因を解決に動く ├ 他部署との調整 ├ 環境問題の解決 └ 認識合わせの場作り 合間: POとバックログ整理を相談 開発者の悩みを聞く 夕: チームの健康状態確認 次の改善アクション検討 // サーバントリーダーシップ Servant Leader = 奉仕するリーダー = チームのために動く // 良いSMの兆候 @チームが動いてる SMがいなくても回る @阻害要因が消える 気づくと問題が解決 @チームが成長してる 自己組織化が進む
POINT —
SMの仕事は「目に見えにくい」のが特徴。良いSMがいるチームは自然に回っているように見えるので、評価されにくい難しさがあります。「いなくなって初めて偉大さがわかる」のがSM。陰の立役者です。★★★
04

開発者

DEVELOPERS

開発者は「どう作るか」を決める役割。プログラマだけでなく、デザイナー・QA・データエンジニア等、インクリメント作成に必要な全ての人を含みます。Scrumでは職能で分けず、1つのチームとして扱います。

role.developers 開発者チームのあり方
// 開発者チームの特徴
┌─────────────────────────────┐ │ 開発者（3〜10人推奨） │ │ │ │ ◇ クロスファンクショナル │ │ プログラマ・デザイナー │ │ QA・データエンジニア等 │ │ 必要な全スキルを内包 │ │ │ │ ◇ 自己管理 │ │ 誰がやるか自分で決める │ │ 進め方も自分で決める │ │ │ │ ◇ 全員が責任を持つ │ │ 「自分の仕事」はない │ │ 「チームの仕事」だけ │ └─────────────────────────────┘ 推奨サイズ: 3〜10人 < 3: 相互作用が乏しい > 10: 調整コスト爆発
// 開発者の責任 @計画 スプリントバックログ作成 タスクの分解と見積もり @実行 インクリメント作成 品質を保証（DoDを守る） @調整 毎日の進捗確認 必要に応じてプラン修正 @改善 自分たちのやり方を見直す 技術的負債と向き合う // スキル横断の意味 専門性は持つ →「Tシェイプ人材」 深い専門 + 周辺知識 兼任は前提 「フロント担当」固定× 「全員フロント書ける」○ // 鉄則 "It's not my job" 禁止 チームの仕事に境界なし
TIP —
Scrumガイド2020で「開発チーム」から「開発者」へ呼称変更されました。これは「チーム内チーム」の発想を排除するため。PO・SM・開発者をまとめて「スクラムチーム」と呼びます。1つのチーム、1つのゴール、です。★★
05

スプリントとプランニング

SPRINT & PLANNING

Scrumの心臓部がスプリント。1ヶ月以下の固定期間で、その期間中にインクリメントを作り上げます。スプリントの始まりがプランニング。「何を・どう作るか」をチーム全員で決める2〜8時間の集中タイムです。

sprint.planning スプリント期間とプランニングの3トピック
// スプリントの流れ
┌─────────────────────────┐ │ プランニング (Day 1) │ │ Why / What / How │ └──────────┬──────────────┘ │ ┌─────▼─────┐ │ 開発作業 │ ← デイリー │ (毎日継続) │ (毎朝15分) │ │ │ + リファイ │ │ ンメント │ └─────┬──────┘ │ ┌──────────▼──────────────┐ │ レビュー (最終日午前) │ │ ステークホルダーへデモ │ └──────────┬──────────────┘ │ ┌──────────▼──────────────┐ │ レトロ (最終日午後) │ │ チームの振り返り │ └─────────────────────────┘ → 翌スプリントへ続く
// プランニングの3トピック @Topic 1: Why（なぜ） POが提示 「このスプリントで なぜ価値を生むか」 → スプリントゴールを作る @Topic 2: What（何を） バックログから選ぶ 「今回これをやる」 → スプリントバックログ(項目) @Topic 3: How（どう） タスク分解と段取り 「こうやって作る」 → スプリントバックログ(計画) // 期間別タイムボックス 1週間スプリント: 最大 2h 2週間スプリント: 最大 4h 4週間スプリント: 最大 8h // 重要 スプリントゴールは 1スプリント1つ 全員が同じ方向を見るための 北極星になる // アンチパターン タスク詰め込み過ぎ ゴール不明瞭 POが来ない
POINT —
「スプリントゴール」はScrumで最重要な概念の1つ。バックログ項目をただ並べるだけでは不十分で、「このスプリントで達成したい1文」を作ること。「ログイン機能のリリース」「決済フローの体験完成」など、動詞で語れる目標がチームの集中を生みます。★★★
06

デイリースクラム

DAILY SCRUM

毎朝15分の同期ミーティング。「朝会」「スタンドアップ」とも呼ばれます。これは進捗報告会ではありません。チームがスプリントゴールに向かっているかを確認し、必要なら計画を調整するためのもの。SMやPOへの報告ではなく、開発者同士の確認と調整です。

event.daily デイリーの本当の意味
// 古典型と現代型
[古典：3つの質問] ① 昨日やったこと ② 今日やること ③ 困っていること → 報告会化しがち → 一方通行になりがち [現代：ゴールから逆引き] ◇ スプリントゴールに 向かえているか？ ◇ 計画を変える必要は？ ◇ ヘルプが要る部分は？ → 全員で適応的に → 議論と調整の場に ─────────────────────── 15分タイムボックス厳守 立ったままが定番 ───────────────────────
// 良いデイリーの兆候 @時間が守れる 15分ピッタリで終わる 延びるなら深掘りは別途 @議論が生まれる 「それ詰まってるなら 今日ペアでやろう」 → 即時の調整 @ゴールが見えてる 「ゴールに対し進捗どう？」 → 全員が同じ景色 // アンチパターン SMへの報告会 → 「昨日は○○しました」 と一人ずつ報告 → これは進捗会議 → 本来の意図と違う 議題が広がりすぎ → 設計議論が始まる → 1人1時間に → タイムボックス破綻 遅刻・欠席が常態化 → 同期が機能しない → デイリーの意味喪失 // 鉄則 毎日 同時刻 同場所 15分 全員参加
TIP —
立ったままやるのは「長引かせないため」の工夫です。座ると無意識に長くなる。リモートワークでは難しいですが、カメラON・15分タイマー表示などで同様の緊張感を作るのが鉄則です。★★
07

スプリントレビュー

SPRINT REVIEW

スプリント終盤に「動くものを披露する」のがレビュー。プレゼンや進捗報告ではなく「触ってもらう」場。ステークホルダーを呼び、その場でフィードバックを受け、バックログを更新する──これがScrumのフィードバックループの核です。

event.review レビューの構成と落とし穴
// レビューの段取り
時間: 1週なら1h / 2週なら2h 4週なら最大4h 参加者: スクラムチーム + ステークホルダー (顧客・営業・経営層・運用) 進め方: 1. スプリントゴール振り返り 2. インクリメントのデモ ↓ 触ってもらう・操作 3. フィードバック収集 4. 市場・状況の変化共有 5. バックログ調整方針 ━━━━━━━━━━━━━━━━━━━━━ プレゼンではなく 協働の場である ━━━━━━━━━━━━━━━━━━━━━
// 良いレビューの条件 @動くものを見せる ✓ 実環境（or 近い環境） ✓ 実データ（or 近いデータ） ✗ スライドだけ ✗ 「動画でご覧ください」 @双方向 ✓ ステークホルダーが触る ✓ その場で議論 ✗ 開発側が一方的説明 @率直なFB歓迎 ✓ 違うと言われても受け止め ✓ バックログに反映 ✗ 防衛的な言い訳 // やってはいけないこと 完成度を盛る → 動かないものを 動くように見せる → 本来の意図と真逆 ステークホルダー欠席 → フィードバック取れず → レビューの意味喪失 // 結論 "Working software over PowerPoint"
POINT —
レビューにステークホルダーが来ないのはScrum失敗の代表的兆候。「忙しくて」と言われるなら、POが招集の責任を持つ必要があります。顧客が見ないものを作っても価値検証できない。これは妥協してはいけない一線です。★★★
08

レトロスペクティブ

SPRINT RETROSPECTIVE

スプリントの最後にやる「チーム自身の振り返り」。プロダクトの話ではなく「自分たちの働き方」を振り返ります。原則12「定期的に振り返り改善する」の体現の場。Scrumが他のフレームワークと違うのは、「改善の儀式」を制度化している点にあります。

event.retrospective レトロの定番フォーマット
// 代表的なレトロ手法
手法軸特徴
KPTKeep/Problem/Try日本で最も普及
YWTやったこと/わかったこと/次やること学習重視
Start/Stop/Continue始める/やめる/続ける行動明確
Mad/Sad/Glad怒/悲/喜感情起点
4LsLiked/Learned/Lacked/Longed多面的
セーリングボート追い風/錨/島視覚的

※ 毎回同じ手法だと飽きる。3〜4回ごとに変えるのが定番。

// レトロの段取り（5フェーズ） 1. 場を設定 (Set the Stage) 雰囲気作り 安全な場であることを確認 2. データ収集 (Gather Data) 事実を持ち寄る 出来事・数値・感情 3. 洞察を生む (Generate Insights) なぜそうなった？ 何が原因？ 4. 何をする (Decide What to Do) 次スプリントの具体行動 担当者・期限つきで 5. 締めくくり (Close) 今日の感謝・気づき // 鉄則 話したことが 外に漏れない 心理的安全性が前提 // 改善は1〜2個に絞る 大量の改善案は どれも実行されない 1〜2個に絞り着実に
裏技 —
レトロが形骸化したら飽きが原因。手法を変えるだけで雰囲気が一新します。FunRetro・Miro・EasyRetro等のオンラインツールはテンプレートが豊富で、いろんな手法を試しやすい。マンネリは改善の敵です。★★

バックログとDoD

BACKLOG & DEFINITION OF DONE

Scrumの3つの成果物はプロダクトバックログ・スプリントバックログ・インクリメント。それぞれに付随するコミットメントとして、プロダクトゴール・スプリントゴール・Done の定義（DoD）があります。特にDoDは品質を保証する命綱です。

artifacts.backlog 3つの成果物と3つのコミットメント
// 成果物の関係
プロダクトバックログ （長期的に成長） 　　│ 　　│ ← プランニングで選ぶ 　　▼ スプリントバックログ （このスプリント分） 　　│ 　　│ ← 開発作業 　　▼ インクリメント （動くソフトウェア） ━━━━━━━━━━━━━━━━━━━━━ 各成果物のコミットメント [PBL] → プロダクトゴール 長期的な目標 [SBL] → スプリントゴール このスプリントの目的 [Inc] → Doneの定義(DoD) 完成の基準
// プロダクトバックログ 特徴: ✓ 1つだけ（プロダクトに1つ） ✓ 上から優先順位順 ✓ 上ほど詳細・下ほど粗い ✓ 動的に変化し続ける // スプリントバックログ 構成: ① スプリントゴール（Why） ② 選んだPBL項目（What） ③ 実行計画・タスク（How） // インクリメント 条件: ✓ DoDを満たしていること ✓ 動くこと ✓ リリース可能であること // DoD の例 @コード ✓ コードレビュー通過 ✓ Lintパス @テスト ✓ ユニットテスト追加 ✓ 統合テスト通過 @ドキュメント ✓ APIドキュ更新 @デプロイ ✓ ステージング環境動作 // DoDがないチーム 完成基準が人で違う 品質がバラつく 技術的負債が累積
POINT —
DoD（Definition of Done）はチームで合意した文書として明文化しておくこと。「Doneとは何か」が人によって違うと、品質が崩壊します。新メンバー加入時の説明資料にもなる重要文書。最低でも10項目程度は欲しいところです。★★★
