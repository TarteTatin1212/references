スケーリングと組織編

SCALING AGILE — FROM ONE TEAM TO ENTERPRISE

01

なぜスケーリングは難しいか

WHY SCALING IS HARD

アジャイルは元々小さなチームを前提に設計されました。Scrumは3〜10人、XPも同様。これを100人・1000人規模に拡大しようとすると、コミュニケーションコスト・依存関係・調整負担が指数的に膨らみます。「スケーリング＝単純な掛け算」では絶対に成功しません。

scaling.difficulty スケーリングの根本問題
// 規模による課題変化
規模主な課題
1チーム（〜10人）チーム内の協働
2〜3チーム（〜30人）同期と依存管理
5〜10チーム（〜100人）アーキテクチャ・調整
10〜50チーム（〜500人）組織設計・文化
50チーム以上会社全体の経営課題

※ 規模が変わると課題の性質が変わる。1チームの解は10チームには効かない。

// コミュニケーションパスの数 N人チームのコミュニケーションパス: P = N × (N-1) / 2 @5人: 10 パス @10人: 45 パス @20人: 190 パス @50人: 1,225 パス @100人: 4,950 パス → 急激に膨らむ → 全員が情報を共有できなくなる // ダンバー数 人類学者 Robin Dunbar の研究: 人間が安定した社会関係を 保てる人数 = 約150人 組織がこれを超えると 形式的なルールが必須に → 文化が変質する // スケーリングの3課題 @1: 同期 複数チームの作業を揃える → スプリント同期・PI Planning @2: 依存関係 チーム間で互いに必要なもの → アーキテクチャ・API契約 @3: アライメント 全員が同じ方向を向く → ビジョン・ロードマップ // 結論 スケーリング ≠ 「アジャイルチームの 単純な複製」 新たな問題群が現れる それに対応する仕組みが必要
POINT —
「うまく回っている1チーム」を10コピーすれば10チームでうまく回るは完全な間違い。規模の問題は質的に異なる。これを理解せずに「スプリント同期しよう」「合同レトロやろう」と表層を真似ると失敗します。スケーリングは組織全体の設計問題です。★★★
02

LeSS

LARGE-SCALE SCRUM

LeSS（Large-Scale Scrum）は「Scrumをそのまま大きくする」哲学。Craig Larman & Bas Vodde が提唱。「シンプルを保つ」を最重視し、新しい役割や階層を増やさない方針です。Scrumの原則に忠実なため、Scrum熟練チームには馴染みやすい構造です。

scaling.less LeSSの構造
// LeSSの基本構造
[LeSS (2〜8チーム)] ┌─────────────────────┐ │ 1人のプロダクトオーナー │ │ 1つのプロダクトBL │ └──────────┬──────────┘ │ ┌──────┬──┴──┬──────┐ ▼ ▼ ▼ ▼ Team1 Team2 ... Team8 │ │ │ │ 各チームはScrumを実践 [LeSS Huge (8チーム以上)] エリア(Area)で分割 各エリアにエリアPO配置

※ PO は1人。バックログも1つ。これがLeSSの絶対原則。

// LeSSの10原則（抜粋） @1. Large-Scale Scrum is Scrum → スケールしてもScrum @2. Empirical Process Control → 経験的プロセス制御 @3. Whole-Product Focus → 全プロダクト視点 → 個々の機能ではなく 全体を最適化 @4. Customer-Centric → 顧客中心 @5. Continuous Improvement → 完全性への絶え間ない改善 @6. Lean Thinking → リーン思想 @7. Systems Thinking → システム思考 // LeSSの特徴的なイベント スプリントプランニング1 全チーム合同 「何を作るか」分担 スプリントプランニング2 各チーム個別 「どう作るか」 合同レビュー 全チームが集まり 全インクリメントをデモ 合同レトロ 「私たちの連携を改善」 // 採用基準 @LeSS が合う ✓ 1つのプロダクトに集中 ✓ Scrum 熟練済み ✓ シンプル志向の組織 @LeSS が合わない ✗ 複数プロダクト並走 ✗ 既存階層を残したい ✗ 「役職」を増やしたい // 採用企業 JPMorgan, BMW シンガポール政府 …シンプル志向の組織
TIP —
LeSS の公式サイトless.worksには豊富な日本語資料があります。『Large-Scale Scrum』（Larman & Vodde）は分厚いが必読。「Scrumに新しい階層を持ち込まない」頑なな姿勢が特徴。「シンプルさは複雑さを征服する」という著者の信念が貫かれています。★★
03

SAFe

SCALED AGILE FRAMEWORK

SAFe（Scaled Agile Framework）は、大企業向けに体系化された最も詳細なスケーリング手法。Dean Leffingwell が提唱。役職と階層が多く、ルールも詳細。「LeSSと真逆の方向性」と評されることも。世界の大企業で最も採用率が高いフレームワークです。

scaling.safe SAFeの構造
// SAFeの4階層
┌──────────────────────────┐ │ Portfolio (ポートフォリオ) │ │ 戦略・投資判断 │ └────────────┬─────────────┘ │ ┌─────────────▼────────────┐ │ Large Solution │ │ (任意・超大規模時のみ) │ └────────────┬─────────────┘ │ ┌─────────────▼────────────┐ │ Essential (ART) │ │ Agile Release Train │ │ 5〜12 チームの束ね単位 │ │ PI Planning で同期 │ └────────────┬─────────────┘ │ ┌─────────────▼────────────┐ │ Team │ │ Scrum or Kanban 単一チーム │ └──────────────────────────┘ PI = Program Increment (10週間)
// SAFeの特徴的な要素 @PI Planning 10週間ごとに 全チーム合同で計画 数百人が一同に会する 「2日間の超大イベント」 @ART (Agile Release Train) 5〜12チームの束ね単位 共通の目標で動く Release Train Engineer (RTE) が運営 @役職リスト Release Train Engineer Solution Train Engineer System Architect Product Manager Business Owner Lean Portfolio Manager Epic Owner ... 多数 // SAFeの賛否両論 @肯定派 ✓ 大企業に即適用可 ✓ 詳細手順書あり ✓ コンサル支援豊富 ✓ 経営層に説明しやすい @否定派 ✗ 階層が多すぎ ✗ 役職が多すぎ ✗ 「アジャイル」と呼べるか? ✗ ScrumGod が批判 "Dark SAFe"とも揶揄される // 業界の声 「"SAFeは大企業の 既存階層を温存できる "アジャイル風"の仕組み"」 ↑ 厳しい批判 「"完璧でなくても 移行のための足場として価値"」 ↑ 擁護論 // 採用企業 Cisco, Lockheed Martin 富士通, 多くの保険会社 規模・規制重視の組織で人気
POINT —
SAFe は「大企業の既存階層を温存できる」のが採用率の高さの理由。批判派は「これはアジャイルではなくウォーターフォール風アジャイル」と切り捨て、擁護派は「移行段階の現実解」と評価。正しい答えはなく、組織の成熟度・現実制約による選択です。★★★
04

Spotifyモデル

SPOTIFY MODEL

Spotify が公開した独自の組織モデル。Squad・Tribe・Chapter・Guildという階層構造で、フレームワークというより「文化」として広まりました。「Spotifyモデルは Spotify でも完全にはやってない」とSpotify社員が認めた逸話も。憧れと現実のギャップが大きい例です。

scaling.spotify Spotifyの4階層
// Spotifyモデルの4階層
━━━━━━━━━━━━━━━━━━━━━━━━━ Squad (チーム) 6〜12人の自律的単位 1つのミッション Scrum や Kanban を選択 Tribe (部族) 関連する Squad の集合 ダンバー数で40〜150人 Chapter (章) 横断的な専門グループ 例: 全 Squad のフロント担当 技術交流・教育の場 Guild (ギルド) 関心に基づく緩い集まり 例: AI ガード, アジャイル研究 自発的参加
// Spotifyモデルの特徴 @Squadの自律性 各 Squad が Scrum/Kanban/Scrumban 自由に選択 「ミニスタートアップ」 @マトリクス組織 Squad: ミッション単位 Chapter: 専門単位 → 両方の上司がいる @文化重視 「Failure Friday」 失敗を学びに変える文化 「Hack Week」 自由開発期間 // 公開の経緯 2012: Henrik Kniberg らが Squad モデルを公開 業界に衝撃 2014: ビデオでさらに広まる 多くの企業が真似 2016: Spotify エンジニアが 「我々もこの通りには やっていない」と発言 業界騒然 2020〜: 「Spotify モデル神話」 の見直しが進む // 業界の教訓 @ベストプラクティスの罠 他社の成功事例を そのまま真似ても 自社の文脈が違うと失敗 @「モデル」の幻想 Squad/Tribe は言葉 本質は「自律」「信頼」 これらの文化が重要 // 結論 Spotify モデルは 「インスピレーション」 として参考にせよ 「処方箋」として 使うのは危険
裏技 —
「Spotify は Spotify モデルをやっていない」とは、2016年の社内ブログで明かされた事実。「組織モデルは進化し続けるもの」であり、2012年のモデルを2024年に丸コピーするのはアジャイル精神に反します。「真似より対話」──各組織が自分の文脈で模索すべきです。★★
05

Scrum@Scale

SCRUM@SCALE

Scrum@ScaleはJeff Sutherland（Scrum共同創始者）が提唱したスケーリング手法。「Scrum of Scrums」の発展形で、複数のScrumを「フラクタル構造」で組み上げます。LeSSと似ていますが、より「経営戦略」を扱う領域まで踏み込みます。

scaling.scrum-at-scale フラクタル構造
// フラクタル構造
[単一 Scrum チーム] PO + SM + 開発者 ↓ スケール [Scrum of Scrums] 5チームの代表者で同期 → "EAT" (Executive Action Team) → "EMS" (Executive MetaScrum) ↓ さらにスケール [Scrum of Scrum of Scrums] 同じパターンを繰り返す フラクタル ━━━━━━━━━━━━━━━━━━━━━ 各階層が「Scrum」の構造 同じパターンの拡大 ━━━━━━━━━━━━━━━━━━━━━
// Scrum@Scaleの構造 @2つのサイクル SDP (Scrum Master Cycle) 「どう作るか」 How サイクル SoS, SoSoS と階層構造 PDP (Product Owner Cycle) 「何を作るか」 What サイクル MetaScrum, EMS と階層 @SoS (Scrum of Scrums) 代表が定期同期 「他チームを止めていること 止めていないこと」 を共有 @EAT Executive Action Team 経営層レベルの Scrum 組織の障害を除去 @EMS Executive MetaScrum プロダクト戦略の意思決定 // LeSSとの違い LeSS: 経営に踏み込まない 「Scrum を大きく」のみ Scrum@Scale: 経営戦略まで Scrum 化 「組織全体の SoS 化」 // メリット ✓ 完全な対称性 ✓ 経営層まで巻き込める ✓ Scrum 思考の徹底 // デメリット ✗ 経営層の理解が必須 ✗ 階層が深くなると同期コスト ✗ 経営層が Scrum 嫌うと崩壊 // 採用 3M, John Deere, Bayer SoS 部分のみ取り入れ ハイブリッド運用が多い
POINT —
「Scrum of Scrums」という用語は元々 Jeff Sutherland が1996年に IDX 社で導入したのが起源。Scrum@Scaleはこれを体系化したもの。「経営層を Scrum 化する」という野心が特徴。経営層の協力が前提なので、トップダウンの覚悟が要ります。★★

フレームワーク選択

CHOOSING A FRAMEWORK

結局「どれを採用すべきか」。答えは「組織の文化と現状」で決まります。同じ会社でも、部門ごとに違うフレームワークを採用するのも普通。「フレームワーク選びは目的ではなく手段」──組織の成熟度を見極めて選ぶことが大事です。

scaling.choose どれを選ぶか判断軸
// フレームワーク比較
軸LeSSSAFeSpotifyScrum@Scale
複雑さ低高中中
規模〜810〜100柔軟柔軟
階層数少多中中
役職少多中少
経営層関与少強関与緩い強関与
習得難易低高中中
// 判断のヒント @Scrum熟練組織なら → LeSS シンプル維持できる @大企業・規制業界なら → SAFe 既存階層と整合性 経営層に説明しやすい @文化的に自律性高いなら → Spotify インスパイア ガイドラインに留める @経営層もアジャイル理解 → Scrum@Scale 全社的に統一思想 // 「正解」はないので 小さく始める まず1チームでアジャイル 実績作る 徐々に拡げる 2〜3チームに増やしてみる 問題点を観察 必要になったら導入 「フレームワーク必要」 と感じてから検討 // 注意 フレームワーク先行は失敗 「SAFe導入する」 と決めてから現場展開 → 形だけになる 現場発で必要を確認 「同期に困った」 「アライメントが取れない」 → そこで初めて検討 // 結論 フレームワークは 手段であって目的ではない
TIP —
アジャイルコーチに「うちはどのフレームワークが良いですか？」と聞いた時、即答する人は怪しい。良いコーチは「まず現状を見せてください」と返します。フレームワーク選びは、現状分析の後。逆順は失敗パターンです。★★★

スケーリングのアンチパターン

SCALING ANTI-PATTERNS

スケーリングの失敗例には明確なパターンがあります。「役職だけ増やす」「会議体で同期しようとする」「ツール導入で解決する」──これらは失敗への高速道路。事前に知っておけば回避できます。

scaling.anti-patterns よくある失敗パターン
// 5大アンチパターン
  • 「アジャイル部署」を作る ── 既存組織の上に乗せるだけで形骸化
  • 役職だけ増やす ── 肩書だけ変わって動きは旧来通り
  • 会議体で同期 ── 「同期ミーティング」が増殖し、開発時間が消える
  • ツールで解決 ── JIRA導入で「アジャイルになった」と勘違い
  • コンサル丸投げ ── 外部の言う通りに導入して文化が育たない
失敗パターンの共通項
「形」だけ真似て
「文化」を見ない
→ 表層的なアジャイル化
成功の条件
小さく始めて学ぶ
現場の声を聞く
失敗を許容する
// アンチパターン詳細 @「アジャイル部署」設立 問題: 既存組織と並立して 「アジャイル推進部」を作る → 既存階層は変わらない → 部署対立が生じる 対策: 既存組織を変える 並立ではなく統合 @役職だけ増える 「Agile Coach」 「Chief Agile Officer」 「Agile Project Manager」 → 肩書だけ変わる → 仕事内容は前と同じ 対策: 役職より仕事を変える @同期会議の増殖 Scrum of Scrums Daily of Dailies Retrospective of Retros → 会議のための会議 → 開発時間が消える 対策: 非同期コミュニケーション ドキュメント化 必要最小限の同期 @ツール幻想 「JIRA使えばアジャイル」 「Confluenceで文化醸成」 → ツールは道具 → 文化は別の問題 対策: ツールは最後に決める まず実践・観察・改善 @コンサル丸投げ 外部コンサルに依存 → 撤退後に元の形に戻る → 内製能力育たず 対策: コーチング受けつつ 内部人材を育てる
POINT —
最大のアンチパターンは「形だけ真似る」こと。Spotify モデルの図、SAFe のロードマップ、PI Planning の写真。「見える部分」を模倣しても、本質である「文化」「信頼」「自律」は移植できません。組織変革は時間がかかる。近道はない、と腹を括る覚悟が必要です。★★★
08

組織変革の現実

REALITY OF TRANSFORMATION

最後に組織変革の厳しい現実を伝えます。「アジャイル変革プロジェクト」の大半は失敗します。なぜか。「人と文化」の問題を「手法と仕組み」で解決しようとするから。本巻の締めくくりとして、変革の本質的な難しさに向き合います。

transformation.reality 変革の難しさと真因
// 変革が失敗する原因
  • 経営層の理解不足 ── 数字で評価する旧来の思考が抜けない
  • 中間管理職の抵抗 ── 自分の役割が脅かされる恐怖
  • 現場の燃え尽き ── 既存業務＋アジャイル導入で疲弊
  • 短期成果の要求 ── 「3ヶ月で成果出せ」で形だけになる
  • 失敗を許さない文化 ── 「アジャイルは失敗から学ぶ」が機能しない

※ 統計的には「アジャイル変革プロジェクトの 50%以上が失敗」と複数調査で示されている。

// 成功する変革の共通項 @1: 経営層のコミット 形だけのスポンサーシップでなく 経営層自身がアジャイル学ぶ 経営層がスクラム実践 @2: 時間をかける 最低 3〜5年の長期視点 3ヶ月で成果は無理 @3: 内部人材を育てる 外部コンサルに依存しない 内部に文化を蓄積 @4: 失敗を許容する 「失敗 = 学び」の文化 心理的安全性が前提 @5: 小さく始める パイロットチーム1つから 成功事例を作る 徐々に拡げる // コンウェイの法則 「システムの設計は、 それを設計した組織の コミュニケーション構造を 模倣する」 — Melvin Conway, 1968 // 含意 組織を変えなければ システム(ソフト)も変わらない 組織変革は本気のシステム改革 // 鉄則 「文化は朝食に戦略を食う」 — Peter Drucker どんなに完璧な戦略でも 組織文化に逆らえば負ける // 結論 スケーリングは フレームワーク選びではなく 文化変革である その覚悟と時間がないなら 小規模なまま留まる選択も 立派なアジャイル
POINT —
Peter Drucker の「文化は朝食に戦略を食う」は、組織変革の格言。どんな完璧なアジャイルフレームワークを導入しても、文化（信頼・自律・失敗許容）が育たない限り形骸化します。「アジャイル変革は人の問題」──ここを腹に落とすことが、最大の出発点です。★★★
