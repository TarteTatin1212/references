01

アジャイルとは何か

WHAT IS AGILE

アジャイルは「方法論」ではなく「価値観」です。具体的な手順の集まりではなく、ソフトウェア開発に対する考え方の集合を指します。だからこそ「Scrumをやればアジャイル」は半分正解で半分間違い。まずはここを押さえます。

agile.definition アジャイルの定義と誤解
// アジャイルの三層構造
┌─────────────────────────┐ │ 価値観・原則 │ ← アジャイル宣言・12原則 │ （Mindset） │ ├─────────────────────────┤ │ 手法・フレームワーク │ ← Scrum / XP / Kanban │ （Methods） │ ├─────────────────────────┤ │ プラクティス │ ← TDD / ペアプロ │ （Practices） │ CI/CD / レトロ └─────────────────────────┘
  • 上から下へ：抽象 → 具体
  • 下から上へ：実装 → 思想
  • 全部つながって初めてアジャイル
// よくある誤解と真実 // ❌ 誤解1： // 「アジャイル＝Scrum」 → ScrumはアジャイルのOne of them // ❌ 誤解2： // 「アジャイル＝速い開発」 → 速さは結果。本質は"適応" // ❌ 誤解3： // 「ドキュメントを書かない」 → 必要なものは書く // ✅ 正しい理解： アジャイル = { 価値観: "変化への適応", 原則: "動くものを早く", 手法: ["Scrum", "XP", "Kanban"] }
POINT —
アジャイルは「やり方」より「あり方」。Scrumの儀式を全部こなしても、価値観がついてこなければ単なる「ミーティングが多い開発」になります。★★★
02

なぜ生まれたのか

WHY AGILE EMERGED

アジャイルを理解する最短ルートは、「何に対するアンチテーゼなのか」を知ることです。アジャイルはウォーターフォール開発の苦い経験から生まれた反動として誕生しました。ここを押さえると、なぜ「動くソフトウェア」「変化への対応」が掲げられるのかが腹落ちします。

waterfall.vs.agile 2つの開発スタイルの根本的な違い
// ウォーターフォールの罠
要件定義 ──┐ ↓ 設計 ──┐ ↓ 実装 ──┐ ↓ テスト ──┐ ↓ リリース 問題：最後まで動くものが見えない 途中で要件が変わると詰む ユーザーの本当の声が届かない
側面ウォーターフォールアジャイル
計画最初に全部必要な分だけ
動くもの最後に1回2週間ごと
変更歓迎されない歓迎する
顧客初期と最終のみ常に一緒
// アジャイルが解決したかった問題 問題1: 「作ったけど使われない」 → 半年かけて作ったら 要件が古くなっていた → 解決策: 短く回す 問題2: 「直前にバグ大量発覚」 → 統合テストで地獄を見る → 解決策: 継続的統合 問題3: 「ユーザーが望んでない物」 → 仕様書通りなのに不評 → 解決策: 早期フィードバック // 結論 アジャイル = "不確実性への適応戦略"
TIP —
ウォーターフォールが悪いのではありません。要件が固まっていて、変化が少ない領域（ビル建築、組み込みの一部）では今でも有効です。不確実性が高い領域でこそアジャイルが力を発揮します。★★
03

アジャイル宣言と12原則

THE AGILE MANIFESTO

2001年、米国ユタ州スノーバードに17人の開発者が集まり、たった68語の宣言を書き上げました。これが「アジャイルソフトウェア開発宣言」。シリーズ全巻の土台になる文書なので、まずは全体を眺めておきます。詳細は本館 Vol.2 で1つずつ丁寧に解説します。

manifesto.4values 4つの価値（左より右が大事という意味ではない）
// アジャイルソフトウェア開発宣言
プロセスやツール
←<←
個人と対話
包括的なドキュメント
←<←
動くソフトウェア
契約交渉
←<←
顧客との協調
計画に従うこと
←<←
変化への対応

上記4つを価値とする
※ 左にも価値はあるが、右により価値を置く

// 12原則の全体像（要約） @顧客満足 1. 早く継続的に価値を届ける 2. 変更要求を歓迎する @頻繁な提供 3. 動くソフトウェアを数週で 4. ビジネスと開発者は毎日協働 @モチベーション 5. 意欲ある個人を中心に 6. 対面コミュニケーション @持続性 7. 動くソフトウェアが進捗 8. 持続可能なペースを保つ @卓越性 9. 技術的卓越と設計に注意 10. シンプルさが本質 @自己組織化 11. 自己組織化チームから創発 12. 定期的に振り返り改善 // → Vol.2 で1つずつ深掘り
POINT —
宣言を読むときの最大の注意点は「左より右」を「左を捨てる」と読まないこと。ドキュメントも契約も計画も必要です。ただ、それらを目的化せず動くもの・対話・協調・適応を優先する、という意味です。★★★
04

主要な手法の地図

METHODS LANDSCAPE

アジャイル傘下には複数の手法があります。Scrum・XP・Kanban・Leanが四天王。それぞれ得意分野が違い、現場では組み合わせて使うのが定番です。「ScrumをベースにXPのプラクティスを取り入れる」が世界中で最もよく見るパターンです。

methods.comparison 4大手法の守備範囲
// 手法ごとの注力ポイント
┌─────────┬─────────────────────┐ │ Scrum │ チームの動かし方 │ │ │ プロジェクト管理 │ ├─────────┼─────────────────────┤ │ XP │ エンジニアリング │ │ │ コード品質・技術 │ ├─────────┼─────────────────────┤ │ Kanban │ ワークフロー可視化 │ │ │ フロー最適化 │ ├─────────┼─────────────────────┤ │ Lean │ ムダ取り・全体最適 │ │ │ 思想・経営 │ └─────────┴─────────────────────┘ ↓ 現場での組み合わせ ↓ Scrum + XP のプラクティス ＝ 世界の定番構成
// 各手法の代表的なキーワード Scrum { 役割: PO / SM / 開発者 イベント: スプリント, デイリー, レビュー, レトロ 成果物: バックログ, インクリメント } XP { 技術: TDD, ペアプロ, リファクタリング, CI, シンプル設計 } Kanban { 原則: 可視化, WIP制限, フロー管理, 継続改善 } Lean { 思想: ムダの排除, 全体最適, 学習の増幅 }
裏技 —
手法を純粋に1つだけ採用している現場はほぼありません。「Scrum-but（スクラムバット）」「ScrumBan」など、現場ごとにアレンジが入ります。大事なのは原則を理解した上で選び取ること。形だけ真似ると形骸化します。★★

アジャイルの1スプリント

A WEEK IN AGILE

抽象的な話が続いたので、具体的な1スプリント（2週間）がどう流れるかを覗いてみます。これがアジャイル開発者の典型的な日々のリズムです。「忙しそう」に見えるかもしれませんが、慣れるとこのリズムが心地よくなります。

sprint.timeline 2週間スプリントの典型的な流れ
// 2週間スプリント・カレンダー
Week 1 ───────────────────── Mon プランニング（午前） ├ 何を作るか決める └ タスク分解 開発スタート（午後〜） Tue デイリー15分 + 開発 Wed デイリー15分 + 開発 Thu デイリー15分 + 開発 Fri デイリー15分 + 開発 バックログリファインメント Week 2 ───────────────────── Mon デイリー15分 + 開発 Tue デイリー15分 + 開発 Wed デイリー15分 + 開発 Thu デイリー15分 + 開発（締切） Fri レビュー（午前） ├ 成果をデモ └ ステークホルダーFB レトロスペクティブ（午後） ├ 良かった点 └ 改善点 ↑ 翌週月曜から次のスプリント
// アジャイルなチームメンバーの1日 09:30 出社・前日のレビュー 10:00 デイリースクラム(15分) ├ 昨日やったこと ├ 今日やること └ 困っていること 10:15 開発・ペアプロ 12:00 ランチ 13:00 開発・コードレビュー 15:00 リファインメント参加 16:00 開発・CIでテスト 18:00 退勤 // 持続可能なペース（原則8） 残業は基本なし 毎日動くものを統合 毎日進捗を共有 // → 詳細は本館 Vol.3 Scrum編へ
TIP —
スプリント期間は1〜4週間で選択可能ですが、現場の8割は2週間スプリントです。短すぎると儀式の比率が増え、長すぎると軌道修正が遅れる。2週間が「ちょうどいい」とされる定番のチューニングです。★★
06

向く現場・向かない現場

WHEN AGILE FITS

アジャイルは万能薬ではありません。「うちもアジャイルやろう」と上が言い出して大失敗──これが現場あるあるです。アジャイルが得意な領域・苦手な領域を最初に把握しておくと、無理筋な導入を避けられます。

agile.fit アジャイル適性の判定基準
// 向く・向かないの判定
FIT — 向いている
  • 要件が不確実
  • 顧客と頻繁に対話可能
  • 市場変化が速い
  • 機能を段階的に出せる
  • チームに学習意欲
NOT FIT — 向いていない
  • 要件が完全に固定
  • 規制で全要件事前承認
  • 1回しか出せない（衛星等）
  • 顧客が指示を出さない
  • 変更を絶対認めない契約

※ 灰色地帯も多い。半分しか当てはまらないなら「ハイブリッド」も選択肢。

// アジャイル適性チェック if (要件.変更頻度 == "high") { return "アジャイル向き"; } if (顧客.参加可能 == false) { return "導入要注意"; } if (プロジェクト.種別 == "探索的") { return "アジャイル最適"; } if (規制.事前承認必須 == true) { return "ハイブリッド推奨"; } // 現場の鉄則 小さく始めて合うか試す 全部か無かで考えない 部分的導入も立派な選択
POINT —
「アジャイルかウォーターフォールか」の二択ではありません。1つの組織内で、プロジェクト特性に応じて使い分けるのが現実的。経営層には「適材適所」と伝えるのが鉄則です。★★★

このシリーズの歩き方

本シリーズは「本館9巻」＋「別館9巻」の二部構成です。本館は「どうやるか」の実践リファレンス、別館は「なぜそうなのか」を歴史・比較・構造から掘り下げる思索編。本館を読んだ直後に別館を開くのが推奨ルートです。
