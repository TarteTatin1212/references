カンバンとリーン編

KANBAN & LEAN — THE FLOW PARADIGM

Scrumのスプリント区切りに対し、カンバンは「流れ続ける」パラダイムです。トヨタ生産方式から生まれた日本由来の思想を、ソフトウェア開発に応用したのがカンバン（David J. Anderson, 2010）。リーンソフトウェア開発と合わせて、「ムダの排除」「フロー最適化」「リードタイム短縮」を実現します。

01

トヨタからソフトへ

FROM TOYOTA TO SOFTWARE

「カンバン」「リーン」のルーツはトヨタ生産方式。大野耐一が体系化し、戦後日本の自動車産業を世界一に押し上げた製造哲学です。これが1990年代に Womack & Jonesによって「リーン生産方式」として欧米に紹介され、2000年代にソフトウェア開発に応用されました。日本人として知っておきたい系譜です。

history.toyota トヨタ生産方式の系譜
// 思想の系譜
1948年 トヨタ │ │ 大野耐一が体系化 │ ジャストインタイム │ 自働化 │ 看板（かんばん） ↓ 1950s〜1980s トヨタ生産方式 日本の製造業に普及 │ │ Womack & Jones らが │ 欧米に紹介 ↓ 1990年 『The Machine That Changed the World』 「Lean」と命名 │ │ Mary & Tom Poppendieck が │ ソフトウェアに翻訳 ↓ 2003年 『Lean Software Development』 │ │ David J. Anderson が │ 看板をソフトに応用 ↓ 2010年 『Kanban』 システム化される
// トヨタ生産方式の2本柱 @Just In Time 必要なものを 必要なときに 必要なだけ作る → 在庫を最小化 @自働化 (じどうか) 異常があれば 機械が自分で止まる → 不良を増やさない // 「かんばん」(看板) とは 語源: 工程間で部品を 注文する「指示書」 役割: 後工程の必要量に 応じて前工程が動く (プル型) 対比 (プッシュ型): 上流が下流に押し付ける → 在庫過多になりがち // 日本人として誇る系譜 大野耐一『トヨタ生産方式』 野中・竹内 Scrum 起源 石川馨 QC・PDCA SECIモデル 知識創造 → アジャイルの源流の 多くが日本由来
POINT —
アジャイル界には「日本に学べ」という時代がありました。1980年代の日本企業の強さの研究から、Scrum・カンバン・リーンが派生したのです。カイゼン・カイカク・KAIZEN・KANBAN・MUDA──これらが英語でそのまま使われるのは、日本の経営哲学への敬意の証です。★★★

カンバンボードの基本

KANBAN BOARD

カンバンの中心にあるのがカンバンボード。「作業を見える化」するシンプルな仕組みで、これが現代のソフトウェア開発を変えました。JIRA・Trello・Asana──現在使われるツール全てがカンバンボード形式です。

TIP —
物理ボード（ホワイトボード＋付箋）には独特の力があります。デジタルツールより「触れる感触」「歩いて見に行く」「全員が同時に見る」が強い。リモート時代でも、オフィス出社日には物理ボードを併用する現場が増えています。★★
03

WIP制限の魔法

WIP LIMITS

カンバン最大の発明がWIP制限（Work In Progress Limit）。「同時着手数に上限を設ける」シンプルなルールですが、これが現場のフローを劇的に改善します。「忙しいのに進まない」という現場の謎は、WIP制限を入れた瞬間に解消することが多いのです。

kanban.wip WIP制限の効果
// WIP制限なし vs あり
[WIP制限なし] DOING: タスクが10個 全部少しずつ進む 完了がいつまで来ない ↓ コンテキスト切替 割込み増加 品質低下 [WIP制限あり (例: 3)] DOING: 3個まで 3個完了するまで 新規着手NG ↓ 集中して終わる 完了が速い 品質も上がる ━━━━━━━━━━━━━━━━━━━━━ 「始めるのをやめ、 終わらせるのを始めよう」 ━━━━━━━━━━━━━━━━━━━━━
// WIP制限がなぜ機能するか @リトルの法則 リードタイム = WIP数 ÷ スループット WIPを減らせば リードタイムは縮む (数学的に保証される) @マルチタスクの害 人間は同時に2つに集中不能 切替コスト 15〜30分 → タスク多いほど効率低下 @詰まりが見える WIP上限に達すると 「もう着手できない」状態 → 詰まり箇所が明確化 → 全員でその列を助けに行く // WIP制限の決め方 経験則: 各列のWIP = チーム人数 × 1.5 例: 5人なら WIP = 7〜8 段階的調整: まず緩めの制限から始める 徐々に厳しく 「窮屈」と感じる手前が最適 // よくある抵抗 「上限に達したから着手しない」 → 「サボってる」と誤解される 対策: 経営層に説明 「WIPを減らすほど早く終わる」 「数学的に証明されている」
POINT —
名言「"Stop Starting, Start Finishing"」（始めるのをやめ、終わらせるのを始めよ）はカンバンの心臓部。10個を半端に進めるより、3個を完了させる方が遥かに価値が出ます。WIP制限はそれを仕組みで強制する装置。詳細は別館 Vol.6 で。★★★
04

フロー最適化

FLOW OPTIMIZATION

カンバンは「流れ（フロー）」の最適化を目指します。スプリント区切りがないので、「常に何かを流し続ける」パラダイム。ボトルネック（詰まり）を見つけて解消し、流れを滑らかに保つのが運用の核です。

kanban.flow フロー観察とボトルネック
// ボトルネックの発見
Dev → QA → Deploy ──────────────────────────── 3 cards 8 cards 2 cards ↑ ↑ ↑ 普通 溜まってる 普通 QA がボトルネック ↓ 対策: ・QA担当を増やす ・自動テストで負荷軽減 ・QA前段で品質チェック ━━━━━━━━━━━━━━━━━━━━━ 最も遅い工程が 全体の速度を決める (Goldratt の制約理論) ━━━━━━━━━━━━━━━━━━━━━
// フローを観察する指標 @累積フロー図 (Cumulative Flow Diagram) 各列のカード数を時系列で → 詰まりが視覚化 @リードタイム分布 カードごとの所要時間 → ばらつき・予測可能性 @スループット 単位時間あたり完了数 → チームの実力 // フロー阻害要因 外部依存待ち 「他チームのレビュー待ち」 割り込み作業 「至急のバグ対応」 技術的負債 「設計が悪くて時間かかる」 知識属人化 「Aさんしかできない」 // フロー改善の手順 1. 観察 (1〜2週間) 2. ボトルネック特定 3. 仮説を立てる 4. 改善試行 5. 効果測定 6. 次のボトルネックへ // 鉄則 全工程を均等改善するより ボトルネックに集中せよ
裏技 —
「制約理論（Theory of Constraints）」はEliyahu Goldratt が提唱した、ボトルネック最適化の理論。著書『The Goal』は名著。「ボトルネック以外を最適化しても全体は速くならない」という非直感的な真実を、小説形式で教えてくれます。★★
05

リードタイムとサイクルタイム

LEAD & CYCLE TIME

カンバン運用の2大指標がリードタイムとサイクルタイム。混同されがちですが、明確に違う概念です。これを理解すると、「顧客満足」と「チーム効率」を別々に最適化できるようになります。

kanban.metrics リードタイム vs サイクルタイム
// 2つの時間軸
┌────────────────────────┐ │ 顧客から見た時間 │ │ │ │ 依頼 ━━━━━━━━━━━ 完了 │ │ ← リードタイム → │ │ (Lead Time) │ └────────────────────────┘ ┌────────────────────────┐ │ 開発から見た時間 │ │ │ │ 待機 → 着手 ━━━ 完了 │ │ ←サイクル→ │ │ (Cycle Time) │ └────────────────────────┘ リードタイム = 待機時間 + サイクルタイム 「待機時間」が 最適化の大きな鉱脈
// 各指標の意味 @Lead Time カード作成 → 完了 顧客の体感速度 「依頼してから何日?」 @Cycle Time 着手 → 完了 開発側の純粋作業時間 「動き出してから何日?」 @待機時間 Lead Time - Cycle Time =「順番待ちしてる時間」 // 改善のヒント Lead Timeを縮めたい → バックログを短く → 不要なものを切る → 優先順位を厳しく Cycle Timeを縮めたい → ストーリー小さく → WIP制限厳しく → 自動化推進 // 計測のしかた @散布図 完了したカードを点に X軸: 完了日 Y軸: サイクルタイム → ばらつきが見える @SLE設定 Service Level Expectation 「85%が10日以内」と公言 予測可能性の指標 // 鉄則 平均より「85%値」を見る 少数の外れ値が 体感を悪化させる
POINT —
「平均リードタイム5日」と言われても、実体験は「ばらつき」で決まります。85%パーセンタイル（85%のカードが○日以内）を指標にするのが現代の主流。SLE（Service Level Expectation）として顧客に公言することで、信頼性のあるチームに育ちます。★★★

リーン7つのムダ

SEVEN WASTES OF LEAN

トヨタ生産方式に由来する「7つのムダ」。Mary & Tom Poppendieck がソフトウェア版に翻訳しました。製造業の「在庫過多」「運搬」を、ソフトウェアの「未完成機能」「タスク切替」に対応させる視点が斬新で、現在もレトロで使える診断ツールです。

lean.7wastes ソフトウェア開発7つのムダ
// 7つのムダ・対応表
製造業ソフトウェア
在庫未完成機能 / 部分完成
余分な処理過剰機能・過剰設計
運搬引き継ぎ・部署間移送
待ち承認待ち・レビュー待ち
動作タスク切替・コンテキスト変更
欠陥バグ・手戻り
作りすぎ使われない機能

※ Poppendieckの本『リーンソフトウェア開発』（2003）で体系化されたソフト版。

// 7つのムダ・詳細 @1: 未完成機能 「ほぼできてる」が滞留 → 価値を生まない在庫 → 完成優先で消化 @2: 過剰機能 「将来使うかも」で実装 → 8割は使われない → YAGNI で防ぐ @3: 引き継ぎ 人から人への情報伝達 → 情報損失が起きる → クロス機能チームで削減 @4: 待ち 承認待ち・レビュー待ち → 単純に時間の浪費 → セルフサービス化で削減 @5: タスク切替 別作業に切り替える → 集中力15〜30分浪費 → WIP制限で削減 @6: 欠陥 バグ・手戻り → 最大のムダ → TDD・自動テストで削減 @7: 作りすぎ 使われない機能の量産 → リーン最大の敵 → 検証→学習で削減 // レトロでの使い方 「今週、どのムダが多かった?」 → 7軸での議論
TIP —
「7つのムダ」でレトロを構造化すると、改善行動が出やすい。「今週は『待ち』が多かった、誰の承認待ち?」「『過剰機能』を作ってないか?」──ピンポイントで詰めると改善案が具体的になります。★★

Scrumbanという折衷

SCRUMBAN — THE HYBRID

Scrum と Kanban は対立する概念ではなく、組み合わせ可能です。両者を融合したScrumbanは、Corey Ladas が2008年に提唱。「Scrumのチーム運営＋Kanbanのフロー管理」として現場で広く使われています。完全Scrumから卒業しつつあるチームに向く形態です。

scrumban Scrum・Kanban・Scrumbanの比較
// 3つの比較
ScrumScrumbanKanban
スプリント有(1〜4週)任意無
役割3つ厳格緩い無
イベント5つ厳格選択的無 or 任意
WIP制限暗黙的明示的必須
変更スプリント外柔軟常時OK
適合新規開発過渡期保守・運用
// Scrumban の典型構成 @Scrumから採用 ✓ スプリント (任意) ✓ デイリースクラム ✓ レトロスペクティブ ✓ プランニング(軽量化) @Kanbanから採用 ✓ WIP制限を明示 ✓ プル型ワークフロー ✓ リードタイム計測 ✓ 累積フロー図 // よくある移行パターン Scrum → Scrumban 運用が回りすぎて スプリントが形式的に → カンバン的要素を追加 Kanban → Scrumban 振り返りや計画が 足りないと感じる → スクラム儀式を追加 // どんな現場で有効か @保守＋新規ミックス 保守は流す 新規はスプリントで @オンコール持ち 割り込み多い Scrum純度は保てない @小〜中規模チーム 独自運用が許される 柔軟性が活きる // 注意点 "なんとなくScrumban" は危険 両方の良さを意図的に選び取る
POINT —
Scrumbanを名乗る現場の多くは、「Scrum-but の言い訳」になっていることがあります。「儀式やってないから Scrumban です」は危ない。「両方の長所を意図的に選ぶ」のがScrumbanの本来。「面倒だから簡略化」のSukurumbanとは別物です。★★★
08

Scrum vs Kanban の選択

SCRUM OR KANBAN

現場で「Scrumとカンバン、どちらを採用すべきか」と問われることがあります。答えは「仕事の性質」で決まります。計画立てて区切って進める仕事はScrum、流れる仕事・差し込み多い仕事はカンバン。これを言語化できると、上層部にも説明しやすくなります。

choose.method 手法選択の判断基準
// 仕事の種類で選ぶ
Scrum が合う
  • 新規プロダクト開発
  • 明確な目標がある
  • 2週間単位で計画OK
  • 差し込みが少ない
  • POが専任で動ける
  • 機能を束ねてリリース
Kanban が合う
  • 運用・保守業務
  • 差し込み多い
  • 個々の依頼が独立
  • 機能ごとに即リリース
  • サポートチーム
  • インフラ・SREチーム

※ 「両方やる」のもアリ。プロダクトチームはScrum、運用チームはKanban、等。

// 選択フローチャート Q1. 目標は固定 or 流動的? ├ 固定的 → Scrum 候補 └ 流動的 → Kanban 候補 Q2. 差し込み頻度は? ├ 少ない → Scrum 候補 └ 多い → Kanban 候補 Q3. リリース戦略は? ├ 機能束ねる → Scrum └ 都度リリース → Kanban Q4. 中間的な状況なら? → Scrumban // 移行のタイミング @Scrumから Kanban へ ✓ プロダクトが安定期 ✓ 機能追加が運用化 ✓ スプリントが形骸化 @Kanbanから Scrumへ ✓ 新規開発フェーズ突入 ✓ チームに計画リズム必要 ✓ 振り返り機会増やしたい // 現実の使い分け例 プロダクトチーム: Scrum 新機能開発に集中 運用チーム: Kanban 問い合わせ流れ続ける QAチーム: Scrumban 両方のチームと連携 // 鉄則 手法は手段 仕事に合わせて選ぶ 逆ではない
POINT —
Scrum vs Kanban は「宗教戦争」になりがち。「Scrumこそ正義」「いやKanbanこそ柔軟」と派閥対立する場面を見かけますが、仕事の性質に合わせて選ぶのが大人の態度。1つの組織で複数の手法を使い分けるのが現代の主流です。★★★
