Q.01 スプリントが終わらず、毎回未完了タスクが残ります SPRINT
A. ANSWER — まず原因を診断する
// 典型的な原因5つ
  • 詰め込み過ぎ ── ベロシティ無視で計画
  • ストーリーが大きすぎる ── 1スプリント内に収まらない
  • 受け入れ基準が曖昧 ── 「完了」の定義が共有されてない
  • 外部依存待ち ── 他チーム・承認の待ち時間
  • 割り込みが多い ── スプリント外作業が発生
NG な対処
残業で完了させる
DoD を下げる
完了基準を緩める
OK な対処
次スプリント分を減らす
ストーリーを分割する
割り込み源を可視化
// 処方箋（手順） 1. 過去3スプリントのベロシティを観察 → 平均値を出す 2. 次スプリントを平均値の80%で計画 → 余白を作る 3. ストーリーを8ポイント以下に分割 → 大きいものは禁忌 4. 受け入れ基準をGiven-When-Thenで明文化 → 「完了」を一意に 5. 割り込みをバッファ枠で吸収 → 1日1ポイント等 6. レトロで外部依存を可視化 → 待ち時間を計測 // 関連: Vol.4 ストーリー分割 // 関連: Vol.6 WIP制限
TIP —
「とにかく残業で終わらせる」は最悪。持続可能なペース（原則8）違反です。スプリントが終わらないのは「計画ミス」のシグナル。残業で隠さず、「次回の計画を見直す材料」として正直に受け止めるのが鉄則です。★★★
Q.02 プロダクトオーナーが決めてくれず、開発が止まります PRODUCT OWNER
A. ANSWER — POの状況を診断する
// POが決められない3パターン
  • 権限がない ── 上司に毎回お伺いが必要
  • 兼任で時間がない ── 本業との二足の草鞋
  • 判断材料が不足 ── ユーザー視点・データが不在

POの「決断力不足」は、本人の問題というより構造的問題。原因に応じて対処が変わります。

NG な対処
POを批判する
開発者が代わりに決める
「決まらないから止める」
OK な対処
権限委譲を交渉する
判断材料を一緒に集める
意思決定の場を定期化
// パターン別の処方箋 @権限がない場合 → SM が権限委譲交渉 経営層に「POに決定権を」 と働きかける → POを本物のPOにする @兼任の場合 → 兼任は致命的 「専任PO配置」を交渉 → 不可能なら PO代行(Proxy)を立てる 定期的な意思決定窓口に @判断材料不足の場合 → ユーザー調査・データ収集 開発チームも巻き込み → 「材料が揃ったから決められる」 状況を作る // 緊急避難策 意思決定が止まる時: 「タイムアウト方式」 - POに 24h で決めてもらう - 期限を過ぎたら開発者の判断 - 後で覆ったら追加タスクへ // 関連: Vol.3 POの責任 // 関連: Vol.8 組織変革の現実
POINT —
POが決められない問題の9割は「構造」が原因。本人を責めても解決しません。「POが機能する組織構造」を作るのがScrumMasterと経営層の責任。PO専任配置・権限委譲──ここに投資しない組織はアジャイルが回りません。★★★
Q.03 見積もりが毎回外れます。どうすれば精度が上がりますか ESTIMATION
A. ANSWER — 「当てる」のではなく「学ぶ」
// 見積もりが外れる原因
  • 絶対時間で見積もる ── 人間の認知に反する
  • ストーリーが大きすぎる ── 不確実性が膨張
  • 1人で見積もる ── アンカリングや認識ズレ
  • 「やったことない作業」 ── スパイクが必要
  • 過剰な精度を求められる ── 心理的プレッシャー
改善のヒント
✓ ストーリーポイント使用
✓ 8点超は分割
✓ プランニングポーカー
✓ 不明はスパイク
よくある罠
✗ 「1ポイント＝何時間？」
✗ ベロシティを目標化
✗ 個人ノルマで管理
✗ 完璧主義
// 見積もり精度向上の手順 1. ポイント制に切り替える 時間見積もりをやめる 2. 過去ストーリーから基準点を作る 「これが3点」を決める 3. プランニングポーカーで議論 全員独立判断 → 議論 4. 8点超は必ず分割 13点・21点は禁忌 5. 不確実ならスパイクを打つ 調査だけのタスク 6. ベロシティを観測値として扱う 目標値・KPIにしない // 別館 Vol.4 で深掘り なぜポイントなのか なぜフィボナッチなのか 不確実性円錐とは何か // 鉄則 「当てる」より「学ぶ」 ハズレ続きでも データが溜まれば 予測可能性が上がる
裏技 —
見積もり精度より「リードタイムの予測可能性」を指標にするのが現代の主流。「85%パーセンタイル」で「85%のストーリーは10日以内」と言える方が、見積もり個別の精度より価値があります。当てることに執着しない勇気が大事。★★
Q.04 レトロスペクティブが形骸化して、毎回同じ話に RETRO
A. ANSWER — 手法と空気を変える
// レトロ形骸化の原因
  • 毎回同じKPT ── 思考パターンが固定化
  • 改善案が実行されない ── 出しても変わらない無力感
  • 本音が出ない ── 心理的安全性が低い
  • SMが主導しすぎ ── 自己組織化が育たない
  • 議論が抽象的 ── 具体的な行動に落ちない
// 改善策 @手法を変える KPT に飽きたら: ✓ YWT (やった・分かった・次) ✓ Mad/Sad/Glad ✓ Start/Stop/Continue ✓ 4Ls ✓ セーリングボート → 3〜4回ごとに変える @1〜2個に絞る 改善案大量より 「次スプリントで実行する1つ」 確実に実行する @担当・期限を決める 「○○が来週までに」 曖昧な改善は実行されない @時々ファシリ役を変える SM以外がファシリ → 視点が変わる @心理的安全性を作る 「ここで話したことは外に出さない」 ルールを明文化 // 関連: Vol.3 レトロ // 関連: 心理的安全性
POINT —
レトロ形骸化の最大の原因は「改善案が実行されない」こと。「毎回出しても何も変わらない」と分かると、チームは出さなくなります。1つでもいいから必ず実行する。次のレトロで「前回の改善どうだった？」と振り返る。この「PDCA回しの儀式」が機能して初めてレトロは価値を生みます。★★★
Q.05 経営層にアジャイルの価値を説明できません MANAGEMENT
A. ANSWER — 経営層の言語で語る
// 経営層が求める情報
  • 数字 ── ROI・コスト削減・売上向上
  • リスク低減 ── 大失敗を避けられる根拠
  • 競争優位 ── 他社より速く市場対応
  • 具体例 ── 「あの会社もこれで成功」

「アジャイル宣言」「Scrumガイド」を熱く語っても伝わりません。経営層の言語で価値を翻訳する必要があります。

// 経営層への翻訳例 @×悪い説明 「アジャイル宣言を実践 個人と対話を重視 動くソフトウェアを優先」 → 「で、それで何が良いの?」 @○良い説明 「2週間ごとにリリース 市場反応を見ながら方向修正 → 不要機能を作らずコスト50%減 → 競合より3ヶ月早く市場投入」 → 「具体的でわかる」 // 使える数字 DORA 4 Key Metrics Eliteチーム vs Lowチーム 生産性 2倍 リリース頻度 200倍 復旧時間 1/2000 → 客観データで説得力 McKinsey 調査 アジャイル組織は 顧客満足度+30% 従業員満足度+30% 業績+20% // 説明のコツ @小さく実証する 1チームでパイロット 3ヶ月で成果データ → 拡大を説得 @経営層の関心軸を聞く コスト?スピード?品質? → その軸でメリットを語る // 関連: Vol.7 DORA Metrics // 関連: Vol.8 組織変革
TIP —
経営層の最大の関心は「リスクとコスト」。アジャイルの「短いリリースサイクル」を、「失敗の小型化・早期発見」と訳すと刺さります。「半年作って失敗するより、2週間で失敗できる方が損害が小さい」──これが経営層の言語です。★★★
Q.06 リモートワークで対話が減り、チームが分断しています REMOTE
A. ANSWER — 仕掛けないと対話は起きない
// リモート特有の問題
  • 偶発的な会話が消える ── 廊下立ち話なし
  • 非言語情報が乏しい ── 表情・空気が伝わらない
  • 遠慮が増える ── 「忙しそうだから後で」
  • 新メンバーが孤立 ── 関係性を築きにくい
// リモート対応のテクニック @1: 雑談チャンネル/時間 Slack の雑談部屋 朝の5分雑談タイム 「お茶会」セッション @2: 常時接続の場 Gather, oVice, Tandem 「同じ部屋にいる感覚」 気軽に話しかけられる @3: カメラON文化 デイリーはカメラON 表情が見える 「孤独感」が減る @4: 非同期化 全部リアルタイム不要 Loom 等で動画コミュ 時差越え可能 @5: 物理オフィスデー 月1〜2回の集合日 「人を知る」時間 対話の貯金を作る @6: ペアプロ・モブプロ リモートでこそ強力 画面共有+音声+チャット 集中時間の共有 // ファシリ的工夫 ラウンドロビン: デイリーで順番に発言 「次は○○さん」 発言機会を均等に チェックイン質問: 「今日の気分を一言」 人柄が見える // 関連: 原則6 対面の会話
POINT —
アジャイル原則6「対面の会話が最も効率的」はリモート時代でも有効ですが、「対面」を「カメラON＋音声」と読み替えるのが現代解釈。物理対面ベストですが、リモートでも「意図的に対話の場を作れば」ある程度補えます。「対話は仕掛けないと起きない」──これが現代の鉄則です。★★★
Q.07 スピード優先で技術的負債が積み上がっています TECH DEBT
A. ANSWER — 負債を可視化し、定期返済する
// 技術的負債が積む原因
  • 機能追加のプレッシャー ── 改善時間がない
  • POが改善ストーリーを後回し ── 価値が見えにくい
  • テスト不足 ── 怖くてリファクタできない
  • 境界線がない ── 「ついでに直す」が積み上がる
NG な対処
「リファクタリング
スプリント」を提案
→ 経営層に拒否される
OK な対処
機能開発に組み込む
「20%ルール」など
毎スプリント返済
// 処方箋 1. 負債を可視化する バックログに「負債リスト」 影響を金額換算してみる 2. 20%ルールを導入 各スプリントの20%を 負債返済に充てる → 機能開発と並行 3. ボーイスカウト・ルール 触ったコードは綺麗にして去る 日常的な微改善 4. 新規機能のDoDに品質基準 「テストカバレッジ80%以上」 「Lint通過」 → 新規負債を作らない 5. 経営層に負債コストを提示 「このまま放置で 1年後にリリース速度半減」 数字で説明 // 比喩で説明する 技術的負債 = 金融負債 ✓ 利息（メンテコスト） ✓ 元本（修正にかかる時間） ✓ 放置 = 利息累積 ✓ 計画的返済が合理的 // 関連: Vol.5 リファクタリング // 関連: Vol.7 自動テスト
TIP —
Martin Fowler の「技術的負債 4象限」──「無謀/慎重 × 意図的/不注意」の組み合わせで負債を分類すると、対処が明確になります。「不注意で無謀な負債」は最悪、「意図的で慎重な負債」はビジネス上の合理的選択。負債を全部否定するのも極端です。★★
Q.08 デイリースクラムが進捗報告会になってしまっています DAILY
A. ANSWER — 質問を変える
// 報告会化の原因
  • SMやリーダーに向かって話している
  • 「3つの質問」を機械的に消化
  • スプリントゴールを忘れている
  • 議論ではなく順番待ち
// 改善策 @質問を変える 古典：3つの質問 ① 昨日何やった? ② 今日何やる? ③ 困りごとは? → 報告会化しやすい 現代：スプリントゴール起点 ① ゴールに対する進捗は? ② 計画を変える必要は? ③ 助け合える部分は? → 議論と調整に @立ち話形式に戻す 座らない カメラ近く → 緊張感を維持 @SMに向かって話さない チーム全員に向かって 「みんなに伝えるべき情報」のみ @議論はその場で発生させる 「Aさん詰まってる → 今日ペアでやろう」 即決即実行 @時々スキップしてみる 「今週はデイリー無し」 本当に必要だったか確認 → 形骸化を防ぐ // 関連: Vol.3 デイリースクラム // 関連: Scrumガイド 2020 改訂
POINT —
Scrum Guide 2020 で「3つの質問」が削除されたのは、まさにこの問題のため。「形式」より「目的」──スプリントゴールに向かう議論であれば、どんな形式でも構いません。毎日同じ質問を機械的に消化するのが最も避けたい形です。★★★
Q.09 ベロシティが評価指標にされて、ポイント水増しが起きています VELOCITY
A. ANSWER — グッドハートの法則
// グッドハートの法則

「測定値が目標になった時、それは良い測定値ではなくなる」

  • ベロシティをKPI化 → 水増し発生
  • 個人評価に使う → チームワーク崩壊
  • チーム間比較 → そもそも比較不能
  • 目標値設定 → 信頼性消失
// 経営層への説明テンプレ @ベロシティとは ストーリーポイント = チーム内の相対的見積もり = チームごとの単位 @比較不能の理由 Aチームの 5点 ≠ Bチームの 5点 単位がそもそも違う メートルとフィートを 単純比較できないのと同じ @評価に使う弊害 ✗ ポイント水増し → 見積もり信頼性崩壊 ✗ 大きいものから着手 → 価値順序が乱れる ✗ レビューが甘くなる → 品質低下 // 代替指標を提案 使うべきメトリクス: ✓ リードタイム ✓ デプロイ頻度 ✓ 顧客満足度 ✓ MTTR (障害復旧時間) ✓ 変更失敗率 → DORA 4 Key Metrics 使うべきでない: ✗ ベロシティ ✗ コード行数 ✗ コミット数 ✗ チケット解決数 // 鉄則を経営層と共有 「ベロシティは観測値、 KPI化したら死ぬ」 これを上層部に 最初に明示するのが SM の重要任務 // 関連: Vol.4 ベロシティ // 関連: Vol.7 DORA
POINT —
「グッドハートの法則」はあらゆる指標に当てはまる金言。「ベロシティを評価指標に」と言われた時点で、それは「アジャイルではなくなる」サイン。SMが断固として守るべき線です。経営層にDORA 4 Key Metricsを代替提案するのが現実解です。★★★
Q.10 アジャイルやってる体だけで、本質的に変わってない気がします CULTURE
A. ANSWER — Cargo Cult Agile に陥っている可能性
// Cargo Cult Agile の症状
  • 儀式は全部やっているが、何も改善されない
  • JIRAは完璧だが、対話は減った
  • ベロシティは安定だが、価値が出てない
  • 「アジャイルやってます」と言える形だけ
  • 変化への適応が起きていない

「カーゴ・カルト」とは、南太平洋の島民が米軍機を真似て木製の管制塔を作った逸話から。「形だけ真似て本質が伴わない」状態の比喩です。

// 本質に立ち返るチェックリスト @4つの価値、機能してる? ✓ 個人と対話 > プロセス → 対話が増えたか? ✓ 動くソフトウェア > ドキュメント → リリース頻度上がったか? ✓ 顧客との協調 > 契約 → 顧客の声が反映されてるか? ✓ 変化への対応 > 計画 → 計画修正が許される文化か? @12原則、生きてる? 特に確認すべき: - 原則1: 早く継続的に価値提供 - 原則4: ビジネス側と毎日協働 - 原則8: 持続可能なペース - 原則11: 自己組織化チーム - 原則12: 定期的な振り返り // 立て直しの手順 1. 原点回帰のレトロ 「私たちは本当に アジャイル宣言の 価値観を実践してる?」 全員で議論 2. 1つだけ本質的改善を選ぶ 「対話が減った」なら 雑談時間導入から 小さく始める 3. 形より結果を測る 「儀式やってるか」ではなく 「価値が出てるか」 4. 必要なら儀式を捨てる勇気 形骸化したデイリーは 一時休止してみる 本当に必要か検証 // 関連: 別館 Vol.2 4価値の構造 // 関連: 別館 Vol.9 (予定) なぜ失敗するか
POINT —
「儀式を全部やってるのに何も変わらない」状態は、業界で「Dark Scrum」「Zombie Scrum」とも呼ばれます。本質は「価値観」。儀式は道具に過ぎません。「我々は本当に変化に適応する組織になったか」──この問いを年1回でも全員で議論する習慣が、形骸化を防ぎます。詳細は別館 Vol.9 で。★★★
