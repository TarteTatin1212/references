EXTREME PROGRAMMING — TECHNICAL EXCELLENCE

01

XPの全体像

XP AT A GLANCE

XPは「優れたソフトウェア開発手法を極限まで実践する」という哲学。1996年、Kent Beck がクライスラー社の給与システム（C3プロジェクト）で実践し、1999年に書籍『Extreme Programming Explained』で世に広めました。「極端（extreme）」とは「良いと分かっていることを徹底的にやる」という意味です。

xp.overview XPの構造と歴史
// XPの「極端」の意味
良いプラクティスXPの極端化
コードレビューは大事→常にレビュー（ペアプロ）
テストは大事→テストを先に書く（TDD）
設計は大事→毎日設計し直す（リファクタ）
統合テストは大事→毎日何度も統合（CI）
顧客との対話は大事→顧客がチームに常駐（On-site Customer）
短い反復は大事→1〜2週の反復
// XPの歴史 1996: Kent Beck が Chrysler C3 プロジェクトで実践 1999: 『XP Explained』初版 2000: 第1次 XP ブーム 2004: 『XP Explained』第2版 (大幅改訂・進化) 現代: XP単独より、Scrum+XP プラクティスとして普及 // XPの3層構造 @5つの価値 (Values) コミュニケーション シンプリシティ フィードバック 勇気 尊敬 @原則 (Principles) 価値を実践に 橋渡しする @プラクティス (Practices) TDD, ペアプロ, CI, リファクタリング, シンプル設計 etc 計13プラクティス // 結論 XP は技術主義派の総本山 コード品質を妥協しない
POINT —
XP単独で導入している現場は今や少数派。「Scrum＋XP」が世界の定番です。Scrumのチーム運営とXPの技術プラクティスを組み合わせる──これが世界の主流になっています。Scrumだけだと「品質が崩壊する」事例が多発した教訓です。★★★
02

XPの5つの価値

FIVE VALUES OF XP

XPの価値観は5つ。コミュニケーション・シンプリシティ・フィードバック・勇気・尊敬。アジャイル宣言が抽象的なのに対し、XPの価値はもっと具体的・現場的。「今日のあなたのコードレビューは、この5つに従っていますか？」と問えるレベルの実用性があります。

xp.values 5つの価値の意味
// 5つの価値（XP第2版）
  • Communication（コミュニケーション）
    問題の多くは伝達不足。話し、聞き、共有する
  • Simplicity（シンプリシティ）
    「今日必要な最も単純な解決策」を選ぶ勇気
  • Feedback（フィードバック）
    早く・頻繁に・正直に。失敗も成功もすぐ見る
  • Courage（勇気）
    正直に話す。動かないコードを捨てる。難問に取り組む
  • Respect（尊敬）
    互いを尊重する。技能・意見・人格すべて
// 価値の相互関係 @5価値は連動する 勇気 + 尊敬 なしには コミュニケーション困難 フィードバック なしには シンプリシティは危険 (本当に必要か判断不能) @日々の使い方 「この設計、複雑すぎない?」 → Simplicity を問う 「今動かないこと、言える?」 → Courage を問う 「この機能、誰のため?」 → Communication を問う 「本当にユーザーに届いた?」 → Feedback を問う 「あの人の意見、検討した?」 → Respect を問う // 鉄則 価値はコード品質の判断軸 具体的な行動に翻訳できる
TIP —
レトロスペクティブで「この1週間、5つの価値どれが弱かったか？」と問うと、いい議論になります。Courageが弱い → 言いにくいことが言えてない、Respect が弱い → 意見対立で角が立った等、診断ツールとして優秀です。★★
03

テスト駆動開発（TDD）

TEST DRIVEN DEVELOPMENT

TDDは「テストを先に書いてから実装する」という、業界の常識をひっくり返した手法。Kent Beck が体系化し、現在のテスト文化のデファクトになりました。Red → Green → Refactorの3ステップを高速回転させるのがTDDの核です。

xp.tdd TDDの3ステップ
// Red-Green-Refactor サイクル
┌────────────────────┐ │ RED │ │ 失敗するテストを書く │ │ まだ実装はない │ │ → テスト失敗(赤) │ └─────────┬──────────┘ │ ┌─────────▼──────────┐ │ GREEN │ │ 最小限の実装 │ │ テストを通すだけ │ │ → テスト成功(緑) │ └─────────┬──────────┘ │ ┌─────────▼──────────┐ │ REFACTOR │ │ コードを綺麗に │ │ テストは緑のまま │ │ → 設計が改善 │ └─────────┬──────────┘ │ └─→ 次のRED へ ━━━━━━━━━━━━━━━━━━━━━ 数分〜十数分で1サイクル ━━━━━━━━━━━━━━━━━━━━━
// 具体例：消費税計算 @Step 1: RED test('100円の税込価格は110円', () => { expect(addTax(100)).toBe(110); }); → addTax 未実装で失敗 @Step 2: GREEN function addTax(price) { return price * 1.1; } → テスト通過 @Step 3: REFACTOR const TAX_RATE = 0.10; function addTax(price) { return price * (1 + TAX_RATE); } → マジックナンバー除去 // TDDの効果 ✓ 設計の改善 「テスト可能な設計」を 強制される → モジュール化進む ✓ デグレ防止 リファクタの安全網 恐れずに変更できる ✓ ドキュメント化 テストが仕様書を兼ねる 「コードはこう動く」が明確
POINT —
TDDは「テスト技法」ではなく「設計技法」です。「先にテストを書く」ことで、テストしやすい＝モジュラーな設計が自然に育ちます。Kent Beck 自身が「TDDは設計の道具だ」と繰り返し述べています。詳細は別館 Vol.5 で。★★★
04

ペアプログラミング

PAIR PROGRAMMING

2人で1台のキーボードを共有して開発するのがペアプロ。「人件費が2倍では？」が初見の反応ですが、「品質・知識共有・教育コスト」を含めると割安と立証されています。XPの中で最も議論を呼んだプラクティスです。

xp.pair ドライバーとナビゲーター
// ペアプロの役割
┌─────────────────────────┐ │ Driver (運転手) │ │ キーボード操作 │ │ 「目の前の数行」に集中 │ └─────────────────────────┘ ↕ 会話 ┌─────────────────────────┐ │ Navigator (案内人) │ │ 俯瞰的に見る │ │ 「全体の方向」「次の手」 │ │ バグ・タイポを指摘 │ └─────────────────────────┘ 数分〜15分で交代 両者が両方の役を体験

※ 集中力の高い「フロー状態」に2人で入る感覚。1人より深く考えられる。

// ペアプロの効果 @品質 バグが30〜50%減少 常時レビュー状態 → 投入後の修正コスト激減 @知識共有 「あの人にしか触れない」 コードが消える バス係数 (Bus Factor) ↑ @教育 シニア+ジュニアのペアは 優れた OJT ペアプロ1ヶ月 ≒ OJT 数ヶ月 @集中持続 1人だと迷走する局面でも 2人なら抜け出せる // 数字で見る効果 ペアプロの追加工数: 約15%(2倍ではない) ペアプロの削減効果: バグ修正・教育・属人化解消 → 総合では15%プラスに → 長期では30%以上プラス // よくある誤解 「人件費2倍」 → 違う。生産性は1.5〜1.8倍 → コストは1.85倍程度 → 品質込みなら得 // 鉄則 全時間ペアでなくていい 重要な部分・難所だけ
裏技 —
「モブプログラミング」はペアプロの拡張版で、3〜5人で1台を共有。新メンバー教育や、技術的に難しい局面で特に有効。Hunter Industriesの事例（モブプロでパフォーマンス激増）は有名です。フルタイムモブが現実的でなくても、難所だけモブは強力な武器になります。★★
05

リファクタリング

REFACTORING

リファクタリングは「外部の振る舞いを変えず、内部構造を改善する」こと。Martin Fowler の1999年の名著『Refactoring』で体系化されました。XPの世界では、これを「機能追加とは別の活動」として明確に区別し、毎日コツコツ続けるのが原則です。

xp.refactor リファクタリングの哲学
// リファクタリングの2つの帽子
帽子機能追加リファクタリング
目的新しい振る舞い同じ振る舞い
テスト追加・更新変更しない
判定仕様通り動く前と同じ動き

※ 同時に両方の帽子をかぶらないのが鉄則。混ぜると変更の意図が曖昧になり、バグの温床になる。

  • 変数名を改善する
  • 重複コードをまとめる
  • 長いメソッドを分割
  • クラスを分離・統合
// 代表的なリファクタリング技法 @Extract Method 長いメソッドを分割 // Before function checkout() { // 100行... 税金計算 / 在庫確認 / 送信 } // After function checkout() { calculateTax(); checkInventory(); sendOrder(); } @Rename Variable 意味のある名前に // Before const d = new Date(); // After const orderDate = new Date(); @Inline / Extract Variable ばらしたり束ねたり @Move Method メソッドを別クラスへ // 鉄則 テストがあるからこそ リファクタリングできる テストなしのリファクタは 「祈り」と呼ばれる // ボーイスカウト・ルール "Always leave the code cleaner than you found it" 来たときより綺麗に去る
POINT —
リファクタリングは「リファクタリングタスク」として独立させるものではなく、日々の開発の中で常時実行するもの。「リファクタリング・スプリント」「リファクタリング・タスク」を作るのはアンチパターン。毎日のボーイスカウト・ルールで実践するのが本来の姿です。★★★
06

シンプル設計

SIMPLE DESIGN

XPの設計哲学は「今日必要な最も単純な設計」。「将来きっと必要になる」と先回りで複雑な設計を入れることを強く否定します。これはYAGNI原則（You Aren't Gonna Need It）として知られます。シンプルさは怠惰ではなく、最高難度の技術です。

xp.simple Beckの4ルール
// シンプル設計の4ルール
  • テストを通す ── 仕様を満たす
  • 意図を表現する ── 読めばわかる
  • 重複がない ── DRY原則
  • 要素が最小限 ── 余計な機能を作らない

※ 順位は上から。テスト通過が最優先、要素最小化は最後。

YAGNI 違反
「将来きっと多言語化
するから i18n 入れよう」
→ 多言語化、結局しない
YAGNI 遵守
「今日は日本語だけ。
必要になったらその時」
→ 必要時にきれいに導入
// YAGNI 原則 "You Aren't Gonna Need It" = あなたにそれは要らない // なぜシンプルか @予測は当たらない 「将来要る」の8割は要らない → 先回り設計は無駄 @複雑さは負債 早期に入れた抽象化は 間違っていれば 剥がすコストが膨大 @変更コスト低い シンプルなら 必要になった時 すぐ拡張できる // 思考実験 "将来こうしたい" と考えた時: ① 今すぐやるか ② やめるか → 「後でやる」は最悪 曖昧な設計を作るだけ // シンプルとイージーは違う Simple(単純): 根本的にシンプル → 努力の成果 Easy(楽): 手数が少ない → 必ずしも単純でない Rich Hickey "Simple Made Easy" の名講演を参照
TIP —
シンプル設計の対極にあるのが「Astronaut Architecture」──宇宙飛行士のように抽象階層を積み上げる過剰設計。「いま読めて、いま動いて、いま変えやすい」を満たせばOK。将来のための今を犠牲にしないこと。★★★
07

継続的インテグレーション

CONTINUOUS INTEGRATION

XPが業界に与えた最大の遺産が継続的インテグレーション（CI）でしょう。「全員が毎日何度もコードを統合する」──当時は革命的な提唱でした。これがなければ現代のDevOps文化は成立しません。詳細は本館 Vol.7 でじっくり扱いますが、ここで概念を押さえます。

xp.ci 統合地獄から継続的統合へ
// 統合の歴史
[統合地獄の時代] 各人が長期間ブランチで作業 │ │ 数週間〜数ヶ月 ↓ いざ統合 → 大量コンフリクト → 動かない → 数日〜数週デバッグ [CIの時代] 全員が毎日マスターに統合 │ │ 1日に複数回 ↓ 小さな差分の統合 → コンフリクト最小 → 自動テストで品質維持 ━━━━━━━━━━━━━━━━━━━━━ 「痛いことは頻繁に」 ━━━━━━━━━━━━━━━━━━━━━
// XPが提唱したCI要素 @頻繁な統合 最低でも1日1回 理想は数時間に1回 → 差分を小さく保つ @自動ビルド ボタン1回でビルド成功 人手介在不要 @自動テスト ビルド時に全テスト実行 失敗時は即発覚 @速い失敗 10分以内に結果を知る → 待たない・止めない // 現代のCIツール GitHub Actions GitLab CI CircleCI Jenkins Travis CI // CIに含まれる流れ 1. git push 2. ビルド開始 3. 単体テスト 4. 静的解析 5. 統合テスト 6. 結果通知 7. NG なら即修正 // 詳細は Vol.7 で CI から CD / CDP へ
POINT —
CIの原典的定義は Martin Fowler の「Continuous Integration」（2006）。「少なくとも1日1回、できれば1日複数回、メインラインに統合する」が核。長期ブランチは禁忌──これがXP発祥のCI原則です。★★★
08

ScrumとXPの融合

SCRUM + XP HYBRID

現代の現場は「Scrumの骨格＋XPの技術プラクティス」の組み合わせが定番です。Scrumは「何を作るか」を扱うが「どう作るか」には踏み込まない。だからXPの技術プラクティスを組み合わせることで初めて持続可能な開発になります。

scrum-xp.combo 役割分担と統合像
// Scrum vs XP の補完関係
領域ScrumXP
WHAT○ バックログ管理—
WHO○ 3-5-3 構造—
WHEN○ スプリント○ 反復
HOW—○ 技術プラクティス
QUALITY△ DoDの一部○ TDD/リファクタ
COLLABORATION○ イベント○ ペアプロ

※ Scrumは「枠組み」、XPは「中身」。両者は本来排他ではなく補完。

// 典型的なScrum+XP構成 @Scrumから採用 ✓ スプリント(2週間) ✓ プランニング ✓ デイリー ✓ レビュー ✓ レトロ ✓ PO/SM/開発者の3役割 ✓ プロダクトバックログ @XPから採用 ✓ TDD ✓ ペアプロ ✓ リファクタリング ✓ CI/CD ✓ シンプル設計 ✓ コーディング規約 ✓ コードの共同所有 // この組み合わせの効果 Scrum単独: 品質が時間で崩壊 技術的負債が累積 「儀式だけ立派」状態 XP単独: 技術は素晴らしいが 顧客との接続が弱い 経営層に説明しにくい Scrum + XP: 両者の弱点を補完 顧客価値 + 技術品質 持続可能な開発 // 鉄則 Scrumだけは半分 XPだけも半分 合わせて全部
POINT —
「ScrumとXPは対立する」と書く古い書籍もありますが、これは誤解。スノーバード会議の17人にはScrum派もXP派もいて同じ宣言に署名しました。「Scrumの上にXPプラクティスを乗せる」が現代の世界標準。「Scrum-but」と呼ばれるアレンジの大半は、実はScrum+XPの形をしています。★★★
