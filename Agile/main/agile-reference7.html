01

CI/CDの全体像

CI/CD OVERVIEW

CI/CDは3つの段階を持ちます。CI（Continuous Integration）＝継続的統合、CD（Continuous Delivery）＝継続的デリバリー、CD（Continuous Deployment）＝継続的デプロイ。略すと同じ「CD」ですが、明確に違う概念です。ここを区別できると、現場で混乱しません。

cicd.overview CI / Delivery / Deployment の違い
// 3段階の関係
┌──────────────────────────────┐ │ CI │ │ Continuous Integration │ │ コードを統合し、テストする │ │ → ビルド+自動テストが成功 │ └────────────┬─────────────────┘ │ ┌────────────▼─────────────────┐ │ Continuous Delivery │ │ いつでも本番に出せる状態を │ │ 自動的に維持 │ │ → デプロイは人間がボタン押す│ └────────────┬─────────────────┘ │ ┌────────────▼─────────────────┐ │ Continuous Deployment │ │ テスト通過したら自動で │ │ 本番デプロイまで実行 │ │ → 完全自動・人間介在なし │ └──────────────────────────────┘ 上から下へ自動化が進む
// 3つの違い・詳細比較 @CI (Continuous Integration) 目的: 統合を継続的に 対象: ソースコード 自動化: ビルド・テスト 結果: 「動くマスター」維持 @Continuous Delivery 目的: いつでも出せる状態 対象: デプロイ可能パッケージ 自動化: 環境準備・配信 結果: ボタン1つでリリース @Continuous Deployment 目的: 完全自動デプロイ 対象: 本番環境 自動化: 本番デプロイまで 結果: 1日に数十回リリース // 段階的に成熟する レベル0: 手動ビルド・手動テスト → 統合は怖い行為 レベル1: CI 導入 → コミット毎に自動テスト → マスターは常に動く レベル2: Continuous Delivery → ステージング環境まで自動 → 本番リリースは人間判断 レベル3: Continuous Deployment → 全工程自動 → 1日数十回のデプロイ // 業界のトップ企業 Amazon: 1秒に1回デプロイ Netflix: 1日数千回 Etsy: 1日50回以上 → レベル3の世界 // 鉄則 段階的に進める いきなりレベル3は無謀
POINT —
「Continuous Delivery」と「Continuous Deployment」を混同しないこと。前者は「いつでも出せる」状態、後者は「自動で出す」仕組み。規制業界（金融・医療）では Continuous Delivery 止まり、Web系では Continuous Deployment まで進むのが定番です。★★★
02

テストピラミッド

TEST PYRAMID

自動テストには階層があります。Mike Cohn が提唱したテストピラミッド──下からユニット・統合・E2Eの順に積み上げる構造。「下を厚く、上は薄く」が鉄則。逆ピラミッド（アイスクリームコーン）になっている現場は危険信号です。

test.pyramid テスト階層の理想と現実
// テストピラミッド（理想）
▲ 数少・遅・脆 ╱│╲ E2E (UI) ╱ │ ╲ ブラウザ操作テスト ╱──┼──╲ ╱ │ ╲ 統合テスト ╱ │ ╲ API・DB連携 ╱─────┼─────╲ ╱ │ ╲ ユニットテスト ╱ │ ╲ 関数・クラス単位 ▼ 数多・速・安定 ▼ 下層ほど: ✓ 数が多い ✓ 実行が速い ✓ 安定している ✓ 修正コストが低い

※ 比率の目安：ユニット70% / 統合20% / E2E 10%

// 各層の特徴 @ユニットテスト 対象: 関数・クラス単位 速度: ミリ秒 個数: 数百〜数千 ツール: Jest, JUnit, pytest 例: 「税込価格を返す関数」 @統合テスト 対象: 複数モジュール連携 速度: 数秒〜数十秒 個数: 数十〜百 ツール: Postman, REST Assured 例: 「APIからDBへの保存」 @E2Eテスト 対象: UI から DB まで 速度: 数十秒〜分 個数: 十〜数十 ツール: Cypress, Playwright, Selenium 例: 「注文完了までの一連操作」 // アンチパターン：アイスクリームコーン ▲ E2Eテスト多い │ │ ← 大量のUIテスト │ 遅い・壊れやすい │ 修正コスト爆発 ▼ 手動テスト さらに上を覆う → 「自動化したつもり」で 実は何も改善されない // なぜ下層を厚くするか 速度: 数秒で全実行 特定: バグの場所が明確 維持: 仕様変更時に壊れにくい 数: 多い → カバレッジ高い // 鉄則 「下に厚く投資せよ」
TIP —
現実には「テスト不足のままE2Eだけ書く」現場が多い。これはアイスクリームコーンの典型。「E2Eテストがあるから安心」と思い込みがちですが、E2Eは遅く壊れやすいので、頻繁な実行に向きません。下層から育てるのが正攻法です。★★
03

ブランチ戦略

BRANCHING STRATEGY

Git の運用方法、ブランチ戦略。代表的な3パターン──Git Flow・GitHub Flow・GitLab Flow。それぞれ哲学が違い、プロジェクトの特性で選びます。「複雑すぎるブランチ運用は、それ自体がアンチパターン」──現代の流れはシンプル化に向かっています。

git.branch-strategies 3大ブランチ戦略
// 3戦略の比較
戦略ブランチ数適合
Git Flow多（master/develop/feature/release/hotfix）パッケージ製品・規制業界
GitHub Flow少（main/feature のみ）Web系・継続的デプロイ
GitLab Flow中（main/環境ブランチ）段階的デプロイ環境

※ 「Git Flow」は2010年に Vincent Driessen が提唱。当時革新的だったが、現在は「複雑すぎる」と見直しが進む。

// Git Flow（複雑） master ───●─────●──────●→ 本番 develop ──●─●─●─●─●─●─●─●→ 開発中 feature/X ╲ ╲╱ release/1.0 ╲─●─●─● hotfix ╲────● ブランチ5種類 ルール多い 学習コスト高い // GitHub Flow（シンプル） main ──●─●─●─●─●→ feature/X ╲ ╱ feature/Y ╲ ╱ ╲ ╱ ブランチ2種類 PRでマージ 本番=main // 各戦略のメリット・デメリット @Git Flow ✓ リリース管理が緻密 ✗ 学習コスト高い ✗ 長期ブランチが発生 ✗ コンフリクト頻発 @GitHub Flow ✓ シンプル・直感的 ✓ 継続的デプロイ向き ✗ 環境管理は別途必要 @GitLab Flow ✓ 環境ごとブランチ ✓ ステージング→本番管理 △ 中間的な複雑さ // 現代の流れ 「シンプル化」 → 多くの現場が Git Flow から GitHub Flow へ移行 「短命ブランチ」 → 数時間〜1日で main にマージ → 長期ブランチを避ける
POINT —
Vincent Driessen 自身が2020年に「Git Flow は2010年代の戦略だった。継続的デリバリーには合わない」と発言しています。「Git Flow が定番」と思い込んでいる現場は、一度立ち止まる価値があります。シンプルな運用ほどフローが速い──カンバンと同じ原則です。★★★
04

トランクベース開発

TRUNK-BASED DEVELOPMENT

シンプル化の究極形がトランクベース開発（TBD）。「全員が main（トランク）に直接コミットする」──常識を覆す運用です。Google・Facebook・Microsoft等の超大規模組織が採用しており、1日に複数回main にマージするのが標準。フィーチャートグルとセットで使われます。

git.trunk-based トランクベースの哲学
// TBDの基本
[短命ブランチ TBD] main ●━●━●━●━●━●━●━●━●→ ↑ ↑ ↑ ↑ 全員が1日に複数回 小さな変更を直接マージ ブランチ寿命: 数時間〜1日 PR: あってもごく短時間 [長命ブランチ (典型的)] main ●━━━━━━━━━━━━━━━━● feature ●━━━━━━━━━━━━━━━━● ↑ 1週間〜1ヶ月 TBD の哲学: 「統合は痛い。だから頻繁に」
// TBDのルール @1. ブランチは短命 寿命 < 1日 数時間で main にマージ @2. 小さな変更 1コミット = 1論理単位 PRは数百行以内 @3. 未完成機能は隠す フィーチャートグル使用 リリースとデプロイを分離 @4. 強力なCI すべてのコミットで 全テスト実行 main は常に動く状態 @5. 信頼の文化 メンバー全員がmainに直接書く → 互いに信頼 // なぜTBDが効くか @リトルの法則 WIP = 進行中ブランチ数 WIPを減らせば リードタイムが縮む @統合コスト 長期ブランチは 必ず統合コストを生む → 短くすれば回避 @フィードバック速度 即マージで 即統合テスト実行 → 問題発見が速い // 採用企業 Google: ほぼ全社的に TBD Facebook: 大半が TBD Microsoft: 多くのチームで TBD LinkedIn, Salesforce など // DORA研究の結論 『Accelerate』(2018) で ハイパフォーマンスチームの 共通点として TBD が浮上 「短命ブランチ」と 「頻繁な統合」が パフォーマンス指標と相関 // アンチパターン 「TBDだから main に とりあえずpush」 → テスト不十分でNG TBD には強力なテスト必須
裏技 —
トランクベース開発を学ぶならtrunkbaseddevelopment.comが公式リソース。『Accelerate』（Forsgren, Humble, Kim）はDORAレポートを書籍化した名著で、「ハイパフォーマンスなチームの共通点」を統計的に示しています。「ブランチ寿命の短さ」がパフォーマンスと相関する事実は衝撃的です。★★★
05

デプロイパイプライン

DEPLOYMENT PIPELINE

コードコミットから本番デプロイまでの「自動化された流れ」がデプロイパイプライン。Jez Humble の『Continuous Delivery』（2010）で体系化されました。各ステージでチェックを行い、合格したら次へ──失敗したらラインを止めて修正。これがソフトウェアの「生産ライン」です。

pipeline.stages 標準的なパイプライン構成
// 標準的なステージ
[1] Commit Stage（数分以内） ├ コンパイル ├ 単体テスト └ 静的解析 │ NG → 失敗通知 OK ↓ [2] Integration Stage（〜10分） ├ 統合テスト └ 軽量受け入れテスト │ OK ↓ [3] Staging Stage（〜30分） ├ E2Eテスト ├ 負荷試験 └ セキュリティスキャン │ OK ↓ [4] Production Deploy ├ Blue-Green └ カナリアリリース │ ↓ 監視・観測
// 各ステージの設計原則 @Fail Fast 早いステージで失敗を検出 → 後段で失敗するほど高コスト @Stage Independence 各ステージは独立 並列実行可能なものは並列で @Idempotency (冪等性) 同じインプット → 同じアウトプット リトライ可能 @Visibility パイプライン全体を見える化 どこで止まったか即わかる // CI/CDツール GitHub Actions: GitHub統合 GitLab CI: GitLab統合 CircleCI: 設定柔軟 Jenkins: オンプレ定番 ArgoCD: K8s向けGitOps Spinnaker: Netflix製・大規模 // YAMLでパイプライン定義の例 @.github/workflows/main.yml name: CI/CD on: push: branches: [main] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v3 - run: npm install - run: npm test deploy: needs: test steps: - run: ./deploy.sh // 鉄則 "If it hurts, do it more often" 痛いことほど頻繁に → 自動化が完成する
POINT —
パイプラインは「ソフトウェアの生産ライン」。トヨタ生産方式の「異常で止める（自働化）」がそのまま適用されます。パイプラインが赤くなったら全員で修正。これがチームの規律になります。「赤いビルドを放置するな」は鉄則です。★★★
06

フィーチャートグル

FEATURE TOGGLES

フィーチャートグル（フィーチャーフラグ）は「コードに組み込んだスイッチ」。デプロイとリリースを分離します。未完成機能をmainにマージしても、トグルOFFにしておけばユーザーには見えない。これがトランクベース開発を可能にする秘密兵器です。

feature.toggles トグルの4種類
// 4種のフィーチャートグル
種類寿命用途
Release短期未完成機能の隠蔽
Experiment中期A/Bテスト
Ops長期機能の緊急停止
Permission永続有料機能・ロール別

※ Martin Fowler が分類。用途に応じて使い分ける。

// 基本的な使い方 if (featureFlag.isEnabled('new-checkout')) { newCheckout(); } else { legacyCheckout(); } // メリット @デプロイとリリースの分離 デプロイ: コードを配置 リリース: 機能を有効化 → 別々のタイミングで実行 @段階的ロールアウト 最初は内部ユーザーのみ → 5%のユーザーに公開 → 50%に拡大 → 100% @緊急停止 問題が起きたら即OFF ロールバック不要 @A/Bテスト パターンAとB両方デプロイ ユーザー群でランダム振分 // アンチパターン トグル爆発 気づくと数百のトグル → コードがif地獄 → デバッグ困難 トグル放置 Release トグルなのに リリース後も削除されない → 死んだコード残存 // 対策 @命名規則 種類_期限_名称 例: REL_2024Q1_new_search @期限設定 Release は3ヶ月以内に 削除がルール @定期的な棚卸し 四半期ごとに不要トグル削除 // ツール LaunchDarkly (デファクト) Split.io Unleash (OSS) ConfigCat 独自実装 (シンプル用途) // 結論 トグルは強力 だが管理コストもある
TIP —
Martin Fowler の記事「Feature Toggles」は最も読まれているリソースの1つ。「リリース型トグルは寿命を短く」「パーミッション型は別管理」など、種類別の運用が大事。「とりあえずトグル」は地獄への道です。★★
07

リリース戦略

RELEASE STRATEGIES

本番への新バージョンの出し方には複数のパターンがあります。Blue-Green・カナリア・ローリング・シャドウ──それぞれの安全性とコストのトレードオフを理解しておくと、現場で適切な選択ができます。

release.strategies 主要なリリース戦略
// 4つの戦略
[Blue-Green] Blue (現行) → Green (新) 一斉切替 即座のロールバック可能 コスト: 2倍のリソース必要 [Canary] 1% → 10% → 50% → 100% 段階的に流し増やす 問題発生時の影響限定 [Rolling] サーバを順次入れ替え サービス停止なし 一般的・低コスト [Shadow] 本番トラフィックを 新環境にも複製送信 実トラフィックで検証 ユーザーには影響ゼロ
// 各戦略の詳細 @Blue-Green Deployment 仕組み: Blue (現行本番) Green (新環境準備) ロードバランサーで切替 メリット: ✓ 即座のロールバック ✓ ダウンタイムゼロ デメリット: ✗ 2倍のリソース ✗ DBマイグレーション複雑 @Canary Release 仕組み: 一部ユーザーから段階公開 メトリクスを観察しながら拡大 メリット: ✓ 問題早期発見 ✓ 影響範囲限定 デメリット: ✗ 段階管理の手間 ✗ 監視体制必須 @Rolling Deployment 仕組み: N台のサーバを順次切替 1台ずつ新バージョンに メリット: ✓ シンプル ✓ 既存インフラで実現 デメリット: ✗ ロールバック遅い ✗ 旧新混在期間あり @Shadow Deployment 仕組み: 本番トラフィック複製 新環境で並行処理 結果は破棄 メリット: ✓ 実トラフィックで検証 ✓ ユーザー影響ゼロ デメリット: ✗ 副作用ある処理に不向き // 業界別の典型 Webサービス: Canary が定番 モバイル: 段階配信 エンタープライズ: Blue-Green 金融: Shadow → Blue-Green
POINT —
Netflix・Google等は「カナリア＋自動ロールバック」を組み合わせています。「メトリクスが悪化したら自動でロールバック」──Spinnaker等のツールで実現可能。人間の判断を介さずに安全性を確保するのが、現代のハイパフォーマンスチームの定番です。★★★
08

DevOps文化

DEVOPS CULTURE

CI/CD の延長線上にあるのがDevOps。「開発と運用の壁を取り払う」文化運動です。「You Build It, You Run It」（作った人が運用する）が原則。Amazon の Werner Vogels が広めたこの言葉が、DevOpsの本質を端的に表しています。

devops.culture DevOpsの本質
// CALMS フレームワーク
  • Culture（文化）── 協働と信頼の文化を醸成
  • Automation（自動化）── 反復作業を全自動化
  • Lean（リーン）── ムダ排除・フロー最適化
  • Measurement（計測）── データで判断・継続改善
  • Sharing（共有）── 知識・成功・失敗の共有

※ Damon Edwards & John Willis が提唱した DevOps 5原則。

DevOps 以前
開発：「動いた、リリースしろ」
運用：「動かない、ロールバック」
互いに責任転嫁
DevOps 以後
開発と運用が同じチーム
「自分たちで作り、運用する」
責任を共有
// DevOpsの起源 2009: Velocity Conference John Allspaw & Paul Hammond "10+ Deploys Per Day" Flickrの事例発表 → DevOps運動の始まり 2010: DevOpsDays Ghent 「DevOps」用語確立 2013: 『The Phoenix Project』 Gene Kim 著 小説形式でDevOpsを伝授 2016: 『The DevOps Handbook』 実践的なガイド 2018: 『Accelerate』 Forsgren, Humble, Kim DORAレポート書籍化 // DORA 4 Key Metrics @デプロイ頻度 Elite: 1日複数回 High: 週〜日 Medium: 月 Low: 半年以上 @変更リードタイム Elite: 1日以内 High: 1週間 Medium: 1ヶ月 Low: 数ヶ月 @変更失敗率 Elite: 0〜15% High: 16〜30% Medium: 16〜30% Low: 46〜60% @復旧時間 (MTTR) Elite: 1時間以内 High: 1日 Medium: 1日〜1週間 Low: 1週間以上 // 結論 DevOps = ツールではない DevOps = 文化である ツールはCALMSの「A」だけ 他の4つは組織と人の話
POINT —
「DevOps部署を作る」のはアンチパターン。これは新しい壁を作るだけ。本来のDevOpsは「開発と運用の壁を消す」運動です。Site Reliability Engineering（SRE）も同じ哲学。Google の『SRE Book』は無料公開されており必読。「You Build It, You Run It」を組織でどう実現するかが、DevOps成功の鍵です。★★★
