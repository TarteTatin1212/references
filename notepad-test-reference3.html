Mapperテストとは

WHAT IS MAPPER TEST

MapperはSQLそのもの。モックで偽物にしたら「SQLが正しいか」は永遠に確かめられません。だからMapperだけは本物のDBに繋いで、実際にSQLを走らせて検証します。

@MybatisTest Mapper＋DBだけ起動
// Service と Mapper の違い
Service(Vol.2)Mapper(今回)
確かめるロジックSQL
DB使わない
(モック)		本物を使う
手法Mockito@MybatisTest
速さ一瞬やや遅い
POINT ── @MybatisTest は軽い結合テスト

@SpringBootTest はアプリ全体を起動しますが、@MybatisTest はMapperとDB機能だけを起動します。Controllerなどは読み込まないので、フル起動より速い。「DBに関わる部分だけ」をピンポイントで試す道具です。

// テスト対象：NoteMapper（おさらい） @Mapper public interface NoteMapper { @Select("SELECT * FROM memos ORDER BY id") List<Note> findAll(); @Insert("INSERT INTO memos (title, content) VALUES (#{title}, #{content})") void insert(Note note); @Select("SELECT id, title, content FROM memos WHERE id = #{id}") Note findById(long id); @Update("UPDATE memos SET title=#{title}, content=#{content} WHERE id=#{id}") void update(Note note); @Delete("DELETE FROM memos WHERE id=#{id}") void delete(Long id); } // これらのSQLが「本当に動くか」を // 実際にDBで走らせて確かめる。
02

テスト用DBを決める

CHOOSE TEST DB

ここで判断が必要です。@MybatisTest はデフォルトで組み込みDB（H2）を探します。でもこのアプリにはH2の依存がありません。2つの道があります。

H2 or MySQL 2つの選択肢
// 2つの道
案A：H2を追加推奨

軽量な組み込みDBを依存に足す。テストのたびにメモリ上にDBを作る。速い・他人の環境でも動く・本番DBを汚さない。ただしMySQLと文法が少し違う点に注意。

案B：実MySQLに繋ぐ

@AutoConfigureTestDatabase(replace=NONE) で本番と同じMySQLを使う。本番に忠実だが、テスト用DBの用意・起動が必要で、データを汚すリスクも。

POINT ── 学習・小規模ならH2が定番

H2はテストのたびに真っさらなDBをメモリに作って捨てるので、再現性が高く速い。本シリーズは案A（H2）で進めます。本番MySQL特有のSQLを使う場合だけ案Bを検討、という住み分けです。

// なぜ判断が要るのか // @MybatisTest のデフォルト動作： // 「組み込みDB(H2など)に自動で繋ぐ」 // でも pom.xml に H2 が無い → エラー： // Failed to replace DataSource with an // embedded database ... // ─── 解決の2案 ─── // 案A：H2を足す（推奨） // → デフォルト動作がそのまま使える // 案B：実MySQLを使う @MybatisTest @AutoConfigureTestDatabase( replace = Replace.NONE) // 置換しない // → application.properties のMySQLに繋ぐ // → テスト用DBの準備が別途必要 // このシリーズは案A（H2）で進める。
03

H2の準備（推奨案）

SETUP H2

案AのH2を準備します。依存を1つ足し、テスト用のテーブル定義を置くだけ。これでテスト実行時に、メモリ上に memos テーブルが自動で用意されます。

pom.xml + schema.sql 2ステップで準備完了
// 準備の2ステップ
  • pom.xml に H2 を追加（test スコープ）
  • src/test/resources に schema.sql を置き、memos テーブルの CREATE 文を書く
POINT ── schema.sql は自動で実行される

Spring Bootは、テスト起動時に src/test/resources/schema.sql を自動で読み込んでテーブルを作成します。だからテストコードでテーブルを作る必要はありません。H2はメモリ上なので、テストが終われば跡形もなく消えます。

// ① pom.xml に追加 <dependency> <groupId>com.h2database</groupId> <artifactId>h2</artifactId> <scope>test</scope> </dependency> // ② src/test/resources/schema.sql CREATE TABLE memos ( id BIGINT AUTO_INCREMENT PRIMARY KEY, title VARCHAR(30), content TEXT ); // ③（任意）初期データ data.sql // src/test/resources/data.sql に // INSERT文を置けば、毎テスト前に投入される INSERT INTO memos (title, content) VALUES ('サンプル', '本文'); // 本番のテーブル定義(MySQL)に合わせる。 // H2はMySQL構文の多くをそのまま解釈する。
04

テストクラスの骨格

SKELETON

Mapperテストの基本形です。@MybatisTest を付け、テスト対象の NoteMapper を @Autowired で注入します。モックではなく本物のMapperです。

NoteMapperTest.java 骨格
// Vol.2との違い

Vol.2は @Mock で偽物のMapperでした。今回は @Autowired で本物のMapperを注入します。

本物なので、呼べば実際にSQLが走り、H2のDBが読み書きされます。だから戻り値だけでなく「DBが実際にどう変わったか」まで確かめられます。

TIP ── @MybatisTest が全部用意する

@MybatisTest を付けるだけで、H2の起動・schema.sqlの実行・Mapperの登録・トランザクション管理まで自動でお膳立てされます。あとは @Autowired でMapperを受け取って呼ぶだけ。

// src/test/java/jp/example/mapper/ // NoteMapperTest.java package jp.example.mapper; import static org.assertj.core.api.Assertions.*; import org.junit.jupiter.api.Test; import org.mybatis.spring.boot.test.autoconfigure .MybatisTest; import org.springframework.beans.factory .annotation.Autowired; import jp.example.entity.Note; @MybatisTest class NoteMapperTest { @Autowired NoteMapper noteMapper; // 本物のMapper // ↓ ここに各テストを書く } // 案B（実MySQL）の場合はクラスに追加： // @AutoConfigureTestDatabase( // replace = Replace.NONE)
05

insert と件数検証

INSERT

最初のテスト。insert でメモを追加し、本当に1件増えたかを確かめます。ここで実コードの「採番キーを取得しない」制約に正直に向き合います。

insert_1件増える 採番キー無しに対応
// 採番キーが取れない問題

このアプリの insert には @Options(useGeneratedKeys...) が付いていません。だから insert 後に note.getId() しても採番されたidは取れません（0のまま）。

なので「insert→そのidでfindById」という検証はできない。代わりに「findAllで件数が1増えたか」で確かめます。

POINT ── これはリファクタの伏線

本来、登録直後にidを使いたい場面は多い（登録後に詳細へ遷移など）。「insertに採番キー取得を足す」のはリファクタリング編の改善候補です。今は現状の制約に合わせて件数で検証します。

// insert：件数で確かめる @Test void insert_メモが1件増える() { // given：登録前の件数を数える int before = noteMapper.findAll().size(); Note note = new Note(); note.setTitle("新しいメモ"); note.setContent("本文です"); // when：insert実行 noteMapper.insert(note); // then：1件増えたか List<Note> after = noteMapper.findAll(); assertThat(after).hasSize(before + 1); // 中身も確認：最後の1件が登録したもの Note last = after.get(after.size() - 1); assertThat(last.getTitle()) .isEqualTo("新しいメモ"); } // ※ note.getId() は使えない（0のまま）。 // 採番キー取得が未設定のため。 // だから findAll の件数＋中身で検証する。

update・delete・findById

UPDATE / DELETE / findById

残りのSQLも確かめます。初期データを前提に、findByIdで取得・updateで変更・deleteで削除し、それぞれDBが意図通り変わったか検証します。

CRUD verification 取得・更新・削除
// 実行結果（イメージ）
JUnit
NoteMapperTest
✓ insert_メモが1件増える
✓ findById_該当を取得

✓ delete_1件減る

TIP ── 「変更→読み直して確認」

updateやdeleteは戻り値がありません。だから「変更した後、findAllやfindByIdで読み直して、DBが実際に変わったか」を確かめます。書いて終わりではなく、読み直して結果を見るのがDBテストの基本です。

// update：内容が変わるか @Test void update_内容が変わる() { // given：まず1件入れる Note n = new Note(); n.setTitle("元タイトル"); noteMapper.insert(n); // 入れた1件のidを取得（findAllの末尾） List<Note> list = noteMapper.findAll(); long id = list.get(list.size()-1).getId(); // when：更新 Note upd = new Note(); upd.setId(id); upd.setTitle("新タイトル"); upd.setContent("新本文"); noteMapper.update(upd); // then：読み直して確認 Note after = noteMapper.findById(id); assertThat(after.getTitle()) .isEqualTo("新タイトル"); } // delete：1件減るか @Test void delete_1件減る() { Note n = new Note(); n.setTitle("消す"); noteMapper.insert(n); List<Note> list = noteMapper.findAll(); long id = list.get(list.size()-1).getId(); int before = list.size(); // when noteMapper.delete(id); // then assertThat(noteMapper.findAll()) .hasSize(before - 1); } // findById(id) は H2 では採番済みidが // 取れる（findAll経由で取得すればOK）。
07

自動ロールバックの仕組み

AUTO ROLLBACK

気になりませんか?「insertやdeleteを何度も実行したら、DBがどんどん汚れるのでは?」——大丈夫です。@MybatisTest はテストごとに自動でロールバックします。

rollback テスト後に巻き戻る
// 各テストの流れ
① テスト開始 トランザクション開始
↓
② テスト本体 insert / update / delete 実行
↓
③ テスト終了 自動でロールバック（全部巻き戻す）
POINT ── だから何度でも安全

各テストは独立したトランザクションで動き、終了時に必ず巻き戻されます。だから「テストAでinsertした1件」が「テストB」に影響しない。毎回まっさらな状態から始まる。テストの順序に依存しないのが鉄則です。

// なぜ汚れないのか // @MybatisTest は内部で // @Transactional を効かせている。 // → 各テストメソッドが1つの // トランザクションで包まれる // → テスト終了時、commitせず rollback // → DBへの変更が全部取り消される // ─── 具体例 ─── // テストAで insert（3件→4件） // テストA終了 → rollback → 3件に戻る // テストBで delete（3件→2件） // テストB終了 → rollback → 3件に戻る // → テストAとBは互いに影響しない // → 何度実行しても同じ結果（再現性） // ─── 鉄則 ─── // テストは「前のテストの結果」に // 依存して書いてはいけない。 // 各テストが自己完結すること。
08

ハマりどころ

PITFALLS

Mapperテスト特有の詰まりポイントを3つ。DBが絡むぶん、単体テストとは違う罠があります。

よくある詰まり 3つの罠
// 詰まりポイント
  • H2が無くてDataSourceエラー：依存追加を忘れると「embedded database」エラー。Sec.02-03で対応済み
  • schema.sql の構文差：MySQL専用構文はH2で動かないことがある。標準的なSQLで書く
  • 採番idを当てにする：このアプリはinsert後にidが取れない。findAll経由で取得する
POINT ── H2とMySQLの方言差

H2はMySQL互換モードもありますが、完全一致ではありません。AUTO_INCREMENT など基本構文は通りますが、MySQL固有の関数等は注意。テストDBと本番DBが違うことのリスクは、案B（実MySQL）を選ぶ理由にもなります。学習段階ではH2で十分です。

// 罠①：DataSource エラー // Failed to replace DataSource with an // embedded database. ...add h2... // → pom.xml に H2 を追加（Sec.03） // 罠②：schema.sql が見つからない // テーブルが無くて Table "MEMOS" not found // → src/test/resources/ に置いたか確認 // （src/main ではなく src/test） // 罠③：採番idを当てにする // NG：insert後にidを使う noteMapper.insert(note); long id = note.getId(); // 0のまま！ // OK：findAllの末尾から取る List<Note> list = noteMapper.findAll(); long id = list.get(list.size()-1) .getId();
