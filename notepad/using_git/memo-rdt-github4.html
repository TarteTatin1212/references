01

この巻のミッション

MISSION

Vol.GH4 のゴールは「複数の要求を整理して、次にやることを可視化する」。1つの Issue を追うのは Vol.3 までで習得。今回は「バックログ全体を俯瞰し、優先順位を管理する」。これが要求工学の「管理」フェーズの本丸です。

「1つの要求」から「要求全体」へバックログ管理
// Vol.GH3 までとの違い
観点Vol.GH1-3Vol.GH4
対象1つの要求要求全体
軸深さ（Issue→PR→Close）広さ（バックログ）
道具Issue・PRラベル・Milestone・Project
問い「これは完成した？」「次に何を？」
// 別冊S1の理論を運用
  • MoSCoW（別冊S1）→ ラベルで可視化
  • リリース計画 → マイルストーン
  • 変更管理（CR票）→ Issue で運用
  • スコープクリープ防止 → Won't ラベルで明示
// Vol.GH4 のゴール // ─── バックログを整理 ─── GitHub Issues に： #1 US-001 メモ作成 [must][closed] #2 US-002 カテゴリ分類 [should][closed] #3 US-003 タグ付け [could] #4 US-004 検索 [could] #5 CR-001 フォルダ分け [wont][v2] ↓ ラベル・Milestone・Projectで整理 ↓ 「次に着手すべきは #3」が一目 ★★★ // ─── 可視化される情報 ─── - 何が完了/未完了か - 各要求の優先度（MoSCoW） - v1 に含むもの/v2 送り - 今スプリントでやること // → 「迷わず次に進める」状態を作る
POINT — 「見えない要求」は管理できない

頭の中やテキストファイルにある要求はすぐ忘れ、抜け、優先順位も曖昧に。GitHub の Issue＋ラベル＋Project に乗せると、「全要求が一覧で見え、優先度で並び、次が分かる」。見える化が管理の第一歩。1人開発でも、要求が3つを超えたら管理ツールの価値が出ます。

02

完成イメージ

FINAL STATE

Vol.GH4 終了時、ラベルで色分けされた Issue 一覧、v1.0 マイルストーンの進捗バー、Todo/In Progress/Done の Project ボードが揃います。「プロジェクトの全体像」が一目で見える状態です。

Project ボードの完成像カンバン方式
// 揃うもの
  • ラベル：must/should/could/wont＋種類別
  • Milestone：v1.0（進捗 50% など）
  • Project ボード：Todo / In Progress / Done の3列

カンバン方式で、Issue がカードとして列を移動。「今どこまで進んでいるか」が視覚的に分かります。

// Project ボードのイメージ
github.com / .../projects / 1
📋 Todo
#4 US-004 検索 could
#3 US-003 タグ could
🔨 In Progress
#3 US-003 タグ付け実装中 story
✅ Done
#1 US-001 ✓ must
#2 US-002 ✓ should
// → Todo/進行中/完了が一目 ★★★ // → カードをドラッグで列移動 // → 「今の状況」が視覚的に分かる
TIP — カンバンは「仕掛かり過多」を防ぐ

Project ボードのIn Progress（進行中）列は、同時に進めるタスクを制限する効果があります。In Progress に5個も並んでいたら「手を広げすぎ」のサイン。1つずつ完了させるのが効率的（別冊S2のスコープ管理にも通じる）。「着手中は1〜2個まで」を意識すると、確実に前進します。

03

ラベル設計

LABEL DESIGN

ラベルはIssue の分類タグ。本シリーズではMoSCoW（優先度）と種類（story/bug/enhancement）の2軸で設計します。別冊S1で学んだ MoSCoW を、ここで色付きラベルとして実体化します。

2軸のラベル設計優先度 × 種類
// 軸① 優先度（MoSCoW）
ラベル色意味
must赤必須
should緑重要
could黄あれば
wont灰今回やらない
// 軸② 種類
ラベル意味
user-story機能要求（US）
bug不具合
enhancement改善要望
question検討事項

1つの Issue に「優先度1つ＋種類1つ」を付けるのが基本。

// ラベルの作成と運用 // ─── 作成手順 ─── // Issues タブ → Labels → New label 名前: must 説明: 必須（MoSCoW: Must） 色: #d73a4a（赤） // ↓ 同様に作成 should → #0e8a16（緑） could → #fbca04（黄） wont → #cccccc（灰） user-story → #1d76db（青） bug → #d73a4a（赤・既定） enhancement → #a2eeef（水色・既定） // ─── ラベルの威力：絞り込み ─── // Issues の検索バーで： is:open label:must → 未完了の必須要求だけ表示 ★ is:open label:could → 後回しOKな要求 label:bug → バグだけ一覧 // ─── MoSCoW が運用に直結 ─── // 別冊S1で決めた優先度が… Must（メモCRUD・カテゴリ）→ closed Should（絞り込み）→ closed Could（タグ・検索）→ 残り ↓ 「must/should は完了、 次は could に着手」が一目 ★★★
POINT — ラベルは「増やしすぎない」

ラベルは便利ですが増やしすぎると逆に分からなくなります。本シリーズは優先度4＋種類4程度に抑える。「priority-high-urgent-critical」みたいに細分化しすぎると運用が破綻。別冊S2のKISSがラベル設計にも効きます。「色を見れば分かる」シンプルさを保ちましょう。

04

マイルストーンでリリース計画

MILESTONE

マイルストーンは「いつまでに、何を」のグループ。複数の Issue を束ね、リリース単位（v1.0 など）で管理します。別冊S1の「v1で何を作るか」がここで形になります。

マイルストーン＝リリースの束進捗が見える
// マイルストーンの役割
  • 複数 Issue をリリース単位で束ねる
  • 進捗バー（完了/全体）が自動表示
  • 期限を設定できる
// 本シリーズのマイルストーン
Milestone含む Issue
v1.0US-001〜004（Must〜Could）
v2.0同期・共同編集（Won't行き）
// MoSCoW との関係

Won't = v2.0 マイルストーンに置く。「今やらない」を「v2でやる」と前向きに記録。スコープクリープ防止（別冊S1）。

// マイルストーンの作成と進捗 // ─── 作成手順 ─── // Issues → Milestones → New milestone Title: v1.0 Due date: （任意） Description: メモ帳アプリ初版 US-001〜004（Must/Should/Could） // → 各 Issue の右サイドで // Milestone に v1.0 を設定 // ─── マイルストーンの進捗表示 ───
Milestones
● v1.0 ████████░░░░ 50% complete 2 closed · 2 open #1 US-001 メモ作成 ✓ closed #2 US-002 カテゴリ ✓ closed #3 US-003 タグ ○ open #4 US-004 検索 ○ open ● v2.0（将来） #5 CR-001 フォルダ分け #6 複数端末同期
// → 進捗バーで「半分終わった」が一目 ★ // → v1.0 で何が残っているか分かる // ─── 別冊S1の MoSCoW が形に ─── v1.0 = Must + Should + Could v2.0 = Won't（フォルダ・同期・共同編集） // → 「やらない」を「次でやる」に整理 ★★★ // → スコープが守られる
POINT — マイルストーンの進捗バーが「あと少し」を見せる

マイルストーンの進捗バーは、モチベーション維持に効きます。「v1.0 が 75% 完了、あと1つ」と見えるとゴールが近いと実感できる。逆に「全然進んでない」も可視化される。個人開発で挫折しがちな人ほど、進捗バーの効果は大きい。要求を Issue 化する地味な作業が、ここで報われます。

05

Project ボードで可視化

PROJECT BOARD

GitHub Projectsはカンバン方式のタスク管理。Issue をカードとしてTodo → In Progress → Done の列で動かす。「今どこまで進んでいるか」を最も直感的に表せるツールです。

カンバンボードの構築Todo / 進行中 / 完了
// Project 作成手順
  1. リポジトリの Projects タブ → New project
  2. テンプレート Board を選択
  3. 列：Todo / In Progress / Done（既定であることが多い）
  4. Issue を ＋ Add item でボードに追加
  5. カードをドラッグで列移動
// 自動化（Workflows）
  • Issue が Close されたら自動で Done 列へ
  • PR がマージされたら自動移動
  • → 手動移動の手間が減る

Issue/PR と連動するのが GitHub Projects の強み。

// カンバンの運用フロー // ─── 1つの要求がボードを移動 ─── US-003（タグ機能）の場合： [Todo] #3 US-003 タグ付け ↓ 着手したら [In Progress] #3 US-003（feature/tag-crud で開発中） ↓ PR マージ＋Closes #3 [Done] #3 US-003 ✓ // → 自動で Done へ（Workflows）★★★ // ─── ボードと本編の対応 ─── 本編Vol.5 でタグ実装に着手 → ボードで #3 を In Progress へ 本編Vol.6 でタグ付け完成 → PR → Closes #3 → ボードで #3 が自動 Done へ // ─── 「次にやること」が明確に ─── [Todo] を見れば： #4 US-004 検索 ← 次はこれ [In Progress] を見れば： 今やっていること（1〜2個に絞る） [Done] を見れば： これまでの成果（達成感）★ // → 迷わず開発を進められる
POINT — Project は「Issue の見せ方を変える」だけ

誤解しやすいですが、Project ボードはIssue を別の形で表示しているだけ。Issue 自体が情報の本体で、Project は「カンバンというビュー」。だからラベル・マイルストーン・Issue 本文はそのまま活きる。Issue をちゃんと作っていれば、Project は並べ替えて見せるだけ。土台は Issue です。

06

変更要求(CR)の運用

CHANGE REQUEST

別冊S1で学んだ変更要求（CR票）とスコープクリープ。開発中に「この機能も欲しい」は必ず出ます。それを拒否せず、Issue として記録し、影響を見て判断する運用を学びます。

変更要求を Issue で受け止める拒否でも全受容でもなく
// 変更要求の3つの行き先
CASE 1
v1 に入れる
影響小・価値大 → 現マイルストーンに追加。
CASE 2
v2 に回す
影響大・急がない → v2.0 マイルストーン＋wont ラベル。
CASE 3
却下
WHYに合わない → close（理由を記録）。
// CR を Issue で記録する意味
  • 判断の経緯が残る（なぜv2に回したか）
  • 後で「あの話どうなった？」に答えられる
  • スコープクリープを可視化して防ぐ
// 変更要求 Issue のテンプレート // ─── CR を Issue 化（別冊S1のCR票）─── ## 変更要求 CR-001 title: [変更要求] フォルダ分けも欲しい labels: enhancement, wont milestone: v2.0 ## 背景・理由 メモが増えるとタグだけでは 整理しきれない ## 影響分析 - 工数: +3日 - 既存機能: 一覧UIに変更必要 - DB: folders テーブル追加 - テスト: 全一覧テストに影響 ## 判断 v1ではタグのみ。 フォルダは v2.0 に回す。 （理由：v1スコープを守るため） // → 拒否ではなく「v2で検討」と記録 ★★★ // → 判断の経緯が永久に残る // ─── スコープクリープ防止の運用 ─── 新しい要望が来たら： 1. まず Issue 化（忘れない） 2. 影響分析（工数・波及） 3. MoSCoW で判断 → Must/Should なら今回 → Could なら余裕があれば → Won't なら v2 へ（wont ラベル） 4. マイルストーンに割り当て // → 「とりあえず作る」を防ぐ ★★★ // → スコープが守られ、完成する
NG — 思いつきで即実装

「いいこと思いついた」とその場で実装開始。スコープが膨張し、v1 がいつまでも完成しない。別冊S1のスコープクリープそのもの。

OK — まず Issue 化して判断

思いつきも一旦 Issue に記録。影響を見て、v1/v2/却下を判断。記録することで忘れず、判断することで膨張を防ぐ。

POINT — 「却下」も記録する価値がある

変更要求を却下するときも、Issue に理由を書いて Closeします。「なぜやらないと決めたか」が残ると、後で同じ提案が来たときに即答できる。「これは○月に検討して、WHYに合わないので見送った」と。却下の記録は、判断の一貫性を保ちます。要求工学の「管理」は、やることだけでなく「やらないことの管理」でもあります。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

Issue・タスク管理で陥りやすい運用上の地雷を5つ。★★★ 印は特に陥りやすい。

Vol.GH4 ハマりどころ 5選運用の落とし穴

① ラベルを作りすぎて運用崩壊 ★★★

  • 細分化しすぎて「どれを付ければ」と迷う
  • 対処：優先度4＋種類4程度に抑える（KISS）

② Issue を作るが更新しない ★★★

  • 作りっぱなしで状態が古いまま
  • 対処：着手で In Progress、完了で Close を習慣化

③ Project にIssueを追加し忘れる

  • Issue は作ったがボードに乗っていない
  • 対処：Issue 作成時に Project を設定／Workflowsで自動追加

④ In Progress が大量に溜まる

  • あれこれ同時着手で全部中途半端
  • 対処：In Progress は1〜2個まで（仕掛かり制限）

⑤ マイルストーンが「いつか」のまま

  • 期限なしで延々と先延ばし
  • 対処：ゆるくても期限を置く（個人開発でも目標に）
