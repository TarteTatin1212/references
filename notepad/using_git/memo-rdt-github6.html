01

この巻のミッション

MISSION

Vol.GH6 のゴールは「品質を仕組みで守る」こと。① コミットメッセージを規約化して履歴を読みやすく、② main を保護して「壊れたコードが直接入る」のを防ぐ。人の注意力ではなく、ルールと仕組みで品質を保ちます。

「気をつける」から「仕組みで防ぐ」へ品質のゲート
// 2つの仕組み
仕組み1
コミット規約
feat/fix/docs の型で統一。履歴が読みやすく、自動分類も可能に。
仕組み2
ブランチ保護
main への直接pushを禁止。PR・レビュー・テスト通過を必須に。
// なぜ「仕組み」が要るか

「気をつけてコミットする」「壊れたコードはpushしない」は人の注意力頼み。疲れていたり急いでいると破綻する。ルールを仕組みに落とすと、注意力に関係なく守られます。

// Vol.GH6 で作る品質ゲート // ─── これまで（Vol.GH1-5）─── コミット：何となく feat: / fix: を使用 main：直接 push できてしまう状態 ↓ // ─── Vol.GH6 で整備 ─── ① コミット規約（Conventional Commits） feat / fix / docs / test / refactor... → 履歴が構造化される → 「何のコミットか」が型で分かる ★ ② ブランチ保護 main は直接 push 禁止 PR 経由のみ （将来）テスト通過を必須に → 壊れたコードが main に入らない ★★★ // ─── 効果 ─── // 履歴：feat だけ並べれば「追加機能」一覧 // main：常に「動く・レビュー済み」状態 // → 個人開発でも品質が保てる // → チーム開発・CI連携の土台に
POINT — 個人開発でも「仕組み化」する意味

「1人なのに規約や保護は大げさ」と思うかもですが、1人だからこそ仕組みが効く。レビューしてくれる他人がいないので、ルールと仕組みが代わりの番人になる。さらに業務に入ったときに身についているのが大きい。Conventional Commits もブランチ保護も現場の標準。今のうちに習慣化しておきましょう。

02

Conventional Commits

CONVENTIONAL COMMITS

Conventional Commits は、コミットメッセージの業界標準フォーマット。type(scope): 要約 の形で書きます。本シリーズでこれまで使ってきた feat(memo): ... が、実はこの規約でした。正式に整理します。

Conventional Commits の構造type(scope): 要約
// 基本構造

type(scope): 要約 の3要素：

  • type：変更の種類（feat/fix/docs等）
  • scope：変更の範囲（memo/category等・任意）
  • 要約：何をしたか（簡潔に）
// 本文とフッター
  • 本文：詳細（なぜ・どう）
  • フッター：Refs #1 Closes #1（トレーサビリティ）

本シリーズで使ってきた形そのもの。無意識に規約に従っていたのです。

// Conventional Commits の形 // ─── 基本形 ─── type(scope): 要約 本文（なぜ・どうしたか） フッター（Refs #1 など） // ─── 本シリーズの実例 ─── feat(memo): メモ作成機能を追加 US-001 を実装。 - @NotBlank/@Size でバリデーション - 単体テスト4件（受入条件AC1-4） Refs #1 └─────────────────────────┘ ↑ これがConventional Commits ★★★ // ─── 各部の意味 ─── feat → 変更の種類（新機能） (memo) → 範囲（メモ機能） メモ作成... → 要約（50字以内が目安） 本文 → 詳細（72字で改行が慣習） Refs #1 → Issue 紐付け（Vol.GH2で学習） // ─── なぜこの形が良いか ─── // ・type で変更の種類が一目 // ・履歴を type で分類・集計できる // ・自動でCHANGELOG生成も可能 // ・チームで統一感が出る ★★★
POINT — 本シリーズは最初から規約に従っていた

振り返ると、本シリーズの GitHub版は Vol.GH1 から chore: プロジェクト初期化、feat(memo): ...、test: ...、docs: README作成 とすべて Conventional Commits で書いてきました。Vol.GH6 で「実はこれが業界標準の規約だった」と種明かし。良い習慣を自然に身につけていたのです。あとは type の使い分けを正確にするだけです。

03

type の使い分け

COMMIT TYPES

Conventional Commits のtypeは数種類。feat / fix / docs / test / refactor / chore など。それぞれ「どんな変更か」を表します。正しく使い分けると、履歴が「変更の種類別」に読めるようになります。

主要な type 一覧これだけ覚えれば十分
type意味例
feat新機能メモ作成
fixバグ修正空タイトル不具合
docsドキュメントREADME更新
testテストテスト追加
refactorリファクタ挙動変えず整理
chore雑務設定・依存更新
style整形インデント等
// 迷いやすい区別
  • feat vs fix：新しく作る vs 壊れたのを直す
  • refactor：挙動を変えずコードを整える（機能追加・修正でない）
  • chore：ビルド設定・依存関係など雑務
// type 別のコミット例 feat(search): キーワード検索を追加 // → 新機能（US-004・本編Vol.7） fix(memo): 空タイトルで500エラーを修正 // → バグ修正 docs: READMEにセットアップ手順を追加 // → ドキュメント（GitHub版Vol.5） test: タグ付けのロールバックテスト追加 // → テスト（本編Vol.6） refactor(service): 検索ロジックを整理 // → 挙動は変えず、コードを綺麗に chore: H2をテスト依存に追加 // → 設定・依存関係（本編Vol.3） // ─── type で履歴を絞り込む ─── $ git log --oneline --grep="^feat" a1b2 feat(search): キーワード検索 3c4d feat(memo): タグ付け 5e6f feat(memo): メモ作成 // → 追加機能だけ一覧（CHANGELOG的）★★★ $ git log --oneline --grep="^fix" // → バグ修正だけ一覧 // ─── 迷ったときの判断 ─── // 「機能が増えた？」→ feat // 「バグを直した？」→ fix // 「動きは同じでコード整理？」→ refactor // 「ドキュメントだけ？」→ docs // 「テストだけ？」→ test
POINT — refactor は「挙動を変えない」が定義

初学者が迷いやすいのが refactor。これは「機能を変えず、コードを綺麗にする」変更専用。「リファクタついでにバグも直した」はfix と refactor を分けるべき。なぜなら「refactor なのに挙動が変わった＝予期せぬ影響」を防ぐため。refactor のコミットはテストが通ったままであることが保証になります（本シリーズのテストが効く）。

04

良いコミットメッセージ

GOOD COMMIT MESSAGE

type だけでなく、要約の書き方も重要。「何をしたか」を簡潔・具体的に。良いメッセージは未来の自分とチームへの説明です。RDT版では要求(US)との対応も書きます。

良い要約・悪い要約具体的に・命令形で
// 要約のルール
  • 50字以内を目安
  • 「何をしたか」を具体的に
  • 命令形・現在形（「追加する」より「追加」）
  • 句点で終わらない
// 本文に書くこと
  • なぜその変更をしたか（背景）
  • どう変えたか（要点）
  • US/受入条件との対応（RDT版）
  • Refs #1 でトレーサビリティ
// 良い例・悪い例 // ─── 悪い要約 ─── ✗ fix: 修正 ✗ feat: いろいろ追加 ✗ update ✗ wip // → 何をしたか分からない ★ // → 後で履歴を見ても役に立たない // ─── 良い要約 ─── ✓ feat(search): キーワード検索を追加 ✓ fix(memo): 空タイトルで500エラーを修正 ✓ test(category): 重複名のテストを追加 // → 何をしたか一目 ★★★ // ─── RDT版の理想形（本文付き）─── feat(search): キーワード検索を実装 US-004 の前半を実装。 - title/content のLIKE検索 - 動的SQL（<if><where>）で 条件の有無に対応 - 全て #{} で安全に（SQLi対策） - 結合テスト3件追加 Refs #4 └──────────────────────┘ ↑ なぜ・どう・US対応・紐付け ★★★ // ─── 「3か月後の自分」テスト ─── // このメッセージを3か月後に読んで // 「何をしたか」分かるか？ // 分からなければ書き直す
POINT — 「なぜ」は本文に、「何を」は要約に

コミットメッセージの黄金律：要約に「何を」、本文に「なぜ」。コードの diff を見れば「どう変えたか（How）」は分かるので、コミットメッセージには「なぜ変えたか（Why）」を残すのが最も価値が高い。「空タイトルで500エラーが出ていたため修正」のように背景を書く。これは要求工学（別冊S1）やドキュメント（GitHub版Vol.5）の WHY 重視と一貫しています。

05

ブランチ保護とは

BRANCH PROTECTION

ブランチ保護は、main などの重要ブランチに「ルール」を課す機能。「直接pushを禁止」「PR必須」「テスト通過必須」などを設定。壊れたコードが main に入るのを仕組みで防ぐガードレールです。

ブランチ保護で何ができるかmain を守るルール
// 設定できる主なルール
  • 直接pushを禁止：main へは PR 経由のみ
  • PRレビュー必須：承認なしにマージ不可
  • ステータスチェック必須：テスト（CI）通過が条件（Vol.GH8で連携）
  • 会話の解決必須：レビューコメント未解決ではマージ不可
// なぜ main を守るか

main は「常に動く状態」であるべき（GitHub Flow＝Vol.GH2）。直接pushを許すと未テスト・未レビューのコードが入るリスク。保護で「ゲートを通った変更だけ」を main に。

// ブランチ保護の効果 // ─── 保護なし（これまで）─── $ git push origin main // → 誰でも直接 push できる // → 未テストのコードも入る // → main が壊れるリスク ★ // ─── 保護あり（Vol.GH6で設定）─── $ git push origin main remote: error: GH006: Protected branch update failed. remote: error: Required status check ... or pull request required. // → 直接 push が拒否される ★★★ // → PR 経由でしか main を変更できない // ─── 守られる開発フロー ─── feature ブランチで作業 ↓ push（feature は自由） ↓ PR 作成 ↓ [ゲート] レビュー承認 + テスト通過 ↓ クリアして初めて main にマージ可能 ★★★ // → main は常に「ゲートを通った」状態 // → 「動く・レビュー済み・テスト通過」
POINT — 保護は「自分も縛る」のが正しい

ブランチ保護を設定すると自分も main に直接pushできなくなります。「不便」と感じるかもですが、それが目的。「急いでるから今回だけ直接push」が事故の元。自分すら縛ることで、品質が一貫して保たれる。設定で「管理者は例外」にもできますが、個人開発こそ自分を縛る練習として、例外なしを推奨します。

06

main を保護する

PROTECT MAIN

実際に main ブランチを保護します。GitHub の Settings → Branches で設定。本シリーズでは「PR必須・直接push禁止」を基本に、将来のCI（Vol.GH8）に備えてステータスチェックの土台も作ります。

ブランチ保護ルールの設定Settings → Branches
// 設定手順
  1. リポジトリの Settings タブ
  2. 左メニュー Branches
  3. Add branch protection rule
  4. Branch name pattern に main
  5. 下記のチェックを入れる
  6. Create / Save
// 本シリーズで入れる設定
  • Require a pull request before merging（PR必須）
  • Require approvals（承認必須・1人）
  • Require conversation resolution（会話解決必須）
  • （将来）Require status checks（テスト通過必須・Vol.GH8）
// 保護ルールの設定イメージ
Settings / Branches / Add rule
Branch name pattern: main Protect matching branches: ☑ Require a pull request before merging ☑ Require approvals: 1 ☑ Require conversation resolution before merging ☐ Require status checks to pass （Vol.GH8 で CI 設定後に有効化） ☑ Do not allow bypassing the above settings （管理者も例外なし） [ Create ]
// ─── 設定後の動作確認 ─── $ git switch main $ echo "test" >> README.md $ git commit -am "docs: test" $ git push ! [remote rejected] main -> main (protected branch hook declined) // → 直接push が拒否される ★★★ // → 正しく保護されている証拠 // ─── 正しいフロー（保護後）─── $ git switch -c docs/update-readme $ git commit -am "docs: READMEを更新" $ git push -u origin docs/update-readme // → feature ブランチは自由 // → PR を作ってマージ ★
POINT — 1人開発でも「承認1人」は自分でできる

「承認必須・1人」を設定すると、自分のPRを自分で承認することになります。「自作自演では？」と思うかもですが、これが Vol.GH3 のセルフレビューそのもの。PRの Files changed を見て、受入条件を確認して、承認ボタンを押す。承認という明示的なステップが、雑なマージを防ぐ。1人でもゲートを通る習慣が、業務での質を作ります。

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

Vol.GH6 ハマりどころ 5選先に知れば回避できる

① 保護後に自分が push できず焦る ★★★

  • これは正常な動作（保護が効いている）
  • 対処：feature ブランチ→PR の正規フローで

  • 「fix: 修正」「update」など中身不明

  • feat と fix と docs が1コミットに
  • 対処：1コミット1目的（type も1つに決まる）
