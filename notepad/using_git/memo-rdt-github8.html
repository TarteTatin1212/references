この巻のミッション

MISSION

Vol.GH8 のゴールは「テストを自動化し、品質ゲートを完成させる」。これまで別冊S3で書いてきたテストを手で実行していましたが、「実行し忘れ」があると意味がない。CIでpush のたびに自動実行し、テストが通らないコードはマージできない状態を作ります。

「手で実行」から「自動実行」へRDTのTを自動化
// これまでの課題
  • テストは書いた（別冊S3・各本編）
  • でも実行は手動（実行し忘れがありうる）
  • 「push 後にテストが落ちていた」に気づけない
// CIで解決
  • push/PR で自動的にテスト実行
  • 落ちたら即通知
  • ブランチ保護と連携してマージをブロック
// RDT の完成形

R（要求）→D（設計）→T（テスト）が自動で回る。Vol.GH6 の品質ゲートにテスト通過が加わり、「壊れたコードが main に入らない」が完成。

// Vol.GH8 で完成する品質ゲート // ─── これまで（手動テスト）─── コードを書く ↓ （テストを手で実行…忘れることも）★ ↓ push ↓ （落ちてても気づかない可能性） // ─── Vol.GH8（CI自動化）─── コードを書く ↓ push / PR ↓ CI が自動でテスト実行 ★★★ ↓ ✓ 全部 green → マージ可能 ✗ 1つでも fail → マージ不可 ↓ main は常に「テスト通過済み」 // ─── 品質ゲートの完成 ─── Vol.GH6: PR必須・レビュー必須 ＋ Vol.GH8: テスト通過必須 ★★★ ＝ 「壊れたコードは絶対に main に入らない」 // → RDT の T（テスト）が // 仕組みで保証される
POINT — テストは「自動実行されて初めて意味を持つ」

どんなに良いテストを書いても、実行されなければ無意味。「忙しくてテストせずに push」「実行を忘れた」——人は必ずミスします。CIはテスト実行を人の意志から切り離し、機械的に毎回実行する。別冊S3で築いたテストが、CIによって「書いただけ」から「常に守ってくれる」存在に変わる。これがRDT版のテスト戦略の完成です。

02

CIとは何か

WHAT IS CI

CI（継続的インテグレーション）とは、コードの変更を頻繁に統合し、その都度自動でビルド・テストする習慣・仕組み。「早く・小さく・自動で」検証することで、問題を早期に発見します。

CI の考え方早期発見が価値
// CI の核心
  • 変更を頻繁に統合（ためこまない）
  • その都度自動でビルド・テスト
  • 問題を早く・小さいうちに発見
// なぜ早期発見が大事か

別冊S1の1:10:100の法則。バグも見つかるのが遅いほど直すコストが増大。CIで「push の瞬間に発見」できれば、最も安く直せる。

// CI で自動化すること
  • ビルド：コンパイルが通るか
  • テスト：全テストが緑か（別冊S3）
  • （発展）静的解析・カバレッジ等
// CI の価値 // ─── CI がない世界 ─── 各自が手元で開発 ↓ たまにまとめて統合 ↓ 「コンパイル通らない」 「他人の変更とぶつかった」 「テストが落ちてる」 → まとめて大量の問題が噴出 ★ → 原因特定が困難（どの変更?） // ─── CI がある世界 ─── 変更を頻繁に push ↓ push のたびに自動ビルド・テスト ↓ 問題があれば即発見 → 「さっきの変更が原因」と明確 ★★★ → 小さいうちに直せる // ─── 別冊S1 の 1:10:100 ─── バグ発見が早いほど安い： push直後に発見： 1（即直せる） レビューで発見： 10 本番で発見： 100 ★★★ // → CI は「最速で発見」する仕組み // → 別冊S1の要求工学と同じ思想 // ─── CI / CD ─── CI（継続的インテグレーション）: ビルド・テストの自動化 ← Vol.GH8 CD（継続的デリバリー/デプロイ）: リリースの自動化 ← 発展（本シリーズ範囲外）
POINT — CI は「1:10:100」を最小化する仕組み

別冊S1の1:10:100の法則（問題は発見が遅いほど高くつく）は、バグにも当てはまります。CIは「push の瞬間にテスト」で、バグを最速・最安で発見する仕組み。「本番で発覚（100倍）」を「push直後に発覚（1倍）」に変える。要求工学（別冊S1）の早期発見思想を、テストとビルドに適用したのがCI。RDT版の理念が、運用にまで一貫しています。

03

GitHub Actions の構造

GITHUB ACTIONS STRUCTURE

GitHub Actions は GitHub に組み込まれたCI/CD ツール。.github/workflows/ に YAML ファイルを置くだけで、push などをきっかけに自動でジョブを実行。基本構造を理解します。

ワークフローの構成要素YAML で定義
// 主要な概念
用語意味
Workflow自動化の定義全体（YAML 1ファイル）
Trigger（on）きっかけ（push/PR等）
Job実行単位（ビルド・テスト等）
Stepジョブ内の各手順
Action再利用可能な部品
// 置き場所

.github/workflows/ci.yml。このパスに YAML を置くと、GitHub が自動で認識。

// YAML の注意

インデントが命（スペース2つ、タブNG）。階層構造をインデントで表す。

// GitHub Actions の構造 # .github/workflows/ci.yml name: CI # ワークフロー名 on: # トリガー（きっかけ） push: branches: [ main ] pull_request: branches: [ main ] # → main への push/PR で起動 ★ jobs: # 実行する仕事 build: # ジョブ名 runs-on: ubuntu-latest # 実行環境 steps: # 手順（上から順に） - name: コード取得 uses: actions/checkout@v4 - name: JDK設定 uses: actions/setup-java@v4 - name: テスト実行 run: ./mvnw test # ─── 構造の階層 ─── Workflow（ci.yml） └ on（トリガー） └ jobs └ build（ジョブ） └ steps └ checkout（部品を使う） └ setup-java └ test（コマンド実行）★ # ─── uses と run ─── uses: 既存の Action（部品）を使う run: シェルコマンドを実行 # → 部品の再利用＋自前コマンド
POINT — Action は「再利用できる部品」

GitHub Actions の uses: は「他人が作った再利用可能な部品」を使う仕組み。actions/checkout（コード取得）や actions/setup-java（Java環境構築）など、定番作業は既製の Action を使えば一行で済む。自分でシェルを書くのは run:。「車輪の再発明をしない」のはコードと同じ（別冊S2のDRY）。Marketplace に無数の Action があります。

04

CIワークフローを書く

WRITE WORKFLOW

本シリーズ用の CI ワークフローを書きます。push/PR でビルド＆テスト。Spring Boot プロジェクトの定番構成です。別冊S3のテスト（単体＋H2結合）を自動で回します。

本シリーズの CI ワークフロービルド＋テスト
// 作成手順
  1. リポジトリに .github/workflows/ フォルダを作る
  2. ci.yml を作成
  3. トリガー・ジョブ・ステップを記述
  4. コミット＆push
  5. Actions タブで実行を確認
// ステップの流れ
  • コードを取得（checkout）
  • JDK 17 をセットアップ
  • 依存をキャッシュ（高速化）
  • テスト実行（mvnw test）

単体＋H2結合テストは外部DB不要なので、この構成で動く。

// .github/workflows/ci.yml name: CI on: push: branches: [ main ] pull_request: branches: [ main ] jobs: test: runs-on: ubuntu-latest steps: # 1. コードを取得 - name: Checkout uses: actions/checkout@v4 # 2. JDK 17 をセットアップ - name: Set up JDK 17 uses: actions/setup-java@v4 with: java-version: '17' distribution: 'temurin' cache: maven # 依存をキャッシュ（高速化） # 3. テスト実行 ★★★ - name: Run tests run: ./mvnw test # → 単体＋H2結合テストが走る # → 別冊S3のテストが自動実行 ★ # ─── キャッシュの効果 ─── # cache: maven で依存DLを再利用 # → 2回目以降の CI が高速に ★ # ─── FULLTEXT（Vol.8）のテストは? ─── # Testcontainers は CI でも動く（Dockerあり） # が、まずは H2 で動くテストから自動化 # → 段階的に拡張（YAGNI）★ # ─── コミット ─── $ git add .github/workflows/ci.yml $ git commit -m "ci: GitHub Actions でテスト自動化" $ git push # → push した瞬間に CI が動き出す ★★★
POINT — まず「H2で動くテスト」から自動化する

CI の最初の一歩は「外部依存なしで動くテスト」から。本シリーズの単体テストと H2 結合テストは外部DBが不要なので、CI で素直に動きます。Vol.8 の FULLTEXT（Testcontainers）は Docker が要りますが、GitHub Actions は Docker 対応なので追加可能。ただし最初から全部欲張らず、動くものから（YAGNI＝別冊S2）。「まず緑にする」→「徐々に拡張」がCI構築のコツです。

05

CIの動作とPR連携

CI IN ACTION

CI を push すると自動で動き出します。Actions タブで実行状況を確認。PR ではチェック結果が表示され、テストの成否が一目で分かる。Vol.GH3 の PR フローと連携します。

CI の実行とPRでの表示緑・赤が一目
// CI 実行の流れ
  1. feature ブランチで開発→push
  2. CI が自動起動
  3. Actions タブで進行を確認
  4. PR を作るとPR画面にチェック結果
  5. 緑なら安心してマージ
// PR での表示
  • 緑のチェック：全テスト通過
  • 赤の×：テスト失敗（詳細ログへ）
  • 黄の●：実行中

レビュアーは「テストが通っているPR」だけを安心してレビューできる。

// CI の実行とPR連携 // ─── Actions タブ（実行状況）───
Actions / CI ✓ ci: テスト自動化 main CI · 42s ✓ feat(memo): ピン留め追加 CI · 38s ✓ test (ubuntu-latest) ✓ Checkout ✓ Set up JDK 17 ✓ Run tests 42 tests passed ★
// ─── PR 画面でのチェック表示 ───
Pull Request #9 feat(memo): ピン留め機能を追加 ✓ All checks have passed ✓ CI / test (pull_request) This branch has no conflicts [ Merge pull request ] ← 緑なら安心 ★★★
// ─── テストが落ちた場合 ───
Pull Request #10 ✗ Some checks were not successful ✗ CI / test (pull_request) MemoSearchTest.キーワード検索 FAILED expected: 1 but was: 0 [ Details ] ← ログで原因確認
// → 落ちた箇所が即分かる ★★★ // → マージ前に修正できる
POINT — CIは「レビュー前の門番」になる

CIがあると、レビュアーの負担が減ります。「テストが通っているか」をCIが先にチェックしてくれるので、人間は「設計やロジックの妥当性」という人にしかできない判断に集中できる。Vol.GH3 のセルフレビューも、CIが緑なことを前提に「動くのは保証済み、では設計は?」と進められる。機械にできることは機械に、人は人にしかできないことを——役割分担の好例です。

06

ステータスチェック必須化

REQUIRED STATUS CHECKS

CIの仕上げ。Vol.GH6 のブランチ保護に「CIが通らないとマージ不可」を追加します。これで「テストが落ちたコードは物理的に main に入れない」。品質ゲートが完全に閉じます。

ブランチ保護にCIを連携テスト通過を必須に
// 設定手順
  1. Settings → Branches → main の保護ルール編集
  2. Require status checks to pass をオン
  3. 必須にするチェック（CI の test）を選択
  4. 保存
// 効果

CI が通らない PR はマージボタンが押せなくなる。Vol.GH6 の「PR必須・レビュー必須」に「テスト通過必須」が加わる。

// 品質ゲートの完成

これで main へのマージには①PR ②レビュー承認 ③テスト通過の全てが必要。3重のゲート。

// ステータスチェック必須化 // ─── ブランチ保護の追加設定 ───
Settings / Branches / main ☑ Require a pull request （Vol.GH6で設定済み） ☑ Require approvals: 1 （Vol.GH6で設定済み） ☑ Require status checks to pass before merging ★NEW ☑ CI / test ← 必須に指定
// ─── 完成した品質ゲート ─── main にマージするには： ① PR を作る（Vol.GH3） ② レビュー承認（Vol.GH6） ③ CI のテストが全部通る（Vol.GH8）★★★ ↓ 全部クリアして初めて マージ可能 // ─── テストが落ちたPRの状態 ───
✗ Required statuses must pass ✗ CI / test — Failing Merging is blocked [ Merge ] ← グレーアウト（押せない）★★★
// → テストが落ちたコードは // 物理的に main に入れない ★★★ // ─── RDT の完成 ─── R 要求 → Issue・テンプレート D 設計 → 3層・PR・レビュー T テスト → CIで自動実行・必須化 ★★★ // → 要求から検証まで仕組みで一貫
POINT — 「3重のゲート」で main を守る

Vol.GH8 完了時、main へのマージには①PR ②レビュー承認 ③CI通過の3つが必須に。これが3重の品質ゲート。①で変更を可視化、②で人間が設計を確認、③で機械が動作を保証。人と機械の両面から守る。「急いでるから」「自分のコードだから」という例外を仕組みで許さない。RDT版のT（テスト）が、運用レベルで完成した瞬間です。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

CI構築で陥りやすい地雷を5つ。★★★ 印は特に頻発。最初のCI構築は試行錯誤が普通なので、落ち着いて対処します。

Vol.GH8 ハマりどころ 5選CI構築の試行錯誤

① YAML のインデント崩れ ★★★

  • スペースとタブ混在・階層ズレ
  • 対処：スペース2つで統一、タブ禁止。エディタで可視化

② ローカルで通るがCIで落ちる ★★★

  • 環境差（JDKバージョン・タイムゾーン・依存）
  • 対処：CIのJDK/環境をローカルと揃える

③ ワークフローが起動しない

  • 置き場所・ファイル名・on の設定ミス
  • 対処：.github/workflows/*.yml を正確に、on を確認

④ DBが必要なテストがCIで失敗

  • Testcontainers 等で Docker/設定不足
  • 対処：まずH2テストから。Testcontainersは段階的に

⑤ CIが遅すぎる

  • 毎回ゼロから依存DL
  • 対処：cache: maven で依存をキャッシュ
// 主要な対処 # ─── ① YAML インデント ─── NG: jobs: test: # タブ → エラー ★ runs-on: ... # インデントずれ OK: jobs: test: # スペース2つ ★ runs-on: ubuntu-latest steps: - uses: ... # さらに2つ # ─── ② 環境を揃える ─── # ローカルが JDK17 なら CI も17 java-version: '17' # ← ローカルと一致 ★ # ─── ③ 起動確認 ─── # Actions タブにワークフローが # 表示されない → パス/ファイル名を確認 .github/workflows/ci.yml ✓ # ─── ⑤ キャッシュで高速化 ─── cache: maven # → 依存DLを再利用、CIが速くなる ★ # ─── CI構築は試行錯誤が普通 ─── # 失敗ログを読んで直すの繰り返し # → 落ちても焦らず、ログから原因特定 # → 別冊S7「ログで原因を追う」と同じ ★
