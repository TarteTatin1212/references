03

PRとは何か

WHAT IS PR

Pull Request（PR・通称プルリク）は「私のブランチを main に取り込んでください」という提案。提案だから差分が見え、議論でき、承認・却下できる。GitHub の最重要機能です。

PRの本質と構成要素変更提案＝議論の場
CONCEPT 1
変更の提案書
base へ compare を取り込む提案。何を・なぜを明示。
CONCEPT 2
レビューの場
マージ前に他人（or自分）の目を通す。コメントで議論。
CONCEPT 3
CIのトリガー
PR作成・更新で自動テストが走る（Vol.GH8で実装）。
CONCEPT 4
要求達成の記録
RDT版では受入条件チェックの場。Closesで要求完成を記録。
// PR が扱うのは「ブランチ間の差分」
  • base：取り込み先（main）
  • compare：取り込むブランチ（feature/memo-crud）
  • その差分（diff）が PR の中身
// PR の4タブ // ─── Conversation ─── PR本文・コメント・議論 受入条件チェックリスト （RDT版の主役） // ─── Commits ─── PR内のコミット一覧 Refs #1 付きコミットが並ぶ （Vol.GH2 で作ったもの） // ─── Files changed ─── 変更ファイルの diff 緑(+)＝追加、赤(-)＝削除 セルフレビューはここで // ─── Checks ─── CI（自動テスト）の結果 Vol.GH8 で GitHub Actions を設定 → テストが緑じゃないとマージ不可 に設定できる ★★★ // ─── PR と git merge の違い ─── git merge（Vol.GH2まで）： レビューなし・履歴のみ PR（今回）： レビュー・議論・差分可視化 ・受入条件確認・CI連携 → 業務はPRが標準 ★★★
POINT — 1人開発でも「自分PR」で練習する

「1人なのにPR？」と思うかもですが、1人だからこそ練習。①変更が整理される、②受入条件を最終確認できる、③業務に入った時に慣れている。RDT版では特に「受入条件チェックの場」として価値があります。自分でレビューして自分でマージは変ではなく標準練習です。

04

ハンズオン — PRを作る

HANDS-ON

Vol.GH2 で push した feature/memo-crud から PR を作ります。受入条件チェックリストを本文に、最後に Closes #1 を書く。これが RDT版のPRの型です。

PR作成 — 6ステップ受入条件付き
  1. feature/memo-crud を push 済みであることを確認
  2. GitHub で黄色いバナー Compare & pull request をクリック
  3. base=main / compare=feature/memo-crud を確認
  4. タイトル：feat(memo): メモCRUD実装（US-001）
  5. 本文に受入条件チェックリスト＋Closes #1
  6. Create pull request
// PR本文テンプレート（RDT版）
  • ## 概要：どのUSを実装したか
  • ## 受入条件：チェックボックス（Issueからコピー）
  • ## テスト：テスト結果
  • Closes #1：Issue 紐付け＆自動Close
// PR作成手順 // Step 1: push 確認 $ git push -u origin feature/memo-crud remote: Create a pull request for 'feature/memo-crud' on GitHub: remote: https://github.com/.../pull/new/ feature/memo-crud // Step 2-6: GitHub Web で PR 作成 // 本文に貼る内容： ## 概要 US-001（メモ作成）を実装しました。 memos の CRUD が動作します。 ## 受入条件 - [x] AC1 保存→一覧表示 - [x] AC2 空タイトルはエラー - [x] AC3 200文字は登録できる - [x] AC4 201文字はエラー ## テスト - MemoValidationTest（AC2-4） - MemoServiceTest（日時セット） - MemoControllerTest（AC1 MockMvc） - 6 tests, all green ## 設計メモ - 3層構造（Controller/Service/Mapper） - @NotBlank/@Size で受入条件を実装 Closes #1 // ↑ これでマージ時にIssue#1が // 自動Close ★★★ // → Create pull request クリック // → PR #5 が誕生
POINT — Closes は「PR本文」か「マージされるコミット」に

Closes #1 が効くのはPR本文かマージされるコミットメッセージに書いた時。PRのコメント欄に書いても効きません（よくある失敗）。本文に書けば、マージした瞬間に Issue #1 が自動Close。途中のコミットは Refs #1、完成PRの本文に Closes #1——この使い分けが Vol.GH2 からの一貫した方針です。

セルフレビューと受入条件確認

SELF REVIEW

マージ前にFiles changed で diff を読み、受入条件が全部満たされたかを確認します。RDT版では「コードの質」と「要求の達成」の両方をチェックします。

2軸のセルフレビューコードと要求
// ① コードのレビュー（Files changed）

別冊S2の設計原則で自問：

  • SRP：説明に「と」が入るクラスは無いか
  • 命名：Manager/Util/Helper が残っていないか
  • 層：Mapper が Controller を触っていないか
  • テスト：書ける構造か（DIされているか）
// ② 要求のレビュー（受入条件）

  • @AC1〜@AC4 が全部テストされているか
  • 異常系（空・超過）も書いたか
  • テストが全部緑か

気づきは行コメントで残す。「なぜこう書いたか」を未来の自分へ。

POINT — Files changed を必ず1回見てからマージ

作成して数秒で即マージはセルフレビューゼロ。意識的に「Files changed を1回見る」習慣を。自分のコードでも「他人のつもりで読む」と粗が見える。これは業務でレビュアーの視点を養う練習にもなります。急がば回れ。

06

Closes でトレース完結

CLOSING THE TRACE

RDT版の集大成。PR をマージすると Closes #1 が発動し、Issue #1 が自動Close。要求→実装→完了の一本の線が完結します。3つのマージ戦略も合わせて学びます。

3つのマージ戦略本シリーズは Merge commit
戦略履歴本シリーズ
Merge commit全コミット保持＋マージコミット採用
Squash1コミットに圧縮—
Rebase線形に並べる—

本シリーズは Merge commit。理由：Refs #1 付きの層ごとコミットを履歴に残したいから（トレーサビリティ重視）。Squash すると個々のコミットが消え、層ごとの記録が失われます。

// マージ手順
  1. 受入条件が全部 [x] を確認
  2. Merge pull request ボタン
  3. Confirm merge
  4. Issue #1 が自動Close されたか確認
  5. Delete branch でブランチ削除
// マージ後に起きること // ─── Merge ボタンを押すと… ─── PR #5 が main にマージ ↓ 本文の "Closes #1" が発動 ↓ Issue #1 が自動的に Closed ↓ Issue #1 に記録： "yourname closed this in #5" ↓ US-001 完成が永久に記録される ★★★ // ─── 完結したトレーサビリティ ───
Issue #1 — US-001（Closed）
● Closed ## ユーザーストーリー As a ... I want ... so that ... ## 受入条件 - [x] AC1 〜 [x] AC4（全部完了） ────────────── ● referenced in commits: feat(entity)... feat(mapper)... (Refs #1 のコミット群) ● closed via PR #5 (Closes #1)
// → Issue を見れば全部分かる： // 何を作るはずだったか（US） // どう実装されたか（コミット） // いつ完成したか（PR #5）★★★

POINT — Squash か Merge commit かは「何を残したいか」

本シリーズがMerge commitを選ぶのは、Refs #1 付きの層ごとコミットを履歴に残したいから。「entity追加→mapper追加→…→test追加」の開発の物語が残ります。Squash すると物語が1行に圧縮される。トレーサビリティ重視なら Merge commit、main の見た目重視なら Squash。プロジェクトの方針で選びます。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

PR操作で踏みやすい地雷を5つ。★★★ 印は特に頻発。

Vol.GH3 ハマりどころ 5選先に知れば回避できる

① Closes #1 を書いたのにCloseされない ★★★

  • 原因A：PRのコメント欄に書いた（本文でないと効かない）
  • 原因B：マージしていない（マージ時に発動）
  • 対処：PR本文に Closes #1、マージで発動

② base ブランチを間違える

  • 原因：base が main 以外になっている
  • 対処：PR作成画面で base=main を目視確認

③ Merge ボタンが押せない（コンフリクト）★★★

  • 原因：main が進んで feature と競合
  • 対処：ローカルで git merge origin/main → コンフリクト解決 → push

④ マージ後にブランチが残る

  • 対処：PR下部の Delete branch をクリック
  • 予防：Settings で「マージ後自動削除」をON

⑤ 受入条件が [ ] のままマージ

  • 問題：要求が未達なのに完成扱いに
  • 対処：マージ前に全部 [x] を確認する習慣を
// コンフリクト解決（③） // PR画面で「This branch has conflicts」 $ git switch feature/memo-crud $ git fetch origin $ git merge origin/main CONFLICT (content): Merge conflict in src/.../templates/memos/list.html // エディタで該当ファイルを開くと： <<<<<<< HEAD <h1>メモ一覧</h1> ======= <h1>メモ帳</h1> >>>>>>> origin/main // → どちらを残すか決め、マーカーを削除 $ git add . $ git commit -m "merge: mainを取り込みコンフリクト解決" $ git push // → PR が自動更新、マージ可能に ★★★ // ─── ① Closes が効く場所 ─── 効く: PR本文に Closes #1 マージされるコミットに Closes #1 効かない: PRのコメント欄に Closes #1 ★★★ （よくある失敗）
