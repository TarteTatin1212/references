TIP — MoSCoW をラベルにする

本編Vol.1 で決めたMoSCoW（Must/Should/Could/Won't）を、そのままGitHub のラベルにできます。must（赤）should（緑）could（黄）。Issue 一覧を見れば「次に何を作るべきか」が一目。要求と運用が繋がります。

03

準備するもの

PREREQUISITES

揃えるのは4つ。Git本体・GitHubアカウント・認証方法・Eclipse EGit。30分で終わります。ここを丁寧にやれば本番で詰まりません。

準備チェックリストこの4つ
// 必要なもの
  • Git本体がインストール済み（git --version で確認）
  • GitHubアカウントを作成済み（無料）
  • 認証方法を決めた（HTTPS+PAT 推奨）
  • Eclipse EGitが有効（標準同梱）
// 初回のユーザー設定

Gitインストール直後に必ず実行：

  • git config --global user.name "名前"
  • git config --global user.email "GitHubのメール"

これを忘れると別人名義でコミットされます。

// インストール確認と初期設定 $ git --version git version 2.43.0 // → 出ればOK。出なければ git-scm.com から // ─── 初回ユーザー設定 ─── $ git config --global user.name "Your Name" $ git config --global user.email "you@example.com" // ─── 認証：HTTPS+PAT（推奨）─── // GitHub右上アバター → Settings // → Developer settings // → Personal access tokens (classic) // → Generate new token // → scope: repo にチェック // → トークンをコピー（二度と表示されない） // push 時に： // Username: yourname // Password: ghp_xxxx（PATを貼る） // ※2021年8月でパスワード直入力は廃止 // → PAT必須 ★★★
POINT — パスワード直入力は廃止済み

かつてはGitHubのログインパスワードで push できましたが、2021年8月に廃止。今は PAT か SSH鍵 が必須。「パスワード聞かれてログインパスワード入れたら通らない」がハマりどころNo.1。PATを使ってください。

04

ハンズオン（コマンドライン版）

CLI HANDS-ON

本編Vol.1 で作ったメモ帳プロジェクトを GitHub に置きます。git init から push まで。4つの基本コマンドで完了します。

手順 — 7ステップこれで GitHub に置ける
  1. プロジェクトフォルダに移動（cd）
  2. git init でGit管理開始
  3. .gitignore を用意（Spring Boot用）
  4. README.md を作成
  5. git add / commit で初コミット
  6. GitHubでリポジトリ作成（Web）
  7. git remote add / push で送信
// .gitignore（Spring Boot用）
  • target/：mvn build 生成物
  • .classpath .project .settings/：Eclipse個人設定
  • *.log：ログ
  • application-local.properties：個人のDB接続情報
// 全コマンド一覧 $ cd ~/workspace/memo $ git init # .gitignore 作成 $ cat > .gitignore << 'EOF' target/ .classpath .project .settings/ bin/ .idea/ *.iml *.log application-local.properties EOF # README.md 作成 $ cat > README.md << 'EOF' # memo-app Spring Boot + MyBatis のメモ帳アプリ。 要求・設計・テスト駆動（RDT）で開発。 ## 環境 - Java 17 / Spring Boot 3.x - MyBatis / MySQL 8.0 ## テーブル - memos / categories / tags - memo_tags（中間テーブル） ## 開発方針 - 要求(US) → 設計 → 実装 → テスト - 要求は Issue で管理 EOF # 初コミット $ git add . $ git commit -m "chore: プロジェクト初期化 - Spring Boot + MyBatis + MySQL - 3テーブル設計のDDL - 要求(US-001〜004)を定義済み" # GitHub でリポジトリ作成後… $ git branch -M main $ git remote add origin https://github.com/yourname/memo-app.git $ git push -u origin main // → GitHub に置けた ★★★
POINT — 初コミットのメッセージにも要求を残す

RDT版では、初コミットのメッセージにも「要求を定義済み」と記録します。chore: プロジェクト初期化 / 要求(US-001〜004)を定義済み。コミット履歴を遡れば「いつ要求が固まったか」も分かる。履歴が開発の記録になるのが RDT版の思想です。

05

Eclipse EGit 版

EGIT VERSION

コマンドが苦手な方向けの GUI 版。Eclipse の EGit（標準同梱）で同じことを画面操作で完結させます。

EGit 手順 — Share から Push右クリックで完結
  1. プロジェクト右クリック → Team → Share Project...
  2. Git 選択 → Create Repository → Finish
  3. .gitignore と README.md を作成（New → File）
  4. Git Staging ビューで全ファイルをStaged
  5. コミットメッセージ入力 → Commit and Push...
  6. URI入力 → GitHubのURL貼付け → 認証（PAT）
// ビューの開き方
  • Git Staging：Window → Show View → Other → Git → Git Staging
  • History：プロジェクト右クリック → Team → Show in History
// コマンド ↔ EGit 対応マップ
コマンドライン
Eclipse EGit
git init
Team → Share Project → Create Repository
git add .
Git Staging で ＋＋（全Stagedへ）
git commit -m
Git Staging でメッセージ → Commit
git push -u origin main
Commit and Push / Team → Push Branch
git log
Team → Show in History
TIP — 両方使えるのがベスト

業務では「コミットはEGit、面倒な操作はコマンド」のハイブリッドが主流。EGitはdiffの可視化が秀逸、コマンドは履歴整理が速い。両方使い分けられると強いです。

後の Vol.GH3（PR編）で、コミットメッセージに Refs #1 と書くと、Issue #1 とコミットが自動で紐付く。「この要求が、どのコードで実現されたか」が追跡できる。これがトレーサビリティです。

POINT — 「要求の見える化」が開発を変える

頭の中やテキストファイルにある要求はすぐ忘れられ、散逸します。GitHub の Issue に置けば「やることリスト」が常に見える。完成したら Close、変更が出たら新 Issue。要求が生き物として管理される。これが要求工学の「管理」フェーズの実践です。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

Vol.GH1 で新人がよく踏む地雷を5つ。★★★ 印は特に頻発。

Vol.GH1 ハマりどころ 5選先に知れば回避できる

① fatal: refusing to merge unrelated histories ★★★

  • 原因：GitHub側でREADME作成＋ローカルでも初コミット
  • 対処：git pull origin main --allow-unrelated-histories
  • 予防：GitHub で空リポジトリを作る（README追加しない）

② Authentication failed（パスワード弾かれる）★★★

  • 原因：ログインパスワードを入力した
  • 対処：PATを作成して入力（Section 03）

③ target/ がコミットされた

  • 原因：.gitignore を書く前に add した
  • 対処：git rm -r --cached target/ → 再コミット
  • 予防：.gitignore は git init 直後に作る

④ Issue にラベルが付けられない

  • 原因：ラベルを事前に作っていない
  • 対処：Issues → Labels → New label で must/should/could を先に作成

⑤ Issue の受入条件チェックボックスが効かない

  • 原因：- [ ] の書式ミス（ハイフン＋スペース＋角括弧）
  • 対処：- [ ] 内容 の形を正確に（スペース必須）
// 主要エラーの対処 // ─── ① unrelated histories ─── $ git pull origin main --allow-unrelated-histories $ git push // ─── ③ target/ 巻き込み ─── $ echo "target/" >> .gitignore $ git rm -r --cached target/ $ git add .gitignore $ git commit -m "chore: ignore target dir" // ─── ⑤ チェックボックスの正しい書式 ─── 正: - [ ] 空タイトルはエラー └┬┘└┬┘ │ └ 半角スペース └ ハイフン+スペース 誤: -[] 内容 // スペースなし *[ ] 内容 // アスタリスク - [x] 内容 // x はチェック済み
