この巻のミッション

MISSION

Vol.GH11 のゴールは「他者が参加できるプロジェクトにする」こと。これまで Vol.GH1〜10 で「一人で安全・確実に開発する」仕組みを作りました。今回はそれを「みんなで育てる」段階へ。貢献のルール・ライセンス・行動規範を整えます。

「一人」から「みんな」へ開かれたプロジェクト
// これまでとの違い

Vol.GH1〜10 は「開発者（自分）のための仕組み」。Vol.GH11 は「参加者（他者）のための仕組み」。視点が内から外へ。

POINT — 公開とは「他者を迎え入れる」こと

リポジトリを公開するのは、単にコードを見せるだけではありません。「他者が使い・貢献できる場を作る」こと。そのためには「どう参加するか（CONTRIBUTING）」「使ってよいか（LICENSE）」「安心できるか（行動規範）」を示す必要がある。これらがないと、訪問者は「貢献していいか分からない」「使っていいか不安」になる。Vol.GH11 は、プロジェクトを「開く」ための準備です。

02

コントリビューションとは

WHAT IS CONTRIBUTION

コントリビューション（貢献）とは、他者がプロジェクトに参加すること。コード修正だけでなく、バグ報告・ドキュメント改善・提案も貢献。これまで学んだIssue・PR・Forkが、貢献の手段になります。

貢献の種類と流れコードだけではない
// 貢献の種類
  • コード：機能追加・バグ修正のPR
  • 報告：バグ報告・改善提案のIssue
  • ドキュメント：READMEの誤字修正等
  • レビュー：他者のPRを確認

コードを書かなくても貢献できる。

// Fork & PR の流れ

外部の人はリポジトリを Fork（複製）し、自分の複製で作業→PRで本家に提案。これまでの PR（Vol.GH3）の応用。

// これまでの仕組みが活きる

Issue（Vol.GH4）、PR（Vol.GH3）、テンプレート（Vol.GH7）、CI（Vol.GH8）——全部が貢献を受け入れる土台。

// コントリビューションの全体像 // ─── 貢献はコードだけではない ─── コード貢献: バグ修正・機能追加のPR 報告貢献: 「ここがバグ」とIssue 「こんな機能が欲しい」と提案 ドキュメント貢献: READMEの誤字修正・説明追加 レビュー貢献: 他者のPRを確認・コメント // → 誰でも何かしら貢献できる ★ // ─── 外部貢献者の Fork & PR ─── 本家リポジトリ ↓ ① Fork（自分のアカウントに複製） あなたの Fork ↓ ② clone して作業 ↓ ③ ブランチで修正（Vol.GH2） ↓ ④ Fork に push ↓ ⑤ 本家に PR（Vol.GH3） 本家のメンテナがレビュー ↓ CI（Vol.GH8）が走る ↓ OK ならマージ // → 外部の人も安全に貢献できる ★★★ // ─── これまでの仕組みが土台に ─── Issue（Vol.GH4） → 報告・提案の受付 PR（Vol.GH3） → コード貢献の受付 テンプレート（GH7）→ 記述の統一 CI（Vol.GH8） → 貢献の品質を自動チェック 保護（Vol.GH6） → mainを守る // → 全部が「貢献を受け入れる」仕組み ★★★
POINT — これまでの仕組みは「貢献を受け入れる土台」だった

Vol.GH1〜10 で作った仕組み——Issue、PR、テンプレート、CI、ブランチ保護——は、実は「他者の貢献を安全に受け入れる」ための土台でもありました。外部の人がPRを出しても、CIが品質をチェックし、ブランチ保護がmainを守る。「知らない人のコードでも、仕組みが安全を保証」する。RDT版で積み上げた「仕組みで品質を守る」が、コミュニティ運営でこそ真価を発揮します。

03

CONTRIBUTING.md

CONTRIBUTING GUIDE

CONTRIBUTING.md は「貢献の手引き」。「どう環境を作るか」「どうPRを出すか」「どんなルールか」を書く。これがあると貢献者が迷わない。RDT版の規約（コミット規約・テスト必須等）を明文化します。

貢献の手引きを書く迷わせない
// CONTRIBUTING に書くこと
  • 環境構築：セットアップ手順
  • 開発フロー：ブランチ・コミット規約
  • テスト：テストの実行・追加方法
  • PRの出し方：レビューの流れ
// RDT版のルールを明文化

これまで暗黙だった「Conventional Commits（Vol.GH6）」「テスト必須（Vol.GH8）」「受入条件を書く（Vol.GH7）」を文書化。

// 配置

ルートまたは .github/ に CONTRIBUTING.md。GitHub がPR作成時に自動でリンク表示。

// CONTRIBUTING.md（RDT版） # コントリビューションガイド メモ帳アプリへの貢献歓迎です！ ## 環境構築 1. このリポジトリを Fork 2. clone して `./gradlew bootRun` 3. 環境変数 DB_PASSWORD を設定（README参照） # → Vol.GH5,7 の内容 ## 開発フロー 1. Issue を作成（or 既存Issueを確認） 2. ブランチを切る: `feature/xxx` # → Vol.GH2 3. 変更してコミット # → Conventional Commits（Vol.GH6） 例: `feat(memo): タグ機能を追加` 4. テストを追加・実行 # → 全テスト緑が必須（Vol.GH8） 5. PR を作成 # → テンプレートに沿って（Vol.GH7） ## コミット規約 - feat: 機能追加 - fix: バグ修正 - docs: ドキュメント （Conventional Commits 準拠・Vol.GH6） ## テスト - 新機能には必ずテストを追加 - `./gradlew test` で全緑を確認 - CI が自動で確認します（Vol.GH8） ## PR のルール - 受入条件を満たすこと - CI が緑であること - レビュー承認後マージ // ─── 効果：PR作成時に表示 ───
Open a pull request 📋 CONTRIBUTING.md を読みましたか？ 本文（テンプレート・Vol.GH7）: ## 受入条件 - [ ] ...
// → 貢献者が迷わず参加できる ★★★
POINT — CONTRIBUTING は「暗黙のルールの明文化」

CONTRIBUTING.md の本質は「これまで暗黙だったルールを言葉にする」こと。あなたの頭の中にある「コミットはこう書く」「テストは必須」「PRはこう出す」——これらは、新しい貢献者には分かりません。文書化することで「誰でも同じやり方で参加できる」。RDT版で積み上げた規約（Vol.GH6,7,8）を明文化すれば、品質を保ったまま、参加の門戸が開く。これは別冊S2の「暗黙知を形式知に」する設計でもあります。

04

ライセンスの選択

CHOOSING A LICENSE

ライセンスは「このコードをどう使ってよいか」を定める法的な許諾。ライセンスがないと、他者は使えない（法的にグレー）。代表的なライセンスの特徴と、選び方を理解します。

ライセンスの種類と選択使ってよいかを定める
// ライセンスがないと

ライセンス未記載のコードは「全権利が作者に」＝他者は法的に使えない（公開していても）。使ってもらうにはライセンスが必須。

// 代表的なライセンス
ライセンス特徴
MIT最も自由・シンプル
Apache 2.0特許条項あり
GPL派生物も公開義務
// 選び方
  • 自由に使ってほしい→ MIT（個人プロジェクトの定番）
  • 派生物も公開させたい→ GPL
  • 迷ったらMIT（シンプル・寛容）
// ライセンスの選択 // ─── ライセンスがないと… ─── ライセンス未記載のコード = 著作権法上「全権利が作者」 = 他者は使用・改変・再配布できない // → 公開していても「使えない」★ // → 使ってもらうにはライセンス必須 // ─── MIT ライセンス（定番）─── # LICENSE ファイル MIT License Copyright (c) 2024 あなたの名前 Permission is hereby granted, free of charge, to any person obtaining a copy... （自由に使用・改変・再配布してよい） （ただし無保証） // → 最も自由・シンプル ★ // → 「自由に使ってOK、自己責任で」 // ─── ライセンスの選択フロー ─── 自由に使ってほしい？ → Yes: MIT（個人の定番）★ 派生物も公開させたい？ → Yes: GPL 特許も考慮？ → Apache 2.0 迷ったら？ → MIT（寛容・シンプル） // ─── GitHub での追加（簡単）─── # リポジトリで Add file → Create # ファイル名 "LICENSE" と打つと # GitHub がライセンス選択UIを出す ★ # → MIT等を選ぶと雛形が入る # → 名前・年を埋めるだけ // ─── 注意：法的な話 ─── // ライセンスは法的な許諾 // → 業務・組織では法務に確認を // → 個人学習なら MIT で十分 ★
POINT — 「公開＝自由に使える」ではない、ライセンスが必須

よくある誤解：「GitHubで公開したから、誰でも使っていい」——これは間違い。ライセンスがないコードは、法的に「全権利が作者」で、他者は使えません。「使ってほしい」ならライセンスを明示する必要がある。個人プロジェクトならMIT（自由・シンプル）が定番。GitHub にはライセンス選択UIがあり、簡単に追加できる。「公開」と「使用許諾」は別物と覚えておきましょう。

05

行動規範とコミュニティ

CODE OF CONDUCT

行動規範（Code of Conduct）は「参加者がどう振る舞うべきか」を定める。誰もが安心して参加できる場を作るためのルール。コードの品質と同じく、コミュニティの健全さも仕組みで守るという発想です。

安心して参加できる場づくり振る舞いのルール
// 行動規範とは

参加者の振る舞いの基準。「敬意を持って」「建設的に」「差別をしない」等。安心して参加できる場を保証する。

// なぜ必要か
  • 誰もが萎縮せず参加できる
  • 問題行動への対応基準が明確
  • プロジェクトの価値観を示す
// 定番のテンプレート

Contributor Covenant という広く使われる雛形がある。ゼロから書かず、定番を採用（別冊S2のKISS：車輪の再発明をしない）。

// 行動規範 // ─── CODE_OF_CONDUCT.md ─── # 行動規範 ## 私たちの約束 このプロジェクトの参加者として、 私たちは誰もが harassment のない 体験ができるよう努めます。 ## 期待される行動 - 他者への敬意を持つ - 建設的なフィードバック - 異なる視点を尊重する - コミュニティのために協力する ## 許容されない行動 - 差別的な言動 - 個人攻撃・嫌がらせ - 他者のプライバシー侵害 ## 対応 違反は [連絡先] へ報告してください。 メンテナが適切に対応します。 // ─── Contributor Covenant（定番）─── # 広く使われる行動規範のテンプレート # https://www.contributor-covenant.org # → ゼロから書かず定番を採用 ★ # → 多くのOSSが採用（信頼性） // ─── 「コミュニティも仕組みで守る」 ─── コード品質: CI・テスト・規約で守る （Vol.GH6,8） ↓ 同じ発想 コミュニティ: 行動規範で守る ★★★ 「安心して参加できる場」 // ─── GitHub の表示 ─── # CODE_OF_CONDUCT.md を置くと # リポジトリの「Community」タブに表示 # → プロジェクトの健全さを示す ★
POINT — 「コードの品質」と「コミュニティの健全さ」は両輪

RDT版ではコードの品質を仕組み（CI・テスト・規約）で守ってきました。行動規範は、同じ発想をコミュニティに適用したもの。「安心して参加できる場」を、ルールという仕組みで保証する。優れたコードがあっても、コミュニティが荒れていれば誰も参加しない。逆に、健全なコミュニティは多くの貢献を集める。技術と人、両方の健全さが、長く続くプロジェクトを作ります。

06

貢献を受け入れる運営

ACCEPTING CONTRIBUTIONS

外部からPRが来たとき、どう対応するか。これまでのレビュー・CIがそのまま活きます。さらに「貢献者への感謝」「丁寧なフィードバック」という、運営の心構えも大切です。

外部PRへの対応仕組み＋心構え
// 外部PRの対応フロー
  1. CIが自動で走る（Vol.GH8）
  2. CONTRIBUTING に沿っているか確認
  3. コードをレビュー（建設的に）
  4. 必要なら修正を依頼
  5. OKならマージ＋感謝
// メンテナの心構え
  • 感謝を伝える：貢献は善意
  • 建設的に：否定でなく改善提案
  • 明確に：何を直すか具体的に
  • 行動規範に沿って：敬意を持つ
// 仕組みが支える

CI（Vol.GH8）が品質を自動チェックするので、メンテナは「設計・方針」のレビューに集中できる。

// 貢献を受け入れる運営 // ─── 外部PRが来たとき ───
Pull request from contributor-x feat(search): 検索履歴を追加 ✓ CI / test (passed) ✓ CI / build (passed) メンテナのコメント: "素晴らしい貢献をありがとうございます！🎉 1点だけ、テストを追加いただけますか？ それ以外は完璧です。"
// → 感謝＋建設的フィードバック ★★★ // ─── レビューの心構え（行動規範）─── NG: 「このコードはダメ」 // → 否定的・貢献者が萎縮 ★ OK: 「ありがとうございます！ ここをこう変えると、より良くなります」 // → 感謝＋具体的な改善提案 ★★★ // ─── 仕組みが運営を支える ─── CI（Vol.GH8） → テストの合否を自動判定 → メンテナは「動くか」を確認不要 ブランチ保護（Vol.GH6） → 未レビューのマージを防ぐ CONTRIBUTING（今回） → 「ルールはここ」と示せる // → 仕組みがある分、人は「対話」に集中 ★ // ─── 「貢献は善意」を忘れない ─── // 外部貢献者は時間を使って // 善意で貢献してくれている // → 感謝と敬意を持って対応 ★★★ // → 良い運営が良い貢献を呼ぶ
POINT — 「仕組み」が整うほど、人は「対話」に集中できる

面白いことに、仕組み（CI・規約・保護）が整うほど、メンテナは人との対話に集中できます。「テストが通るか」はCIが判定するので、メンテナは「設計の方針」「貢献者への感謝」に時間を使える。仕組みが機械的な確認を肩代わりし、人は人にしかできないこと（対話・判断・感謝）に注力する。RDT版で築いた仕組みは、「良いコミュニティ運営」の余裕を生むのです。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

コミュニティ運営で陥りやすい点を5つ。コードと違い「人」が関わるので、技術以外の配慮も必要です。

Vol.GH11 ハマりどころ 5選人が関わる難しさ

① ライセンスを付けずに公開 ★★★

  • 「公開＝使ってOK」と誤解
  • 対処：LICENSE を必ず追加（Section 04）

② CONTRIBUTING がなく貢献者が迷う

  • 「どう参加すれば?」が不明
  • 対処：手引きを明文化（Section 03）

③ 否定的なレビューで貢献者が離れる ★★★

  • 厳しい言葉で萎縮させる
  • 対処：感謝＋建設的に（Section 06・行動規範）

④ 行動規範を「不要」と軽視

  • 小さなプロジェクトでも安心の土台
  • 対処：定番テンプレートを採用（Section 05）

⑤ PRを放置する

  • 貢献者の善意を無駄にする
  • 対処：早めに反応（せめて「確認します」）
