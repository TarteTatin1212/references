02

タグとは何か

WHAT IS A TAG

タグは「特定のコミットに付ける名前付きの印」。ブランチが「動き続ける線」なら、タグは「固定された一点」。「このコミットが v1.0.0 だ」と永久に動かない目印を付けます。

タグ vs ブランチ動かない印
// タグとブランチの違い
ブランチタグ
性質動く線固定の点
用途開発を進める節目を記録
移動コミットで進む動かない
例main, feature/xv1.0.0
// 2種類のタグ
  • 軽量タグ：単なる印（名前だけ）
  • 注釈付きタグ：作者・日付・メッセージ付き（推奨）

リリースには注釈付きタグを使う。「誰が・いつ・何を」が残る。

// タグの概念 // ─── ブランチは動く、タグは固定 ─── コミット履歴： A---B---C---D---E ← main（動き続ける） | v1.0.0 ← タグ（Cに固定・動かない） // main は E, F, G... と進むが // v1.0.0 は C を指したまま ★★★ // → 「v1.0.0 の時点」がいつでも分かる // ─── タグでできること ─── $ git checkout v1.0.0 // → v1.0.0 の時点のコードに戻れる ★ // → 「v1.0.0 で何が起きたか」を調査 // ─── 軽量タグ vs 注釈付きタグ ─── # 軽量タグ（印だけ） $ git tag v1.0.0 # 注釈付きタグ（推奨）★ $ git tag -a v1.0.0 -m "初回リリース" // → 作者・日付・メッセージが記録される // → リリースには注釈付きを使う ★★★
POINT — タグは「動かない」からこそ信頼できる

タグの価値は「一度打ったら動かない」こと。ブランチ（main）は新しいコミットでどんどん進みますが、タグ v1.0.0 は永久にその時点を指し続ける。だから「v1.0.0 のコードが見たい」「v1.0.0 に戻したい」がいつでもできる。「あの完成版の状態」を永久保存する栞（しおり）がタグ。リリースの土台になる、シンプルで強力な仕組みです。

03

セマンティックバージョニング

SEMANTIC VERSIONING

バージョン番号「v1.2.3」には意味があります。セマンティックバージョニング（SemVer）は、番号の付け方のルール。MAJOR.MINOR.PATCH の3つで「何が変わったか」を表現します。

MAJOR.MINOR.PATCH の意味番号で変更の規模を示す
// 3つの数字の意味
位置名前上げるとき
1番目MAJOR互換性を壊す変更
2番目MINOR後方互換の機能追加
3番目PATCH後方互換のバグ修正
// Conventional Commits との対応

Vol.GH6 のコミット規約と綺麗に対応：

  • feat（機能追加）→ MINOR を上げる
  • fix（バグ修正）→ PATCH を上げる
  • 互換性破壊 → MAJOR を上げる
// セマンティックバージョニング // ─── v1.2.3 の構造 ─── v 1 . 2 . 3 │ │ └ PATCH：バグ修正 │ └──── MINOR：機能追加（互換あり） └──────── MAJOR：互換性を壊す変更 // ─── バージョンが上がる例 ─── v1.0.0 初回リリース（US-001〜005） ↓ バグ修正（fix） v1.0.1 検索の不具合を修正 ↓ 機能追加（feat） v1.1.0 エクスポート機能を追加 ↓ さらに機能追加 v1.2.0 ページネーション追加 ↓ 互換性を壊す大改修 v2.0.0 DB構造を一新（移行が必要） // ─── 上の位が上がると下はリセット ─── v1.2.3 → MINOR上げ → v1.3.0 （PATCHは0に） v1.3.0 → MAJOR上げ → v2.0.0 （MINOR/PATCHは0に） // ─── Conventional Commits と対応（Vol.GH6）─── feat: → MINOR ↑（機能追加） fix: → PATCH ↑（バグ修正） 互換性破壊 → MAJOR ↑ // → コミットの type からバージョンが決まる // → 規約が一貫している ★★★
POINT — バージョン番号は「ユーザーへのメッセージ」

SemVer の本質は「番号を見れば、アップデートの影響が分かる」こと。PATCH 上げ（v1.0.0→v1.0.1）なら安心して更新（バグ修正だけ）、MAJOR 上げ（v1→v2）なら要注意（互換性が壊れる＝移行作業が要る）。番号がユーザーへのメッセージになる。適当に番号を振るのでなく、変更の規模を番号で正直に伝えるのが SemVer の約束です。

04

タグを打つ

CREATE A TAG

実際にv1.0.0 のタグを打ちます。US-001〜005 が完成した今が最初のリリースの好機。注釈付きタグを作り、GitHub に push します。push を忘れないのがポイント（Section 07）。

v1.0.0 タグの作成と push注釈付きタグ
// タグを打つ手順
  1. main が最新・テスト緑（CI通過）を確認
  2. 注釈付きタグを作成（-a -m）
  3. GitHub に push（--tags または push origin v1.0.0）
  4. GitHub のタグ一覧で確認
// タグを打つタイミング
  • main にあるべき変更が全て入った状態
  • CIが緑（テスト通過・Vol.GH8）
  • → 「リリースできる品質」が保証された時点
// タグの作成と push // ─── ① 状態確認 ─── $ git switch main $ git pull # main が最新・CI緑を確認 ★ // ─── ② 注釈付きタグを作成 ─── $ git tag -a v1.0.0 -m "v1.0.0 初回リリース メモ帳アプリ初版。 - メモのCRUD（US-001） - カテゴリ分類（US-002） - タグ付け（US-003） - 検索（US-004） - ピン/アーカイブ（US-005）" // → 作者・日付・メッセージが記録 ★ // ─── ③ GitHub に push ─── $ git push origin v1.0.0 # または全タグまとめて： $ git push --tags // → タグを忘れず push（重要・Section 07）★★★ // → 通常の push ではタグは送られない！ // ─── ④ 確認 ─── $ git tag v1.0.0 $ git show v1.0.0 tag v1.0.0 Tagger: あなた <...> Date: ... v1.0.0 初回リリース ... // ─── GitHub上でも確認 ───
github.com/.../tags
🏷 v1.0.0 Latest v1.0.0 初回リリース 3 commits to main since this tag
// → タグが GitHub に表示される ★★★
POINT — タグは「git push --tags」しないと共有されない

最も多いミス：タグを打ったのに GitHub に出てこない。原因は、通常の git push ではタグが送られないから。タグはgit push origin v1.0.0（個別）または git push --tags（全部）で明示的に pushする必要があります。「ローカルにタグはあるのに GitHub にない」と思ったら、まず push を確認。これは Section 07 のハマりどころでも再掲します。

05

GitHub Releases

GITHUB RELEASES

GitHub Releases は、タグを「正式なリリース」として公開する機能。タグにリリースノート（変更点の説明）を添え、必要なら成果物（jar等）を添付。ユーザーが「最新版」を入手できる場所になります。

タグからリリースを作成タグ＋説明＋成果物
// タグとリリースの違い
  • タグ：コミットへの印（git の機能）
  • リリース：タグに説明・成果物を添えた公開物（GitHub の機能）

リリースはタグの上に乗る。タグなしにリリースは作れない。

// 作成手順
  1. GitHub の Releases → Draft a new release
  2. タグ v1.0.0 を選択
  3. タイトル・リリースノートを記入
  4. 必要なら成果物（jar）を添付
  5. Publish release
// GitHub Releases // ─── タグ → リリース ─── git tag v1.0.0 ← ① 印（git） ↓ GitHub Release ← ② 公開物（GitHub） + リリースノート + 成果物（任意） // ─── リリース画面のイメージ ───
github.com/.../releases
📦 v1.0.0 — 初回リリース Latest 🏷 v1.0.0 ## ✨ 新機能 - メモのCRUD（US-001） - カテゴリ分類（US-002） - タグ付け（US-003） - 全文検索（US-004） - ピン留め・アーカイブ（US-005） ## 📦 Assets memo-app-1.0.0.jar (24.5 MB) Source code (zip) Source code (tar.gz)
// ─── リリースの価値 ─── - ユーザーが「最新の安定版」を入手 - 「v1.0.0 で何ができるか」が一目 - 成果物（jar）をダウンロード可能 - 過去のリリースも一覧で残る ★★★ // ─── CI と連携（Vol.GH8）─── // GitHub Actions で「タグ push 時に // 自動でビルド→リリース作成」も可能 // → リリースの自動化（発展）★
POINT — リリースは「ユーザーが見る入口」

GitHub Releases は「ユーザーがこのアプリと最初に出会う場所」になることが多い。OSS なら、訪問者はReleases を見て「最新版は何か・何ができるか」を判断する。だからリリースノートを丁寧に書くのが大事（Section 06）。タグが「git の技術的な印」なら、リリースは「ユーザーに向けた発表」。GitHub版Vol.5 の README と並ぶ、対外的な顔です。

06

リリースノートとCHANGELOG

RELEASE NOTES & CHANGELOG

リリースで重要なのが「何が変わったか」の記録。リリースノート（そのリリースの変更点）とCHANGELOG（全リリースの履歴）。Vol.GH6 のConventional Commits があれば、これらを半自動で生成できます。

変更点を記録するコミットから生成できる
// リリースノートとCHANGELOG
  • リリースノート：1つのリリースの変更点（Releases に書く）
  • CHANGELOG.md：全リリースの履歴（リポジトリに置く）
// 書き方（ユーザー視点）

技術的詳細でなく「ユーザーにとって何が変わったか」を書く。新機能・改善・修正で分類。

// Conventional Commits から生成

Vol.GH6 の規約があれば、feat→新機能、fix→修正と自動分類できる。GitHub の「Generate release notes」機能も使える。

// CHANGELOG.md（全リリース履歴） # Changelog ## [1.0.0] - 2024-XX-XX ### Added（新機能） - メモのCRUD機能（US-001） - カテゴリによる分類（US-002） - 複数タグ付け（US-003） - キーワード全文検索（US-004） - ピン留め・アーカイブ（US-005） # 将来のリリース例： ## [1.1.0] - 将来 ### Added - エクスポート/インポート機能 ### Fixed（修正） - 検索結果の並び順の不具合 // ─── Conventional Commits から生成 ─── # Vol.GH6 の規約に沿ったコミット： feat(memo): メモ作成を追加 feat(search): 全文検索を追加 fix(memo): 空タイトルエラーを修正 ↓ 自動分類 ### Added - メモ作成 - 全文検索 ### Fixed - 空タイトルエラー // → type からセクションが決まる ★★★ // ─── GitHub の自動生成機能 ─── # Releases 作成時に # 「Generate release notes」ボタン # → マージされたPRから自動でノート生成 # → PRタイトルが整っていると綺麗（Vol.GH6,7）★ // ─── 「Keep a Changelog」形式 ─── Added 新機能 Changed 既存機能の変更 Fixed バグ修正 Removed 削除された機能 // → 標準的な分類（覚えておくと便利）
POINT — 良いコミット規約が、良いリリースノートを生む

Vol.GH6 でConventional Commits（feat/fix/docs）を整えた効果が、ここで開花します。type で分類されたコミットから、リリースノートを半自動生成できる。「feat → 新機能」「fix → 修正」と並べるだけ。日々の丁寧なコミットが、リリース時の作業を楽にする。RDT版で積み上げた規律が、最後にリリースという形で報われるのです。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

リリース・バージョン管理で陥りやすい地雷を5つ。★★★ 印は特に頻発。

Vol.GH9 ハマりどころ 5選先に知れば回避できる

① タグを push し忘れる ★★★

  • 通常の push ではタグが送られない
  • 対処：git push origin v1.0.0 / --tags（Section 04）

② バージョン番号の付け方に迷う ★★★

  • 機能追加なのに PATCH を上げる等
  • 対処：feat→MINOR、fix→PATCH、破壊→MAJOR（Section 03）

③ タグとリリースを混同する

  • タグ＝git の印、リリース＝GitHub の公開物
  • 対処：タグを打ってからリリースを作る（Section 05）

④ 間違ったコミットにタグを打つ

  • 未テストの状態でタグ
  • 対処：タグを削除して打ち直し（下記）。CI緑を確認してから

⑤ リリースノートが技術的すぎる

  • 内部の変更ばかりでユーザーに伝わらない
  • 対処：「ユーザーにとって何が変わったか」を書く（Section 06）
// 主要な対処 // ─── ① タグの push ─── $ git tag -a v1.0.0 -m "..." $ git push origin v1.0.0 # 忘れずに ★★★ // ─── ④ タグの打ち直し ─── # ローカルのタグを削除 $ git tag -d v1.0.0 # リモートのタグも削除 $ git push origin :refs/tags/v1.0.0 # 正しいコミットで打ち直し $ git tag -a v1.0.0 -m "..." $ git push origin v1.0.0 // → タグは消して打ち直せる ★ // ※ ただし公開後の打ち直しは混乱の元 // なので「CI緑を確認してから打つ」が基本 // ─── ② バージョン番号の判断 ─── バグ修正だけ → PATCH（v1.0.1） 機能追加（互換） → MINOR（v1.1.0） 互換性を壊す → MAJOR（v2.0.0） // → 迷ったら「ユーザーへの影響」で考える // ─── 初回は v1.0.0、開発中は v0.x ─── # 開発途中（不安定）: v0.1.0, v0.2.0... # 最初の安定版: v1.0.0 ★ # → 本シリーズは US-001〜005 完成で v1.0.0
