01

この巻のミッション

MISSION

Vol.GH7 のゴールは2つ。① 機密情報を Git に載せない仕組みを作る（.gitignore・環境変数）、② Issue/PR テンプレートでチームの記述を統一する。「事故を防ぐ」と「品質を揃える」の両方を、仕組みで実現します。

2つのテーマ安全と統一
// テーマ① シークレット管理

DBパスワード・APIキーなどの機密情報を Git に載せない。本シリーズの application.properties には DB 接続情報があり、これを安全に扱う必要があります。

// テーマ② テンプレート

Issue や PR を作るとき、毎回ゼロから書くと記述がバラバラに。テンプレートで「受入条件を必ず書く」などを仕組み化。RDT版の品質を保ちます。

// 共通する考え方

どちらも「人の注意力でなく仕組みで」（Vol.GH6 と同じ思想）。機密の流出も記述のばらつきも、仕組みで防ぐ。

// Vol.GH7 で整備するもの // ─── ① シークレット管理 ─── memo-app/ ├ .gitignore ← 機密を除外 ├ application.properties │ （機密を直書きしない） └ application.properties.example （ダミー値の見本を共有） // → DBパスワードは環境変数で注入 // → Git には載せない ★★★ // ─── ② テンプレート ─── .github/ ├ ISSUE_TEMPLATE/ │ ├ user_story.md ← US用 │ └ bug_report.md ← バグ用 └ pull_request_template.md ← PR用 // → Issue/PR作成時に自動で雛形 ★★★ // → 「受入条件を書く」を仕組み化 // ─── 効果 ─── // 安全：機密が流出しない // 統一：誰が書いても同じ品質 // → チーム開発・公開リポジトリの土台
POINT — 機密流出は「取り返しがつかない」事故

機密情報の流出は、他の事故と違って取り返しがつきません。一度 GitHub に push したパスワードは履歴に永久に残り、削除しても誰かが見た・コピーした可能性が消えない。だから「流出してから対処」では遅く、「最初から載せない」が唯一の正解。Vol.GH7 は予防がすべてです。

02

機密情報が流出する事故

SECRET LEAK

まず「どうやって流出するか」を知ります。多くはうっかり。DBパスワードを書いた設定ファイルを、何気なく git add . して push——これだけで世界中に公開されます。仕組みを知れば防げます。

流出のパターンと被害うっかりが大半
// よくある流出パターン
  • application.properties に本番DBパスワードを直書き→ push
  • APIキーをコードに埋め込み→ push
  • git add . で意図せず機密ファイルを含める
  • 公開リポジトリで世界中に公開
// 被害
  • DBへ不正アクセス
  • APIキーの不正利用（高額請求）
  • 履歴に残り消せない（削除しても痕跡）
// GitHub の検知

GitHub にはシークレットスキャンがあり、APIキー等を push すると警告が出ることも。でも頼り切らず、最初から載せないのが基本。

// 流出が起きる瞬間 // ─── 危険な application.properties ─── spring.datasource.url= jdbc:mysql://localhost:3306/memo_app spring.datasource.username=root spring.datasource.password=MyR3alP@ss ↑ 本物のパスワード ★ // ─── うっかり push ─── $ git add . # 全部追加（危険） $ git commit -m "設定追加" $ git push // → パスワードが GitHub に公開 ★★★ // ─── GitHub の警告 ───
⚠ Secret scanning alert Possible database password detected in application.properties
// ─── 履歴からは消えない ─── // 後でパスワードを消してコミットしても… $ git log -p # 過去の履歴を見ると - password=MyR3alP@ss ← 履歴に残る！ // → 一度 push したら漏れたと考える ★★★ // → パスワードは即変更が必要 // ─── だから「最初から載せない」─── // .gitignore で除外（Section 03） // 環境変数で注入（Section 04）
POINT — git add . は便利だが危険、中身を確認する習慣を

機密流出の多くは git add .（全部追加）が原因。便利ですが、意図しないファイルまで含めてしまう。対策は①.gitignore で機密を除外（Section 03）、②git status でadd する前に中身を確認する習慣。「何をコミットするか」を意識する。Vol.GH6 の「1コミット1目的」とも通じる、コミットを丁寧にの精神です。

03

.gitignore の設計

GITIGNORE

.gitignore は「Git に追跡させないファイル」のリスト。機密情報・ビルド成果物・IDE設定などを除外します。Spring Boot プロジェクトでは定番の .gitignore があります。Vol.GH1 で作った簡易版を本格化します。

.gitignore に書くもの機密・成果物・IDE
// 除外すべき3カテゴリ
CAT 1
機密情報
パスワード・APIキー・.env など。絶対に載せない。
CAT 2
ビルド成果物
target/ build/ *.class。ソースから再生成可能。
CAT 3
IDE設定
.idea/ .vscode/。個人環境依存。
// 記法
  • 1行1パターン
  • / 末尾はディレクトリ
  • * はワイルドカード
  • # はコメント
// .gitignore（Spring Boot版） # ===== 機密情報（最重要）===== application-local.properties application-secret.properties .env *.key *.pem # → DBパスワード・APIキーを含むファイル ★★★ # ===== ビルド成果物 ===== target/ build/ *.class *.jar !*.gradle/wrapper/*.jar # → ソースから再生成できる # ===== IDE 設定 ===== .idea/ *.iml .vscode/ .settings/ .classpath .project # → 個人環境に依存 # ===== OS 固有 ===== .DS_Store Thumbs.db // ─── 動作確認 ─── $ git status Untracked files: README.md src/... # application-local.properties は # 表示されない（無視されている）★ // ─── gitignore.io が便利 ─── # https://www.toptal.com/developers # /gitignore # 「Java, Spring, IntelliJ」等を指定すると # 定番の .gitignore を生成してくれる ★
POINT — .gitignore は「最初に」作る

.gitignore はプロジェクトの最初（Vol.GH1）に作るのが鉄則。後から追加すると、既にコミット済みのファイルは追跡され続ける（Section 07 のハマりどころ）。本シリーズは Vol.GH1 で簡易版を作りました。「機密を含むファイルを作る前に、それを .gitignore に書く」順序が安全。gitignore.io のような定番テンプレート生成サービスを使うと漏れがありません。

04

機密を外に出す（環境変数）

ENVIRONMENT VARIABLES

機密を .gitignore で除外するだけでなく、そもそもコードに書かない。環境変数で外から注入します。Spring Boot は環境変数を自然に読めるので、「設定の見本だけ Git に、実際の値は環境変数で」が定石です。

環境変数で機密を注入見本と実値を分ける
// 考え方

機密をコードや設定ファイルに直書きせず、環境変数から読む。Git には「見本（ダミー値）」だけ載せ、実際の値は各自の環境変数で。

// 2つのファイル
  • application.properties：環境変数を参照する形（Gitに載せる）
  • .properties.example：見本（ダミー値・Gitに載せる）
// メリット
  • 機密が Git に載らない
  • 環境ごとに値を変えられる（開発/本番）
  • 新メンバーは見本を見て設定できる
// 環境変数による機密管理 // ─── application.properties（Gitに載せる）─── spring.datasource.url= ${DB_URL:jdbc:mysql://localhost:3306/memo_app} spring.datasource.username= ${DB_USER:root} spring.datasource.password= ${DB_PASSWORD} // ↑ 環境変数 DB_PASSWORD から読む // ${名前:デフォルト} で既定値も指定可 ★ // ─── 環境変数を設定（各自の環境）─── # Mac/Linux $ export DB_PASSWORD="MyR3alP@ss" # Windows (PowerShell) > $env:DB_PASSWORD="MyR3alP@ss" // → 実際のパスワードは環境変数に // → Git には載らない ★★★ // ─── application.properties.example ─── # 見本（Gitに載せる・ダミー値） # 各自 DB_PASSWORD 環境変数を設定すること spring.datasource.password=${DB_PASSWORD} # 例: export DB_PASSWORD="your_password" // ─── 新メンバーの手順 ─── 1. .example を見て必要な環境変数を知る 2. 自分の環境変数を設定 3. アプリ起動 // → 機密を共有せず、各自で設定 ★★★ // → READMEにも手順を記載（GitHub版Vol.5）
POINT — 「設定の構造」は共有、「秘密の値」は各自

環境変数方式の本質は「設定の構造（どんな項目が要るか）は共有し、秘密の値だけ各自で持つ」こと。.example ファイルで「DB_PASSWORD が必要」と伝えつつ、実際の値は載せない。新メンバーは見本を見て自分の値を設定。これは GitHub版Vol.5 のREADME のセットアップ手順とセットで機能します。「動かし方は教える、秘密は渡さない」のが安全な共有です。

05

Issue テンプレート

ISSUE TEMPLATE

テーマ②に移ります。Issue テンプレートは、Issue 作成時に自動で雛形を表示する仕組み。RDT版では「ユーザーストーリー用」「バグ報告用」を用意し、受入条件を必ず書くことを仕組み化します。

Issue テンプレートで記述を統一.github/ISSUE_TEMPLATE/
// テンプレートの効果
  • Issue 作成時に雛形が自動表示
  • 「受入条件を書く」を忘れない
  • 誰が書いても同じ構造
// 置き場所

.github/ISSUE_TEMPLATE/ に Markdown ファイルを置く。複数置くと「どのテンプレートを使うか」選択肢が出る。

// 本シリーズの2種
  • user_story.md：US用（WHO/WHAT/WHY＋受入条件）
  • bug_report.md：バグ用（再現手順＋期待/実際）
// Issue テンプレート（RDT版） // ─── .github/ISSUE_TEMPLATE/user_story.md ─── --- name: ユーザーストーリー about: 新機能の要求（US） labels: user-story --- ## ユーザーストーリー - **As a** （誰が） - **I want** （何を） - **so that** （なぜ） ## 受入条件 - [ ] AC1: - [ ] AC2: - [ ] AC3: ## 優先度（MoSCoW） - [ ] Must / Should / Could / Won't // → 別冊S1の要求工学がそのまま雛形に ★★★ // ─── .github/ISSUE_TEMPLATE/bug_report.md ─── --- name: バグ報告 about: 不具合の報告 labels: bug --- ## 概要 （何が起きているか） ## 再現手順 1. 2. ## 期待する動作 ## 実際の動作 ## 環境 - OS / ブラウザ / バージョン // → バグ報告に必要な情報を漏れなく ★ // → 別冊S3「バグ修正は再現テストから」に繋がる
POINT — テンプレートが「RDTの型」を強制する

Issue テンプレートの真価は「RDT版の型を仕組みで強制」すること。ユーザーストーリーの Issue を作れば、必ず WHO/WHAT/WHY と受入条件の欄が出る。「受入条件を書き忘れる」がなくなる。これは別冊S1の要求工学を、運用に落とし込んだもの。良い習慣を、テンプレートで自動化する。人の記憶に頼らない品質維持です。

PR テンプレート

PR TEMPLATE

PR テンプレートは、PR 作成時に本文の雛形を自動表示。Vol.GH3 で手書きした「受入条件チェックリスト＋Closes」を、テンプレート化して毎回自動に。RDT版のPRの型を仕組みにします。

PR テンプレートで品質を揃える.github/pull_request_template.md
// 置き場所

.github/pull_request_template.md（1ファイル）。PR 作成時に本文に自動挿入される。

// 含める項目（Vol.GH3の型）
  • 概要：どのUSを実装したか
  • 受入条件チェックリスト
  • テスト結果
  • Closes #（Issue紐付け）
// 効果

Vol.GH3 で手書きしていた内容が毎回自動に。「受入条件チェックを忘れる」「Closesを書き忘れる」がなくなる。

// PR テンプレート（RDT版） // ─── .github/pull_request_template.md ─── ## 概要 （どのUS/Issueを実装したか） ## 受入条件 - [ ] AC1: - [ ] AC2: - [ ] AC3: ## テスト - [ ] 単体テスト追加 - [ ] 結合テスト追加 - [ ] 全テスト green ## セルフレビュー - [ ] Files changed を確認した - [ ] 設計原則を満たす（SRP/命名） - [ ] 機密情報を含まない ## 関連 Issue Closes # // ─── 効果：PR作成画面 ───
Open a pull request タイトル: feat(search): 検索を実装 本文（テンプレートが自動挿入）: ## 概要 US-004 を実装... ## 受入条件 - [x] AC1: キーワード検索 - [x] AC2: カテゴリ絞り込み ... ## セルフレビュー - [x] 機密情報を含まない ★ Closes #4
// → Vol.GH3 の手書きが自動に ★★★ // → 受入条件・Closes・機密チェックを // 毎回忘れず実施できる // ─── 「機密チェック」を入れる意味 ─── // PR時に「機密を含まないか」を // チェック項目に → 流出の最終防壁 ★
POINT — PRテンプレートに「機密チェック」を入れる

PR テンプレートに「機密情報を含まない」のチェック項目を入れると、マージ前の最終防壁になります。Section 02 の流出事故を、「PR で必ず確認する」仕組みで防ぐ。これは Vol.GH3 のセルフレビュー＋Vol.GH7 のシークレット管理の合わせ技。テンプレートは「チェックすべきこと」を毎回思い出させる仕組み。RDT版の品質ゲートが、また一つ強化されます。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

シークレット管理・テンプレートで陥りやすい地雷を5つ。★★★ 印は特に重大。機密関連は事故が深刻なので要注意。

Vol.GH7 ハマりどころ 5選機密事故は深刻

① .gitignore 追加前にコミット済み ★★★

  • 既に追跡中のファイルは .gitignore で消えない
  • 対処：git rm --cached ファイル で追跡解除

② push 済み機密を「消せばOK」と思う ★★★

  • 履歴に残る＋既に見られた可能性
  • 対処：パスワード・キーを即変更（無効化）。これが最優先

③ テンプレートが表示されない

  • 置き場所・ファイル名の誤り
  • 対処：.github/ISSUE_TEMPLATE/、.github/pull_request_template.md を正確に

④ YAML フロントマターの崩れ

  • Issue テンプレートの --- 部分のインデントミス
  • 対処：name/about/labels の記法を守る

⑤ .example を更新し忘れる

  • 設定項目が増えたのに見本が古い
  • 対処：設定追加時に .example も更新（Doc as Code）
// 主要な対処 // ─── ① 既にコミット済みの除外 ─── # .gitignore に追加しただけでは消えない $ git rm --cached application-local.properties # → Git の追跡から外す（ローカルは残る） $ git commit -m "chore: 機密ファイルを追跡から除外" // → 以降は .gitignore で無視される ★ // ─── ② push 済み機密の対応（最重要）─── # もし機密を push してしまったら： 1. パスワード/キーを即変更（無効化） ★★★ # これが最優先。履歴削除より先 2. .gitignore に追加＋追跡解除 3. 必要なら履歴クリーン（高度） # 「消したから大丈夫」は通用しない # → 漏れた前提で、値そのものを変える ★★★ // ─── テンプレート配置の確認 ─── .github/ ├ ISSUE_TEMPLATE/ │ ├ user_story.md ✓ │ └ bug_report.md ✓ └ pull_request_template.md ✓ # → 正確なパス・ファイル名が必須 ★
