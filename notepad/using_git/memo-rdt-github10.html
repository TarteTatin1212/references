01

この巻のミッション

MISSION

Vol.GH10 のゴールは「リリース後の保守を仕組み化する」こと。v1.0.0 を出したら終わりではありません。依存ライブラリの脆弱性対応・更新は、アプリを安全に保ち続けるために不可欠。これをDependabot で自動化します。

「作って終わり」ではない保守という仕事
// リリース後に起きること

v1.0.0 を出した後も、世界は動きます：

  • 使っているライブラリに脆弱性が発見される
  • ライブラリの新バージョンが出る
  • 放置するとセキュリティリスクが蓄積
// 保守の難しさ

脆弱性情報を人が常に追うのは無理。「気づいたら古くて危険なライブラリだらけ」になりがち。仕組みで自動化が必要。

// Dependabot

GitHub の機能。依存ライブラリを監視し、更新や脆弱性を自動でPRで知らせる。「人が追わなくても済む」仕組み。

// 保守という継続的な仕事 // ─── 開発のライフサイクル ─── 要求 → 設計 → 実装 → テスト → リリース（v1.0.0・Vol.GH9） → 保守 ← ここがVol.GH10 ★★★ ↓ 新しい要求 → また開発… // → リリースは終わりでなく通過点 // ─── 放置されたプロジェクトの末路 ─── 1年放置すると… - ライブラリが何十個も古い - 既知の脆弱性が放置されている - いざ更新しようにも差が大きすぎて困難 // → 「塵も積もれば」の負債 ★ // ─── Dependabot による自動化 ─── Dependabot が依存を監視 ↓ 更新/脆弱性を検知 ↓ 自動で更新PRを作成 ↓ CI（Vol.GH8）が通れば安全に確認 ↓ マージ // → 人が追わなくても保守が回る ★★★ // ─── 本シリーズの依存（保守対象）─── Spring Boot, MyBatis, MySQL Driver, JUnit, Mockito, AssertJ, Jackson（Vol.11で追加）... // → これらを最新・安全に保つ
POINT — 「リリースは通過点、保守は継続」

初学者は「リリースしたら完成」と思いがちですが、実際はリリースは通過点。アプリが使われ続ける限り、依存ライブラリの更新・脆弱性対応という保守が続きます。これを怠ると、「動いてはいるが危険なアプリ」になる。Vol.GH7 の機密管理と同じくセキュリティの継続。Dependabot で「保守を仕組みに組み込む」のが、長く使われるアプリの条件です。

02

依存関係とは

DEPENDENCIES

依存関係（dependencies）とは、アプリが使っている外部ライブラリのこと。Spring Boot、MyBatis、Jackson——これらは他人が作ったコード。便利ですが、その品質・安全性に依存することになります。

依存関係の仕組み他人のコードを使う
// 依存関係とは

アプリは多くのライブラリの上に成り立つ。pom.xml/build.gradle に書いた依存が、アプリの土台。

// 直接依存と推移的依存
  • 直接依存：自分が明示的に追加（Spring Boot等）
  • 推移的依存：依存が依存しているもの（自動で入る）

1つのライブラリが数十個の推移的依存を連れてくることも。全部が保守対象。

// 依存の二面性
  • メリット：車輪の再発明をしない（別冊S2のKISS）
  • リスク：他人のコードの問題を抱え込む
// 依存関係の構造 // ─── pom.xml の依存（直接依存）─── <dependencies> <dependency> spring-boot-starter-web </dependency> <dependency> mybatis-spring-boot-starter </dependency> <dependency> jackson-databind // Vol.11で追加 </dependency> </dependencies> // → 自分で追加した依存（直接依存） // ─── 推移的依存（自動で入る）─── spring-boot-starter-web ├ spring-web ├ spring-webmvc ├ tomcat-embed-core ├ jackson-databind └ ... （数十個） // → 1個追加すると芋づる式に ★ // → 全部が「保守対象」になる ★★★ // ─── 依存ツリーを見る ─── $ mvn dependency:tree com.example:memo-app ├ spring-boot-starter-web │ ├ spring-web:6.x │ ├ tomcat-embed-core:10.x │ └ ... ├ mybatis-spring-boot-starter └ ... // → 何に依存しているか全体が見える // ─── 依存の二面性（別冊S2）─── // メリット：自分で書かなくていい（KISS） // リスク：他人のコードの脆弱性を抱える // → だから「監視と更新」が必要 ★★★
POINT — 「使う依存」も自分の責任範囲

ライブラリを使うのは賢い選択（別冊S2：車輪の再発明をしない）。でも「使ったライブラリの問題は、自分のアプリの問題」になる。Jackson に脆弱性が見つかれば、Jackson を使う自分のアプリも危険。「他人のコードだから関係ない」とは言えない。だから依存を把握し、更新を続けるのが使う側の責任。便利さの代償として、保守がついてくるのです。

03

脆弱性とアップデート

VULNERABILITY & UPDATE

ライブラリには脆弱性（セキュリティ上の欠陥）が見つかることがあります。発見されると修正版がリリースされる。古いまま放置すると攻撃の的に。更新の種類とタイミングを理解します。

脆弱性と更新の判断SemVerと連動
// 脆弱性とは

ライブラリのセキュリティ上の欠陥。攻撃に悪用される。発見されるとCVE（脆弱性ID）が付き、修正版が出る。

// 更新の種類（SemVer・Vol.GH9）
更新リスク対応
PATCH（x.x.1）低すぐ更新
MINOR（x.1.x）中テスト後
MAJOR（1.x.x）高慎重に
// 判断の基準
  • 脆弱性修正：優先的に更新
  • PATCH更新：安心して更新（バグ修正のみ）
  • MAJOR更新：互換性破壊→慎重にテスト
// 脆弱性とアップデート // ─── 脆弱性の例 ───
⚠ Dependabot security alert jackson-databind 2.x.x has a known vulnerability (CVE-XXXX-XXXX) Severity: High Fixed in: 2.x.y
// → 「使っているライブラリが危険」と通知 // → 修正版に更新すべき ★★★ // ─── SemVer と更新リスク（Vol.GH9）─── 2.15.2 → 2.15.3（PATCH） バグ/脆弱性修正のみ → 安心して更新 ★ 2.15.2 → 2.16.0（MINOR） 機能追加（互換あり） → テスト通れば更新 2.15.2 → 3.0.0（MAJOR） 互換性破壊 → コード修正が必要かも → 慎重にテスト ★★★ // ─── 更新の判断フロー ─── 脆弱性あり？ → Yes: 優先的に更新（セキュリティ） PATCH/MINOR？ → CIが通れば更新（Vol.GH8） MAJOR？ → 変更内容を確認・慎重にテスト // ─── CI が安全網になる（Vol.GH8）─── // 更新してもテストが緑なら // 「壊れていない」と判断できる ★★★ // → 別冊S3：テストは変更の安全網 // → 依存更新でもテストが守る
POINT — 依存更新の安全性は「CI（テスト）」が保証する

依存ライブラリの更新は「壊れないか不安」になりがち。でも Vol.GH8 で作ったCI（自動テスト）があれば安心。更新PRに対してCIが走り、テストが緑なら「壊れていない」と判断できる。これは別冊S3の「テストは変更の安全網」そのもの。日々のテスト（RDT版の核心）が、依存更新という保守作業も支える。テストへの投資が、保守の場面でも報われます。

04

Dependabot の設定

DEPENDABOT SETUP

Dependabot を有効にします。① 脆弱性アラート（GitHub が自動で警告）、② バージョン更新（.github/dependabot.yml で設定）。設定すると更新を自動でPRにしてくれる。

Dependabot の2つの機能アラートと自動更新
// ① セキュリティアラート

GitHub が自動で脆弱性を検知して警告。設定不要（リポジトリ設定でON）。脆弱性が見つかると更新PRを自動作成。

// ② バージョン更新

.github/dependabot.yml を置くと、定期的に依存をチェックし、新しいバージョンがあれば更新PRを作成。

// 設定手順
  1. Settings → Security → Dependabot を有効化
  2. .github/dependabot.yml を作成
  3. パッケージエコシステム（maven/gradle）を指定
  4. 更新頻度（weekly等）を設定
// .github/dependabot.yml version: 2 updates: # Maven の依存を監視 - package-ecosystem: "maven" directory: "/" schedule: interval: "weekly" # 毎週チェック # 一度に来るPR数を制限 open-pull-requests-limit: 5 # PRにラベルを付ける（Vol.GH4） labels: - "dependencies" # GitHub Actions 自体も更新対象（Vol.GH8） - package-ecosystem: "github-actions" directory: "/" schedule: interval: "weekly" // ─── 設定後の動き ─── 毎週月曜（例）： Dependabot が依存をチェック ↓ 新バージョンあり 更新PRを自動作成 ↓ CI（Vol.GH8）が自動実行 ↓ テスト緑 あなたが確認してマージ ★ // ─── セキュリティアラートの画面 ───
Security / Dependabot alerts ⚠ 2 open - jackson-databind: High → Dependabot がPR作成済み - (another): Moderate
// → 脆弱性を見逃さない仕組み ★★★
POINT — 「open-pull-requests-limit」で PR の洪水を防ぐ

Dependabot を有効にすると、最初は大量の更新PRが来て圧倒されがち。open-pull-requests-limit で同時に開くPR数を制限すると、少しずつ対応できる。また schedule: weekly で頻度を抑える。「全部一気に」ではなく「無理なく続けられるペース」に設定するのがコツ。保守は継続が大事なので、自分が対応しきれる量に調整します。

05

更新PRへの対応

HANDLING UPDATE PRS

Dependabot が作る更新PRへの対応方法。これまで学んだPR・CI・レビューがそのまま活きます。CIが緑かを確認し、マージ。RDT版で積み上げた仕組みが、保守でも回ります。

更新PRのレビューとマージCIが判断材料
// 更新PRの中身

Dependabot のPRには変更内容・リリースノート・互換性情報が自動で書かれる。「何が変わるか」が分かる。

// 対応の判断
  1. PRの内容（更新の種類・SemVer）を確認
  2. CIが緑かを確認（Vol.GH8）
  3. PATCH/MINORでCI緑 → マージ
  4. MAJOR → 変更内容を慎重に確認
// これまでの仕組みが活きる
  • PR（Vol.GH3）：更新もPR経由
  • CI（Vol.GH8）：自動テストで安全確認
  • ブランチ保護（Vol.GH6）：main を守る
// 更新PRへの対応 // ─── Dependabot のPR画面 ───
Pull requests ⬆ Bump jackson-databind from 2.15.2 to 2.15.3 by dependabot PATCH update（バグ/脆弱性修正） ✓ CI / build (passed) ✓ CI / test (passed) Release notes: - セキュリティ修正 CVE-XXXX
// ─── 対応フロー ─── PATCH更新（2.15.2→2.15.3）： CI緑 → 安心してマージ ★ （バグ/脆弱性修正のみ） MINOR更新（2.15→2.16）： CI緑 → マージ （機能追加・互換あり） MAJOR更新（2.x→3.x）： ⚠ 変更内容を確認 CI緑でも互換性破壊の可能性 → リリースノートを読む → 必要ならコード修正 ★★★ // ─── これまでの仕組みが活きる ─── 更新PR → CI（Vol.GH8）が自動テスト → ブランチ保護（Vol.GH6）でmain守る → セルフレビュー（Vol.GH3）で確認 → マージ // → 保守も同じワークフロー ★★★ // → 積み上げた仕組みが報われる // ─── まとめてマージも可能 ─── # Dependabot にコメントで指示 @dependabot merge @dependabot rebase # → CI通過後に自動マージ等もできる
POINT — 更新PRこそ「CIの真価」が発揮される

依存更新は「テストがあって本当に良かった」と実感する場面。「Jackson を 2.15.2→2.15.3 に上げて大丈夫?」——手動で全機能を確認するのは無理。でもCI（Vol.GH8）が全テストを自動実行し、緑なら「壊れていない」と即判断できる。RDT版でテストを積み上げてきたからこそ、依存更新を恐れず続けられる。テストは「今を守る」だけでなく「未来の保守を支える」投資なのです。

06

保守を続ける文化

MAINTENANCE CULTURE

保守は「仕組み」だけでなく「習慣」。Dependabot を入れても、PRを放置したら意味がない。少しずつ・定期的に対応する文化が、アプリを長く健全に保ちます。RDT版の集大成として、開発の全体像を振り返ります。

保守を習慣にする小さく継続
// 保守を続けるコツ
  • 定期的に：週1回など、更新PRを見る時間を作る
  • 小さく：溜めずにこまめに対応
  • CIを信じる：緑なら安心してマージ
  • 放置しない：放置が最大の負債
// RDT版の開発サイクル全体

GitHub版で学んだ流れ：

  • 要求（Issue）→ ブランチ → PR → CI → マージ → リリース → 保守
  • そして保守から新しい要求へ循環
// 開発は円環

リリースで終わりでなく、保守を経て次の開発へ。要求工学（別冊S1）の「要求は進化する」が、運用でも回り続ける。

// 保守を含む開発の全体像 // ─── GitHub版で築いた開発サイクル ─── ┌──────────────────────────┐ │ │ 要求(Issue) │ ↓ Vol.GH4 │ ブランチ │ ↓ Vol.GH2 │ 実装＋コミット │ ↓ Vol.GH6（規約） │ PR │ ↓ Vol.GH3 │ CI（自動テスト） │ ↓ Vol.GH8 │ レビュー＋マージ │ ↓ Vol.GH6（保護） │ リリース │ ↓ Vol.GH9 │ 保守（依存更新） │ ↓ Vol.GH10 │ 新しい要求 ────────────────────┘ // → 円環。開発は続く ★★★ // ─── 保守の習慣 ─── ✓ 週1回、更新PRを確認する時間 ✓ 溜めずにこまめに対応 ✓ CI緑なら安心してマージ ✓ 脆弱性アラートは優先対応 // → 「小さく継続」が健全さを保つ ★ // ─── RDT版 GitHub編で学んだこと ─── // 要求の追跡（トレーサビリティ） // 品質を仕組みで守る（規約・保護・CI） // 機密を守る（シークレット管理） // 成果を世に出す（リリース） // 安全を保ち続ける（保守）★★★ // → 開発の全工程を一周した
POINT — 「小さく継続」が放置の負債を防ぐ

保守の敵は「放置」。「後でまとめて」と溜めると、更新の差が大きくなり、対応が困難に。「Spring Boot を一気に2世代上げる」のは大変ですが、こまめに1段ずつなら楽。Dependabot＋CI があれば、週に数分の確認で保守が回る。「小さく、継続的に」——これは RDT版全体を貫く「1機能ずつ確実に」の精神と同じ。健全なアプリは、地道な保守から生まれます。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

依存管理・保守で陥りやすい地雷を5つ。★★★ 印は特に頻発。

Vol.GH10 ハマりどころ 5選先に知れば回避できる

① 更新PRを放置する ★★★

  • 溜まって差が大きくなり対応困難に
  • 対処：定期的に・小さく対応（Section 06）

② MAJOR更新を無確認でマージ ★★★

  • 互換性破壊でアプリが壊れる
  • 対処：MAJORは変更内容を確認・テスト（Section 03,05）

③ dependabot.yml の YAML ミス

  • インデント崩れで設定が効かない
  • 対処：YAML のインデントを正確に（Section 04）

④ テストがないと更新の安全が判断できない

  • CIがないと「壊れたか」分からない
  • 対処：CI（Vol.GH8）が前提（Section 03,05）

⑤ 更新でテストが落ちて放置

  • 落ちたPRをそのまま放置
  • 対処：落ちたら原因を確認、修正してから
// 主要な対処 // ─── ② MAJOR更新の確認 ─── 2.x → 3.x（MAJOR）の更新PR： - CIが緑でも油断しない - リリースノートで破壊的変更を確認 - 必要ならコード修正 - 別ブランチで十分テストしてから // → MAJORは「慎重に」が鉄則 ★★★ // ─── ⑤ 落ちたPRへの対応 ─── 更新PRでCIが赤： → 更新で何かが壊れた → ログを見て原因特定 → コード修正 or 更新を見送り // → 赤を放置しない（別冊S7：握りつぶさない）★ // ─── ③ YAML の確認 ─── NG: インデントがズレている OK: version: 2 updates: - package-ecosystem: "maven" directory: "/" schedule: interval: "weekly" // → スペース2つのインデントを揃える ★ // ─── 保守の鉄則 ─── ✓ こまめに対応（放置しない） ✓ MAJORは慎重に ✓ CIを安全網にする ✓ 落ちたら原因究明 ✓ 脆弱性は優先 ★★★
