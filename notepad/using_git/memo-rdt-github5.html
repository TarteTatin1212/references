01

この巻のミッション

MISSION

Vol.GH5 のゴールは「リポジトリを他人が読める状態にする」。コードと Issue は揃いましたが、初見の人が全体像を掴めるドキュメントがまだありません。README とその周辺ドキュメントで、「何のアプリか・どう動かすか・どう作られているか」を伝えます。

「動くコード」だけでは不十分伝わるリポジトリへ
// これまで揃ったもの・足りないもの

揃ったもの（Vol.GH1〜4）：

  • コード（履歴付き）
  • Issue（要求の管理）
  • PR（変更の記録）

足りないもの：

  • 初見の人向けの全体説明（README）
  • 要求・設計・テストの方針ドキュメント
  • 動かし方の手順書
// Doc as Code

ドキュメントもコードと同じリポジトリで管理。Markdown で書き、Git で履歴管理。コードと一緒に育てます。

// Vol.GH5 で作るドキュメント構成 memo-app/ ├ README.md ← プロジェクトの顔 ├ docs/ │ ├ requirements.md ← 要求（US一覧） │ ├ design.md ← 設計方針 │ ├ testing.md ← テスト方針 │ └ screens.md ← 画面設計（特別編） ├ src/ └ pom.xml // ─── 各ドキュメントの役割 ─── README.md: 「何のアプリ？どう動かす？」 → 初見の人が最初に読む requirements.md: 「何を作るか」（US-001〜004） → Issue と対応 design.md: 「どう作るか」（3層・テーブル） → 別冊S2,S4 の方針 testing.md: 「どうテストするか」 → 別冊S3 のテスト戦略 // → RDT版で作ってきた // 要求・設計・テストを文章化 ★★★
POINT — ドキュメントは「未来の他人」のために書く

ドキュメントの読者は「3か月後の自分」「引き継ぎ相手」「採用面接でリポジトリを見る人」。コードは「どう動くか」を語りますが、「なぜそう作ったか・どう動かすか」はドキュメントにしかない。GitHub版Vol.3 のトレーサビリティが「未来への手紙」だったように、README も未来への手紙です。

02

なぜドキュメントが要るか

WHY DOCUMENT

「コードを読めば分かる」は半分正しく半分間違い。コードはHow（どう動くか）を語りますが、What（何を）・Why（なぜ）・How to run（どう動かすか）はドキュメントが担います。要求工学（別冊S1）の WHY とも繋がります。

コードとドキュメントの分担語れることが違う
問い答えるもの
どう動くか（How）コード
何を作るか（What）ドキュメント
なぜ作るか（Why）ドキュメント
どう動かすか（Run）README
なぜこの設計かdesign.md
// ドキュメントがないと起きること
  • 新メンバーが環境構築で1日溶かす
  • 「なぜこの設計？」が誰も分からない
  • 同じ質問が何度も繰り返される
  • リポジトリを見た人が使い方が分からず去る
// ドキュメントが答える3つの問い // ─── What（何を）─── // requirements.md / README の概要 「Spring Boot + MyBatis のメモ帳アプリ。 メモをカテゴリ・タグで整理できる」 // ─── Why（なぜ）─── // requirements.md の WHY（別冊S1） 「思いついたアイデアを忘れる前に すぐ記録し、後で整理したい」 // → 要求工学のWHYがそのまま // ─── How to run（どう動かす）─── // README のセットアップ手順 1. MySQLでDB作成 2. application.properties設定 3. mvn spring-boot:run 4. localhost:8080/memos // ─── コードが答える How（どう動く）─── // → ソースを読めば分かる // → でも上の3つはコードにない ★★★ // → だからドキュメントが必要
POINT — 「動かし方」が書いていないリポジトリは使われない

OSS でもよくあるのが「README にセットアップ手順がない」問題。どんなに良いコードでも、動かし方が分からなければ誰も使えません。最低限「① 必要な環境 ② セットアップ手順 ③ 起動方法」を書く。これだけでリポジトリの価値が跳ね上がる。本シリーズの README にも必ず入れます。

03

README の構成

README STRUCTURE

README はリポジトリの「顔」。GitHub でリポジトリを開くと最初に表示されます。定番の構成に沿って書けば、初見の人が迷いません。本シリーズ用の構成を設計します。

README の定番セクション上から読める順に
// README の構成（上から）
  1. タイトル＋概要：何のアプリか1〜2文
  2. バッジ：技術スタック・ビルド状態（任意）
  3. 機能：何ができるか（US ベース）
  4. 技術スタック：使用技術
  5. セットアップ：環境構築・起動手順
  6. テーブル構成：ER の概要
  7. ドキュメントへのリンク：docs/ へ
  8. 開発方針：RDT（要求→設計→テスト）
// 順番の意図

「概要→使い方→詳細」の順。読者は上から読むので、最初に全体像、次に動かし方を置きます。

// README のプレビューイメージ
github.com/yourname/memo-app
📝 memo-app
Spring Boot + MyBatis のメモ帳アプリ。 メモをカテゴリ・タグで整理できる。 Java 17 Spring Boot 3.x MyBatis MySQL 8.0
✨ 機能
- メモのCRUD（US-001） - カテゴリ分類（US-002） - 複数タグ付け（US-003） - 検索（US-004・予定）
🚀 セットアップ
1. DB作成 ... 2. 設定 ... 3. 起動 ...
📚 ドキュメント
- 要求 → docs/requirements.md - 設計 → docs/design.md - テスト → docs/testing.md
// → 初見の人が上から読んで // 全体像→使い方→詳細 と辿れる ★★★
POINT — README の冒頭1文で「何のアプリか」を即答する

README で最も重要なのが冒頭の1〜2文。「Spring Boot + MyBatis のメモ帳アプリ」のように、「何のアプリで、何の技術か」を即答する。長い前置きは不要。読者は3秒で「自分に関係あるか」を判断します。要求工学の WHO/WHAT（別冊S1）が、ここで1文に凝縮されます。

04

README を書く

WRITE README

実際に README.md を書きます。Markdown で記述し、コミット。セットアップ手順は「コピペで動く」レベルの具体性が鉄則です。Vol.GH1 で作った簡易版を、本格的な README に育てます。

README.md の本文コピペで動く具体性
// 書く手順
  1. リポジトリ直下の README.md を開く（Vol.GH1で作成済み）
  2. 定番構成（Section 03）に沿って加筆
  3. セットアップは実際のコマンドを書く
  4. docs/ へのリンクを張る
  5. コミット（docs: READMEを充実）
// セットアップの具体性

「DBを用意」ではなく実際のSQL・コマンドを書く。読者がコピペで動かせるのが理想。

// README.md（抜粋） # 📝 memo-app Spring Boot + MyBatis のメモ帳アプリ。 メモをカテゴリ・タグで整理できます。 要求→設計→テスト駆動（RDT）で開発。 ## ✨ 機能 - メモの作成・編集・削除（US-001） - カテゴリでの分類・絞り込み（US-002） - 複数タグでの整理（US-003） - キーワード検索（US-004・実装予定） ## 🛠 技術スタック | 分類 | 技術 | |------|------| | 言語 | Java 17 | | FW | Spring Boot 3.x | | ORM | MyBatis | | DB | MySQL 8.0 | | テスト | JUnit5 / Mockito / AssertJ | ## 🚀 セットアップ ### 1. データベース作成 ```sql CREATE DATABASE memo_app CHARACTER SET utf8mb4; ``` ### 2. 設定 `application.properties` に接続情報： ``` spring.datasource.url= jdbc:mysql://localhost:3306/memo_app spring.datasource.username=root spring.datasource.password=（各自） ``` ### 3. 起動 ```bash ./mvnw spring-boot:run ``` ブラウザで http://localhost:8080/memos ## 📚 ドキュメント - [要求定義](docs/requirements.md) - [設計方針](docs/design.md) - [テスト方針](docs/testing.md) - [画面設計](docs/screens.md) ## 🔄 開発方針（RDT） 要求(US) → 設計 → 実装 → テスト の順で1機能ずつ実装。 要求は Issue、変更は PR で管理。
POINT — コマンドは「そのまま実行できる」形で書く

セットアップ手順はコードブロック（```）で囲み、コピペで動く形に。「MySQLでデータベースを作ってください」よりCREATE DATABASE memo_app; と具体的に書く。読者の「で、具体的に何を打てば？」を先回り。Markdown のコードブロックはコピーボタンも付くので、読者が楽です。

05

設計ドキュメント（docs/）

DESIGN DOCS

README から分離した詳細ドキュメントを docs/ に置きます。RDT版で作ってきた要求・設計・テスト方針を文章化。別冊（S1〜S5）の理論が、ここでこのプロジェクト固有の方針として結実します。

docs/ の3大ドキュメント要求・設計・テスト
// requirements.md（要求）
  • WHO/WHAT/WHY（別冊S1）
  • US-001〜004 の一覧と受入条件
  • MoSCoW 優先度
  • → Issue と対応
// design.md（設計）
  • レイヤード3層（別冊S2）
  • テーブル構成・ER図（別冊S4）
  • 主要な設計判断（削除戦略など）
// testing.md（テスト）
  • テストピラミッド方針（別冊S3）
  • 単体/結合の使い分け
  • H2 の使用方針
// docs/design.md（抜粋） # 設計方針 ## アーキテクチャ レイヤードアーキテクチャ（3層）： ``` Controller（HTTP） ↓ Service（業務ロジック） ↓ Mapper（SQL） ``` - 依存は上から下のみ - 各層は単一責任（SRP） - DI でテスト容易性を確保 ## テーブル構成 ``` categories ─1対多─ memos memos ─多対多─ tags（memo_tags経由） ``` | テーブル | 役割 | |---------|------| | memos | メモ本体 | | categories | カテゴリ（マスタ）| | tags | タグ（マスタ）| | memo_tags | 中間テーブル | ## 主要な設計判断 ### 削除戦略 - カテゴリ削除 → メモは残す （ON DELETE SET NULL） 理由: ユーザーのメモは失わせない - タグ削除 → 紐付けのみ消す （ON DELETE CASCADE） ### 重複防止 - カテゴリ名・タグ名は UNIQUE 制約 理由: 正しさはDBに守らせる // → 別冊S2,S4 の方針が // このプロジェクトの判断として残る ★★★ // → 「なぜこの設計か」が読める
POINT — 「設計判断」と「その理由」をセットで残す

design.md で最も価値があるのは「主要な設計判断＋理由」。「カテゴリ削除は SET NULL（理由：メモを失わせない）」のように。判断だけでなく WHY を残すと、後で「なぜこうなってる？」に答えられる。別冊S2のトレードオフの判断を文章化したもの。未来の自分が設計変更を考えるときの土台になります。

06

Markdown 記法の要点

MARKDOWN BASICS

ドキュメントは Markdown で書きます。GitHub は Markdown を自動でHTML表示。覚えるべき記法は10個程度。これだけで読みやすいドキュメントが書けます。

よく使う Markdown 記法これだけ覚えれば十分
記法意味
# 見出し見出し（#が多いほど小）
**太字**太字
- 項目箇条書き
1. 項目番号付きリスト
[文字](URL)リンク
`コード`インラインコード
```コードブロック
| A | B |表
- [ ] 項目チェックボックス
> 引用引用

#の後・-の後にはスペースが必須。これを忘れると見出し・リストにならない。

// Markdown の書き方と表示 // ─── 見出し（# の後にスペース）─── # 大見出し → 最大 ## 中見出し → 中 ### 小見出し → 小 // ─── リスト ─── - 項目1 // - の後にスペース必須 - 項目2 - ネスト // 2スペースでインデント 1. 手順1 // 番号付き 2. 手順2 // ─── リンク ─── [要求定義](docs/requirements.md) └文字┘ └────URL────┘ // ─── コードブロック ─── ```bash ./mvnw spring-boot:run ``` // → 言語名を書くと色付け（シンタックスハイライト） // ─── 表 ─── | 分類 | 技術 | |------|------| | 言語 | Java | // → |で区切り、2行目は区切り線 // ─── チェックボックス（Issueでも使用）─── - [x] 完了した項目 - [ ] 未完了の項目 // → GitHub Vol.1 の受入条件と同じ ★
POINT — GitHub のプレビューで確認しながら書く

GitHub 上でファイルを編集するとき、Preview タブで表示を確認できます。「表が崩れてないか」「リンクが効くか」を書きながらチェック。ローカルなら VS Code のプレビュー機能も便利。「書く→プレビュー→直す」を繰り返せば、Markdown はすぐ慣れます。本シリーズの Issue・PR でも使ってきた記法なので、もう半分は身についています。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

ドキュメント作成で陥りやすい地雷を5つ。★★★ 印は特に頻発。

Vol.GH5 ハマりどころ 5選Markdown＋運用の罠

① 見出し・リストが効かない ★★★

  • 原因：# や - の後にスペースがない
  • 対処：#␣見出し -␣項目（␣＝半角スペース）

② コードブロックが閉じていない ★★★

  • 原因：``` の閉じ忘れ → 以降が全部コード扱い
  • 対処：開始と終了の ``` をペアで

③ 表が崩れる

  • 原因：2行目の区切り線（|---|---|）がない／列数不一致
  • 対処：ヘッダー行の下に区切り線、各行の | 数を揃える

④ ドキュメントが古くなる ★★★

  • 原因：コードを変えてもドキュメントを更新しない
  • 対処：機能追加の PR でドキュメントも一緒に更新（Doc as Code）

⑤ READMEが長すぎる

  • 原因：詳細を全部 README に書く
  • 対処：README は概要、詳細は docs/ に分離（リンク）
// 主要な対処 // ─── ① スペースの有無 ─── NG: #見出し // スペースなし→ただの文字 OK: # 見出し // スペースあり→見出しに ★ NG: -項目 OK: - 項目 // - の後にスペース ★ // ─── ③ 表の正しい書き方 ─── | 分類 | 技術 | // ヘッダー |------|------| // 区切り線（必須！）★ | 言語 | Java | // データ行 // → 区切り線がないと表にならない // → 各行の | の数を揃える // ─── ④ Doc as Code の実践 ─── // 機能追加時の PR に含めるもの： - 実装コード - テスト - 関連ドキュメントの更新 ★★★ （README の機能一覧、design.md など） // → コードとドキュメントを // 同じ PR で更新すれば古くならない // ─── コミット例 ─── $ git commit -m "docs: README作成 - 機能一覧（US対応） - セットアップ手順 - docs/ へのリンク Refs #1"
