01

この巻のミッション

MISSION

Vol.GH2 のゴールは2つ。①feature ブランチで機能を作る、②コミットを Issue に紐付けてトレーサビリティを繋ぐ。①は基本マナー、②がRDT版の核心。「要求→ブランチ→コミット→クローズ」を一本の線にする。

トレーサビリティとは何か要求と実装を線で結ぶ
// トレーサビリティ＝追跡可能性

別冊S1で学んだ要求工学の「管理」。その重要概念がトレーサビリティ（追跡可能性）。「この要求は、どのコードで実現されたか」「このコードは、どの要求のためか」を双方向に辿れること。

// なぜ追跡できると嬉しいか
  • 「なぜこのコードがある？」→ Issue を辿れば要求が分かる
  • 「US-001 は完成した？」→ Issue の Close 状態で分かる
  • 変更時の影響範囲が辿れる
  • 監査・引き継ぎが楽になる
// Vol.GH2 で繋ぐ「線」 // ─── 要求から完了までの一本の線 ─── US-001（本編Vol.1で定義） ↓ Issue #1（GitHub版Vol.1で起票） ↓ feature/memo-crud（このVol.GH2で作成） ↓ コミット（Refs #1 で紐付け） ├ feat(entity): Memo追加 ├ feat(mapper): MemoMapper ├ feat(service): MemoService ├ feat(controller): MemoController ├ feat(view): メモ画面3つ └ test: 受入条件AC1-4のテスト ↓ PR（Vol.GH3で作成予定） ↓ Closes #1 でマージ ↓ Issue #1 が自動Close = US-001 完成 ★★★ // → 「要求が、いつ、どう実現されたか」 // が GitHub 上で完全に追跡できる
POINT — RDT版のブランチ名は要求と対応させる

本シリーズではブランチ名を US と対応させます。feature/memo-crud（US-001）、feature/category-crud（US-002）。ブランチ名を見れば「どの要求の作業中か」が分かる。ブランチも要求トレーサビリティの一部です。

03

準備とブランチ戦略

BRANCH STRATEGY

本シリーズのブランチ戦略はシンプルな GitHub Flow。main は常に動く、機能は feature ブランチで作る。小規模個人開発に過剰な Git Flow は採用しません（別冊S2の「過剰設計を避ける」と同じ思想）。

GitHub Flow を採用する理由規模に合った戦略
// ブランチ戦略の選択
戦略複雑さ向き
GitHub Flow低小〜中規模・継続デプロイ
Git Flow高大規模・リリース管理厳格
trunk-based低CI/CD成熟チーム

本シリーズは GitHub Flow。main + feature だけ。メモ帳に Git Flow は過剰（別冊S2の判断軸）。

// GitHub Flow の3ルール
  1. main は常にデプロイ可能（動く状態）
  2. 作業はfeature ブランチで
  3. 完成したらPR経由で main へ（Vol.GH3）
// 使うコマンド（予習） $ git switch -c feature/X // 作成＋切替 $ git switch main // 切替 $ git branch // 一覧・現在地 $ git merge feature/X // 取り込み $ git branch -d feature/X // 削除 $ git log --graph // 履歴グラフ // ─── ブランチ命名規則 ─── feature/memo-crud // US-001 feature/category-crud // US-002 feature/tag-crud // US-003 fix/title-validation // バグ修正 // → 「種類/対象」の形 // → 要求(US)と対応 ★★★ // ─── switch vs checkout ─── // switch = ブランチ切替専用（新しい・推奨） // checkout = 切替＋ファイル復元（混乱しやすい） // → 本シリーズは switch 採用
POINT — ブランチ戦略も「過剰設計を避ける」

Git Flow はdevelop/release/hotfix など多数のブランチを使う重厚な戦略。大規模には有効ですが、メモ帳の個人開発には過剰。別冊S2のYAGNI・規模に合った構造と同じ判断で、シンプルな GitHub Flow を選びます。道具は規模に合わせる。

04

ハンズオン（コマンドライン版）

CLI HANDS-ON

本編Vol.2 の Memo CRUD を、feature ブランチで作り、Issue #1 に紐付けながらコミットします。受入条件・層ごとにコミットを分けるのがポイントです。

手順 — feature ブランチで実装管理Issue紐付きコミット
  1. main から feature/memo-crud を作成＆切替
  2. Memo Entity を作成 → add → commit（Refs #1）
  3. Mapper を作成 → add → commit（Refs #1）
  4. Service を作成 → add → commit（Refs #1）
  5. Controller を作成 → add → commit（Refs #1）
  6. View 3画面 → add → commit（Refs #1）
  7. テスト（受入条件） → add → commit（Refs #1）
  8. feature ブランチを push

すべてのコミットに Refs #1。Issue #1 と紐付き、トレーサビリティが繋がります。

// CLI 手順 // Step 1: ブランチ作成＆切替 $ git switch -c feature/memo-crud Switched to a new branch 'feature/memo-crud' // Step 2: Entity コミット $ git add src/main/java/.../entity/Memo.java $ git commit -m "feat(entity): Memoクラス追加 - @NotBlank/@Size でバリデーション （受入条件 AC2-4 に対応） - 5フィールド（id/title/content/ createdAt/updatedAt） Refs #1" // ↑ Refs #1 で Issue #1 に紐付く ★★★ // Step 3-6: 同様に層ごとコミット $ git commit -m "feat(mapper): MemoMapper追加 Refs #1" $ git commit -m "feat(service): MemoService追加 - create時にcreatedAt/updatedAtをセット Refs #1" $ git commit -m "feat(controller): MemoController追加 - @Validated + BindingResult で バリデーション（AC2対応） - PRGパターン Refs #1" $ git commit -m "feat(view): メモ画面3つ追加 Refs #1" // Step 7: テストコミット（RDT版の肝） $ git add src/test/ $ git commit -m "test: US-001の受入条件をテスト化 - MemoValidationTest（AC2-4） - MemoServiceTest（日時セット） - MemoControllerTest（AC1 MockMvc） - 6 tests, all green Refs #1" // ↑ テストも独立コミット ★★★ // Step 8: push $ git push -u origin feature/memo-crud
POINT — テストを独立コミットにする意味

RDT版ではテストを独立したコミットにします（or 実装とセット）。「どの受入条件を、いつテスト化したか」が履歴に残る。理想はテストファースト（テストを先にコミット → 実装で緑に）。これがRed-Green-Refactorの履歴版。テストの存在が履歴で証明されるのが業務でも重要です。

EGIT VERSION

EGit でも同じことができます。Issue 紐付け（Refs #1）はコミットメッセージに書くだけなので、EGit でも全く同様に機能します。

EGit でのブランチ＆コミットGUI で完結

// History で確認

Team → Show in History で、Refs #1 付きコミットが層ごとに並ぶのを視覚的に確認できます。

EGit の History ビューは、Refs #1 付きコミットが層ごとに積み上がる様子が視覚的に見えます。各コミットをクリックすれば diff も確認できる。「どの層で何を変えたか」を目で追えるのが EGit の強み。コマンドの git log --graph より直感的です。

Issue 紐付けとトレーサビリティ

ISSUE LINKING

RDT版の核心。Refs #1 と Closes #1 の使い分け、GitHub上での自動リンク、双方向トレーサビリティの実現方法を学びます。要求と実装が GitHub 上で一本の線になる瞬間です。

Refs と Closes の使い分け紐付けとクローズ
// 2つのキーワード
Refs #1
参照（紐付けのみ）
Issue #1 にリンクを張る。Issueは開いたまま。作業途中のコミットに。
Closes #1
クローズ
マージ時に Issue #1 を自動Close。完成を表すコミット/PRに。
// 使い分けの方針
  • 作業途中の各コミット → Refs #1（紐付けだけ）
  • 完成を表すPR or 最終コミット → Closes #1（自動Close）
  • Closes は main にマージされた時に発動
// 他のキーワード

Closes / Fixes / Resolves は同じ（自動Close）。Refs / See は参照のみ。

POINT — トレーサビリティは「未来の自分」への手紙

半年後、「なんでこのコードこう書いたんだっけ」となったとき。コミットの Refs #1 から Issue #1 を開けば、当時の要求・受入条件・判断が全部読める。トレーサビリティは未来の自分（と引き継ぎ相手）への手紙。書くのは少し手間ですが、受け取る価値は計り知れません。

07

ハマりどころ＋NEXT STEP

PITFALLS & NEXT

ブランチ＋Issue紐付けで踏みやすい地雷を5つ。★★★ 印は特に頻発。

Vol.GH2 ハマりどころ 5選先に知れば回避できる

① 未コミットでブランチ切替できない ★★★

  • エラー：Your local changes would be overwritten
  • 対処：git stash で退避 → 切替 → git stash pop

② Refs #1 を書いてもリンクされない

  • 原因：push していない（ローカルのみ）／番号違い
  • 対処：push 後に GitHub で確認。Issue 番号を再確認（#1 が正しいか）

③ Closes #1 を途中コミットに書いてしまった ★★★

  • 問題：まだ完成してないのに Issue がCloseされる（マージ時）
  • 対処：途中は Refs #1、完成PRだけ Closes #1

④ main で作業してしまった

  • 状況：feature を切り忘れて main にコミット
  • 対処：git reset --soft HEAD~1 → git switch -c feature/X → commit し直し

⑤ push 時に rejected（pull忘れ）

  • 原因：リモートが先行（Web編集など）
  • 対処：git pull → コンフリクト解決 → push
// 主要エラーの対処 // ─── ① stash で退避 ─── $ git stash $ git switch main $ git switch feature/memo-crud $ git stash pop // ─── ④ main で誤コミット ─── $ git log --oneline -1 abc123 (main) feat: Memo追加 ← main!! $ git reset --soft HEAD~1 // → mainを1つ戻す（変更は残る） $ git switch -c feature/memo-crud // → 新ブランチへ（変更を持っていく） $ git commit -m "feat: Memo追加 Refs #1" // → 正しいブランチでコミットし直し ★★★ // ─── ③ Refs と Closes の使い分け ─── 途中コミット → Refs #1 （紐付けのみ） 完成のPR → Closes #1 （自動Close） // 途中で Closes を書くと、マージ時に // まだ完成してない Issue が閉じる ★★★
