01

要求の獲得

PHASE: REQ ELICITATION

最初にやるのはコードでも設計でもなく「要求の獲得」です。WHO（誰のため）/ WHAT（何を）/ WHY（なぜ）。中でもWHY が一番抜けやすく、一番重要。「メモ帳を作る」だけでは何も決まりません。誰の・どんな困りごとを・なぜ解決するかを言葉にします。

要求工学の全体フロー 本シリーズの背骨
// なぜ「考えてから書く」のか

設計を飛ばして書いたコードは最初の1週間は速い。でも1か月後から壊れ始める。要求を固めずに作ると「作ったけど誰も使わない」が起きます。

要求工学は5つの活動の反復です。一直線ではなく、何度も行き来します。

[1] 獲得 → 困りごとを引き出す ↓ [2] 分析 → 本当に必要なものへ ↓ [3] 仕様化 → 文書・モデルに残す ↓ [4] 検証 → これで良いか確かめる ↓ [5] 管理 → 変化を追跡し続ける ↑___ ループ（要求は変わる）
// WHO / WHAT / WHY の3問 // メモ帳と一口に言っても… WHO 誰が使う？ - 自分専用？ - チーム共有？ - 不特定多数？ WHAT 何を作る？ - 思いつきの記録？ - 議事録？ - TODO？日記？ WHY なぜ作る？ ← 最重要 - 既存アプリの何が不満？ - 何を解決したい？ // ─── 同じ「メモ帳」でも… ─── // 「自分用の議事録ツール」と // 「チームの共有日記」では // 全く別物になる ★★★ // だから最初に WHO/WHAT/WHY を固める
ペルソナを立てる 「誰のため」を1人に絞る
// 本シリーズのペルソナ

要求の獲得では架空の代表ユーザー（ペルソナ）を立てます。「みんなのため」は「誰のためでもない」。1人に絞ると要求が明確になります。

PERSONA
田中さん（32歳・会社員）
利用場面：通勤中にスマホで思いつきをメモ、夜PCで整理。ゴール：思いついた瞬間に素早く記録したい。不満：既存アプリは検索しても目的のメモが出ない。技術レベル：中。
// 獲得段階で答えるべき質問 // 本シリーズの回答： Q. 誰が使う？ A. "個人ユーザー（単一ユーザー前提・v1）" Q. どんな場面？ A. "思いつき・TODO・記録を カテゴリとタグで整理したい" Q. 既存の何が不満？ A. "分類軸が1つしかなく、 後から探しにくい" Q. どのデバイス？ A. "まずWebブラウザ（PC）。 同期はv2以降" // → この回答が、3テーブル設計 // （memos + categories + tags） // の「なぜ」になる ★★★ // → 分類軸を2つ持ちたい // = カテゴリ＋タグ
POINT — 「要求」と「要件」と「仕様」は別物

要求＝ユーザーの生の声（「探しやすいといいな」）。要件＝測定可能に明文化（「全文検索が500ms以内」）。仕様＝どう実現するか（MATCH AGAINST を使う）。曖昧な声を検証可能な条件に変えるのが要求工学の仕事です。Vol.1 ではこの3語の区別を体に入れます。

02

要求の分析

PHASE: REQ ANALYSIS

獲得した生の要求を「機能要求・非機能要求・制約」の3つに仕分けます。さらにMoSCoW法で「v1で何を作るか」を決める。全部作ろうとすると終わらない。「やらないこと」を明示するのが分析の核心です。

3カテゴリへの仕分け 機能 / 非機能 / 制約
// メモ帳アプリの仕分け
カテゴリ内容
機能要求作成・編集・削除・一覧・カテゴリ分類・タグ付け・検索
非機能要求一覧1秒以内・データ永続化・入力検証
制約単一ユーザー前提(v1)・Web(PC)・Spring Boot/MySQL
// 非機能要求は「数値」で書く
  • NG：「速い」「使いやすい」（測定不能）
  • OK：「一覧表示は1000件で1秒以内」（測定可能）

数値化が要求工学の鉄則。後で検証できる形にします。

// 非機能要求（NFR）を数値で @PERF-001 一覧表示性能 メモ1000件でも一覧表示が 1秒以内 @PERF-002 検索性能 全文検索を 500ms 以内（P95） @DATA-001 データ保全 登録内容はDBに永続化され、 アプリ再起動でも失われないこと @VALID-001 入力検証 タイトルは必須・200文字以内 色は #xxxxxx 形式 @USAB-001 使用性 マニュアルなしで メモ作成・検索・削除ができること // → これらが Vol.7,8,12,13 で // テストの「合格ライン」になる ★★★
MoSCoW法で優先順位 v1のスコープを決める
// 4分類でスコープを守る
MUST
必須
無いとリリース不可。メモCRUD・一覧。
SHOULD
重要
無くても動く。カテゴリ・検索。
COULD
あれば嬉しい
余裕があれば。タグ・ピン留め。
WON'T
今回やらない
v2候補。同期・共同編集。

目安比率：Must 60% / Should 20% / Could 20%。Must が80%超なら絞り込み不足のサイン。

// 本シリーズの MoSCoW（全14巻の地図） [Must] — Vol.2,3 ・メモCRUD（作成/編集/削除/一覧） ・カテゴリ管理（マスタCRUD） [Should] — Vol.4,5,6,7 ・カテゴリ別の絞り込み ・タグ付け（多対多） ・複合検索 [Could] — Vol.8,9,10,11 ・全文検索 ・ピン留め/並び替え/アーカイブ ・一括操作/エクスポート [Won't（今回のv1範囲外）] ・複数端末同期 ・共同編集 ・画像添付 // → 本シリーズの全14巻は // この優先順位に沿って並ぶ ★★★ // → Vol.12,13 は品質強化 // （バリデーション・性能）
POINT — 「Won't（やらない）」を書く勇気

初学者ほど「あれもこれも」と機能を盛り込んで完成しない。Won't を明示すると「やらなくていい」と決まり、心が軽くなります。同期・共同編集は v1 ではやらない。これを最初に宣言するのがプロのスコープ管理です。

03

仕様化（ストーリー＋受入条件）

PHASE: REQ SPECIFICATION

機能要求をユーザーストーリー形式で書き、各ストーリーに受入条件（Given/When/Then）を添えます。これが後のテストの設計図になります。「受入条件＝テストケース」と考えてOK。要求とテストが直結する瞬間です。

ユーザーストーリー（Connextra形式） As a / I want / so that
// 書式

As a〔ユーザー役割〕として
I want〔機能・行動〕
so that〔理由・価値〕

so that（理由）が肝。「なぜそれが欲しいか」を必ず書く。理由が書けない機能は本当に要るのか疑う。

// INVEST チェック

良いストーリーは6条件を満たします：

  • Independent（独立）
  • Negotiable（対話の余地）
  • Valuable（価値がある）
  • Estimable（見積もれる）
  • Small（小さい）
  • Testable（テストできる）
// 本シリーズのユーザーストーリー US-001（Vol.2） As a 思いつきを記録したいユーザー I want すぐにメモを作成・保存したい so that アイデアを忘れる前に書き留められる US-002（Vol.3） As a メモを整理したいユーザー I want カテゴリでメモを分類したい so that 種類ごとに把握できる US-003（Vol.5） As a 横断的に整理したいユーザー I want メモに複数タグを付けたい so that カテゴリと別の軸でも絞れる US-004（Vol.7,8） As a 過去のメモを探したいユーザー I want キーワードで検索したい so that 目的のメモを素早く見つけられる // → 各 US が各 Vol に対応 ★★★ // → 「so that」が WHY を支える
受入条件（Given / When / Then） = 後のテストケース
// 書式

Given〔前提条件〕
When〔行動〕
Then〔期待される結果〕

受入条件は正常系だけでなく異常系・境界値も書きます。「該当なし」「空入力」も忘れずに。

// なぜ受入条件が重要か
  • 「完成」の定義が明確になる
  • そのままテストコードに変換できる
  • 認識のズレを防げる
// US-001（メモ作成）の受入条件 @AC1 正常系 Given: タイトル「買い物」を入力した状態 When : 保存ボタンを押す Then : メモが保存され、一覧に表示される @AC2 異常系（空タイトル） Given: タイトルが空の状態 When : 保存ボタンを押す Then : エラーが表示され、保存されない @AC3 境界値（200文字） Given: タイトルがちょうど200文字 When : 保存する Then : 正常に保存される @AC4 境界値の外（201文字） Given: タイトルが201文字 When : 保存する Then : エラーが表示される // ─── これが Vol.2 で… ─── @Test void 空タイトルはエラーになる() { // @AC2 がそのまま JUnit テストに ★★★ } // → 受入条件 = テストケース // → 要求とテストが直結する
POINT — 受入条件はテストの設計図

Given/When/Then は、JUnit の「準備（Arrange）→ 実行（Act）→ 検証（Assert）」＝AAAとそのまま対応します。Given=Arrange、When=Act、Then=Assert。要求段階で受入条件を書いておけば、Vol.2 以降でテストコードに翻訳するだけ。要求とテストは地続きです。

04

設計①アーキテクチャ・データ

PHASE: DESIGN ARCHITECTURE & DATA

要求が固まったら設計へ。まずアーキテクチャ（全体構造）とデータ設計を決めます。「変更コストの大きい決定」を先に。メモ帳のような小〜中規模アプリではレイヤード（層を分ける）で十分。過剰設計は避けます。

レイヤード・アーキテクチャ Spring の3層構造そのもの
// 関心を層で分ける

フロー文書の「UI→ロジック→データ」は、Spring では「Controller→Service→Mapper」に対応します。

[Controller層] 画面・入力受付 ↓ 呼ぶ [Service層] 業務ロジック ↓ 呼ぶ [Mapper層] DB保存・読込

鉄則：依存は上から下へ。下の層が上の層を知ってはいけない。Mapper が Controller を呼ぶ、は禁止。

// フロー文書 → Spring 翻訳 // 原典（JS/レイヤード） Spring版 UI層 → Controller ロジック層 → Service データ層 → Mapper (MyBatis) // ─── 規模に合った構造を選ぶ ─── // メモ帳は小〜中規模 // → クリーンアーキテクチャは過剰 // → レイヤードで必要十分 ★★★ // ─── フォルダ構成は設計そのもの ─── src/main/java/com/example/memo/ ├ controller/ // 画面 ├ service/ // ロジック ├ mapper/ // DB └ entity/ // データ構造 // → 見ただけで「何をするアプリか」 // が分かるのが理想
データ設計 — 名詞を探す 要求の「名詞」がエンティティ
// 要求から名詞を抽出

要求文に出てくる名詞がデータの候補です。本シリーズの要求から：

  • 「メモを作る」→ memos
  • 「カテゴリで分類」→ categories
  • 「タグを付ける」→ tags

3つの名詞 = 3テーブル。要求が自然にデータ構造を導きます。

// 「事実は時点で確定する」
  • created_at：一度決めたら変えない
  • updated_at：保存のたびに更新

「いつの事実か」を意識して設計します。

// データ設計：3エンティティ Memo（メモ・トランザクション系） - id 一意識別子 - title タイトル（必須・200字） - content 本文 - category_id カテゴリ参照（Vol.4〜） - created_at 作成日時（不変） - updated_at 更新日時（毎回更新） Category（カテゴリ・マスタ系） - id - name 必須・UNIQUE - color #xxxxxx - sort_order 並び順 Tag（タグ・マスタ系） - id - name 必須・UNIQUE memo_tags（中間テーブル・多対多） - memo_id - tag_id // ─── 識別子の設計判断 ─── // 連番 vs UUID // → 本シリーズは学習目的で連番(AUTO_INCREMENT) // → 業務で同期を考えるなら UUID 推奨 // → 「戻せる選択肢」の観点では // 後から移行も可能 ★
POINT — YAGNI：必要になるまで作らない

「将来 UUID にするかも」「同期するかも」で最初から重い仕組みを入れるのは過剰設計。本シリーズは連番＋localStorageではなくMySQLで素直に始めます。「将来のため」の8割は使われない（YAGNI）。測定してから最適化、迷ったら戻せる選択肢。これが設計の判断軸です。

05

設計②3テーブルの全体像

PHASE: DESIGN TABLE DESIGN

データ設計を具体的なDDL（CREATE TABLE）に落とします。1対多（categories）と多対多（tags）の両方を持つのが本シリーズの特徴。関係性は要求から導かれることを確認しながら進めます。

関係性の判定（要求ベース） なぜ1対多と多対多か
// 要求から関係性を導く

「メモは1つのカテゴリに属する」（要求）
→ メモ：カテゴリ ＝ 多：1 → 1対多

「メモに複数タグ、タグも複数メモに」（要求）
→ メモ：タグ ＝ 多：多 → 多対多（中間テーブル必須）

// 判定式
  • 両側に複数あり得る → 多対多
  • 片方だけ複数 → 1対多
  • 両方1つ → 1対1（同テーブルで足りる）

関係性は要件で決まる。詳しくは別冊S1で。

// DDL：3テーブル + 中間テーブル CREATE TABLE categories ( id BIGINT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(50) NOT NULL UNIQUE, color VARCHAR(7) NOT NULL, sort_order INT DEFAULT 0, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); CREATE TABLE memos ( id BIGINT AUTO_INCREMENT PRIMARY KEY, title VARCHAR(200) NOT NULL, content TEXT, category_id BIGINT, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, FOREIGN KEY (category_id) REFERENCES categories(id) ON DELETE SET NULL // メモは残す ); CREATE TABLE tags ( id BIGINT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(50) NOT NULL UNIQUE ); CREATE TABLE memo_tags ( memo_id BIGINT NOT NULL, tag_id BIGINT NOT NULL, PRIMARY KEY (memo_id, tag_id), FOREIGN KEY (memo_id) REFERENCES memos(id) ON DELETE CASCADE, FOREIGN KEY (tag_id) REFERENCES tags(id) ON DELETE CASCADE ); // → 削除戦略も設計の一部 ★★★ // categories削除 → memoは残す(SET NULL) // memo削除 → タグ紐付けも消す(CASCADE)
POINT — 削除戦略は設計で決める

FK の ON DELETE は設計判断。SET NULL（メモは残す）、CASCADE（連動削除）、RESTRICT（削除拒否）。本シリーズは「メモは大事だから残す、関連付けは連動」という方針。「これを消したらアレはどうなる？」を最初に決めるのが事故を防ぐ鉄則です。

07

テストの土台を整える

PHASE: TEST TEST FOUNDATION

本シリーズはテストを書きながら開発します。Vol.1 ではテストの土台を整え、テストピラミッドの考え方と最初の1本を体験します。本格的なテストは Vol.2 から、要求の受入条件を JUnit に翻訳していきます。

テストピラミッドと Spring 翻訳 下を厚く、上を薄く
// 3層の比率と Spring での道具
層割合Springでの道具
単体70%JUnit5 + Mockito + AssertJ
結合20%@MybatisTest + H2/Testcontainers
E2E10%MockMvc（Spring MVC Test）
// 設計のコツ（テスト容易性）

「保存」「現在時刻」など外部依存と、純粋な計算ロジック（検索・ソート・バリデーション）を分離すると、単体テストが圧倒的に書きやすい。

テストが書きにくい＝設計を見直すサイン。これは本シリーズの重要な指針です。

// フロー文書 → Spring テスト翻訳 // 原典(JS/Jest) Spring版 Jest 単体 → JUnit5 + Mockito InMemory Fake → @MybatisTest + H2 Playwright E2E → MockMvc // ─── 依存性注入（DI）でテスト容易に ─── // フロー文書の「Storeを外から注入」は // Spring では @RequiredArgsConstructor @Service @RequiredArgsConstructor public class MemoService { private final MemoMapper mapper; // ↑ テストでは Mockito で // モックに差し替えられる ★★★ } // テスト時： @Mock MemoMapper mapper; @InjectMocks MemoService service; // → 本物のDBなしでロジックをテスト
最初のテストを動かす JUnit が動く環境を確認
// テストの配置と作成手順
  1. Spring Initializr 生成時に spring-boot-starter-test が自動で入っている
  2. テストは src/test/java/com/example/memo/ 配下
  3. 右クリック → New → JUnit Test Case
  4. まず「JUnit が動くか」だけ確認する1本を書く
// AAA パターン（3ブロック）
  • Arrange（準備）＝ 受入条件の Given
  • Act（実行）＝ When
  • Assert（検証）＝ Then
// SmokeTest.java（最初の1本） package com.example.memo; import static org.assertj.core.api.Assertions.*; import org.junit.jupiter.api.Test; class SmokeTest { @Test void JUnitが動作する() { // Arrange（準備） int a = 2; int b = 3; // Act（実行） int sum = a + b; // Assert（検証） assertThat(sum).isEqualTo(5); } } // → 右クリック → Run As → JUnit Test // → 緑のバー（成功）が出れば // テスト環境OK ★★★ // ─── テスト名は日本語でOK ─── // 「何を検証しているか」が分かる名前を // 例：空タイトルはエラーになる() // 200文字は登録できる() // → Vol.2 から本格的に書いていく
POINT — テストは「将来の自分を救う投資」

テストの価値は書いた瞬間ではなく、機能追加やリファクタの時に出ます。「動いた」を確認するためではなく、「これからも安心して変更できる」状態を作るために書く。Vol.1 では土台だけ。Vol.2 から受入条件をテストに翻訳していきます。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPL VERIFY & PITFALLS

土台が整ったら動作確認。/health で疎通を確認し、JUnit の最初の1本を緑にします。Vol.1 で初学者が必ず踏む環境系の地雷を5つ先回りします。

Vol.1 ハマりどころ 5選 環境構築の定番地雷

① DB接続できない（Communications link failure）★★★

  • 原因：MySQL が起動していない／ポート違い／パスワード違い
  • 対処：MySQL 起動確認、application.properties の url/username/password を再確認

② Unknown database 'memo_app'

  • 原因：DB を作っていない
  • 対処：CREATE DATABASE memo_app; を実行

③ Lombok の getter/setter が赤線 ★★★

  • 原因：Eclipse に Lombok プラグインが未導入
  • 対処：lombok.jar を実行してEclipseにインストール → Eclipse再起動

④ ポート8080が使用中（Port already in use）

  • 原因：別のアプリ（or 前回起動分）が8080使用中
  • 対処：前のプロセス停止、または server.port=8081 に変更

⑤ map-underscore... が効かず全フィールド null

  • 原因：map-underscore-to-camel-case=true 書き忘れ
  • 対処：application.properties に追記 → 再起動（Vol.2 で効いてくる）
// 疎通確認チェックリスト // ① Spring 起動 コンソールに： Started MemoApplication in X seconds // → 起動成功 // ② DB接続＋データ確認 ブラウザ: http://localhost:8080/health 表示: OK / categories = 3 // → DB接続＆初期データOK ★★★ // ③ JUnit 動作 SmokeTest を Run As → JUnit Test 緑のバー // → テスト環境OK // ─── ③が赤いとき（DB接続NG）─── Caused by: com.mysql.cj.jdbc.exceptions .CommunicationsException: Communications link failure // チェック順： // 1. MySQL は起動しているか // 2. ポート3306は合っているか // 3. username/password は正しいか // 4. memo_app DBは存在するか
TIP — エラーは「下から読む」

Spring の長いスタックトレースは一番下の「Caused by:」が真の原因。上から読むと心が折れます。下から読んで「Caused by」を探すのが鉄則。エラーメッセージは敵ではなく道案内です。
