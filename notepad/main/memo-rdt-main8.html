01

要求の確認（US-004の完成）

PHASE: REQREVISIT REQUIREMENT

Vol.7 で検索（LIKE）は動きましたが、「メモが大量になっても速い」という非機能要求がまだ。US-004 を本当に完成させるには、性能も満たす必要があります。Vol.8 は US-004 の仕上げです。

機能要求と非機能要求動く＋速い
// Vol.7 で足りなかったこと

Vol.7 で「検索できる」（機能要求）は満たしました。でも US-004 には「快適に使える」（非機能要求）も暗黙に含まれます。メモ1万件で検索に数秒かかったら使えない。

// 機能 vs 非機能（別冊S1）
種類例巻
機能要求検索できるVol.7
非機能要求速い・快適Vol.8
// 「測定してから」が前提

別冊S2の「測定してから最適化」通り、まず Vol.7 で素直に LIKE 実装→遅いと測定で確認→FULLTEXT で改善という順序を踏みます。

// US-004 の非機能要求 @AC6 性能（非機能要求） Given: メモが大量（例: 1万件）ある When : キーワード検索する Then : 実用的な速度で結果が返る （体感で待たされない） @AC7 改善後も正しさ維持 Given: FULLTEXT に変更した When : Vol.7 と同じ検索をする Then : Vol.7 と同じ結果が返る （高速化で結果が変わらない） // ─── @AC6,7 が Vol.8 の核心 ★★★ ─── // @AC6：速くする（非機能） // @AC7：速くしても結果は同じ // → 最適化で壊さないことを保証 // → 既存テストが守ってくれる ★ // ─── 「動く」の次は「使える」─── // 機能要求（動く）= Vol.7 // 非機能要求（速い・使える）= Vol.8 // → 両方満たして US-004 完成 ★★★

02

受入条件をテストケース化

PHASE: REQ→TESTAC TO TEST CASES

性能改善のテストは2種類。①速くなったか（性能）、②結果は変わらないか（正しさ）。特に②が重要。最適化で結果が変わったら本末転倒。Vol.7 のテストが「壊していない保証」になります。

性能改善のテスト戦略速さと正しさ
// 2つの観点
観点確認内容
正しさVol.7と同じ結果が返る
性能大量データで速い
// 正しさのテスト（最重要）

Vol.7 の検索テストがそのまま使える。FULLTEXT に変えても同じテストが緑なら、結果は変わっていない。これが「最適化で壊していない」保証（別冊S3の変更の安全網）。

// 性能のテスト（補助的）

性能テストは環境依存で不安定。CI で厳密に測るより、大量データで「実用的か」を手動確認が現実的。

// 性能改善のテスト戦略 // ─── ① 正しさ（既存テストが守る）★★★ ─── // Vol.7 の MemoSearchTest がそのまま // 「キーワードで絞り込める」等のテストが // FULLTEXT に変えても緑なら… // → 結果が変わっていない証明 ★★★ // → リファクタの安全網（別冊S3） @Test void キーワードで絞り込める() { // Vol.7 と同じテスト // 実装が LIKE→FULLTEXT に変わっても // このテストは変えない // → 緑なら「動作は同じ」★ } // ─── ② 性能（手動測定が現実的）─── // 大量データを投入して測る // LIKE: 1万件で 800ms // FULLTEXT: 1万件で 20ms // → 改善を数値で確認 ★ // 自動の性能テストは環境依存で不安定 // → 別冊S3「sleep/時間依存は避ける」 // → 性能は手動測定＋ログで確認が無難 // ─── 最適化の鉄則 ─── // 「速くする」より先に // 「結果を変えない」を保証する // → 既存テストを緑に保つ ★★★
POINT — 最適化の前に「テストが緑」を確認する

性能改善（リファクタの一種）の鉄則：「変更前にテストが緑、変更後もテストが緑」。Vol.7 の検索テストがあるおかげで、LIKE→FULLTEXT に変えても「結果が変わっていない」と即確認できる。これが別冊S3の「テストは変更の安全網」の真価。テストがあるから、安心して最適化できる。テストなしの最適化は「直したつもりが壊した」の温床です。

03

仕様化：LIKEの限界を測る

PHASE: SPECMEASURE THE LIMIT

いきなり FULLTEXT にせず、まず「LIKE が本当に遅いか測る」。別冊S2の「推測するな、測定せよ」。EXPLAIN で実行計画を見て、LIKE がインデックスを使えていないことを確認します。

LIKEの遅さを測定で確認推測でなく測定
// なぜ LIKE '%word%' は遅いか

別冊S6・S7 の復習。LIKE '%word%' は先頭が % なのでインデックスが効かない。全行を1つずつ調べるフルスキャンになる。

// EXPLAIN で実行計画を見る

MySQL の EXPLAIN で「どう実行されるか」が分かる。type: ALL が出たらフルスキャン（遅い）の証拠。

// 測定の手順（別冊S2）
  1. 大量データを投入
  2. LIKE 検索の時間を測る
  3. EXPLAIN で実行計画を確認
  4. 遅さの原因を特定
// LIKE の限界を測定 // ─── EXPLAIN で実行計画 ─── EXPLAIN SELECT * FROM memos WHERE title LIKE '%会議%'; // 結果： +------+-------+--------+ | type | rows | Extra | +------+-------+--------+ | ALL | 10000 | Using | ← フルスキャン ★ | | | where | +------+-------+--------+ // type=ALL = 全行スキャン（遅い） // rows=10000 = 1万行を全部調べる ★★★ // ─── 時間を測る ─── // メモ10件： 1ms（一瞬） // メモ1000件： 80ms // メモ10000件： 800ms（体感で遅い）★ // メモ100000件： 8秒（実用不可）★★★ // ─── なぜ通常インデックスが効かないか ─── // 索引は「先頭から探す」（別冊S6） LIKE '会議%' → 前方一致 → 索引が効く LIKE '%会議%' → 中間一致 → 索引が効かない ★ LIKE '%会議' → 後方一致 → 索引が効かない // メモ検索は「どこに含まれるか」なので // '%会議%' が必要 → 通常索引では無理 // → FULLTEXT が必要（Section 04）★★★ // ─── 別冊S2 の実践 ─── // 「遅そう」で最適化せず、測ってから ★ // メモ帳v1が数百件なら LIKE で十分 // 大量を想定するなら FULLTEXT へ
POINT — EXPLAIN は「SQLの健康診断」

EXPLAIN は SQL の前に付けるだけで「どう実行されるか」を教えてくれる、性能チューニングの必須ツール。type: ALL（フルスキャン）や rows が大きいと遅いサイン。「なんとなく遅い」ではなくEXPLAIN で原因を特定してから対策する。これが別冊S2の「推測するな、測定せよ」の実践。当てずっぽうの最適化を避ける道具です。

04

設計①：FULLTEXTインデックス

PHASE: DESIGNFULLTEXT INDEX

遅さの原因が分かったのでFULLTEXT インデックスを設計します。全文検索専用の索引で、「文章の中の単語」を高速に探せる。日本語にはngram パーサが必要な点も押さえます。

FULLTEXT の仕組みと日本語対応MATCH ... AGAINST
// FULLTEXT インデックスとは

通常の索引が「列の値そのもの」を索引化するのに対し、FULLTEXT は「文章を単語に分解して索引化」。だから「文章中のどこに含まれるか」を高速に探せる。

// 日本語の壁と ngram

FULLTEXT は元々英語向け（スペースで単語分割）。日本語はスペースがないので、ngram パーサ（N文字ずつ区切る）が必要。MySQL 5.7+ で利用可能。

// 検索構文

LIKE の代わりに MATCH(列) AGAINST(キーワード) を使う。

// FULLTEXT インデックスの設計 // ─── DDL：FULLTEXT インデックス追加 ─── // 日本語対応のため ngram パーサを指定 ALTER TABLE memos ADD FULLTEXT INDEX ft_memo (title, content) WITH PARSER ngram; // ↑ 日本語をN文字ずつ索引化 ★★★ // ─── 検索構文（LIKE → MATCH AGAINST）─── // Vol.7（LIKE・遅い）: WHERE title LIKE '%会議%' // Vol.8（FULLTEXT・速い）: WHERE MATCH(title, content) AGAINST(#{keyword} IN BOOLEAN MODE) // → 索引を使って高速検索 ★★★ // ─── EXPLAIN で改善を確認 ─── EXPLAIN SELECT * FROM memos WHERE MATCH(title,content) AGAINST('会議'); +----------+------+ | type | rows | +----------+------+ | fulltext | 1 | ← 索引使用！★★★ +----------+------+ // type=fulltext = FULLTEXT索引を使用 // rows=1 = 全行スキャンしない（速い） // ─── 性能の改善 ─── LIKE FULLTEXT 1万件: 800ms → 20ms ★★★ 10万件: 8秒 → 50ms ★★★ // → 桁違いに速くなる // ─── ngram の仕組み（bigram例）─── // 「会議メモ」を2文字ずつ索引化： // 会議 / 議メ / メモ // → 「会議」で検索すると索引ヒット ★
POINT — 日本語の全文検索は ngram パーサが必須

FULLTEXT は英語前提（スペース区切り）なので、そのままだと日本語が検索できません。日本語はngram パーサ（文章をN文字ずつ区切って索引化）が必要。MySQL なら WITH PARSER ngram。これを忘れると「FULLTEXT を作ったのに日本語がヒットしない」とハマります（Section 08）。言語によって全文検索の設定が変わる——日本語特有の注意点です。

05

設計②：段階的移行の判断

PHASE: DESIGNGRADUAL MIGRATION

FULLTEXT は強力ですが万能ではない。「いつ LIKE から FULLTEXT に切り替えるか」は判断が要ります。別冊S2のトレードオフとYAGNIで、段階的な移行を設計します。

LIKE と FULLTEXT の使い分けトレードオフの判断
// それぞれの長所・短所
LIKEFULLTEXT
速度（大量）遅い速い
少量データ十分速い速い
実装簡単索引・ngram設定
部分一致柔軟単語単位の癖
H2テスト可不可（要MySQL）
// 判断基準
  • 少量（〜数千件）：LIKE で十分（KISS）
  • 大量・高頻度検索：FULLTEXT
  • 本シリーズは「学習目的で FULLTEXT を体験」
// 段階的移行の設計 // ─── 別冊S2 の判断軸を適用 ─── if (データ量が少ない && 検索が稀) { // LIKE で十分（YAGNI） // 索引・ngramの複雑さは不要 → Vol.7 の LIKE のまま ★ } else if (大量データ || 高頻度検索) { // 性能が問題 → FULLTEXT → Vol.8 の FULLTEXT へ移行 ★★★ } // ─── 移行を段階的に ─── 1. Vol.7：LIKE で動かす（まず動く） 2. 測定：遅いことを EXPLAIN で確認 3. Vol.8：FULLTEXT に切り替え 4. 既存テストで「結果同じ」を確認 // → いきなり FULLTEXT にしない // → 「動く→測る→直す」の順 ★★★ // ─── FULLTEXT の注意（万能でない）─── // ・短すぎる単語は検索できないことがある // （ngramのトークン長設定に依存） // ・LIKE ほど柔軟な部分一致でない場合も // → 要件に応じて LIKE 併用もあり // → 「銀の弾丸はない」（別冊S2）★ // ─── 本シリーズの方針 ─── // 学習目的で FULLTEXT を実装・体験 // 「性能改善の流れ」を一通り経験する // → 実務で判断できる土台を作る ★
POINT — FULLTEXT も「銀の弾丸」ではない

FULLTEXT は速いですが万能ではありません。短い単語が検索できない、LIKE ほど柔軟な部分一致が苦手、設定が複雑、H2 でテストできない——といったトレードオフがあります（別冊S2）。「速いから常に FULLTEXT」ではなく「データ量・検索頻度・要件で選ぶ」。少量データなら LIKE のシンプルさが勝つことも。道具の長所と短所を知って選ぶのが設計です。

06

実装

PHASE: IMPLIMPLEMENTATION

設計に沿って実装。DDL（FULLTEXT追加）とMapper の SQL（LIKE→MATCH AGAINST）。Vol.7 の動的SQLのキーワード部分だけを差し替えます。カテゴリ・タグの条件はそのまま。

DDL と Mapper の変更キーワード部分を差し替え
// 変更箇所

変えるのはキーワード検索の部分だけ。Vol.7 の動的SQLの構造（<if><where>）は維持し、LIKE を MATCH AGAINST に置き換え。

// 作業手順
  1. DDL で memos に FULLTEXT インデックス追加（ngram）
  2. MemoMapper.xml のキーワード <if> を MATCH AGAINST に
  3. カテゴリ・タグの条件はそのまま
// 既存構造を活かす

SearchForm・Service・Controller・View は変更不要。Mapper の SQL だけ変える＝影響範囲が小さい（別冊S2の低結合の恩恵）。

// 実装の変更 // ─── DDL（FULLTEXT追加）─── ALTER TABLE memos ADD FULLTEXT INDEX ft_memo (title, content) WITH PARSER ngram; // ─── MemoMapper.xml（キーワード部分）─── <!-- Vol.7（LIKE）--> <!-- <if test="keyword != null and keyword != ''"> AND (m.title LIKE CONCAT('%',#{keyword},'%') OR m.content LIKE CONCAT('%',#{keyword},'%')) </if> --> <!-- Vol.8（FULLTEXT）★★★ --> <if test="keyword != null and keyword != ''"> AND MATCH(m.title, m.content) AGAINST(#{keyword} IN BOOLEAN MODE) </if> // → #{} で安全（別冊S6）★ // → カテゴリ・タグの<if>はそのまま // ─── 変更不要なもの ─── SearchForm → そのまま MemoService → そのまま Controller → そのまま search.html → そのまま // → Mapper の SQL だけ変更 ★★★ // → 別冊S2「低結合」の恩恵 // = 影響範囲が小さい // ─── IN BOOLEAN MODE とは ─── // FULLTEXT の検索モードの一つ // 部分的なマッチや演算子(+/-)が使える // → 柔軟な検索が可能
POINT — 変更が「Mapper の SQL だけ」で済む設計の価値

LIKE→FULLTEXT という大きな変更なのに、変えるのはMapper の SQL だけ。SearchForm も Service も Controller も View も無傷。これは Vol.1 から積み上げた3層構造・低結合（別冊S2）のおかげ。「検索のロジックは Mapper に閉じ込めた」から、実装方法を変えても他に波及しない。良い設計は、後の変更を局所化する——その恩恵を実感する場面です。

07

テスト（性能と正しさ）

PHASE: TESTTEST

FULLTEXT は H2 で動かないため、テスト戦略を見直します。正しさは Vol.7 のテストを実MySQL（Testcontainers）で実行。性能は手動測定。別冊S3の「H2 vs Testcontainers」の判断が活きます。

Testcontainers で実MySQLテストFULLTEXTはH2不可
// なぜ Testcontainers か

FULLTEXT・ngram はMySQL固有機能で H2 にない（別冊S3）。だから検索テストは実MySQLが必要。Testcontainers で Docker の MySQL を起動してテストします。

// 正しさのテスト

Vol.7 の検索テストを実MySQL で実行。FULLTEXT に変えても同じ結果が返ることを確認（@AC7）。

// 段階的なテスト方針
  • カテゴリ・タグ検索：H2 でもOK（FULLTEXT不要）
  • キーワード検索：Testcontainers（実MySQL）
// Testcontainers での結合テスト // ─── 依存追加（build.gradle）─── testImplementation 'org.testcontainers:mysql' // ─── MemoFulltextTest.java ─── @SpringBootTest @Testcontainers class MemoFulltextTest { @Container static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0"); // → Docker で本物の MySQL を起動 ★★★ @DynamicPropertySource static void props( DynamicPropertyRegistry r) { r.add("spring.datasource.url", mysql::getJdbcUrl); // 起動したMySQLに接続 } @Autowired MemoMapper mapper; // @AC7: FULLTEXTでも結果は同じ @Test void キーワードで絞り込める() { // Vol.7 と同じ検証内容 SearchForm form = new SearchForm(); form.setKeyword("会議"); var result = mapper.search(form); // 「会議」を含むメモだけ（Vol.7と同じ） assertThat(result) .extracting(Memo::getTitle) .contains("会議メモ"); // → FULLTEXTでも結果同じ ★★★ // → 最適化で壊していない証明 } } // ─── 性能の手動測定 ─── // 大量データ投入スクリプトで1万件作成 // LIKE版とFULLTEXT版の時間を比較： // LIKE: ~800ms // FULLTEXT: ~20ms // → 改善を数値で確認（@AC6）★ // → 自動性能テストは環境依存で不安定 // なので手動測定が現実的（別冊S3）
POINT — 「実DB固有機能」のテストは Testcontainers

別冊S3で触れた「H2 vs Testcontainers」の判断が、ここで現実に。FULLTEXT・ngram はMySQL固有で H2 では動かない。こういう「本番DB固有の機能」を使う部分は、H2 では不十分でTestcontainers（実MySQL）が必要。一方、カテゴリ・タグ検索など標準SQLの部分は H2 で十分速い。「テストしたい対象に応じてDBを選ぶ」——速さと忠実さのトレードオフ（別冊S3）の実践です。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザで確認。検索結果が Vol.7 と同じか、速くなったかを確認。Vol.8 で踏みやすい FULLTEXT 特有の地雷を5つ先回りします。

Vol.8 ハマりどころ 5選FULLTEXT特有の地雷

① 日本語がヒットしない ★★★

  • 原因：ngram パーサ指定なしで FULLTEXT 作成
  • 対処：WITH PARSER ngram を付けて作り直す

② H2 のテストで FULLTEXT が動かない ★★★

  • 原因：H2 は MATCH AGAINST 非対応
  • 対処：キーワード検索のテストは Testcontainers（Section 07）

③ 短い単語が検索できない

  • 原因：ngram_token_size より短い語
  • 対処：トークンサイズ設定を確認／LIKE 併用も検討

④ 既存データが検索に出ない

  • 原因：FULLTEXT 索引作成前のデータ
  • 対処：通常は自動で索引化されるが、確認・再構築

⑤ AGAINST に直接ユーザー入力を文字列連結

  • 原因：${} で埋め込み→インジェクション
  • 対処：必ず #{keyword}（別冊S6）
