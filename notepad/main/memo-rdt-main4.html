01

要求の確認（US-002の続き）

PHASE: REQREVISIT REQUIREMENT

Vol.3 で categories マスタを作りましたが、まだメモと繋がっていません。US-002 の「カテゴリで分類」を本当に実現するには、メモにカテゴリを紐付ける必要があります。Vol.4 は US-002 の後半戦です。

US-002 の完成に必要なものマスタだけでは足りない
// Vol.3 で足りなかったこと

Vol.3 では categories の箱を作りました。でも US-002「メモをカテゴリで分類したい」には、メモとカテゴリの紐付けが必要。これが Vol.4 のスコープ。

// 要求を分解
  • メモにカテゴリを設定できる（memos に category_id）
  • 一覧でカテゴリ名を表示（JOIN）
  • カテゴリで絞り込み（WHERE）
// 新しい受入条件の追加

US-002 に絞り込みの受入条件を追加します。要求は進化する（別冊S1の変更管理）。Vol.3 時点では「分類できる」だったのが、Vol.4 で「絞り込める」に育ちます。

// US-002 の受入条件（Vol.4で追加分） @AC5 カテゴリ設定（正常系） Given: カテゴリ「仕事」が存在する When : メモ作成時に「仕事」を選ぶ Then : メモにカテゴリが紐付く @AC6 カテゴリ名の表示 Given: 「仕事」カテゴリのメモがある When : 一覧を表示する Then : メモ行に「仕事」と表示される @AC7 未分類の表示 Given: カテゴリ未設定のメモがある When : 一覧を表示する Then : 「未分類」と表示される（エラーにしない） @AC8 カテゴリ絞り込み Given: 複数カテゴリのメモがある When : 「仕事」で絞り込む Then : 「仕事」のメモだけ表示される // ─── @AC7 が設計を左右する ★★★ ─── // 「未分類」を許すか？ // → 許す（category_id は NULL 可） // → だから LEFT JOIN（後述） // → 要求が JOIN の種類を決める
POINT — 「未分類を許すか」は要求で決まる

@AC7「未分類のメモを表示する」が、設計を大きく左右します。「メモは必ずカテゴリを持つ」なら INNER JOIN・NOT NULL。「未分類もOK」ならLEFT JOIN・NULL許容。本シリーズは「思いついたらすぐメモ（カテゴリは後で）」というペルソナ（田中さん）に合わせ、未分類を許す。要求がJOINの種類を決めるのです。

PHASE: REQ→TESTAC TO TEST CASES

JOIN・絞り込みの受入条件をテストケースに変換します。これらは「DBの動作」なので、ほとんど結合テスト（@MybatisTest + H2）になります。別冊S3の「JOIN結果は実DBで検証」がそのまま活きます。

テストケース表と層振り分けJOIN系は結合テスト
// テストケース一覧
検証期待層
カテゴリ名が取れる「仕事」結合
未分類のメモNULL/未分類結合
カテゴリ絞り込み該当のみ結合

JOIN・絞り込みの結果は実DBでないと検証できない（別冊S3）。Service の「絞り込みありなしの分岐」だけ単体で。

// 受入条件 → テスト層の振り分け // ─── 結合テスト（@MybatisTest+H2）─── @AC6 カテゴリ名表示 → JOINの結果を実DBで確認 @AC7 未分類の表示 → LEFT JOINでNULLになるか確認 @AC8 絞り込み → WHERE句が効くか確認 // → 全部「SQLの動作」 ★★★ // → モックでは検証不可 // → 実DBの結合テストが必須 // ─── 単体テスト（Mockito）─── Service の絞り込み分岐： categoryId が null → findAll() categoryId あり → findByCategory() // → ロジックの分岐はモックで検証 ★ // → 「正しいMapperメソッドを呼ぶか」 // ─── 別冊S3の判断がそのまま ─── // 「何が正しさを守るか」で層を選ぶ： // JOIN・WHERE = SQL（DB）→ 結合 // 分岐ロジック = Java → 単体

JOIN は「2つのテーブルをどう繋ぐか」の指定。少しのミス（ON句の間違い、JOINの種類間違い）で結果が激変します。しかもモックでは検証不可（別冊S3）。だからJOIN を書いたら必ず実DB（H2）で結合テスト。「想定した行数・想定したカテゴリ名が取れるか」を確認するのが鉄則です。

03

仕様化：JOINと絞り込み

PHASE: SPECSPECIFICATION

要求を技術仕様に。一覧表示の SQL（JOIN）と絞り込みの URL 設計を決めます。「未分類も表示」（@AC7）からLEFT JOINが確定します。

JOIN の種類と URL 設計LEFT JOIN が要求から決まる
// なぜ LEFT JOIN か
JOIN種類未分類メモは
INNER JOIN消える（NG）
LEFT JOIN残る（@AC7に合致）

@AC7「未分類も表示」を満たすにはLEFT JOIN。INNER だと category_id が NULL のメモが消えてしまう。要求が JOIN を決める。

// 絞り込みの URL 設計
  • 全件：GET /memos
  • 絞り込み：GET /memos?categoryId=1
  • クエリパラメータで絞り込み条件を渡す
// 仕様：SQL と画面 // ─── 一覧の SQL（LEFT JOIN）─── SELECT m.id, m.title, m.content, m.category_id, c.name AS category_name, c.color AS category_color FROM memos m LEFT JOIN categories c ON m.category_id = c.id ORDER BY m.updated_at DESC; // → 未分類（category_id=NULL）も // category_name=NULL で残る ★★★ // ─── 絞り込みの SQL ─── // categoryId 指定時のみ WHERE 追加 ... WHERE m.category_id = #{categoryId} // ─── 完成画面イメージ ─── ┌──────────────────────────────┐ │ メモ一覧 │ │ [全部▼] ← カテゴリで絞り込み │ ├──────────────────────────────┤ │ タイトル │ カテゴリ │更新 │ ├──────────────────────────────┤ │ 会議メモ │●仕事 │11:23 │ │ 買い物 │（未分類）│10:05 │ ←@AC7 │ Java学習 │●学習 │09:30 │ └──────────────────────────────┘ ↑ @AC6 カテゴリ名表示 ↑ @AC7 未分類も消えない
POINT — INNER と LEFT を間違えると要求違反

INNER JOINは「両方に存在する行だけ」、LEFT JOINは「左（memos）は全部残す」。もし INNER で書くと、カテゴリ未設定のメモが一覧から消える——@AC7 違反です。「未分類も見せたい」という要求が LEFT JOIN を要求している。この対応を意識すると、JOINの選択を間違えません。

04

設計①：関連とデータ取得

PHASE: DESIGNRELATION & FETCH

memos に category_id を追加し、Memo が Category を持つデータ構造に設計します。Vol.1 の DDL（FK・ON DELETE SET NULL）がここで活きます。「メモを消してもカテゴリは残る、カテゴリを消してもメモは残る」方針です。

Memo に Category を持たせるオブジェクトの関連
// Memo エンティティの拡張

Memo に2つ追加：

  • categoryId：Long・FK（DBの値）
  • category：Category 型・JOINで取得した実体

categoryId は「参照」、category は「実体」。一覧表示では category.name を使います。

// DDL の変更（Vol.1で設計済み）

memos に category_id を追加。ON DELETE SET NULL＝カテゴリ削除時、メモの category_id を NULL に（メモは残す）。Vol.1 の削除戦略が効きます。

// Memo.java の拡張 + DDL @Data public class Memo { private Long id; @NotBlank @Size(max = 200) private String title; private String content; // ↓ Vol.4 で追加 private Long categoryId; // FK（参照） private Category category; // 実体（JOIN） private LocalDateTime createdAt; private LocalDateTime updatedAt; } // ─── DDL：memos に FK 追加 ─── ALTER TABLE memos ADD COLUMN category_id BIGINT, ADD FOREIGN KEY (category_id) REFERENCES categories(id) ON DELETE SET NULL; // ↑ カテゴリ削除→メモのcategory_id // をNULLに（メモは残る）★★★ // ─── 削除戦略の確認（Vol.1設計）─── カテゴリ「仕事」を削除 ↓ ON DELETE SET NULL 「仕事」だったメモ → category_id=NULL ↓ 一覧では「未分類」と表示（@AC7） // → メモは絶対に消えない ★★★ // → 「メモは大事」という設計思想
POINT — categoryId（参照）と category（実体）を分ける

Memo に2つの「カテゴリ」を持たせます。categoryId（保存・更新で使う FK の値）とcategory（JOIN で取得したカテゴリの実体）。「保存するときは id、表示するときは実体」。この分離が、フォーム送信（id を受け取る）と一覧表示（名前を出す）の両方を自然に扱えます。

05

設計②：resultMapとassociation

PHASE: DESIGNRESULTMAP & ASSOCIATION

JOIN で取得した「メモ＋カテゴリ」を1つの Memo オブジェクトに組み立てる仕組みが、MyBatis の resultMap と association。1対多（正確には多対1）の関連をオブジェクトにマッピングする設計です。

resultMap で JOIN 結果を組み立てるassociation = 1つの関連
// なぜ resultMap が必要か

JOIN の結果はフラットな1行（memo列＋category列）。これを Memo オブジェクト（中に Category を持つ）に組み立てるには、「どの列がどのプロパティか」のマッピング指示が要る。それが resultMap。

// association とは
  • association：1つの関連オブジェクト（Memo→1つのCategory）
  • collection：複数の関連（後の Vol.5 タグで登場）

Vol.4 は多対1（メモ→1カテゴリ）なのでassociationを使います。

// resultMap + association（XML） // ─── JOIN結果（フラット）─── id│title │category_id│cat_name│cat_color 1 │会議メモ│1 │仕事 │#d63131 2 │買い物 │NULL │NULL │NULL // ↓ これを Memo オブジェクトに組み立てる // ─── MemoMapper.xml ─── <resultMap id="MemoWithCategory" type="Memo"> <id property="id" column="id"/> <result property="title" column="title"/> <result property="content" column="content"/> <result property="categoryId" column="category_id"/> <!-- ↓ 関連オブジェクト --> <association property="category" javaType="Category"> <id property="id" column="category_id"/> <result property="name" column="category_name"/> <result property="color" column="category_color"/> </association> </resultMap> // → JOIN1行が Memo(+Category) に ★★★ // → 未分類は category=null（NULLなら作らない） // ─── 使う select ─── <select id="findAllWithCategory" resultMap="MemoWithCategory"> SELECT m.*, c.name AS category_name, c.color AS category_color FROM memos m LEFT JOIN categories c ON m.category_id = c.id ORDER BY m.updated_at DESC </select>
POINT — association は「多対1」、collection は「1対多」

MyBatis の関連マッピングは2種類。association＝1つの関連（メモは1つのカテゴリを持つ＝多対1）。collection＝複数の関連（メモは複数のタグを持つ＝1対多/多対多、Vol.5）。「単数なら association、複数なら collection」と覚えればOK。Vol.4 は単数なので association です。

Mapper と Service（絞り込み分岐）動的に WHERE を足す
// Mapper の追加メソッド
  • findAllWithCategory()：全件（JOIN）
  • findByCategoryId(Long)：絞り込み（JOIN＋WHERE）
// Service の絞り込み分岐

// CategoryMapper / Service の追加 // ─── MemoMapper.java（追加）─── List<Memo> findAllWithCategory(); List<Memo> findByCategoryId( Long categoryId); // ─── MemoService（絞り込み分岐）─── public List<Memo> findForList( Long categoryId) { if (categoryId == null) { // 絞り込みなし → 全件 return mapper.findAllWithCategory(); } // 絞り込みあり return mapper.findByCategoryId(categoryId); } // → この分岐を単体テストで検証 ★ // ─── findByCategoryId の XML ─── <select id="findByCategoryId" resultMap="MemoWithCategory"> SELECT m.*, c.name AS category_name, c.color AS category_color FROM memos m LEFT JOIN categories c ON m.category_id = c.id WHERE m.category_id = #{categoryId} ORDER BY m.updated_at DESC </select>
Controller と View（絞り込みUI）@RequestParam とドロップダウン
// Controller の絞り込み受け取り

@RequestParam(required=false) Long categoryId で絞り込み条件を受け取る。必須でないので、なければ null＝全件。

// View の2要素
  • 絞り込みドロップダウン：カテゴリ選択で再表示
  • メモ作成フォーム：カテゴリ選択（select）追加
// MemoController + View @GetMapping public String list( @RequestParam(required = false) Long categoryId, Model model) { model.addAttribute("memos", service.findForList(categoryId)); // 絞り込みドロップダウン用に // 全カテゴリも渡す model.addAttribute("categories", categoryService.findAll()); model.addAttribute("selectedId", categoryId); return "memos/list"; } // ─── list.html 絞り込みUI ─── <form method="get" action="/memos"> <select name="categoryId" onchange="this.form.submit()"> <option value="">すべて</option> <option th:each="c : ${categories}" th:value="${c.id}" th:text="${c.name}" th:selected= "${c.id == selectedId}"> </option> </select> </form> // ─── 一覧でカテゴリ名表示（@AC6,7）─── <td th:text= "${memo.category != null ? memo.category.name : '未分類'}"></td> // → category が null なら「未分類」★★★ // → @AC7 をビューで実装 // ─── new.html カテゴリ選択 ─── <select th:field="*{categoryId}"> <option value="">（未分類）</option> <option th:each="c : ${categories}" th:value="${c.id}" th:text="${c.name}"></option> </select>

// MemoServiceFilterTest.java（単体） @ExtendWith(MockitoExtension.class) class MemoServiceFilterTest { @Mock MemoMapper mapper; @InjectMocks MemoService service; // 絞り込みなし → findAll を呼ぶ @Test void categoryIdがnullなら全件取得() { service.findForList(null); verify(mapper) .findAllWithCategory(); verify(mapper, never()) .findByCategoryId(any()); // → 正しいメソッドを呼んだか ★ } // 絞り込みあり → findByCategoryId @Test void categoryId指定なら絞り込み() { service.findForList(1L); verify(mapper) .findByCategoryId(1L); verify(mapper, never()) .findAllWithCategory(); } } // → 分岐ロジックはモックで十分 ★★★ // → DBの動作（JOIN）は結合テストで // → 別冊S3の使い分けを実践
POINT — 同じ機能を「2つの層」でテストする意味

絞り込み機能を、単体（分岐ロジック）と結合（SQL動作）の両方でテストしました。一見二度手間ですが、役割が違います。単体は「正しいメソッドを選ぶか」、結合は「そのSQLが正しい結果を返すか」。分業することで、失敗時に「ロジックの問題か、SQLの問題か」が即わかる。これがテストピラミッドの実践です。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザで確認。カテゴリ選択・名前表示・未分類表示・絞り込みを目視。Vol.4 で踏みやすい JOIN 特有の地雷を5つ先回りします。

Vol.4 ハマりどころ 5選JOIN特有の地雷

① カテゴリ名が表示されない（null）★★★

  • 原因：resultMap の column 名と SQL の別名が不一致
  • 対処：c.name AS category_name と column="category_name" を一致させる

② findAll が resultType のままで JOIN が効かない

  • 原因：select に resultType="Memo" を指定（resultMap でない）
  • 対処：resultMap="MemoWithCategory" に変更

③ 未分類メモが一覧から消える ★★★

  • 原因：INNER JOIN で書いた（@AC7違反）
  • 対処：LEFT JOIN に修正（結合テストで検知できる）

④ 絞り込みドロップダウンで全部消える

  • 原因：「すべて」の option の value が空でない／categoryId に空文字が渡る
  • 対処：value="" を空に、Controller で空文字を null 扱い

⑤ 編集時にカテゴリが選択済みにならない

  • 原因：th:field を使っていない／selected 未指定
  • 対処：th:field="*{categoryId}" で自動選択
