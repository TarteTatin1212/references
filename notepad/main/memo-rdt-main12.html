04

設計①：バリデーション

PHASE: DESIGNVALIDATION DESIGN

別冊S11のBean Validationを本格適用します。フォームクラスに@NotBlank/@Size 等を宣言、@Validated で自動検証。これまで断片的だった検証を体系化。インポート（Vol.11）にも同じルールを適用します。

Bean Validation の本格適用宣言的に検証
// フォームに検証を宣言（別冊S11）

各フォームクラスに検証ルールをアノテーションで宣言。検証ロジックを毎回書かない（KISS）。

// 検証ルール（メモ帳）
  • タイトル：必須・100文字以内
  • 本文：2000文字以内
  • カテゴリ名：必須・50文字以内
// 親切なメッセージ（別冊S11）

各ルールに「直し方が分かるメッセージ」を付ける。「タイトルは100文字以内で入力してください」。

// バリデーションの設計 // ─── MemoForm に検証を宣言（別冊S11）─── import jakarta.validation.constraints.*; @Data public class MemoForm { @NotBlank(message = "タイトルを入力してください") @Size(max = 100, message = "タイトルは100文字以内で入力してください") private String title; @Size(max = 2000, message = "本文は2000文字以内で入力してください") private String content; @NotNull(message = "カテゴリを選択してください") private Long categoryId; } // → ルールを宣言、親切なメッセージ ★★★ // ─── CategoryForm も同様 ─── @NotBlank(message = "カテゴリ名は必須") @Size(max = 50) private String name; // ─── Controller で @Validated ─── @PostMapping public String create( @Validated @ModelAttribute MemoForm form, BindingResult br, // 検証結果（直後に置く） Model model) { if (br.hasErrors()) { // 検証エラー → 入力画面に戻す（@AC1） setupForm(model); return "memos/form"; } service.create(form); return "redirect:/memos"; } // → サーバ側で必ず検証（別冊S11・@AC5）★★★ // ─── インポートにも同じルール（Vol.11）─── // Validator を使い、インポートデータも // 同じ @NotBlank/@Size で検証できる // → 「入力源が違っても同じルール」（別冊S11）★
POINT — BindingResult は検証対象の「直後」に置く

Spring の落とし穴：@Validated を付けた引数の直後に BindingResult を置く必要があります。create(@Validated MemoForm form, BindingResult br) の順序が重要。間に他の引数を挟むと検証エラーで例外が飛ぶ（400エラー）。「検証対象とBindingResultはペア」と覚える。これは初学者がハマりやすい点（Section 08）。順序を守れば、br.hasErrors() で検証結果を扱えます。

設計②：例外の集約

PHASE: DESIGNEXCEPTION HANDLING

別冊S7の集大成。業務例外を独自例外クラス（RuntimeException継承）で定義し、@ControllerAdvice で一括ハンドリング。Vol.3 のUNIQUE違反500エラーを、ここで親切なエラー画面に変えます。

@ControllerAdvice で例外を集約（別冊S7）白い500を親切な画面に
// 独自例外クラス（別冊S7）

RuntimeException を継承した業務例外を定義。MemoNotFoundException、DuplicateCategoryException 等。右クリック→New→Classで作成。

// @ControllerAdvice で集約（別冊S7）

全Controllerの例外を1か所で捕まえる。例外の種類に応じた親切なエラー画面を返す。Controller は try-catch 不要。

// 表示とログの分離（別冊S7）
  • ユーザー：分かりやすいメッセージ
  • ログ：技術詳細・スタックトレース
// 例外の集約（別冊S7） // ─── 独自例外（RuntimeException継承）─── public class MemoNotFoundException extends RuntimeException { // 非チェック public MemoNotFoundException(Long id) { super("メモが見つかりません: id=" + id); } } // → RuntimeException継承（別冊S7）★ // → @Transactionalで自動ロールバック // ─── @ControllerAdvice で集約（別冊S7）─── import org.springframework.web.bind .annotation.*; @Slf4j @ControllerAdvice // 全Controller横断 ★★★ public class GlobalExceptionHandler { // メモ未検出 → 404（@AC3） @ExceptionHandler( MemoNotFoundException.class) @ResponseStatus(HttpStatus.NOT_FOUND) public String handleNotFound( MemoNotFoundException e, Model model) { log.warn(e.getMessage()); // ログ model.addAttribute("message", "メモが見つかりませんでした"); return "error/404"; // 親切な画面 } // カテゴリ重複 → 親切に（@AC2）★★★ // Vol.3の500エラーをここで解消！ @ExceptionHandler( DuplicateCategoryException.class) public String handleDuplicate( DuplicateCategoryException e, Model model) { model.addAttribute("message", "そのカテゴリ名は既に存在します"); return "error/conflict"; } // 想定外 → 汎用＋詳細隠す（@AC4）★★★ @ExceptionHandler(Exception.class) public String handleAll( Exception e, Model model) { log.error("想定外エラー", e); // 詳細はログ model.addAttribute("message", "エラーが発生しました"); // 簡潔に return "error/500"; } } // → 全例外を集約、親切な画面＋ログ ★★★
POINT — Vol.3 の「500エラー」が、ここでついに解消される

本シリーズの伏線回収：Vol.3 でカテゴリ名を重複させると出た「白い500エラー」を、Vol.12 でついに親切なエラー画面に変えます。当時は「これは Vol.12 で解消します」と先送りにしていました（別冊S7でも触れた）。DataIntegrityViolationException や独自の DuplicateCategoryException を @ControllerAdvice で捕まえ、「そのカテゴリ名は既に存在します」と表示。長く引っ張った伏線が、守りの集大成として回収されます。

例外クラスとServiceの実装投げる側を作る

  1. exception パッケージを作成
  2. 右クリック→New→Class で例外クラス
  3. RuntimeException を継承（別冊S7）
  4. Service で適切に例外を投げる

error/ 配下に404・conflict・500の画面を作成。分かりやすいメッセージ＋戻るリンク（別冊S7,S8）。

POINT — Controller は「投げっぱなし」、表示は集約に任せる（別冊S7）

実装の要点は「Service は例外を投げるだけ、Controller は catch しない、表示は @ControllerAdvice」という役割分担（別冊S7）。各 Controller に try-catch を書くとコードが重複し汚れる。代わりに例外を投げっぱなしにし、@ControllerAdvice が一括で親切な画面に変換。これで Controller は正常系だけを書けてスッキリ。「関心の分離」（別冊S2）の実践です。

07

テスト（正常系と異常系）

PHASE: TESTTEST

バリデーション・例外ハンドリングをMockMvc で検証。異常系（不正入力・例外）が中心。「弾かれるか・親切なエラーか」を確認します。既存テスト（正常系）も緑のままを保ちます。

MockMvc で異常系を検証弾く・親切に
// 検証ポイント
  • @AC1：空タイトル→検証エラー
  • @AC2：重複→親切なエラー画面
  • @AC3：存在しないメモ→404

異常系テストで堅牢性を保証。既存の正常系テストも壊していないか確認。

// MemoValidationTest.java（MockMvc） @WebMvcTest(MemoController.class) class MemoValidationTest { @Autowired MockMvc mockMvc; @MockBean MemoService service; // @AC1 空タイトルは検証エラー @Test void 空タイトルは弾かれる() throws ... { mockMvc.perform(post("/memos") .param("title", "") .param("categoryId", "1")) // 入力画面に戻る（保存しない） .andExpect(view().name( "memos/form")) .andExpect(model() .attributeHasFieldErrors( "memoForm", "title")); // → 空タイトルが弾かれる（@AC1）★★★ // → service.create は呼ばれない verify(service, never()) .create(any()); } // @AC3 存在しないメモは404 @Test void 存在しないメモは404() throws ... { // Service が例外を投げる設定 when(service.findById(99999L)) .thenThrow( new MemoNotFoundException(99999L)); mockMvc.perform(get("/memos/99999")) .andExpect(status().isNotFound()) .andExpect(view().name( "error/404")); // → 404画面（白い500でない）★★★ // → @ControllerAdvice が効いている } // @AC1（正常）正しい入力は通る @Test void 正しい入力は保存される() throws ... { mockMvc.perform(post("/memos") .param("title", "正常なメモ") .param("categoryId", "1")) .andExpect(redirectedUrl( "/memos")); verify(service).create(any()); // → 正常系も維持（壊していない）★ } }
POINT — 「正しい入力は通る」も必ずテストする

異常系テスト（弾く）に集中するあまり、「正しい入力はちゃんと通る」のテストを忘れがち。検証を厳しくしすぎて「正常な入力まで弾く」バグもある。だから「不正は弾く」と「正常は通す」の両方をテストする。「空タイトルは弾く」かつ「正しいタイトルは保存される」。守りを固めても、正常な利用を妨げない——これを両方のテストで保証します。

08

Vol.12 ハマりどころ 5選検証・例外特有の地雷

① BindingResult の位置ミス ★★★

  • @Validated の直後に置かないと400エラー
  • 対処：検証対象の直後に BindingResult（Section 04）

② 例外を握りつぶす ★★★

  • catch して何もしない（別冊S7）
  • 対処：投げて @ControllerAdvice に任せる（Section 05）

③ 画面検証だけでサーバ検証なし ★★★

  • JS回避で不正データが通る（別冊S11）
  • 対処：@Validated でサーバ検証（Section 04）

④ エラー画面に技術詳細を露出

  • スタックトレースが見える（別冊S7）
  • 対処：ユーザーには簡潔に、詳細はログ（Section 05）

⑤ @ControllerAdvice が効かない

  • 例外がチェック例外、または catch済み
  • 対処：RuntimeException継承、catchしない（別冊S7）
