01

要求の確認（新US-005）

PHASE: REQNEW REQUIREMENT

いつも通りまず要求から。ただし今回は新しい要求（追加US）。当初の要求定義（Vol.1）になかった、使い込んで生まれたニーズです。別冊S1の「要求は進化する・変更管理」の実践として、新US-005を定義します。

新US-005 と背景使って生まれた要求
// なぜ新しい要求が生まれたか

US-001〜004 で基本機能は完成。使ってみると「メモが増えて、よく使うものが探しにくい」「古いメモが邪魔だが消したくない」という新たな不満が出てきました。これが新しい要求の源泉（別冊S1）。

// 新US-005
US-005 / SHOULD
メモを整理して使いやすくしたい
As a メモが増えたユーザー
I want 重要なメモを固定し、不要なメモを隠したい
so that よく使うメモにすぐ辿り着ける
// 3つの機能に分解
  • ピン留め：重要メモを上に固定
  • アーカイブ：不要メモを隠す（消さない）
  • 並び替え：好きな順で見る
// 新US-005 の受入条件 @AC1 ピン留め Given: 通常のメモがある When : ピン留めする Then : そのメモが一覧の最上部に来る @AC2 ピン解除 Given: ピン留めされたメモ When : ピンを解除する Then : 通常の位置（更新順）に戻る @AC3 アーカイブ（論理削除）★ Given: 通常のメモがある When : アーカイブする Then : 通常一覧から消えるが、データは残る （アーカイブ一覧で見れる） @AC4 アーカイブ解除（復元） Given: アーカイブされたメモ When : 復元する Then : 通常一覧に戻る @AC5 アーカイブは検索にも出ない ★★★ Given: アーカイブされたメモ When : 検索する Then : 検索結果に出ない（論理削除の徹底） @AC6 並び替え Given: 複数のメモがある When : 並び順を変える（更新/作成/名前） Then : 指定順で表示される // ─── @AC3,5 が論理削除の核心 ★★★ ─── // 「消えるが残る」「検索にも出ない」 // → ソフトデリート（別冊S8）

当初の要求定義（Vol.1）にピン留めやアーカイブがなかったのは失敗ではありません。使ってみて初めて分かる要求は必ずある（別冊S1の変更管理）。むしろ「最初に全部予測しよう」とすると過剰設計（YAGNI＝別冊S2）。動くものを作り、使い、足りないものを足す。この健全な進化が、新US-005を生みました。要求は育つものです。

02

受入条件をテストケース化

PHASE: REQ→TESTAC TO TEST CASES

新US-005の受入条件をテストに。特に重要なのが@AC5「アーカイブは検索にも出ない」。別冊S8で学んだ論理削除の「条件付け忘れ」を、テストで防ぎます。これは結合テストで検証します。

テストケースと論理削除の検証漏れ防止が肝
// テストケース一覧
検証期待層
ピン留め→上に来る最上部結合
アーカイブ→通常一覧に出ない除外結合
アーカイブ→検索にも出ない除外結合
復元→通常一覧に戻る復活結合
並び替え指定順結合

@AC5（検索にも出ない）が最重要テスト。論理削除は全クエリで除外が必要なので、検索でも漏れないか確認（別冊S8）。

// 受入条件 → テスト層 // ─── 全部結合テスト（@MybatisTest+H2）─── // 状態・並び順はSQLの動作 → 実DBで @AC1 ピン留め → ORDER BY pinned DESC @AC3 アーカイブ → 通常一覧で archived=false @AC5 アーカイブは検索にも出ない ★★★ → 検索SQLにも archived=false が必要 → 「アーカイブしたメモが検索に出ない」 を明示的にテスト // ─── @AC5 が論理削除の安全網 ─── // 別冊S8の「条件付け忘れ」を防ぐ @Test void アーカイブ済みは検索に出ない() { // 「会議」を含むメモを2件用意 // 1件をアーカイブする // → 「会議」で検索 // → アーカイブしてない1件だけ出る ★★★ } // → もし検索SQLに archived 条件を // 付け忘れたら、このテストが落ちる // → 漏れを自動検出 ★★★ // ─── 並び替えのテスト ─── @AC6 並び替え → 名前順なら ORDER BY title ASC → 結果の順序を assertThat で検証 → containsExactly（順序も含めて検証）
POINT — 論理削除は「除外されること」を全箇所でテスト

論理削除（アーカイブ）のテストで最重要なのは「アーカイブ済みが、あらゆる一覧・検索から除外される」こと。通常一覧だけでなく検索結果（@AC5）でも除外を確認。別冊S8で学んだ「条件付け忘れ」は、このテストがあれば自動で検出できる。「除外されるべきものが、ちゃんと除外されるか」——消極的な確認こそ、論理削除の安全網です。

UI設計のポイント：アーカイブと削除を別物として見せる。アーカイブは「とりあえず隠す（戻せる）」、削除は「本当に消す（戻せない）」。ボタンも分け、アーカイブは気軽に、削除は確認付きで。別冊S8の「消す前に隠す」がこの設計。ユーザーは「アーカイブなら気軽に押せる」と分かり、メモ整理が捗ります。操作の取り返しのつかなさを、UIの重さに反映させます。

設計①：状態列とアーカイブ

PHASE: DESIGNSTATE COLUMNS

別冊S8で学んだ状態管理を実装に落とします。memos にpinned/archived/sort_order 列を追加。アーカイブは論理削除（フラグ）で実現。状態の更新メソッドを設計します。

状態列の追加と更新真偽値で状態管理
// DDL：状態列を追加
  • pinned：BOOLEAN（ピン留め）
  • archived：BOOLEAN（アーカイブ＝論理削除）
  • sort_order：INT（手動並び順・任意）

別冊S8通り2状態は真偽値でシンプルに（KISS）。

// 状態更新のMapper

状態だけを更新する軽量なUPDATEメソッド。メモ全体を更新せず、該当の列だけ変える。

// 状態列の設計 // ─── DDL（別冊S8 Section 03）─── ALTER TABLE memos ADD pinned BOOLEAN NOT NULL DEFAULT false, ADD archived BOOLEAN NOT NULL DEFAULT false, ADD sort_order INT NOT NULL DEFAULT 0; // → 既存メモは全て false（通常状態）★ // ─── Memo クラスに追加 ─── public class Memo { // ... 既存フィールド private boolean pinned; private boolean archived; private int sortOrder; } // ─── 状態更新の Mapper ─── // ピン留め切替 <update id="updatePinned"> UPDATE memos SET pinned = #{pinned} WHERE id = #{id} </update> // アーカイブ（論理削除） <update id="updateArchived"> UPDATE memos SET archived = #{archived} WHERE id = #{id} </update> // → archived=true でアーカイブ // → archived=false で復元 ★★★ // → DELETE は使わない（論理削除） // ─── アーカイブ = 論理削除（別冊S8）─── 物理削除（既存・確定削除用）: DELETE FROM memos WHERE id=#{id} 論理削除（アーカイブ）: UPDATE memos SET archived=true ... // → 同じ「消す」でも全く別物 ★★★ // → アーカイブは戻せる
POINT — 状態更新は「列だけ」を変える軽量UPDATE

ピン留めやアーカイブの切替は、メモ全体を更新せず、該当の列（pinned/archived）だけを UPDATE します。タイトルや本文を巻き込まないので軽量で安全。「ピン留めボタンを押したら、なぜか本文が消えた」のような事故を防ぐ。「変えたい1列だけを変える」のが、状態更新の鉄則。SQLも SET pinned = ? だけのシンプルなものになります。

05

設計②：並び順とクエリ集約

PHASE: DESIGNSORT & QUERY

別冊S8の複数キーORDER BYと論理削除のクエリ集約を設計します。「ピンを上に、アーカイブを除外」を全一覧クエリで一貫させる。条件付け忘れを防ぐ設計が鍵です。

並び順とクエリの設計条件の一貫性
// 並び順（複数キー・別冊S8）

ピン留め最優先 → 選択された並び順。複数キーの ORDER BY。

// 論理削除の徹底（別冊S8）

全ての通常一覧・検索に archived = false を付ける。付け忘れ防止のため、メソッドを役割で分ける。

  • findActive：通常一覧（archived=false）
  • findArchived：アーカイブ一覧（archived=true）
  • 検索（search）にも archived=false を追加
// 並び順の安全な切替

ユーザーの並び順指定はホワイトリストで安全な値に変換（別冊S6,S8）。${} は使わない。

// 並び順とクエリ集約 // ─── 通常一覧（複数キーORDER BY）─── <select id="findActive" resultMap="MemoFull"> SELECT m.*, ... FROM memos m LEFT JOIN categories c ... WHERE m.archived = false // 論理削除除外 ★★★ ORDER BY m.pinned DESC, // ①ピン優先 ${orderBy} // ②選択された順 </select> // ※ orderBy はホワイトリスト検証済みの値のみ // ─── 並び順のホワイトリスト（別冊S6,S8）─── String orderBy = switch (sort) { case "name" -> "m.title ASC"; case "created" -> "m.created_at DESC"; default -> "m.updated_at DESC"; }; // → ユーザー入力を直接SQLに入れない ★★★ // → 許可した順序だけ通す（安全） // ─── アーカイブ一覧 ─── <select id="findArchived" ...> SELECT ... FROM memos m ... WHERE m.archived = true // 逆の条件 ORDER BY m.updated_at DESC </select> // ─── 検索にも archived 条件（@AC5）★★★ ─── <select id="search" ...> SELECT ... FROM memos m ... <where> AND m.archived = false // 検索でも除外 ★★★ <if test="keyword..."> ... </if> </where> </select> // → 別冊S8「全クエリに条件」の徹底 // → 付け忘れると@AC5テストが落ちる
POINT — メソッド名で「何を取るか」を明示して付け忘れを防ぐ

論理削除の条件付け忘れ（別冊S8の最頻出バグ）を防ぐ設計上の工夫：findActive（通常）/ findArchived（アーカイブ）と名前で役割を明示する。「findAll」のような曖昧な名前だと「archived 条件を付けるべきか」が分からない。名前に意図を込める（別冊S2の良い命名）ことで、使う側も実装側も間違えにくくなります。検索の archived 条件は @AC5 のテストが守ります。

テスト（論理削除の漏れ防止）

PHASE: TESTTEST

状態と論理削除を結合テストで検証。特に@AC5「アーカイブが検索に出ない」を重点的に。別冊S8の「除外されることのテスト」で、論理削除の漏れを防ぎます。

結合テスト — 状態と論理削除除外の検証が肝
// 検証ポイント
  • @AC1：ピン留めが上に来る（順序）
  • @AC3：アーカイブが通常一覧に出ない
  • @AC5：アーカイブが検索にも出ない ★★★
  • @AC4：復元で戻る
  • @AC6：並び替えの順序

@AC5 が論理削除の安全網。検索SQLの archived 条件付け忘れを検出。

Vol.9 ハマりどころ 5選状態管理・論理削除の地雷

① 検索に archived 条件を付け忘れ ★★★

  • アーカイブしたメモが検索に出る
  • 対処：全クエリに条件＋@AC5テスト（別冊S8）

② ピン留めが上に来ない

  • ORDER BY で pinned DESC を先頭にしていない
  • 対処：ORDER BY pinned DESC, ... の順序

③ 並び替えで ${} 直接埋め込み ★★★

  • ユーザーの sort を ORDER BY に直接
  • 対処：ホワイトリストで変換（別冊S6,S8）

④ boolean のデフォルト値漏れ

  • 既存メモの pinned/archived が NULL
  • 対処：DDL で NOT NULL DEFAULT false

⑤ アーカイブを物理削除と混同

  • アーカイブで DELETE してしまう
  • 対処：アーカイブは UPDATE（archived=true）
