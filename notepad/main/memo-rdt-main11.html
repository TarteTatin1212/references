01

要求の確認（データ移行）

PHASE: REQREVISIT REQUIREMENT

いつも通りまず要求から。使い込むと「大事なメモが消えたら困る」「別環境にメモを移したい」というニーズが。これがデータ移行（エクスポート/インポート）の要求。新しいユーザーストーリー US-006 として定義します。

US-006 と背景バックアップと移行
// なぜデータ移行が必要か

メモが増えると「失いたくない」という気持ちが強まります。バックアップ、別環境への移行、他アプリからの取り込み——これらが US-006。

// US-006
US-006 / COULD
メモをバックアップ・移行したい
As a 大事なメモを持つユーザー
I want メモをファイルに書き出し、読み込みたい
so that バックアップや別環境への移行ができる

  • インポート：ファイル→DB（難・検証必須）

POINT — 「往復で元に戻る」がエクスポート/インポートの正しさ

エクスポート/インポートの最も重要な性質が@AC3「往復の一貫性」。エクスポートしたファイルをインポートすると、元のメモが復元される。これが崩れると「バックアップしたのに復元できない」という最悪の事態に。別冊S10の「エクスポートの形式をインポートが受け入れる」を、テストで保証する（Section 07）。往復して一致するか——これがデータ移行の信頼性の核心です。

受入条件をテストに。特に重要なのが@AC3（往復）と@AC4（不正データ検証）。別冊S10の「インポートは検証してから」「部分失敗は報告」を、テストで実証します。

POINT — 「往復テスト」は1つで多くを保証する強力なテスト

@AC3 の往復テスト（export→import→一致）は、1つで「エクスポートが正しい」「インポートが正しい」「両者の形式が合っている」を同時に保証する強力なテスト。もし途中でデータが欠ける・形式がずれるなら、往復で一致しなくなるので検出できる。「ラウンドトリップテスト」とも呼ばれ、シリアライズ/パースの正しさを検証する定番。データ移行機能には必ず入れたいテストです。

エクスポートで悩むのが「どこまで含めるか」。タイトル・本文だけでなく、Vol.9 で追加したカテゴリ・タグ・pinned・archived も含めると、完全なバックアップになる。「往復で完全に元に戻る（@AC3）」ためには、復元に必要な情報を全て含める必要がある。JSON なら構造的に全部入れられる（別冊S10）。「何を保存すれば元に戻るか」を考えて形式を設計します。

04

設計①：エクスポート

PHASE: DESIGNEXPORT DESIGN

まず易しいエクスポートから（別冊S10：易しい方から）。メモを取得し、Jackson で JSON にシリアライズ、ファイルとしてダウンロード。自前で書かずライブラリに任せます（別冊S10）。

エクスポートの設計Jacksonでシリアライズ
// 処理の流れ
  1. 全メモを取得（カテゴリ・タグ込み）
  2. エクスポート用のDTOに変換
  3. Jackson で JSON にシリアライズ
  4. UTF-8 でダウンロードレスポンス
// エクスポート用DTO

Memo エンティティをそのまま出すのでなく、エクスポート用のDTOを作る。「ファイルに出すべき情報だけ」を整理（id等の内部情報は除く判断も）。

// なぜDTOか

エンティティ直出しだと内部構造が漏れる＋将来の変更で形式が壊れる。DTOで「外部に出す形」を固定（別冊S2の関心の分離）。

// エクスポートの設計 // ─── エクスポート用DTO ─── @Data public class MemoExport { private String title; private String content; private String category; // 名前（idでなく） private List<String> tags; // 名前の配列 private boolean pinned; private boolean archived; } // → 「外部に出す形」を固定 ★ // → idは出さない（環境で変わるので） // → カテゴリ/タグは名前で（移行しやすい） // ─── Service：エクスポート ─── public String exportToJson() throws ... { List<Memo> memos = mapper.findAllWithDetails(); List<MemoExport> dtos = memos.stream() .map(this::toExport) .toList(); // Jackson でシリアライズ（別冊S10） return objectMapper .writerWithDefaultPrettyPrinter() .writeValueAsString(dtos); // → List → JSON文字列 ★ } // ─── Controller：ダウンロード ─── @GetMapping("/export") public ResponseEntity<byte[]> export() throws ... { String json = service.exportToJson(); byte[] body = json.getBytes( StandardCharsets.UTF_8); // UTF-8 ★★★ return ResponseEntity.ok() .header("Content-Disposition", "attachment; filename=memos.json") .contentType( MediaType.APPLICATION_JSON) .body(body); // → UTF-8 でダウンロード ★★★ }
POINT — エクスポートは「DTO」で外部形式を固定する

エクスポートで Memo エンティティを直接 JSON 化しないのが定石。理由：①内部構造（id等）が漏れる、②エンティティを変更すると外部形式が勝手に変わる（互換性破壊）。代わりにMemoExport という専用DTOを作り、「外部に出す形」を固定する。これは別冊S2の関心の分離。「内部の都合」と「外部との約束」を分けることで、将来の変更に強いエクスポートになります。

05

設計②：インポートと検証

PHASE: DESIGNIMPORT & VALIDATION

別冊S10の集大成。難しいインポートを設計します。パース→検証→登録→報告の流れ。「外部データは信用しない」（別冊S10）に従い、1行ずつ検証し、不正行を弾いて報告します。

インポートの設計（検証中心）信じる前に確かめる
// インポートの流れ（別冊S10）
  1. ファイルを UTF-8 で読み込み
  2. Jackson で JSON をパース
  3. 1件ずつ検証（必須・長さ等）
  4. 正常な行だけ登録（カテゴリ/タグは名前で解決）
  5. 結果（成功/失敗）を報告
// 検証内容（別冊S10）
  • タイトルが空でないか（@NotBlank相当）
  • 長さは適切か（@Size相当）
// 部分失敗の方針（別冊S9,S10）

正しい行は登録、不正行は理由付きで報告（方針B+C）。「97件成功、3件失敗（理由）」。

// インポートの設計 // ─── インポート結果の型 ─── @Data public class ImportResult { private int successCount; private List<String> errors; // 例: ["行3: タイトルが空", ...] } // ─── Service：インポート（別冊S10）─── @Transactional // 登録をまとめる（別冊S5） public ImportResult importFromJson( MultipartFile file) throws ... { // ① UTF-8 で読み込み（別冊S10）★★★ String json = new String( file.getBytes(), StandardCharsets.UTF_8); // ② パース（Jackson・別冊S10） List<MemoExport> rows = objectMapper.readValue(json, new TypeReference<>(){}); ImportResult result = new ImportResult(); int line = 0; for (MemoExport row : rows) { line++; // ③ 検証（別冊S10）★★★ if (row.getTitle() == null || row.getTitle().isBlank()) { result.addError( "行" + line + ": タイトルが空"); continue; // この行はスキップ } if (row.getTitle().length() > 100) { result.addError( "行" + line + ": タイトルが長すぎ"); continue; } // ④ 登録（カテゴリ/タグは名前で解決） registerMemo(row); result.incrementSuccess(); } // ⑤ 結果を返す（別冊S10：報告）★★★ return result; } // → 検証してから登録、失敗は報告
POINT — インポートの「カテゴリ/タグは名前で解決」

エクスポートでカテゴリ・タグを名前で出した（id でなく）のは、インポート時に「名前で既存を探す、なければ作る」ができるから。id は環境ごとに違うので使えない。「仕事」というカテゴリ名でインポート→既存の「仕事」があれば紐付け、なければ新規作成。「名前ベースで解決」することで、別環境への移行が可能になる。これがエクスポートで名前を使った理由です。

06

実装

PHASE: IMPLIMPLEMENTATION

設計に沿って実装。エクスポート（ダウンロード）、インポート（アップロード＋検証）、結果表示。Jackson は Spring 同梱なので追加不要。MultipartFile でファイルを受け取ります。

Controller の実装ダウンロードとアップロード
// インポートControllerのポイント
  • @RequestParam MultipartFile file で受け取る
  • Service で検証＋登録
  • 結果（成功/失敗）を画面に表示
  • パースエラー等は例外ハンドリング（Vol.12へ）
// MemoController（インポート） // ─── インポート画面表示 ─── @GetMapping("/data") public String dataPage() { return "memos/data"; } // ─── インポート実行 ─── @PostMapping("/import") public String importMemos( @RequestParam("file") MultipartFile file, RedirectAttributes ra) { if (file.isEmpty()) { ra.addFlashAttribute("error", "ファイルを選択してください"); return "redirect:/memos/data"; } try { ImportResult result = service.importFromJson(file); // 結果を報告（別冊S10）★★★ ra.addFlashAttribute("result", result); } catch (Exception e) { // パース失敗等（別冊S7） ra.addFlashAttribute("error", "ファイルの形式が不正です"); } return "redirect:/memos/data"; } // ─── data.html（画面）─── <!-- エクスポート --> <a href="/memos/export"> メモをエクスポート（JSON） </a> <!-- インポート --> <form method="post" action="/memos/import" enctype="multipart/form-data"> <input type="file" name="file" accept=".json"> <button>取り込む</button> </form> // → enctype が重要（ファイル送信）★ <!-- 結果表示（別冊S10）--> <div th:if="${result}"> <p th:text="${result.successCount} + '件取り込みました'"></p> <ul th:if="${!result.errors.isEmpty()}"> <li th:each="e : ${result.errors}" th:text="${e}"></li> </ul> </div> // → 成功件数＋失敗理由を表示（別冊S10）★★★
POINT — ファイルアップロードは enctype="multipart/form-data" が必須

ファイルをアップロードするフォームは、enctype="multipart/form-data" が必須。これがないとファイルが送信されない（ファイル名だけ送られる等）。初学者がよく忘れる罠（Section 08）。また accept=".json" で選択時にJSONファイルを促すとUXが良い（別冊S8）。ファイル送信は通常のフォーム送信と仕組みが違うことを覚えておきましょう。

PHASE: TESTTEST

エクスポート/インポートをテスト。核心は@AC3（往復で一致）と@AC4（不正データの検証）。別冊S10の「往復テスト」「検証＋報告」を実装します。日本語の保持（@AC5）も確認します。

往復テストと異常データテスト信頼性を保証
// 検証ポイント
  • @AC3：export→importで元に戻る ★★★
  • @AC4：不正行を弾いて報告 ★★★
  • @AC5：日本語が化けない

往復テストがエクスポート/インポート両方の正しさを一度に保証（別冊S10）。

// MemoImportExportTest.java（結合） @SpringBootTest @Transactional class MemoImportExportTest { @Autowired MemoDataService service; @Autowired MemoMapper mapper; // @AC3 往復で一致 ★★★ @Test void エクスポートしてインポートすると元に戻る() throws Exception { // メモ2件を用意（日本語・タグ付き） setupMemos(); // エクスポート String json = service.exportToJson(); // DBをクリア（全削除） mapper.deleteAll(); // インポート（ファイルに見立てて） MultipartFile file = toFile(json); ImportResult r = service.importFromJson(file); // 元の2件が復元される ★★★ assertThat(r.getSuccessCount()) .isEqualTo(2); var restored = mapper.findAllWithDetails(); assertThat(restored) .extracting(Memo::getTitle) .containsExactlyInAnyOrder( "会議メモ", "買い物"); // → 往復で元に戻る ★★★ // → 日本語も保持（@AC5） } // @AC4 不正行を弾いて報告 ★★★ @Test void 不正な行は登録されず報告される() throws Exception { // 正常1件＋不正1件（タイトル空）のJSON String json = """ [ {"title":"正常メモ","content":"a"}, {"title":"","content":"b"} ] """; MultipartFile file = toFile(json); ImportResult r = service.importFromJson(file); // 正常1件は登録、不正1件は失敗 assertThat(r.getSuccessCount()) .isEqualTo(1); assertThat(r.getErrors()) .hasSize(1) .first().asString() .contains("タイトルが空"); // → 検証＋報告（別冊S10）★★★ } }
POINT — テストデータの JSON は「テキストブロック」で書くと読みやすい

Java 17 のテキストブロック（"""..."""）を使うと、テスト用の JSON をそのままの形で書ける。エスケープ地獄（"{\"title\":...}"）を避けられて、テストが読みやすい。不正データのテスト（@AC4）では「わざと壊れたJSON」を用意するので、テキストブロックが便利。本シリーズは Java 17 なので、こうしたモダンな構文も活用します。テストの可読性も品質のうちです。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザで確認。エクスポート→ファイル確認→インポートの往復を実際に試す。Vol.11 で踏みやすいデータ入出力特有の地雷を5つ先回りします。

Vol.11 ハマりどころ 5選データ入出力特有の地雷

① enctype 忘れでファイルが送られない ★★★

  • multipart/form-data がないと送信失敗
  • 対処：form に enctype を付ける（Section 06）

② 文字コード未指定で文字化け ★★★

  • UTF-8 を明示せず日本語が化ける
  • 対処：read/write 両方で UTF-8（別冊S10・Section 04,05）

③ インポートを無検証で登録 ★★★

  • 不正データがそのままDBに
  • 対処：1件ずつ検証（別冊S10・Section 05）

④ パース失敗で500エラー

  • 壊れたJSONで例外→白い500画面
  • 対処：try-catchで親切なエラー（別冊S7・Vol.12へ）

⑤ 巨大ファイルで止まる

  • アップロードサイズ上限・メモリ
  • 対処：max-file-size 設定、件数制限（別冊S9,S10）
