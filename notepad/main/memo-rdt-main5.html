設計①：中間テーブルと多対多

PHASE: DESIGNJUNCTION TABLE

別冊S4で学んだ中間テーブル memo_tags を実装に落とします。複合主キーで重複を防ぎ、CASCADE で削除を連動。Memo にタグのリストを持たせます。

Memo にタグリストを持たせるList<Tag> 型
// エンティティの拡張
  • Tag：id, name（Vol.3のCategoryと同型）
  • Memo に tags を追加：List<Tag> 型

カテゴリは単数（Category）、タグは複数（List<Tag>）。これが1対多と多対多の違い。

// 中間テーブル（別冊S4＋Vol.1設計）
  • 複合主キー（memo_id, tag_id）：同じ組合せ重複防止
  • CASCADE：メモ削除→紐付けも消す
// Tag.java + Memo拡張 + 中間テーブル // ─── Tag.java（Vol.3 Categoryと同型）─── @Data public class Tag { private Long id; @NotBlank @Size(max = 50) private String name; } // ─── Memo に tags を追加 ─── @Data public class Memo { private Long id; private String title; private String content; private Long categoryId; private Category category; // 単数（1対多） // ↓ Vol.5 で追加 private List<Tag> tags; // 複数（多対多）★★★ private LocalDateTime createdAt; private LocalDateTime updatedAt; } // → category は1つ、tags は複数 ★ // ─── 中間テーブル（Vol.1で設計済み）─── CREATE TABLE memo_tags ( memo_id BIGINT NOT NULL, tag_id BIGINT NOT NULL, PRIMARY KEY (memo_id, tag_id), -- ↑ 複合主キー：重複防止（別冊S4） FOREIGN KEY (memo_id) REFERENCES memos(id) ON DELETE CASCADE, -- ↑ メモ削除→紐付けも消す FOREIGN KEY (tag_id) REFERENCES tags(id) ON DELETE CASCADE ); // ─── 削除戦略の確認 ─── // メモ削除 → memo_tags の該当行も消える // （CASCADE）。でも tags 本体は無関係 ★★★ // タグ削除 → memo_tags の該当行も消える // （CASCADE）。でも memos 本体は無関係
POINT — 単数は association、複数は collection

別冊S4・Vol.4 の復習。Memo が持つcategory は単数（1つ）なので association、tags は複数（List）なので collection。「1つの関連は association、複数の関連は collection」。Java の型でも Category category（単数）と List<Tag> tags（複数）で対応します。型を見れば association か collection か分かるのです。

設計②：collectionで複数取得

PHASE: DESIGNCOLLECTION MAPPING

JOIN で展開された複数行を「1メモ＋タグリスト」に組み立てるのが collection。Vol.4 の association（単数）の複数版です。フラットなSQL結果を、ネストしたオブジェクトに戻します。

resultMap + collection複数行を1オブジェクトに
// collection の役割

JOIN結果は「1メモがタグ数だけ複数行」（Section 03）。collection は同じ id の行をまとめ、タグを List に集約します。

// association と collection の対比
associationcollection
関連数1つ複数
Java型CategoryList<Tag>
使う巻Vol.4Vol.5
// id 要素が「まとめる鍵」

resultMap の <id> が「同じメモか判定する鍵」。これで複数行が1メモにまとまります。

// resultMap + collection（XML） // ─── JOIN結果（複数行）─── id│title│tag_id│tag_name 1 │会議 │1 │重要 1 │会議 │2 │急ぎ ← 同じid=1 2 │買物 │NULL │NULL // ↓ collection で組み立て // Memo(id=1, tags=[重要,急ぎ]) // Memo(id=2, tags=[]) // ─── MemoMapper.xml ─── <resultMap id="MemoWithTags" type="Memo"> <id property="id" column="id"/> <!-- ↑ idが「まとめる鍵」★★★ --> <result property="title" column="title"/> <result property="content" column="content"/> <!-- 複数タグを List に集約 --> <collection property="tags" ofType="Tag"> <id property="id" column="tag_id"/> <result property="name" column="tag_name"/> </collection> </resultMap> // → ofType="Tag" = リストの要素の型 // → tag_id が NULL なら追加しない（@AC4）★★★ // ─── association と collection 併用 ─── // 1つの resultMap に両方書ける： <association property="category"... → カテゴリ（単数） <collection property="tags"... → タグ（複数） // → メモ＋カテゴリ＋タグ全部を // 1回のSQLで取得できる ★★★
POINT — <id> 要素を忘れると行がまとまらない

collection 利用時の最頻出ミス：resultMap に <id> を書き忘れる。<id> が「どの行が同じメモか」を判定する鍵なので、これがないとMyBatis が行をまとめられず、タグ1つごとに別メモが作られる（メモが重複）。collection を使うときは <id> 必須。これは Section 08 のハマりどころでも再掲します。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザで確認。複数タグの表示・タグなしメモを目視。Vol.5 で踏みやすい collection 特有の地雷を5つ先回りします。

Vol.5 ハマりどころ 5選collection特有の地雷

① メモが重複表示される ★★★

  • 原因：resultMap に <id> がない（行がまとまらない）
  • 対処：<id property="id" column="id"/> を必ず書く

② タグが1つしか表示されない

  • 原因：collection でなく association を使った／resultType を使った
  • 対処：複数は collection + ofType

③ タグなしメモが消える ★★★

  • 原因：INNER JOIN で書いた（@AC4違反）
  • 対処：LEFT JOIN（結合テストで検知できる）

④ tags が null になる（空リストでない）

  • 原因：タグ0個のとき MyBatis が null を返す場合がある
  • 対処：View で null チェック、または Service で空リスト初期化

⑤ tag_id の別名と column 名が不一致

  • 原因：t.id をそのまま使うと memos.id と衝突
  • 対処：t.id AS tag_id と別名を付け、column を一致させる

POINT — 「メモ重複」は collection の最頻出バグ

collection で最も多いバグがメモの重複表示。原因はほぼ <id> 要素の書き忘れ。MyBatis は <id> で「同じレコードか」を判定し、同じならタグだけリストに追加します。<id> がないと判定できず、タグの数だけメモが増殖。「メモが2倍に増えた」と思ったら、まず <id> を疑いましょう。
