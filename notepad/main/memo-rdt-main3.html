重複名(@AC3)・並び順(@AC4)は「DB絡み」なので結合テストで検証。バリデーション（空・文字数）は単体で。テスト層の振り分けも設計です。

POINT — 「どの層でテストするか」も設計判断

バリデーション（空・文字数）はJavaのロジックなので単体テスト。重複名・並び順はDBの制約・並び替えに依存するので結合テスト。「何が正しさを保証しているか」でテスト層を選ぶ。Javaが守るなら単体、DBが守るなら結合。テスト層の選択は、責務の所在の理解です。

04

設計①：データと制約

PHASE: DESIGNDATA & CONSTRAINTS

Category のデータ構造を確定します。UNIQUE制約（重複禁止）がマスタの肝。受入条件 @AC3（重複エラー）を「どこで守るか」——DBの制約か、Javaのチェックか。設計判断します。

Category エンティティと UNIQUE 制約重複は誰が守るか
// Category の5フィールド
  • id：Long・主キー
  • name：String・必須・50字・UNIQUE
  • color：String・#xxxxxx
  • sortOrder：Integer・並び順
  • createdAt：LocalDateTime
// 重複を「DBで守る」設計

受入条件 @AC3（重複エラー）はDB の UNIQUE 制約で守ります。Java で「事前に findByName してチェック」は避けます。理由：

  • 同時アクセスで両方とも「重複なし」と判定→両方INSERTの競合が起きる
  • DB の UNIQUE 制約なら確実に1つだけ

→ 正しさはDBに守らせる（別冊S3）。

// Category.java + DDL package com.example.memo.entity; import java.time.LocalDateTime; import jakarta.validation.constraints.*; import lombok.Data; @Data public class Category { private Long id; @NotBlank(message = "カテゴリ名は必須") @Size(max = 50, message = "50文字以内") private String name; private String color; private Integer sortOrder; private LocalDateTime createdAt; } // ─── DDL（Vol.1で設計済み）─── CREATE TABLE categories ( id BIGINT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(50) NOT NULL UNIQUE, // ↑ @AC3 を DB が守る ★★★ color VARCHAR(7) NOT NULL, sort_order INT DEFAULT 0, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); // ─── 「@AC2と@AC3で守る場所が違う」 ─── @AC2 空の名前 → @NotBlank（Javaが守る） @AC3 重複名 → UNIQUE制約（DBが守る） @AC4 並び順 → ORDER BY（SQLが守る） // → 「何を、誰が守るか」を設計で決める ★★★
POINT — 重複チェックはJavaでやらない

初学者がやりがちな失敗：「INSERT前にfindByNameで重複チェック」。これは同時アクセスで破綻します（2人が同時にチェック→両方「なし」→両方INSERT）。DBのUNIQUE制約に任せ、違反時の例外を捕まえるのが正解。「正しさの最後の砦はDB」。これは別冊S3でも詳しく扱います。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザでも確認。カラーピッカー・並び順・重複エラーを目視します。Vol.3 で踏みやすい地雷を5つ先回りします。

Vol.3 ハマりどころ 5選マスタ＋結合テストの地雷

① @MybatisTest で schema.sql が読まれない ★★★

  • 原因：src/test/resources/schema.sql が無い／application-test.properties 未設定
  • 対処：test配下に schema.sql を置く、H2 を依存に追加

② H2 で MySQL 構文エラー（AUTO_INCREMENT 等）

  • 原因：H2 と MySQL の方言差
  • 対処：H2 の MySQL 互換モード jdbc:h2:mem:test;MODE=MySQL

③ 重複テストが DuplicateKeyException で落ちない

  • 原因：schema.sql に UNIQUE 制約を書き忘れ
  • 対処：name VARCHAR(50) NOT NULL UNIQUE を確認

④ input type=color が #000000 になる

  • 原因：newForm() でデフォルト色を未セット
  • 対処：c.setColor("#2d5485") をセット（Section 06）

⑤ 並び順が効かない

  • 原因：ORDER BY sort_order を書き忘れ／同値で id 順未指定
  • 対処：ORDER BY sort_order ASC, id ASC の2段ソート
// 結合テストのセットアップ // ─── src/test/resources/schema.sql ─── CREATE TABLE categories ( id BIGINT AUTO_INCREMENT PRIMARY KEY, name VARCHAR(50) NOT NULL UNIQUE, color VARCHAR(7), sort_order INT DEFAULT 0, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); // ─── application-test.properties ─── spring.datasource.url= jdbc:h2:mem:test;MODE=MySQL; DB_CLOSE_DELAY=-1 spring.sql.init.mode=always // ─── テスト実行結果 ─── CategoryValidationTest ✓ 空のカテゴリ名はエラー ✓ _50文字はOK CategoryMapperTest ✓ 登録したカテゴリがsort順で取れる ✓ 重複するカテゴリ名は登録できない all green ★★★ // ─── ブラウザ確認 ─── // /categories で： // - カラーピッカーで色選択（@AC1目視） // - sort_order昇順で並ぶ（@AC4目視） // - 同名登録でエラー画面（@AC3目視）
POINT — H2 は「速くて安定」だが MySQL ではない

H2 インメモリDBは高速・テスト終了で消えるのが利点。ただしMySQL とは方言が違うので、MODE=MySQL で互換性を上げます。それでも完全互換ではないので、本番に近い検証が必要なら Testcontainers で実MySQLを使う選択肢も（別冊S3で触れます）。速さと忠実さのトレードオフです。
