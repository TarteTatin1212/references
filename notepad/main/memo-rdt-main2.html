02

受入条件をテストケース化

PHASE: REQ→TESTAC TO TEST CASES

RDT版の心臓部。受入条件（Given/When/Then）をテストケースの一覧に変換します。「テストを先に設計する」。実装より先にテスト設計をすると、「何ができれば完成か」が明確になります。

受入条件 → テストケース表テスト設計技法を適用
// バリデーションのテストケース

タイトル：1〜200文字必須という要件から、代表値・境界値・異常系を選びます。

入力期待技法
空文字エラー異常系
1文字OK下限境界
200文字OK上限境界
201文字エラー境界の外
通常文字列OK代表値
nullエラー異常系

1機能につき正常系1〜2＋異常系3〜5が目安。異常系は意識しないと抜けます。

// Given/When/Then → AAA への変換 // 受入条件の構造が… Given（前提） → Arrange（準備） When （行動） → Act（実行） Then （結果） → Assert（検証） // → そのまま JUnit の3ブロックに ★★★ // ─── @AC2 を例に変換してみる ─── // 受入条件： // Given: タイトルが空 // When : 保存する // Then : エラー、保存されない // ↓ テストケースに @Test void 空タイトルはバリデーションエラー() { // Arrange（Given） Memo memo = new Memo(); memo.setTitle(""); // Act + Assert（When + Then） // → バリデーションが効くか検証 // → 詳細は Section 07 } // → テスト名は「何を検証するか」を // 日本語で書く（@AC2 がそのまま名前に）
POINT — テストを先に設計する＝完成の定義

テストケース表を先に作ると「これが全部通れば完成」というゴールが明確になります。実装中に「あとどれだけ？」と迷わない。受入条件＝テストケース＝完成の定義。この3つが一致するのが RDT の美しさです。

仕様化：CRUDのエンドポイント

PHASE: SPECSPECIFICATION

要求を技術的な仕様に落とします。CRUD をURLとHTTPメソッドの設計に変換。「何を」（要件）から「どう」（仕様）への橋渡しです。

URL設計（RESTful）要件→仕様の変換
// 6つのエンドポイント
操作メソッドURL
一覧(R)GET/memos
新規フォームGET/memos/new
登録(C)POST/memos
編集フォームGET/memos/{id}/edit
更新(U)POST/memos/{id}

// HTML の制約と PRG
  • HTML標準は GET と POST のみ（PUT/DELETE不可）
  • 更新・削除も POST で実現、URL末尾の動詞で識別
  • POST後は必ず redirect（PRGパターン）
// 仕様：完成画面イメージ // http://localhost:8080/memos ┌──────────────────────────┐ │ メモ一覧 [+ 新規] │ ├──────────────────────────┤ │ID│ タイトル │更新 │操作│ ├──────────────────────────┤ │ 1│買い物リスト│11:23 │編 削│ │ 2│Java学習 │10:05 │編 削│ └──────────────────────────┘ // ─── PRG パターンの流れ ─── [保存ボタン] POST /memos ↓ 処理 ↓ redirect:/memos ← 重要 [GET /memos] 一覧表示 ↓ F5リロードしても二重登録なし ★★★ // ─── @AC1 がここに対応 ─── // "保存→一覧に表示" // = POST /memos → redirect → GET /memos

URL設計やHTTPメソッドの選択は仕様（どう作るか）。要件（何を満たすか）を満たす限り実装者の裁量です。ただしRESTfulな慣習に従うと、他人が読みやすく業務でも通用する。「自由だが、定石に乗る」のが賢い仕様化です。

04

設計①：データと層

PHASE: DESIGNDATA & LAYER

Vol.1 で決めた3層構造とmemosテーブルに沿って、Memo のデータ構造を確定します。テスト容易性を意識した設計をします。「純粋ロジックと外部依存を分ける」のが鍵。

Memo エンティティとテスト容易性分離が単体テストを楽にする
// Memo の5フィールド
  • id：Long・主キー
  • title：String・必須・200字
  • content：String・任意
  • createdAt：LocalDateTime・不変
  • updatedAt：LocalDateTime・毎回更新
// テスト容易性のための分離

フロー文書の指針：「保存・現在時刻などの外部依存」と「純粋ロジック」を分離。

  • 外部依存：DB保存（Mapper）、現在時刻（now()）
  • 純粋ロジック：バリデーション（タイトル文字数チェック）

バリデーションを純粋に切り出すと、DBなしで単体テストできます。

// Memo.java + バリデーション設計 package com.example.memo.entity; import java.time.LocalDateTime; import jakarta.validation.constraints.*; import lombok.Data; @Data public class Memo { private Long id; @NotBlank(message = "タイトルは必須です") @Size(max = 200, message = "200文字以内") private String title; private String content; private LocalDateTime createdAt; private LocalDateTime updatedAt; } // ─── @NotBlank / @Size は ─── // Bean Validation（jakarta.validation） // → @AC2(空) と @AC3,4(文字数) に対応 ★★★ // → 受入条件が「アノテーション」になった // ─── 外部依存と純粋ロジックの分離 ─── 純粋ロジック（テスト楽）： - @NotBlank / @Size のバリデーション 外部依存（モックで差し替え）： - mapper.insert() ← DB保存 - LocalDateTime.now() ← 現在時刻 // → Service が現在時刻をセットし // Mapper はモック化できる ★
POINT — Bean Validation で受入条件をコードに埋める

@NotBlank（空チェック）と@Size(max=200)（文字数）は、受入条件 @AC2〜@AC4 をそのままアノテーション化したもの。要求が宣言的にコードへ刻まれる。後の Section 07 で、このアノテーションが効くかをテストします。要求・コード・テストが三位一体です。

設計②：クラスと責務

PHASE: DESIGNCLASS & RESPONSIBILITY

3層それぞれのクラスと責務を設計します。単一責任（SRP）を守る。「変更理由は1つだけ」。説明に「と」が入ったら分割のサインです。

クラス設計と責務分割SRP を守る
// 各クラスの単一責任
CONTROLLER
MemoController
HTTPの世話だけ。URL対応・Model詰め・redirect。
SERVICE
MemoService
業務ロジックだけ。日時セット・Mapper呼び出し。
MAPPER
MemoMapper
SQLだけ。データ取得・保存。ロジック無し。
// 命名は「業務の言葉」で
  • NG：processData()（何をする？）
  • OK：findAll() create()（読めば分かる）
  • Manager/Util/Helper が付いたら責務が曖昧のサイン
// 責務分割の検証（SRP） // ─── 「と」が入ったら分割 ─── 悪い設計： MemoManager 「メモを保存し、かつ画面を更新し、 かつ検索する」 → 変更理由が3つ → SRP違反 良い設計： MemoMapper 「保存する」だけ MemoController 「画面を扱う」だけ MemoService 「ロジックを束ねる」だけ → 各クラス変更理由は1つ ★★★ // ─── 依存方向（上から下）─── MemoController ↓ 呼ぶ（OK） MemoService ↓ 呼ぶ（OK） MemoMapper // MapperがControllerを呼ぶ ← 禁止 // 下が上を知ってはいけない ★★★ // ─── テスト容易性とDI ─── // @RequiredArgsConstructor で // 依存を外から注入 → // テストでモックに差し替え可能
POINT — 「テストが書きにくい＝設計が悪い」

フロー文書の重要な指針：テストが書きにくいと感じたら、それは設計を見直すサイン。責務がごちゃ混ぜだとモック化できず、テストが書けません。SRPを守る＝テストしやすい。設計品質とテスト容易性は表裏一体です。

実装

PHASE: IMPLIMPLEMENTATION

設計が固まったので実装。Entity→Mapper→Service→Controller→View の順に。クラス作成は毎回「右クリック → New」を明示します。設計に沿って書くのが基本。迷ったら設計（前のPhase）に戻ります。

Mapper（インターフェース＋XML）SQLだけの層
// 作成手順
  1. com.example.memo 右クリック → New → Package → ...mapper
  2. mapper 右クリック → New → Interface → MemoMapper
  3. resources に mapper フォルダ → MemoMapper.xml
// MemoMapper.java package com.example.memo.mapper; import java.util.List; import org.apache.ibatis.annotations.Mapper; import com.example.memo.entity.Memo; @Mapper public interface MemoMapper { List<Memo> findAll(); Memo findById(Long id); void insert(Memo memo); void update(Memo memo); void deleteById(Long id); } // MemoMapper.xml（抜粋） <mapper namespace="...mapper.MemoMapper"> <select id="findAll" resultType="Memo"> SELECT id, title, content, created_at, updated_at FROM memos ORDER BY updated_at DESC </select> <insert id="insert" useGeneratedKeys="true" keyProperty="id"> INSERT INTO memos (title, content, created_at, updated_at) VALUES (#{title}, #{content}, #{createdAt}, #{updatedAt}) </insert> <!-- findById/update/deleteById も同様 --> </mapper>
Service と Controllerロジック層とHTTP層
// 作成手順
  1. service パッケージ作成 → MemoService クラス
  2. controller パッケージ作成 → MemoController クラス
  3. Vol.1 で作った HealthController は削除してOK

Service が日時セットの業務ロジックを担当。Controller は HTTP の世話に専念。

// MemoService.java @Service @RequiredArgsConstructor public class MemoService { private final MemoMapper mapper; public List<Memo> findAll() { return mapper.findAll(); } public Memo findById(Long id) { return mapper.findById(id); } public void create(Memo memo) { LocalDateTime now = LocalDateTime.now(); memo.setCreatedAt(now); memo.setUpdatedAt(now); mapper.insert(memo); } public void update(Memo memo) { memo.setUpdatedAt( LocalDateTime.now()); mapper.update(memo); } public void delete(Long id) { mapper.deleteById(id); } } // MemoController.java（要点） @Controller @RequestMapping("/memos") @RequiredArgsConstructor public class MemoController { private final MemoService service; @GetMapping public String list(Model m) { m.addAttribute("memos", service.findAll()); return "memos/list"; } @PostMapping public String create( @Validated @ModelAttribute Memo memo, BindingResult br) { if (br.hasErrors()) { return "memos/new"; // @AC2,3,4: バリデーションエラー時 // は保存せずフォームに戻る ★★★ } service.create(memo); return "redirect:/memos"; // PRG } // new/edit/update/delete は省略（同型） }
POINT — @Validated + BindingResult が受入条件を守る

Controller の @Validated @ModelAttribute Memo ＋ BindingResult が、Entity の @NotBlank/@Size を発動させます。エラーがあれば保存せずフォームに戻る＝受入条件 @AC2〜@AC4 の動作。要求がコードの分岐として実装された瞬間です。

単体テスト — バリデーション@AC2〜@AC4 を JUnit に

  1. src/test/java/.../entity に MemoValidationTest
  2. Bean Validation の Validator で title を検証

Service の単体テスト（Mockito）Mapper をモック化
// Mockito で外部依存を差し替え

Service の業務ロジック（create時に日時をセット）を、DBなしでテストします。Mapper をモック化（差し替え）。

  • @Mock：Mapper のニセモノを作る
  • @InjectMocks：Service にモックを注入
  • verify()：Mapper が呼ばれたか検証

POINT — Red-Green-Refactor サイクル

理想はテストファースト。①失敗するテストを書く（Red）→②通す最小実装（Green）→③整える（Refactor）。特に純粋ロジック（バリデーション）はテストを先に書けます。本Vol.2 では実装を先に見せましたが、慣れたら「受入条件→テスト→実装」の順を試してください。バグ修正は必ず再現テストを先に。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザでも確認。テストは内部品質、ブラウザ確認は体験の確認。両方やります。Vol.2 で踏みやすい地雷を5つ先回りします。

Vol.2 ハマりどころ 5選CRUD＋テストの地雷

① BindingException: findById not found ★★★

  • 原因：Mapper.xml の namespace/id ミス
  • 対処：namespace を完全修飾名に、id をメソッド名と一致

② @Validated を付けてもバリデーションが効かない ★★★

  • 原因：Controller の引数に @Validated が無い／BindingResult の位置が違う
  • 対処：@Validated @ModelAttribute Memo memo, BindingResult br の順（BindingResultは検証対象の直後）

③ MockMvc テストで 404 / テンプレートエラー

  • 原因：Thymeleaf テンプレート（memos/new.html）未作成
  • 対処：list/new/edit.html を作成してから MockMvc テスト実行

④ Mockito の verify が「0 interactions」

  • 原因：@InjectMocks より先に @Mock が初期化されていない／@ExtendWith 忘れ
  • 対処：@ExtendWith(MockitoExtension.class) を付ける

⑤ created_at が NULL で入る

  • 原因：Service の create() で setCreatedAt を呼んでいない
  • 対処：Service で LocalDateTime.now() をセット（テストで検知できる）
// テスト実行と確認 // ─── テストを全部走らせる ─── // プロジェクト右クリック // → Run As → JUnit Test 実行結果： MemoValidationTest ✓ 空タイトルはエラーになる ✓ _200文字は登録できる ✓ _201文字はエラーになる MemoServiceTest ✓ 作成時にcreatedAtとupdatedAtが入る MemoControllerTest ✓ メモ作成すると一覧にリダイレクト ✓ 空タイトルはフォームに戻る 6 tests, 0 failures ★★★ // ─── ②の正しい引数順 ─── @PostMapping public String create( @Validated @ModelAttribute Memo memo, BindingResult br) { // ←直後に置く ... } // BindingResult が検証対象の // 直後にないと例外になる ★★★ // ─── ブラウザ確認 ─── // http://localhost:8080/memos // → 一覧 → 新規 → 保存 → 編集 → 削除 // → 空タイトルでエラーが出るか // （受入条件@AC2を目視確認）
POINT — テストとブラウザ確認は役割が違う

テストは「ロジックが正しいか」を高速・自動で。ブラウザ確認は「ユーザー体験として成立するか」を目視で。どちらも必要。テストが緑でも、画面が崩れていればユーザーは困る。逆に画面が動いても、エッジケースはテストでしか守れません。
