いつも通りまず要求から。Vol.1 で定義した最後のストーリー US-004（検索）を確認します。メモが増えると一覧では探しきれない。「キーワードで探す」「条件を組み合わせて絞る」——これが検索の価値です。

Vol.7 では キーワード（部分一致）＋カテゴリ＋タグを組み合わせた検索を実装。LIKE 検索が中心です。大量データ向けの全文検索（FULLTEXT）は次の Vol.8。

POINT — 「条件なしで全件」も立派な仕様

@AC5「何も入力せず検索→全件」は見落としやすいが重要な受入条件。検索フォームで空のまま検索したとき、エラーにせず全件返すのが親切な挙動。これは別冊S6の動的SQL <where> が「条件0個なら WHERE を出さない」で自然に実現します。「入力なし」という境界もテストするのが、別冊S3のテスト設計です。

04

設計①：検索条件オブジェクト

PHASE: DESIGNSEARCH FORM OBJECT

複数の検索条件を1つのオブジェクト（SearchForm）にまとめます。バラバラの引数より凝集度が高く、条件追加にも強い。別冊S2の設計原則をここで実践します。

SearchForm クラスの設計条件をまとめる
// SearchForm の役割

検索条件（キーワード・カテゴリID・タグIDリスト）を1つのオブジェクトに。Controller はこれ1個を受け取り、Mapper にそのまま渡す。

// メリット
  • 引数が1個で済む
  • 条件追加がフィールド追加だけ
  • 「検索条件」という概念がコードに表れる
  • Mapper の <if test> で各フィールドを判定
// 作成手順

form パッケージ右クリック → New → Class → SearchForm。

// SearchForm.java package com.example.memo.form; import java.util.List; import lombok.Data; @Data public class SearchForm { // キーワード（タイトル/本文の部分一致） private String keyword; // カテゴリ（単一） private Long categoryId; // タグ（複数） private List<Long> tagIds; } // ─── 引数1個 vs バラバラ ─── バラバラ（NG）: search(String keyword, Long categoryId, List<Long> tagIds, ...) // → 条件が増えると引数も増える ★ まとめる（OK）: search(SearchForm form) // → 引数1個。条件追加はフィールド追加 ★★★ // → 別冊S2の「凝集度を高める」 // ─── Controller での受け取り ─── @GetMapping("/search") public String search( @ModelAttribute SearchForm form, Model model) { // keyword=会議&categoryId=1&tagIds=2 が // 自動で SearchForm にバインドされる ★ model.addAttribute("memos", service.search(form)); // 検索フォーム再表示用にカテゴリ・タグも model.addAttribute("form", form); return "memos/search"; }
POINT — フォームオブジェクトは @ModelAttribute で自動バインド

Spring の @ModelAttribute SearchForm form は、クエリパラメータを自動で SearchForm のフィールドにマッピングします。?keyword=会議&categoryId=1 が form.keyword="会議" form.categoryId=1 に。手作業のパラメータ取得が不要。「URLのパラメータ名＝フィールド名」にしておけば、Spring が全部やってくれる。これも凝集度を高める設計の恩恵です。

05

設計②：動的SQLの組み立て

PHASE: DESIGNDYNAMIC SQL DESIGN

別冊S6の集大成。SearchForm の各条件を、<if><where><foreach> で動的SQLに組み立てます。条件があれば WHERE に足し、なければ足さない。全て #{} で安全に。

動的SQLの設計（別冊S6の実践）if/where/foreach
// 各条件の組み立て方
条件SQL断片タグ
キーワードtitle/content LIKE<if>
カテゴリcategory_id =<if>
タグIN (memo_tags)<foreach>
// <where> が整える

<where> が「条件0個なら WHERE 省略」「先頭 AND 削除」を自動化（別冊S6）。@AC5（条件なし→全件）も自然に実現。

// 安全性

全て #{}。${} は使わない（別冊S6）。SQLインジェクション対策を徹底。

// 検索の動的SQL（MemoMapper.xml） <select id="search" resultMap="MemoFull"> SELECT DISTINCT m.*, c.name AS category_name, c.color AS category_color FROM memos m LEFT JOIN categories c ON m.category_id = c.id <where> <!-- @AC1 キーワード --> <if test="keyword != null and keyword != ''"> AND (m.title LIKE CONCAT('%', #{keyword}, '%') OR m.content LIKE CONCAT('%', #{keyword}, '%')) </if> <!-- @AC2 カテゴリ --> <if test="categoryId != null"> AND m.category_id = #{categoryId} </if> <!-- @AC3 タグ（IN句）--> <if test="tagIds != null and !tagIds.isEmpty()"> AND m.id IN ( SELECT memo_id FROM memo_tags WHERE tag_id IN <foreach item="id" collection="tagIds" open="(" separator="," close=")"> #{id} </foreach> ) </if> </where> ORDER BY m.updated_at DESC </select> // ─── 条件で SQL がこう変わる ─── // 全部指定 → WHERE ... AND ... AND ... // キーワードのみ → WHERE title LIKE ... // なし → WHERE なし（全件）★★★ // ─── 全て #{} で安全 ─── // #{keyword} #{categoryId} #{id} // → SQLインジェクション対策（別冊S6）★★★
POINT — タグ検索（多対多）は IN ＋サブクエリ＋foreach

タグでの絞り込みは多対多なので、少し工夫が要ります。「指定タグのいずれかが付いたメモのID」を中間テーブルから取り、m.id IN (...) で絞る。<foreach> でタグIDリストを (2, 5, 7) に展開（別冊S6）。SELECT DISTINCT を付けるのは、複数タグ一致でメモが重複しないようにするため。多対多の検索は IN ＋サブクエリが定石です。

// Controller + 検索View // ─── MemoController（検索）─── @GetMapping("/search") public String search( @ModelAttribute SearchForm form, Model model) { model.addAttribute("memos", service.search(form)); // フォーム再表示用 model.addAttribute("categories", categoryService.findAll()); model.addAttribute("allTags", tagService.findAll()); model.addAttribute("form", form); return "memos/search"; } // ─── search.html（検索フォーム）─── <form method="get" action="/memos/search" th:object="${form}"> <!-- キーワード（入力値が残る）--> <input type="text" name="keyword" th:value="*{keyword}"> <!-- カテゴリ --> <select name="categoryId"> <option value="">すべて</option> <option th:each="c : ${categories}" th:value="${c.id}" th:text="${c.name}" th:selected= "${c.id == form.categoryId}"> </option> </select> <!-- タグ（複数）--> <span th:each="t : ${allTags}"> <input type="checkbox" name="tagIds" th:value="${t.id}"> <span th:text="${t.name}"></span> </span> <button>検索</button> </form> <!-- 検索結果（一覧と同じ表示）--> <div th:each="memo : ${memos}"> ... </div> // → 結果0件なら「該当なし」も表示
POINT — 検索後も「入力した条件」を残す

検索のUX で大事なのが「検索後も入力値が残る」こと。キーワードを打って検索→結果が出たとき、入力欄が空に戻ると「条件を変えて再検索」がやりにくい。th:value="*{keyword}" で入力値を保持し、カテゴリ・タグも選択状態を維持。「ユーザーの操作を無駄にしない」のが親切な検索画面です。

テスト（条件の組み合わせ）

PHASE: TESTTEST

検索を結合テストで検証。各条件単独・複合・条件なしを網羅。「ヒットするメモが出て、ヒットしないメモが消える」を厳密に確認します（Section 02 のPOINT）。

結合テスト — 検索の組み合わせ@AC1〜5 を実DBで
// テストデータ設計

「絞り込めること」を示すため、条件に合うメモと合わないメモを用意：

  • メモA：「会議」仕事カテゴリ・重要タグ
  • メモB：「買物」私用カテゴリ・タグなし

各検索でAが出てBが消えることを検証。

// MemoSearchTest.java（結合・H2） @MybatisTest class MemoSearchTest { @Autowired MemoMapper mapper; // @BeforeEach でメモA・Bを用意（省略） // @AC1 キーワード @Test void キーワードで絞り込める() { SearchForm form = new SearchForm(); form.setKeyword("会議"); var result = mapper.search(form); // Aだけ（Bは消える） assertThat(result) .extracting(Memo::getTitle) .containsExactly("会議メモ"); // → 「買物」は出ない ★★★ } // @AC4 複合条件（AND） @Test void 複合条件で絞り込める() { SearchForm form = new SearchForm(); form.setKeyword("会議"); form.setCategoryId(workId); form.setTagIds(List.of(importantId)); var result = mapper.search(form); // 全条件を満たすAだけ assertThat(result).hasSize(1); // → AND で全部効いている ★★★ } // @AC5 条件なし → 全件 @Test void 条件なしなら全件() { SearchForm form = new SearchForm(); // 何も設定しない var result = mapper.search(form); // 全件（A・B両方） assertThat(result).hasSize(2); // → <where>が空→WHERE省略→全件 ★★★ } // @AC3 タグで絞り込み @Test void タグで絞り込める() { SearchForm form = new SearchForm(); form.setTagIds(List.of(importantId)); var result = mapper.search(form); // 重要タグ付きのAだけ assertThat(result).hasSize(1); } } // → 条件の有無・組み合わせを網羅 ★★★ // → 動的SQLが正しく変化するか実証
POINT — 動的SQLは「条件の組み合わせ」をテストで網羅

動的SQLは条件の有無で生成されるSQLが変わるため、「条件あり」「条件なし」「複合」を網羅してテストします。特に@AC5（条件なし→全件）は <where> が正しく空になるかの検証。もし <if> の test 条件にミスがあると、「条件を入れたのに絞れない」「空なのに0件」といったバグが出る。組み合わせテストが動的SQLの正しさを保証します。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザで確認。各条件・複合・空検索を目視。Vol.7 で踏みやすい動的SQL特有の地雷を5つ先回りします。

Vol.7 ハマりどころ 5選動的SQL特有の地雷

① 空文字とnullの判定漏れ ★★★

  • キーワード未入力で空文字 "" が来る→ LIKE '%%' で全件のはずが意図せず動く
  • 対処：keyword != null and keyword != '' の両方を test に

② <foreach> で空リストエラー ★★★

  • tagIds が空で IN () になり構文エラー
  • 対処：<if test="tagIds != null and !tagIds.isEmpty()"> で囲む

③ タグ複数指定でメモが重複表示

  • JOIN で1メモが複数行に
  • 対処：SELECT DISTINCT または IN＋サブクエリ（Section 05）

④ ${} を使ってインジェクション穴

  • うっかり ${keyword} と書く
  • 対処：必ず #{}（別冊S6）。${} は使わない

⑤ 検索後に入力値が消える

  • th:value 未設定でフォームが空に戻る
  • 対処：form の値を th:value で再表示（Section 06）
// テスト＆ブラウザ確認 // ─── テスト結果 ─── MemoSearchTest ✓ キーワードで絞り込める ✓ タグで絞り込める ✓ 複合条件で絞り込める ✓ 条件なしなら全件 all green ★★★ // ─── ① 空文字の判定 ─── NG: <if test="keyword != null"> // → 空文字 "" のとき true になり // LIKE '%%' が走る（実害は薄いが無駄） OK: <if test="keyword != null and keyword != ''"> // → 空文字なら条件を足さない ★★★ // ─── ② foreach の空リスト対策 ─── NG: tagIds=[] で IN () ← 構文エラー ★ OK: <if test="tagIds != null and !tagIds.isEmpty()"> ... <foreach> ... </if> // → 空なら IN 句ごと出さない ★★★ // ─── ブラウザ確認 ─── // /memos/search で： // - キーワードのみ検索（@AC1） // - カテゴリのみ（@AC2） // - タグのみ（@AC3） // - 全部指定（@AC4・AND） // - 空検索→全件（@AC5） // - 検索後も条件が残る（⑤）
POINT — 動的SQLは「実際に生成されたSQL」をログで確認

動的SQLのデバッグでは、実際に生成されたSQLをログ出力すると一発で原因が分かります。application.properties に MyBatis のSQLログ設定を入れると、条件によってどんな SQL になったかがコンソールに出る。「条件を入れたのに WHERE に出ていない」「IN () になっている」が目で見える。「動的に何が生成されたか」を見るのが、動的SQLデバッグの王道です。
