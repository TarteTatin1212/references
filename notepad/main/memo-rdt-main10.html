01

要求の確認（一括操作）

PHASE: REQREVISIT REQUIREMENT

いつも通りまず要求から。Vol.9 でアーカイブ・削除を実装しましたが1件ずつ。使ってみると「古いメモを何十件もまとめて整理したい」というニーズが。これも使い込んで生まれた要求（US-005の延長）です。

一括操作の要求と背景1件ずつは面倒
// なぜ一括操作が必要か

Vol.9 のアーカイブは便利ですが、50件のメモを整理するには50回ボタンを押す。これは苦痛（別冊S9）。「選んでまとめて」が欲しい。

// US-005 の拡張
US-005 拡張 / SHOULD
複数メモをまとめて整理したい
As a メモが大量にあるユーザー
I want 複数のメモを選んで一括でアーカイブ/削除したい
so that 整理の手間を大幅に減らせる
// 一括操作の対象
  • 一括アーカイブ：選んだメモをまとめてアーカイブ
  • 一括削除：選んだメモをまとめて削除（要確認）
// 一括操作の受入条件 @AC1 複数選択して一括アーカイブ Given: 複数のメモがある When : 3件を選んで一括アーカイブ Then : 選んだ3件がアーカイブされる （選んでないメモは変わらない） @AC2 複数選択して一括削除 Given: 複数のメモがある When : 3件を選んで一括削除（確認後） Then : 選んだ3件が削除される @AC3 0件選択で実行 Given: メモがある When : 何も選ばず一括操作を実行 Then : エラーにならず、何も起きない （または「選択してください」）★ @AC4 一括処理の原子性 Given: 複数メモを一括処理 When : 処理が途中で失敗する Then : 全件処理されない（中途半端にしない） @AC5 一括削除は件数確認 ★★★ Given: 複数メモを選んで削除 When : 削除ボタンを押す Then : 「N件削除します」と確認が出る // ─── @AC3 空選択・@AC5 件数確認が肝 ─── // 別冊S9：空リストの罠・件数表示
POINT — 「0件選択」も必ず仕様に含める

一括操作で見落としやすいのが@AC3「0件選択で実行」。ユーザーが何も選ばずに「一括アーカイブ」を押すことは必ず起きる。このときエラー（IN () 構文エラー・別冊S9）になってはいけない。「何も起きない」または「選択してください」と親切に。複数選択UIは「0個選択」という境界を常に考慮する——別冊S3の境界値、別冊S9の空リストの罠です。

受入条件をテストケース化

PHASE: REQ→TESTAC TO TEST CASES

一括操作の受入条件をテストに。「選んだものだけ処理、選んでないものは無傷」が肝。さらに@AC3（空リスト）の境界を必ずテスト。別冊S9の「空リストの罠」を防ぎます。

一括処理のテストケース選択と非選択・空リスト
// テストケース一覧
検証期待層
選んだ3件をアーカイブ3件のみ変化結合
選んでないメモ無傷結合
選んだ3件を削除3件のみ削除結合
空リストで実行エラーなし結合
0件選択の判定スキップ単体

「選んだものだけ・選んでないものは無傷」を両方検証（別冊S9）。空リスト（@AC3）も必須。

// 受入条件 → テスト層 // ─── 結合テスト（@MybatisTest+H2）─── @AC1 一括アーカイブ → 選んだ3件が archived=true → 選んでない2件は archived=false ★★★ （両方確認 = 別冊S9） @AC2 一括削除 → 選んだ3件が消える → 選んでないメモは残る @AC3 空リストでエラーなし ★★★ → ids=[] で実行 → 例外が出ない（別冊S9の罠回避） @AC4 原子性 → 単一SQLで全件処理 or 全件失敗 // ─── 単体テスト（Mockito）─── @AC3 空リストの判定 → Service が空なら Mapper を呼ばない → verify(mapper, never()).bulkArchive() // ─── 「選んでないものは無傷」が肝 ─── // 一括処理は「対象だけ」を変えるべき // → 選んでないメモまで変わったら大事故 // → IN句の対象が正しいか検証 ★★★ // ─── 空リストの境界（別冊S3,S9）─── // 「0件選択」は必ず起きる境界 // → IN () エラーを防ぐ仕組みをテスト
POINT — 一括処理は「対象外が無傷」を必ず確認する

一括処理のテストで最も重要なのは「選んだものだけが処理され、選んでないものは一切変わらない」こと。「3件選んでアーカイブ→その3件だけ archived=true、残り2件は false のまま」。もしIN句の条件にミスがあると、選んでないメモまで処理される大事故に。だから「対象が変わった」と「対象外が無傷」の両方を検証する。一括処理は影響が大きいので、テストも厳密にします。

04

設計①：IN句でまとめて処理

PHASE: DESIGNBULK WITH IN CLAUSE

別冊S9の集大成。一括処理をIN句＋foreachで「まとめて1回のSQL」に。ループで1件ずつではなく、単一SQLで全件処理。DB往復1回で効率的、かつ原子的です。

IN句による一括SQL（別冊S9の実践）まとめて1回
// 設計方針（別冊S9）

ループで1件ずつ（N回）ではなく、IN句で1回。UPDATE ... WHERE id IN (...)。Vol.7 の <foreach> を再利用。

// 必要な Mapper メソッド
  • bulkArchive(ids)：一括アーカイブ
  • bulkDelete(ids)：一括削除
// なぜ単一SQLか（別冊S9）
  • DB往復が1回（効率）
  • 単一SQLは原子的（@AC4）
  • シンプル（KISS・別冊S2）
// 一括処理のMapper（別冊S9） // ─── MemoMapper.java（追加）─── void bulkArchive(List<Long> ids); void bulkDelete(List<Long> ids); // ─── MemoMapper.xml（IN句＋foreach）─── // 一括アーカイブ <update id="bulkArchive"> UPDATE memos SET archived = true WHERE id IN <foreach item="id" collection="list" open="(" separator="," close=")"> #{id} </foreach> </update> // → 全て #{} で安全（別冊S6）★ // 一括削除 <delete id="bulkDelete"> DELETE FROM memos WHERE id IN <foreach item="id" collection="list" open="(" separator="," close=")"> #{id} </foreach> </delete> // ─── 実行されるSQL ─── UPDATE memos SET archived=true WHERE id IN (1, 2, 3); // → 3件を1回で更新（DB往復1回）★★★ // ─── ループ（N回）との違い（別冊S9）─── NG: for(id) { update(id); } // N回往復 OK: bulkArchive(ids); // 1回往復 ★★★ // → 別冊S9：DB往復を減らす // → 別冊S6 のN+1対策と同じ原則
POINT — foreach の collection は List なら "list"

MyBatis の <foreach> で、引数が単一の List（bulkArchive(List<Long> ids)）の場合、collection="list" と指定します（MyBatis のデフォルト名）。あるいは @Param("ids") を付けて collection="ids" とも書ける。Vol.7 のタグ検索では @Param 経由でしたが、単一List引数なら "list" が使える。この違いでハマることがあるので、@Param で明示するのが確実です（Section 08）。

05

設計②：空リストと原子性

PHASE: DESIGNEMPTY LIST & ATOMICITY

別冊S9の2大ポイントを設計に。空リスト（@AC3）は IN () エラーになるのでService でガード。原子性（@AC4）は単一SQLで自然に確保。「0件なら何もしない」を仕組みで保証します。

空リスト対策と原子性の設計別冊S9の核心
// 空リスト対策（別冊S9）

IN () は構文エラー。空リストが来たらService でガードして Mapper を呼ばない。

  • Service で ids.isEmpty() をチェック
  • 空なら何もせず return
// 原子性（別冊S9）

単一の UPDATE/DELETE ... WHERE id IN (...) はそれ自体が原子的（DB保証）。追加の @Transactional は不要。

// 設計の要点
  • 空リストガードはServiceに（業務ロジック）
  • 単一SQLで原子性を自動確保
  • シンプルさを保つ（KISS）
// 空リストガードと原子性 // ─── Service で空リストガード（@AC3）─── @Service public class MemoService { public void bulkArchive(List<Long> ids) { // 空リストガード（別冊S9）★★★ if (ids == null || ids.isEmpty()) { return; // 何もしない } mapper.bulkArchive(ids); // → IN () エラーを未然に防ぐ ★ } public void bulkDelete(List<Long> ids) { if (ids == null || ids.isEmpty()) { return; } mapper.bulkDelete(ids); } } // ─── 原子性は単一SQLで自動確保（@AC4）─── // UPDATE ... WHERE id IN (1,2,3) は // 1文のSQL → 原子的（DB保証） // → 3件全部成功 or 全部失敗 // → @Transactional は不要（別冊S9）★★★ // ─── なぜ単一SQLが優れるか（別冊S9）─── ループ + @Transactional: 原子性は得られるが DB往復 N回 単一SQL（IN句）: 原子性も効率も両立 ★★★ + コードもシンプル（KISS・別冊S2） // ─── ガードを Service に置く理由 ─── // 「空なら何もしない」は業務ルール // → Service の責務（別冊S2の層分離） // → Mapper は「SQLを実行するだけ」に保つ
POINT — 「単一SQL」が効率・原子性・単純さを同時に実現

別冊S9で学んだ通り、一括処理は「単一SQL（IN句）」が三拍子そろう：①効率（DB往復1回）、②原子性（DBが自動保証、@Transactional不要）、③単純さ（コードがシンプル・KISS）。ループ＋@Transactional でも原子性は得られますが、DB往復がN回で非効率。「1文のSQLで表現できるなら、それが最善」。空リストのガードだけ Service で足せば完成です。

テスト（一括処理と境界）

PHASE: TESTTEST

一括処理を結合テストで検証。「選んだものだけ・選んでないものは無傷」、そして@AC3（空リスト）の境界。別冊S9の「空リストの罠」を、テストで確実に防ぎます。

結合テスト — 一括処理と空リスト対象と境界の検証
// 検証ポイント
  • @AC1：選んだ3件がアーカイブ、残りは無傷
  • @AC2：選んだ3件が削除、残りは残る
  • @AC3：空リストでエラーなし ★★★

@AC3（空リスト）は別冊S9の罠回避を保証する重要テスト。Service のガードが効くか確認。

// MemoBulkTest.java（結合・H2） @MybatisTest @Import(MemoService.class) // Serviceも対象 class MemoBulkTest { @Autowired MemoService service; @Autowired MemoMapper mapper; // @AC1 選んだものだけアーカイブ ★★★ @Test void 選んだメモだけ一括アーカイブ() { // メモ5件を用意 Memo m1=memo("1"); mapper.insert(m1); Memo m2=memo("2"); mapper.insert(m2); Memo m3=memo("3"); mapper.insert(m3); Memo m4=memo("4"); mapper.insert(m4); Memo m5=memo("5"); mapper.insert(m5); // 3件を一括アーカイブ service.bulkArchive(List.of( m1.getId(), m2.getId(), m3.getId())); // 選んだ3件は archived=true assertThat(mapper.findById(m1.getId()) .isArchived()).isTrue(); // 選んでない m4,m5 は archived=false ★★★ assertThat(mapper.findById(m4.getId()) .isArchived()).isFalse(); assertThat(mapper.findById(m5.getId()) .isArchived()).isFalse(); // → 対象だけ・対象外は無傷（別冊S9）★★★ } // @AC3 空リストでエラーなし ★★★ @Test void 空リストでもエラーにならない() { // 空リスト・null で実行 assertThatNoException() .isThrownBy(() -> { service.bulkArchive(List.of()); service.bulkArchive(null); }); // → IN () エラーが出ない // → Service のガードが効く（別冊S9）★★★ } // @AC2 一括削除 @Test void 選んだメモだけ一括削除() { // 3件削除、残りは残る（省略） } }
POINT — 空リストのテストは「例外が出ないこと」を検証する

@AC3（空リスト）のテストは、AssertJ の assertThatNoException().isThrownBy(...) で「例外が出ないこと」を明示的に検証します。普通のテストは「結果が正しいか」を見ますが、ここでは「エラーにならないこと自体」がテスト対象。別冊S9の「空リストの罠（IN () エラー）」を、Service のガードが確実に防いでいるか保証する。「異常が起きないこと」も立派なテストです。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザで確認。複数選択・一括処理・件数確認を目視。Vol.10 で踏みやすい一括処理特有の地雷を5つ先回りします。

Vol.10 ハマりどころ 5選一括処理特有の地雷

① 空リストで IN () エラー ★★★

  • 0件選択で実行→構文エラー
  • 対処：Service で空リストガード（別冊S9・Section 05）

② foreach の collection 名ミス ★★★

  • 単一List引数で collection="ids" にして動かない
  • 対処：@Param("ids") を付ける、または "list"

③ 一括削除に件数確認なし ★★★

  • 選択した全件が一瞬で消える
  • 対処：JS で件数確認（別冊S9・Section 06）

④ ループで1件ずつ処理

  • N回のDB往復で遅い
  • 対処：IN句でまとめて1回（別冊S9・Section 04）

⑤ 選んでないメモまで処理

  • IN句の対象指定ミス
  • 対処：「対象外が無傷」をテスト（Section 07）
// テスト＆ブラウザ確認 // ─── テスト結果 ─── MemoBulkTest ✓ 選んだメモだけ一括アーカイブ ✓ 選んだメモだけ一括削除 ✓ 空リストでもエラーにならない ★★★ MemoStateTest（既存・Vol.9） ✓ 全て緑 all green ★★★ // ─── ② foreach の collection 名 ─── NG: void bulkArchive(List<Long> ids); <foreach collection="ids" ...> // → 単一List引数だと "ids" で動かない場合 ★ OK: void bulkArchive(@Param("ids") List<Long> ids); <foreach collection="ids" ...> // → @Param で明示すれば確実 ★★★ // （または collection="list" を使う） // ─── ① 空リストガード ─── OK: if (ids == null || ids.isEmpty()) return; mapper.bulkArchive(ids); // → IN () を未然に防ぐ（別冊S9）★★★ // ─── ブラウザ確認（別冊S8,S9）─── // - 複数チェック→一括アーカイブ→3件消える // - 選んでないメモは残る（@AC1）★★★ // - 一括削除→「3件削除します」確認（@AC5） // - 0件で実行→「選択してください」（@AC3） // - 「N件処理しました」表示（別冊S9）
POINT — 一括削除は「全選択→削除」を必ず自分で試す

ブラウザ確認で必ず試すべきなのが「全選択して一括削除」。これは最も危険な操作で、件数確認が正しく出るかを体で確かめる。「全選択（100件）→削除→『100件削除します』と出るか」。もし確認なしで消えたら大事故（別冊S9）。自分が一番怖い操作を、自分で試して安全を確認する。一括処理は影響が大きいぶん、ブラウザ確認も入念に行います。
