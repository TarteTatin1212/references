// US-003 の受入条件（Vol.6で追加分） @AC5 タグを付けて保存 Given: メモ編集で「重要」「急ぎ」を選ぶ When : 保存する Then : メモに2つのタグが紐付く @AC6 タグを付け替える Given: 「重要」が付いたメモ When : 「重要」を外し「急ぎ」を付けて保存 Then : タグが「急ぎ」だけになる @AC7 タグを全部外す Given: タグが付いたメモ When : 全タグのチェックを外して保存 Then : タグが0個になる @AC8 保存の原子性（途中失敗）★トランザクション Given: タグ保存の途中でエラーが起きる When : 保存処理が失敗する Then : 元のタグ状態に戻る（中途半端にしない） // ─── @AC8 がトランザクションの核心 ★★★ ─── // 「途中失敗で元に戻る」 // → @Transactional が必要（別冊S5） // → 全削除→再INSERTを安全にする
POINT — 「付け替え」は内部的に「全削除→全追加」

タグの付け替え（@AC6）は、人間の感覚では「重要を外して急ぎを足す」ですが、実装では「既存タグを全削除して、新しいタグを全追加」が定石（別冊S5）。差分計算（どれを消してどれを足すか）よりシンプルで確実。ただしこれはトランザクション（@AC8）が前提。全削除した後にINSERTが失敗したら、タグが全部消える悲劇を防ぐためです。

PHASE: REQ→TESTAC TO TEST CASES

タグ付けの受入条件をテストに。付ける・付け替える・全外し（@AC5,6,7）は結合テスト、原子性（@AC8）はロールバックのテスト。別冊S5のトランザクション理論が、テストで実証されます。

テストケースと層振り分け原子性のテストが新しい
// テストケース一覧
検証期待層

タグ付け替え古いの消え新が残る結合

// 受入条件 → テスト層 // ─── 結合テスト（@MybatisTest+H2）─── @AC5 付けて保存 → 全削除→INSERTで2件紐付くか @AC6 付け替え → 全削除→新INSERTで入れ替わるか @AC7 全外し → 全削除のみ、0件になるか @AC8 原子性（ロールバック）★★★ → 途中で例外→元の状態に戻るか → これがトランザクションの検証 // ─── 単体テスト（Mockito）─── 保存処理の順序： 1. deleteTagsByMemoId が呼ばれる 2. その後 addTag が呼ばれる → verify で呼び出し順序を検証 // ─── @AC8 のテストが核心 ─── // 「全削除した後にINSERT失敗」を // わざと起こし、元に戻るか確認 // → トランザクションが効いている証明 ★★★ // → 別冊S5の理論をテストで実証
POINT — 「ロールバックのテスト」を書く意味

@AC8 のロールバックのテストは、本シリーズで初めて「失敗時の挙動」を検証します。正常系（成功）だけでなく「失敗したとき正しく元に戻るか」をテストする。これがないと、@Transactional の付け忘れ（別冊S5のあるある①）に気づけません。「壊れたときに正しく壊れるか」をテストするのが、堅牢なシステムの条件です。

04

設計①：全削除→再INSERT

PHASE: DESIGNDELETE-ALL & RE-INSERT

タグ付けの実装方針を「全削除→再INSERT」に決めます（別冊S5）。差分更新よりシンプルで確実。中間テーブルへの操作メソッドを設計します。

なぜ「全削除→再INSERT」か差分更新より確実
// 2つの方針の比較
差分更新全削除→再INSERT
処理消す/足すを計算全消し→全追加
複雑さ高い低い
バグ入りやすい入りにくい
前提—トランザクション

全削除→再INSERTを採用（KISS＝別冊S2）。差分計算は複雑でバグの温床。シンプルが正義。

// 必要な Mapper メソッド
  • deleteTagsByMemoId(memoId)：紐付け全削除
  • addTag(memoId, tagId)：1件追加
// 中間テーブル操作の Mapper // ─── MemoMapper.java（追加）─── // 紐付けを全削除 void deleteTagsByMemoId(Long memoId); // 紐付けを1件追加 void addTag( @Param("memoId") Long memoId, @Param("tagId") Long tagId); // ─── MemoMapper.xml ─── <delete id="deleteTagsByMemoId"> DELETE FROM memo_tags WHERE memo_id = #{memoId} </delete> <insert id="addTag"> INSERT INTO memo_tags (memo_id, tag_id) VALUES (#{memoId}, #{tagId}) </insert> // ─── 全削除→再INSERT の流れ ─── タグを[急ぎ]に付け替える： 1. DELETE FROM memo_tags WHERE memo_id=1 ← 既存全削除 （重要・急ぎ が消える） 2. INSERT memo_tags(1, 急ぎ) ← 再追加 結果：急ぎ だけになる ★ // → 「今のタグ＝送られたtagIds」に // 完全に置き換える // → シンプル・確実（KISS）
POINT — 「全削除→再INSERT」は KISS の実践

差分更新（「どのタグを消して、どれを足すか」を計算）は賢く見えますが、バグの温床。「全部消して、欲しいものだけ入れ直す」方が圧倒的にシンプルで確実（別冊S2の KISS）。性能が問題になる規模でもないので（メモ1件のタグは数個）、シンプルさを優先。ただしトランザクションが前提——次の Section 05 で境界を引きます。

05

設計②：トランザクション境界

PHASE: DESIGNTRANSACTION BOUNDARY

別冊S5の集大成。「全削除→再INSERT」を1つのトランザクションにまとめます。@Transactional を Service メソッドに付ける。途中失敗で全部ロールバック。境界の引き方を設計します。

@Transactional の境界設計Service の1メソッドに
// トランザクションの範囲

「メモにタグを付ける」という1つの業務を、1つの @Transactional メソッドにまとめます（別冊S5）。

  • メモ本体の更新
  • タグ紐付けの全削除
  • 新タグの複数INSERT

これら全部を「全部か、ゼロか」に。1つでも失敗したら全部ロールバック。

// どこに付けるか（別冊S5）
  • Service のメソッドに付ける（業務の単位）
  • Controller・Mapper には付けない
// @Transactional の境界 import org.springframework .transaction.annotation.Transactional; @Service @RequiredArgsConstructor public class MemoService { private final MemoMapper mapper; // 複数操作 → @Transactional 必須 ★★★ @Transactional public void updateWithTags( Memo memo, List<Long> tagIds) { // ① メモ本体を更新 memo.setUpdatedAt( LocalDateTime.now()); mapper.update(memo); // ② 既存タグ紐付けを全削除 mapper.deleteTagsByMemoId( memo.getId()); // ③ 新タグを再INSERT if (tagIds != null) { for (Long tagId : tagIds) { mapper.addTag( memo.getId(), tagId); } } // → ①②③全部が1トランザクション ★★★ // → 途中で例外なら全部ロールバック } } // ─── @Transactional がなかったら ─── // ② で全削除した後 ③ で失敗すると… // → タグが全部消えたまま！（別冊S5） // → @Transactional が「全削除→再INSERT」を // 安全にする ★★★ // ─── 境界の原則（別冊S5）─── // 業務の単位 = トランザクションの単位 // = Service の1メソッド // 「メモにタグを付ける」業務全体が // 1つの「全部か、ゼロか」
POINT — トランザクションが「大胆な全削除」を可能にする

Section 04 で「全削除→再INSERT」というシンプルな方針を選べたのは、@Transactional があるから。「全削除した後に失敗しても元に戻る」と保証されているので、安心して大胆に全削除できる。トランザクションがなければ、怖くて差分更新せざるを得ない。トランザクションが、シンプルな実装を可能にする。設計とトランザクションは表裏一体です（別冊S5）。

実装

PHASE: IMPLIMPLEMENTATION

設計に沿って実装。Controller（複数 tagIds 受け取り）、Service（@Transactional）、View（タグチェックボックス）。既存のメモ更新にタグ付けを組み込みます。

Controller と View複数値の受け取りと表示
// Controller の変更

既存の更新メソッドに @RequestParam List<Long> tagIds を追加。required=false で全外し（0個）にも対応。

// View の変更（編集画面）

全タグをチェックボックスで表示。付いているタグは checked に。Thymeleaf で現在のタグ判定。

// MemoController + 編集View // ─── 編集フォーム表示（タグ含む）─── @GetMapping("/{id}/edit") public String editForm( @PathVariable Long id, Model model) { model.addAttribute("memo", service.findByIdWithTags(id)); model.addAttribute("allTags", tagService.findAll()); return "memos/edit"; } // ─── 更新（タグ付け含む）─── @PostMapping("/{id}") public String update( @PathVariable Long id, @Validated @ModelAttribute Memo memo, BindingResult br, @RequestParam(required = false) List<Long> tagIds) { // 複数値 ★ if (br.hasErrors()) { return "memos/edit"; } memo.setId(id); service.updateWithTags(memo, tagIds); // ↑ @Transactional のメソッド ★★★ return "redirect:/memos"; } // ─── edit.html タグチェックボックス ─── <div th:each="tag : ${allTags}"> <input type="checkbox" name="tagIds" th:value="${tag.id}" th:checked="${#lists.contains( memo.tags.![id], tag.id)}"> <span th:text="${tag.name}"></span> </div> // → 全タグ表示、付いてるのはchecked ★ // → memo.tags.![id] = 現在のタグID一覧
Service（@Transactional）の実装Section 05 のコード
// 実装のポイント
  • @Transactional をメソッドに付ける
  • 更新→全削除→再INSERT の順
  • tagIds が null なら全外し（INSERT スキップ）
  • 例外は握りつぶさない（別冊S5）
// MemoService.updateWithTags @Transactional public void updateWithTags( Memo memo, List<Long> tagIds) { // ① メモ本体更新 memo.setUpdatedAt( LocalDateTime.now()); mapper.update(memo); // ② 既存タグ全削除 mapper.deleteTagsByMemoId( memo.getId()); // ③ 新タグ再INSERT（@AC5,6） if (tagIds != null && !tagIds.isEmpty()) { for (Long tagId : tagIds) { mapper.addTag( memo.getId(), tagId); } } // tagIds が空 → ②だけ実行 = 全外し（@AC7） // 例外は catch せず伝播させる ★★★ // → 失敗時 Spring が自動ロールバック } // ─── 動作まとめ ─── // tagIds=[急ぎ] → 重要削除・急ぎ追加（付け替え） // tagIds=[]/null → 全削除のみ（全外し） // 途中失敗 → 全ロールバック（原子性）
POINT — null と空リストの両方に備える

チェックボックスを0個選んで保存すると、tagIds はnull（送られない）。Service 側で tagIds != null && !tagIds.isEmpty() と両方チェックすると安全。これで「全タグを外す（@AC7）」が正しく動きます。「複数選択UIは0個もありうる」を忘れずに。null チェック漏れは Section 08 のハマりどころでも扱います。

テスト（ロールバックも検証）

PHASE: TESTTEST

タグ付けを結合テストで検証。付ける・付け替える・全外しに加え、「途中失敗でロールバック」を検証します。別冊S5の原子性を、テストで実証する回です。

結合テスト — タグ付けの正常系@AC5,6,7 を実DBで
// 検証ポイント
  • @AC5：タグを付けて保存→2件紐付く
  • @AC6：付け替え→古いの消え新が残る
  • @AC7：全外し→0件になる

@SpringBootTest を使い、@Transactional の Service を実際に動かして検証します（Mapper だけの @MybatisTest では @Transactional のService が動かないため）。

結合テスト — ロールバックの検証@AC8 原子性
// 原子性のテスト方法

わざと失敗を起こす。存在しない tag_id を渡すと、FK制約違反でINSERTが例外。このとき全削除も取り消され、元のタグが残るか検証。

  • 既存タグ「重要」を付けたメモを用意
  • 存在しない tag_id（999）で更新を試みる
  • 例外発生→ロールバック
  • 元の「重要」が残っていることを確認

これで@Transactional が効いていることを実証します。

// ロールバックのテスト（@AC8） // 途中失敗で元に戻るか @Test void 保存途中の失敗で元のタグに戻る() { // Arrange：重要タグを付けておく Tag important = tag("重要"); tagMapper.insert(important); Memo m = memo("会議"); mapper.insert(m); service.updateWithTags(m, List.of(important.getId())); // 今：重要 が付いている // Act：存在しないタグIDで更新（失敗する） assertThatThrownBy(() -> service.updateWithTags(m, List.of(999L))) // 存在しないID .isInstanceOf( DataIntegrityViolationException.class); // → FK制約違反で例外 // → @Transactional でロールバック ★★★ // Assert：元の「重要」が残っている var saved = service .findByIdWithTags(m.getId()); assertThat(saved.getTags()) .extracting(Tag::getName) .containsExactly("重要"); // → 全削除されたはずの「重要」が // ロールバックで戻っている ★★★ // → @Transactional の動作を実証！ } // ─── このテストが証明すること ─── // もし @Transactional を付け忘れたら… // → 全削除が確定し、INSERTだけ失敗 // → タグが消えたまま // → このテストが落ちて気づける ★★★
POINT — ロールバックのテストが @Transactional 付け忘れを検出

このロールバックテストの真価は、「@Transactional を付け忘れたら落ちる」こと。別冊S5のあるある失敗①（付け忘れ）を、テストが自動で検出してくれます。正常系のテストだけだと、付け忘れに気づけません。「失敗時に正しく元に戻るか」を明示的にテストすることで、トランザクションの守りが本当に効いているか確認できます。

08

ブラウザ確認＋ハマりどころ

PHASE: IMPLVERIFY & PITFALLS

テストが緑になったらブラウザで確認。タグの選択・付け替え・全外しを目視。Vol.6 で踏みやすいトランザクション・複数値特有の地雷を5つ先回りします。

Vol.6 ハマりどころ 5選トランザクション特有の地雷

① @Transactional 付け忘れ ★★★

  • 全削除後にINSERT失敗→タグ消失
  • 対処：複数操作の Service に必ず @Transactional（別冊S5）
  • 検知：ロールバックテスト（Section 07）が落ちる

② @Transactional が効かない（自己呼び出し）★★★

  • 同クラス内呼び出しはプロキシを通らない（別冊S5）
  • 対処：Controller→Service の呼び出しなら問題なし

③ 全外しでエラー（tagIds が null）

  • チェック0個で tagIds=null → NPE
  • 対処：tagIds != null && !isEmpty() でチェック（Section 06）

④ 編集画面でタグが選択済みにならない

  • th:checked の判定ミス
  • 対処：#lists.contains(memo.tags.![id], tag.id)

⑤ 例外を握りつぶしてロールバックされない

  • try-catch で例外を消すと Spring が検知できない（別冊S5）
  • 対処：トランザクション内では例外を伝播させる
// テスト＆ブラウザ確認 // ─── テスト結果 ─── MemoTagServiceTest ✓ タグを付けて保存できる ✓ タグを付け替えられる ✓ タグを全部外せる ✓ 保存途中の失敗で元のタグに戻る all green ★★★ // ─── ③ null チェックの徹底 ─── NG: for (Long id : tagIds) { // tagIds=null でNPE ★ mapper.addTag(...); } OK: if (tagIds != null && !tagIds.isEmpty()) { // 両方チェック ★ for (Long id : tagIds) { mapper.addTag(...); } } // ─── ⑤ 例外を握りつぶさない ─── NG: @Transactional public void updateWithTags(...) { try { ... } catch (Exception e) { // 握りつぶし → ロールバックされない ★ } } OK: 例外はそのまま伝播させる（catchしない） // ─── ブラウザ確認 ─── // メモ編集画面で： // - タグを複数チェック→保存→一覧で確認（@AC5） // - 再編集→チェック変更→付け替え（@AC6） // - 全チェック外し→保存→タグ0個（@AC7） // - 付いてるタグがchecked表示される（④）
POINT — トランザクションは「目に見えない」から、テストで守る

トランザクションの怖さは「正常時は付け忘れても動いてしまう」こと。タグ付けが成功している限り、@Transactional がなくても見た目は同じ。問題は失敗したときだけ露見します。だからロールバックテスト（Section 07）が不可欠。「普段は見えないが、いざというとき効く」守りを、テストで可視化する。見えない品質こそテストで守るのです。
