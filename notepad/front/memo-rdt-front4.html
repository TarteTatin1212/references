01

CSSの基本構造

CSS BASICS

CSSは「どの要素を、どう飾るか」を書くもの。セレクタ（どれを）＋プロパティ（何を）＋値（どう）の3点。本編の style.css も、この基本の積み重ねです。まず構造を確認します。

CSSの3要素セレクタ・プロパティ・値
// ルールの形

セレクタ { プロパティ: 値; }。「.btn という要素を、背景色を珊瑚色に」。

// セレクタの種類
記法対象
h1全 h1 タグ
.btnclass="btn"
#mainid="main"
.card pcard内のp
// 読み込み（F1復習）

th:href="@{/css/style.css}" で読み込む。

// CSSの基本構造 /* ─── ルールの形 ─── */ .btn { background: #d2694a; /* 値 */ color: #fff; padding: 6px 16px; } /* セレクタ .btn の要素を飾る */ /* ─── セレクタいろいろ ─── */ h1 { ... } /* 全h1 */ .btn { ... } /* class="btn" */ .card p { ... } /* card内のp */ .btn:hover { ... } /* マウス乗せた時 */ /* ─── 本シリーズの方針 ─── */ /* ・class中心（タグ直接指定は最小限）*/ /* ・1つの style.css にまとめる */ /* ・上から「全体→部品」の順で書く */ /* ─── ファイルの構成（本編style.css）─── */ 1. :root /* デザイン変数 */ 2. * / body /* 全体の土台 */ 3. .container /* レイアウト枠 */ 4. .btn など /* 共通部品 */ 5. .memo-table /* 機能別 */ 6. @media /* スマホ対応 */ /* → 「土台→部品→個別」の順 ★ */
POINT — CSSは「上から土台→部品→個別」の順で書く

CSSファイルは書く順番が大事。本編の style.css は「①変数 → ②全体の土台（body等）→ ③共通部品（btn等）→ ④機能別（テーブル等）→ ⑤スマホ対応」の順。大きいものから小さいものへ。この順序だと「どこに何を書いたか」が探しやすく、後から崩れにくい。CSSが散らかる原因の多くは「順番がバラバラ」。まず構成を決めて書くのが、整ったCSSの第一歩です。

02

:root でデザイン変数

CSS VARIABLES

CSS設計の最重要テクニックが :root での変数（カスタムプロパティ）。色や余白を1か所にまとめて名前を付け、各所から呼び出す。「色を変えたい」が1行の修正で済むようになります。

CSS変数の威力1か所変えれば全部変わる
// :root とは

ページ全体の変数置き場。--accent: #d2694a; のように名前付きで値を定義。

// 使い方

var(--accent) で呼び出す。色を変えたいときは:root の1行だけ直せば、使っている全箇所に反映。

// 本編のカラーパレット

主色　背景　線。これらを変数化。

// 何を変数にするか
  • 色（テーマカラー・背景・線）
  • 余白の基準値
  • 角丸・影など繰り返す値
// :root でデザイン変数 /* ─── 変数を定義（本編style.css）─── */ :root { --ink: #2a2f38; /* 文字色 */ --paper: #f5f3ec; /* 背景 */ --accent: #3a4a6a; /* 主色 */ --line: #d8d4c6; /* 罫線 */ --radius: 4px; /* 角丸 */ } /* ─── 使う：var() で呼び出す ─── */ body { background: var(--paper); color: var(--ink); } .btn { background: var(--accent); border-radius: var(--radius); } /* ─── 変数の威力 ─── */ /* テーマ色を変えたい！ */ 変数なし: #3a4a6a を使った全箇所を探して置換 ★ → 漏れ・ミスが起きる 変数あり: --accent: #3a4a6a → #c0392b に1行変更 → 使用箇所すべてに反映 ★★★ /* ─── このシリーズの色設計 ─── */ /* 本編=藍紺 / 別冊=水浅葱 / GitHub=チャコール */ /* → 各シリーズで :root だけ差し替え ★ */ /* → 構造は同じ、色だけ変える */
POINT — 「同じ値が2回以上出たら変数にする」

CSS変数を使う判断は単純：「同じ値（色・余白）が2回以上出てきたら変数にする」。主色 #3a4a6a はボタン・見出し・リンクで何度も使う→ --accent に。これは本編で学んだDRY（繰り返さない）のCSS版。変数化すれば「テーマ色を変えたい」が1行で済み、色の不統一も防げる。実際このフロント別冊シリーズも、:root の色だけ変えて珊瑚×砂色にしています。

03

クラスの命名

NAMING CLASSES

クラス名は「見た目」でなく「役割」で名付けるのが鉄則。.red（赤い）ではなく .btn-danger（危険なボタン）。役割で名付ければ、後で色を変えても名前が嘘にならない。命名は地味ですが重要です。

役割ベースの命名見た目で名付けない
// 良い命名・悪い命名
悪い（見た目）良い（役割）
.red.error
.big-text.title
.box1.memo-card
// なぜ役割で名付けるか

.red だと「赤をやめて青に」したとき名前が嘘になる。.error なら色を変えても意味が通る。

// 命名の一貫性

本編では .btn/.cat-tag/.tag-chip/.flash など、機能を表す名前で統一。

// クラスの命名 /* ─── 見た目で名付ける（NG）─── */ .red { color: red; } /* → 「赤をやめたい」とき名前が嘘に ★ */ .mt20 { margin-top: 20px; } /* → 余白を変えたら名前と不一致 ★ */ /* ─── 役割で名付ける（OK）─── */ .error { color: var(--accent); } /* → 色を変えても「エラー」の意味は不変 ★ */ .btn { ... } .btn-danger { ... } /* 削除ボタン等 */ /* ─── 本編で使った命名（役割ベース）─── */ .container /* レイアウト枠 */ .btn /* ボタン共通 */ .memo-table /* メモ一覧の表 */ .cat-tag /* カテゴリの札 */ .tag-chip /* タグの小札 */ .flash /* 通知メッセージ */ .error /* エラー文 */ /* → 全部「何のためか」が分かる名前 ★★★ */ /* ─── 複数語はハイフンで ─── */ .memo-card .tag-chip .btn-danger /* → ケバブケース（小文字+ハイフン）が定番 */ /* ─── 共通＋個別の組み合わせ ─── */ <button class="btn btn-danger">削除</button> /* btn=共通スタイル + btn-danger=赤色 ★ */ /* → 共通部分を再利用できる */
POINT — 「色を変えたら名前が嘘になる」名前は避ける

命名の良し悪しを判断する簡単なテスト：「もし色やサイズを変えたら、名前が嘘にならないか?」。.red は青にしたら嘘。.error は何色でも「エラー」のまま正しい。.big は小さくしたら嘘。.title はサイズに関係なく正しい。「見た目」は変わるが「役割」は変わりにくい。だから役割で名付ける。これは本編の「意図を表す名前を付ける」という設計原則の、CSS版です。

04

テーブルのレイアウト

TABLE LAYOUT

メモ一覧で使うテーブルのスタイリング。border-collapse で罫線を整え、ヘッダーを区別し、行に余白を。本編Vol.2〜4 の .memo-table を題材に、読みやすい表の作り方を学びます。

テーブルの整え方罫線・余白・ヘッダー
// テーブルの基本
  • border-collapse: collapse：罫線を重ねて1本に
  • th/td に padding：行に余白
  • th に背景色：ヘッダーを区別
// 幅の調整

width: 100% で横いっぱい。列幅は内容に応じて自動、または th に width 指定。

// 操作列の工夫

編集・削除ボタンの列は white-space: nowrap で折り返さない。

// .memo-table（本編Vol.2〜4） /* ─── 基本 ─── */ .memo-table { width: 100%; border-collapse: collapse; /* → 罫線を重ねて1本に ★ */ } /* ─── セルの余白と罫線 ─── */ .memo-table th, .memo-table td { padding: 8px 10px; border-bottom: 1px solid var(--line); text-align: left; } /* ─── ヘッダーを区別 ─── */ .memo-table th { background: var(--paper-2); font-size: 13px; } /* ─── 操作列は折り返さない ─── */ .ops { white-space: nowrap; } /* → 編集/削除が縦に崩れない ★ */ /* ─── 空行の案内 ─── */ .empty { text-align: center; color: var(--muted); } /* ─── 仕上げ：ホバーで行を強調（任意）─── */ .memo-table tbody tr:hover { background: #faf8f2; } /* → マウスを乗せた行が分かりやすく ★ */
POINT — テーブルは「border-collapse」で罫線を整える

表を作って「罫線が二重に見える」のは、border-collapse を指定していないから。デフォルトでは各セルの罫線が離れて二重になる。border-collapse: collapse を table に付けると、隣接する罫線が重なって1本になり、すっきりします。これはテーブルスタイリングの最初の一手。あとは th/td に padding で余白を与えれば、ぐっと読みやすい表になります。

05

カードとフォームのレイアウト

CARD & FORM LAYOUT

詳細画面のカード、入力画面のフォームのレイアウト。flexbox や余白設計で、整然と並べる。本編の .content・.field を題材に、見やすい配置のコツを学びます。

カードとフォームの配置余白とflexで整える
// カード（詳細表示）

本文を枠で囲い、内側に余白。border＋padding＋border-radius で「カード」に。改行を保つ white-space: pre-wrap も。

// フォームの項目（field）

各項目を .field で縦に積む。label を上、input を下、間に余白。input は width:100% で揃える。

// flexbox で横並び

タグのチップやボタンを横に並べるには display:flex ＋ gap。折り返しは flex-wrap。

// カードとフォーム /* ─── カード（詳細の本文）─── */ .content { background: #fff; padding: 16px; border: 1px solid var(--line); border-radius: 6px; white-space: pre-wrap; /* → 改行・空白を保って表示 ★ */ } /* ─── フォームの各項目 ─── */ .field { margin-bottom: 16px; } .field label { display: block; /* labelを上に */ font-weight: bold; margin-bottom: 4px; } .field input, .field textarea { width: 100%; /* 横幅を揃える */ padding: 8px; border: 1px solid var(--line); border-radius: 4px; } /* ─── flexで横並び（タグチップ等）─── */ .tag-checks { display: flex; flex-wrap: wrap; /* 入りきらなければ折返し */ gap: 8px; /* 要素間の余白 ★ */ } /* → チェックボックスがきれいに並ぶ ★★★ */ /* ─── エラー時の入力欄を赤枠に ─── */ .is-invalid { border-color: #c0392b; } /* → th:errorclass="is-invalid" と連携（F3）★ */
POINT — 横並びは「float」でなく「flexbox」で

要素を横に並べたいとき、古い float ではなくdisplay: flex ＋ gap を使うのが現代の定番。flex は並べ方・揃え方・余白を簡単に制御でき、gap で要素間の間隔もすっきり。flex-wrap: wrap を足せば画面幅に応じて自動で折り返してくれる（次節のレスポンシブにも繋がる）。タグチップ、ボタン群、メニューなど、横並びは全部 flexで考えると整理できます。

06

レスポンシブ対応

RESPONSIVE DESIGN

レスポンシブとは、PC・スマホ・タブレットなど画面幅に応じて見た目を調整すること。@media で「幅が狭いとき」のスタイルを追加します。本編のHTML教材も、この仕組みでスマホ対応しています。

@media で画面幅対応狭いとき調整
// レスポンシブの考え方

基本のスタイルを書き、@media で「幅が狭いときだけ」上書き。PC向けを基本に、スマホで調整するのが分かりやすい。

// viewport の指定（必須）

HTMLの <head> に <meta name="viewport"> が必要。これがないとスマホで正しく縮小されない。

// よくある調整
  • 余白を狭く（padding削減）
  • 横並びを縦並びに（flex-direction）
  • 文字サイズを少し小さく
// レスポンシブ対応 /* ─── ① HTMLに viewport（必須）─── */ <meta name="viewport" content="width=device-width, initial-scale=1.0"> /* → これがないとスマホで縮小されない ★★★ */ /* ─── ② 基本スタイル（PC向け）─── */ .container { max-width: 760px; padding: 24px 20px; } /* ─── ③ @media で狭い時を上書き ─── */ @media (max-width: 720px) { .container { padding: 16px 12px; /* 余白を狭く */ } h1 { font-size: 20px; /* 見出し小さく */ } .filter { flex-direction: column; /* 縦並びに */ } } /* → 画面幅720px以下で適用 ★★★ */ /* ─── ブレークポイントの目安 ─── */ ~480px スマホ（縦） ~768px タブレット 769px~ PC /* → 本シリーズは720px前後で1段階だけ */ /* → 凝りすぎず「狭い時の最低限」でOK ★ */ /* ─── flexは自然にレスポンシブ ─── */ .tag-checks { display: flex; flex-wrap: wrap; } /* → 幅が狭まると自動で折り返す */ /* → @media なしでもある程度対応 ★ */
POINT — まず viewport、それから @media

スマホ対応で一番忘れがちなのが、HTMLの <meta name="viewport">。これが無いと、スマホはPC幅で表示して全体を縮小するため、文字が極小になる。まずこれを入れる。その上で @media (max-width: ...) で「幅が狭いときだけ」余白や並びを調整する。凝りすぎず「狭い時の最低限の調整」で十分。flex-wrap を使っておけば、@media なしでもある程度自動で対応してくれます。

07

CSSのあるある失敗

CSS PITFALLS

CSSで初心者がハマる失敗を5つ。!important の乱用、値の直書き、viewport 忘れなど。知っていれば散らからないCSSが書けます。

CSSあるある 5選先に知れば回避できる

① 色・余白を直書きでバラ撒く ★★★

  • 同じ色が各所に散り、変更困難
  • 対処：:root の変数に（Section 02）

② !important を乱用

  • 後で上書きできず詰む
  • 対処：セレクタを工夫、importantは最終手段

③ 見た目でクラス命名

  • .red など、変更で名前が嘘に
  • 対処：役割で命名（Section 03）

④ viewport 忘れでスマホ崩れ ★★★

  • スマホで文字が極小
  • 対処：meta viewport を入れる（Section 06）

⑤ CSSの変更が反映されない

  • ブラウザキャッシュが原因のことも
  • 対処：スーパーリロード／開発時はcache=false
// あるある失敗の対処 /* ─── ① 値の直書き ─── */ NG: .btn { background: #3a4a6a; } .title { color: #3a4a6a; } .link { color: #3a4a6a; } /* → 同じ色が散在、変更が大変 ★ */ OK: :root { --accent: #3a4a6a; } .btn { background: var(--accent); } /* → 1か所で管理 ★★★ */ /* ─── ② !important 乱用 ─── */ NG: .btn { color: red !important; } /* → 後で上書きできない ★ */ OK: セレクタを具体的に書いて優先度で解決 /* ─── ⑤ 反映されない時 ─── */ /* application.properties で開発時: */ spring.thymeleaf.cache=false /* + ブラウザでスーパーリロード */ /* (Ctrl+Shift+R / Cmd+Shift+R) ★ */ /* ─── 良いCSSの条件 ─── */ ✓ 値は :root の変数に ✓ !important は最終手段 ✓ クラスは役割で命名 ✓ viewport を入れる ✓ 反映されない時はキャッシュ疑う ★★★
POINT — 「変更が反映されない」はキャッシュをまず疑う

CSSを直したのに画面が変わらないとき、コードが正しければブラウザのキャッシュが原因のことが多い。ブラウザは一度読んだCSSを使い回すため。対処はスーパーリロード（Ctrl+Shift+R / Cmd+Shift+R）。開発中は application.properties に spring.thymeleaf.cache=false を入れておくと、テンプレートの変更も即反映される。「直したのに変わらない」で悩む時間を、この知識で減らせます。
