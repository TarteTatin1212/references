01

チェックボックスの仕組み

HOW CHECKBOXES WORK

チェックボックスは「オン/オフ」を選ぶ入力。テキスト入力と違う特徴があります。チェックしたものだけ送信され、同じ name にすると複数値になる。まずこの基本を理解します。

チェックボックスの特徴テキスト入力とは違う
// チェックボックスの基本
  • type="checkbox" でオン/オフ
  • チェックした分だけ送信される
  • 未チェックは送られない（重要）
// テキスト入力との違い
textcheckbox
送信常にチェック時のみ
値入力文字value属性
複数―同nameで可
// value が大事

チェック時に何の値を送るかは value で決まる。タグなら value="${tag.id}"。

// チェックボックスの仕組み // ─── 基本 ─── <input type="checkbox" name="agree" value="yes"> # チェックすると agree=yes が送信 # チェックしないと…何も送られない ★ // ─── テキストとの違い ─── <input type="text" name="title"> → 空でも title= が送られる <input type="checkbox" name="ok"> → 未チェックなら ok 自体が送られない ★★★ # この違いが後で重要（空リスト対策） // ─── 同じ name で複数 ─── <input type="checkbox" name="tagIds" value="1"> 重要 <input type="checkbox" name="tagIds" value="2"> 急ぎ <input type="checkbox" name="tagIds" value="3"> 会議 1と3をチェック → tagIds=1&tagIds=3 が送信 # → 同じ name で複数値 ★★★ # → サーバーは List で受け取る（次節） // ─── value の役割 ─── value="${tag.id}" # → チェック時にこの id が送られる # → 「どのタグを選んだか」を id で伝える ★
POINT — チェックボックスは「未チェックなら送られない」

チェックボックスの最重要の特性：チェックしたものだけ送信され、未チェックは送られない。テキスト入力が「空でも送る」のと対照的。だから「1つもチェックしないと、その項目自体が送られてこない」。これが本編Vol.10 の「空リスト対策」に繋がる——「0件選択」だとサーバーに tagIds が届かず null になる。複数選択を扱うなら、この特性を必ず頭に入れておきましょう。

02

複数選択とListバインド

LIST BINDING

同じ name のチェックボックス群は、サーバー側でList として受け取れます。本編Vol.5 のタグ選択がこれ。チェックしたタグIDのリストが List<Long> tagIds に入る。画面の複数選択とJavaのListが繋がる仕組みです。

複数値→Listへ画面とJavaを繋ぐ
// Listバインドの流れ
  1. 同じ name="tagIds" で複数チェック
  2. tagIds=1&tagIds=3 が送信
  3. サーバーが List<Long> に変換
  4. [1, 3] として受け取る
// Form クラス側（本編Vol.5）

private List<Long> tagIds; と定義すれば、複数値が自動でListに。

// th:each でチェックボックス生成

タグ一覧を th:each で回し、各タグのチェックボックスを生成。

// 複数選択とListバインド // ─── 画面：th:each でチェックボックス生成 ─── <div class="tag-checks"> <label th:each="tag : ${tags}" class="tag-check"> <input type="checkbox" name="tagIds" th:value="${tag.id}"> <span th:text="${tag.name}"></span> </label> </div> # → タグの数だけチェックボックス生成 ★ # → 全部 name="tagIds"（同じ名前） // ─── 送信されるデータ ─── 「重要」(id=1)と「会議」(id=3)をチェック → tagIds=1&tagIds=3 // ─── サーバー：Listで受け取る（Vol.5）─── # Form クラス: public class MemoForm { private String title; private List<Long> tagIds; # List ★★★ } ↓ tagIds = [1, 3] として受け取れる # → 画面の複数選択 = JavaのList ★★★ // ─── Controller（本編Vol.6）─── @PostMapping("/memos") String create(@ModelAttribute MemoForm form) { # form.getTagIds() で [1,3] が取れる service.create(form); # タグ付け } # → サーバーで memo_tags に保存（Vol.6）
POINT — 「同じ name」が複数値を List にまとめる鍵

複数選択の核心は「チェックボックスを全部同じ name にする」こと。name="tagIds" で統一すると、チェックした値が tagIds=1&tagIds=3 と同じキーで複数送られ、サーバー側の List<Long> tagIds に自動でまとまる。もし name をバラバラ（tagId1, tagId2…）にすると、Listにならず扱いづらい。「複数選びたいものは、同じ name で並べる」——これが画面とサーバーを繋ぐ約束です。

03

th:field でチェック状態

CHECKED STATE

編集画面では、既に付いているタグを最初からチェック済みにしたい。th:field="*{tagIds}" を使えば、選択済みのタグが自動でチェックされる。手動で checked を判定する必要がありません。

th:field の自動チェック選択済みを反映
// 編集時の課題

メモ編集で「既に付いているタグ」をチェック済みで表示したい。手動だと面倒。

// th:field が解決

th:field="*{tagIds}" ＋ th:value="${tag.id}"。tagIds に含まれる id のチェックボックスを自動でcheck。

// 手動との比較

手動なら th:checked="${#lists.contains(...)}" と書く必要が。th:field なら一発。

// th:field でチェック状態 // ─── th:field で自動チェック ─── <label th:each="tag : ${tags}"> <input type="checkbox" th:field="*{tagIds}" th:value="${tag.id}"> <span th:text="${tag.name}"></span> </label> # → memoForm.tagIds に tag.id が含まれれば # 自動で checked になる ★★★ // ─── 仕組み ─── 編集対象メモのタグ = [1, 3]（重要・会議） memoForm.tagIds = [1, 3] ↓ th:field が判定 tag.id=1（重要）→ ☑ checked tag.id=2（急ぎ）→ ☐ tag.id=3（会議）→ ☑ checked # → 既存のタグが最初からチェック済み ★ // ─── 手動だと面倒（参考）─── th:field なし: <input type="checkbox" name="tagIds" th:value="${tag.id}" th:checked="${#lists.contains( memoForm.tagIds, tag.id)}"> # → contains 判定を自分で書く ★ th:field あり: <input type="checkbox" th:field="*{tagIds}" th:value="${tag.id}"> # → 判定不要、自動 ★★★ // ─── F3の th:field と一貫 ─── # テキストもチェックボックスも同じ th:field # → 型に応じて適切に処理してくれる ★
POINT — th:field はチェックボックスでも「状態の自動反映」

F3 で学んだ th:field は、テキスト入力では value を、セレクトでは selected を自動生成しました。チェックボックスでは checked を自動判定します。th:field="*{tagIds}" と th:value="${tag.id}" を組めば、tagIds に含まれる id のチェックボックスだけ checkedに。「th:field は、入力欄の種類を問わず"現在の状態"を反映する」万能属性。手動の contains 判定が不要になります。

04

一括操作のチェックボックス

BULK SELECTION

本編Vol.10 の一括操作。一覧の各行にチェックボックスを置き、選んだメモをまとめてアーカイブ・削除。タグ選択（F6前半）と同じ仕組みで、メモIDのリストを送ります。

一括操作のUI行ごとにチェック
// 構成
  • 一覧全体を1つの form で囲む
  • 各行に name="ids" のチェックボックス
  • 一括ボタン（アーカイブ／削除）
// formaction で送信先を分岐

同じ選択を、アーカイブと削除で使い分け。ボタンの formaction で送信先を変える（本編Vol.10）。

// 送られるデータ

チェックした行の ids=1&ids=5&ids=8 が List<Long> ids に。タグ選択と同じ。

// 一括操作のチェックボックス <!-- 一覧全体を form で囲む --> <form id="bulkForm" method="post"> <!-- 一括ボタン（送信先を分岐）--> <button type="submit" th:formaction="@{/memos/bulk/archive}"> 一括アーカイブ </button> <button type="submit" th:formaction="@{/memos/bulk/delete}" onclick="return confirmDelete()"> 一括削除 </button> <!-- 各行にチェックボックス --> <tr th:each="memo : ${memos}"> <td> <input type="checkbox" name="ids" th:value="${memo.id}" class="memo-check"> </td> <td th:text="${memo.title}"></td> </tr> </form> // ─── 送られるデータ ─── id=1,5,8 をチェック → ids=1&ids=5&ids=8 → サーバー List<Long> ids = [1,5,8] ★ // ─── formaction で分岐（Vol.10）─── 同じチェック選択を… [一括アーカイブ] → /memos/bulk/archive [一括削除] → /memos/bulk/delete # → 選択は共通、操作で送信先を変える ★★★ // ─── Controller（本編Vol.10）─── @PostMapping("/memos/bulk/archive") String bulkArchive( @RequestParam(required=false) List<Long> ids) { ... } # → タグ選択と全く同じ仕組み ★
POINT — タグ選択も一括操作も「同じ name で List」の応用

本編Vol.5 のタグ複数選択と、Vol.10 の一括操作は全く同じ仕組みです。「同じ name のチェックボックス → List で受け取る」。タグなら name="tagIds"、一括操作なら name="ids"。一度この型を理解すれば、「複数のものを選んで一度に処理する」UI は全部同じパターンで作れる。違いはformaction で送信先を分けるくらい。応用が効く、汎用的なテクニックです。

05

全選択を作る（JS）

SELECT ALL

一括操作で便利な「全選択」チェックボックス。これはJavaScriptが必要な数少ない場面（本シリーズ初のJS）。ヘッダーのチェックで全行を一括オン/オフします。シンプルなJSで実現できます。

全選択チェックボックスJSで一括操作
// なぜ JS が必要か

「全選択を押したら全行をチェック」はブラウザ上の動き。サーバーは関係ない。だからJavaScriptで行う。

// 仕組み
  1. ヘッダーに「全選択」チェックボックス
  2. 変化したら全 .memo-check を同じ状態に
// JSの置き場所（F1復習）

static/js/bulk-select.js に書き、th:src="@{/js/bulk-select.js}" で読み込む。

// 全選択（static/js/bulk-select.js） // ─── 画面：全選択チェックボックス ─── <th> <input type="checkbox" id="selectAll"> </th> # 各行は class="memo-check"（前節） // ─── JS：全選択の制御 ─── document.addEventListener('DOMContentLoaded', () => { const all = document .getElementById('selectAll'); const checks = document .querySelectorAll('.memo-check'); // 全選択チェックの変化を監視 all.addEventListener('change', () => { checks.forEach(c => { c.checked = all.checked; // 全行を同じ状態に ★ }); }); }); # → 全選択ON → 全行ON / OFF → 全行OFF ★★★ // ─── 読み込み（F1復習）─── <script th:src="@{/js/bulk-select.js}"></script> # → static/js/ から読み込む // ─── DOMContentLoaded の意味 ─── # HTMLの読み込み完了後にJSを動かす # → 要素が存在してから操作（安全）★ # → これがないと「要素が無い」エラーも // ─── 本シリーズ初のJS ─── # 前半(Vol.1〜6)はJSなしで来た # 一括操作で初めてJSが必要に ★ # → 詳しくはフロント別冊F7で
POINT — JSは「サーバーでできないブラウザ上の動き」に使う

「全選択」はJavaScriptが必要な典型例。なぜなら「ボタンを押した瞬間に画面のチェックを変える」のは、サーバーに行かずブラウザ上で完結する動きだから。Thymeleaf（サーバー）は画面を描くまでが仕事で、描いた後のその場の操作はJSの領域。「表示はThymeleaf、その場の動きはJS」という役割分担。本シリーズは前半JSなしでしたが、ここで初めて登場。詳しくはフロント別冊F7 で扱います。

06

選択件数の確認

SELECTION COUNT

一括削除のような危険な操作では、「何件選んだか」を確認させるのが親切。0件なら警告、削除前に件数を表示。本編Vol.10 の「件数確認」を、JSで実装します。

削除前の件数確認誤操作を防ぐ
// なぜ確認するか
  • 一括削除は取り返しがつかない
  • 0件選択で実行 → 何も起きないと不親切
  • 件数を見せて誤操作を防ぐ
// JSで実装
  1. チェック済みの数を数える
  2. 0件なら「選択してください」
  3. 1件以上なら「N件削除しますか?」
// 本編Vol.10 と対応

サーバー側でも空リストガード（Vol.10）。画面でもサーバーでも守る（多層防御）。

// 選択件数の確認（JS） // ─── 削除前の確認関数 ─── function confirmDelete() { // チェック済みの数を数える const n = document .querySelectorAll( '.memo-check:checked').length; // 0件なら警告（@AC3・Vol.10） if (n === 0) { alert('メモを選択してください'); return false; # 送信中止 ★ } // 件数を見せて確認 return confirm( n + '件を完全に削除します。\n' + 'この操作は取り消せません。'); # → OKで送信、キャンセルで中止 ★★★ } // ─── ボタンに紐付け（前節）─── <button onclick="return confirmDelete()"> 一括削除 </button> # → false を返すと送信されない ★ // ─── :checked セレクタ ─── '.memo-check:checked' # → チェック済みの要素だけ取得 # → .length で件数が分かる ★ // ─── 多層防御（本編Vol.10）─── 画面（JS）: 0件なら警告・件数確認 + サーバー: 空リストガード（Vol.10） # → 画面が突破されてもサーバーが守る ★★★ # → フロント別冊F4・本編の「多層防御」 // ─── 件数表示でUX向上 ─── # 「3件削除します」と具体的に # → 「全部消えるところだった」と気づける ★
POINT — 危険な操作ほど「件数を見せて」確認する

一括削除のような取り返しのつかない操作では、「3件を削除します」と件数を具体的に見せて確認するのが鉄則。ただ「削除しますか?」より、件数があると「全選択で100件消えるところだった!」と気づける。また0件選択は警告して送信を止める。これは画面側の親切。ただし画面のJSは回避できるので、本編Vol.10 のようにサーバー側でも空リストガードを必ず入れる。画面とサーバー、両方で守るのが安全です。

07

複数選択のあるある失敗

PITFALLS

チェックボックス・複数選択で初心者がハマる失敗を5つ。name の不統一、value 忘れ、0件未対応など。知っていれば回避できます。

複数選択あるある 5選先に知れば回避できる

① name がバラバラで List にならない ★★★

  • tagId1, tagId2 と別名にする
  • 対処：全部同じ name="tagIds"（Section 02）

② value 忘れで何が選ばれたか不明

  • value がないと "on" が送られる
  • 対処：th:value="${tag.id}"（Section 01）

③ 0件選択を想定しない ★★★

  • 未チェックだと項目が送られず null
  • 対処：JS警告＋サーバーガード（Section 06）

④ 編集時にチェックが反映されない

  • 既存タグがチェックされない
  • 対処：th:field を使う（Section 03）

⑤ JSが要素を見つけられない

  • HTML読込前にJSが走る
  • 対処：DOMContentLoaded（Section 05）
// あるある失敗の対処 // ─── ① name を統一 ─── NG: name="tagId1" name="tagId2" # → List にまとまらない ★ OK: name="tagIds"（全部同じ）★★★ // ─── ② value を付ける ─── NG: <input type="checkbox" name="ids"> # → チェックすると "on" が送られる ★ OK: th:value="${memo.id}" # → どのメモか id で分かる // ─── ③ 0件対策（画面＋サーバー）─── 画面: if(n===0){ alert(...); return false; } サーバー: if(ids==null||ids.isEmpty()) return; # → 両方で守る（多層防御）★★★ // ─── ④ 編集時のチェック ─── NG: name="tagIds"（th:fieldなし） # → 既存タグがチェックされない OK: th:field="*{tagIds}" th:value="${tag.id}" # → 自動でチェック済みに ★ // ─── ⑤ JSのタイミング ─── OK: document.addEventListener( 'DOMContentLoaded', () => {...}); # → HTML読込後に実行（安全）★ // ─── 良い複数選択の条件 ─── ✓ name を統一して List に ✓ value で値を明示 ✓ 0件は画面＋サーバーで対策 ✓ 編集は th:field でチェック反映 ✓ JSは DOMContentLoaded 後 ★★★
POINT — 複数選択の基本は「name 統一・value 明示・0件対策」

複数選択で押さえるべき3点：①name を統一（List にまとめる）、②value を明示（何を選んだか伝える）、③0件を対策（未チェックだと送られない特性への備え）。特に③は画面（JS警告）とサーバー（空リストガード）の両方で守る——本編Vol.10 で学んだ通り。チェックボックスの「チェックした分だけ送られる」という特性を理解していれば、これらの対策がなぜ必要かも腑に落ちます。
