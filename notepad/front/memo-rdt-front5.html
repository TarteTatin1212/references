FRONT.05 ── LIST, FILTER & CONDITIONAL DISPLAY

対応する本編はVol.4・5。一覧画面にカテゴリ列・タグチップを表示し、絞り込みセレクトで「このカテゴリだけ」を表示する。選んだ値を保持するth:selected、状況に応じて表示を変える条件分岐。F2 の th:each/th:if と F4 の CSS を組み合わせて、実用的な一覧を完成させます。「データを見やすく・絞り込めるように」が今回のテーマです。

💡 一覧・絞り込みでつまずく点

① 絞り込みは <form method="get">（検索はGET）／② 選択を保持するのは th:selected="${現在値 == 候補値}"／③ onchange="this.form.submit()" で選んだ瞬間に送信／④ タグ等の子リストは入れ子の th:each で展開。これらを押さえれば、絞り込み付き一覧が作れます。

01

一覧に情報を足す

ENRICH THE LIST

F2 で作った素朴な一覧に、本編Vol.4・5 でカテゴリ・タグが加わります。一覧は「ぱっと見て分かる」のが命。何の情報を、どう見せるかを設計します。

一覧の進化情報を見やすく足す
// 一覧に足すもの
  • カテゴリ列（Vol.4）：色付きの札で
  • タグ列（Vol.5）：複数の小札で
  • 絞り込み（Vol.4）：カテゴリで選別
// 見やすさの原則
  • 色で分類を一目で（カテゴリ色）
  • チップで複数情報をコンパクトに
  • 絞り込みで必要な分だけ表示
// 使う道具

F2 の th:each/th:if＋F4 の CSS（チップ・札のスタイル）の合わせ技。

// 一覧の進化 // ─── F2時点（素朴）─── | タイトル | 更新日時 | 操作 | | 会議メモ | 05-01 | 編集 | // ─── Vol.4・5（情報リッチ）─── | タイトル | カテゴリ | タグ | 更新 | 操作 | | 会議メモ | [仕事] | 重要 急ぎ | 05-01| 編集 | | 買い物 | [私用] | ― | 05-01| 編集 | ↑色付き札 ↑小札（複数） // → 一目で分類とタグが分かる ★★★ // ─── 絞り込み（Vol.4）─── カテゴリで絞り込み: [仕事 ▼] ↓ | 会議メモ | [仕事] | ... | （仕事のメモだけ表示） // → 必要な分だけ見られる ★ // ─── 使う技術 ─── th:each 行・タグの繰り返し（F2） th:if カテゴリ有無の出し分け（F2） th:href 絞り込みリンク（F2） <select> 絞り込みフォーム（今回） CSS チップ・札のスタイル（F4） // → これまでの総合 ★★★
POINT — 一覧は「情報量」と「見やすさ」のバランス

一覧画面の設計で大事なのは「必要な情報を、見やすく」のバランス。情報を詰め込みすぎるとごちゃつき、少なすぎると役に立たない。メモ帳なら「タイトル・カテゴリ・タグ・更新日時」が適量。カテゴリは色付き札、タグは小さなチップと視覚的に区別すれば、情報が多くても整理されて見える。「何を見せ、何を省くか」も画面設計の一部です。

02

カテゴリ列の表示

CATEGORY COLUMN

本編Vol.4 で、メモにカテゴリが紐づきました。一覧にカテゴリ名を色付きの札で表示します。カテゴリが未設定（null）の場合の出し分けも、th:if で対応します。

カテゴリ列の組み立て色付き札＋null対応
// 表示の方針
  • カテゴリあり：色付きの札で名前
  • カテゴリなし（null）：「未分類」
// 色を動的に（th:style）

カテゴリごとの色を th:style で札の背景に。データ駆動の色付け。

// null チェックは必須（F2復習）

memo.category が null のとき .name を触ると落ちる。th:if で守る。

// カテゴリ列の表示 <!-- 一覧の各行のカテゴリ列 --> <td> <!-- カテゴリあり：色付き札 --> <span th:if="${memo.category != null}" class="cat-tag" th:style="'background:' + ${memo.category.color}" th:text="${memo.category.name}">仕事</span> <!-- カテゴリなし：未分類 --> <span th:if="${memo.category == null}" class="cat-none">未分類</span> </td> # → ありなしで出し分け（F2のth:if）★★★ # → null を直接触らないので安全 /* ─── CSS（F4の知識）─── */ .cat-tag { display: inline-block; padding: 2px 10px; border-radius: 12px; /* 丸い札 */ font-size: 12px; color: #fff; } .cat-none { color: var(--muted); font-size: 12px; } // ─── th:style で動的な色 ─── th:style="'background:' + ${memo.category.color}" # category.color が #3a4a6a なら # → style="background:#3a4a6a" ★ # → カテゴリごとに違う色の札に
POINT — データ駆動の色は th:style で、固定の色は CSS で

色の付け方には2種類あります。カテゴリごとに変わる色（データに入っている）は th:style="'background:' + ${...}" で動的に。全カテゴリ共通の形・サイズ（角丸・余白・文字色）は .cat-tag クラスでCSSに。「変わるものは th:style、変わらないものは CSS」と分ければ、見た目が整いつつ、データに応じた表現もできる。役割分担がきれいになります。

03

タグチップの表示（入れ子each）

TAG CHIPS

本編Vol.5 で、1つのメモに複数のタグが付くようになりました。一覧の各行で、タグの数だけ小さなチップを並べます。「行の繰り返し」の中で「タグの繰り返し」——入れ子の th:each です。

入れ子の th:each行の中でタグを繰り返す
// 入れ子の繰り返し

外側：th:each="memo : ${memos}"（行）。内側：th:each="tag : ${memo.tags}"（タグ）。各メモのタグ分チップを並べる。

// タグが空の場合

タグなしのメモは th:if="${#lists.isEmpty(memo.tags)}" で「―」表示。

// チップのスタイル（F4）

小さく・横並び・折り返し。display:inline-block や flex で。

// タグチップの表示（入れ子each） <!-- 一覧の各行（外側のeach）--> <tr th:each="memo : ${memos}"> ... <!-- タグ列：内側のeach --> <td> <span th:each="tag : ${memo.tags}" class="tag-chip" th:text="${tag.name}">タグ</span> <!-- タグが無いとき --> <span th:if="${#lists.isEmpty(memo.tags)}" class="cat-none">―</span> </td> </tr> # 外: memoを繰り返す # 内: その memo.tags を繰り返す ★★★ # → メモごとに、タグの数だけチップ // ─── 表示イメージ ─── 会議メモ | [重要][急ぎ][会議] ← 3チップ 買い物 | ― ← タグなし Java学習 | [学習] ← 1チップ /* ─── CSS（F4）─── */ .tag-chip { display: inline-block; padding: 1px 8px; margin: 1px; border-radius: 10px; font-size: 11px; background: var(--paper-2); } /* → 小さな丸チップが横に並ぶ ★ */ // ─── データの前提（本編Vol.5）─── # memo.tags は List<Tag> # MyBatis の collection で取得済み # → 画面は「あるものを並べるだけ」 ★
POINT — 入れ子の each は「外＝親、内＝子のリスト」

入れ子の th:each は、外側が親（メモ）、内側が子のリスト（そのメモのタグ）。th:each="memo : ${memos}" の中で th:each="tag : ${memo.tags}" と書く。内側は外側で取り出した memo の中の tags を繰り返す。これで「各メモごとに、そのタグだけを並べる」ができる。本編Vol.5 の MyBatis collection で memo.tags が取れているので、画面は「あるものを並べるだけ」。サーバーとの連携が活きる場面です。

04

絞り込みセレクト

FILTER SELECT

本編Vol.4 のカテゴリ絞り込み。セレクトボックスでカテゴリを選ぶと、そのカテゴリのメモだけ表示。検索系はGETフォームを使い、選んだ瞬間に送信する工夫も加えます。

絞り込みフォームGET＋自動送信
// なぜ GET か

絞り込み・検索はGET。理由：URLに条件が残り（?categoryId=3）、ブックマーク・共有・戻るができる。POST は登録・更新用。

// 選んだ瞬間に送信

onchange="this.form.submit()" で、セレクトを変えた瞬間にフォーム送信。ボタン不要で快適。

// オプションの生成

カテゴリ一覧を th:each で <option> に展開。「すべて」の選択肢も用意。

// 絞り込みセレクト <!-- GETフォームで絞り込み --> <form method="get" th:action="@{/memos}"> <label>カテゴリで絞り込み：</label> <select name="categoryId" onchange="this.form.submit()"> <!-- すべて（絞り込み解除）--> <option value="">すべて</option> <!-- カテゴリ一覧を展開 --> <option th:each="cat : ${categories}" th:value="${cat.id}" th:text="${cat.name}"></option> </select> </form> // ─── 送信されるURL ─── 「仕事」を選ぶと… → GET /memos?categoryId=3 ★ → URLに条件が残る（共有・戻る可能） // ─── なぜ GET か ─── GET（絞り込み・検索）: URLに条件 → ブックマーク可・戻れる ★ POST（登録・更新）: URLに残らない → 二重送信を防ぐ // ─── onchange の効果 ─── onchange="this.form.submit()" # → 選んだ瞬間にフォーム送信 ★ # → 「絞り込みボタン」を押す手間が省ける // ─── Controller（本編Vol.4）─── @GetMapping("/memos") String list(@RequestParam(required=false) Long categoryId, Model model) # → categoryId で絞り込んで返す
POINT — 絞り込み・検索は GET、登録・更新は POST

フォームの method の使い分けは明確：絞り込み・検索は GET、登録・更新・削除は POST。GET は条件がURLに残るので、「このカテゴリの一覧」をブックマーク・共有・ブラウザの戻るで再現できる。検索結果のURLを送れるのもGETのおかげ。一方、データを変える操作はPOST（URLに残らず、リロードでの二重送信を防ぐ）。本編のPRGパターンとも一貫した、Webの基本ルールです。

05

th:selected で選択保持

PRESERVE SELECTION

絞り込んだ後、セレクトボックスが「すべて」に戻ってしまうと不便。今選んでいるカテゴリを保持するのが th:selected。「現在の値と一致する選択肢」を選択状態にします。

th:selected の保持選んだ状態を維持
// 問題

「仕事」で絞り込んだのに、セレクトが「すべて」表示だと今の絞り込みが分からない。

// 解決：th:selected

th:selected="${cat.id == selectedCategoryId}" ＝「現在選択中のIDと一致する選択肢」を selected に。

// 必要なもの

Controller が「現在選んでいるID」を selectedCategoryId として渡す（本編Vol.4）。

// th:selected で選択保持 // ─── 問題：絞り込み後にリセット ─── 「仕事」を選ぶ → /memos?categoryId=3 → 一覧は仕事だけ（正しい） → でもセレクトは「すべて」表示 ★ → 今の絞り込みが分からない // ─── 解決：th:selected ─── <option th:each="cat : ${categories}" th:value="${cat.id}" th:text="${cat.name}" th:selected="${cat.id == selectedCategoryId}"></option> # → 現在のIDと一致する選択肢が selected ★★★ # → 「仕事」が選ばれた状態を維持 // ─── 仕組み ─── selectedCategoryId = 3（仕事）のとき cat.id=1 → th:selected="false" cat.id=3 → th:selected="true" ★ 選択 cat.id=5 → th:selected="false" # → id=3 の option だけ selected に // ─── Controller が現在値を渡す（Vol.4）─── @GetMapping("/memos") String list(@RequestParam(required=false) Long categoryId, Model model) { model.addAttribute("memos", ...); model.addAttribute("categories", ...); model.addAttribute("selectedCategoryId", categoryId); # 現在値 ★ return "memos/list"; } # → 選んだIDを画面に渡すから保持できる // ─── th:field なら自動（F3復習）─── # フォームオブジェクトを使う場合は # th:field="*{categoryId}" で selected も自動 ★ # （絞り込みは単純なので th:selected を明示）
POINT — 「選んだ状態の保持」はユーザーの迷子を防ぐ

絞り込みやフォームで「選んだ状態が保持される」のは、地味ですがユーザー体験を大きく左右します。絞り込んだのにセレクトが初期値だと、ユーザーは「今、何で絞り込んでいるか分からず迷子」に。th:selected（または th:field）で現在の状態を画面に映すことで、「今こうなっている」が一目で分かる。本編Vol.4 の Controller が「現在値」を渡すのは、この保持のためです。

06

条件で表示を変える

CONDITIONAL DISPLAY

状況に応じて表示を変えるテクニックの整理。「結果が0件なら案内」「絞り込み中なら解除リンク」など。F2 の th:if を発展させ、ユーザーに親切な一覧を作ります。

状況に応じた表示親切な出し分け
// よくある条件表示
  • 結果0件 → 「該当なし」案内
  • 絞り込み中 → 「解除」リンク
  • 件数表示 → 「3件中」
  • ピン留め → 📌マーク
// 使う構文

th:if/th:unless（F2）、三項演算子、#lists.isEmpty など。

// 親切さの工夫

「0件」のときただ空白でなく「メモがありません。作成しますか?」と次の行動を促す。

// 条件で表示を変える // ─── ① 0件の案内 ─── <tr th:if="${#lists.isEmpty(memos)}"> <td colspan="5" class="empty"> メモがありません <a th:href="@{/memos/new}">作成する</a> </td> </tr> # → 空でも次の行動を促す（親切）★★★ // ─── ② 絞り込み中の解除リンク ─── <a th:if="${selectedCategoryId != null}" th:href="@{/memos}">絞り込みを解除</a> # → 絞り込み中だけ「解除」を表示 ★ // ─── ③ 件数表示 ─── <p th:text="'全 ' + ${#lists.size(memos)} + ' 件'"></p> # → 「全 3 件」 // ─── ④ ピン留めマーク（Vol.9）─── <span th:if="${memo.pinned}">📌</span> # → pinned=true のメモだけ📌 // ─── ⑤ 三項で簡潔に ─── <td th:text="${memo.archived} ? '(アーカイブ済)' : '通常'"></td> # → 状態をラベルで出し分け ★ // ─── 親切な一覧の条件 ─── ✓ 0件でも次の行動を示す ✓ 絞り込み状態が分かる ✓ 件数が見える ✓ 状態（ピン・アーカイブ）が一目 ★★★
POINT — 「0件」のときこそ、画面の親切さが問われる

一覧が0件のとき、ただ空白だとユーザーは「壊れた? 読み込み中?」と不安に。良い画面は「メモがありません。作成しますか?」と状況の説明＋次の行動を示す。これは本編の「ユーザーに分かりやすく」の画面版。「データがある時」だけでなく「無い時・絞り込んで0件の時」まで設計するのが、親切なフロントエンド。th:if での空チェックは、こうした配慮の入り口です。

07

一覧表示のあるある失敗

LIST PITFALLS

一覧・絞り込みで初心者がハマる失敗を5つ。null 未対応、選択保持忘れ、GET/POST 取り違えなど。知っていれば回避できます。

一覧表示あるある 5選先に知れば回避できる

① カテゴリ null で落ちる ★★★

  • 未分類メモで .name を直接参照
  • 対処：th:if で null チェック（Section 02）

② 絞り込み後に選択が戻る ★★★

  • セレクトが「すべて」に戻る
  • 対処：th:selected で保持（Section 05）

③ 絞り込みを POST にする

  • URLに残らず共有・戻るができない
  • 対処：絞り込みは GET（Section 04）

④ 入れ子 each の変数名が衝突

  • 外も内も同じ変数名にする
  • 対処：memo/tag と別名に（Section 03）

⑤ 0件のとき空白で不親切

  • 何も出ず壊れて見える
  • 対処：空の案内を出す（Section 06）
// あるある失敗の対処 // ─── ① null チェック ─── NG: th:text="${memo.category.name}" # category=null で落ちる ★ OK: th:if="${memo.category != null}" th:text="${memo.category.name}" // ─── ② 選択保持 ─── NG: th:selected 無し → 毎回「すべて」 OK: th:selected="${cat.id == selectedCategoryId}" // ─── ③ GET/POST ─── NG: <form method="post"> で絞り込み OK: <form method="get"> で絞り込み ★ // ─── ④ 変数名の衝突 ─── NG: <tr th:each="x : ${memos}"> <span th:each="x : ${x.tags}"> # 同名 x ★ OK: <tr th:each="memo : ${memos}"> <span th:each="tag : ${memo.tags}"> # 別名 ★ // ─── 良い一覧の条件 ─── ✓ null は th:if で守る ✓ 選択は th:selected で保持 ✓ 絞り込みは GET ✓ 入れ子は変数名を分ける ✓ 0件は親切に案内 ★★★
POINT — 一覧トラブルの多くは「null」と「状態保持」

一覧画面のトラブルで多いのは2つ：①カテゴリ等の null を未対応で落ちる、②絞り込みやソートの状態が保持されない。①は th:if での null チェック、②は th:selected/th:field での状態保持で解決。どちらも「データが無い・選択がある」という"通常じゃない状態"への配慮。正常系（全部データがある）だけでなく、「null・空・選択中」も想定するのが、堅牢な一覧画面のコツです。
