02

静的HTML vs サーバー生成

STATIC VS SERVER-RENDERED

普通のWebサイトの静的HTMLと、このアプリのサーバー生成HTMLは別物です。静的HTMLは「書いた内容がそのまま表示」、サーバー生成は「サーバーがデータを埋めて完成させる」。この違いを理解することが出発点です。

2種類のHTMLの違いいつ中身が決まるか
// 静的HTML

ファイルに書いた内容がそのまま表示される。中身は固定。ダブルクリックで開ける。

// サーバー生成HTML（Thymeleaf）

th:text 等の「印」を書いておき、サーバーがデータを埋めて完成HTMLを作る。中身は表示のたびに変わる。

// 比較
静的サーバー生成
中身固定動的
開き方直接サーバー経由
データ埋め込み済毎回挿入
// 静的 vs サーバー生成 // ─── 静的HTML（中身は固定）─── <h1>会議メモ</h1> // → いつ見ても「会議メモ」 // → メモが変わっても固定のまま ★ // ─── サーバー生成（Thymeleaf）─── <h1 th:text="${memo.title}">タイトル</h1> // → サーバーが memo.title を埋める // → メモごとに中身が変わる ★★★ // → 「タイトル」は仮表示（後述） // ─── サーバー処理の前後 ─── 処理前（テンプレート）: <h1 th:text="${memo.title}">タイトル</h1> ↓ サーバーが memo.title=会議メモ を埋める 処理後（ブラウザに届くHTML）: <h1>会議メモ</h1> // → ブラウザは普通のHTMLとして受け取る ★ // ─── だから直接開くと動かない ─── ファイルをダブルクリック → th:text が処理されない → 「タイトル」（仮表示）のまま ★ → 必ずサーバー経由で見る
POINT — タグの中の文字は「仮表示（プレビュー用）」

<h1 th:text="${memo.title}">タイトル</h1> の「タイトル」という文字に注目。これはサーバー処理後には消えて、実際の値（会議メモ等）に置き換わります。なぜ書くかというと、デザイナーが静的に開いたときのプレビューのため。Thymeleaf の賢いところで、「サーバーなしでも、おおよその見た目が分かる」。本番では必ず th:text の値で上書きされます。

03

Thymeleafの立ち位置

WHERE THYMELEAF SITS

Thymeleafは、Spring Boot で標準的に使われるテンプレートエンジンです。「テンプレート（ひな型）」に「データ」を流し込んで、完成HTMLを生成します。HTMLの中に特別な属性（th:〜）を書くのが特徴です。

テンプレートエンジンの仕事ひな型＋データ＝完成HTML
// テンプレートエンジンとは

ひな型（テンプレート）にデータを流し込み、完成したHTMLを作る道具。料理に例えれば「型」に「具材」を入れて成形するイメージ。

// Thymeleafの特徴
  • HTMLの属性として書く（th:〜）
  • 普通のHTMLとしてそのまま開ける（仮表示）
  • Spring と統合されている
// なぜ採用されるか

HTMLとして自然で読みやすく、デザイナーとも協業しやすい。Spring Boot の定番。

// テンプレートエンジンの流れ // ─── 3つの材料 ─── ① テンプレート（list.html） <tr th:each="m : ${memos}"> <td th:text="${m.title}"></td> </tr> ② データ（Controllerが渡す） memos = [会議メモ, 買い物, Java学習] ③ Thymeleafが合成 ↓ 完成HTML: <tr><td>会議メモ</td></tr> <tr><td>買い物</td></tr> <tr><td>Java学習</td></tr> // → データの数だけ行が増える ★★★ // ─── th:属性の正体 ─── th:text → 中身のテキストを差し替え th:each → 繰り返し th:if → 条件で表示/非表示 th:href → リンク先を組み立て // → 全部「HTMLの属性」として書く ★ // → 普通のブラウザは th: を無視するので // 静的に開いても崩れない // ─── 他の選択肢との違い ─── JSP : 古い・非推奨 Thymeleaf : Spring Boot 標準・推奨 ★ React/Vue : フロント分離型（本シリーズ対象外）
POINT — Thymeleafは「HTMLを壊さない」のが強み

Thymeleaf の最大の特徴は「属性として書くので、HTMLとして自然」なこと。th:text は普通のブラウザなら無視されるので、テンプレートをそのまま開いてもデザインが崩れません。これによりデザイナーとプログラマーが同じファイルを扱える。古い JSP のようにHTMLの中にコードが混ざって読めなくなることもない。「HTMLを尊重する」のが、Thymeleaf が好まれる理由です。

04

画面ファイルの置き場所

WHERE FILES GO

Thymeleafテンプレートには決まった置き場所があります。src/main/resources/templates/ 配下です。Controller が返すビュー名と、ファイルのパスが対応します。この約束を理解しましょう。

テンプレートの配置ルールビュー名＝パス
// 置き場所の基本

テンプレートは src/main/resources/templates/ 配下。Controller が返す文字列（ビュー名）が、この配下のファイルパスに対応する。

// 対応の例
Controllerの戻り値ファイル
"memos/list"templates/memos/list.html
"memos/form"templates/memos/form.html
"error/404"templates/error/404.html
// 機能ごとにフォルダ分け

memos/・categories/・error/ のように機能でフォルダ分けすると整理しやすい。

// テンプレートの配置 // ─── ディレクトリ構造 ─── src/main/resources/templates/ ├ memos/ │ ├ list.html # メモ一覧 │ ├ form.html # 作成/編集 │ └ show.html # 詳細 ├ categories/ │ ├ list.html │ └ form.html └ error/ ├ 404.html └ 500.html // ─── Controller との対応 ─── @GetMapping("/memos") public String list(Model model) { model.addAttribute("memos", ...); return "memos/list"; // → templates/memos/list.html を表示 ★ } // 戻り値の文字列 = templates配下のパス // （.html は省略） // ─── Model でデータを渡す ─── model.addAttribute("memos", memoList); ↓ テンプレートで ${memos} として使える ★★★ // → サーバー → 画面 のデータの受け渡し
POINT — 「ビュー名」と「ファイルパス」は鏡うつし

Controller の return "memos/list" と、ファイル templates/memos/list.html は必ず一致します。これがズレると「テンプレートが見つからない」エラーに（フロント別冊Vol.9で扱います）。覚え方は「templates/ を起点に、戻り値の文字列＋.html」。Model に addAttribute した名前が、テンプレートで ${名前} として使える——このサーバーと画面の橋渡しが、画面表示の基本構造です。

05

画面が表示されるまでの流れ

REQUEST TO RENDER

URLを開いてから画面が出るまで、裏で何が起きているかを追います。ブラウザ→Controller→Service→Mapper→DB→テンプレート→ブラウザという一連の流れ。これが分かると、画面の不具合の原因も特定しやすくなります。

リクエストから描画まで裏側の旅
// 一覧画面が出るまで
  1. ブラウザが /memos をリクエスト
  2. Controller が受け取る
  3. Service→Mapper→DB でメモ取得
  4. Model にデータを載せる
  5. テンプレート（list.html）にデータを流す
  6. 完成HTMLをブラウザに返す
// フロントが関わるのは

①リクエストと⑤テンプレートと⑥表示。間（②〜④）は本編の領域。

// 画面表示の全行程 // ─── 一覧 /memos を開く ─── 🌐 ブラウザ 「GET /memos ください」 │ ← フロント（リンク・URL）❖ ▼ 🎯 MemoController list() メソッドが受付 │ ▼ ⚙ MemoService → MemoMapper → 🗄 DB メモ一覧を取得 │ ← 本編 main の領域 ▼ 📦 Model model.addAttribute("memos", list) │ ▼ 📄 list.html（テンプレート） th:each で memos を行に展開 │ ← フロント（Thymeleaf）❖ ▼ 🌐 ブラウザ 完成HTMLを表示 ← フロント（CSS・JS）❖ // ─── 不具合の切り分けに役立つ ─── 「データが表示されない」とき： - DBにデータがある？（本編） - Modelに載せた？（本編） - th:each の書き方は？（フロント）★ - ${名前} が一致してる？（フロント）★ // → 流れが分かると原因を絞れる ★★★
POINT — 流れが分かると「どこが悪いか」を切り分けられる

「データが画面に出ない」とき、原因は複数の場所にありえます。DBが空（本編）、Modelに載せ忘れ（本編）、th:each の書き方ミス（フロント）、${名前} の不一致（フロント）。流れを知っていれば、上流から順に確認できる。「DBにはある→Modelにも載った→なら画面側の問題」と切り分けられる。これがデバッグの基本。フロント別冊Vol.9 で詳しく扱います。

06

CSSとJSの置き場所

CSS & JS LOCATION

CSS と JavaScript は、テンプレートとは別の場所src/main/resources/static/ に置きます。これらは静的ファイルとして、そのままブラウザに配信されます。テンプレートからの読み込み方も確認します。

静的ファイルの配置と読み込みstatic配下に置く
// 置き場所
  • CSS → static/css/
  • JS → static/js/
  • 画像 → static/images/
// テンプレートと違う点

テンプレートはサーバーが加工するが、static はそのまま配信。CSS/JSは加工不要なので static に置く。

// 読み込みは @{} で

テンプレートから th:href="@{/css/style.css}" で読み込む。@{} が正しいパスに解決してくれる。

// CSS/JS の配置と読み込み // ─── 配置 ─── src/main/resources/static/ ├ css/ │ └ style.css └ js/ └ bulk-select.js # Vol.10で使用 // ─── テンプレートから読み込む ─── <link rel="stylesheet" th:href="@{/css/style.css}"> <script th:src="@{/js/bulk-select.js}"></script> // → @{/css/...} は static/ を起点に解決 ★ // ─── @{} を使う理由 ─── NG: href="/css/style.css" // → アプリの公開パスが変わると壊れる ★ OK: th:href="@{/css/style.css}" // → コンテキストパスを自動で考慮 ★★★ // ─── ブラウザからのアクセス ─── http://localhost:8080/css/style.css // → static/css/style.css が配信される // → static/ の部分はURLに出ない ★ // ─── 本シリーズのJS事情 ─── // 前半（Vol.1〜6）はJSなし // JS登場は Vol.10 の一括選択のみ // → サーバーサイド中心だから ★
POINT — リンクやパスは必ず @{} で書く

CSS・JS・画像・リンクのパスは、直接書かず必ず @{} で囲みます。th:href="@{/css/style.css}" のように。理由は、アプリの公開パス（コンテキストパス）が変わっても、自動で正しいパスに解決してくれるから。href="/css/..." と直書きすると、デプロイ環境によってパスがずれて読み込めなくなる。「パスは @{} で組み立てる」のが Thymeleaf の鉄則です。フロント別冊Vol.2 で詳しく扱います。
