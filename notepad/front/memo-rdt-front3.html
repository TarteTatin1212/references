01

フォームの仕組み

HOW FORMS WORK

フォームはユーザーの入力をサーバーに送る仕組み。<form> で囲み、送信ボタンを押すと、入力内容がController に届く。Thymeleaf は、この送信と受け取りを楽にする構文を用意しています。

フォーム送信の流れ入力→送信→受取
// フォームの基本
  • <form method="post" action="..."> で囲む
  • 入力欄（input/textarea/select）を置く
  • 送信ボタンで Controller に届く
// Thymeleafが楽にすること
  • フォームとJavaオブジェクトを結ぶ
  • 各項目を自動でひも付け
  • エラーをその場に表示
  • 入力値を自動で保持
// 受け取る側（本編）

Controller が @ModelAttribute MemoForm で受け取る（本編Vol.2）。

// フォーム送信の全体像 // ─── 画面（フォーム）─── <form th:object="${memoForm}" method="post" th:action="@{/memos}"> <input th:field="*{title}"> <button>保存</button> </form> │ 送信 ▼ // ─── サーバー（本編Vol.2）─── @PostMapping("/memos") String create(@ModelAttribute MemoForm form) # → title 等が form に詰まって届く ★★★ // ─── Thymeleafが面倒を見る4つ ─── ① th:object → フォーム全体を1オブジェクトに ② th:field → 各項目を自動ひも付け ③ th:errors → エラーをその場に表示 ④ 入力保持 → エラー時に値が消えない // ─── 表示(F2)との違い ─── 表示: サーバー → 画面（${...} で出す） 入力: 画面 → サーバー（*{...} で送る）★ # → 向きが逆。記号も ${} → *{} に
POINT — フォームは「双方向」、だから専用構文がある

表示（F2）はサーバー→画面の一方向でしたが、フォームは双方向：①画面に初期値を表示し、②ユーザーが編集し、③サーバーに送り、④エラーなら値を保ったまま戻す。この複雑なやり取りを、Thymeleaf は th:object/th:field/th:errors の3点セットで楽にしてくれます。手で value="..." や name="..." を書く必要がなくなるのが利点です。

02

th:object でフォームを束ねる

TH:OBJECT

th:object は「このフォームは、どのJavaオブジェクトと対応するか」を宣言します。th:object="${memoForm}" で「このフォームは memoForm」と結ぶ。以降、中の項目は *{...} で短く書けます。

th:object の役割フォームと箱を結ぶ
// th:object とは

<form th:object="${memoForm}"> ＝「このフォームの入力は memoForm という箱に入る」と宣言。

// ${} と *{} の関係

object を宣言すると、中では *{title} ＝ ${memoForm.title} の短縮形。毎回 memoForm を書かなくて済む。

// Controller が箱を用意

表示前に Controller が model.addAttribute("memoForm", new MemoForm()) で空の箱を渡す（本編Vol.2）。

// th:object の使い方 // ─── フォームにオブジェクトを結ぶ ─── <form th:object="${memoForm}" method="post"> <!-- 以降 *{} は memoForm の中身 --> <input th:field="*{title}"> <input th:field="*{content}"> </form> // ─── ${} と *{} の関係 ─── th:object="${memoForm}" を宣言すると… *{title} = ${memoForm.title} *{content} = ${memoForm.content} # → *{} は object 内への近道 ★★★ # → memoForm を毎回書かずに済む // ─── Controller が箱を渡す（本編Vol.2）─── @GetMapping("/memos/new") String newForm(Model model) { model.addAttribute("memoForm", new MemoForm()); # 空の箱 ★ return "memos/form"; } # → 表示時に空の memoForm を用意 # → これがないと th:object でエラー ★ // ─── 編集時は値入りの箱 ─── # 編集画面では既存メモの値を詰めた # memoForm を渡す → 既存値が初期表示される
POINT — th:object には「Controllerが用意した箱」が必要

th:object="${memoForm}" を使うには、Controller が表示前に model.addAttribute("memoForm", ...) で箱（オブジェクト）を渡しておく必要があります。新規作成なら空の箱、編集なら既存値を詰めた箱。これを忘れると「memoForm が無い」エラーに（フロント別冊F9の頻出トラブル）。「フォームを表示する前に、対応する箱を Model に載せる」——これが本編Vol.2 のController と画面の約束です。

03

th:field で項目を結ぶ

TH:FIELD

th:field は入力欄1つ1つを、オブジェクトの項目に結ぶ魔法の属性。th:field="*{title}" と書くだけで、name・id・value が自動生成される。手書きの手間とミスが激減します。

th:field の自動生成3つを一度に
// th:field が生成するもの

th:field="*{title}" 1つで、name・id・value の3つを自動生成。

属性生成される値
nametitle（送信時のキー）
idtitle（label連携用）
value現在の値（保持用）
// 各入力欄での使い方

input・textarea・select・checkbox すべてで th:field が使える。型に応じて適切に処理される。

// th:field の使い方 // ─── テキスト入力 ─── <input type="text" th:field="*{title}"> ↓ サーバー処理後 <input type="text" id="title" name="title" value="会議メモ"> # → name/id/value が自動生成 ★★★ // ─── テキストエリア ─── <textarea th:field="*{content}"></textarea> # → 中身に現在値が入る // ─── セレクト（本編Vol.4）─── <select th:field="*{categoryId}"> <option value="">未分類</option> <option th:each="c : ${categories}" th:value="${c.id}" th:text="${c.name}"></option> </select> # → 現在の categoryId が自動で selected に ★ // ─── チェックボックス（F6で詳しく）─── <input type="checkbox" th:field="*{tagIds}" th:value="${tag.id}"> # → リストに含まれていれば自動でcheck ★ // ─── label との連携 ─── <label for="title">タイトル</label> <input th:field="*{title}"> # → id=title が自動生成されるので for と連携 ★
POINT — th:field を使えば name/value を手書きしない

th:field の最大の利点は「name・id・value を手で書かなくていい」こと。手書きだと name="title" のタイプミスで送信されない、value の書き忘れで入力値が保持されないといったミスが起きがち。th:field="*{title}" なら全部自動・一貫。しかもselect の selected、checkbox の checked も自動判定。「入力欄は th:field で結ぶ」と覚えれば、フォームのミスが激減します。

04

th:errors でエラー表示

TH:ERRORS

入力が不正なとき、th:errors でエラーメッセージをその場に表示します。本編Vol.12 のバリデーション（@NotBlank 等）と連携し、「タイトルを入力してください」を該当欄の近くに出せます。

th:errors の表示該当欄にエラー
// 基本

th:errors="*{title}" ＝「title の検証エラーがあれば、そのメッセージを表示」。エラーが無ければ何も出ない。

// 本編との連携（Vol.12）

Form クラスの @NotBlank(message="...") のメッセージが、ここに表示される。サーバー側の検証と画面が連動。

// エラー時のスタイル

th:errorclass で、エラー時だけ赤枠などのクラスを付けられる。

// th:errors の使い方 // ─── 基本：該当欄の下に出す ─── <input th:field="*{title}"> <span class="error" th:errors="*{title}"></span> # → titleにエラーがあればメッセージ表示 # → 無ければ span ごと消える ★ // ─── 本編Vol.12 との連携 ─── # Form クラス側: @NotBlank(message = "タイトルを入力してください") @Size(max = 100, message = "100文字以内で") private String title; ↓ 検証エラー時 # 画面側 th:errors に message が表示 ★★★ # 「タイトルを入力してください」 // ─── エラー時だけスタイルを変える ─── <input th:field="*{title}" th:errorclass="is-invalid"> # → エラー時だけ is-invalid クラスが付く # → CSSで赤枠などにできる ★ // ─── 全エラーをまとめて表示 ─── <ul th:if="${#fields.hasErrors('*')}"> <li th:each="e : ${#fields.errors('*')}" th:text="${e}"></li> </ul> # → フォーム上部に全エラーを一覧表示 ★ // ─── 検証の流れ（本編Vol.12）─── 入力 → @Validated で検証 → エラーあり → 同じフォーム画面に戻す → th:errors がメッセージ表示 ★★★
POINT — エラーメッセージは Form クラスの message で決まる

画面に出る「タイトルを入力してください」という文言は、画面側ではなく Form クラスの @NotBlank(message="...") で定義されています（本編Vol.12）。画面の th:errors は「そのメッセージを表示する場所」を指定するだけ。だから文言を変えたいときは Form クラスを直す。「検証ルールとメッセージはサーバー側、表示位置は画面側」という役割分担を理解すると、エラー表示が自在になります。

05

入力値の保持

PRESERVING INPUT

検証エラーで戻ったとき、ユーザーが入力した値が消えないのは大事なUX。Thymeleaf では th:field を使い、Controller が同じフォームオブジェクトを返すだけで、入力値が自動で保持されます。

入力保持の仕組み消えないのは自動
// なぜ保持が大事か

長文を書いて1か所のミスで全部消えるのは最悪のUX。エラー時も入力を保ったまま修正できるべき。

// 仕組み（自動）
  1. 送信された値は form に入る
  2. 検証エラーで同じ form を返す
  3. th:field が form の値を value に
  4. → 入力値がそのまま表示
// 手動だと大変

th:field なしで value を手書きすると、保持を自分で書く羽目に。th:field なら自動。

// 入力値の保持 // ─── Controller（本編Vol.2,12）─── @PostMapping("/memos") String create(@Validated @ModelAttribute MemoForm form, BindingResult br) { if (br.hasErrors()) { return "memos/form"; # 同じformを返す ★ # → form には送信値が入ったまま } service.create(form); return "redirect:/memos"; } # → エラー時、form をそのまま画面へ ★★★ // ─── 画面（自動で保持）─── <input th:field="*{title}"> # → form.title の値が value に入る # → ユーザーが打った「会議メモ」が残る ★ # → 手で value を書く必要なし // ─── 手動だと（th:field なし）─── 面倒: <input name="title" th:value="${memoForm.title}"> # → value を自分で書く必要あり ★ 楽: <input th:field="*{title}"> # → name も value も自動 ★★★ // ─── 保持されない原因（F9で詳しく）─── # ・エラー時に new Form() を作り直している # ・redirect している（値が消える） # → エラー時は return "view"（リダイレクトしない）★
POINT — エラー時は「リダイレクトせず、同じformを返す」

入力値が保持される鍵は、Controller が検証エラー時にリダイレクトせず、送信された form をそのまま画面に返すこと（return "memos/form"）。もし redirect: するとform の値が失われ、入力が消えます。「成功したらリダイレクト（PRG）、エラーなら同じ画面を返す」——本編Vol.2・12 のこの使い分けが、入力保持を支えています。th:field と組めば、保持は完全に自動です。

06

作成フォームを組み立てる

BUILD THE FORM

学んだ th:object・th:field・th:errors を全部使って、本編Vol.2 のメモ作成フォームを組み立てます。新規・編集兼用にする工夫も。これが実際の form.html です。

memos/form.html の全体3点セットの合わせ技
// 使う構文
  • th:object：memoForm と結ぶ
  • th:field：各入力欄をひも付け
  • th:errors：エラーをその場に
// 新規・編集の兼用

送信先（action）をid の有無で切り替え。id があれば編集（/memos/{id}）、なければ新規（/memos）。

// 1ファイルで両対応

新規も編集も同じ form.html。重複を避ける（本編の関心の分離・F8で深掘り）。

// templates/memos/form.html <!-- 新規/編集で送信先を切り替え --> <form th:object="${memoForm}" method="post" th:action="${id} != null ? @{/memos/{id}(id=${id})} : @{/memos}"> <!-- タイトル --> <div class="field"> <label for="title">タイトル</label> <input type="text" th:field="*{title}" th:errorclass="is-invalid"> <span class="error" th:errors="*{title}"></span> </div> <!-- 本文 --> <div class="field"> <label for="content">本文</label> <textarea th:field="*{content}" rows="6"></textarea> <span class="error" th:errors="*{content}"></span> </div> <button type="submit">保存</button> <a th:href="@{/memos}">キャンセル</a> </form> # → object/field/errors の3点セット完成 ★★★ # → 新規も編集も1ファイルで対応
POINT — 新規と編集は「1つのフォーム」で兼用できる

新規作成フォームと編集フォームはほぼ同じなので、1つの form.html で兼用するのが定石。違いは送信先（action）だけ——新規は /memos、編集は /memos/{id}。これを th:action="${id} != null ? ... : ..." の三項演算子で切り替えます。同じ画面を2つ作らないのは、本編で学んだ「重複を避ける（DRY）」の画面版。コードが減ればバグも減ります。

07

フォームのあるある失敗

FORM PITFALLS

フォームで初心者がハマる失敗を5つ。th:object の箱の渡し忘れ、*{} と ${} の混同など。知っていれば数分で抜けられます。

フォームあるある 5選先に知れば回避できる

① Controllerが箱を渡し忘れ ★★★

  • th:object で「memoForm が無い」エラー
  • 対処：表示前に addAttribute（Section 02）

② *{} と ${} を混同

  • object内は *{}、外は ${}
  • 対処：field/errors は *{}

③ エラー時に redirect して値が消える ★★★

  • 入力が全部消える
  • 対処：エラー時は return "view"（Section 05）

④ BindingResult の位置ミス

  • @Validated の直後に置かないと400
  • 対処：検証対象の直後（本編Vol.12）

⑤ ファイル送信で enctype 忘れ

  • ファイルが送られない（本編Vol.11）
  • 対処：enctype="multipart/form-data"
// あるある失敗の対処 // ─── ① 箱の渡し忘れ ─── NG: 新規画面で memoForm を渡さない # → th:object でエラー ★ OK: model.addAttribute("memoForm", new MemoForm()); return "memos/form"; // ─── ② *{} と ${} ─── object内（フォーム項目）: *{title} object外（一覧表示等）: ${memo.title} # → field/errors は必ず *{} ★★★ // ─── ③ エラー時のreturn ─── NG: if(br.hasErrors()) return "redirect:/memos/new"; # → 入力値が消える ★★★ OK: if(br.hasErrors()) return "memos/form"; # → form を返すので値が残る ★ // ─── ⑤ ファイル送信（本編Vol.11）─── NG: <form method="post"> OK: <form method="post" enctype="multipart/form-data"> // ─── 良いフォームの条件 ─── ✓ 表示前に箱を addAttribute ✓ field/errors は *{} ✓ エラー時はリダイレクトしない ✓ BindingResult は直後に ✓ ファイルは enctype 指定 ★★★
POINT — フォームトラブルの多くは「箱」と「returnの仕方」

フォーム関連のトラブルで多いのは2つ：①Controller が表示前にフォームの箱（オブジェクト）を渡し忘れ（th:object でエラー）、②エラー時に redirect して入力値が消える。どちらもサーバー側（Controller）の書き方が原因で、画面側は正しいことが多い。「画面が変なときは、Controller も疑う」。本編とフロント別冊を両方理解していると、こうした切り分けが速くなります。
