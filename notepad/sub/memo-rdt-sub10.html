01

別冊S10のミッション

MISSION

別冊S10は「外部とデータをやり取りする設計を理解する」回。本編Vol.11 でメモのエクスポート（書き出し）・インポート（取り込み）を実装します。これまではアプリ内で完結していましたが、ファイルという「外の世界」と繋がる。新しい注意点が出てきます。

「外の世界」と繋がる新しい責任
// なぜエクスポート/インポートが必要か

使い込むと「メモをバックアップしたい」「別のアプリに移したい」「大量のメモを一気に取り込みたい」というニーズが。これがデータ入出力の価値。

// 「外」と繋がる難しさ
  • 形式：CSV? JSON? どう書き出す
  • 文字コード：日本語が化けないように
  • 信頼性：取り込むデータは正しいとは限らない
// 別冊S10のゴール
  • CSV/JSONの使い分けが分かる
  • 文字コードの罠を避けられる
  • パース/シリアライズを理解
  • 取り込みデータを検証できる
// データ入出力の全体像 // ─── これまで（アプリ内で完結）─── ブラウザ ⇔ アプリ ⇔ DB // → 全部自分の管理下 // ─── Vol.11（外部と繋がる）─── エクスポート（書き出し）: DB → ファイル（CSV/JSON）→ 外部 「メモをバックアップ・移行」 インポート（取り込み）: 外部 → ファイル → アプリ → DB 「メモを一括登録・復元」 // ─── 外と繋がる3つの注意点 ─── ① 形式（CSV/JSON） → どう表現するか（Section 03） ② 文字コード（UTF-8） → 日本語が化けないか（Section 04） ③ 信頼性 → 取り込むデータは信用できない ★★★ → 検証が必要（Section 06） // ─── 「外部データは信じない」が鍵 ─── // 自分が作ったデータと違い、 // インポートデータは何が来るか不明 // → 別冊S6のSQLi、S7の例外と同じ精神
POINT — 「外部データは信用しない」が入出力の鉄則

データ入出力で最も大事な心構えは「外から来るデータは信用しない」。エクスポート（書き出し）は自分のデータなので安心ですが、インポート（取り込み）は何が来るか分からない——壊れたCSV、不正な値、文字化け、悪意あるデータ。別冊S6のSQLインジェクション、S7の例外設計と同じく、「外部入力は疑ってかかる」。インポートには必ず検証（Section 06）を入れます。

02

エクスポートとインポート

EXPORT & IMPORT

エクスポート（書き出し）とインポート（取り込み）は、向きが逆。エクスポートはDB→ファイル、インポートはファイル→DB。難しさも違います。インポートの方が圧倒的に難しい——理由を理解します。

2つの方向と難易度の違いインポートが難しい
// エクスポート（書き出し）

DBのメモをファイル形式に変換して出力。自分のデータなので形式が分かっている。比較的簡単。

// インポート（取り込み）

ファイルを読み込んでDBに登録。外から来るので何が入っているか不明。検証・エラー処理が必要で難しい。

// 難易度の比較
エクスポートインポート
方向DB→ファイルファイル→DB
データ源自分（信頼）外部（不信）
難易度易難
検証不要必須
// エクスポートとインポート // ─── エクスポート（簡単）─── DB のメモ ↓ 取得（既存のfindActive等） List<Memo> ↓ ファイル形式に変換（シリアライズ） CSV/JSON 文字列 ↓ ダウンロード ユーザーのファイル // → 自分のデータ → 形式が確実 → 簡単 ★ // ─── インポート（難しい）─── ユーザーのファイル ↓ アップロード CSV/JSON 文字列（中身は不明！） ↓ パース（解析） ↓ 【検証】← ここが重要 ★★★ ↓ 正しいものだけ List<Memo> ↓ DB に登録 DB // → 外部データ → 何が来るか不明 → 検証必須 // ─── インポートが難しい理由 ─── 壊れたCSV（列が足りない） 不正な値（タイトルが空、日付が変） 文字化け（文字コード違い） 巨大ファイル（メモリ不足） 重複データ（既存と被る） // → これら全てに対処が必要 ★★★ // → 「取り込めない場合」の設計が大事
POINT — エクスポートから作り、インポートは慎重に

実装の順序として「エクスポートを先、インポートを後」が定石。エクスポートは自分のデータを出すだけで簡単・安全。インポートは検証・エラー処理が必要で難しい。まずエクスポートで「どんな形式で出すか」を確定させ、その形式をインポートで受け入れる。本編Vol.11 もこの順序。易しい方から作り、難しい方を慎重に——別冊S2の段階的アプローチです。

03

CSVとJSONの使い分け

CSV VS JSON

データ形式の代表CSVとJSON。どちらもテキスト形式ですが、得意分野が違います。CSVは表（フラット）、JSONは構造（ネスト）。メモ帳のデータにどちらが向くかを考えます。

CSV と JSON の特性表 vs 構造
// CSV（カンマ区切り）
  • 表形式（行と列）
  • Excelで開ける
  • シンプルだがネストが苦手
  • メモのタグ（複数）が表現しにくい
// JSON（構造化）
  • 階層・配列を表現できる
  • メモ＋タグ（複数）が自然
  • プログラムで扱いやすい
  • Excelでは開けない
// 使い分け
  • CSV：Excelで見たい・単純な表
  • JSON：タグ等の構造・完全なバックアップ
// CSV と JSON の比較 // ─── CSV（表形式）─── id,title,content,category 1,会議メモ,明日10時,仕事 2,買い物,牛乳と卵,私用 // → シンプル、Excelで開ける // → でもタグ（複数）が困る ★ // tags 列に "重要;急ぎ" と詰める？ // → 区切り文字地獄（別冊S4の教訓） // ─── JSON（構造化）─── [ { "id": 1, "title": "会議メモ", "content": "明日10時", "category": "仕事", "tags": ["重要", "急ぎ"] // 配列で自然 ★ }, { "id": 2, "title": "買い物", "tags": [] } ] // → タグ（複数）を配列で表現 ★★★ // → メモの構造をそのまま保存できる // ─── メモ帳での使い分け ─── CSV エクスポート: 「Excelで一覧を見たい」 → タグは諦めるか ; 区切りで JSON エクスポート: 「完全バックアップ・タグも含めて」 → 構造をそのまま保存 ★ // → 用途で選ぶ（別冊S2のトレードオフ） // → 本編Vol.11 は両方 or どちらかを実装
POINT — タグ（多対多）があるなら JSON が自然

メモ帳にはタグ（多対多・別冊S4）があります。これはCSVだと表現が苦しい——1メモに複数タグを「重要;急ぎ」と詰め込むと、別冊S4で学んだ「カンマ区切り地獄」の再来。一方JSON なら配列 ["重要","急ぎ"] で自然。「構造のあるデータは JSON、単純な表は CSV」。メモの完全バックアップには JSONが向きます。データの構造に形式を合わせるのがポイントです。

04

文字コードの罠

CHARACTER ENCODING

日本語を扱うアプリで避けて通れない文字コード。UTF-8 が標準ですが、Excel が Shift_JIS を期待するなどのズレで文字化けが起きます。仕組みを知れば防げます。

文字コードと文字化けUTF-8 が基本
// 文字コードとは

文字をバイト列に変換する規則。同じ「あ」でも、UTF-8 と Shift_JIS では違うバイトになる。書く時と読む時で規則が違うと文字化け。

// 標準は UTF-8

現代の標準はUTF-8。Web、JSON、多くのシステムが UTF-8。基本は UTF-8 で統一。

// Excel の罠（CSV）

Excel は CSV をShift_JIS（環境による）で開こうとし、UTF-8 の日本語が文字化けすることがある。BOM付きUTF-8で回避できる場合も。

// 文字コードの罠 // ─── 文字化けの仕組み ─── 「あ」を UTF-8 で書く → E3 81 82 それを Shift_JIS で読む → 文字化け ★ // → 書く規則と読む規則が違うと化ける // ─── 基本方針：UTF-8 で統一 ─── // 書き出し時に明示的に UTF-8 response.setCharacterEncoding("UTF-8"); response.setContentType( "text/csv; charset=UTF-8"); // → 文字コードを明示する ★★★ // ─── Excel の CSV 文字化け ─── // UTF-8 のCSVをExcelで開くと // 日本語が化けることがある（環境依存） // 対策①：BOM付きUTF-8 // 先頭に EF BB BF を付ける // → Excelが UTF-8 と認識しやすい byte[] bom = {(byte)0xEF, (byte)0xBB, (byte)0xBF}; // 対策②：JSON を使う（Excelで開かない） // → 文字化け問題を回避 // ─── ファイル読み込み時も明示 ─── // インポート時、UTF-8 で読む new InputStreamReader( inputStream, StandardCharsets.UTF_8); // → 文字コードを指定して読む ★★★ // → 指定しないと環境依存で化ける // ─── 鉄則 ─── // 書く時も読む時も「UTF-8」を明示 // → 暗黙のデフォルトに頼らない ★★★
POINT — 文字コードは「明示する」、デフォルトに頼らない

文字化けの最大の原因は「文字コードを指定せず、暗黙のデフォルトに任せる」こと。デフォルトはOS・環境で変わるので、開発機では動いても本番で化ける。対策はシンプル：書く時も読む時も、明示的に UTF-8 を指定する。StandardCharsets.UTF_8 を必ず渡す。「環境のデフォルトを信用しない」——文字コードを明示するだけで、文字化けの大半は防げます。

05

シリアライズとパース

SERIALIZE & PARSE

データ変換の2つの向き。シリアライズ（オブジェクト→文字列）とパース（文字列→オブジェクト）。エクスポートはシリアライズ、インポートはパース。ライブラリに任せるのが定石です。

シリアライズとパースの方向ライブラリに任せる
// 2つの変換
変換向き用途
シリアライズオブジェクト→文字列エクスポート
パース文字列→オブジェクトインポート
// 手書きしない（ライブラリ）

CSV/JSONの変換は自前で書かない。ライブラリを使う：

  • JSON：Jackson（Spring標準）
  • CSV：OpenCSV / Apache Commons CSV

手書きパースはエスケープ・改行・引用符でハマる（別冊S2のKISS：車輪の再発明をしない）。

// シリアライズとパース // ─── JSON シリアライズ（エクスポート）─── // Jackson（Spring標準）を使う ObjectMapper mapper = new ObjectMapper(); String json = mapper.writeValueAsString(memos); // → List<Memo> → JSON文字列 ★ // ─── JSON パース（インポート）─── List<Memo> memos = mapper.readValue( json, new TypeReference<List<Memo>>(){}); // → JSON文字列 → List<Memo> ★ // ─── 手書きパースの罠（NG）─── // CSVを自前でsplit(",")すると… String[] parts = line.split(","); // → "東京, 大阪" のようにカンマを含む // 値で破綻 ★ // → 引用符・改行・エスケープも自前は地獄 // → ライブラリに任せる（別冊S2のKISS）★★★ // ─── CSV ライブラリ（OpenCSV例）─── // 書き出し CSVWriter writer = new CSVWriter(...); writer.writeNext(new String[]{ "会議, 重要", "明日10時"}); // → カンマを含む値も正しくエスケープ ★ // 読み込み CSVReader reader = new CSVReader(...); String[] row = reader.readNext(); // → 引用符・改行を正しく解釈 ★★★ // ─── 鉄則 ─── // CSV/JSON のパースは自前で書かない // → 枯れたライブラリを使う ★★★
POINT — CSV/JSONのパースは「自前で書かない」

初学者がやりがちな失敗が「CSVを split(",") で自前パース」。一見動きますが、値にカンマが含まれる（"東京, 大阪"）、改行が入る、引用符のエスケープで必ず破綻します。CSV/JSON は仕様が意外と複雑。だから枯れたライブラリ（Jackson, OpenCSV）に任せるのが鉄則（別冊S2のKISS・車輪の再発明をしない）。「自分で書くべきでないものは書かない」のも、立派な設計判断です。

06

取り込み時のバリデーション

IMPORT VALIDATION

インポートの核心「取り込むデータを検証する」。外部ファイルは何が入っているか分からない（Section 01）。タイトルが空、列が足りない、不正な値——検証してから登録します。別冊S7の例外、Vol.12のバリデーションに繋がります。

インポートデータの検証信じる前に確かめる
// 検証すべきこと
  • 形式：列が揃っているか
  • 必須：タイトルが空でないか
  • 型：日付が正しい形式か
  • 長さ：タイトルが長すぎないか
  • 参照：カテゴリが存在するか
// 既存のバリデーションを活用

本編Vol.2 で作った@NotBlank/@Size 等のバリデーション（Vol.12 で本格化）を、インポートデータにも適用。「画面入力もファイル入力も同じルールで検証」。

// 検証のタイミング

DB登録の前に検証。不正なら登録せず、エラーを集めて報告（Section 07）。

// インポートの検証 // ─── 検証の流れ ─── ファイル ↓ パース List<ImportRow>（生データ） ↓ 【1行ずつ検証】★★★ - タイトルが空でないか（@NotBlank） - 長さは適切か（@Size） - 日付は正しいか - カテゴリは存在するか ↓ 検証OKのものだけ List<Memo> ↓ DB登録 // ─── 検証の実装イメージ ─── for (ImportRow row : rows) { // 必須チェック if (row.getTitle() == null || row.getTitle().isBlank()) { errors.add("行" + n + ": タイトルが空です"); continue; // この行はスキップ } // 長さチェック if (row.getTitle().length() > 100) { errors.add("行" + n + ": タイトルが長すぎます"); continue; } // 検証OK → 登録対象に valid.add(toMemo(row)); } // → 不正な行を除き、正しい行だけ登録 ★★★ // ─── 既存バリデーションの再利用 ─── // 画面入力で使う @NotBlank @Size を // インポートデータにも適用できる // → 「入力源が違っても同じルール」★ // → 本編Vol.12 のバリデーションと統合 // ─── 別冊S6,S7 とのつながり ─── // S6：外部入力はSQLインジェクション対策 // S7：不正データは例外で扱う // S10：取り込みデータは検証してから // → 「外部入力は疑う」が一貫 ★★★
POINT — 「画面入力」も「ファイル入力」も同じ検証ルールで

インポートのバリデーションは、画面入力と同じルールを適用するのが理想。「タイトルは必須・100文字以内」というルールは、入力源（画面/ファイル）に関わらず一貫させるべき。本編Vol.2 で作り Vol.12 で本格化するバリデーションを、インポートでも再利用する。「どこから来たデータでも、同じ品質基準で守る」。検証ロジックを一元化すれば、漏れもなくなります。

07

部分的な取り込み失敗

PARTIAL IMPORT FAILURE

インポートで「100行中3行が不正」だったらどうするか。別冊S9の部分的失敗の3方針がここでも登場。全部やめる/正しい分だけ取り込む/結果を報告。インポートでは「報告」が特に重要です。

取り込み失敗の方針報告が重要
// 3方針（別冊S9の再登場）
方針A
全か無か
1行でも不正なら全部取り込まない。厳格。
方針B
正しい分だけ
不正行をスキップし正しい行だけ登録。
方針C
報告する
「97行取り込み、3行失敗（理由）」と結果を返す。
// インポートの推奨

方針B＋Cが親切：正しい行は取り込み、失敗した行と理由を報告。ユーザーは「どこを直せばいいか」分かる。

// 部分的な取り込み失敗 // ─── 方針B+C：取り込み＋報告（推奨）─── ImportResult result = importService.importMemos(file); result の中身： 成功: 97件 失敗: 3件 - 行5: タイトルが空 - 行12: カテゴリ「xxx」が存在しない - 行88: 日付の形式が不正 // → 正しい97件は取り込む（方針B） // → 失敗3件を理由付きで報告（方針C）★★★ // → ユーザーは「どこを直すか」分かる // ─── 結果のUI（別冊S8,S9）─── ┌──────────────────────────────┐ │ インポート結果 │ │ ✓ 97件を取り込みました │ │ ✗ 3件は取り込めませんでした │ │ 行5: タイトルが空 │ │ 行12: カテゴリが存在しない │ │ 行88: 日付が不正 │ └──────────────────────────────┘ // → 透明性が高い（別冊S8の安心感）★ // ─── 方針A（全か無か）の場合 ─── // 「1行でも不正なら全部やめる」 // → @Transactional でロールバック（別冊S5） // → 厳格だが「97件も無駄」になる // → 用途次第（給与データ等の厳格な場面向け） // ─── 判断（別冊S1,S9）─── メモのインポート → 方針B+C（柔軟・親切） 厳格な業務データ → 方針A（全か無か） // → 要求とユーザーの期待で選ぶ ★★★
POINT — インポートは「何が失敗したか」を必ず伝える

インポートで最悪なのは「黙って一部が取り込まれない」こと。ユーザーは「100件入れたのに97件しかない、なぜ?」と混乱する。だから「97件成功、3件失敗（行5: タイトルが空...）」と理由付きで報告する（方針C）。これは別冊S7の「ユーザーに分かりやすく伝える」、別冊S8の「安心感」の実践。失敗を隠さず、直し方が分かるように伝えるのが親切なインポートです。

08

データ入出力あるある失敗

DATA I/O PITFALLS

データ入出力で初心者が踏みやすい失敗を5つ。文字化け・自前パース・検証なしなど。別冊S2・S6・S7・S9の知識で回避できます。

データ入出力あるある 5選先人の轍を踏まない

① 文字コードを指定せず文字化け ★★★

  • デフォルト任せで日本語が化ける
  • 対処：UTF-8 を明示（Section 04）

② CSVを split(",") で自前パース ★★★

  • カンマ・引用符・改行で破綻
  • 対処：ライブラリを使う（Section 05）

③ インポートデータを検証せず登録 ★★★

  • 不正データがそのままDBに
  • 対処：登録前に検証（Section 06）

④ 失敗を黙って無視

  • 一部取り込めなかったのに報告なし
  • 対処：成功/失敗を報告（Section 07）

⑤ 巨大ファイルでメモリ不足

  • 全行を一度にメモリに載せる
  • 対処：ストリーム処理・件数制限（別冊S9の大量配慮）
