別冊S11のミッション

MISSION

別冊S11は「入力を正しく検証する設計を理解する」回。本編Vol.12 でバリデーションを本格化します。これまで各所に断片的にあった検証（Vol.2 の @NotBlank、Vol.11 のインポート検証）を、体系的に整理。「不正な入力からアプリを守る」理論を固めます。

02

なぜ検証するのか

WHY VALIDATE

「なぜ入力を検証するのか」。理由は3つ：正しいデータを保つ・アプリを壊さない・セキュリティ。「ユーザーは正しく入力してくれる」という楽観は禁物。別冊S6・S10の「外部入力は信用しない」と一貫します。

検証する3つの理由楽観は禁物
// 検証する3つの理由
理由1
データの正しさ
空タイトルや異常値がDBに入らないようにする。
理由2
アプリの安定
想定外の入力でクラッシュしないようにする。
理由3
セキュリティ
悪意ある入力（インジェクション等）を防ぐ。
// 「ユーザーは間違える」前提

ユーザーは悪気なく間違える。長すぎる入力、空欄、変な文字。さらに悪意ある攻撃者もいる。「正しく使ってくれる」は幻想。

// 検証しないと起きること // ─── 理由1：データの正しさ ─── 検証なし → 空タイトルのメモがDBに → 一覧で「（無題）」が大量に → データが汚れる ★ // ─── 理由2：アプリの安定 ─── 検証なし → 100万文字のタイトル → DB列の上限超過でエラー → 想定外の入力でクラッシュ ★ // ─── 理由3：セキュリティ ─── 検証なし → 悪意ある入力 → SQLインジェクション（別冊S6） → XSS（スクリプト埋め込み） → セキュリティ事故 ★★★ // ─── 「ユーザーは間違える」前提 ─── 楽観: 「みんな正しく入力するはず」 → 幻想 ★ 現実: - 悪気なく間違える（空欄、長すぎ） - 勘違いで変な値 - 悪意ある攻撃者 → だから検証が必要 ★★★ // ─── 別冊S6,S10 と一貫 ─── // S6：SQLインジェクション → 入力を疑う // S10：インポートデータ → 検証する // S11：あらゆる入力 → 検証する // → 「外部入力は信用しない」が一貫 ★★★
POINT — 「ユーザーは正しく入力する」という楽観を捨てる

バリデーションの出発点は「ユーザーは間違えるし、悪意ある人もいる」という前提。「みんな正しく使ってくれるはず」という楽観は、データ汚染・クラッシュ・セキュリティ事故を招く。これは別冊S6（SQLインジェクション）、別冊S10（インポート検証）と同じ精神。「あらゆる外部入力は、正しいと証明されるまで疑う」——防御の基本姿勢です。

03

検証する場所（多層防御）

WHERE TO VALIDATE

「どこで検証するか」。答えは「複数の層で」（多層防御）。画面（JS）・サーバ（Java）・DB（制約）。それぞれ役割が違う。「画面で検証したから安心」は危険です。

多層での検証各層に役割
// 3つの検証層
層役割
画面（JS）即フィードバック（UX）
サーバ（Java）本当の防御線
DB（制約）最後の砦
// 画面検証だけでは危険 ★★★

JSの検証は簡単に回避できる（ブラウザの開発者ツール、APIを直接叩く）。だからサーバ側の検証が必須。画面検証はUXのため、サーバ検証は守りのため。

// 本シリーズの方針

サーバ検証（Bean Validation）を中心に、DB制約（UNIQUE等）で最後の砦。画面検証は補助。

// 多層防御 // ─── 3層の検証 ─── ① 画面（JS） 即座にフィードバック 「タイトルを入力してください」 → UX向上（待たせない） ※ 簡単に回避できる ★ ② サーバ（Java/Bean Validation）★★★ @NotBlank @Size など → 本当の防御線 → 必ず通る・回避不可 ③ DB（制約） NOT NULL, UNIQUE, 長さ制限 → 最後の砦 → アプリのバグでも守る // ─── 画面検証だけの危険 ─── // JSの検証は… - ブラウザの開発者ツールで無効化できる - APIを直接叩けば素通り - JSが動かない環境もある // → 画面検証「だけ」は穴だらけ ★★★ // → サーバ検証が必須 // ─── なぜ多層か ─── // 1層が破られても次の層が守る // 画面すり抜け → サーバが弾く // サーバのバグ → DB制約が弾く // → 「多層防御」で堅牢に ★★★ // ─── 本シリーズの実例 ─── 画面: th:errors で表示 サーバ: @NotBlank @Size（Vol.2,12） DB: UNIQUE（Vol.3）, NOT NULL // → 3層で守ってきた
POINT — 「サーバ側検証」は絶対に省略しない

最も重要な原則：「サーバ側の検証は絶対に省略しない」。画面（JS）の検証は簡単に回避できるので、それだけに頼るのは無防備。攻撃者は API を直接叩いてくる。画面検証はUX（親切さ）のため、サーバ検証は守りのためと役割が違う。「画面でチェックしてるから大丈夫」は最も危険な思い込み。本シリーズは Bean Validation でサーバ検証を必ず行います。

04

Bean Validation

BEAN VALIDATION

Java の標準的な検証の仕組みBean Validation。アノテーション（@NotBlank, @Size 等）でルールを宣言。Spring と統合され、@Validated で自動検証。Vol.2 で少し使いましたが、ここで体系的に理解します。

アノテーションによる検証宣言的に書く
// 主なアノテーション
注釈意味
@NotNullnull でない
@NotBlank空白でない（文字列）
@Size長さ範囲
@Min/@Max数値範囲
@Emailメール形式
@Pattern正規表現
// 宣言的の利点

検証ルールをフィールドに宣言するだけ。検証ロジックを毎回書かない（別冊S2のKISS）。ルールが一目で分かる。

// Bean Validation // ─── フォームクラスにルールを宣言 ─── public class MemoForm { @NotBlank(message = "タイトルは必須です") @Size(max = 100, message = "タイトルは100文字以内") private String title; @Size(max = 2000, message = "本文は2000文字以内") private String content; @NotNull(message = "カテゴリを選択") private Long categoryId; } // → ルールをフィールドに宣言 ★ // → 検証ロジックを毎回書かない（KISS） // ─── Controller で自動検証 ─── @PostMapping public String create( @Validated @ModelAttribute MemoForm form, BindingResult br) { // 検証結果 if (br.hasErrors()) { // 検証エラーがあれば入力画面に戻す return "memos/form"; } // 検証OK → 処理続行 ★ service.create(form); return "redirect:/memos"; } // → @Validated で自動検証 // → BindingResult にエラーが入る // ─── 画面でエラー表示（Thymeleaf）─── <input th:field="*{title}"> <span th:errors="*{title}"></span> // → 「タイトルは必須です」が表示 ★ // ─── ネストした検証 ─── // @Valid でオブジェクト内も検証 // リストの各要素も検証できる
POINT — 検証ルールは「宣言」する、ロジックを書かない

Bean Validation の良さは「検証を宣言的に書ける」こと。if (title == null || title.isEmpty()) という検証ロジックを毎回書く代わりに、@NotBlank と宣言するだけ。これは別冊S2のKISS（自前で書かない）の実践。ルールがフィールドに集まるので「何が必須か」が一目で分かる。検証ロジックの重複もなくなる。宣言的なコードは、読みやすく間違いにくいのです。

05

防御的プログラミング

DEFENSIVE PROGRAMMING

防御的プログラミングとは、「想定外の入力・状態に備えてコードを書く」考え方。null チェック、引数の検証、早期リターン。「起きないはず」を信じない。バリデーションの精神をコード全体に広げます。

防御的プログラミングの実践想定外に備える
// 主な技法
  • 引数の検証：メソッド冒頭で前提を確認
  • 早期リターン：異常なら早く抜ける
  • null 安全：Optional, null チェック
  • 不変性：変更されない設計
// ガード節（早期リターン）

メソッド冒頭で異常な条件を弾く（ガード節）。「正常な道」をネストの外に出す。読みやすく安全。

// やりすぎ注意

ただし全部に過剰な防御は冗長（YAGNI・別冊S2）。「外部との境界」を重点的に守る。内部のメソッドまで毎回チェックは過剰。

// 防御的プログラミング // ─── ガード節（早期リターン）─── public void bulkArchive(List<Long> ids) { // 冒頭で異常を弾く（ガード節）★ if (ids == null || ids.isEmpty()) { return; // 早期リターン（別冊S9でも実践） } // ここから先は ids が有効と保証される mapper.bulkArchive(ids); } // → 「正常な道」が読みやすい ★ // ─── 引数の検証 ─── public Memo findById(Long id) { // 前提を確認（防御） Objects.requireNonNull(id, "idは必須"); ... } // ─── null 安全（Optional）─── public Optional<Memo> find(Long id) { return Optional.ofNullable( mapper.findById(id)); } // 呼び出し側： service.find(id) .orElseThrow(() -> new MemoNotFoundException(id)); // → null を型で表現、見落とし防止 ★ // ─── やりすぎ注意（別冊S2のYAGNI）─── 過剰: // 内部の全メソッドで全引数をチェック // → 冗長・読みにくい ★ 適切: // 「外部との境界」を重点的に守る // - Controller（入力の入口） // - public API のメソッド // - インポート（別冊S10） // → 信頼境界で守る ★★★
POINT — 防御は「信頼境界」で重点的に

防御的プログラミングは大事ですが、「全部に過剰な防御」は逆効果（冗長・読みにくい）。重点を置くべきは「信頼境界」——外部とアプリの境目（Controller、public API、インポート）。ここをしっかり守れば、内部は信頼して書ける。「入口で守り、中は信頼」というメリハリ。別冊S2のYAGNIと同じく、必要なところに必要なだけの防御が、良いバランスです。

06

エラーメッセージの設計

ERROR MESSAGE DESIGN

検証で「エラーをどう伝えるか」も重要。ユーザーが「何を・どう直せばいいか」分かるメッセージが理想。別冊S7（ユーザー向け表示）・別冊S8（UX）と繋がる、親切な検証の設計です。

良いエラーメッセージ直し方が分かる
// 良いメッセージの条件
  • 何が問題か：「タイトルが空です」
  • どう直すか：「100文字以内で入力」
  • 具体的に：「現在120文字」
  • その場で：該当フィールドの近くに表示
// 悪いメッセージ
  • 「エラー」（何が?）
  • 「入力値が不正です」（どこが?）
  • 技術的すぎる（「ConstraintViolation」）
// 別冊S7,S8 との連携

別冊S7「ユーザーに分かりやすく」、別冊S8「迷わせない」が、エラーメッセージにも当てはまる。

// エラーメッセージの設計 // ─── 悪いメッセージ ─── ✗ 「エラー」 → 何が問題か不明 ★ ✗ 「入力値が不正です」 → どのフィールドか不明 ✗ 「ConstraintViolationException」 → 技術的すぎる（別冊S7）★ // ─── 良いメッセージ ─── ✓ 「タイトルを入力してください」 → 何を・どうするか明確 ★ ✓ 「タイトルは100文字以内で 入力してください（現在120文字）」 → 具体的・直し方が分かる ★★★ ✓ 該当フィールドの真下に表示 → どこが問題か一目（別冊S8） // ─── Bean Validation の message ─── @NotBlank(message = "タイトルを入力してください") @Size(max = 100, message = "タイトルは100文字以内で入力してください") // → message でユーザー向け文言を指定 ★ // ─── 表示位置（別冊S8：迷わせない）─── <input th:field="*{title}"> <span class="error" th:errors="*{title}"></span> // → 該当フィールドの真下にエラー // → 「どこを直すか」が一目 ★★★ // ─── 複数エラーをまとめて ─── // 1つ直す→次のエラー…の繰り返しは苦痛 // → 全エラーを一度に表示するのが親切 // → Bean Validation は全項目を検証して返す
POINT — エラーメッセージは「直し方」まで伝える

良いエラーメッセージは「何が問題か」だけでなく「どう直すか」まで伝える。「タイトルが長すぎます」より「タイトルは100文字以内で入力してください（現在120文字）」。ユーザーが「あと20文字減らせばいい」と即分かる。これは別冊S7（ユーザーに分かりやすく）、別冊S8（迷わせない）の実践。エラーは「責める」のでなく「導く」もの。親切なエラーが、良いUXを作ります。

07

バリデーションと例外の連携

VALIDATION & EXCEPTION

別冊S7（例外）と別冊S11（バリデーション）がどう連携するか。「入力エラーはバリデーション、業務エラーは例外」と使い分け。本編Vol.12 では両方を@ControllerAdvice で統合します。

バリデーションと例外の使い分け守りの両輪を統合
// 使い分け
種類手段例
入力エラーバリデーション空タイトル
業務エラー例外重複・未検出
システムエラー例外DB障害
// なぜ使い分けるか
  • 入力エラー：入口で防げる→バリデーション＋フォーム再表示
  • 業務エラー：処理して初めて分かる→例外
// Vol.12 での統合

バリデーションエラーはBindingResult、業務例外は@ControllerAdvice。両方で親切なエラー表示。

// バリデーションと例外の連携 // ─── 入力エラー → バリデーション ─── // 「タイトルが空」は入口で防げる @Validated MemoForm form, BindingResult br ↓ if (br.hasErrors()) return "form"; // 再表示 // → バリデーションで弾く（別冊S11）★ // ─── 業務エラー → 例外 ─── // 「カテゴリ名が重複」は // DBに問い合わせて初めて分かる if (categoryExists(name)) throw new DuplicateException(name); // → 例外で対処（別冊S7）★ // ─── @ControllerAdvice で統合（Vol.12）─── @ControllerAdvice class GlobalExceptionHandler { // 業務例外をハンドリング（別冊S7） @ExceptionHandler(DuplicateException.class) public String handleDup(...) { return "error/conflict"; } } // → 入力エラー：BindingResult＋フォーム // → 業務例外：@ControllerAdvice // → 両方で親切なエラー表示 ★★★ // ─── 守りの全体像（S7＋S11）─── 入力 ↓ バリデーション（別冊S11） → 入力エラーは入口で弾く ↓ OK 業務処理 ↓ 例外（別冊S7） → 業務/システムエラーを対処 ↓ @ControllerAdvice で集約表示 // → 守りの両輪が連携 ★★★
POINT — 「入口で防げるか」でバリデーションと例外を分ける

使い分けの基準は「入口で防げるか」。「タイトルが空」は入力を見ただけで分かる→バリデーション。「カテゴリ名が重複」はDBに聞いて初めて分かる→例外。前者はフォーム再表示で「直してもらう」、後者は例外で「対処する」。本編Vol.12 では、別冊S7（例外）と別冊S11（バリデーション）の両方の理論を統合し、あらゆるエラーを親切に扱います。守りの両輪が揃う回です。

08

バリデーションあるある失敗

VALIDATION PITFALLS

バリデーションで初心者が踏みやすい失敗を5つ。画面検証だけ・過剰防御・不親切なメッセージなど。別冊S2・S6・S7の知識で回避できます。

バリデーションあるある 5選先人の轍を踏まない

① 画面検証だけでサーバ検証なし ★★★

  • JS回避でサーバに不正データ
  • 対処：サーバ検証を必ず（Section 03）

② 検証ロジックを各所に重複

  • 同じチェックをあちこちに手書き
  • 対処：Bean Validation で宣言的に（Section 04）

③ エラーメッセージが不親切 ★★★

  • 「エラー」だけで直し方不明
  • 対処：何を・どう直すかを伝える（Section 06）

④ 過剰な防御で冗長

  • 内部の全メソッドで全チェック
  • 対処：信頼境界で重点的に（Section 05）

⑤ 入力エラーを例外にする（逆も）

  • 空タイトルで例外を投げる等
  • 対処：入口で防げるかで使い分け（Section 07）

POINT — バリデーションは「守りの集大成」

バリデーションは、別冊の守りの理論が集まる分野です。S2（KISS：宣言的に書く）、S6（外部入力を疑う）、S7（例外との連携・親切な表示）、S10（インポートの検証）が全て関わる。本編Vol.12 では、これらを統合して「あらゆる不正入力からアプリを守る」仕組みを作る。RDT版で積み上げた「外部入力は信用しない」という一貫した姿勢が、ここで完成形になります。
