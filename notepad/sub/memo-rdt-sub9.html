01

別冊S9のミッション

MISSION

別冊S9は「まとめて処理する設計を理解する」回。本編Vol.10 で「複数メモを選んでまとめてアーカイブ/削除」を実装します。1件ずつの操作をN件まとめて行うとき、効率・安全・UXの3つを考える必要があります。

なぜ一括処理が必要か1件ずつは面倒
// Vol.9 の限界

  • 一括処理のトランザクションを設計

POINT — 一括処理は「効率」と「整合性」の両方が要る

一括処理は単に「速くする」だけではありません。「50件まとめてアーカイブ→途中で失敗→25件だけアーカイブされた中途半端な状態」を防ぐ整合性も必要（別冊S5のトランザクション）。「効率（まとめて1回）」と「整合性（全部か、ゼロか）」の両立が一括処理の設計。本編Vol.10 では、この両方を意識して実装します。

02

1件ずつ vs まとめて

ONE-BY-ONE VS BULK

複数件を処理する方法は2つ。「ループで1件ずつN回」と「SQLでまとめて1回」。後者が圧倒的に効率的。別冊S6のN+1問題と同じ「DB往復を減らす」考え方です。

2つのアプローチの比較DB往復が決定的
// 1件ずつ（ループ）

Java のループで1件ずつUPDATE。N件ならN回のDB往復。実装は素直だが遅い。

// まとめて（バルク）

1回のSQLで全件処理。WHERE id IN (...) や バルクINSERT。DB往復が1回。

// 比較
1件ずつまとめて
DB往復N回1回
速度遅い速い
実装素直IN句等
// 1件ずつ vs まとめて // ─── 1件ずつ（ループ・非効率）─── for (Long id : ids) { mapper.updateArchived(id, true); // → UPDATE ... WHERE id=? を // N回実行（DB往復N回）★ } // 50件 → 50回のDB往復 // → N+1問題（別冊S6）と同じ構造 // ─── まとめて（バルク・効率的）─── mapper.archiveByIds(ids); // → UPDATE ... WHERE id IN (1,2,...,50) // 1回のSQL（DB往復1回）★★★ // 50件 → 1回のDB往復 // ─── SQL の違い ─── 1件ずつ: UPDATE memos SET archived=true WHERE id=1; UPDATE memos SET archived=true WHERE id=2; ... ×50 まとめて: UPDATE memos SET archived=true WHERE id IN (1,2,3,...,50); // → 1文で50件更新 ★★★ // ─── 別冊S6 とのつながり ─── // N+1問題：ループ内で個別クエリ → 遅い // 一括処理：まとめて1回 → 速い // → 「DB往復を減らす」は共通の原則 ★★★
POINT — 「ループ内クエリ」を見たら一括化を検討

別冊S6のN+1問題と同じく、「ループの中でDBクエリを実行している」のを見たら、一括化（バルク）できないかを検討する習慣を。「N件をループで1件ずつUPDATE」は、ほとんどの場合「WHERE id IN (...)」で1回にまとめられる。DB往復は処理時間の大半を占めるので、これを減らすと劇的に速くなる。「まとめられるものはまとめる」が効率化の基本です。

03

IN句による一括処理

BULK WITH IN CLAUSE

一括処理の主役IN句。WHERE id IN (1, 2, 3) で複数IDをまとめて指定。MyBatis では本編Vol.7 で使った <foreach> でIDリストを展開します。もちろん #{} で安全に（別冊S6）。

IN句 + foreach の一括処理複数IDをまとめて
// IN句の基本

WHERE id IN (リスト) で「リストのいずれかに一致」。一括の更新・削除・取得に使える。

// foreach でリスト展開（Vol.7）

MyBatis の <foreach> で、IDリスト [1,2,3] を (1,2,3) に展開。本編Vol.7 のタグ検索で既出。

// 一括操作の種類
  • 一括アーカイブ：UPDATE ... WHERE id IN
  • 一括削除：DELETE ... WHERE id IN
  • 一括取得：SELECT ... WHERE id IN
// IN句 + foreach（本編Vol.10で実装） // ─── 一括アーカイブ ─── <update id="archiveByIds"> UPDATE memos SET archived = true WHERE id IN <foreach item="id" collection="ids" open="(" separator="," close=")"> #{id} </foreach> </update> // → ids=[1,2,3] を (1,2,3) に展開 // → 全て #{} で安全（別冊S6）★★★ // ─── 一括削除 ─── <delete id="deleteByIds"> DELETE FROM memos WHERE id IN <foreach item="id" collection="ids" open="(" separator="," close=")"> #{id} </foreach> </delete> // ─── 実行されるSQL ─── UPDATE memos SET archived=true WHERE id IN (1, 5, 8, 12, 23); // → 5件を1回で更新 ★★★ // ─── 本編Vol.7 の foreach 再利用 ─── // タグ検索で使った foreach と同じ技術 // → 一度学んだ技術が一括処理にも活きる // → 「IDリストを IN 句に展開」は定番 ★
POINT — IN句のリストが「空」のときに注意

本編Vol.7 でも触れた罠：IN句に渡すリストが空だと IN () となり構文エラー。一括操作で「1件も選択せずに実行」されたときに起きる。対策は「リストが空なら処理をスキップ」（<if test="ids != null and !ids.isEmpty()"> やService側のチェック）。一括処理は「0件選択」という境界を必ず考慮する。これは別冊S3の境界値テストの対象でもあります。

04

一括処理とトランザクション

BULK & TRANSACTION

一括処理はトランザクションと深く関わります（別冊S5）。「50件まとめて処理」が途中で失敗したら? 「全部か、ゼロか」にするか、「成功した分は残す」か。設計判断が必要です。

一括処理の原子性全部か、ゼロか
// 単一SQLなら自動で原子的

UPDATE ... WHERE id IN (...) という1文のSQLは、それ自体が原子的（全件成功 or 全件失敗）。DBが保証。

// 複数SQLなら @Transactional

一括処理が複数のSQLに分かれる場合（例：アーカイブ＋関連更新）は、@Transactional でまとめる（別冊S5）。

// ループ処理の落とし穴

もしループで1件ずつ処理する場合、@Transactional がないと途中失敗で中途半端に。25件目で失敗→24件だけ処理済み。@Transactional 必須。

// 一括処理のトランザクション // ─── ① 単一SQL（自動で原子的）─── UPDATE memos SET archived=true WHERE id IN (1,2,...,50); // → 1文のSQLは原子的 // → 50件全部成功 or 全部失敗 // → DBが保証（追加の対策不要）★ // ─── ② ループ処理（@Transactional必須）─── @Transactional // ← 必須 ★★★ public void archiveAll(List<Long> ids) { for (Long id : ids) { mapper.updateArchived(id, true); // もし25件目で例外が出たら… } } // @Transactional あり： // → 全件ロールバック（0件処理）★ // @Transactional なし： // → 24件処理済み・25件目以降未処理 // → 中途半端（別冊S5）★★★ // ─── 本シリーズの方針 ─── // 可能なら単一SQL（IN句）で原子的に // → シンプルかつ安全（KISS・別冊S2） mapper.archiveByIds(ids); // 1文で完結 ★ // ─── 別冊S5 の復習 ─── // 複数操作 → @Transactional // 一括処理も「複数件の操作」なので // 原子性を意識する ★★★
POINT — 可能なら「単一SQL」にすると原子性が自動で得られる

一括処理は「ループで@Transactional」より「単一SQL（IN句）」の方が優れることが多い。単一の UPDATE ... WHERE id IN (...) なら、原子性はDBが自動保証し、性能も良い（DB往復1回）。ループ＋@Transactional は原子性は得られるがDB往復がN回。「1文のSQLで表現できないか」をまず考えるのが、一括処理設計の出発点。シンプルさと安全性が同時に手に入ります。

05

部分的失敗の扱い

PARTIAL FAILURE

一括処理の難所「一部だけ失敗したら?」。50件中3件が条件を満たさない場合、「全部やめる」「できる分だけやる」「失敗を報告」のどれを選ぶか。要求次第で変わる設計判断です。

部分的失敗の3つの方針要求で決める
// 3つの方針
方針A
全か無か
1件でも失敗なら全部ロールバック。一貫性重視。
方針B
できる分だけ
成功分はそのまま、失敗分はスキップ。完遂重視。
方針C
報告する
「50件中47件成功、3件失敗」と結果を伝える。
// 本シリーズの方針

メモのアーカイブ・削除は方針A（全か無か）がシンプルで安全。単一SQLなら自然にこうなる。複雑な部分失敗処理はYAGNI（別冊S2）。

// 部分的失敗の方針 // ─── 方針A：全か無か（本シリーズ）─── UPDATE memos SET archived=true WHERE id IN (1,2,3); // → 1件でもエラーなら全件ロールバック // → 中途半端を作らない（別冊S5）★ // → メモのアーカイブはこれで十分 // ─── 方針B：できる分だけ ─── // 例：メール一括送信 // 「3件失敗しても、47件は送りたい」 for (Long id : ids) { try { send(id); } catch (Exception e) { failed.add(id); // 失敗を記録 // → 続行（別冊S7：握りつぶしでなく記録） } } // → 完遂を優先する処理向け // ─── 方針C：報告する ─── BulkResult result = archiveAll(ids); // result: 成功47件、失敗3件、失敗理由 // → ユーザーに「47件アーカイブしました // （3件は既にアーカイブ済み）」 // → 透明性が高い ★ // ─── 判断は要求次第 ─── 取り消し可能・一貫性重視 → 方針A 完遂優先（メール等） → 方針B 透明性重視 → 方針C // → 「何を大事にするか」で選ぶ（別冊S1,S2）★★★ // → メモ帳はシンプルに方針A
POINT — 部分失敗の方針は「要求から」決める

部分的失敗の扱いは技術ではなく要求の問題。「メモのアーカイブ」なら全か無か（方針A）でいい。でも「100人へのメール一括送信」なら、3件失敗で全部やめたら困る（方針B）。「何を優先するか」は要求とユーザーの期待から決まる（別冊S1）。本シリーズはメモ操作なのでシンプルに方針A。「銀の弾丸はない、要求で選ぶ」——別冊S2の判断の考え方です。

06

大量データへの配慮

LARGE DATA CARE

一括処理は「大量」になると別の問題が出ます。IN句に1万個、1回のトランザクションで10万件——これらは限界や副作用がある。別冊S2の「測定してから」で、現実的な配慮を学びます。

大量一括処理の注意点限界とチャンク分割
// 大量処理の問題
  • IN句の上限：DBによって個数制限（数千〜）
  • 長いトランザクション：ロックが長く他をブロック（別冊S5）
  • メモリ：大量データを一度に載せると枯渇
// 対策：チャンク分割

大量なら「100件ずつに分割して処理」（チャンク）。一度に全部やらず、適度な単位で。

// メモ帳での現実

個人のメモが数千件を一度に一括操作することは稀。YAGNI（別冊S2）で、メモ帳v1は普通のIN句で十分。「大量になったら分割」と知っておけばOK。

// 大量データへの配慮 // ─── 問題：IN句に大量のID ─── WHERE id IN (1, 2, ..., 100000) // → DBのIN句上限を超える可能性 // → SQLが巨大化、性能劣化 ★ // ─── 対策：チャンク分割 ─── // 100件ずつに分けて処理 List<List<Long>> chunks = partition(ids, 100); for (List<Long> chunk : chunks) { mapper.archiveByIds(chunk); // → 100件ずつ × 複数回 } // → IN句の上限を回避 // → トランザクションも短く保てる ★ // ─── 長いトランザクションの害（別冊S5）─── // 10万件を1トランザクションで処理 // → 長時間ロック → 他の処理がブロック ★ // → チャンクごとにコミットも検討 // （ただし原子性とのトレードオフ） // ─── メモ帳での現実（YAGNI・別冊S2）─── // 個人のメモ = 多くて数百〜数千件 // 一度の一括操作 = せいぜい数十件 // → 普通の IN句で全く問題ない ★ // → チャンク分割は「大量になったら」 // → 推測で先に複雑化しない（測定してから） // ─── 別冊S2 の判断 ─── // 「大量データの心配」は // 実際に大量になってから対処 // → 早すぎる最適化を避ける ★★★
POINT — 「大量対策」も測定してから（早すぎる最適化を避ける）

チャンク分割などの大量データ対策は重要ですが、メモ帳v1で先回りして実装するのは過剰（別冊S2のYAGNI・早すぎる最適化）。個人のメモが「一度に1万件一括操作」する場面はまずない。「そういう問題と対策がある」と知っておき、実際に大量になったら測定して対処する。知識として持ちつつ、今は普通のIN句でシンプルに——これが現実的な判断です。

07

一括処理のUX

BULK OPERATION UX

一括処理はUX（別冊S8）も重要。複数選択のしやすさ、「何件選択中か」の表示、破壊的操作の確認。「まとめて消す」は影響が大きいので、慎重なUX設計が要ります。

一括操作のUX設計選択と確認
// 複数選択のUX
  • チェックボックスで複数選択
  • 「全選択」チェックボックス
  • 「N件選択中」の表示
// 一括操作の確認（別冊S8）
  • 一括アーカイブ：復元可能→確認は軽め or なし
  • 一括削除：取り返しつかない→必ず確認＋件数表示

「50件を完全に削除します。よろしいですか?」と件数を見せるのが重要。

// 結果のフィードバック

「50件をアーカイブしました」と結果を伝える（別冊S8の安心感）。

// 一括操作のUX // ─── 複数選択UI ─── ┌──────────────────────────────┐ │ ☑全選択 [選択した3件を ▼] │ │ [アーカイブ][削除] │ ├──────────────────────────────┤ │ ☑ 古いメモ1 │ │ ☑ 古いメモ2 │ │ ☑ 古いメモ3 │ │ ☐ 重要メモ（選択しない） │ └──────────────────────────────┘ // → チェックで複数選択 // → 「3件選択中」が分かる ★ // ─── 破壊的操作の確認（別冊S8）─── 一括削除ボタン → 「選択した50件を完全に削除します。 この操作は取り消せません。」 → [削除する] [キャンセル] // → 件数を明示 ★★★ // → 「50件」を見せることで誤操作防止 // ─── 一括アーカイブ（復元可・軽め）─── 一括アーカイブ → 確認は軽め or なし（復元できるから） → 「50件をアーカイブしました [元に戻す]」★ // → 別冊S8「軽い操作は取り消しで対応」 // ─── 0件選択の扱い ─── 何も選択せず実行 → 「メモを選択してください」 → エラーにせず親切に促す ★ // → Section 03 の「空リスト」をUXでも配慮 // ─── 結果のフィードバック ─── 処理後：「50件をアーカイブしました」 // → 何件処理したか伝える（安心感）★★★
POINT — 一括削除は「件数」を見せて誤操作を防ぐ

一括操作で最も危険なのが一括削除。1件の削除より影響が桁違いに大きい。だから確認ダイアログに「50件を削除します」と件数を明示する。「あ、50件も選んでた、想定と違う」と気づくチャンスを与える。別冊S8の「取り返しのつかない操作ほど確認」が、一括処理では特に重要。件数表示は、大事故を防ぐ最後の砦です。

08

一括処理あるある失敗

BULK PITFALLS

一括処理で初心者が踏みやすい失敗を5つ。ループ内クエリ・空リスト・確認なし一括削除など。別冊S5〜S8の知識で回避できます。

一括処理あるある 5選先人の轍を踏まない

① ループ内で1件ずつクエリ ★★★

  • N件をループでN回DB往復（遅い）
  • 対処：IN句でまとめて1回（Section 02,03）

② 空リストで IN () エラー ★★★

  • 0件選択で IN () 構文エラー
  • 対処：空ならスキップ（Section 03）

③ ループ処理に @Transactional なし

  • 途中失敗で中途半端（別冊S5）
  • 対処：単一SQL化、またはトランザクション

④ 一括削除に確認なし ★★★

  • 大量データが一瞬で消える
  • 対処：件数表示＋確認（別冊S8・Section 07）

⑤ アーカイブにも archived 条件忘れ

  • 一括対象に既にアーカイブ済みが混入
  • 対処：対象クエリにも状態条件（別冊S8）
// あるある失敗の対処 // ─── ① ループ内クエリ → 一括化 ─── NG: for (Long id : ids) mapper.updateArchived(id, true); // → N回のDB往復 ★ OK: mapper.archiveByIds(ids); // → 1回のDB往復（IN句）★★★ // ─── ② 空リスト対策 ─── NG: ids=[] で archiveByIds(ids) // → IN () 構文エラー ★ OK: if (ids == null || ids.isEmpty()) { return; // 何もしない } mapper.archiveByIds(ids); // → 空なら処理スキップ ★★★ // ─── ④ 一括削除の確認 ─── NG: 削除ボタン → 即DELETE // → 50件が一瞬で消える ★★★ OK: 「50件を削除します」確認 → OKで実行 // → 件数を見せて誤操作防止（別冊S8）★ // ─── 良い一括処理の条件 ─── ✓ IN句でまとめて1回 ✓ 空リストはスキップ ✓ 原子性（単一SQL or @Transactional） ✓ 一括削除は件数表示＋確認 ✓ 対象クエリにも状態条件 ★★★
POINT — 一括処理は「効率・安全・UX」の総合演習

一括処理は、これまで学んだ理論の総合演習です。効率（IN句・別冊S6のN+1）、安全（トランザクション・別冊S5）、UX（確認・別冊S8）、境界（空リスト・別冊S3）が全部関わる。1つの機能に複数の理論が絡むからこそ、それぞれの理論を理解していると綺麗に設計できる。本編Vol.10 は、別冊で積み上げた知識を一度に活かす場面です。
