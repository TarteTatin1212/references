01

別冊S1のミッション

MISSION

別冊S1は「要求工学を理論として理解する」回。本編Vol.1 で「手順」は体験しました。でも「なぜその手順なのか」が分からないと応用が利きません。理論を知ると、未知の場面でも判断できるようになります。

なぜ要求工学を学ぶのか手戻りコストの法則
// 1:10:100 の法則

ソフトウェア工学の経験則に「1:10:100」があります。要求段階で見つけたバグの修正コストを1とすると：

  • 要求段階での修正 → コスト 1
  • 設計段階での発覚 → コスト 10
  • リリース後の発覚 → コスト 100

早く見つけるほど安い。だからコードを書く前の要求工学に投資する価値があります。

// 別冊S1で得られる視点
  • 要求工学の5活動を「なぜ」で説明できる
  • WHYの重要性を具体例で語れる
  • INVESTでストーリーの良し悪しを判定できる
  • 変更管理の作法を理解する
// 手戻りコストの実例 // ─── ケース：検索機能 ─── 要求段階で気づく： 「あ、検索は部分一致だよね」 → ストーリーを1行修正 → コスト：5分 設計段階で気づく： 「LIKE '%x%' だとインデックス効かない」 → テーブル設計を見直し → コスト：半日 リリース後に気づく： 「1万件で検索が10秒かかる」 → 全文検索エンジン導入 → DBスキーマ変更 → 全機能の再テスト → コスト：2週間 + 信頼失墜 // → 同じ問題でも // 気づくタイミングで // コストが100倍変わる ★★★ // だから「考えてから書く」
POINT — 「速く作る」より「正しく作る」

初学者は「とにかく早くコードを書きたい」と焦ります。でも間違ったものを速く作っても、作り直しで結局遅い。要求工学は「正しいものを作る」ための投資。急がば回れが、結果的に最速です。

02

要求工学の5活動の本質

5 ACTIVITIES

要求工学は獲得・分析・仕様化・検証・管理の5活動。本編では流れを体験しましたが、各活動が「何のためにあるか」を理解すると、抜け漏れに気づけます。

5活動それぞれの「問い」各活動の存在理由
活動問い成果物
獲得何が欲しい？何に困ってる？生の要求・ペルソナ
分析本当に必要なのは？構造化要求・優先度
仕様化決まったことは？US・受入条件・NFR
検証これで本当に良い？レビュー・プロト
管理何が変わった？変更要求・トレース
// 一直線ではなく「反復」

5活動は上から下へ一度流れて終わりではありません。仕様化していて要求の矛盾に気づいたら獲得に戻る。検証で方向ズレが見つかったら分析に戻る。行き来するのが正常です。

// 各活動の「サボると起きること」 獲得をサボると： → 誰も使わないものを作る → 「作ったけど的外れ」 分析をサボると： → 全部入りで終わらない → 優先度なしで迷走 仕様化をサボると： → 「完成」の定義が曖昧 → 認識ズレで揉める 検証をサボると： → 作ってから「これじゃない」 → 手戻り爆発 管理をサボると： → 変更が無秩序に増殖 → スコープ崩壊（スコープクリープ） // → 5活動はそれぞれ // 別の事故を防いでいる ★★★ // → どれも省略できない
POINT — 「反復」を恐れない

「もう要求は決まったから」と前に戻らないのは危険信号。設計や実装で要求の問題に気づいたら、要求に戻るのが正しい。早く戻るほど傷は浅い。これは設計フロー（別冊S2予定）にも共通する大原則です。

03

WHYが最重要な理由

WHY MATTERS MOST

WHO/WHAT/WHY のうち、WHY が一番抜けやすく、一番重要。「なぜそれを作るのか」が分かっていないと、機能の取捨選択ができず、優先順位も付けられない。WHY こそが羅針盤です。

WHYが羅針盤になる機能判断の基準
// WHY があると判断できる

本シリーズのWHY：「分類軸が1つしかなく後から探しにくい、を解決する」。

このWHYがあると、機能の採否を判断できます：

  • カテゴリ機能 → 採用（分類軸を増やす＝WHYに直結）
  • タグ機能 → 採用（別の分類軸＝WHYに直結）
  • 検索機能 → 採用（探しやすく＝WHYに直結）
  • 絵文字スタンプ → 却下（WHYと無関係）

WHYが機能の「採否の物差し」になります。

// ユーザーストーリーの「so that」 // so that = WHY を文に埋め込む装置 As a メモを整理したいユーザー I want タグを付けたい so that カテゴリと別の軸でも絞れる ↑ これがWHY // ─── so that が書けない時 ─── As a ユーザー I want ダークモードが欲しい so that … ？？？ // 「so that」が書けない＝ // WHYが無い＝本当に要るか怪しい // → 立ち止まって考える ★★★ // ─── 5回の「なぜ」（なぜなぜ分析）─── 「検索が欲しい」 → なぜ？「メモが探せないから」 → なぜ？「件数が多いから」 → なぜ？「毎日記録するから」 → なぜ？「思いつきを逃したくないから」 → なぜ？「アイデアが価値だから」 // → 本当のWHY（=価値）に到達 ★★★ // → 表面の要望を掘ると本質が見える
NG — WHYなしの機能追加

「競合にあるから」「なんとなく便利そう」で機能を足す。WHYがないので優先順位も付かず、際限なく増える。完成しない。

OK — WHYで取捨選択

すべての機能をWHYに照らして採否を判断。WHYに貢献しない機能は勇気を持って却下（or Won't行き）。スコープが守れる。

POINT — 「なぜなぜ分析」で本当のWHYを掘る

表面の要望（WHAT）を「なぜ？」と5回繰り返すと、本当の価値（WHY）にたどり着きます。「検索が欲しい」の奥には「アイデアを逃したくない」がある。本質のWHYが分かると、検索以外の解決策も見えてくる。これがトヨタ発祥のなぜなぜ分析です。

04

要求・要件・仕様の境界

REQUEST / REQUIREMENT / SPEC

混同しやすい「要求・要件・仕様」の3語。この境界が曖昧だと業務会話が噛み合いません。「曖昧な声 → 検証可能な条件 → 技術的記述」という変換の流れを押さえます。

3語の変換フロー曖昧 → 明確 → 技術
STAGE 1
要求 (Request)
ユーザーの生の声。曖昧でOK。「探しやすいといいな」
STAGE 2
要件 (Requirement)
システムが満たすべき明文化された条件。測定可能。「全文検索500ms以内」
STAGE 3
仕様 (Specification)
どう実現するかの技術記述。「MATCH AGAINST + FULLTEXT INDEX」
// 変換の責任は誰に
  • 要求 → 要件：要求分析者・SEの仕事
  • 要件 → 仕様：設計者・開発者の仕事
  • 個人開発では全部自分。だから境界を意識する
// 検索機能で3段階を追う // ─── 要求（曖昧でOK）─── ユーザー： 「メモがすぐ探せるといいな」 // ─── 要件（測定可能に）─── @SEARCH-001 キーワードで全文検索でき、 1000件に対して500ms以内（P95）で 結果を返すこと @SEARCH-002 該当0件の場合は 「該当なし」を明示すること // ─── 仕様（技術的に）─── - memos.content に FULLTEXT INDEX - MeCab パーサで日本語分かち書き - MATCH(content) AGAINST(? IN BOOLEAN MODE) で検索 - 結果が空なら専用ビューを返す // → 同じ「検索」でも // 抽象度が3段階で変わる ★★★ // → 要件は「何を」、仕様は「どう」
POINT — 要件に「実装」を混ぜない

よくある失敗が要件段階で技術を決めてしまうこと。「Reactで作る」は要件ではなく仕様（実装選定）。要件は「何を満たすか」に集中し、「どう作るか」は仕様に分離する。混ぜると技術に縛られて要求が歪む。これは要求バグの一種です（Section 06）。

05

INVEST原則の深掘り

INVEST PRINCIPLE

良いユーザーストーリーの6条件INVEST。本編で名前は出しましたが、ここで1つずつの意味と、満たさないと何が困るかを深掘りします。ストーリーの品質チェックリストとして使えます。

INVEST 6条件の詳細良いストーリーの条件
頭文字意味満たさないと
Independent独立している順序に縛られ並行作業不可
Negotiable交渉の余地詳細固定で柔軟性なし
Valuable価値がある作っても誰も喜ばない
Estimable見積もれる計画が立たない
Small小さい1スプリントで終わらない
Testableテスト可能完成判定ができない
// 本シリーズで特に重要な2つ
  • V（価値）：so that が書けるか＝WHYがあるか（Section 03）
  • T（テスト可能）：受入条件が書けるか＝後のJUnitに直結
// INVEST 違反の例と修正 // ─── S（Small）違反 ─── 悪い例（大きすぎる）： US: 「メモアプリの全機能を実装する」 → 数か月かかる。見積もれない 分割後： US-001: メモを作成・保存できる US-002: メモを一覧表示できる US-003: メモを検索できる → 各1〜2週間。見積もれる ★ // ─── T（Testable）違反 ─── 悪い例（検証不能）： US: 「使っていて気持ちいいUI」 → 「気持ちいい」をどう測る？ 修正後： US: 「メモ作成は2タップ以内で開始できる」 → タップ数で検証可能 ★ // ─── I（Independent）違反 ─── 悪い例（依存しすぎ）： US-A: タグ機能（US-Bが前提） US-B: タグマスタ（US-Aが前提） → 循環依存、どちらも始められない 修正後： US-B を先に独立して完成させ、 US-A はそれを利用する形に整理 ★
POINT — Tが書けないストーリーは要注意

INVESTの中で本シリーズが最重視するのはT（Testable）。受入条件（Given/When/Then）が書けないストーリーは、完成判定もテストもできない。「気持ちいい」「使いやすい」のような曖昧表現は数値化する。受入条件が書けて初めて、要求が実装可能になるのです。

06

要求バグの4類型

REQUIREMENT DEFECTS

要求にも「バグ」があります。曖昧性・欠落・実装混入・検証不能の4類型。コードのバグと違って目に見えにくいのが厄介。先に類型を知っておけば、要求レビューで発見できます。

要求バグ4類型と対策レビューの観点
DEFECT 1
曖昧性
「サクサク動く」。人によって解釈が違う。→ 数値化（2秒以内）
DEFECT 2
欠落
「メモを作れる」だけ。→ 編集・削除・一覧も書く
DEFECT 3
実装混入
「Reactで作る」。技術が要求に混入。→ 要求と技術選定を分ける
DEFECT 4
検証不能
「気持ちいい」。測れない。→ 計測可能な指標に
// 特に「欠落」が怖い

欠落は気づきにくい。「作成」だけ書いて「削除」を忘れると、実装後に「あれ、消せない」と発覚。CRUD は4つセットで確認する癖をつけます。

// 要求バグの検出例 // ─── 曖昧性バグ ─── バグ：「検索は速く」 修正：「検索は500ms以内（1000件・P95）」 // ─── 欠落バグ ─── バグ：要求に「メモ作成」しかない 検出：CRUD チェック C(作成) ✓ R(一覧) ✗ ← 欠落！ U(編集) ✗ ← 欠落！ D(削除) ✗ ← 欠落！ 修正：4操作すべて要求に追加 // ─── 実装混入バグ ─── バグ：「MyBatisのXMLで検索する」 （要求に技術が混入） 修正：「キーワードで検索できる」 （技術は仕様段階で決める） // ─── 検証不能バグ ─── バグ：「直感的に使える」 修正：「初見ユーザーがマニュアルなしで メモ作成・検索・削除を完了できる」 // → 4類型を知っていれば // 要求レビューで先に潰せる ★★★
POINT — 「曖昧な形容詞」は要求バグのサイン

「速い」「使いやすい」「分かりやすい」「きれい」——形容詞が出たら要注意。それは曖昧性バグの可能性大。「速い → 何秒？」「使いやすい → 何タップ？」と数値に変換する。これだけで要求の品質が一段上がります。

07

機能要求と非機能要求

FUNCTIONAL VS NON-FUNCTIONAL

機能要求（何ができるか）と非機能要求（どう動くか）。初学者は機能要求ばかり考えて非機能要求を忘れがち。でも非機能要求がアーキテクチャを決める。軽視できません。

非機能要求の種類（FURPS+）忘れがちな観点
// 非機能要求のカテゴリ
観点問い本シリーズの例
性能どれだけ速い？一覧1秒・検索500ms
可用性落ちない？(v1では対象外)
信頼性データ消えない？DB永続化
使用性使いやすい？マニュアル不要
保守性直しやすい？3層・テスト有
セキュリティ安全？入力検証・SQLi対策
// なぜ非機能がアーキを決めるか

例：「検索500ms以内」という非機能要求があると：

  • LIKE検索では遅い → FULLTEXT INDEX が必要
  • これがテーブル設計（アーキ）を左右する

非機能要求 → アーキテクチャの因果は強い。

// 非機能要求が設計を決める例 // ─── 非機能要求 ─── @PERF: メモ1000件で 全文検索500ms以内 // ─── これがアーキを決める ─── if (件数 < 100) { // LIKE '%word%' で十分 SELECT * FROM memos WHERE content LIKE '%word%'; } else if (件数 > 1000 && 500ms以内) { // LIKEでは遅い！ // → FULLTEXT INDEX 必須 ALTER TABLE memos ADD FULLTEXT(content); SELECT * FROM memos WHERE MATCH(content) AGAINST('word'); } // → 非機能要求がスキーマ設計を変える // → だから要求段階で決めておく ★★★ // → 本シリーズ Vol.8 で実装 // ─── 非機能要求を忘れると ─── // LIKE で作る → リリース後に遅いと発覚 // → FULLTEXT 化 → スキーマ変更 → 全テスト // → 1:10:100 の法則（Section 01）★★★
POINT — 非機能要求は「最初に数値で」

機能要求は思いつきやすいですが、非機能要求は意識しないと抜ける。しかも後から追加すると作り直しになりやすい（性能・セキュリティは特に）。要求段階で数値とともに宣言しておけば、設計で織り込める。非機能要求こそ最初にが鉄則です。

08

変更管理 — 要求は必ず変わる

CHANGE MANAGEMENT

要求工学の5番目の活動「管理」。開発中に「やっぱりこの機能も欲しい」は必ず出ます。変更を拒否せず、影響を確認した上で受け入れる作法を学びます。これはGitHub版の Issue/PRと連動します。

変更要求への向き合い方拒否でも全受容でもなく
// 変更要求の3つの対応
BAD 1
全部拒否
「仕様は決まった」と一切受けない。→ 使えないものが完成
BAD 2
全部受容
言われるまま追加。→ スコープ崩壊（スコープクリープ）
GOOD
影響を見て判断
工数・他機能への波及を確認してから、入れるか・v2に回すか決める。
// 変更要求票（CR票）

変更は記録します。「いつ・何を・なぜ・影響は」を残す。GitHub ではIssueがこの役割を果たします。

// 変更要求票（CR票）の例 @change_request CR-001 起票日: 2026-XX-XX 対象: US-003（タグ機能） 変更内容: タグだけでなくフォルダ分けも 追加したい 理由: メモが増えるとタグだけでは 整理しきれない 影響分析: ├ 工数: +3日 ├ 既存機能: 一覧画面UIに変更 ├ DB: foldersテーブル追加必要 └ 判断: v1ではタグのみ、 フォルダはv2に回す // ─── GitHub では Issue で管理 ─── Issue #12 title: [機能要望] フォルダ分け labels: enhancement, v2 milestone: v2.0 body: ## 要望 フォルダでメモを整理したい ## 影響 - foldersテーブル追加 - 一覧UIの変更 ## 判断 v2 に回す // → 変更管理 = 要求工学の「管理」 // → GitHub版 で実践する ★★★ // → 拒否せず、記録し、判断する
POINT — スコープクリープという病

「ついでにこれも」「せっかくだからあれも」で際限なく機能が増える現象をスコープクリープと呼びます。プロジェクトが終わらなくなる最大の死因。対策は「変更は歓迎、ただし影響を見てv1かv2かを判断」。MoSCoW の Won't を守る規律が効きます。
