02

動的SQLとは

WHAT IS DYNAMIC SQL

動的SQLとは、条件によって組み立てが変わるSQL。検索フォームで「キーワードだけ」「カテゴリだけ」「両方」と入力が変わると、WHERE 句も変わる必要があります。これを実現するのが動的SQLです。

静的SQL vs 動的SQL条件で変わる
// 静的SQL（これまで）

Vol.2〜6 の SQL は固定。「全件取得」「IDで1件」など、毎回同じ形でした。

// 動的SQL（検索で必要）

検索は入力次第で WHERE 句が変わる。条件があれば足す、なければ足さない。SQL を実行時に組み立てる。

// 素朴な実装（NG）の問題

Java で if 文を使って文字列連結すると、SQLインジェクションの危険＋コードが汚くなる。だから MyBatis の動的SQL機能を使います（Section 03）。

// 静的 vs 動的 // ─── 静的SQL（固定）─── SELECT * FROM memos WHERE id = #{id} // → 形が変わらない // ─── 動的SQL（条件で変化）─── // 検索条件のパターン： 入力なし: SELECT * FROM memos キーワードのみ: SELECT * FROM memos WHERE title LIKE '%keyword%' キーワード＋カテゴリ: SELECT * FROM memos WHERE title LIKE '%keyword%' AND category_id = 1 // → 条件の有無で WHERE が伸び縮み ★★★ // ─── 素朴な文字列連結（NG）─── String sql = "SELECT * FROM memos WHERE 1=1"; if (keyword != null) { sql += " AND title LIKE '%" + keyword + "%'"; // 危険！★ } // → SQLインジェクション（Section 04） // → コードも読みにくい // → MyBatisの動的SQLを使うべき（Section 03）
POINT — 「WHERE 1=1」ハックは使わない

動的SQLの素朴な実装で「WHERE 1=1」から始めて AND を足していくテクニックがあります。動きはしますが、文字列連結＝SQLインジェクションの危険。MyBatis には専用の動的SQL機能（<where> <if>）があり、安全かつ綺麗に書けます（Section 03）。「動くけど危ない方法」より「安全な専用機能」を使うのが鉄則です。

03

MyBatisの動的SQL

MYBATIS DYNAMIC SQL

MyBatis は動的SQL専用のタグを持ちます。<if>（条件分岐）、<where>（WHERE句の自動整形）、<foreach>（リスト展開）。これで安全かつ綺麗に動的SQLが書けます。本編Vol.7 で実際に使います。

if / where / foreach動的SQLの3タグ
// 3つの主要タグ
タグ役割
<if>条件が真なら含める
<where>WHEREとAND/ORを自動整形
<foreach>リストを展開（IN句など）
// <where> の賢さ

<where> は「条件が1つもなければ WHERE を出さない」「先頭の AND/OR を消す」を自動でやってくれる。手で書くと面倒な整形を肩代わり。

// MyBatis 動的SQL（本編Vol.7で使用） // ─── if + where ─── <select id="search" resultMap="MemoFull"> SELECT * FROM memos m <where> <if test="keyword != null and keyword != ''"> AND m.title LIKE CONCAT('%', #{keyword}, '%') </if> <if test="categoryId != null"> AND m.category_id = #{categoryId} </if> </where> ORDER BY m.updated_at DESC </select> // ─── <where> が自動でやること ─── // 条件0個 → WHERE を出さない（全件） // 条件1個目の AND を消す // → 「WHERE AND ...」のミスを防ぐ ★★★ // ─── foreach（IN句・タグ検索）─── <if test="tagIds != null"> AND m.id IN ( SELECT memo_id FROM memo_tags WHERE tag_id IN <foreach item="id" collection="tagIds" open="(" separator="," close=")"> #{id} </foreach> ) </if> // → tagIds=[1,2,3] を (1,2,3) に展開 ★ // ─── 全部 #{} を使う（安全）─── // 動的SQLでも値は #{} で渡す // → SQLインジェクション対策（Section 04,05）
POINT — <where> が「WHERE AND」のミスを防ぐ

動的SQLで陥りやすいのが「WHERE AND title...」のような構文エラー（最初の条件の AND が余る）。<where> タグは先頭の AND/OR を自動で削除し、条件が0個なら WHERE 自体を出さない。手で文字列を組むと必ず踏むミスを、MyBatis が肩代わり。専用機能を使えば、定番のバグを回避できるのです。

04

SQLインジェクション

SQL INJECTION

検索はユーザー入力をSQLに使うため、最も注意すべきセキュリティ問題がSQLインジェクション。悪意ある入力でDBを操作される攻撃です。仕組みと対策を理解すれば、確実に防げます。

SQLインジェクションの仕組み入力でSQLを乗っ取る
// 攻撃の仕組み

文字列連結でSQLを組むと、入力にSQLの一部を混ぜ込める。これがインジェクション（注入）。

  • 入力欄に ' OR '1'='1 と入れる
  • SQLが WHERE x='' OR '1'='1' になる
  • 常に真→全件漏洩
// もっと危険な例
  • '; DROP TABLE memos; --
  • → テーブルごと削除
// 対策：プレースホルダ

値を「データ」として渡す（SQLの一部にしない）。MyBatis の #{} がこれ。これだけで防げる。

// SQLインジェクションと対策 // ─── 脆弱なコード（文字列連結）─── String sql = "SELECT * FROM users WHERE name = '" + 入力 + "'"; // 入力が普通なら： // "田中" → WHERE name = '田中' ✓ // 入力が攻撃なら： // "' OR '1'='1" // → WHERE name = '' OR '1'='1' // → 条件が常に真 → 全件漏洩 ★★★ // "'; DROP TABLE users; --" // → テーブル削除 ★★★（壊滅的） // ─── 安全なコード（プレースホルダ）─── // MyBatis の #{} を使う SELECT * FROM users WHERE name = #{name} // #{} は値を「パラメータ」として渡す // → 入力が「' OR '1'='1」でも // 「' OR '1'='1」という名前の検索になる // （SQLとして解釈されない）★★★ // → 安全 // ─── プレースホルダの仕組み ─── SQL本体: SELECT ... WHERE name = ? パラメータ: ["' OR '1'='1"] // → SQL構造とデータが分離 // → データはSQLとして解釈されない // → これがインジェクション対策の本質 ★★★
POINT — 「文字列連結でSQLを作らない」が絶対の鉄則

SQLインジェクション対策は実はシンプル：「ユーザー入力を文字列連結でSQLに混ぜない」。MyBatis なら #{} を使うだけ。これでSQL構造とデータが分離され、入力がSQLとして解釈されません。「自分のアプリは大丈夫」は禁物。個人開発でも、検索を作る瞬間から #{} を徹底する習慣が大事です。

05

#{} と ${} の違い

HASH VS DOLLAR

MyBatis には2つの埋め込み記法があります。#{}（安全）と ${}（危険）。この違いを知らないと、うっかりSQLインジェクションの穴を開けてしまいます。原則 #{} だけを使うのが鉄則です。

#{} と ${} の決定的な違い片方は危険
// 2つの記法
#{}${}
方式プレースホルダ文字列置換
安全性安全危険
用途値（ほぼ全部）列名・ソート順（稀）
原則これを使う極力使わない
// 結論

原則 #{} だけ。${} は「列名を動的に変える」など値ではないものを埋める稀なケースのみ、しかもユーザー入力は絶対に入れない。

// #{} と ${} の動作の違い // ─── #{} プレースホルダ（安全）─── WHERE title = #{keyword} // 実際の動作： SQL: WHERE title = ? param: [入力値] // → 入力はデータ扱い、SQLにならない ★ // ─── ${} 文字列置換（危険）─── WHERE title = '${keyword}' // 実際の動作： // keyword をそのまま文字列に埋め込む // 入力 "田中" → WHERE title = '田中' // 入力 "' OR '1'='1" → // WHERE title = '' OR '1'='1' ★★★ // → SQLインジェクション！ // ─── ${} の正当な用途（稀）─── // 列名やソート順を動的にしたいとき ORDER BY ${sortColumn} // ※ #{} は値専用なので列名には使えない // ※ ただし sortColumn にユーザー入力を // 直接渡すのは厳禁！ // → ホワイトリストで検証してから使う if (!List.of("title", "updated_at") .contains(sortColumn)) { throw new IllegalArgumentException(); } // → 許可した列名だけ通す ★ // ─── 本シリーズの方針 ─── // 検索は全部 #{} で書く // ${} は使わない（必要なら厳重に検証）
POINT — 迷ったら #{}、${} は赤信号だと思う

実務での合言葉：「#{} を使う、${} は見たら立ち止まる」。${} は文字列をそのまま埋め込むので、ユーザー入力が入ると即インジェクション。本シリーズの検索（本編Vol.7）は全て #{} で書きます。${} が必要になるのは「ソート列を動的に」など稀なケースだけで、その時もホワイトリスト検証が必須。${} は危険信号と覚えましょう。

06

N+1問題

N+1 PROBLEM

性能の代表的な落とし穴N+1問題。「1回の検索の後、各行ごとに追加クエリが走る」現象。メモ一覧で各メモのタグを取るとき、うっかり踏みやすい。本編Vol.4,5 で JOIN を使ったのは、実はこれを避けるためでした。

N+1問題の仕組みと対策クエリが激増する
// N+1問題とは

「メモ一覧（1回）＋各メモのタグ取得（N回）」＝合計 N+1 回のクエリ。メモが100件なら101回のDB往復。これが遅さの原因。

// なぜ起きるか
  • メモ一覧を取る（1回）
  • 各メモについて「このメモのタグは？」を個別に問い合わせ（N回）
// 対策：JOIN でまとめて取る

本編Vol.4,5 のJOIN ＋ resultMap（association/collection）が対策。1回のクエリで全部取る。これがN+1を防ぐ。

// N+1問題の発生と対策 // ─── N+1 が起きる実装（NG）─── // 1. メモ一覧を取る List<Memo> memos = mapper.findAll(); // → SELECT * FROM memos （1回） // 2. 各メモのタグを個別に取る for (Memo m : memos) { m.setTags( mapper.findTagsByMemoId(m.getId())); // → SELECT ... WHERE memo_id=? // （メモの数だけ＝N回）★★★ } // メモ100件なら： // 1（一覧）＋100（各タグ）＝101クエリ ★ // → DB往復が多すぎて遅い // ─── 対策：JOINで1回（本編Vol.5）─── SELECT m.*, t.id AS tag_id, t.name AS tag_name FROM memos m LEFT JOIN memo_tags mt ON m.id=mt.memo_id LEFT JOIN tags t ON mt.tag_id=t.id // → 1回のクエリで全メモ＋全タグ ★★★ // → resultMap の collection で組み立て // → これが N+1 対策だった！ // ─── 本編Vol.5 の伏線回収 ─── // Vol.5 で collection を使って // JOINで一括取得したのは、 // 実は N+1 を避けるためでもあった ★★★ // 「1回のSQLでまとめて取る」 // = 性能の基本
POINT — 本編Vol.5 の JOIN は N+1 対策でもあった

本編Vol.5 でcollection を使って JOIN で一括取得したことを覚えていますか。あれは「複数行を1オブジェクトにまとめる」だけでなく、N+1問題を避ける意味もありました。もし「メモ一覧→各メモのタグを個別取得」すると N+1。JOIN で1回にまとめるのが正解。設計（Vol.5）が性能（N+1対策）を兼ねていた——良い設計は複数の問題を同時に解きます。

インデックスの基礎

INDEX BASICS

検索を速くするインデックス。本の「索引」と同じで、探す列にあらかじめ目次を作る。ただし万能ではなく、トレードオフがある。別冊S2の「測定してから最適化」と合わせて理解します。

インデックスの効果とコスト読み速・書き遅
// インデックスとは

本の索引（目次）と同じ。「タイトルで探す」が多いなら、title に索引を作ると探すのが速くなる。

// トレードオフ（別冊S2）
効果
読み（検索）速くなる
書き（INSERT/UPDATE）遅くなる（索引更新）
容量増える
// いつ貼るか

「測定してから」（別冊S2）。最初から全列にインデックスは過剰。遅い検索が判明してから、その列に貼る。

// インデックスの考え方 // ─── 索引なし（フルスキャン）─── SELECT * FROM memos WHERE title = '会議' // → 全行を1つずつ確認（遅い） // メモ1万件なら1万回チェック ★ // ─── 索引あり ─── CREATE INDEX idx_title ON memos(title); // → titleの「目次」ができる // → 目次から一発で見つける（速い）★★★ // ─── PRIMARY KEY は自動で索引 ─── // id（主キー）は自動でインデックス // → WHERE id=? が速いのはこのため // FK も多くの場合インデックス推奨 // ─── LIKE '%word%' は索引が効かない ─── SELECT * FROM memos WHERE title LIKE '%会議%' // → 前方一致でない（先頭に%） // → 通常のインデックスが効かない ★ // → 全文検索（FULLTEXT）が必要 // → 本編Vol.8 で扱う // ─── 別冊S2の「測定してから最適化」─── // 1. まず普通に LIKE で実装（Vol.7） // 2. 遅いと分かったら FULLTEXT（Vol.8） // → 推測で最初から最適化しない ★★★
POINT — LIKE '%...%' は普通のインデックスが効かない

重要な事実：LIKE '%会議%'（前後に%）は通常のインデックスが効きません。索引は「先頭から探す」ので、途中一致には使えない。だから本編Vol.7 の LIKE 検索は小規模なら問題ないが、大規模だと遅い。その対策が本編Vol.8 の全文検索（FULLTEXT）。別冊S2の「測定してから最適化」通り、まず LIKE で作り、遅ければ FULLTEXT へ段階的に進めます。

08

クエリ設計あるある失敗

QUERY PITFALLS

クエリ設計で初心者が踏みやすい失敗を5つ。SQLインジェクション・N+1・SELECT * など。知っていれば回避できるものばかりです。

クエリ設計あるある 5選先人の轍を踏まない

① 文字列連結でSQLを作る ★★★

  • SQLインジェクションの穴
  • 対処：必ず #{} を使う（Section 04,05）

② ${} にユーザー入力を入れる ★★★

  • ${} は文字列置換＝危険
  • 対処：値は #{}、${} は使わない

③ N+1問題に気づかない ★★★

  • ループ内で個別クエリ
  • 対処：JOIN でまとめて取る（Section 06）

④ 何でも SELECT *

  • 不要な列まで取得、JOINで列名衝突
  • 対処：必要な列を明示（特にJOIN時）

⑤ 推測で早すぎる最適化

  • 遅くないのに複雑なインデックス・キャッシュ
  • 対処：測定してから（別冊S2・Section 07）
// あるある失敗の対処 // ─── ④ SELECT * の問題（JOIN時）─── NG: SELECT * FROM memos m LEFT JOIN categories c ... // → m.id と c.id が両方「id」で衝突 ★ // → resultMap でマッピングできない OK: SELECT m.id, m.title, c.id AS category_id, c.name AS category_name FROM memos m LEFT JOIN categories c ... // → 必要な列を明示、別名で衝突回避 ★ // → 本編Vol.4,5 で実践済み // ─── ⑤ 早すぎる最適化を避ける ─── // 別冊S2の判断軸： 1. まず素直に実装（LIKE検索） 2. 動かして測定 3. 遅ければ最適化（FULLTEXT/INDEX） // → 「遅くなりそう」で複雑にしない ★★★ // → メモ帳v1は数百件想定 // → LIKE で十分（Vol.7） // → 必要になったら FULLTEXT（Vol.8） // ─── 良いクエリの条件 ─── ✓ #{} で安全 ✓ JOIN で N+1 回避 ✓ 必要な列を明示 ✓ 測定に基づく最適化 ★★★
POINT — 「安全→正しさ→速さ」の順で考える

クエリ設計の優先順位は①安全（インジェクション対策）→②正しさ（正しい結果）→③速さ（性能）。まず #{} で安全に、次に正しい結果を返し、最後に遅ければ最適化。速さを最初に追って安全を疎かにするのは本末転倒。別冊S2の「測定してから最適化」と合わせ、安全と正しさを土台に、速さは必要に応じてが鉄則です。
