01

別冊S4のミッション

MISSION

別冊S4は「データ構造を理論で理解する」回。本編Vol.4 で 1対多を、Vol.5 で多対多を実装します。でも「なぜそのテーブル構成か」を理論で理解しないと、未知のデータ構造で迷います。データ設計は「後から最も変更しにくい層」。だからこそ理論が重要です。

なぜデータモデリングが最重要か後から変えにくい
// データ構造は変更コストが最大

設計フロー文書の指摘：「データ構造は後から最も変更しにくい層。保存形式を間違えると、リリース後に全機能を巻き込んで作り直し」。

  • コードは直せる、画面も直せる
  • でもテーブル構成の変更は全機能に波及
  • 既存データの移行（マイグレーション）も必要
// 別冊S4で得る視点
  • 正規化の意味と「やりすぎ」の境界
  • 3つの関連を見分けられる
  • 多対多に中間テーブルが要る理由
  • ER図を読み書きできる
// データ設計ミスの代償 // ─── 悪い設計でスタートすると… ─── // 例：タグをカンマ区切りで保存 memos テーブル tags VARCHAR: "重要,あとで,アイデア" ↑ 1カラムに複数値 // 最初は動く。でも後で… 「タグで絞り込みたい」 → LIKE '%重要%' で無理やり検索 → "重要度" も引っかかる（誤検出）★ 「タグの名前を変更したい」 → 全メモの文字列を置換… 「タグごとの件数を出したい」 → 文字列を分解して集計…地獄 // → リリース後に気づく // → 中間テーブルへの移行が必要 // → 全機能を作り直し ★★★ // → 最初に正しく設計していれば // 全部 SQL 一発で済んだ // → だからデータモデリングは最重要
POINT — データ設計は「1:10:100」が最も効く層

別冊S1の1:10:100の法則。データ設計はこれが最も極端に効く層です。要求段階でテーブル構成を正しく決めれば安い。リリース後に「カンマ区切りを中間テーブルに」と気づくと、データ移行＋全機能改修＋全テストで100倍のコスト。データ設計こそ、最初に丁寧に。

02

データモデリングとは

WHAT IS DATA MODELING

データモデリングとは、現実世界の情報をテーブルと関連で表現する設計作業。設計フロー文書の言葉では「要求に出てくる名詞がデータの候補」。メモ帳の「メモ・カテゴリ・タグ」がそれです。

名詞を見つけて関連を引く3ステップ
// モデリングの3ステップ
  1. 名詞を抽出：要求文の名詞＝エンティティ候補
  2. 属性を決める：各エンティティが持つデータ
  3. 関連を引く：エンティティ間の関係（1対多など）
// メモ帳での実践
  • 「メモを作る」→ Memo
  • 「カテゴリで分類」→ Category
  • 「タグを付ける」→ Tag

要求が自然にエンティティを導く。これが本編Vol.1 でやったこと。

// 要求 → エンティティ → 関連 // ─── ① 名詞を抽出 ─── 要求：「メモをカテゴリで分類し、 タグも付けて整理したい」 名詞：メモ・カテゴリ・タグ → 3エンティティ ★ // ─── ② 属性を決める ─── Memo: id, title, content, 日時 Category: id, name, color, sort_order Tag: id, name // ─── ③ 関連を引く ─── メモ ── 1つのカテゴリに属する → メモ:カテゴリ = 多:1 → 1対多 メモ ── 複数のタグを持つ タグ ── 複数のメモに付く → メモ:タグ = 多:多 → 多対多 ★★★ // → 関連の種類は「両側の数」で決まる // → 次の Section 03 で詳説
POINT — 「動詞」は関連、「名詞」はエンティティ

要求文を分解するコツ：名詞＝エンティティ（テーブル）、動詞＝関連や操作。「メモをカテゴリで分類する」なら、名詞「メモ・カテゴリ」がテーブル、「分類する」が両者の関連。要求の文法構造がデータ構造のヒントになります。これは要求工学（別冊S1）とデータ設計の橋渡しです。

03

3つの関連

3 RELATIONSHIPS

エンティティ間の関連は3種類。1対1・1対多・多対多。それぞれ実装方法が違う。「両側にいくつあるか」で見分けます。本編Vol.4 が 1対多、Vol.5 が多対多です。

3つの関連と実装方法両側の数で見分ける
TYPE 1
1対1
片方に1つずつ。例：ユーザーとプロフィール。同じテーブルか、FK1つ。
TYPE 2
1対多
片方が複数持つ。例：カテゴリと複数のメモ。多側にFK。本編Vol.4。
TYPE 3
多対多
両方が複数。例：メモとタグ。中間テーブル必須。本編Vol.5。
// 見分け方の質問

2つのエンティティ A と B について：

  • 「1つの A は複数の B を持つ？」
  • 「1つの B は複数の A を持つ？」

両方YES → 多対多。片方だけYES → 1対多。両方NO → 1対1。

// 3つの関連の実装 // ─── 1対多（メモ←→カテゴリ）─── // 「多」側（memos）に FK を置く memos id, title, category_id(FK) categories id, name // 1カテゴリ → 複数メモ // 1メモ → 1カテゴリ // → 多側にFK 1本 ★（Vol.4） // ─── 多対多（メモ←→タグ）─── // 中間テーブルが必要 memos tags id, title id, name \\ / memo_tags（中間テーブル） memo_id(FK), tag_id(FK) // 1メモ → 複数タグ // 1タグ → 複数メモ // → FKだけでは表現できない // → 中間テーブルが要る ★★★（Vol.5） // ─── なぜ多対多はFKで無理か ─── // memos に tag_id を置くと… // → 1メモ1タグしか持てない（NG） // tags に memo_id を置くと… // → 1タグ1メモしか付かない（NG） // → どちらも多対多を表現できない // → 中間テーブルが必然 ★★★
POINT — 1対多のFKは「多側」に置く

1対多でFKをどちらに置くかでよく迷います。答えは「多側」。メモ（多）とカテゴリ（1）なら、memos に category_id。逆に categories に memo_id を置くと「1カテゴリ1メモ」になってしまう。「多が1を指す」と覚えればOK。本編Vol.4 でこれを実践しました。

04

正規化の基礎

NORMALIZATION

正規化とは、データの重複と矛盾を防ぐテーブル設計の技法。「1つのカラムに複数値を入れない」「重複データを別テーブルに分ける」。第1〜第3正規形を、難しい用語抜きで実用的に理解します。

正規化の3段階（実用版）用語より「なぜ」
// 第1正規形（1NF）

1つのカラムに1つの値。カンマ区切りで複数値はNG。

  • NG：tags = "重要,あとで"
  • OK：1行1値に分解（中間テーブルへ）
// 第2正規形（2NF）

主キーの一部だけで決まる項目を分離。中間テーブルで特に重要。

// 第3正規形（3NF）

他の列から決まる列を分離。「カテゴリ名」をメモに直接持たず、categories に分けるのがこれ。

// 実用的な目安

「同じデータが2か所に重複していたら分ける」。これだけ意識すれば、自然と第3正規形くらいになります。

// 正規化の実例 // ─── 非正規形（NG）─── memos id│title │category_name│category_color│tags 1 │会議 │仕事 │#d63131 │重要,急ぎ 2 │買物 │仕事 │#d63131 │ ↑重複！ ↑重複！ ↑複数値！ // 問題： // ・カテゴリ色を変えたい→全メモ修正 // ・「仕事」の名前変更→全メモ修正 // ・タグで検索できない（文字列） // ─── 正規化後（OK）─── // 第3正規形：カテゴリを分離 memos: id, title, category_id(FK) categories: id, name, color // → カテゴリ色は1か所だけ ★ // → 名前変更も1か所 ★ // 第1正規形：タグを分解（中間テーブル） tags: id, name memo_tags: memo_id, tag_id // → タグで検索・集計できる ★★★ // ─── 正規化の効果 ─── 更新の一貫性：1か所直せば全部反映 検索の容易さ：SQLで絞り込める 矛盾の防止：重複がないから矛盾しない
POINT — 正規化のやりすぎ＝過剰設計

正規化は重要ですが、やりすぎると JOIN だらけで遅く・複雑に。別冊S2のYAGNI・KISSと同じく、「実用上必要な正規化」で止めるのが正解。一般に第3正規形までで十分。性能が問題になったら、あえて重複を持つ「非正規化」を検討（測定してから＝別冊S2）。正規化も判断、原理主義にならない。

05

多対多と中間テーブル

MANY-TO-MANY

本編Vol.5 の核心「多対多」を理論で先取り。中間テーブル（memo_tags）の正体、複合主キー、なぜこれが必要か。多対多は3テーブル設計の最重要にして最難関です。

中間テーブルの正体多対多を「2つの1対多」に分解
// 多対多 = 2つの1対多

多対多は直接表現できません。中間テーブルを挟んで「2つの1対多」に分解します。

  • memos ←1対多→ memo_tags
  • tags ←1対多→ memo_tags

中間テーブル memo_tags は「どのメモにどのタグが付いたか」の組み合わせを記録。

// 中間テーブルの構造
  • memo_id（FK → memos）
  • tag_id（FK → tags）
  • 複合主キー（memo_id, tag_id）＝同じ組み合わせの重複防止
// 中間テーブルの仕組み // ─── データのイメージ ─── memos memo_tags tags 1 会議 memo_id│tag_id 1 重要 2 買物 1 │ 1 2 急ぎ 3 学習 1 │ 2 3 あとで 2 │ 3 3 │ 1 3 │ 2 // 読み方： // メモ1（会議）→ タグ1(重要),2(急ぎ) // メモ2（買物）→ タグ3(あとで) // メモ3（学習）→ タグ1(重要),2(急ぎ) // → 1メモに複数タグ ★ // → 1タグが複数メモに ★ // → 多対多が表現できた ★★★ // ─── DDL（Vol.1で設計済み）─── CREATE TABLE memo_tags ( memo_id BIGINT NOT NULL, tag_id BIGINT NOT NULL, PRIMARY KEY (memo_id, tag_id), -- ↑ 複合主キー = 同じ組合せ重複防止 FOREIGN KEY (memo_id) REFERENCES memos(id) ON DELETE CASCADE, -- ↑ メモ削除→紐付けも消す FOREIGN KEY (tag_id) REFERENCES tags(id) ON DELETE CASCADE ); // ─── 削除戦略の違い（Vol.1設計）─── カテゴリ（1対多）: ON DELETE SET NULL → メモは残る、未分類になる 中間テーブル（多対多）: ON DELETE CASCADE → メモ削除で紐付けも消す（紐付けだけ） → メモ本体・タグ本体は無関係 ★★★
POINT — 中間テーブルの複合主キーが「重複」を防ぐ

中間テーブルの主キー (memo_id, tag_id) は重要。これにより「同じメモに同じタグを2回付ける」が物理的に不可能になります。別冊S3の「正しさはDBに守らせる」がここでも効く。アプリ側で重複チェックする必要がなく、DBが整合性を保証。本編Vol.6 でこの恩恵を実装で体験します。

ER図の読み書き

ER DIAGRAM

ER図（Entity-Relationship図）は、テーブルと関連を図で表現する設計図。チームでの共通言語です。記号の意味と、メモ帳アプリのER図を読み書きできるようになります。

ER図の記法とメモ帳のER図設計の共通言語
// 関連の記号（カラスの足記法）
記号意味
──┤1（ちょうど1）
──<多（0以上）
1 ──< 多1対多
多 >──< 多多対多
// ER図を描く意味
  • 全体像が一目：テーブル数・関連が見える
  • チームの共通言語：認識を揃えられる
  • 実装前の検証：紙の上で矛盾に気づける

設計フロー文書の「検証フェーズ」＝紙のスケッチで確認、がこれ。

// メモ帳アプリの ER 図
┌─────────────┐ │ categories │ │─────────────│ │ id (PK) │ │ name │ │ color │ │ sort_order │ └──────┬──────┘ │ 1 │ │ 多 (1対多) ┌──────┴──────┐ │ memos │ │─────────────│ │ id (PK) │ │ title │ │ content │ │ category_id │───FK └──────┬──────┘ │ 1 │ 多 ┌──────┴──────┐ │ memo_tags │ ← 中間テーブル │─────────────│ │ memo_id(PK) │ │ tag_id (PK) │ └──────┬──────┘ │ 多 │ 1 ┌──────┴──────┐ │ tags │ │─────────────│ │ id (PK) │ │ name │ └─────────────┘
// memos ─1対多─ categories（Vol.4） // memos ─多対多─ tags（memo_tags経由・Vol.5） // → これがメモ帳アプリの全体像 ★★★
POINT — ER図は「実装前」に描く

ER図の真価は実装前に発揮されます。紙やツールでテーブルと関連を描いてみると、矛盾や抜けに気づける（設計フロー文書の「検証」）。「あれ、タグの色はどこに持つ？」「未分類を許すなら category_id は NULL 可だな」——図にすると考えが整理される。コードを書く前の数分の投資が、大きな手戻りを防ぎます。

07

関連設計の判断

DESIGN DECISIONS

関連設計には判断が伴います。「1対多か多対多か」「FKをどちらに」「削除時どうするか」。別冊S2のトレードオフの判断軸が、ここでも活きます。

関連設計の3つの判断要求から決める
// 判断① 1対多か多対多か

要求で決まる。「メモは1カテゴリ」なら1対多、「メモは複数タグ」なら多対多。将来の変化も考慮。

// 判断② 削除時の挙動
戦略意味本シリーズ
SET NULL参照を空にカテゴリ削除
CASCADE連動削除中間テーブル
RESTRICT削除拒否—
// 判断③ 中間テーブルに属性を持たせるか

多対多の中間テーブルに「付けた日時」「並び順」などを持たせることも。本シリーズはシンプルにFKのみ（YAGNI）。

// 関連設計の判断例 // ─── 判断① カテゴリは1対多で良いか ─── 要求：「メモは1つのカテゴリに属する」 → 1対多で十分 // もし「メモを複数カテゴリに」なら？ → 多対多に変更（中間テーブル） → 要求次第で構造が変わる ★ // ─── 判断② 削除戦略（要求ベース）─── Q. カテゴリ「仕事」を消したら 「仕事」のメモはどうなる？ A1. メモも消す → CASCADE （でもメモは大事…却下） A2. 未分類にする → SET NULL ★採用 （メモは残る・本シリーズの方針） A3. 削除させない → RESTRICT （メモがある限りカテゴリ削除不可） // → 「メモは大事」という価値観が // SET NULL を選ばせる ★★★ // ─── 判断③ 中間テーブルの属性 ─── // シンプル版（本シリーズ）： memo_tags(memo_id, tag_id) // 拡張版（必要なら）： memo_tags(memo_id, tag_id, created_at, sort_order) // → 「タグを付けた順」を保持したいなら // → でも今は不要（YAGNI）★ // → 必要になったら足す（別冊S2）
POINT — 削除戦略は「価値観」を反映する

FKの ON DELETE は単なる技術選択ではなく「何を大事にするか」の表明。本シリーズがカテゴリ削除→メモは SET NULL（残す）を選ぶのは、「ユーザーのメモは絶対に失わせない」という価値観。一方中間テーブルは CASCADE（紐付けだけ消す）。「何を守り、何を消すか」を要求と価値観から決めるのが関連設計です。

08

モデリングあるある失敗

MODELING PITFALLS

データモデリングで初心者が必ず1つは踏む失敗を5つ厳選。知っていれば回避できるものばかり。特に「カンマ区切り地獄」は要注意です。

モデリングあるある 5選先人の轍を踏まない

① カンマ区切りで複数値 ★★★

  • tags = "重要,急ぎ" のように1カラムに複数
  • 検索・集計・名称変更が地獄に
  • → 中間テーブルで第1正規形に

② 同じデータを複数テーブルに重複 ★★★

  • カテゴリ名をメモにも保存
  • 名前変更時に全部直す羽目に
  • → FKで参照、名前は1か所（第3正規形）

③ 1対多のFKを「1側」に置く

  • categories に memo_id を置く
  • → 1カテゴリ1メモになる（NG）
  • → FKは「多側」に

④ 多対多をFK1本で表現しようとする

  • memos に tag_id を1つ置く
  • → 1メモ1タグしか持てない
  • → 中間テーブルが必須

⑤ 何でも1つの巨大テーブルに

  • メモもカテゴリもタグも1テーブルに詰める
  • → NULLだらけ・意味不明
  • → エンティティごとに分ける
// ①カンマ区切り地獄の詳細 // ─── 一見動くが破綻する ─── memos tags VARCHAR: "重要,急ぎ,あとで" // 破綻ポイント： // 1. タグで絞り込めない SELECT * FROM memos WHERE tags LIKE '%重要%' // → "重要度" も引っかかる（誤検出）★ // → "最重要" も引っかかる // 2. タグ名を変更できない // "重要"→"高優先" にしたい // → 全メモの文字列を置換… // → "重要度" まで壊す危険 ★ // 3. タグ別の件数が出せない // → 文字列を分解して集計…悪夢 // ─── 正しい設計（中間テーブル）─── SELECT m.* FROM memos m JOIN memo_tags mt ON m.id=mt.memo_id JOIN tags t ON mt.tag_id=t.id WHERE t.name = '重要' // → 完全一致で正確 ★★★ // → 名称変更は tags の1行だけ // → 件数も GROUP BY 一発
POINT — 「カンマ区切り」を見たら設計を疑う

1カラムにカンマ区切りで複数値を入れたくなったら、それは多対多を中間テーブルで表現すべきサイン。「最初は楽」ですが、検索・集計・更新で必ず破綻します（別冊S1の1:10:100）。本編Vol.5 で正しく中間テーブルを作るのは、この地獄を避けるため。「複数値はカラムでなく行で」が鉄則です。
