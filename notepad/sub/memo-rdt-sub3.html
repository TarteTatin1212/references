01

別冊S3のミッション

MISSION

別冊S3は「テストを戦略として理解する」回。本編で「単体は多く、結合は要所で」を実践しました。でも「なぜその配分か」「どんなケースを書けば十分か」を理論で理解すると、無駄なく効果的なテストが書けます。テストはセンスではなく戦略です。

テストを学ぶと何が変わるか安心して変更できる
// テストの根本的な価値

テストフロー文書の核心：「テストは『動いた』を確認するためではなく、『これからも安心して変更できる』状態を作るために書く」。

  • テストの価値は書いた瞬間ではなく、変更時に出る
  • テストがあればリファクタや機能追加が怖くない
  • 「壊れたらテストが教えてくれる」安心感
// 別冊S3のゴール
  • テストピラミッドの「なぜ」を説明できる
  • 境界値・異常系の選び方が分かる
  • モックと実DBを適切に使い分けられる
  • 「ダメなテスト」を見抜ける
// テストがある世界・ない世界 // ─── テストがない世界 ─── コードを変更 ↓ 「動くかな…」と手動で全画面確認 ↓ 見落とし → 本番でバグ発覚 ↓ 「触るのが怖い」コードになる ↓ 誰も直せない → 塩漬け ★ // ─── テストがある世界 ─── コードを変更 ↓ テスト実行（数秒） ↓ 緑 → 安心してコミット 赤 → 壊れた箇所が即わかる ↓ 「いつでも直せる」コードになる ★★★ // → テストは「変更する勇気」をくれる // → これが投資としての価値
POINT — テストは「コスト」ではなく「投資」

「テストを書く時間がもったいない」は初学者の典型的な誤解。テストは将来の自分を救う投資。書く時は少し手間でも、変更・リファクタ・バグ修正で何倍も回収できます。特に「半年後に機能追加」のとき、テストの有無で天と地の差が出ます。

02

テストは何のために書くか

WHY TEST

テストの目的は「正しさの確認」だけではありません。変更の安全網、仕様の記録、設計の改善。テストには複数の役割があります。これを理解すると、テストの書き方が変わります。

テストの4つの役割確認だけじゃない
ROLE 1
正しさの確認
意図通り動くかを確認。最も基本的な役割。
ROLE 2
変更の安全網
変更で壊れたら教えてくれる。リファクタの命綱。
ROLE 3
仕様の記録
テストを読めば仕様が分かる。実行可能なドキュメント。
ROLE 4
設計の改善
テストしにくい＝設計が悪いサイン。設計を磨く鏡。
// RDT版での意味

本シリーズでは受入条件＝テスト。テストが「要求の記録（ROLE 3）」を兼ねる。テスト名 空タイトルはエラーになる() を読めば、要求が分かります。

// テスト名が「仕様」になる // ─── テストを読むと仕様が分かる ─── class MemoValidationTest { 空タイトルはエラーになる() // 仕様 _200文字は登録できる() // 仕様 _201文字はエラーになる() // 仕様 } // → テスト名を並べると「タイトルの仕様」 // が浮かび上がる ★★★ // → 実行可能なドキュメント // ─── バグ修正は再現テストから ─── // テストフロー文書の鉄則： // 「バグ修正は必ず再現テストを // 先に書いてから直す」 1. バグを再現するテストを書く（赤） 2. バグを修正する（緑） 3. このテストが再発を防ぐ ★★★ // → 同じバグが二度と出ない // → これがテストの「変更の安全網」
POINT — テスト名は日本語で「何を検証するか」

テストフロー文書の指針：テスト名は「何を検証しているか」が分かる日本語で。test1() ではなく 空タイトルはエラーになる()。こうするとテスト一覧が仕様書になる。失敗時も「何が壊れたか」が名前で即わかる。Javaのテストメソッド名に日本語が使えるのは大きな利点です。

03

テストピラミッド

TEST PYRAMID

テスト戦略の最重要概念テストピラミッド。下（単体）を厚く、上（E2E）を薄く。なぜこの形が良いのか、理論で理解します。本編で実践した配分の根拠です。

ピラミッドの形と理由速さ・安定性・コスト
▲ E2E (10%)
遅い・脆い・高コスト
─────────────
▰▰ 結合 (20%)
中速・DB絡み
───────────────────
▰▰▰▰ 単体 (70%)
速い・安定・低コスト
// なぜ下を厚くするか
層速度安定性
単体速い(ms)高い
結合中(秒)中
E2E遅い(分)脆い

単体は速くて安定なので、たくさん書ける。E2Eは遅くて脆いので、最小限に。

// 逆ピラミッド（アンチパターン） // ─── やってはいけない形 ─── ▰▰▰▰ E2E（多い）← 全部E2Eで確認 ───────── ▰▰ 結合 ─── ▲ 単体（少ない） // 問題： // - テストが遅い（毎回ブラウザ起動） // - 脆い（画面変更で大量に壊れる） // - 失敗原因が特定しにくい // → 「アイスクリームコーン」 // と呼ばれるアンチパターン ★ // ─── 本シリーズの配分（Spring版）─── 単体（70%）： - バリデーション（@NotBlank/@Size） - Service のロジック（Mockito） → JUnit + Mockito + AssertJ 結合（20%）： - Mapper の SQL（UNIQUE/並び順/JOIN） → @MybatisTest + H2 E2E相当（10%）： - 主要フロー（保存→一覧） → MockMvc // → 下を厚く、上を薄く ★★★ // → 本編Vol.2,3 でこの通り実践した
POINT — E2Eで全部やろうとしない

テストフロー文書の警告：「E2Eで全バリデーションを網羅しようとする（遅く・脆くなる）」のは避ける。バリデーションは単体で網羅し、E2E（MockMvc）は主要フロー1〜2本だけ。「空タイトル」「201文字」などのエッジケースを全部 MockMvc で書くと、遅くて壊れやすいテスト群になります。適材適所が鉄則です。

04

AAAパターン

ARRANGE-ACT-ASSERT

1つのテストは「準備→実行→検証」の3ブロックで書く。これがAAA（Arrange-Act-Assert）。受入条件のGiven/When/Thenと完全対応します。読みやすく、壊れにくいテストの基本構造です。

AAA と Given/When/Then要求とテストの対応
// 3ブロックの対応
受入条件テスト(AAA)内容
GivenArrange前提・準備
WhenAct実行する操作
ThenAssert期待結果の検証
// AAA を守る効果
  • 読みやすい：3ブロックで構造が一目
  • 1テスト1検証：Assert が肥大化しない
  • 要求と対応：受入条件がそのまま写せる
// AAA の実例（コメントで明示） @Test void 空タイトルはエラーになる() { // ── Arrange（準備）= Given ── Memo memo = new Memo(); memo.setTitle(""); // ── Act（実行）= When ── var errors = validator.validate(memo); // ── Assert（検証）= Then ── assertThat(errors).isNotEmpty(); } // → 受入条件@AC2が3ブロックに ★★★ // ─── 1テスト1検証が原則 ─── 悪い例： @Test void 全部テスト() { // 空もテスト、201文字もテスト、 // 重複もテスト… → 何が壊れたか不明 } 良い例： @Test void 空はエラー() { ... } @Test void _201文字はエラー() { ... } // → 1テスト1検証 // → 失敗時に原因が即わかる ★★★
POINT — Assert の肥大化に注意

1つのテストで何でもかんでも検証（Assert）すると、何が壊れたか分からなくなります。1テスト＝1つの振る舞いの検証が理想。「空もチェック、文字数もチェック、重複もチェック」を1メソッドに詰めない。テストを分ける勇気が、失敗原因の特定を楽にします。

05

テスト設計技法

TEST DESIGN TECHNIQUES

無限の入力を全部は試せません。代表値・境界値・異常系を賢く選ぶ。これがテスト設計技法。本編で「200文字OK・201文字NG」を選んだ根拠がここにあります。

3つの代表的技法どんなケースを書くか
// ① 同値分割

入力を「同じ結果になるグループ」に分け、各グループから1つ選ぶ。1〜200文字は全部「OK」グループ→代表で1つ。

// ② 境界値分析

バグは境界に潜む。「200文字OK、201文字NG」のように境界の両側をテスト。> と >= の書き間違いを検出。

// ③ 異常系

空・null・想定外の入力。意識しないと抜ける。テストフロー文書の目安：1機能につき正常系1〜2＋異常系3〜5。

// タイトル（1〜200文字必須）のテスト設計 // ─── 同値分割 + 境界値分析 ─── 入力 期待 技法 ───────────────────────── 空文字 エラー 異常系・境界 1文字 OK 下限境界 100文字 OK 代表値（同値） 200文字 OK 上限境界 ★ 201文字 エラー 上限境界の外 ★ null エラー 異常系 ───────────────────────── // → 200と201の両方を書くのが肝 ★★★ // → if (len > 200) と if (len >= 200) // の書き間違いを検出できる // ─── 境界値が重要な理由 ─── // バグは「ちょうど境目」に潜む @Size(max = 200) // もし max = 199 と書き間違えたら？ // → 200文字テストが落ちて気づける ★★★ // → 100文字だけテストしても気づけない // ─── 異常系は意識して書く ─── // 正常系（OK系）は思いつきやすい // 異常系（空・null・超過）は // 意識しないと抜ける // → 「正常1〜2 + 異常3〜5」を目安に ★
POINT — 境界値「ちょうど」と「±1」を狙う

境界値テストのコツは「境界そのもの」と「境界の外側1つ」を両方書くこと。max=200 なら200（OK）と201（NG）。これで >/>=/</<= のオフバイワンエラー（1ずれバグ）を検出できます。「真ん中の値」だけのテストは境界バグを見逃すのです。

06

モック vs 実DB

MOCK VS REAL DB

本編Vol.2 はモック（Mockito）、Vol.3 は実DB（H2）を使いました。どう使い分けるか。これはテスト戦略の重要な判断。「何を検証したいか」で決まります。

モックと実DBの使い分け何を検証したいか
// 使い分けの基準
検証したいこと使うもの
Serviceのロジックモック
「Mapperが呼ばれたか」モック
SQLが正しいか実DB(H2)
UNIQUE制約の動作実DB
JOINの結果実DB
並び順実DB
// モックの本質と限界
  • モックは「ニセモノ」：戻り値を自分で決める
  • 「呼ばれたか」「何が渡されたか」は検証できる
  • でも「DBが本当にどう動くか」は分からない
  • → DBの挙動（制約・JOIN）は実DBでしか検証できない
// モックと実DBの役割の違い // ─── モック（Mockito）でできること ─── @Mock MemoMapper mapper; @InjectMocks MemoService service; @Test void 日時がセットされる() { Memo m = new Memo(); service.create(m); // 「createdAtがセットされたか」= Serviceのロジック assertThat(m.getCreatedAt()).isNotNull(); // 「insertが呼ばれたか」= 呼び出しの検証 verify(mapper).insert(m); } // → DBは登場しない。ロジックだけ検証 ★ // → 速い・どこでも動く // ─── モックでは検証できないこと ─── // 「UNIQUE制約で重複が弾かれるか」 // → モックは戻り値を自分で決めるので // 「重複時に例外」を再現できない ★ // ─── 実DB（@MybatisTest+H2）の出番 ─── @MybatisTest class CategoryMapperTest { @Autowired CategoryMapper mapper; @Test void 重複は登録できない() { mapper.insert(cat("仕事")); assertThatThrownBy(() -> mapper.insert(cat("仕事"))) .isInstanceOf( DataIntegrityViolationException.class); } // → 本物のDBがUNIQUE制約を発動 ★★★ // → これはモックでは不可能 }
H2 vs Testcontainers（実MySQL）速さと忠実さ
観点H2(インメモリ)Testcontainers
速度速い遅い(起動コスト)
忠実さMySQL方言は不完全本物のMySQL
環境不要Docker必要
向き日常の結合テスト本番に近い検証

本シリーズは H2 が基本。MySQL固有機能（FULLTEXT等）の検証が必要な巻ではTestcontainersを検討します（Vol.8の全文検索など）。速さと忠実さのトレードオフ（別冊S2の判断軸）。

// トレードオフの判断（別冊S2と連動） // ─── H2 を選ぶ理由（日常）─── if (検証したいのが) { 基本的なCRUD・UNIQUE・並び順 → H2 で十分（速い）★ } // ─── Testcontainers が要る場面 ─── if (検証したいのが) { MySQL固有機能： - FULLTEXT INDEX（全文検索） - MySQL特有の関数 → H2では再現できない → Testcontainers で実MySQL ★★★ } // ─── 「迷ったら戻せる選択肢」─── // まず H2 で始める（速い・手軽） // 不足を感じたら Testcontainers へ // → 段階的に判断（YAGNI）★ // → 別冊S2の設計判断と同じ思想
POINT — DI があるからモックも実DBも選べる

本編Vol.2 で @RequiredArgsConstructor によるDIを入れたおかげで、テストでMapper をモックに差し替えられました。別冊S2の「低結合がテスト容易性を生む」がここで効いています。設計（DI）→テスト容易性→モックと実DBの選択肢。すべて繋がっています。

07

テストの独立性

TEST INDEPENDENCE

テストフロー文書の重要原則：各テストは互いに独立させる。前のテストの状態に依存しない。順序を変えても、1つだけ実行しても通る。これが守れないと「たまに落ちるテスト」になります。

独立性が崩れると何が起きるかフレーキーテストの正体
// 独立性の鉄則
  • テスト間でデータを共有しない
  • 実行順序に依存しない
  • 1つだけ実行しても通る
  • 各テスト後に状態をリセット
// 独立性が崩れる典型
  • テストAがDBに登録 → テストBがそれに依存
  • テストの実行順が変わると突然落ちる
  • 「たまに落ちる」フレーキー（不安定）テストに

Spring では @Transactional で自動ロールバックが便利。

// 独立性を保つ仕組み // ─── @Transactional で自動ロールバック ─── @MybatisTest // デフォルトでロールバックされる class CategoryMapperTest { @Test void テストA_登録() { mapper.insert(cat("仕事")); // このテスト終了後、自動でロールバック // → DBは空に戻る ★★★ } @Test void テストB_別の登録() { // テストAの「仕事」は残っていない // → 各テストがクリーンな状態から開始 ★ mapper.insert(cat("仕事")); // 重複しない } } // ─── @BeforeEach でリセットする手も ─── @BeforeEach void setup() { // 各テスト前にデータをクリア mapper.deleteAll(); } // ─── 独立性が崩れた例（NG）─── static Long savedId; // テスト間で共有 ← NG @Test void テスト1() { savedId = service.create(...); // 保存 } @Test void テスト2() { service.findById(savedId); // テスト1に依存！ // → テスト2だけ実行すると savedId=null // → 落ちる ★★★ }
POINT — sleep() は使わない、条件待ちを使う

テストフロー文書の禁止事項：sleep()（固定待ち）は使わない。「3秒待てば終わってるだろう」は環境次第で破綻します（速いマシンでは無駄、遅いマシンでは足りない）。「条件が満たされるまで待つ」仕組みを使う。本シリーズの MockMvc は同期的なので待ち不要ですが、非同期処理を扱う時は覚えておきましょう。

08

やってはいけないテスト

TEST ANTI-PATTERNS

テストにもアンチパターンがあります。検証なしテスト、カバレッジ至上主義、脆いテスト。これらを知っておけば「テストを書いたのに意味がない」を避けられます。

テストアンチパターン 5選避けるべき書き方

① 検証のない「実行だけ」テスト ★★★

  • メソッドを呼ぶだけで assert が無い
  • カバレッジは上がるが何も保証しない
  • → 必ず Assert を書く

② カバレッジ100%を目標化 ★★★

  • 数値のために無意味なテストを量産
  • → 目安はライン80%・ブランチ70%で十分
  • カバレッジは手段であって目的ではない

③ E2Eで全網羅（アイスクリームコーン）

  • 遅く・脆くなる
  • → 細かいケースは単体で（Section 03）

④ テスト間でデータ共有（順序依存）

  • 「たまに落ちる」フレーキーテストの原因
  • → 各テストを独立させる（Section 07）

⑤ sleep() で固定待ち

  • 環境次第で破綻、無駄に遅い
  • → 条件待ちを使う
// アンチパターンの実例 // ─── ① 検証なしテスト（無意味）─── @Test void createのテスト() { service.create(new Memo()); // assert が無い！ // → 例外が出なければ「成功」扱い // → 何も保証していない ★ } 良い例： @Test void 作成で日時がセットされる() { Memo m = new Memo(); service.create(m); assertThat(m.getCreatedAt()) .isNotNull(); // ← 検証あり ★ } // ─── ② カバレッジ至上主義の罠 ─── // 「100%にしろ」と言われ… @Test void getterのテスト() { Memo m = new Memo(); m.setTitle("a"); assertThat(m.getTitle()) .isEqualTo("a"); // → Lombokのgetterをテスト… // → カバレッジは上がるが無意味 ★ } // 目安：ライン80% / ブランチ70% // → 「意味のあるテスト」を優先 // ─── 良いテストの条件 ─── ✓ 検証（Assert）がある ✓ 1テスト1検証 ✓ 独立している ✓ 名前で意図が分かる ✓ 意味のあるケースを選んでいる ★★★
POINT — カバレッジは「目的」ではなく「目安」

カバレッジ（テストが通ったコードの割合）は便利な指標ですが、100%を目標にすると本末転倒。Lombokのgetterや単純な委譲をテストしても意味は薄い。「重要なロジック・境界・異常系がテストされているか」が本質。ライン80%・ブランチ70%で十分というのが現実的な目安です。数より質。
