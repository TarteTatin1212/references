02

例外とは何か

WHAT IS EXCEPTION

例外（Exception）とは、「正常な処理を続けられない事態」を伝える仕組み。エラーを戻り値で返すのではなく、処理を中断して呼び出し元に投げ上げる。これにより、エラー処理を集約できます。

例外 vs 戻り値でのエラーなぜ例外が良いか
// 戻り値でエラーを返す方式の問題

古い方式：エラーを戻り値（-1, null, false）で表す。問題：

  • 毎回戻り値チェックが必要（忘れると素通り）
  • 正常な戻り値とエラー値が混ざる
  • エラー処理があちこちに散らばる
// 例外方式の利点
  • エラー時は自動で処理が中断（チェック忘れがない）
  • 正常な値とエラーが分離
  • エラー処理を1か所に集約できる（Section 04,06）
  • 「何が起きたか」を型で表現（例外クラス）
// 戻り値方式 vs 例外方式 // ─── 戻り値でエラー（古い・NG）─── Memo memo = mapper.findById(id); if (memo == null) { // 毎回チェック必須 // エラー処理…忘れたら NPE ★ return "error"; } // → チェック忘れで素通り → バグ // → エラー処理が散らばる // ─── 例外方式（推奨）─── Memo memo = service.findById(id); // もし見つからなければ Service が例外を投げる // → ここに到達した時点でmemoは必ず存在 // → チェック不要、コードが素直 ★★★ // Service 内： public Memo findById(Long id) { Memo memo = mapper.findById(id); if (memo == null) { throw new MemoNotFoundException(id); // → 例外を投げる ★ } return memo; } // ─── 例外の流れ ─── throw（投げる） ↓ 処理が中断され、呼び出し元へ catch（捕まえる）or 上へ伝播 ↓ どこかで処理される // → エラー処理を1か所に集約できる ★★★
POINT — 例外は「正常系のコードを汚さない」ための仕組み

例外の最大の価値は「正常系のコードがエラーチェックで埋もれない」こと。戻り値方式だと if (result == null) が随所に挟まり、本来やりたい処理が見えなくなる。例外なら正常系をまっすぐ書き、エラーは投げて集約処理に任せる。「うまくいく道を素直に書く」のが例外の思想。本シリーズで既に DataIntegrityViolationException 等に触れてきました。

03

チェック例外と非チェック例外

CHECKED VS UNCHECKED

Java の例外は2種類。チェック例外（コンパイラが処理を強制）と非チェック例外（RuntimeException）（強制しない）。別冊S5のトランザクションでも触れた重要な区別。使い分けの方針を理解します。

2種類の例外と使い分けRuntimeException 中心が今風
チェック例外非チェック例外
親ExceptionRuntimeException
処理強制（try/throws必須）任意
例IOExceptionNullPointer等
用途回復可能な異常プログラムの誤り・回復困難
// 現代の主流

Spring など現代のフレームワークは非チェック例外（RuntimeException）中心。理由：チェック例外はthrows の伝播でコードが汚れるから。Spring の例外（DataAccessException等）は全て RuntimeException。

// 別冊S5との接続

@Transactional のロールバックはデフォルトで RuntimeException のみ（別冊S5）。これも「RuntimeException 中心」の設計思想の表れ。

// チェック例外と非チェック例外 // ─── チェック例外（処理を強制）─── public void read() throws IOException { // IOException は Exception の子 // → 呼び出し側は try/catch か // throws が必須（コンパイラが強制）★ } // 呼び出し側： try { read(); } catch (IOException e) { // 処理が強制される } // ─── 非チェック例外（任意）─── public void process() { throw new IllegalArgumentException(); // RuntimeException の子 // → try/catch も throws も不要 ★ } // ─── 本シリーズの方針 ─── // 独自例外は RuntimeException を継承 public class MemoNotFoundException extends RuntimeException { // ← 非チェック public MemoNotFoundException(Long id) { super("メモが見つかりません: " + id); } } // → throws を書かなくていい // → コードが汚れない ★★★ // → Spring の流儀に合う // ─── 別冊S5 の復習 ─── // @Transactional はデフォルトで // RuntimeException のみロールバック // → 独自例外もRuntimeExceptionなら // 自動でロールバックされる ★
POINT — 独自例外は RuntimeException を継承する

アプリ独自の例外（MemoNotFoundException など）を作るときは、RuntimeException を継承するのが現代の主流。理由：①throws の伝播でコードが汚れない、②Spring の例外と統一感、③@Transactional が自動でロールバック（別冊S5）。チェック例外は「回復可能で、呼び出し側に対処を強制したい」稀なケースのみ。迷ったら RuntimeExceptionです。

04

どこで捕まえるか

WHERE TO CATCH

例外設計で最も重要な判断「どこで catch するか」。原則は「投げるのは発生箇所、捕まえるのは対処できる場所」。やみくもに try-catch を散らすのは逆効果。集約して捕まえるのが定石です。

例外を捕まえる場所の設計層と責務で決める
// 原則：投げる場所と捕まえる場所
  • 投げる：異常を検知した場所（Service/Mapper）
  • 捕まえる：対処できる場所（多くは最上位＝Controller層の外側）
// 3層での流れ（別冊S2）
層例外の扱い
MapperDB例外が発生
Service業務的な例外を投げる（基本catchしない）
Controller基本素通り
集約ハンドラここで一括catch
// 集約の利点

各所で try-catch せず、1か所（集約ハンドラ）で捕まえる。エラー処理が散らばらない（Section 06）。

// 例外を捕まえる場所 // ─── NG：あちこちで try-catch ─── @GetMapping("/{id}") public String show(@PathVariable Long id) { try { Memo m = service.findById(id); ... } catch (Exception e) { // 全メソッドで同じcatch…重複 ★ } } // → 全Controllerメソッドにtry-catch // → コードが重複・汚れる // ─── OK：集約して捕まえる ─── // Controllerは素直に書く @GetMapping("/{id}") public String show(@PathVariable Long id, Model model) { model.addAttribute("memo", service.findById(id)); // 例外は投げっぱなしでOK return "memos/show"; } // → try-catch なし、素直 ★ // 集約ハンドラで一括catch（Section 06） @ExceptionHandler(MemoNotFoundException.class) public String handleNotFound(...) { // 全Controllerの NotFound をここで処理 ★★★ return "error/404"; } // ─── 捕まえる場所の判断基準 ─── // 「ここで対処できるか?」 // できない → 投げて上へ // できる → catch して対処 // → 多くは「最上位で集約」が答え ★★★
POINT — 「とりあえず try-catch」は最悪のパターン

初学者がやりがちな「不安だからとりあえず全部 try-catch」は最悪。①コードが try-catch だらけで読めない、②catch して握りつぶすとエラーが消える（別冊S5の罠）、③エラー処理が散らばる。正しくは「対処できない例外は投げて、最上位で集約処理」。本シリーズは Spring の @ExceptionHandler / @ControllerAdvice で集約します（Section 06）。catch は「対処できるときだけ」。

05

ユーザーに何を見せるか

WHAT TO SHOW USERS

例外が起きたとき、ユーザーに見せるものと開発者が記録するものを分けます。技術的な詳細（スタックトレース）をユーザーに見せてはいけない。セキュリティとUXの両面で重要です。

ユーザー向け表示 vs ログ記録見せるものを分ける
// ユーザーに見せてはいけないもの
  • スタックトレース（クラス名・行番号）
  • SQL文・テーブル構造
  • 内部のファイルパス
  • → 攻撃者へのヒント＋ユーザーには無意味
// ユーザーに見せるべきもの
  • 分かりやすいメッセージ「メモが見つかりません」
  • 次の行動「一覧に戻る」リンク
  • 必要なら問い合わせ番号（ログと紐付け）
// 開発者向け（ログ）

技術的詳細はログに記録（Section 07）。ユーザーには見せず、開発者だけが見る。

// ユーザー表示とログの分離 // ─── NG：例外をそのまま露出 ───
HTTP 500 Internal Server Error java.lang.NullPointerException at com.example.memo.service .MemoService.findById(MemoService.java:42) at com.example.memo.controller ... SELECT * FROM memos WHERE id=999
// → ユーザーは意味不明 ★ // → 攻撃者にDB構造のヒント ★★★ // → 「壊れてる」印象 // ─── OK：親切なエラー画面 ───
😢 メモが見つかりませんでした お探しのメモは削除されたか、 URLが間違っている可能性があります。 [ メモ一覧に戻る ]
// → 何が起きたか分かる ★ // → 次の行動が示される ★ // → 技術詳細は隠す ★★★ // ─── 裏ではログに記録 ─── log.warn("メモ未検出: id={}", id); // → 開発者はログで原因を追える // → ユーザーには見せない // ─── 分離の原則 ─── ユーザー向け: 分かりやすく・行動を示す 開発者向け（ログ）: 技術的詳細・原因究明用 // → 「見せる相手」で内容を変える ★★★
POINT — スタックトレースの露出はセキュリティリスク

エラー画面にスタックトレースやSQL文が出るのは、単に「見栄えが悪い」だけでなくセキュリティリスク。攻撃者に「使っている技術・テーブル構造・脆弱性のヒント」を与えてしまう。本番環境では必ず技術的詳細を隠し、ユーザーには親切なメッセージだけ。詳細はログに記録（Section 07）。これは別冊S6のSQLインジェクション対策と並ぶ、基本のセキュリティ意識です。

06

Springの例外ハンドリング

SPRING EXCEPTION HANDLING

Spring は例外を集約処理する仕組みを持ちます。@ExceptionHandler（コントローラ単位）と @ControllerAdvice（全体）。これで「例外→親切なエラー画面」を1か所で実現。本編Vol.12 で実装します。

@ControllerAdvice で集約全体の例外を1か所で
// 2つの仕組み
  • @ExceptionHandler：特定の例外を処理するメソッド
  • @ControllerAdvice：全コントローラ横断で例外を集約
// 集約の流れ
  1. Service が例外を投げる
  2. Controller は素通り（catchしない）
  3. @ControllerAdvice が捕まえる
  4. 例外の種類に応じたエラー画面を返す
// 例外と画面の対応
  • NotFound → 404 画面
  • その他 → 500 画面（詳細は隠す）
// @ControllerAdvice（本編Vol.12で実装） import org.springframework.web.bind .annotation.*; @ControllerAdvice // 全Controller横断 ★★★ public class GlobalExceptionHandler { // メモ未検出 → 404画面 @ExceptionHandler( MemoNotFoundException.class) public String handleNotFound( MemoNotFoundException e, Model model) { log.warn(e.getMessage()); // ログに記録 model.addAttribute("message", "メモが見つかりませんでした"); return "error/404"; // 親切な画面 } // 重複エラー → 専用メッセージ @ExceptionHandler( DataIntegrityViolationException.class) public String handleDuplicate( Model model) { model.addAttribute("message", "既に同じ名前が存在します"); return "error/conflict"; } // 想定外 → 500画面（詳細隠す） @ExceptionHandler(Exception.class) public String handleAll( Exception e, Model model) { log.error("想定外エラー", e); // 詳細はログ model.addAttribute("message", "エラーが発生しました"); // ユーザーには簡潔に return "error/500"; } } // ─── 効果 ─── // 全Controllerの例外がここに集約 ★★★ // → Controller は try-catch 不要 // → エラー処理が1か所に // → ログ記録とユーザー表示を分離 // → 別冊S2の「関心の分離」の実践
POINT — @ControllerAdvice は「例外処理の関心を分離」する

@ControllerAdvice の本質は「エラー処理という関心事を1か所に集約」すること（別冊S2の関心の分離）。各 Controller は正常系だけを書き、エラー処理は @ControllerAdvice に任せる。これで「Controller は本来の仕事に集中」「エラー処理は統一」が両立。例外を投げる側（Service）と処理する側（Advice）が分離され、それぞれが単純になります。本編Vol.12 の中心的な仕組みです。

07

ログとの付き合い方

LOGGING

例外を捕まえたらログに記録します。ログは開発者が後で原因を追うための記録。ログレベルの使い分けと、「何を記録し、何を記録しないか」を理解します。

ログレベルと記録の指針開発者向けの記録
// ログレベル（重要度順）
レベル用途
ERROR想定外・要対応の異常
WARN想定内の異常（未検出等）
INFO通常の動作記録
DEBUG開発時の詳細
// 記録の指針
  • 例外はスタックトレース付きで記録（原因究明用）
  • パスワード・個人情報は記録しない（漏洩リスク）
  • WARN/ERROR を使い分け（想定内/外）
// ログの使い分け import lombok.extern.slf4j.Slf4j; @Slf4j // Lombokでloggerを自動生成 @Service public class MemoService { public Memo findById(Long id) { Memo m = mapper.findById(id); if (m == null) { // 想定内の異常 → WARN log.warn("メモ未検出: id={}", id); throw new MemoNotFoundException(id); } return m; } } // ─── ハンドラでのログ ─── @ExceptionHandler(Exception.class) public String handleAll(Exception e) { // 想定外 → ERROR（スタックトレース付き） log.error("想定外エラー発生", e); // → 第2引数に例外を渡すと // スタックトレースも記録される ★ return "error/500"; } // ─── レベルの使い分け ─── log.error(...) // 想定外・対応必要（DB障害等） log.warn(...) // 想定内の異常（未検出・重複） log.info(...) // 通常動作（起動・重要処理） log.debug(...) // 開発時のみ（本番は出さない） // ─── 記録してはいけないもの ─── // ✗ パスワード // ✗ 個人情報（不必要に） // ✗ クレジットカード番号等 // → ログ漏洩時のリスク ★★★ // → 「ログに何を残すか」も設計
POINT — ログは「未来の自分のデバッグ」のため

ログの目的は「本番で問題が起きたとき、原因を追えること」。エラーが起きても、ログがなければ「何が起きたか分からない」。逆に適切なログがあれば「いつ・どこで・何が」が追える。ただしパスワードや個人情報は記録しない（漏洩リスク）。「原因究明に必要な情報だけを、機密は除いて記録」が鉄則。GitHub版Vol.5 のドキュメントと同じく、未来への記録です。

08

例外設計あるある失敗

EXCEPTION PITFALLS

例外処理で初心者が踏みやすい失敗を5つ。握りつぶし・とりあえずcatch・詳細露出など。別冊S5・S7の知識で確実に回避できます。

例外設計あるある 5選先人の轍を踏まない

① 例外の握りつぶし ★★★

  • catch して何もしない→エラーが消える（別冊S5）
  • 対処：必ず記録or再スロー。空catchは禁止

② スタックトレースをユーザーに露出 ★★★

  • 500エラー画面に技術詳細
  • 対処：詳細はログ、ユーザーには簡潔なメッセージ（Section 05）

③ とりあえず全部 try-catch

  • コードが try-catch だらけ
  • 対処：集約ハンドラで一括処理（Section 04,06）

④ catch (Exception e) で何でも捕まえる

  • 想定外の例外まで握って隠す
  • 対処：具体的な例外型でcatch、想定外は上へ

⑤ 例外メッセージが不親切

  • 「エラー」だけでは原因不明
  • 対処：「何が・なぜ」を含める（id等の文脈）
// あるある失敗の対処 // ─── ① 握りつぶし（最悪）─── NG: try { service.save(memo); } catch (Exception e) { // 何もしない ★★★ } // → エラーが闇に消える // → 「保存できてないのに成功扱い」 // → 別冊S5：ロールバックもされない OK: // catch せず投げる（集約処理に任せる） service.save(memo); // または記録して再スロー // ─── ⑤ 不親切な例外メッセージ ─── NG: throw new RuntimeException("エラー"); // → 何のエラーか不明 ★ OK: throw new MemoNotFoundException( "メモが見つかりません: id=" + id); // → 「何が・どのid」が分かる ★★★ // → ログを見れば原因究明できる // ─── 良い例外設計の条件 ─── ✓ 握りつぶさない（記録or投げる） ✓ ユーザーには簡潔・行動を示す ✓ 開発者にはログで詳細 ✓ 集約ハンドラで一括処理 ✓ メッセージに文脈（何が・なぜ）★★★
POINT — 「握りつぶし」は最も罪深いアンチパターン

例外処理で絶対にやってはいけないのが握りつぶし（catch して何もしない）。エラーが静かに消え、「動いているように見えて実は壊れている」最悪の状態を作る。別冊S5ではトランザクションのロールバックも発動しなくなる。デバッグも不可能。「捕まえたら、最低でも記録する」。空の catch ブロックを見たら、それはバグの温床だと疑いましょう。
