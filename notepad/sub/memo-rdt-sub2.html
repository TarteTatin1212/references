01

別冊S2のミッション

MISSION

別冊S2は「設計原則を理論として理解する」回。本編Vol.2 で「SRPで分けた」「DIで注入した」を体験しました。でも「なぜそれが良いのか」を理論で理解すると、未知の場面でも判断できます。設計はセンスではなく、原則の積み重ねです。

設計を学ぶと何が変わるか壊れないコードへ
// 設計の有無で起きる差

フロー文書の名言：「設計を飛ばして書き始めたコードは、最初の1週間は速いが、1か月後から壊れ始める」。

  • 設計なし：書くのは速い → 変更で破綻 → 全体作り直し
  • 設計あり：最初は少し慎重 → 変更に強い → 長く使える
// 設計を見直すサイン（症状）
  • 新機能の追加に毎回たくさんの箇所を触る（結合度が高い）
  • 同じバグが繰り返し出る（責務が分かれていない）
  • どこに何のコードがあるか分からない（凝集度が低い）

これらの症状が出たら設計に立ち戻るサインです。

// 別冊S2のゴール // このS2を読み終えると、こう言える： ✓ "SRP違反は「説明に『と』が入る」で 見抜ける" ✓ "OCPは「拡張は足すだけ、既存は 触らない」" ✓ "凝集度を高く、結合度を低く、が あらゆる設計の合言葉" ✓ "YAGNI：必要になるまで作らない。 『将来のため』の8割は使われない" ✓ "迷ったら『戻せる選択肢』を選ぶ" // → 設計を「言葉」で語れるようになる ★★★ // → コードレビューで的確な指摘ができる
POINT — 設計原則は「禁止リスト」ではなく「判断の軸」

SOLIDなどの原則を「絶対守るべき掟」と捉えると、過剰設計に走ります。原則は「迷ったときの判断の軸」。状況に応じて適用度を変えるもの。メモ帳アプリのような小規模では原則を緩く、要点だけ守るのが正解です。

02

設計の目的は3つ

3 GOALS OF DESIGN

そもそも「なぜ設計するのか」。フロー文書が挙げる3つの目的——変更容易性・理解容易性・正しさの保証。「コードを速く書くため」ではありません。この目的を理解すると、設計判断がブレなくなります。

設計の3つの目的速く書くためではない
GOAL 1
変更容易性
後から直しやすい。要求は必ず変わる（別冊S1）。変更に強い構造を作る。
GOAL 2
理解容易性
他人と未来の自分が読める。半年後の自分が5分で理解できるか。
GOAL 3
正しさの保証
意図した通りに動く。テストできる構造＝正しさを守れる構造。
// 3目標とテストの関係

注目すべきはGOAL 3（正しさ）とテストの関係。テストできる構造（DI・SRP）は、そのまま変更容易性にも貢献する。設計品質とテスト容易性は同じものを別の角度から見ているのです。

// 3目標がメモ帳アプリでどう効くか // ─── 変更容易性 ─── // 「保存先を localStorage から DB に // 変えたい」→ Mapper だけ差し替え // → Controller/Service は無傷 ★ // ─── 理解容易性 ─── // MemoController を見れば // 「URLと画面の対応」が分かる // MemoService を見れば // 「業務ロジック」が分かる // → 探す場所が明確 ★ // ─── 正しさの保証 ─── // Mapper をモック化して // Service のロジックを単体テスト // → 「日時が必ずセットされる」を保証 ★ // ─── 速く書くことは目的ではない ─── // むしろ最初は少し遅くなる // でも「変更・理解・正しさ」を得る // → トータルでは圧倒的に速い ★★★
POINT — 「往復運動」が正しい設計プロセス

フロー文書の重要な指摘：設計は一直線ではなく往復運動。実装で問題に気づいたら設計に戻る、設計で問題に気づいたら要求に戻る。「もう決まったから」と無視すると辻褄合わせの汚いコードになる。早く戻るほど傷は浅い。これは別冊S1の「1:10:100」とも繋がります。

03

SRP — 単一責任の原則

SINGLE RESPONSIBILITY

SOLID のS。「クラスを変更する理由は1つだけ」。本編Vol.2 で「説明に『と』が入ったら分割」と実践しました。ここでは、なぜSRPが変更容易性を生むかを理論で理解します。

SRPの本質と効果変更理由を1つに
// 「変更理由」で責務を見る

SRPの「責任」とは「変更を引き起こす理由」。1つのクラスが複数の理由で変更されると、ある変更が別の機能を壊します。

// メモ帳の MemoManager（悪い例）

もし MemoManager が「保存・画面更新・検索」を全部やると：

  • DB変更でこのクラスを触る
  • 画面変更でも触る
  • 検索仕様変更でも触る

→ 3つの理由で変更される。1つの修正が他を壊すリスク。

// SRP違反 → 分割 // ─── 悪い例：全部入り ─── class MemoManager { void save(Memo m) { ... } // DB変更で触る void updateScreen() { ... } // 画面変更で触る List search(String k) { ... } // 検索変更で触る } // → 変更理由が3つ = SRP違反 ★ // ─── 良い例：責務で分割 ─── class MemoMapper { void insert(Memo m) { ... } } // 変更理由：DB class MemoService { void create(Memo m) { ... } } // 変更理由：業務ロジック class MemoController { String list(Model m) { ... } } // 変更理由：画面 // → 各クラス変更理由は1つ ★★★ // ─── 分割のサイン ─── // クラスの説明文に「と」「かつ」が入る // → 「保存し、かつ画面更新する」 // → 即・分割を検討 // クラス名に Manager/Util/Helper が付く // → 責務が曖昧なサイン // → 「何をするか」を具体的な名前に
POINT — Manager / Util / Helper は黄信号

クラス名にManager・Util・Helperが付いたら「責務が曖昧」のサイン。これらの名前は「何でも入れていい箱」になりがち。MemoManager より MemoService（業務）/MemoMapper（保存） のように「何をするか」を具体名で。名前が設計を導きます。

04

OCP — 開放閉鎖の原則

OPEN-CLOSED

SOLID のO。「拡張に開き、修正に閉じる」＝機能追加は足すだけ、既存コードは触らない。メモ帳アプリで「保存先を切り替えたい」場面を例に、OCPの実践を見ます。

OCPとインターフェース拡張は足すだけ
// OCPの実現手段＝抽象化

OCPはインターフェース（抽象）で実現します。具体的な実装を直接使わず、「契約（インターフェース）」に依存する。

// メモ帳での例

フロー文書の例：保存先を localStorage / IndexedDB で切り替え。Spring版ならMySQL / 別DBの切り替え。

  • MemoRepository インターフェースを定義
  • 実装を MyBatis版、JPA版 など差し替え可能に
  • Service はインターフェースだけ知る
// ただし3回ルール

フロー文書の警告：OCPのための抽象化も、必要になってから。最初から全部インターフェース化は過剰設計。

// OCP：保存先を差し替え可能に // ─── 抽象（インターフェース）─── public interface MemoRepository { List<Memo> findAll(); void save(Memo m); } // ─── 実装A：MyBatis版 ─── @Repository public class MyBatisMemoRepository implements MemoRepository { private final MemoMapper mapper; // MyBatisで実装 } // ─── 実装B：将来のJPA版 ─── // @Repository // class JpaMemoRepository // implements MemoRepository { ... } // → 足すだけ。既存は触らない ★★★ // ─── Service は抽象に依存 ─── @Service public class MemoService { private final MemoRepository repo; // 具体実装を知らない // → 保存先が変わっても無傷 ★★★ } // ─── ただし注意（YAGNI）─── // 本シリーズ本編では Mapper を直接使う // （Repository層は入れていない） // 理由：保存先を変える予定がない // → 「必要になってから」抽象化する ★ // → これが OCP と YAGNI のバランス
POINT — OCPと過剰設計は紙一重

OCPは強力ですが、「あらゆる箇所をインターフェースで抽象化」すると過剰設計。本シリーズ本編が Mapper を直接使い、Repository 層を入れていないのは「保存先を変える予定がないから」。OCPは「変わりそうな箇所」だけに適用する。変化を予測して、変わる所だけ柔らかくが極意です。

05

凝集度と結合度

COHESION & COUPLING

フロー文書が「あらゆる設計の合言葉」と呼ぶ「高凝集・低結合」。凝集度＝関連する処理がまとまっている度、結合度＝モジュール間の依存の強さ。この2つを意識すると設計が一段上がります。

凝集度を高く・結合度を低く設計の合言葉
// 凝集度（Cohesion）

関連する処理は1つにまとめる。メモの操作はメモ関連のクラスに集約。バラバラに散らばっていると「どこに何があるか分からない」（理解容易性の低下）。

  • 高凝集：MemoService にメモのロジックが集まっている
  • 低凝集：メモのロジックがあちこちに散在
// 結合度（Coupling）

モジュール間の依存を弱く。依存が強いと、片方の変更がもう片方に波及する。

  • 低結合：引数で渡す（依存が明示的・弱い）
  • 高結合：グローバル変数共有（依存が暗黙・強い）
// 結合度の強弱を比較 // ─── 高結合（悪い）─── class MemoService { void create(Memo m) { // グローバルな状態を直接触る GlobalState.currentUser...; GlobalDB.connection...; // → どこから何に依存するか不明 ★ // → テストでモック化できない } } // ─── 低結合（良い）─── @Service @RequiredArgsConstructor class MemoService { // 依存を「引数（コンストラクタ）」で受け取る private final MemoMapper mapper; void create(Memo m) { mapper.insert(m); // → 依存が明示的・弱い ★★★ // → テストでモック差し替え可 } } // ─── DI が低結合を実現する ─── // @RequiredArgsConstructor = // 依存性注入（DI） // → 「外から依存を渡す」 // → 結合度が下がる // → テスト容易性が上がる ★★★ // 本編Vol.2 でやった「DIでモック化」は // この「低結合」の恩恵だった
3層構造は凝集度・結合度の実装理論が形になる
層凝集（集めるもの）
ControllerHTTP関連の処理
Service業務ロジック
MapperSQL・データアクセス

各層が高凝集（関連処理が集まる）。層間は低結合（DIで疎結合・上から下の依存のみ）。レイヤード構造は高凝集・低結合の実装そのものです。

// 高凝集・低結合の効果 // ─── 高凝集の効果 ─── // 「メモの保存を直したい」 // → MemoMapper を見れば全部ある // → 探さなくていい（理解容易性）★ // ─── 低結合の効果 ─── // 「DB を MySQL から変えたい」 // → Mapper だけ差し替え // → Service/Controller は無傷 // → 変更が波及しない（変更容易性）★ // ─── 逆（低凝集・高結合）の地獄 ─── // 低凝集：メモのロジックが // Controller/Service/Util に散在 // → 1つの修正で3箇所探す // 高結合：全部が密に絡む // → 1つ変えると全部壊れる // → これが「スパゲッティコード」★★★
POINT — DRYを意識しすぎない

フロー文書の重要な但し書き：DRY（繰り返すな）を意識しすぎない。「形が似てるだけ」で共通化しない。共通化するのは「変更理由が同じとき」だけ。たまたま似たコードを無理に共通化すると、片方だけ変えたい時に困ります。偶然の重複と本質的な重複を見分けるのが大事です。

06

KISS・YAGNI・過剰設計

KISS / YAGNI / OVER-ENGINEERING

設計で最も陥りやすい罠が「過剰設計」。「将来のため」「拡張性のために」が出たら立ち止まる。KISS（シンプルに）とYAGNI（必要になるまで作るな）が歯止めです。

過剰設計を防ぐ2原則「将来のため」は要注意
KISS
Keep It Simple
シンプルに保て。賢く見せようとして複雑にするな。読めるコードが良いコード。
YAGNI
You Aren't Gonna Need It
必要になるまで作るな。「将来のため」の8割は使われない。
// 過剰設計の典型例
  • 使わないインターフェース（実装が1つしかないのに抽象化）
  • 使わない設定の柔軟性（誰も変えない設定を外部化）
  • 「将来複数DBになるかも」で全部Repository化
  • メモ帳にクリーンアーキテクチャ（規模に合わない）
// 3回ルール

抽象化・共通化は「同じ問題が3回現れてから」。1回目はベタ書き、2回目は我慢、3回目で抽出。

// 過剰設計 vs 適切な設計 // ─── 過剰設計（メモ帳に） ─── // 「将来に備えて」全部抽象化 MemoRepositoryInterface MemoRepositoryFactory AbstractMemoRepository MemoRepositoryStrategy MemoRepositoryProxy // → メモ1種類しかないのに // 5階層の抽象化 ★ // → 読めない・理解できない // → YAGNI違反 // ─── 適切（本シリーズ）─── MemoMapper // インターフェース MemoMapper.xml // SQL // → これで十分 ★★★ // → シンプル・読める・テストできる // ─── 規模に合った構造を選ぶ ─── 規模 推奨アーキテクチャ 数画面・短命 シンプルMVC 小〜中規模 レイヤード ← メモ帳はここ 長期・複雑 レイヤード+DDD/クリーンアーキ // メモ帳にクリーンアーキは過剰 ★★★ // ─── マジックナンバーには名前を ─── // KISS だが「定数化」はする 悪: if (title.length() > 200) 良: static final int MAX_TITLE = 200; if (title.length() > MAX_TITLE)
NG — 「将来のため」の作り込み

「いつか複数DB対応するかも」「拡張性のために」で使わない抽象化を大量に。読めない・テストしにくい・誰も使わない。YAGNI違反の典型。

OK — 必要十分のシンプルさ

今の要求をシンプルに満たす。拡張が必要になったらその時に足す（OCPで足しやすくはしておく）。KISS + YAGNI。

POINT — 「測定してから最適化する」

フロー文書の判断のコツ：推測で重い仕組みを入れない。「遅くなりそうだからキャッシュを」ではなく「測って遅かったらキャッシュを」。本シリーズが localStorage ではなく素直に MySQL を使い、最初から FULLTEXT INDEX を入れないのも同じ思想。早すぎる最適化は諸悪の根源です。

07

レイヤードアーキテクチャ

LAYERED ARCHITECTURE

本シリーズが採用するレイヤード（層を分ける）アーキテクチャを、理論として整理します。なぜ層を分けるか、依存方向の鉄則、フォルダ構成の意味。Spring の3層がこれの実装です。

層を分ける意味と鉄則依存は上から下へ
// なぜ層を分けるか

各層が「1つの関心」に集中することで、高凝集・低結合が自然に実現します。

  • Controller層：画面・入力受付・表示
  • Service層：業務ロジック
  • Mapper層：保存・読み込み
// 鉄則：依存は上から下へ

下の層が上の層を知ってはいけない。Mapper が Controller を呼ぶのは禁止。これを破ると循環依存が生まれ、すぐ破綻します。

// フォルダ構成は設計そのもの

フォルダを見ただけで「何をするアプリか」が分かるのが理想。

// レイヤードの依存方向 // ─── 正しい依存（上から下）─── [Controller層] 画面・HTTP ↓ 呼ぶ（OK） [Service層] 業務ロジック ↓ 呼ぶ（OK） [Mapper層] DB // ─── 禁止される依存 ─── [Mapper層] ↓ 呼ぶ（NG！） [Controller層] // → 下が上を知る = 循環の元 ★★★ // ─── フォルダ構成（設計の表現）─── src/main/java/com/example/memo/ ├ controller/ // 画面・HTTP │ ├ MemoController │ └ CategoryController ├ service/ // 業務ロジック │ ├ MemoService │ └ CategoryService ├ mapper/ // DB │ ├ MemoMapper │ └ CategoryMapper └ entity/ // データ構造 ├ Memo └ Category // → フォルダ = 層 = 関心の分離 // → 見ただけで構造が分かる ★★★ // ─── フロー文書の翻訳 ─── // 原典：ui/ domain/ storage/ utils/ // Spring：controller/ service/ // mapper/ entity/ // → 思想は同じ「関心の分離」
POINT — 層をまたぐ密結合が最大の敵

フロー文書のレビュー観点：「データ層がUIを直接触っていないか」。Mapper の中で画面更新を呼ぶ、Service が HTTP の詳細を知る——こういう層をまたいだ密結合が一番危険。各層は自分の関心だけ。Controller は HTTP、Service は業務、Mapper は SQL。越境させないのが鉄則です。

08

トレードオフの判断軸

TRADE-OFFS

設計には「絶対の正解」がない場面が多い。厳密さ vs 柔軟性、読み速さ vs 書き速さ。フロー文書が示す判断軸を持っておくと、迷ったときにブレません。

4つのトレードオフ軸迷ったときの判断材料
トレードオフ判断のヒント
厳密さ vs 柔軟性型・制約を強くすると安全だが手間増
読み速さ vs 書き速さインデックスは読み速・書き遅
抽象化 vs 具体性抽象化は再利用↑だが理解↓
開発速度 vs 長期保守プロジェクト寿命で重視軸が変わる
// 判断の3つのコツ
  1. 迷ったら「戻せる選択肢」を選ぶ（localStorage→IndexedDBは後で変えられる）
  2. 測定してから最適化（推測で重い仕組みを入れない）
  3. 3回ルール（抽象化は3回目に登場してから）
// メモ帳での実際のトレードオフ // ─── ① ID：連番 vs UUID ─── 連番: + シンプル、人が読める - 推測されやすい、同期で衝突 UUID: + 衝突しない、同期向き - 長い、人が読めない 判断: 本シリーズは連番 理由: v1は同期しない（YAGNI） 後でUUIDに移行も可能（戻せる）★ // ─── ② 検索：LIKE vs FULLTEXT ─── LIKE: + 実装が簡単、すぐ動く - 件数増で遅い FULLTEXT: + 高速、スコア順 - INDEX設定が必要 判断: 小規模はLIKE、大規模でFULLTEXT → 測定してから切り替え（Vol.8）★ // ─── ③ Form と Entity：分離 vs 兼用 ─── 分離: + 役割が明確、安全 - クラスが増える 兼用: + シンプル - バリデーションが混在 判断: Vol.2〜は兼用、Vol.12 で分離 → 必要になったタイミングで分ける ★ // → トレードオフに「正解」はない // → 状況・寿命・規模で判断する ★★★
POINT — 「迷ったら戻せる選択肢」が最強の指針

設計判断で最も実用的なのが「戻せる選択肢を選ぶ」。連番→UUID、LIKE→FULLTEXT、兼用→分離。どれも後から変更できる。逆に戻せない決定（DBの種類、言語選定など）は慎重に。「変えにくい決定ほど慎重に、変えやすい決定は気軽に」。これが設計のリスク管理です。
