01

別冊S5のミッション

MISSION

別冊S5は「データの整合性を守る理論を理解する」回。本編Vol.6 でメモのタグ付けを実装します。これは「memos を更新し、memo_tags に複数INSERT」という複数の操作。途中で失敗したら？——この問いに答えるのがトランザクションです。

なぜトランザクションが必要か中途半端を防ぐ
// 複数操作の「中途半端」問題

本編Vol.6 の「メモにタグ3つを付ける」は、内部で複数のINSERTになります。もし2つ目で失敗したら？

  • 1つ目のタグ：保存済み
  • 2つ目のタグ：失敗
  • 3つ目のタグ：未実行

→ 中途半端な状態。「タグ1つだけ付いた壊れたメモ」が残る。これを防ぐのがトランザクション。

// トランザクションの約束

「全部成功か、全部なかったことに」。途中で失敗したら最初の状態に戻す（ロールバック）。中途半端を許さない。

// トランザクションがない世界の悲劇 // ─── タグ付けで途中失敗 ─── メモにタグ[重要,急ぎ,あとで]を付ける INSERT memo_tags(memo=1, tag=重要) ✓ INSERT memo_tags(memo=1, tag=急ぎ) ✓ INSERT memo_tags(memo=1, tag=あとで) ✗ 失敗！ （DB接続切れ等） // トランザクションなしだと… 結果：重要・急ぎは保存、あとでは未保存 → 中途半端な状態 ★ → ユーザー「あれ、あとでタグは？」 → データが意図と違う // ─── 銀行の例（古典）─── A口座から1万円引く ✓ B口座に1万円足す ✗ 失敗 // → トランザクションなし： // 1万円が消えた！（Aから引いたまま） // → トランザクションあり： // 全部取り消し、Aの1万円は戻る ★★★ // → 「全部か、ゼロか」が整合性の鍵
POINT — 「単一の操作」には不要、「複数の操作」に必要

トランザクションが必要なのは「1つの業務が複数のDB操作にまたがる」とき。Vol.2〜5 の単一INSERTには不要でした（1操作は元々原子的）。でも Vol.6 の「メモ更新＋タグ複数INSERT」は複数操作なので必要。「この業務、途中で失敗したら困る複数操作か?」がトランザクション要否の判断基準です。

02

トランザクションとは

WHAT IS TRANSACTION

トランザクションとは「ひとまとまりの処理」。複数のDB操作を1つの単位として扱い、「全部成功（コミット）」か「全部取り消し（ロールバック）」のどちらかにする仕組みです。

コミットとロールバック2つの結末
// トランザクションの3つの操作
BEGIN
開始
トランザクションを始める。ここから一連の操作。
COMMIT
確定
全部成功 → 変更を確定。DBに永続化。
ROLLBACK
取り消し
失敗 → 全部なかったことに。開始前に戻す。
// 流れ

BEGIN →（複数の操作）→ 全部OKなら COMMIT、1つでも失敗なら ROLLBACK。「中間状態が外から見えない」のが特徴。

// トランザクションの流れ // ─── 成功パターン ─── BEGIN; // 開始 UPDATE memos ... // 操作1 ✓ INSERT memo_tags ... // 操作2 ✓ INSERT memo_tags ... // 操作3 ✓ COMMIT; // 全部確定 ★ // → 変更がDBに永続化される // ─── 失敗パターン ─── BEGIN; // 開始 UPDATE memos ... // 操作1 ✓ INSERT memo_tags ... // 操作2 ✓ INSERT memo_tags ... // 操作3 ✗ 例外！ ROLLBACK; // 全部取り消し ★★★ // → 操作1,2もなかったことに // → BEGIN前の状態に完全に戻る // ─── Spring では自動 ─── // @Transactional を付けると… // メソッド開始 = BEGIN // 正常終了 = COMMIT // 例外発生 = ROLLBACK // → BEGIN/COMMIT/ROLLBACKを // 手で書かなくていい ★★★（Section 05）
POINT — 「中間状態が見えない」のが重要

トランザクション中の中途半端な状態は、外（他のユーザーや処理）からは見えません。タグを3つINSERTしている途中、他の人が一覧を見ても「0個」か「3個」しか見えない（1個や2個の中間は見えない）。COMMIT した瞬間に全部見える。「全部見えるか、何も見えないか」。これがデータの一貫性を守ります。

03

ACID特性

ACID PROPERTIES

トランザクションが守るべき4つの性質 ACID。原子性・一貫性・分離性・永続性。難しい用語ですが、「中途半端を作らない」ための4つの約束と理解すれば十分です。

ACID — 4つの約束頭文字で覚える
頭字性質意味
A原子性
Atomicity		全部か、ゼロか
C一貫性
Consistency		ルールを保つ
I分離性
Isolation		互いに干渉しない
D永続性
Durability		確定後は消えない
// 本シリーズで特に重要な2つ
  • A（原子性）：タグ複数INSERTが「全部か、ゼロか」（Vol.6）
  • I（分離性）：同時アクセス時の干渉防止（Section 07）
// ACID をメモ帳で理解 // ─── A 原子性（Atomicity）─── // タグ3つINSERT // → 全部成功 or 全部なし // → 「タグ1個だけ」は許さない ★★★ // ─── C 一貫性（Consistency）─── // 制約（FK・UNIQUE等）を破らない // → 存在しないtag_idは紐付けられない // （FK制約が一貫性を守る） // ─── I 分離性（Isolation）─── // 2人が同時にタグ付けしても // 互いの中間状態が見えない // → 混ざらない（Section 07） // ─── D 永続性（Durability）─── // COMMITしたら、停電してもデータは残る // → DBがディスクに確実に記録 // ─── 4つは「中途半端を作らない」─── // ための異なる角度の保証 ★★★ // A: 操作の中途半端なし // C: ルール違反の状態なし // I: 他者との混ざりなし // D: 確定後の消失なし
POINT — ACID は「DBが頑張ってくれる」

ACID の多くはDB（MySQL等）が自動的に保証してくれます。私たちが意識するのは主にA（原子性）＝トランザクションの境界をどこに引くか。「この一連の操作は全部か、ゼロか」とまとめるのが開発者の仕事。Spring では @Transactional でメソッドに境界を引くだけ。残りはDBとフレームワークがやってくれるのです。

04

原子性とロールバック

ATOMICITY & ROLLBACK

ACID で最重要な原子性（Atomicity）。「全部か、ゼロか」を実現するのがロールバック。失敗時に開始前の状態へ完全に巻き戻す。本編Vol.6 のタグ付けで、この恩恵を受けます。

ロールバックが「全部か、ゼロか」を作る巻き戻しの仕組み
// Vol.6 のタグ付けで考える

「メモのタグを[重要,急ぎ,あとで]に更新」は、内部で：

  1. 既存のタグ紐付けを全削除
  2. 新しいタグを順にINSERT

途中で失敗したら？ ロールバックで削除も取り消し、INSERTも取り消し。元のタグ状態に戻る。

// 「全削除→再INSERT」が安全な理由

タグ更新は「全部消して付け直す」のが定石。トランザクションがあるので、途中失敗しても元に戻るから安全。差分更新よりシンプルで確実です。

// タグ更新とロールバック // ─── タグを付け替える処理 ─── @Transactional // ← これが原子性を保証 public void updateTags( Long memoId, List<Long> tagIds) { // 1. 既存の紐付けを全削除 mapper.deleteTagsByMemoId(memoId); // 2. 新しいタグを順にINSERT for (Long tagId : tagIds) { mapper.addTag(memoId, tagId); } } // ─── 途中失敗時の動き ─── deleteTagsByMemoId(1) ✓ 削除 addTag(1, 重要) ✓ addTag(1, 急ぎ) ✗ 例外！ // @Transactional のおかげで… ROLLBACK → 削除も取り消し（タグが戻る） → 重要のINSERTも取り消し → 元のタグ状態に完全復帰 ★★★ // → 「全削除したのに付け直せなかった」 // という最悪の事態を防ぐ // ─── トランザクションなしの恐怖 ─── // deleteで全タグ消えた後に // INSERTが失敗したら… // → タグが全部消えたまま！★ // → だから @Transactional が必須
POINT — 「全削除→再INSERT」はトランザクションが前提

多対多の更新で「既存を全削除して新しく付け直す」のは定石ですが、トランザクションがあって初めて安全です。もしトランザクションなしで「全削除→INSERT失敗」したら、タグが全部消えたままの悲劇に。@Transactional があるから、大胆に全削除→再INSERTできる。トランザクションがシンプルな実装を可能にするのです。

05

@Transactionalの仕組み

HOW @TRANSACTIONAL WORKS

Spring では @Transactional アノテーション1つでトランザクションが効きます。BEGIN/COMMIT/ROLLBACK を手で書く必要がない。その仕組みと正しい付け方を理解します。

@Transactional の動作メソッド境界がトランザクション
// @Transactional が自動でやること
  • メソッド開始時に BEGIN
  • 正常に戻ったら COMMIT
  • 例外が出たら ROLLBACK

メソッドの実行範囲＝トランザクションの範囲。だから「複数操作を1メソッドにまとめ、@Transactional を付ける」が基本形。

// どこに付けるか
  • Service のメソッドに付ける（業務の単位）
  • Controller や Mapper には付けない
  • クラスに付けると全メソッドに適用
// @Transactional の正しい使い方 import org.springframework .transaction.annotation.Transactional; @Service @RequiredArgsConstructor public class MemoTagService { private final MemoMapper mapper; // 複数操作 → @Transactional 必須 @Transactional public void updateTags( Long memoId, List<Long> tagIds) { mapper.deleteTagsByMemoId(memoId); for (Long id : tagIds) { mapper.addTag(memoId, id); } } // → メソッド全体が1トランザクション // → 例外で全ロールバック ★★★ } // ─── 仕組み：プロキシ（AOP）─── // Spring は @Transactional メソッドを // 「プロキシ」で包む： 呼び出し ↓ [プロキシ] BEGIN ↓ 実際のメソッド実行 ↓ [プロキシ] 正常 → COMMIT 例外 → ROLLBACK // → 開発者は中身を書くだけ ★ // → 境界管理はSpringが代行 // ─── 付ける場所の指針 ─── Controller : 付けない（HTTPの世話） Service : 付ける（業務の単位）★ Mapper : 付けない（SQLの実行） // → 別冊S2の3層と対応 // → 業務ロジックの境界＝トランザクション
POINT — トランザクションは「Service層」に引く

@Transactional はService のメソッドに付けます。理由：Service は「1つの業務」を表す単位だから（別冊S2）。「メモにタグを付ける」という業務全体が1トランザクション。Mapper（個々のSQL）や Controller（HTTP）には付けない。「業務の境界＝トランザクションの境界」。3層構造とトランザクションが綺麗に対応します。

06

ロールバックの条件

ROLLBACK CONDITIONS

意外な落とし穴：@Transactional は「どんな例外でもロールバック」ではありません。デフォルトでは非チェック例外（RuntimeException）でのみロールバック。チェック例外ではロールバックしない。ここを知らないとハマります。

ロールバックされる例外・されない例外デフォルトの挙動に注意
// デフォルトの挙動
例外の種類ロールバック
RuntimeException系される
Error系される
チェック例外
(IOException等)		されない！

チェック例外ではロールバックされないのがデフォルト。これは罠。

// 対処
  • 本シリーズの例外（DB系）は大半がRuntimeExceptionなので普通はOK
  • チェック例外でもロールバックしたいなら @Transactional(rollbackFor = Exception.class)
// ロールバック条件の詳細 // ─── デフォルト：RuntimeException のみ ─── @Transactional public void updateTags(...) { mapper.deleteTagsByMemoId(memoId); // DB系の例外は RuntimeException // （DataAccessException系） mapper.addTag(...); // → 失敗時 RuntimeException // → 自動でロールバック ★ } // ─── 罠：チェック例外 ─── @Transactional public void process() throws IOException { mapper.insert(...); // ✓ throw new IOException(); // → チェック例外 // → デフォルトではロールバックされない！★ // → insertが残ってしまう } // ─── 対処：rollbackFor 指定 ─── @Transactional(rollbackFor = Exception.class) public void process() throws IOException { // → 全ての例外でロールバック ★★★ } // ─── もう一つの罠：自己呼び出し ─── // 同じクラス内のメソッド呼び出しでは // @Transactional が効かない // （プロキシを経由しないため）★★★ public void a() { b(); // b の @Transactional は効かない！ } @Transactional public void b() { ... } // → 別クラスに分けるか、設計を見直す
POINT — 「自己呼び出し」で @Transactional が効かない罠

もう一つの有名な罠：同じクラス内のメソッド呼び出しでは @Transactional が効きません。Spring はプロキシ（Section 05）で包んでトランザクションを効かせますが、クラス内部の直接呼び出しはプロキシを経由しないから。対処は別のService クラスに分ける。本シリーズでは1メソッド完結なので問題になりませんが、知っておくと将来のデバッグが楽です。

07

分離レベルと同時実行

ISOLATION LEVELS

ACID のI（分離性）。複数のトランザクションが同時に動くときの干渉をどこまで防ぐか。これが分離レベル。個人開発では深入り不要ですが、業務で必須の知識として概要を押さえます。

同時実行の問題と分離レベル業務で効いてくる
// 同時実行で起きる問題
  • ダーティリード：他者の未確定データを読む
  • 反復不能読み取り：同じ読み取りで結果が変わる
  • ファントムリード：途中で行が増減する
// 分離レベル（強い順）
レベル特徴
SERIALIZABLE最強・遅い
REPEATABLE READMySQLの既定
READ COMMITTED多くのDBの既定
READ UNCOMMITTED最弱・速い

強いほど安全だが遅い（別冊S2のトレードオフ）。通常はDBの既定で十分。

// 分離レベルの考え方 // ─── トレードオフ（別冊S2）─── 強い分離（SERIALIZABLE） + 安全（干渉なし） - 遅い（ロック多い） 弱い分離（READ UNCOMMITTED） + 速い - 危険（未確定データが見える） // → 厳密さ vs 速度のトレードオフ // ─── 本シリーズの方針 ─── // 個人開発・単一ユーザー前提（v1制約） // → 同時実行はほぼ起きない // → DBの既定（MySQL: REPEATABLE READ）で十分 // → 分離レベルを意識しなくてOK ★ // ─── 業務で必要になる場面 ─── // 在庫管理：同時に注文が来たら？ // 座席予約：同じ席を2人が取ったら？ // → 分離レベル＋ロックが効いてくる // → メモ帳v1では不要（YAGNI） // ─── 指定する場合（参考）─── @Transactional( isolation = Isolation.SERIALIZABLE) // → 必要になったら指定（通常は不要）
POINT — 個人開発では分離レベルは「既定でOK」

分離レベルは同時アクセスが多い業務システムで重要ですが、本シリーズは単一ユーザー前提（v1制約）なので、DBの既定のままで問題ありません。「将来 多人数対応するなら考える」（YAGNI＝別冊S2）。今は「@Transactional で原子性を守る」だけに集中。分離レベルは「そういう概念がある」と知っておけば十分です。

08

トランザクションあるある失敗

TRANSACTION PITFALLS

トランザクションで初心者が踏みやすい失敗を5つ。知っていれば回避できるものばかり。特に「@Transactional の付け忘れ」と「自己呼び出し」は要注意です。

トランザクションあるある 5選先人の轍を踏まない

① 複数操作なのに @Transactional 付け忘れ ★★★

  • タグ複数INSERTにトランザクションなし
  • 途中失敗で中途半端なデータ
  • → 複数DB操作のService に必ず付ける

② 自己呼び出しで効かない ★★★

  • 同クラス内の呼び出しはプロキシを通らない
  • → 別クラスに分ける（Section 06）

③ チェック例外でロールバックされない

  • デフォルトは RuntimeException のみ
  • → rollbackFor を指定（Section 06）

④ 例外を catch して握りつぶす

  • try-catch で例外を捕まえて何もしない
  • → 例外が伝播しないとロールバックされない
  • → catch するなら再スローするか、handle する

⑤ トランザクションを長く持ちすぎ

  • 外部API呼び出しなど重い処理をトランザクション内に
  • → DBロックが長く、性能劣化
  • → トランザクションは短く保つ
// ④例外の握りつぶし（要注意） // ─── NG：例外を握りつぶす ─── @Transactional public void updateTags(...) { try { mapper.deleteTagsByMemoId(memoId); mapper.addTag(...); // 失敗 } catch (Exception e) { // 何もしない（握りつぶし）★ } } // → 例外が外に出ない // → Springがロールバックを発動できない // → 中途半端なデータが残る ★★★ // ─── OK：例外を伝播させる ─── @Transactional public void updateTags(...) { mapper.deleteTagsByMemoId(memoId); mapper.addTag(...); // 例外が出たらそのまま外へ伝播 // → Springがロールバック ★★★ } // ─── ⑤トランザクションは短く ─── NG: @Transactional public void process() { mapper.insert(...); callExternalApi(); // 重い・遅い ★ mapper.update(...); } // → API呼び出し中もDBロック OK: // 外部API は トランザクション外で // DB操作だけを @Transactional で囲む
POINT — 「例外を握りつぶさない」は鉄則

④の例外の握りつぶしは特に危険。catch して何もしないと、Spring が例外を検知できず、ロールバックが発動しません。結果、中途半端なデータが残る。「トランザクション内では、例外を握りつぶさず外に伝播させる」。どうしても catch するなら、適切に処理するか再スローする。これはエラーハンドリング（Vol.12 で本格的に扱う）の基本でもあります。
