01

別冊S8のミッション

MISSION

別冊S8は「ユーザー体験（UX）を設計する」回。これまでは「機能が動く」ことを中心に作ってきました。でも本編Vol.9 以降は「使いやすさ」「安心して使える」という、より人間寄りの設計が問われます。技術だけでなく「使う人の気持ち」を設計に取り込みます。

「動く」の先にある「使える」UXの視点
// 本編Vol.9 で扱う機能
  • ピン留め：よく使うメモを上に固定
  • 並び替え：更新順・作成順・名前順

どれも「使い勝手」に関わる機能。技術的には簡単でも、UX設計の判断が肝。

// 別冊S8のゴール
  • 物理削除と論理削除を使い分けられる
  • ソフトデリートを設計できる
  • 状態（ピン・アーカイブ）の持たせ方が分かる
  • 「取り消せる」UXを設計できる
// 「動く」から「使える」へ // ─── これまで（機能中心）─── US-001〜004： メモを作る・分類・タグ・検索 → 「できる」を実現 // ─── Vol.9以降（UX中心）─── 「よく使うメモが下に埋もれる」 → ピン留めで上に固定したい 「終わったメモを消したいが、 後で見たいかも」 → アーカイブ（消さずに隠す） 「うっかり大事なメモを削除した」 → 取り消せると安心 // ─── UXの問い ─── // 「技術的にできるか」ではなく // 「ユーザーが快適・安心か」★★★ // → 別冊S1の WHY に立ち返る // → 「なぜこの機能が要るか」

ピン留めやアーカイブのような機能は、「なぜそれが欲しいか（WHY）」を理解しないと的外れになります（別冊S1）。「アーカイブ＝消す」と勘違いすると、ユーザーの「消したくないけど隠したい」という本当の願いを外す。UX設計とは「ユーザーの本当の願いを技術で叶える」こと。要求工学（別冊S1）と地続きです。

02

UXとは何か

WHAT IS UX

UX（ユーザー体験）とは、ユーザーがアプリを使う中で得る体験全体。見た目（UI）だけでなく、「迷わない」「安心できる」「ストレスがない」を含む。良いUXは「使う人の気持ちを先回り」します。

UXの構成要素UIだけではない
// UX を支える要素
要素1
分かりやすさ
迷わず操作できる。何が起きるか予測できる。
要素2
安心感
取り返しがつく。間違えても怖くない。
要素3
効率
よく使う操作が簡単。無駄な手順がない。
// UI と UX の違い
  • UI：見た目・操作部品（ボタン等）
  • UX：使った体験全体（UIを含む）
// UX を具体例で // ─── 悪いUX ─── 削除ボタンを押す → 即座に消える（確認なし） → 取り消せない → 「あ、間違えた…」😱 // → 不安・ストレス ★ // ─── 良いUX ─── 削除ボタンを押す → 「削除しますか?」確認 → 削除しても「取り消す」リンク → またはアーカイブ（後で戻せる） → 「間違えても大丈夫」😌 // → 安心して使える ★★★ // ─── メモ帳でのUX設計 ─── ピン留め： よく使うメモを上に → 「探す手間」を減らす（効率） アーカイブ： 消さずに隠す → 「消すのが怖い」を解消（安心） 並び替え： 好みの順で見られる → 「自分のやり方で」（自由度） // ─── UX = 気持ちの設計 ─── // 技術 + 「使う人の気持ち」★★★
POINT — 良いUXは「取り返しがつく」

UX設計で最も重要な原則の一つが「取り返しがつく」こと。人は間違えます。間違えても「元に戻せる」と分かっていれば、安心して操作できる。逆に「一度押したら取り返しがつかない」と、ユーザーは怖くて操作をためらう。本編Vol.9 のアーカイブ（消さずに隠す）は、まさに「取り返しがつく削除」。安心感がUXの土台です。

03

物理削除と論理削除

HARD VS SOFT DELETE

「消す」には2種類あります。物理削除（DELETE）＝データを本当に消す。論理削除（ソフトデリート）＝消したことにする（フラグを立てるだけ）。本編Vol.9 のアーカイブは論理削除の応用です。

2つの「消す」の違い本当に消す vs 隠す
物理削除論理削除
動作DELETE文フラグを立てる
データ本当に消える残る（隠れる）
復元不可可能
容量減る増え続ける
例完全削除アーカイブ・ゴミ箱
// 使い分け
  • 物理削除：本当に要らない・容量・法令
  • 論理削除：復元の可能性・履歴・安心感
// メモ帳での適用

アーカイブ＝論理削除（隠すが残す）。完全削除＝物理削除。2段階にすると安心。

// 物理削除と論理削除 // ─── 物理削除（本当に消す）─── DELETE FROM memos WHERE id = #{id}; // → データが消える // → 復元不可 ★ // ─── 論理削除（消したことにする）─── // archived フラグを立てるだけ UPDATE memos SET archived = true WHERE id = #{id}; // → データは残る // → 一覧では archived=false だけ表示 // → archived=true は「隠れる」 // → 戻すのも簡単（false に戻す）★★★ // ─── 一覧の取得（論理削除考慮）─── // 通常の一覧（アーカイブ済みを除く） SELECT * FROM memos WHERE archived = false ORDER BY updated_at DESC; // アーカイブ一覧（隠したものだけ） SELECT * FROM memos WHERE archived = true; // ─── メモ帳の2段階削除（安心設計）─── 通常メモ ↓ アーカイブ（論理削除） アーカイブ済み（隠れるが残る） ↓ 完全削除（物理削除） 本当に消える // → 2段階で「うっかり消し」を防ぐ ★★★ // → ゴミ箱と同じ発想
POINT — 「ゴミ箱」は論理削除の身近な例

OS のゴミ箱が論理削除の典型。「削除」してもすぐ消えずゴミ箱に移動（隠れる）、後で復元できる。本当に消すのは「ゴミ箱を空にする」とき。本編Vol.9 のアーカイブも同じ発想。「1段階目で隠す（論理削除）、2段階目で本当に消す（物理削除）」。この2段階が「うっかり消し」からユーザーを守る定番のUXパターンです。

04

ソフトデリートの設計

SOFT DELETE DESIGN

ソフトデリート（論理削除）を実装するときの設計を学びます。フラグの持ち方、一覧クエリへの影響、注意点。便利ですが「全クエリに条件追加が必要」という落とし穴もあります。

ソフトデリートの設計と注意点便利だが落とし穴も
// フラグの持ち方
  • boolean フラグ：archived（true/false）
  • 日時：archived_at（NULLなら未アーカイブ）

日時方式は「いつアーカイブしたか」も分かる。本シリーズはシンプルにboolean。

// 落とし穴：全クエリに条件追加

論理削除を入れると、通常の一覧・検索すべてに archived = false を足す必要がある。忘れると「消したはずのメモが出る」。

// トレードオフ（別冊S2）

安心（復元可能）と複雑さ（全クエリに条件）のトレードオフ。メモ帳のように復元が嬉しい場面では論理削除が勝つ。

// ソフトデリートの設計 // ─── テーブル設計 ─── ALTER TABLE memos ADD COLUMN archived BOOLEAN NOT NULL DEFAULT false; // → archived フラグを追加 ★ // ─── 操作 ─── // アーカイブする（論理削除） UPDATE memos SET archived=true WHERE id=#{id}; // 復元する（戻す） UPDATE memos SET archived=false WHERE id=#{id}; // 完全削除する（物理削除） DELETE FROM memos WHERE id=#{id}; // ─── 落とし穴：全クエリに条件 ★★★ ─── // 通常一覧 SELECT * FROM memos WHERE archived = false ★ ← 必須 // 検索（Vol.7,8）にも追加が必要！ SELECT * FROM memos <where> AND archived = false ★ ← 忘れがち ... 他の検索条件 ... </where> // カテゴリ別件数などにも影響 // → 「archived=false」を入れ忘れると // アーカイブ済みメモが漏れて表示される ★ // ─── 対策：デフォルトで除外する設計 ─── // 共通の検索条件にする・ビューを使う等 // → 「条件追加を忘れない」工夫 // → 本シリーズは各クエリに明示的に追加
POINT — ソフトデリートは「全クエリへの影響」を忘れない

論理削除の最大の落とし穴は「通常クエリ全部に archived = false を追加し忘れる」こと。一覧・検索・件数集計——すべてに影響します。1か所忘れると「アーカイブしたはずのメモが検索に出る」バグに。導入するなら「全クエリへの影響を洗い出す」のが鉄則。だからこそテスト（別冊S3）で「アーカイブ済みが一覧に出ない」を検証するのが重要です。

05

状態の持たせ方

STATE MANAGEMENT

ピン留め・アーカイブのような「状態」をどう持つか。boolean フラグが基本ですが、状態が増えると設計の判断が要ります。本編Vol.9 の pinned・archived の設計を考えます。

状態フラグの設計boolean か enum か
// メモが持つ状態
  • pinned：ピン留めされているか（true/false）
  • archived：アーカイブされているか（true/false）

それぞれ独立した boolean。「ピン留めかつアーカイブ」も理論上ありうる（実際は稀）。

// boolean を増やす vs enum
方式向く場面
複数boolean独立した状態
enum（status）排他的な状態

ピンとアーカイブは独立なので boolean が自然。「下書き/公開/削除済み」のような排他的状態なら enum。

// 状態の持たせ方 // ─── 複数 boolean（独立状態）─── class Memo { private boolean pinned; // ピン留め private boolean archived; // アーカイブ } // → 2つは独立（組み合わせ自由） // → ピン留め＆未アーカイブ 等 ★ // ─── enum（排他的状態）─── // もし「状態が排他的」なら： enum MemoStatus { DRAFT, PUBLISHED, ARCHIVED } // → 同時に1つの状態しか取れない // → メモ帳のピン/アーカイブには不向き // （独立なので boolean が正解） // ─── 状態の判断基準 ─── Q. 2つの状態は同時に成立する? YES → 別々の boolean ★ NO（排他的）→ enum // ─── 状態とクエリの組み合わせ ─── 通常一覧： WHERE archived = false ORDER BY pinned DESC, ← ピンを上に updated_at DESC // → pinned=true が先頭に来る // → 状態が「並び順」にも影響 ★★★ // → 次の Section 06 で詳説
POINT — 「独立か排他か」で boolean か enum を選ぶ

状態設計の判断軸：「複数の状態が同時に成立するか」。ピン留めとアーカイブは独立（ピン留めしつつアーカイブも理論上可）なので別々の boolean。一方「下書き→公開→アーカイブ」のように1つしか取れない（排他的）ならenum（status列）が適切。boolean を増やしすぎると組み合わせが爆発するので、排他的なら enum でまとめるのが整理のコツです。

06

並び順の設計

SORT ORDER DESIGN

並び順は地味ですがUXの要。「ピン留めを上に」「更新順／作成順／名前順を選べる」。複数キーのソートとユーザーが選べる並び替えの設計を学びます。本編Vol.9 で実装します。

並び順の設計複数キー＋ユーザー選択
// ピン留めを最優先に

並び順は複数キーで決まる。「まずピン留め、その中で更新順」。ORDER BY pinned DESC, updated_at DESC。

// ユーザーが選べる並び替え
  • 更新順（既定）
  • 作成順
  • 名前順（タイトル）

並び順をユーザーが選べると自由度が上がる。ただし安全に実装（別冊S6の ${} 注意）。

// ソート列の安全性

ソート列を動的にするとき、ユーザー入力を直接 ORDER BY に入れない（別冊S6）。ホワイトリストで許可した列だけ。

// 並び順の設計 // ─── 複数キーソート（ピン優先）─── SELECT * FROM memos WHERE archived = false ORDER BY pinned DESC, ← まずピン留めを上 updated_at DESC; ← その中で更新順 // → ピン留めメモが常に先頭 ★★★ // ─── ユーザーが並び順を選ぶ ─── // URL: /memos?sort=created // SearchForm に sort を追加 // ─── 安全なソート列の扱い（別冊S6）─── NG: ORDER BY ${sort} // → ユーザー入力を直接 → インジェクション ★ OK: ホワイトリストで検証 String orderBy = switch (sort) { case "created" -> "created_at DESC"; case "title" -> "title ASC"; default -> "updated_at DESC"; }; // → 許可した値だけ → 安全 ★★★ // → Mapper には検証済みの値を渡す // ─── 並び順とピンの組み合わせ ─── // どの並び順でも「ピン留めは上」を維持： ORDER BY pinned DESC, ${検証済みの並び順} // → ピン優先は常に先頭キー ★ // ─── UX：既定値を賢く ─── // 何も選ばなければ「更新順」 // → 最近触ったメモが上＝自然 ★
POINT — ソート列を動的にするなら必ずホワイトリスト

並び替えでソート列をユーザーが選べるようにするとき、別冊S6の ${} の危険が再来します。ORDER BY ${sort} でユーザー入力を直接使うとSQLインジェクション。必ず「許可した列名だけ通すホワイトリスト」（switch文等）で検証してから使う。「ユーザーが選ぶ＝ユーザー入力＝危険」。動的なソートこそ、安全性の意識が必要です。

07

取り消せるUX

UNDOABLE UX

良いUXの核心「取り消せる」。削除前の確認、削除後の取り消し（Undo）、アーカイブという緩衝。ユーザーが「間違えても大丈夫」と感じられる設計を学びます。

「間違えても大丈夫」の設計3つの安全策
// 取り消せるUXの3パターン
策1
事前確認
「削除しますか?」のダイアログ。実行前に止める。
策2
事後Undo
削除後「取り消す」リンク。実行後に戻せる。
策3
緩衝（アーカイブ）
即削除せず一旦隠す。後でいつでも戻せる。
// メモ帳での適用
  • 完全削除 → 事前確認（取り返しつかないから）
  • アーカイブ → 緩衝（いつでも復元可）
// 取り消せるUXの実装パターン // ─── 策1：事前確認（物理削除）─── // 完全削除は取り返しつかない // → 確認ダイアログ <button onclick= "return confirm('完全に削除しますか?')"> 完全削除 </button> // → 誤クリックを1段階で止める ★ // ─── 策3：緩衝（アーカイブ）─── // アーカイブは論理削除＝戻せる // → 確認すら不要（戻せるから） [アーカイブ] → 隠れる [アーカイブ一覧で復元] → 戻る // → 「間違えても1クリックで戻る」★★★ // ─── 削除の安心レベル ─── 通常メモ ↓ アーカイブ（緩衝・戻せる） ↓ 確認不要（安全だから） アーカイブ済み ↓ 完全削除（取り返しつかない） ↓ 確認必須（危険だから）★ 本当に消える // ─── 確認の使い分け ─── 戻せる操作（アーカイブ）：確認不要 → 気軽に操作できる ★ 戻せない操作（完全削除）：確認必須 → 慎重にさせる ★ // → 「危険度に応じた摩擦」を設計 ★★★ // → 安全な操作は軽く、危険な操作は重く
POINT — 「危険度に応じた摩擦」を設計する

確認ダイアログは「摩擦（手間）」です。摩擦は多すぎるとうざく、少なすぎると事故る。コツは「危険度に応じて摩擦を変える」。戻せるアーカイブは摩擦ゼロ（即実行）で気軽に。戻せない完全削除は摩擦あり（確認）で慎重に。「全部に確認」も「全部確認なし」も悪いUX。操作の危険度を見極めて、適切な摩擦を置くのが設計です。

08

UX設計あるある失敗

UX PITFALLS

UX・状態設計で初心者が踏みやすい失敗を5つ。確認の乱用・論理削除の条件漏れ・ソート列の危険など。技術と気持ちの両面から回避策を学びます。

UX設計あるある 5選先人の轍を踏まない

① 論理削除の条件漏れ ★★★

  • 一部クエリに archived=false を入れ忘れ
  • アーカイブ済みが漏れて表示
  • 対処：全クエリを洗い出し、テストで検証（Section 04）

② 確認ダイアログの乱用

  • 何でも「本当に?」→ うざい・形骸化
  • 対処：危険な操作だけ確認（Section 07）

③ ソート列にユーザー入力を直接 ★★★

  • ${sort} でインジェクション
  • 対処：ホワイトリスト検証（別冊S6・Section 06）

④ 取り消せない破壊的操作

  • 確認もUndoもなく即完全削除
  • 対処：アーカイブ（緩衝）か確認を入れる

⑤ 状態が増えすぎて管理不能

  • boolean を乱立させ組み合わせ爆発
  • 対処：排他的なら enum でまとめる（Section 05）
// あるある失敗の対処 // ─── ① 論理削除の条件漏れ検出 ─── // テストで守る（別冊S3）： @Test void アーカイブ済みは一覧に出ない() { // メモを作ってアーカイブ service.archive(memoId); // 通常一覧を取得 var list = service.findActive(); // アーカイブ済みが含まれない assertThat(list) .noneMatch(m -> m.getId().equals(memoId)); // → 条件漏れがあれば落ちる ★★★ } // ─── ③ ソート列の安全化 ─── NG: ORDER BY ${userSort} OK: switch で許可リスト → 検証済み値のみ // ─── 良いUX設計の条件 ─── ✓ 戻せる操作を基本に（アーカイブ） ✓ 危険な操作だけ確認（摩擦を使い分け） ✓ 論理削除は全クエリに条件＋テスト ✓ ソート列はホワイトリスト ✓ 状態は独立=boolean/排他=enum ★★★
POINT — 論理削除のバグは「テストでしか気づけない」

論理削除の条件漏れ（①）は見た目では気づきにくい。アーカイブした直後は問題なくても、「検索したらアーカイブ済みが混ざる」のは特定の操作でしか露見しない。だから「アーカイブ済みが各画面に出ないこと」をテストで網羅するのが唯一の確実な防御。別冊S3の「テストは変更の安全網」が、論理削除という横断的な変更でこそ威力を発揮します。
